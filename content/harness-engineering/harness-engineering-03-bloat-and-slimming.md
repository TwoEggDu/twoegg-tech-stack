## 这篇解决什么问题

外面讲 Harness 的文章基本只讲"该加什么"。一个 Harness 跑到 Growth 后期会发现一个反直觉的事实：

**Harness 真正的健康指标不是规则有多全，而是有没有勇气主动放弃。**

这篇要回答的是：

- Bloat 的具体信号长什么样
- 哪些规则该删、哪些该合、哪些该降级为"建议"
- 瘦身的实操步骤
- 为什么"主动放弃"反而提高 AI 一次交付可用率

## Bloat 的五个具体信号

不是"CLAUDE.md 超过 N 行"这种粗暴指标。Bloat 的真实信号要细一点。

### 信号一：AI 在该执行和该确认之间反复横跳

- 表现：同一个任务里，AI 写两行就停下来问你确认，确认完写两行又停
- 根因：多条规则互相约束，每条都触发了"该问人"，但实际只有一条相关

### 信号二：规则之间出现"该看哪条"的冲突

- 表现：AI 说"按 A 规则我应该这样做，但 B 规则又说不能这样"
- 根因：规则在不同时期写的，没人做过整体一致性审查

### 信号三：Skill 被触发后 AI 还要花一半时间澄清这个 Skill 干嘛的

- 表现：触发 Skill 之后第一句话不是开始做事，是"这个 Skill 是要我做 X 对吗"
- 根因：Skill 的 description 写得既不像"什么时候触发"也不像"做什么"

### 信号四：一条规则要打三层比喻才能解释清楚

- 表现：CLAUDE.md 里某条规则后面跟着三段"也就是说"
- 根因：规则本身在表达一个不该被规则化的偏好

### 信号五：删一条规则之后整体表现反而变好

- 表现：你心血来潮删了一条不那么重要的规则，AI 一次交付可用率反而上去
- 根因：这条规则之前其实在制造混淆

## 瘦身的三类手术

针对 Bloat 的处理不是"全删"，是分类处理。

### 手术一：删

- 适用：那条规则从来没被违反过、或者违反了也没人在意
- 操作：直接删
- 警告：删之前确认它真的没有在保护某个隐式假设。Drift 期的"看起来没人在意"经常是因为代码已经偷偷符合那条规则——删了之后下一次有人改代码就会破

### 手术二：合

- 适用：两三条规则其实在讲同一件事的不同侧面
- 操作：合成一条，原来的多条规则用一句话承载
- 警告：合并后要重新走一次一致性检查，避免合出新的内部冲突

### 手术三：降级

- 适用：那条规则其实是偏好不是约束、违反它不会出错只是不优雅
- 操作：从硬规则降级为软建议，或者从 CLAUDE.md 挪到 Skill 描述里
- 警告：降级不是删——AI 偶尔还是要能看到，只是不让它在每次任务都触发

## 瘦身的实操步骤

不是凭感觉删。下面是一个可重复的瘦身流程。

### Step 1：清理输入

- 把 CLAUDE.md / 全部 Skill 描述 / 任何 AGENTS.md 都打印出来或拷到一个文件里
- 给每条规则编号

### Step 2：分类标注

- 每条规则标三个标签之一：被违反过 / 没被违反过但保护着某假设 / 一次性偏好
- 第一类是金子（保留）
- 第二类要查代码（可能是 Drift）
- 第三类是 Bloat 候选

### Step 3：冲突扫描

- 把所有规则两两交叉，找"AI 同时满足这两条"是否成立
- 不成立的规则对 → 合并或择一删

### Step 4：影响面预估

- 删之前问：这条规则保护的是什么场景
- 在最近 N 次任务里能找到至少 1 次该场景吗
- 找不到 → 候选删
- 找得到 → 候选降级

### Step 5：分批执行

- 一次只动 3-5 条规则
- 删后跑 1 周看效果
- AI 行为变好 → 继续
- AI 行为变差 → 立刻 revert，那条规则是真的有用

## 为什么"主动放弃"反而提高一次交付可用率

这是一个反直觉但可验证的事实。原因有三：

### 原因一：AI 的注意力是有限资源

- 每加一条规则，AI 在每次任务开场要把它过一遍
- 规则太多时，"过一遍"变得机械化、AI 自己会跳过细节
- 删掉低价值规则等于把注意力让给高价值规则

### 原因二：规则冲突的代价指数级

- N 条规则两两交叉是 N² 量级
- 100 条规则可能有几千对潜在冲突
- 删一条规则可能消除几十个冲突路径

### 原因三：人的维护成本也是 Harness 一部分

- Harness 越厚，人越懒得审计
- 没人审计的 Harness 必然 Drift
- 主动瘦身是给未来的自己留口气

## 一个判断"该不该加"的快速门禁

加规则之前，过这三关：

1. **这条规则保护的失败模式，最近一个月真的发生过吗？**
   - 没发生过 → 不该加
2. **这条规则能不能写成机械化门禁（git hook / CI check）？**
   - 能 → 写成机械化的，不写进 CLAUDE.md
   - 不能 → 进入第 3 关
3. **这条规则跟现有规则有没有冲突？**
   - 有 → 先处理冲突再加
   - 没有 → 可以加

过不了三关的，先放进"候选区"——可以是项目根的一个 `harness-pending.md`——攒够 5 条再审一次。多数候选会自己消失。

## 收束

Bloat 不是 Harness 设计错误，是它健康长大的自然产物。

但 Bloat 不处理就会变成 Drift，Drift 不处理就会变成 Sunset。瘦身这一步是 Harness 从 Growth 进入长期稳定运营的必经之路。

最短结论是：

**Harness 真正的健康指标，不是它能管多少事，而是它愿意主动放弃多少事。**
