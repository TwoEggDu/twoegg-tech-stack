## 这篇解决什么问题

AI Coding 最容易让人产生错觉的地方是：只要模型再强一点，工具再智能一点，AI 就能稳定接手更多工程任务。

实际用下来不是这样。

模型确实能写代码，能读文件，能跨文件修改，也能跑测试。但它进入一个真实项目时，面对的不是一道孤立编程题，而是一套带历史包袱的工程环境：目录结构、生成代码、业务约定、构建脚本、版本边界、测试入口、发布门禁、团队口头规则。

没有这些上下文，AI 很容易写出一种很危险的代码：语法正确，风格也像项目，但业务语义是错的。

比如它可能会：

- 修改了生成代码目录，却不知道下次生成会覆盖
- 只改了客户端缓存，漏掉服务器协议和数据定义
- 新增字段时用了看起来合理的类型，却没有遵守配置表单位约定
- 写完代码后没有跑对应生成器、构建或烟测
- 在 Claude、Codex、DeepSeek 之间切换时丢掉了上一轮已经确认的边界

这些问题不是单靠 prompt 能解决的。prompt 能改善一次回答，但不能把项目规则、执行流程和验证闭环稳定沉淀下来。

我把这层东西称为自己的 **AI Coding Harness Engineering**。

这里的 Harness 不是替代 CLAUDE.md 或 Skill，而是站在它们之上的组织层：它决定什么时候读取上下文、什么时候加载领域规则、什么时候执行任务、什么时候验证和回写。

它要解决的不是“AI 能不能写代码”，而是：

> **怎样让 AI 在明确边界内，稳定接手工程任务，并且每一步都能被验证、交接和复盘。**

## 为什么 AI Coding 不能只靠更强模型

更强模型能提高上限，但不能替你补齐工程系统。

一个真实项目里的 AI Coding 失败，通常不是模型完全不会写代码，而是它缺少四类信息。

第一类是 **项目上下文**。它不知道项目是什么结构，哪些目录是生成物，哪些模块互相依赖，哪些命令必须在修改后执行。这个问题前面用 [CLAUDE.md]({{< relref "ai-empowerment/ai-empowerment-05-claude-md-project-context.md" >}}) 解决了一部分。

第二类是 **领域规则**。它不知道某个 UI 框架的生成代码不能改，不知道配置表字段之间的隐式依赖，不知道协议生成之后还有客户端和服务器两套产物。这个问题前面用 [Skill]({{< relref "ai-empowerment/ai-empowerment-06-skill-domain-knowledge.md" >}}) 解决了一部分。

第三类是 **任务流程**。它不知道一个需求应该先改数据定义，还是先改 UI；不知道什么时候该停下来让人确认；不知道一个阶段的输出应该是工作单、骨架、代码，还是风险清单。这个问题在 [AI 辅助开发工作流]({{< relref "ai-empowerment/ai-empowerment-07-dev-workflow-integration.md" >}}) 里已经开始出现。

第四类是 **验证闭环**。它不知道改完之后该跑什么，构建失败哪些可以忽略，哪些必须处理；也不知道一次失败应该沉淀成规则、测试、文档，还是只是一条临时提醒。

这四类信息如果都靠人每次临场解释，AI Coding 就会一直停留在“单次辅助”阶段。你会发现自己并没有真正省下判断成本，只是把“写代码”换成了“反复纠正 AI”。

Harness 要做的事，就是把这些临场解释变成系统。

## Harness 的最小模型

我现在不把 Harness 理解成一个平台，也不把它理解成一个超长 prompt。

对我来说，第一版 Harness 只需要五层：

```text
Context：AI 做事前必须知道什么
Rules：哪些约定不能违反
Workflow：这类任务按什么步骤推进
Checks：怎么证明它没有乱来
Memory：这次踩坑怎样变成下次少犯错
```

这五层不是新造一套分类，而是对前面四类缺口的工程化回应：Context 对应项目上下文，Rules 对应领域规则，Workflow 对应任务流程，Checks 对应验证闭环，Memory 负责把失败反馈重新写回前四层。

这五层各自解决不同问题。

### Context：上下文入口

上下文入口回答的是：AI 进项目后先读什么。

例如在这个站点里，AI 写文章前必须知道：

- 根 `doc-plan.md` 只做系列级路由，不放篇级清单
- `docs/*-series-plan.md` 才是某个系列的 canonical plan
- `content/` 里是实际发布文章
- frontmatter 和 Hugo shortcode 的引号规则很容易导致构建失败
- 私仓 Unity C++ 源码不能直接搬进公开文章

这些东西如果长期靠口头提醒，很容易在重复任务中遗漏。更稳的做法，是把它们写在稳定入口里，例如 `AGENTS.md`、系列 plan、Harness 主流程文档。

### Rules：领域规则

规则回答的是：哪些事情看起来能做，但做了会出错。

比如：

- 生成代码目录不能手改
- 文章写完后要先回写系列 plan，再考虑根 `doc-plan.md`
- `DATA-TODO` 只能标真实待补数据，不能用虚构数字填上
- 不确定版本和平台时，要写 `[待核查]` 或补验证设计

规则和上下文的区别在于：上下文让 AI 知道项目是什么，规则限制 AI 不能怎么做。

### Workflow：任务流程

流程回答的是：一类任务应该怎样推进。

拿这篇文章本身举例，我没有直接让 AI 写正文，而是先跑了一次内容生产 Harness：

```text
选题输入
-> 编辑定位
-> 技术深挖
-> 验证设计
-> 初稿骨架
-> 人工确认主线
-> 正文起草
-> 风险审查
-> 解释桥接
-> 系列一致性
-> 最小改写
-> Hugo 构建
-> 回写系列状态
-> 反馈沉淀
```

这条流程的重点不是显得正式，而是防止 AI 在边界没确认时直接写正文。

代码任务也一样。修一个 bug、加一个小功能、重构一个模块，不应该共用同一个模糊指令。每类任务都要有自己的阶段、输入、输出和停止点。

### Checks：验证门禁

验证回答的是：怎样证明 AI 的输出可用。

文章任务里，最小验证是 Hugo 构建、frontmatter、relref、系列状态回写。代码任务里，验证可能是单测、构建、代码生成、关键路径手测、CI、静态分析、截图或日志。

这里最重要的判断是：**没有验证动作的 AI 任务，不应该被视为完成。**

AI 说“我已经改好了”没有意义。真正有意义的是：

- 改了哪些文件
- 跑了哪些命令
- 哪些检查通过
- 哪些检查没跑，为什么没跑
- 哪些风险仍然要人工确认

验证门禁这一层不在本系列里完整展开。它和 [质量护栏]({{< relref "code-quality/quality-gate-how-to-connect-automation-and-ai-review.md" >}}) 的思路是同一件事：人做判断，机器做可重复证明，流程负责把证明接起来。

### Memory：反馈沉淀

记忆回答的是：这次犯过的错，下次怎样不再犯。

我不太相信“在聊天里提醒过 AI 一次，它以后就记住了”。跨对话、跨模型、跨工具之后，这种记忆基本靠不住。

更可靠的做法是把错误沉淀回文件：

| 错误类型 | 沉淀位置 |
|---|---|
| 项目级规则遗漏 | `AGENTS.md` |
| 某类任务流程遗漏 | `.agents/harness/*.md` |
| 文章写法问题 | `docs/article-writing-method.md` |
| 系列边界变化 | `docs/*-series-plan.md` |
| 需要真实数据 | 正文 `DATA-TODO` |
| 需要项目经验 | 正文 `EXPERIENCE-TODO` |

Harness 真正开始变强，不是第一次写出来的时候，而是它能吸收失败的时候。

## 我会如何搭建 v0

第一版不要做大。

我现在更倾向于从一个高频、低风险、反馈快的任务开始。对这个站点来说，就是内容生产；对一个游戏项目来说，可能是“新增一个小功能的骨架代码”“修一个明确 bug”“补一条自动化验证”。

v0 的目录可以很简单：

```text
.agents/
  harness/
    content-production.md
    agent-compatibility.md
  skills/
    col-editor/
    col-drilldown/
    col-verify/
    col-draft/
    col-risk/
    col-bridge/
    col-consistency/
    col-rewrite/
docs/
  ai-empowerment-series-plan.md
  article-production-workflow.md
  article-writing-method.md
```

这里有三个关键点。

第一，**核心规程（Harness Core）要写成普通 Markdown**。不要把它绑死在某个 Agent 的专属命令里。在具备文件读取能力的环境里，Codex、Claude、DeepSeek 都可以读取；如果没有，就需要粘贴必要上下文。工具能力差异放到工具适配层（Agent Adapter），核心规程保持通用。

第二，**阶段产物要固定格式**。编辑工作单、七层深挖稿、验证设计、初稿骨架、风险清单、交接状态（Harness State），这些都应该有相对稳定的输出格式。格式稳定，下一阶段才能接得住。

第三，**停止点要写清楚**。什么时候只能输出工作单，什么时候必须等作者确认，什么时候可以写正文，什么时候必须跑构建。没有停止点，AI 最容易一路顺滑地把错误扩散到后面。

我现在给 v0 定的目标不是“AI 自动写完一切”，而是：

> 对同一类任务，AI 能稳定知道该读什么、产出什么、停在哪里、怎么验证。

这已经足够有价值。

## 一次任务在 Harness 里的完整流转

一次任务进入 Harness 后，不应该直接变成“请帮我改”。

我更希望它先进入一个状态机。

这里的状态机不是指一定要写一个运行时框架，而是把一次 AI 任务拆成几个固定阶段，每个阶段都有输入、输出和停止条件。

```text
Intake
-> Context
-> Plan
-> Execute
-> Verify
-> Handoff
-> Learn
```

### Intake：任务进入

先判断任务类型。

这是写文章、修 bug、加功能、审稿、重构、补测试，还是只是在做方案讨论？任务类型不同，后面的上下文和验证完全不同。

这一步的输出不需要复杂，最少要说清：

- 当前任务是什么
- 属于哪一类任务
- 需要读哪些上下文
- 是否有明确停止点

### Context：装载上下文

根据任务类型读取上下文。

写文章就读系列 plan、写法规范、相邻文章。改代码就读项目结构、相关模块、测试入口、生成规则。做发布或构建任务就读交付流程和质量门禁。

上下文不是越多越好。太多无关上下文会干扰模型判断。Harness 要做的是选择正确上下文，而不是把整个仓库都塞进去。

### Plan：阶段计划

AI 应该先输出计划或工作材料，而不是马上改最终产物。

这一步的价值在于让人先检查方向。如果方向错了，此时修正成本最低。

对这篇文章来说，Day 1 就先产出了编辑工作单、深挖分析、验证设计和初稿骨架。等主线确认后，才进入正文起草。

### Execute：执行

执行阶段才真正改文件或写正文。

但执行不是自由发挥。它应该严格使用前面确认过的边界：

- 不重讲 05 / 06 / 07
- 不编造效率数字
- 需要真实数据的位置用 `DATA-TODO`
- 需要项目经验的位置用 `EXPERIENCE-TODO`

代码任务同理。确认过“不改生成目录”，执行阶段就不能再碰生成目录。

### Verify：验证

执行之后必须验证。

文章任务跑 `hugo`。代码任务跑对应测试、构建、生成器或最小手测。验证失败就回到执行或计划阶段，不应该把失败藏在最后一句“可能还需要你检查一下”里。

### Handoff：交接

跨 Agent 工作时，交接不能只靠上一轮聊天记录。

我现在会要求每个阶段结束时输出一个交接状态（Harness State）：

```text
Harness State
- Task: 当前任务
- Stage: 当前阶段
- Context read: 已读取的关键文档
- Decisions: 已经确认的边界和判断
- Open questions: 需要作者或下一个 agent 确认的问题
- Next stage: 建议下一步
- Changed files: 已修改文件
- Verification: 已运行检查
```

这个状态块看起来很普通，但它至少能降低跨 Agent 切换时的重复解释成本：你可以从 Codex 切到 Claude，再切到 DeepSeek，不必每次都从项目背景重新讲起。

### Learn：沉淀

最后一步不是“结束对话”，而是判断这次有没有可复用经验。

如果 AI 因为规则缺失犯错，就补规则。如果因为上下文入口不清楚犯错，就补上下文入口。如果因为没有验证才漏问题，就补检查项。如果只是一次性偏好，就不要写进全局规则。

沉淀要克制。Harness 不是越厚越好。

## 怎么验证 Harness 真的有效

我现在不急着用“AI Coding 率”做第一指标。

这个指标很容易被算虚：AI 生成了很多行代码，不等于它承担了更多工程判断；AI 写了大部分骨架，也可能把最难判断的部分留给人返工。

v0 更适合先看五个简单指标：

| 指标 | 看什么 |
|---|---|
| 一次交付可用率 | AI 第一次输出能不能直接进入下一阶段 |
| 人工返工点 | 人主要改结构、事实、风格、边界还是链接 |
| 规则违例数 | 是否违反项目规则、生成规则、目录边界、frontmatter |
| 验证通过率 | 构建、测试、生成器、检查命令是否一次通过 |
| 反馈沉淀率 | 失败是否回写成规则、检查项或 TODO |

这些指标不需要一开始就自动化。先用很朴素的方式记录几次任务就够。

我真正关心的是两件事。

第一，同类错误有没有减少。

如果 AI 上周错在“没读 series plan”，这周还错在同一个地方，优先说明 Harness 没有把规则稳定送进流程；同时也要排除模型能力和上下文窗口限制。

第二，人是不是把精力从“反复解释规则”转移到了“做真正判断”。

AI Coding 的价值不在于替人按键，而在于把重复、模式化、可验证的部分接走，让人回到架构、业务语义、边界取舍和风险判断上。

## 什么时候扩展，什么时候停下来

不是所有东西都该放进 Harness。

适合进入 Harness 的任务通常有四个特征：

- 高频出现
- 规则稳定
- 失败模式重复
- 验证方式明确

比如文章生产、配置表检查、协议生成、UI 生成代码使用、构建前检查、PR 风险扫描，都比较适合。

不适合的任务也很明确：

- 一次性探索
- 需求本身还没想清楚
- 核心是产品判断或审美判断
- 没有可运行验证
- 规则还在频繁变化

这类任务强行 Harness 化，反而会让流程变重。

我会用几条红线判断是否该停下来修 Harness：

- 同类错误连续复发
- AI 总是越过人工确认点
- 检查命令长期写着但不执行
- 规则过时后没人维护
- 文档越来越厚，但实际任务没有更稳

遇到这些信号，不应该继续堆更多规则，而应该回头问：哪个阶段没站住？上下文入口错了，还是停止点错了？验证是不是太重，所以大家都绕过了？

## 收束

AI Coding Harness Engineering 的核心，不是让 AI 变成完全自动的工程师。

它更像是给 AI 工作搭一条可控轨道：任务怎么进来，读什么上下文，按什么阶段走，在哪里停，怎么验证，失败后沉淀到哪里。

在这个意义上，Harness 接住的是前面几篇文章的结果：

- [CLAUDE.md]({{< relref "ai-empowerment/ai-empowerment-05-claude-md-project-context.md" >}}) 解决全局上下文
- [Skill]({{< relref "ai-empowerment/ai-empowerment-06-skill-domain-knowledge.md" >}}) 解决领域规则
- [跨层开发工作流]({{< relref "ai-empowerment/ai-empowerment-07-dev-workflow-integration.md" >}}) 解决任务链路
- Harness 把它们接成流程、验证和反馈闭环

最短结论是：

**不要先追求 AI 写了多少代码，先让 AI 每一步都可理解、可验证、可交接、可复盘。**

如果后续真实跑过几轮任务，下一篇更适合单独展开：怎样记录 AI Coding 的指标，怎样复盘失败，怎样判断 Harness 真的让系统变好了。
