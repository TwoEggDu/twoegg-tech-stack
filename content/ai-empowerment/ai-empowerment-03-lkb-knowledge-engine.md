## 一、LKB 的定位：不是问答系统，是内容加工厂

LKB（LLM Knowledge Base）是一个由 AI 维护的项目知识库。它的目录结构很简单：

```
KB/
├── CLAUDE.md        # 工作流规范（AI 必须遵守的规则）
├── raw/             # 原始来源（只读）
│   ├── articles/    # 文章、竞品分析
│   ├── papers/      # 论文
│   ├── books/       # 书籍章节
│   └── assets/      # 图片等附件
├── wiki/            # AI 维护的知识层
│   ├── index.md     # 内容索引
│   ├── log.md       # 操作日志
│   ├── concepts/    # 概念页
│   ├── entities/    # 实体页
│   ├── sources/     # 来源摘要
│   ├── comparisons/ # 比较分析
│   └── queries/     # 查询存档
└── concepts/        # 高层概念文档
```

核心原则：
- `raw/` **只读不写**——原始资料是单一事实来源，AI 绝不修改
- `wiki/` **AI 全权维护**——创建、更新、交叉引用、保持一致性
- 人的角色是**策划来源和指导分析方向**

LKB 不直接面向团队成员。它的产出——`wiki/` 里的结构化文档——才是最终被发布到 Hugo Wiki、被 Dify 索引的内容。

## 二、三个核心工作流

### Ingest（摄入）

把原始资料变成结构化知识。

**触发**：有人把新资料放进 `raw/` 目录，然后让 AI 处理。

**流程**：
1. AI 完整读取原始文件，提取关键信息和主要论点
2. 在 `wiki/sources/` 创建来源摘要页——一句话总结、主要论点、关键细节
3. 在 `wiki/concepts/` 创建或更新概念页——整合新来源的见解
4. 在 `wiki/entities/` 创建或更新实体页——涉及的人物、系统、模块
5. 更新 `index.md` 索引和 `log.md` 日志

一次 Ingest 通常会产出 3-5 个 wiki 页面。这些页面之间用 Obsidian 风格的 `[[双括号链接]]` 交叉引用。

### Query（查询）

基于 wiki 内容回答问题。

**触发**：有人提问。

**流程**：
1. AI 读取 `wiki/index.md` 定位相关页面
2. 读取相关 wiki 页面获取上下文
3. 综合 wiki 内容生成回答
4. 如果回答有价值，存档到 `wiki/queries/`

关键区别：Query 不是通用问答。AI 被要求**优先基于 wiki 内容回答**，信息不足时明确标注并用通用知识补充。这保证了回答的可追溯性。

### Lint（健康检查）

保持 wiki 的质量和一致性。

**检查项**：
- 矛盾检测：不同页面之间的冲突声明
- 结构检查：孤立页面（无入链）、断链
- 内容质量：页面过于简短、缺少交叉引用

## 三、为什么 Claude Code 能做 Dify 做不了的事

Dify 的 RAG 问答和 LKB 的 Query 看起来都是"问答"，但底层能力完全不同：

| 能力 | Dify RAG | LKB + Claude Code |
|------|----------|-------------------|
| 读自然语言文档 | 通过向量检索，效果不错 | 直接读取，理解更完整 |
| 读代码 | 向量检索效果差 | 直接读源文件，跨文件追踪 |
| 读配置表 | 无法理解 Excel 结构 | 可以读 Excel、理解字段关系 |
| 跨层关联 | 只能检索单篇文档 | 能同时读协议、服务器代码、配置表 |
| 产出 | 一次性回答 | 持久化的结构化文档 |

举一个例子。策划问"梦灵自动上阵的逻辑是什么"。

Dify 的检索结果取决于 wiki 里有没有写过相关文档。如果没有，它答不了。

Claude Code 可以直接读 `S_PetModule.cs`、`C_PetModule.cs`、`T_TemporaryTeam.cs`，追踪调用链，理解自动上阵的完整逻辑，然后把结论写成一篇结构化文档。这篇文档进入 wiki 后，Dify 下次就能回答了。

## 四、实际案例：M2 技能书系统 Ingest

项目中有一个实际的 Ingest 案例：分析竞品 M2 的技能书系统。

原始资料：一篇包含 20+ 截图的竞品分析文章，放在 `raw/articles/M2-技能书系统.md`。

Ingest 后的产出：

- `wiki/sources/m2-技能书系统.md` — 来源摘要，提取核心机制和数值结构
- `wiki/concepts/技能书系统.md` — 概念页，整合技能书的获取、穿戴、升级、图鉴等子系统
- `wiki/queries/技能书系统-系统改动清单.md` — 如果要实现类似系统，需要改动哪些模块
- `wiki/queries/技能书系统-通用系统依赖.md` — 技能书系统依赖哪些通用基础设施

从一篇原始文章，变成了 4 个结构化页面，每个页面聚焦不同维度，相互交叉引用。

## 五、文档质量对 Dify 检索效果的影响

把同样的知识以两种形式喂给 Dify：

**原始形式**：直接把竞品分析文章上传。文章结构是叙述性的，信息夹在截图说明和分析推测之间。

**经 LKB 整理后**：概念页有清晰的章节结构——功能概述、获取方式、穿戴规则、升级路径、数值结构。每个章节的信息密度高，没有冗余叙述。

向量检索的准确度在第二种形式上明显更好。原因是：

- 分块后每个 chunk 的信息集中度更高
- 章节标题和内容的语义一致性更强
- 没有叙述性过渡句干扰匹配

这就是 LKB 的真正价值——**不是替代 Dify，而是提升喂给 Dify 的内容质量**。
