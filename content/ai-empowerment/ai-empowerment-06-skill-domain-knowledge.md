## 这篇解决什么问题

[上一篇]({{< relref "ai-empowerment/ai-empowerment-05-claude-md-project-context.md" >}})讲了 CLAUDE.md 怎么给 AI 提供项目全局上下文。但 CLAUDE.md 有一个天然限制：**它是全局加载的，所有对话都会读**。

游戏项目有大量领域专业知识——FairyGUI 的 UI 组件命名约定、配置表字段之间的依赖关系、战斗系统的配置语义规则——这些信息量大、场景专用。全部塞进 CLAUDE.md 会导致上下文膨胀，跟当前任务无关的信息反而干扰 AI 的判断。

Skill 解决的是：**按需注入领域知识**。

## 这篇要回答什么

1. CLAUDE.md 和 Skill 的边界在哪
2. Skill 是什么，怎么运作的
3. 三个真实的 Skill 案例
4. 什么时候该写 Skill，什么时候靠 CLAUDE.md 就够
5. 维护成本和收益怎么评估

## 一、CLAUDE.md vs Skill 的边界

| | CLAUDE.md | Skill |
|---|---|---|
| 加载方式 | 每次对话自动加载 | 按需激活（用户触发或 AI 判断） |
| 内容特征 | 项目全局信息 | 特定领域的专业知识 |
| 典型内容 | 架构、目录、构建命令、已实现系统 | FairyGUI 约定、配置表规则、战斗配置语义 |
| 信息量 | 数百行，尽量精简 | 可以很长，因为只在需要时加载 |
| 更新频率 | 随架构变化更新 | 随领域规则变化更新 |

简单的判断标准：**如果这个信息跟 80% 以上的开发任务相关，放 CLAUDE.md；如果只跟某个特定类型的任务相关，做成 Skill。**

## 二、Skill 是什么

Skill 是一个 Markdown 文件，包含特定领域的指令、规则和参考信息。它以 Claude Code 的 Skill 机制加载——当用户执行相关任务（比如 `/fgui`）或 AI 判断当前任务匹配时，Skill 的内容被注入到对话上下文中。

一个 Skill 文件的典型结构：

```markdown
---
description: 触发条件描述
---

# Skill 名称

## 适用场景
什么时候用这个 Skill

## 核心规则
1. 规则一
2. 规则二

## 参考信息
相关文件路径、命名约定、示例代码

## 常见错误
容易踩的坑和正确做法
```

## 三、三个真实案例

### FGUI Skill — FairyGUI UI 开发规范

**问题**：FairyGUI 有自己的代码生成约定——生成的类在 `Fairy_Gen/` 目录，永远不能手动编辑；自定义实现在生成文件旁边创建继承类。通用 AI 不知道这些约定，经常建议直接修改生成文件。

**Skill 内容**：
- FairyGUI 的包/组件/绑定命名规则
- 生成代码目录结构和文件命名约定
- 自定义实现的正确方式（继承 vs 修改）
- 多语言变体的资源组织方式
- 常见错误：直接编辑 `Fairy_Gen/` 下的文件

**效果**：AI 在做 UI 相关任务时，不再建议编辑生成代码，能正确创建自定义继承类。

### Gameplay Skill — GameEditor 配置系统规则

**问题**：游戏配置系统的字段之间有复杂的语义依赖。比如一个技能配置里，`TargetType` 为"敌方"时 `Range` 字段才有意义；`EffectType` 为"持续"时才需要填 `Duration`。这些依赖关系不在代码里（代码只做运行时校验），在策划的配置约定里。

**Skill 内容**：
- 各配置编辑器的字段清单和类型
- 字段间的依赖关系和约束条件
- 引用链：哪些字段引用其他配置表的 ID
- 常见配置错误的排查路径

**效果**：AI 在辅助配置工作时，能提醒"这个字段需要先设置 X 为 Y 才有效"，而不是机械地填写所有字段。

### Excel-MCP Skill — 配置表读写约定

**问题**：项目的 Excel 配置表有特定的组织方式——表头在第几行、哪些列是 ID 列、不同 Sheet 的关系。直接让 AI 读写 Excel 容易出错。

**Skill 内容**：
- Excel MCP 工具的正确使用方式
- 项目配置表的目录结构和命名约定
- 表头格式和数据类型规范
- 跨表引用的验证规则

**效果**：AI 通过 MCP 工具读写 Excel 时，能正确识别表结构，不会误删表头或破坏引用关系。

## 四、什么时候该写 Skill

不是所有领域知识都需要做成 Skill。判断标准：

**该写 Skill**：
- 这个领域有**明确的规则**（不是模糊的最佳实践，是"这样做会出错"的硬规则）
- 规则**不在代码里可读**（代码只做运行时校验，规则在约定层面）
- AI **经常在这个领域犯同一类错误**（说明通用知识不够）
- 这类任务**频率足够高**（每周至少触发几次）

**不需要 Skill，CLAUDE.md 就够**：
- 信息量不大（几行就能说清楚）
- 几乎所有任务都需要这个信息
- 信息可以从代码中直接推断

**都不需要，让 AI 读代码就行**：
- 信息就在代码注释和函数名里
- 不涉及隐式约定或业务规则

## 五、维护成本和收益

Skill 的维护成本主要是：
- **初始编写**：梳理领域规则并结构化，通常 2-4 小时
- **持续更新**：规则变化时更新，取决于领域稳定性
- **验证**：确认 AI 在 Skill 加载后的行为符合预期

收益怎么评估：

| 指标 | 衡量方式 |
|------|---------|
| 错误减少 | Skill 覆盖领域内 AI 犯错的频率下降 |
| 沟通成本 | 开发者不再需要反复纠正 AI 的同一类错误 |
| 任务完成率 | AI 能独立完成的领域任务比例提升 |

经验值：一个写得好的 Skill，能把特定领域的 AI 犯错率从 30-40% 降到 5-10%。前提是规则梳理准确、覆盖了常见错误场景。
