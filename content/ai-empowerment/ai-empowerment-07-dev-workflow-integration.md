## 这篇解决什么问题

前面六篇分别讲了知识管理闭环和 AI 上下文管理。这篇把这些能力串起来，看 AI 在日常开发工作流中的实际角色。

游戏功能开发不是写一个文件就完事。一个新系统（比如头衔系统）需要跨越五层：数据定义 → 协议 → 服务器模块 → 客户端模块 → Unity UI。每一层有自己的约定、模式和容易出错的地方。

这篇不是 AI 编程教程，而是讲清楚：**在这个跨层联动的过程中，AI 该做什么、不该做什么、人的判断在哪里不可替代。**

## 这篇要回答什么

1. 游戏功能开发的五层结构是什么
2. AI 在每一层的具体角色
3. 一个完整功能从零到落地的 AI 辅助过程
4. 人和 AI 的边界线

## 一、五层结构

一个典型的游戏功能开发涉及以下五层：

```
第一层：数据定义（DP.Data）
  数据结构、枚举、配置表定义
      ↓
第二层：协议定义（DP.Data/Protocol）+ 代码生成
  请求/响应消息 → build_protocol.bat → GenClient + GenServer
      ↓
第三层：服务器模块（DP.Service.Logic/Modules）
  协议处理、业务逻辑、数据库操作
      ↓
第四层：客户端模块（DP.Client/Modules）
  客户端逻辑抽象（netstandard2.1，不依赖 Unity）
      ↓
第五层：Unity UI（TopHeroUnity/GameScripts/HotFix）
  FairyGUI 绑定、UI 逻辑、表现层
```

这五层之间有严格的依赖关系——改了第一层，后面四层都可能受影响。跨层一致性是最容易出 Bug 的地方。

## 二、AI 在每一层的角色

### 第一层：数据定义

AI 擅长：
- **参照已有模式生成数据结构**。项目里已经有 20+ 个系统的数据定义，AI 可以参照背包系统的 `Bag.Types.cs`、`Item.cs` 的模式来创建新系统的数据结构
- **配置表定义**。根据策划需求生成 `T_*.cs` 配置表类的骨架

人来判断：
- 数据结构的粒度和拆分方式
- 哪些字段需要持久化、哪些是运行时计算
- 与已有系统的数据边界

### 第二层：协议定义 + 代码生成

AI 擅长：
- **保持请求/响应的对称性**。参照已有协议（如 `P_Bag.cs`）的命名和结构模式
- **提醒运行代码生成器**。CLAUDE.md 里写了协议修改后需要执行 `build_protocol.bat`

人来判断：
- 协议的粒度：是一个大消息还是拆成多个小消息
- 通知 vs 请求/响应的选择
- 协议版本兼容性

### 第三层：服务器模块

AI 擅长：
- **生成模块骨架**。参照已有模块（如 `S_BagModule.cs` 的 ItemController、BagProvider 拆分方式）生成新模块的结构
- **一致性检查**。确保协议处理器和协议定义匹配，确保数据库操作和 ORM 实体一致

人来判断：
- 业务逻辑的正确性（伤害公式、奖励计算、条件判断）
- 并发和事务边界
- 性能敏感路径的实现方式

### 第四层：客户端模块

AI 擅长：
- **保持客户端/服务端模块对称**。服务端有 `S_TitleModule.cs`，客户端应该有 `C_TitleModule.cs`，网络处理、数据缓存的模式是一致的
- **协议处理代码的生成**。请求发送、响应处理、通知监听的模式高度统一

人来判断：
- 客户端缓存策略
- UI 更新时机和方式
- 弱网和断线重连的处理

### 第五层：Unity UI

AI 擅长：
- **FairyGUI 生成代码的使用方式**（如果加载了 FGUI Skill）
- **参照已有 UI 类的模式**创建新窗口的骨架代码

人来判断：
- 交互设计和动画细节
- 性能优化（哪些 UI 需要对象池、哪些可以每次创建）
- 与其他 UI 系统的协调

## 三、一个完整案例：头衔系统

以头衔系统为例，看 AI 辅助开发的完整过程。

### 起点

策划需求：玩家可以获得并展示头衔，头衔有激活条件，激活后提供属性加成。

### AI 辅助过程

**第一步：数据和协议定义**

开发告诉 AI："参照背包系统的模式，帮我定义头衔系统的数据结构和协议。"

AI 读取已有的 `Bag.Types.cs`、`D_Bag.cs`、`P_Bag.cs`，理解项目的数据定义模式，生成：
- `D_TitleData.cs` — 头衔存档数据
- `T_Title.cs` — 头衔配置表
- `P_Title.cs` — 头衔协议（激活请求、装备请求、同步通知）

开发审核：调整字段命名、确认持久化粒度、删除不需要的协议消息。

**第二步：运行代码生成**

AI 提醒执行 `build_protocol.bat`。这一步是纯自动化。

**第三步：服务器模块**

AI 参照 `S_BagModule.cs` 的模块拆分方式，生成 `S_TitleModule.cs` 的骨架——协议处理器注册、条件检查框架、数据库操作的 CRUD。

开发填写：激活条件的具体判断逻辑（需要理解业务规则）、属性加成的计算（需要理解属性系统）、条件适配器的注册（`TitleConditionAdapter.cs`）。

**第四步：客户端模块**

AI 生成 `C_TitleModule.cs`，包括协议请求/响应处理、本地数据缓存、状态查询接口。

开发审核：确认缓存策略、确认 UI 更新通知方式。

**第五步：Unity UI**

AI（加载 FGUI Skill 后）生成 UI 窗口的骨架类，正确继承 FairyGUI 生成的基类。

开发实现：具体的交互逻辑、动画效果、与头衔模块的数据绑定。

### 结果

五层代码中，AI 生成了大约 60-70% 的骨架代码。开发的工作集中在**业务判断、边界 Case 处理和 UI 细节**——这些是 AI 做不好的部分，也是开发真正应该花时间的部分。

## 四、人和 AI 的边界线

| AI 做 | 人做 |
|-------|------|
| 骨架代码生成 | 业务逻辑的正确性 |
| 模式复用（参照已有系统） | 架构决策（该不该拆、怎么拆） |
| 跨层一致性检查 | 性能和并发的判断 |
| 重复性代码（协议处理、CRUD） | 边界 Case 和异常处理 |
| 命名和格式的一致性 | 需求理解和设计取舍 |

一句话总结：**AI 做重复的事，人做判断的事。**

如果发现 AI 在某个领域频繁犯错，说明这个领域需要一个 [Skill]({{< relref "ai-empowerment/ai-empowerment-06-skill-domain-knowledge.md" >}})。如果发现团队在某个知识上频繁被问，说明这个知识需要进入 [知识闭环]({{< relref "ai-empowerment/ai-empowerment-01-team-knowledge-closed-loop.md" >}})。

## 收束

这个系列讲了两件事：

1. **知识管理闭环**（01-04）：Dify 暴露缺口 → LKB 生产知识 → Wiki 沉淀 → Dify 越来越能答
2. **AI 辅助开发**（05-07）：CLAUDE.md 给全局上下文 → Skill 注入领域规则 → AI 在跨层联动中做骨架、人做判断

这两件事不是独立的。知识闭环让团队的知识不再丢失，AI 辅助开发让开发者把时间花在真正需要判断的地方。它们共同解决的问题是：**让 AI 从个人的零散使用变成团队的系统化能力。**
