## 这篇解决什么问题

[上一篇]({{< relref "ai-empowerment/ai-empowerment-01-team-knowledge-closed-loop.md" >}})建立了知识闭环的整体模型。这篇展开第一层：**怎么让全团队都能从文档里获得 AI 回答**。

策划不会去翻代码，QA 不会去查 Git 日志。他们需要的是一个问答入口——用自然语言问，AI 从已有文档里找答案。这就是 Dify 在闭环里的角色。

## 一、技术选型：为什么是 Dify + Ollama

选型不是比功能列表，是看约束条件。我们的约束：

| 约束 | 选择结果 |
|------|---------|
| 内网部署，数据不能出公司 | 排除所有云端 API 方案 |
| 不想依赖外部服务的可用性 | 本地模型，Ollama 运行时 |
| 策划和 QA 也要用 | 需要 Web 界面，不能是命令行 |
| 团队 20 人，不值得自研 | 用成熟的开源平台 |
| 硬件有限，一台 RTX 3080 | 7B 模型是上限 |

Dify 提供了开箱即用的 RAG 问答平台：知识库管理、向量索引、对话界面、嵌入脚本。Ollama 提供本地模型运行时。两者通过 API 对接，部署在同一台服务器上。

Qwen2.5:7b 是当前中文效果最好的 7B 级别模型，显存占用约 6GB，在 RTX 3080 10GB 上有充足余量。单次推理约 2-4 秒，日常 20 人并发不高的情况下足够支撑。

## 二、硬件需求

| 项目 | 配置 | 备注 |
|------|------|------|
| 显卡 | NVIDIA RTX 3080 (10GB) | 7B 模型占 ~6GB，余量充足 |
| 内存 | 16GB+ | Docker + Ollama 同时运行 |
| 磁盘 | 预留 30GB+ | Docker 镜像 ~10GB + 模型 ~5GB + 索引 |

如果想升级到 14B 模型，显存不够的部分会自动溢出到内存，速度会下降但能跑。

## 三、部署核心流程

部署分四步。这里不写操作手册，讲每一步的关键决策点。

### 3.1 部署 Dify

Docker Compose 一键启动。关键决策：

- Dify 版本选稳定版，不要跟最新的 main 分支
- `.env` 里的 `SECRET_KEY` 必须改掉默认值
- 首次启动拉镜像需要 10-30 分钟

### 3.2 安装 Ollama 和模型

Ollama 装在宿主机上（不在 Docker 里），这样 GPU 访问最直接。

关键配置：环境变量 `OLLAMA_HOST=0.0.0.0`，否则 Docker 里的 Dify 无法访问宿主机上的 Ollama。

### 3.3 在 Dify 中接入模型

模型供应商选 Ollama，Base URL 填 `http://host.docker.internal:11434`。这是 Docker 访问宿主机的特殊地址，Windows 下开箱即用。

### 3.4 创建知识库并导入文档

Hugo Wiki 的 `content/` 目录下全是 `.md` 文件，可以批量上传到 Dify 知识库。上传后 Dify 自动做向量化索引。

索引方式选"高质量模式"——虽然慢一些，但检索准确度明显好于"经济模式"。

## 四、什么内容该入库

不是所有文档都值得入库。满足以下任一条件的内容才值得：

| 条件 | 示例 |
|------|------|
| 被多人重复问到 | "角色属性在哪配？""打包流程是什么？" |
| 涉及团队特有流程或配置 | 构建流程、分支策略、服务器环境 |
| Bug 排查后的经验总结 | 某个功能为什么表现异常、根因和修复方案 |
| 设计与实现存在差异 | 策划设计的是 A，实际做成了 B |
| 代码层面的关键逻辑说明 | 伤害计算公式、匹配算法流程 |

入库文档分四类，每类都有固定模板：

1. **功能实现说明** — 某个系统的核心逻辑，开发撰写
2. **Bug 复盘** — 问题排查经验沉淀，排查人撰写
3. **设计-实现差异记录** — 策划和开发共同确认
4. **FAQ** — 反复被问到的问题整理成问答对

模板的价值不在格式统一，而在**降低写文档的心理门槛**——填空比从零写容易得多。

## 五、系统提示词设计

提示词是整个系统的"人格"，决定了 AI 回答的风格和边界。核心原则：**答不了就说答不了，绝不编造。**

```
你是公司内部知识库助手。请严格根据提供的文档内容回答问题。

规则：
1. 如果文档中有相关内容，请基于文档给出清晰、准确的答案
2. 如果文档中没有相关信息，请明确回复"知识库中暂无此内容"，不要编造答案
3. 回答请使用中文，简洁准确
4. 如有来源文档，请在回答末尾注明文档名称
```

"知识库中暂无此内容"这句话是闭环的触发点。它告诉团队：这里有一个知识缺口需要填补。

## 六、向量检索的天然局限

Dify 的 RAG 问答效果取决于两个因素：文档质量和检索准确度。

检索准确度在自然语言文档上表现不错，但在以下场景有天然短板：

- **代码**：函数名、变量名在语义空间里没有好的表示，分块后上下文丢失
- **配置表**：Excel 表头和数值的语义关系很难被向量化捕捉
- **跨文档关联**：问题的答案需要综合多篇文档时，检索召回率下降

这些短板不是 Dify 的问题，是向量检索这个技术路线的固有限制。

解决方案不是换更好的检索引擎，而是**提升喂给 Dify 的文档质量**。一篇经过 LKB 整理的结构化文档，比一篇原始策划文档的检索效果好得多。
