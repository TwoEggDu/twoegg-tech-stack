## 这篇解决什么问题

你每次和 AI 对话都要重新解释："我们项目用 Unity 2022.3，有一个 TopHeroSLN 的 C# 解决方案，里面有 20 个项目，客户端 DLL 复制到 Unity 的这个目录……"

这不是 AI 的问题，是**上下文缺失**的问题。AI 没有长期记忆，每次对话都是白纸一张。

CLAUDE.md 解决的就是这个：**让 AI 每次对话都自动获得项目上下文**。

## 这篇要回答什么

1. CLAUDE.md 是什么，为什么它有效
2. 该写什么、不该写什么
3. 一个真实项目的 CLAUDE.md 长什么样
4. 有和没有 CLAUDE.md 的效果差异
5. 怎么维护，怎么防止过时

---

## 一、CLAUDE.md 是什么

CLAUDE.md 是放在项目根目录的 Markdown 文件。Claude Code 每次启动对话时会自动读取并加载到上下文中。

它的作用类似于你给一个新同事写的"项目入门指南"——但读者不是人，是 AI。

**为什么它有效**：AI 的能力 = 模型能力 × 上下文质量。同一个模型，在有项目上下文和没有项目上下文时表现差异巨大。CLAUDE.md 把每次都要口头说的信息固化成文件，AI 每次对话自动加载，效率提升是持续的。

## 二、该写什么

CLAUDE.md 的内容应该是**AI 无法从代码本身推断出来，但对理解项目至关重要的信息**。

### 必须写的

**项目概述和架构**：项目是什么、用什么技术栈、主要组成部分。AI 需要知道它面对的是一个 Unity MMO 而不是一个 Web 应用。

**目录结构和代码组织**：哪个目录放什么代码，项目之间的依赖关系。这决定了 AI 在哪里找代码、怎么理解模块边界。

**常用命令**：构建命令、运行命令、代码生成命令。AI 需要知道 `build_protocol.bat` 干什么、在哪里执行。

**开发模式**：添加新功能的标准流程（先定义数据 → 定义协议 → 生成代码 → 实现服务器 → 实现客户端 → Unity UI）。这让 AI 的建议符合团队的工作方式。

**重要约束**：Unity 版本、netstandard2.1 的兼容性要求、不要手动编辑生成代码的目录。这些是容易踩坑的红线。

### 不该写的

**会快速过时的细节**：当前正在修的 Bug、本周的工作计划。这些信息的有效期太短，维护成本高于收益。

**可以从代码推断的信息**：某个函数的具体实现、某个类的字段列表。AI 可以直接读代码获取这些。

**已经在 README 或其他文档里的内容**：不要重复。可以引用文件路径让 AI 按需读取。

## 三、一个真实项目的演进

我们的 DP 项目 CLAUDE.md 从零开始，经历了几个阶段：

### 阶段一：基础骨架（~50 行）

最初只写了项目概述、目录结构和构建命令。这已经让 AI 从"完全不知道你的项目"变成"知道这是一个什么样的项目"。

### 阶段二：架构说明（~200 行）

加入了三层架构（TopHeroSLN 逻辑层、服务器运行时、Unity 客户端运行时）、项目列表、数据流、代码共享模型。AI 开始能理解"改 DP.Data 里的协议后需要运行代码生成器"这种跨模块关系。

### 阶段三：系统清单和开发模式（~500 行）

加入了已实现系统的完整清单（背包、抽卡、任务、宠物等）、每个系统涉及的文件列表、添加新功能的标准流程。AI 现在能参照已有系统的模式来实现新功能。

### 当前状态

500+ 行，涵盖：项目概述、目录结构、常用命令、架构分层、20 个项目说明、数据流、代码共享模型、Unity 架构、已实现系统清单、开发模式、文件位置参考、重要约束。

## 四、有 vs 没有的效果差异

| 场景 | 没有 CLAUDE.md | 有 CLAUDE.md |
|------|---------------|-------------|
| "帮我加一个新协议" | 不知道协议定义在哪，不知道要运行代码生成 | 直接在 DP.Data/Protocol/ 创建，提醒你运行 build_protocol.bat |
| "背包系统在哪" | 全局搜索，可能找到无关结果 | 直接定位到 S_BagModule.cs、C_BagModule.cs、P_Bag.cs |
| "帮我实现一个新系统" | 按通用模式写，不符合项目约定 | 参照已有系统的模式，遵守 netstandard2.1 约束 |
| "这个 DLL 是从哪来的" | 不知道 DLL 依赖关系 | 知道 DP.Client.dll 来自 TopHeroSLN/Code/DP.Client/ |

效果差异在项目越复杂时越明显。20 个子项目、6 个 Git 子模块、跨项目 DLL 依赖——没有上下文的 AI 在这种复杂度下几乎无法有效工作。

## 五、维护策略

CLAUDE.md 的最大风险是过时。一个描述错误的架构比没有描述更糟糕。

**谁更新**：修改了架构或添加了新系统的开发者。在 PR 描述或提交说明中顺手更新。

**什么时候更新**：
- 添加了新的游戏系统
- 修改了项目结构或依赖关系
- 改变了构建或部署流程
- 发现 AI 因为 CLAUDE.md 的过时信息做出了错误判断

**怎么防止过时**：
- 只写变化慢的信息（架构、约定、流程），不写变化快的信息（当前任务、临时状态）
- 定期让 AI 自检：问 AI "根据 CLAUDE.md，项目里应该有 X，帮我确认一下还在不在"
- 把 CLAUDE.md 的更新纳入新功能开发的 checklist
