## 一个反复出现的场景

策划问："背包容量上限在哪配的？"

开发 A 回答了。过了两周，另一个策划又问了同样的问题。开发 A 不在，开发 B 花了二十分钟翻代码才找到答案。

再过一个月，新入职的策划又问了。这次没人记得上次谁回答的，答案又从零开始查。

这个问题的本质不是"没有 AI"，而是**团队知识没有沉淀**。答案存在于某个人的脑子里、某次聊天记录里、某段代码注释里——但它不在一个所有人都能找到的地方。

上一个 AI 聊天工具不能解决这个问题。AI 能搜索的前提是有东西可搜。

## 这篇要回答什么

1. 团队知识管理的核心矛盾是什么

5. 闭环的飞轮效应怎么启动

## 先给一句总判断

**AI 赋能团队知识管理，核心不是某个工具，而是一个闭环：Dify 暴露知识缺口，LKB 填补缺口，Wiki 沉淀知识，Dify 变得越来越能答。**

## 一、团队知识管理的核心矛盾

游戏团队的知识分布天然不均匀。

策划知道设计意图，但不知道代码怎么实现的。开发知道实现细节，但不知道策划为什么这么设计。QA 知道什么场景会出问题，但不知道根因。每个人都掌握拼图的一块，但没有人能看到全貌。

更麻烦的是，这些知识的载体五花八门：

| 知识类型 | 实际载体 | 问题 |
|---------|---------|------|
| 系统设计意图 | 策划脑子里、飞书文档里 | 文档过时、版本不一致 |
| 实现细节 | 代码注释、提交日志 | 只有写的人能找到 |
| 配置规则 | Excel 表头注释 | 注释不全、约定俗成 |
| 踩坑经验 | 聊天记录、口头传授 | 人走就没了 |
| 排查结论 | Bug 单里的一行备注 | 下次遇到同样问题还是从零查 |

这些知识有一个共同特征：**产生的时候没有被系统化记录，需要的时候找不到**。

传统的解决方案是"建文档库、写 Wiki"。但写文档这件事本身就是问题——开发不愿意写，写了没人维护，维护了没人看。文档库最终变成一个信息墓地。

## 二、为什么"建一个文档库"不够

文档库解决的是**存储**问题，但团队知识管理需要解决的是四个问题：

1. **发现**：知道哪些知识缺失
2. **生产**：把零散信息变成可用的文档
3. **沉淀**：让文档持续更新而不是写完就死
4. **消费**：让需要的人能快速找到并理解

传统文档库只解决了第 3 步的一半（存），但它不解决第 1 步（怎么知道该写什么）、不降低第 2 步的成本（写文档依然很痛苦）、也没有好的第 4 步方案（全文搜索在面对模糊问题时很弱）。

AI 能在这四个环节都提供增量价值——但前提是你把这四个环节串成一个闭环，而不是在某个点上孤立地用 AI。

## 三、三层架构

我们在一个 20 人的 Unity MMO 团队里跑通了一套方案，核心是三层：

```
┌───────────────────────────────────────────────┐
│  第一层：Dify — 团队唯一问答入口                  │
│  策划/QA/开发都在这里问                           │
│  基于 Hugo Wiki 的文档做语义检索和回答              │
│  答不了的问题 → 暴露知识缺口                       │
└──────────────────┬────────────────────────────┘
                   │
                   v
┌───────────────────────────────────────────────┐
│  第二层：LKB + Claude Code — 知识生产引擎         │
│  读代码 + 读配置表 + 读原始资料                    │
│  生成结构化文档 → 回流到 Hugo Wiki                 │
└──────────────────┬────────────────────────────┘
                   │
                   v
┌───────────────────────────────────────────────┐
│  第三层：Hugo Wiki — 知识沉淀层                   │
│  Markdown 文件 → Git 管理 → Jenkins 自动部署      │
│  Dify 重新索引 → 下次能答                         │
└───────────────────────────────────────────────┘
```

### 第一层：Dify — 门面

Dify 是团队所有人提问的统一入口。它做的事情很简单：接收自然语言问题，从 Hugo Wiki 的文档库里做语义检索，把相关文档喂给本地 LLM（Qwen2.5:7b），生成回答。

关键设计决策：

- **内网部署**，数据不出公司
- **本地模型**（Ollama + Qwen2.5:7b），不依赖外部 API
- **嵌入 Hugo Wiki 页面**，不需要单独打开另一个工具
- **系统提示词要求：答不了就说"知识库中暂无此内容"**，不编造

最后一点至关重要。当 Dify 回复"知识库中暂无此内容"时，这不是失败——这是系统在告诉你**哪里有知识缺口**。

### 第二层：LKB + Claude Code — 后厨

LKB（LLM Knowledge Base）是一个由 AI 维护的项目知识库。它跑在 Claude Code 上，能做 Dify 做不了的两件事：

1. **读代码**。Dify 的向量检索对代码效果差——代码的调用关系和上下文在分块后会丢失。但 Claude Code 能直接读源文件、跨文件追踪调用链、理解项目架构。

2. **关联分析**。策划问"背包容量在哪配的"，答案可能涉及 Excel 配置表的某个字段、服务器模块的某段代码、客户端 UI 的某个绑定——这种跨层关联只有能同时读代码和配置的工具才能做到。

LKB 的工作流分三步：

- **Ingest（摄入）**：把原始资料（策划文档、竞品分析、会议纪要）丢进 `raw/` 目录，AI 读取并生成结构化的 wiki 页面
- **Query（查询）**：基于已有 wiki 内容回答问题，有价值的回答存档
- **Lint（健康检查）**：定期检查 wiki 的矛盾、断链、过时信息

LKB 不是另一个问答系统——它是 Dify 的**上游内容加工厂**。它产出的结构化文档，比直接丢原始策划文档给 Dify 的检索效果好得多。

### 第三层：Hugo Wiki — 仓库

Hugo Wiki 是所有知识的最终沉淀地。Markdown 文件，Git 管理，Jenkins 自动部署到内网。

它同时服务两个消费者：
- 人直接浏览 Wiki 站点
- Dify 通过向量索引检索

每次有新文档提交，Jenkins 构建后可以触发 Dify 重新索引。

## 四、为什么代码类知识必须"前置到文档层"

这是整个方案里最反直觉的一点：**AI 问答系统不能直接查代码。**

向量检索的工作原理是把文档分块、向量化、存入索引，然后用语义相似度匹配查询。这对自然语言文档效果不错，但对代码来说：

- 代码的"含义"高度依赖上下文。一个函数名在分块后脱离了调用方和被调用方，语义就丢失了
- 代码的检索粒度和文档不一样。你搜"背包容量在哪配的"，匹配到的代码片段可能是无关的变量声明
- 代码频繁变化。索引更新的成本和频率都比文档高

所以正确的做法不是把代码扔进 Dify 让它检索，而是：

```
策划提问 → Dify 答不了 → 开发用 Claude Code 查代码
→ 开发把结论写成文档 → 文档入 Wiki → Dify 能答了
```

这个流程里，LKB + Claude Code 的价值就是**大幅降低"查代码 → 写文档"这一步的成本**。开发不用从零写，AI 读完代码后生成初稿，开发审核修改即可。

## 五、闭环的飞轮效应

这套系统有一个自增强特性：

```
团队提问 → Dify 暴露缺口 → LKB 生产文档 → Wiki 沉淀
    ↑                                            │
    └────────── Dify 索引更新，能答更多 ──────────┘
```

- 用得越多，暴露的缺口越多
- 缺口越多，生产的文档越多
- 文档越多，AI 能答的越多
- 能答的越多，更多人愿意用

启动飞轮的关键是**前几圈要有人推**。具体来说：

1. 先部署 Dify，让团队开始问——哪怕一开始答不好
2. 收集 Dify 回复"知识库中暂无此内容"的记录
3. 按频率排序：什么问题被问得最多但答不了
4. 用 LKB + Claude Code 针对高频缺口生产文档
5. 文档入库后验证 Dify 能否正确回答

前 10 篇文档可能要手动推。但一旦团队看到"我提的问题两天后 AI 就能答了"，参与度会显著提升。
