`本地分档只能回答"第一次怎么判"，不能回答"上线后怎么长期保持判得对"。`

所以这篇文章真正要讲的，不是再写一套新的 `if-else`，而是：

`怎样把机型分档接成一个可回写、可灰度、可回滚的线上治理系统。`

如果把这条平台主线往后收，你也可以把它理解成这样：

- `URP平台-02` 立的是默认判定
- `URP平台-03` 立的是线上纠偏链

## 本地分档为什么一上线就会失真

本地分档最容易给人一种错觉：

- 启动时判过一次
- 玩家也能手动改
- 似乎这件事就结束了

实际上，线上环境会持续改变这条判断成立的条件。

最常见的失真来源有五类：

1. **系统与驱动在变**
某个机型最初测得没问题，不代表后续 OTA、GPU 驱动或图形 API 路径变化后仍然没问题。

2. **内容在变**
新活动、新角色、新地图、新后处理和新 `Renderer Feature` 上线后，旧规则表不一定还成立。

3. **设备在老化**
同样是"某型号手机"，电池健康度、散热条件、存储老化和后台环境都可能把真实预算拉低。

4. **玩家在覆盖你的默认判断**
你给了"自动 / 高 / 中 / 低"四个选项后，线上真实生效档位不再等于"系统自动判定档位"。

5. **线上问题不是按型号名出现的**
很多事故最后真正稳定复现的维度，不是"某某手机"，而是：
- 某个 `GPU` 家族
- 某个 `OS` 补丁版本
- 某条渲染路径
- 某个资源或热更快照

所以机型分档一旦进线上，就不能再只靠"启动时判一次 + 本地存一份 `PlayerPrefs`"活着。

## 线上要回收哪些分档信号

如果你想把机型分档做成可治理系统，第一步不是急着上 `Remote Config`，而是先想清楚：

`线上到底要回收哪些信号，才能知道这次分档是对是错。`

我更建议至少回收下面这五组。

### 1. 设备指纹

不要只记机型名，至少要补齐：

- 机型名
- `GPU / SoC` 家族
- 图形 API（`Metal / Vulkan / OpenGL ES / D3D12`）
- `OS` 大版本与补丁版本
- 分辨率与刷新率
- 内存容量和可用存储量

这组信息负责回答：

`到底是哪一类设备在出问题。`

### 2. 当前生效档位与来源

只记"现在是 `Mid` 档"不够，你还要知道它是怎么来的。

建议至少区分下面几种来源：

- `local_detect`：本地设备识别结果
- `benchmark`：首次 Benchmark 覆盖了本地识别
- `remote_override`：线上规则表强制覆盖
- `player_manual`：玩家手动选择
- `fallback_safe_mode`：异常保护强制降到安全档

这组信息负责回答：

`是规则判错了，还是玩家自己把档位拉高了。`

### 3. 版本与内容快照

如果线上问题无法和具体内容快照绑定，后面几乎没法定位。

分档相关遥测里至少要带上：

- 客户端构建号
- 资源版本
- 热更快照版本
- 远程配置规则版本

否则最后你只能看到：

- 某机型卡

却看不到：

- 是哪一版规则表把它抬到了高档
- 是哪一版资源快照把阴影预算推爆了

### 4. 性能与热稳定信号

线上分档真正要看的，不是某一次单帧，而是持续表现。

建议至少回收：

- 关键场景 `avg / p95 / p99 frame time`
- 启动后前 3 分钟与后 15 分钟的帧时间差
- 关键场景首帧与稳定阶段差异
- 是否触发过热保护或安全降档
- 崩溃、`ANR`、明显卡顿是否集中在某档位

这一组信号负责回答：

`这档不是"理论上能跑"，而是"线上是否真的稳"。`

### 5. 玩家手动改档行为

玩家手动改档本身就是很值钱的反馈。

例如：

- 自动判 `High`，大量玩家手动改成 `Mid`
- 自动判 `Low`，玩家改成 `High` 也依然稳定

这两种情况都说明规则表值得调整。

所以我通常会单独统计：

- 自动档默认值
- 玩家最终选择档位
- 改档后是否更容易崩溃、卡顿或退出

这不是为了迎合玩家偏好，而是为了知道：

`你的默认判断到底保守了，还是激进了。`

## 规则表不要写成一大坨机型名单

很多团队第一次做线上治理时，会很自然地写成这样：

```json
{
  "iPhone13,4": "Mid",
  "iPhone14,2": "High",
  "SM-S9180": "Mid",
  "SM-S9280": "High"
}
```

这东西短期能救火，但长期会越来越脏。

更稳的做法是把规则拆成三层：

1. **硬性黑白名单**
用于明确已知高风险或明确可信的设备 / 驱动组合。

2. **能力分组规则**
例如按 `GPU` 家族、图形 API、内存或分辨率分层，而不是只按型号名写死。

3. **实验与灰度规则**
例如只对 5% 用户把某个 `GPU` 家族从 `High` 降到 `Mid`，观察一周后再决定是否全量。

一个更可维护的规则表示意，大概像这样：

```json
{
  "version": 12,
  "hard_blocks": [
    {
      "match": {
        "gpu_family": "mali-g57",
        "os_version": "android-14.0"
      },
      "force_tier": "low",
      "reason": "shadow_instability"
    }
  ],
  "recommended_rules": [
    {
      "match": {
        "gpu_family": "adreno-730+",
        "graphics_api": "vulkan",
        "resolution_height_lte": 2400
      },
      "tier": "high"
    },
    {
      "match": {
        "gpu_family": "mali-g57|g68"
      },
      "tier": "mid"
    }
  ],
  "experiments": [
    {
      "id": "downgrade_g78_shadow_test",
      "match": {
        "gpu_family": "mali-g78"
      },
      "rollout": 0.1,
      "force_tier": "mid"
    }
  ]
}
```

关键不是 `JSON` 语法本身，而是这张表应该具备三个能力：

- 可版本化
- 可灰度
- 可快速回滚

## 生效优先级链要先写清，不要线上再猜

线上规则系统最怕的，不是规则少，而是优先级混乱。

我更建议在设计之初就把"最终生效档位"的来源顺序写死。

一个更稳的优先级链可以是这样：

1. **安全强制覆盖**
已知会崩、会花屏、会触发严重事故的组合，线上直接强制降档或禁用某功能。

2. **玩家手动选择**
只要没有踩到安全强制覆盖，玩家手动选择可以覆盖默认推荐值。

3. **灰度 / 实验规则**
用于小流量观察分档调整是否有效。

4. **远程推荐规则**
按设备能力给出默认推荐档位。

5. **本地识别 + Benchmark**
作为没有远程规则命中时的默认起点。

6. **安全兜底**
什么都拿不到时，先落到 `Mid` 或 `Low`，不要猜最高档。

伪代码可以像这样：

```csharp
public static QualityTier ResolveFinalTier(DeviceFingerprint fp)
{
    if (RemoteRules.TryGetHardBlock(fp, out var forcedSafeTier))
        return forcedSafeTier;

    if (PlayerSetting.TryGetManualTier(out var manualTier) &&
        RemoteRules.IsTierAllowed(fp, manualTier))
        return manualTier;

    if (RemoteRules.TryGetExperimentTier(fp, out var experimentTier))
        return experimentTier;

    if (RemoteRules.TryGetRecommendedTier(fp, out var remoteTier))
        return remoteTier;

    if (LocalDetector.TryDetectTier(fp, out var localTier))
        return localTier;

    return QualityTier.Mid;
}
```

这里最重要的，不是代码写法，而是：

`安全覆盖、玩家手动、远程推荐、本地识别这几层到底谁先谁后，必须在系统设计时说清楚。`

## 灰度和回滚不是"发布功能"的专属能力

很多团队只把灰度和回滚理解成：

- 新玩法上线要灰度
- 线上事故要回滚

其实机型分档同样非常适合灰度。

因为分档规则本质上也是在改线上体验：

- 你把一批设备从 `High` 降到 `Mid`
- 你把某条 `Renderer Feature` 在某组设备上关闭
- 你把某个 `GPU` 家族从 `Vulkan` 切回 `OpenGL ES`

这些动作都应该走一套更稳的灰度流程。

一个很实用的最小流程是：

1. 先选一组明确的目标设备指纹
2. 只放量 1%~5%
3. 盯下面几类指标：
- `p95 frame time`
- 崩溃率 / `ANR`
- 首次进入关键场景耗时
- 玩家手动改档比例
4. 观察一个固定窗口
5. 再决定是否扩大到 20%、50%、100%

回滚条件也要提前写清，不要等数据烂了再临时讨论。

例如：

- 某组设备崩溃率超过稳定版基线 `30%`
- `p95 frame time` 恶化超过 `20%`
- 玩家手动把推荐档位改低的比例显著上升

一旦触发，就直接回滚到上一版规则表，而不是重发客户端。

这也是机型分档线上治理最大的价值之一：

`很多错档问题，本来不该等到发新版客户端才能修。`

## 线上回写的目标不是"永远判对"，而是"持续收紧误差"

机型分档很难做到一次写完就永远正确。

因为你面对的是：

- 会变的系统
- 会变的内容
- 会变的驱动
- 会变的玩家设备状态

所以更稳的目标不是追求"永不出错"，而是建立一条闭环：

1. 本地先给出初始判断
2. 线上收集设备指纹、真实帧时间、稳定性和玩家改档行为
3. 用规则表小流量修正
4. 用灰度观察是否真的改善
5. 把稳定有效的规则沉淀成下一版默认判断

这样做的好处是，机型分档不再只是启动时的一次决定，而会逐渐演化成一套长期可维护的设备治理系统。

## 一条最小可执行路径

如果项目现在还没有成型的线上分档治理，我会建议先从下面这条最小路径开始：

1. 先给每次会话记录 `设备指纹 + 当前档位 + 档位来源 + 客户端版本 + 资源快照`
2. 先把关键场景 `avg / p95 frame time` 和崩溃率按档位切维
3. 先做一张版本化远程规则表，不要再把线上修正规则散在代码里
4. 先支持"按设备组合强制降档"的安全覆盖
5. 先让规则表支持 1%~5% 的灰度放量和一键回滚

只要这五步立住，你的机型分档就已经从"本地技巧"进入"线上系统"了。

## 小结

`URP平台-02` 解决的是"第一次怎么判"，这篇解决的是"上线后怎么持续判得对"。

如果把整件事压成一句话，我会这样收：

`机型分档一旦进线上，就不再只是本地设备识别，而是一个要靠设备指纹、版本快照、帧时间、稳定性、玩家改档行为持续回写，再用远程规则、灰度和回滚不断修正的运营系统。`
