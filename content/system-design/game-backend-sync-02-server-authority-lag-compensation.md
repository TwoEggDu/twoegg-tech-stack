## 问题空间：权威与延迟的对立

服务端权威（Server Authority）是多人游戏最基础的防作弊机制——服务端说什么是真实的，什么就是真实的。但这个模型有一个本质代价：**玩家的每次操作都需要一个完整的往返延迟（RTT）才能见效**。

假设玩家的 RTT 是 100ms（北京到上海的正常延迟），那么：

1. 玩家按下"向前移动"（t=0）
2. 输入到达服务端（t=50ms）
3. 服务端处理，推送新位置（t=55ms，含服务端 Tick 延迟）
4. 新位置到达客户端（t=105ms）
5. 玩家的角色才在屏幕上移动

这 105ms 的延迟，在 FPS 或动作游戏里是完全不可接受的——它会让游戏感觉"踩在棉花上"，操作完全失去直觉。

另一端是客户端预测（Client-Side Prediction）：让客户端先执行操作，不等服务端确认。但预测可能出错，修正的时候出现"橡皮筋效应"（位置突然跳回去），同样让玩家抓狂。

问题就是：**如何在保持服务端权威的前提下，让玩家感受不到延迟？**

这不是一个可以"完全解决"的问题，而是一组工程上的权衡——理解这组权衡，才能做出合适的实现选择。

---

## 抽象模型：三层延迟来源

在讨论解决方案之前，先把延迟的来源分清楚。网络游戏里有三种延迟，它们的成因和应对手段完全不同：

### 1. 输入延迟（Input Latency）

玩家操作 → 服务端确认 → 客户端显示结果，完整 RTT。这是最直观的延迟，客户端预测就是针对这个问题的。

### 2. 展示延迟（Render/Interpolation Delay）

为了让其他玩家的移动看起来流畅（而非因为网络抖动跳来跳去），客户端对其他实体做插值，会人为引入约 1-2 个网络 Tick 的展示延迟（通常 50-100ms）。这是为了平滑视觉效果必须付出的代价。

### 3. 处理延迟（Server Tick Delay）

服务端不是实时处理每个包，而是按 Tick（通常 20-64Hz）批量处理。输入包可能等待最多一个 Tick 才被处理，平均增加半个 Tick 的延迟（约 8-25ms）。

三种延迟叠加，让"实际延迟感"远超纯 RTT 数字。理解这三层，才知道优化手段针对的是哪个环节。

## 具体机制：客户端预测与服务端回滚

### 客户端预测（Client-Side Prediction）

核心思路：**客户端不等服务端确认，先本地执行玩家输入，把结果直接渲染出来**。与此同时，把输入包发送给服务端。

实现上，客户端需要维护一个**输入缓冲区（Input Buffer）**：

```
输入缓冲区（客户端维护）：
[序号 n, 输入 Ik, 预测后状态 Sk] → 发送给服务端
[序号 n+1, 输入 Ik+1, 预测后状态 Sk+1] → 发送给服务端
...（等待服务端确认的输入队列）
```

服务端收到输入后，以权威方式计算结果，并把**服务端确认帧编号（Ack Frame）**和对应**权威状态（Authoritative State）**返回给客户端。

客户端收到服务端确认后，做如下处理：

1. 从输入缓冲区删除序号 ≤ Ack Frame 的所有条目（这些已被确认，不再需要）
2. 如果服务端返回的权威状态与客户端预测状态**一致**，什么都不用做，继续预测
3. 如果**不一致**（预测错误），触发**状态回滚与重播（Rollback & Replay）**

### 预测错误时的回滚与重播

预测错误是客户端预测最难处理的部分。正确流程：

```
1. 强制修正到服务端权威状态（覆盖客户端错误状态）
2. 将输入缓冲区中所有未确认的输入，按顺序重新执行一遍
3. 得到"修正后的当前预测状态"
4. 渲染修正后的状态
```

这个过程完全在客户端本地完成，通常在一帧内处理完毕（因为缓冲的输入数量有限，一般不超过 RTT/Tick = 约 5-10 帧的输入）。

**橡皮筋效应的本质**：如果修正前后的状态差异过大（比如服务端发现玩家被击退了 5 米，但客户端预测认为没有），视觉上就会出现突然的位置跳变——这就是"橡皮筋"。缓解手段是对状态差异做平滑插值，但过度平滑又会引入视觉欺骗感，需要根据游戏类型精细调整。

### 延迟补偿（Lag Compensation）

客户端预测解决了"本地操作感"的问题，但还有另一个问题：**射击判定**。

假设玩家 A（RTT 100ms）看到玩家 B 在位置 (10, 0, 5)，扣扳机打了一枪。但这个"看到"其实是 50ms 前的历史位置——因为 B 的状态经过了展示延迟。与此同时，服务端当前时刻的 B 实际上已经移动到了 (10, 0, 8)。

如果服务端用当前状态做碰撞检测，A 的这枪会丢失——他明明"打中了"，服务端却说没打中。

**延迟补偿的核心机制**：服务端保存世界状态的历史快照（通常维护 1-2 秒的历史窗口）。当服务端收到玩家 A 的射击命令时，**回溯到玩家 A 发射时"看到的"历史时刻**，在那个历史快照上做碰撞检测，然后以那次检测的结果为准。

```
服务端处理射击命令：
1. 从命令包中取出客户端时间戳 T_client
2. 找到时间 T_client 对应的历史世界快照
3. 在历史快照中做射线检测（Raycast）
4. 判定是否命中，应用伤害
5. 恢复到当前快照，继续正常处理
```

这个机制有一个有趣的副作用：**高延迟玩家打低延迟玩家时有优势**——因为高延迟玩家的历史回溯窗口更长，服务端需要回到更早的历史状态，被打的玩家"那时"的位置可能已经逃走了，但在高延迟玩家的视角上还在原地。这是一个工程上无法完全消除的公平性问题，通常用延迟上限（如 200ms）来限制历史回溯深度。

## 工程实现架构：Source Engine 方案

Valve 的 Source Engine（CS:GO、Team Fortress 2 等游戏的引擎）是延迟补偿工程化落地的经典参考，其设计在 GDC 2001 的技术论文（"Latency Compensating Methods in Client/Server In-game Protocol Design and Optimization"）中有详细说明。

Source Engine 的完整架构由四个部分组成：

### 1. Input Buffer（客户端输入缓冲）

客户端每次 Tick 把玩家输入打包（移动方向、视角、开火等），加上序列号和本地时间戳，发送服务端。同时本地预测执行这个输入，更新角色状态。所有未被确认的输入保存在缓冲区中。

### 2. Snapshot Interpolation（快照插值）

对于其他玩家的实体，客户端不做预测，而是在两个服务端快照之间插值。客户端维护一个**展示时间（Render Time）= 当前时间 - 插值延迟**，通常插值延迟设为 2 个 Tick 周期（例如 64Hz 服务器的话约 31ms）。

这保证了其他玩家移动的视觉流畅性，代价是你"看到"的其他玩家是略微过去的状态——这正是延迟补偿需要处理的问题。

### 3. Server Rewind（服务端回滚）

服务端维护一个环形缓冲区，保存过去 N 帧（通常对应 1 秒历史）的所有实体快照。处理射击命令时：

```csharp
// 伪代码示意
void ProcessShootCommand(ShootCmd cmd)
{
    float clientTime = cmd.ClientTimestamp;
    // 限制最大回滚时间（防高延迟滥用）
    clientTime = Mathf.Clamp(clientTime, CurrentTime - MaxRewindTime, CurrentTime);
    
    WorldSnapshot historical = snapshotBuffer.GetAtTime(clientTime);
    
    // 在历史快照中做射线检测
    RaycastHit hit;
    if (historical.Raycast(cmd.Origin, cmd.Direction, out hit))
    {
        ApplyDamage(hit.entityId, cmd.Damage);
    }
}
```

### 4. Client Reconciliation（客户端协调）

服务端确认帧的返回包含：Ack 序列号 + 权威状态。客户端收到后，丢弃已确认的输入，如果预测有误，回滚并重播剩余未确认输入。

Source Engine 的一个关键工程细节：移动类操作（走路、跑步）通常预测误差很小，因为物理逻辑简单；而受伤、被击退等操作误差可能很大——对这类操作，Source Engine 采用更激进的状态修正（允许更大的视觉跳变），而不是平滑插值，因为玩家需要快速感知"我被打了"的信息。

## 工程边界：哪些场景不适合这套方案

### 预测成本过高的逻辑

如果游戏逻辑复杂到客户端无法快速重播（例如复杂物理模拟、大量 AI 决策），每次回滚重播会消耗大量 CPU，可能导致帧率崩溃。此时需要分层处理：只对玩家角色自身做预测，复杂逻辑仍然等服务端确认。

### 高欺骗性场景

延迟补偿本质上是"相信客户端告知的时间戳"。如果客户端篡改时间戳，让服务端回溯到很久以前做判定，就可以制造"超级远程射击"等作弊行为。防御措施：严格限制最大回滚时间（通常不超过 200ms），并对时间戳做合法性验证。

### 高延迟玩家的公平性问题

如前所述，延迟补偿对高延迟玩家有隐性优势。竞技性游戏通常设置延迟上限（如 150-200ms），超过上限的玩家会被踢出或匹配到专属服务器。这是一个游戏设计决策，不只是工程问题。

### 移动端的弱网环境

移动端网络抖动（Jitter）远大于 PC，输入包乱序到达概率更高。此时需要更大的 Input Buffer（防止乱序导致输入丢失），同时服务端需要更激进的插值策略。KCP 协议（快速可靠 UDP）在移动端比原生 TCP/UDP 表现更好，是移动游戏后端的常见选择。

## 最短结论

- **服务端权威是防作弊的必要条件**，但纯权威模型带来完整 RTT 的延迟感，FPS/动作游戏无法接受。
- **客户端预测**解决了"本地操作感"问题：本地先执行，等服务端确认，不一致时回滚重播。代价是需要维护输入缓冲区，以及处理回滚时的视觉修正（橡皮筋）。
- **延迟补偿**解决了"射击判定"问题：服务端回溯到玩家"看到的"历史时刻做碰撞检测，让玩家"打到了就是打到了"。代价是对高延迟玩家有轻微公平性倾斜，需用最大回滚时间限制。
- Valve Source Engine 是这套方案的工程参考标准：Input Buffer + Snapshot Interpolation + Server Rewind + Client Reconciliation 四件套，是现代 FPS 网络同步的基础架构蓝图。
