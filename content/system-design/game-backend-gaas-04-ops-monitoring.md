## 不监控的代价

2023 年，某款手游在大版本更新后，由于新副本掉落逻辑存在 Bug，玩家可以无限循环领取特定道具。这个问题在上线后 6 小时才被发现（一名运营人员看到游戏论坛玩家在晒异常道具数量），此时已有数万玩家获得了大量本不该有的道具，经济系统严重失衡，后续的处置（回收道具、补偿正常玩家）引发更大的舆情危机。

如果有监控：特定道具的日产出量对比历史基线出现了 50 倍的异常，5 分钟内就能告警，运营团队在 30 分钟内可以通过远程配置关闭问题副本，将损失控制在最小范围。

**监控不是可选项，是 Live Service 游戏的基础设施。** 问题不是"要不要监控"，而是"监控什么、怎么告警、谁来响应"。

## 两类指标的分工

游戏运营监控的指标分为两大类：

**运营指标（Business Metrics）：** 反映游戏的商业健康状况，由运营/产品团队关注。异常通常意味着玩家行为变化、活动效果偏差或经济系统问题。

**技术指标（Technical Metrics）：** 反映系统的运行状况，由工程团队关注。异常通常意味着服务器过载、代码 Bug 或基础设施故障。

两类指标需要分开监控，但需要联动分析：技术指标的异常（如接口错误率上升）往往是运营指标异常（如留存率下降）的先行信号。

## 游戏运营的核心指标体系

### DAU / MAU（活跃用户数）

- **DAU（Daily Active Users）：** 每天有登录行为的去重用户数
- **MAU（Monthly Active Users）：** 每月有登录行为的去重用户数
- **DAU/MAU 比率：** 反映用户粘性，比率越高说明用户每月越频繁回来玩（完全粘性 = 1.0，大多数游戏在 0.2-0.4）

**监控要点：** 每日 DAU 对比昨日同期、上周同期、上月同期。绝对值不如趋势重要——如果 DAU 连续 3 天下滑超过 10%，无论当前绝对值多少，都需要立刻排查原因。

### 次日留存率

已经在上一篇文章详细讨论过。从监控角度，需要关注的是**滚动 7 日平均次日留存率**的趋势，单日数据受样本量影响波动较大。

### 崩溃率（Crash Rate）

```
崩溃率 = 崩溃会话数 / 总会话数 × 100%
```

需要按平台（iOS / Android / PC）和客户端版本分别监控。新版本上线后，崩溃率是最先需要盯的指标——如果新版本崩溃率比上一版本高出 50% 以上，需要立刻评估是否回滚或紧急修复。

**行业参考：** 移动游戏的崩溃率通常控制在 0.1%-1% 以下，超过 2% 会对留存产生明显影响。

### 付费率 / ARPU

- **付费率（Conversion Rate）：** 当日活跃用户中有付费行为的比例。通常在 1%-5% 之间，高付费深度游戏可以到 10%+。
- **ARPU（Average Revenue Per User）：** 每日活跃用户的平均收入 = 当日总收入 / 当日 DAU
- **ARPPU（Average Revenue Per Paying User）：** 每付费用户的平均收入 = 当日总收入 / 当日付费用户数

**监控要点：** 付费率的突然下降（超过 20%）可能意味着支付链路故障（不是玩家不想付，是付不了），需要立刻联动检查支付接口的错误率。

## 技术健康指标

### 服务器资源

| 指标 | 告警阈值建议 | 备注 |
|------|-------------|------|
| CPU 使用率 | > 80% 持续 5 分钟 | 预留弹性，CPU 100% 意味着请求已经在排队 |
| 内存使用率 | > 85% | 接近 OOM 的先兆 |
| 磁盘使用率 | > 80% | 日志滚满磁盘会导致服务崩溃 |
| 网络入/出带宽 | > 70% 峰值容量 | 带宽饱和导致延迟急剧上升 |

### 接口 P99 延迟

P99 延迟是"99% 的请求在 X 毫秒内完成"，比平均延迟更能反映用户的真实体验（平均值容易被少量快速请求拉低）。

**参考阈值：**
- 游戏 API 接口：P99 < 200ms（正常），P99 > 500ms（告警），P99 > 1000ms（严重）
- 对战服务（实时）：RTT < 50ms（优秀），RTT > 150ms（问题边界）

### 错误率

```
错误率 = HTTP 5xx 响应数 / 总请求数 × 100%
```

正常游戏后端的 5xx 错误率应该低于 0.1%。如果错误率上升到 1%，已经是需要紧急响应的级别——每 100 个请求有 1 个失败，玩家会明显感知到。

## 告警阈值的设计方法

### 绝对阈值：简单直接

最简单的告警方式：当指标超过固定阈值时触发。

```yaml
# 示例告警规则
- name: "API错误率过高"
  condition: "error_rate > 1%"
  duration: "5 minutes"  # 持续 5 分钟才触发，避免瞬时抖动
  severity: "critical"
  notify: "oncall-engineer"

- name: "CPU使用率过高"
  condition: "cpu_usage > 80%"
  duration: "5 minutes"
  severity: "warning"
  notify: "ops-channel"
```

**适用场景：** 技术指标（有明确的"正常范围"）。

**局限：** 业务指标（如 DAU、付费率）没有固定的"正常值"，随游戏生命周期、活动节奏波动，用固定阈值会产生大量误报。

### 同比/环比异常检测：更智能

对于业务指标，比较当前值与历史同期（同比）或与前一时间段（环比）的偏差：

- **同比（Year-over-year / Week-over-week）：** 今天 DAU 对比上周同一天的偏差超过 20%，告警
- **环比（Day-over-day）：** 今日付费率对比昨日偏差超过 30%，告警

这类异常检测对游戏数据更合理，因为它自动适应了周末/工作日的差异、赛季周期的波动等正常模式。

更复杂的方案是**统计异常检测（Statistical Anomaly Detection）**，基于历史数据建立预期值范围（均值 ± N 倍标准差），超出范围时告警。这类方案可以通过 AWS CloudWatch Anomaly Detection、Datadog 的 Monitor 等直接使用，不需要自研。

## 告警疲劳：太多告警等于没告警

告警疲劳是实际运营中最常见的问题之一：团队的告警通道（Slack、钉钉、PagerDuty）每天收到几百条告警，大部分是误报或低优先级信息，结果工程师习惯性忽视，等到真正的严重事故时也无人响应。

**对抗告警疲劳的原则：**

**1. 告警必须可操作（Actionable）**

每一条告警都应该对应一个明确的处置预案："收到这条告警，应该做什么"。如果一条告警让工程师的第一反应是"这是什么，我该怎么办"，说明告警本身设计不清晰。

**2. 按严重程度分级（Severity）**

```
P0（Critical）：服务完全不可用，影响大量用户，立即唤醒值班工程师
P1（High）：核心功能受损，影响部分用户，30 分钟内响应
P2（Medium）：非核心功能异常，值班期间处理
P3（Low）：潜在问题，工作时间处理
```

P0 告警才需要发送短信/电话，P2/P3 只发 Slack/钉钉消息。深夜收到 P3 告警电话会迅速摧毁团队对告警系统的信任。

**3. 合并重复告警（Alert Grouping）**

同一根因引发的多条关联告警，合并成一条。例如：数据库连接池耗尽，可能同时触发 CPU 高、接口超时、错误率高三条告警，这三条应该被聚合显示，工程师一眼就能看出是同一个根因，而不是以为有三个独立问题。

**4. 定期审查告警规则**

每季度审查：哪些告警从来没有引发过真实问题（误报）？哪些真实问题发生时没有产生对应告警（漏报）？根据历史数据调整阈值。

## 事故响应的基本流程

告警触发只是开始，事故响应流程决定了从发现问题到恢复服务的时间。

```
1. 发现（Detection）
   告警触发 → 值班工程师确认 → 判断是否为真实事故

2. 定级（Classification）
   评估影响范围（多少用户受影响？哪些功能不可用？）→ 确定 P0/P1/P2

3. 指挥（Incident Commander）
   P0/P1 事故指定一名 IC（Incident Commander）统筹协调
   IC 负责：对外沟通（对玩家的公告）+ 对内协调（调度工程资源）
   IC 不亲自写代码，专注于信息同步和决策

4. 处置（Mitigation）
   工程师并行排查：定位根因（看日志、看监控）+ 临时止血（关功能/回滚部署）
   目标：先恢复服务，再查根因。"完美的修复"不如"快速止血"

5. 复盘（Post-mortem）
   事故解决后 24-48 小时内完成复盘文档
   复盘文档包含：时间线、根因分析、影响评估、改进措施
   复盘文化：对事不对人，目标是系统改进，不是追责
```

**MTTR（平均恢复时间）** 是衡量事故响应效率的核心指标。优秀的 Live Service 团队通常能将 P0 事故的 MTTR 控制在 30 分钟以内。

## 游戏开服当天的特殊监控策略

游戏开服（新游上线或大版本更新）是风险最高的时刻，正常的监控策略需要加强：

**高频采样：** 将监控指标的采样频率从每分钟提高到每 10-30 秒。开服初期，问题会在几分钟内从萌芽变成灾难，需要更快的感知速度。

**缩短告警窗口：** 平时需要"持续 5 分钟"才触发的告警，开服当天改为"持续 1 分钟"。宁可多一些误报，也不要错过早期预警。

**人工盯盘：** 开服当天，指派专人持续盯着监控大盘（不是等告警，而是主动看）。第一个小时尤为关键，这是流量最高、系统最脆弱的时段。

**降级预案就位：** 开服前准备好所有关键功能的快速关闭脚本（通过 Feature Flag 或直接的运维脚本），发现问题可以秒级关闭受影响功能，而不是等工程师研究怎么操作。

**扩容预备：** 开服前将服务器容量预先扩展到预期峰值的 150%，而不是依赖弹性扩容追着流量走（弹性扩容有几分钟的延迟，对开服洪峰来说太慢）。

## 工程边界

- 监控系统本身也会出故障（告警服务挂了、指标收集断了）。监控系统需要有自己的健康检查，并且监控系统的告警通道要与被监控的服务解耦（用不同的基础设施）。
- 指标可见不等于可理解。数百条监控指标堆在一个大屏上，值班工程师看到异常也需要时间理解上下文。设计监控大盘时，核心指标要突出，关联指标要分组，帮助快速定位。

---

## 最短结论

游戏运营监控的核心是：**业务指标（DAU/留存/付费率）体现游戏健康，技术指标（延迟/错误率/崩溃率）体现系统健康，两者需要联动分析才能快速定位问题根因。**

告警设计的最高原则是：每条告警对应一个可执行的处置方案，按严重程度分级，定期清理误报。告警疲劳是监控系统失效的慢性病——比没有监控更危险，因为它让团队失去对告警的信任。

开服当天加密监控、备好降级预案，是 Live Service 工程文化最直接的体现。
