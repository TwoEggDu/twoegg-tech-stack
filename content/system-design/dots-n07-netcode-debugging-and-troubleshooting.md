> 验证环境：Unity 6000.0.x · com.unity.netcode 1.4.x · com.unity.entities 1.3.x


NetCode 最后拼的不是概念，而是证据链。很多多人问题表面上都像“网络有点抖”，但实际根因可能完全不同：输入链错了、快照太重了、权威边界漂了、预测回滚不稳定，或者只是表现层在伪装问题。

N07 的目标不是把工具名背一遍，而是给一条稳定的排障顺序：先分症状，再找责任层，再补证据。只要顺序反了，排障就会变成无限试错。

---

## 先分症状，不要先猜原因

NetCode 的症状可以先粗分成几类。

第一类是输入延迟感，表现为按了没反应，或者本地反馈明显晚一拍。第二类是位置抖动和橡皮筋，通常和预测、回滚或权威修正有关。第三类是命中不一致，本地看中了，服务器没认，或者反过来。第四类是回滚尖峰，表现为偶发大卡顿，常常意味着修正太频繁或重放太重。

症状分型的意义在于，它能先告诉你应该查哪一层：

- 输入延迟感，优先查输入链和 Tick 对齐。
- 抖动与回弹，优先查预测、插值和权威修正。
- 命中不一致，优先查 Snapshot、事件回写和权威裁决。
- 回滚尖峰，优先查重放成本、同步预算和对象数量。

---

## 排障顺序应该从哪一层开始

正确的顺序不是“先看所有日志”，而是先找最上游的边界是否成立。

第一步看世界职责：Client World、Server World、Ghost、Authority 有没有被写乱。第二步看输入链：CommandData 是否按 Tick 进入、是不是把结果伪装成输入。第三步看同步链：Snapshot 里是否塞了太多字段，Ghost 是否同步了不该同步的对象。第四步看预测链：是否真的可重放，重放后结果是否稳定。第五步看插值和表现层：远端显示是否把问题放大了。

这个顺序的核心原则是：越上游的问题越便宜，越下游的问题越容易伪装成别的问题。不要一上来就盯 UI、动画或者某个窗口，它们往往只是把真正的问题遮住了。

---

## 一条有效的证据链长什么样

排障不能只靠截图或体感，必须能把“症状”变成“可复现的证据”。

最少要留三类证据：

- `Tick` 和输入记录，证明某一帧到底收到了什么。
- `Snapshot` 或权威状态记录，证明服务器最终裁决了什么。
- 预测与回滚日志，证明本地和权威之间怎么偏离、怎么修正。

如果没有这三类证据，调试就只能靠猜。NetCode 问题越复杂，越需要把每次偏差都绑定到具体 Tick 和具体对象上。

下面是一个最小证据记录器的示意：

```csharp
public sealed class NetcodeTrace
{
    public void RecordInput(int tick, string entity, string input)
    {
        // 记录 CommandData / 本地输入
    }

    public void RecordSnapshot(int tick, string entity, string state)
    {
        // 记录权威快照或修正结果
    }

    public void RecordPrediction(int tick, string entity, string note)
    {
        // 记录预测偏差与回滚点
    }
}
```

这段代码不重要的是 API，重要的是三件事必须分开记。只要输入、权威和预测混在一条日志里，后面就很难看出谁先错。

---

## 常见症状对应的排查方向

如果玩家说“操作很迟钝”，先查输入链有没有被错放到状态链里，再查本地预测有没有被禁掉。

如果玩家说“角色会被拉回去”，优先查预测和回滚是否稳定，再查权威状态是否按 Tick 对齐。

如果玩家说“技能看起来命中了，但服务器没算”，先查 Snapshot、事件回写和命中判定的权威层，不要先怀疑表现层。

如果玩家说“偶发大卡顿”，先查同步预算和回滚重放成本，再查是否某些对象被错误地全量同步了。

这类排查的关键不是“猜哪个工具更强”，而是先把症状映射回责任层。

---

## 哪些地方最容易误判

第一，表现层抖动不一定是网络问题，有时只是动画或镜头把本来就存在的小修正放大了。

第二，预测失败不一定是算法错了，也可能是输入链不稳定、Tick 不对齐，或者权威状态本身就不干净。

第三，回滚尖峰不一定是单个系统慢，常常是对象太多、同步太宽或者每次修正都触发了过重的重放。

所以排障时要避免一个习惯：看到症状就立刻去改最后一层。很多时候真正应该先改的是最上游的数据边界。

---

## 小结

NetCode 排障的核心不是工具，而是顺序：先分症状，再找责任层，再收证据。

只要你能把输入、快照、预测、插值和表现分开看，延迟、抖动、错位和回滚尖峰就不再是一个混乱的大问题，而是几种可以逐层定位的故障类型。
