## 游戏后端的攻击者画像

普通 Web 应用的攻击者通常追求两件事：数据（个人信息黑市价值）和算力（挖矿、跳板）。游戏后端的攻击者还有第三件事——**虚拟财富套现**。

这个区别至关重要。一个电商平台被攻击，攻击者需要拿到支付信息才能变现。一个游戏后端被攻击，攻击者可能只需要多刷几次副本奖励，就能在二级市场卖掉游戏货币。变现链条更短、风险更低，这直接推高了游戏后端的攻击吸引力。

另一个差异是**实时对战的作弊动机**。在 PvP 游戏里，作弊者可以通过外挂、变速器、数据包篡改获得竞争优势，再以高段位账号售卖或接代练获利。这类攻击不需要突破服务端，仅在客户端层面就能产生价值，但一旦客户端验证不充分，攻击行为就会延伸到服务端。

理解攻击者的动机，是构建威胁模型的第一步。

---

## 威胁建模的基本思路

威胁建模（Threat Modeling）的核心是把"我们有什么"和"他们想要什么"对应起来，找到交叉点，那就是攻击面。

一个适合游戏后端的简化框架：

```
攻击者目标 → 可利用的入口点 → 当前的防御层次 → 防御缺口
```

**第一步：枚举攻击者目标**

对游戏后端，攻击者目标通常包括：
- 批量获取游戏货币/稀有道具（可变现）
- 修改自己或他人的存档数据（获得优势或破坏竞争对手）
- 控制高价值账号（批量售号、代练接单）
- 制造服务不可用（竞争对手打压、勒索）

**第二步：枚举入口点**

游戏后端的主要入口点：
- 客户端发起的 HTTP/WebSocket API
- 游戏大厅/匹配服务
- 存档读写接口
- 支付回调接口
- 后台管理系统

**第三步：识别防御层次**

每个入口点上，现有的防御措施是什么？是否存在绕过的可能？

这个过程不依赖工具，只需要一张纸和对自己系统足够的诚实。

## 三类主要威胁的攻击路径

### 威胁一：接口刷取 → 洗货币

**攻击路径：**

攻击者发现某个奖励接口（比如每日任务完成、副本通关奖励）存在以下问题之一：
- 无频率限制，可以高频调用
- 服务端未校验业务前置条件（任务是否真的完成）
- 依赖客户端传参决定奖励数量

攻击者用脚本批量调用该接口，短时间内积累大量游戏货币，随后通过游戏内交易系统转移给"洗钱"账号，再由该账号在二级市场出售。

**关键弱点：**
客户端传参决定奖励数量，是这类攻击中最常见的设计失误。例如：

```
POST /api/quest/complete
{
  "quest_id": 101,
  "reward_coins": 1000  // 攻击者直接改成 999999
}
```

服务端如果直接使用这个 `reward_coins`，等于把奖励权力交给了攻击者。

**防御要点：**
服务端根据 `quest_id` 查配置表确定奖励，绝不依赖客户端传入的奖励数值。奖励领取接口需要校验任务完成状态（该状态只能由服务端的业务逻辑写入，不接受客户端申报）。

---

### 威胁二：数据篡改 → 改存档

**攻击路径：**

存档篡改有两种主要手段：

**手段 A - 直接修改内存/本地文件：** 客户端在本地运行，攻击者可以用 Cheat Engine 之类的工具修改内存中的数值，再通过游戏的"同步存档"接口将篡改后的数据上传到服务端。如果服务端对上传的存档数据做的验证不够，篡改就会持久化。

**手段 B - 回放历史存档（存档回滚攻击）：** 攻击者在某个有利时机（如资产最多时）备份存档，消耗大量资源后，向服务端重新上传旧版本存档，将资产回滚到有利状态。

**关键弱点：**
服务端将整个存档 blob 视为可信输入，只做格式校验而不做业务合理性校验（比如货币突然增加了 10 倍是否合理）。

**防御要点：**
- 服务端维护关键资产的权威记录，不完全依赖客户端上传
- 对存档变更做增量校验（不允许未经业务流程产生的资产突增）
- 存档版本号单调递增，拒绝接受比当前版本更旧的存档

---

### 威胁三：账号劫持 → 盗号卖号

**攻击路径：**

账号劫持的常见手段：

1. **凭证填充（Credential Stuffing）：** 攻击者使用从其他平台泄露的账号密码（撞库），大量尝试登录游戏账号。玩家复用密码的比例很高，这类攻击成功率不低。

2. **会话令牌劫持：** 如果 Token 通过不安全的渠道传输（HTTP 明文），或者 Token 有效期过长（数周甚至永久有效），攻击者一旦截获就可以长期控制账号。

3. **钓鱼/社工：** 伪造游戏官网登录页，诱导玩家输入账号密码。这类攻击针对高价值账号定向实施，防御难度较高。

4. **API 爆破：** 对登录接口进行密码字典爆破。如果没有登录频率限制，攻击者可以系统地尝试弱密码。

**关键弱点：**
登录接口无频率限制；Token 不过期；账号异常登录无告警。

**防御要点：**
- 登录失败多次后触发 CAPTCHA 或临时锁定
- Token 设置合理有效期（Access Token 15-30 分钟，Refresh Token 7-30 天）
- 异地/异设备登录触发二次验证
- 监控单 IP 的登录失败率

## 服务端验证的基本原则

所有上述攻击路径都绕不过一个根本问题：**服务端是否真正执行了业务逻辑，还是在转发客户端的结论？**

一个帮助判断的问题：

> "如果一个技术能力很强的攻击者，可以完全控制客户端发出的任何 HTTP 请求，他能做什么？"

如果答案是"他可以让系统执行本不该执行的业务结果"，就说明服务端把业务决策权错误地交给了客户端。

**几个具体原则：**

- **输入验证（Input Validation）：** 所有客户端传入的参数，视为不可信输入，校验类型、范围、枚举值。
- **业务前置条件校验（Precondition Check）：** 领取奖励前，先验证奖励条件是否成立（由服务端自己记录的状态，不是客户端传来的状态）。
- **结果不依赖客户端申报（Result Ownership）：** 奖励金额、战斗结果、等级提升，由服务端根据服务端状态计算，不接受客户端"我打赢了，给我 X 奖励"的自报。
- **最小权限（Least Privilege）：** 每个接口只能访问和修改与其职责相关的数据，游戏客户端不应该有直接写入任意字段的权限。

## 安全设计的"安全失效"原则

**安全失效（Fail Secure / Fail Safe）** 是安全设计中最重要的原则之一，但在游戏后端中经常被忽视。

这个原则的含义是：**当系统遇到异常状态时，应当默认进入更安全的状态，而不是更宽松的状态。**

游戏后端的常见反例：

- 风控服务调用超时，默认放行（应该：默认拦截或降级处理）
- 签名验证模块报错，为了不影响用户体验，跳过验证继续执行（应该：返回错误，拒绝处理）
- 奖励配置查询失败，回退到固定值 9999（应该：拒绝发放奖励，记录错误）

"为了不影响体验而放宽安全校验"是一个非常危险的工程决策路径。每次这样的决策，都在攻击面上留下一个可利用的缝隙。

**工程边界的建议：**

安全机制应当独立于业务逻辑，且其失败应该是显式可见的（告警、日志）。不应该设计成"安全校验失败了但业务继续跑，没人知道"的隐式降级。

## 威胁建模的工程化落地

威胁模型不是写完就束之高阁的文档，它是工程决策的输入。实际落地的最小步骤：

1. **定期（至少每季度）更新威胁清单** — 特别是在新功能上线前，用攻击者视角过一遍设计文档。

2. **为每个高价值接口建立"最坏情况"假设** — 如果这个接口被无限调用会怎样？如果参数全被篡改会怎样？

3. **在 Code Review 中加入安全检查点** — 涉及资产变更、账号状态变更的代码，审查服务端是否做了完整的业务前置条件校验。

4. **安全事件的快速响应机制** — 一旦发现异常的批量请求或资产突增，要能在分钟级别触发告警和临时封禁，而不是靠人工每天巡检。

---

## 最短结论

游戏后端的安全威胁建立在"虚拟资产可变现"这个特殊动机上，攻击面比普通 Web 应用更广。

三类核心威胁（接口刷取、数据篡改、账号劫持）的根源都指向同一个失误：**服务端把本该自己掌握的业务决策权，通过信任客户端输入而让渡给了攻击者。**

防御体系的起点不是安全组件的堆叠，而是一个简单的认知转变：**客户端数据永远不可信，服务端负责执行业务逻辑，而不只是转发客户端的结论。**
