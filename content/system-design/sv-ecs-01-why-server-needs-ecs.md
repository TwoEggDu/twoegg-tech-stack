## 从一个错误的出发点说起

很多人接触服务端 ECS 的第一个念头是：「客户端用 Unity DOTS 效果很好，服务端也用 ECS 应该也行吧。」

这个逻辑不完全错，但出发点是错的。把客户端的解决方案平移到服务端，往往会在错误的地方用力——强行引入渲染相关的概念，照搬 DOTS 的 World 管理方式，或者直接嵌入 Unity 的 Job System。

服务端需要 ECS，不是因为「客户端 ECS 好用」，而是因为**服务端本身的计算问题和 ECS 擅长解决的问题高度匹配**。这两件事之间是独立成立的。要想用好服务端 ECS，必须先从服务端自己的问题出发。

---

## 服务端的核心计算问题

一个典型的 MMO 游戏服务端，或者任何有大量 AI 对象的游戏服务端，每帧需要做什么？

- 10000 个 NPC，每个 NPC 需要运行一帧 AI 状态机（当前状态 → 条件判断 → 下一状态）
- 5000 颗子弹，每帧更新位置、检测碰撞、判断生命周期
- 200 个 BOSS，每个运行更复杂的行为树，读取周边玩家数量、距离、仇恨值
- 50 个房间，每个房间有独立的地图状态，需要独立的仿真时钟

这些对象有一个共同特征：**同构**。绝大多数普通 NPC 的数据结构是相同的，绝大多数子弹的数据结构是相同的。对它们的操作也是相同的：同一套 AI 逻辑、同一套物理更新公式。

这个问题和客户端 ECS 要解决的问题**一模一样**：大量同构对象的高频批量更新。ECS 架构擅长的正是这件事——把同类对象的数据紧凑地排列在连续内存里，然后顺序遍历执行 System。

不同之处在于，服务端还叠加了一组客户端没有的约束，这些约束会深刻影响架构选择。但在讲约束之前，先要理解传统服务端是怎么处理这个问题的，以及问题出在哪里。

---

## 传统服务端架构的问题

游戏服务端最经典的架构是**线程池 + 消息队列**模型：

- 一个主线程负责接收网络消息，推入消息队列
- 若干工作线程从队列中取消息，执行对应的处理逻辑
- 每条消息对应一个对象（「NPC_1234 执行一次 AI」「子弹_5678 推进一帧」）
- 处理完毕后，如有需要，把结果推回给客户端

这个模型在对象数量少的时候运行良好。问题在于，**每条消息的处理都需要随机访问堆内存**。

假设有 10000 个 NPC，每个 NPC 是堆上独立分配的对象，持有它的行为树、状态机、位置、仇恨表等数据。每次处理「NPC_1234 执行 AI」，CPU 需要：

1. 从消息队列读出 NPC 的指针
2. 解引用指针，跳转到 NPC 对象所在的内存地址（大概率不在缓存里）
3. 读取行为树根节点的指针，再解引用（又一次缓存 miss）
4. 读取当前状态指针，再解引用
5. 执行逻辑，写回结果

10000 个 NPC 的一帧更新 = 10000 次以上的随机内存访问。现代 CPU 的 L3 缓存大约 8-32 MB，10000 个 NPC 对象很容易就超出缓存容量，导致大量缓存 miss。每次缓存 miss 大约惩罚 100-300 个 CPU 周期。

更糟糕的是，**随着对象数量增长，情况会非线性恶化**。对象越多，缓存竞争越激烈，每次访问的命中率越低，单核性能瓶颈提前到来。很多团队在对象数量从 1000 增长到 10000 时，发现 CPU 占用翻了不止十倍，根源就在这里。

## 服务端和客户端的差异对比

虽然计算问题相似，但服务端和客户端在整体架构上有根本差异：

| 维度 | 客户端 ECS | 服务端 ECS |
|---|---|---|
| **主要瓶颈** | 渲染调用、物理、动画 | AI 批量更新、网络 I/O、状态同步 |
| **并发模型** | 单 World，Job System 并行 | 多 World（每个房间一个），可多线程 |
| **持久化** | 无（重启加载场景） | 有（存档、断线重连、回放） |
| **横向扩展** | 不需要 | 需要（跨节点分片、跨服） |
| **语言选择** | C#（Unity DOTS）、Rust（Bevy） | C++（Flecs/EnTT）、Rust（Bevy）、Go（自研） |
| **渲染层** | 必须（Entities.Graphics） | 完全不需要 |

这张表说明了一件事：直接把 Unity DOTS 搬到服务端是不合适的。Unity DOTS 的设计重心在渲染和客户端物理，它的多 World 支持是事后补充的，它的持久化和网络 I/O 层几乎是空白的。服务端需要的是一个**轻量、headless、对 I/O 友好的 ECS 内核**。

---

## ECS 在服务端能解决什么

明确了差异，再来看 ECS 在服务端能真正带来什么收益。

**1. 同构对象的批量更新**

ECS 的核心数据结构是 Archetype + Chunk（或类似的分组存储方案）。相同组件组合的实体，数据紧凑排列在连续内存块里。遍历 10000 个 NPC，实际上是顺序扫描一段连续内存——CPU prefetcher 可以预取下一个对象的数据，缓存命中率极高。

同样是 10000 个 NPC 的一帧 AI 更新，ECS 架构下可以做到几乎全部在 L2/L3 缓存内完成，和传统堆对象的随机访问模型相比，性能差距可以达到 5-10 倍。

**2. 多 World 隔离**

服务端通常有多个游戏房间同时运行。ECS 的 World 对象天然是一个独立的状态容器——每个房间一个 World，World 之间完全隔离，可以在不同线程上并行 tick，不需要任何锁。房间创建就是创建新的 World，房间销毁就是销毁 World，生命周期管理非常清晰。

**3. 系统解耦**

ECS 的 System 通过 Query 声明自己依赖哪些组件，不依赖其他 System 的内部状态。AI System、移动 System、战斗 System 各自操作各自关心的组件集合，互不耦合。这对服务端代码的可维护性帮助极大——在客户端项目里，游戏逻辑往往会随着迭代而深度耦合，服务端的问题更严重，因为服务端代码通常没有客户端那么频繁的重构机会。

---

## ECS 在服务端解决不了什么

不要把 ECS 当成银弹。有三件事是 ECS 架构本身解决不了的。

**网络 I/O 仍然是异步的。** ECS 的 System 在同步的 World Tick 里执行，但网络收发包是异步的（epoll、io_uring、或者 Tokio 异步运行时）。这两个世界之间需要设计一个消息边界——异步 I/O 线程收到数据包，写入消息缓冲区，下一个 Tick 开始时 System 消费这些消息。这个边界的设计是服务端 ECS 架构里最容易出问题的地方，后续会专门讲。

**持久化仍然需要设计映射层。** ECS 的 Chunk 是 SoA（Structure of Arrays）布局，数据库通常是行式存储（或列式，但语义不同）。把 ECS 的实体状态保存到数据库，需要一个序列化 / 快照层。Flecs 提供了内置的 REST API 可以导出 World 状态，EnTT 则需要自己写序列化。这个层的设计复杂度不容小觑。

**横向扩容仍然需要外部协调。** ECS 是单节点的仿真内核，它不知道「另一台机器上也有 NPC」这件事。如果一个游戏世界需要多台机器协同，需要在 ECS 之外设计消息总线（Redis Pub/Sub、Kafka、NATS）来协调节点间的状态同步。SpatialOS 曾经尝试把 ECS 本身做成分布式的，这条路的结论留到后面讨论。

---

## 小结

服务端需要 ECS，是因为服务端的核心计算问题（大量同构对象高频批量更新）和 ECS 的设计目标高度吻合，而不是因为「客户端 ECS 好用」。传统线程池 + 堆对象模型在对象数量增长时会遭遇严重的缓存 miss 问题，ECS 的连续内存布局可以从根本上缓解这个问题。

但服务端和客户端的约束有根本差异：无渲染层、有异步 I/O、有持久化需求、有多 World 管理、有横向扩展需求。这五个约束决定了服务端 ECS 的架构选择不能照搬客户端的方案，也是后续框架选型（Flecs vs EnTT vs Bevy）的核心判据。
