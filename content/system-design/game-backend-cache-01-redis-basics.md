## 问题空间：为什么游戏后端特别需要 Redis

很多游戏后端项目在早期是这样跑的：MySQL 一张大表，所有请求直接查库，登录验证、排行榜计算、匹配队列一股脑全走关系型数据库。这条路能走多远？通常到内测阶段就会遇到第一面墙——数据库 QPS 被压垮，慢查询堆积，排行榜刷新卡成 PPT。

游戏后端的读写特征和电商、内容平台非常不同。玩家登录验证高频、高并发；排行榜需要实时排序但不需要持久化每一次变化；匹配队列需要毫秒级出队；道具数量需要原子加减。这些需求有一个共同特点：**对速度的要求远高于对关系查询的需求**，对原子性的要求远高于对事务一致性的要求。

Redis 恰好是为这类场景设计的。它把数据结构直接暴露在网络接口层，用 C 实现单线程事件循环，避免锁竞争，在普通服务器上轻松达到 10 万 QPS 以上。但 Redis 不是万能药——它的每个数据结构都有自己的边界，用错结构不仅解决不了问题，还会给你制造新问题。

本文的目标就是把这五种结构与游戏场景的对应关系讲清楚，让你在设计系统时能直接选到合适的结构，而不是每次都从 String 开始试错。

## 抽象模型：五种结构，五类问题

Redis 的数据结构不是语言数据类型，而是**针对特定操作模式优化的存储容器**。选型的核心问题永远是：你最频繁的操作是什么？范围查询、集合运算、还是原子计数？

理解每种结构之前，先建立一个心智模型：Redis 的所有命令复杂度都是公开的，O(1) 的命令可以随便调，O(N) 和 O(log N) 的命令需要估算 N 的规模。游戏场景里经常犯的错是把 N=全服玩家数 传给一个 O(N) 命令，然后惊讶于 Redis 阻塞。

### String：最简单但最常被滥用

String 是 Redis 最基础的结构，底层可以存任意字节序列，最大 512 MB。游戏后端的典型用法：

**Token 存储**：玩家登录后生成 JWT 或自定义 Token，以 `session:uid:{uid}` 为 key 存入 Token 字符串，设置 TTL（通常 7-30 天）。这个场景用 String 是对的，因为 Token 本身就是一个不可拆分的字符串，没有内部字段需要单独更新。

**计数器**：`INCR`、`INCRBY`、`DECR` 都是 O(1) 原子操作。活跃用户数、每日登录次数、API 调用限流（Rate Limiting）全都适合 String + INCR。特别注意：`INCR` 的原子性来自 Redis 的单线程模型，不需要额外加锁，这是 Redis 计数器比数据库计数器快一个量级的根本原因。

**分布式锁的原始实现**：`SET key value NX EX seconds`，用 NX（Not Exists）做互斥，EX 设置超时防死锁。这是分布式锁的最简形式，但有 ABA 问题（锁过期后被其他进程获取，原进程误删新锁），生产环境通常需要配合 Lua 脚本做 value 校验。

String 被滥用的典型场景是把整个玩家对象序列化成 JSON 塞进去。每次更新任何一个字段都要读出整个 JSON、修改、再写回，不仅浪费带宽，还在高并发下产生竞态。这个场景正确的结构是 Hash。

### Hash：玩家对象的天然容器

Hash 是 field-value 映射，一个 key 下可以存多个字段。游戏后端最典型的用法是玩家缓存对象：

```
HSET player:uid:10001 name "张三" level 50 gold 9999 last_login 1712345678
```

Hash 的优势在于**字段级别的读写**：只需要更新金币时，`HSET player:uid:10001 gold 8888`，不需要读出整个对象。`HGET`、`HSET`、`HMGET`（批量读取指定字段）都是 O(1) 或 O(N)（N 是请求的字段数，不是 Hash 的总字段数）。

**容量警告**：Hash 有两种底层编码，当字段数量少于 128 且所有 value 都小于 64 字节时，Redis 使用 ziplist（压缩列表）存储，内存极其紧凑；超过这个阈值后转换为 hashtable，内存占用跳升。在设计玩家缓存时需要控制字段数量，把变化频率极低的字段（如账号注册信息）留在 MySQL，把热字段（level、gold、online_status）放 Redis Hash。

**不适合场景**：Hash 没有原生的过期控制到字段级别（只能对整个 key 设 TTL）。如果需要字段级别过期，比如每个 Buff 各自有过期时间，应该给每个 Buff 用独立的 String key，而不是把所有 Buff 塞进一个 Hash。

### List：消息队列和活动流

List 是双端链表，支持从头部（Left）和尾部（Right）的 O(1) push/pop。游戏后端的两个核心用法：

**消息队列（简单版）**：生产者 `RPUSH queue:mail uid:10001:msg`，消费者 `BLPOP queue:mail 30`（阻塞 pop，超时 30 秒）。BLPOP 在队列为空时阻塞，不轮询，CPU 友好。这是游戏内邮件系统、系统通知的常见实现方式。

**活动流 / 最近记录**：`LPUSH recent:battles:{uid} {battle_id}`，然后 `LTRIM recent:battles:{uid} 0 99` 只保留最近 100 条。每次推新记录+裁剪是 O(N)（N 是裁剪数量），但因为总长度被 TRIM 控制，实际上是常数时间。玩家的战斗记录、好友动态、聊天记录前 N 条都适合这个模式。

**不适合场景**：List 没有去重能力，也不能按 score 排序。如果要的是"排名"或"唯一元素集合"，List 是错误的选择。

### ZSet：排行榜和优先级队列的专属结构

ZSet（Sorted Set）是 Redis 里最"游戏友好"的数据结构。每个 member 关联一个 float 类型的 score，所有 member 按 score 自动保持有序。底层是 ziplist（小数量）+ skiplist（大数量）。

**排行榜**：
```
ZADD leaderboard:pvp 9500.0 "uid:10001"
ZREVRANK leaderboard:pvp "uid:10001"      # 获取降序排名，O(log N)
ZREVRANGE leaderboard:pvp 0 99 WITHSCORES # 获取 Top 100，O(log N + M)
ZSCORE leaderboard:pvp "uid:10001"         # 获取玩家分数，O(log N)
```

这套操作的时间复杂度都是 O(log N)，N 是排行榜总人数。全服 100 万玩家的排行榜，`ZREVRANK` 的耗时和 1000 人的排行榜几乎没有差别，这是数据库 `ORDER BY` 无法比拟的优势。

**同分排名问题**：当两个玩家分数完全相同时，ZSet 按 member 的字典序排列，这通常不符合游戏需求（一般要求先达到分数的玩家排名更靠前）。工程解法是把时间戳编码进 score，例如 `score = 实际分数 * 1e10 + (MAX_TIMESTAMP - 当前时间戳)`，用高位存分数、低位存时间偏移，保证同分时按时间先后排序。

**优先级队列**：匹配系统中，用玩家的等待时长或段位作为 score，`ZPOPMIN` 优先取出得分最低（等待最久或段位最接近）的玩家进行匹配。

### Set：去重与关系运算

Set 是无序不重复集合，底层是 intset（全整数）或 hashtable。游戏后端典型用法：

**去重**：`SADD daily:login:20260404 uid:10001`——每日登录去重统计。相比 HyperLogLog（近似计数），Set 的精确计数在集合规模可控（万级以下）时是更简单的选择。

**好友关系**：把玩家的好友列表存为 Set，用 `SINTERSTORE` 求两个玩家的共同好友，`SDIFFSTORE` 求差集（"Ta 有的你没有的好友"）。Set 的集合运算是 O(N*M) 或 O(N+M)，在好友列表规模有限（几百人）的情况下完全可以接受。

**黑名单 / 白名单**：`SISMEMBER blacklist:uid uid:10001`——O(1) 判断玩家是否在黑名单。

**不适合大规模数据**：如果要统计全服数亿个事件去重 UV，Set 的内存消耗是不可接受的，应该用 HyperLogLog（误差率约 0.81%，内存固定 12 KB）。

## 具体实现：选型的判断依据

工程中选数据结构的核心问题：

1. **操作是对整体还是对字段？** 对整体（读/写整个值）→ String；对字段（只更新某几个字段）→ Hash
2. **需要顺序吗？** 按插入顺序 → List；按分数排序 → ZSet；无序 → Set 或 Hash
3. **需要唯一性吗？** 需要去重 → Set 或 ZSet；不需要 → List 或 String
4. **操作频率多高？** O(N) 的命令在 N > 1 万时要开始警惕；KEYS、SMEMBERS 等命令在生产环境应当禁用或有条件使用

## Redis 不是主数据库

必须在这里明确一个工程边界：**Redis 不应该是游戏后端的主数据库**。

原因有三。第一，Redis 的持久化（RDB 快照 + AOF 追加日志）是为缓存场景设计的，不是为强一致性数据存储设计的。AOF + 每秒 fsync 在宕机时最多丢失 1 秒数据，这对排行榜可以接受，对玩家金币不可以接受。第二，Redis 的内存是有限资源，全量玩家数据放内存不经济。第三，Redis 不支持复杂的关系查询（JOIN、子查询），游戏里的账单流水、背包历史这类需要关系分析的数据天然属于关系型数据库。

正确的架构是：MySQL / PostgreSQL 作为 source of truth，Redis 作为热数据缓存层，承接高频读写；玩家下线或缓存过期后，数据从 MySQL 重新加载。

## 内存模型与过期策略基础

Redis 的内存管理有几个关键知识点：

**过期策略**：Redis 使用"惰性删除 + 定期删除"组合。惰性删除是访问 key 时检查是否过期，过期则删除；定期删除是每隔 100ms 随机抽查一批 key，删除已过期的。这意味着已过期的 key 不会立即释放内存，在高 key 数量的场景下，内存实际使用可能比预期高。

**内存淘汰（maxmemory-policy）**：当内存达到上限时，Redis 有 8 种淘汰策略，游戏后端最常用的是 `allkeys-lru`（淘汰最近最少使用的 key）或 `volatile-lru`（只淘汰设置了 TTL 的 key）。**一定要显式配置 maxmemory 和淘汰策略**，否则 Redis 默认不淘汰，内存耗尽后所有写操作会返回 OOM 错误。

**大 key 问题**：单个 key 的 value 超过 10 KB 就要警惕，超过 100 KB 要主动优化。大 key 在序列化/反序列化时会阻塞单线程的 Redis，在网络传输上消耗带宽，在淘汰时造成内存抖动。

## 最短结论

Redis 的价值在于把"对的数据结构"直接放在网络边界上——选错结构，Redis 只是一个贵的 HashMap；选对结构，它才是游戏后端的速度护盾。
