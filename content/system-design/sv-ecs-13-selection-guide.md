## 问题空间：选型困境的真正来源

读完一个技术系列之后，通常有两种困境。

第一种：每个框架看起来都不错，不知道选哪个。

第二种：每个框架都有缺点，不知道哪个缺点对我影响最小。

这两种困境有同一个根源：**还没有把自己的约束条件说清楚**。

选型不是"哪个框架更好"的问题，而是"在我的约束条件下，哪个框架的缺点我最能接受"的问题。四个维度——语言约束、同时在线规模、团队现有技术栈、是否共享客户端代码——是把候选框架从五六个过滤到一两个的最有效工具。

但在进入框架选型之前，先回答一个更重要的问题。

---

## 先问：你真的需要 ECS 吗？

ECS 不是"高性能游戏服务端"的同义词。它是一种特定的架构模式，有它最适合的问题类型。

**什么时候根本不需要 ECS？**

### 场景一：小规模回合制游戏

扑克、麻将、棋牌类游戏，单房间同时在线 2–10 人，房间内状态量极小，游戏逻辑以"事件驱动"为主（玩家操作 → 状态跳转 → 广播结果）。

这类游戏的服务端用状态机 + 事件总线就够了，引入 ECS 只会增加概念复杂度，让每个实习生都需要先学 Archetype 才能改一个 Bug。

### 场景二：逻辑简单的弱实时游戏

休闲竞技、跑酷、消消乐类。游戏逻辑主要跑在客户端，服务端只做结果验证和防作弊，没有大量同构实体需要高频更新。

这类游戏的服务端用传统 OOP 架构（玩家对象、房间管理器）完全胜任，不需要 ECS 的数据局部性优化，因为根本没有那么多实体。

### 场景三：超低并发的 PvE 游戏

单人或 2–4 人的合作 PvE，房间内实体数量不超过 200，Tick 逻辑简单，服务端只需要每秒跑几次状态校验。

**何时才真正需要 ECS？**

- 单房间/地图内同时存在 500 个以上活跃实体，且每帧都需要对大量实体做同构处理
- 有大量相似的 Entity，它们只在少数 Component 的组合上有差异（NPC、子弹、道具等）
- 游戏逻辑高度并行化，System 之间的依赖关系清晰、数量众多
- 需要在客户端和服务端共享游戏逻辑代码（Shared Simulation）

如果你的项目在上述四点上都打叉，先认真考虑传统架构，不要为了用 ECS 而用 ECS。

## 四个判断维度

确认需要 ECS 之后，用四个维度来定位：

### 维度一：语言约束（Language）

这是最硬的约束，没有例外。

- 团队只有 **C/C++ 能力**：Flecs（C99 核心，C++ wrapper）或 EnTT（C++17）
- 团队只有 **C# 能力**（Unity 技术栈）：Arch、LeoECS（Leopotam）、DefaultEcs
- 团队有 **Rust 能力**（或愿意投入学习成本）：Bevy ECS
- 团队同时有 **C++ 和 C#**（服务端 C++，客户端 Unity）：Flecs 或 EnTT 服务端，另起项目

语言约束不是"我们能学新语言吗"，而是"我们能在项目周期内，让所有维护这个服务端的工程师都达到生产级别的熟练度吗"。初创团队做新项目可以评估引入 Rust，但中途切换语言的风险极高。

### 维度二：同时在线规模（Scale）

不是"未来的最大规模"，而是"Day 1 上线时的真实规模"，加上合理的增长预期。

| 规模分级 | 单房间/实例实体数 | 并发房间数 | 备注 |
|---------|----------------|---------|------|
| 小规模 | < 500 | < 100 | 绝大多数独立游戏和小团队项目 |
| 中等规模 | 500–5000 | 100–1000 | 主流多人游戏，MOBA/FPS 竞技 |
| 大规模 | 5000–100000 | 1000+ | MMO、开放世界、大型 Battle Royale |
| 超大规模 | > 100000 | N/A（单实例） | EVE Online 类、SpatialOS 类 |

绝大多数游戏的目标规模在小到中等之间。超大规模需要的是分布式 ECS 架构（上一篇讨论的 SpatialOS 思路），而不是换一个更好的单进程框架。

### 维度三：团队现有技术栈（Team Stack）

技术栈不只是"我们用什么语言"，还包括：

- 现有的网络库是什么？（ENet、KCP、gRPC、Netty）
- 现有的持久化方案是什么？（Redis、MySQL、自研内存数据库）
- CI/CD 和监控基础设施是否成熟？

ECS 框架需要集成到现有技术栈里，集成摩擦越小越好。一个和你现有网络层无缝集成的框架，比一个性能稍好但需要重写整个 I/O 层的框架更值得选。

### 维度四：是否共享客户端代码（Code Sharing）

这个维度经常被低估。

如果服务端和客户端使用同一门语言，可以共享游戏逻辑代码（移动预测、伤害计算、技能系统）。这不仅减少了维护两份代码的成本，更重要的是保证了"客户端预测的逻辑"和"服务端权威逻辑"是同一份代码，确定性更容易保证。

- **C# 技术栈**（Unity 客户端）：可以共享 C# 游戏逻辑，Arch/LeoECS 是有意义的选择
- **C++ 技术栈**（UE 客户端）：可以共享 C++ 游戏逻辑，Flecs/EnTT 是自然的选择
- **异构技术栈**（Unity + C++ 服务端）：代码共享困难，通常只能共享协议定义（Protobuf）

## 各框架的四维定位

| 框架 | 语言 | 适用规模 | 典型技术栈 | 代码共享 |
|------|-----|---------|----------|---------|
| **Flecs** | C/C++ | 中等–大规模 | 独立 C++ 服务端，UE 插件 | C++ 客户端（UE） |
| **EnTT** | C++17 | 小–中等规模 | C++ 游戏服务端，Minecraft 生态 | C++ 客户端（UE） |
| **Bevy ECS** | Rust | 小–中等规模 | 全 Rust 技术栈，安全优先 | Rust 客户端（罕见） |
| **Arch** | C# (.NET 6+) | 小–中等规模 | .NET 独立服务端，Unity 共享代码 | Unity 客户端 |
| **LeoECS (Leopotam)** | C# | 小规模 | Unity 服务端逻辑共享 | Unity 客户端（核心用例） |
| **DefaultEcs** | C# | 小–中等规模 | .NET 服务端，Monogame | C# 客户端 |

**Flecs 的特点**：功能最完整，内置 Relationship、Observer、REST API 监控界面。学习曲线较陡，但文档是这几个框架里最好的。适合需要大量 Query 灵活性和运行时反射的游戏。

**EnTT 的特点**：API 最符合 C++ 程序员的直觉，头文件-only，嵌入方便。Minecraft（Java Edition 的某些工具链）和大量独立游戏使用。规模上限比 Flecs 稍低，但对中等规模完全够用。

**Bevy ECS 的特点**：Rust 的所有权模型天然防止了 System 间的数据竞争，编译期就能捕获很多并发错误。性能出色，但 Rust 的学习曲线是这里最陡的。适合追求安全性的小团队，不适合需要快速迭代的项目。

**Arch 的特点**：性能在 C# ECS 框架里名列前茅，API 设计现代（Source Generator 生成高效查询代码）。适合 Unity 团队把服务端逻辑抽出来跑在 .NET 上，最大化代码共享。

**LeoECS (Leopotam) 的特点**：设计极简，几乎零运行时开销，没有反射。核心用例是 Unity 客户端的 ECS 逻辑层，服务端复用时仍然适用，但缺少服务端常用的 Query 功能（兴趣管理、Component 索引）。适合小型团队、逻辑量不大的项目。

## 决策树

按三个问题依次过滤，通常可以把候选框架缩短到 1–2 个。

```
Step 1：语言
├── C/C++    → 候选：Flecs, EnTT
├── C#       → 候选：Arch, LeoECS, DefaultEcs
├── Rust     → 候选：Bevy ECS
└── 其他     → 自研或寻找对应生态框架

Step 2：规模（在 Step 1 的候选里过滤）
├── 小规模   → 候选内所有框架均可
├── 中等规模 → C++：Flecs 或 EnTT 均可；C#：Arch 优先；Rust：Bevy ECS
└── 大规模   → C++：Flecs 优先（关系型查询、运行时反射更完整）

Step 3：代码共享需求
├── 需要共享 Unity 逻辑  → C# 路线：Arch 或 LeoECS
├── 需要共享 UE 逻辑     → C++ 路线：Flecs 或 EnTT
└── 不需要共享           → 按 Step 1 + Step 2 的结果选最熟悉的
```

## 三个典型项目的选型示例

### 示例 A：竞技 FPS，60 人房间，C++ 后端，UE5 客户端

**约束**：C++ 语言、中等规模（每房间约 300–500 活跃实体）、UE5 客户端可以共享 C++ 物理逻辑。

**过滤结果**：C++ 路线 → 中等规模 → Flecs 或 EnTT 均可 → 需要共享 UE 逻辑 → Flecs（有官方 UE 插件，Query 灵活性更好）。

**最终建议**：Flecs，配合 Flecs-ue 插件，把移动预测和碰撞检测逻辑放在共享的 C++ 模块里，客户端和服务端都链接同一个库。

### 示例 B：开放世界 MMO，单地图 5000+ 玩家，C++ 后端，Unity 客户端

**约束**：C++ 语言、大规模（单地图 5000+ 玩家 + 大量 NPC）、Unity 客户端（C# 和 C++ 异构）。

**过滤结果**：C++ 路线 → 大规模 → Flecs 优先 → 异构技术栈，代码共享通过 Protobuf 协议定义。

**附加考虑**：5000+ 玩家的规模可能需要分区，需要在 Flecs 之上设计区域分区和跨区消息路由（参考上一篇的进程间消息路由方案）。单纯换框架解决不了超大规模的问题。

### 示例 C：4 人合作 PvE，Unity 客户端，小团队（4 人），希望最快落地

**约束**：C# 语言、小规模（单房间约 100–200 实体）、Unity 技术栈、团队资源有限。

**过滤结果**：C# 路线 → 小规模 → Arch 或 LeoECS 均可 → 需要共享 Unity 逻辑 → LeoECS（团队如果已经在客户端用了 LeoECS，服务端直接复用，学习成本为零）。

**附加考虑**：如果团队没有在客户端用任何 ECS，先评估是否真的需要 ECS——4 人 PvE、100 实体的规模，传统 OOP 架构同样可以胜任，不必引入 ECS 的额外复杂度。

## 工程边界

**框架锁定的风险不来自框架本身，来自把业务逻辑写进 System 里。** 如果 System 代码直接依赖 Flecs 的 `ecs_query_t` 或 EnTT 的 `registry.view()`，换框架就是重写所有 System。解决方法是在 System 和框架之间加一层接口（Adapter），业务逻辑只依赖接口，不直接调用框架 API。这个代价值得付，尤其是在项目早期。

**选型文档要记录"为什么不选 X"。** 决策过程中被排除的框架，以及排除原因，应该写进 ADR（Architecture Decision Record）。6 个月后新来的工程师看到 Flecs 时，会问"为什么不用 EnTT"，这个问题应该有书面答案，而不是"问一下老员工"。

**不要在需求未稳定前锁定框架。** 如果游戏类型还在调整（从 PvP 改成 PvE，从 60 人改成 10 人），这时候锁定服务端框架是过早决策。用最简单的方案先跑起来，等需求稳定后再做选型是更务实的做法。

---

## 最短结论

ECS 是手段，服务端性能和可维护性才是目标。

不需要 ECS 的项目用了 ECS，得到的是复杂度，不是性能。需要 ECS 的项目用错了框架，得到的是迁移成本，不是速度。四维判断的目的是在正确的时候做正确的决定：先确认是否需要 ECS，再按语言→规模→代码共享顺序过滤，最后选最熟悉的那个——因为工程实践的差异，往往比框架本身的性能差异更影响最终结果。
