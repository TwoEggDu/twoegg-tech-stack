> 一句总判断：`帧同步不是"把输入发给所有人"这么简单。它要求技能系统的每一层——从浮点运算到 Buff 聚合到物理碰撞——都满足确定性约束，否则一帧的微小偏差会在数十帧后滚雪球成完全不同的战斗结果。`

22 篇把"技能触发了"拆成了 5 个含义，并讲清了帧同步和状态同步两种模型下这 5 个含义的时序差异。

那一篇的结论是：帧同步让含义 2 到 5 在所有端完全一致，代价是含义 1 到 3 之间有不可消除的输入延迟。

但这句话里有一个隐含的前提没有展开：

`"所有端完全一致"不是帧同步自动保证的，而是需要你的技能系统满足确定性约束才能成立。`

如果技能系统的任何一层——浮点计算、随机数序列、Buff 求值顺序、物理碰撞——在不同机器上产生了哪怕一个 bit 的差异，帧同步的"所有端一致"就不成立了。

这就是帧同步下技能系统真正难的地方：不是网络层怎么传输入，而是怎么保证技能系统的每一次计算在所有端产生完全相同的结果。

## 确定性约束传导到技能系统的 5 个层面

帧同步的核心公理只有一句话：

`给定相同的初始状态和相同的输入序列，任何一台机器运行出来的结果必须完全一致。`

这句话看起来很简单，但它传导到技能系统后，至少有 5 个层面需要重新审视。

### 层面 1：浮点数

这是帧同步下第一个必须解决的问题，也是最容易被低估的问题。

IEEE 754 浮点运算在理论上是确定性的——相同的操作数和相同的运算应该产生相同的结果。但实际上，不同平台、不同编译器、不同优化等级下，浮点运算的结果可能不一致。

原因有几个：

1. **编译器优化**。编译器可能重新排列浮点运算的顺序（比如把 `a + b + c` 优化为 `(a + c) + b`），而浮点加法不满足结合律，不同顺序可能产生不同结果。
2. **FMA 指令**。某些处理器支持 Fused Multiply-Add 指令，它的精度和分开做乘法再做加法是不同的。如果一台机器用了 FMA 而另一台没用，结果就会有微小差异。
3. **x87 和 SSE 的精度差异**。x86 架构上，x87 浮点单元默认用 80 位精度计算然后截断到 64 位，而 SSE 直接用 64 位。同一段代码在不同执行路径下可能走不同的浮点单元。
4. **跨平台差异**。如果你的游戏需要在 PC 和移动端之间帧同步，ARM 和 x86 的浮点行为差异更大。

技能系统里浮点数无处不在：伤害公式、移速计算、冷却倒计时、碰撞检测、Buff 数值聚合。任何一处浮点不一致，都会导致帧同步崩溃。

解决方案有两条路：

**路径一：定点数（Fixed-Point）**

把所有游戏逻辑层的数值运算从浮点数替换为定点数。定点数本质是整数运算，不存在平台差异问题。

```csharp
// 定点数示例：用 int64 表示，低 16 位为小数部分
public struct FPNumber
{
    private long rawValue;
    public static readonly int FractionBits = 16;

    public static FPNumber operator +(FPNumber a, FPNumber b)
    {
        return new FPNumber { rawValue = a.rawValue + b.rawValue };
    }

    public static FPNumber operator *(FPNumber a, FPNumber b)
    {
        return new FPNumber
        {
            rawValue = (a.rawValue * b.rawValue) >> FractionBits
        };
    }
}
```

定点数的代价是：精度有限（取决于小数位数）、开发体验差（所有数值字面量都要转换）、和引擎的渲染/物理层交互时需要频繁转换。

**路径二：确定性浮点（Deterministic Float）**

不替换浮点数，而是约束所有浮点运算走相同的执行路径。具体做法包括：

- 编译器禁止浮点重排序优化（如 MSVC 的 `/fp:strict`，GCC 的 `-ffp-contract=off`）
- 统一使用 SSE 指令，禁用 x87
- 禁用 FMA 指令
- 所有平台使用相同的编译器和编译选项

这条路的可行性取决于你是否能完全控制编译环境。如果是 PC 独占游戏且所有玩家用相同的构建版本，通常可行。如果是跨平台游戏，几乎不可能走这条路。

技能系统里的实际建议：如果你选择帧同步，优先考虑定点数。虽然开发成本更高，但它提供了最强的确定性保证，不依赖编译器行为。

### 层面 2：随机数

技能系统里有大量随机行为：暴击判定、闪避计算、随机目标选择、散布角度、随机 Buff 触发概率。

帧同步下，所有客户端必须使用完全相同的随机数序列。

做法很直接：所有客户端使用同一个伪随机数生成器（PRNG），用同一个种子初始化，并且按完全相同的顺序消费随机数。

```csharp
public class DeterministicRandom
{
    private uint state;

    public DeterministicRandom(uint seed)
    {
        state = seed;
    }

    // xorshift32，简单且跨平台一致
    public uint Next()
    {
        state ^= state << 13;
        state ^= state >> 17;
        state ^= state << 5;
        return state;
    }
}
```

关键约束不在 PRNG 算法本身——任何确定性 PRNG 都行。关键约束在于：

1. **种子必须全端一致。** 通常在对局开始时由服务器或房主生成一个种子广播给所有玩家。
2. **消费顺序必须全端一致。** 如果一端先算了暴击再算了闪避，另一端先算了闪避再算了暴击，即使两端用同一个种子和同一个 PRNG，后续所有随机数都会错开。

第二点是更容易出错的地方。技能系统里随机数的消费顺序通常由逻辑帧内的执行顺序决定，而执行顺序又取决于实体遍历顺序、Buff 触发顺序、投射物碰撞顺序等等。

如果这些顺序在不同端不一致，随机数序列就会分叉。

实践建议：把随机数消费封装到确定性随机管理器中，按固定的调用协议使用，禁止任何"随便取一个随机数"的写法。

### 层面 3：逻辑帧与渲染帧分离

帧同步下，游戏逻辑必须运行在固定步长的逻辑帧上（比如每秒 30 逻辑帧或 60 逻辑帧），而不是跟着渲染帧率走。

如果技能系统的 Tick 和渲染帧绑定，那么不同机器的帧率差异会导致逻辑执行次数不同，确定性立刻崩溃。

这个约束对技能系统的影响是：

1. **所有时间相关的计算用逻辑帧数表示，不用秒数。** 冷却不是"3.5 秒"，而是"105 逻辑帧"（假设逻辑帧率 30fps）。前摇不是"0.2 秒"，而是"6 逻辑帧"。
2. **`deltaTime` 是常量。** 技能系统里不能出现 `Time.deltaTime` 这类和实际帧率相关的值。
3. **渲染层只做插值。** 渲染帧率可以是 60fps、120fps 或任意值，但渲染层只在两个逻辑帧之间做视觉插值，不参与任何逻辑计算。

```csharp
// 帧同步下的冷却推进
public void TickCooldown()
{
    // remainingFrames 是逻辑帧数，不是秒数
    if (remainingFrames > 0)
        remainingFrames--;
}

// 渲染层只负责插值表现
public float GetCooldownDisplayRatio(float interpolation)
{
    float current = remainingFrames - interpolation;
    return Mathf.Max(0, current / totalFrames);
}
```

### 层面 4：Buff / Effect 求值顺序

08 篇已经讨论过 Buff 的叠层、刷新、覆盖和聚合顺序问题。

在状态同步下，Buff 求值顺序不一致的最坏后果是"客户端预测的伤害和服务端不同"——服务端的权威结果会纠正它。

但在帧同步下，Buff 求值顺序不一致的后果是致命的：两端算出不同的伤害值，从此所有后续状态全部分叉。

帧同步对 Buff 求值顺序的要求是：

1. **Modifier 聚合顺序必须固定。** 不能依赖 Hash 表的遍历顺序或者 Buff 的添加时间戳（时间戳在不同端可能有微小差异）。最稳的做法是给每个 Modifier 分配一个全局确定性 ID，按 ID 排序后聚合。
2. **同帧添加多个 Buff 的处理顺序必须固定。** 如果同一逻辑帧内有多个 Buff 被添加或移除，它们的处理顺序不能依赖事件到达顺序，而必须按确定性规则（比如 Buff 类型优先级 + ID）排序。
3. **快照时机必须统一。** 如果某些 Buff 使用快照（Snapshot）语义，快照的采样时机必须在所有端完全一致。

### 层面 5：物理和碰撞

技能系统的命中判定通常依赖碰撞检测。在帧同步下，碰撞检测也必须确定性。

这意味着：

- 不能使用引擎内置的非确定性物理引擎（比如 Unity 的 PhysX 默认模式）
- 需要使用确定性物理引擎，或者自己实现简化的碰撞检测
- 碰撞检测的遍历顺序、浮点计算路径、接触点排序都必须确定性

实践中有两种做法：

**简化碰撞**：很多帧同步游戏不使用完整的物理引擎做技能命中判定，而是用简化的几何检测——圆形 vs 圆形、矩形 vs 圆形、扇形检测等。这些几何检测用定点数实现后天然确定性。

**确定性物理引擎**：如果游戏需要完整的物理交互（比如刚体碰撞影响技能弹道），则需要引入确定性物理引擎。开源方案如 Box2D 的定点数移植版，或者商业方案如 Photon Quantum 内置的确定性物理。

## 输入收集与帧锁定

确定性只解决了"相同输入产生相同输出"这一半。另一半是：怎么保证所有端在每个逻辑帧收到相同的输入。

### 经典帧锁定（Lockstep）

最基础的帧同步模型是帧锁定：

1. 每个玩家在逻辑帧 N 采集自己的输入
2. 把输入发送给所有其他玩家（或中继服务器）
3. 等待收到所有玩家在帧 N 的输入
4. 所有端用同一组输入执行帧 N 的逻辑
5. 推进到帧 N+1，重复

这个模型的问题是：如果某个玩家的输入包晚到了，所有人都要等。

等待导致卡帧，卡帧导致手感差。

### 延迟帧（Input Delay）

为了缓解网络抖动的影响，通常会引入输入延迟：

玩家在帧 N 采集的输入，不在帧 N 执行，而是延迟 D 帧后在帧 N+D 执行。

```text
帧 N:   采集输入 → 标记为"帧 N+D 的输入"→ 发送
帧 N+D: 收到所有玩家的输入 → 统一执行
```

D 越大，容忍的网络延迟越高——因为有更多时间等待远端输入到达。
D 越小，手感越直接——但网络抖动时更容易卡帧。

格斗游戏通常的 input delay 是 3-5 帧（50-83ms @ 60fps）。对于技能系统来说，这意味着玩家按下技能键后要等 3-5 帧才能看到技能开始执行。

### 格斗游戏的 input delay 模型

格斗游戏是帧同步技能系统最极端的场景。

在格斗游戏里，技能系统的每一帧都是关键的——1 帧的差异可能决定一个连招是否成立，一个防御是否有效。

格斗游戏对 input delay 的处理通常是：

1. 固定 input delay（比如 3 帧），并在 UI 上告知玩家
2. 根据网络状况动态调整 input delay（可能在 2-6 帧之间浮动）
3. 结合回滚重播来减少 input delay 的感知影响

第三种就是下面要讲的 GGPO 模型。

## 回滚重播（Rollback Netcode / GGPO）

纯帧锁定 + 输入延迟的模型在网络条件好的时候没问题，但在延迟较高或抖动较大时，要么卡帧（等待输入），要么输入延迟很大（手感差）。

GGPO（Good Game Peace Out）提出了一种更激进的方案：预测执行 + 回滚重播。

### 基本原理

1. 不等待远端输入。如果帧 N 还没收到对方的输入，就预测对方的输入（通常预测为"和上一帧相同"）。
2. 用预测输入执行帧 N 的逻辑，继续推进。
3. 当真实输入到达时，检查预测是否正确。
4. 如果预测正确，什么都不用做。
5. 如果预测错误，回滚到分歧帧，用真实输入重新执行从分歧帧到当前帧的所有逻辑。

```text
当前帧: 10
最后确认帧: 7（帧 8、9、10 的远端输入都是预测的）

收到远端帧 8 的真实输入，发现和预测不同:
  1. 保存当前帧 10 的状态
  2. 回滚到帧 7 的状态快照
  3. 用帧 8 的真实输入重新执行帧 8
  4. 用帧 9 的预测输入重新执行帧 9
  5. 用帧 10 的预测输入重新执行帧 10
  6. 结果替换当前状态
```

### GGPO 对技能系统的额外要求

回滚重播对技能系统提出了两个额外要求：

**第一，状态快照。**

回滚要求你能保存和恢复任意逻辑帧的完整状态。

对技能系统来说，这意味着你需要能快照：

- 所有角色的状态机当前状态（Idle / Windup / Active / Recovery / Cooldown）
- 所有激活中的技能实例的完整运行时数据
- 所有 Buff 的当前叠层、剩余时间、Modifier 列表
- 所有冷却的剩余帧数
- 所有资源（蓝、怒气等）的当前值
- 随机数生成器的当前状态
- 所有投射物的位置、速度和生命周期状态

如果技能系统的状态分散在各种对象的各种字段里，快照和恢复的成本会非常高。

这就是为什么 18 篇讨论的 Overwatch ECS 架构天然适合回滚——ECS 的所有状态都在 Component 里，快照就是复制一组 Component 数据，恢复就是替换回来。

**第二，重新执行 N 帧。**

回滚后需要从分歧帧重新执行到当前帧。如果当前帧和分歧帧之间差了 5 帧，就需要在一帧的时间内重新执行 5 帧的逻辑。

对技能系统来说，这意味着 Tick 函数必须足够快。如果一次技能 Tick 耗时 1ms，5 帧回滚就需要额外 5ms。在 60fps（16.6ms 一帧）的预算下，5ms 的回滚成本是可以接受的。但如果回滚帧数更多，或者技能 Tick 更重，就会超预算。

实践中的限制：大多数 GGPO 实现会限制最大回滚帧数（比如 7-8 帧）。如果网络延迟太高导致需要回滚超过最大帧数，就退化为帧等待。

### GGPO 在技能系统中的限制

GGPO 适用的场景有明确的边界：

1. **人数限制。** GGPO 最适合 2 人对战（如格斗游戏），因为只需要预测一个对手的输入。4 人已经是实践上限，更多人时预测错误率和回滚成本都急剧上升。
2. **表现层回滚。** 逻辑层回滚相对简单，但表现层（动画、特效、音效）的回滚非常困难。大多数 GGPO 实现选择不回滚表现层，而是在预测错误时做视觉修正（比如角色位置瞬间跳变）。这对格斗游戏的技能系统来说还可以接受，但对需要复杂特效表现的 RPG 技能系统来说，视觉跳变会非常明显。
3. **状态量限制。** 快照的大小直接影响内存占用和拷贝开销。格斗游戏的战斗状态相对紧凑（两个角色 + 有限的投射物），快照成本低。但一个有 20 个角色、每个角色 10 个 Buff、场上 50 个投射物的 MOBA 场景，快照成本就不可忽视了。

## 帧同步 vs 状态同步的技能系统工程成本对比

| 维度 | 帧同步 | 状态同步 |
|------|------|------|
| 确定性要求 | 强制——浮点、随机、物理都必须确定性 | 不要求——服务端是权威 |
| 数值层改造 | 定点数库或确定性浮点约束 | 无特殊要求 |
| 随机数管理 | 统一种子 + 严格消费顺序 | 服务端决定随机结果 |
| Buff 聚合 | 必须固定顺序，不能依赖运行时状态 | 服务端权威结果纠正差异即可 |
| 物理/碰撞 | 确定性物理引擎或简化碰撞 | 服务端权威碰撞 |
| 快照/回滚 | 需要完整状态快照能力（GGPO） | 只需局部预测状态的回滚 |
| 网络带宽 | 只传输入，带宽极低 | 传状态同步包，带宽随实体数增长 |
| 反作弊 | 弱——所有客户端拥有完整状态 | 强——服务端拥有权威状态 |
| 前期成本 | 高——确定性基础设施投入大 | 中——预测和回滚有复杂度但可以渐进 |
| 后期维护 | 低——确定性保证一旦建立就很稳 | 中高——每个新技能都要考虑预测策略 |
| 调试难度 | 中——Desync 排查需要帧对比工具 | 中——预测回滚问题需要对照客户端和服务端日志 |
| 适合场景 | 格斗（2-4 人）、RTS、棋牌、动作对战 | MMORPG、FPS、大规模 MOBA |

核心判断：帧同步的技能系统工程成本是前置的——你必须在项目早期就建立完整的确定性基础设施。一旦建立，后续添加新技能的边际成本很低（因为所有技能都在同一个确定性框架里运行）。

状态同步的技能系统工程成本是分散的——每个新技能都需要考虑"哪些该预测、哪些该等确认、预测错了怎么回滚"。但前期不需要大规模基础设施投入。

---

## 和 18 篇（Overwatch 确定性模拟）的关系

18 篇讨论的 Overwatch ECS 方案，本质上是在状态同步的网络架构（服务器权威 + 客户端预测）上，用帧同步的思想（确定性模拟）来保证预测质量。

具体来说：

1. Overwatch 有服务器，服务器拥有权威状态——这是状态同步。
2. 客户端本地运行确定性模拟来预测——这是帧同步的思想。
3. 当服务端权威状态和客户端预测不一致时，客户端回滚到服务端状态并重新模拟——这是 GGPO 回滚的思想。

所以 Overwatch 不是纯帧同步，也不是传统状态同步，而是两者的混合：

`服务器权威保证最终一致性，确定性模拟保证预测质量，回滚重播处理预测错误。`

这个混合模型之所以可行，是因为 Overwatch 的 ECS 架构天然满足了确定性模拟的前提：所有状态在 Component 里、System 执行顺序固定、没有对象级副作用。

如果你的技能系统是传统 OOP 架构，要达到同样的效果，需要自己保证这三个前提——这就是工程成本差异的来源。

## 这一篇真正想留下来的结论

帧同步下的技能系统，难点不在网络层，而在确定性保证。

浮点数不确定，伤害公式就会分叉。
随机数消费顺序不确定，暴击判定就会分叉。
逻辑帧和渲染帧不分离，Tick 次数就会分叉。
Buff 聚合顺序不确定，属性计算就会分叉。
碰撞检测不确定，命中判定就会分叉。

任何一处分叉，在帧同步下都是致命的——因为帧同步没有服务器来纠正差异。一帧的微小偏差会在后续每一帧被放大，最终导致两端的战斗结果完全不同。

GGPO 回滚重播可以改善手感（减少 input delay），但它不解决确定性问题——它假设确定性已经成立，在此基础上处理预测和回滚。

所以这一篇最短的结论可以记成：

`帧同步下的技能系统，第一优先级不是怎么传输入，而是怎么让技能执行链的每一层——浮点、随机、帧率、Buff 聚合、碰撞——在所有端产生完全相同的结果。确定性是帧同步的地基，地基不稳，上面什么模型都不成立。`
