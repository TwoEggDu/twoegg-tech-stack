## 问题：玩家为什么会被分到错误的服务器

某个 MMO 项目上线了跨服竞技场功能。系统设计了三台逻辑服，每台服务器处理独立的房间状态。部署后出现了一个诡异的 bug：同一局游戏的两个玩家，操作结果不一致——一人看到自己打出了暴击，另一人那边显示对方是普通攻击。

排查发现，两个玩家的请求在某些时刻被 L7 负载均衡的 Round-Robin 分配到了**不同的逻辑服节点**。同一个房间的战斗状态在节点 A 上，但玩家 B 的某次请求落到了节点 B，节点 B 上没有这个房间的状态，于是创建了一个空的状态对象并"成功"处理了请求。

这个问题的根源不是代码 bug，而是**用无状态服务的负载均衡策略去处理有状态服务**。

负载均衡在 Web 后端是一个相对简单的问题——随机分配就行，每台服务器的行为是幂等的。但在游戏后端，有状态的逻辑服、长连接的网关、需要强一致路由的房间系统，让负载均衡策略变得复杂。理解这套机制，才能避免上面这类"分到了错误服务器"的灾难性 bug。

## 抽象模型：负载均衡的三个层次

负载均衡按工作的网络层次划分为三类，每类解决不同级别的问题：

### DNS 负载均衡（L3/L4 之前）

DNS 层面的负载均衡最简单：域名解析返回多个 IP，客户端随机选择一个连接。

优点是零基础设施成本。缺点也很明显：
- DNS 有 TTL 缓存，节点下线后客户端仍可能连接失效 IP
- 无法感知服务器实际负载，只做简单的 IP 分发
- 客户端可能缓存 DNS 结果，导致流量倾斜

游戏后端很少单独依赖 DNS 负载均衡，通常只作为 CDN 和全球入口的分发手段。

### L4 负载均衡（传输层）

L4 负载均衡（如 LVS、HAProxy TCP 模式）工作在传输层，看到的是 TCP/UDP 包，不理解应用层协议。它根据源 IP + 目标 IP + 端口做转发决策，性能极高。

适合游戏后端的场景：
- 网关层的 TCP 连接分发（不需要理解 WebSocket 帧内容）
- UDP 游戏通信的分发（如 KCP、ENet 协议）
- 需要极低延迟的转发

缺点：无法基于应用层内容（如 player_id）做路由决策，更无法感知会话状态。

### L7 负载均衡（应用层）

L7 负载均衡（如 Nginx、Envoy）工作在应用层，可以读取 HTTP 头、WebSocket 握手包、自定义协议头。它可以基于内容做复杂路由：

- 根据 player_id header 路由到特定逻辑服
- 根据 URL 路径区分游戏服务和管理服务
- 检查 JWT token 并做简单鉴权

缺点：每个连接都需要解析到应用层，CPU 开销高于 L4，延迟也略高。

## 具体实现：负载均衡算法

### 轮询（Round Robin）

最简单的算法：请求依次分发给每台服务器，循环往复。适合无状态、请求处理时间均匀的场景。

**不适合游戏后端有状态服务**：如果玩家第一个请求落在服务器 A，第二个请求落在服务器 B，而游戏状态存在服务器 A，玩家就会"找不到自己"。

### 最少连接（Least Connections）

把新请求分给当前连接数最少的服务器。对长连接场景比轮询稍好，但同样无法解决有状态路由问题。

### 粘性会话（Sticky Session / Session Affinity）

**这是游戏后端长连接场景的核心需求。** 粘性会话保证同一个客户端的所有请求都路由到同一台服务器。

实现方式：
1. **基于 Cookie**（HTTP/HTTPS）：负载均衡在响应里种一个 Cookie（如 `server_id=A`），后续请求带着这个 Cookie，LB 识别后路由到 A。
2. **基于源 IP 哈希**：对客户端 IP 做哈希，相同 IP 永远路由到同一台服务器。简单，但客户端 IP 可能变化（移动网络），或多个客户端共享同一个 NAT IP。
3. **基于自定义 Header**：在 WebSocket 握手时，客户端携带 player_id 或 session_token，LB 解析后路由。这是游戏后端最精确的方式。

**粘性会话的陷阱**：当目标服务器宕机，绑定在该服务器上的所有会话都需要重建。这不是 bug，是游戏服务器的正常故障处理场景——需要配套的重连和状态恢复机制。

### 一致性哈希（Consistent Hashing）

一致性哈希是分布式系统中解决节点增减时"影响范围最小化"的经典算法。它把 key（如 room_id 或 player_id）映射到一个虚拟的哈希环上，每个节点负责环上的一段区间。

**关键特性**：当新增或删除一个节点时，只有该节点相邻范围内的 key 需要重新映射，其余 key 的路由不变。对比简单取模（`hash(key) % N`），增删节点时几乎所有 key 都需要重映射，这在有状态服务中是灾难性的。

游戏后端使用一致性哈希的典型场景：
- 分布式 Redis 集群中 key 到节点的路由
- 匹配服务中将房间 ID 路由到特定逻辑服节点
- 跨服通信中确定"玩家 X 的状态在哪个节点"

## 抽象模型：服务发现

负载均衡解决"如何分发请求"，服务发现解决"我要发给谁——目标服务在哪里"。

### 为什么不能写死 IP

在开发环境写死服务器 IP 是常见习惯，但在生产环境是灾难：
- 服务器故障需要手动更新配置并重启依赖方
- 自动扩缩容新起的实例 IP 不在配置文件里
- 金丝雀部署时新旧版本共存，路由规则无法动态调整

服务发现的核心模型：

```
服务实例启动
  → 向注册中心注册（IP + Port + 元数据）
  → 定期发送心跳

调用方
  → 查询注册中心："逻辑服"有哪些实例？
  → 获取实例列表
  → 通过负载均衡算法选择一个
  → 建立连接并调用

注册中心
  → 监控心跳
  → 心跳超时 → 从列表中剔除该实例
```

### Consul / etcd / K8s Service 解决了什么

**Consul**

Consul 是一个集成了服务注册、健康检查、KV 存储、DNS 接口的综合工具。

游戏后端常用 Consul 的理由：
- 支持多数据中心的服务发现（跨机房）
- 健康检查支持 HTTP、TCP、脚本三种方式，灵活
- DNS 接口让遗留系统不改代码就能使用服务发现
- Watch 机制支持服务列表变化时的实时通知

**etcd**

etcd 是一个强一致的分布式 KV 存储，Kubernetes 用它做整个集群的配置存储。游戏后端单独使用 etcd 做服务发现时，通常是因为已经依赖了 etcd（如 K8s 环境），利用其 Watch 机制实现服务列表的实时更新。

etcd 的优势在于**强一致性**（基于 Raft 协议），缺点是内置功能较少，健康检查需要自己实现。

**Kubernetes Service**

在 K8s 环境下，Service 对象是最自然的服务发现机制：
- ClusterIP Service 提供集群内虚拟 IP，kube-proxy 维护 iptables/ipvs 规则
- DNS 形式（`service-name.namespace.svc.cluster.local`）让服务间调用不依赖 IP
- EndpointSlice 实时追踪 Pod 的上下线状态

K8s Service 对无状态服务非常友好，但对游戏后端有状态路由的支持较弱——默认的 kube-proxy 做的是随机/轮询，不支持基于 player_id 的粘性路由。通常需要配合 Headless Service + 自定义客户端负载均衡来实现有状态路由。

## 游戏后端 vs 传统 Web：长连接导致的关键差异

| 维度 | 传统 Web 后端 | 游戏后端（长连接） |
|------|---------------|-------------------|
| 连接生命周期 | 毫秒级，请求完即关 | 分钟到小时级，持续维持 |
| 节点扩容 | 新节点立刻分担流量 | 新节点只接收新连接，存量连接不迁移 |
| 节点缩容 | 摘掉即可，当前请求最多等几秒 | 需要等所有连接断开 or 主动迁移会话 |
| 节点宕机 | 重启，用户重试即可 | 会话状态丢失，玩家直接断线 |
| 路由策略 | 无状态，随机分配即可 | 有状态，必须保证请求路由到正确节点 |
| 健康检查频率 | 几秒一次即可 | 需要更细粒度（心跳超时通常 30-60s）|

**长连接最大的负载均衡挑战**：连接一旦建立，流量就固定在某台机器上了。这意味着：

1. 新加入的节点短期内连接数很少，无法立刻分担压力（需要等新连接建立）
2. 老节点无法通过 LB 自然卸载连接，必须主动断连或等连接自然断开
3. 流量的"重新平衡"远比无状态服务慢

实际工程处理：对于网关层，通常不依赖 LB 做实时负载均衡，而是在客户端重连时选择连接数最少的网关节点（通过注册中心查询）。

## 具体实现：游戏后端的典型服务发现流程

以一个房间制游戏为例：

```
1. 玩家发起匹配请求
   → 网关收到，转发给匹配服
   
2. 匹配服查询服务注册中心
   → 获取所有逻辑服节点列表
   → 根据节点当前房间数选择最空闲节点
   → 在注册中心记录：房间 R 分配给逻辑服节点 N
   
3. 匹配完成，告知玩家
   → 玩家发送"加入房间 R"请求
   → 网关查询路由表：房间 R 在节点 N
   → 转发给节点 N
   
4. 逻辑服节点 N 宕机
   → 注册中心心跳超时，标记节点 N 下线
   → 网关收到节点 N 下线通知
   → 节点 N 上所有玩家连接断开
   → 玩家重连，匹配服检测到需要重新分配房间
```

健康检查的设置是关键细节：太频繁（<1s）会给注册中心造成压力；太稀疏（>30s）会导致宕机后很久才被发现，期间请求持续失败。游戏后端通常设置 5-10s 心跳，3 次失败视为下线。

## 工程边界：负载均衡不能解决的问题

### 脑裂（Split Brain）

如果注册中心本身出现网络分区，一部分节点认为某服务在线，另一部分认为它下线，路由决策就会分裂。这是一个分布式一致性问题，需要注册中心本身有强一致性保障（如 etcd 的 Raft），不是配置更多负载均衡能解决的。

### 热点问题（Hot Spot）

一致性哈希减少了节点增减时的迁移量，但无法消除热点。如果某个 room_id 的哈希值集中在同一个节点，该节点会比其他节点承受更大压力。解决方案是虚拟节点（Virtual Nodes）——每个物理节点在哈希环上有多个虚拟位置，分散热点。

### 有状态服务的故障恢复

负载均衡能检测到节点宕机并停止路由，但无法恢复宕机节点上丢失的内存状态。游戏后端需要在逻辑层设计 checkpoint（定期把内存状态快照到 Redis 或 DB），负载均衡只负责"别把请求发给挂掉的节点"。

---

## 最短结论

游戏后端的负载均衡核心难点不是"如何分发请求"，而是"如何保证有状态的请求永远路由到正确的节点"——长连接的粘性会话和一致性哈希是解决这个问题的两把钥匙，服务发现是让这套机制在节点动态变化时仍然工作的基础设施。
