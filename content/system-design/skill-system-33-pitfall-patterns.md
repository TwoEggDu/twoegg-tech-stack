> 一句总判断：技能系统的 bug 有规律。绝大多数线上问题可以归进这 7 类，知道类型就知道该先查哪一层。

做技能系统的人迟早会意识到一件事：

大多数 bug 不是凭空出现的新问题，而是同一类问题换了个皮再来一次。

第一次遇到"Buff 叠层炸了"，你可能花两天才定位到 Modifier 聚合顺序。第二次遇到类似症状，如果你还是从头排查，说明排障流程没有沉淀下来。

坑型录的价值不在于"记录了哪些坑"，而在于"下次再遇到时能直接走对路径"。

这篇做的事情很简单：

把技能系统里最常见的 7 类坑，按**信号 -> 根因 -> 排查步骤 -> 防护建议**的结构整理出来。目标不是给每种 bug 写一份修复教程，而是让你看到某个现象时，能在 30 秒内判断它属于哪一类，然后直接走对应的排查路径。

## 这篇要回答什么

这篇主要回答 4 个问题：

1. 技能系统线上问题最常归进哪 7 类。
2. 每一类最典型的信号长什么样。
3. 每一类最稳的排查顺序是什么。
4. 每一类在架构层面到底是怎么埋下去的，以及怎么提前防住。

这 7 类覆盖了从属性计算到网络同步、从配置管理到跨系统时序的完整面。它们不是按技术栈分的，而是按**你在战斗日志里看到的症状**分的。

在正式开始之前先说明一个前提：这里的"坑型"不是按严重程度排的，而是按**你在日常排障中遇到它们的频率**排的。前三类（叠加溢出、快照错误、命中重复）几乎占了所有战斗数值类 bug 的大头。后四类（生命周期泄漏、预测回滚、配置断裂、时序竞争）更偏向系统级问题，出现频率低一些但排查成本高得多。

## 第一类：Buff 叠加溢出

### 典型信号

1. 角色伤害突然暴增到不合理的数值，或者变成无敌状态。
2. 某个 Buff 的叠层数显示异常：99 层、-1 层、或者一个明显不该出现的数字。
3. 重复施加同一个 Buff 后数值不符合预期，比如叠了 3 层但伤害只算了 1 层的量。
4. 同一个 Buff 在不同角色身上的叠加行为不一致。
5. Buff 被移除后属性没有完全还原。

### 这类坑的核心难点

叠加规则（替换、追加、刷新、上限）没有在一个地方统一定义，导致同一个 Buff 在不同调用路径下走了不同的叠加逻辑。

### 最稳排查步骤

**第一步：打印 Buff 容器的当前叠层数和 Modifier 列表。**

不要只看 UI 上显示的层数。直接从运行时的 BuffContainer 里拿当前的 `stackCount` 和挂在这个 Buff 上的所有 Modifier 实例。大部分溢出问题到这一步就已经能看出端倪，因为你会发现叠层数和 Modifier 数量不匹配。

**第二步：确认 Buff 的 StackPolicy 配置。**

检查这个 Buff 的配置表里 StackPolicy 字段写的是 Replace、Stack 还是 Refresh。然后跟实际运行时的行为对比。常见的情况是：配置里写了 Stack，但代码里某条路径绕过了叠加检查直接 Add 了一个新的 Modifier 实例。

**第三步：检查 Modifier 聚合是否因叠层数放大。**

如果 Modifier 的数值是 `baseValue * stackCount`，而 stackCount 没有上限，那聚合结果会随叠层数线性增长。更隐蔽的情况是百分比 Modifier 的叠加：10% 叠 10 层不是 100%，而是 `1.1^10 = 2.59`，这取决于你的聚合方式是加法还是乘法。

**第四步：检查是否缺少 MaxStack 上限。**

翻配置表看这个 Buff 有没有配 MaxStack。如果没配，再看代码里 BuffContainer 的 AddStack 方法有没有硬编码的兜底上限。很多项目在早期开发时不设上限，因为"策划说这个 Buff 最多叠 3 层"，但代码没有强制执行这个约束。

### 根因解释

Buff 叠加溢出的根因不是"缺一个上限检查"这么简单。真正的问题是 Modifier 的聚合顺序依赖了 Buff 的添加顺序，而不是优先级。

当一个 Buff 叠加时，新的 Modifier 被追加到聚合链的末尾。如果聚合链是按顺序计算的（先加后乘、先 base 后 percent），那叠加顺序会影响最终结果。在高频叠加场景下（比如 DOT 类 Buff 每秒刷新一次），添加和移除的时序混乱会导致聚合链里出现"幽灵 Modifier"：Buff 已经被标记移除，但它的 Modifier 还留在聚合链里参与计算。

另一个常见的触发条件是"Buff 替换"逻辑写错了。配置上说 StackPolicy 是 Replace（同类 Buff 只保留最新的一个），但代码实现时先 Add 后 Remove，而不是先 Remove 后 Add。在 Add 和 Remove 之间的那个 Tick 里，两个 Buff 同时存在，两组 Modifier 同时生效。如果这个 Tick 恰好触发了一次伤害计算，结果就是双倍增益。

### 防护建议

- 在 BuffDef 配置层强制要求 StackPolicy 和 MaxStack 字段，缺了就过不了配置校验。
- Modifier 聚合后做一次 Clamp，确保最终值不会超过策划定义的属性上下限。
- 叠加和移除走同一个入口，不允许直接操作 Modifier 列表。
- 对百分比类 Modifier 的叠加方式做显式声明（加法叠加还是乘法叠加），不要留给代码去猜。
- 写一个 Debug 命令，可以在运行时 dump 指定角色身上所有 Buff 的叠层数、剩余时间和 Modifier 明细，方便线上排查。

## 第二类：快照时机错误

### 典型信号

1. Buff 已经消失了但伤害还在生效，表现为"幽灵伤害"。
2. 加了攻击力 Buff 后立刻放技能但伤害没变，要等下一次才生效。
3. 同一个技能对不同目标的伤害不一致，在不该不一致的情况下。
4. DOT 效果的每跳伤害会随角色属性实时变化，策划说它不应该变。
5. 切换武器后技能伤害仍然按旧武器计算。

### 这类坑的核心难点

快照应该在什么时间点采集属性值，没有统一约定。有的 Effect 在 Apply 时快照，有的在 Execute 时快照，有的 DOT 每跳都重新采集，但这些策略没有被显式声明出来。

### 最稳排查步骤

**第一步：确认 Effect 的快照策略。**

从 EffectDef 配置里找到这个 Effect 的 SnapshotPolicy 字段。它应该是 OnApply、OnExecute 或 Periodic 三者之一。如果配置里没有这个字段，那说明快照策略是代码里隐式决定的，这本身就是坑的来源。

**第二步：打印快照时刻的 Source Attribute 值。**

在 Effect 的快照点加日志，打印 Caster 当前的所有相关属性（攻击力、暴击率、技能增伤等）。然后和你期望的值对比。

**第三步：对比快照值和当前 Attribute 值。**

如果 Effect 是 OnApply 快照的，那在 Effect 实际造成伤害的时刻，Caster 的属性可能已经变了。把这两个时间点的属性值拉出来对比，看差异在哪个属性上。

**第四步：检查快照之后是否有 Modifier 又修改了同一个属性。**

这是最隐蔽的情况。快照发生在 T1 时刻，但在 T1 之后、Effect 实际结算之前，另一个 Buff 的 Modifier 修改了被快照的属性。如果快照存的是引用而不是值拷贝，那快照数据会被"污染"。

### 根因解释

快照问题的根因是：快照 Modifier 和实时 Modifier 混在了同一个聚合链里，没有做区分。

当一个 Effect 声称自己是"快照型"时，它应该在快照时刻把需要的属性值拷贝出来，后续计算只用这份拷贝。但如果属性系统没有提供"对某个查询冻结数值"的能力，快照实现通常会退化为"在 Apply 时读一次当前值存到 Effect 实例上"。这个做法在大多数情况下可以工作，但它有一个前提：Apply 时刻和实际结算时刻之间，不会有其他逻辑修改被快照的属性。在有 Buff 互相触发的场景下，这个前提很容易被打破。

### 防护建议

- 在 EffectDef 层强制声明 SnapshotPolicy，不允许留空。
- 快照时做值拷贝，不要存属性引用。
- 在战斗日志里记录每次快照的时间戳和采集值，方便回溯。
- 如果项目里 DOT 类 Effect 数量多，考虑提供"每跳重算"和"首跳快照"两种模板，让策划显式选择。
- 在编辑器的技能预览面板里标注每个 Effect 的快照策略，策划能直观看到"这个伤害用的是哪一刻的攻击力"。

快照问题在 MOBA 和 MMO 里尤其常见。因为这两个品类的 Buff 更新频率高、属性变化剧烈，快照时机差几帧就可能差一个 Buff 的增益。如果你的项目属于这两个品类，建议在项目早期就把快照策略文档化。

## 第三类：命中重复计算

### 典型信号

1. 一次攻击打出了双倍甚至三倍伤害。
2. 范围技能对同一个目标触发了多次 Effect Apply。
3. 投射物穿过目标时 OnHit 触发了多次。
4. 链式闪电跳到同一个目标两次。
5. 战斗日志里同一帧出现了同一对 source-target 的多条伤害记录。

### 这类坑的核心难点

碰撞回调和命中判定的去重逻辑不在同一个地方。碰撞是物理引擎管的，命中判定是技能系统管的，两边各自觉得自己没问题。

### 最稳排查步骤

**第一步：检查碰撞回调是否有 hitSet 去重。**

在投射物或范围触发器的碰撞回调里，检查是否维护了一个 `HashSet<EntityId>`。每次触发时先检查目标是否已经在 set 里。如果没有这个 set，那重复命中几乎是必然的。

**第二步：检查投射物是否在同一帧触发了多次 OnTriggerEnter。**

Unity 的 `OnTriggerEnter` 在 Collider 形状重叠的情况下，可能在同一个 FixedUpdate 周期内对同一对碰撞体触发多次回调。用日志打印每次回调的帧号和碰撞对信息，看是否有同帧重复。特别注意复合碰撞体（一个 GameObject 上挂了多个 Collider）的情况，每个 Collider 会独立触发回调。

**第三步：检查范围检测的实现方式。**

`Physics.OverlapSphere` 返回的是某一时刻所有重叠的 Collider，如果你在多帧连续调用它（比如范围 DOT），同一个目标会在每帧都出现在结果里。确认你是在每次检测后做了去重，还是在 Effect 的 Apply 层做了幂等保护。

**第四步：检查 Effect 的 Apply 是否有幂等保护。**

即使碰撞回调层面做了去重，Effect Apply 本身也应该有幂等检查。特别是在 Effect 有"每次命中叠加一层"逻辑的情况下，如果 Apply 没有幂等保护，重复回调会直接导致叠层数翻倍。

### 根因解释

物理引擎的碰撞回调不保证只触发一次。这是由物理引擎的工作方式决定的。

在 Unity 里，`OnTriggerEnter` 和 `OnTriggerStay` 的触发时机取决于 FixedUpdate 的步进频率和 Collider 的运动速度。当一个投射物以高速穿过一个 Collider 时，根据插值方式不同，可能在进入和穿出的过程中触发多次。当范围技能使用持续性 Trigger 时，`OnTriggerStay` 每个 FixedUpdate 都会触发。如果命中判定直接绑定在这些回调上，而没有在技能系统层面做一次"这次 Cast 已经命中过这个目标了"的检查，重复计算就是必然的。

### 防护建议

- 每次 Cast 创建一个 hitSet，在整个 Cast 生命周期内去重。投射物、范围检测、链式效果都共用这个 set。
- Effect Apply 层加幂等检查：同一个 CastId + TargetId 组合只 Apply 一次。
- 碰撞检测和命中判定做分层：碰撞回调只负责报告"谁碰到了谁"，命中判定在技能系统层面统一处理。
- 对穿透型投射物，用 hitSet + 穿透计数器配合，而不是依赖 OnTriggerExit 来"重置"命中状态。
- 在 Debug 模式下对每次 Effect Apply 记录 CastId + TargetId + FrameCount，如果同一组合在同一帧出现两次，直接报警。

这一类坑在 ARPG 和弹幕类游戏里出现频率最高，因为这些品类的碰撞体密度大、帧间移动距离长。如果你的投射物速度经常超过自身碰撞体长度的 2 倍以上，建议在物理层面就开启 CCD（Continuous Collision Detection），而不是在技能层面做补偿。

## 第四类：生命周期泄漏

### 典型信号

1. 角色已经死亡但 VFX 还在播放，或者持续对已死亡角色造成伤害。
2. 技能被打断了但 Buff 没有被移除，角色带着不该有的属性加成继续战斗。
3. 定时器在技能结束后仍然触发，导致异常的周期性事件。
4. 长时间战斗后内存持续增长，Profiler 里 Effect / Timer / VFX 对象计数只增不减。
5. 切换场景后战斗音效还在播放。

### 这类坑的核心难点

技能中断时的清理路径不完整。正常结束走 OnEnd，被打断走 OnCancel，角色死亡走 OnOwnerDeath，但这三条路径清理的东西不一样。

### 最稳排查步骤

**第一步：检查 Ability 的 OnEnd / OnCancel 是否清理了所有挂载物。**

列出这个 Ability 在执行过程中创建的所有东西：Buff、Effect、Timer、VFX、音效、投射物。然后逐一检查 OnEnd 和 OnCancel 里是否都有对应的清理逻辑。很多时候 OnEnd 写得很完整，但 OnCancel 只清理了一部分。

**第二步：检查 Buff 是否注册了 OnOwnerDeath 回调。**

角色死亡是一个特殊的中断路径。如果 Buff 没有监听 Owner 的死亡事件，那角色死亡后 Buff 的定时器可能继续运行，持续触发效果，甚至尝试对一个已销毁的 Entity 执行操作。

**第三步：检查 Timer / Coroutine 是否在 OnDestroy 中取消。**

如果技能系统使用协程或定时器来驱动周期性效果，那在宿主对象销毁时，这些协程和定时器必须被显式取消。Unity 的协程在 MonoBehaviour 销毁时会自动停止，但自定义的 Timer 系统不会。

**第四步：用 Profiler 对比战斗前后的对象计数。**

在战斗开始前和结束后分别做一次快照，比较 Effect、Buff、Timer、VFX 这几类对象的实例数量。如果战斗结束后数量没有回到战斗前的水平，就有泄漏。

### 根因解释

生命周期泄漏的根因是：正常结束和异常中断走了不同的清理路径，而异常中断的路径不完整。

这个问题在架构层面是这样产生的：开发技能时，第一版代码通常只考虑了正常流程。OnEnd 里把该清理的都清理了。然后策划要求支持技能打断，于是加了 OnCancel。但 OnCancel 通常是后加的，写的人不一定知道 OnEnd 里清理了哪些东西，所以很容易漏。再后来角色死亡需要清理所有进行中的技能，又加了一条新的中断路径。三条路径各自维护各自的清理列表，越往后越不同步。

还有一类更隐蔽的泄漏来源：事件订阅。Buff 在 Apply 时订阅了 Owner 的某个事件（比如 OnDamageTaken），但在移除时没有取消订阅。Buff 实例本身可能已经不在 BuffContainer 里了，但因为事件系统还持有它的引用，GC 无法回收它，它的回调还会继续被触发。这种泄漏不会在对象计数里体现出来（Buff 没有被重复创建），而是表现为"某个回调被触发了但不知道是谁注册的"。

### 防护建议

- 统一清理入口：无论是正常结束、被打断还是 Owner 死亡，都最终走同一个 Cleanup 方法。OnEnd / OnCancel / OnOwnerDeath 只负责设置中断原因，不负责具体清理。
- 用"资源登记表"模式：Ability 每创建一个 Buff / Effect / Timer / VFX，都登记到一个列表里。Cleanup 时遍历这个列表逐一释放。
- 定期做泄漏检测：在战斗结束后检查是否有属于该战斗的对象仍然存活，在 Debug 模式下直接报错。
- 对长生命周期的 Buff（比如装备被动），单独管理，不走技能的 Cleanup 路径。

泄漏问题有一个常见的"温水煮青蛙"效应：单次战斗泄漏的对象可能只有几个，不影响性能。但如果玩家持续战斗半小时不退出，泄漏对象累积到几千个的时候才会看到帧率下降或内存告警。到那时候再排查，已经很难定位到是哪一场战斗、哪一个技能泄漏的了。所以泄漏检测应该在每场战斗结束时就做，而不是等到 Profiler 报警。

## 第五类：预测回滚撕裂

### 典型信号

1. 技能释放后角色位置"跳了一下"，像是瞬移了一小段距离然后又回来。
2. 血条先掉再回再掉，出现明显的视觉抖动。
3. VFX 播放到一半突然消失，或者同一个特效播了两次。
4. 客户端和服务端的技能结果不一致，客户端显示命中但服务端判定未命中。
5. 高延迟环境下技能表现完全混乱，低延迟环境下正常。

### 这类坑的核心难点

预测性变更和服务端确认之间的表现层没有做平滑过渡。客户端为了手感先预测执行了技能效果，但服务端拒绝后回滚时，表现层直接硬切，玩家看到的就是"闪了一下"。

### 最稳排查步骤

**第一步：确认是否是预测被拒绝导致的回滚。**

在网络层的日志里搜索这次技能释放的 CastId，看服务端是否返回了拒绝或修正消息。如果确认是回滚，记录客户端预测的结果和服务端确认的结果之间的差异。

**第二步：检查预测窗口是否过大。**

测量从客户端发出 Cast 请求到收到服务端确认之间的时间差。如果这个窗口超过了 200ms，客户端可能已经走了太远：伤害数字显示了、VFX 播放了、血条变了。回滚的代价越大，撕裂感越明显。

**第三步：检查 VFX / 动画是否绑定了预测性状态。**

如果 VFX 的播放直接绑定在客户端预测的"命中"事件上，那回滚时 VFX 就会突然消失。检查表现层是否区分了"预测性播放"和"确认性播放"。

**第四步：检查回滚时是否有 Lerp / 平滑过渡而非硬切。**

当服务端修正了角色位置或血量时，客户端是直接设置新值（硬切），还是从当前值 Lerp 到目标值（平滑过渡）。硬切在逻辑上是正确的，但视觉上是灾难。

### 根因解释

预测回滚撕裂的根因是：表现层直接绑定了预测状态，回滚时无法优雅撤回。

在客户端预测模型里，预测发生在逻辑层。客户端在发出请求后立刻在本地执行技能逻辑，包括伤害计算、状态变更、Buff 施加等。表现层（VFX、动画、音效、UI）监听逻辑事件来做渲染。问题出在：如果表现层监听的是"逻辑状态变了"这个事件，那预测时触发一次，回滚时逻辑状态改回去又触发一次，确认时再触发一次。表现层会看到三次事件，但只有最后一次是对的。

更隐蔽的情况是：预测和确认的结果一致（服务端没有拒绝），但因为网络延迟导致确认消息到达时，客户端已经在本地执行了第二帧的逻辑。确认消息触发了一次"确认性"事件，叠加在本地已有的状态上，导致效果被应用了两次。这种"确认反而导致重复"的 bug 比预测被拒绝更难发现。

### 防护建议

- 表现层不监听预测事件，只监听确认事件。预测窗口内用"预估动画"（一个较轻的前摇或提示特效）填充，确认后再播放完整效果。
- VFX 和逻辑解耦：VFX 有自己的播放状态机，不直接绑定战斗逻辑的属性变更。
- 位置和血量的回滚用 Lerp 过渡，持续时间可以根据网络延迟动态调整。
- 对无法撤回的表现（比如已经播完的音效），不做回滚处理，只在确认后补偿差异。
- 在高延迟环境下做自动化测试，专门验证预测回滚路径。很多预测回滚的 bug 在局域网环境下永远不会复现，因为延迟太低，预测窗口太短。

值得一提的是，不是所有项目都需要客户端预测。如果你的项目是 PVE 为主、对操作延迟不敏感，服务端权威 + 客户端纯播表现可能比做预测回滚更稳。预测回滚的工程复杂度远高于它表面看起来的样子。

## 第六类：配置依赖断裂

### 典型信号

1. "这个技能昨天还好的今天就 crash 了"，代码没改，只改了配置。
2. NullReferenceException 在 Effect Apply 时抛出，堆栈指向一个看起来没问题的 Effect。
3. 改了一个 Effect 的 ID 后多个技能同时失效。
4. 新建了一个 Buff 但忘了把它注册到某个全局表里，运行时查不到。
5. 策划复制了一个技能配置做修改，忘了更新里面引用的 Effect ID，导致新技能执行的是旧 Effect。

### 这类坑的核心难点

技能、Effect、Buff 之间的引用关系没有显式的依赖图。你不知道改了一个 Effect 的 ID 会影响哪些技能，也不知道删了一个 Buff 定义会不会让某个 Effect 在 Apply 时找不到目标。

### 最稳排查步骤

**第一步：检查 SkillDef 引用的 EffectSpec ID 是否仍然存在。**

从报错的 Skill 开始，拿到它配置里引用的所有 EffectSpec ID，然后逐一检查这些 ID 在 Effect 配置表里是否还存在。如果某个 ID 被改名或删除了，就是断裂点。

**第二步：检查 Effect 引用的 Buff ID 是否匹配。**

同样的检查往下做一层：Effect 配置里引用的 Buff ID，在 Buff 配置表里是否存在。Buff 配置里引用的 Modifier 模板 ID 是否存在。一层层往下查，直到找到断裂点。

**第三步：用依赖图工具可视化引用链。**

如果项目里有配置依赖可视化工具，直接生成这个 Skill 的完整引用链。如果没有这个工具，手动画一张：Skill -> Effect -> Buff -> Modifier，标出每一层的 ID 引用关系。断裂点在可视化后通常一目了然。

**第四步：检查最近的配置变更记录。**

在版本控制里看最近 24 小时内配置表的变更记录。重点关注 ID 重命名、条目删除和新增。配置依赖断裂几乎总是由"改了一个地方但没有更新引用它的其他地方"引起的。

**第五步：检查是否有"隐式依赖"。**

有些引用关系不在配置表里，而是在代码里硬编码的。比如某个 Effect 的逻辑代码里写了 `BuffManager.AddBuff("buff_burning")`，这个字符串 ID 不会出现在 EffectDef 的配置里，所以配置校验器也扫不到。遇到配置校验通过但运行时仍然断裂的情况，就要往这个方向查。

### 根因解释

配置依赖断裂的根因是：配置之间的引用是字符串 ID，没有编译期检查。

在代码世界里，你重命名一个类或方法，编译器会立刻告诉你哪些地方引用了旧名字。但在配置世界里，Skill 引用 Effect 用的是字符串 ID，Effect 引用 Buff 也是字符串 ID。这些引用关系只有在运行时才会被解析，没有任何静态检查机制。更糟糕的是，策划通常在 Excel 或编辑器里改配置，而不是在 IDE 里。他们没有"全局搜索引用"的习惯，也没有工具支持这个操作。

### 防护建议

- 在 CI 阶段跑配置校验器：检查所有 ID 引用是否完整、是否有孤立的配置条目、是否有循环引用。
- 配置编辑器里加引用检查：当策划试图删除或重命名一个 ID 时，弹出"以下 N 个配置引用了这个 ID"的警告。
- 建立依赖图可视化工具：不需要很复杂，能从一个 Skill ID 开始画出完整的引用链就够了。
- 配置表做版本校验：每次加载时检查配置版本号和运行时依赖表是否匹配。

配置依赖断裂有一个显著的组织层面特征：它几乎总是在"多人同时改配置"的时候出现。一个策划改了 Effect ID，另一个策划不知道，继续用旧 ID 创建新技能。如果配置表是 Excel 或 JSON 散文件，Git 合并时冲突不会体现在引用关系上。这类问题仅靠代码审查是防不住的，必须靠工具。

## 第七类：跨系统时序竞争

### 典型信号

1. 命中音效有时在伤害数字之前播放，有时在之后，顺序不稳定。
2. 动画事件和逻辑事件的执行顺序每次运行不一样。
3. 编辑器里时序正常但真机上不对，或者高帧率正常但低帧率出问题。
4. 帧率低时技能行为不同：某个效果被跳过了，或者触发了两次。
5. 两个同时释放的技能偶尔产生不同的结算顺序，导致结果不确定。

### 这类坑的核心难点

动画事件、物理回调、逻辑 Tick 的执行顺序在不同帧率下不确定。Unity 的 Update、LateUpdate、FixedUpdate 和 AnimationEvent 各自有自己的调度节奏，当帧率波动时，它们之间的相对顺序会变。

### 最稳排查步骤

**第一步：在关键时序点加时间戳日志。**

在技能执行链的每个关键节点（Cast 开始、动画事件触发、碰撞检测、Effect Apply、VFX 播放）加上 `Time.frameCount` 和 `Time.time` 的日志。跑几次，对比这些节点的帧号和时间戳是否稳定。

**第二步：确认 AnimationEvent 是在 Update 还是 LateUpdate 触发。**

Unity 的 AnimationEvent 默认在 Animator 的 Update 阶段触发，但具体时机取决于 Animator 的 UpdateMode 设置（Normal / Animate Physics / Unscaled Time）。如果 Animator 设成了 Animate Physics，AnimationEvent 会在 FixedUpdate 触发，这和你在 Update 里做的逻辑就不在同一个调度周期了。

**第三步：确认物理回调是在 FixedUpdate 还是 Update 触发。**

`OnTriggerEnter`、`OnCollisionEnter` 这些回调在 FixedUpdate 周期触发。如果你的技能逻辑在 Update 里跑，那物理回调和技能逻辑之间就有一帧到多帧的延迟，具体取决于当前帧率和 FixedUpdate 的频率。

**第四步：检查是否依赖了 Time.deltaTime 的精度。**

如果技能的持续时间、冷却时间或间隔时间用 `Time.deltaTime` 累加来判断，那在帧率波动时精度会下降。60fps 时 deltaTime 是 16ms，30fps 时是 33ms。如果判断条件是 `elapsed >= 0.5f`，在 30fps 下可能跳过 0.5 直接到 0.533。

**第五步：在不同帧率下复现。**

用 `Application.targetFrameRate` 把帧率锁到 15、30、60 分别跑一次，看问题是否只在特定帧率下出现。如果是，那几乎可以确定是时序竞争问题。

### 根因解释

跨系统时序竞争的根因是：Unity 的 Update / LateUpdate / FixedUpdate / AnimationEvent 执行顺序在帧率波动时不确定。

Unity 的主循环在每一帧会按固定顺序执行：FixedUpdate（可能执行 0 次或多次）-> Update -> LateUpdate。但 FixedUpdate 的执行次数取决于上一帧花了多少时间。如果上一帧很慢（比如 GC 卡了一下），这一帧可能连续执行多次 FixedUpdate 来补偿。这意味着物理回调和逻辑更新之间的相对时序不是固定的。AnimationEvent 更复杂：它的触发取决于动画的播放进度，而播放进度又取决于 deltaTime。如果动画在两帧之间跨过了事件点，事件会被"追赶"触发，但触发的上下文可能和你预期的不一样。

举一个具体的例子：一个攻击动画在第 0.3 秒有一个 hitEvent，在第 0.5 秒有一个 endEvent。如果某一帧的 deltaTime 是 0.25 秒（4fps 卡顿），动画从 0.2 秒直接跳到 0.45 秒，hitEvent 被追赶触发了，但 endEvent 还没到。下一帧 deltaTime 恢复正常（0.016 秒），动画走到 0.466 秒，还是没到 endEvent。再下一帧才触发 endEvent。结果就是 hitEvent 和 endEvent 之间的间隔变得不稳定。如果你的技能逻辑假设"hitEvent 之后 X 毫秒内一定会收到 endEvent"，这个假设在低帧率下就不成立了。

### 防护建议

- 关键时序用逻辑帧号而非实际时间。定义一个全局的 LogicFrame 计数器，所有技能时序判断都基于这个帧号，不基于 `Time.time` 或 `Time.deltaTime`。
- AnimationEvent 只做标记，不做执行。AnimationEvent 触发时只设置一个标志位（比如 `hitFrameReached = true`），真正的逻辑在技能系统的统一 Tick 里检查这个标志位后执行。
- 把物理检测的结果缓冲到下一个逻辑 Tick 再处理，而不是在物理回调里直接执行技能逻辑。
- 对帧率敏感的逻辑做帧率无关设计：用固定步长模拟代替 deltaTime 累加。

时序竞争问题有一个让人崩溃的特点：它在开发机上几乎不复现。开发机通常是高配 PC，帧率稳定在 60fps 以上，FixedUpdate 和 Update 的执行次数比例非常稳定。但到了低端手机上，帧率掉到 20-30fps，FixedUpdate 可能一帧执行两三次，AnimationEvent 的触发时机就开始漂移了。所以这一类问题的测试，必须在目标机型上跑，或者至少在开发机上把帧率锁到 20fps 跑一轮。

## 怎么用这张表

在讲总表之前，先说一下使用方式。

这 7 类坑不是互斥的。一个线上 bug 完全可能同时涉及两类甚至三类。比如"角色释放技能后死亡，复活后发现攻击力异常高"，这里面既有生命周期泄漏（死亡后 Buff 没清理），又有 Buff 叠加溢出（复活后重新叠加导致叠层数异常）。

遇到这种复合型问题时，建议先按最明显的信号分类，走那一类的排查步骤。如果走到一半发现问题不在这一层，再切到另一类的排查路径。不要试图同时排查两类问题，那样只会让日志更混乱。

另外，这张表的"最先该查"列是给你省时间用的。它不是完整的排查流程，而是告诉你：在打开代码之前，先看哪一个数据点最有可能直接揭示问题。很多时候一个战斗 bug 的排查时间从"两天"缩短到"两小时"，不是因为你技术更好了，而是因为你第一步就查对了地方。

## 这篇真正想留下来的结论

技能系统的 bug 不是随机事件。

当你排过足够多的问题之后，你会发现绝大多数线上事故都可以归进上面这 7 类。知道类型，就知道该先查哪一层、该打什么日志、该问策划什么问题。

7 类坑的共性是一样的：**它们都不是单点问题，而是系统间交互的边界问题。**

Buff 叠加溢出是配置层和运行时聚合层的边界没对齐。快照时机错误是 Effect 和属性系统的时间约定没统一。命中重复是物理引擎和技能系统的职责没划清。生命周期泄漏是正常路径和异常路径的清理没合并。预测回滚是逻辑层和表现层的绑定方式没做缓冲。配置依赖是静态配置和运行时解析的校验没前置。时序竞争是 Unity 多个调度阶段和技能逻辑之间的时序没收敛。

解决这些问题的思路也是一样的：不是在出问题的地方加一个 if 判断，而是在架构层面把边界显式化。

该统一的入口统一，该显式声明的策略显式声明，该前置的校验前置到 CI，该解耦的表现和逻辑解耦。

最后一个建议：把这 7 类坑做成团队内部的 checklist。每次有人报了一个战斗 bug，先花 30 秒对着这个表判断它属于哪一类，再开始排查。时间长了你会发现，不是 bug 变少了，而是排查时间从"每次两天"变成了"每次两小时"。

如果你正在搭建或维护一套技能系统，建议把这 7 类坑和你们的战斗日志对一遍。大概率你会发现，最近三个月排的 bug 里，有超过一半可以直接对号入座。

坑型录不是写完就结束的东西。随着项目推进，你可能会发现第八类、第九类坑型。但只要你保持"先分类再排查"的习惯，新坑型也会自然沉淀进这个框架里。重要的不是这里列了 7 类还是 10 类，而是你的团队有没有一个共享的、结构化的排障起点。
