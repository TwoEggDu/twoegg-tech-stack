## 问题空间：会话不是一个变量，是一台状态机

在不少游戏后端的早期实现里，"游戏房间"只是数据库里的一行记录，用一个 `status` 字段标记当前状态。开发者在各个业务逻辑里直接读写这个字段，用 `if status == "playing"` 来做分支判断。

这种做法在玩家数量少、并发低的时候能跑通。但一旦规模扩大，问题就开始涌现：

- 结算逻辑和玩家离开逻辑同时触发，两者都尝试写入结果，数据不一致
- 玩家在结算阶段还能发送游戏操作包，服务端处理了一个无效指令
- DS 进程崩溃后，房间记录停留在"游戏中"状态，永远无法被清理
- 匹配系统把玩家分配到一个已经满员的房间

这些问题的根源是同一个：**没有把会话当作状态机来设计**。状态机的价值不在于"我知道当前是什么状态"，而在于"我知道哪些操作在当前状态下是合法的，哪些是非法的"——以及状态转换必须经过哪些验证。

---

## 抽象模型：五阶段生命周期

一个多人游戏会话的完整生命周期可以划分为五个阶段，每个阶段有明确的进入条件、合法操作集合和退出触发器：

```
[等待玩家 WAITING]
       ↓ 玩家数量达到要求 / 房主强制开始
[游戏中 IN_GAME]
       ↓ 游戏结束条件满足（时间/分数/对象消灭）
[结算中 SETTLING]
       ↓ 结算数据写入完成
[关闭中 CLOSING]
       ↓ 所有资源释放完毕
[已销毁 DESTROYED]
```

每个阶段对应一组明确的状态约束：

| 阶段 | 允许玩家加入 | 允许游戏操作 | 允许主动离开 | DS 可被回收 |
|------|------------|------------|------------|------------|
| WAITING | 是 | 否 | 是 | 否 |
| IN_GAME | 受限（重连） | 是 | 是（断线处理） | 否 |
| SETTLING | 否 | 否 | 否（等待结算完成） | 否 |
| CLOSING | 否 | 否 | 否 | 否（等待写入完成） |
| DESTROYED | 否 | 否 | 否 | 是 |

任何绕过这个状态机的操作都是潜在的数据一致性漏洞。

## 阶段一：房间创建

### 触发方：匹配系统 vs 玩家主动创建

房间的创建有两种模式，它们的责任归属不同：

**匹配系统创建**（常见于竞技类游戏）：匹配系统完成组队后，向 DS 编排层申请一个可用 DS，将 DS 的地址写入会话记录，再将加入凭证下发给所有玩家。这种模式下，玩家无法选择进入哪个 DS，匹配系统是唯一的创建发起方。

**玩家主动创建**（常见于派对类游戏、自定义房间）：玩家主动触发建房请求，后端分配 DS 并返回房间 ID 和连接地址。这种模式需要额外处理"房主离开后房间怎么办"的问题。

两种模式的共同点是：**房间进入 WAITING 状态之前，DS 进程必须已经启动并完成初始化**。如果 DS 还未就绪就把地址下发给玩家，会导致客户端连接失败的大量报错。

### 创建时的初始化数据

会话创建时，后端需要确定并持久化以下信息：

```json
{
  "session_id": "sess_abc123",
  "ds_host": "10.0.1.5",
  "ds_port": 7777,
  "map": "desert_canyon",
  "mode": "tdm",
  "max_players": 10,
  "current_players": 0,
  "status": "WAITING",
  "created_at": 1712345678,
  "expected_players": ["player_001", "player_002", "..."]
}
```

`expected_players` 列表在竞技模式中尤为重要——它是后续玩家加入验证的白名单。

## 阶段二：玩家加入

玩家加入不是一个简单的"建立连接"操作，而是一个多步验证流程。

### 验证流程

```
客户端发送加入请求（含 Token）
          │
          ▼
    Token 验证
    ├─ 是否有效（签名校验）
    ├─ 是否过期（时间戳检查）
    └─ 是否属于此 session（session_id 绑定）
          │ 通过
          ▼
    房间状态检查
    ├─ status == WAITING？（不在 IN_GAME 之前不可加入）
    └─ DS 是否可达？
          │ 通过
          ▼
    容量检查
    └─ current_players < max_players？
          │ 通过
          ▼
    重复加入检测
    └─ 该 player_id 是否已在房间内？（防止客户端重复请求）
          │ 通过
          ▼
    加入成功：current_players++，广播玩家列表更新
```

**Token 设计的要点**：加入 Token 必须绑定到具体的 session_id 和 player_id，并设置较短的有效期（通常 30-120 秒）。这防止了 Token 被截获后用于进入其他会话的攻击。

**重复加入检测**：客户端网络抖动可能导致同一个加入请求被发送多次。服务端必须用幂等设计处理这种情况——第二次收到同一 player_id 的加入请求时，直接返回成功，不做重复的数据修改。

### WAITING → IN_GAME 的转换条件

不同游戏类型的转换条件不同：

- **固定人数模式**：`current_players == max_players` 时自动开始
- **倒计时模式**：等待期满后，有最低人数就开始（比如 5/10 人也可开始）
- **房主控制模式**：房主发送"开始游戏"指令，服务端检查最低人数后执行转换

状态转换必须是原子的——不能出现"一半玩家看到游戏已开始，另一半还在等待"的情况。DS 确认状态切换后，批量广播通知所有已连接玩家。

## 阶段三：玩家离开

玩家离开有两种性质完全不同的情形，服务端的处理逻辑必须区分对待。

### 主动离开

玩家点击"退出游戏"按钮，客户端发送离开请求后断开连接。

处理流程：
1. 将玩家标记为"已离开"状态（而不是立刻从玩家列表中删除）
2. 广播"玩家XX已离开"给其他玩家
3. 根据游戏规则决定是否触发提前结束逻辑（例如，某些游戏中有玩家离开会强制结束）
4. 如果是 WAITING 阶段主动离开，且该玩家是房主，需要处理房主转移或解散房间

### 超时离开（断线）

客户端网络中断，DS 在心跳超时后判定玩家已断线。

与主动离开的关键区别：**断线的玩家可能重连**，因此不能立即清理其游戏状态。

处理流程：
1. 心跳超时，将玩家标记为"断线中"状态，记录断线时间戳
2. 启动重连等待计时器（通常 30-120 秒，根据游戏类型调整）
3. 等待计时器期间，该玩家的游戏实体保持存在（位置、血量等状态不变或交给 AI 托管）
4. 重连计时器超时后，若玩家仍未回来，则触发"超时离开"，按永久离开处理

这个设计的核心原则是：**在玩家有机会重连之前，不要破坏其游戏状态**。

## 阶段四：结算

结算阶段是整个生命周期中数据一致性风险最高的阶段。

### 结算触发

游戏结束条件满足后，DS 上的游戏逻辑触发结算。**第一个操作是状态切换**：将会话状态从 `IN_GAME` 改为 `SETTLING`。

状态切换完成后：
- 拒绝所有新的游戏操作包（客户端可能还有在途数据包）
- 拒绝新玩家加入

### 数据写入顺序

结算数据的写入顺序至关重要，必须遵循"先写结果，再广播"的原则：

```
1. 计算最终结果（胜负、得分、KDA 等）
2. 将结果写入持久化存储（数据库/排行榜服务）
         ↓ 等待写入确认
3. 更新会话状态为 CLOSING
4. 广播结算结果给所有客户端
5. 等待客户端确认收到（或超时）
```

**为什么要先写再广播？** 如果先广播，客户端在播放结算动画时 DS 崩溃了，结果就永远不会写入存储，玩家看到了"赢了"的动画，但积分没有增加。先写入确保了即使后续步骤失败，核心数据已安全落地。

### 多服务写入的协调

结算通常需要写入多个服务：

- **战绩服务**：保存本局的详细战斗记录
- **积分/排名服务**：更新玩家的 ELO / MMR
- **道具/奖励服务**：发放对局奖励
- **统计服务**：更新各类计数器

这些写入不要求全部在同一事务内完成（分布式事务代价极高），但需要保证**战绩写入优先于奖励发放**，防止玩家重复领取奖励。常见的做法是用消息队列将结算事件发布出去，各下游服务按序消费。

## 阶段五：关闭与销毁

### 会话销毁时的资源清理清单

当结算数据写入完成、所有客户端均已收到结算通知（或超时），进入 CLOSING 阶段，按以下顺序清理资源：

1. **DS 进程自主退出**：DS 调用 Agones SDK 的 `Shutdown()` 方法（或通知自建注册中心），然后进程正常退出
2. **数据落库确认**：确认所有异步写入任务已完成（或设置最大等待超时）
3. **通知匹配层**：释放 DS 资源，让编排层可以将此 DS 回收进 Ready 池或销毁
4. **清理临时数据**：删除 Redis 中的会话缓存、玩家位置数据等临时状态
5. **更新会话状态**：将数据库中的会话记录标记为 `DESTROYED`，写入结束时间戳

### 异常终止的清理

DS 进程崩溃时，上述正常流程不会执行。需要有外部的守护机制来检测"僵尸会话"：

- **心跳监控**：会话服务定期检查 DS 的心跳，超时后主动触发清理流程
- **状态幂等化**：清理逻辑必须设计为幂等的，多次执行不会产生副作用
- **未完成结算的处理**：如果 DS 在 `IN_GAME` 状态崩溃，需要决策策略——是标记本局无效、平局，还是用最后一次状态快照推算结果

## 工程边界

**状态机必须持久化**：会话状态不能只存在于 DS 进程的内存中。DS 崩溃后，外部系统需要能读到正确的状态来执行清理。使用 Redis 存储实时状态 + 数据库持久化历史记录是常见的组合。

**状态转换必须加锁**：多个并发请求可能同时尝试触发同一个状态转换（例如，两个玩家同时满足结束条件）。使用 Redis 的原子操作或数据库的乐观锁确保状态只被转换一次。

**客户端必须感知状态变化**：会话状态转换后，必须立即通知所有连接的客户端。不要依赖客户端主动轮询——这会造成客户端和服务端状态不同步的窗口期。

---

## 最短结论

游戏会话是一台五阶段状态机（WAITING → IN_GAME → SETTLING → CLOSING → DESTROYED），每个阶段定义了合法的操作集合。三个高风险点：玩家加入需要 Token + 容量 + 幂等三重验证；断线与主动离开必须分开处理，断线玩家需要保留重连窗口；结算必须先写存储再广播结果，防止 DS 崩溃导致的数据丢失。
