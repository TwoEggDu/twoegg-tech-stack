> 程序觉得"基建做完了"，策划和美术觉得"什么都用不起来"。这中间差的不是代码量，而是接口设计。

技能系统的基建不是"把代码写好"就行了。

实际生产里，技能系统的日常使用者不是程序——是策划在配数值、配公式、配效果组合；是美术在挂动画事件、调 VFX 时序、对齐表现节奏。如果程序搭的基建，策划和美术用不起来、看不懂、不敢碰，那这套基建对团队来说就等于不存在。

这一篇要回答：程序为技能系统搭基建，应该分几层搭、每层给谁用、每层该提供什么接口、常见的断裂点在哪。

---

## 三层基建

技能系统的基建可以分成三层。

```text
┌──────────────────────────────┐
│         管道层 Pipeline       │  配表 → 导出 → 校验 → 热重载 → CI
├──────────────────────────────┤
│         工具层 Tooling        │  预览器 / 依赖图 / 校验器 / 调试视图
├──────────────────────────────┤
│         运行时层 Runtime      │  参数暴露 / 行为组合 / 事件回调 / 表现隔离
└──────────────────────────────┘
```

这三层不是递进关系，而是并行服务关系：

- 运行时层决定策划和美术"能做什么"
- 工具层决定他们"能不能看见自己做了什么"
- 管道层决定他们"改完之后多久能看到结果"

很多团队只搭了运行时层，工具层和管道层几乎没有。结果就是：系统能力很强，但只有程序员能用。

## 运行时层给策划的接口

运行时层给策划的接口，核心是回答一个问题：哪些东西策划可以自己改，哪些东西必须找程序。

### 可配参数

这是最基础的一层。策划应该能在配置表或编辑器里直接修改的东西：

- 伤害值、治疗值、护盾值
- 冷却时间、施法时间、前摇时长
- 作用范围（半径、角度、长宽）
- 目标数量上限
- 消耗（MP、能量、弹药）
- 技能等级成长系数

这些参数的特征是：改了之后不会影响系统行为的正确性，只影响数值结果。程序要做的是把它们从代码里抽出来，放进 SkillDef / SpellDef 的数据结构。

### 可组合行为

比可配参数更进一步——策划不只改数字，而是选择"这个技能由哪些行为拼出来"：

| 组合维度 | 策划能选什么 |
|---------|------------|
| Effect 列表 | 一个技能触发哪些 Effect（伤害、位移、召唤、区域） |
| Buff 叠加规则 | 同一个 Buff 重复施加时是刷新、叠层还是独立计时 |
| 目标筛选模式 | 敌方 / 友方 / 自身 / 全体，加上标签过滤 |
| 施法条件 | 需要的前置状态（站立、空中、持有特定武器） |
| 连锁规则 | 命中后触发另一个效果、击杀后刷新冷却 |

每个单元自身已经被程序实现好了，策划做的是选择和组合。程序要确保每个行为单元有清晰的输入输出定义，并且组合的合法性可以被校验。

### 必须写代码的部分

不是所有东西都能配置化。必须明确告诉策划"这些事情需要找程序"的清单至少包括：

- 新的目标选择逻辑（比如"选择血量最低的三个友方"这种自定义筛选规则）
- 自定义碰撞检测形状（圆形扇形之外的不规则形状）
- 新的 Modifier 类型（对属性的操作方式不在现有框架内）
- 跨系统交互（技能触发剧情、技能改变地形、技能影响 AI 行为树）
- 性能敏感的计算逻辑

把这条线画清楚比什么都重要。策划最怕的不是"这个不能配"，而是"不知道哪些能配哪些不能配"。

### 公式系统的暴露程度

公式系统是运行时层给策划最敏感的接口之一。

`27 篇`已经讨论过公式系统的复杂度边界。这里从基建视角补一个判断标准：

```text
Level 0：纯参数填写      → 策划填数字，程序写公式
Level 1：模板选择        → 策划选公式模板，填参数
Level 2：表达式编辑      → 策划在受限语法里写公式（如 base_atk * 1.5 + skill_level * 10）
Level 3：脚本能力        → 策划能写 Lua / 可视化脚本
```

大多数项目应该停在 Level 1 到 Level 2 之间。Level 3 听起来灵活，但维护成本极高，出了 bug 排查链条会变得很长。

选定暴露程度之后，程序要保证公式求值是统一的、可调试的、有错误报告的。不要让策划写了一个表达式却不知道它实际被怎样求值。

## 运行时层给美术的接口

运行时层给美术的接口，核心不是"给美术调参数"，而是给美术提供可靠的时间线锚点和生命周期管理。

### AnimEvent 回调

`28 篇`已经定义了 AnimEvent 的命名标准。从基建视角看，程序需要确保的是：

- AnimEvent 的回调机制是稳定的——不会因为帧率波动导致事件丢失
- 回调的触发时机和技能逻辑时间线是对齐的——`OnSkillFire` 对应的确实是逻辑层的 ActiveStart
- 美术能在动画编辑器里直接看到可用的 AnimEvent 列表，而不是去问程序

关键的回调锚点：

| 回调 | 美术用来做什么 |
|------|-------------|
| OnCastStart | 播放起手特效、蓄力光效 |
| OnSkillFire | 生成投射物 VFX、播放发射音效 |
| OnSkillHit | 播放受击特效、受击音效、镜头震动 |
| OnSkillEnd | 停止循环特效、播放收招音效 |

程序不需要美术理解技能状态机内部怎么转的，但要保证这四个锚点的行为是一致的、可预期的。

### VFX 生命周期管理

美术最常遇到的 VFX 问题不是"怎么做特效"，而是"特效应该什么时候销毁"。

程序应该提供的 VFX 生命周期接口：

```text
1. 跟随技能状态：技能进入某阶段时自动播放，离开时自动停止
2. 跟随 Buff 状态：Buff 添加时播放，Buff 移除时停止
3. 独立计时：播完自然时长后自动销毁
4. 手动控制：由其他事件显式触发 Play / Stop
```

对美术来说，最好用的是第 1 种——"我只需要告诉系统这个 VFX 属于哪个阶段，系统自动管生命周期"。

最容易出问题的是美术不清楚自己的 VFX 走的是哪种模式，结果要么提前消失，要么永远挂在场景里。

### 表现层与逻辑层的隔离

`09 篇`定义了表现和逻辑的解耦原则。对美术来说，运行时基建需要保证一个底线：

表现可以延迟、可以跳过、可以降级，但不能影响逻辑。

具体来说：

- 低端设备关闭了粒子特效，技能伤害照算
- 网络延迟导致动画晚播了两帧，命中判定不受影响
- 表现层崩了（VFX 加载失败），技能逻辑不会跟着崩

程序在这一层要做的不是让美术理解隔离原理，而是让隔离成为默认行为——美术做表现时根本不需要担心"我改了特效会不会把逻辑搞坏"。

## 工具层

工具层的职责是让策划和美术能看见自己做了什么、做对了没有。

`12 篇`已经系统讲过编辑器、依赖图、校验器和调试视图。这里从"程序应该提供什么"的视角做一次重新整理。

### 技能预览器

不启动完整游戏就能看到技能效果。这是策划和美术迭代效率最大的瓶颈——每改一次配置要启动完整游戏、进关卡、找怪物、释放技能，一个来回 3 到 5 分钟，一天下来有效迭代次数被严重压缩。

预览器应该做到：

- 选一个技能，立刻看到施法动画、VFX、范围指示器
- 能拖时间轴，手动控制技能在哪个阶段
- 能切换施法者模型（不同角色体型对挂接点有影响）
- 不需要战斗系统、不需要 AI、不需要完整场景

`12 篇`已经强调过：不要在预览器里重写一套运行时逻辑。预览器看表现，运行时管真相。

### 依赖图

一个技能引用了哪些 Effect、Buff、动画、VFX、音效。技能数量超过 50 个以后这件事变得关键——策划改一个 Effect 参数，需要知道多少技能引用了它；美术替换一个 VFX，需要知道哪些技能受影响。

依赖图至少支持正向查询（这个技能依赖什么）、反向查询（这个资源被谁引用）、可视化展示（节点图或树形，不是纯文本）。

### 校验器

提交前自动检查配置合法性。校验器是成本最低、收益最高的工具——在策划保存配置的那一刻就告诉他：引用的 EffectDef 不存在、冷却时间填了负数、动画资源路径无效、Buff 叠加规则和效果类型矛盾、公式里引用了不存在的变量。

校验器最重要的一条原则：错误信息必须是业务级的，不是代码级的。

```text
坏的报错：NullReferenceException at SkillDefLoader.cs:127
好的报错：技能"火球术"引用的效果"Burn_DOT"不存在，请检查 EffectDef 表第 43 行
```

### 调试视图

运行时查看技能状态机、Buff 列表、Modifier 聚合链。

`12 篇`提出调试视图应该围绕"一次技能释放"给出可读事件链。从基建视角补充一点：调试视图至少应该有两个模式。

模式一：策划模式——显示业务信息。

```text
[火球术] 施法成功
  → 触发效果：对目标造成 120 点火焰伤害
  → 触发效果：附加 Burn Buff，持续 3 秒，每秒 15 点伤害
  → 冷却开始：5.0 秒
```

模式二：程序模式——显示系统信息。

```text
[12:31:08.121] CastRequest(skillId=1003, caster=Entity_42)
[12:31:08.122] ValidationPassed(constraints=[MP>=30, CD==0])
[12:31:08.123] EnterPhase(Windup, duration=0.3)
[12:31:08.356] SpawnProjectile(prefab=proj_fireball, id=3007)
[12:31:08.601] HitConfirmed(target=Entity_78, hitbox=Sphere_r2)
[12:31:08.602] ExecuteEffect(DamageEffect, value=120)
[12:31:08.603] ApplyBuff(BurnBuff, stacks=1, duration=3.0)
```

策划看第一种就够了。程序排查问题看第二种。如果只有第二种，策划看不懂；如果只有第一种，程序排不了线上问题。

## 管道层

管道层的职责是让"改完配置到看到结果"这条路尽可能短、尽可能安全。

### 配表到运行时的自动化流水线

```text
策划改配置 → 保存 → 本地校验 → 导出为运行时格式 → 自动部署到开发环境
```

每一步都应该是自动的。策划不应该需要手动运行导出脚本、手动拷贝文件、手动触发构建。`27 篇`已经讲了管道细节，这里只补一条：程序的责任不只是搭起来，还要保证它不会悄悄坏掉。

### 热重载

改了配置不需要重启游戏。这一条对迭代效率的影响是量级的——重启游戏迭代一天 30 次，热重载一天 200 次。

| 可热重载 | 不可热重载 |
|---------|----------|
| 数值参数（伤害、冷却、范围） | 新增的 Effect 类型 |
| Buff 参数（持续时间、叠层上限） | 新增的目标选择模式 |
| 公式表达式 | 新增的碰撞检测形状 |
| VFX 资源引用 | 运行时代码变更 |
| 音效资源引用 | 状态机结构变更 |

左边的改动频率远高于右边。只要左边的东西能热重载，策划和美术的日常工作效率就能得到保障。

### CI 集成

配置变更触发回归测试。CI 应该在配置合入主干之前跑全量校验（SkillDef / EffectDef / BuffDef 合法性）、引用完整性检查（悬空引用、循环依赖）、关键技能的自动化测试（`13 篇`定义的回归用例），以及生成变更报告。

CI 的价值不是"帮策划检查"，而是"在改动合入主干之前拦住问题"。

### 错误反馈闭环

校验失败或导出失败时，告诉策划具体哪里错了、怎么改。一个完整的反馈应该包含：哪个文件哪一行、出了什么问题（业务语言）、可能的修复建议。做到前两条就已经超过大多数项目了——很多团队的管道报错还停留在堆栈信息阶段，策划看到之后唯一能做的就是把截图发给程序。

## 五个常见断裂点

基建搭了，但团队用不起来。原因通常出在以下五个地方。

### 断裂点一：工具只能程序员用

编辑器、校验器、依赖图都有了，但操作方式是命令行的，输出是 JSON 的，没有 UI。程序觉得"功能全了"，策划觉得"完全看不懂"。

解决方案不是做完美 GUI，而是至少做到：有最小可用的可视化界面、输出人类可读、操作不超过三步就能看到结果。

### 断裂点二：预览器和实际运行时行为不一致

`12 篇`重点警告过这个问题。预览器为了不依赖完整运行时，往往自己做一套简化逻辑。时间一长两边就会漂移，策划在预览器里调好了效果，进游戏一看完全不一样。这个断裂一旦发生，预览器就会被整个团队抛弃。

解决方案：尽可能复用运行时代码路径，只 mock 外部依赖（网络、完整场景、AI）。做不到就明确标注预览器的保证边界。

### 断裂点三：校验规则和配表规范不同步

程序写了校验规则，但策划后来新增了 Effect 类型或改了配表结构，校验器没跟着更新。结果是合法配置被拒绝、不合法配置被放行，策划开始绕过校验器提交。

解决方案：校验规则和配表 schema 共用一份定义，新增字段或类型时校验规则自动跟着更新。

### 断裂点四：管道报错信息是代码级的而非业务级的

```text
坏的报错：
  System.ArgumentOutOfRangeException: Index was out of range.
  at SkillExporter.Export(String path) in SkillExporter.cs:line 234

好的报错：
  导出失败：技能"冰冻新星"（第 87 行）的 EffectList 引用了 ID=2045 的效果，
  但 EffectDef 表中不存在 ID=2045。
  建议：检查 EffectDef 表是否有遗漏，或者技能配置中的 EffectID 是否填错。
```

策划不知道 SkillExporter.cs 在哪，也不知道 Index was out of range 意味着什么。解决方案：在管道每个环节加一层业务级错误翻译——哪个技能、哪个字段、什么问题、怎么改。

### 断裂点五：基建做了但没人教怎么用

这是最容易被忽视但杀伤力最大的断裂点。

程序花两个月搭了工具链，写了技术文档，就认为基建完成了。但策划和美术从来没被培训过怎么用这些工具。结果是策划还在手动导出 Excel，美术还在口头问程序"这个特效挂哪里"，工具链使用率不到 20%。

解决方案不是写更长的文档，而是做一次面对面的工作坊、在工具里加引导提示、指定一个对接人前两周随时答疑、收集反馈把高频问题做进工具默认行为。

## 这一篇真正想留下来的结论

程序为技能系统搭基建，最容易犯的错误是只站在程序视角评估"基建够不够"。运行时稳定、架构合理、代码可维护——这些只是程序自己的基建。策划和美术需要的是另一套东西：

- 运行时层：清晰的可配 / 可组合 / 必须写代码的边界
- 工具层：看得懂的预览器、查得到的依赖图、拦得住的校验器、分角色的调试视图
- 管道层：自动化的导出流水线、能热重载的配置通道、说人话的错误信息

五个断裂点里，技术性最低但杀伤力最大的是最后一个——基建做了但没人教怎么用。

如果只记一句话：

`技能系统的基建不是"让代码跑起来"，而是"让不写代码的人也能高效、安全地产出内容"。程序的基建做没做好，不是程序说了算，是策划和美术的日常工作效率说了算。`
