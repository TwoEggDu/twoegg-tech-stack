## 问题从一次需求评审开始

一家游戏工作室在评审"玩家换装后立刻刷新排行榜头像"这个需求时，陷入了争论。排行榜服务说："头像数据在账号服务，我需要账号服务推送更新。"账号服务说："排行榜刷新不是我的职责，我只管存头像。"API 网关说："那我是不是要在更新头像的接口里，顺便触发排行榜刷新？"

这场争论暴露的不是需求问题，而是**服务边界没有划清楚**的问题。当一个功能需求需要三个服务的负责人一起开会才能确定归属，说明这三个服务的边界存在模糊地带。

服务边界划分是游戏后端架构中最需要前期投入的决策，因为它一旦固化下来，后续的改动成本极高。

---

## 按什么维度划分服务边界

业界有多种划分服务的方式：按技术层（数据层、业务层、表现层），按功能（CRUD 服务、查询服务），按业务领域（DDD Bounded Context）。在游戏后端，**按业务领域划分**是最有生命力的方式，原因如下：

**业务领域的变化节奏天然不同**。支付逻辑和战斗逻辑完全不在同一个时间维度上变化——支付跟着渠道策略和合规要求走，战斗跟着版本设计走。把它们放在同一个服务里，任何一边的变更都会拖累另一边的发布节奏。

**业务领域对存储的要求天然不同**。排行榜的核心操作是有序集合读写，Redis Sorted Set 是天然契合。聊天历史需要高吞吐写入和时间范围查询，Kafka + Cassandra 或 MongoDB 更合适。战斗数据是高频的内存状态，根本不需要持久化。如果一个服务需要同时优化完全不同的存储模式，说明它承载了不属于同一个领域的职责。

**业务领域的扩缩容需求天然不同**。匹配服务在新赛季开始时压力最大，战斗服在玩家在线峰值时压力最大，排行榜在结算期压力最大。这些峰值时间通常不重叠，独立扩缩可以节省大量资源。

DDD（领域驱动设计）中的 Bounded Context 概念给了一个实用的划分原则：**同一个名词在不同上下文中有不同含义，就说明边界在这里**。"玩家" 在账号服务里是身份和认证，在战斗服里是实体和属性，在排行榜里是分数和排名。这三个"玩家"可以共享一个 PlayerID，但它们的状态、生命周期、一致性要求完全不同。

## 账号服务：一切的信任根基

**核心职责**：玩家注册、登录、Token 发放与验证、基础个人信息（昵称、头像 URL、绑定渠道）的存储。

账号服务是整个后端的信任根基。其他所有服务在处理请求前，都需要问账号服务一个问题：这个 Token 合法吗？对应的 PlayerID 是什么？

因此账号服务有几个独特的特性：
- **读多写少**：登录后的大多数请求只是验证 Token，Token 可以是 JWT 自验证（无需查库）或短期缓存在 Redis 中
- **变更频率极低**：账号逻辑一旦稳定，几乎不需要改动，这使它非常适合独立出来
- **故障影响是全局的**：账号服务挂了，玩家无法登录，所有业务都受影响——这意味着它需要最高的可用性保障，独立部署才能给它专属的资源保证

**边界注意**：账号服务只管**身份**，不管**玩家状态**。玩家当前在哪个战斗房间里、玩家的战斗属性是什么——这些不属于账号服务。

常见错误：把"玩家数据"都堆进账号服务，导致账号服务变成一个什么都装的大杂烩。账号服务应该只存那些在任何业务场景下都不变的基础身份信息。

---

## 匹配服务：无状态入口，有状态路由

**核心职责**：接受玩家匹配请求，按照规则（段位、延迟、模式）组成对局，分配战斗服房间，通知玩家进入。

匹配服务是少见的同时具有两种特性的服务：它的**业务逻辑是无状态的**（匹配算法不依赖历史状态），但它管理的**匹配队列是有状态的**（玩家在队列里等待，需要保持会话）。

这种特性决定了匹配服务的架构：核心的匹配算法可以水平扩展，但队列状态需要统一的存储（通常是 Redis）。

**独立理由**：
- 匹配算法迭代频繁（新模式、新排队逻辑），不应该影响其他服务
- 峰值负载和战斗服不同步，新模式上线时匹配压力急剧上升
- 匹配涉及跨服玩家路由，逻辑相对独立

**边界注意**：匹配服务只负责"把玩家送进房间"，不负责"玩家在房间里做什么"。一旦玩家进入战斗房间，匹配服务的职责就结束了。

常见错误：让匹配服务直接管理战斗房间的状态，或者让匹配服务感知战斗结果。这会造成匹配服务和战斗服之间的强耦合，任何一方发布都要协调另一方。

---

## 战斗服务：有状态的核心，轻易不拆细

**核心职责**：承载玩家实时对局，处理输入指令，推送同步状态，管理房间生命周期，产出结算数据。

战斗服是游戏后端中最难拆分的服务，原因是它天生有状态且对延迟极度敏感。

**为什么不能拆太细**：
- 一局战斗的完整上下文（所有玩家的位置、血量、技能冷却、物理状态）必须在同一个进程的内存里，才能在帧间隔（15ms~50ms）内完成计算和推送
- 跨进程调用意味着序列化 + 网络往返，即使是最快的 gRPC 也要增加 1ms+ 的延迟，对竞技游戏不可接受
- 游戏规则（技能判定、伤害计算、碰撞检测）本质上是一个无法拆分的计算整体

战斗服通常作为独立集群运行，一个房间绑定到一个战斗服进程实例，通过 PlayerID 或 RoomID 路由，不允许跨实例。

**什么可以拆出去**：战斗结束后的**结算**可以异步化，战斗服产出结算报告后，由结算服务负责把战斗结果转换为背包变更、排行榜更新、成就解锁等副作用。这样战斗服本身的职责保持纯粹，只负责"打架"，不负责"打完之后的所有事情"。

**边界注意**：战斗服不应该直接操作背包数据库或账号数据。它产出事件，其他服务消费事件，这样结算失败了可以重试，不需要回滚战斗状态。

## 排行榜服务：读写分离，适合独立

**核心职责**：接收分数更新，维护有序排名，提供排名查询（单人排名、段位列表、附近排名）。

排行榜的技术特性非常鲜明：
- 写操作是**幂等的点更新**（更新某个玩家的分数），可以高度并发
- 读操作是**范围查询**（前 100 名、自己附近的排名），需要有序数据结构
- Redis Sorted Set 能以 O(log N) 的复杂度完成这两类操作，是排行榜的标准实现

排行榜适合独立的原因：结算季、赛季结束时的访问量峰值极高，但这个峰值和战斗峰值完全不同步。独立排行榜服务可以在结算时单独扩容 Redis 集群，不影响其他服务。

**边界注意**：排行榜服务只负责维护分数和排名，不负责解读"这个分数意味着什么"。段位计算（分数 → 段位名称）可以是排行榜服务的一部分，也可以由配置服务提供规则，排行榜按规则转换。

**头像同步问题**：回到开头的例子——排行榜展示的头像 URL，可以在写入排行榜分数的同时一并写入（来源：账号服务的事件通知）。排行榜服务不应该在查询时实时调用账号服务获取头像，那是查询热路径，延迟不可接受，也是错误的边界理解。

## 聊天服务：异步写入，独立扩展

**核心职责**：接收消息发送请求，存储聊天历史，推送消息给在线玩家。

聊天和游戏核心逻辑的关系极弱（玩家在聊天时，游戏状态不依赖聊天），但技术要求特殊：
- 消息写入量大，频率高，允许短暂延迟
- 历史消息查询需要时间范围索引
- 实时推送依赖 WebSocket 或 SSE 长连接

这些特性使聊天适合独立，且内部可以使用和其他服务完全不同的技术栈（Kafka 做写缓冲，Cassandra 或 MongoDB 做历史存储）。

**边界注意**：聊天服务不做内容审核的实时决策——内容审核是一个独立的能力，可以是同步拦截也可以是异步举报处理，不应该让聊天服务的核心路径等待审核结果。

## 支付服务：合规驱动的强隔离

**核心职责**：接收购买请求，校验渠道支付凭证（Apple IAP / Google Play / 第三方支付），扣款，通知发货服务。

支付服务独立的驱动力**不是技术原因，而是合规和安全原因**：
- 支付数据受到严格的数据保护要求，需要独立的审计日志，不能和游戏业务数据混合存储
- 渠道对接逻辑（IAP 验证、Webhook 签名校验）变更频繁，独立服务可以快速适配
- 支付服务的安全漏洞风险极高，需要最严格的代码审查和访问控制

**边界注意**：支付服务只负责**确认钱收到了**，不负责发放道具。发货是另一个服务（或支付的下游事件消费者）的职责。这个边界的本质是：支付成功和道具发放是两个独立的事件，中间必须有可重试的机制——支付成功但发货失败，应该能补发，而不是退款。

## 服务间调用链过深的问题

调用链过深是微服务腐化的典型症状。假设玩家点击"领取任务奖励"，触发了：API 网关 → 任务服务 → 背包服务 → 成就服务 → 排行榜服务 → 通知服务，共 5 次串行 RPC 调用，累计延迟 10ms+，而且任何一个服务超时都导致整个请求失败。

**收缩策略**：
1. **异步化非关键路径**：成就解锁和排行榜更新通常不需要在主请求里同步完成，可以改为异步事件。主请求只保留对玩家体验有直接影响的同步调用。
2. **数据冗余换调用减少**：背包服务不需要每次查询都调用账号服务获取玩家名字，可以在背包写入时冗余存储玩家名字（接受最终一致性）。
3. **聚合服务（BFF）**：如果 App 端需要同时展示多个服务的数据（个人主页：账号信息 + 排行榜名次 + 最近战绩），可以引入一个 BFF（Backend for Frontend）层，并发调用多个服务，在 BFF 层聚合结果，客户端只发一个请求。

## 工程边界：哪些服务不该被这套模型覆盖

游戏后端里有一些服务具有跨领域特性，不适合被强行放进业务领域模型：

- **日志和监控**：这是横切关注点，不应该有服务依赖它，而是所有服务向它推送数据
- **配置中心**：提供运行时配置（活动参数、匹配规则、限流阈值），所有服务都依赖它，但配置中心本身不依赖任何业务服务
- **游戏数值管理**：道具属性、技能参数这类数据，属于内容管理系统的职责，不应该分散到各业务服务里，应该有独立的数据管理和热更新机制

---

## 最短结论

游戏后端的服务边界应该沿着业务领域切割，同时让数据存储归属和变更节奏对齐——不同领域不共享数据库，边界清楚的标志是：修改一个功能只需要改一个服务。
