M01 解决的是“该不该上 DOTS”，M03 解决的是“长期 Hybrid 边界怎么活”。到了 M02，问题就变得更具体了：**如果决定试点，第一刀到底切哪一层，才能既拿到收益，又不把项目结构切坏**。

很多团队不是死在“不会写 ECS”，而是死在“第一刀切错了”。一上来先碰 UI、先碰动画、先碰资源加载，或者先想把整个角色框架搬进 ECS，最后往往得到一套更复杂、却没有稳定收益的混合结构。第一阶段迁移真正该做的，不是证明“DOTS 很强”，而是证明**这条迁移路径在你的项目里值得继续走**。

---

## 为什么第一刀最容易切错

第一阶段迁移最危险的误判，是把“用户最容易看见的问题”当成“最适合先迁的问题”。卡顿最明显的也许是 UI 首开、角色动画顿一下、场景切换最后一帧抖动，但这些现象未必是高密度仿真层的问题，更未必适合拿 DOTS 当第一把刀。

DOTS 第一阶段最该承担的任务，不是“把最痛的地方都修掉”，而是先验证三件事：

- 你的项目里是不是真的存在一层高密度、同构、批量更新的仿真压力。
- 这层压力能不能被从当前 MonoBehaviour 结构里安全切出来。
- 切出来之后，团队能不能用 A/B 基线证明收益真实存在。

如果这三件事都没有先被回答，迁移动作就很容易从“试点”滑成“重写”。

## 为什么不要从 UI 开始

UI 常常是第一阶段最不该碰的层。原因不是 UI 不重要，而是它天然站在 `Representation Layer`，负责展示、交互和平台耦合，和 DOTS 最擅长的高密度仿真层不是同一个问题空间。

从 UI 开始迁移通常会立刻遇到三类坏耦合：

- UI 读的是大量状态快照，但本身不是权威仿真层。
- UI 强依赖 Unity Object、事件回调、资源引用和编辑器工作流。
- UI 的痛点很多时候来自重建、布局、图集、过度刷新，而不是数据布局。

这也是为什么 M03 要把 `Managed -> ECS` 的边界单独拿出来讲。第一阶段如果先碰 UI，往往会把边界问题提前引爆：仿真层还没切出来，表现层已经和桥接层缠在一起了。那时你得到的不是 DOTS 试点，而是一套谁都不敢再动的过渡结构。

## 先挑 Pilot System，不要挑“最显眼的系统”

第一阶段不该先挑“最核心”的系统，而该先挑**最适合做 Pilot System 的系统**。一个合格的试点系统，通常同时满足下面四个条件：

- 它有明显的高密度、同构、批量更新特征。
- 它和 UI、动画、资源引用之间的直接耦合较弱。
- 它的输入和输出边界清楚，能通过桥接层接入现有项目。
- 它可以建立 A/B 基线，迁移前后都能稳定测量。

这意味着第一阶段最适合切出的，通常不是“玩家角色总控系统”，而是更像下面这些层：

- RTS 或塔防里的大规模单位移动与状态推进。
- 弹幕、子弹池、命中体积查询这类高频仿真对象。
- 大量 NPC 的简单感知、筛选、路径前处理。
- 高密度 Buff、冷却、定时器、范围判定等批量规则层。

它们的共同点不是“业务重要”，而是“能在不撕裂整套项目结构的前提下，先把 DOTS 真正擅长的收益拉出来”。

## 第一阶段最适合切出的仿真层长什么样

如果要给第一阶段一个简单画像，它通常长这样：

- 输入来自现有输入系统、AI 决策或服务器下发。
- 中间是一段高密度仿真层，负责批量状态推进。
- 输出不是直接驱动 UI，而是产出位置、状态标记、命中结果、播放事件这类桥接数据。
- Managed 层再把这些结果翻译成动画、VFX、UI 和声音。

也就是说，第一阶段迁移的目标不是“把角色从 MonoBehaviour 彻底搬走”，而是先切出一段可以独立验证的 `Simulation Layer`。只要这层成立，你后面无论继续扩张还是及时止损，结构都还是清楚的。

一个最小迁移路线，通常可以压成下面这张图：

```text
现有输入 / AI / 服务端结果
    -> Managed 桥接层
    -> Pilot System（ECS 仿真层）
    -> 输出位置 / 状态 / 事件快照
    -> 动画 / UI / VFX / Audio 消费
```

这条链路里最重要的不是“ECS 写了多少行”，而是仿真层有没有被独立出来，桥接层有没有保持窄接口。

## 第一阶段怎么落刀才稳

真正动手时，比“迁哪层”更重要的是“迁的顺序”。最稳的做法，通常不是一边迁一边猜，而是按下面五步收束：

### 1. 先冻结问题和基线

在动任何系统之前，先写清楚：

- 当前瓶颈到底是什么。
- 准备用什么指标判断迁移值不值。
- 哪个系统被视为第一阶段试点。
- 什么结果算通过，什么结果算退出。

如果这一步没写，团队很快就会把“试点”做成“持续扩写”。

### 2. 只切仿真层，不切表现层

第一阶段最重要的边界纪律，是先让 ECS 接管“算什么”，不要急着接管“怎么显示”。位置、命中、状态推进、计时器递减，这些都适合先切；UI、动画、摄像机、声音，先留在 Managed 层。

### 3. 桥接层保持窄，不做业务中台

桥接层只负责两件事：

- 把输入翻成 ECS 可消费的数据。
- 把 ECS 结果翻成表现层可消费的数据。

不要在桥接层里继续堆复杂规则，否则它会立刻变成第二套业务核心。

### 4. 并行保留旧路径做 A/B

第一阶段最好保留一条旧实现作为对照，而不是直接把旧路径删掉。哪怕只保留一个关卡、一个战斗场景、一个 1000 单位压测场景，也比“主观感觉更快了”要可靠得多。

### 5. 明确试点完成后要么扩张，要么止损

Pilot System 的价值在于帮你判断方向，不在于“证明团队已经迁了一部分”。如果试点收益不稳定、边界越来越糊、维护成本过高，就应该停；如果收益稳定、桥接清楚、回归可控，再决定继续切下一层。

## 一个最小可落地的第一阶段写法

下面这个例子不是版本绑定教程，只是说明“第一阶段迁移该把什么放哪边”：

```csharp
public struct UnitMoveCommand : IComponentData
{
    public float2 DesiredDirection;
    public float DeltaTime;
}

public struct UnitMoveResult : IComponentData
{
    public float3 Position;
    public bool IsMoving;
}

// Managed 侧采样输入或 AI 决策，写入桥接数据。
public sealed class UnitMoveBridge : MonoBehaviour
{
    public void PushMove(float2 direction, float deltaTime)
    {
        // 这里只负责把输入送进 ECS，不做仿真规则。
    }
}

// ECS 侧只负责批量推进单位移动。
public partial struct UnitMoveSystem : ISystem
{
    public void OnUpdate(ref SystemState state)
    {
        // 读取 UnitMoveCommand，批量更新位置与状态，
        // 输出 UnitMoveResult 供动画 / UI / VFX 消费。
    }
}
```

这个写法的重点不是具体 API，而是职责：

- 输入留在 Managed 侧采样。
- 仿真留在 ECS 侧批量推进。
- 结果输出成桥接数据，不直接劫持动画和 UI。

这就是第一阶段该追求的最小结构。

## A/B 基线和退出条件必须先写

M01 已经讲过“没有验证链就别上 DOTS”。到了 M02，这条原则要进一步收紧：**第一阶段试点必须在开工前就写出通过条件和退出条件**。

最低限度也要有下面这张表：

| 项目 | 迁移前 | 迁移后目标 | 退出条件 |
|------|--------|------------|----------|
| 单位数 / 子弹数 / 仿真对象数 | 固定压测场景 | 不低于当前规模 | 无法稳定复现 |
| 主线程时间 | 当前基线 | 有明确下降 | 收益不稳定或回归频繁 |
| 结构边界 | 当前 MonoBehaviour 路径 | Managed / ECS 边界更清楚 | 桥接层越来越厚 |
| 维护成本 | 当前团队可控 | 回归和调试成本可接受 | 维护难度明显上升 |

这张表的作用不是项目管理，而是防止试点越做越像信仰工程。

## 第一阶段最不该碰的几层

如果你现在就准备开刀，下面这几层通常应该明确排除在第一阶段之外：

- UI 和界面状态机。
- 角色动画总控、镜头系统、音频触发。
- 资源加载、Addressables、流式 IO 这类非仿真瓶颈。
- 编辑器工作流强绑定的技能配置、关卡 Authoring 工具。
- 整个玩家角色框架和所有战斗逻辑一起打包重迁。

第一阶段应该做的是“切小、切深、切可测”，而不是“切广、切满、切到没人敢回退”。

## 常见误区

最常见的误区之一，是把第一阶段迁移理解成“先把系统改写成 ECS 语法”。这会让团队以为工作量等于收益，结果却常常只是把原有逻辑换了一种写法。

第二个误区，是把试点系统选成最关键、最复杂、最不可回退的系统。这样的试点一旦失手，项目很难快速止损。

第三个误区，是试点一跑起来就顺手把动画、UI、事件、资源层一起改了。这样一来，最后很难判断收益到底来自 DOTS，还是来自你同时改动了五件事。

## 小结

第一阶段迁移的关键，不是“先迁最重要的层”，而是先挑一个高密度、弱耦合、可测量的 `Pilot System`。
别从 UI 和表现层开始，先切出一段能独立成立的 `Simulation Layer`，再用桥接层把它接回现有项目。
只要基线、退出条件和桥接边界在开工前就冻结住，第一阶段迁移就还是工程试点，不会滑成结构重写。
