## 问题空间

某 MMORPG 主数据库服务器硬件故障，宕机时间 4 小时。玩家数据回滚到 3 小时前的备份，3 小时内获得的所有装备、经验、货币全部消失。论坛炸了，工单数量是平日的 50 倍，运营团队手动补偿玩家耗费两周时间。

这个故事在游戏行业里重复发生，几乎每年都有知名游戏中招。技术上，灾难恢复是可以被工程化的问题，但很多团队在第一次灾难来临之前，从未认真定义过自己能接受多少数据丢失、多少停服时间。

定义这两个问题是一切灾难恢复工程的起点。

---

## 抽象模型

### RTO 与 RPO 的游戏语境定义

**RTO（Recovery Time Objective，恢复时间目标）**：从灾难发生到服务完全恢复，最多允许多长时间？

对游戏服务：
- 休闲手游：RTO 4 小时可接受（大多数玩家晚上回来时服务已恢复）
- 竞技 PC 游戏：RTO 1 小时已经是重大事故
- 电竞赛事直播期间：RTO 15 分钟可能导致数百万美元赔偿

**RPO（Recovery Point Objective，恢复点目标）**：恢复后，数据最多丢失多久之前的操作？

对游戏服务，RPO 的直接翻译是：**玩家最多会丢失多少游戏时间内的进度**。
- 休闲手游：RPO 1 小时通常可以接受（玩家理解并谅解）
- 付费道具场景：RPO 5 分钟——如果玩家刚充值了 300 元并获得了道具，这个操作被回滚，即使有补偿机制也会引发强烈不满和退款纠纷
- 竞技赛事结果：RPO 0（赛果记录不能丢，必须有实时备份）

**RTO 和 RPO 决定了你需要为备份和冗余花多少钱**。RPO 越小、RTO 越短，成本越高。明确定义这两个数字，才能合理选择技术方案，而不是盲目追求"越高越好"。

## 具体实现

### 数据库备份层次

游戏数据库的备份体系通常分三层，层层递进：

#### 第一层：全量备份

定期对数据库做完整快照，通常每天一次（或每 6 小时一次，取决于 RPO 要求）。

```
时间轴：
00:00  全量备份 A（100GB）
06:00  全量备份 B（102GB）
12:00  全量备份 C（104GB）
18:00  全量备份 D（106GB）
```

全量备份是兜底层，能从任意备份时间点完整恢复。
**问题**：备份间隔内的数据丢失无法恢复（如果 RPO 是 1 小时，6 小时全量备份不够）。

#### 第二层：增量备份

在全量备份之间，每隔一段时间（如 1 小时）备份自上次备份以来变化的数据块（InnoDB 的增量基于 redo log 或 xtrabackup 的增量能力）。

```
00:00  全量备份 A
01:00  增量备份（相对于 A 的变化）
02:00  增量备份（相对于 01:00 的变化）
...
06:00  全量备份 B（替代 A + 所有增量）
```

恢复时：加载全量 A + 按序应用所有增量，恢复到任意整点。
**问题**：RPO 仍然是 1 小时（最后一次增量备份到故障时间点的数据丢失）。

#### 第三层：Binlog/WAL 实时复制

对于 MySQL 是 binlog，对于 PostgreSQL 是 WAL（Write-Ahead Log）。这是数据库事务级别的操作日志，每一个 INSERT/UPDATE/DELETE 都被记录。

```
全量备份 A（00:00）
+ 增量备份（01:00）
+ binlog（01:00 → 故障时刻，近乎实时）
= 恢复到故障前最后一个提交的事务
```

Binlog 复制实现方式：
- **本地保留**：binlog 写到本地磁盘，定期上传对象存储（OSS/S3）。如果服务器硬件全损，binlog 也会丢失，因此上传频率决定最终 RPO。
- **主从复制**：binlog 实时流式复制到从库（Replica），从库始终与主库保持接近实时的数据同步（通常延迟 < 1s）。主库故障时，从库可以立即提升为新主库。

```
主库 Master  →  binlog 实时流  →  从库 Replica 1
                               →  从库 Replica 2（异地）
```

#### 三层备份的 RPO 对比

| 备份方案 | RPO | 存储成本 | 恢复复杂度 |
|---------|-----|---------|-----------|
| 仅全量备份（1次/天）| 最多 24 小时 | 低 | 简单 |
| 全量 + 增量（1次/小时）| 最多 1 小时 | 中 | 中等 |
| 全量 + 增量 + binlog 上传 | 最多 5 分钟 | 中高 | 中等 |
| 全量 + 主从实时复制 | < 1 秒 | 高（多倍机器成本）| 复杂（主从切换）|
| 多主同步复制（如 Galera）| 0（同步提交）| 极高 | 极复杂 |

### 冷备、热备、多主复制的成本对比

**冷备（Cold Standby）**：备份文件存在对象存储，需要时手动恢复。
- RTO：数小时（恢复时间取决于数据量）
- 成本：极低（只需对象存储费用）
- 适用：开发环境、非关键数据

**热备（Hot Standby / 主从复制）**：从库实时跟随主库，主库故障时手动或自动切换。
- RTO：1-5 分钟（检测故障 + 提升从库 + 更新连接指向）
- 成本：2× 机器成本（一主一从）
- 适用：大多数游戏生产数据库的标准方案

**多主复制（Multi-Master）**：多个节点都可以接受写入，任一节点故障不影响写入。
- RTO：秒级（其他节点继续处理请求）
- 成本：3× 以上，且写入延迟增加（需要等待多节点确认）
- 适用：对写入可用性要求极高的场景，游戏中较少见（游戏写入通常可以接受短暂中断）

### 宕机演练（Chaos Engineering）

备份和主从复制写好了，但你真的确认它们能用吗？很多团队在真正需要从备份恢复时才发现：备份文件损坏、恢复步骤文档过时、从库复制延迟已经落后 6 小时、自动切换脚本有 bug。

Chaos Engineering 的核心思想是：**主动制造故障，在可控条件下验证恢复流程，而不是等待真实灾难**。

#### 演练类型

**数据库主库宕机演练**：
1. 在低峰期，人工关闭主库进程（或切断主库网络）
2. 观察从库提升为主库的时间（自动 vs 人工）
3. 测量应用层感知到的服务中断时间
4. 验证写入在切换后能正常恢复
5. 复盘：发现了哪些预期外的问题？

**备份恢复演练（定期，建议每月）**：
1. 取最近一次全量备份 + 增量备份 + binlog
2. 在隔离测试环境中完整执行恢复流程
3. 验证恢复后的数据完整性（通过校验和、业务数据一致性检查）
4. 记录实际恢复耗时，与 RTO 对比

```bash
# 备份恢复验证脚本示意
restore_from_backup() {
    local backup_date=$1
    echo "开始从 ${backup_date} 的备份恢复..."

    # 1. 下载全量备份
    aws s3 cp s3://game-backup/${backup_date}/full.xbstream /tmp/
    
    # 2. 解压并应用增量
    xtrabackup --prepare --target-dir=/tmp/full --incremental-dir=/tmp/inc1
    
    # 3. 应用 binlog
    mysqlbinlog /tmp/binlog.* | mysql -u root -p
    
    # 4. 验证关键数据
    mysql -e "SELECT COUNT(*) FROM players WHERE updated_at > NOW() - INTERVAL 1 HOUR"
    
    echo "恢复完成，耗时: ${SECONDS}s"
}
```

**网络分区演练**：模拟两个 Zone 之间的网络中断，验证跨 Zone 的玩家状态迁移是否能正确处理（等待重试还是直接报错）。

#### 演练的组织要求

技术上的 Chaos Engineering 需要组织层面的支持：
- 演练必须有明确的**成功标准**（RTO/RPO 是否达标），而不是"看看会发生什么"
- 演练前必须有**回滚计划**（如果演练导致真正的数据问题怎么办）
- 演练结果必须形成**改进 Action**，否则演练价值归零

### 多区域容灾

对于有全球玩家的游戏，单一数据中心存在单点风险。多区域容灾指在地理分散的两个（或多个）数据中心运行备用系统。

#### 架构模式

**主备模式（Active-Passive）**：
- 主区域：承载全部流量
- 备区域：保持热备状态，数据实时从主区域同步，但不接受外部流量
- 切换：主区域故障时，DNS 切换到备区域（TTL 低才能快速生效）

```
正常状态：
  玩家 → DNS → 主区域（上海）→ 主数据库

故障切换：
  主区域（上海）故障
  ↓
  运维触发切换脚本
  ↓
  DNS 更新指向备区域（北京）
  ↓
  备区域从库提升为主库
  ↓
  玩家 → DNS（更新后）→ 备区域（北京）→ 新主库
```

**主主模式（Active-Active）**：两个区域都承载流量，互相作为对方的备份。玩家按地理位置路由到就近区域。但游戏的全服玩法（排行榜、拍卖行）需要跨区域同步，实现复杂度很高。

#### 跨区域数据同步的延迟问题

主备模式下，从主区域到备区域的数据同步不是零延迟。典型跨地区复制延迟：
- 同城两机房：< 5ms
- 上海 ↔ 北京：约 30ms
- 上海 ↔ 新加坡：约 50-80ms
- 上海 ↔ 北美：约 150-200ms

30ms 的网络延迟意味着从库落后主库最多约 30-100ms（取决于事务提交频率）。

游戏场景中，切换到备区域后需要评估：
- **玩家数据的 RPO**：最近 30-100ms 内的充值记录、对局结果是否可能丢失？
- **全局状态的一致性**：排行榜、每日活动进度是否需要特殊处理？

### 游戏特有问题：进行中的对局

Web 服务的容灾切换相对简单，因为大多数操作是无状态的短请求。但游戏中进行中的对局是有状态的长会话，宕机时：

1. **帧同步游戏**：游戏状态在服务端（帧指令序列）。服务端宕机后，客户端失去连接，整局对局通常需要判负/重算，或者将对局数据视为作废（结算给所有玩家保底奖励）。
2. **状态同步游戏**（MMO 副本）：副本状态（怪物 HP、玩家位置）存在服务端内存中。宕机后数据丢失，玩家重新登录时副本通常需要重置（从检查点恢复或完全重置）。
3. **棋牌/卡牌类**：每一步操作通常持久化到数据库，宕机后可以从最后一个持久化步骤恢复，但需要额外的"对局状态重建"逻辑。

**最低可接受方案**：对局中宕机时，对所有参与玩家发放"对局异常补偿"（货币/道具），而不是尝试恢复对局。这是大多数游戏的实际做法。

**进阶方案**：对局状态检查点（Checkpoint）——每隔 30-60 秒将对局关键状态持久化到 Redis 或数据库。宕机恢复后，从最近检查点重建对局。实现复杂，适合高价值对局（赛事对局、高级别竞技排名赛）。

```python
# 对局检查点伪代码
class MatchCheckpointer:
    def __init__(self, redis_client, match_id: str):
        self.redis = redis_client
        self.match_id = match_id
        self.checkpoint_interval = 30  # 秒

    async def save_checkpoint(self, game_state: GameState):
        key = f"match:{self.match_id}:checkpoint"
        data = {
            'tick': game_state.current_tick,
            'players': serialize_players(game_state.players),
            'entities': serialize_entities(game_state.entities),
            'timestamp': time.time()
        }
        await self.redis.set(key, json.dumps(data), ex=3600)  # 1小时过期

    async def load_checkpoint(self) -> Optional[GameState]:
        key = f"match:{self.match_id}:checkpoint"
        data = await self.redis.get(key)
        if data:
            return deserialize_game_state(json.loads(data))
        return None
```

## 工程边界

### RTO/RPO 与成本的现实约束

没有无限预算可以实现 RTO=0、RPO=0。典型游戏公司的实际情况：

- **小型独立游戏**：全量备份 + 每日一备已足够，重要数据（充值）走主从复制。RTO 4 小时，RPO 1 小时。
- **中型网游**：主从复制（同城）+ binlog 异地归档。RTO 30 分钟，RPO 5 分钟。
- **头部竞技/MMO**：多区域主备（Active-Passive）+ 实时 binlog 复制。RTO 5 分钟，RPO < 1 分钟。

**投入产出边界**：将 RPO 从 5 分钟压到 1 分钟，需要显著增加数据库同步频率和存储成本，但玩家感知的差异可能不大（两种情况下玩家都会感到不满并申请补偿）。真正的投入应该放在减少故障发生频率和加速故障检测上，而不是无限压缩 RPO。

## 最短结论

灾难恢复不是备份做了就完事，定期演练并量化实测 RTO/RPO 与目标的差距，是唯一能在真实宕机时不手忙脚乱的方法。
