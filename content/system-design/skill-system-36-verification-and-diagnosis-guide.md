> "技能不对"不是诊断信息。真正的诊断从把"不对"分到具体的层开始。

技能系统的 Bug 报告有一个共同特征：描述都很模糊。

"这个技能打不出来。"
"伤害好像不对。"
"联网之后技能就不正常了。"

这些描述的信息量几乎为零，因为"不对"可以落在 6 个完全不同的层。
如果拿到 Bug 之后直接开始猜，大概率会在错误的层里浪费大量时间。

这篇的目标是给出一套**分层诊断流程**——先把"不对"归到具体的层，再在那个层里用对应的工具和方法定位根因。
不是概念解释，是一个可以直接拿来用的诊断参考。

---

## 这篇要回答什么

这篇主要回答 4 个问题：

1. "技能不对"到底可能在哪 6 层出问题。
2. 每层用什么方法验证。
3. 每层最常见的误判是什么。
4. 怎样从表面症状快速收敛到具体层。

---

## 分层诊断总览

先给一套最小决策树：

```
"技能不对"
  └→ 技能能不能放出来？
        ├→ 不能 → [Level 0: 校验层]
        │           └→ 检查 CanActivate 返回值和失败原因
        └→ 能 → 技能放出来之后命中了吗？
              ├→ 没命中 → [Level 1: 目标/碰撞层]
              │             └→ 可视化碰撞范围 + targeting 日志
              └→ 命中了 → 伤害数值对不对？
                    ├→ 不对 → [Level 2: Effect/Modifier 层]
                    │           └→ 打印 Modifier 聚合链
                    └→ 对 → 表现是否匹配逻辑？
                          ├→ 不匹配 → [Level 3: 表现层]
                          │             └→ 逻辑/表现时间线对比
                          └→ 匹配 → 多人联网时一致吗？
                                ├→ 不一致 → [Level 4: 同步层]
                                │             └→ 三端事件对比
                                └→ 一致 → 下次改完还能正常吗？
                                      └→ [Level 5: 回归层]
                                            └→ CI 测试 + 配置校验
```

6 层的核心逻辑：每一层以前一层"正常"为前提。
如果校验层就出了问题，不需要往后查。
如果单机一切正常但联网有问题，直接跳到同步层。

下面逐层展开。

---

## Level 0：校验层 — 技能能不能放出来

这是最前置的问题。技能按了没反应，在确认输入本身正常的情况下，几乎必然是校验层的问题。

### 检查方法

1. 打开技能校验的 Debug 日志（通常需要一个编译开关或运行时 Flag）。
2. 打印 `CanActivate` 的返回值和具体失败原因。
3. 逐条检查校验链：冷却、资源、Tag 阻塞、状态限制。
4. 如果连 `CanActivate` 都没有被调用到，回退到输入层——检查输入请求是否到达了技能系统。

### 日志关键字

```
[Skill] CanActivate failed: CooldownNotReady (remaining=1.2s)
[Skill] CanActivate failed: ResourceInsufficient (mana=10, cost=50)
[Skill] CanActivate failed: TagBlocked (has=Silence, blocks=Skill.Active)
[Skill] InputRequest dropped: buffer window expired (age=320ms, max=200ms)
```

### 常见误判

- **"技能放不出来所以是 CD 的 bug"**。实际上经常是 Silence Tag 或类似的状态阻塞标签没有被正确移除。CD 是校验链中最容易被直觉锁定的环节，但它恰恰通常是正确的——出问题更多的是 Tag 阻塞和状态条件。
- **"按键没反应所以是输入系统的 bug"**。输入系统可能完全正常，问题出在输入缓冲窗口的配置：请求到了，但到达的时间点不在合法窗口内，被静默丢弃了。这种情况下输入层日志没有异常，校验层也没有收到请求，看起来像是"什么都没发生"。
- **"校验通过了但技能没放出来"**。如果 `CanActivate` 返回 true 但技能确实没有执行，问题在校验层之后——通常是 Activate 流程本身的某个环节断了（比如 Montage 没有找到、Ability 的 Activate 逻辑内部 early return 了）。这已经不是校验层的问题。

### 修复方向

- 输入缓冲的配置和机制 → 04 篇（输入与施法请求）
- 校验链的完整设计 → 05 篇（校验与约束）

---

## Level 1：目标/碰撞层 — 放出来但没命中

技能成功释放了，但没有产生任何命中结果。

### 检查方法

1. 开启碰撞范围的可视化调试（DrawGizmo / Debug Visualization）。在运行时画出技能的实际碰撞区域，和你以为的区域做对比。
2. 打印 targeting 的完整结果：候选目标列表、过滤条件、最终通过的目标。
3. 对于弹道类技能，打印弹体的轨迹和碰撞事件。
4. 检查碰撞的 Layer Mask 配置——这是最高频的配错项。

### 日志关键字

```
[Skill] Targeting: candidates=5, afterFilter=0, reason=LayerMismatch
[Skill] Targeting: 0 targets found (area center=(3.2, 0, 5.1), radius=4.0)
[Skill] HitResult: no valid targets in range
[Skill] Projectile: spawned at T=0, no collision detected after 3.0s, destroyed
```

### 常见误判

- **"碰撞检测有 bug"**。碰撞检测本身极少有 bug（用的是引擎的物理系统），绝大多数情况下是 Layer Mask 配错了。攻击方的技能碰撞层和被攻击方的物理层不在同一个 Mask 组里，碰撞系统正确地返回了"没碰到"。
- **"范围技能打不到"**。如果是范围检测（球形、扇形）打不到目标，第一件事不是查半径，而是查高度差。很多范围技能的碰撞检测在 XZ 平面上是对的，但目标站在台阶上或不同的 Y 高度，不在碰撞体积内。
- **"弹道方向不对"**。确认弹体的发射方向是跟随准心、角色朝向、还是其他参考系。方向参考系选错了，弹体就是射向另一个方向，碰撞检测无从命中。

### 修复方向

- 目标选择、碰撞过滤和命中判定的完整设计 → 06 篇（目标选择与命中）

---

## Level 2：Effect/Modifier 层 — 命中了但伤害不对

技能命中了目标，但最终造成的效果（伤害、Buff 数值、状态变化）和预期不一致。

### 检查方法

1. 打印 Effect 的完整 Apply 链：原始属性值 → 所有参与的 Modifier 及其类型 → 最终输出值。
2. 对照设计公式，手算一遍预期值。然后把每一步的实际值和手算值对比，找出第一个分歧点。
3. 检查 Modifier 的聚合顺序：加法和乘法的执行顺序对最终结果影响很大。
4. 如果有 Buff 叠加，检查叠加层数是否正确、是否存在上限溢出。

### 日志关键字

```
[Effect] Apply: source=PlayerATK(120), target=EnemyDEF(30)
[Modifier] Chain: Add(+50) → Mul(×1.5) → Add(+20) → Final=275
[Effect] Damage result: raw=275, afterDEF=245, afterArmor=210, final=210
[Buff] Stack overflow: max=5, attempted=6, clamped to 5
```

### 常见误判

- **"公式写错了"**。公式本身通常是对的，更常见的情况是 Modifier 聚合顺序不符合设计预期。同一批 Modifier 里的加法和乘法，执行顺序不同结果完全不同。`(100 + 50) × 1.5 = 225` 和 `100 × 1.5 + 50 = 200`，差了 25。
- **"数值太大了，肯定溢出"**。先别急着找溢出，更多情况下是 Buff 叠加出了问题。一个本应独占的 Buff 叠了多层，或者叠加上限没有生效，每多一层数值就多算一倍。可以参考 33 篇中的第一类坑型分析。
- **"是不是取了错误的属性"**。确认 Effect 读取的属性来源是正确的——是施法者的攻击力还是目标自己的攻击力？是基础值还是当前值（已包含 Buff 加成）？属性来源选错，后面所有计算都会偏。

### 修复方向

- Effect 系统的完整结构 → 07 篇（效果系统）
- Buff/Modifier 的设计与聚合规则 → 08 篇（Buff/Modifier）
- 常见数值类坑 → 33 篇（坑型录）

## Level 3：表现层 — 伤害对了但表现不对

逻辑层的数值完全正确，但玩家看到的视觉/音频表现和逻辑不一致。

### 检查方法

1. 把逻辑层的战斗事件日志和表现层的播放日志放在同一个时间线上对比。找出两者的时间差。
2. 检查 AnimEvent 是否在正确的动画帧触发。动画文件被替换或修改后，事件帧可能偏移。
3. 检查 VFX 的生命周期：Spawn 时间、挂接点、销毁时间。
4. 检查音效的触发时机是否匹配。

### 日志关键字

```
[Presentation] AnimEvent 'DamagePoint' fired at frame 18, expected frame 12
[Presentation] Event queue: LogicDamage T=0.20, VFXSpawn T=0.35 (delta=150ms)
[VFX] Spawn failed: socket 'Weapon_Tip' not found, fallback to root
[VFX] Particle destroyed at T=0.8, expected lifetime=2.0 (pool recycled early)
```

### 常见误判

- **"动画时间不对"**。逻辑层和表现层用的时间基准经常不是同一个。逻辑层用固定步长（比如 20ms/tick），表现层用 `deltaTime`。如果不做时间对齐，两边的"第 0.3 秒"可能差几十毫秒，帧率波动时差距更大。
- **"VFX 没播"**。VFX 系统本身工作正常，只是 VFX 挂接点（Socket）的名字改了。美术修改了骨骼或重新导出了模型，Socket 名字从 `Weapon_Tip` 变成了 `weapon_tip` 或 `WeaponTip`，挂接失败后 VFX 要么没出现，要么出现在了错误的位置。
- **"表现延迟了"**。如果表现始终比逻辑晚一个固定量（比如固定晚 1 帧），通常不是 Bug 而是表现层有一帧的 pipeline 延迟。这个延迟是设计上可接受的还是需要补偿，取决于项目对同步精度的要求。

### 修复方向

- 逻辑层和表现层的解耦方案 → 09 篇（动画与表现解耦）

## Level 4：同步层 — 单机对了但联网不对

单机模式下一切正常，一到联网就出现不一致。

### 检查方法

1. 同时开两个客户端和一个服务端（或权威端），三端同时录制日志。
2. 把三端的技能事件时间线对齐（用服务端时间戳做基准）。找出第一个分歧事件。
3. 检查预测/确认/回滚日志。如果用了客户端预测，确认 PredictionKey 的分配和确认流程是否完整。
4. 检查关键状态的快照：HP、位置、Buff 列表。在分歧事件之前，三端的快照是否一致。

### 日志关键字

```
[Sync] PredictionKey allocated: K=42, ability=Fireball
[Sync] Server confirmed: K=42, result=Hit, damage=210
[Sync] Server rejected: K=42, reason=TargetAlreadyDead, rollback initiated
[Sync] Desync detected: client HP=80, server HP=100, delta=20
[Sync] State correction applied: position snapped from (3,0,5) to (3,0,4.2)
```

### 常见误判

- **"网络延迟太大"**。延迟是不同步的表现条件，不是根因。高延迟只是放大了已有的不同步问题。在低延迟下测试也能复现的不同步，和延迟无关。可以参考 34 篇的具体案例。
- **"是不是丢包了"**。丢包是网络层的事，但大部分技能层面的不同步不是丢包导致的。如果你的同步协议在 TCP 上跑，丢包由传输层保证重传，根本到不了技能逻辑。如果在 UDP 上跑，也应该有应用层的确认机制。真正的丢包导致的问题看起来像"事件缺失"而不是"数值不同"。
- **"客户端预测有问题"**。先确认问题是预测回滚不正确，还是根本没有预测。如果某个技能效果在客户端完全没有预测（等服务端确认后才执行），那不是"预测有问题"，而是"没做预测所以有延迟感"。两者的修复方向完全不同。

### 修复方向

- 多人同步的完整架构 → 11 篇（多人同步）
- 具体的不同步案例分析 → 34 篇（不同步案例）

## Level 5：回归层 — 都对了但下次改完又坏了

当前版本一切正常，但每次有人改了配置或代码之后就出问题。这不是运行时 Bug，而是工程保障缺失。

### 检查方法

1. CI 上的技能回归测试是否通过。如果 CI 里没有技能相关测试，这本身就是问题。
2. 公式夹具的 expected value 是否跟着策划配置更新了。策划改了技能参数但测试用例的预期值没更新，测试要么报假阳性（正确结果被判为失败），要么报假阴性（错误结果碰巧匹配了旧预期值）。
3. 配置校验器是否覆盖了最近的变更。新增的 EffectSpec 或 Modifier 引用是否在校验器的检查范围内。

### 日志关键字

```
[CI] Skill regression test FAILED: 3/50 cases
[CI] FAIL: Fireball damage expected=210, actual=275 (config updated, fixture stale)
[Validator] Config broken reference: EffectSpec_042 references ModifierDef_099 (deleted)
[Validator] New config entry SkillDef_New_003 not covered by any test case
```

### 常见误判

- **"测试环境和线上不同"**。先检查最基本的一件事：测试用的配置文件和线上版本是不是同一份。配置不同步是回归测试失灵的头号原因。
- **"测试用例过时了"**。这个判断本身通常是对的，但重点不是"去更新用例"，而是"为什么用例没有自动跟随配置更新"。如果每次改配置都要手动更新对应的测试用例，这个流程迟早会断。可持续的方式是让测试夹具从配置自动生成预期值，或者至少在配置变更时自动标记受影响的测试用例。
- **"CI 太慢了所以没人跑"**。如果回归测试耗时太长导致开发者绕过，问题不在"CI 太慢"而在回归测试的粒度设计。技能回归测试应该有两档：一档是秒级的公式验证（每次提交跑），一档是分钟级的完整流程测试（每日跑或合入主干前跑）。如果只有一档完整测试且耗时 30 分钟，开发者跳过是必然的。

### 修复方向

- 测试策略和回归保障 → 13 篇（测试与回归）

| 症状 | 最先查 | Level | 参考篇目 |
|------|--------|-------|---------|
| 技能按了没反应 | 校验日志（CanActivate 返回值） | 0 | 04 / 05 |
| 放出来但打不中 | 碰撞可视化 + Layer Mask 配置 | 1 | 06 |

使用方式：拿到一个"技能不对"的 Bug，先从这张表的第一行开始，逐行排除。大部分问题会在前 3 层就定位到。

## 跨层症状：看起来是 A 层，根因在 B 层

有些症状看起来属于某一层，但根因在另一层。这类问题是分层诊断中最容易绕弯路的地方。

列几个实际项目中遇到过的典型例子：

### "伤害为零" — 看起来是 Level 2，实际是 Level 1

症状：技能播了动画和 VFX，但目标血量没变化。
直觉判断：Effect 没有正确 Apply，去查 Modifier。
实际根因：技能的命中检测没有返回任何有效目标。Effect 确实执行了 Apply 流程，但 `targets` 列表为空，所以什么都没发生。表现层的动画和 VFX 是预播放的，不依赖命中结果。

定位方法：先打印 HitResult 的目标数量。如果为 0，问题不在 Level 2，回退到 Level 1。

### "技能在特定时机放不出来" — 看起来是 Level 0，实际是 Level 3

症状：某个技能在连招中放不出来，其他时候正常。
直觉判断：校验条件在连招中不满足，去查校验链。
实际根因：前一个技能的动画还没播完，AnimEvent 中的"允许取消"事件还没触发，输入被正确地缓冲了但等待窗口过期后被丢弃。问题在表现层的动画事件时机配置，不在校验层。

定位方法：检查 `CanActivate` 是否被调用。如果根本没被调用，问题在输入或表现层的门控逻辑，不是校验层本身。

### "联网伤害不对" — 看起来是 Level 4，实际是 Level 2

症状：两个客户端看到的伤害数字不一样。
直觉判断：同步出了问题。
实际根因：两个客户端的 Buff 状态本身就不同。客户端 A 有一个本地 Buff 没有同步到客户端 B，导致两端用不同的 Modifier 列表计算伤害。根因在 Buff 同步的遗漏，属于 Level 2（Buff 管理）和 Level 4（同步）的交叉地带。

定位方法：先在服务端确认权威伤害值，再分别和两个客户端对比。如果服务端值和其中一个客户端一致，另一个不一致，查不一致那端的 Modifier 列表差异。

## 诊断的两个反模式

在给出结论之前，单独提两个在实际项目中反复出现的诊断反模式。

### 反模式一：跨层猜测

典型场景：技能打不出来（Level 0 的问题），但开发者直接去查网络同步（Level 4）。

原因是"上次那个 Bug 是同步的问题"，于是这次也先查同步。
这是用经验替代流程。分层诊断的核心价值就是避免跨层猜测——每一层的前提是前一层正常，如果校验层还没确认通过，后面的层根本不需要看。

### 反模式二：只查日志不做对比

打印了一堆日志，但只看单端的。

同步类问题只看客户端日志永远看不出来。表现类问题只看逻辑层日志也看不出来。
对比是诊断的核心手段：逻辑和表现对比、客户端和服务端对比、预期值和实际值对比。
单端日志只能告诉你"发生了什么"，对比才能告诉你"哪里不对"。

### 反模式三：修好了但不知道为什么

改了一个值，Bug 消失了，就提交了。

这种修复方式有两个风险。第一，你可能只是掩盖了症状。比如把碰撞范围从 4 改到 6，"打不到"的 Bug 消失了，但根因是 Layer Mask 配错——下一个半径为 3 的技能还会出同样的问题。第二，你没法写有效的回归测试，因为你不知道什么条件会让它再次出现。

分层诊断的产出不应该只是"修好了"，而应该是"根因在 Level X 的 Y 环节，修复方式是 Z"。这个信息才是回归保障的基础。

---

## 这篇真正想留下来的结论

技能系统的 Bug 定位之所以困难，不是因为每一层特别复杂，而是因为层数多且表面症状相似。

"技能不对"可以是 6 层中任意一层的问题，但开发者往往凭直觉直接跳到自己最熟悉的那一层查，跳对了算运气好，跳错了就浪费半天。

分层诊断流程的价值在于：

1. **先定层，再定因**。把"不对"分到具体的层，缩小搜索空间。
2. **每层有固定的检查方法和日志关键字**。不依赖经验和直觉，新人也能执行。
3. **层间有前提关系**。前一层正常才需要查下一层，避免无效排查。

最后一个补充：这篇列出的所有日志关键字，前提是你的技能系统已经埋了这些日志。如果你的系统现在还没有分层日志，那在遇到第一个难定位的 Bug 之前，先把每一层的关键日志加上。事后加日志的成本远高于事前——因为事后你还要重现 Bug 才能验证日志是否有效。

这篇不是一次性阅读的文章，更适合作为一个随时可以回来查的诊断参考。
当你下次拿到一个"技能不对"的 Bug 报告时，打开这篇，从 Level 0 开始逐层检查，通常不会超过 3 层就能定位到问题所在的层——然后去那一层对应的系列篇目里找具体的解决方案。
