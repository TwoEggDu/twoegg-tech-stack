> LoL 的技能系统不是技术上最先进的，但它证明了一件事：`技能系统怎样在 10 年 Live Service 中持续演化而不崩。`

很多人讨论技能系统架构时，喜欢聊 GAS 的 Tag 系统有多灵活、Dota 2 的数据驱动有多优雅。

但如果把视角拉到工程运营层面，英雄联盟面对的约束是另一个量级：

- 160+ 英雄，每个英雄 4 个主动技能 + 被动，加上装备主动效果、召唤师技能、符文效果
- 每两周一次 patch，每次 patch 可能涉及 20-40 个英雄的数值调整
- 全球多区域部署，要求所有区域的比赛结果在相同规则下可复现
- 竞技公平性是底线——任何客户端可以绕过的逻辑都不能信任

这些约束叠加在一起，决定了 LoL 的技能系统架构不是"怎样做得更酷"，而是"怎样在持续变化中不出事"。

Riot 在多次 GDC 分享中（包括 "Evolving the Server-Side Architecture" 等演讲）提到了他们在这个过程中踩过的坑和做出的选择。这篇就从这些公开信息出发，拆解 Live Service 下技能系统的核心工程决策。

## 这篇要回答什么

1. 为什么 MOBA 选择服务端权威模型，客户端只做表现预测。
2. 技能脚本化经历了怎样的演化路径，最终为什么走向数据驱动 + 脚本混合。
3. 每两周一次 patch 的数据管道具体怎样运作。
4. 冠军设计流程中，技能系统的工程约束怎样影响设计决策。
5. 反作弊在技能层有哪些具体体现。
6. 这些经验对自研系统有什么可借鉴的。

## 服务端权威模型：为什么 MOBA 不做客户端预测

### 核心选择

LoL 从架构上选择了纯服务端权威模型：所有技能逻辑（命中判定、伤害计算、Buff 应用、CD 管理）全部在服务端执行，客户端不参与任何游戏逻辑的计算。

这和 FPS 游戏的常见做法正好相反。

FPS 类游戏（如 Overwatch、Valorant）通常会做客户端预测 + 服务端回滚，因为射击手感对延迟极其敏感，玩家需要"按下开火的瞬间就看到结果"。

MOBA 不需要这个。

原因很直接：

- **操作粒度不同**。MOBA 的技能释放是离散事件（点击 → 施法前摇 → 效果生效），不是连续输入流。30ms 的输入延迟在施法前摇（通常 0.2-0.5 秒）面前几乎不可感知。
- **公平性优先级极高**。客户端预测意味着客户端需要运行游戏逻辑的副本，这就给了作弊者可乘之机。对于电竞项目，这个风险不可接受。
- **状态复杂度高**。10 个英雄同时可能有几十个 Buff、光环、被动效果在叠加，客户端要精确复制这些状态几乎不可能不出错。

### 客户端做什么

客户端不运行逻辑，但它不是纯粹的"终端显示器"。

客户端做的事情是**表现预测**：

- 玩家按下技能键后，客户端立刻播放施法动画和粒子效果，不等服务端确认
- 移动指令发出后，客户端立刻开始移动角色，不等服务端回包
- 弹道类技能发出后，客户端立刻显示弹道飞行

但所有这些都只是"视觉预测"。实际的命中判定、伤害数字、状态变化，全部来自服务端的后续确认包。

如果服务端结果和客户端预测不一致（比如技能被打断、目标进入无敌状态），客户端会用插值或淡出的方式修正表现，而不是硬切。

### 代价

纯服务端权威的代价是显而易见的：

- **最低延迟受限于网络 RTT**。玩家的操作到看见结果，至少有一个 RTT 的延迟。这也是为什么 LoL 对网络质量要求较高。
- **服务端计算压力大**。所有 10 个玩家的所有技能逻辑都在服务端跑，一场比赛中高峰期每秒可能有数百次技能相关的逻辑计算。
- **表现层修正需要额外工程**。客户端需要一套完整的"预测 → 确认 → 修正"管线，这本身就是不小的工作量。

但对于 LoL 这样的竞技游戏，这些代价是值得的。

用 Riot 自己的说法：宁可让所有玩家都感受到 30ms 的延迟，也不能让任何一个玩家有机会通过修改客户端获得优势。

## 技能脚本化与热更新演化

### 早期：硬编码时代

LoL 最初的技能实现方式很直接——C++ 硬编码。

每个英雄的每个技能就是一段 C++ 代码。施法逻辑、伤害公式、特殊效果，全部写在代码里。

这种方式在英雄数量少的时候没什么问题。但当英雄数量开始增长，问题很快暴露：

- **编译部署成本高**。改一个技能的数值就要重新编译、部署服务端。
- **策划无法独立调试**。任何技能调整都需要程序员介入。
- **跨英雄复用困难**。相似的技能逻辑在不同英雄间重复实现。
- **patch 风险高**。动任何一段代码都可能影响其他英雄。

### 中期：脚本化

Riot 逐步将技能逻辑从 C++ 迁移到脚本层。

这个过程不是一步到位的，而是渐进式的：

- 先把数值（伤害、冷却、范围等）从代码里拆出来，做成可配置的数据
- 然后把简单的逻辑（比如"对目标造成伤害并施加减速"）用脚本表达
- 最后把更复杂的逻辑（比如多阶段技能、条件触发）也移到脚本

Riot 使用的脚本系统经过了多次迭代。从公开信息来看，他们内部使用了一套基于 Lua 或类似语言的脚本方案，服务端在运行时解释执行这些脚本。

脚本化带来的直接好处：

- 策划可以在不重新编译的情况下修改技能行为
- 新英雄的开发周期缩短
- 技能逻辑可以在不停服的情况下热更新（受限于脚本层能表达的范围）

### 当前：数据驱动 + 脚本混合

经过多年演化，LoL 的技能系统最终走向了一个分层架构：

**数据层**：纯数值配置。伤害、冷却、施法距离、持续时间等。这一层可以随时热更新，不需要任何代码变更。

**脚本层**：行为逻辑。技能的施法流程、目标选择规则、多阶段逻辑、条件触发。这一层可以通过脚本更新，但需要更谨慎的测试。

**引擎层**：底层系统。命中检测、物理模拟、寻路、视野计算。这一层是 C++ 实现的，变更需要完整的发版流程。

这三层的更新频率和风险等级递增：

| 层级 | 更新方式 | 频率 | 风险 |
|------|---------|------|------|
| 数据层 | 热更新配置 | 随时（hotfix） | 低 |
| 脚本层 | 脚本部署 | 双周 patch | 中 |
| 引擎层 | 编译部署 | 赛季大版本 | 高 |

### 能力预算

Riot 在 GDC 分享中提到的一个重要概念是**能力预算（Complexity Budget）**。

每个新英雄的技能不是"技术上能做到就可以做"，而是要评估：

- 这个技能需要多少独特的系统支持？
- 后续平衡调整的维护成本有多高？
- 出 Bug 的时候排查难度有多大？
- 这个机制是否可以被未来的英雄复用？

能力预算不是一个精确的数字，而是一个评估框架。

每个赛季会有一定的"预算"用于引入新机制（比如新的控制类型、新的位移逻辑、新的资源系统），超出预算的设计要么简化，要么推迟。

## 平衡数据管道

### 双周 patch 的数据流

LoL 的 patch 节奏是大约每两周一次。一个 patch 的数据流大致如下：

**设计师调数阶段**（patch 周期第 1-5 天）

- 平衡团队分析上一个 patch 的数据：各英雄胜率、选取率、Ban 率、对局数据
- 确定本次 patch 的调整目标
- 设计师在内部工具中修改数值，生成 changelist

**内部测试阶段**（第 5-8 天）

- 变更部署到内部测试环境
- QA 团队验证所有改动是否按预期生效
- 自动化测试检查是否有回归问题（比如改了 A 英雄的数值，导致 B 英雄的某个交互出 Bug）
- 平衡团队在内部对局中验证手感

**PBE 阶段**（第 8-11 天）

- 变更部署到 PBE（Public Beta Environment）服务器
- 玩家社区测试，收集反馈
- 如果发现严重问题，可以在这个阶段回滚或修改

**正式上线**（第 11-14 天）

- 最终版本部署到全球所有正式服务器
- 部署后监控异常数据

### 热更新数值 vs 需要 patch 的逻辑

不是所有改动都需要等双周 patch。

**可以随时热更新的**：

- 数值调整（伤害、冷却、法力消耗、持续时间等）
- 某些简单的行为开关（比如临时禁用某个技能效果）

这类改动走的是数据层热更新通道，可以在几小时内部署到正式服。这就是 Riot 发 "hotfix" 的方式——通常在 patch 上线后 1-2 天，如果发现某个英雄数值明显偏强或偏弱，会用 hotfix 快速调整。

**必须等 patch 的**：

- 技能机制变更（比如把点选技能改成方向技能）
- 新增或移除技能效果（比如给技能加一段位移）
- 交互逻辑修改（比如修改某个技能与特定道具的交互方式）

这类改动涉及脚本层甚至引擎层的变更，需要完整的测试流程。

这个区分直接影响了技能系统的架构设计——数值层必须和逻辑层干净地分离，否则"快速 hotfix"就不可能实现。

## 冠军设计流程

### 从概念到上线

一个新英雄从概念到正式上线，通常需要一年甚至更长时间。流程大致分为：

**概念阶段**

- 确定英雄的主题方向（比如"影子刺客"、"虚空战士"）
- 定义英雄在游戏中的定位（位置、角色、独特性）
- 初步构思技能组（Kit）的设计方向

**原型阶段**

- 用快速原型验证核心机制
- 在这个阶段，技能实现可能非常粗糙——用现有英雄的技能拼凑来测试手感
- 关键目标是验证"这个机制好不好玩"，而不是"技术上怎么实现"

**迭代阶段**

- 技能逐步从原型走向正式实现
- 和美术、音效团队协作，加入表现层
- 反复平衡测试

**PBE 阶段**

- 部署到 PBE，接受玩家测试
- 根据反馈进行最后调整

**正式上线**

- 通常在一个大 patch 中上线
- 上线后密切监控数据，准备 hotfix

### 为什么酷炫技能会被砍

这是一个经常被玩家质疑的问题：为什么有些看起来很酷的技能设计最终没有上线？

答案通常不是"做不出来"，而是**维护不了**。

Riot 的设计师在多次分享中提到，一个技能被砍通常是因为：

**交互组合爆炸**

160+ 英雄意味着一个新技能需要和所有现有技能正确交互。一个看起来简单的"反弹弹道"机制，需要测试它和每一个阻挡类、穿透类、吸收类技能的交互。测试矩阵是 N 的平方级别。

**平衡性不可控**

有些机制在概念上很酷，但数值调整空间极窄。要么太强要么太弱，中间没有合理的平衡点。这类机制在 Live Service 中是灾难，因为每次环境变化都可能让它突然变得极端。

**Bug 排查成本高**

越独特的机制，出 Bug 时越难定位。如果一个技能用到了其他技能都不用的底层接口，那么这个接口出问题时，只有这一个英雄受影响，但排查可能需要深入引擎层。

**复用价值低**

如果一个复杂机制只用在一个英雄身上，它的实现和维护成本就全部分摊在这一个英雄上。Riot 更倾向于投入资源实现可以被多个英雄复用的系统（比如通用的弹道系统、通用的护盾系统）。

这就是能力预算的实际体现——不是"能不能做"，而是"值不值得在 10 年 Live Service 中持续维护"。

## 反作弊在技能层的体现

### 服务端权威是基础

前面讲的服务端权威模型本身就是最重要的反作弊手段。因为客户端不运行游戏逻辑，所以大部分"修改技能效果"类的作弊在架构层就被阻止了。

但服务端权威不能解决所有问题。

### 自动瞄准（Scripting）检测

Scripting 是 LoL 中最常见的作弊类型之一。它不修改游戏逻辑，而是通过读取客户端内存中的游戏状态（敌方位置、技能 CD、弹道轨迹等），自动计算最优操作时机和方向。

服务端检测 Scripting 的思路：

- **输入模式分析**。人类玩家的反应时间、操作精度有自然波动。如果一个玩家持续以超人类的反应速度精确释放技能，统计上就是异常的。
- **预判准确率监控**。方向技能（如 Ezreal 的 Q）需要预判目标移动方向。人类的预判准确率有上限，持续超出这个上限就是可疑的。
- **操作间隔分析**。脚本通常有固定的执行间隔或者过于稳定的操作节奏，这和人类的随机性不同。

这些检测不是实时阻止，而是后台统计分析，积累足够证据后进行封禁。

### CD 篡改校验

虽然客户端不运行逻辑，但仍然有人尝试通过修改客户端发送的数据包来绕过冷却。

服务端的校验很简单——因为所有 CD 管理都在服务端，所以客户端发来的"释放技能"请求，服务端只需要检查这个技能当前是否在 CD 中。如果在 CD 中，请求直接丢弃。

客户端能做到的只是在本地显示上骗过玩家自己（比如隐藏 CD 显示），但无法实际影响游戏逻辑。

### 战争迷雾

战争迷雾是另一个关键的反作弊点。

LoL 的战争迷雾是**服务端控制的信息过滤**：

- 服务端知道所有单位的位置
- 但服务端只向每个客户端发送该客户端有权看到的信息
- 在战争迷雾覆盖区域内的敌方单位信息根本不会发送到客户端

这意味着"透视"类作弊在数据层就被阻止了——客户端内存里根本没有被迷雾覆盖的敌方单位数据。

这个设计对服务端的压力不小。服务端需要为每个玩家（或每个队伍）单独计算信息过滤，相当于维护多份裁剪后的世界状态。但对于竞技游戏来说，这个代价是必要的。

### 回放与审计

LoL 的服务端会记录完整的比赛日志，包括每个时间点的所有输入和状态。这不仅用于回放功能，也用于赛后的作弊审计。

如果一个玩家被举报，系统可以回放整场比赛，从服务端视角检查该玩家的每一个操作是否存在异常模式。

## 对自研系统的启示

从 LoL 的经验中可以提炼出几个对自研技能系统有价值的设计原则。

### 可热更新数值层 + 可 patch 逻辑层

这是 LoL 技能系统架构最核心的分层思想：

```text
┌─────────────────────────────────────┐
│          数值层 (Data)              │  ← hotfix 级别，随时更新
│  伤害、冷却、范围、持续时间...       │
├─────────────────────────────────────┤
│          逻辑层 (Script)            │  ← patch 级别，双周更新
│  施法流程、目标选择、条件触发...     │
├─────────────────────────────────────┤
│          引擎层 (Engine)            │  ← 赛季级别，低频更新
│  命中检测、物理、寻路、视野...       │
└─────────────────────────────────────┘
```

自研系统从一开始就应该按这个分层设计。

具体来说：

- 数值层用纯数据配置（JSON / ScriptableObject / 配置表），部署时只需要替换数据文件
- 逻辑层用脚本或可序列化的行为树，部署时需要脚本解释器热加载
- 引擎层用编译语言实现，变更走完整发版流程

即使项目初期不需要频繁更新，这个分层也值得做。因为一旦进入运营期，没有这个分层就意味着每次调数值都要走完整的发版流程。

### 能力预算概念

大多数自研项目不会有 160+ 英雄的规模，但能力预算的思维方式仍然适用。

每当设计一个新的技能机制时，问自己三个问题：

1. **交互成本**：这个机制需要和现有多少个系统正确交互？每增加一个交互就增加了测试和维护成本。
2. **调参空间**：这个机制是否有足够的数值调整空间？如果数值敏感度极高（差 5% 就从不可用变成超模），后期平衡就会很痛苦。
3. **复用价值**：这个机制是否可以被其他角色 / 技能复用？如果只用一次，实现成本就全部分摊在一个技能上。

### 服务端权威的取舍

并非所有项目都需要 LoL 级别的服务端权威。

| 项目类型 | 建议模型 |
|---------|---------|
| 竞技 MOBA / RTS | 纯服务端权威 |
| PvE 合作（如 MMO 副本） | 服务端权威 + 有限客户端预测 |
| 单机 + 可选联机 | 客户端权威 + 服务端校验 |
| 纯单机 | 客户端权威 |

选择哪种模型取决于：公平性需求有多高、延迟容忍度有多大、服务端成本预算有多少。

### 从第一天就考虑"改"的成本

LoL 的技能系统架构不是为了"做出来"优化的，而是为了"改得动"优化的。这个思维方式值得所有 Live Service 项目借鉴：

- 技能配置能不能在不重新编译的情况下修改？
- 修改一个技能的数值需要几步操作、几个人参与？
- 出了平衡问题能不能在 1 小时内 hotfix？
- 新增一个技能机制需要改多少个文件？

如果答案是"改一个数值需要重新编译部署"，那就需要在技术选型阶段重新考虑架构。

---

## 结论

LoL 的技能系统给我们的最大启发不是任何一个具体的技术方案，而是一个工程理念：

**技能系统的核心挑战不是"做出来"，而是"改得动、查得清、扛得住"。**

改得动——Live Service 意味着技能参数和机制会持续变化，架构必须支持低成本的修改。查得清——160+ 英雄的交互组合意味着 Bug 排查空间巨大，系统必须有足够的可观测性。扛得住——竞技游戏不允许"偶尔出错"，服务端权威和反作弊必须在架构层保障公平性。

这三个"得"不是功能需求，而是架构约束。它们不会出现在策划案里，但会决定技能系统在第三年、第五年还能不能健康运转。对于正在设计自研技能系统的团队，LoL 的经验指向一个简单但容易被忽视的结论：在写第一行技能代码之前，先回答"这个系统两年后怎么改"。
