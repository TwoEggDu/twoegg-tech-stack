P01 到 P06 解决的是“物理世界怎么运转、数据怎么拆、查询怎么问、Baking 怎么转”。到了 P07，问题会变得很现实：**一旦物理慢了、抖了、穿了、命中错了，到底该先看什么**。

这篇不打算记工具清单。工具名和窗口名会随版本变化，但诊断顺序不会。真正有用的是先建立症状分类，再沿着证据链往下找：是固定步长的问题，还是接触对暴涨，还是 Query 过多，还是 Baking 数据就已经错了。

## 验证环境与口径

本文按 `Unity 6 / Entities 1.x / Unity Physics / Havok Physics` 的常见调试口径来写。工具窗口和统计项会随版本变化，所以本文优先讲诊断顺序，不把某个窗口名写成永远不变的真理。

这篇不展开：

- `P01` 的世界模型细节。
- `P03` 的 Query 选型细节。
- `P06` 的 Baking 转换细节。

## 先按症状分类，不先按工具分类

物理问题看起来很多，其实常见症状只有几类。

| 症状 | 更像哪类问题 | 第一反应不该是什么 |
|---|---|---|
| 帧时间突然升高 | 接触对、Broadphase、Query 过多 | 先换工具乱看 |
| 角色抖动或站不稳 | 固定步长、权威交接、查询时机 | 先改参数碰运气 |
| 穿透、错位、回弹异常 | 体积模型、事件回流、同步点 | 先怀疑渲染 |
| 命中偶发丢失 | Query 类型选错、回写时机不对 | 先加更多 Raycast |
| 只有某些场景出错 | Baking 数据、场景导入差异 | 先怪运行时主链 |

这张表的目的不是给答案，而是先把“问题属于哪一层”分开。因为调试最怕的不是找不到工具，而是把烘焙错误、查询错误和仿真错误混成一类。

## 诊断顺序：先复现，再看链路

比较稳的顺序是从便宜证据往贵证据走。

1. 先确认可复现，并把输入和场景固定住。
2. 再看是不是固定步长、系统顺序或者世界交接点出了问题。
3. 再看是 Query 过多、接触对暴涨，还是事件回流太密。
4. 最后才去看具体组件、具体体积和具体数据来源。

这个顺序很重要，因为很多团队会反着来：先打开一堆工具，再在所有统计项之间跳来跳去，最后只得到“看起来都可能有问题”。那不是诊断，那是信息噪声。

可以把它压成一条简单原则：**先确认问题属于哪条链，再去看这条链上的哪个环节爆了**。

## 最小证据链：记录每帧物理状态，而不是只记体感

下面这段伪代码只表达一件事：调试要留下结构化证据，不要只留下“我感觉抖了”。

```csharp
public struct PhysicsFrameProbe
{
    public int FixedStepIndex;
    public int BodyCount;
    public int QueryCount;
    public int ContactPairCount;
    public float StepMs;
}

public partial struct PhysicsProbeSystem : ISystem
{
    public void OnUpdate(ref SystemState state)
    {
        var probe = new PhysicsFrameProbe
        {
            FixedStepIndex = 0,
            BodyCount = 0,
            QueryCount = 0,
            ContactPairCount = 0,
            StepMs = 0
        };

        // 真实项目里，这里记录的是“症状证据”，不是为了追求完整 API 演示。
        // 同一段问题至少要能回答：
        // 1. 这一帧有多少物理体
        // 2. 做了多少查询
        // 3. 接触对是否异常上升
        // 4. 固定步长是否抖动
    }
}
```

这类记录的意义在于，你后面可以把“某个场景帧率差”拆成可比较的证据：是 Query 上去了，还是 contact pair 上去了，还是每个 fixed step 的耗时本来就不稳定。

## 边界与代价

P07 最容易踩的坑，是把调试当成“找一个万能窗口”。实际上，调试真正关心的是证据链，不是界面。

第二个坑，是把性能问题都归咎于单次查询。单次 `Raycast` 可能不贵，但一大批实体在热循环里发查询，再叠上回写和结构变更，就会把问题放大成链路问题。

第三个坑，是把抖动直接理解成渲染问题。很多时候抖动来自固定步长、权威交接、查询时机或者事件回流，而不是渲染层。

第四个坑，是把 Baking 错误和运行时错误混在一起。一个场景只在某些构建里错，优先看转换链；一个场景只在高负载时错，优先看运行时链。

## 常见误区

第一个误区，是只记工具名不记症状。工具是手段，症状才是入口。

第二个误区，是一上来就找“最强性能优化”。诊断的目标不是优化，而是先定位层级。

第三个误区，是把所有物理问题都往 Query 上推。很多问题其实出在固定步长、接触对、事件回流或者 Baking 数据。

第四个误区，是把一次排障经验当成永久真理。版本变了、包变了、场景变了，工具和窗口名都可能变，但“先复现、再分层、再找证据”的顺序不该变。

## 小结

P07 的核心不是教你背调试工具，而是让你按症状和证据链定位问题。

物理慢、抖、穿、错位，不要先猜结论，先看它属于固定步长、Query、接触对、事件回流还是 Baking 的哪一层。

下一步应读：`DOTS-P04｜CollisionEvents / TriggerEvents：命中事件怎样安全地回写 ECS 世界`

理由：当你能区分“主动查询”与“被动事件”之后，物理调试就能从一团噪声里拆出真正的回写链路。
