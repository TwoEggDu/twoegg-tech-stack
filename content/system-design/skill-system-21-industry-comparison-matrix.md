> 没有最好的技能框架，只有`最匹配你项目约束的方案`。这篇把 6 种方案放到同一张 8 维矩阵里，让选型有据可判。

技能系统选型是一个经常被简化的决策。

常见的讨论方式是列功能清单：这个框架支持 Buff 吗？支持多人同步吗？有编辑器吗？
然后选"功能最多"的那个。

但功能多不代表合适。

一个 3 人独立团队用 GAS 来做单机 Roguelike，和一个 40 人团队用自研框架做竞技 MOBA，面对的约束完全不同。
如果选型时没有把约束维度拆开，最后要么过度工程，要么在关键路径上发现框架不够用。

这一篇要做的事，就是把选型维度从"功能清单"升级到`架构和工程维度`，然后给出每种方案在每个维度上的具体判断。

---

## 这篇要回答什么

这篇主要回答 5 个问题：

1. 6 种技能系统方案各自的核心定位是什么。
2. 从 8 个架构和工程维度看，每种方案的优势和短板分别在哪。
3. 完整的 8×6 矩阵总表长什么样。
4. 给定一个项目类型，应该怎样做选型决策。
5. 什么时候该自研，什么时候不该。

---

## 6 种方案的一句话定位

在进入矩阵之前，先用一句话把每种方案的核心定位锚住。

| 方案 | 一句话定位 |
|------|------|
| **GAS（Unreal Gameplay Ability System）** | Unreal 引擎的官方技能框架，把 Ability、Effect、Attribute、Tag 和网络预测做成了引擎级基础设施 |
| **Dota 2 数据驱动** | Valve 为 MOBA 场景设计的声明式技能模型，用 KV 配置 + Modifier 事件驱动覆盖标准技能，Lua 兜底复杂逻辑 |
| **守望先锋（Overwatch）** | 暴雪为竞技 FPS 优化的 ECS 架构技能系统，以确定性模拟和服务器权威为第一优先级 |
| **LoL（League of Legends）** | Riot 为大规模 MOBA 运营设计的脚本化技能系统，强调热更新能力和策划迭代速度 |
| **PoE（Path of Exile）** | GGG 为极深度 Build 系统设计的组合求值引擎，核心挑战在组合空间管理而非单技能执行 |
| **Unity 自研** | 不依赖特定引擎框架，根据项目需求从零搭建，灵活性最高但工程量完全自负 |

这 6 种不是"同一类东西的 6 个版本"，而是`面向不同核心约束的 6 种回答`。

## 维度一：数据驱动程度

这个维度衡量的是：策划能在不改代码的前提下，配出多大比例的技能。

| 方案 | 判断 |
|------|------|
| **GAS** | 中等。GE 和 Modifier 可配置，但 Ability 逻辑通常需要 C++ 或蓝图，纯配置无法完成复杂技能 |
| **Dota 2** | 高。标准 MOBA 技能 30-40% 纯 KV 配置可完成，50-60% 需要 KV + Lua 混合，只有约 10% 必须碰 C++ |
| **守望先锋** | 低。ECS 架构下技能行为由 Component + System 硬编码，策划主要调数值参数，不配逻辑 |
| **LoL** | 高。技能逻辑用脚本语言编写，支持热更新，策划 / 技术策划可以直接写技能脚本 |
| **PoE** | 中高。Gem 的基础行为是代码定义的，但 Modifier 聚合规则和组合兼容性是数据驱动的 |
| **自研** | 取决于设计。可以做到任意程度，但数据驱动程度越高，前期框架投入越大 |

**核心判断**：数据驱动不是越高越好。如果项目技能数量少于 30 且行为高度差异化（比如竞技 FPS），硬编码反而更可控。数据驱动的收益在`技能数量大、模式可归纳`时才显现。

一个容易忽略的陷阱：数据驱动程度高，不代表策划迭代速度快。
如果配置语法复杂、缺乏校验工具、出错后排查困难，高数据驱动反而会拖慢策划的实际产出。
Dota 2 的 KV 配置就面临这个问题——配置表达力很强，但策划直接编辑文本文件，没有可视化反馈。

## 维度二：网络同步模型

这个维度衡量的是：框架对多人同步提供了什么级别的内置支持。

| 方案 | 判断 |
|------|------|
| **GAS** | 强。内置客户端预测、服务器权威确认、AbilityTask 的网络生命周期管理，是 6 种方案里多人支持最完整的 |
| **Dota 2** | 强但不可移植。Source 2 引擎提供完整的服务器权威 + 快照插值，但这是引擎级能力，不是技能框架层的 |
| **守望先锋** | 极强。确定性模拟 + 服务器权威 + 客户端预测回滚，为竞技 FPS 的严格同步需求定制 |
| **LoL** | 中强。服务器权威架构，客户端主要做表现，同步模型针对 MOBA 的回合感节奏优化 |
| **PoE** | 服务器权威。所有计算服务端完成，客户端只负责表现和输入，没有客户端预测 |
| **自研** | 零。网络同步层完全需要自己实现，这是自研方案里工程量最大的单项 |

**核心判断**：如果项目需要竞技级同步（FPS 对战、MOBA），这个维度的权重应该排在第一位。自研方案在这个维度上的劣势最大——一个可靠的预测 + 回滚系统，保守估计需要 2-3 个有经验的程序员投入 3-6 个月。

补充一个常被忽略的细节：GAS 的网络同步不是开箱即用的完美方案。
它的预测模型要求开发者正确处理 AbilityTask 的网络生命周期、正确区分 Local Predicted / Server Initiated / Server Only 的执行路径。
如果不理解这些边界就直接用，出来的网络行为会比不用预测还差。

## 维度三：效果组合能力

这个维度衡量的是：技能效果之间能以多复杂的方式互相修改和组合。

| 方案 | 判断 |
|------|------|
| **GAS** | 中等。GE 的 Modifier 支持多种聚合类型，Tag 系统提供条件组合，但没有 PoE 级别的组合深度 |
| **Dota 2** | 中等。Modifier 可以叠加和覆盖，但技能之间的组合关系是隐式的（通过 Modifier 交叉生效），没有显式组合层 |
| **守望先锋** | 低。英雄技能高度独立，组合主要靠英雄间配合而非技能系统内部的修改关系 |
| **LoL** | 中等。技能之间通过被动标记、状态交互产生组合，但深度有限，主要靠数值交互而非行为修改 |
| **PoE** | 极强。Support Gem 直接修改 Active Gem 的行为规则，Passive Tree 提供 1300+ 节点的修改来源，Keystone 改变计算规则本身 |
| **自研** | 取决于设计。可以做到任意深度，但组合层每深一级，测试和平衡成本都指数增长 |

**核心判断**：效果组合能力是技能系统最容易过度设计的维度。除非你的核心玩法就是"Build 构建"（ARPG、卡牌构建），否则中等组合能力通常够用。PoE 级别的组合深度需要配套的数值团队和长期平衡投入。

一个值得记住的经验法则：
组合深度每增加一级（从"效果独立"到"效果可叠加"到"效果可修改行为规则"），测试矩阵的规模大约增长一个数量级。
如果你的 QA 团队只有 2 个人，PoE 级别的组合深度是测不过来的。

## 维度四：运行时性能模型

这个维度衡量的是：框架在运行时的性能特征和瓶颈在哪。

| 方案 | 判断 |
|------|------|
| **GAS** | 中等。UObject 体系带来 GC 压力，Tag 查询在 Ability 数量多时有累积开销，但 C++ 底层保证了单次操作性能 |
| **Dota 2** | KV 解析层有开销，Lua 脚本层比 C++ 慢一个量级，但 MOBA 的实体数量有限（10 英雄 + 召唤物），规模不是瓶颈 |
| **守望先锋** | 强。ECS 架构天然缓存友好，确定性模拟避免了浮点数不一致带来的额外同步开销 |
| **LoL** | 脚本层有固有性能上限，但服务器只需处理 10 个英雄 + 小兵，实际负载可控 |
| **PoE** | Modifier 聚合是性能热点。300+ Modifier 的脏标记 + 按需重算策略是关键优化，全量重算不可持续 |
| **自研** | 完全取决于实现。可以针对项目特征做极致优化，但也可能因为缺乏经验写出性能灾难 |

**核心判断**：运行时性能在实体数量少（MOBA：10 英雄）时很少成为瓶颈。性能真正成为约束的场景是：大规模战场（100+ 实体同时释放技能）、高频 Modifier 聚合（PoE 模型）、或竞技 FPS 的每帧确定性模拟。

需要特别注意的是：脚本化方案（LoL 模式、Dota 2 的 Lua）的性能上限比编译型方案低一个量级。
在实体数量少的场景这不是问题，但如果你的项目可能扩展到大规模战场（比如 MMO 的公会战），脚本层会成为性能瓶颈。
这时候要么在关键路径上退回 C++/C#，要么一开始就不选纯脚本化方案。

## 维度五：工具链成熟度

这个维度衡量的是：策划和技术策划的日常工作流是否有配套工具支撑。

| 方案 | 判断 |
|------|------|
| **GAS** | 中高。Unreal Editor 本身提供蓝图可视化、数据资产编辑和调试工具，但 GAS 专用工具需要项目自己补（比如 GE 聚合可视化） |
| **Dota 2** | 中等。Workshop Tools 提供 KV 编辑和 Lua 调试，但没有可视化技能编辑器，策划直接编辑文本文件 |
| **守望先锋** | 高（内部）。暴雪有完整的内部工具链，但外部团队无法使用 |
| **LoL** | 高（内部）。Riot 有成熟的技能脚本编辑和测试工具，但同样是内部工具 |
| **PoE** | 中等（内部）。GGG 有内部数值工具和 Build 模拟器，但社区工具（如 Path of Building）填补了大量外部需求 |
| **自研** | 零起步。工具链完全需要自建，这是自研方案里最容易被低估的成本 |

**核心判断**：工具链的实际投入通常占技能系统总工程量的 30-50%。选型时如果只看运行时架构不看工具链，会严重低估落地成本。GAS 在这个维度上有显著优势，因为 Unreal Editor 的基础设施可以直接复用。

这里有一个容易踩的坑：守望先锋和 LoL 的工具链成熟度虽然标为"高"，但那是它们各自内部团队多年积累的结果。
如果你想参考它们的架构思想自研，工具链不会跟着一起来。
所以在评估这些方案的"借鉴价值"时，要把工具链的自建成本单独算进去。

## 维度六：学习曲线与团队适配

这个维度衡量的是：团队从零上手到能正常产出技能内容，需要多久。

| 方案 | 判断 |
|------|------|
| **GAS** | 陡峭。概念多（ASC、GE、GA、GC、Tag），C++ 门槛高，但一旦上手后框架一致性强，新人可以在已有模式里高效工作 |
| **Dota 2** | 中等。KV 配置对策划友好，Lua 脚本对技术策划友好，但 Modifier 事件模型的心智模型需要时间建立 |
| **守望先锋** | 高门槛。ECS + 确定性模拟 + 网络预测回滚，需要团队有扎实的系统编程和网络编程经验 |
| **LoL** | 低。脚本化模型对策划友好，热更新缩短迭代周期，但脚本质量管理在团队规模扩大时是挑战 |
| **PoE** | 高门槛。组合系统的数值设计和平衡需要深厚的数值经验，框架层面的 Modifier 聚合管线理解成本高 |
| **自研** | 取决于设计者水平。如果主程经验足够，可以设计出团队友好的架构；如果经验不足，自研系统很容易变成只有作者自己能改的黑箱 |

**核心判断**：学习曲线不只看框架复杂度，更要看团队组成。如果团队主力是策划和技术策划，LoL 模式（脚本化）最友好。如果团队主力是 C++ 程序员，GAS 最合适。如果团队只有 1-2 个全栈程序员，自研反而可能是最实际的选择——因为框架逻辑完全在自己掌控中。

一个更细的判断维度：区分"框架上手时间"和"框架内产出时间"。

GAS 的上手时间长（1-3 个月才能理解全貌），但上手后在框架内产出新技能很快——因为模式统一，新人照着已有技能改就行。
自研框架的上手时间可能很短（作者自己当然最熟），但如果架构不一致，每加一个新技能都可能遇到新问题。

所以如果团队规模会扩大（从 3 人到 15 人），GAS 的长期团队效率优势会越来越明显。
如果团队始终是 2-3 人核心，自研的灵活性优势更大。

## 维度七：扩展边界

这个维度衡量的是：当需求超出框架设计假设时，框架能被扩展到什么程度。

| 方案 | 判断 |
|------|------|
| **GAS** | 中高。C++ 源码可读可改，但 GAS 的 UObject 继承体系一旦偏离了设计假设（比如非 Actor 实体需要 ASC），改造成本很高 |
| **Dota 2** | 有上限。KV + Lua 的表达力有天花板，超出天花板只能请求 Valve 在引擎层扩展（对 Mod 社区是硬限制） |
| **守望先锋** | 外部不可扩展。这是暴雪内部引擎，不存在外部扩展的可能性 |
| **LoL** | 外部不可扩展。Riot 的技能脚本系统是内部基础设施，外部团队无法使用 |
| **PoE** | 外部不可扩展。GGG 的组合引擎不是公开框架 |
| **自研** | 无限。所有代码都是你的，可以往任何方向扩展。但代价是所有扩展也都需要你自己维护 |

**核心判断**：守望先锋、LoL、PoE 三种方案对外部团队没有实际可用性——它们的价值在于`架构思想可借鉴`，而不是代码可复用。真正可选的只有 GAS、Dota 2（Mod 场景）和自研。扩展边界这个维度上，自研有绝对优势，GAS 有条件优势。

GAS 扩展边界的一个具体限制值得展开：
GAS 假设 AbilitySystemComponent 挂在 Actor 上。如果你的项目有"非 Actor 实体也需要 Ability"的需求（比如 ECS 架构里的纯数据实体、或者物品系统需要技能效果），GAS 的改造成本会很高。

Dota 2 的扩展上限体现在另一个地方：
KV 配置的 Action 类型是引擎内置的有限集。如果你需要一种引擎没有提供的 Action 类型（比如自定义物理位移），只能退到 Lua 甚至请求 C++ 扩展。对于 Mod 社区来说，这就是硬墙。

## 维度八：Live Service 支撑

这个维度衡量的是：框架对长期运营（热更新、版本迭代、AB 测试）的支撑能力。

| 方案 | 判断 |
|------|------|
| **GAS** | 弱。C++ 编译部署周期长，Unreal 的热更新支持有限，版本迭代需要完整的构建和发布流程 |
| **Dota 2** | 强。KV 配置和 Lua 脚本都支持热更新，Valve 可以在不停服的情况下调整技能数值和逻辑 |
| **守望先锋** | 中等。暴雪有内部热修复能力，但 ECS 架构的硬编码特性限制了不发版本就能改的范围 |
| **LoL** | 极强。脚本化技能系统天然支持热更新，Riot 的双周补丁节奏依赖这个能力 |
| **PoE** | 强。服务端权威架构意味着大量变更可以只改服务端，客户端更新频率可以更低 |
| **自研** | 取决于设计。如果前期规划了脚本层或数据驱动层，可以做到很好的热更新支持；如果全部硬编码，每次改动都需要发版 |

**核心判断**：如果项目是需要频繁调平衡的 Live Service 游戏（MOBA、MMO），热更新能力是刚需。GAS 在这个维度上有明显短板。自研方案如果一开始就考虑了脚本层（Lua / C# 热更），可以做到和 LoL 模式接近的迭代速度。

一个经常被问到的问题：GAS 就不能做热更新吗？

技术上可以通过蓝图层做部分逻辑热更新，或者用 Unreal 的 Pak 补丁机制做资产更新。
但 GAS 的核心逻辑（Ability 执行、GE 聚合）是 C++ 编译的，如果要改这些层的行为，就需要完整的构建和发布流程。

这就是为什么 LoL 选择了脚本化——Riot 需要每两周做一次英雄平衡调整，如果每次调整都要走 C++ 编译部署，运营节奏根本跟不上。

## 怎么读这张矩阵

在看完整表之前，先说一个读法上的注意事项。

这张表里"取决于设计"和"取决于实现"出现了很多次，全部集中在"自研"这一列。
这不是在回避判断，而是在说一个事实：`自研方案的所有维度表现，完全取决于你的设计决策和实现质量。`

这既是自研的最大优势（完全可控），也是最大风险（完全自负）。

所以读这张表的正确方式是：

1. 先忽略"自研"列，看其他 5 种方案在 8 个维度上的固有特征
2. 然后问自己：我的项目在哪 2-3 个维度上不能妥协
3. 最后看是否有某个方案在这些维度上都够用
4. 如果没有，才考虑自研——而且要明确自研时需要重点投入哪些维度

## 完整 8×6 矩阵总表

| 维度 | GAS | Dota 2 | 守望先锋 | LoL | PoE | 自研 |
|------|------|------|------|------|------|------|
| 数据驱动程度 | 中 | 高 | 低 | 高 | 中高 | 取决于设计 |
| 网络同步模型 | 强 | 强（引擎级） | 极强 | 中强 | 服务器权威 | 零 |
| 效果组合能力 | 中 | 中 | 低 | 中 | 极强 | 取决于设计 |
| 运行时性能 | 中 | 中 | 强 | 中 | 热点在聚合 | 取决于实现 |
| 工具链成熟度 | 中高 | 中 | 高（内部） | 高（内部） | 中（内部） | 零起步 |
| 学习曲线 | 陡峭 | 中 | 高门槛 | 低 | 高门槛 | 取决于设计者 |
| 扩展边界 | 中高 | 有上限 | 不可扩展 | 不可扩展 | 不可扩展 | 无限 |
| Live Service | 弱 | 强 | 中 | 极强 | 强 | 取决于设计 |

这张表不是用来"选最多格子里写了强的那个"，而是用来做`约束匹配`——先明确你项目最重的 2-3 个约束维度，再看哪种方案在这些维度上最合适。

## 选型决策树

下面按项目类型给出具体的选型路径。
每种类型先明确最重的约束维度，再给推荐方向和关键判断。

### 竞技 FPS（类守望先锋）

**最重约束**：网络同步（确定性模拟 + 预测回滚）、运行时性能（每帧模拟）
**推荐方向**：如果用 UE，GAS + 自研预测层。如果用 Unity 或自研引擎，参考守望先锋的 ECS 架构思想自建。
**关键判断**：不要选数据驱动方案——竞技 FPS 的技能数量少（每英雄 3-4 个）且行为高度差异化，硬编码比配置更可控。
**典型风险**：在这个品类里最常见的错误是低估预测回滚的复杂度。如果你的技能会产生位移（冲刺、闪烁、击退），回滚时需要还原物理状态，这远比回滚数值属性困难。

### MOBA（类 Dota 2 / LoL）

**最重约束**：数据驱动程度（100+ 英雄的技能产出效率）、Live Service（双周补丁节奏）
**推荐方向**：LoL 的脚本化模式最适合。如果团队 C++ 能力强且用 UE，GAS 可行但需要额外补热更新能力。
**关键判断**：MOBA 的技能数量大（500+），如果没有策划可直接写的脚本层或配置层，产出效率会成为瓶颈。
**典型风险**：MOBA 项目最容易低估的是英雄间交互测试的规模。N 个英雄的两两交互测试矩阵是 N^2 级别，100 个英雄就是 10000 个交互对。如果框架没有标准化的 Modifier 聚合规则，每个交互对都可能出 Bug。

### ARPG / MMO（类 PoE / 暗黑）

**最重约束**：效果组合能力（Build 多样性）、Modifier 聚合性能（数百个同时生效）
**推荐方向**：自研组合层 + 参考 PoE 的聚合管线设计。GAS 可以作为基础但需要在 Modifier 聚合层做大量扩展。
**关键判断**：如果核心玩法是 Build 构建，组合层的设计质量直接决定游戏深度。这种项目最不应该"选一个现成框架就够了"。
**典型风险**：ARPG 项目最容易出问题的地方是 Modifier 聚合顺序。如果一开始没有明确定义"增加/乘法/额外/最终"的聚合阶段，后面加新词缀时会发现每个词缀的效果取决于代码里碰巧的执行顺序，而不是设计意图。

### 复杂 Build 类（Roguelike、卡牌构建）

**最重约束**：效果组合能力（规则可组合）、扩展边界（需要频繁加新机制）
**推荐方向**：自研。这类项目的核心乐趣就在组合的涌现性，框架必须完全可控。
**关键判断**：不要用 GAS——GAS 的设计假设是"一个 Ability 做一件事"，和 Roguelike 的"所有东西都可以互相影响"存在根本性的假设冲突。
**典型风险**：Roguelike 的组合涌现很容易产生开发者没预料到的极端 Build。如果没有设计循环防护和数值封顶机制（参考 20 篇 PoE 的对策），某些组合会让游戏直接崩溃或变得毫无挑战。

### 中小团队（1-5 程序员）

**最重约束**：学习曲线（不能花 3 个月学框架）、工具链（不能花 6 个月造工具）
**推荐方向**：如果用 UE，GAS 是最务实的选择——虽然学习曲线陡峭，但上手后框架一致性强，不需要自己造基础设施。如果用 Unity，轻量自研 + 参考系列 00-13 的分层思想。
**关键判断**：中小团队最大的风险不是"框架不够强"，而是"自研框架写到一半发现维护不了"。如果没有至少一个有完整技能系统经验的主程，不建议从零自研。

### UE 项目（已确定用 Unreal）

**最重约束**：和引擎生态的集成度（蓝图、网络、资产管线）
**推荐方向**：GAS 是默认选择。除非有明确的理由（比如 GAS 的 UObject 开销无法接受，或项目需要 ECS 架构），否则不应该在 UE 项目里自研技能框架。
**关键判断**：GAS 最大的优势不是"功能多"，而是"和 UE 其他系统的集成是现成的"。网络同步、蓝图可视化、资产管线、调试工具都不需要从零开始。

## 什么时候自研 vs 什么时候不该自研

### 应该自研的信号

1. **框架的核心假设和项目需求冲突**。比如你的项目需要非 Actor 实体有 Ability（GAS 假设 Ability 挂在 Actor 上），或者你的项目需要确定性模拟（GAS 用浮点数）。
2. **项目的核心差异化就在技能系统上**。如果技能系统是游戏最核心的卖点（Roguelike 的涌现组合、独特的战斗节奏），那它应该完全可控。
3. **团队有足够经验**。至少有一个程序员完整搭过一个技能系统、踩过主要的坑、知道哪些层不能省。

### 不该自研的信号

1. **团队没有完整技能系统经验**。第一次做技能系统就从零自研，大概率会在 Modifier 聚合、多人同步、或工具链这三个点上严重低估工作量。
2. **技能系统不是项目核心差异化**。如果你的游戏核心是剧情、社交或经济系统，技能系统只需要"够用"，那用现成框架比自研更经济。
3. **项目已经选了 UE 且需要多人同步**。GAS 在这个场景下的综合优势太大，自研很难做到同等水平的网络同步集成。
4. **时间紧迫**。一个可用的自研技能系统（含最小工具链）保守估计需要 3-6 个月。如果项目节奏不允许这个投入，就不该自研。

### 自研的常见失败模式

- **只建了执行层，忘了工具链**。技能可以跑了，但策划改一个数值要重新编译。
- **只考虑了单人，忘了同步**。单人模式下跑得很好，一加联网就崩。
- **Modifier 系统从简单加法开始，后来发现需要聚合顺序但已经到处是硬编码**。
- **没有脏标记机制，Modifier 数量上升后性能急剧下降**。

这些失败模式在系列前文的对应篇章里都有拆解（08 篇 Modifier、11 篇同步、12 篇工具链）。
如果决定自研，建议至少把这几篇的核心约束先过一遍。

### 自研的最小可行投入估算

如果决定自研，下面是一个粗略的工作量估算，基于"一个有经验的游戏程序员全职投入"的假设：

| 层 | 最小可用版本 | 生产级版本 |
|------|------|------|
| 数据模型 + 生命周期 | 2-3 周 | 4-6 周 |
| 执行链（输入 → 校验 → 命中 → 效果） | 3-4 周 | 6-8 周 |
| Buff / Modifier 聚合 | 2-3 周 | 4-8 周 |
| 最小编辑器 / 配置工具 | 3-4 周 | 8-12 周 |
| 多人同步（如需要） | 6-10 周 | 12-24 周 |
| 测试框架 | 1-2 周 | 3-4 周 |

加起来：

- **单人游戏最小可用版本**：约 3-4 个月
- **联网游戏生产级版本**：约 8-15 个月

这还没算内容制作和平衡调优的时间。
所以"自研"不是一个轻率的决定——它是一个需要明确预算和时间线的工程决策。

## 选型时的常见误区

最后补几个在选型讨论中经常出现的误区。

### 误区一："GAS 什么都能做，选它就完了"

GAS 确实覆盖面广，但它有几个硬性约束：
绑定 UE 引擎、UObject 体系的性能开销、C++ 编译部署周期、不适合需要极端组合深度的项目。
如果你的项目不在 UE 上，或者核心玩法需要 GAS 设计假设之外的能力，GAS 就不是答案。

### 误区二："自研总是更好，因为更灵活"

灵活性是有代价的。
自研意味着所有基础设施（网络同步、工具链、调试工具）都要自己建。
如果团队规模不足或经验不够，这些基础设施的质量会远低于成熟框架，最终的"灵活性"变成了"每个地方都有不同的写法，没人能维护"。

### 误区三："参考了 X 的架构思想 = 和 X 一样好"

架构思想只是 30% 的工作量。
真正的工程量在实现细节、边界处理、工具链和长期维护上。
参考守望先锋的 ECS 架构思想自研，和实际达到守望先锋的工程质量，中间差的不是"思想"而是"人年"。

### 误区四："技能数量少就不需要框架"

即使只有 10 个技能，如果它们之间有 Buff 交互、有属性修改、有多人同步需求，"10 个独立函数"很快就会变成维护噩梦。
需不需要框架，不只看技能数量，还要看技能之间的交互密度。

## 这一篇真正想留下来的结论

技能系统选型不是选"最强的框架"，而是选`约束匹配度最高的方案`。

如果把这个判断展开成三层：

**第一层**：明确你项目最重的 2-3 个约束维度（网络同步？数据驱动？组合深度？热更新？团队规模？）。
**第二层**：在这 2-3 个维度上，对比可选方案的实际支撑能力。不要看"是否支持"，要看"支持到什么程度、代价是什么"。
**第三层**：如果没有一个现成方案在你的关键维度上都够用，才考虑自研。自研不是"更好的选择"，而是"没有更匹配的选择时的兜底方案"。

如果把整篇压到最短一句话：

`先列约束，再选方案。约束决定了哪些维度不能妥协，方案决定了在这些维度上你能走多远。`
