## 问题的本质：如何证明请求的合法性

HTTP 是无状态协议。服务端收到一条请求时，它看到的只是一组字节——IP 地址、请求头、请求体。它无法天然地知道这条请求是从真实客户端发出的、代表着哪个玩家、是否已经被篡改过。

对于普通 Web 应用，这个问题已经有了成熟的方案——Session/Cookie，或者更现代的 Token 机制。但游戏后端面临一些额外挑战：

- **客户端环境不可信：** 游戏客户端运行在玩家设备上，可以被反编译、被调试、被 Hook。攻击者可以提取客户端硬编码的密钥，可以修改网络包，可以用脚本模拟合法请求。
- **高频调用：** 游戏的实时性要求接口能处理每秒数百次调用，鉴权机制不能有太高的性能开销。
- **多端覆盖：** 同一账号可能同时活跃在手机、PC、第三方平台多个入口，鉴权体系需要统一管理。

因此，游戏接口安全需要解决的核心问题是：**在不可信环境中，用尽可能低的开销，确认一条请求的来源身份，并保证内容未被篡改。**

## JWT：无状态的身份证明

### 结构

JWT（JSON Web Token）由三部分构成，用 `.` 连接：

```
Header.Payload.Signature
```

- **Header：** 声明令牌类型（JWT）和签名算法（通常是 HS256 或 RS256）
- **Payload：** 包含声明（Claims），比如用户 ID、角色、过期时间（`exp`）、签发时间（`iat`）
- **Signature：** 用密钥对 `Base64(Header).Base64(Payload)` 进行签名

```json
// Payload 示例
{
  "sub": "player_123456",
  "role": "player",
  "iat": 1711900000,
  "exp": 1711900900,
  "server_region": "cn-north"
}
```

### 验证流程

服务端收到 JWT 后：

1. 解析 Header，确认签名算法（**注意：必须在服务端固定允许的算法列表，拒绝接受 `alg: none`**）
2. 用对应密钥重新计算签名，与 Token 中的 Signature 对比
3. 验证 `exp` 字段，确认 Token 未过期
4. 从 Payload 中提取 `sub`（用户 ID），作为后续业务逻辑的身份依据

**核心优势：** 服务端无需查询数据库即可完成验证。多个服务节点共享同一密钥（或公钥），即可独立验证 Token，天然适合分布式部署。

### HS256 vs RS256

| 算法 | 密钥类型 | 适用场景 |
|------|----------|----------|
| HS256 | 对称密钥（共享秘密） | 单体服务或所有服务都可持有密钥 |
| RS256 | 非对称密钥（私钥签名，公钥验证） | 多服务架构，签发服务持有私钥，其他服务只持有公钥 |

游戏后端通常推荐 RS256：认证服务（Auth Service）独立持有私钥，其他微服务只需要公钥即可验证 Token，不需要共享敏感密钥。

## OAuth 2.0：第三方登录的标准流程

大多数游戏都支持微信、Apple ID、Google 等第三方账号登录。这里用到的就是 OAuth 2.0 的 **Authorization Code Flow**（授权码流程）。

### 流程概述

```
玩家点击"微信登录"
    ↓
游戏客户端 → 微信 OAuth 服务器（用户授权）
    ↓
微信返回 Authorization Code（一次性，短效）
    ↓
游戏客户端 → 游戏后端（携带 Authorization Code）
    ↓
游戏后端 → 微信服务器（用 Code 换取 Access Token）
    ↓
游戏后端用微信 Access Token 查询用户的 OpenID / UnionID
    ↓
根据 OpenID 找到或创建游戏账号，签发游戏自己的 JWT
    ↓
返回游戏 JWT 给客户端
```

**关键点：**

- 游戏客户端拿到的是游戏自己签发的 JWT，不是微信的 Token。游戏后端对自己的 Token 有完全的控制权（有效期、权限范围、吊销机制）。
- Authorization Code 是一次性的，用完即失效，防止中间人截获后重复使用。
- 与微信服务器的通信发生在游戏后端（Server-to-Server），不经过客户端，密钥不暴露给玩家设备。

## 接口签名：防止参数篡改

身份鉴权（我是谁）解决了之后，还需要解决**完整性验证（我发的内容有没有被改过）**的问题。

在游戏场景中，攻击者可能在客户端发出请求后、到达服务端之前，通过中间人代理（如 Charles、Fiddler）修改请求参数。例如把"购买 10 个道具"改成"购买 10000 个道具"。

接口签名的目标是让这类篡改在服务端可被检测。

### 签名方案

一个常见的游戏接口签名方案包含以下要素：

```
签名 = HMAC-SHA256(
    sorted_params + timestamp + nonce,
    client_secret
)
```

其中：
- **sorted_params：** 请求参数按 key 排序后的字符串拼接（防止参数顺序不同导致签名不匹配）
- **timestamp：** 当前 Unix 时间戳（秒）
- **nonce：** 本次请求的随机字符串（一次性使用）
- **client_secret：** 客户端密钥，由服务端签发，不在请求参数中传输

请求体示例：

```json
{
  "item_id": 1001,
  "quantity": 10,
  "timestamp": 1711900000,
  "nonce": "a3f8c2d1",
  "sign": "5e7a9b3c..."
}
```

服务端收到请求后，用相同算法重新计算签名，与请求中的 `sign` 比对。不一致则拒绝请求。

**注意事项：**

- `client_secret` 需要妥善保管，不能出现在客户端代码中明文硬编码（实际上移动端客户端很难完全保护密钥，这是所有客户端鉴权方案的共同局限）。
- 签名只能保证"参数在传输过程中未被篡改"，不能阻止攻击者在客户端侧构造伪造请求（如果密钥泄露）。签名是深度防御的一层，不是唯一防线。

## 防重放攻击：Nonce + TTL + 黑名单

即使有了签名，攻击者仍然可以捕获一条合法请求，原封不动地重复发送——这就是**重放攻击（Replay Attack）**。

例如：捕获一条"完成任务，领取奖励"的合法请求，重放 1000 次，每次都能通过签名验证，因为签名本身是合法的。

防重放的三要素：

### 1. Timestamp（时间戳窗口）

服务端检查请求中的 `timestamp`，如果与服务器当前时间差超过 N 秒（通常 5-300 秒），则拒绝请求。这将重放攻击的有效窗口限制在了时间戳窗口内。

### 2. Nonce（一次性随机数）

每条请求携带一个唯一的随机字符串（Nonce）。服务端在 Timestamp 窗口内记录所有已处理的 Nonce（存入 Redis，设置与窗口等长的过期时间）。如果相同的 Nonce 出现第二次，则拒绝请求。

```python
# Redis 实现示例
def check_nonce(nonce: str, ttl_seconds: int = 300) -> bool:
    key = f"nonce:{nonce}"
    # SET NX（不存在才写入）
    was_new = redis.set(key, "1", ex=ttl_seconds, nx=True)
    return was_new  # True = 首次使用，允许; False = 已使用，拒绝
```

### 3. 已用 Nonce 黑名单

结合 Timestamp 窗口，Nonce 的生命周期只需要与窗口等长，之后可以自动过期清除，不需要永久存储。这将 Redis 的内存消耗控制在可接受的范围内。

## Token 的有效期设计

Token 的有效期是安全性与用户体验的权衡。Token 越短效越安全，但频繁重新登录影响体验。

游戏通常采用**双 Token 机制**：

| Token | 有效期 | 用途 |
|-------|--------|------|
| Access Token | 15 分钟 - 1 小时 | 每次 API 请求时携带，服务端无状态验证 |
| Refresh Token | 7 天 - 30 天 | 当 Access Token 过期时，用来换取新的 Access Token |

**刷新流程：**

```
客户端携带 Access Token 发送请求
    ↓
服务端：Token 过期，返回 401
    ↓
客户端用 Refresh Token 请求 /auth/refresh
    ↓
服务端验证 Refresh Token，签发新的 Access Token（和可选的新 Refresh Token）
    ↓
客户端用新 Access Token 重试请求
```

**Refresh Token 的吊销：** Refresh Token 需要在服务端维护状态（数据库或 Redis），支持主动吊销（账号封禁、用户主动登出、安全事件时强制踢号）。

## 常见漏洞速查

### 漏洞 1：alg: none 攻击

JWT 的 Header 可以声明 `"alg": "none"`，部分早期库会对此绕过签名验证，直接信任 Payload。

**防御：** 服务端硬编码允许的算法列表，拒绝接受 `none` 算法。

### 漏洞 2：签名密钥强度不足

HS256 使用的密钥如果太短或可预测，攻击者可以暴力破解。

**防御：** 使用至少 256 位（32 字节）的高熵随机密钥，通过密钥管理服务（KMS）生成和轮换。

### 漏洞 3：时间戳窗口过宽

Timestamp 窗口设置为 30 分钟甚至 1 小时，让攻击者有足够时间发起重放攻击。

**防御：** 在用户体验可接受的前提下，将窗口设置尽量短（300 秒已经是比较宽松的上限）。

### 漏洞 4：Token 通过 URL 参数传递

某些团队图方便，将 JWT 放在 URL 查询参数中（`?token=xxx`），导致 Token 出现在服务器日志、浏览器历史、Referrer 头中。

**防御：** Token 必须通过 HTTP Header 传递（`Authorization: Bearer <token>`），或者 Cookie（设置 `HttpOnly` + `Secure`）。

## 工程边界

接口安全机制的局限性需要明确：

**客户端密钥的固有困境：** 移动端客户端（APK/IPA）可以被逆向，硬编码的密钥可以被提取。接口签名更多是抬高攻击成本，而非绝对防御。服务端的业务逻辑校验（频率限制、业务前置条件验证）是最后一道防线。

**JWT 无法即时吊销：** JWT 的设计是无状态的，一旦签发，在过期时间内原则上无法撤销（除非引入服务端黑名单，但这会失去无状态的优势）。对于需要即时踢号的场景，需要在 Access Token 有效期短（分钟级别）和维护 Token 黑名单之间做选择。

---

## 最短结论

证明"这是真实玩家发的合法请求"需要三层机制联合工作：JWT 负责身份声明（我是谁），接口签名负责完整性保证（内容没被改），Nonce + Timestamp 负责唯一性保证（这条请求只执行一次）。

三层缺一不可，但任何一层都不是万能的。接口安全的上限取决于服务端业务逻辑的严密程度——如果服务端本身不对业务前置条件做校验，再完善的鉴权机制也保护不了业务数据。
