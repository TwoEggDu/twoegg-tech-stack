> "技能不同步"不是一种 bug，而是至少 5 种不同层的问题在表现层汇聚成了同一个症状。

这是技能系统深度第 34 篇。[第 11 篇]({{< relref "system-design/skill-system-11-multiplayer-sync.md" >}}) 讲的是多人同步下技能执行链"应该怎么拆"，这篇讲的是"拆错了或者细节出了问题时，会以什么形式炸出来"。

五种根因全部来自真实项目里反复出现过的 desync 报告，每一种都经历过"QA 说不同步 → 先被误导 → 最后才定位到真正原因"的完整过程。

5. 怎样建立"看到不同步先查哪一层"的诊断习惯

## 为什么"不同步"最难排

大多数 bug 是确定性的：传错参数就必定崩溃，逻辑写反就必定算错。但"不同步"不一样，它难排的原因至少有三层。

**第一层：表现层汇聚。**

五种完全不同的根因最终看起来都是同一个症状："两边不一样"。QA 报上来的描述几乎一模一样，但背后需要查的层完全不同。

**第二层：时序依赖。**

局域网 5ms 下一切正常，20ms 下偶尔出现，50ms 下才稳定复现。办公室 WiFi 测不出来，手机 4G 才能看到，但 4G 本身的抖动又让复现变得不确定。

**第三层：非必现。**

只有特定时序组合才能触发。两个玩家恰好在同一逻辑帧窗口内各释放一个技能，且两个技能恰好有互相抵消的判定——少一个条件都不会出问题。跑 50 局才遇到一次，但线上百万局里就是每天都有投诉。

这三层叠在一起，**传统的"断点 → 复现 → 修复"流程基本不适用**。你需要的是分层诊断：先判断不同步发生在哪一层，再在那一层里找具体原因。

## 5 种根因的全景

在展开每一种之前，先把五种根因在网络链路上的位置画出来：

```
客户端 A 输入 → [网络传输] → 服务端接收 → 服务端执行 → [网络传输] → 客户端 B 接收
     │                          │                │                          │
     ▼                          ▼                ▼                          ▼
  根因 1：输入到达顺序       根因 2：校验状态    根因 3：命中判定          根因 5：表现绑定
                                                  时刻的世界状态

                            根因 4：预测窗口过大
                        （贯穿客户端预测到服务端确认的整条链）
```

接下来逐一拆解。

## 根因 1：输入到达顺序不一致

### 现象

两个玩家几乎同时释放技能。玩家 A 在自己屏幕上看到"我先放出来了"，玩家 B 在自己屏幕上也看到"我先放出来了"。

如果这两个技能存在互相抵消的判定（比如两个护盾相撞时只有先放的那个生效），两边都认为自己应该生效。服务端裁定了一个结果，必然有一边被回滚——如果客户端用了乐观预测，玩家看到的就是护盾先出现后消失。

### 容易被误导的方向

QA 报："护盾互相抵消的判定有 bug，两边结果不一样。"

命中判定本身没有 bug。A 端按 A 先放来算，B 端按 B 先放来算，两边在各自的输入顺序下都算对了。**问题不在"怎么算"，而在"拿什么顺序算"。**

### 真正根因

两个客户端的输入通过 UDP 到达服务端的顺序是不确定的。A 的操作物理时间上比 B 早 3ms，但 A 的网络多抖了 10ms，服务端实际上先收到了 B 的输入。同时两个客户端本地都有输入缓冲区，各自先执行了自己的输入（本地预测），两端跑出了不同的执行序列。

```
物理时间  ──────────────────────────────────────▶

客户端 A：  [A 释放技能]         [收到 B 的输入]
            ├─ 本地预测：A 先 ─┤

客户端 B：         [B 释放技能]  [收到 A 的输入]
                   ├─ 本地预测：B 先 ─┤

服务端：                [收到 B] [收到 A]
                        ├─ 裁定：B 先 ─┤
```

### 定位方法

在客户端和服务端都打印每条输入到达时的**逻辑帧号**（不是物理时间戳，因为物理时间本身就不同步）。如果两端发送帧号相同但服务端接收帧号不同，就是这个问题：

```
[Client A] Input sent at logic frame 1042
[Client B] Input sent at logic frame 1042
[Server]   Received B input at frame 1042, A input at frame 1043
```

### 修法

**服务端统一排序。** 收集一个逻辑帧窗口内的所有输入后，按确定性规则排序（比如按玩家 ID 做 tiebreaker），然后再执行。不依赖网络到达顺序。

**客户端以服务端确认为准。** 可以先按"我先"来预测执行，但必须准备好在服务端确认不一致时回滚。

## 根因 2：校验状态不同步

### 现象

客户端显示 CD 已好（图标转完了），玩家按下技能按钮，本地也播了施法动画，但服务端回一个"还在 CD"的拒绝。施法动画硬生生被中断。或者反过来：客户端觉得还在 CD 所以按钮是灰的，但实际上服务端那边 CD 已经好了。

### 容易被误导的方向

QA 报："CD 计算有 bug，两边不一样。"

程序查 CD 计算逻辑，两边公式完全一致，单元测试也没问题。**问题不在公式本身，而在公式的输入值不一样。**

### 真正根因

一个 Buff 修改了 CD 冷却速率（比如"急速光环：CD 速度 +30%"）。这个 Buff 在客户端被预测性 Apply——本地 CD 按 1.3 倍速在走。但服务端还没确认（可能 Buff 施加条件在服务端不满足，或者仅仅因为延迟还没到），CD 还是按 1.0 倍速走。

```
时间线 ──────────────────────────────────────▶

客户端：  [预测 Apply 急速 Buff]  CD 按 1.3x 走  →  CD 好了！
服务端：  [还没确认急速 Buff]     CD 按 1.0x 走  →  CD 还剩 0.8s
```

0.8 秒的差距足以让客户端认为"可以放了"而服务端说"不行"。

### 定位方法

在两端分别打印 `CooldownRemaining` 和影响它的 Modifier 列表。如果客户端多了未被服务端确认的 Modifier，就是这个问题：

```
[Client] CooldownRemaining=0.0, Modifiers=[HasteAura(+30%)]
[Server] CooldownRemaining=0.8, Modifiers=[]
```

### 修法

**CD 计算用服务端权威值。** 客户端可以做预测性的 CD 显示（让玩家觉得反应快），但施法请求的校验必须等服务端确认。

**客户端预测 CD 时不计入未确认的 Buff 效果。** 给每个 Modifier 标记 `confirmed` 标志，CD 预测计算时忽略所有 `unconfirmed` 的 Modifier：

```csharp
float GetPredictedCooldownRate()
{
    float rate = baseCooldownRate;
    foreach (var mod in activeModifiers)
    {
        // 只计入已确认的 Modifier
        if (mod.isConfirmed)
            rate *= mod.cooldownRateMultiplier;
    }
    return rate;
}
```

## 根因 3：命中判定时刻的世界状态不一致

### 现象

玩家 A 打了一枪，准星对着目标 B 的头，本地判定命中，血条掉了——但服务端说没命中，血条又回去了。或者反过来：B 在自己屏幕上明明躲到掩体后面了，但服务端判定被命中。这是射击类和 ARPG 类项目里最高频的投诉。

### 容易被误导的方向

QA 报："碰撞检测有 bug。"

碰撞检测本身是对的，给同样的位置数据结果完全一致。**问题不在碰撞检测逻辑，而在碰撞检测用的位置数据不一样。**

### 真正根因

客户端 A 在时刻 T 开火，此时目标 B 在位置 P1。开火请求经过 RTT/2 到达服务端，此时 B 已经移动到位置 P2。服务端直接按 P2 做判定——即使客户端确实瞄准了，服务端也会判定未命中。

```
时刻 T（客户端 A 开火）：

  客户端 A 看到：
    A ────────● B 在 P1（准星对准）
              命中！

时刻 T + RTT/2（服务端收到请求）：

  服务端世界状态：
    A ────────          ● B 已经在 P2
              P1 已空
              未命中！
```

P1 和 P2 之间的距离 = 移速 * RTT/2。B 每秒移动 5 米、RTT 100ms，偏差就是 0.25 米——对身体宽度 0.6 米的角色，足以导致"打中变没打中"。

### 定位方法

在客户端开火时记录目标位置，在服务端收到请求时记录目标位置。如果 delta 和 RTT/2 * 移速量级吻合，就是这个问题：

```
[Client A] Fire at T=1042, target pos=(10.0, 0, 5.0)
[Server]   Received fire at T=1044, target pos=(10.25, 0, 5.0)
[Server]   Delta=0.25m, hit radius=0.3m → marginal miss
```

### 修法

**服务端做 Lag Compensation。** 服务端收到开火请求后，不用当前时刻的世界状态做判定，而是回退到客户端开火时刻的世界状态快照，在那个快照里做碰撞检测。

具体做法：服务端维护最近 N 帧的世界状态历史（至少覆盖最大可接受 RTT）。收到开火请求时，根据客户端声称的开火时刻查找最接近的历史快照，回退角色位置后做判定。

```
服务端 Lag Compensation 流程：
1. 收到开火请求，附带客户端时间戳 T_client
2. 在历史快照池里找到 T_client 最近的快照
3. 把所有角色位置回退到那个快照的状态
4. 在回退后的状态里做碰撞检测
5. 恢复当前状态
```

对投射物类技能（火球、箭矢）：投射物飞行轨迹用服务端权威，客户端做外推预测保证视觉连续性。

**重要限制：** Lag Compensation 不是无限回退的。超过 RTT 阈值（通常 200-300ms）的请求应该直接按当前状态判定，否则会出现"躲到掩体后面一秒了还被打中"的体验问题。

## 根因 4：预测窗口过大

### 现象

玩家释放技能后，施法动画播了、火球飞出去了、目标血条掉了。然后突然——动画回退、火球消失、血条回到原来的数值，紧接着"正确"的结果重新播一遍。或者更微妙的版本：血条先掉到 60%，跳回 80%，再慢慢掉到 65%——"血条在抖动"。

### 容易被误导的方向

QA 报："表现层有 bug，动画会突然跳。"

表现层忠实地跟随了逻辑状态的变化——动画跳是因为逻辑状态跳了。**问题不在表现层怎么播，而在逻辑层预测了太多内容。**

### 真正根因

客户端预测性地执行了整条技能链：

1. 预测了施法成功（合理）
2. 预测了投射物命中（勉强合理）
3. 预测了伤害最终值（包括暴击、抗性减算等完整计算）
4. 预测了目标死亡（不合理）

服务端回来的结果和步骤 3、4 不一致——也许服务端那边目标有一个客户端不知道的临时护盾，也许目标残血但没到 0。客户端被迫回滚大量状态，回滚幅度越大，玩家感受到的"跳变"就越明显。

```
预测的内容越多 → 服务端否决的可能性越大 → 回滚幅度越大 → 视觉跳变越严重

预测范围：
  ✓ 施法激活（几乎不会被否决）
  ✓ CD 开始（很少被否决）
  ✓ 资源消耗（偶尔被否决）
  △ 投射物命中（有时被否决）
  ✗ 伤害最终值（经常被否决）
  ✗ 目标死亡（经常被否决）
```

### 定位方法

在客户端记录每次预测了哪些状态变更，以及服务端确认后回滚了哪些：

```
[Client] Predicted: [CastStart, CDStart, ManaDeduct, Hit, Damage=340, TargetDeath]
[Server] Confirmed: [CastStart, CDStart, ManaDeduct, Hit, Damage=280, TargetAlive]
[Client] Rollback:  [Damage 340→280, TargetDeath→Alive]
```

如果 Rollback 列表经常出现 Damage 和 Death 相关条目，说明预测范围过宽。

### 修法

**收窄预测范围。** 施法激活/CD 开始/资源消耗可以预测；命中判定谨慎预测；最终伤害值和目标死亡不预测，等服务端。

**表现层做平滑过渡。** 血条回滚时做 200ms 插值过渡，不要硬切：

```csharp
// 硬切（差体验）
healthBar.SetValue(serverConfirmedHP);

// 平滑过渡（好体验）
healthBar.SmoothTo(serverConfirmedHP, duration: 0.2f);
```

## 根因 5：表现层和逻辑层绑定

### 现象

服务端否决了一次命中判定，逻辑层回滚了伤害——但客户端的击中特效已经播出去了。粒子已经发射、音效已经响过。玩家看到的是：明明有击中特效但伤害没扣。更严重的：预测了目标死亡，播了死亡动画和击杀播报，结果被服务端否决——目标"死而复生"。

### 容易被误导的方向

QA 报："网络延迟太大了，需要优化网络。"

网络延迟是触发条件但不是根因。50ms 就足以让一次命中判定被否决。**问题不在延迟多大，而在表现元素绑定了错误的事件源。**

### 真正根因

VFX 和 SFX 的触发直接绑定了预测性状态的变更事件：

```csharp
// 问题代码：表现直接绑定了预测事件
skillSystem.OnPredictedHit += (target) =>
{
    // 这些一旦执行就不可撤回
    VFXPool.Spawn(hitSpark, target.position);
    AudioManager.Play(hitSound);
    CameraShake.Trigger(intensity: 0.3f);
};
```

当预测说"命中了"，粒子立刻发射、音效立刻播放、相机立刻震动。但预测是可以被否决的——而这些表现元素是**不可撤回的一次性行为**，和预测性逻辑绑定后就形成了"表现无法回滚"的困局。

### 定位方法

检查每个 VFX/SFX 的触发绑定的事件源：

```
[审计清单]
击中火花 → 绑定了 OnPredictedHit    ← 预测事件，有风险
施法动作 → 绑定了 OnPredictedCast   ← 预测事件，风险低（施法很少被否决）
伤害数字 → 绑定了 OnPredictedDamage  ← 预测事件，高风险！
击杀播报 → 绑定了 OnPredictedKill   ← 预测事件，极高风险！
```

如果关键表现元素（伤害数字、击杀播报、死亡动画）绑定的是预测事件而非确认事件，就是这个问题。

### 修法

按表现元素的重要性分两类处理：

**关键表现等确认后再播放。** 伤害数字、死亡动画、击杀播报必须绑定 `OnConfirmedXxx` 事件：

```csharp
// 伤害数字：等服务端确认
skillSystem.OnConfirmedDamage += (target, amount) =>
{
    DamageNumberUI.Show(target.position, amount);
};

// 死亡：等服务端确认
skillSystem.OnConfirmedKill += (target) =>
{
    target.PlayDeathAnimation();
    KillFeed.Show(attacker, target);
};
```

**非关键表现可以预测播放，但做好"已播放就不撤回"的兜底。** 击中火花、施法动作、音效这些即使和最终结果不一致，玩家也不会明显感知"出错了"。允许预测性播放，但一旦播了就不撤回——"一闪而过的火花"比"火花消失"更不违和。

总结成一个表格：

```
表现元素       绑定事件          可否撤回    策略
─────────────────────────────────────────────────
施法动作       OnPredictedCast   可以        预测播放
击中火花       OnPredictedHit    不可以      预测播放，不撤回
施法音效       OnPredictedCast   不可以      预测播放，不撤回
伤害数字       OnConfirmedDamage N/A         等确认
死亡动画       OnConfirmedKill   N/A         等确认
击杀播报       OnConfirmedKill   N/A         等确认
```

## 5 种根因的诊断流程

看到"不同步"时，不要凭直觉猜，按下面这棵决策树从上到下过一遍：

```
看到不同步
  │
  ├─ 1. 两端的输入执行顺序是否一致？
  │     打印两端的输入到达逻辑帧号
  │     不一致 → 根因 1：输入到达顺序
  │     一致 ↓
  │
  ├─ 2. 两端的校验状态（CD/Buff/资源）是否一致？
  │     打印两端的 CooldownRemaining 和 Modifier 列表
  │     不一致 → 根因 2：校验状态不同步
  │     一致 ↓
  │
  ├─ 3. 命中判定时刻的目标位置是否一致？
  │     打印客户端开火时目标位置 vs 服务端收到请求时目标位置
  │     差异超过命中范围 → 根因 3：命中时刻世界状态不一致
  │     一致 ↓
  │
  ├─ 4. 客户端预测了多少内容？
  │     检查预测列表和回滚列表
  │     回滚列表经常出现 Damage/Death → 根因 4：预测窗口过大
  │     回滚量小 ↓
  │
  └─ 5. 表现元素绑定了预测事件还是确认事件？
        审计 VFX/SFX 的事件绑定
        关键表现绑定了预测事件 → 根因 5：表现层和逻辑层绑定
```

实际项目里一个 desync 报告可能同时涉及多种根因。比如"血条先掉后回再掉"可能同时是根因 4 和根因 5。但诊断时仍然建议从上到下过，因为上层的根因修完后下层的症状可能自然消失。

---

## 诊断的基础设施

上面的定位方法都依赖一个前提：你能拿到两端的日志并对齐。

如果你的项目还没有以下基础设施，建议优先补上：

- **逻辑帧同步日志。** 每个关键事件（输入到达、状态变更、命中判定）都附带逻辑帧号。逻辑帧号是两端可以对齐的唯一可靠锚点，物理时间戳不可靠。
- **预测/确认日志。** 客户端每次预测性执行和每次收到服务端确认时都打一条日志。包含预测了什么、确认了什么、回滚了什么。
- **状态快照对比工具。** 能在任意时刻把两端的关键状态（位置、血量、Buff 列表、CD 状态）dump 出来做 diff。不需要实时，事后分析即可。

这三样加起来通常一两天就能搭完，但在排查 desync 问题时的价值是决定性的。没有它们，你只能靠猜。

## 这篇真正想留下来的结论

"不同步"不是一种 bug。它是至少五种不同层的问题在玩家屏幕上的投影。

```
  输入层    │  校验层    │  判定层    │  预测层    │  表现层
  输入乱序  │  状态分歧  │  时刻偏差  │  预测过宽  │  绑定错误
     │           │           │           │           │
     ▼           ▼           ▼           ▼           ▼
  ┌──────────────────────────────────────────────────────┐
  │            玩家看到的：两边不一样                       │
  └──────────────────────────────────────────────────────┘
```

五种根因的修法完全不同，但诊断方法有一个共同模式：**先确定不同步发生在哪一层，再在那一层里找差异。** 不要从表现层的症状直接猜根因——"两边不一样"这五个字能对应的根因太多了。

如果记一个最小习惯：看到不同步，先打两端的逻辑帧日志做对比。输入顺序对不对、状态值一不一致、位置差多少——三步下来，五种根因至少能排除三种。
