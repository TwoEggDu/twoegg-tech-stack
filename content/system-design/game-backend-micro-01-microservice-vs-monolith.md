## 一个团队的微服务故事

某中型手游团队在上线半年后开始拆服务。理由很充分：单体代码库已经 40 万行，编译一次要三分钟，登录模块和战斗模块的发布经常互相拖累。架构师拍板：把账号、匹配、战斗、好友、排行榜全部拆开，每个服务一个 Git 仓库，独立部署。

六个月后，他们在事故复盘会上承认：线上 P0 故障频率从每月 1 次变成了每周 2 次。新增一个跨服务的业务功能，原来改 3 个文件，现在需要改 5 个仓库、协调 3 个团队、写 2 份 API 文档。发布窗口不是变短了，而是变长了，因为多个服务必须同步发版。

他们拆的不是微服务，他们制造的是一个**分布式单体**。

---

## 微服务的真实收益是什么

在被过度宣传的架构词汇里，微服务大概是被误解最深的一个。它的核心价值不是"每个功能一个服务"，而是以下三点有明确边界的收益：

**独立部署**：单体中任何模块发布都要全量部署。微服务允许高频变更的模块（比如活动系统）独立发布，不影响低频模块（比如账号系统）。前提是这两个模块确实有不同的发布频率，且它们之间的接口足够稳定。

**独立扩缩容**：排行榜在跨服赛季结束时读请求暴增，但这个峰值和战斗服的峰值不重叠。把排行榜独立出来，可以针对性地扩容 Redis + 查询服务，而不是把整个后端都扩一遍。独立扩缩的前提是负载特征明显不同。

**技术栈自由**：匹配算法用 Go 写性能更好，支付回调需要 Java 生态的成熟中间件，这些差异在单体里很难共存。微服务允许不同服务用不同语言，但代价是运维复杂度线性增加。

注意这三点都有前提条件。如果你的服务发布频率差不多、负载峰值同步、团队只有一种技术栈，那么拆微服务得到的是成本，不是收益。

## 微服务的真实代价

**网络延迟不可忽视**。进程内函数调用通常在纳秒到微秒级，而跨服务的 HTTP/gRPC 调用即使在同机房内也有 0.5ms~2ms 的延迟。单次调用不明显，但如果一个游戏操作需要串行调用 5 个服务，延迟就叠加到 5~10ms，对实时战斗来说已经不可接受。

**分布式事务是持续的痛**。玩家购买道具需要扣钱包余额 + 增加背包物品。在单体里是一个数据库事务，回滚简单。在微服务里，钱包和背包是两个服务，要么引入 Saga 模式（每一步都要补偿逻辑），要么用两阶段提交（性能损耗大、死锁风险高）。每一个跨服务的状态一致性需求都是额外的复杂度预算。

**运维复杂度不是线性增加，而是指数级增加**。10 个微服务意味着：10 套独立的 CI/CD 流水线、10 套配置管理、10 套日志收集、独立的服务发现、独立的健康检查、独立的容量规划。如果没有完善的基础设施（K8s、服务网格、分布式链路追踪），这些运维工作会压垮团队。

**接口版本管理是长期债务**。服务 A 调用服务 B，B 修改了接口，A 需要同步更新。在单体里编译时就能发现，在微服务里可能在运行时才爆炸。版本兼容问题会随着服务数量增长而积累。

## 游戏后端的特殊性

游戏后端有几个特征让它和普通 Web 后端的架构决策不一样：

**高频小消息**。玩家每帧发送操作指令，服务器推送同步状态，消息频率可以达到每秒 20~60 次。这种场景对延迟极度敏感，每增加一次跨服务调用都是可见的代价。

**有状态服务**。战斗服务必须维护房间状态，玩家 A 的攻击命中玩家 B，这个计算必须在同一个进程上下文里完成，不能轻易水平分散。有状态服务的扩缩容和无状态服务完全不同，需要一致性哈希或显式路由。

**实时性要求**。竞技类游戏的服务端帧率可能是 20~64Hz，对应帧间隔是 15ms~50ms。如果处理一帧的逻辑需要等待多次跨服务调用，整个房间的节奏就会崩掉。实时战斗的核心逻辑通常必须在同进程内完成。

**玩家生命周期复杂**。一个玩家从登录到游戏结束，会涉及账号验证、会话管理、匹配等待、战斗过程、结算、背包更新、排行榜刷新……这条链路上任何一个环节的超时都会影响玩家体验。链路越长，中间断点越多，端到端的可用性就越低（10 个 99.9% 的服务串联，整体可用性是 99%）。

## 什么阶段适合什么架构

### 起步阶段：单体是对的

项目刚开始，需求不稳定，团队规模小（3~8 人），领域边界还没有沉淀清楚。这个阶段拆微服务是自毁。一个设计良好的单体——模块化代码结构、清晰的内部接口、独立的数据表前缀——已经能满足 95% 的需求，同时保留未来拆分的可能性。

判断标准：单体能支撑当前和未来 12 个月的需求，就没必要拆。

### 增长阶段：按痛点拆

当以下痛点出现时，才值得考虑拆分某个模块：
- 某个模块的发布频率明显高于其他模块，且每次全量发布风险都很高
- 某个模块的峰值负载和其他模块完全不同，独立扩缩能显著降低成本
- 某个模块需要完全不同的技术栈（比如实时战斗用 C++ 或 Rust）
- 某个模块需要不同的合规或安全隔离要求（比如支付）

这个阶段应该是**按业务域谨慎拆分**，而不是把所有模块都拆开。先拆最痛的那一个，验证拆分带来的收益是否覆盖了引入的复杂度。

### 规模化阶段：微服务网格

当团队规模达到 30 人以上，多个小组并行开发不同功能，而且已经有完善的基础设施支撑（服务发现、链路追踪、灰度发布），微服务才真正能发挥优势。这个阶段的拆分是组织驱动的，每个服务对应一个小组的边界（康威定律）。

## 分布式单体：比单体更糟糕的形态

分布式单体的症状：
- 多个服务共享同一个数据库，通过直连数据库共享状态
- 服务之间存在大量同步调用链，任何一个服务挂掉都会导致全链路崩溃
- 发布时必须协调多个服务同时上线，没有真正做到独立部署
- 新增功能需要修改多个仓库，变更成本比单体还高

分布式单体是拆分过程中最常见的陷阱。它同时拥有微服务的运维复杂度和单体的耦合问题，两者的缺点全占，优点一个没有。

形成原因通常是：拆了服务边界，但没有拆数据边界。每个服务有独立的代码仓库，但它们还是直连同一个 MySQL，或者通过数据库表来共享状态。

正确的做法是：服务边界必须和数据边界同步拆分。每个服务拥有自己的数据存储，其他服务只能通过 API 访问，不能直连数据库。

## 游戏公司的真实拆分路径

观察过多家游戏公司的后端架构演进，有一个相对典型的路径：

**第一阶段（0~1年）**：大单体。一个 Go 或 Java 项目，按功能划分包（账号、背包、战斗、社交……），共享一个数据库，一起发布。

**第二阶段（1~2年）**：支付和账号先拆。支付因为合规要求必须隔离，账号因为被所有业务依赖且变更频率低，先独立出来。其他模块还是单体。

**第三阶段（2~3年）**：战斗服独立。当玩家量上来之后，战斗服的技术要求（低延迟、有状态、可能需要 C++）和其他业务差异太大，作为独立集群运行。

**第四阶段（3年以上）**：按业务域拆分。社交（好友、聊天）、内容（活动、商城）、对局（匹配、排行榜）逐步拆开，每个域有自己的团队和发布节奏。

每一次拆分都有明确的驱动力，不是因为"微服务是趋势"，而是因为旧有架构已经出现了无法通过优化解决的瓶颈。

## 工程边界：哪些问题微服务解决不了

微服务不能解决以下问题：
- **糟糕的代码质量**：把一堆烂代码拆成多个微服务，得到的是多堆烂代码
- **团队协作问题**：如果团队沟通效率低，微服务会放大这个问题，而不是解决它
- **数据库瓶颈**：如果问题是单个 MySQL 撑不住，微服务帮不上忙，应该先优化查询、引入缓存、读写分离
- **实时战斗的延迟问题**：战斗逻辑的延迟是网络延迟 + 计算延迟，拆成微服务只会增加延迟

把微服务当成银弹，是很多团队踩坑的根本原因。

---

## 最短结论

游戏后端应该先用单体起步、按真实痛点拆分、用数据边界确认拆分是否彻底——拆服务之前先问：这个拆分带来的运维成本，有没有被独立扩缩或独立发布的收益覆盖？
