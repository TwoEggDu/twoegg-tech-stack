## 一次"全部用 REST"的代价

某团队在微服务改造完成后，发现了一个奇怪的现象：玩家完成战斗到看到结算界面，需要等待 3~4 秒。往前追，战斗结算的调用链是这样的：

战斗服 → POST /settlement（结算服）→ POST /inventory/update（背包服）→ POST /achievement/check（成就服）→ POST /leaderboard/update（排行榜服）→ POST /notification/send（通知服）

全程 5 次串行 HTTP 调用，每次平均 200ms（包含网络往返、JSON 序列化、服务处理时间），累计超过 1 秒，加上战斗服本身的处理时间，3~4 秒是必然结果。

更糟的是，成就服务有一天因为数据库慢查询出现了 2 秒超时，整条链路全部卡住，玩家看到的是结算界面长时间 loading，然后超时失败。最后他们发现，成就更新对玩家的**即时体验没有影响**——成就通知本来就是弹出式的，完全可以异步处理。

这个故事说明的问题是：**通信模式的选择不是技术偏好，而是业务语义的体现**。

---

## 两种通信模型的本质差异

理解 gRPC 和消息队列（MQ）的区别，要先理解它们背后的通信模型：

**同步调用（RPC）的语义**：服务 A 调用服务 B，A 等待 B 的响应，然后根据响应决定下一步。这隐含了一个假设：A 的后续逻辑**依赖** B 的执行结果。调用期间，A 的处理线程（或协程）被阻塞等待。

**异步消息的语义**：服务 A 发布一个事件到消息中间件，然后继续执行下一步，不等待任何消费者的响应。消费者 B 在自己的节奏下处理这个事件。这隐含了一个假设：A 只需要通知"某件事发生了"，不关心 B 什么时候处理、处理结果是什么。

这两种语义的差别，就是选择通信模式的核心判断标准：**我的下一步操作是否依赖对方的响应？**

如果是，用同步 RPC（gRPC）。如果不是，用异步消息（MQ）。

## 为什么不全用 HTTP/REST

很多人的第一反应是：HTTP 用得最熟，直接 REST 调用就行了，为什么要引入 gRPC 这种额外复杂度？

REST over HTTP 在服务间通信上有几个具体问题：

**协议效率**：HTTP/1.1 的文本协议（JSON）序列化和反序列化开销明显。一个包含 100 个字段的结构，JSON 序列化后可能是 5KB，而 Protobuf 序列化只有 1KB。在服务间每秒数万次调用的场景下，这个差异是可见的。

**类型安全**：REST 接口的类型约束依赖文档（OpenAPI/Swagger）。调用方和被调用方的接口类型是通过人工维护对齐的，字段名拼错、类型不匹配，在编译期无法发现，只在运行时崩溃。

**接口版本管理**：REST 没有内建的向后兼容机制，字段变更需要依靠约定（不删字段、只加字段）。gRPC + Protobuf 有明确的字段编号规则，版本演化有约束可依。

**双向流**：游戏服务间的一些场景需要服务端主动推送数据（比如战斗服向匹配服报告房间状态），HTTP/1.1 不支持服务端推送，需要轮询或 WebSocket 打补丁。gRPC 原生支持双向流（Bidirectional Streaming）。

## gRPC 的优势和代价

**优势**：

- **强类型**：Protobuf IDL 定义接口，编译生成客户端和服务端的 stub 代码，字段类型在编译期校验。接口变更必须修改 .proto 文件，变更是可见的、可 review 的。
- **高效序列化**：Protobuf 是二进制协议，比 JSON 序列化快约 3~10 倍，消息体积小约 3~5 倍。在高频调用场景下，这个性能差距是显著的。
- **双向流**：支持 Server Streaming、Client Streaming 和 Bidirectional Streaming，适合游戏服务中的状态同步场景。
- **HTTP/2 复用**：gRPC 基于 HTTP/2，多个 RPC 调用可以在同一条 TCP 连接上并发，减少连接开销。
- **内建截止时间（Deadline）**：每个 gRPC 调用都可以设置 deadline，超时后客户端取消请求，避免无限等待。

**代价**：

- **Protobuf 管理负担**：所有服务共享的 .proto 文件需要统一管理（通常放在一个独立的 proto 仓库），接口变更需要所有依赖方同步更新，增加了协调成本。
- **服务耦合**：gRPC 调用是直接的服务依赖。如果服务 B 宕机，服务 A 的调用立即失败。需要加入熔断（Circuit Breaker）、重试、fallback 机制才能保证可用性。
- **调试复杂度**：二进制协议不能直接用 curl 调试，需要专门的工具（grpcurl、Postman gRPC）。

## 消息队列的优势和代价

**优势**：

- **解耦**：发布方不需要知道消费方是谁，消费方可以随时上下线，不影响发布方。
- **削峰**：突发流量写入 MQ，消费者按自己的处理能力消费，避免下游服务被打垮。
- **可重试**：消息消费失败后，MQ 可以重试，消费者只需要实现幂等逻辑。
- **扇出**：一个事件可以被多个消费者同时消费，不需要发布方逐一通知。

**代价**：

- **延迟不确定**：消息从发布到被消费有延迟，取决于消费者的处理速度和 MQ 的积压情况。不适合需要立即得到结果的场景。
- **最终一致性**：使用 MQ 意味着放弃了即时一致性，接受最终一致性。这在某些业务场景（比如支付扣款）是不可接受的。
- **幂等要求**：MQ 可能重复投递消息（网络抖动时），消费者必须实现幂等逻辑，保证重复处理同一条消息不产生重复效果。
- **调试困难**：异步链路的问题排查比同步调用链路复杂，需要在消息里携带完整的链路 ID。

## 游戏后端的通信矩阵

用具体的游戏场景来说明选择原则：

| 场景 | 通信方式 | 原因 |
|------|----------|------|
| 客户端请求验证 Token（API 网关 → 账号服） | 同步 gRPC | 必须立即得到验证结果，请求不能继续直到确认身份 |
| 玩家匹配成功，通知战斗服创建房间（匹配服 → 战斗服） | 同步 gRPC | 需要立即获知房间 ID 和服务器地址，然后返回给客户端 |
| 战斗结束，更新玩家背包（战斗服 → 背包服） | 异步 MQ | 背包更新不需要在战斗服响应链路里完成，可以延迟，且支持重试 |
| 战斗结束，更新排行榜（战斗服 → 排行榜服） | 异步 MQ | 排行榜更新有延迟可以接受，需要扇出给多个消费者 |
| 玩家购买道具，扣钱包余额（支付服 → 钱包服） | 同步 gRPC + Saga | 强一致性要求，扣款失败需要立即知道 |
| 战斗结束，触发成就检查（战斗服 → 成就服） | 异步 MQ | 成就不需要即时更新，失败可重试，不影响结算主流程 |
| 发送系统公告（运营系统 → 通知服） | 异步 MQ | 通知可以延迟，推送量大需要削峰 |
| 战斗服向匹配服报告房间满人（战斗服 → 匹配服） | 同步 gRPC | 匹配服需要立即标记房间不可用，否则可能继续投递玩家 |

## gRPC 双向流在游戏中的特殊场景

gRPC 的 Bidirectional Streaming 在游戏后端有一个重要用途：**服务间的状态同步**。

战斗服需要定期向网关服汇报每个房间里的玩家连接状态（哪些玩家在线、哪些断线重连），网关服也需要向战斗服推送玩家的重连事件。这种双向、持续的通信场景，用普通 RPC（每次单独调用）开销太高，用 MQ（需要两个 topic 互相发）很笨重，而 gRPC 双向流可以在一条持久连接上双向传递数据，是这个场景的最优解。

## 分布式调用链的可观察性

无论是 gRPC 还是 MQ，分布式服务之间的调用链路可观察性都是必须要处理的问题，否则线上出问题时排查极其困难。

**TraceID 传递**是最基础的要求：每个外部请求在入口（API 网关）生成一个全局唯一的 TraceID，在服务间调用时，TraceID 必须被显式传递。

- gRPC：在 Metadata（类似 HTTP Header）中传递 `x-trace-id`
- MQ：在消息的 Header 或 Payload 中携带 `trace_id` 字段

**结构化日志**：每条日志必须包含 TraceID，这样可以用 TraceID 在日志系统（ELK、Loki）里过滤出一次请求的完整链路。

**链路追踪**：引入 OpenTelemetry（OTel）标准，接入 Jaeger 或 Tempo 等分布式追踪系统，可以可视化地看到一次请求经过了哪些服务、每个服务耗时多少、在哪里超时或报错。这在定位串行调用链性能问题时极其有价值。

**关键点**：异步 MQ 的 TraceID 传递容易被忽视，但它其实更重要——因为异步链路出问题时，没有 TraceID 就完全无法串联上下游的日志。必须在消息格式规范中强制要求 trace_id 字段。

## 幂等设计：MQ 可靠性的前提

使用 MQ 的服务必须为消费者实现幂等逻辑，因为 MQ 在故障恢复时可能重复投递消息（Kafka 的 at-least-once 语义）。

游戏后端的幂等实现通常依赖业务唯一键：
- 背包更新：以 `(playerID, transactionID)` 作为幂等键，同一笔交易的背包更新只执行一次
- 排行榜更新：以 `(playerID, matchID)` 作为幂等键，同一场对局的分数只更新一次
- 成就解锁：天然幂等，已解锁的成就再次处理不会产生副作用

在 Redis 里存储最近处理过的幂等键（TTL 设为消息可能延迟的最大时间），消费者处理前先查幂等键，如果已存在则跳过，如果不存在则处理并写入幂等键。

---

## 工程边界：通信模式不该做的事

同步 gRPC 不应该承担的职责：
- **日志和统计数据上报**：这类"发出去不管结果"的场景必须用 MQ 或 UDP 上报，不应该用同步调用
- **广播通知**：给所有在线玩家推送公告，不应该用 RPC 遍历调用，应该用 MQ 扇出或推送服务

异步 MQ 不应该承担的职责：
- **需要事务保证的操作**：比如"扣钱 + 发货"必须同时成功或同时失败，MQ 的异步解耦反而使事务管理更复杂
- **需要立即反馈的用户操作**：玩家点击"购买"按钮，必须立即知道成功还是失败，不能异步化

---

## 最短结论

选同步 gRPC 还是异步 MQ，唯一的判断标准是：下一步逻辑是否依赖对方的响应——依赖就用 gRPC，不依赖就用 MQ，强行同步化只是把延迟累积进请求链路，强行异步化则是把一致性问题留给自己处理。
