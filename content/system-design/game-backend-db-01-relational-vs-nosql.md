## 这篇文章在解决什么问题

游戏后端项目启动的第一周，技术选型会议上必然出现这个问题：我们用 MySQL 还是 MongoDB？

这个问题反复出现，原因不是团队技术水平不够，而是这两个问题在很多场合被搞混了：

- 第一个问题：**数据本身的结构特征是什么？**
- 第二个问题：**系统对数据的访问模式是什么？**

只回答第一个，会选错技术。只回答第二个，会设计出难以维护的 schema。两个问题都回答了，选型才有依据。

游戏项目里同时存在关系型和非关系型数据库，几乎是行业常态，不是"历史包袱"，而是因为游戏数据本身就不是同质的——玩家账号、背包物品、排行榜积分、行为日志，它们的结构特征和访问压力完全不同。

---

## 关系型数据库的核心能力

MySQL 和 PostgreSQL 的本质优势，不是"有表结构"，而是**事务与一致性保证**。

**ACID 是真实的工程保障**，不是理论概念。在游戏场景里，以下操作必须是原子的：

- 玩家购买道具：从金币账户扣款 + 背包增加物品，两个操作要么全成功，要么全回滚
- 排行榜积分更新：战斗结束写入分数 + 更新段位，不允许写了分数但段位更新失败的中间状态
- 好友关系建立：A 的好友列表加 B + B 的好友列表加 A，对称关系不能只写一半

PostgreSQL 比 MySQL 在事务隔离级别上更严格（真正支持 Serializable 而不是退化为 Snapshot Isolation），如果你的游戏有复杂的跨表一致性需求，PostgreSQL 值得认真考虑。

**关系型数据库擅长的另一件事是联表查询**。"查询在某个公会里、等级大于 60、本周有登录的活跃玩家"，这类多条件过滤在 SQL 里是自然的。MongoDB 的聚合管道（Aggregation Pipeline）虽然也能做，但写起来更繁琐，优化也更难。

关系型的约束：**Schema 必须预先定义**。加一列要 ALTER TABLE，对大表来说这个操作有锁风险（虽然 MySQL 5.6+ 和 PostgreSQL 12+ 的 Online DDL 缓解了很多），但本质上，表结构变更有成本。

---

## 非关系型数据库的核心能力

MongoDB 的核心不是"灵活的文档结构"，而是**水平扩展和写入吞吐**。

文档模型（Document Model）解决的问题是：当一条逻辑记录的字段不固定时，不需要为每种变化新增列。游戏里典型的例子是**每个玩家的技能树配置**——不同职业的玩家，技能树结构完全不同，强行用关系表来存会出现大量 NULL 列，或者需要 EAV（Entity-Attribute-Value）这类反范式设计。

MongoDB 的另一个优势是**原生支持嵌套文档**。玩家的装备信息（武器、防具、饰品），连同每件装备的附魔属性、强化等级，可以直接以嵌套 JSON 的形式存在一个文档里，读取时一次 I/O 拿到所有数据，不需要 JOIN。

**水平分片（Sharding）是 MongoDB 的原生能力**。关系型数据库（尤其是 MySQL）的分库分表需要借助外部中间件（ShardingSphere、Vitess），配置复杂。MongoDB 的 Sharded Cluster 是内置的，对于需要快速扩容的游戏后端，这个特性有实际价值。

MongoDB 的约束：**不支持真正的跨文档 ACID 事务**（4.0 版本后虽然加入了多文档事务，但性能代价显著，不适合高频使用）。如果业务逻辑需要跨多个集合的原子更新，MongoDB 不是好选择。

## 游戏数据的特征分析

游戏数据不是铁板一块，分开看：

### 玩家基础档案
字段相对稳定：user_id、昵称、等级、经验、创建时间、最后登录时间。这是**高读低写**的数据——每次进入游戏都读，但修改频率较低。关系型数据库的缓存友好性（MySQL 的 Query Cache，或配合 Redis）和索引能力在这里有优势。

典型选择：**MySQL / PostgreSQL**

### 背包与物品
每个玩家持有的物品数量从几个到几千个不等，每件物品有独立的实例属性（强化等级、附魔、耐久度）。这是**中等写频率 + 复杂结构**的数据。

如果用关系型，一般设计成 `item_instance` 表（每行一件物品），用 `user_id` 索引。这是最正确的设计，后面文章会详细说。
如果用 MongoDB，一个文档里嵌套所有物品，读取快但写入时更新单件物品需要用 `$set` 精确操作，不注意会踩坑。

两种方案都可行，关键是别用 JSON 大字段把所有物品塞成一列——这个选择在两种数据库里都是错的。

### 排行榜
**排行榜不适合任何一种数据库的 ORDER BY 查询**。不管是 `SELECT ... ORDER BY score DESC LIMIT 100` 还是 MongoDB 的 `sort()`，在千万级数据量下都扛不住实时查询压力。

排行榜的正确方案是 **Redis ZSet（有序集合）**，数据库只负责持久化，Redis 负责实时排名计算。这个选择和关系型 vs 非关系型无关，是另一个维度的问题。

### 行为日志与统计数据
玩家的行为日志（登录、战斗、消费）是**高写低读**的数据，写入频率极高，查询通常是离线统计而非实时业务逻辑。

这类数据适合用 MongoDB（高吞吐写入）或者直接走消息队列（Kafka）后落到列式存储（ClickHouse）做分析。用 MySQL 存行为日志是典型的错误选型——写入瓶颈会很快出现，而且 B-Tree 索引对时间序列数据的压缩效率也不高。

## 常见错误选型的根源

### 全上 MongoDB
动机通常是："文档模型灵活，我们游戏数据结构经常变，不想每次改表"。

问题在于：当需要跨集合事务时（购买道具扣金币 + 加物品），MongoDB 的多文档事务性能差，要么妥协一致性，要么接受性能下降。核心业务数据（账号、道具、货币）的一致性不能妥协。

### 全上 MySQL
动机通常是："关系型数据库成熟稳定，团队熟悉"。

问题在于：当日志数据、行为数据开始膨胀，MySQL 的写入性能成为瓶颈。加上游戏内容频繁迭代，物品属性字段频繁扩展，ALTER TABLE 的运维成本开始爆发。

### 正确思路
不是"选一个"，而是**按数据特征分类放置**：

| 数据类型 | 推荐方案 | 理由 |
|---------|---------|------|
| 玩家账号、货币、核心状态 | MySQL / PostgreSQL | 事务保证，高一致性 |
| 背包物品实例 | MySQL（item_instance 表） | 结构规整，需要行级锁 |
| 技能树、自定义配置 | MongoDB | 结构变化多，嵌套文档适合 |
| 排行榜 | Redis ZSet（DB 做持久化） | 实时排名性能 |
| 行为日志 | MongoDB 或 Kafka+ClickHouse | 高写吞吐，离线分析 |

## 工程边界：数据库不该吞掉什么

**数据库不是缓存层**。大量游戏后端把热数据（在线玩家状态、好友列表）直接放在数据库读，每次请求都打 DB，这是数据库承担了缓存的职责，迟早撑不住。

**数据库不是消息队列**。用一张 `event_queue` 表来实现异步任务派发，靠轮询 `status = 'pending'` 来消费，这是反模式。轮询产生大量无效 IO，高并发下锁竞争严重。

**数据库不是文件系统**。把大型二进制（录像、截图）存在 BLOB 字段里，是典型的职责越界。

---

## 最短结论

游戏数据天然是异构的，用数据特征（结构稳定度、访问模式、一致性要求）来驱动选型，而不是用团队习惯或某种技术的"灵活性"。
