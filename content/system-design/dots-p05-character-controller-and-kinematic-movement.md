P01 已经说明了物理世界怎样接进 ECS，P02 解释了物理数据怎么拆，P03 和 P04 分别讲了查询与事件回流。到了角色控制这一步，最容易暴露的事实是：**Character Controller 之所以难，不是因为碰撞很复杂，而是因为它把输入、移动、碰撞、动画和权威边界都压在同一条链上**。

很多团队一开始都会把角色控制当成纯 Physics 问题，后来才发现它其实是边界问题。你要决定谁提供输入、谁计算位移、谁负责碰撞修正、谁写回最终位置、谁驱动动画表现。只要这几个角色没分开，代码就会很快写散。

---

## 为什么角色控制不是纯物理问题

角色控制看起来像“往前走、撞墙、上台阶、下坡”，但实际上它至少跨了四层。

第一层是输入。玩家按下什么、服务器接受什么、当前帧要不要预测，先得说清楚。  
第二层是运动意图。角色想往哪里走、是不是要跳、是不是要进入冲刺。  
第三层是碰撞和空间修正。这个意图能不能落地，是否被墙、坡、台阶挡住。  
第四层是表现。最终的位置、朝向、速度，怎样交给动画和表示层。

如果把这些都压进一个“角色控制器”类里，短期能跑，长期一定会变成边界混乱。因为每次修改，不是在改一个物理参数，而是在改多条链路的交接方式。

## 角色控制器真正站在哪条边界上

在 DOTS 里，更稳的理解是：角色控制器站在 `Simulation Layer` 和 `Representation Layer` 的交界上。

它不是单纯的物理体，也不是单纯的动画驱动器。它通常要做三件事：

- 接收输入或意图
- 把意图转换成可验证的移动请求
- 将修正后的结果交给仿真和表现两侧

这意味着，角色控制器最怕的不是“碰撞多”，而是“写入点太多”。你既在输入阶段改一次，又在物理阶段改一次，又在动画阶段再改一次，最后同一帧里谁都以为自己是权威。

## 运动链应该怎么切

更干净的做法，是把 kinematic movement 看成一个三段链。

```text
Input / Intent
-> Movement Solve
-> Collision Resolve
-> Write Back
```

最小伪代码可以这样写：

```csharp
public partial struct CharacterMoveSystem : ISystem
{
    public void OnUpdate(ref SystemState state)
    {
        // 1. 读输入 / 意图
        // 2. 计算想要移动的位移
        // 3. 用 Physics Query 做碰撞修正
        // 4. 把最终结果写回仿真状态
        //
        // 动画、特效、镜头只读这里输出的结果，不反向改同一份权威数据。
    }
}
```

这里的重点不是 API，而是权威方向。输入和意图先进入仿真链，碰撞修正只负责约束位移，最终写回只发生一次。表现层拿到的是“已经决定好的结果”，不是再反过来抢权。

## 为什么它总和动画、输入、资源一起出问题

角色控制最容易散，往往不是物理写错，而是边界没切清。

比如：

- 输入系统直接改 Transform，物理系统又在下一步覆写它。
- 动画根运动既改位移又改朝向，物理修正时又把结果打回。
- 镜头和角色状态耦合，导致每次位移修正都牵动表现层。

这些问题的共同点是：**角色移动不是唯一权威，多个系统都在试图改同一份状态**。只要发生这种情况，调试时你永远看见的是“最后结果不对”，却很难知道是哪一层先写坏了。

所以更稳的原则是：

- 输入只产出意图
- 仿真只算位移和碰撞修正
- 表现层只消费结果

如果哪一层要反向改前一层，就说明边界已经开始失控。

## Kinematic 为什么更容易写散

Kinematic 移动看起来比动态体简单，因为你不用完整交给物理积分，但它其实更考验边界设计。

原因很直接。动态体很多时候“交给物理就行”，而 kinematic 角色则必须自己负责移动意图、碰撞约束、贴地、台阶、斜坡、分离和重叠修正。你不是少写了物理，而是把更多判断提前搬到了自己的系统里。

这也是为什么角色控制经常会和 `Physics Query`、`DistanceQuery`、`CollisionEvents` 连在一起。它们不是零散工具，而是同一条边界链上的不同环节。问题一旦没切好，角色控制器就会变成一堆补丁：地面一个补丁，台阶一个补丁，斜坡一个补丁，贴墙又一个补丁。

## 什么时候该用，什么时候不该用

如果你的项目里只是少量角色、控制规则简单、移动行为不需要复杂修正，轻量的运动系统可能比完整角色控制器更合适。

如果你的角色控制需要稳定地处理：

- 地面检测
- 坡道与台阶
- 贴墙与滑动
- 网络同步
- 动画和表示层解耦

那它就不再是一个“随便写个脚本就行”的问题，而是必须明确边界的仿真组件。

## 常见踩坑

第一个坑，是把角色控制器写成“直接改位置”的快捷方式。这样最容易绕过碰撞和同步边界。

第二个坑，是把输入、移动和表现混在同一个系统里。看起来少了系统，实际上只是把责任揉烂了。

第三个坑，是想让 kinematic 角色同时拥有动态体和控制器的全部权威。结果通常是互相覆盖。

第四个坑，是在角色控制里反复写结构变更。控制问题本来应该是局部位移修正，被写成了世界布局修改。

## 小结

Character Controller 与 kinematic movement 的核心，不是“怎么走得更像物理”，而是“怎么把输入、碰撞、位移和表现层的权威切清楚”。

只要把它当成边界问题而不是纯物理问题，角色控制链就能稳定拆开；否则你后面修的每一个 bug，都会变成对同一条混乱链路的补丁。
