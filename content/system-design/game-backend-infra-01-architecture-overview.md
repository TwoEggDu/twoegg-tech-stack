## 问题：一张"大泥球"是怎么长出来的

很多项目的后端都有一个共同的起点：一台服务器，一个进程，搞定所有事情。

登录验证在这里，战斗逻辑在这里，数据存取在这里，推送消息也在这里。早期运转顺畅，毕竟代码量小，心智负担低。直到某一天：

- 战斗逻辑的 CPU 峰值把数据库连接池打满了
- 一个新功能需要读玩家数据，但登录服和战斗服各自维护了一份 session 缓存
- 推送消息的长连接数暴增，把原本处理业务逻辑的线程池也卡住了
- 一台逻辑服宕机，所有在线玩家同时断线

这时候你才意识到：问题不是出在某行代码上，而是架构层面没有划清各层的职责与边界。

游戏后端的架构挑战和传统 Web 后端有本质区别。HTTP 的"请求-响应"模型天然无状态，Web 应用可以通过横向扩容和 Load Balancer 轻易水平扩展。但游戏后端大量依赖**长连接**、**有状态会话**和**实时推送**，这三件事叠在一起，决定了它必须分层——每一层解决一类问题，不能混着来。

## 抽象模型：四层职责划分

一个典型的多人在线游戏后端，可以抽象为四层：

```
客户端
   │
   ▼
┌──────────────────────────────┐
│         接入层 / 网关          │  ← 连接管理、鉴权、路由
└──────────────────────────────┘
   │
   ▼
┌──────────────────────────────┐
│         逻辑服                │  ← 游戏规则、状态机、业务逻辑
└──────────────────────────────┘
   │
   ▼
┌──────────────────────────────┐
│         数据服 / 存储层        │  ← 持久化、缓存、数据一致性
└──────────────────────────────┘
   │
   ▼
┌──────────────────────────────┐
│         推送服 / 消息分发层    │  ← 跨服推送、广播、通知
└──────────────────────────────┘
```

这不是唯一的分法，但它覆盖了绝大多数游戏后端中反复出现的职责边界问题。

### 接入层 / 网关（Gateway）

网关是所有客户端连接的入口。它的核心职责只有三件事：

1. **连接管理**：维护 TCP / WebSocket 长连接，处理心跳、断连、重连
2. **鉴权**：验证 token，确保连接属于合法玩家
3. **路由**：把通过鉴权的消息转发给对应的逻辑服

网关本身**不执行业务逻辑**。它不知道"战斗开始"意味着什么，它只知道"这条消息要转给 Room 服务的某个节点"。

网关通常是**无状态的**，或者说它持有的状态极度轻量——只有连接映射表（connection_id → player_id → target_service）。这使得网关可以水平扩展：多台网关机器共享同一套路由配置，任何一台宕机都不影响其他连接。

**边界警告**：网关不应该持有游戏会话状态。一旦网关开始缓存玩家的装备列表或技能冷却状态，它就从路由器变成了状态服务，水平扩展能力立刻丧失。

### 逻辑服（Game Logic Server）

逻辑服是游戏真正的大脑。战斗计算、状态机跳转、道具扣减、匹配逻辑——全部在这里。

逻辑服通常是**有状态的**。一个房间（Room）的所有玩家状态都驻留在同一个逻辑服节点的内存里，因为这些状态需要以极低延迟互相访问。如果每次判断"子弹是否击中"都要走一次数据库查询，帧率立刻崩溃。

有状态意味着水平扩展的规则不同于无状态服务：你不能把同一个房间的逻辑拆给两台机器同时处理。扩展的方式是**分片**（sharding）——不同房间分配给不同节点，单个节点内部的逻辑仍然是单线程或轻量并发。

**边界警告**：逻辑服不该直接操作数据库。逻辑服关心的是"当前这局的状态"，持久化是数据服的职责。如果逻辑服直接写数据库，峰值战斗负载会直接打垮存储层。

### 数据服 / 存储层（Data Service）

数据服负责所有需要持久化的数据操作，包括：

- 玩家档案（等级、货币、背包）
- 战局结算记录
- 排行榜
- 跨会话的冷数据查询

数据服通常是**无状态的服务 + 有状态的存储**：服务本身可以水平扩展，存储（MySQL、Redis、MongoDB）负责维护数据一致性。

数据服的关键设计问题是**读写分离**和**缓存策略**。玩家道具数量这类高频读写数据走 Redis，结算记录走 MySQL 主库写入，历史查询走从库读取。

**边界警告**：数据服不处理游戏规则。它不知道"金币扣减到 0 以下应该报错"——这个规则属于逻辑服。数据服只管存什么、怎么存、存得一致不一致。

### 推送服 / 消息分发层（Push Service）

推送服解决一个特定问题：**如何把消息从服务端主动推送给客户端，尤其是跨逻辑服节点的推送**。

典型场景：玩家 A 在逻辑服节点 1，玩家 B 在逻辑服节点 2，节点 1 需要通知节点 2 的玩家"好友上线了"——这条消息需要穿越节点边界，找到玩家 B 当前连接的网关，再推送出去。

推送服通常依赖一个中央的**连接路由表**（player_id → gateway_id → connection_id），配合消息队列或 pub/sub 系统（如 Redis Pub/Sub、Kafka）实现跨节点广播。

**边界警告**：推送服不负责消息的生成逻辑。它不判断"该不该推送"，只负责"怎么把消息送达"。业务规则（谁应该收到这条消息）属于逻辑服的职责。

## 具体实现：连接的生命周期

把四层映射到一条完整的玩家连接生命周期，能让职责划分更清晰：

```
1. 连接建立
   客户端 → TCP 握手 → 网关
   网关分配 connection_id，进入未鉴权状态

2. 鉴权
   客户端发送 Login 包（含 token）
   网关验证 token（调用 Auth 服务，或本地验证 JWT）
   验证通过 → 建立 player_id ↔ connection_id 映射
   网关通知路由表：player X 在 gateway Y

3. 进入游戏逻辑
   客户端发送加入房间、匹配等请求
   网关根据 player_id 路由到目标逻辑服节点
   逻辑服维护房间状态，处理游戏消息

4. 游戏中的数据读写
   逻辑服读取玩家背包 → 调用数据服（通过 RPC 或 REST）
   战局结束 → 逻辑服发结算数据给数据服持久化

5. 推送消息
   逻辑服产生需要推送的事件（如队友击杀、状态变化）
   调用推送服接口，推送服查路由表，找到目标网关，投递消息

6. 断线
   网关检测到连接断开（心跳超时或 FIN 包）
   清理连接映射，通知逻辑服 player X 下线
   逻辑服处理玩家断线状态（AI 接管 or 直接退出房间）
   推送服清理该 player 的路由条目
```

每一个步骤都在明确的层内完成，不跨层直接操作数据。这就是分层的意义：任何一层出问题，影响范围是可控的。

## 和传统 Web 后端的关键差异

| 维度 | 传统 Web 后端 | 游戏后端 |
|------|---------------|----------|
| 连接模型 | HTTP 短连接，请求完即关 | TCP/WS 长连接，会话持续数小时 |
| 状态管理 | 服务无状态，状态全在 DB | 逻辑服有状态，内存热数据 |
| 消息方向 | 客户端主动请求 | 双向，服务端主动推送 |
| 扩展方式 | 加机器即扩容 | 有状态分片，不能随意迁移 |
| 延迟要求 | 几百毫秒可接受 | 竞技类需要 <50ms |
| 故障模型 | 单服宕机，重启即恢复 | 宕机丢失内存状态，需要 checkpoint 机制 |

传统 Web 框架（Spring Boot、Django 等）的设计假设是"无状态 + DB 持久化"，这个假设在游戏后端中不成立。直接把 Web 框架用于游戏后端，通常会在长连接管理和有状态会话两个地方遭遇严重问题。

## 架构变体：不同游戏类型的四层形态

四层架构不是固定的，它会随游戏类型产生变体：

**休闲手游（消除、卡牌）**

逻辑相对简单，可以合并部分层次。逻辑服可以是无状态的（每次请求从 Redis 读取玩家状态，计算结果写回），类似 Web 后端。网关层甚至可以用 Nginx + WebSocket 代替自研网关。

**MOBA / 竞技类**

对延迟极度敏感。逻辑服通常是 dedicated game server，状态全在内存，帧同步或状态同步要求网络包间隔 <50ms。推送服通常直接由逻辑服承担（房间内的推送走直连，不经过额外的推送层）。数据服在战局中几乎不参与，战局结束后才做结算持久化。

**MMO**

最复杂的形态。逻辑服按地图/区域分片，玩家跨区需要跨节点迁移状态（handoff）。网关需要支持动态路由（玩家从 A 区走到 B 区，网关需要切换目标逻辑服）。数据服压力最大，全服玩家的背包、装备读写都在这里。推送服需要支持全服广播（系统公告）和定向推送（组队通知）。

## 工程边界：这张图不能解决什么

分层架构解决了**职责划分**问题，但它不负责以下工程问题：

- **跨节点的数据一致性**：两台逻辑服同时操作同一个玩家的数据，怎么保证不冲突？这是分布式事务问题，不是架构图能解决的。
- **连接迁移**：玩家断线重连后，内存状态如何恢复？需要 checkpoint 机制和状态序列化设计。
- **热更新**：逻辑服的代码如何在不停服的情况下更新？这涉及进程管理和状态迁移策略。
- **监控与追踪**：跨层的请求链路如何追踪？需要 tracing 系统（OpenTelemetry 等）和统一的日志格式。

---

## 最短结论

游戏后端的四层划分（网关、逻辑服、数据服、推送服）本质上是在分离"连接管理"、"游戏规则"、"数据持久化"、"消息投递"这四类完全不同的技术挑战——每一层职责越纯粹，整个系统就越容易独立扩展和故障隔离。
