# Dedicated Server 架构：无头模式、服务器与客户端的代码共享边界

## 问题空间：为什么有了 Listen Server 还需要 Dedicated Server？

你在做一款 8 人竞技射击游戏，测试阶段用的是 Listen Server——其中一个玩家的客户端同时承担服务器职责。一切看起来运作正常，直到你遇到以下几个让你头痛的问题：

1. 那个承担 Host 职责的玩家拥有零延迟，其他玩家却要承受 80ms 的延迟惩罚。竞技公平性从架构层面就已经崩了。
2. Host 玩家的机器配置低，画面帧率掉到 30fps，连带所有玩家的游戏逻辑帧率也跟着抖动。
3. Host 玩家断线，整局游戏立刻结束。没有迁移机制，没有持久化，一切归零。
4. 你想做防作弊校验，但 Host 玩家的客户端同时运行着"权威服务器"代码，这意味着它能看到所有玩家的位置——Wall Hack 在这里几乎没有技术门槛。

这四个问题都指向同一个根本缺陷：**Listen Server 把玩家身份和服务器职责混在了同一个进程里**。

Dedicated Server（简称 DS）的诞生就是为了从架构上切断这种耦合。

## 抽象模型：Dedicated Server 的定义

从最精简的定义出发：

**Dedicated Server 是一个不渲染画面、不接受玩家直接输入、只运行游戏逻辑的独立进程。**

没有窗口，没有 GPU 调用，没有音频输出，没有"本地玩家"这个概念。它的所有输出只有一种：通过网络把游戏状态广播给连接的客户端。

把这个定义和 Listen Server 并排放：

| 维度 | Listen Server | Dedicated Server |
|------|--------------|-----------------|
| 渲染 | 有（Host 玩家看画面） | 无 |
| 本地玩家 | 有（Host 就是一个玩家） | 无 |
| 延迟公平性 | Host 有优势 | 所有玩家对等 |
| 崩溃影响 | Host 断线，游戏结束 | 服务器崩溃，可重启迁移 |
| 防作弊能力 | 弱（Host 可以看全局状态） | 强（客户端无法访问服务器内存） |
| 部署方式 | 随客户端分发 | 独立运行在数据中心 |

这个对比表揭示了一件事：Listen Server 本质上是个"便利性方案"，适合小规模合作或临时对战；而 Dedicated Server 是"工程性方案"，为竞技公平和生产可靠性而生。

## 代码共享边界：哪些能共享，哪些不能

这是 DS 架构里最容易踩坑的地方。

很多团队一开始这样想：客户端和服务器都是同一套游戏逻辑，共享代码理所当然。这个思路本身没问题，但"共享的边界在哪里"必须非常清晰。

### 可以共享的代码

**游戏规则逻辑（Game Rules）**：伤害计算公式、技能触发条件、胜负判定逻辑——这些必须在客户端（预测）和服务器（权威计算）上一致运行，共享代码是正确做法。

**移动预测逻辑（Movement Prediction）**：客户端用本地输入做预测性移动，服务器用相同的公式做权威计算，结果要能对得上。共享同一套数学代码能保证两端行为一致。

**数据结构和序列化**：网络数据包的结构定义、序列化/反序列化逻辑必须两端完全一致，否则消息解析就会出错。

**状态机定义**：角色的状态机（移动/跳跃/攻击）往往在服务端运行权威版本、客户端运行预测版本，共享状态机定义合理。

### 绝对不能共享的代码

**渲染相关（Rendering）**：所有引擎的渲染管线调用——Unity 的 `Camera`、`Renderer`、`Shader`，Unreal 的 `USceneComponent` 渲染部分——在 DS 上根本不存在 GPU 上下文，调用即崩溃。

**音频系统（Audio）**：`AudioSource`、`AudioListener`、声音优先级管理全部属于这个范畴。DS 上没有输出设备，这些组件不应该被初始化。

**输入系统（Input）**：`Input.GetAxis`、`InputSystem.RegisterInputDevice`——DS 上没有键盘、没有手柄、没有触摸屏。任何依赖物理输入设备的代码都不应该在 DS 上运行。

**UI 系统（UI）**：Canvas、UI 元素、HUD——在没有渲染上下文的环境里这些系统既没意义也会报错。

**平台特定客户端服务**：成就系统、社交 SDK、广告 SDK——这些依赖于有"玩家身份"的运行环境。

一个简单的心智模型：**任何"让玩家看见或感知"的代码，都不属于 DS 的职责范围。**

## 服务器权威（Server Authority）的真实含义

"服务器权威"这个词被滥用了，很多人把它理解成"所有逻辑都在服务器跑"——这个理解是错的，也做不到（延迟让它不现实）。

正确的模型是**服务器是最终裁判，但客户端允许提前预测**。

具体拆解：

**服务器决定的事情（权威计算）**：
- 玩家是否命中了目标（命中判定）
- 角色当前的真实血量
- 物品是否真的被拾取
- 游戏规则条件是否满足（比赛是否真的结束）
- 玩家的合法位置（防止传送作弊）

**客户端模拟的事情（预测/表现）**：
- 本地角色的移动（先行渲染，等服务器校正）
- 特效播放、命中反馈（表现性质，不影响游戏结果）
- UI 状态（弹药量的本地显示，可以比服务器确认快一帧）
- 音效触发

这个分工产生了两个经典问题：**客户端预测回滚（Client Prediction Rollback）**和**延迟补偿（Lag Compensation）**，它们都是对"服务器权威"这个模型的工程补丁，日后会单独成文。

## DS 上各子系统职责的变化

当一个游戏系统从"客户端/主机混合运行"迁移到 DS 架构，各个子系统的职责会发生明确的分工。

### 物理模拟

DS 上的物理模拟是**权威物理**：碰撞检测结果、刚体位置更新都以服务器为准。客户端做的是**预测性物理**，结果要定期和服务器对齐。

但 DS 上不需要运行布料模拟（Cloth Simulation）、GPU 粒子物理——这些是纯表现效果，没有任何游戏逻辑依赖。在 DS 上运行它们只是在白白浪费 CPU。

### AI 系统

DS 上的 AI 是**决策型 AI**：寻路、行为树决策、攻击目标选择——这些结果需要广播给客户端。

客户端上的 AI 是**表现型 AI**：动画混合、视线射线用于播放表情、环境感知用于调整动作姿态——这些不影响游戏逻辑，可以只在客户端本地运行。

一个常见错误：把"敌人朝向"的精确旋转插值放在服务端 AI 里，导致服务端每帧都在做高精度的四元数运算——但这个旋转数据其实只是为了客户端动画表现，服务端不需要这个精度。

### 输入处理

DS 上不存在"本地玩家输入"，但它处理**所有连接玩家的网络输入包**：

1. 客户端在本地读取输入，打上时间戳，打包成输入命令（Input Command）
2. 发送给服务器
3. 服务器接收、排序、应用到游戏逻辑

输入包的时间戳对延迟补偿至关重要——服务器需要知道这个输入是在哪个历史帧发出的，才能正确校验命中结果。

## 为什么竞技类/合作类游戏必须用 DS

技术理由上面已经说清楚了，这里只做一个工程层面的总结：

**竞技游戏（FPS、MOBA、格斗）**：需要对等延迟和防作弊。Listen Server 的 Host 优势在职业赛场是不可接受的。作弊检测需要服务器作为信任根——客户端本地跑的代码永远可以被修改。

**合作游戏（副本、合作战役）**：需要持久化和断线重连。副本进行到一半 Host 玩家断线，其他玩家的游戏进度不应该丢失。DS 的进程独立性保证了玩家连接状态和游戏状态可以分离。

**大规模多人（MMO-like 区域）**：需要服务端托管多个实例、动态扩缩容、负载均衡。Listen Server 的部署模型根本不支持这些。

## 工程边界

一些在实际工程中需要明确的边界条件：

**不是所有游戏都需要 DS**：棋牌类、回合制策略、单机附带联机——用 Listen Server 或纯服务端逻辑（无 DS）完全足够，引入 DS 是额外的运维成本。

**DS 不等于消灭了客户端逻辑**：客户端的预测、插值、表现层逻辑反而在 DS 架构下变得更重要，因为网络延迟的存在要求客户端能"看起来流畅"，而不是傻等服务器返回。

**DS 的冷启动时间是运维指标**：游戏房间从创建到可接受连接，这段时间直接影响匹配等待体验。DS 需要尽量减少启动时的资产加载时间。

## 最短结论

Dedicated Server 的本质是**把服务器职责从玩家进程里剥离出来**，代价是增加了部署复杂度，收益是公平性、可靠性和安全性。

代码共享的边界很清晰：**游戏规则和数学计算可以共享，所有"让人看见/听见/摸到"的代码不能在 DS 上运行。**

服务器权威的正确理解是**服务器是最终裁判，不是唯一执行者**——客户端需要持续预测，服务器负责校正。
