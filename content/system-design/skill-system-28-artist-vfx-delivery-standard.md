> 美术说"效果做好了"，程序说"时序不对"。根因不是谁做错了，而是两边从来没有对齐过"什么叫做好了"。

技能表现在大多数项目里都是美术和程序之间最高频的协作点之一。

一个技能的完整表现至少涉及：

- 角色动画（起手、挥出、收招）
- VFX 特效（蓄力光效、投射物、爆炸、命中火花）
- 音效（施法音、飞行音、命中音）
- 镜头反馈（震动、缩放）
- UI 反馈（飘字、指示器）

这五层东西分属不同的人或团队制作，但最终要在运行时的同一条时间线上对齐。

如果没有一套统一的交付标准和验收规范，团队迟早会陷入一个循环：

```text
美术：效果做好了，按需求文档做的
程序：挂上去时序不对
美术：你给的时间点不对
程序：我给的是逻辑时间点，你的特效太长了
策划：反正现在看起来不对，你们再调调
```

这个循环的根因不是谁做错了，而是"美术的时间参考系"和"程序的时间参考系"从来没有被显式对齐过。

这一篇要回答 5 个问题：

1. 美术视角的技能表现制作全流程到底有几段。
2. 四条时间线怎样对齐——输入时间线、技能逻辑时间线、动画时间线、VFX 时间线。
3. 挂接点的命名规范应该长什么样。
4. 美术自检清单应该包含哪些项。
5. "美术说效果对了但程序说时序不对"的根因到底是什么。

## 美术视角的全流程

从美术视角看，一个技能表现的制作至少有五段。

### 第一段：概念

收到策划的技能设计文档，理解技能的核心表现诉求。

例如："这是一个火球术，蓄力 0.3 秒，发射一个带拖尾的火焰投射物，命中后爆炸。"

### 第二段：制作

分头制作各类资源：

- 动画师做施法动作
- 特效师做火球 VFX 和爆炸 VFX
- 音效设计做施法音和爆炸音

这一步的关键问题是：三个人各自制作时，参考的时间线是不是同一条。

### 第三段：挂接

把制作好的资源挂到角色或技能系统上：

- 动画挂到 Animator Controller
- VFX 挂到角色骨骼节点或技能发射点
- 音效绑定到 AnimEvent 或技能事件

这一步是最容易出问题的环节，因为"挂到哪个点"和"什么时候触发"的信息往往靠口头沟通。

### 第四段：调时序

在编辑器或运行时里调整各类资源的触发时机，让视觉效果和游戏手感匹配。

这一步经常需要反复迭代，因为：

- 动画的打击帧和逻辑的生效帧不在同一时刻
- VFX 的起播点和投射物的发射点不在同一时刻
- 音效的峰值和视觉冲击的峰值不在同一时刻

### 第五段：验收

在完整的游戏环境里验证最终效果。

验收的标准是什么？如果没有定义，就只能靠"策划觉得可以了"这种主观判断。

## 四条时间线的对齐约定

`09 篇`已经提出了四条时间线的概念。这一篇从美术视角把对齐约定具体化。

### 时间线 1：输入时间线

玩家按下按钮的时刻。这是整条链的起点。

美术需要知道的是：从按下按钮到技能逻辑开始，有没有延迟。

### 时间线 2：技能逻辑时间线

技能系统内部的阶段划分：

```text
[Windup 0.0-0.3s] → [Active 0.3-0.5s] → [Recovery 0.5-0.8s]
```

这条时间线上有几个关键锚点：

| 锚点 | 含义 | 美术关心什么 |
|------|------|-------------|
| CastStart | 施法开始 | 起手动画从这里开始播 |
| WindupEnd / ActiveStart | 前摇结束，生效开始 | 投射物从这里发射，近战从这里开始判定 |
| HitConfirm | 命中确认 | 受击特效和受击动画从这里触发 |
| RecoveryStart | 收招开始 | 收招动画从这里开始 |
| CastEnd | 施法结束 | 所有残留表现应该在这之前或这之后收尾 |

### 时间线 3：动画时间线

动画资源自己的时间轴：

```text
动画帧：[0f 起手] → [12f 抬手] → [18f 挥出] → [24f 收招] → [30f 结束]
```

动画时间线上需要标记的关键帧：

| AnimEvent 名称 | 位置 | 含义 |
|---------------|------|------|
| OnCastStart | 第 0 帧 | 施法起手（和逻辑的 CastStart 对齐） |
| OnSkillFire | 第 18 帧 | 发射 / 近战命中窗口开启（和逻辑的 ActiveStart 对齐） |
| OnSkillHit | 根据投射物飞行时间 | 命中确认（近战时和 OnSkillFire 接近） |
| OnSkillEnd | 第 30 帧 | 施法结束（和逻辑的 CastEnd 对齐） |

### 时间线 4：VFX 时间线

VFX 自己的生命周期：

```text
[蓄力光效 0.0-0.3s] → [投射物 0.3-0.6s] → [爆炸 0.6-1.2s] → [残留烟雾 1.2-2.0s]
```

VFX 时间线的关键约束：

- 蓄力光效必须在 CastStart 时开始、ActiveStart 时结束或消退
- 投射物 VFX 必须在 OnSkillFire 时生成
- 爆炸 VFX 必须在 HitConfirm 时触发
- 残留烟雾可以晚于技能逻辑结束，但不能无限期存在

### 对齐规则

四条时间线的对齐不是"所有东西同一帧发生"，而是明确每个锚点的主从关系：

```text
CastStart     → 动画从这里开始       → 蓄力 VFX 从这里播
ActiveStart   → 动画到打击帧         → 投射物 VFX 从这里生成
HitConfirm    → 受击动画从这里播     → 爆炸 VFX 从这里触发 → 命中音效从这里播
CastEnd       → 收招动画到这里结束   → 残留 VFX 开始淡出
```

`技能逻辑时间线是主时间线。动画时间线、VFX 时间线、音效时间线都对齐到逻辑锚点，而不是互相对齐。`

## 挂接点命名规范

挂接点命名不统一是美术和程序之间最高频的低级摩擦。

### AnimEvent 命名标准

我建议所有 AnimEvent 使用下面的命名约定：

| 事件名 | 触发时机 | 谁订阅 |
|--------|---------|--------|
| `OnCastStart` | 施法动画开始 | 技能系统：确认动画已经开始播放 |
| `OnSkillFire` | 发射 / 命中窗口开启 | 技能系统：生成投射物或打开命中窗口 |
| `OnSkillHit` | 命中确认 | 表现系统：播放受击反馈 |
| `OnSkillEnd` | 施法动画结束 | 技能系统：确认可以进入下一个状态 |
| `OnCancelWindow` | 可取消窗口开启 | 输入系统：开始接受取消输入 |
| `OnVfxSpawn_{Name}` | 特定 VFX 触发点 | 表现系统：实例化特定 VFX |
| `OnSfxPlay_{Name}` | 特定音效触发点 | 音效系统：播放特定音效 |

命名规则：

- 统一用 `On` 前缀，表示这是一个事件通知
- 技能流程相关用 `Skill` 或 `Cast` 前缀
- 表现相关用 `Vfx` 或 `Sfx` 前缀
- 可参数化的用下划线加名称后缀

### VFX Prefab 命名和结构

VFX Prefab 的命名建议：

```text
vfx_{技能名}_{阶段}.prefab

示例：
vfx_fireball_charge.prefab      -- 蓄力阶段
vfx_fireball_projectile.prefab  -- 投射物
vfx_fireball_impact.prefab      -- 命中爆炸
vfx_fireball_residue.prefab     -- 残留效果
```

VFX Prefab 的内部结构建议：

```text
vfx_fireball_projectile
├── Root (空节点，代表挂接原点)
│   ├── Core (核心粒子发射器)
│   ├── Trail (拖尾)
│   └── Glow (光照 / 辉光)
```

Root 节点的 Transform 应该是干净的（0,0,0 / 无旋转 / 1,1,1 缩放）。所有偏移在子节点上做。

### SFX 命名

```text
sfx_{技能名}_{阶段}.wav

示例：
sfx_fireball_cast.wav
sfx_fireball_fly.wav
sfx_fireball_impact.wav
```

## 美术自检清单

在美术提交技能表现资源之前，建议跑一遍下面的自检清单。

### 1. 时序对齐

- [ ] 起手动画的第 0 帧和 CastStart 对齐
- [ ] 打击帧 / 发射帧和逻辑的 ActiveStart 对齐
- [ ] AnimEvent 都已经标记在正确的帧上

### 2. VFX 生命周期

- [ ] 每个 VFX Prefab 都有明确的生命周期（不是无限循环）
- [ ] Looping VFX 有停止接口（Stop() 后能正确淡出）
- [ ] 投射物 VFX 的 Trail 在投射物销毁后不会突然消失（有残留淡出）

### 3. 中断安全

- [ ] 如果技能在前摇阶段被打断，蓄力 VFX 能正确停止
- [ ] 如果技能在 Active 阶段被打断，投射物 VFX 仍然能正常走完（或安全销毁）
- [ ] 如果角色在收招阶段死亡，残留 VFX 不会永远挂在场景里

### 4. 循环与重复释放

- [ ] 连续释放两次同一技能，VFX 不会叠加出异常视觉
- [ ] 持续技能（Channel）的循环 VFX 在持续期间表现稳定，不会逐渐偏移

### 5. 分档适配

- [ ] 高画质和低画质各有一套 VFX 方案（至少是粒子数量分档）
- [ ] 低画质下关闭的效果不会导致视觉信息缺失（关键信息用非粒子方式保底）

### 6. 性能预算

- [ ] 单个技能的 VFX 峰值粒子数在预算范围内
- [ ] Draw Call 数量在预算范围内
- [ ] 没有使用 Mesh 粒子发射器的超大面数网格

### 7. 挂接点正确性

- [ ] VFX 挂在正确的骨骼节点上（手部 / 武器端点 / 胸口 / 脚底）
- [ ] Root 节点 Transform 干净

### 8. 音效时序

- [ ] 施法音效和起手动画同步
- [ ] 命中音效和 HitConfirm 同步
- [ ] 没有音效在技能结束后还在播放

### 9. 命名规范

- [ ] AnimEvent 命名符合约定
- [ ] VFX Prefab 命名符合约定
- [ ] SFX 文件命名符合约定

### 10. 资源引用

- [ ] 所有引用的材质、贴图、动画都已经提交
- [ ] 没有引用临时路径或个人工作目录的资源

## "美术说效果对了但程序说时序不对"的根因

这个问题在几乎所有项目里都会出现。它有三个典型根因。

### 根因一：参考的时间线不是同一条

美术调时序时，参考的是动画时间线——"我在第 18 帧放了一个事件，视觉上看起来对"。

程序看的是技能逻辑时间线——"ActiveStart 在 0.3 秒，但你的 AnimEvent 在 0.36 秒"。

这 0.06 秒的差异在编辑器里很难看出来，但在网络同步场景下会导致：

- 本地表现比服务器确认早了两帧
- 受击反馈在伤害结算之前就播了
- 投射物发射和伤害判定不在同一帧

解决方案：明确技能逻辑时间线是主时间线。动画的关键帧必须对齐到逻辑锚点，而不是反过来。

### 根因二：动画时长变了但逻辑参数没跟着改

动画师优化了动作，把总时长从 1.0 秒改成了 0.8 秒。但逻辑侧的 Windup / Active / Recovery 参数还是按 1.0 秒配的。

结果是：动画已经播完了，逻辑还在 Recovery 阶段；或者动画还在挥刀，逻辑已经结束了。

解决方案：

- 动画时长变更必须通知程序和策划
- 更好的做法是在校验器里检测：AnimClip 时长和 SkillDef 的总时序是否匹配
- 如果差异超过一个阈值，自动报警

### 根因三：VFX 生命周期和技能生命周期脱节

VFX 的循环时长是 2.0 秒，但技能的 Active 窗口只有 0.5 秒。

或者 VFX 没有正确响应 Stop() 调用，导致技能结束了 VFX 还在播。

解决方案：

- VFX 的生命周期必须有明确的开始和结束接口
- 技能系统在状态切换时显式调用 VFX 的 Stop / Destroy
- 校验器检查：VFX 的自然时长是否和它所服务的技能阶段匹配

## 与 09 篇的衔接

`09 篇`从程序视角定义了"技能逻辑和表现系统应该怎样耦合、怎样解耦"。

这一篇从美术视角补完了另一半：

- `09 篇`说"动画事件可以参与桥接但不该独占技能执行权"——这篇定义了 AnimEvent 的命名标准和职责边界
- `09 篇`说"受击反馈不要直接绑在伤害函数里"——这篇定义了 HitConfirm 锚点和受击 VFX 的触发规则
- `09 篇`说"四条时间线要分开看"——这篇把四条时间线的对齐约定具体化到了可执行的表格

两篇合起来读，就能把"技能表现应该怎么做"这件事从架构原则落到交付规范。

## 这篇真正想留下来的结论

美术和程序在技能表现上的协作摩擦，大多数不是能力问题，而是规范缺失。

四件事能解决大部分问题：

1. 对齐到同一条主时间线——技能逻辑时间线是主线，动画和 VFX 对齐到逻辑锚点。
2. 统一挂接点命名——AnimEvent、VFX Prefab、SFX 文件都有约定俗成的命名规则。
3. 美术自检清单——提交前自己过一遍 10 项检查，把最常见的问题在源头拦住。
4. 校验器兜底——动画时长和逻辑时序的匹配、VFX 生命周期的完整性，由工具自动检查。

如果只记一句话：

`美术说"效果做好了"和程序说"时序不对"之间的鸿沟，不是靠来回调参数填上的，而是靠一套双方都能看懂的时间线约定和交付规范消除的。`
