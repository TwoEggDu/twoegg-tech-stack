> 技能系统要的是“逻辑正确”，表现系统要的是“手感成立”。这两者必须对齐，但绝不能互相吞掉。

很多项目里，技能和表现最早的关系都很直接：

```csharp
public void CastFireball()
{
    animator.Play("Cast_Fireball");
    SpawnEffect("Fireball_Muzzle");
    target.TakeDamage(120);
    audioSource.PlayOneShot(fireballSfx);
}
```

这在原型期很省事，因为所有东西都在一个地方。

但只要项目进入真实生产，很快就会遇到下面这些问题：

- 动画改了时长，伤害点突然不对了
- 同一个技能，玩家角色和 AI 怪物要用不同动作资源
- 网络延迟下，本地表现先播了，服务器结果还没回来
- 投射物实际晚命中，但伤害已经先结算了
- 近战挥刀视觉上还没碰到，逻辑上已经命中了

也就是说，技能系统和表现系统之间真正难的不是“要不要联动”，而是：

`逻辑时序和表现时序怎样对齐，但又不被彼此绑死。`

这一篇就专门处理这件事。

---

## 先拆开几条不同的时间线

一旦谈到技能和表现，很多讨论会自动混成“这个技能什么时候生效”。

但实际至少有四条不同时间线同时存在。

### 1. 输入时间线

玩家什么时候按下按钮、什么时候松开按钮、什么时候给出下一个输入。

### 2. 技能逻辑时间线

技能什么时候进入：

- Windup
- Active
- Recovery
- Cooldown

### 3. 动画时间线

动作什么时候：

- 起手
- 抬手
- 挥出
- 收招

### 4. 表现时间线

VFX / SFX / Camera Shake / Hit Flash / Damage Number 什么时候出现。

如果不先把这四条线拆开，系统后面一定会反复遇到一句模糊的话：

`“这个技能现在什么时候算生效？”`

而更准确的问题其实应该是：

- 技能逻辑什么时候进入 Active
- 动画什么时候进入打击姿态
- 投射物什么时候发射
- 受击特效什么时候播
- 伤害数字什么时候跳

---

## 最重要的不是“同步”，而是“对齐规则明确”

很多人会说技能和动画要同步。

这句话不算错，但还是太粗了。

因为真正需要的不是“所有东西同一帧发生”，而是：

`每个关键点由谁驱动、谁依赖谁、谁可以晚一点、谁绝不能错位。`

我建议至少把下面四个点显式定义出来。

### 1. 前摇点

技能已经开始，但效果还没真正送达。

这时常见动作包括：

- 播起手动画
- 锁定朝向
- 禁止某些输入
- 预热音效

### 2. 生效点

技能逻辑真正进入 Active 的关键时刻。

例如：

- 生成投射物
- 打开近战命中窗口
- 立即结算治疗
- 启动持续区域

### 3. 受击点

目标侧真正收到命中反馈的时刻。

例如：

- 扣血
- 受击特效
- 受击动画
- 数字飘字
- 镜头震动

### 4. 收尾点

技能动作和表现进入结束阶段。

例如：

- 收招动作
- 结束残留特效
- 允许下一个输入接入

这四个点不一定重合。

而“技能系统与表现解耦”的第一步，就是承认它们本来就不应该被强行写成同一个瞬间。

---

## 一个最常见的错误：把动画事件当成技能系统本体

很多项目会把技能生效完全交给动画事件：

```csharp
// 动画事件
void OnCastHitFrame()
{
    ApplyDamage();
}
```

这类做法前期很顺，但后面会有四个明显问题。

### 1. 技能逻辑失去主导权

动画变成了逻辑的唯一驱动器。

于是：

- 没动画就没法放技能
- 动画切掉了逻辑就丢
- AI / 网络回放 / 非角色实体很难复用

### 2. 表现资源变化会直接改逻辑结果

动画师如果改了动作长度、插值或事件点，技能结算时机就跟着变。

### 3. 逻辑调试很难做

当“为什么没命中”时，你要去查：

- 生命周期
- 动画状态机
- 事件点有没有被打到

### 4. 网络同步非常脆

本地动画先播和远端状态确认之间，很容易错位。

所以更稳的关系应该是：

`动画事件可以参与驱动表现关键点，也可以参与通知技能系统，但技能系统的主时序不应该完全寄生在动画资源上。`

---

## 更稳的关系：技能逻辑主时序 + 表现事件桥接

我更建议把关系做成：

```text
技能系统维护主生命周期
-> 在关键阶段抛出表现事件
-> 动画/VFX/SFX/相机系统订阅这些事件
-> 某些动画事件再反向通知技能系统“某个窗口已到”
```

也就是说，这里不是单向依赖，而是“有主有辅”的双向桥接。

### 技能系统主导什么

- 生命周期状态
- 是否真正进入 Active
- 命中是否成立
- 冷却和恢复

### 表现系统主导什么

- 动画播放
- 特效实例化
- 音效播放
- 镜头反馈
- UI 飘字与提示

### 动画事件适合负责什么

- 通知“投射物发射窗口已到”
- 通知“近战命中窗口开启/关闭”
- 通知“可取消窗口开启”

但不适合独占：

- 技能是否已经成功进入生命周期
- 技能是否最终命中
- 冷却和资源状态

## 可以把技能关键点显式建成事件

如果只想先把骨架立住，我建议至少有一套显式 Skill Event。

例如：

```csharp
public enum SkillEventType
{
    CastStarted,
    WindupStarted,
    HitWindowOpened,
    ProjectileSpawned,
    HitConfirmed,
    RecoveryStarted,
    CastEnded,
    CastInterrupted
}
```

然后系统在关键节点发事件：

```csharp
Publish(new SkillEvent
{
    type = SkillEventType.ProjectileSpawned,
    caster = cast.context.caster,
    skill = cast.context.skill,
    target = cast.context.primaryTarget
});
```

一旦这层存在，很多表现就可以从技能逻辑里拆出去：

- VFX 监听 `ProjectileSpawned`
- 音效监听 `CastStarted` / `HitConfirmed`
- UI 监听 `CooldownStarted`
- 相机系统监听 `HitConfirmed`

## 近战和投射物，在表现对齐上要分开看

### 近战技能

近战最重要的是：

- 动作姿态和命中窗口对齐
- 判定开启关闭准确
- 命中反馈和刀感一致

这里动画参与度非常高，但也更容易绑死。

更稳的做法通常是：

- 技能系统控制命中窗口
- 动画事件通知窗口开启时机
- 命中仍然由统一命中系统结算

### 投射物技能

投射物最重要的是：

- 发射点对齐
- 飞行过程可信
- 命中反馈发生在真实碰撞或命中确认时

这里最大禁忌是：

`视觉上投射物还没碰到，逻辑上伤害已经先结算。`

所以投射物类技能更适合把：

- 发射表现
- 飞行表现
- 命中表现

严格分成三个阶段，而不是在起手时一次性播完。

## 受击反馈最好不要直接绑在伤害函数里

很多系统会写成：

```csharp
void ApplyDamage(Entity target, int amount)
{
    target.HP -= amount;
    target.PlayHitAnimation();
    SpawnHitVfx(target.position);
    ShowDamageNumber(amount);
}
```

这类写法的问题是：

- 所有命中反馈被写死在伤害逻辑里
- 以后要区分格挡、护盾吸收、无敌、假命中、预警命中就会很难拆

更稳的方式是：

- 伤害系统只产出命中/伤害结果
- 表现系统根据结果决定播什么反馈

例如：

```text
HitResult:
  - target = Goblin_01
  - outcome = Blocked
  - damageApplied = 0
  - shieldAbsorbed = 120
```

这时候表现层就能更自然地决定：

- 播格挡火花
- 不播掉血数字
- 播护盾破裂音效

## Timeline / Event 更适合做“桥”，不适合做“本体”

无论你用的是：

- Unity 动画事件
- Timeline
- 自定义事件轨
- 甚至状态机行为回调

这些工具最适合做的，其实都是“桥接”。

它们适合：

- 标记发射点
- 标记命中窗口
- 标记可取消窗口
- 标记镜头和音效时点

它们不适合：

- 代替技能生命周期
- 代替命中结算
- 代替资源和冷却逻辑

一句话说就是：

`Timeline / Event 应该告诉系统“现在到了哪个表现点”，但不该自己变成技能系统。`

---

## 一个更稳的落地方式

如果放到 Unity 常见工程里，我更推荐下面这套关系：

### 技能层

- 管生命周期
- 管命中是否合法
- 管什么时候进入 Active / Recovery

### 表现协调层

- 订阅 Skill Event
- 负责动画、VFX、SFX、Shake、UI 的分发

### 动画桥接层

- 从 Animator / Timeline 收到关键点事件
- 回调技能系统：
  - 打开发射窗口
  - 打开命中窗口
  - 打开可取消窗口

### 命中反馈层

- 根据 HitResult 决定播什么表现

这几层一分开，项目后面会稳很多，因为：

- 换动画资源不必重写技能逻辑
- AI 和玩家可以复用同一套技能框架
- 网络同步时也更容易做本地预测和远端修正

---

## 这一篇真正想留下来的结论

技能逻辑和表现系统不应该彼此独立到毫无关系，但也绝不能互相吞掉。

更稳的关系是：

- 技能系统维护主生命周期和命中结果
- 表现系统订阅关键事件做动画、特效、音效和反馈
- 动画事件只负责桥接关键窗口，不负责独占技能执行权

所以这一篇最短的结论可以记成：

`动画与表现要和技能逻辑对齐，但不能成为技能系统本体。技能系统管“是否真正发生”，表现系统管“如何让玩家感到发生了”。`
