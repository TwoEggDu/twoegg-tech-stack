P01 已经把 `Physics World` 的地图立住了，P02 解释了物理数据怎么拆，P03 则把“主动查询物理世界”这条路讲清了。到 P04，新的问题开始出现：**当物理世界主动告诉你“谁碰到了谁”时，为什么不能直接在事件回调里改 Entity，为什么一定要多一层写回流程**。

这不是一个“API 用法”问题，而是一个边界问题。事件发生在物理执行链里，ECS 世界里的实体却受调度顺序、结构变更和同步点约束。只要把这两层混在一起，代码很快就会变成一边收事件、一边改结构、一边又被下一步系统覆盖。

---

## 为什么事件不能直接改 ECS 结构

CollisionEvents 和 TriggerEvents 看起来很像“有事发生了，立刻处理掉”。在普通对象模型里，这种直觉经常成立：回调一来，直接改对象字段，甚至直接销毁对象。DOTS 里不能这么干，因为事件触发的那一刻，往往正处在物理主链内部。

这里有三个原因。

第一，物理事件通常发生在固定步长的执行中间，不是任意时刻。你如果在这里直接改结构，就等于在一个还没结束的仿真批次里打断它。

第二，事件回调拿到的是“发生了什么”，不是“你现在可以随便重排世界”。它是结果，不是写权限。

第三，结构变更会触发 Archetype 迁移、Chunk 重排或同步点延后。把这些动作塞进物理回调里，最后很难判断是物理慢了，还是写回把主链打断了。

所以 P04 最重要的结论是：**事件可以采集结果，但不能直接承担最终写回职责**。

## 正确的写回链条长什么样

最稳的做法，是把流程切成四段。

```text
Physics World 产生事件
-> 事件系统只收集事实
-> 写入临时缓冲 / 命令缓冲
-> 在安全的 ECS 同步点统一提交
-> 后续系统读取已提交结果
```

这条链里最关键的是“临时缓冲”和“统一提交”。前者让事件先落地，后者让结构变更回到正确时机。

一个最小化的伪代码可以长这样：

```csharp
public struct HitEvent
{
    public Entity A;
    public Entity B;
}

public partial struct CollectHitEventsSystem : ISystem
{
    public void OnUpdate(ref SystemState state)
    {
        // 这里只做两件事：
        // 1. 收集 CollisionEvents / TriggerEvents 事实
        // 2. 记录到临时容器或 ECB
        //
        // 不在这里直接 AddComponent / RemoveComponent / DestroyEntity。
    }
}

public partial struct ApplyHitResultsSystem : ISystem
{
    public void OnUpdate(ref SystemState state)
    {
        // 在安全同步点统一提交：
        // - 标记命中状态
        // - 写入伤害请求
        // - 延迟结构变更
    }
}
```

这里的意图不是让你照抄框架，而是明确职责分层。采集阶段只负责“发生了什么”，提交阶段才负责“要把什么写回 ECS”。

## 哪些东西可以写回，哪些东西不该直接写

事件回流时，最容易写乱的是“把所有后续动作都压成一次即时修改”。

可以安全写回的，通常是这几类：

- 纯数据标记，例如命中标志、冷却触发、待处理请求
- 需要统一提交的状态请求，例如伤害、击退、失活
- 不改变当前物理主链布局的结果缓存

不该直接写回的，通常是这些：

- 立即销毁大量实体
- 在物理回调里改动同一批正在被物理步进读取的结构
- 把表现层逻辑和仿真层逻辑混成一个回调

这里的原则很简单：**事件回调只负责收事实，ECS 系统负责做决定**。如果你在事件回调里就开始做最终决定，后面很容易绕不过同步点。

## 为什么 ECB 在这里还是核心工具

事件回写和 `EntityCommandBuffer` 的关系非常直接。ECB 的作用不是“让代码更像 DOTS”，而是把结构性动作放到后面统一提交。

当事件只是记录“谁撞到了谁”时，ECB 可以先把它们排起来；当后续系统确认了命中结果，再一次性执行销毁、加组件、切状态这些动作。这样做有两个好处。

第一，物理主链不会被中途打断。第二，命中逻辑、状态逻辑、结构变更逻辑终于能分开看。

如果你把“收事件”和“改世界”绑在一个回调里，后面所有问题都会变得模糊：到底是碰撞判断错了，还是写回时机错了，还是结构变更太频繁了，最后根本分不清。

## 常见踩坑

第一个坑，是把事件当成同步消息直接消费，回调里立刻改实体。这样看似省一步，实际最容易踩同步点。

第二个坑，是把 TriggerEvents 当成“无代价碰一下就算数”。Trigger 不是免费逻辑，它同样会占用事件链和后续写回预算。

第三个坑，是把命中结果和表现层反馈混在一起。比如在物理回调里直接播动画、闪特效、改 UI，最后会让仿真层和表示层一起抖。

第四个坑，是认为“只要事件已经产生，就可以随便改结构”。不行。结构变更仍然要尊重 ECS 的提交边界。

## 小结

CollisionEvents 和 TriggerEvents 的价值，不是让你在回调里立即改世界，而是给你一条安全的事实回流路径。

最稳的写法，是先采集事件，再缓冲，再在安全同步点统一提交；只要把“收事实”和“改结构”分开，物理事件链就不会和 ECS 主链打架。
