## 问题空间：双向通信问题和它的解法

游戏服务端有一个 HTTP 天然解决不了的问题：**服务端需要主动推送数据给客户端**。

在 HTTP 的请求-响应模型里，只有客户端能发起请求。服务端没有办法主动说"玩家 B 刚刚移动到了这里"——除非客户端去轮询。轮询的代价大家都知道：要么频率太低错过更新，要么频率太高浪费资源，而且每次都要建立新连接（HTTP/1.1 的情况下）。

WebSocket 的出现解决了这个问题。但它解决的方式，从根本上决定了它的能力边界。

---

## 抽象模型：WebSocket 是什么

### HTTP Upgrade：一次特殊的握手

WebSocket 不是一个独立的协议，它是在 HTTP 连接上通过 `Upgrade` 机制建立的全双工 TCP 连接。

```
// 客户端发起 WebSocket 握手
GET /game-ws HTTP/1.1
Host: game.example.com
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==
Sec-WebSocket-Version: 13

// 服务端同意升级
HTTP/1.1 101 Switching Protocols
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Accept: s3pPLMBiTxaQ9kYGzzhZRbK+xOo=
```

握手完成后，这个 TCP 连接就变成了一个双向的消息通道。双方都可以随时发送数据，不再需要请求-响应的问答模式。

### WebSocket 的核心特性

- **全双工**：服务端可以随时主动推送，不需要等客户端请求
- **持久连接**：连接建立后保持，不需要每次交互重建
- **帧格式**：有轻量的帧头（2-10 字节），支持文本帧和二进制帧
- **运行在 TCP 之上**：这是它的能力上限，也是它的根本局限

**关键认知**：WebSocket 不是一个新的传输协议，它是 TCP 的应用层封装。TCP 的所有特性（三次握手、有序交付、队头阻塞、拥塞控制）WebSocket 都完整继承。

## WebSocket 适合的游戏场景

### H5 游戏和微信小游戏

浏览器和小游戏运行环境是一个沙盒环境。浏览器出于安全策略，不允许 Web 页面直接使用原生 UDP socket（WebRTC DataChannel 是唯一例外，后面会讲）。

在这个约束下，WebSocket 是**在浏览器/小游戏环境内能得到的最接近长连接的方案**：
- 可以做服务端主动推送
- 支持二进制帧（配合 Protobuf 使用）
- 连接复用，不需要每次轮询

对于受限于运行环境的 H5 游戏，WebSocket 不是"最好的选择"，而是"可用选择里最合适的"。

### 低频状态同步：回合制和卡牌游戏

回合制游戏的状态更新频率极低——每次只在玩家操作时更新。这类游戏的延迟容忍度高（200-500ms 可接受），消息频率低（每秒不超过 5 次）。

在这种负载下，TCP 的队头阻塞完全不是问题——因为包发得这么稀，同时有多个包在传输的概率很低。WebSocket 的双向推送能力完全满足需求，开发体验也远好于自建 UDP 基础设施。

典型适合 WebSocket 的游戏类型：
- 棋牌游戏（象棋、扑克、麻将）
- 回合制策略游戏
- 放置游戏、Idle 游戏
- 轻量社交游戏（农场、装扮）
- 文字 RPG / 文字冒险游戏

这些游戏里，WebSocket 是工程上的优选——快速开发、工具链成熟、运维简单。

### 实时通知和推送

游戏内的聊天系统、好友上线通知、系统公告、活动倒计时推送，这些都是低频但需要实时性的消息。WebSocket 是标准解法，没有必要为这类需求引入 UDP。

## WebSocket 不适合的场景

### 高频小包：延迟是结构性问题

FPS 游戏需要每帧（16ms）发送一次位置更新和输入状态，每个包大约 50-200 字节。在一个 64 人服务器上，每帧服务端要处理和广播约 6400 条消息。

在 WebSocket（TCP）上，这个场景的根本问题是：

**队头阻塞依然存在**。如果第 N 帧的某个包在传输中丢失，第 N+1 帧的更新会被内核缓冲区暂扣，等到重传完成才交给应用层。这段等待时间通常是 1-3 个 RTT（20-150ms），足以让游戏画面出现明显卡顿。

Nagle 算法也是隐患。虽然可以通过 `TCP_NODELAY` 关闭，但标准 WebSocket 服务器框架（如 Node.js ws 库）默认不一定开启此选项，需要手动配置。

### 延迟极敏感：物理上不可绕过

FPS 和格斗游戏的延迟感知阈值约 50ms。超过这个阈值，玩家会明显感觉到"输入响应迟钝"。

TCP 的重传机制在有丢包时会引入至少 1 RTT 的额外延迟，在高峰期国内跨运营商网络中 RTT 50-100ms 很正常，这意味着一次丢包就会直接触发 50-100ms 的卡顿，而且这不是调参能解决的——它是 TCP 协议的运作机制。

UDP 允许应用层"放弃旧包、用最新包"的语义，在同样的网络条件下，有丢包时的表现比 TCP 好得多。

### 为什么不能绕过这个限制

有些开发者尝试在 WebSocket 之上实现"跳过旧包"的逻辑：收到消息后，如果序列号比当前最新的低，就丢弃它。

这个逻辑本身是正确的，但解决不了根本问题。**问题不在应用层看到的消息顺序，而在内核缓冲区的等待**。TCP 在把数据交给应用层之前，就已经完成了有序重组。你的应用层看到的是"晚来但按序"的数据流，等待的延迟早就发生了，数据已经"迟到"了。

你不能告诉 TCP："这个序列号的包丢了，不用重传，直接给我后面的包"——TCP 没有这个接口。这个接口只有 UDP 能给你。

## WebRTC DataChannel：浏览器端的例外

WebRTC 是专门为浏览器设计的实时通信标准，它的 DataChannel 提供了在浏览器中使用 UDP 语义的能力。

```javascript
// WebRTC DataChannel 可以配置为不可靠模式
const dataChannel = peerConnection.createDataChannel("game", {
    ordered: false,     // 不保证顺序
    maxRetransmits: 0,  // 不重传，丢了就丢了
});
```

这是在浏览器环境里唯一能够真正实现 UDP 语义的方案。

但 WebRTC 有一个重要的使用场景定位：它的设计目标是 **P2P（点对点）通信**，主要用于视频/音频通话和 P2P 文件传输。用于游戏客户端到服务端通信时，有几个非平凡的工程问题：

**ICE/STUN/TURN 的复杂性**：WebRTC 的连接建立过程需要 ICE（Interactive Connectivity Establishment）协商，涉及 STUN 服务器用于 NAT 穿透、TURN 服务器用于中继。这比直接连 UDP 服务端复杂得多。

**连接建立延迟高**：ICE 协商通常需要几百毫秒，不适合需要快速重连的游戏场景。

**服务端实现复杂**：服务端需要实现 WebRTC 信令服务器和 DTLS/SRTP 处理，工程量远超普通 UDP 服务端。

**结论**：WebRTC DataChannel 适合 P2P 游戏（比如 P2P 对战避开服务器延迟）或有特殊浏览器 UDP 需求的场景，不适合作为通用的游戏服务端协议。

## 工程边界：游戏类型和协议选择的对应关系

综合传输层、可靠性需求、运行环境三个维度，给出一个完整的对应关系：

| 游戏类型 | 运行环境 | 推荐方案 | 理由 |
|---------|---------|---------|------|
| 棋牌/回合制 | 任意 | WebSocket + Protobuf | 低频同步，双向推送足够 |
| H5 休闲游戏 | 浏览器 | WebSocket | 浏览器约束，WebSocket 是上限 |
| 微信小游戏 | 小游戏环境 | WebSocket | 同浏览器限制 |
| MMORPG | 原生 App | TCP + 自定义协议 或 KCP | 频率中等，可选择 |
| MOBA | 原生 App | KCP over UDP | 高频，延迟敏感 |
| FPS/TPS | 原生 App | 原生 UDP + KCP/ENet | 极高频，延迟极敏感 |
| 格斗/赛车 | 原生 App | 原生 UDP + 帧同步 | 帧同步逻辑，最低延迟 |
| H5 竞技游戏 | 浏览器 | WebRTC DataChannel | 唯一浏览器 UDP 选项 |

### 关键判断

- 是否在浏览器/小游戏沙盒环境？是 → WebSocket（或 WebRTC DataChannel）
- 是否回合制/低频？是 → WebSocket 够用
- 是否高频（> 20Hz）且延迟极敏感？是 → 必须 UDP
- 没有明确性能问题但想快速开发？WebSocket 先跑起来，数据说明问题再优化

## 最短结论

WebSocket 解决了浏览器/小游戏环境的双向通信问题，对于低频状态同步游戏是完整可靠的解法。它不是妥协，而是在约束条件下的正确选择。

但 WebSocket 基于 TCP，它无法绕过 TCP 的队头阻塞。竞技类游戏（FPS/MOBA/格斗）在有丢包的网络条件下，TCP 的重传等待会直接表现为玩家感知到的卡顿，这不是应用层代码能修复的问题。

原生 UDP 的不可替代性不在于它"更快"，而在于它提供了一个能力：当旧数据不再有价值时，可以真正放弃它。这个能力在高频实时游戏中是必须的。
