## 问题从一次购买说起

玩家点击"购买"，后端需要完成三件事：

1. **货币服务**：扣除玩家 100 金币
2. **道具服务**：给玩家背包增加一把剑
3. **日志服务**：记录这笔交易流水

三步全成功，皆大欢喜。但真实系统里，网络会超时，服务会崩溃，数据库会死锁。如果第一步扣了钱，第二步添加道具时服务宕机，你得到的是一个丢了钱又没拿到剑的愤怒玩家。

这就是**分布式事务问题**的核心：如何在多个独立服务之间保证"要么全做，要么全不做"？

---

## 分布式事务的根本难题

### 单机事务为什么简单？

在单库场景下，ACID 事务由数据库引擎保障。`BEGIN → 操作 → COMMIT`，一旦提交失败，引擎自动回滚，调用方几乎感知不到复杂性。这套机制之所以能工作，是因为**数据在同一个进程的控制范围内**。

### 跨服务后，控制权碎了

微服务拆分之后，货币数据库、道具数据库、日志数据库各自独立。你不可能用一个 `BEGIN` 把三个数据库锁住。

**两阶段提交（2PC）** 是学术界给出的经典方案：协调者先向所有参与者发送"准备"请求，收到全员 OK 后再发送"提交"请求。理论上完美，工程上代价极高：

- **阻塞问题**：准备阶段所有资源被锁定，等待协调者决策。协调者宕机，所有参与者无限等待。
- **性能代价**：每次事务至少两轮网络往返，高并发时吞吐量断崖式下跌。
- **实现复杂**：协调者需要持久化事务状态，恢复逻辑极为复杂。

游戏后端动辄每秒数千次购买操作，2PC 的代价无法接受。这迫使我们接受一个现实：**在分布式系统里，强一致性是昂贵的，最终一致性才是工程常态。**

## Saga 模式：用补偿替代回滚

Saga 模式的核心思想很直接：**把一个跨服务的长事务拆成一系列本地事务，每个本地事务成功后发布事件或消息触发下一步；如果某步失败，执行反向补偿操作撤销已完成的步骤。**

这不是"回滚"，而是"补偿"——补偿是一个正向的业务操作，将系统状态恢复到事务开始前的逻辑等价状态。

### 两种实现方式

#### 编排式（Choreography）

每个服务监听事件并自主决定下一步行动：

```
货币服务扣款成功 → 发布 MoneyDeducted 事件
道具服务监听该事件 → 添加道具 → 发布 ItemAdded 事件
日志服务监听该事件 → 记录流水 → 发布 LogRecorded 事件
```

失败时，各服务监听失败事件并执行补偿：

```
道具服务添加失败 → 发布 ItemAddFailed 事件
货币服务监听该事件 → 退款 → 发布 MoneyRefunded 事件
```

**优点**：服务完全解耦，没有中央协调者，天然适合事件驱动架构。  
**缺点**：业务流程分散在各个服务里，整体流程难以追踪，调试困难，很快变成"服务之间的意大利面"。

#### 协调式（Orchestration）

引入一个 Saga 协调者（Orchestrator），显式定义完整的业务流程：

```
Saga Orchestrator:
  step1: 调用货币服务.扣款()
    on_failure: 无需补偿（尚未改变其他状态）
  step2: 调用道具服务.添加道具()
    on_failure: 调用货币服务.退款()
  step3: 调用日志服务.记录()
    on_failure: 调用道具服务.删除道具(), 调用货币服务.退款()
```

**优点**：业务流程集中可见，可以很方便地追踪事务状态，易于审计和调试。  
**缺点**：协调者本身变成一个有状态的关键服务，需要做好持久化和高可用。

**工程建议**：游戏后端的购买、任务完成等复杂业务流程，优先用**协调式**。流程可见性在线上排查问题时价值极高。

## 游戏购买流程的 Saga 设计

以购买道具为例，设计一个完整的 Saga：

### 正向流程

```
1. 创建订单（订单服务）→ 状态: PENDING
2. 扣除货币（货币服务）→ 状态: MONEY_DEDUCTED
3. 增加道具（道具服务）→ 状态: ITEM_ADDED
4. 记录日志（日志服务）→ 状态: COMPLETED
```

### 补偿流程

每一步都需要定义对应的补偿操作：

| 步骤 | 正向操作 | 补偿操作 |
|------|----------|----------|
| 创建订单 | 插入 PENDING 订单 | 将订单标记为 CANCELLED |
| 扣除货币 | 扣减余额 | 退还等额货币（带补偿流水记录） |
| 增加道具 | 插入背包记录 | 删除对应背包记录 |
| 记录日志 | 插入流水 | 补偿日志几乎无意义，通常跳过 |

**关键设计点**：日志服务的失败是否需要补偿前面的步骤？这取决于业务定义。通常游戏里"道具已给到玩家但日志没记录"比"日志记了但道具没给"危害更小。可以将日志记录设计为**尽力而为（best-effort）**，失败后通过异步重试修复，不触发全链路补偿。

## 最终一致性：游戏场景的可接受性分析

最终一致性意味着：**系统在没有新的更新操作的情况下，最终会达到一致状态，但在达到一致之前，不同节点/服务的数据可能短暂不一致。**

### 哪些游戏数据可以接受最终一致性？

**可以接受**：
- **排行榜**：排名延迟几秒甚至几十秒更新，玩家感知不强。
- **成就系统**：达成条件和徽章显示之间有短暂延迟，完全可接受。
- **好友在线状态**：心跳间隔本身就引入了几十秒的不确定性。
- **活动统计数据**：服务端统计面板延迟分钟级别是正常的。

**不可接受或需谨慎**：
- **货币余额**：玩家对金币数量极度敏感，"扣了钱但界面没变"会引发大量投诉。但注意，这里要求的其实是**读自己写（Read-Your-Writes）一致性**，而非全局强一致性。
- **道具实际持有**：玩家购买后必须能立即使用道具，不能"等一会儿再试"。
- **PVP 对局结算**：涉及积分、段位的变更，错误会直接影响竞技公平性。

这个分类告诉我们：不是"所有游戏数据都需要强一致性"，而是要**按数据特性分类设计一致性策略**。

## 幂等设计：分布式系统的安全网

Saga 的补偿机制要能工作，每个服务的接口必须是**幂等的**——同一个请求执行多次，结果与执行一次相同。

为什么幂等如此重要？网络超时后，调用方往往会重试。如果不加保护，可能出现：
- 扣了两次货币
- 道具发放了两份
- 日志重复记录

### 请求 ID / 去重 Key 方案

最常见的幂等实现：**调用方生成全局唯一的请求 ID，服务端用该 ID 去重。**

```go
// 请求体
type PurchaseRequest struct {
    RequestID  string  // UUID，由客户端生成
    PlayerID   int64
    ItemID     int32
    Price      int64
}

// 服务端处理
func (s *ItemService) AddItem(req PurchaseRequest) error {
    // 先查去重表
    if s.deduplicator.Exists(req.RequestID) {
        return nil // 幂等返回成功
    }
    // 执行实际操作
    err := s.db.InsertItem(req.PlayerID, req.ItemID)
    if err != nil {
        return err
    }
    // 记录已处理
    s.deduplicator.Mark(req.RequestID, TTL: 24h)
    return nil
}
```

去重表可以用 Redis 实现，Key 为 RequestID，设置合理的 TTL（通常 24 小时足够覆盖所有重试窗口）。

**注意**：去重记录的写入和业务操作的执行，最好在同一个本地事务里完成，避免"操作成功但去重标记没写入"的竞态。

## 什么场景不适合 Saga

Saga 是对强一致性的妥协，它有明确的适用边界。以下场景**不适合**使用 Saga，应该考虑其他方案：

### 金融级强一致性场景

游戏内的充值、提现（如果涉及真实货币）通常需要对接支付渠道，这类操作对一致性要求极高。宁可用**单服务单库**，甚至单机数据库来保障 ACID，也不应拆分到多个服务用 Saga 维护。拆分带来的维护成本远大于收益。

### 对"部分完成"状态零容忍的场景

有些业务不允许出现中间状态——即使是短暂的。比如一个需要原子交换的 PVP 匹配（A 必须和 B 配对，不存在"A 匹配成功但 B 没收到"的状态）。这类场景应该在**单服务内**处理，或者用分布式锁 + 单库事务。

### 补偿逻辑无法定义的场景

有些操作在业务上不可逆：发出去的系统公告、触发的成就通知、已经播放的动画特效。这类操作本身就应该放到 Saga 的末尾，作为"收尾步骤"，而不是设计成可补偿的中间步骤。

---

## 工程边界总结

| 维度 | Saga 适合 | 不适合 Saga |
|------|-----------|------------|
| 一致性要求 | 最终一致性可接受 | 强一致性必须保证 |
| 事务步骤数 | 2-8 步 | 超过 10 步（复杂度爆炸） |
| 补偿可定义性 | 每步都能定义反向操作 | 存在不可逆操作 |
| 数据跨越边界 | 多服务多库 | 单服务单库 |
| 并发量 | 高并发场景 | 低并发且简单场景（2PC 反而简单） |

---

## 最短结论

分布式事务没有银弹。游戏后端的正确姿势是：**用 Saga 处理跨服务的业务流程，用幂等 + 去重保障补偿可重试，对货币等敏感数据保障读自己写一致性，对排行榜等数据接受最终一致性；遇到必须强一致的场景，勇敢地回到单服务单库，不要为了"微服务纯洁性"而引入不必要的复杂度。**

理解分布式一致性的本质，是从"能跑"到"可靠"的关键一跃。
