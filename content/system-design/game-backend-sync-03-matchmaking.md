## 问题空间：匹配系统要平衡三个互相对立的目标

玩家点击"开始匹配"，期待在合理时间内，进入一场公平的、满员的对局。这三个期待背后是三个工程目标——**公平性（Match Quality）、等待时间（Wait Time）、满员率（Fill Rate）**——而这三个目标天然对立：

- 追求极高公平性（只匹配评分完全相近的玩家）→ 等待时间变长，满员率下降
- 追求极低等待时间（随机匹配）→ 公平性崩溃，玩家流失
- 追求极高满员率（强行凑够人数）→ 可能把天梯顶端玩家和新手凑在一起

匹配系统的本质是**在这三个维度上做动态权衡**，而不是找到一个"最优解"。ELO 评分、动态扩圈、DS 分配都是服务于这个权衡的具体机制，理解它们的局限性和适用边界，比单纯学会实现更重要。

## 抽象模型：ELO 评分的核心逻辑

### 期望胜率公式

ELO 评分系统由国际象棋棋手 Arpad Elo 在 1960 年代提出，后被电子游戏广泛采用。其核心思想是用**期望胜率**来建模两个玩家之间的实力差距：

```
E_A = 1 / (1 + 10^((R_B - R_A) / 400))
```

其中：
- `R_A`、`R_B` 是两个玩家的当前评分
- `E_A` 是玩家 A 的期望胜率（0 到 1 之间）
- 分母中的 400 是缩放因子（控制分数差距对胜率预测的敏感度）

直觉上：两人评分相同时，`E_A = 0.5`（50% 胜率）；A 比 B 高 200 分时，`E_A ≈ 0.76`（76% 胜率）；差 400 分时，`E_A ≈ 0.91`。

### K 因子：分数更新幅度的控制器

对局结束后，按以下公式更新双方分数：

```
R_A_new = R_A + K * (S_A - E_A)
```

其中：
- `S_A` 是实际结果（赢 = 1，输 = 0，平 = 0.5）
- `K` 是更新系数，控制每局比赛对评分的影响幅度

K 因子是 ELO 系统中最重要的调节参数：

- **K 值大（如 32）**：分数变动快，新玩家能快速找到正确段位，但"分数噪声"大，一次偶然失误可能让你掉很多分
- **K 值小（如 10）**：分数稳定，但收敛慢，新玩家需要打很多局才能反映真实实力
- **动态 K 值**：新号玩家使用较大 K（快速定位），高分/高局数玩家使用较小 K（保护稳定分段）。这是现代游戏的标准做法。

### ELO 的局限性

ELO 是一个相对成熟但并不完善的模型，有几个显著问题：

1. **收敛慢（Cold Start Problem）**：新玩家从默认分（如 1000）出发，可能需要 30-50 局才能到达真实段位，这期间他会破坏别人的匹配体验。
2. **分数通胀/通缩**：如果系统中玩家总数变化（大量新号涌入），分数会出现系统性漂移。
3. **不反映实力不确定性**：ELO 只有一个数字，无法表达"我对这个评分的确定程度"——一个刚开始打的新号 1200 分，和一个打了 500 局的稳定 1200 分，在 ELO 里完全一样。

## 更精准的模型：Glicko-2 与 TrueSkill

### Glicko-2：引入评分不确定度

Glicko-2 是 ELO 的直接改进，在评分数字之外增加了两个额外维度：

- **RD（Rating Deviation，评分偏差）**：表示对当前评分的不确定程度。RD 越大，说明对这个玩家的实力越不确定（新号 RD 高，老号 RD 低）
- **σ（Rating Volatility，波动性）**：表示玩家实力本身的稳定性。一个时而超常发挥时而摸鱼的玩家，σ 值高

Glicko-2 的匹配策略可以优先匹配 RD 接近的玩家，让新玩家更快定位，老玩家保持稳定分段。大量竞技游戏（包括《League of Legends》早期版本）采用过 Glicko-2。

### TrueSkill：专为多人对局设计

微软 Research 开发的 TrueSkill（用于 Xbox Live）解决了 ELO 和 Glicko-2 的两个核心限制：

1. **多人对局（1v1 以外）**：ELO 天然是 1v1 模型，扩展到 5v5 时需要做团队合并（把队伍整体算作一个 ELO），精度较差。TrueSkill 原生支持多人对局。
2. **概率建模**：TrueSkill 用贝叶斯推断，每个玩家的实力是一个高斯分布 `N(μ, σ²)`，μ 是均值（估计实力），σ 是标准差（不确定性）。公开展示的分数通常是 `μ - 3σ`（保守估计，减少"被低估"的抱怨）。

TrueSkill 的工程代价是计算复杂度显著高于 ELO，在大量并发匹配时需要注意计算性能。微软开源了 TrueSkill 的变体（TrueSkill Through Time），也有第三方 Python/Go 实现可用。

## 具体实现：匹配队列的工程架构

### 优先级队列与匹配票（Match Ticket）

玩家点击"开始匹配"，后端创建一个**匹配票（Match Ticket）**：

```json
{
  "ticket_id": "uuid-xxx",
  "player_id": "player-abc",
  "created_at": 1712200000,
  "rating": 1850,
  "rating_uncertainty": 45,
  "preferred_region": ["ap-east"],
  "game_mode": "ranked_5v5"
}
```

匹配系统维护一个按时间排序的队列（优先级队列，等待时间越长优先级越高），周期性（例如每秒一次）从队列中取出一批 Ticket，尝试组建对局。

### 动态扩圈策略（Expanding Search）

最关键的工程机制是**等待越久，匹配范围越宽**。一个典型的动态扩圈策略：

| 等待时间 | 允许评分差值 | 允许区域扩展 |
|----------|-------------|-------------|
| 0-15s    | ±50         | 首选区域    |
| 15-30s   | ±100        | 首选 + 次选区域 |
| 30-60s   | ±200        | 所有区域    |
| 60s+     | ±400        | 所有区域 + 跨段位容忍 |

这个策略的参数调整是纯数据驱动的，需要根据游戏的实际匹配数据（等待时间分布、玩家评分分布）持续迭代。新游戏上线初期玩家稀疏，通常需要更激进的扩圈参数。

### 批量匹配算法

每次匹配周期，系统从队列中取出一批 Ticket，尝试找到符合条件的分组：

```
// 简化的 5v5 匹配流程（伪代码）
func tryMatch(tickets []Ticket) []Match {
    sort by wait_time desc  // 等待最久的优先
    
    for each candidate_captain in tickets:
        range = getSearchRange(candidate_captain.wait_time)
        group = findTickets(tickets, candidate_captain.rating ± range, count=9)
        
        if len(group) == 9:  // 找到 9 个对手组成 10 人对局
            balance = balanceTeams(group + [candidate_captain])
            if balance.quality >= threshold:
                emit Match(balance.team_a, balance.team_b)
                remove matched tickets from queue
```

团队平衡（`balanceTeams`）是一个 NP-hard 问题的近似，通常用贪心算法：按评分排序后交替分配，使两队平均评分差最小。

### 开源框架：Open Match

谷歌开源的 **Open Match** 是目前最成熟的匹配系统框架，运行在 Kubernetes 上，设计了 Director/MMF（Matchmaking Function）分离的架构：

- **Frontend**：接收玩家 Ticket，写入 Redis
- **Query Service**：按条件从 Redis 批量查询候选 Ticket
- **MMF（Matchmaking Function）**：开发者自定义的匹配逻辑，以 gRPC 服务形式插入
- **Director**：调用 MMF，拿到匹配结果后触发 DS 分配

## 具体实现：房间创建与 DS 分配

### Agones：Kubernetes 上的游戏服务器管理

匹配系统找到一组玩家后，需要为他们分配一个专用游戏服务器（Dedicated Server，DS）。现代云原生方案通常用 **Agones**（谷歌与 Ubisoft 联合开源）管理游戏服务器的生命周期。

完整流程：

```
1. 匹配系统产生匹配结果（10个玩家的分组）
2. Director 调用 Agones GameServerAllocation API
3. Agones 从就绪的 GameServer 池中分配一个空闲实例
4. GameServer 状态从 Ready → Allocated，收到玩家列表
5. Agones 返回 GameServer 的 IP:Port
6. Director 将 IP:Port 分发给所有匹配到的玩家
7. 玩家客户端直连 GameServer，开始游戏
```

Agones 的 GameServer 池维护策略：为保证玩家快速进入游戏，需要提前预热一批空闲 GameServer 实例（Fleet 自动扩缩容），避免玩家等待容器冷启动（通常 10-30 秒）。

### 区域就近分配

DS 分配时还需要考虑区域：匹配到的玩家可能来自不同地区，需要选择对所有玩家延迟总和最小的 DS 所在区域。常见做法：

1. 客户端在进入匹配前，先 Ping 测各个候选区域的延迟
2. 把延迟数据附在 Match Ticket 里
3. Director 拿到匹配结果后，选择玩家组的"延迟加权最优"区域分配 DS

## 反作弊在匹配层的卡口

匹配层是反作弊系统的第一道卡口，在玩家进入对局之前就应该隔离异常玩家：

### 同 IP 多开检测

同一 IP 地址短时间内出现多个活跃 Ticket，可能是"小号刷分"或"代练陪练"行为：

```
规则：同 IP 在 24 小时内创建超过 N 个不同账号的 Ticket → 触发人工审核 + 进入隔离队列
```

注意：宿舍、网吧等场景可能有合理的多人同 IP 情况，规则需要有例外处理（如同 IP 账号历史行为正常则豁免）。

### 异常玩家隔离队列（Penalty Pool）

已被检测到异常行为（封号记录、举报率异常高、外挂检测命中）的玩家，不直接封号，而是进入**隔离匹配队列**——只与同类可疑玩家匹配，延迟更长，体验更差，促使其自然离开，同时减少对正常玩家的影响。这是业界称为"影子匹配（Shadow Pool）"的策略。

### 段位保护与分组合法性验证

匹配结果产生后，在触发 DS 分配之前，应做一轮合法性校验：

- 评分差值是否在允许范围内（防止后端 Bug 产生极不平衡的匹配）
- 是否存在已被封号玩家（实时查询封号状态，防止封号处理延迟）
- 是否存在同账号玩家（防止同人小号被分到对立队伍）

## 匹配系统的监控指标

匹配系统运行效果的评估，依赖一套核心监控指标：

### 效率指标

| 指标 | 定义 | 健康阈值（参考） |
|------|------|-----------------|
| 平均等待时间（Average Wait Time） | 从创建 Ticket 到进入游戏的时间 | < 60s（竞技模式） |
| P95 等待时间 | 95% 的玩家的最长等待时间 | < 180s |
| 放弃率（Abandon Rate） | 主动取消匹配的 Ticket 比例 | < 5% |
| Ticket 积压数 | 当前队列中等待超过 120s 的 Ticket 数 | 趋势持续上升 = 警报 |

### 质量指标

| 指标 | 定义 | 监控方式 |
|------|------|----------|
| 匹配质量分布（Match Quality Score） | 每场对局的双队评分差分布 | 直方图，关注长尾 |
| 实际胜率 vs 预期胜率差值 | 验证评分系统准确性 | 每日回归分析 |
| 满员率（Fill Rate） | 实际开局人数 / 期望人数 | 人数不足开局 = 问题 |

### 异常检测

- **评分急剧变化**：同一账号短时间内评分变化超过阈值（可能被代打）
- **神秘断线**（Disconnect Spike）：某段时间内断线率异常升高，可能是 DS 分配到的区域有问题
- **匹配系统延迟**：Ticket 创建到分配 DS 的服务端处理时间（不含网络），应 < 1s

## 工程边界：匹配系统的隐性成本

匹配系统看起来不复杂，但工程实践中有几个经常被低估的成本：

### 玩家分布的长尾问题

大部分玩家集中在中等段位，极高和极低段位的玩家稀少。顶端 1% 的玩家可能需要等待数分钟才能找到对手，而你不得不为他们单独设计扩圈策略（甚至允许跨服匹配）。《守望先锋》OWL 职业玩家在公开服务器上遇到的长时间匹配就是这个问题的典型案例。

### 多人组队的匹配公平性

当玩家可以组队（Party）参与匹配时，整队的实力评估变得复杂——一个高手带低分号组队，整体评分中间值可能偏低，但实际战斗力远超显示评分。许多游戏（如《Valorant》）对组队的评分差有明确限制，或者对组队使用单独的匹配队列。

### 评分系统的博弈行为

玩家会主动对评分系统进行博弈（"放水"、"刷分"、"保段"等行为），这些行为会系统性地污染评分数据。检测这类行为需要在匹配系统之外另建行为分析系统，是一个独立的大课题。

## 最短结论

- **ELO 是起点，不是终点**。简单游戏用 ELO 够用，竞技性强的游戏应升级到 Glicko-2（带不确定度）或 TrueSkill（支持多人对局、贝叶斯推断）。
- **动态扩圈是等待时间的核心调节杆**。等待越久越放宽匹配范围，参数需要基于真实数据持续调整。
- **DS 分配的速度决定玩家进入游戏的速度**。Agones + 预热 GameServer 池是云原生场景的标准方案。
- **反作弊在匹配层就应该卡**。Shadow Pool 比封号更有效，同 IP 多开检测是最低成本的前置卡口。
- **监控三件套**：平均等待时间、P95 等待时间、匹配质量分布——这三个指标能反映匹配系统 80% 的健康状况。
