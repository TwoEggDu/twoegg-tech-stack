## 问题空间：服务端如何知道连接还活着

Web 后端程序员对"连接管理"的印象通常是：请求来了，处理完，连接关掉（或归还连接池）。状态短暂，生命周期清晰。

游戏服务端的情况完全不同。一个玩家登录后，它和服务器之间的 TCP/UDP 连接可能持续几个小时。在这段时间里，服务端需要持续知道这条连接是否还活着——因为连接对应着游戏状态：玩家的位置、血量、所在副本、好友列表。

如果一条连接"悄无声息地死掉"而服务端没有发现，就会发生以下任何一种情况：
- 服务端一直往一个实际上已经断开的连接发包，浪费资源
- 玩家的游戏状态卡在最后一帧，队伍里的人看到他"原地不动"
- 内存里的连接对象和玩家状态没有被清理，长期积累造成内存泄漏
- 下次玩家重连时，系统认为旧会话还在，产生状态冲突

理解连接管理，需要先理解一个容易被忽视的基础设施：NAT。

---

## 抽象模型：NAT 和静默断开

绝大多数玩家不是直接连接到公网的，他们在 NAT（Network Address Translation）设备后面——家用路由器、运营商的 4G/5G 网关、企业防火墙。

NAT 设备维护一个映射表：

```
内网地址:端口 ↔ 外网地址:端口（NAT 分配）
```

当玩家发出一个连接请求，NAT 设备为这条连接创建一条映射记录，并把内网地址"翻译"成外网地址转发出去。服务端看到的是 NAT 的外网地址，看不到玩家的真实内网地址。

这里有一个关键问题：**NAT 映射表的记录不是永久的**。大多数 NAT 设备对"空闲"连接（一段时间内没有数据通过）会自动删除映射记录。

这个超时时间因设备而异：
- 家用路由器（TCP）：通常 5-10 分钟
- 运营商 4G NAT（TCP）：通常 2-5 分钟
- 企业防火墙（UDP）：可能只有 30 秒！

一旦 NAT 映射记录删除，客户端发出的包无法路由到服务端（因为 NAT 不知道这个包该转发给谁），服务端发出的包也无法路由到客户端（同理）。

**从 TCP 层面看**，连接并没有正式断开（没有 FIN/RST 包），双方的 TCP 状态仍然是 ESTABLISHED。这叫做**半开连接（Half-open Connection）**或**幽灵连接（Ghost Connection）**——连接在两端的内核看起来是活的，但实际上数据根本传不到对方。

这就是心跳的根本存在理由：**不是为了传输有价值的数据，而是为了维持 NAT 映射表记录，以及让双方能及时发现连接已经不通了**。

## 具体实现：心跳设计

### 基本结构

心跳的标准实现模式：

```
客户端定时发 Ping → 服务端收到后立刻回 Pong
```

不是服务端主动发 Ping，原因是：
1. 服务端是多连接的，主动给每个客户端发 Ping 会显著增加服务端的发包频率
2. 客户端发 Ping 可以同时验证上行链路是否正常，Pong 验证下行链路
3. 如果客户端不发 Ping，服务端没有收到任何消息，比收到错误消息更容易区分

### 心跳间隔和超时的合理取值

```
Ping 间隔：15-30 秒（大多数游戏）
超时判定：3 次未收到 Pong，或超过 60-90 秒没有任何数据

为什么不能 Ping 太频繁：
- 增加带宽消耗（虽然每个心跳包很小，但几百万连接的累积量不小）
- 增加服务端处理压力
- 手机耗电（频繁唤醒网络栈）

为什么不能 Ping 间隔太长：
- NAT 超时可能快于心跳间隔（尤其 UDP 场景，一些 NAT 30 秒超时）
- 断线检测滞后，玩家游戏体验变差
```

实际工程中，对于 UDP 游戏：Ping 间隔 10-15 秒。对于 TCP/WebSocket 游戏：Ping 间隔 20-30 秒。

### 心跳消息的设计细节

心跳包不应该用游戏逻辑消息 ID。通常用一个专用的 Ping/Pong 消息类型，它：
- 携带时间戳（客户端发出时的时间），用于计算 RTT
- 极小（8-16 字节），不给序列化层增加开销
- 不触发游戏逻辑处理，直接在网络层处理后返回

```protobuf
message Ping {
    int64 client_timestamp_ms = 1;
}
message Pong {
    int64 client_timestamp_ms = 1;  // 原样返回
    int64 server_timestamp_ms = 2;  // 服务端时间戳
}
```

客户端收到 Pong 时：`RTT = now() - ping.client_timestamp_ms`，这个 RTT 数据对延迟优化和客户端断线预警都很有用。

### 优化：任何数据包都算"活跃"

不要仅仅靠 Ping/Pong 判断连接存活。每次收到任何数据包（位置更新、技能操作等），都应该更新连接的"最后活跃时间"。这样在游戏进行中（持续有数据包），就不需要额外的心跳包。

```python
def on_receive_any_packet(connection, packet):
    connection.last_active_time = current_time()  # 更新活跃时间
    # ... 处理具体消息
```

只有在连接进入"空闲"状态（没有游戏数据交换）时，才依赖心跳包维持。

## 连接数上限：C10K 和 C100K 问题

### 传统阻塞 I/O 的瓶颈

早期服务端编程的经典模型是：每个连接对应一个线程（或进程）。

```
连接 1 → 线程 1
连接 2 → 线程 2
...
连接 1000 → 线程 1000
```

每个线程占用约 8MB 栈内存（Linux 默认），1000 个线程就是 8GB 内存。更严重的问题是**上下文切换开销**：1000 个线程竞争 CPU，调度器的开销本身就会消耗大量 CPU 时间。

1999 年，Dan Kegel 提出 **C10K 问题**：如何用一台服务器同时处理 1 万个连接？这在当时是一个工程挑战，而现在已经是基础要求。当前游戏服务端的目标是 **C100K（10 万并发连接）**，大型 MMO 甚至需要 C1M（百万连接）。

### epoll：非阻塞 I/O 的核心机制

解决 C10K 问题的关键是**I/O 多路复用**，在 Linux 上最高效的实现是 **epoll**。

```c
// 创建 epoll 实例
int epfd = epoll_create1(0);

// 注册所有连接的 socket fd
for (int i = 0; i < conn_count; i++) {
    struct epoll_event ev;
    ev.events = EPOLLIN | EPOLLET;  // 监听可读事件，边缘触发
    ev.data.fd = connections[i].fd;
    epoll_ctl(epfd, EPOLL_CTL_ADD, connections[i].fd, &ev);
}

// 等待事件：只有有数据的 fd 才会返回
int n = epoll_wait(epfd, events, MAX_EVENTS, timeout_ms);
for (int i = 0; i < n; i++) {
    // 处理 events[i].data.fd 上的数据
}
```

epoll 的工作模式：
- 不是轮询所有连接，而是让内核监视，只在有事件发生时通知
- 1 个线程可以管理 10 万个连接的事件通知
- 内存消耗和连接数无关，只取决于"同时有事件的连接数"

**epoll 的实际连接数上限**由以下因素决定：

```
1. 文件描述符数量限制
   - 每个进程默认 ulimit -n 1024，需要调整到 65535 或更高
   - 系统级 /proc/sys/fs/file-max

2. 内存：每个连接的应用层状态（连接对象 + 读写缓冲区）
   - 假设每个连接 4KB（缓冲区 + 状态），10 万连接 = 约 400MB

3. CPU：处理实际数据的能力（epoll 本身开销很低）
```

```bash
# 生产环境调整
ulimit -n 1000000                    # 文件描述符上限
sysctl -w net.core.somaxconn=65535   # 连接队列长度
sysctl -w net.ipv4.tcp_max_syn_backlog=65535
```

### io_uring：Linux 5.1+ 的下一代 I/O 模型

io_uring 是 Linux 5.1（2019年）引入的异步 I/O 接口，进一步减少了 epoll 的系统调用开销：

- epoll 需要在每个事件上调用 `recv()`，每次系统调用有开销
- io_uring 通过共享内存的提交队列（SQ）和完成队列（CQ），批量提交和接收 I/O 操作，大幅减少系统调用次数

对于 C100K 以上的游戏服务端，io_uring 在高 QPS 场景下有 20-30% 的延迟改善（取决于包大小和处理逻辑）。2024 年后新建的高性能游戏服务端，io_uring 已经是值得考虑的选项。

## 工程边界：连接泄漏的排查

连接泄漏是游戏服务端常见的长期 bug。症状是：服务器运行数小时后，`/proc/自进程PID/fd` 里的文件描述符数量持续上涨，最终达到 `ulimit` 上限，新连接被拒绝，服务宕机。

### 排查方法

**第一步：确认泄漏存在**

```bash
# 查看进程打开的 fd 数量
ls /proc/$(pidof game_server)/fd | wc -l

# 查看所有 TCP 连接状态分布
ss -tn state all | awk '{print $1}' | sort | uniq -c
```

如果 CLOSE_WAIT 状态的连接持续增加，通常意味着：服务端收到了 FIN（对端关闭），但服务端代码没有关闭对应的 socket，连接卡在 CLOSE_WAIT 状态不释放。

**第二步：追踪连接生命周期**

在每个连接创建时打印一条日志（包含 fd 号、创建时间），在关闭时打印另一条。用日志分析工具找出"只有创建没有关闭"的 fd。

```
[CONN_OPEN]  fd=1234 addr=192.168.1.100:56789 time=2026-04-04T10:00:00
[CONN_CLOSE] fd=1234 reason=timeout time=2026-04-04T10:45:00
```

**第三步：检查异常退出路径**

在错误处理、panic/exception 路径中，检查 socket 是否被正确关闭。C++ 中通过 RAII 管理 socket 生命周期，Go 中通过 defer 确保 `conn.Close()` 被调用。

```go
func handleConnection(conn net.Conn) {
    defer conn.Close()  // 确保任何退出路径都关闭连接
    // ...
}
```

**第四步：监控指标**

把以下指标接入监控系统（Prometheus/Grafana）：

- 当前活跃连接数
- 进程打开的 fd 数量
- CLOSE_WAIT / TIME_WAIT 连接数
- 每分钟新建/关闭连接数

这些指标的异常趋势（缓慢上涨而非平稳）通常是泄漏的早期信号，在服务宕机前几小时就能发现。

## 服务端主动断开连接的时机

不是所有连接都等玩家自己断。服务端需要主动关闭连接的场景：

**超时断线**：超过设定时间没有收到任何数据（包括心跳），服务端认为客户端网络彻底断开，主动关闭连接并清理玩家状态。

**违规检测**：反外挂系统检测到异常（外挂、速度修改、非法数据包），立即断开连接并封号。断开连接的顺序：先在逻辑层清理玩家状态（防止其他玩家看到幽灵玩家），再关闭 socket。

**服务重启**：灰度重启（Graceful Shutdown）时，停止接受新连接，给现有连接一个宽限期（通常 30-60 秒）让玩家迁移到其他节点，超时后强制关闭剩余连接。

**会话冲突**：同一账号在另一台设备上登录，服务端向旧连接发送"账号在其他地方登录"通知，然后关闭旧连接。

**主动关闭的正确姿势**：

```
1. 发送"即将断开"的消息给客户端（让客户端有机会弹出提示）
2. 等待一个短暂延迟（约 100ms，让消息发出）
3. 调用 close()/conn.Close()
4. 清理内存中的连接对象和玩家状态
```

不要先清理状态再关闭 socket，也不要连通知都不发就直接关——良好的断开流程能让客户端表现得更优雅，减少玩家投诉。

## 最短结论

游戏长连接管理的核心问题是：NAT 设备会静默删除空闲连接的映射记录，导致连接在双端看起来活着但实际无法传输数据。心跳的第一职责是维持 NAT 映射，第二职责才是检测断线。

epoll 让单个服务端进程管理 10 万级别的连接成为可能，连接数的真正瓶颈是内存和文件描述符配置，不是 epoll 本身。

连接泄漏是长期运行的游戏服务端的常见慢性病，通过连接数监控和生命周期追踪日志可以在宕机前发现并修复。

服务端主动断开连接是正常运营的一部分，有规范的断开流程（通知 → 延迟 → 关闭 → 清理）能显著提升玩家体验。
