## 服务端权威不是一个开关

许多开发者把"服务端权威"理解为一个设计原则：把重要计算放服务端就行了。但落地时会发现，"放到服务端"本身就是一系列工程决策的集合，每一项都有成本，都有取舍。

更准确的问法是：**这个操作的最终裁决必须在服务端做吗？如果绕过服务端验证，攻击者能获得什么？**

---

## 哪些操作必须在服务端做最终裁决

判断标准是：**如果客户端能自行决定这件事的结果，攻击者能从中获益多少？**

### 必须服务端裁决的操作

**伤害计算**：客户端只上报"我射击了，目标是谁"，伤害值由服务端根据武器配置、距离、护甲计算。如果允许客户端上报伤害值，攻击者可以直接上报 999999 伤害。

**物品获取/消耗**：拾取物品、使用道具、消耗资源必须经过服务端验证库存状态。客户端不持有权威库存。

**技能释放合法性**：技能冷却时间、法力消耗必须在服务端验证。客户端可以绕过本地冷却检查，但服务端维护独立的冷却状态。

**位置最终状态**：玩家坐标的最终值由服务端决定。服务端拒绝不合理的位置更新时，向客户端发送强制纠正包。

**游戏结果**：击杀判定、目标达成、比赛结算完全在服务端计算，客户端上报的结果数据仅作参考。

### 可以延迟验证的操作

**特效与音效**：纯表现层，不影响游戏状态，客户端自主决定。

**本地预测（客户端预测）**：玩家移动、技能预输入等先在客户端执行以降低延迟感，服务端验证后对不合法操作进行回滚。这是延迟补偿的核心机制。

**UI 状态**：金币显示、血条数字等是服务端状态的本地镜像，异常也只影响显示不影响实际游戏状态。

## 输入合法性验证

### 输入速率限制

每个玩家每单位时间能发出的操作包数量是有上限的：

```python
class InputRateLimiter:
    def __init__(self, max_actions_per_second, burst_limit):
        self.max_rate = max_actions_per_second
        self.burst_limit = burst_limit
        self.tokens = burst_limit  # 令牌桶
        self.last_refill = time.now()
    
    def allow(self):
        now = time.now()
        elapsed = now - self.last_refill
        self.tokens = min(
            self.burst_limit, 
            self.tokens + elapsed * self.max_rate
        )
        self.last_refill = now
        if self.tokens >= 1:
            self.tokens -= 1
            return True
        return False  # 丢弃此包
```

速度外挂通过加快游戏内时钟使客户端每秒发出更多操作包，超出速率限制的包被服务端直接丢弃。

**阈值的设置**：正常 Tick 率 × 延迟容差系数。例如客户端 60Hz 输入，允许 2 倍容差即 120 包/秒。超出视为异常。

### 操作合法性校验矩阵

不同操作有不同的合法性约束：

| 操作类型 | 验证项目 |
|---|---|
| 移动 | 位移距离 ≤ max_speed × delta_time × tolerance |
| 射击 | 武器装弹状态、冷却时间、弹药数量 |
| 技能 | 冷却时间、资源消耗、目标范围合法性 |
| 交互 | 与交互目标的距离 ≤ 交互范围 |
| 传送 | 目标坐标是否在合法传送点列表内 |

拒绝非法操作时，服务端不仅要拒绝，还要将当前权威状态同步给客户端（防止客户端和服务端状态永久分叉）。

### 操作序列完整性

单个操作合法，但操作序列可能不合法。例如：
- 在未拿出武器的状态下射击
- 在施法中途（有打断限制的技能）又释放另一个技能
- 在死亡状态下移动

服务端维护玩家的状态机，每个操作必须符合当前状态的允许转换。

## 物理边界检查

### 服务端物理模拟的取舍

完整的服务端物理模拟（Physics Server）能防止穿墙、传送、碰撞无效等问题，但成本显著：

- CPU 成本：物理模拟是计算密集型操作，每 Tick 都要运行
- 精度要求：服务端物理必须与客户端物理保持足够一致，否则合法操作会被误判
- 延迟复杂性：需要配合延迟补偿处理客户端和服务端的物理状态差异

实际项目中，通常不做完整物理模拟，而是做**轻量级合法性检查**：

```python
def validate_move(player, new_position, delta_time):
    # 1. 速度检查
    displacement = distance(player.position, new_position)
    max_allowed = player.max_speed * delta_time * TOLERANCE
    if displacement > max_allowed:
        return False, "speed_violation"
    
    # 2. 碰撞检查（简化版）
    if intersects_solid_geometry(player.position, new_position):
        return False, "wall_clip"
    
    # 3. 高度合法性（防止飞行外挂）
    if not is_grounded(new_position) and not player.can_fly:
        if new_position.y > player.position.y + JUMP_TOLERANCE:
            return False, "fly_hack"
    
    return True, None
```

碰撞检测通常使用简化的碰撞体（胶囊体或 AABB），而非完整的渲染网格，以控制计算成本。

### 地图边界与禁区

维护服务端的地图合法区域定义，拒绝坐标落在地图外或禁区内的移动请求。这是防止传送外挂的最直接手段。

## 服务端回放验证

### 什么是服务端回放验证

对于射击类游戏，服务端记录一段时间内所有玩家的输入和位置历史，当收到一个"命中"事件时，服务端回放（Replay）那一时刻的游戏状态，独立验证这一命中是否物理上可能发生。

```
客户端发来：
  timestamp: T
  shooter_id: A
  target_id: B  
  hit_position: (x, y, z)

服务端回放：
  1. 从历史缓冲区找到 T 时刻 A 和 B 的位置
  2. 计算 A 在 T 时刻的视角方向
  3. 做射线检测：从 A 位置沿视角方向发射射线
  4. 射线是否命中 B 的碰撞体（考虑 B 在 T 时刻的位置）
  5. 是否有遮挡物阻挡（防穿墙子弹）
  
验证通过 → 接受命中，计算伤害
验证失败 → 拒绝命中
```

### 历史缓冲区的大小

缓冲区需要保存多长时间的历史？这取决于你允许的最大延迟：

- 支持到 150ms 延迟 → 保存 150ms 历史
- 支持到 300ms 延迟 → 保存 300ms 历史

历史缓冲区的内存成本随玩家数量线性增长。100 个玩家、300ms 缓冲、每帧存储约 100B → 每玩家 30 帧 × 100B = 3KB，100 玩家 = 300KB，可接受。

### 延迟补偿场景下的权威边界

延迟补偿本质上是一种"时间旅行"：服务端回到过去验证操作。这带来了一个争议性设计问题：

**场景**：玩家 A（延迟 200ms）向已经躲到掩体后面的玩家 B 开枪。在 A 的视野中，B 还没躲好（因为 A 看到的是 200ms 前的状态）。服务端回放 200ms 前的状态，发现 B 当时确实在 A 的视线内，命中有效。

但对 B 来说，他明明已经躲好了，却仍然被打死——这是"被过去击中"。

这是延迟补偿的固有矛盾，没有完美解法，只有取舍：

- **偏向射击方**：大多数 FPS（如 Overwatch、CS2）的选择，减少射击手的延迟感
- **限制补偿时间**：只补偿到最大阈值（如 100ms），减少"时间旅行"的极端感
- **目标位置不回溯**：只回溯射击方的视角，目标始终使用当前位置

选哪种方案是游戏设计决策，但无论哪种，服务端都必须是最终裁决者。

## 成本与性能考量

服务端权威检查不是免费的，每增加一个验证步骤都有计算成本：

| 验证类型 | 相对计算成本 | 适用场景 |
|---|---|---|
| 速度/距离检查 | 低（O(1) 数学计算）| 所有移动操作 |
| 状态机转换检查 | 低（O(1) 状态查表）| 所有操作 |
| 射线检测（简化碰撞体）| 中（取决于场景复杂度）| 射击命中验证 |
| 完整物理模拟 | 高（每 Tick 全量）| 仅对关键操作 |
| 行为统计分析 | 高（异步处理）| 离线/后台管道 |

实践中，逐帧的轻量级检查在服务器主循环里做，重量级分析（统计、回放、机器学习）在异步管道里处理。

---

## 工程边界

- 服务端验证过于严格会误拒合法操作（特别是高延迟玩家），容差系数需要基于真实网络数据校准
- 客户端预测 + 服务端回滚的架构下，回滚频率过高会造成"橡皮筋"现象，影响游戏体验
- 服务端物理状态和客户端物理状态的同步是一个复杂工程问题，引擎选型（Deterministic Physics vs Float Physics）会影响方案
- 回放验证的历史缓冲区是内存成本，需要根据最大在线人数和目标延迟容忍度规划容量

---

## 结论

服务端权威的实现是一组从成本和体验双向约束下做取舍的工程决策：轻量级验证放主循环，重量级分析走异步管道，延迟补偿范围由游戏设计目标决定，但最终裁决权永远在服务端。
