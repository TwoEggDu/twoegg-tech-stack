## 问题空间

某 FPS 游戏新赛季上线，在线人数从平日的 3 万涨到 15 万，维持两周后回落到 5 万。运营团队事后查账单，发现那两周的云费用是平日的 8 倍，但 On-Demand 实例没有及时缩容，高峰期结束后又多付了两周的费用。全年云账单超出预算 40%。

这不是极端案例，而是游戏公司云成本管理失控的典型画像。游戏业务有其特殊性：流量高度集中在特定时段（晚高峰、节假日、新版本上线），峰谷比可能达到 10:1，但底层云资源如果不精细规划，就会持续按峰值付费。

云成本优化不是运维的副业，而是影响游戏利润率的工程问题。

---

## 抽象模型

### 游戏服务器云成本的构成

游戏服务器的云成本主要来自四类：

```
典型中型多人游戏月云账单构成（参考值）：

计算实例（EC2/CVM）            45-55%
  - 游戏逻辑服
  - 匹配服、网关服
  - 数据库服务器

网络带宽（出口流量）            20-30%
  - 服务端 → 客户端的状态同步数据
  - 比 Web 服务高得多（持续长连接 + 高频推送）

存储（块存储 + 对象存储）       5-10%
  - 数据库磁盘
  - 备份文件（对象存储）
  - 游戏资源包 CDN 源站

托管数据库（RDS/云数据库）     10-15%
  - 通常比自建数据库贵，但运维成本低
  - 读写副本数量决定这项费用的弹性

其他（负载均衡、日志存储等）   5-10%
```

带宽成本的占比在游戏场景中远高于普通 Web 服务，是成本优化中经常被低估的方向。

## 具体实现

### Spot Instance：哪些组件能用，哪些不能

Spot Instance（AWS）/ 竞价实例（腾讯云/阿里云）是云厂商利用空闲算力以折扣价（通常 20-70% off）提供的实例，但云厂商可以在给出 2 分钟通知后中断实例。

这个"随时可能被中断"的特性，决定了 Spot Instance 只适合**无状态或可容忍中断**的组件。

#### 适合使用 Spot Instance 的组件

**无状态逻辑服（战斗服、物理模拟）**：如果游戏对局状态不完全依赖单台服务器内存（有 checkpoint 机制或对局可以重新分配），这类服务中断后影响的只是进行中的对局，可以通过客户端重连恢复。

```
对局分配策略（支持 Spot 中断）：
  匹配服把对局分配到 Spot 实例池
  每个对局有独立 ID，状态定期写入 Redis
  实例收到中断通知（2分钟警告）时：
    1. 停止接收新对局分配
    2. 将进行中的对局迁移到其他实例（或发放补偿后结束）
    3. 上报中断通知到控制层，触发补充实例
```

**离线批处理任务**：排行榜计算、战报分析、反作弊数据分析、AI 训练（游戏 bot 训练）。这类任务支持检查点保存，实例中断后从检查点继续，完全适合 Spot。

**压测 bot 节点**：压测本身就是临时任务，Spot 中断只是提前结束某些 bot，可以接受。

#### 不适合使用 Spot Instance 的组件

**数据库服务器**：主库宕机意味着所有写操作中断，2 分钟内完成主从切换极其困难，风险极高。

**匹配服**：匹配服中断会让等待匹配的玩家全部掉线重排，用户体验极差。而且匹配服通常维护全局玩家等待队列，这个状态迁移复杂。

**网关/接入层**：玩家连接都接入网关，中断后所有在线玩家断开。即使有自动重连机制，断线体验也是负面的。

**有状态的缓存层（如 Redis 主节点）**：Redis 存储了大量会话状态，Spot 中断会导致大量状态丢失。

#### Spot 使用的工程要点

1. **多可用区 + 多实例类型**：Spot 中断通常是可用区级别的，同时配置多个可用区和实例类型，降低同时中断的概率。
2. **中断通知处理**：AWS 提供 EC2 Metadata 中断通知（`/latest/meta-data/spot/termination-time`），实例应轮询这个接口，收到通知后做 graceful shutdown。
3. **Spot 与 On-Demand 混合**：不要 100% 使用 Spot，保留基础 On-Demand 比例（如 30%），确保即使全部 Spot 中断，服务也不会完全不可用。

```python
# EC2 Spot 中断通知轮询（部署在实例内）
import requests
import time

def check_spot_interruption():
    try:
        resp = requests.get(
            'http://169.254.169.254/latest/meta-data/spot/termination-time',
            timeout=1
        )
        if resp.status_code == 200:
            # 收到中断通知，有 ~2 分钟处理时间
            return True
    except:
        pass
    return False

while True:
    if check_spot_interruption():
        graceful_shutdown()
        break
    time.sleep(5)
```

### Reserved Instance：预估基础负载，覆盖 70%

Reserved Instance（预留实例）是以 1 年或 3 年合同提前锁定实例使用权，换取 30-70% 的折扣。

**适用逻辑**：将服务器负载分为"基础负载"和"弹性负载"两层。

```
在线玩家数的典型日曲线：
  
  ████████████████
  ██              ████████████
  ██                          ████
  ██                              ████████████████

  基础负载（全天持续存在）：    ← 用 Reserved Instance 覆盖
  弹性峰值（晚高峰、版本上线）：← 用 On-Demand 覆盖
```

**具体操作**：

1. 取过去 3 个月的每日最低在线峰值（避免节假日等异常值），这是稳定的基础负载。
2. 根据基础负载计算需要多少实例（通常 CPU 利用率 60-70% 基准）。
3. 用 Reserved Instance 覆盖这部分，签 1 年合同（3 年风险太高，游戏生命周期不确定）。
4. 超出基础负载的部分用 On-Demand，极端峰值可以混入 Spot。

**经验比例**：RI 覆盖 65-75% 的稳定负载，On-Demand 覆盖 20-25%，Spot 覆盖 5-10%。

**RI 的风险**：如果游戏在 RI 合约期内下线，RI 费用仍然要付。AWS 提供 RI Marketplace 可以转让，但价格会有折损。腾讯云、阿里云也有类似的 RI 提前退订机制，损失不同。

### 弹性扩缩容的成本陷阱

#### 陷阱一：只有扩容策略，没有缩容策略

这是最常见的错误。工程师设置了"CPU > 70% 时自动扩一台"，但没有设置"CPU < 30% 持续 15 分钟时缩容"。结果：高峰期扩容到 50 台，高峰过后仍然挂着 50 台 On-Demand 实例，每小时多付 40 台的费用。

**正确配置**：扩容和缩容策略成对出现，且缩容策略要有足够的冷却时间（避免振荡）。

```yaml
# AWS Auto Scaling 策略示例
ScaleOutPolicy:
  metric: CPUUtilization
  threshold: 70%
  comparison: GreaterThanThreshold
  evaluation_periods: 2  # 连续2个采样周期（约2分钟）超阈值才扩容
  adjustment: +2          # 每次扩2台
  cooldown: 300s          # 扩容后5分钟内不再扩容

ScaleInPolicy:
  metric: CPUUtilization
  threshold: 30%
  comparison: LessThanThreshold
  evaluation_periods: 10  # 连续10个周期低于阈值（约10分钟）才缩容
  adjustment: -1           # 每次缩1台（缩容要保守）
  cooldown: 600s           # 缩容后10分钟内不再缩容
```

缩容比扩容需要更保守：错误扩容的代价是多付几十分钟费用，错误缩容的代价是服务不可用。

#### 陷阱二：扩容速度跟不上流量增长

新版本上线，玩家在发布后的 5 分钟内涌入，On-Demand 实例从启动到可以接受游戏连接需要 3-5 分钟（镜像拉取 + 程序启动 + 预热）。如果扩容触发太晚，中间有几分钟的服务降级窗口。

**解决方案**：
1. **预测性扩容（Scheduled Scaling）**：已知版本更新时间，提前 30 分钟手动触发扩容，而不是等待自动触发。
2. **保持预热实例（Warm Pool）**：AWS 提供 Warm Pool 功能，预先保有已启动但未接入流量的实例，需要时可以在秒级接入，代价是为这些实例支付费用（但只需 On-Demand 价格的一部分，因为 Warm Pool 实例处于停止状态）。

#### 陷阱三：缩容时的玩家数据保全

游戏服务器缩容时，不能简单地 kill 实例。如果实例上有玩家正在游戏中，需要：
1. 将实例标记为"不接受新连接"（deregister from load balancer）
2. 等待现有对局结束（或超时强制结束）
3. 玩家数据持久化到数据库
4. 然后才能安全终止实例

这个 graceful drain 过程可能需要 30-60 分钟（等待最长对局结束）。自动扩缩容框架需要支持自定义的 lifecycle hook，而不是直接强制终止。

### 带宽成本优化

游戏出口带宽是账单中容易被忽视的大项。具体来说，成本来自**服务端向客户端发送的数据量**。

#### 带宽估算

```
单玩家带宽：
  帧同步游戏：约 10-50 KB/s（取决于游戏帧率和指令密度）
  状态同步 MMO：约 5-20 KB/s
  FPS 射击游戏：约 20-80 KB/s

5 万并发玩家，平均 30 KB/s：
  = 50,000 × 30 KB/s = 1.5 GB/s 出口带宽

AWS 出口带宽定价（us-east-1，约）：
  前 10TB/月：$0.09/GB
  10TB-40TB：$0.085/GB
  
  1.5 GB/s × 3600s × 24h × 30天 = 约 3,888 TB/月
  ≈ $330,000/月（仅带宽费用）
```

这个数字让很多团队震惊——带宽账单可能比计算实例还贵。

#### 带宽优化策略

**策略一：减少每玩家带宽用量**
- **Delta 压缩**：只发送状态变化，而不是每帧发送完整状态
- **AOI 截流**：AOI 范围外的实体不同步（前面 AOI 章节已讲）
- **优先级降频**：远距离实体降低同步频率（从 30Hz 降至 5Hz）
- **数据压缩**：对同步包应用 zlib/LZ4 压缩（需要权衡 CPU 开销与带宽节省）

**策略二：CDN 分发 vs 直连**

游戏资源包（更新包、DLC）的下载流量可以通过 CDN 分发，避免从源站直出带宽。CDN 价格通常比云厂商出口带宽便宜。

但游戏**实时通信数据**（位置同步、命中判定）不适合走 CDN（CDN 是缓存层，不支持双向低延迟通信）。有一个折中方案：使用**网络加速服务**（如 AWS Global Accelerator、腾讯云 GA），将玩家流量接入就近的云骨干网，减少公网绕路，同时可能获得比 CDN 更低的带宽价格。

**策略三：区域出口流量管理**

同一云厂商内部（如同一 VPC 内、同一区域内）的流量通常不计费或费率极低。跨区域和出公网的流量才是大头。架构设计上，尽量将高带宽组件放在同一区域、同一 VPC 内部通信。

### 真实案例：某 FPS 游戏新赛季账单分析

以下是一个基于真实模式的账单分析（数据已脱敏）。

**背景**：某 FPS 游戏，平日稳定 5 万在线，新赛季第 1 周峰值 18 万在线，第 2 周回落到 8 万，第 3 周稳定在 6 万。

**账单对比（月均）：**

| 费用类型 | 赛季前平日 | 赛季第1周 | 赛季第3周 |
|---------|-----------|----------|---------|
| 计算（RI覆盖部分）| $18,000 | $18,000 | $18,000 |
| 计算（On-Demand）| $2,000 | $28,000 | $6,000 |
| 带宽（出口）| $9,000 | $32,000 | $11,000 |
| 数据库（RDS）| $5,000 | $8,000 | $5,500 |
| 其他 | $2,000 | $3,000 | $2,200 |
| **合计** | **$36,000** | **$89,000** | **$42,700** |

**关键发现：**

1. **带宽是第一增长因子**：在线人数从 5 万增加到 18 万（3.6×），带宽成本从 $9,000 增加到 $32,000（3.5×），接近线性增长。带宽优化投入产出比最高。

2. **On-Demand 第 2 周的错误**：第 2 周在线人数回落到 8 万，但工程师没有及时缩容 On-Demand 实例（高峰时扩到 80 台逻辑服，结束后只缩到 50 台），多付了约 $6,000。

3. **RI 的价值**：RI 覆盖了稳定基础负载对应的 40 台逻辑服，相比 On-Demand 节省约 40%，全年节省约 $86,000。

4. **Spot 的潜在空间**：离线的战报分析、反作弊批处理、AI 对战 bot 训练全部用 On-Demand，迁移到 Spot 预计可节省该部分 60%，约 $800/月。

## 工程边界

### 成本优化与工程复杂度的权衡

Spot Instance 的使用需要中断处理逻辑，RI 规划需要精确的流量预测，带宽优化需要协议层改动。每一项优化都增加系统复杂度。

**建议优先级：**
1. 首先确保缩容策略正确配置（零额外工程量，立竿见影）
2. 然后做 RI 规划（只需要采购决策，不需要改代码）
3. 再考虑 Spot 接入（需要 graceful shutdown 逻辑）
4. 最后优化每玩家带宽用量（需要协议层改动，测试周期长）

**另一个边界**：成本优化的时机。游戏上线初期，稳定性比成本更重要，不应该为了省 20% 的成本而引入 Spot 中断风险。游戏运营稳定（上线 3-6 个月后）再系统性做成本优化，是更合理的时序。

---

## 最短结论

游戏云成本优化的最大回报来自三件事：缩容策略与扩容策略同等重视、RI 覆盖稳定基础负载、带宽用量从协议层降低，其余都是边际收益。
