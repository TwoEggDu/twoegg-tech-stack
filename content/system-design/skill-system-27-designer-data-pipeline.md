> 策划每天做的事情，不是"在表里填一个数"，而是驱动一条从配置到运行时的完整管道。管道断了，数填对也没用。

很多程序会觉得策划的工作就是"填数值表"。

但如果你真正站在策划视角看，他每天面对的链路远比"填一行"复杂：

- 在 Excel 或编辑器里改一个字段
- 保存后触发导出
- 导出产物进入构建管线
- 构建通过后部署到开发环境
- 开发环境里进游戏验证效果
- 如果效果不对，回头重新改

这条链路里，任何一个环节断了，策划的迭代效率就会直接崩掉。

而更麻烦的是，技能系统的配置远比一般的数值表复杂。它不只有数字，还有引用关系、公式、事件绑定、资源路径。所以这条管道如果没有被显式设计过，团队很快就会掉进"改了一个数，线上崩了"的坑。

这一篇要回答 5 个问题：

1. 策划视角的完整数据管道到底有几段。
2. SkillDef 哪些字段应该暴露给策划、哪些应该隐藏、为什么。
3. 公式系统的复杂度边界在哪——策划应该写公式还是填参数。
4. 多人协作时同一张配置表的版本控制怎么做。
5. "改了一个数线上崩了"的防护链应该长什么样。

---

## 策划视角的技能配置全流程

如果把策划配一个技能的完整过程摊开来看，至少有六段。

### 第一段：数值表 / 编辑器里编辑

策划在 Excel、Google Sheets 或者自研编辑器里完成配置。

这一步要决定的是：技能的基础参数是什么、引用了哪些效果和 Buff、目标选择模式是什么。

### 第二段：本地校验

保存之后，最好立刻有一层校验能在本地告诉策划"这行配置是不是合法的"。

例如：

- 冷却时间填了负数
- 引用的 EffectDef 不存在
- 技能声明了投射物但没配 ProjectileSpell
- 伤害类型和目标类型矛盾

这层校验越早越好，因为一旦等到进游戏才发现，来回一趟可能要 5 到 10 分钟。

### 第三段：导出

从编辑格式（Excel / JSON / 编辑器私有格式）导出为运行时格式。

常见的运行时格式包括：

- 二进制 FlatBuffers / Protocol Buffers
- 序列化 JSON / MessagePack
- ScriptableObject（Unity）
- KV 文本（Valve 系引擎）

这一步最关键的不是"选哪种格式"，而是导出过程是不是自动的、可重复的、有错误报告的。

### 第四段：CI 校验

导出产物进入 CI 管线后，跑一遍更完整的校验：

- 引用完整性检查（所有 ID 能否解析）
- 公式语法检查
- 依赖图是否有循环
- 和上一个版本的 diff 报告

### 第五段：部署到开发环境

通过 CI 的配置自动部署到开发服务器或本地模拟环境。

### 第六段：进游戏验证

策划在开发环境里打开游戏，释放技能，看效果是否符合预期。

## 这六段里，最容易断的是哪几段

根据我的经验，最容易断的是第二段和第三段。

### 第二段断裂：没有本地校验

很多团队的策划改完配置后，唯一的验证方式就是"进游戏试"。

结果是：

- 一个拼写错误要花 10 分钟才能发现
- 一个引用丢失要等到运行时报空引用才知道
- 策划的迭代速度被"启动游戏"的时间绑死

### 第三段断裂：导出不稳定

导出脚本是手动跑的、导出顺序不固定、导出结果不确定性高。

最典型的症状：

- "我本地导出是对的，CI 导出就不对"
- "导出脚本崩了但没人发现，用了昨天的旧数据上线"
- "导出顺序变了，ID 映射表全乱了"

## 配表结构设计：SkillDef 哪些字段暴露、哪些隐藏

这一层决定了策划的工作界面到底长什么样。

从 `02 篇`我们知道 SkillDef 至少有这些字段：

```csharp
public class SkillDef
{
    public string id;
    public string displayName;
    public CostDef cost;
    public CooldownDef cooldown;
    public CastPolicyDef castPolicy;
    public TargetingDef targeting;
    public List<EffectDef> effects;
    public List<TagId> requiredTags;
    public List<TagId> blockedTags;
}
```

但策划不需要看到所有字段。我建议按下面三层分类。

### 暴露层：策划每天都要改的字段

| 字段 | 说明 | 为什么暴露 |
|------|------|-----------|
| displayName | 技能显示名 | 策划命名 |
| cost.amount | 消耗数值 | 数值平衡核心 |
| cooldown.duration | 冷却时间 | 数值平衡核心 |
| cooldown.maxCharges | 最大充能数 | 影响释放节奏 |
| castPolicy.windup | 前摇时间 | 手感调校 |
| castPolicy.activeWindow | 生效窗口 | 手感调校 |
| castPolicy.recovery | 后摇时间 | 连招节奏 |
| targeting.range | 施法距离 | 战斗设计 |
| targeting.radius | 范围半径 | 范围技设计 |
| effects（列表选择） | 引用哪些效果 | 技能行为组装 |

### 受限层：策划偶尔需要改、但有约束的字段

| 字段 | 说明 | 约束 |
|------|------|------|
| castPolicy.castType | 施法类型 | 只能从枚举里选 |
| targeting.mode | 目标选择模式 | 只能从枚举里选 |
| cost.type | 消耗资源类型 | 只能从已注册类型选 |
| requiredTags | 施放前置标签 | 从标签库选 |
| blockedTags | 施放禁止标签 | 从标签库选 |

### 隐藏层：策划不该碰的字段

| 字段 | 说明 | 为什么隐藏 |
|------|------|-----------|
| id | 技能唯一标识 | 程序生成或严格管控 |
| cost.consumeOnCastStart | 消耗时机 | 架构决策 |
| cost.refundIfInterrupted | 中断退款 | 架构决策 |
| cooldown.cooldownGroup | 冷却分组 | 跨技能约束 |
| targeting.mask | 碰撞层 | 物理系统细节 |
| targeting.requireLineOfSight | 视线检查 | 寻路系统细节 |

这样分层之后，策划的编辑界面可以做得很干净：常改的放最上面，偶尔用的折叠，不该碰的直接不显示。

## 公式系统的复杂度边界

技能数值经常需要公式。最典型的比如：

```text
damage = base_damage + attack_power * coefficient
```

但这里有一个非常关键的决策：策划应该写公式，还是填参数？

### 方案 A：策划填参数，程序写公式模板

策划只需要填：

```text
base_damage = 100
coefficient = 1.2
```

程序在代码里定义公式模板：

```csharp
float damage = baseDamage + caster.attackPower * coefficient;
```

这种方案的好处是：

- 公式不会出语法错误
- 策划不需要理解变量作用域
- 运行时性能有保障

缺点是：

- 每种新公式都要程序加模板
- 策划想试一个新计算逻辑，必须等程序

### 方案 B：策划直接写公式表达式

策划在配置里写：

```text
damage = {base_damage} + {caster.attack_power} * {coefficient}
```

运行时用表达式解析器求值。

这种方案的好处是：

- 策划自由度高
- 不需要等程序加模板

缺点是：

- 公式可能有语法错误
- 性能可能有问题（每帧解析字符串）
- 变量名拼错了只有运行时才会报错
- 公式越写越复杂，最后没人看得懂

### 我的建议：混合方案

大多数项目最终会走到一个折中：

- 常规公式用模板，策划填参数
- 少数需要灵活性的地方开放表达式，但有语法校验和白名单变量
- 表达式在导出时预编译成字节码，不在运行时做字符串解析

关键判断标准是：

`如果一个公式的变量只有 2 到 3 个、结构固定，就用模板。如果策划确实需要在同一个字段里尝试不同的计算逻辑，才开放表达式。`

不要因为"以后可能需要灵活"就从第一天开始做完整的脚本系统。大多数项目用不到那个复杂度。

## 多人协作的版本控制

技能数值表在真实项目里最头疼的问题之一是：多个策划同时改同一张表。

### 问题本质

如果配置存成 Excel（.xlsx），Git 合并是不可能的——它是二进制文件，冲突只能手动解。

如果配置存成 JSON / CSV，Git 可以合并，但一张几百行的 CSV 里两个人各改了一行，diff 也非常难读。

### 三种常见策略

#### 策略一：Text Export + Git 合并

把 Excel 导出为行结构稳定的 CSV 或 JSON，策划在 Excel 里编辑，但提交到版本控制的是导出后的文本文件。

```text
策划编辑 Excel → 导出 CSV → 提交 CSV 到 Git
```

优点：Git 能 diff、能合并。

缺点：Excel 和 CSV 之间可能出现不一致。需要 CI 校验"Excel 重新导出后 CSV 是否和仓库一致"。

#### 策略二：行级锁 / 分区

把技能表按英雄、按技能组或按策划负责区域分成多张子表。

```text
skill_table_warrior.csv     -- 策划 A 负责
skill_table_mage.csv        -- 策划 B 负责
skill_table_common.csv      -- 需要审批才能改
```

优点：物理上避免了冲突。

缺点：跨表引用变得稍微复杂。有些全局字段（比如公共冷却组）需要单独一张共享表。

#### 策略三：自研编辑器 + 服务端存储

完全放弃 Excel，策划在自研编辑器里编辑，数据存在服务端数据库。

编辑器负责字段级锁定和冲突检测。

优点：粒度最细，冲突最少。

缺点：开发成本最高。如果编辑器不稳定，策划的工作流会直接断掉。

### 我的建议

大多数团队在项目中前期最稳的选择是策略一加策略二的组合：

- 配置以文本格式进版本控制
- 按功能区域分表
- CI 检查导出一致性
- 公共配置需要 Code Review 审批

## 防崩链："改了一个数线上崩了"的防护

这是技能配置管道里最严重的问题。它的根因通常是：配置变更没有经过任何校验就直接到了运行时。

一条合格的防崩链至少有三层。

### 第一层：保存时校验器

策划保存配置的瞬间，校验器立刻检查：

```text
[ERROR] skill_fireball: cooldown.duration = -1, 冷却时间不能为负
[ERROR] skill_fireball: effects[0] 引用的 EffectDef "dmg_fire_v2" 不存在
[WARN]  skill_fireball: castPolicy.windup = 0, 瞬发技能是否确认不需要前摇？
```

这层拦截的是最低级的错误——拼写、越界、引用丢失。

### 第二层：CI 管线校验

配置提交后，CI 跑完整校验：

```text
1. 重新从源文件导出，检查和提交的导出产物是否一致
2. 全量引用完整性检查
3. 公式语法检查
4. 依赖图循环检测
5. 和上一个版本的 diff 报告（哪些技能被改了、改了什么字段）
6. 可选：关键公式的数值回归测试
```

这层拦截的是跨配置的关联错误——引用断裂、循环依赖、数值异常。

### 第三层：灰度发布

即使通过了前两层，新配置也不应该直接全量发布。

更稳的流程是：

1. 先部署到内部测试服
2. 策划和 QA 在测试服验证
3. 灰度到一小部分线上用户
4. 确认没有异常后全量推送

这层拦截的是校验器覆盖不到的逻辑错误——比如"数值合法但游戏体验崩了"。

### 三层加起来

```text
保存时校验  →  CI 管线校验  →  灰度发布
拦低级错误     拦关联错误       拦体验错误
```

少任何一层，管道就不完整。

## 三个项目的策划工作流对比

不同项目给策划提供的工作界面差异很大。拿三个有公开信息的案例来看。

### Dota 2：KV 文本配置

策划在 KeyValues 文本文件里直接编辑技能定义。

```text
"AbilityCooldown"  "12.0 10.0 8.0 6.0"
"AbilityManaCost"  "90 100 110 120"
```

特点：

- 策划能看到所有字段，包括行为标志和事件绑定
- 没有图形化编辑器，门槛高
- 表达能力强，但复杂技能最终要退到 Lua 或 C++
- 版本控制直接用文本 diff

### Unreal GAS：Blueprint + DataAsset

策划通过 Blueprint 编辑 GameplayAbility，通过 DataAsset 配置 GameplayEffect。

特点：

- 有图形化节点编辑器
- 策划可以拖连线定义技能流程
- 但 Blueprint 的版本控制很痛苦——二进制文件，合并困难
- 复杂逻辑容易让 Blueprint 图变得不可读

### LoL：内部脚本系统

LoL 的技能配置使用内部的脚本和配置系统，策划通过专用工具链编辑。

特点：

- 有专门的数值配置层和脚本逻辑层分离
- 内部工具链高度定制
- 策划在工具里操作，不直接碰底层数据格式
- 校验和预览深度集成

### 对比总结

| 维度 | Dota 2 KV | GAS Blueprint | LoL 内部工具 |
|------|-----------|---------------|-------------|
| 策划入口 | 文本文件 | 图形编辑器 | 专用工具 |
| 暴露程度 | 全暴露 | 中等 | 按层暴露 |
| 版本控制 | 文本 diff 友好 | 二进制不友好 | 内部管理 |
| 公式灵活度 | AbilitySpecial 变量 | Attribute 计算 | 脚本层 |
| 校验 | 弱 | 编译期检查 | 强 |
| 迭代速度 | 快（文本热加载） | 中等 | 取决于工具链 |

没有绝对最优方案。但有一个共同趋势：

`随着项目规模变大，策划的工作界面一定会从"直接编辑底层数据"向"在受约束的编辑器里操作"演进。`

## 这篇真正想留下来的结论

策划配技能不只是"填一行数据"。它驱动的是一条从编辑、校验、导出、CI、部署到验证的完整管道。

这条管道的健康程度，直接决定了团队的迭代速度和线上稳定性。

三个最关键的设计决策是：

1. SkillDef 的字段分层暴露——不是所有字段策划都该看到。
2. 公式系统的复杂度边界——大多数场景模板加参数就够了，不要过早引入脚本。
3. 防崩链的三层覆盖——保存时校验拦低级错误、CI 拦关联错误、灰度拦体验错误。

如果只记一句话：

`策划的迭代速度不取决于他填表的速度，而取决于从"改了一个数"到"在游戏里看到结果"这条管道有多短、有多稳。`
