## 为什么要区分这三个概念

在很多团队的日常用语里，"内容"、"资源"和"配置"经常混着用。策划说"我更新了内容"，美术说"我提交了资源"，程序说"我改了配置"——但他们可能在说同一个文件，也可能在说完全不同的东西。

这种模糊在开发期没有太大问题。但当交付链路开始正式运转——需要定义质量门、需要自动化检查、需要追踪变更来源——模糊的概念就会制造混乱。

一个典型的场景：某次构建后功能异常，排查发现是一个配置表引用了一个不存在的资源 ID。问题是——这算内容问题、资源问题还是配置问题？该由策划修、美术修还是程序修？是导表流程的问题还是资源管线的问题？

如果三个概念的边界没有划清，这类问题每次都要靠人临时判断，无法系统性预防。

## 三个概念的定义

### 内容

内容是玩家在游戏中**直接感知到的素材**。

- 角色模型、场景模型、UI 贴图——玩家看到的视觉元素
- 动画片段、过场动画——玩家看到的动态表现
- 音效、配音、背景音乐——玩家听到的声音
- 关卡布局、剧情文本——玩家体验到的游戏设计

内容的产出者通常是美术和策划。内容的特征是**体积大、变更频繁、对最终体验有直接影响**。

### 资源

资源是内容经过**引擎处理后的运行时产物**。

一个美术在 Maya 里做的 FBX 模型是"内容"。这个 FBX 经过 Unity 的导入器处理，变成引擎内部的 Mesh、Material、Texture 对象后，就变成了"资源"。

内容和资源的关系：

```
内容（源文件）    →  导入 / 转换  →  资源（运行时产物）
FBX              →  Unity Importer →  Mesh + Material
PSD / TGA        →  TextureImporter → Texture2D (ASTC/ETC2)
WAV              →  AudioImporter   → AudioClip (Vorbis/ADPCM)
```

资源的特征是**格式由引擎决定、受平台约束、是构建产物的组成部分**。

同一份内容在不同平台上会产生不同的资源——同一张贴图在 iOS 上可能是 ASTC 6x6 格式，在 Android 上可能是 ETC2 格式，在微信小游戏上可能需要进一步压缩。

### 配置

配置是**驱动游戏逻辑和行为的结构化数据**。

- 技能参数：伤害值、冷却时间、射程范围
- 掉落表：什么怪掉什么物品、概率多少
- 商城定价：商品 ID、价格、上架时间
- 关卡配置：出生点、怪物波次、通关条件
- 环境配置：服务器地址、CDN 地址、Feature Flag

配置的产出者通常是策划（游戏逻辑配置）和程序（环境和系统配置）。配置的特征是**体积小、结构化、对游戏行为有直接控制力**。

配置出错的代价往往比内容出错更严重。一个贴图分辨率不对，最多画面模糊；一个掉落表概率填错，可能导致经济系统崩溃。

## 三者的边界

| 维度 | 内容 | 资源 | 配置 |
|------|------|------|------|
| **产出者** | 美术、策划 | 引擎导入器（自动） | 策划、程序 |
| **源格式** | FBX / PSD / WAV / 关卡编辑器 | 引擎内部格式 | Excel / JSON / ScriptableObject |
| **版本管理** | 通常在 Git LFS 或 Perforce | 通常不入版本库（构建时生成） | 在 Git 中管理 |
| **验证方法** | 规范检查（格式/大小/命名） | 导入设置检查 + 构建后校验 | 类型检查 + 引用完整性 + 范围校验 |
| **变更频率** | 高（跟随版本迭代） | 跟随内容变更 | 非常高（策划频繁调整） |
| **交付方式** | 跟随首包或热更新资源包 | 跟随首包或热更新资源包 | 可独立热推（配置热更新） |
| **出错代价** | 画面/音频异常 | 加载失败/格式错误 | 逻辑错误/经济系统失衡 |

## 边界模糊时的典型问题

### 问题一：配置引用了不存在的资源

策划在掉落表里填了一个物品 ID，对应的物品图标资源还没有提交。构建成功了（因为配置和资源分开打包），但运行时加载物品图标时报错。

**根因**：配置和资源的引用关系没有在提交时验证。

**解法**：配置验证阶段加入引用完整性检查——配置中引用的所有资源 ID 必须在资源清单中存在。

### 问题二：内容更新了但资源没有重新导入

美术更新了一个角色模型的 FBX，提交到版本库。但引擎的导入设置没有变，CI 的增量构建认为"导入设置没变所以不需要重新导入"，最终打出的包里还是旧的模型。

**根因**：内容变更没有正确触发资源重新生成。

**解法**：构建管线中加入内容变更检测——当源文件的哈希值变化时，强制重新导入对应资源。

### 问题三：环境配置和游戏配置混在一起

服务器地址、CDN 地址这类环境配置和技能参数、掉落表这类游戏配置放在同一套导表系统里。发版时切换环境（从测试服切到正式服）需要修改导表，修改导表的同时不小心碰到了策划刚更新的技能参数。

**根因**：两类配置的变更频率和变更来源不同，但管理方式相同。

**解法**：环境配置和游戏配置分开管理。环境配置走构建参数或环境变量，不走导表系统。
