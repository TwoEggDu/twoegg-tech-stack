## 这篇解决什么问题

代码可以回滚，数据不能。数据库是服务端最关键的有状态组件——丢数据 = 丢玩家。本篇讲数据库运维的五个核心环节：备份、Schema 迁移、分库分表、一致性和灾难恢复。

## 备份策略

### 三层备份

| 层级 | 方式 | 频率 | 保留时间 |
|------|------|------|---------|
| 自动全量备份 | 数据库全量快照 | 每日（低峰期） | 30 天 |
| 增量备份 / Binlog | 记录变更日志 | 持续 | 7 天 |
| 跨区域备份 | 复制到不同地域的存储 | 每日 | 7 天 |

### 时间点恢复（Point-in-Time Recovery）

全量备份 + Binlog 可以恢复到任意时间点。场景：玩家充值数据在 14:30 被 Bug 写坏了——恢复到 14:29 的状态。

### 备份验证

**备份不验证等于没有备份。** 常见事故：备份跑了半年，真出事时发现备份文件是损坏的。

验证方式：定期（每周/每月）将备份还原到测试环境，验证数据完整性。自动化这个流程——不要依赖人工执行。

## Schema 迁移

### 版本化迁移

Schema 变更必须是版本化的、可追踪的、可重放的。

```
migrations/
  V001__create_player_table.sql
  V002__add_player_level_column.sql
  V003__create_guild_table.sql
```

工具选择：

| 工具 | 生态 | 特点 |
|------|------|------|
| Flyway | Java / 通用 | 成熟，SQL-first |
| EF Migrations | .NET | 与 Entity Framework 集成 |
| Alembic | Python | SQLAlchemy 生态 |
| golang-migrate | Go | 轻量 |

### 只前进原则

迁移脚本只有"前进"版本，没有"回退"版本。原因：回退脚本难以保证正确性，特别是涉及数据变换的迁移。出问题时通过"前进修复"（写新的迁移脚本修复数据）解决。

### 部署时的 Schema 迁移

Schema 变更和代码部署通常在同一次发布中进行。关键原则——**Schema 变更必须向后兼容**：

| 操作 | 安全做法 | 危险做法 |
|------|---------|---------|
| 加列 | 加列并设默认值 | — |
| 改列名 | 新建列 → 双写 → 迁移数据 → 删旧列 | 直接 RENAME |
| 删列 | 先让代码不再读写该列 → 下一次发布再删 | 代码和 Schema 同时删 |
| 改类型 | 新建列 → 迁移数据 → 切换 → 删旧列 | 直接 ALTER TYPE |

**核心逻辑**：部署过程中新旧版本代码会共存。Schema 变更必须让新旧代码都能正常运行。

### 迁移测试

所有迁移脚本必须先在 Staging 环境跑过。用生产数据的脱敏副本来测试——空库上跑成功不代表生产库上也能成功（数据量、索引、约束都不同）。

## 分库分表

### 为什么要分

单库的容量和性能有上限。当单表数据量达到千万级、单库 QPS 达到瓶颈时，需要分库分表。

### 游戏数据库的常见分片方式

| 分片维度 | 适用场景 | 示例 |
|---------|---------|------|
| 按区服 | 区服隔离的游戏 | 1 服用 db_1，2 服用 db_2 |
| 按玩家 ID | 全区全服的游戏 | player_id % 16 决定数据在哪个分片 |
| 按时间 | 日志类数据 | 每月一张表 |

### 分片键选择

分片键的选择决定了查询效率：

| 查询场景 | 好的分片键 | 差的分片键 |
|---------|-----------|-----------|
| 查某玩家的数据 | player_id | server_id（需要跨分片查） |
| 查某区服的排行榜 | server_id | player_id（需要跨分片聚合） |
| 查全服排行榜 | — | 任何键都需要跨分片聚合 |

没有完美的分片键——选择时要优先满足最频繁的查询模式。

### 跨分片查询

跨分片查询的性能远差于单分片查询。应对策略：

- 尽量避免跨分片查询（通过分片键设计）
- 对于必须跨分片的查询（如全服排行榜），使用异步聚合 + 缓存
- 将跨分片查询的结果定时预计算到单独的聚合表

## 数据一致性

### 游戏后端的一致性挑战

游戏后端通常是分布式架构——多个服务操作不同的数据。经典问题：玩家购买道具时，扣钱和发货是两个操作。扣钱成功但发货失败怎么办？

### 事务边界

| 策略 | 适用 | 代价 |
|------|------|------|
| 本地事务 | 单库操作 | 简单，但限于单库 |
| 分布式事务（2PC） | 跨库操作 | 性能差，实现复杂 |
| 最终一致性 + 补偿 | 跨服务操作 | 需要补偿机制，短暂不一致 |

游戏后端最常用的是**最终一致性 + 补偿**：先执行操作，如果部分失败则通过补偿机制修复。例如扣钱成功但发货失败——定时任务扫描"扣钱但未发货"的记录并重试发货。

### 幂等性

补偿机制的前提是操作必须是幂等的——重试不会产生副作用。例如"发放道具"操作必须通过唯一的交易 ID 防止重复发货。

## 灾难恢复

### RPO 和 RTO

| 指标 | 含义 | 游戏服务端参考值 |
|------|------|----------------|
| RPO（恢复点目标） | 最多能接受丢多少数据 | < 5 分钟（增量备份间隔） |
| RTO（恢复时间目标） | 最多能接受停多久 | < 30 分钟 |

RPO 决定备份频率，RTO 决定恢复方案的选择。

### 恢复方案

| 场景 | 恢复方式 | 预计时间 |
|------|---------|---------|
| 单表数据损坏 | 从备份恢复该表 + Binlog 重放 | 分钟级 |
| 整库不可用 | 切换到从库（主从复制） | 秒级（如果有从库） |
| 机房级故障 | 切换到异地灾备 | 分钟到小时级 |

### 灾难恢复演练

和备份验证一样——**不演练的灾备方案等于没有方案**。建议每季度做一次灾难恢复演练，验证 RTO 是否达标。

## 小结

- [ ] 是否有自动化的每日全量备份 + 持续增量备份
- [ ] 备份是否定期验证（还原到测试环境）
- [ ] Schema 迁移是否版本化且向后兼容
- [ ] 分片键是否按最频繁的查询模式选择
- [ ] 跨服务操作是否有幂等性和补偿机制
- [ ] RPO 和 RTO 是否有明确目标且定期演练验证
