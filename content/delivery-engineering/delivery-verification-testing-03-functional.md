## 这篇解决什么问题

功能验证是四层验证模型的第一层，也是最基础的一层。但"功能验证"不是一个笼统的概念——它至少包含冒烟测试、功能测试、回归测试三类，加上自动化策略和多平台差异处理。本篇逐一拆解。

## 功能验证的层次结构

| 层次 | 名称 | 目的 | 执行时机 | 执行时长 |
|------|------|------|---------|---------|
| L1 | 冒烟测试 | 验证最小核心路径可用 | 每次构建 | 10-15 分钟 |
| L2 | 功能测试 | 验证新功能和修改功能 | 功能提交后 | 1-2 小时 |
| L3 | 回归测试 | 验证已有功能未被破坏 | 发版前 | 4-8 小时 |
| L4 | 探索性测试 | 发现用例未覆盖的问题 | 发版前 | 人工，时间不固定 |

四层不互相替代。冒烟测试通过只说明"能跑起来"，不代表功能正确。

## 冒烟测试

冒烟测试是最小的功能验证集合——如果冒烟测试不通过，后续一切验证都不需要跑。

### 冒烟测试的设计原则

| 原则 | 说明 |
|------|------|
| 最小关键路径 | 只覆盖核心功能的主干链路 |
| 快速执行 | 控制在 10-15 分钟内完成 |
| 高可靠性 | 误报率必须极低，否则团队会忽视 |
| 全自动化 | 人工冒烟测试不可接受——太慢且不稳定 |

### 游戏项目的冒烟测试清单示例

- [ ] 客户端启动不崩溃
- [ ] 登录流程正常完成
- [ ] 主界面加载成功，核心 UI 元素可见
- [ ] 进入核心玩法场景不崩溃
- [ ] 基础操作响应正常（移动、攻击、交互）
- [ ] 与服务端通信正常（至少一个请求-响应验证）

冒烟测试失败 = 这个构建不可用，立即通知构建负责人。

## 功能测试

功能测试覆盖新增功能和修改功能，验证它们是否符合需求规格。

| 类型 | 覆盖范围 | 用例来源 |
|------|---------|---------|
| 新功能测试 | 本次迭代新增的功能 | 需求文档 → 测试用例 |
| 修改功能测试 | 本次迭代修改的功能 | 变更记录 → 影响分析 → 测试用例 |

功能测试的关键不是写多少用例，而是覆盖等价类和边界条件。

## 回归测试

回归测试验证"已有功能没有被新代码破坏"。回归测试的难点不是怎么测，而是测什么——全量回归太慢，选择性回归怕漏。

### 回归范围的确定

回归范围由影响分析决定（参考 V04-05 的变更追踪和影响分析）：

| 变更类型 | 回归范围 | 说明 |
|---------|---------|------|
| UI 层修改 | 相关界面 + 导航链路 | 界面变更通常影响范围有限 |
| 业务逻辑修改 | 相关功能模块 + 上下游 | 需要追踪调用链 |
| 数据层修改 | 所有使用该数据的功能 | 数据变更影响范围广 |
| 底层框架修改 | 全量回归 | 基础设施变更影响全局 |
| 配置/资源修改 | 使用该配置/资源的功能 | 通常可以精确定位 |

### 回归测试的优先级

回归用例太多时按优先级排列：

1. **P0 回归**：核心付费链路、登录注册、数据持久化——每次必跑
2. **P1 回归**：主要玩法功能、社交功能——功能迭代时跑
3. **P2 回归**：辅助功能、设置页面——大版本时跑
4. **P3 回归**：边缘场景、低频功能——全量回归时跑

## 自动化策略

不是所有测试都适合自动化。自动化的投入产出比取决于执行频率和稳定性。

| 适合自动化 | 不适合自动化 |
|-----------|------------|
| 冒烟测试（每次构建跑） | 探索性测试（需要人的直觉） |
| 核心回归测试（频繁执行） | UX 体验测试（需要主观判断） |
| API 接口测试（输入输出明确） | 视觉还原度测试（像素级对比成本高） |
| 数据校验测试（规则明确） | 首次功能测试（功能不稳定时自动化浪费） |

### 自动化的优先级

1. 冒烟测试——必须自动化
2. P0 回归测试——优先自动化
3. API / 协议测试——性价比高
4. P1 回归测试——逐步自动化
5. 其余——根据频率和稳定性决定

## Unity 项目的测试框架

| 框架 | 适用场景 | 特点 |
|------|---------|------|
| Unity Test Framework (Edit Mode) | 纯逻辑测试 | 不需要运行时环境，速度快 |
| Unity Test Framework (Play Mode) | 需要运行时的测试 | 启动 Player，可以测试 MonoBehaviour |
| 自定义自动化框架 | 冒烟测试、功能回归 | 录制回放或脚本驱动 UI 操作 |

Play Mode 测试可以在 CI 中通过 Unity 命令行运行，配合构建流水线实现自动化回归。

## 多平台功能差异

同一功能在不同平台可能表现不同，功能验证需要覆盖平台差异点：

| 差异维度 | 示例 | 验证方式 |
|---------|------|---------|
| 输入方式 | 触屏 vs 手柄 vs 键鼠 | 各平台分别验证操作响应 |
| 支付流程 | App Store / Google Play / 微信支付 | 各渠道支付链路独立验证 |
| 推送通知 | APNs / FCM / 厂商推送 | 各平台推送到达率验证 |
| 分享功能 | 系统分享 / 微信 / QQ | 各社交平台分享和回调验证 |
| 文件系统 | 沙盒差异、权限模型差异 | 存档读写在各平台验证 |

## 小结

- [ ] 是否区分了冒烟、功能、回归、探索性四类功能验证
- [ ] 冒烟测试是否控制在 15 分钟内且全自动化
- [ ] 回归范围是否基于影响分析确定
- [ ] 回归用例是否有优先级分级
- [ ] 自动化策略是否按投入产出比排列优先级
- [ ] 多平台功能差异是否有专门的验证覆盖
