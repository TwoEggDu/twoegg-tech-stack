## 这篇解决什么问题

V05-06 讲了 Catalog/Manifest 版本比对的原理。V06-06 讲了 CDN 部署。这一篇把两者串成完整的热更新资源管线：从构建产物到用户设备上资源替换生效的端到端流程。

## 完整的热更新资源流程

```
构建端                         CDN                          客户端
──────                        ────                         ──────
1. 构建新版 Bundle             4. 部署 Bundle                7. 启动时拉取远程 Manifest
2. 生成新版 Manifest           5. 验证部署完成               8. 与本地 Manifest 比对
3. 计算增量包（可选）           6. 部署 Manifest              9. 下载差异文件
                                                           10. 校验完整性
                                                           11. 原子替换
                                                           12. 更新本地 Manifest
                                                           13. 生效（重新加载或重启）
```

### 构建端：增量包的生成

**全量更新**：客户端下载所有变更的 Bundle。简单但下载量大。

**增量更新（补丁包）**：只下载 Bundle 内变更的部分。

增量更新的两种粒度：

| 粒度 | 做法 | 下载量 | 复杂度 |
|------|------|--------|--------|
| **Bundle 级** | 变更的 Bundle 整个重新下载 | 较大 | 低 |
| **文件级 diff** | 对每个变更的 Bundle 计算二进制 diff（如 bsdiff） | 最小 | 高 |

**推荐**：大多数项目用 Bundle 级增量就够了。文件级 diff 只在包体极度敏感（微信小游戏）或用户网络极差的场景下值得投入。

### 客户端：热更新的生命周期

客户端热更新的完整生命周期：

**检查阶段**：
```
1. 启动时请求远程 Manifest（带 no-cache header）
2. 比对远程和本地 Manifest
3. 计算需要下载的 Bundle 列表和总大小
4. 如果无更新 → 直接进入游戏
5. 如果有更新 → 检查设备剩余空间 → 进入下载阶段
```

**下载阶段**：
```
6. 显示下载进度（已下载 / 总大小）
7. 下载到临时目录（不覆盖当前在用的文件）
8. 支持断点续传（记录每个文件的下载偏移）
9. 并行下载（2-4 个连接，移动网络下限制为 2 个）
10. 单文件下载失败 → 重试 3 次 → 仍失败则暂停并提示用户
```

**校验阶段**：
```
11. 对每个下载的文件计算哈希
12. 与 Manifest 中的哈希比对
13. 不一致 → 删除该文件 → 标记为需要重新下载
14. 全部一致 → 进入替换阶段
```

**替换阶段**：
```
15. 将临时目录的文件移动到正式目录（原子操作）
16. 更新本地 Manifest 为新版本
17. 清理临时目录
18. 标记热更新完成
```

**生效阶段**：
```
19. 方案 A：重启游戏（最安全，确保所有资源从新版本加载）
19. 方案 B：运行时热替换（不重启，但需要处理已加载资源的刷新）
```

### 回退机制

热更新的每一步都可能失败。回退机制确保失败时用户仍然可以正常游戏：

| 失败点 | 回退行为 |
|--------|---------|
| 远程 Manifest 拉取失败 | 使用本地 Manifest 继续（上一个版本的资源） |
| 下载中断 | 保留已下载的文件，下次启动继续（断点续传） |
| 校验失败 | 删除临时目录的所有文件，使用旧版本资源 |
| 替换失败 | 临时目录不动，旧版本仍然完整可用 |
| 新版本启动崩溃 | 下次启动检测到上次崩溃，回退到旧版本 Manifest |

**最后一条是最关键的**：如果新版本的热更新包导致了崩溃，用户下次启动时应该能自动回退到旧版本。

实现方式：
```
启动时检查：
  if (上次启动是否正常退出 == false && 上次有热更新) {
    // 上次崩溃且有热更新 → 可能是热更新导致的
    回退到首包内置的 Manifest
    清除热更新缓存
    标记为"已回退"
    上报回退事件（用于监控）
  }
```

### 热更新与首包的兼容性

热更新的核心约束：**新版热更包必须和旧版首包兼容。**

不兼容的典型情况：
- 热更包的代码引用了新版首包才有的 API（旧首包用户崩溃）
- 热更包的 Bundle 依赖了新版首包才有的公共 Bundle
- 热更包的序列化格式变了，旧版 runtime 无法反序列化

**兼容性矩阵**：
```
首包 v1.0 + 热更 v1.0.1 → 必须兼容
首包 v1.0 + 热更 v1.0.2 → 必须兼容
首包 v1.0 + 热更 v1.1.0 → 可能不兼容（Minor 版本变更可能有 API 变化）
```

当 Minor 版本变更导致首包和热更不兼容时，需要强制用户到商店更新首包。

### 热更新的验证策略

热更新发布前的验证：

| 验证项 | 方法 |
|--------|------|
| 增量正确性 | 用旧版首包 + 新版热更包启动，验证功能正常 |
| 多版本兼容 | 用最近 3 个版本的首包分别测试新热更包 |
| 回退验证 | 模拟热更后崩溃，验证自动回退机制 |
| 断网恢复 | 下载到一半断网，恢复后验证断点续传 |
| 空间不足 | 设备空间不足时验证提示和处理逻辑 |

## 常见事故与排障

**事故：热更后旧用户崩溃**。热更包引用了 v1.1 首包才有的 Shader 变体。v1.0 首包的用户下载热更包后，加载该 Shader 时找不到对应变体，渲染异常。

**根因**：热更包的构建环境和首包不一致。首包用 Unity v1.0 构建，热更包用 Unity v1.1 构建——两个版本的 Shader 编译产出不同。

**预防**：热更包必须用和首包相同的 Unity 版本和构建配置构建。CI 中锁定热更构建环境。

**事故：热更后内存增长最终 OOM**。热更包更新了部分 Bundle，但旧 Bundle 和新 Bundle 的公共资源引用不兼容。运行时同时加载了新旧两份公共资源，内存翻倍。

**根因**：公共 Bundle 的版本被热更新跳过了。新 Bundle 引用的公共资源 ID 和旧公共 Bundle 里的不一致，引擎无法共享，各自加载一份。

**预防**：热更新包必须包含所有依赖变化的 Bundle——不只是直接修改的 Bundle，还包括被间接影响的公共 Bundle。

## 小结与检查清单

- [ ] 热更新是否下载到临时目录（不直接覆盖正式文件）
- [ ] 文件替换是否是原子操作
- [ ] 是否支持断点续传
- [ ] 下载文件是否逐个校验哈希
- [ ] 校验失败是否有重新下载和回退机制
- [ ] 新版热更崩溃时是否能自动回退到旧版本
- [ ] 热更包是否和最近 3 个版本的首包兼容
- [ ] 热更包的构建环境是否和首包一致
- [ ] 热更发布前是否经过多版本兼容测试
