## 这篇解决什么问题

项目结构是交付链路最底层的基础设施。它不会直接导致崩溃或性能问题，但会间接决定：编译要多久、打包怎么分组、多人协作冲突多不多、新人进来能不能快速上手。

## 为什么这个问题重要

项目结构的问题是温水煮青蛙——开始时没感觉，等到痛的时候已经改不动了。

常见的痛点：

- 编译时间从 10 秒涨到 3 分钟——因为所有代码在同一个编译域，改一行触发全量编译
- 两个程序同时改了同一个文件夹下的不同文件，合入时产生目录级冲突
- 新人看着几百个平铺的文件夹不知道哪些是核心代码、哪些是第三方、哪些是临时实验
- AssetBundle 打包时发现依赖关系混乱——A 文件夹的资源引用了 B 文件夹的资源，但 A 和 B 不在同一个 Bundle 里

这些问题的根因都一样：**项目结构没有按照编译边界、打包边界和职责边界来设计。**

## 本质是什么

项目结构设计的本质是回答三个问题：

1. **什么和什么放在一起**（分组原则）
2. **什么和什么不能互相依赖**（依赖规则）
3. **什么能变、什么不能变**（稳定性分层）

### 分组原则：按功能模块还是按文件类型

两种经典的分组方式：

**按文件类型**（不推荐作为顶层结构）：
```
Assets/
├── Scripts/
├── Textures/
├── Models/
├── Audio/
├── Prefabs/
└── Scenes/
```

问题：角色系统的代码在 Scripts/，贴图在 Textures/，Prefab 在 Prefabs/——修改一个角色需要跨三个目录操作。多人协作时，所有人都在 Scripts/ 目录下工作，冲突高发。

**按功能模块**（推荐作为顶层结构）：
```
Assets/
├── Core/                  # 框架层：生命周期、事件、工具类
├── Modules/
│   ├── Character/         # 角色系统：代码 + 资源 + 配置
│   ├── Combat/            # 战斗系统
│   ├── UI/                # UI 系统
│   ├── VFX/               # 特效系统
│   └── Audio/             # 音频系统
├── ThirdParty/            # 第三方插件和 SDK
├── Config/                # 配置数据
├── Art/                   # 美术源资源（大文件）
└── Editor/                # 编辑器扩展
```

优点：每个模块的代码、资源、配置放在一起，修改一个模块的影响范围可预期。模块之间的依赖关系清晰——如果 Combat 引用了 Character 的代码，这个依赖在目录结构上是可见的。

### 依赖规则：分层和单向依赖

项目结构不只是文件夹的划分，还包括文件夹之间的依赖方向。

**分层模型**：

```
应用层（游戏逻辑、UI、关卡）         ← 最不稳定，变化最频繁
     ↓ 依赖
模块层（角色、战斗、背包、任务）       ← 按功能拆分的独立模块
     ↓ 依赖
框架层（事件系统、对象池、资源管理）    ← 基础能力，变化少
     ↓ 依赖
基础层（数学库、工具类、平台抽象）      ← 最稳定，几乎不变
```

**依赖规则**：
- 上层可以依赖下层，下层不能依赖上层
- 同层之间尽量不互相依赖；如果必须依赖，通过接口或事件解耦
- 第三方库只被框架层或基础层封装后暴露，应用层不直接依赖第三方 API

违反依赖规则的后果：
- 基础层改了一行，应用层全部受影响（依赖方向反转）
- 两个模块互相依赖，无法单独编译或单独测试（循环依赖）
- 替换一个第三方 SDK 需要改几十个文件（直接依赖泄漏）

### 稳定性分层：高频变更和低频变更分开

项目中不同区域的变更频率差异很大：

| 区域 | 变更频率 | 影响范围 |
|------|---------|---------|
| 游戏逻辑 / UI | 每天多次 | 局部 |
| 模块代码 | 每周几次 | 模块内 |
| 框架层 | 每月几次 | 全局 |
| 第三方 SDK | 每版本 0-1 次 | 全局 |
| 引擎版本 | 每大版本 0-1 次 | 全局 |

高频变更区域和低频变更区域应该在目录结构上隔离。这样：
- 编译时，高频区域的改动不触发低频区域的重新编译
- 打包时，高频区域的资源可以独立热更新，不拖动低频区域
- 协作时，高频区域的冲突不影响低频区域的稳定性

## 项目结构与交付链路的关系

项目结构的设计直接影响交付链路的三个环节：

**编译段**。目录结构和编译域（asmdef）的划分一一对应。结构混乱 = 编译域混乱 = 编译时间长 = 迭代慢。

**资源管线**。目录结构影响 AssetBundle 的打包分组。模块化的目录结构让 Bundle 的依赖关系清晰；平铺的结构让 Bundle 之间的依赖变成一团乱麻。

**多人协作**。目录结构影响代码冲突的频率和 CODEOWNERS 的划分。模块化的结构让每个团队/个人负责自己的目录；平铺的结构让所有人都在同一个目录下竞争。

## 常见错误做法

**项目初期不设计结构，等"大了再整理"**。等到项目有几百个文件后再调整目录结构，意味着修改所有的资源引用路径、GUID 映射和 AssetBundle 配置。成本远大于一开始就设计好。

**照搬其他项目的结构**。项目结构应该从自己项目的模块划分和团队分工推导出来，不应该照搬模板。照搬的结构往往和实际的开发节奏不匹配。

**美术资源和代码混在同一层级**。美术资源的变更频率高、文件体积大、版本管理方式不同（LFS）。和代码混在一起会让 Git 操作变慢、冲突排查困难。美术资源应该在顶层目录就和代码分开。

## 小结与检查清单

- [ ] 顶层目录是否按功能模块组织（而非按文件类型）
- [ ] 依赖方向是否单向（上层依赖下层，不反过来）
- [ ] 第三方库是否封装后暴露（应用层不直接调用第三方 API）
- [ ] 高频变更区域和低频变更区域是否在目录结构上隔离
- [ ] 美术大文件是否和代码分开管理
- [ ] 目录结构是否与编译域（asmdef）划分对齐
- [ ] 目录结构是否能让新人在 5 分钟内理解"核心代码在哪、第三方在哪、配置在哪"
