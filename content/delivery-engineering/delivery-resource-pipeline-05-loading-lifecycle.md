## 这篇解决什么问题

资源打好包后，运行时需要把它们从磁盘加载到内存、实例化为引擎对象、在使用完毕后释放。这条链路的工程设计决定了加载速度和内存稳定性。

## 本质是什么

资源在运行时有一个完整的生命周期：

```
请求加载 → 磁盘读取 → 反序列化 → 注册到引擎 → 被引用使用 → 引用释放 → 卸载 → 内存回收
```

这条链路中有三个关键的工程决策点：

### 决策一：同步还是异步

| 方式 | 行为 | 优点 | 缺点 |
|------|------|------|------|
| **同步** | 调用时阻塞主线程直到加载完成 | 代码简单、使用即得 | 阻塞主线程导致卡顿 |
| **异步** | 调用后立即返回，加载完成后回调 | 不阻塞主线程 | 需要管理加载状态和回调 |

**原则**：
- 启动阶段的关键资源可以同步加载（反正用户在等加载画面）
- 游戏运行中的资源必须异步加载（否则帧率波动）
- 预加载场景可以在场景切换的 loading 画面中用异步批量加载

**不要做的事**：在 Update 里同步加载资源。一个同步加载耗时 50ms，帧率直接从 60fps 掉到 20fps。

### 决策二：加载时机策略

| 策略 | 做法 | 适用场景 |
|------|------|---------|
| **启动全量加载** | 启动时一次性加载所有资源 | 小体量游戏、资源总量 <200MB |
| **场景预加载** | 进入场景前加载该场景需要的资源 | 有明确场景切换点的游戏 |
| **按需加载** | 用到时才加载 | 开放世界、资源量极大的游戏 |
| **分层加载** | 核心资源预加载 + 非核心按需加载 | 大多数商业游戏 |

**分层加载**是最常用的策略：

```
第一层：启动时加载（UI 框架、基础 Shader、配置数据）→ 必须同步完成才能进入
第二层：场景预加载（场景模型、角色模型、地形）→ 在 loading 画面中异步完成
第三层：运行时按需（特效、音效、非关键 UI）→ 进入场景后异步加载
第四层：后台预取（下一个场景的资源、可能用到的角色）→ 空闲时后台加载
```

### 决策三：引用计数与释放

资源加载到内存后，可能被多个对象引用。引用计数确保只有在所有引用者都释放后，资源才被卸载。

```
角色 A 引用了 texture_shared → 计数 +1 → 计数 = 1
角色 B 也引用了 texture_shared → 计数 +1 → 计数 = 2
角色 A 被销毁，释放引用      → 计数 -1 → 计数 = 1
角色 B 被销毁，释放引用      → 计数 -1 → 计数 = 0 → 可以安全卸载
```

引用计数的工程要点：

**配对原则**。每一次 Load 必须有对应的 Release。这是最基本的要求，也是最容易出错的地方。如果 Load 了 10 次但只 Release 了 9 次，资源永远不会被卸载。

**所有权明确**。谁 Load 了资源，谁负责 Release。不要出现"A 加载的资源由 B 来释放"的情况——一旦 B 的释放逻辑变更，A 加载的资源就泄漏了。

**自动化 Release**。尽可能用 RAII 模式或 Handle 封装，让引用释放跟随对象销毁自动发生，而不是依赖手动调用。

## 泄漏检测

资源泄漏是内存问题的第一大来源。泄漏表现为：资源被加载后从未释放，或引用计数不归零。

### 泄漏的典型模式

**模式一：Load 无 Release**。代码中加载了资源但没有在任何路径上调用 Release。通常因为异常退出路径没有处理释放。

**模式二：循环引用**。A 引用 B、B 引用 A，两者的引用计数永远不归零。

**模式三：缓存失控**。资源被加入缓存后不再主动清除。缓存无上限，内存持续增长。

**模式四：回调泄漏**。异步加载的回调持有了对资源的引用，但回调注册后没有反注册。

### 泄漏检测手段

**运行时检测**：
- 定期扫描所有已加载资源的引用计数，报告计数 > 0 但无活跃引用者的资源
- 在场景切换后检查是否有上一个场景的资源未被释放
- 长时间运行测试（8-24 小时），监控内存曲线是否持续上升

**构建时检测**：
- 静态分析代码中所有 Load 调用，检查是否有对应的 Release 调用
- 分析异常路径（try-catch、early return）是否遗漏 Release

**工具辅助**：
- Unity Profiler 的 Memory 模块可以查看所有加载的资源和引用计数
- 自定义的资源管理器应该提供 Dump 功能——输出当前所有已加载资源的列表和引用计数

## 预加载与 Shader 预热

两个常被忽视的加载优化：

### 预加载（Preloading）

在场景切换的 loading 画面中提前加载下一个场景需要的资源。目标是让用户进入新场景后不再等待加载。

预加载的优先级排序：
1. 场景基础几何体和地形（没有它们场景无法显示）
2. 主角和常驻 NPC 的模型（第一眼就能看到的东西）
3. UI 框架和 HUD（交互必需）
4. 其他资源按需异步加载

### Shader 预热（Shader Warmup）

Shader 在首次使用时需要编译和缓存。如果不预热，第一次渲染使用该 Shader 的物体时会出现明显的卡顿。

预热策略：
- 在 loading 画面中遍历所有可能使用的 Shader 变体，强制编译
- 使用 ShaderVariantCollection 收集需要预热的变体列表
- 热更新新增 Shader 后，下次启动时补充预热

Shader 预热的详细工程化设计在本站渲染管线栏的 Shader Variant 系列中展开。

## 常见事故与排障

**事故：场景切换后内存不降反升**。切换场景后，旧场景的资源没有被正确卸载。排查路径：用 Profiler 的 Memory 快照对比两个场景的资源列表，找到只在旧场景存在但未被卸载的资源。

**事故：首次进入战斗卡顿 2 秒**。原因可能是：战斗场景的资源没有预加载，或 Shader 没有预热。排查路径：用 Profiler 的 CPU 模块看首帧的加载耗时分布。

**事故：长时间运行后 OOM 崩溃**。内存曲线持续上升不回落。排查路径：每 10 分钟做一次 Memory 快照，对比增量——找到持续增长的资源类型（通常是贴图或音频缓存）。

## 小结与检查清单

- [ ] 游戏运行中是否全部使用异步加载（无 Update 中的同步加载）
- [ ] 加载时机是否有分层策略（预加载 + 按需 + 后台预取）
- [ ] 每一次 Load 是否有配对的 Release
- [ ] 资源引用所有权是否明确（谁加载谁释放）
- [ ] 场景切换后是否有内存检查（旧资源是否全部释放）
- [ ] 是否有 Shader 预热机制
- [ ] 是否有长时间运行测试检测内存泄漏
