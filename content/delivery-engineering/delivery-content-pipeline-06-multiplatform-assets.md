## 这篇解决什么问题

前五篇讲了资源线和配置线的生产管线。但有一个问题在单端开发时不存在，多端交付时会突然暴露：**同一份源资产在不同平台上需要不同的运行时格式和质量参数。**

这一篇讲清楚差异在哪、谁来管、管理策略怎么设计。

## 为什么这个问题重要

多端资源差异管理出问题的典型场景：

- Android 包使用了 iOS 的纹理压缩格式（PVRTC），在 Android 设备上解压开销翻倍
- 微信小游戏的资源没有做额外的体积压缩，主包超出 4MB 限制，提审被驳回
- 低端设备分档使用了高端的贴图分辨率，内存超预算闪退
- 三端资源分开维护，更新时漏掉了某一端，导致内容不一致

这些问题的根因相同：**资源差异管理没有从生产段设计进去，而是在构建段才临时处理。**

## 本质是什么

多端资源差异管理的本质是：**一份源资产 → 多份运行时产物**，由导入设置和构建配置决定产物形态。

```
源资产（一份）
    ↓
导入设置 × 平台 × 质量档位
    ↓
运行时产物（多份）
```

管理的对象不是"三套不同的资源"，而是"一套源资产 + 一组平台差异规则"。

## 三类核心差异

### 纹理差异

纹理是多端差异最大的资源类型：

| 差异维度 | iOS | Android | 微信小游戏 |
|---------|-----|---------|-----------|
| 主流压缩格式 | ASTC | ASTC 或 ETC2 | ASTC（WebGL 2.0+） |
| 低端回退格式 | ASTC 8x8 | ETC2（不支持 ASTC 的旧设备） | 降分辨率 |
| 最大分辨率 | 受内存预算约束 | 受设备分档约束 | 严格受限（内存极小） |
| Alpha 通道 | ASTC 原生支持 | ETC2 对 Alpha 压缩质量较差 | 同 ASTC |
| 法线贴图精度 | ASTC 4x4 | ASTC 4x4 或 ETC2（精度损失） | ASTC 4x4 |

管理策略：

**不维护三套贴图**。美术只产出一份高质量源贴图。平台差异通过引擎的导入设置（TextureImporter 的平台覆盖）自动处理。

**导入设置模板化**。按贴图用途（漫反射 / 法线 / UI / 特效 / 光照贴图）定义导入模板，模板中包含每个平台的压缩格式和最大尺寸。新贴图导入时自动匹配模板。

**设备分档叠加**。在平台差异之上，还有设备分档差异——高端设备使用 2048 分辨率，中端使用 1024，低端使用 512。分档规则与平台规则正交，需要通过 Quality Settings 或自定义配置管理。

### 模型与动画差异

模型的多端差异相对较小，但在极端情况下仍然需要管理：

| 差异维度 | 高端设备 | 中端设备 | 低端设备 / 微信小游戏 |
|---------|---------|---------|-------------------|
| 面数 | 完整模型 | LOD1 | LOD2 或简化模型 |
| 骨骼数 | 完整骨骼 | 减少末端骨骼 | 大幅简化 |
| 动画精度 | 完整曲线 | 压缩曲线 | 进一步压缩 |

管理策略：

**LOD 系统**。模型差异通过 LOD（Level of Detail）系统管理，不需要维护多套模型。LOD 的分级和切换距离按设备分档配置。

**动画压缩参数化**。动画的压缩率通过导入设置控制。高端设备保留更多关键帧精度，低端设备压缩更激进。

### 音频差异

音频的多端差异主要体现在编码格式和质量上：

| 差异维度 | iOS | Android | 微信小游戏 |
|---------|-----|---------|-----------|
| 推荐编码 | AAC / Vorbis | Vorbis | Vorbis / MP3 |
| 采样率 | 44100Hz | 44100Hz | 22050Hz（节省体积） |
| 声道 | 立体声 | 立体声 | 单声道（节省体积） |
| 压缩质量 | 高 | 中-高 | 中-低 |

微信小游戏的音频差异最大——由于包体和内存限制，通常需要降采样率、改单声道、提高压缩率。

## 差异管理的三种策略

### 策略一：导入时分平台配置

最常见的方式。在引擎的资源导入设置中，按平台配置不同的参数。

**优点**：一份源资产，构建时自动产出不同平台的运行时资源。管理简单，美术无感知。

**缺点**：所有差异都必须能通过导入参数表达。如果某个平台需要完全不同的资源（比如微信小游戏需要特制的低面数模型），导入参数无法覆盖。

**适用场景**：纹理压缩格式、分辨率限制、音频编码格式——这些差异可以通过参数化控制。

### 策略二：资源变体

为同一个逻辑资源准备多个物理文件，构建时根据目标平台选择对应的变体。

```
Assets/Characters/Hero/
    hero_body_d.tga         (通用源贴图)
    hero_body_d_mini.tga    (微信小游戏专用低分辨率版本)
```

**优点**：可以处理导入参数无法覆盖的差异（不同平台使用完全不同的资源文件）。

**缺点**：增加了资源管理的复杂度。美术需要维护多份资源，容易遗漏更新。

**适用场景**：微信小游戏需要特制资源、PC 端需要高清贴图而移动端使用标准贴图。

### 策略三：运行时动态加载

不在构建时区分，而是在运行时根据设备能力动态选择资源。

```
// 伪代码
if (DeviceTier == High)
    LoadTexture("hero_2k");
else if (DeviceTier == Medium)
    LoadTexture("hero_1k");
else
    LoadTexture("hero_512");
```

**优点**：一个包适配所有设备。可以通过远程配置动态调整。

**缺点**：所有档位的资源都要打进包里（除非配合按需下载），包体增大。运行时加载逻辑更复杂。

**适用场景**：设备分档差异（同一平台内的高中低端设备），配合 Addressables 或按需下载使用。

### 实际项目中的组合

大多数项目会组合使用三种策略：

- **平台差异**（iOS vs Android vs 微信小游戏）→ 策略一（导入时分平台配置）
- **极端差异**（微信小游戏专用资源）→ 策略二（资源变体）
- **设备分档差异**（同平台高中低端）→ 策略三（运行时动态加载）

关键是：**差异规则在生产段定义，不在构建段临时处理。**

## 微信小游戏的特殊性

微信小游戏的资源差异不只是"格式不同"，而是整体资源策略都需要重新设计：

**主包 4MB 限制**。微信小游戏的主包只有 4MB（有些类别是 8MB 或 12MB）。绝大多数资源必须放在分包或远程 CDN 上，不能打进主包。

**内存极度受限**。小游戏运行在微信的 V8 / JSCore 环境中，可用内存远小于原生 App。资源的总量和峰值都需要大幅压缩。

**WebGL 渲染约束**。部分高级渲染特性在 WebGL 上不可用或性能极差。Shader 需要有 WebGL 兼容版本。

这意味着微信小游戏不能简单地"用同一套资源换个格式"。它可能需要：
- 专门的贴图分辨率上限（比原生端低一档）
- 专门的模型面数预算
- 专门的 Shader 变体（去掉 WebGL 不支持的特性）
- 专门的加载策略（按场景按需下载，不预加载）

在本专栏的 V06（包体管理与分发）和 V07（多端构建）中会详细展开微信小游戏的工程化处理。

## 小结与检查清单

- [ ] 三端资源差异是否通过导入设置统一管理（而非手动维护三套资源）
- [ ] 导入设置模板是否按资源用途分类定义
- [ ] 微信小游戏是否有独立的资源预算（分辨率、面数、内存）
- [ ] 设备分档差异是否与平台差异分开管理
- [ ] 变体资源（如有）是否有自动化的同步检查（更新源资产时变体是否同步更新）
- [ ] 三端的资源质量是否在 CI 中分别验证
