## 这篇解决什么问题

同一个构建产物在不同阶段需要连接不同的后端环境（开发、测试、预发布、生产）。同时，某些功能需要在运行时动态开关（灰度、A/B 测试、紧急关闭）。

环境配置和 Feature Flag 是两种不同的控制机制，但在很多项目中被混在一起管理——这是事故的高发区。

## 为什么这个问题重要

- 发布包连接了测试环境的服务器地址，用户登录后看到测试数据
- Feature Flag 的状态在预发布环境和生产环境不同步，预发布验证通过的功能在生产环境上表现不同
- 紧急关闭某个功能后，下次启动时 Flag 被本地缓存覆盖，关闭失效
- 开发期的 GM 指令开关和 Feature Flag 混在一套系统里，发布时忘记关闭 GM 功能

## 本质是什么

### 环境配置：决定基础设施指向

环境配置解决的是"这个构建产物连接哪套基础设施"：

| 环境 | 服务器 | CDN | 数据库 | 用途 |
|------|--------|-----|--------|------|
| **开发（Dev）** | dev.api.game.com | dev-cdn | dev-db | 日常开发，数据可随时重置 |
| **测试（QA）** | qa.api.game.com | qa-cdn | qa-db | QA 验证，数据相对稳定 |
| **预发布（Staging）** | staging.api.game.com | prod-cdn | staging-db | 发布前最终验证，使用生产 CDN |
| **生产（Production）** | api.game.com | prod-cdn | prod-db | 用户环境 |

环境配置的管理原则：

**通过构建参数注入，不硬编码在代码里。** 代码中不应该出现 `if (env == "production")` 来决定服务器地址。服务器地址应该作为构建参数在构建时注入，或通过环境配置文件在启动时读取。

**环境配置和游戏配置分开管理。** 服务器地址、CDN 地址、日志级别等环境配置走构建参数或环境变量，不走策划的导表系统。两者的变更频率、变更来源和审批流程完全不同。

**预发布环境尽量接近生产环境。** 预发布的 CDN 应该使用生产 CDN（或其镜像），而不是独立的测试 CDN。这样可以提前发现 CDN 配置问题。

### Feature Flag：决定功能可见性

Feature Flag 解决的是"这个用户看到什么功能"：

| Flag 类型 | 用途 | 生命周期 | 控制方式 |
|-----------|------|---------|---------|
| **发布 Flag** | 未完成的功能先合入主干但对用户不可见 | 功能完成后删除 | 代码内默认关闭 |
| **灰度 Flag** | 新功能逐步放量（1% → 10% → 全量） | 全量后删除 | 远程配置服务 |
| **实验 Flag** | A/B 测试 | 实验结束后删除 | 远程配置服务 |
| **运营 Flag** | 活动开关、紧急关闭 | 长期存在 | 远程配置服务 |
| **调试 Flag** | 开发期的 GM 指令、调试面板 | 发布时编译排除 | 编译开关（不是 Feature Flag） |

**关键区分**：调试 Flag 应该用编译开关（`#if DEVELOPMENT_BUILD`）实现，不应该用 Feature Flag 系统。编译开关在发布包中完全不存在；Feature Flag 在发布包中存在但关闭——两者的安全级别不同。

### Feature Flag 的生命周期管理

Feature Flag 最大的工程问题不是"怎么实现"，而是"怎么清理"。

```
创建 → 开发期关闭 → 灰度打开 → 全量打开 → 清理
```

每个 Flag 必须有：
- **创建时间和创建人**
- **预期全量时间**
- **全量后的清理责任人**

如果不管理生命周期，6 个月后代码里会有 50 个 Flag，其中 40 个永远为 true。每个永远为 true 的 Flag 都是一段死代码分支——降低可读性，增加测试路径。

**自动化清理提醒**：Flag 全量超过两个版本未清理，CI 应该发出警告。

### Feature Flag 的存储和同步

| 方案 | 读取时机 | 延迟 | 适用 |
|------|---------|------|------|
| **本地配置文件** | 启动时读取 | 无（但不可远程更新） | 发布 Flag |
| **远程配置服务** | 启动时拉取 + 定时刷新 | 秒级 | 灰度/实验/运营 Flag |
| **服务端下发** | 登录时随用户数据下发 | 登录后 | 需要和用户数据关联的 Flag |

远程 Flag 的同步必须处理两个边界情况：

**网络不可用时的回退**。客户端无法拉取远程配置时，使用上次缓存的配置。如果从未成功拉取过，使用构建时内置的默认值。

**Flag 变更的生效时机**。Flag 状态变更后，正在运行的客户端何时生效？立即生效（可能在游戏中途切换功能，体验不好）还是下次启动时生效（延迟但安全）？

## 环境配置和 Feature Flag 的关系

两者必须独立管理，但有一个交叉点：**Flag 的状态在不同环境可能不同。**

例如一个"新商城"功能的 Flag：
- 开发环境：开启（开发者需要看到新商城来开发和调试）
- 测试环境：开启（QA 需要测试新商城）
- 预发布环境：开启（最终验证）
- 生产环境：关闭（等待灰度放量）

这意味着 Flag 的存储系统需要支持按环境区分状态。如果所有环境共用一套 Flag 配置，在生产环境打开灰度时，开发环境的 Flag 也会变——影响正在开发的下一个版本。

## 常见错误做法

**环境配置用 if-else 判断**。代码里写 `if (isProduction) url = "api.game.com"` 而不是从配置文件读取。这意味着切环境需要改代码、重新编译。

**Feature Flag 没有远程控制能力**。Flag 状态写死在本地配置文件里，修改后需要发版才能生效。紧急关闭功能时发版需要 1-3 天审核——等不起。

**Flag 在代码里直接用字符串引用**。`if (GetFlag("new_shop"))` 散落在 20 个文件里。Flag 拼写错误不会编译报错，只会运行时静默返回 false。应该用常量或枚举统一管理 Flag 名称。

## 小结与检查清单

- [ ] 环境配置是否通过构建参数或配置文件注入（不硬编码）
- [ ] 环境配置是否和游戏配置分开管理
- [ ] Feature Flag 是否有远程控制能力（不需要发版就能开关）
- [ ] 调试功能是否用编译开关而非 Feature Flag
- [ ] 每个 Flag 是否有创建时间、预期全量时间和清理责任人
- [ ] Flag 全量后是否有自动化的清理提醒
- [ ] Flag 配置是否按环境区分状态
