## 这篇解决什么问题

修复一个 Bug 后最怕的事不是修不好，而是修好了旧问题、引入了新问题。回归验证就是要回答"修复是否引入了副作用"。但回归验证的范围怎么定？全量跑一遍不现实（太慢），只跑修改点不够（漏了关联影响）。本篇讲回归范围的确定方法、定向回归和全量回归的选择、自动化回归套件的设计，以及在发布压力下如何用时间盒保证最低验证覆盖。

## 回归范围的确定

回归范围不是拍脑袋定的，要基于影响分析。影响分析的输入是修复的 diff，输出是可能受影响的功能区域。

### 影响分析的三个维度

| 维度 | 方法 | 示例 |
|------|------|------|
| 代码依赖 | 查看修改文件的引用关系 | 修改了 ResourceLoader → 所有使用 ResourceLoader 的模块可能受影响 |
| 数据流 | 追踪修改的数据流向 | 修改了配表解析逻辑 → 所有读取该配表的功能 |
| 运行时路径 | 分析修改代码的执行路径 | 修改了登录流程 → 登录后的初始化链路 |

### 影响范围分级

| 影响级别 | 定义 | 回归策略 |
|---------|------|---------|
| 局部 | 只影响修改点的直接功能 | 修改点 + 直接关联功能 |
| 模块级 | 影响整个模块的行为 | 模块内全量 + 模块接口 |
| 跨模块 | 影响多个模块 | 受影响模块 + 核心流程 |
| 全局 | 影响基础设施/公共组件 | 全量回归 |

## 定向回归 vs 全量回归

| 维度 | 定向回归 | 全量回归 |
|------|---------|---------|
| 范围 | 只验证受影响区域 | 验证所有功能 |
| 耗时 | 1-4 小时 | 1-3 天 |
| 适用 | 影响范围明确、修改范围小 | 大版本发布、基础设施变更 |
| 风险 | 可能遗漏间接影响 | 耗时长，可能延误发布 |
| 自动化要求 | 可手动执行 | 必须依赖自动化 |

### 选择流程

```
修复的影响范围是什么？
├── 局部 → 定向回归
├── 模块级 → 定向回归（扩大到模块）
├── 跨模块 → 定向回归 + 核心流程冒烟
└── 全局 → 全量回归
```

## 自动化回归套件设计

自动化回归不是"把所有测试用例都自动化"，而是分层设计：

### 三层回归套件

| 层级 | 名称 | 内容 | 执行时机 | 耗时 |
|------|------|------|---------|------|
| L1 | 冒烟回归 | 核心流程的最小验证集 | 每次构建后自动执行 | 15-30 分钟 |
| L2 | 功能回归 | 各模块的关键功能验证 | 每日构建/修复验证 | 2-4 小时 |
| L3 | 全量回归 | 完整的功能 + 性能 + 稳定性验证 | 版本发布前 | 1-3 天 |

### L1 冒烟回归的覆盖范围

冒烟回归是"最低保障"——如果这些都过不了，版本绝对不能发：

| 验证点 | 通过标准 |
|--------|---------|
| 启动到主界面 | 不闪退、不黑屏、不卡在加载 |
| 登录流程 | 能正常登录/注册 |
| 核心玩法入口 | 至少一个核心玩法能进入并完成 |
| 支付流程 | 支付接口能正常调起 |
| 热更新流程 | 能检测更新并正常下载 |
| 基础 UI | 主要界面无错位/缺失 |

### 回归用例的维护

| 维护动作 | 频率 | 负责人 |
|---------|------|--------|
| 新增回归用例 | 每次 Bug 修复后（将复现步骤转化为回归用例） | 修复者 |
| 清理过期用例 | 每月 | QA 负责人 |
| 更新自动化脚本 | 功能变更时 | 自动化工程师 |
| 回顾回归有效性 | 每季度 | QA + 开发 |

## 热更新的回归验证

热更新的回归验证在 V08-05 已有详细覆盖，此处强调与缺陷修复相关的要点：

| 验证点 | 方法 | 关注点 |
|--------|------|--------|
| 热更包完整性 | 校验文件列表和 hash | 是否遗漏文件或多包含文件 |
| 增量更新正确性 | 从多个历史版本升级验证 | 差分包是否适用于所有源版本 |
| 功能正确性 | 执行修复点的回归用例 | 修复是否在热更后生效 |
| 兼容性 | 热更前后的存档/缓存兼容 | 旧存档在新版本中是否正常 |

## 时间盒回归

发布压力下不可能做全量回归。时间盒回归是在限定时间内做最高价值的验证：

### 时间盒的三个档位

| 档位 | 时间 | 适用场景 | 验证内容 |
|------|------|---------|---------|
| 紧急 | 1-2 小时 | P0 hotfix | L1 冒烟 + 修复点验证 |
| 标准 | 4-8 小时 | 常规 Bug 修复 | L1 冒烟 + L2 受影响模块 |
| 充分 | 1-3 天 | 版本发布 | L1 + L2 + L3 全量 |

### 时间盒内的优先级排序

时间有限时，先验证什么？

| 优先级 | 验证内容 | 原因 |
|--------|---------|------|
| 1 | 修复点本身 | 确认 Bug 真的修好了 |
| 2 | 修复点的直接关联功能 | 最可能受副作用影响 |
| 3 | 核心流程冒烟 | 确保基本功能没被破坏 |
| 4 | 历史高频 Bug 区域 | 这些区域最容易出问题 |
| 5 | 边缘场景和兼容性 | 如果还有时间再覆盖 |

## 回归度量指标

| 指标 | 定义 | 健康值 | 含义 |
|------|------|--------|------|
| 逃逸率 | 通过回归验证但线上仍出现的 Bug 比例 | < 5% | 回归覆盖是否充分 |
| 回归发现率 | 回归验证中发现新问题的比例 | 10-20% | 过低可能是用例不够，过高说明代码质量差 |
| 自动化覆盖率 | L1+L2 中自动化执行的比例 | > 70% | 回归效率是否足够 |
| 回归耗时趋势 | 每次回归的执行时间变化 | 稳定或下降 | 自动化是否在持续优化 |

逃逸率是最关键的指标——它直接衡量回归验证的质量。如果逃逸率持续高于 5%，说明回归范围或深度有问题，需要复盘。

## 小结

- [ ] 回归范围是否基于影响分析而不是拍脑袋
- [ ] 团队是否有 L1/L2/L3 三层回归套件
- [ ] L1 冒烟回归是否在每次构建后自动执行
- [ ] 每次 Bug 修复后是否将复现步骤转化为回归用例
- [ ] 时间盒回归是否有明确的优先级排序
- [ ] 是否跟踪逃逸率来衡量回归质量
