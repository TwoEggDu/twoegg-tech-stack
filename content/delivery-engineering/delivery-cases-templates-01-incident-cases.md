## 这篇解决什么问题

交付事故不是"别人的事"——每个项目都会踩坑，区别在于有没有从坑里爬出来时带走一些教训。这篇汇总了四类最常见的交付事故模式，每个案例按标准结构呈现：症状-排查过程-根因-修复-预防。目标不是讲故事，而是提供可检索的事故模式库——下次遇到类似症状时能快速定位方向。

## 案例一：热更新事故——CDN 缓存 + AB 依赖

### 症状

热更新推送后，部分用户反馈游戏内 UI 显示异常——按钮文字变成乱码、图标显示为粉红色方块。但团队在测试环境复现不出来，内部设备全部正常。

### 排查过程

| 步骤 | 操作 | 发现 |
|------|------|------|
| 1 | 检查热更新包完整性 | 服务器上的包是对的 |
| 2 | 抓包对比用户下载内容 | 部分用户下载到的 AB Hash 和最新版不一致 |
| 3 | 检查 CDN 缓存状态 | CDN 边缘节点缓存了旧版本的 catalog |
| 4 | 分析 AB 依赖 | 新 UI Bundle 依赖新 Font Bundle，但用户拿到旧 catalog 指向旧 Font |

### 根因

热更新推送时，先上传了新的 AB 文件，再更新 catalog。CDN 缓存了旧版本 catalog 的 TTL 还没过期。部分用户在 TTL 窗口内请求到了旧 catalog，旧 catalog 指向旧 Font Bundle，导致新 UI Bundle 加载到不匹配的字体资源。

### 修复

1. 立即清除 CDN 全部边缘节点的 catalog 缓存
2. 强制客户端重新请求 catalog（带版本号 query string 绕过缓存）
3. 用户重启后恢复正常

### 预防措施

| 措施 | 对应 Volume |
|------|------------|
| catalog 的 CDN TTL 设为 0 或极短（< 60s） | V06-06 CDN 分发 |
| catalog URL 带版本号 query string，保证每次热更获取最新 | V06-06 CDN 分发 |
| 热更推送顺序：先上传所有 AB → 验证完整性 → 最后更新 catalog | V08-06 热更事故模式 |
| AB 依赖关系纳入自动化检查：catalog 引用的所有 AB 必须在 CDN 上存在 | V08-05 热更验证 |

## 案例二：构建事故——环境漂移导致产物不一致

### 症状

CI 构建的 APK 在内部测试设备上运行正常。但同一天用开发机手动打的 APK 安装后闪退。排查发现两个 APK 的 IL2CPP 产物不一致。

### 排查过程

| 步骤 | 操作 | 发现 |
|------|------|------|
| 1 | 对比两个 APK 的文件列表 | libil2cpp.so 的 Hash 不同 |
| 2 | 对比构建日志 | 编译参数一致 |
| 3 | 对比 NDK 版本 | CI 机器 NDK r21e，开发机 NDK r23b |
| 4 | 锁定差异 | r23b 的 Clang 对某个模板展开行为不同，导致运行时 Crash |

### 根因

CI 构建环境通过 Docker 镜像锁定了 NDK 版本，但开发机的 NDK 由开发者自行安装，版本不一致。两个 NDK 版本的编译器对 C++ 模板的处理存在细微差异，导致生成的 native 代码行为不同。

### 修复

1. 回收手动构建的 APK，使用 CI 构建的产物
2. 统一团队 NDK 版本到 r21e

### 预防措施

| 措施 | 对应 Volume |
|------|------------|
| 构建环境 Docker 化，所有构建必须在容器中执行 | V07-02 构建配置 |
| 禁止从开发机直接出包用于外发（CI 是唯一的包产出路径） | V16 CI/CD 管线 |
| CI 构建日志记录完整的环境指纹（OS、SDK、NDK、编译器版本） | V07-01 构建基础 |
| 定期审计开发机环境与 CI 环境的差异 | V07-07 构建优化 |

## 案例三：性能事故——版本回退引发线上帧率下降

### 症状

新版本全量上线后，中低端设备的平均帧率从 28fps 下降到 22fps。Crash 率没有变化，但 DAU 留存率下降了 2 个百分点。

### 排查过程

| 步骤 | 操作 | 发现 |
|------|------|------|
| 1 | 对比性能测试报告 | 测试报告显示帧率达标（测试设备是高端机） |
| 2 | 分析线上数据按设备分层 | 中低端设备帧率下降明显，高端设备无变化 |
| 3 | Profiling 中低端设备 | Draw Call 增加了 40% |
| 4 | 二分定位 | 某美术资源的 LOD 配置被错误修改，低 LOD 层级的面数翻倍 |

### 根因

某次资源合并时，LOD Group 的配置文件冲突被错误解决——保留了高精度版本的 LOD1 覆盖了低精度版本。高端设备因为 GPU 余量大感知不到差异，中低端设备直接帧率崩盘。性能测试只在高端设备上跑，没有覆盖中低端设备的帧率验证。

### 修复

1. 热更新修复 LOD 配置
2. 中低端设备帧率恢复

### 预防措施

| 措施 | 对应 Volume |
|------|------------|
| 性能测试必须覆盖低、中、高三档设备 | V14-03 设备分档 |
| LOD / Mesh 资源变更纳入 CI 自动化检查（面数阈值） | V13-04 性能测试 |
| 资源合并冲突的解决必须有 TA Review | V18-03 多团队协作 |
| 版本灰度期间按设备档位分层监控帧率 | V17-03 线上监控 |

## 案例四：配置事故——数据表错误导致逻辑失败

### 症状

版本更新后，商城购买道具时客户端弹出"数据异常"错误提示。服务端日志显示价格字段解析失败。影响所有用户的商城功能。

### 排查过程

| 步骤 | 操作 | 发现 |
|------|------|------|
| 1 | 检查服务端日志 | 价格字段 type error：expected int, got string |
| 2 | 检查配置表 | 道具价格列有一行值为 "1,000"（带逗号） |
| 3 | 检查配置管线 | 配置验证规则没有覆盖数值格式检查 |
| 4 | 追溯变更 | 策划手动编辑 Excel 时输入了带千位分隔符的数值 |

### 根因

配置管线的验证规则检查了"非空""范围"，但没有检查"格式"。带千位分隔符的数值在 Excel 中显示正常，导出为 JSON 时变成字符串，运行时解析失败。

### 修复

1. 紧急修复配置表，去除千位分隔符
2. 服务端热加载修复后的配置

### 预防措施

| 措施 | 对应 Volume |
|------|------------|
| 配置验证规则增加格式检查（数值列不允许非数字字符） | V02-05 配置验证 |
| 配置变更走 CI 管线，不允许手动部署 | V16 CI/CD 管线 |
| 配置导出后自动化冒烟测试（核心流程走一遍） | V13-03 功能测试 |
| Excel 模板锁定单元格格式（数值列设为"数字"格式，不允许文本） | V02-04 配置管线 |

## 事故模式总结

| 事故类型 | 根因模式 | 核心预防手段 |
|---------|---------|------------|
| 热更新事故 | 缓存 + 依赖不一致 | CDN TTL 策略 + 依赖完整性检查 |
| 构建事故 | 环境漂移 | Docker 化 + CI 唯一出包 |
| 性能事故 | 覆盖不足 + 设备分层缺失 | 多设备档位测试 + 资源变更检查 |
| 配置事故 | 验证规则不完整 | 格式校验 + 自动化冒烟 |

**共同规律**：四类事故的根因都不是"某个人犯了错"，而是"系统缺少一道防线"。修复事故修的是当前问题，预防事故补的是系统的防线缺口。
