## 这篇解决什么问题

Bug 修好了但根因没找到，等于只治了症状。更危险的是"假根因"——把表面原因当成根因，觉得问题已经解决了，下次换个姿势又出。本篇介绍三种根因分析方法：五个为什么适合逻辑链条清晰的问题，时间线分析适合多因素交织的事故，分层定位适合快速缩小排查范围。

## 三种方法的适用场景

| 方法 | 适用场景 | 输出 | 耗时 |
|------|---------|------|------|
| 五个为什么（5-Whys） | 因果链条相对清晰的单一问题 | 从症状到系统原因的因果链 | 30-60 分钟 |
| 时间线分析（Timeline） | 多因素交织的复杂事故 | 事件序列 + 关键转折点 | 1-4 小时 |
| 分层定位（Layer-based） | 需要快速缩小排查范围 | 问题所在的具体层级 | 15-30 分钟 |

实际操作中常常组合使用：先用分层定位缩小范围，再用五个为什么深挖，复杂事故用时间线分析还原全貌。

## 方法一：五个为什么（5-Whys）

核心原则：对每个回答继续追问"为什么"，直到触达系统层面的原因。

### 示例：线上闪退

| 层级 | 问题 | 回答 |
|------|------|------|
| Why 1 | 为什么线上闪退？ | 加载某个 AssetBundle 时空引用异常 |
| Why 2 | 为什么空引用？ | 该 AB 依赖的一个纹理资源不存在 |
| Why 3 | 为什么资源不存在？ | 热更新包里没有包含这个纹理 |
| Why 4 | 为什么热更包没有包含？ | 资源依赖分析工具没有扫描到这个间接依赖 |
| Why 5 | 为什么没扫描到？ | 依赖分析只做了一级依赖，没有递归扫描 |

停在 Why 2（"空引用"）修复 → 只是加个空判断，下次换个资源缺失还会崩。
停在 Why 5（"依赖分析不递归"）修复 → 修依赖分析工具，这一整类问题都不会再发。

### 五个为什么的常见陷阱

| 陷阱 | 表现 | 应对 |
|------|------|------|
| 停得太早 | "因为配置写错了" → 结束 | 继续问：为什么配置会写错？为什么没有校验？ |
| 问歪了 | 中途跑到不相关的方向 | 每一步检查：这个回答是否直接导致了上一层的现象？ |
| 人身归因 | "因为小王粗心了" | 人会犯错是常态，问：系统为什么没有拦住人的错误？ |
| 只有一条链 | 只追一个方向 | 复杂问题可能有多条因果链，必要时分叉追 |

### 关键原则：永远追到系统层面

根因分析的终点不是"谁做错了"，而是"系统的哪个环节允许了这个错误发生"。

| 表面原因（不够深） | 系统原因（足够深） |
|-------------------|------------------|
| 配表写错了 | 配表没有自动化校验规则 |
| 代码没有处理空值 | 防御性编程规范没有在 CR 中检查 |
| 热更包漏了资源 | 资源依赖分析不完整 |
| 测试没覆盖到 | 测试用例设计没有覆盖这个路径的规则 |

## 方法二：时间线分析（Timeline Analysis）

适用于多因素交织的复杂事故——比如线上事故往往不是单一原因，而是多个因素在特定时间窗口内叠加。

### 操作步骤

1. **收集事件**：从所有相关方（开发、运维、QA、客服）收集事件节点
2. **排列时间线**：按时间顺序排列所有事件
3. **标记关键转折点**：哪个事件发生后问题开始恶化
4. **识别因果关系**：哪些事件之间有因果关联
5. **找到窗口期**：问题本可以在哪个时间点被拦截

### 时间线模板

| 时间 | 事件 | 执行人 | 影响 | 本可以拦截？ |
|------|------|--------|------|------------|
| T-7d | 新 Shader 合入主干 | 开发 A | 引入了 Mali GPU 不兼容的指令 | 是：Shader 审查流程 |
| T-3d | 构建通过，验证通过 | CI + QA | 验证设备没有 Mali GPU | 是：兼容性矩阵缺失 |
| T-0d | 版本发布 | 运维 | 正式上线 | 是：灰度范围未覆盖 Mali 设备 |
| T+2h | Mali 设备大量闪退 | 玩家 | 影响 15% 用户 | — |
| T+3h | 定位到 Shader 问题 | 开发 B | 确认根因 | — |
| T+5h | 推送热更修复 | 运维 | 问题解决 | — |

时间线分析的价值不在于还原经过，而在于找到"本可以拦截"的窗口——这些窗口就是防复发措施的切入点。

## 方法三：分层定位（Layer-based Localization）

快速排查时使用。沿着 V02 分类矩阵的根因维度逐层排除：

```
问题出现
├── 是内容问题吗？→ 检查配表/美术资源/文案
│   └── 不是
├── 是资源问题吗？→ 检查 AB 依赖/资源格式/打包配置
│   └── 不是
├── 是配置问题吗？→ 检查环境变量/Feature Flag/服务器配置
│   └── 不是
├── 是代码问题吗？→ 检查最近变更/逻辑分支/边界条件
│   └── 是 → 进入 5-Whys 深挖
├── 是平台问题吗？→ 检查 SDK 版本/OS 兼容性/API 变更
│   └── —
└── 是基础设施问题吗？→ 检查 CI/CDN/服务端日志
    └── —
```

分层定位的关键是**排除法**——快速排除不是的层，锁定是的层，然后用五个为什么深挖。

### 各层的快速排查清单

| 层级 | 排查方法 | 耗时 |
|------|---------|------|
| 内容 | 对比最近的配表/资源变更记录 | 5-10 分钟 |
| 资源 | 检查 AB 构建日志、资源依赖图 | 10-15 分钟 |
| 配置 | 对比各环境配置差异 | 5-10 分钟 |
| 代码 | git log 查最近变更 + 断点/日志复现 | 15-60 分钟 |
| 平台 | 查平台 SDK Release Notes + 设备特定日志 | 10-20 分钟 |
| 基础设施 | 查 CI 日志 + CDN 状态 + 服务端监控 | 10-15 分钟 |

## 根因文档模板

每个 P0/P1 Bug 修复后必须填写根因文档：

| 字段 | 内容 |
|------|------|
| 缺陷编号 | 关联 Jira/工单 ID |
| 症状描述 | 一句话描述现象 |
| 根因分析方法 | 使用了哪种方法（5-Whys/时间线/分层定位） |
| 根因链条 | Why 1 → Why 2 → ... → 系统根因 |
| 系统根因 | 一句话总结系统层面的原因 |
| 修复方案 | 代码层面怎么修 |
| 防复发措施 | 自动化检查/流程变更/知识记录（详见 V15-06） |
| 关联缺陷 | 是否有历史上的类似缺陷 |

## 常见的"假根因"

| 假根因 | 为什么是假的 | 真根因方向 |
|--------|------------|-----------|
| "他写错了" | 人一定会犯错，问系统为什么没拦住 | 缺少自动化校验 |
| "测试没覆盖到" | 这不是原因而是另一个问题 | 测试用例设计规则不完整 |
| "需求不清楚" | 那为什么没有在开发前澄清？ | 需求评审流程缺失 |
| "时间太赶了" | 赶工是常态，问为什么赶工就会出问题 | 质量保障流程在赶工时被跳过 |
| "第三方 SDK 的 Bug" | 你选择了用它，问为什么没有防御 | 缺少第三方 SDK 的异常处理 |

## 小结

- [ ] 团队是否掌握了三种根因分析方法并知道各自适用场景
- [ ] 五个为什么是否追到了系统层面而不是停在表面
- [ ] 复杂事故是否用时间线分析还原了完整经过
- [ ] P0/P1 缺陷是否有根因文档
- [ ] 是否识别并避免了"假根因"（人身归因、测试没覆盖等）
