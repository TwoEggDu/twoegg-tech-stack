## 这篇解决什么问题

如果项目需要上主机平台（PlayStation、Xbox、Nintendo Switch）或 XR 平台（Meta Quest、PlayStation VR2），认证流程和移动端审核完全不同。本篇给出概览，帮助团队评估主机发布的工程投入。

## 主机认证 vs 移动端审核

| 维度 | 移动端审核 | 主机认证 |
|------|-----------|---------|
| 审核方 | 平台人员抽查 | 专门的认证团队逐项检查 |
| 检查项 | 内容合规 + 基本功能 | 200-500 项技术要求 |
| 通过标准 | "没发现明显问题" | "每一项都必须通过" |
| 不通过后果 | 驳回重提 | 驳回 + 详细的失败报告 + 需要逐项修复 |
| 典型周期 | 1-3 天 | 2-4 周 |

### TRC（Technical Requirements Checklist）

每个主机平台都有自己的 TRC——一份必须逐项满足的技术要求清单。TRC 涵盖：

**系统集成**：
- 用户账户管理（登录、登出、多账户切换）
- 成就 / 奖杯系统集成
- 存档系统（自动存档、云存档、多存档槽）
- 手柄输入（包括特殊按键如 PS 键、Xbox 键的处理）

**性能要求**：
- 帧率要求（通常 30fps 最低，某些平台要求 60fps 选项）
- 加载时间上限
- 内存使用限制
- 启动时间限制

**错误处理**：
- 断网状态的处理
- 存储空间不足的处理
- 手柄断开的处理
- 用户中途退出的数据保存

**本地化**：
- 必须支持的语言列表
- 语言切换不重启
- 文本不裁切

**辅助功能**：
- 字幕 / 文本大小选项
- 色盲模式
- 按键自定义

### 常见不通过原因

| 原因 | 频率 | 预防 |
|------|------|------|
| 存档在异常退出时丢失 | 高 | 自动存档 + 写入完成确认 |
| 断网时崩溃 | 高 | 全部网络调用加超时和重试 |
| 手柄断开时无提示 | 中 | 监听手柄连接事件 + 弹窗 |
| 帧率不达标 | 中 | 性能预算必须覆盖目标平台 |
| 本地化文本裁切 | 中 | 所有语言的 UI 适配测试 |

### XR 平台额外要求

XR 平台（Meta Quest、PSVR2）在 TRC 基础上增加了：

- **舒适度要求**：帧率必须达标（通常 72fps+），否则会导致晕动症
- **安全提示**：游戏启动时必须显示安全使用提示
- **Guardian/边界系统**：必须正确响应平台的安全边界
- **性能模式**：必须支持平台的动态分辨率或固定帧率模式

## 主机认证在交付链路中的位置

主机认证的周期（2-4 周）远长于移动端审核。这直接影响发布节奏：

```
移动端：开发 → 验证 → 提审(1-3天) → 发布
主机：  开发 → 内部 TRC 预检 → 提交认证(2-4周) → 修复不通过项 → 重新认证 → 发布
```

**工程建议**：
- 在项目早期就阅读目标平台的 TRC，把要求融入开发流程（而非发布前才看）
- 建立内部 TRC 预检清单，在提交认证前自行逐项检查
- 每次重大功能变更后做一次内部 TRC 回归，不要积累到发布前一次性处理

## 小结

- [ ] 是否在项目早期阅读了目标平台的 TRC
- [ ] 是否有内部 TRC 预检清单
- [ ] 认证周期是否纳入发布计划（至少预留 4 周）
- [ ] 常见不通过原因是否有预防措施
- [ ] XR 平台的帧率和舒适度要求是否纳入性能预算
