上一篇画出了交付链路的全景图。链路中间有一个关键环节是"验证"——构建产物出来后，需要确认它是否满足交付标准。

但"质量好不好"不是一个可以笼统回答的问题。本篇把交付质量拆成四个独立维度，说清每个维度在回答什么问题、怎样度量、出了问题的代价是什么。

## 为什么要拆成四个维度

一个常见的交付事故模式是：功能测试全部通过，发版后在低端设备上大面积闪退。

这不是测试不认真。这是因为功能验证和稳定性验证是两个不同的维度，需要不同的验证方法和通过标准。只做了一个维度的验证，不能替代另一个维度。

四个维度各自独立，互不替代：

| 维度 | 回答的问题 | 不能替代 |
|------|-----------|---------|
| 功能 | 该做的事做了没有？ | 做了不代表跑得快 |
| 性能 | 跑得够快吗？ | 快不代表不崩 |
| 稳定性 | 能一直跑不崩吗？ | 不崩不代表所有设备都能跑 |
| 兼容性 | 所有目标设备都能跑吗？ | 能跑不代表功能对 |

四个维度形成一个闭环：功能对 → 跑得快 → 不会崩 → 所有设备都行。缺任何一个，交付都不完整。

## 维度一：功能

**回答的问题**：这个版本该做的事做了没有？做对了没有？没有把已经做对的事弄坏吧？

功能维度的三层检查：

**冒烟测试**。最小关键路径验证——登录、进主界面、进入核心玩法、退出。如果冒烟测试不过，后续所有验证都没有意义。

**功能验证**。新增和修改的功能是否按需求工作。这一层的难点不是"怎么测"，而是"测多少"——在有限的验证资源下，怎样用最少的用例覆盖最高风险的变更。

**回归验证**。本次变更是否影响了已有功能。回归的范围取决于变更的影响面分析——改了底层框架就需要全面回归，改了某个 UI 面板就只需要定向回归。

**失败代价**：用户看到的是"功能不对"或"功能缺失"——最直接、最容易被感知、最容易产生投诉的质量问题。

## 维度二：性能

**回答的问题**：在目标设备上，帧率、内存、加载时间和包体大小是否在预算范围内？

性能维度的核心概念是**预算**。不是"跑得越快越好"，而是"是否在预算基线内"。

**帧率预算**。目标帧率是多少？在什么设备上？在什么场景下？帧率预算不是一个全局数字，而是按设备分档、按场景类型制定的矩阵。

**内存预算**。整体内存峰值不超过多少？纹理、网格、音频、脚本各占多少？超出预算意味着 OOM 风险，而 OOM 在移动端直接表现为闪退。

**加载时间预算**。首次启动、场景切换、热更新下载各自的时间上限是多少？加载时间超预算不会崩溃，但会导致流失。

**包体预算**。首包大小、分包策略、热更新包大小各自的上限。包体超预算可能导致平台审核不通过（微信小游戏主包 4MB 限制），或者用户在低网速下放弃下载。

**失败代价**：用户感受到的是"卡"、"慢"、"占空间太大"——不一定会崩溃，但会直接影响留存和评分。

## 维度三：稳定性

**回答的问题**：在正常使用和边界条件下，产品能不能持续运行而不崩溃？

稳定性维度关注的不是"能不能跑通一次"，而是"能不能持续跑"。

**Crash 率**。启动 Crash、运行时 Crash、特定操作触发的 Crash。Crash 率是版本健康最硬的指标之一——超过阈值就应该触发回滚或热修复。

**ANR / 卡死**。Android 上的 Application Not Responding，iOS 上的 Watchdog Kill。主线程阻塞超过阈值，系统会直接杀掉进程。用户的感知是"卡住了然后闪退"。

**内存泄漏**。短时间运行没问题，长时间运行内存持续增长，最终 OOM。这类问题需要长时间运行测试才能暴露。

**弱网和异常恢复**。网络断开后重连、切后台再恢复、下载中断后续传——这些边界场景在功能测试中很容易被忽略，但在真实用户环境中频繁发生。

**失败代价**：用户看到的是"闪退"、"卡死"、"数据丢失"——这是最严重的质量问题，直接影响商店评分和用户信任。

## 维度四：兼容性

**回答的问题**：在所有目标设备和平台上，产品能不能正常工作？

兼容性维度的特殊性在于：它不是一个可以在单台设备上验证完的维度。同样的代码和资源，在不同设备上的表现可能完全不同。

**设备兼容性**。不同的 CPU / GPU / 内存组合，不同的屏幕分辨率和宽高比。低端设备可能不支持某些渲染特性，高端设备可能暴露不同的时序问题。

**OS 版本兼容性**。Android 的碎片化最严重——不同厂商的定制 ROM 可能修改了系统行为。iOS 版本更新相对统一，但旧版本的兼容性仍需关注。

**图形 API 兼容性**。不同 GPU 对 Vulkan / Metal / OpenGL ES 的支持程度不同。Shader 编译、纹理格式、渲染精度都可能存在差异。

**平台特有约束**。微信小游戏的内存限制、包体限制和 API 限制与原生 App 完全不同。同一套代码在原生端运行正常，在小游戏端可能完全跑不动。

**失败代价**：用户看到的是"在我的手机上打不开"或"画面异常"——这类问题最难排查，因为开发团队的设备上通常无法复现。

## 四个维度的优先级

四个维度不是等权重的。在资源有限的情况下，验证的优先级通常是：

```
稳定性 > 功能 > 性能 > 兼容性
```

**稳定性最高**，因为 Crash 是最严重的质量问题。一个功能不对但不崩的版本，比一个功能正确但频繁崩溃的版本好得多。

**功能次之**，因为功能缺失或错误是用户最直接能感知的问题。

**性能再次**，因为性能退化通常是渐进的，不会像 Crash 那样突然爆发。但性能问题如果严重到触发 OOM 或 ANR，就会升级为稳定性问题。

**兼容性最后**，不是因为它不重要，而是因为它的验证成本最高（需要大量设备），通常需要通过设备矩阵和优先级排序来管理。

这个优先级不是固定的。如果你的产品面向全球市场、覆盖数百种设备，兼容性的优先级会显著上升。如果你的产品刚上线、用户量还小，稳定性的容忍度可能更高。

关键是：**四个维度必须同时存在于验证体系中**，只是在资源分配上有先后。

## 质量维度与交付链路的关系

这四个维度不只影响验证段。它们在交付链路的每个环节都有对应的工程手段：

| 维度 | 生产段 | 构建段 | 验证段 | 运营段 |
|------|--------|--------|--------|--------|
| 功能 | 需求规格和配置验证 | 编译正确性 | 冒烟 + 功能 + 回归测试 | 用户反馈闭环 |
| 性能 | 资源规范和预算约束 | 打包策略和压缩配置 | 性能基线和回归检测 | 线上性能指标监控 |
| 稳定性 | 代码质量和异常处理 | 构建配置一致性 | Crash 测试和长时间运行 | Crash 率监控和告警 |
| 兼容性 | 资源格式多端适配 | 多端构建正确性 | 设备矩阵测试 | 设备分布和问题上报 |

这意味着质量不是在验证段才开始关注的。每个维度在链路的每个环节都有自己的工程手段。
