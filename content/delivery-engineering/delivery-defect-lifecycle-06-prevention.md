## 这篇解决什么问题

Bug 修好了、回归通过了，大多数团队到这里就结束了。但如果不做防复发，同类问题迟早换个姿势再来一次。防复发是缺陷闭环的最后一步也是最重要的一步——它决定了团队是在"永远救火"还是"越来越少的火"。本篇介绍三级防复发机制、复发率度量方法、Postmortem 模板，以及 V15 全系列的总结。

## 三级防复发机制

防复发不是一种动作，而是三个层级逐步递进的机制：

| 级别 | 名称 | 做什么 | 效果 | 投入 |
|------|------|--------|------|------|
| L1 | 自动化检查 | 在 CI 中添加规则，自动拦截同类问题 | 同类问题在提交阶段被拦截 | 低（几小时） |
| L2 | 流程变更 | 修改产生问题的工作流程 | 同类问题在产生阶段被预防 | 中（需协调） |
| L3 | 知识沉淀 | 记录模式供未来参考 | 团队认知提升，类似问题被更快识别 | 低（几小时） |

三个层级不是互斥的，一个 Bug 的防复发可能同时需要 L1 + L2 + L3。

## L1：自动化检查

自动化检查是最直接的防复发手段——把"人记住不要犯这个错"变成"机器自动拦住这类错"。

### 常见的自动化检查类型

| 检查类型 | 工具/方法 | 示例 |
|---------|---------|------|
| 静态分析规则 | Roslyn Analyzer / Unity CodeAnalysis | 禁止在 Update 中分配大数组 |
| 配表校验规则 | CI 中的配表验证脚本 | 检查 ID 引用是否有效、数值范围是否合理 |
| 资源规范检查 | 资源导入后处理脚本 | 纹理尺寸不超过 2048、AB 不包含冗余依赖 |
| 构建后验证 | 构建后脚本 | 检查最终包体是否包含必要文件 |
| 提交钩子 | Git pre-commit / pre-push hook | 检查提交信息格式、禁止提交大文件 |

### L1 的设计原则

| 原则 | 说明 | 反例 |
|------|------|------|
| 规则明确 | 每条规则有清晰的通过/不通过标准 | "代码质量要好" — 无法自动化判定 |
| 反馈快速 | 检查结果在分钟内返回 | 检查跑 30 分钟才出结果 |
| 误报率低 | 误报会让人忽略真正的警告 | 每次构建都报 20 条无关警告 |
| 可跳过但留痕 | 紧急情况可以跳过，但记录在案 | 跳过后无人知道 |

### L1 的投入产出

一条好的自动化规则可以拦截该类问题的所有未来实例。投入几小时写规则，避免未来数十次相同 Bug 的修复成本。

| 投入 | 产出 |
|------|------|
| 2-4 小时编写规则 | 该类问题在 CI 阶段被永久拦截 |
| 定期维护规则集 | 规则集持续积累，防线越来越密 |

## L2：流程变更

有些问题不是代码检查能拦住的，而是工作流程本身有漏洞。

### 常见的流程变更

| 问题根因 | 流程变更 | 变更后的效果 |
|---------|---------|------------|
| 配表修改后没有走验证 | 配表修改必须触发 CI 校验才能合入 | 配表错误在提交前被发现 |
| 热更包漏资源 | 热更前增加资源清单比对步骤 | 资源遗漏在发布前被发现 |
| 新 Shader 未经兼容性测试 | Shader 变更必须经过 GPU 兼容矩阵验证 | 兼容性问题在合入前被发现 |
| 紧急修复跳过 Code Review | P0 修复可以事后 CR 但必须在 24h 内补 | CR 不被跳过，只是延后 |
| 第三方 SDK 升级引入问题 | SDK 升级必须在隔离分支上验证后再合入 | SDK 问题不会直接影响主干 |

### L2 的注意事项

| 注意 | 说明 |
|------|------|
| 不要过度增加流程 | 每加一步流程就增加一点交付摩擦 |
| 优先自动化 | 能自动化的不要靠人手动执行 |
| 明确例外规则 | 紧急情况怎么跳过、跳过后怎么补 |
| 定期清理 | 不再需要的流程步骤要及时移除 |

## L3：知识沉淀

不是所有问题都能用自动化检查或流程变更来防止。有些问题需要靠人的经验判断，这时候知识沉淀是最后一道防线。

### 知识沉淀的形式

| 形式 | 适用场景 | 维护方式 |
|------|---------|---------|
| 缺陷模式库 | 记录已知的缺陷模式和排查方法 | 每次修复后补充 |
| Code Review 检查清单 | CR 时需要关注的已知陷阱 | 每月更新 |
| Onboarding 文档 | 新人需要知道的历史教训 | 每季度更新 |
| 技术分享 | 复杂问题的根因分析过程 | 每次 P0/P1 后分享 |

### 缺陷模式库的条目格式

| 字段 | 内容 |
|------|------|
| 模式名称 | 一句话描述这类问题（如"AB 间接依赖缺失"） |
| 症状表现 | 用户/测试会看到什么现象 |
| 根因模式 | 什么条件下会产生这个问题 |
| 排查路径 | 怎么快速确认是不是这个问题 |
| 修复模板 | 典型的修复方式 |
| 自动化检查 | 是否已有 L1 规则覆盖 |
| 历史案例 | 关联的 Jira/工单编号 |

## 防复发质量度量

| 指标 | 定义 | 计算方法 | 健康值 |
|------|------|---------|--------|
| 复发率 | 同类缺陷再次出现的比例 | 复发缺陷数 / 已关闭缺陷总数 | < 5% |
| L1 覆盖率 | 有自动化检查规则的缺陷类型比例 | 有 L1 规则的类型数 / 总类型数 | > 50% |
| 防复发执行率 | P0/P1 修复后实际完成防复发措施的比例 | 有防复发措施的 P0P1 数 / P0P1 总数 | > 80% |
| L1 拦截数 | CI 中自动化规则拦截的问题数量 | 月度统计 | 持续增长 |

复发率是终极指标：如果同类问题不再复发，说明闭环真的闭上了。

## Postmortem 模板

每个 P0/P1 Bug 修复后必须完成 Postmortem（事后复盘）：

| 章节 | 内容 |
|------|------|
| 概述 | 一句话描述问题、影响范围、持续时间 |
| 时间线 | 从问题发生到彻底修复的完整事件序列 |
| 影响评估 | 影响了多少用户、持续了多长时间、造成了什么损失 |
| 根因分析 | 使用 V15-03 的方法分析根因（5-Whys/时间线/分层定位） |
| 修复方案 | 做了什么修复、通过什么渠道发布 |
| 防复发措施 | L1/L2/L3 各层级的具体措施 |
| 行动项 | 每个措施的负责人和完成截止日期 |
| 回顾 | 什么做得好、什么可以改进 |

### Postmortem 的关键原则

| 原则 | 说明 |
|------|------|
| 对事不对人 | 聚焦系统和流程，不追责个人 |
| 行动项必须有 Owner | 没有 Owner 的行动项等于没有 |
| 行动项必须有 Deadline | 没有 Deadline 的行动项会被无限拖延 |
| 定期跟踪行动项 | 每周检查未完成的防复发行动项 |
| 公开透明 | Postmortem 对团队公开，而不是只有相关人知道 |

### 行动项跟踪表

| 行动项 | 类型 | 负责人 | 截止日期 | 状态 |
|--------|------|--------|---------|------|
| 添加 AB 递归依赖扫描 CI 规则 | L1 | 开发 A | 修复后 3 天 | 进行中 |
| 热更发布前增加资源清单比对步骤 | L2 | 运维 B | 修复后 1 周 | 待开始 |
| 更新缺陷模式库"AB 依赖缺失"条目 | L3 | QA C | 修复后 2 天 | 已完成 |

## V15 缺陷闭环系列总结

六篇文章从缺陷生命周期的六个阶段构建了完整的闭环体系：

| 阶段 | 篇目 | 核心产出 |
|------|------|---------|
| 全局视角 | 01 缺陷闭环的本质 | 从"修 Bug"升级到"修系统"的认知 |
| 分类 | 02 问题分类体系 | 5 × 6 两维矩阵 + P0-P3 优先级 |
| 根因分析 | 03 根因分析方法论 | 5-Whys / 时间线 / 分层定位三种方法 |
| 修复 | 04 修复策略 | 最小修复范围 + Cherry-Pick 规范 + 三端覆盖 |
| 回归验证 | 05 回归验证设计 | 三层回归套件 + 时间盒回归 |
| 防复发 | 06 防止复发 | 三级防复发机制 + Postmortem 模板 |

闭环的终极检验标准：**复发率持续下降**。如果修过的同类问题不再出现，闭环就是有效的。

V16 将讲交付度量与持续改进——用数据驱动交付体系的优化，与 V15 的防复发度量形成闭环。

## 小结

- [ ] 每次 P0/P1 修复后是否完成了三级防复发措施
- [ ] L1 自动化检查是否在持续积累
- [ ] L2 流程变更是否经过评估，没有过度增加交付摩擦
- [ ] L3 缺陷模式库是否在持续更新
- [ ] 是否跟踪复发率来衡量防复发效果
- [ ] P0/P1 的 Postmortem 是否按模板执行并跟踪行动项
