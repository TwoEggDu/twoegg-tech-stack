## 这篇解决什么问题

V14-02 的预算体系按设备档位分了高/中/低三档。但"高中低"不是一个技术方案——哪些设备是高档？怎么在运行时判定？判定后怎么切换质量配置？运行中温度升高怎么动态降级？

这一篇解决设备分档的完整链路：指纹采集 → 档位映射 → 质量配置 → 动态调整 → 远程更新。

## 设备分档在交付中的位置

设备分档是预算和用户体验之间的桥梁：

```
性能预算（V14-02）→ 设备分档 → 质量配置 → 用户体验
                        ↑
                   设备能力指纹
```

没有设备分档，预算就是一个理论值——因为你不知道当前设备应该执行哪套预算。

## 能力指纹

设备分档的输入是"能力指纹"——一组描述设备硬件能力的参数：

| 参数 | 获取方式 | 权重 |
|------|---------|------|
| GPU 型号 | `SystemInfo.graphicsDeviceName` | 高——GPU 直接决定渲染能力 |
| GPU 显存 | `SystemInfo.graphicsMemorySize` | 高 |
| 系统内存 | `SystemInfo.systemMemorySize` | 高——决定内存预算 |
| CPU 核心数 | `SystemInfo.processorCount` | 中 |
| CPU 频率 | `SystemInfo.processorFrequency` | 中 |
| 屏幕分辨率 | `Screen.width * Screen.height` | 中——影响 GPU 负载 |
| 芯片组 | 平台 API（Snapdragon/Dimensity/A 系列） | 高——已有基准数据 |

### 指纹到档位的映射方式

**方式一：查表法（推荐）**

维护一张设备 → 档位的映射表，按芯片组或 GPU 型号精确映射。

```
Snapdragon 8 Gen 3  → 高档
Snapdragon 8 Gen 2  → 高档
Snapdragon 7 Gen 1  → 中档
Dimensity 8200      → 中档
Snapdragon 680      → 低档
A17 Pro             → 高档
A15                 → 中档
A12                 → 低档
```

优点：精确可控，基于实测数据。缺点：新设备上市时需要更新映射表。

**方式二：评分法（兜底）**

用硬件参数加权计算得分，按得分段分档。用于映射表中没有的未知设备。

```
score = GPU_score * 0.4 + Memory_score * 0.3 + CPU_score * 0.2 + Screen_score * 0.1
高档：score ≥ 80
中档：40 ≤ score < 80
低档：score < 40
```

**实践建议**：查表法为主，评分法为兜底。映射表覆盖 90% 的用户设备，剩下 10% 用评分法自动分档。

## 质量配置

每个档位对应一套质量配置：

| 配置项 | 高档 | 中档 | 低档 |
|--------|------|------|------|
| 目标帧率 | 60 FPS | 30 FPS | 30 FPS |
| 渲染分辨率 | 原生 | 0.85x | 0.7x |
| 阴影 | 实时 + 软阴影 | 实时硬阴影 | 关闭 |
| 后处理 | Bloom + SSAO + TAA | Bloom + TAA | 关闭 |
| LOD 距离倍率 | 1.0 | 0.8 | 0.6 |
| 特效粒子数 | 100% | 60% | 30% |
| 纹理分辨率 | 原图 | 0.5x | 0.25x |
| 同屏角色上限 | 30 | 20 | 10 |

### 质量配置的管理方式

质量配置应该是**数据驱动**的，不是硬编码在代码里的：

1. 配置文件定义每个档位的参数值（JSON/ScriptableObject）
2. 质量管理器在启动时读取当前档位 → 应用对应配置
3. 配置文件可通过热更新替换 → 不发版就能调整质量参数

## 动态降级

静态分档解决的是"启动时用什么配置"。但运行过程中设备状态会变化：

- **热降频**：长时间高负载 → 芯片温度升高 → 系统降低 CPU/GPU 频率 → 实际性能下降
- **电量不足**：低电量模式 → 系统限制性能
- **后台应用争抢**：系统可用内存减少

### 动态降级策略

| 监控指标 | 采集方式 | 降级动作 |
|---------|---------|---------|
| 帧率持续低于目标 | 近 N 帧的平均帧时间 | 降低渲染分辨率、减少特效 |
| 设备温度过高 | 平台 API（iOS `thermalState`） | 降一档质量配置 |
| 可用内存不足 | 平台 API / `SystemInfo` | 释放低优先级缓存、降低纹理分辨率 |
| 电量低于阈值 | 平台 API | 限制帧率到 30 FPS |

**降级不是一步到底**——应该分级执行：

```
正常运行 → 轻度降级（降分辨率）→ 中度降级（降特效 + 分辨率）→ 重度降级（降一档配置）
                ↑                        ↑                          ↑
           帧率低于目标 5%          帧率低于目标 15%          帧率低于目标 30%
```

**回升机制**：温度降下来、帧率恢复后，逐步恢复到原始配置。回升的阈值应该比降级阈值更宽松（避免在临界点反复切换）。

## 远程配置更新

设备分档的映射表和质量配置都应该支持远程更新：

| 更新场景 | 说明 |
|---------|------|
| 新设备上市 | 映射表中添加新芯片组 → 远程下发 |
| 用户反馈特定设备卡顿 | 调整该设备的档位 → 远程下发 |
| 新功能上线 | 调整质量配置参数 → 远程下发 |
| 运营活动（大场景） | 临时降低低档设备的质量上限 → 远程下发 |

远程配置的更新频率通常是"按需"——不需要定期更新，有新设备或有问题时才更新。

## 小结与检查清单

- [ ] 是否有设备分档方案（而非所有设备跑同一套配置）
- [ ] 分档是否基于能力指纹（而非仅看设备型号字符串）
- [ ] 是否有查表法 + 评分法的组合策略（已知设备查表、未知设备评分）
- [ ] 每个档位是否有完整的质量配置表
- [ ] 质量配置是否数据驱动（而非硬编码）
- [ ] 是否有动态降级机制（运行时温度/帧率/内存监控）
- [ ] 动态降级是否分级执行且有回升机制
- [ ] 设备映射表和质量配置是否支持远程更新
