## 这篇解决什么问题

性能问题往往是"温水煮青蛙"——每次提交只退化一点点，累积到某天突然发现帧率不够了。性能测试的目的不是优化，而是在退化发生时及时发现。本篇讲性能测试怎么做、预算基线从哪来、数据怎么自动采集、退化怎么自动检测。

## 性能测试 vs 性能分析

| 维度 | 性能测试（Performance Testing） | 性能分析（Profiling） |
|------|-------------------------------|---------------------|
| 目的 | 检查是否满足预算 | 定位性能瓶颈的原因 |
| 时机 | 每次构建 / 发版前 | 发现退化后 |
| 输出 | 通过 / 不通过 + 数据报告 | 火焰图、调用栈、内存快照 |
| 自动化 | 必须自动化 | 通常手动 |
| 执行者 | CI 系统 + 自动化框架 | 开发工程师 |

性能测试是验证环节，性能分析是调查环节。先测出退化，再用分析工具找原因。

## 性能预算基线

性能测试需要基线——没有基线就没有对比，没有对比就不知道是否退化。

### 基线来源

性能预算基线从 V14 定义（本系列和 V14 形成闭环），核心指标：

| 指标 | 低端设备 | 中端设备 | 高端设备 |
|------|---------|---------|---------|
| 战斗场景 FPS | ≥ 25 | ≥ 30 | ≥ 45 |
| 主界面加载时间 | ≤ 5s | ≤ 3s | ≤ 2s |
| 内存峰值 | ≤ 1.2GB | ≤ 1.5GB | ≤ 2.0GB |
| 启动到首帧 | ≤ 12s | ≤ 8s | ≤ 5s |
| 包体大小（安装包） | ≤ 1.5GB | ≤ 1.5GB | ≤ 1.5GB |

基线不是一成不变的——随着功能增加，预算可能调整，但调整需要经过评审。

### 退化阈值

退化检测不是"低于基线就告警"，而是允许一定波动：

| 告警级别 | 条件 | 响应 |
|---------|------|------|
| 警告 | 指标退化 3-5% | 记录，通知负责人 |
| 阻断 | 指标退化 > 5% | 构建标记为失败 |
| 严重 | 指标超出预算 | 立即修复，不允许合入 |

## 自动化性能数据采集

性能测试的关键是自动化和可重复。手动跑一次测不算测试，因为不可重复、不可比较。

### 采集流程

```
CI 构建 → 部署到测试设备 → 运行 Benchmark 场景 → 采集指标 → 上报数据 → 与基线比对 → 生成报告
```

### Benchmark 场景设计

| 场景 | 测试内容 | 关键指标 |
|------|---------|---------|
| 启动场景 | 冷启动到首帧 | 启动时间、首帧内存 |
| 主界面场景 | 主界面加载和操作 | 加载时间、帧率、内存 |
| 战斗场景（标准） | 固定数量怪物 + 技能释放 | 平均帧率、最低帧率、帧时间方差 |
| 战斗场景（压力） | 大量怪物 + 全屏特效 | 最低帧率、Draw Call、三角面数 |
| 场景切换 | 从 A 场景到 B 场景 | 加载时间、内存峰值 |
| 长时间运行 | 循环执行核心玩法 30 分钟 | 内存趋势、帧率趋势 |

Benchmark 场景必须固定输入——相同的操作序列、相同的数据、相同的场景配置，才能保证数据可比较。

## 设备矩阵

性能测试必须在真实设备上跑——模拟器的性能数据没有参考价值。

| 设备层级 | 定义 | 测试频率 | 典型设备示例 |
|---------|------|---------|------------|
| 低端基准机 | 最低支持的设备 | 每次发版 | 2-3 年前的中低端机型 |
| 中端代表机 | 用户量最大的设备段 | 每次构建 | 当前主流中端机型 |
| 高端代表机 | 高性能设备 | 每次构建 | 当前旗舰机型 |

中端代表机是日常 CI 的主力测试设备——它代表了最大的用户群体。

## 性能回归检测

性能回归检测的核心逻辑：

| 步骤 | 说明 |
|------|------|
| 采集当前数据 | 在 Benchmark 场景中采集本次构建的性能指标 |
| 获取基线数据 | 取最近 N 次构建的平均值作为基线（消除单次波动） |
| 计算偏差 | 当前值与基线的百分比差异 |
| 判定退化 | 偏差超过阈值则标记为退化 |
| 生成报告 | 包含退化指标、对比图表、可能的关联提交 |

### 减少噪声

性能数据天然有波动，以下措施减少误报：

- **多次采样取平均**：同一场景跑 3-5 次，去掉最高最低取平均
- **滑动基线**：基线取最近 5-10 次构建的平均值，而非单次快照
- **设备隔离**：测试设备专用，不跑其他任务
- **温度控制**：设备过热会降频，需要冷却间隔

## 服务端性能测试

客户端性能测试之外，服务端也需要独立的性能验证：

| 测试类型 | 目的 | 方法 |
|---------|------|------|
| 负载测试 | 验证正常负载下的响应时间 | 模拟预期并发量，检查 P99 延迟 |
| 压力测试 | 找到系统的承载上限 | 逐步加压直到性能不可接受 |
| 浸泡测试 | 验证长时间运行的稳定性 | 中等负载运行 8-24 小时，观察资源趋势 |
| 峰值测试 | 验证突发流量的处理能力 | 短时间内大幅加压，检查恢复速度 |

服务端性能测试通常使用 JMeter、Locust、k6 等工具，在独立的测试环境中执行。

## 性能数据的可视化与趋势

采集的性能数据不只用于单次比对，更重要的是看趋势：

- **帧率趋势图**：每次构建的平均帧率，发现渐进式退化
- **内存趋势图**：内存峰值的变化，发现内存增长
- **启动时间趋势图**：启动耗时的变化，发现加载退化
- **退化事件时间线**：标注每次退化对应的提交，方便追溯

## 小结

- [ ] 团队是否区分了性能测试和性能分析
- [ ] 是否定义了性能预算基线（分设备层级）
- [ ] 是否有自动化的性能数据采集流程
- [ ] Benchmark 场景是否固定输入、可重复
- [ ] 是否设置了退化阈值和自动告警
- [ ] 服务端是否有独立的负载/压力/浸泡测试
- [ ] 性能数据是否有趋势可视化
