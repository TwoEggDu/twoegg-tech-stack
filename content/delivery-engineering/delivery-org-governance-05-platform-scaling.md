## 这篇解决什么问题

公司的第一个项目有了 CI 脚本、构建工具、自动化测试。第二个项目启动时，把第一个项目的脚本拷过来改了改。第三个项目又拷了一份。三个项目的脚本各自演化，修一个 Bug 要改三份代码，新功能加了一份忘了另外两份。这篇讨论交付工具链的规模化路径——什么时候从项目脚本升级到跨项目平台，怎么抽象，维护边界怎么划。

## 工具链成熟度的三个阶段

| 阶段 | 名称 | 特征 | 维护者 |
|------|------|------|--------|
| L1 | 项目脚本 | 解决单项目的局部痛点，脚本散落在项目里 | 谁写的谁维护 |
| L2 | 交付管线 | 脚本串成 CI/CD 管线，有统一入口和执行顺序 | 工具链工程师 |
| L3 | 跨项目平台 | 把可复用的能力抽象为平台服务，项目通过配置接入 | 平台团队 |

### L1 → L2：从脚本到管线

这个跨越在 V16 CI/CD 管线系列已经详细讨论。核心是把散落的脚本串成自动化管线——提交触发构建、构建触发测试、测试触发部署，人不再需要记住执行顺序。

### L2 → L3：从管线到平台

这是本篇的重点。L2 的管线解决了单项目的自动化问题，但当公司有多个项目时，每个项目各自维护一套管线配置。L3 把管线中可复用的部分抽象成平台，项目通过配置接入。

## 什么时候该抽平台

**不要过早抽象**。抽平台有成本——设计抽象层、维护兼容性、支持多项目的差异化需求。过早抽象会让平台变成所有项目的公约数，谁都不满意。

### 三条抽平台的前置条件

| 条件 | 说明 | 反例 |
|------|------|------|
| 3+ 项目已确认 | 至少三个项目有同样的需求，不是臆测的"以后可能需要" | 只有一个项目在用，另外两个"计划中" |
| 执行模型一致 | 多个项目的构建、测试、部署流程在核心步骤上相同 | 一个项目用 Unity 一个用 UE，构建流程完全不同 |
| 有专人维护 | 有人（或有团队）专门负责平台的开发和维护 | "大家一起维护"=没人维护 |

**如果三条都满足**：可以开始抽平台。
**如果只满足两条**：做好准备但先不动手。
**如果只满足一条**：继续用项目级脚本，不要强行抽象。

## 平台架构原则

### 规则与执行器分离

平台最核心的架构决策：**把"做什么"和"怎么做"分开**。

```
┌─────────────────────────────────────────┐
│              平台层（Platform）           │
│  ┌──────────┐  ┌──────────┐  ┌────────┐ │
│  │ 构建引擎  │  │ 测试引擎  │  │ 部署引擎│ │
│  └──────────┘  └──────────┘  └────────┘ │
│         ↑            ↑           ↑       │
│     执行器：通用能力，平台团队维护       │
└─────────────────────────────────────────┘
         ↑            ↑           ↑
┌─────────────────────────────────────────┐
│              配置层（Config）             │
│  project_a.yaml  project_b.yaml  ...    │
│         ↑                               │
│     规则：项目差异化配置，项目团队维护    │
└─────────────────────────────────────────┘
```

### 配置驱动的差异化

每个项目通过 YAML/JSON 配置定义自己的交付规则：

```yaml
# project_a.yaml — 配置示例
project:
  name: "ProjectA"
  engine: "unity"
  version: "2022.3"

build:
  platforms: ["android", "ios"]
  il2cpp: true
  addressables: true

quality_gates:
  - name: "unit_test"
    threshold: 100%
    severity: "block"
  - name: "crash_free_rate"
    threshold: 99.5%
    severity: "block"
  - name: "bundle_size"
    threshold: "1.2GB"
    severity: "warn"

deploy:
  canary_stages: [1%, 10%, 50%, 100%]
  rollback: "auto"
```

**关键**：配置层只描述"要什么"（质量门的阈值是多少、灰度分几个阶段），不描述"怎么做"（怎么运行测试、怎么执行灰度）。"怎么做"由平台层的执行器处理。

### 平台需要提供的核心能力

| 能力 | 说明 | 各项目的差异化方式 |
|------|------|--------------------|
| 构建 | 编译、打包、签名 | 配置：平台、编译选项、签名证书 |
| 质量门 | 运行检查并判断是否通过 | 配置：检查项、阈值、严重级别 |
| 部署 | 推送到分发渠道 | 配置：渠道列表、灰度策略 |
| 通知 | 构建结果通知、审批提醒 | 配置：通知渠道、通知规则 |
| 审计 | 记录每次构建、发布、审批 | 配置：保留策略、报告格式 |

## 维护边界

平台和项目之间必须有清晰的维护边界：

| 层级 | 维护者 | 职责 |
|------|--------|------|
| 平台引擎 | 平台团队 | 通用执行器的开发、Bug 修复、性能优化 |
| 平台 SDK/CLI | 平台团队 | 项目接入的入口和文档 |
| 项目配置 | 项目团队 | 本项目的交付规则配置 |
| 项目定制脚本 | 项目团队 | 平台不支持的项目特殊需求 |

### 边界冲突的处理

| 场景 | 处理方式 |
|------|---------|
| 项目想要平台不支持的功能 | 项目团队先用定制脚本解决；如果 2+ 项目都需要，平台团队评估后纳入平台 |
| 平台升级导致项目配置不兼容 | 平台团队提供迁移脚本 + 至少一个版本的向后兼容期 |
| 项目绕过平台直接改底层 | 平台团队有权拒绝支持被修改的部分；项目团队自行承担维护成本 |

## 从 L2 到 L3 的渐进路径

不要一步到位——推荐的渐进抽象路径：

### Phase 1：统一构建配置格式（1-2 月）

把各项目的构建脚本参数化，用统一的 YAML 格式描述，但执行器还是各项目自己的脚本。

### Phase 2：抽取公共执行器（2-3 月）

把各项目相同的执行逻辑提取为共享包（npm package、Python package、NuGet package），各项目通过引用包来使用。

### Phase 3：平台化服务（3-6 月）

把执行器包装成服务——项目提交配置，平台调度执行。此时项目不再直接运行构建脚本，而是调用平台 API。

### Phase 4：自助化（6-12 月）

项目团队可以通过 Web UI 自助配置交付规则，不需要修改代码或找平台团队。

**关键**：每个 Phase 都应该能独立交付价值。不要"全做完才上线"——Phase 1 结束后各项目就已经从统一配置格式中受益了。

## 平台化的三个常见陷阱

### 陷阱一：公约数陷阱

平台只支持所有项目的最小公共需求，每个项目都觉得不够用。

**避免方法**：平台提供扩展点（hook、plugin），项目可以在标准流程中插入自定义步骤。

### 陷阱二：无人维护陷阱

平台抽出来了但没有专人维护。半年后 Bug 没人修、新功能没人加，项目团队又开始自己写脚本。

**避免方法**：L3 的前提是"有专人维护"——如果没有这个条件，不要开始抽平台。

### 陷阱三：过度设计陷阱

平台设计了复杂的插件系统、配置继承、多级覆盖，但日常只用了最基础的功能。

**避免方法**：从实际需求驱动，不从架构美学驱动。先用最简单的方式支持当前需求，等需求真正出现再扩展。

## V18 系列总结

V18 五篇文章从五个维度覆盖了交付体系中"人和流程"的维度：

| 篇章 | 核心贡献 |
|------|---------|
| 01 角色与职责 | RACI 矩阵把模糊职责变成可执行边界 |
| 02 发布审批 | 审批清单 + 变更分级 + 紧急通道 |
| 03 多团队协作 | 分支策略 + CODEOWNERS + 发布列车 |
| 04 工程文化 | 质量/复盘/自动化三根文化支柱 |
| 05 从项目到平台 | 三阶段成熟度 + 渐进抽象路径 |

这五个维度的共同目标：**让交付体系不只是工具的集合，而是组织能力的体现**。工具可以拷贝，组织能力不能拷贝——这就是为什么同样的 CI/CD 工具在不同团队里效果天差地别。
