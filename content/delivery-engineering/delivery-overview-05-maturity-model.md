前四篇建立了交付的定义、链路全景、质量维度和飞轮模型。这些都是"应该怎么做"。这一篇回答的是"我们现在在哪里"。

成熟度模型不是用来打分的，而是用来定位的——知道自己在哪一级，才能知道下一步该投入在哪个环节。

## 五级成熟度

| 级别 | 名称 | 一句话特征 |
|------|------|-----------|
| L1 | 手动发布 | 能把包打出来，但全靠人盯人 |
| L2 | 可重复构建 | 构建自动化了，但验证还靠手动 |
| L3 | 质量门禁 | 关键环节有自动化检查，但没有完整闭环 |
| L4 | 交付管线 | 从构建到发布有完整的自动化管线和质量门 |
| L5 | 持续交付 | 交付是日常操作，飞轮在自转 |

### L1：手动发布

**特征**：
- 构建依赖特定的人或特定的机器
- 没有 CI，构建脚本是某个人电脑上的批处理文件
- 发布前的检查是一份 Word 文档里的清单，靠人逐项勾选
- 出了问题，定位靠"那个最熟悉这块的人"
- 回滚方案是"重新打上一个版本的包"，但没有人验证过这个包还能不能用

**最大风险**：关键人离开后，发布能力断裂。构建环境不可复现，同一份代码打出不同的包。

**改进优先级**：先把构建自动化——让任何人在任何机器上都能打出一致的包。

### L2：可重复构建

**特征**：
- 有 CI 服务器，构建可以一键触发
- 构建脚本版本化管理，构建环境有文档
- 同一份代码在不同机器上能打出一致的包
- 但验证仍然靠手动——QA 拿到包后按清单逐项测试
- 出了问题，修复后重新触发构建，但没有自动化回归

**最大风险**：验证成为瓶颈。每次发版前 QA 手动跑两天测试，开发等着。性能退化和兼容性问题在手动测试中很容易遗漏。

**改进优先级**：在构建管线中加入自动化检查——编译检查、资源规范检查、冒烟测试。不需要一步到位全自动化，先把最高频的问题自动化拦住。

### L3：质量门禁

**特征**：
- 构建后有自动化检查：编译通过、资源合规、包体大小、变体数量
- 冒烟测试自动化，关键路径在构建后自动跑
- 性能基线存在，可以检测明显的性能退化
- 但验证和发布之间没有正式的门禁流程——"差不多了就发"
- 线上监控有基础数据，但没有告警机制和事故响应流程
- 闭环靠人：出了问题修了就行，没有系统性的根因分析和防复发

**最大风险**：质量门是孤立的检查点，没有连成链路。门禁通过了但版本仍然出问题，因为门禁覆盖不全。线上问题的响应依赖个人判断，没有标准流程。

**改进优先级**：连接门禁和发布流程——让验证门和发布门成为正式的阶段转换条件，而不是"参考信息"。建立线上监控告警和事故响应 SLA。

### L4：交付管线

**特征**：
- 从构建到发布有完整的自动化管线
- 质量门、验证门、发布门三道门正式运作，不满足条件不允许推进
- 灰度发布策略明确，观察指标和扩量条件预定义
- 回滚预案经过验证，可以在分钟级执行
- 线上监控有完整的告警体系，关键指标异常自动触发响应
- 缺陷闭环有正式流程：分类 → 根因 → 修复 → 回归 → 防复发
- 多端构建和发布在同一个管线中处理

**最大风险**：管线高度依赖当前的工具和配置，变更管线本身的风险很高。改进速度开始放缓，因为"管线能跑了就不想动了"。

**改进优先级**：建立持续改进循环。每个版本的复盘产出应该被系统性地沉淀到管线中（新的检查规则、新的自动化脚本、新的告警规则），而不是停留在会议纪要里。

### L5：持续交付

**特征**：
- 发版是日常操作，不是"战役"
- 团队对发版的信心来自管线和数据，不是来自"这次特别仔细"
- 每次发版后的复盘产出会自动进入管线改进
- 交付效能指标被持续追踪：Lead Time、Deploy Frequency、MTTR、变更失败率
- 飞轮在自转——内环和外环都在加速，改进段持续产出

**这一级没有"最大风险"**，因为持续交付本身就是一种持续改进的状态。如果团队停止了改进，它就会逐渐退回 L4。

## 怎么用这个模型

### 自评方法

对照飞轮的八个环节，逐个评估当前状态：

| 环节 | L1 手动 | L2 可重复 | L3 有门禁 | L4 管线化 | L5 持续改进 |
|------|---------|----------|----------|----------|-----------|
| 生产 | 无规范 | 有规范但靠人盯 | 有自动化检查 | 门禁拦不合规产出 | 规范随项目演进自动更新 |
| 集成 | 无分支策略 | 有策略但手动合入 | 合入前有 CI 检查 | 合入门禁正式运作 | 冲突预防性解决 |
| 构建 | 手动打包 | CI 自动构建 | 构建后有质量检查 | 多端并行构建 | 构建时间持续优化 |
| 验证 | 人工全手动 | 有冒烟脚本 | 自动化 + 基线 | 四维度覆盖 | 验证策略自适应 |
| 发布 | 手动上传 | 脚本化上传 | 有灰度策略 | 管线化发布 | 按需发布 |
| 监控 | 无 | 有数据但不看 | 有看板 | 有告警 + SLA | 预测性告警 |
| 闭环 | 修了就行 | 有 Bug 追踪 | 有根因分析 | 有防复发机制 | 自动化防御 |
| 改进 | 无 | 偶尔复盘 | 定期复盘 | 复盘产出进管线 | 持续进化 |

团队的整体成熟度等于**短板环节的级别**，不等于各环节的平均值。一个构建达到 L4 但验证停留在 L1 的团队，整体交付能力是 L1——因为验证的缺失会让构建的优势完全失效。

### 改进路线

改进的原则是**补短板，不是拉长板**。

找到当前最弱的环节，把它提升一级。然后重新评估，找到新的最弱环节，再提升一级。每次只提升一个环节一级，不要试图一步到位。

典型的改进路径：

```
第一步：构建从 L1 → L2（自动化构建，让发版不依赖特定的人）
第二步：验证从 L1 → L2（加冒烟测试，让最基本的问题不漏出去）
第三步：集成从 L1 → L2（建立分支策略，减少合入冲突）
第四步：监控从 L1 → L2（接入 Crash 上报，至少知道出了什么问题）
第五步：各环节从 L2 → L3（在每个环节加入质量门禁）
...
```

每一步的投入通常是 2-4 周。10 步之后（大约 6-8 个月），一个 L1 团队可以达到整体 L3。从 L3 到 L4 通常需要更长时间，因为涉及流程设计和组织协调。

## 成熟度与本专栏的关系

本专栏 19 卷的内容，对应的成熟度级别大致分布：

| 级别 | 对应卷 | 重点 |
|------|--------|------|
| L1 → L2 | V03 工程基建、V07 多端构建 | 让构建可重复、可自动化 |
| L2 → L3 | V13 验证与测试、V14 性能与稳定性 | 加入质量门禁 |
| L3 → L4 | V16 CI/CD、V17 灰度上线 | 连成完整管线 |
| L4 → L5 | V15 缺陷闭环、V18 组织治理 | 建立持续改进循环 |
| 全级别 | V01 总论、V19 案例与模板 | 框架和实战参考 |
