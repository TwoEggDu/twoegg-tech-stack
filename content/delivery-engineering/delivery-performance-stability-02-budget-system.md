## 这篇解决什么问题

V14-01 讲了性能工程的四阶段循环，Budget 是第一步。但"定预算"不是拍一个帧率目标就完事——需要三个维度的拆解才能变成可执行的约束。这一篇讲预算怎么定、按什么维度拆、每个类目怎么分配。

## 三维预算矩阵

性能预算是三个维度的交叉：

```
维度一：系统（角色、场景、UI、特效、音频、脚本）
维度二：场景类型（大厅、战斗、过场、加载）
维度三：设备档位（高档、中档、低档）
```

**每个交叉点都有自己的预算值。** 比如"低档设备 + 战斗场景 + 角色系统"的 CPU 帧时间预算可能是 4ms。

不需要把每个交叉点都精确定义——先定最关键的（低档设备 + 战斗场景），再逐步覆盖。

## 预算类目

### 帧时间预算

帧时间 = CPU 帧时间 + GPU 帧时间（取较大值决定实际帧率）。

| 目标帧率 | 总帧时间预算 | CPU 预算 | GPU 预算 |
|---------|-------------|---------|---------|
| 60 FPS | 16.67ms | ≤14ms | ≤14ms |
| 30 FPS | 33.33ms | ≤28ms | ≤28ms |

帧时间预算的分配原则：留 2-3ms 余量给系统开销（垂直同步、线程切换、OS 调度）。

**按系统分配帧时间（战斗场景 30 FPS 为例）**：

| 系统 | CPU 预算 | 说明 |
|------|---------|------|
| 渲染提交 | 6ms | Draw Call 提交、裁剪、排序 |
| 角色逻辑 | 4ms | 动画、AI、技能 |
| 物理 | 3ms | 碰撞检测、物理模拟 |
| UI | 3ms | 布局、渲染 |
| 特效 | 3ms | 粒子系统、后处理 |
| 脚本逻辑 | 4ms | 游戏逻辑、网络、事件 |
| GC + 其他 | 3ms | 垃圾回收、系统开销 |
| **合计** | **26ms** | 留 2ms 余量 |

### 内存预算

内存预算的硬约束来自设备物理内存。超过系统可用内存的阈值就会触发 OOM。

| 类目 | 占比建议 | 说明 |
|------|---------|------|
| 纹理 | 40-60% | 最大单项，也是最易膨胀的 |
| Mesh | 10-20% | 模型顶点和索引 |
| 音频 | 5-10% | BGM 流式播放可控制峰值 |
| 脚本堆 | 10-20% | Mono/IL2CPP 托管堆 |
| Shader | 3-5% | 变体数量是关键因素 |
| 引擎内部 | 10-15% | RenderTexture、Buffer 等 |

**按设备档位设定内存上限**：

| 设备档位 | 物理内存 | 应用内存预算 | 说明 |
|---------|---------|------------|------|
| 高档 | ≥6 GB | ≤1.5 GB | 系统和其他 App 占用约 2-3 GB |
| 中档 | 4 GB | ≤1.0 GB | 中档设备最多的，预算最敏感 |
| 低档 | 2-3 GB | ≤600 MB | iOS 低内存设备 OOM 阈值很低 |

### 包体大小预算

包体大小直接影响下载转化率。V06 已覆盖包体分发的工程实践，这里定预算。

| 渠道 | 推荐首包大小 | 说明 |
|------|------------|------|
| iOS App Store | ≤200 MB | 超过 200 MB 需 Wi-Fi 下载 |
| Google Play | ≤150 MB（APK）/ AAB 拆分后更灵活 | 新兴市场网速慢 |
| 微信小游戏 | 首包 ≤20 MB，子包按需 | 即玩体验要求极小首包 |

### 加载时间预算

| 场景 | 预算 | 说明 |
|------|------|------|
| 冷启动到首屏 | ≤5s（高档）/ ≤8s（低档） | 首次体验直接影响留存 |
| 场景切换 | ≤3s | 频繁操作，用户容忍度低 |
| 热更资源下载 | 进度条 + 后台 | 不阻塞用户操作 |

## 预算推导方法

预算不是拍脑袋定的，是从设备能力倒推的：

```
1. 确定目标设备（最低支持到什么设备）
2. 确定目标帧率（低档 30 FPS / 高档 60 FPS）
3. 从帧率推导帧时间预算（30 FPS → 28ms 可用）
4. 从设备内存推导内存预算（低档 3 GB → 应用 ≤600 MB）
5. 从帧时间和内存总预算分配到各系统
6. 各系统 Owner 认领并确认可行性
7. 不可行的 → 调整分配或降低该系统复杂度
```

**关键步骤是第 6 步**——预算必须由执行者确认，否则就是一纸空文。

## 预算作为 CI 约束

预算定完不是存在文档里就结束。预算要变成 CI 可以检查的约束：

| 预算项 | CI 检查方式 | 拦截条件 |
|--------|-----------|---------|
| 包体大小 | Build Report 解析 | 超过阈值 → 标记失败 |
| 纹理内存 | 资源扫描工具 | 单张纹理超限 → 告警 |
| Shader 变体数 | 构建日志解析 | 变体总数超限 → 标记失败 |
| 帧时间 | 自动化性能跑测 | 帧时间超预算 → 标记失败 |
| 内存峰值 | 自动化内存跑测 | 峰值超预算 → 标记失败 |

V16 CI/CD 系列将完整覆盖性能预算在 CI 中的集成方法。这里先建立概念：**预算必须是可自动检查的，否则就是装饰品。**

## 小结与检查清单

- [ ] 性能预算是否按三个维度拆解（系统、场景、设备）
- [ ] 帧时间预算是否从目标帧率倒推并分配到各系统
- [ ] 内存预算是否按类目设定并区分设备档位
- [ ] 包体大小是否区分渠道设定上限
- [ ] 加载时间是否区分场景类型设定预算
- [ ] 预算是否由各系统 Owner 确认可行性
- [ ] 预算是否有 CI 自动检查机制（而非只存在文档中）
