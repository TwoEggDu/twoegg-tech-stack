## 这篇解决什么问题

很多团队的 Bug 分类只有"功能/性能/崩溃"三种标签——这是纯症状维度，能描述"出了什么问题"，但回答不了"为什么出问题"。结果就是：分类做了等于没做，对后续的根因分析和防复发没有帮助。本篇建立两维分类矩阵：症状维度描述现象，根因维度定位源头，两个维度交叉才能指导修复方向。

## 为什么单维度分类不够

| 分类方式 | 能回答 | 不能回答 | 后果 |
|---------|--------|---------|------|
| 只按症状 | "闪退了" | "为什么闪退？是代码问题还是资源问题？" | 每次都从头排查 |
| 只按根因 | "是代码逻辑错误" | "这个错误导致了什么后果？影响多大？" | 无法判断优先级 |
| 症状 × 根因 | "闪退（症状）+ 资源加载失败（根因）" | — | 同时知道影响和方向 |

## 症状维度：五类表现

| 症状类型 | 定义 | 典型表现 |
|---------|------|---------|
| 功能（Function） | 功能行为不符合预期 | 按钮无响应、数值计算错误、流程走不通 |
| 性能（Performance） | 运行效率不达标 | 帧率下降、加载超时、卡顿掉帧 |
| 稳定性（Stability） | 运行中断或异常退出 | 闪退、ANR、冻结、内存溢出 |
| 兼容性（Compatibility） | 特定环境下不正常 | 某设备黑屏、某系统版本崩溃、某分辨率 UI 错位 |
| 体验（Experience） | 功能正确但体验不好 | 动画卡顿、音效延迟、文案错误、UI 遮挡 |

## 根因维度：六类来源

| 根因类型 | 定义 | 典型情况 |
|---------|------|---------|
| 内容（Content） | 策划/美术/音频等内容数据问题 | 配表数值错误、文案写错、美术资源尺寸不对 |
| 资源（Resource） | 资源管线处理问题 | 图集打包错误、AB 依赖缺失、资源格式不兼容 |
| 配置（Config） | 构建/运行时配置问题 | 环境变量错误、Feature Flag 配错、服务器地址写错 |
| 代码（Code） | 程序逻辑错误 | 空引用、数组越界、竞态条件、逻辑分支遗漏 |
| 平台（Platform） | 平台 SDK/OS 层面问题 | iOS 审核规则变更、Android 权限变化、微信小游戏 API 限制 |
| 基础设施（Infra） | CI/CD/服务器/CDN 等基础设施问题 | 构建脚本报错、CDN 缓存未刷新、服务端配置未同步 |

## 5 x 6 分类矩阵

两个维度交叉形成 30 格矩阵。不是每个格子都常见，下表标注了游戏项目中的高频组合：

| | 内容 | 资源 | 配置 | 代码 | 平台 | 基础设施 |
|---|---|---|---|---|---|---|
| **功能** | 配表错误致功能异常 | - | Feature Flag 配错 | **逻辑 Bug（最高频）** | SDK 接口变更 | - |
| **性能** | 资源规格超标 | **打包策略不合理** | 画质配置未分档 | **算法效率问题** | 平台渲染差异 | CDN 下载慢 |
| **稳定性** | - | **资源加载失败致闪退** | - | **空引用/越界** | OS 版本不兼容 | 服务端超时 |
| **兼容性** | - | 资源格式不兼容 | 分辨率配置缺失 | 设备适配遗漏 | **GPU/驱动兼容问题** | - |
| **体验** | **文案/美术问题（高频）** | 加载顺序不合理 | - | 动画/音效时序错误 | 输入延迟差异 | - |

加粗格子是游戏项目中最常见的组合，团队应优先建立这些格子的快速排查模板。

## 优先级定义（P0-P3）

| 优先级 | 定义 | 响应 SLA | 修复 SLA | 典型场景 |
|--------|------|---------|---------|---------|
| P0 | 阻塞性：核心功能不可用或大面积影响 | 15 分钟内响应 | 4 小时内修复 | 全服闪退、支付失败、数据丢失 |
| P1 | 严重：重要功能异常，有临时绕行方案 | 1 小时内响应 | 24 小时内修复 | 某玩法无法正常进行、严重性能退化 |
| P2 | 一般：非核心功能异常或体验问题 | 4 小时内响应 | 下个版本修复 | UI 显示异常、非关键文案错误 |
| P3 | 轻微：不影响使用的小问题 | 下个工作日 | 排入 Backlog | 边缘 case、美观问题 |

优先级不是固定的。同一个 Bug 在不同时期优先级可能不同——大版本上线前夕，P2 问题可能升为 P1。

## 优先级判定流程

```
是否影响核心功能？
├── 是 → 是否大面积影响？
│       ├── 是 → P0
│       └── 否 → 有无绕行方案？
│               ├── 无 → P0
│               └── 有 → P1
└── 否 → 是否影响用户体验？
        ├── 明显影响 → P2
        └── 轻微/边缘 → P3
```

## 标准化 Bug 报告模板

每份 Bug 报告必须包含以下字段：

| 字段 | 必填 | 说明 |
|------|------|------|
| 标题 | 是 | 格式：[症状类型] 一句话描述现象 |
| 复现步骤 | 是 | 从启动到触发的完整步骤，越详细越好 |
| 预期结果 | 是 | 按设计应该发生什么 |
| 实际结果 | 是 | 实际发生了什么 |
| 复现率 | 是 | 必现 / 高概率 / 偶现 / 仅出现一次 |
| 症状分类 | 是 | 功能/性能/稳定性/兼容性/体验 |
| 根因分类 | 否 | 初报时可为空，根因分析后补填 |
| 优先级 | 是 | P0/P1/P2/P3 |
| 环境信息 | 是 | 设备型号、OS 版本、游戏版本、服务器环境 |
| 日志/截图 | 是 | 崩溃日志、截图、录屏中至少提供一项 |
| 关联构建号 | 是 | 出现问题的具体构建版本 |

## 分类的运营：定期回顾

分类矩阵不是建完就不管。每周或每版本做一次分类分布统计：

- 哪个症状维度的 Bug 最多？→ 反映当前质量短板
- 哪个根因维度的 Bug 最多？→ 反映哪个环节最容易出问题
- 哪个格子的 Bug 在增长？→ 预警信号
- P0/P1 的比例是否在下降？→ 闭环效果的直接体现

## 小结

- [ ] 团队是否使用两维分类而不是只按症状分类
- [ ] 5 × 6 矩阵中的高频格子是否有快速排查模板
- [ ] P0-P3 优先级是否有明确的定义和响应 SLA
- [ ] Bug 报告是否有标准化模板并严格执行
- [ ] 是否定期统计分类分布并识别趋势
