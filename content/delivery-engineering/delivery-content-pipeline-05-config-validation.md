## 这篇解决什么问题

上一篇画出了配置管线的六个环节。其中"验证"是最关键也最容易被跳过的环节。

这一篇把验证拆成四层，说清每层检查什么、在什么时机执行、能拦住什么类型的错误。

## 为什么这个问题重要

配置错误的修复成本随传播距离指数级增长：

| 发现时机 | 修复成本 | 影响范围 |
|---------|---------|---------|
| 填表时（编辑器提示） | 1x | 策划自己改 |
| 导表时（导出工具报错） | 2x | 策划改表 + 重新导出 |
| 构建时（CI 检查报错） | 5x | 策划改表 + 重新导出 + 重新构建 |
| 测试时（QA 发现异常） | 20x | 定位 + 策划改表 + 重新导出 + 重新构建 + 重新验证 |
| 上线后（用户投诉） | 100x+ | 定位 + 修复 + 紧急导表 + 热更新 + 事故复盘 |

从 1x 到 100x 的差距，就是验证环节存在的意义。

## 四层验证模型

### 第一层：类型检查

**检查什么**：每个字段的值是否符合 Schema 定义的数据类型。

| Schema 定义 | 合法值 | 非法值 |
|------------|--------|--------|
| int | 100 | "一百"、3.14、空 |
| float | 3.14 | "三点一四"、空 |
| string | "warrior" | —（字符串几乎任何值都合法，需要配合枚举限制） |
| enum[物理,魔法,真实] | "物理" | "混合"（不在枚举列表中） |
| ref:ItemTable.id | 10001（存在于 ItemTable） | 99999（不存在） |

类型检查是最基础的一层，应该在**导出时**就执行——策划点击"导出"按钮的那一刻就报错，不需要等到 CI。

实施要点：
- Schema 定义必须和配置表绑定（表头注释或独立的 Schema 文件）
- 类型检查失败应该阻止导出，不是只报警告
- 错误信息必须包含：哪张表、哪一行、哪一列、期望什么类型、实际填了什么

### 第二层：范围检查

**检查什么**：数值是否在业务合理的范围内。

类型检查能确认"这个字段是整数"，但不能确认"这个整数在合理范围内"。范围检查补上这一层：

| 字段 | 类型 | 范围约束 | 理由 |
|------|------|---------|------|
| 攻击力 | int | [1, 99999] | 不能为 0 或负数 |
| 暴击率 | float | [0.0, 1.0] | 概率不能超过 100% |
| 冷却时间 | float | [0.1, 600.0] | 不能为 0（无限释放），不能超过 10 分钟 |
| 掉落概率 | float | [0.0, 1.0] | 概率约束 |
| 物品堆叠上限 | int | [1, 9999] | 不能为 0 |
| 价格 | int | [0, 999999] | 0 表示免费，负数非法 |

范围约束应该定义在 Schema 中，和类型检查一起在导出时执行。

高级用法：**条件范围**。某些字段的合理范围取决于其他字段的值。例如"技能伤害类型为百分比时，伤害值范围是 [0.01, 10.0]；为固定值时，范围是 [1, 99999]"。条件范围的表达能力取决于 Schema 语言的设计。

### 第三层：引用完整性

**检查什么**：配置中引用的外部实体是否存在。

游戏配置表之间有大量的引用关系：

```
技能表.特效ID  → 引用 → 特效表.id
掉落表.物品ID  → 引用 → 物品表.id
物品表.图标    → 引用 → 资源清单中的图标路径
关卡表.怪物ID  → 引用 → 怪物表.id
```

引用完整性检查要确保：
- 每一个引用目标都存在（不存在 = 运行时一定报错）
- 被引用的记录没有被标记为"已删除"或"已废弃"
- 引用链没有形成环（A 引用 B，B 引用 A）

引用完整性检查的执行时机应该在**所有表导出完成后**——因为单张表导出时，它引用的其他表可能还没有导出。

实施要点：
- 需要一个全局的引用关系图（哪张表的哪个字段引用了哪张表的哪个字段）
- 引用关系图应该从 Schema 中自动提取，不应该手动维护
- 引用缺失应该是**阻止级**错误——因为运行时一定会崩溃或异常

### 第四层：跨表一致性

**检查什么**：多张表之间的数据是否逻辑一致。

这是最复杂的一层，因为它需要业务知识：

**掉落表与物品表**：掉落表引用的所有物品 ID 是否都在物品表中存在，并且物品的状态是"已上线"而不是"草稿"。

**技能表与动画表**：技能配置引用的动画 ID 是否存在，动画时长是否和技能的施法时间匹配。

**商城表与物品表**：商城上架的物品是否都有定价，价格是否和物品稀有度匹配。

**关卡表与怪物表**：关卡引用的怪物 ID 是否存在，怪物的等级范围是否和关卡推荐等级匹配。

跨表一致性检查通常需要自定义规则，无法完全自动化推导。建议的做法：

```
# 规则示例（伪代码）
rule "掉落物品必须已上线":
  for each row in DropTable:
    item = ItemTable.find(row.item_id)
    assert item.status == "online", "掉落表引用了未上线物品"
```

将规则写成可声明的脚本，在导表完成后统一执行。

## 验证在什么时机执行

| 验证层 | 执行时机 | 耗时 | 阻止级别 |
|--------|---------|------|---------|
| 类型检查 | 导出时（策划点击导出） | 秒级 | 阻止导出 |
| 范围检查 | 导出时 | 秒级 | 阻止导出 |
| 引用完整性 | 全部表导出后 / CI | 秒-分钟级 | 阻止构建 |
| 跨表一致性 | CI | 分钟级 | 阻止构建或警告 |

原则：**越简单的检查越早执行，越复杂的检查越靠后但不能没有。**

## 常见错误做法

**只做类型检查，不做引用完整性**。类型检查只能保证"填的是数字"，不能保证"这个数字对应的东西存在"。引用断裂是配置错误中最高频的类型。

**验证规则硬编码在导表工具里**。随着项目演进，验证规则需要频繁调整。如果规则硬编码在工具里，每次调整都需要程序修改工具。验证规则应该是可声明的（配置文件或脚本），策划主管也能调整。

**验证结果只在控制台打印**。策划看不到控制台输出，或者输出太多找不到关键信息。验证结果应该以结构化报告的形式输出（HTML / 文本摘要），标注问题的表名、行号、字段名和期望值。

## 小结与检查清单

- [ ] 是否有明确的 Schema 定义（类型 + 范围 + 引用关系）
- [ ] 类型检查和范围检查是否在导出时就执行
- [ ] 引用完整性检查是否在全部表导出后自动执行
- [ ] 是否有至少 3-5 条跨表一致性规则
- [ ] 验证结果是否以结构化报告形式输出
- [ ] 验证规则是否可声明（不硬编码在工具里）
- [ ] 引用缺失是否为阻止级错误
