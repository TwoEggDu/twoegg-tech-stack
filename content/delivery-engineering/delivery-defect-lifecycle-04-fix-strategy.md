## 这篇解决什么问题

根因找到了，怎么修？很多开发者修 Bug 时会"顺手"重构周边代码、优化命名、调整架构——在日常开发中这是好习惯，在 hotfix 中这是灾难。修复策略的核心是控制风险：用最小范围修复问题，选对代码合并方式，确认修复在所有目标平台生效。

## 最小修复范围原则

| 规则 | 说明 | 反例 |
|------|------|------|
| 只改导致问题的代码 | 修复不是重构的时机 | 修一个空引用，顺手重命名了 5 个变量 |
| 不改不相关的文件 | 减少 diff 范围，降低回归风险 | 修配表校验，顺手改了 UI 布局 |
| 不升级依赖版本 | 依赖升级可能引入未知变化 | 修一个 Bug 时升级了第三方 SDK |
| 不调整架构 | 架构变更影响面太大 | 修一个性能问题时重构了加载管线 |
| 保持代码风格一致 | 不因修复引入格式差异 | hotfix 里改了缩进规则 |

### 为什么最小范围如此重要

hotfix 的修改越多，回归范围就越大：

| 修改范围 | 回归范围 | 验证耗时 | 引入新问题的概率 |
|---------|---------|---------|----------------|
| 1 个文件 / < 10 行 | 修改点 + 直接关联 | 1-2 小时 | 低 |
| 3-5 个文件 / 30-50 行 | 受影响模块 | 4-8 小时 | 中 |
| 10+ 个文件 / 100+ 行 | 接近全量回归 | 1-2 天 | 高 |

P0 问题要求 4 小时内修复——如果修改范围大到需要 1 天的回归，SLA 就不可能达标。

## Cherry-Pick vs Full Merge

修复完成后，代码怎么合入发布分支？两种方式各有适用场景。

### 策略对比

| 维度 | Cherry-Pick | Full Merge |
|------|------------|------------|
| 操作 | 从开发分支挑选特定 commit 合入发布分支 | 将开发分支整体合入发布分支 |
| 适用 | 线上 hotfix，只需要特定修复 | 计划内的版本发布 |
| 风险 | 可能遗漏依赖的 commit | 可能带入未经验证的变更 |
| 合并冲突 | 少（改动小） | 可能多（累积变更） |
| 追溯性 | commit hash 变了，需要标注来源 | 保持完整的 commit 历史 |

### Cherry-Pick 的操作规范

1. **在开发分支上先修复**：不要直接在发布分支上改代码
2. **验证通过后再 Cherry-Pick**：确认修复在开发分支上验证通过
3. **Cherry-Pick 后在发布分支上再验证一次**：环境差异可能导致不同表现
4. **标注来源**：在 commit message 中注明原始 commit hash
5. **检查依赖**：确认被 Cherry-Pick 的 commit 不依赖其他未合入的 commit

### 什么时候不该用 Cherry-Pick

| 场景 | 原因 | 替代方案 |
|------|------|---------|
| 修复涉及 5+ 个 commit | 依赖关系复杂，容易遗漏 | 评估是否可以做 Full Merge |
| 修复依赖新增的 API | Cherry-Pick 会因缺少 API 编译失败 | 将 API 新增也一起 Cherry-Pick |
| 发布分支和开发分支差异过大 | 代码上下文不同，可能行为不一致 | 在发布分支上单独修复 |

## Hotfix 分支策略

| 分支类型 | 从哪里拉 | 合入哪里 | 生命周期 |
|---------|---------|---------|---------|
| hotfix/xxx | 发布分支 | 发布分支 + 开发主干 | 修复验证后删除 |
| bugfix/xxx | 开发主干 | 开发主干 | 随版本发布 |

Hotfix 分支的关键规则：修复合入发布分支后，**必须同步合入开发主干**。否则下个版本会把这个 Bug 重新带上线。

```
开发主干 ─────────────────────────────
                    ↑ 合入
发布分支 ──┬───── hotfix/xxx ──┬───→ 发布
           └── 拉分支          └── 合入
```

## 三端覆盖确认

游戏项目通常有多个目标平台。修复一个 Bug 时必须确认：

| 确认项 | 问题 | 处理 |
|--------|------|------|
| 是否三端共有 | 这个 Bug 在 iOS/Android/微信小游戏 上都存在吗？ | 逐端验证 |
| 修复是否三端通用 | 修复代码在三个平台上都能编译和运行吗？ | 检查平台宏、条件编译 |
| 是否有平台特有逻辑 | 修复是否需要针对某个平台做特殊处理？ | 加平台分支 |
| 修复渠道是否一致 | 三个平台的修复通过什么方式送达？ | 热更/版本更新/配置推送 |

### 修复交付渠道选择

| 渠道 | 适用场景 | 到达速度 | 限制 |
|------|---------|---------|------|
| 热更新（代码） | 代码逻辑修复 | 小时级 | 受 HybridCLR/DHE 支持范围限制 |
| 热更新（资源） | 资源/配表修复 | 小时级 | AB 依赖关系需要正确 |
| 配置推送 | Feature Flag/服务端配置 | 分钟级 | 只能改配置支持的参数 |
| 版本更新 | 原生代码修复 | 天级（需审核） | iOS 审核 1-3 天 |
| 服务端修复 | 纯服务端问题 | 分钟级 | 需要服务端灰度验证 |

选择原则：**能用配置推送解决的不走热更新，能用热更新解决的不走版本更新**。渠道越轻量，到达用户越快，风险越可控。

## 修复验证清单

修复完成后、合入分支前的验证清单：

| 验证项 | 方法 | 通过标准 |
|--------|------|---------|
| 问题是否修复 | 按原始复现步骤验证 | 问题不再出现 |
| 修改范围是否最小 | Code Review 检查 diff | 没有不相关的修改 |
| 编译是否通过 | 三端编译 | 无编译错误 |
| 自动化测试是否通过 | 跑 CI | 无新增失败 |
| 是否需要同步合入主干 | 检查分支策略 | hotfix 必须同步 |
| 是否需要更新版本号 | 检查发布规范 | 热更需要更新资源版本号 |

## 紧急修复的时间分配

P0 问题的 4 小时 SLA 怎么分配：

| 阶段 | 时间 | 占比 |
|------|------|------|
| 定位根因 | 60-90 分钟 | 30-40% |
| 编写修复 | 30-60 分钟 | 15-25% |
| 验证修复 | 60-90 分钟 | 30-40% |
| 发布上线 | 30-60 分钟 | 15-25% |

如果定位超过 90 分钟还没找到根因，考虑先用临时方案止血（关闭功能/回滚版本），再从容排查。

## 小结

- [ ] hotfix 是否严格遵守最小修复范围原则
- [ ] 团队是否有明确的 Cherry-Pick vs Full Merge 选择标准
- [ ] hotfix 是否同步合入了开发主干
- [ ] 三端覆盖确认是否纳入修复流程
- [ ] 修复交付渠道是否按"最轻量优先"原则选择
- [ ] P0 修复是否有时间分配指导
