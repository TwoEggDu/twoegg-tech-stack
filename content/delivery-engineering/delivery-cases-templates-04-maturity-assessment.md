## 这篇解决什么问题

V01-05 介绍了五级成熟度模型的概念。本篇把它展开为可实际操作的评估工具——8 个交付阶段 x 5 个成熟度级别的完整矩阵，配合自评方法和改进路线图模板。这也是交付工程专栏 19 卷的完结篇——最后一节回顾全部内容，给出使用指南。

## 成熟度评估矩阵

### 五级定义回顾

| 级别 | 名称 | 一句话特征 |
|------|------|-----------|
| L1 | 手动发布 | 能把包打出来，但全靠人盯人 |
| L2 | 可重复构建 | 构建自动化了，但验证还靠手动 |
| L3 | 质量门禁 | 关键环节有自动化检查，但没有完整闭环 |
| L4 | 交付管线 | 从构建到发布有完整的自动化管线和质量门 |
| L5 | 持续交付 | 交付是日常操作，飞轮在自转 |

### 八个交付阶段

| 阶段 | 覆盖内容 | 对应 Volume |
|------|---------|------------|
| 1. 版本管理 | 分支策略、版本号、环境管理 | V04 |
| 2. 构建 | 构建自动化、环境一致性、构建优化 | V07 |
| 3. 资源管线 | AB 打包、资源验证、CDN 分发 | V05, V06 |
| 4. 验证与测试 | 功能测试、性能测试、兼容性测试 | V13 |
| 5. 质量门 | 自动化检查、阈值管理、门禁决策 | V13, V16 |
| 6. 发布与灰度 | 灰度策略、审批流程、回滚机制 | V09, V17, V18 |
| 7. 监控与应急 | 线上监控、告警、热修复、回滚 | V11, V17 |
| 8. 复盘与改进 | Postmortem、效能度量、持续改进 | V15, V17, V18 |

### 完整评估矩阵

#### 阶段 1：版本管理

| 级别 | 特征 |
|------|------|
| L1 | 没有统一的分支策略；版本号由人手动维护；没有环境区分 |
| L2 | 有约定的分支策略（如 Git Flow）；版本号有规则但手动更新 |
| L3 | 分支策略强制执行（Branch Protection）；版本号自动生成 |
| L4 | 多环境管理（dev/staging/prod）完善；Feature Flag 机制就绪 |
| L5 | 分支策略适配多团队；版本协调自动化；环境按需创建和销毁 |

#### 阶段 2：构建

| 级别 | 特征 |
|------|------|
| L1 | 构建在开发者本机执行；构建环境因人而异 |
| L2 | 有 CI 服务器；构建脚本版本化；同代码可出一致的包 |
| L3 | 构建环境 Docker 化；构建失败自动通知并路由到对应角色 |
| L4 | 增量构建优化；构建缓存；多平台并行构建 |
| L5 | 构建时间在预算内；构建异常自动重试和自愈；构建指标持续监控 |

#### 阶段 3：资源管线

| 级别 | 特征 |
|------|------|
| L1 | 资源手动打包；没有资源规范检查 |
| L2 | AB 打包自动化；有基本的资源命名规范 |
| L3 | 资源规范自动化检查（贴图/面数/命名）；CDN 分发自动化 |
| L4 | AB 依赖分析自动化；包体大小监控和预警；热更新管线完整 |
| L5 | 资源管线与构建管线完全集成；跨项目资源标准统一 |

#### 阶段 4：验证与测试

| 级别 | 特征 |
|------|------|
| L1 | 全手动测试；没有测试用例管理 |
| L2 | 有单元测试但覆盖率低；手动功能测试有测试用例 |
| L3 | 单元测试集成到 CI；自动化功能测试覆盖核心路径 |
| L4 | 性能测试自动化；兼容性测试覆盖目标设备矩阵 |
| L5 | 测试覆盖率持续提升；测试结果自动关联到质量门决策 |

#### 阶段 5：质量门

| 级别 | 特征 |
|------|------|
| L1 | 没有质量门；构建成功就可以发布 |
| L2 | 有手动检查清单；关键指标有人看但无阈值 |
| L3 | 关键指标有自动化检查和阈值；不通过阻塞发布 |
| L4 | 质量门覆盖功能/性能/稳定性/资源多维度；门禁决策自动化 |
| L5 | 质量门阈值动态调整（基于历史数据）；跨项目质量基线统一 |

#### 阶段 6：发布与灰度

| 级别 | 特征 |
|------|------|
| L1 | 直接全量发布；没有灰度；回滚靠重新发包 |
| L2 | 有灰度意识但执行不规范；审批靠 IM 消息确认 |
| L3 | 灰度分阶段放量；有审批流程和清单；回滚方案已文档化 |
| L4 | 灰度指标自动监控；审批嵌入 CI/CD 管线；回滚已验证 |
| L5 | 灰度决策半自动化；发布是低风险的日常操作；紧急通道有完善流程 |

#### 阶段 7：监控与应急

| 级别 | 特征 |
|------|------|
| L1 | 没有线上监控；出问题靠用户反馈 |
| L2 | 有基本 Crash 监控；但没有告警规则 |
| L3 | Crash/ANR/性能指标有监控和告警；有 On-call 机制 |
| L4 | 监控按设备分档；告警路由到对应角色；热修复通道就绪 |
| L5 | 监控覆盖全链路；异常模式自动识别；应急响应 SLA 有保障 |

#### 阶段 8：复盘与改进

| 级别 | 特征 |
|------|------|
| L1 | 没有复盘；出了问题修完就结束 |
| L2 | 偶尔开复盘会；但没有标准格式，Action Item 不跟踪 |
| L3 | 每次事故都有 Postmortem；AI 有 Owner 和 Deadline |
| L4 | AI 完成率 > 80%；交付效能指标有度量；复发率有统计 |
| L5 | 复盘驱动持续改进；效能指标持续优化；改进项自动纳入 Sprint |

## 自评方法

### Step 1：逐阶段打分

对照上面的矩阵，为团队在每个阶段评定当前级别。打分规则：

- 一个阶段的级别 = 该阶段**稳定达到**的最高级别
- "稳定达到"意味着不是偶尔做到，而是每次发布都能做到
- 如果部分特征达到 L3、部分还在 L2，记为 L2（取下限）

### Step 2：填写评估表

| 阶段 | 当前级别 | 达标依据 | 缺失项 |
|------|---------|---------|--------|
| 1. 版本管理 | L? | 支撑判断的事实 | 达到下一级缺什么 |
| 2. 构建 | L? | | |
| 3. 资源管线 | L? | | |
| 4. 验证与测试 | L? | | |
| 5. 质量门 | L? | | |
| 6. 发布与灰度 | L? | | |
| 7. 监控与应急 | L? | | |
| 8. 复盘与改进 | L? | | |

### Step 3：计算总体级别

**总体级别 = 8 个阶段中的最低级别**（木桶原理）。

交付链路是串联的——构建 L4 但监控 L1，总体效果约等于 L1。交付效能受限于最薄弱的环节。

### Step 4：识别优先改进项

按以下优先级排序：

1. **最低分阶段**——它拖了整体后腿
2. **同分中影响面最大的阶段**——构建和发布阶段的权重通常高于其他
3. **投入产出比最高的阶段**——从 L1 提升到 L2 通常比从 L3 提升到 L4 性价比更高

## 改进路线图模板

### 模板字段

| 字段 | 说明 |
|------|------|
| 当前总体级别 | 评估得出的当前级别 |
| 目标级别 | 计划达到的级别 |
| 目标时间 | 计划达到目标级别的截止日期 |
| 优先改进阶段 | 最需要提升的阶段（通常 2-3 个） |
| 每阶段改进计划 | 具体的改进动作、Owner、Milestone |
| 资源需求 | 需要的人力、工具、预算 |
| 验收标准 | 怎么判断改进完成 |

### 填写示例

```markdown
# 交付成熟度改进路线图

## 现状
- 当前总体级别：L2
- 瓶颈阶段：质量门（L1）、监控与应急（L1）
- 相对优势：构建（L3）、版本管理（L3）

## 目标
- 目标级别：L3
- 目标时间：2026-10-01（6 个月）

## 改进计划

### Phase 1：质量门从 L1 → L3（Month 1-3）
| 改进项 | Owner | Milestone | 验收标准 |
|--------|-------|-----------|---------|
| 定义核心指标的阈值 | QA Lead | M1 | 至少 5 项指标有明确阈值 |
| 将质量检查集成到 CI | Toolchain | M2 | CI 构建自动执行质量检查 |
| 质量门阻塞发布 | Release Mgr | M3 | 不通过质量门的包无法进入发布流程 |

### Phase 2：监控从 L1 → L3（Month 2-4）
| 改进项 | Owner | Milestone | 验收标准 |
|--------|-------|-----------|---------|
| 接入 Crash 监控 SDK | Dev Lead | M2 | Crash 数据可在看板上查看 |
| 配置告警规则 | Ops | M3 | Crash 率异常自动告警到 On-call |
| 建立 On-call 机制 | Release Mgr | M4 | 每次发布有指定 On-call 人员 |

### Phase 3：巩固与验证（Month 5-6）
| 改进项 | Owner | Milestone | 验收标准 |
|--------|-------|-----------|---------|
| 全流程走一次完整的 L3 交付 | Release Mgr | M5 | 从构建到复盘的完整闭环 |
| 重新评估成熟度 | Tech Lead | M6 | 所有阶段 ≥ L3 |

## 资源需求
- Toolchain 工程师：1 人 x 3 个月（CI 集成和质量门实现）
- 第三方监控服务：每月约 ¥XXX
- 团队培训：2 次（质量门使用 + 监控告警响应）
```

## 评估节奏建议

| 频率 | 目的 |
|------|------|
| 首次评估 | 建立基线，确定改进方向 |
| 每季度 | 跟踪改进进展，更新路线图 |

### 五条阅读路径使用指南

V01-06 定义了五条阅读路径，这里给出每条路径的最终推荐阅读顺序：

#### L1 全景线——适合所有角色的全局认知

从 V01 总论开始，按 Volume 顺序通读每个系列的索引页和第一篇文章。目标是建立交付链路的完整心智模型，不深入细节。

**推荐阅读量**：19 个索引页 + 19 篇首篇 = 约 38 篇

#### L2 工程线——适合开发和工具链工程师

聚焦技术实现：V03 工程基础 → V05 资源管线 → V07 构建 → V08 热更 → V13 验证 → V16 CI/CD。

**推荐阅读量**：约 40 篇核心技术文章

#### L3 深度线——适合想深入特定领域的读者

选择当前工作最相关的 2-3 个系列深读全部文章。例如做 AB 管线的人深读 V05+V06，做 CI/CD 的人深读 V07+V16。

**推荐阅读量**：按需，每个系列 5-7 篇

#### L4 管理线——适合技术负责人和项目经理

聚焦流程和协作：V01 总论 → V04 版本管理 → V09 平台上架 → V17 灰度上线 → V18 组织治理 → V19 成熟度评估。

**推荐阅读量**：约 25 篇管理和流程文章

#### L5 速查线——适合遇到具体问题时查阅

不需要按顺序读。遇到问题时，定位到对应 Volume 的对应文章，查阅检查清单和操作步骤。V19 的四篇文章本身就是速查工具。

**推荐阅读量**：按需，每次 1-2 篇

### 专栏的使用建议

1. **新团队**：从 V01 总论 + V19-04 成熟度评估开始——先知道自己在哪，再决定往哪走
2. **改进中的团队**：根据成熟度评估的瓶颈阶段，定位到对应 Volume 深读
3. **已有成熟流程的团队**：用 V19-02 检查清单对照现有流程，找漏洞
4. **新成员入职**：读 L1 全景线的索引页 + V19-01 事故案例集，快速建立认知

### 写在最后

交付工程的终极目标不是完美的流程或漂亮的管线，而是**让团队有信心说"这个版本可以上线"**——这个信心不是来自某个人的直觉，而是来自系统化的验证、自动化的检查、可量化的指标。

19 卷专栏覆盖了从概念到实践的完整路径。但纸上得来终觉浅——每个团队的项目特征、人员结构、技术栈都不同，最终的交付体系一定是根据自身情况裁剪和迭代的产物。

起步不需要完美——L1 到 L2 的跨越，往往只需要一个 CI 服务器和一个构建脚本。
