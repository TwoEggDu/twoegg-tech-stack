## 这篇解决什么问题

线上出了问题，回滚太重（丢失所有新功能）、配置回退不够（问题不在功能开关覆盖范围）——需要紧急推一个修复。Hotfix 不是"特别快的版本发布"，而是一套有约束的紧急流程：范围限制、加速验证、独立分支、修复后必须回流。搞错了任何一步，Hotfix 本身会成为新问题的来源。

## Hotfix vs 常规发布

| 维度 | 常规发布 | Hotfix |
|------|---------|--------|
| **范围** | 包含所有计划功能和修复 | 仅包含目标问题的最小修复 |
| **分支来源** | develop 分支 | release tag 或 release 分支 |
| **验证深度** | 完整回归测试 | Smoke + 目标修复验证 + 发布配置检查 |
| **审批流程** | 标准审批链 | 加速审批（但不能跳过） |
| **灰度流程** | 标准四阶段放量 | 压缩灰度（种子期可缩短到 4~12h） |
| **后续动作** | 合入 develop，常规迭代 | 修复必须 Cherry-pick 回 develop |

**核心约束**：Hotfix 只修目标问题——"顺便"加功能、"顺便"重构是绝对禁止的。范围越小，风险越低，验证越快。

## Hotfix 分支工作流

```
main/release ─── tag v2.3.1 ──────────────────────────── tag v2.3.2
                    │                                        │
                    └── hotfix/v2.3.2 ── fix commit ── merge ┘
                                              │
                                              └── cherry-pick → develop
```

**步骤说明**：

| 步骤 | 操作 | 要点 |
|------|------|------|
| 1 | 从 release tag 创建 hotfix 分支 | 分支命名：`hotfix/vX.Y.Z`，基于已发布的版本 |
| 2 | 在 hotfix 分支上提交修复 | 仅提交目标问题的最小修复代码 |
| 3 | 在 hotfix 分支上运行加速验证 | Smoke + 修复验证 + 发布配置检查（下文详述） |
| 4 | 合入 main/release 并打 tag | 新版本号递增 patch（v2.3.1 → v2.3.2） |
| 5 | Cherry-pick 回 develop | 确保修复在下一个常规版本中也存在 |
| 6 | 删除 hotfix 分支 | 保持分支卫生 |

**Cherry-pick 冲突处理**：如果 cherry-pick 回 develop 时有冲突，说明 develop 上的代码结构已经变了。此时不能强行合并——需要在 develop 上单独修复同一问题（可能是不同的代码修改），然后创建 follow-up ticket 追踪。

## 加速验证通道

Hotfix 的验证不能跳过，但可以压缩。

### 验证范围

| 验证项 | 常规发布 | Hotfix | 说明 |
|--------|---------|--------|------|
| 冒烟测试（Smoke） | 全量 | 全量 | 确保基本功能不坏，不能省 |
| 修复目标验证 | - | 针对性用例 | 确认目标问题已修复 |
| 回归测试 | 完整回归 | 仅修改涉及模块 | 范围收缩到变更影响面 |
| 发布配置检查 | 标准流程 | 标准流程 | 版本号、资源版本、服务端兼容性 |
| 性能测试 | 完整基准 | 可跳过（除非修改涉及性能） | 最小修复通常不影响性能 |
| 兼容性测试 | 多机型覆盖 | Top 3 机型 | 覆盖范围缩小但不能为零 |

### 验证时间预期

| 验证阶段 | 常规发布 | Hotfix |
|---------|---------|--------|
| CI 构建 | 30~60 min | 30~60 min（无法压缩） |
| 自动化测试 | 2~4h | 30~60 min（仅跑相关模块） |
| 人工验证 | 4~8h | 1~2h（Smoke + 修复验证） |
| 总计 | 6~12h | 2~4h |

**底线**：再紧急的 Hotfix 也不能跳过 Smoke 测试。"修了一个 Crash 引入了另一个 Crash"比不修还糟糕。

## Hotfix 交付渠道选择

| 渠道 | 速度 | 覆盖率 | 适用场景 |
|------|------|--------|---------|
| 热更新（Lua/脚本层） | 快（用户下次启动生效） | 高（自动下载） | 逻辑 Bug、数值错误 |
| 热更新（资源层） | 快（用户下次启动生效） | 高（自动下载） | 资源引用错误、配置表问题 |
| 热更新（IL2CPP/Native） | 中（依赖平台限制） | 中 | C# 代码层 Bug（HybridCLR 等方案） |
| 应用商店紧急提审 | 慢（iOS 1~3 天，Android 数小时） | 渐进式 | Native 层 Bug、SDK 问题 |
| 服务端修复 | 最快（无需客户端更新） | 即时 | 服务端逻辑 Bug、数据修复 |

**优先级**：能服务端修就不推客户端，能热更新就不走应用商店。

## Hotfix 审批加速

紧急不等于无审批——但审批链可以压缩：

| 常规审批 | Hotfix 审批 |
|---------|------------|
| 开发提交 → QA 负责人 → 技术主管 → 发布负责人 | 开发提交 → 技术主管（QA 并行验证） → 发布负责人 |
| 异步评审，可等到工作时间 | 同步评审，随时响应 |
| 标准 MR 模板 | 简化模板（问题描述 + 修复内容 + 影响范围） |

**记录要求**：加速审批不代表不记录——事后必须补充完整的 MR 描述和审批记录。

## Hotfix 后的收尾

Hotfix 完成不是终点——以下收尾动作不能省：

- [ ] Cherry-pick 已合入 develop，下一个常规版本包含此修复
- [ ] 如果 Hotfix 是临时方案（workaround），已创建 follow-up ticket 追踪正式修复
- [ ] Hotfix 的监控数据已确认——修复确实生效，没有引入新问题
- [ ] 事故工单已更新——Hotfix 的时间线、修复内容、验证结果已记录
- [ ] 回归测试用例已补充——导致此 Hotfix 的场景需要加入常规回归用例库
- [ ] 复盘已安排——Hotfix 频率是交付健康度的重要信号（V17-05 详述）

## Hotfix 频率是一个信号

如果团队频繁发 Hotfix（每个版本都有 1~2 次），说明上游质量有问题：

| Hotfix 频率 | 可能原因 | 改进方向 |
|------------|---------|---------|
| 每版本 0~1 次 | 正常范围 | 持续保持 |
| 每版本 2~3 次 | 测试覆盖不足或发布节奏太快 | 增加自动化测试、延长灰度观察期 |
| 每版本 4+ 次 | 质量体系有系统性问题 | 停下来做根因分析，可能需要减慢发布节奏 |
