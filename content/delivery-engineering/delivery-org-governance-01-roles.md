## 这篇解决什么问题

交付出了问题，最常见的场景不是"没人能做"，而是"每个人都觉得该别人做"。版本延期了，开发说"QA 没测完"，QA 说"构建没给我"，构建说"没人告诉我该打包了"。根源是职责边界不清——RACI 矩阵是把模糊职责变成明确契约的最小工具。

## 交付链路上的六个核心角色

| 角色 | 核心职责 | 典型岗位 |
|------|---------|---------|
| 开发（Dev） | 编写代码、修复缺陷、代码评审 | 客户端程序、服务端程序 |
| 测试（QA） | 验证功能、回归测试、报告缺陷 | 测试工程师、QA Lead |
| 技术美术（TA） | 资源规范制定、性能预算监控、管线工具开发 | Technical Artist |
| 工具链工程师（Toolchain） | CI/CD 管线维护、构建系统、自动化脚本 | DevOps、构建工程师 |
| 发布负责人（Release Manager） | 发布节奏规划、灰度策略、审批协调 | 版本经理、制作人 |
| 运维（Ops） | 服务部署、监控告警、线上应急 | SRE、运维工程师 |

**注意**：这六个角色是交付链路的逻辑角色，不是组织架构的岗位。一个人可能身兼多角色（小团队里主程可能同时是 Toolchain + Release Manager），但每个角色的职责必须有人明确承担。

## RACI 矩阵

RACI 四个字母的定义：

- **R — Responsible（执行者）**：实际干活的人
- **A — Accountable（负责人）**：最终为结果负责、有审批权的人，每个任务只能有一个 A
- **C — Consulted（被咨询者）**：执行前需要征求意见的人
- **I — Informed（被通知者）**：执行后需要知会的人

### 交付各阶段的 RACI

| 交付阶段 | Dev | QA | TA | Toolchain | Release Mgr | Ops |
|---------|-----|----|----|-----------|-------------|-----|
| 代码提交与评审 | R/A | I | C | I | I | - |
| 资源制作与检查 | C | I | R/A | C | I | - |
| CI 构建 | C | I | - | R/A | I | - |
| 质量门检查 | C | R | C | A | I | - |
| 功能测试 | C | R/A | - | I | I | - |
| 性能测试 | C | R | A | C | I | - |
| 发布审批 | C | C | C | C | R/A | I |
| 灰度放量 | I | C | - | R | A | C |
| 线上监控 | I | I | - | C | C | R/A |
| 应急回滚 | C | C | - | R | A | R |

### 怎么读这张表

以"发布审批"行为例：Release Manager 是 R/A——她既负责执行审批流程，也是最终拍板人。Dev、QA、TA、Toolchain 是 C——审批前需要确认各自的交付物就绪。Ops 是 I——审批通过后需要知道即将上线。

## 为什么职责边界比职责列表更重要

定义"你该做什么"只完成了一半；定义"什么不是你的事"同样重要：

| 角色 | 职责边界：什么不该做 |
|------|-------------------|
| Dev | 不该自己决定发布时间，不该绕过 QA 直接上线 |
| QA | 不该修 Bug（发现并报告即可），不该拍板是否发布 |
| TA | 不该自行修改构建管线配置 |
| Toolchain | 不该决定功能是否完成（构建成功 ≠ 功能完成） |
| Release Mgr | 不该替团队做技术决策，不该跳过审批流程 |
| Ops | 不该自行决定回滚（需要 Release Mgr 授权） |

**核心原则**：职责模糊时，默认归属 Accountable 角色。如果一件事没人是 A，那就是管理债——必须立刻指定。

## 三种常见的职责模糊场景

### 场景一：构建失败了谁来修？

构建失败的原因可能来自 Dev（代码编译错误）、TA（资源格式不对）、Toolchain（管线配置问题）。但当 CI 红了，常见反应是"等 Toolchain 来看"。

**正确做法**：Toolchain 负责定位到"哪个阶段失败"，然后将修复任务路由到对应角色。构建系统应该在失败消息里标明阶段（编译/资源处理/打包），让路由自动化。

### 场景二：性能回退了谁负责？

性能测试发现帧率下降 15%。Dev 说"我没改渲染"，TA 说"资源没变"。

**正确做法**：QA 负责报告回退事实（R），TA 负责定位是资源还是渲染问题（R），Dev 负责修复代码层面的问题（R），Release Mgr 负责决定是否阻塞发布（A）。注意这里有多个 R——RACI 允许多个 R，但 A 必须唯一。

### 场景三：线上 Crash 谁先响应？

凌晨 3 点 Crash 率飙升。Ops 看到了告警，但 Dev 没人在线。

**正确做法**：Ops 是第一响应人（R），执行预案——如果 Crash 率超过致命阈值，Ops 有权执行回滚（R），但需要在回滚后立即通知 Release Mgr（A）并记录事件。Dev 在下一个工作时段介入排查（R）。预案必须在灰度开始前就写好（V17-01）。

## 落地 RACI 的三个建议

1. **从一张小表开始**——不要试图覆盖所有场景，先画出上面的核心交付阶段矩阵，团队一起 Review
2. **挂到 Wiki 而不是放在脑子里**——RACI 的价值在于"有争议时查表"，不是"开完会后遗忘"
3. **每季度回顾一次**——角色会变、人员会调整、流程会迭代，RACI 要跟着更新

## RACI 与工具集成

RACI 不只是文档，可以嵌入工具链：

| 场景 | 工具集成方式 |
|------|------------|
| CI 失败通知 | 根据失败阶段自动 @ 对应 R 角色（而不是广播给所有人） |
| 发布审批 | 审批系统自动获取 RACI 中的 A 角色作为审批人 |
| 缺陷分配 | Bug 系统根据缺陷分类自动指派给 R 角色 |
| 监控告警 | 告警路由根据告警类型通知对应 R 和 A |
