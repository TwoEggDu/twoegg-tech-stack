## 这篇解决什么问题

V08 系列讲了客户端热更新（资源热更、代码热更、DHE）。服务端也有热更新需求——但动机不同。客户端热更新为了跳过商店审核，服务端热更新为了不重启进程就更新逻辑。本篇覆盖三种级别的服务端热更新。

## 三种热更新级别

| 级别 | 内容 | 风险 | 适用场景 |
|------|------|------|---------|
| L1：配置热推 | 数值、开关、活动参数 | 低 | 调数值、开关功能、紧急屏蔽 |
| L2：代码热重载 | 替换整个程序集或 System | 中 | 修复逻辑 bug、更新业务逻辑 |
| L3：函数级热补丁 | 替换特定函数实现 | 高 | 紧急修复特定函数 bug |

## L1：配置热推

最安全的热更新方式——只改数据，不改代码。运维在管理后台修改配置值，推送到配置中心（Redis / etcd），服务端监听变更事件并加载新配置。

### 适用范围

| 场景 | 示例 |
|------|------|
| 数值调整 | 怪物血量、经验倍率、掉落概率 |
| 功能开关 | 关闭有问题的活动、屏蔽有 bug 的功能 |
| 活动参数 | 活动时间、奖励内容、参与条件 |
| 黑白名单 | 封禁名单、白名单测试 |

### 验证要求

推送前校验格式、检查数值范围（防止配错成负数），先推送到测试服验证，保留上一版配置支持回退，所有实例确认已加载新配置。

**配置热推是最应该优先建设的能力**——覆盖 80% 的紧急修复场景，风险最低，实现最简单。

## L2：代码热重载

### ET Framework 的热重载方案

ET 将代码分为两部分：

| 程序集 | 内容 | 可热重载 |
|--------|------|---------|
| Model | 数据定义（Component） | 否——数据结构变更需要重启 |
| Hotfix | 逻辑实现（System/Handler） | 是——卸载旧程序集、加载新程序集 |

热重载流程：编译新 Hotfix.dll → 推送到服务端实例 → 卸载旧 Assembly → 加载新 Assembly → 重新绑定 System/Handler。

### ECS 中的 System 替换

ECS 架构中逻辑集中在 System、数据在 Component。热重载只替换 System，Component 保持不变——这正是数据与逻辑分离的优势。

### 状态风险

| 风险 | 应对 |
|------|------|
| 局部变量丢失 | 避免在 System 中保存状态——状态放 Component |
| 闭包引用失效 | 热重载后重新注册所有回调 |
| 类型不匹配 | Hotfix 必须和当前 Model 版本配套 |
| 静态变量重置 | 避免用静态变量存储运行时状态 |

验证要点：程序集兼容性检查、单元测试通过、先在一台实例灰度、检查内存泄漏（旧程序集是否正确卸载）。

## L3：函数级热补丁

通过运行时修改函数入口地址（Method Swapping / Detour），将调用重定向到新实现。

| 适用场景 | 风险与限制 |
|---------|-----------|
| 紧急修复单个函数的计算错误 | 平台兼容性——不是所有运行时都支持 |
| 临时绕过有问题的逻辑 | 调试困难——堆栈信息可能混乱 |
| 线上调试（添加更多日志） | 累积补丁后行为难以预测 |

**函数级热补丁应作为应急手段而非常规更新方式。** 打完补丁后，尽快通过正式部署替换补丁代码。

## 决策框架

| 问题类型 | 推荐级别 | 理由 |
|---------|---------|------|
| 数值配错了 | L1 配置热推 | 改配置，风险最低 |
| 活动需要紧急关闭 | L1 配置热推 | 功能开关 |
| 业务逻辑有 bug | L2 代码热重载 | 替换整个 System |
| 单个函数有误，影响严重 | L3 函数级替换 | 应急修复 |
| 需要加新功能 | 滚动更新 | 新功能需要完整测试 |
| 数据结构变更 | 滚动更新 | Model 层变更必须重启 |

## 小结

- [ ] 是否优先建设了配置热推能力（覆盖 80% 的紧急修复）
- [ ] 配置推送是否有格式验证、范围检查和灰度能力
- [ ] 代码热重载是否基于数据/逻辑分离（只换 System）
- [ ] 热重载后是否验证了程序集兼容性和内存泄漏
- [ ] 函数级补丁是否仅用于应急，后续有正式修复计划
- [ ] 是否有清晰的决策框架判断用哪种热更新级别
