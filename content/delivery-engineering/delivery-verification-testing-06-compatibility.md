## 这篇解决什么问题

功能对、性能达标、稳定性过关——但某款设备上花屏、某个 OS 版本闪退、某种屏幕比例 UI 错位。兼容性问题的本质是碎片化——Android 碎片化尤其严重。本篇讲设备矩阵怎么设计、云真机怎么用、GPU 和 OS 兼容性怎么测。

## 设备矩阵设计

不可能覆盖所有设备，设备矩阵的设计原则是用最少的设备覆盖最大的用户群。

### 设备选择维度

| 维度 | 数据来源 | 权重 |
|------|---------|------|
| 市场占有率 | 第三方数据平台（友盟、极光） | 高——覆盖用户基数 |
| 收入贡献 | 游戏内数据分析 | 高——覆盖付费用户 |
| 问题反馈率 | 客服和社区反馈统计 | 中——覆盖问题设备 |
| 芯片代表性 | 芯片市场分布 | 中——覆盖不同 GPU/CPU |
| 系统版本分布 | 应用市场统计 | 中——覆盖系统差异 |

### 设备矩阵模板

| 层级 | 设备数量 | 选择标准 | 测试频率 |
|------|---------|---------|---------|
| 核心设备 | 5-8 台 | 覆盖 60% 以上用户 + 主要芯片平台 | 每次构建 |
| 扩展设备 | 10-15 台 | 覆盖 85% 以上用户 + 问题高发机型 | 每次发版 |
| 全量设备 | 30+ 台 | 覆盖 95% 以上用户 | 大版本发布前 |

### Android 核心设备选择示例

| 芯片 | 代表机型 | 定位 |
|------|---------|------|
| Snapdragon 8 Gen 3 | 旗舰机 | 高端基准 |
| Snapdragon 8 Gen 1 | 次旗舰 | 高端覆盖 |
| Dimensity 9200 | 联发科旗舰 | 芯片差异覆盖 |
| Snapdragon 7 Gen 1 | 中端主力 | 中端基准 |
| Dimensity 1200 | 中端联发科 | 芯片差异覆盖 |
| Snapdragon 680 | 低端机型 | 最低支持线 |

iOS 相对简单——覆盖最近 3-4 代 iPhone + 最近 2 代 iPad 即可。

## 云设备平台

实体设备有限，云设备平台可以大幅扩展覆盖范围。

| 平台 | 特点 | 适用场景 |
|------|------|---------|
| AWS Device Farm | 国际主流，设备更新快 | 海外发行项目 |
| Firebase Test Lab | Google 生态，Android 覆盖好 | Android 为主的项目 |
| 百度 MTC | 国内设备覆盖全 | 国内发行项目 |
| Testin | 国内老牌，支持游戏测试 | 国内发行项目 |
| 腾讯 WeTest | 腾讯生态，微信小游戏支持好 | 微信小游戏项目 |

### 云真机 vs 实体设备

| 维度 | 云真机 | 实体设备 |
|------|--------|---------|
| 设备种类 | 多，数百种 | 有限，需要采购 |
| 成本 | 按使用时长计费 | 一次性采购 + 维护 |
| 性能数据准确性 | 偏差较大（网络延迟、虚拟化） | 准确 |
| 适用测试 | 功能兼容性、UI 适配 | 性能测试、稳定性测试 |
| 自动化集成 | API 支持，可集成 CI | 需要自建设备管理系统 |

原则：性能和稳定性测试用实体设备，兼容性覆盖用云真机扩展。

## GPU 与图形 API 兼容性

GPU 兼容性是游戏项目最常遇到的兼容性问题——不同 GPU 对图形 API 的实现存在差异。

### 图形 API 差异

| 维度 | Vulkan | Metal | OpenGL ES |
|------|--------|-------|-----------|
| 平台 | Android | iOS/macOS | Android（兼容） |
| 着色器语言 | SPIR-V | MSL | GLSL ES |
| 驱动质量 | 参差不齐 | Apple 控制，稳定 | 老设备驱动问题多 |
| 常见问题 | 驱动 bug、特性支持差异 | 版本差异小 | 精度差异、扩展支持 |

### GPU 兼容性测试重点

| 检查项 | 说明 |
|--------|------|
| Shader 编译 | 所有 Shader 是否在目标 GPU 上编译通过 |
| 渲染正确性 | 画面是否正确——花屏、黑屏、闪烁 |
| Shader 精度 | 低精度浮点在部分 GPU 上的表现差异 |
| 纹理格式 | ASTC / ETC2 / PVRTC 在不同 GPU 上的支持 |
| 特性降级 | 不支持的特性是否正确降级（如 Compute Shader） |

### Shader Variant 验证

Shader Variant 爆炸是常见的兼容性炸弹——某些 keyword 组合在特定 GPU 上编译失败或表现异常。

- [ ] 是否有 Shader Variant 的全量编译验证
- [ ] 是否在不同 GPU 架构上验证渲染结果
- [ ] 是否有 Shader 精度降级的兼容性覆盖

## OS 版本兼容性

### Android 版本碎片化

Android 版本碎片化依然严重，需要覆盖的主要版本：

| 版本 | API Level | 覆盖理由 |
|------|-----------|---------|
| Android 14 | 34 | 最新版本，新特性 |
| Android 13 | 33 | 主流版本 |
| Android 12 | 31-32 | 存量用户多 |
| Android 11 | 30 | 最低推荐支持版本 |
| Android 10 | 29 | 根据用户数据决定是否支持 |

### iOS 版本覆盖

iOS 用户更新率高，通常覆盖最近 3 个大版本即可：

| 策略 | 说明 |
|------|------|
| 最新版本 | 必须支持，Apple 审核要求 |
| 前一个大版本 | 必须支持，仍有大量用户 |
| 前两个大版本 | 根据用户数据决定 |

### 系统级行为差异

不同 OS 版本在权限、通知、后台策略上有差异：

| 差异点 | 影响 | 验证方式 |
|--------|------|---------|
| 权限模型 | 运行时权限的弹窗时机和行为 | 各版本分别验证权限流程 |
| 通知渠道 | Android 8+ 要求通知渠道 | 验证通知是否正确显示 |
| 后台限制 | 各版本后台策略不同 | 验证后台任务是否被杀 |
| Scoped Storage | Android 10+ 的存储访问限制 | 验证文件读写是否正常 |

## 屏幕适配

| 类型 | 常见比例 | 测试重点 |
|------|---------|---------|
| 标准屏幕 | 16:9 | 基准适配 |
| 全面屏 | 18:9, 19.5:9, 20:9 | 安全区域、刘海屏适配 |
| 折叠屏 | 展开/折叠两种比例 | 动态切换、布局重排 |
| 平板 | 4:3, 16:10 | 大屏布局、UI 缩放 |

关键验证点：

- [ ] 安全区域是否正确处理（刘海、打孔、圆角）
- [ ] 不同比例下 UI 是否完整可见
- [ ] 折叠屏展开/折叠切换是否正常
- [ ] 平板横竖屏切换是否正确

## V13 系列总结

V13 建立了四层验证体系：

1. **功能验证**——冒烟守底线、回归防退化、自动化提效率
2. **性能验证**——预算基线比对、退化自动检测、设备分层测试
3. **稳定性验证**——长跑找泄漏、Monkey 找崩溃、弱网找断线
4. **兼容性验证**——设备矩阵覆盖、云真机扩展、GPU/OS 差异处理

四层验证通过验证门分级拦截——质量门拦代码问题、验证门拦产品问题、发布门做最终确认。

验证体系不是一次建好就不动的——随着项目规模增长和用户量变化，验证策略需要持续调整。

## 小结

- [ ] 是否设计了分层的设备矩阵（核心/扩展/全量）
- [ ] 设备选择是否基于市场份额和收入贡献数据
- [ ] 是否使用云真机平台扩展兼容性覆盖
- [ ] 是否有 GPU/图形 API 的兼容性测试
- [ ] Shader Variant 是否有全量编译验证
- [ ] 是否覆盖了主要 OS 版本的行为差异
- [ ] 是否测试了不同屏幕比例和折叠屏适配
