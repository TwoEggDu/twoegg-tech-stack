## 这篇解决什么问题

资源管线是交付链路中最重的环节——一个中型游戏项目的资源占构建产物体积的 80-95%。很多团队直接从"怎么用 AssetBundle"或"怎么配 Addressables"开始学，但没有先理解资源管线到底在解决什么问题。

不理解问题就选方案，等于先选了药再去看病。

## 为什么这个问题重要

资源管线设计不当的代价：

- 首次加载时间 30 秒以上——因为打包粒度太粗，加载一个小物件要拉整个大包
- 热更新包体 500MB——因为打包策略不支持增量更新，改了一个贴图要重新下载整个 Bundle
- 运行中内存持续增长最终 OOM——因为加载后没有正确释放资源，引用计数失控
- 热更新后部分用户画面异常——因为 Catalog 版本比对逻辑有 Bug，下载了不完整的资源

## 本质是什么

无论使用什么引擎——Unity、Unreal、Godot、自研——资源管线都要解决同样的四个问题：

### 问题一：打包——怎样把散文件变成可分发的包

开发期，资源是散的：每个贴图一个文件，每个模型一个文件，每个音频一个文件。一个中型项目可能有几万到几十万个资源文件。

发布时不能把这些散文件直接发给用户：
- 散文件在移动设备上的 IO 性能极差（V03 的存储与 IO 基础已经解释了原因）
- 散文件无法有效压缩（每个文件独立压缩的比率低于整体压缩）
- 散文件的数量超出某些文件系统的限制

所以需要**打包**：把逻辑相关的资源组合成少量的包文件。

打包需要回答的核心问题：
- **分组原则**：哪些资源放在同一个包里？按场景分？按功能模块分？按资源类型分？
- **依赖管理**：A 包里的材质引用了 B 包里的贴图，怎么处理？
- **冗余控制**：同一个贴图被两个包引用，是打两份（冗余但独立）还是抽成公共包（无冗余但增加依赖）？
- **粒度权衡**：包太大则加载浪费，包太小则 IO 次数多。最优粒度取决于加载模式和分发方式。

### 问题二：加载——运行时怎么把包里的资源取出来用

打好的包在运行时需要被加载到内存中，资源需要被解包、反序列化、注册到引擎的资源系统中。

加载需要回答的核心问题：
- **同步还是异步**：同步加载阻塞主线程（简单但会卡顿），异步加载不阻塞（流畅但需要管理回调和状态）
- **何时加载**：启动时全量加载（简单但启动慢），进场景时按需加载（快但需要预加载策略），运行时动态加载（灵活但需要处理加载中的过渡状态）
- **加载优先级**：同时请求加载多个资源时，谁先加载？玩家视野内的优先于视野外的？当前场景的优先于预加载的？

### 问题三：版本——怎么知道用户手上的资源是不是最新的

游戏上线后会持续更新资源。用户设备上已有旧版资源，新版资源在服务器上。资源管线需要一套版本机制来回答：

- **哪些资源需要更新**：客户端本地的资源清单和服务器上的清单做比对，找出差异
- **怎样下载差异部分**：全量下载还是增量下载？增量下载的粒度是文件级还是 Bundle 级？
- **更新后怎么生效**：需要重启游戏还是运行时热替换？替换过程中的资源一致性怎么保证？
- **更新失败怎么回退**：下载中断后重启，是从头下载还是断点续传？下载完成但校验失败，是回退到旧版本还是重新下载？

### 问题四：生命周期——加载的资源什么时候释放

资源加载到内存后，需要管理它的生命周期：什么时候可以释放，释放时怎么确认没有人还在使用它。

生命周期管理需要回答的核心问题：
- **引用追踪**：谁在使用这个资源？当所有使用者都释放了引用，资源才能安全卸载
- **泄漏检测**：资源被加载后从未释放，或者释放的时机不对导致引用计数不归零
- **内存预算**：资源总内存是否在预算范围内？超出时优先卸载哪些资源？
- **跨包依赖释放**：A 包卸载时，它依赖的 B 包里的贴图能不能一起卸载？要看 B 包是否还有其他引用者

## 四个问题的关系

这四个问题不是独立的，而是互相约束的：

```
打包策略  ──→  影响加载性能（粒度决定 IO 次数）
    │              │
    │              ↓
    └───→  影响版本管理（粒度决定热更新包大小）
               │
               ↓
         影响生命周期管理（依赖关系决定释放逻辑复杂度）
```

打包粒度细 → 加载 IO 次数多但热更包小 → 依赖关系多但可以精确释放
打包粒度粗 → 加载 IO 次数少但热更包大 → 依赖关系少但释放粗放

没有"最优"的打包粒度。它取决于项目的加载模式（是一次性加载还是按需流式加载）、分发模式（首包为主还是热更新为主）和设备限制（低端设备的 IO 和内存上限）。

## 与交付链路的关系

资源管线横跨交付飞轮的多个环节：

| 交付环节 | 资源管线的职责 |
|---------|--------------|
| 生产段 | 资源规范（V02 已覆盖） |
| 构建段 | 打包、序列化、平台适配 |
| 验证段 | 包体预算检查、加载性能测试、依赖完整性验证 |
| 发布段 | CDN 部署、版本清单发布 |
| 监控段 | 加载失败率、下载成功率、资源缺失上报 |
| 闭环段 | 资源问题的根因分析和防复发 |

资源管线不只是"打包和加载"，它贯穿了构建、验证、发布和运营的每一个阶段。
