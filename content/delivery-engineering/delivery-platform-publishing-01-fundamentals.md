## 这篇解决什么问题

V07 讲了怎么构建三端产物，V08 讲了脚本热更新。构建产物出来后，下一步是让用户拿到它。这就是发布——但发布远不只是"上传到平台"。

## 发布在交付链路中的位置

```
V08 热更构建 ──┐
V07 多端构建 ──┤→ 构建产物 → [验证门] → [发布门] → 发布 → [灰度] → 全量
V06 CDN 部署 ──┘
```

发布位于验证门和灰度之间。它的职责是：**在确认产物合格后，通过平台审核机制把产物送达用户，同时准备好万一出问题的退路。**

## 发布前检查清单

在提交平台审核之前，以下检查必须完成：

### 产物检查
- [ ] 构建产物通过了验证门（功能/性能/稳定性/兼容性）
- [ ] 包体大小在平台限制内
- [ ] 签名有效且未过期
- [ ] 版本号正确递增（不和已发布版本冲突）
- [ ] Build Number 唯一且可追溯到 commit

### 合规检查
- [ ] 隐私政策已更新（如有数据收集变更）
- [ ] 年龄分级已确认
- [ ] 第三方 SDK 的合规声明已更新
- [ ] 敏感权限的使用说明已准备（iOS 隐私标签、Android 数据安全表）

### 回滚准备
- [ ] 上一个稳定版本的产物仍然可用（已归档）
- [ ] 回滚操作有预设脚本（不是手动操作）
- [ ] 热修复通道已就绪（如需紧急修复可走热更）
- [ ] 服务端兼容旧版本客户端（回滚后旧版本能正常连接）

### 发布审批
- [ ] 发布审批流程完成（至少两人确认）
- [ ] Release Notes 已准备
- [ ] 灰度策略已确认（起始比例、观察窗口、扩量条件）

## 发布的三个阶段

### 阶段一：提审

把构建产物提交到平台审核。三个平台的提审方式不同（后续文章展开），但有共同的工程要求：

**提审自动化**。提审操作应该是 CI 管线的一部分，不是手动到平台控制台操作。自动化提审可以：
- 避免人工操作遗漏（忘记选某个配置项）
- 记录提审时间和版本信息
- 在审核通过后自动触发下一步

**提审前冒烟**。提审前在平台特定的真机上跑一次快速冒烟——确保包能安装、能启动、核心流程能走通。这比审核被拒后再排查高效得多。

### 阶段二：审核等待

审核周期因平台而异：

| 平台 | 典型审核周期 | 可预测性 |
|------|------------|---------|
| iOS App Store | 1-3 天 | 低（可能更快或更慢） |
| Google Play | 数小时-1 天 | 中 |
| 微信小游戏 | 1-3 天 | 低 |

审核等待期的工程管理：

**状态追踪**。CI 或运维系统应该轮询审核状态，状态变更时通知团队。不要靠人手动刷新平台控制台。

**并行提审**。三端同时提审（不是等 iOS 通过后再提 Android）。iOS 审核最慢，应该最先提审。

**审核驳回处理**。驳回后需要快速响应：分析驳回原因 → 修复 → 重新构建 → 重新提审。驳回原因应该记录到知识库中，避免同样的原因反复驳回。

### 阶段三：发布上线

审核通过后，不等于自动上线。大多数平台支持"手动发布"——审核通过后由开发者决定何时推送给用户。

**推送策略**：
- 三端尽量同步上线（避免"iOS 已更新但 Android 还没有"导致的服务端兼容压力）
- 如果三端审核时间差异大，可以先发审核通过的端，但服务端必须同时支持新老版本
- 配合灰度发布——先推送 1%，观察指标，再逐步扩量（V17 详细展开）

## 发布的原子性

发布应该是原子操作：**要么完整上线，要么不上线。**

不应该出现的中间状态：
- iOS 已上线但 Android 还没有，服务端只兼容新版本 → Android 用户无法连接
- 客户端已上线但 CDN 上的热更资源还没有部署 → 新版客户端找不到对应资源
- 客户端已上线但服务端还没有部署新版本 → 新接口调用失败

### 一次原子性被破坏的真实事故

某个项目曾经遇到过一次典型的原子性破坏事故。当时 iOS 审核意外提前通过（通常要两天，那次 6 小时就过了），运维同学看到审核通过就习惯性点了发布——但 Android 版本还在审核中，CDN 上的热更资源也还没有部署到位。

接下来发生了什么：

1. iOS 用户更新到新版本后，客户端去拉取 CDN 上的对应版本资源包，发现 404
2. 由于资源下载失败的兜底逻辑不够健壮，客户端卡在了加载界面无限转圈
3. 客服群在 15 分钟内涌入上百条"更新后进不去游戏"的反馈
4. 团队紧急回撤 iOS 版本——但已经有约 8% 的活跃用户更新了，这些用户只能等热更修复
5. 最终修复耗时 4 小时，影响了当天的日活和留存数据

这个事故的根因很简单：**发布流程里没有"三端联动检查"这个环节**，运维可以单独操作任意一端的发布。事后加了两个机制——第一，发布操作统一通过发布管线触发，不再允许手动点发布；第二，管线里加了前置检查，三端审核状态和 CDN 部署状态全部就绪才允许执行发布动作。

**经验教训**：原子性不是靠"大家记住同时操作"来保证的，而是靠**系统约束**——把单端发布按钮从运维手上收走，换成一个统一的发布流水线。

**协调机制**：
1. 三端审核全部通过后才开始发布流程
2. CDN 资源和服务端先部署（V06-06 的先内容后索引原则）
3. 确认 CDN 和服务端就绪后，三端同步推送客户端更新
4. 灰度推送以最慢的端为准

### 实际操作中的发布时间窗口

经验上来看，三端发布的时间窗口选择比很多人以为的重要得多。我更倾向于把发布放在**工作日上午 10-11 点**，原因如下：

- 上午发布意味着全天有充足的监控窗口，团队在岗可以响应问题
- 避开午高峰——很多游戏的活跃峰值在中午 12-14 点，提前一小时发布给了足够的灰度观察时间
- 不要在周五下午发版——这是行业共识但还是有团队踩坑。周五下午发版，周末出问题没人响应
- 不要在大版本更新日、节假日前一天发非紧急版本——用户行为模式变化大，指标基线不准

某次我们在周四晚上 8 点发了一个"小改动"版本，周五早上发现 Crash 率微升（从 0.08% 到 0.15%），但因为是工作日刚开始大家都在处理别的事情，直到下午才有人认真看监控数据。如果是上午发的，这个问题在两小时内就会被注意到。

## 常见事故与排障

### 审核驳回的高频原因

实际操作中最常见的审核驳回原因有这么几类，按我们团队的统计排序：

| 排名 | 驳回原因 | 频率 | 平均延迟影响 |
|------|---------|------|-------------|
| 1 | iOS 隐私标签与实际 SDK 行为不一致 | 每 3-4 次提审遇到一次 | +2-3 天 |
| 2 | 截图/预览视频与实际功能不符 | 首版本或大改版时 | +1-2 天 |
| 3 | 崩溃或严重 UI 问题被审核人员发现 | 偶发 | +2-5 天 |
| 4 | 微信小游戏内容合规（文案、图片） | 每版本需注意 | +1-3 天 |
| 5 | Google Play 目标 API 级别不满足要求 | 年度 API 升级时 | +1-7 天 |

经验上来看，第 1 类问题最坑——因为 iOS 的隐私标签审查在 2023 年之后明显变严了。某次我们集成了一个第三方广告 SDK，SDK 文档说它不收集 IDFA，但审核发现它的二进制里调用了 ATT 相关的 API。结果连续被拒三次，每次来回就是两天，最终找到 SDK 提供商才确认是他们的文档没更新。

**教训**：每次集成新 SDK 时，用 `nm` 或 `otool` 扫描 SDK 二进制文件，看它实际调用了哪些系统 API，不要只信文档。

### 提审前冒烟不过的典型问题

有一类问题特别容易在提审前冒烟阶段暴露——**签名问题**。Android 最常见的是 v1/v2/v3 签名不一致，iOS 最常见的是 Provisioning Profile 过期或者 Entitlements 配置变更后没有重新签名。这些问题在本地开发环境跑不出来（开发用 Debug 签名），只有在 Release 构建+真机安装时才会爆。

建议：CI 管线里加一步**签名验证**，构建完成后自动检查签名完整性，不要等到提审被拒才发现。

## 常见错误做法

**审核通过后立刻自动上线**。平台提供了"审核通过后自动发布"的选项。但这意味着放弃了发布时机的控制权——可能在凌晨上线（没有值班人员监控）或在服务端未就绪时上线。建议使用手动发布。

**三端分别由不同的人独立发布**。iOS 由 A 负责，Android 由 B 负责，微信由 C 负责。三个人各自操作，没有同步协调。应该由一个人或一个自动化流程统一协调三端发布。

**不记录发布操作**。发布时间、发布版本、操作人、灰度策略没有记录。出了问题后无法回溯"这个版本是什么时候发布的、谁操作的、灰度了多少"。

**版本号管理混乱**。某个项目曾经出现过一次乌龙：Android 渠道包的版本号写成了和上一个已发布版本相同的值，提审时 Google Play 直接拒绝了。排查发现是构建脚本里版本号是手动填写的，有人改了代码但忘了改版本号。解决方案很简单——版本号从 Git tag 自动生成，构建脚本不接受手动输入。

## 小结与检查清单

- [ ] 发布前检查清单是否完整执行
- [ ] 提审是否通过 CI 自动化
- [ ] 审核状态是否有自动追踪和通知
- [ ] 三端是否并行提审（iOS 最先）
- [ ] 发布是否使用手动发布（不自动上线）
- [ ] 三端发布是否有统一协调（不各自独立操作）
- [ ] 发布操作是否有记录（时间、版本、操作人）
