## 这篇解决什么问题

微信小游戏的包体限制和原生 App 有结构性差异。不是"包小一点"的问题，而是整个包体策略需要重新设计。本篇覆盖微信小游戏的主包限制、分包机制、CDN 资源加载和内存约束。

## 微信小游戏的结构性约束

| 约束 | 数值 | 对比原生 App |
|------|------|------------|
| 主包上限 | 4MB（普通）/ 8MB（增强）/ 12MB（特殊类目） | 原生端首包 150-300MB |
| 分包总上限 | 20MB（普通）/ 30MB（增强） | 原生端不限制 |
| 可用内存 | 通常 256-512MB（和微信主进程共享） | 原生端通常 1-4GB 独占 |
| 运行环境 | WebGL（V8/JSCore + Canvas/WebGL） | 原生代码（ARM64） |
| 文件系统 | 虚拟文件系统，有缓存限制 | 原生文件系统，空间充裕 |

这些约束意味着：**不能简单地把原生端的包打小一点就上微信——需要完全不同的架构设计。**

## 主包策略

4MB 主包只能放最最核心的东西：

```
主包（4MB 以内）
├── game.js / game.json         (引擎入口和配置)
├── 引擎 WASM 文件               (Unity WebGL 运行时，通常 2-3MB)
├── 启动加载界面资源              (极简的 loading UI)
└── 启动配置文件                  (服务器地址、CDN 地址、版本号)
```

**主包里几乎没有游戏资源**。登录界面、主界面、角色、场景——全部放在分包或 CDN 上。

### 引擎 WASM 的体积优化

Unity 导出的 WebGL 运行时（.wasm 文件）通常是主包中最大的部分。优化手段：

- **Managed Stripping Level 设为 High**：裁剪未使用的 C# 类型
- **Engine Code Stripping**：裁剪未使用的引擎模块
- **压缩**：微信小游戏支持 WASM 的 Brotli/gzip 压缩
- **拆分**：部分 WASM 模块可以延迟加载（需要 Unity 版本支持）

## 分包策略

微信的分包机制允许把内容拆分为多个子包，在需要时动态加载：

```
├── 主包（4MB）           → 启动必需
├── 分包 A: login（2MB）  → 登录和主界面
├── 分包 B: battle（5MB） → 战斗场景
├── 分包 C: shop（3MB）   → 商城
└── ... 总计不超过 20MB
```

### 分包的加载时机

微信提供了两种分包加载方式：

**预下载**（preDownloadSubpackages）：在进入指定页面时自动预下载其他分包。

```json
// game.json
{
  "preDownloadSubpackages": [
    {
      "packages": ["battle"],
      "network": "wifi"  // 只在 Wi-Fi 下预下载
    }
  ]
}
```

**按需加载**：代码中调用 `wx.loadSubpackage()` 在需要时下载。

### 分包的工程设计

分包划分的原则和 V05 的打包策略类似——按功能模块或使用场景划分：

- 登录和主界面一个分包（用户打开后立刻需要）
- 每种核心玩法一个分包（进入时按需加载）
- 活动内容一个分包（随活动上下线）

## CDN 资源：分包之外的大块内容

20-30MB 的分包总量对大多数游戏来说远远不够。额外的资源通过 CDN 加载：

```
CDN (无大小限制)
├── 角色模型和贴图
├── 场景资源
├── 特效资源
├── 音频文件
├── 高清 UI 素材
└── 配置数据
```

CDN 资源的加载方式：

**wx.request / wx.downloadFile**：下载文件到本地临时目录或用户文件系统。

**文件缓存管理**：微信小游戏的用户文件系统有空间限制（通常 200MB）。需要自己管理缓存——哪些文件常驻、哪些用完可删、空间不足时淘汰哪些。

### CDN 资源的版本管理

和 V05-06 讲的 Catalog 版本比对机制相同——但在微信小游戏环境下有额外约束：

- 文件系统操作比原生端慢（虚拟文件系统开销）
- 不能用 SharedArrayBuffer 等高级 API（微信环境限制）
- 缓存清理需要主动管理（系统不会自动清理）

## 内存约束下的资源策略

微信小游戏的内存约束决定了资源策略必须比原生端更激进：

| 策略 | 做法 |
|------|------|
| **纹理降级** | 所有贴图最大 512x512，关键贴图最大 1024 |
| **模型简化** | 面数预算比原生端降低 50-70% |
| **音频压缩** | 单声道、22050Hz、高压缩率 |
| **即用即释** | 离开场景立刻释放该场景的资源，不缓存 |
| **分帧加载** | 大资源分多帧加载，避免单帧内存峰值 |
| **纹理异步上传** | 避免大纹理上传时的内存峰值 |

**内存监控**是微信小游戏交付的必备能力：

```javascript
// 微信 API：监控内存告警
wx.onMemoryWarning(function(res) {
    // level: 5(TRIM_MEMORY_RUNNING_MODERATE), 10(LOW), 15(CRITICAL)
    if (res.level >= 10) {
        // 紧急释放非关键资源
        releaseNonCriticalResources();
    }
});
```

## 微信小游戏与原生端的资源共享

理想情况下，三端共用一套资源管线，微信小游戏端通过参数差异化。但实际中微信小游戏的差异太大，通常需要：

- **独立的纹理分辨率上限**（比原生端低一档）
- **独立的模型 LOD 配置**（更激进的简化）
- **独立的 Shader 变体集**（去掉 WebGL 不支持的特性）
- **独立的音频配置**（降采样率、降质量）

这些差异应该通过 V02-06 讲的多端资源差异管理机制处理——用导入预设和构建配置自动产出，不手动维护两套资源。

## 常见错误做法

**把原生端的资源直接搬到微信小游戏**。原生端 2048x2048 的贴图在微信小游戏上不只是"大"——加载时的内存峰值可能直接导致进程被杀。

**CDN 资源不做缓存管理**。每次启动都重新下载所有资源。用户体验差，CDN 费用高。必须有本地缓存和版本比对。

**不监控微信端的内存使用**。没有接入 `wx.onMemoryWarning`，内存超限时微信直接杀进程——用户看到的是"闪退"，没有任何错误提示。

## 小结与检查清单

- [ ] 主包是否控制在 4MB/8MB 以内（按类目）
- [ ] 分包划分是否按使用场景，有预下载策略
- [ ] CDN 资源是否有本地缓存和版本比对
- [ ] 缓存空间是否有主动管理（淘汰策略）
- [ ] 纹理/模型/音频是否有微信端独立的规格上限
- [ ] 是否接入了 `wx.onMemoryWarning` 并有紧急释放逻辑
- [ ] Shader 是否有 WebGL 兼容版本
