## 这篇解决什么问题

美术资源是游戏交付中体积最大、变更最频繁的生产线。一个中型项目的美术资源量通常是代码量的 10-100 倍。

如果美术资源的生产管线没有质量门，下游的资源管线、构建管线和性能预算都无法保障。

## 为什么这个问题重要

美术资源生产管线的问题通常不会立刻暴露。一个格式不对的贴图可以正常导入引擎、正常构建、正常在编辑器里运行。但它会在这些地方爆发：

- 打包后包体超预算——因为贴图尺寸没有按规范压缩
- 低端设备闪退——因为模型面数超出设备承受范围
- 构建时间暴涨——因为大量资源没有按增量导入的条件提交
- 多端表现不一致——因为导入设置在不同平台上的行为不同

这些问题的修复成本远高于在生产阶段拦住的成本。

## 本质是什么

任何引擎的美术资源生产管线，本质上都是四个环节：

```
导出 → 提交 → 导入 → 验证
```

### 导出：从 DCC 工具产出源文件

美术在 Maya / 3ds Max / Blender 中制作模型，在 Photoshop / Substance Painter 中制作贴图，在 Wwise / FMOD 中制作音频。

导出环节的核心问题是**导出设置的一致性**。同一个项目的所有美术应该使用相同的导出预设：

- FBX 导出：坐标系、缩放比例、骨骼层级、动画烘焙方式
- 贴图导出：色彩空间（sRGB vs Linear）、通道布局（哪些信息放在哪个通道）、位深度
- 音频导出：采样率、通道数、格式

导出设置不一致是美术资源问题的第一大来源。两个美术用不同的 FBX 导出设置，进入引擎后一个模型正常一个模型缩放异常——但两个人都觉得自己没问题。

**质量门**：导出预设文件应该版本化管理，美术不应该手动配置导出参数。

### 提交：源文件进入版本库

美术资源提交到版本库时，是最适合做第一道自动化检查的时机。

提交时可以拦的问题：

- 文件命名不符合规范（大小写、中文、特殊字符）
- 文件大小超出阈值（一个 4K 未压缩的 PSD 可能 200MB+）
- 文件格式不在白名单中（提交了 .max 源文件而不是 .fbx）
- 目录位置错误（角色资源放到了场景目录下）

提交时不该拦的问题：

- 资源内容是否正确（这是美术主管的职责，不是工具的职责）
- 资源性能是否达标（这需要导入引擎后才能判断）

**质量门**：提交钩子（pre-commit hook）或 CI 的提交后检查。规则应该明确、可解释、可豁免。

### 导入：引擎处理源文件

提交到版本库的源文件，在引擎中通过 Importer 转换为运行时资源。这个环节是引擎专属的，不同引擎的处理方式不同。

导入环节的核心问题是**导入设置的一致性和正确性**。

以 Unity 为例，一个贴图的导入设置包括：

- 纹理类型（Default / NormalMap / Sprite）
- 最大尺寸和压缩格式（按平台分别配置）
- Mipmap 是否开启
- sRGB 色彩空间是否正确
- Read/Write Enabled 是否需要

错误的导入设置可能导致：
- 法线贴图被当成普通贴图处理，光照计算错误
- 贴图未压缩，内存和包体翻倍
- Read/Write Enabled 不必要地打开，内存占用翻倍

**质量门**：导入设置应该通过预设（Preset）或自动化脚本统一管控，而不是依赖美术逐个手动设置。

### 验证：导入后的质量检查

资源导入引擎后，可以进行更深层的质量检查：

- 模型面数是否在预算范围内
- 贴图实际分辨率和压缩格式是否符合规范
- 骨骼数量是否在目标平台的限制内
- 材质引用是否完整（有没有丢失 Shader 或纹理）
- Prefab 的 Component 配置是否合理

这一层检查通常通过编辑器扩展工具或 CI 中的自动化脚本执行。

**质量门**：资源验证结果应该以报告形式输出，区分"必须修复"（阻止构建）和"建议优化"（不阻止但记录）。

## 常见错误做法

**把所有检查都放在提交时**。提交钩子如果检查太重（比如需要启动引擎做分析），会严重拖慢美术的提交速度，最终导致美术绕过检查流程。提交时只做轻量检查，重量检查放在 CI。

**导入设置靠人工文档**。写一份"美术导入规范文档"让大家照着设置。结果是每个人的理解不同，设置不一致。导入设置必须通过工具自动化，而不是靠人读文档。

**不区分阻止性和建议性问题**。所有检查不通过都阻止提交。结果是美术频繁被阻止，开始抱怨"工具太严了"，最终工具的可信度下降，真正重要的检查也被忽视。必须区分严重级别——命名错误只是警告，面数超预算 200% 才是阻止。

## 如何验证

你的美术资源生产管线是否健康，可以用以下检查清单：

- [ ] 导出预设是否版本化管理，所有美术使用相同预设
- [ ] 提交时是否有自动化的命名和格式检查
- [ ] 导入设置是否通过 Preset 或脚本统一管控
- [ ] 导入后是否有面数、尺寸、压缩格式的自动化验证
- [ ] 验证结果是否区分阻止性和建议性
- [ ] 新美术入职时，是否能在不读文档的情况下正确提交资源

最后一项是最重要的检验标准：如果新人不读文档也能正确操作，说明管线设计到位；如果必须读完一份长文档才能正确提交，说明管线还是在靠人而不是靠工程。
