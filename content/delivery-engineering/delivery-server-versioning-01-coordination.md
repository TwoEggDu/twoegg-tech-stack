## 这篇解决什么问题

服务端上线后，客户端不会同步更新——有人用最新版本，有人停留在上一版。服务端必须同时服务多个客户端版本。怎么定义版本号、怎么划分兼容边界、怎么制定淘汰策略——这是服务端版本管理的核心问题。

## 版本不对称性

| 维度 | 客户端 | 服务端 |
|------|--------|--------|
| 版本控制权 | 玩家决定是否更新 | 团队控制，部署即生效 |
| 在线版本数 | 可能同时存在 3-5 个版本 | 通常只有一个版本在运行 |
| 更新速度 | 小时到天级（商店审核） | 分钟级（部署流程） |
| 回退能力 | 几乎不可能 | 分钟级回滚 |

核心矛盾：服务端只有一个版本在跑，但要同时服务 N 个客户端版本。

## 版本号定义

服务端版本号推荐使用语义化版本（SemVer）：`Major.Minor.Patch`。

| 字段 | 含义 | 兼容性影响 |
|------|------|-----------|
| Major | 大版本——协议不兼容的变更 | 旧客户端无法使用，必须强制更新 |
| Minor | 功能版本——向后兼容的新增 | 旧客户端可以正常使用，新功能不可见 |
| Patch | 修复版本——bug 修复 | 对客户端完全透明 |

## 同时支持多版本的三种模式

### 模式一：版本路由

不同版本的客户端路由到不同的服务端实例。版本完全隔离，但资源翻倍、运维复杂度高。适合大版本过渡期。

### 模式二：协议兼容层

服务端内部通过适配层处理不同版本的协议差异。只跑一套业务逻辑，但适配层需要持续维护，历史兼容代码会累积。

### 模式三：最低版本强制更新

设定最低客户端版本，低于该版本的客户端被强制要求更新。服务端不需要维护多版本兼容，但影响玩家体验，不能频繁使用。

### 实际做法

| 场景 | 策略 |
|------|------|
| 大版本更新（Major） | 最低版本强制更新 |
| 功能迭代（Minor） | 协议兼容层 |
| Bug 修复（Patch） | 服务端直接部署，客户端无感 |
| 大版本过渡期（1-2 周） | 版本路由 |

## API 版本化模式

| 模式 | 示例 | 优劣 |
|------|------|------|
| URL 路径 | `/api/v2/login` | 直观，但 URL 膨胀 |
| Header 字段 | `X-API-Version: 2` | URL 干净，但不够显式 |
| Payload 版本字段 | `{ "version": 2, "data": {...} }` | 灵活，适合游戏自定义协议 |

游戏服务端更常用第三种——游戏协议通常不走 HTTP REST，而是自定义二进制协议或 Protobuf，在消息头中带版本号最自然。

## 版本兼容矩阵

维护一张兼容矩阵，记录每个服务端版本支持哪些客户端版本：

| 服务端版本 | 最低客户端版本 | 最高客户端版本 | 备注 |
|-----------|--------------|--------------|------|
| 2.3.0 | 2.1.0 | 2.3.x | 当前线上 |
| 2.2.0 | 2.0.0 | 2.2.x | 上一版本 |
| 2.1.0 | 2.0.0 | 2.1.x | 已废弃 |

CI 中可以添加兼容性测试——用不同版本的客户端跑集成测试，确保矩阵中标注"兼容"的组合确实能正常工作。

## 版本淘汰策略

不可能永远支持所有旧版本。淘汰策略四步走：

1. **维护窗口**：同时支持最近 N 个 Minor 版本（推荐 N=2-3）
2. **淘汰公告**：至少提前 2 周通知即将不支持的版本
3. **渐进式淘汰**：先限制新功能，再限制登录，最后完全拒绝
4. **数据驱动**：当某版本的 DAU 占比低于阈值（如 5%），启动淘汰流程

## 小结

- [ ] 服务端版本号是否遵循语义化版本
- [ ] 是否定义了 Major/Minor/Patch 的兼容性规则
- [ ] 是否有明确的多版本兼容策略（路由/适配层/强制更新）
- [ ] 是否维护版本兼容矩阵
- [ ] 是否有版本淘汰策略和提前通知机制
- [ ] CI 是否包含多客户端版本的兼容性测试
