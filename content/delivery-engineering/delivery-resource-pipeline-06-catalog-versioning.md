## 这篇解决什么问题

资源打包和加载解决了"资源怎么打、怎么用"。版本比对解决的是"用户设备上已有旧资源，服务器上有新资源，客户端怎么知道需要下载哪些差异"。

这个环节看似简单（比较两个文件列表），但工程细节决定了热更新的正确性和可靠性。

## 为什么这个问题重要

版本比对出问题的典型场景：

- 热更新后 20% 用户加载异常——Manifest 版本没有正确更新，部分用户下载到了旧版本的资源清单，按旧清单下载了不完整的资源
- CDN 缓存返回旧文件——Manifest 已更新但 CDN 节点的缓存还是旧的，客户端拿到旧 Manifest 后认为"不需要更新"
- 断网恢复后资源残缺——用户在下载更新包时断网，重启后下载了一半的文件和本地的旧文件混在一起

## 本质是什么

资源版本管理的核心是一个**双清单比对 + 可靠下载 + 校验回退**的流程：

```
1. 客户端读取本地 Manifest（上一次已有的资源清单）
2. 从服务器拉取远程 Manifest（最新的资源清单）
3. 比对两份 Manifest，得到差异列表（需要下载的文件）
4. 下载差异文件
5. 校验下载文件的完整性
6. 替换本地文件，更新本地 Manifest
7. 校验失败或下载中断时，回退到上一个完整状态
```

每一步都有工程细节需要处理。

### Manifest 的结构

Manifest 是资源清单——记录了每个资源包的名称、哈希值、大小和依赖关系。

一个 Manifest 至少需要包含：

```json
{
  "version": "1.2.3",
  "build_hash": "abc123",
  "bundles": [
    {
      "name": "bundle_characters",
      "hash": "sha256:def456...",
      "size": 15728640,
      "dependencies": ["bundle_shared_textures"],
      "crc": 1234567890
    }
  ]
}
```

关键字段：

| 字段 | 用途 |
|------|------|
| version | 标识 Manifest 对应的游戏版本 |
| build_hash | 标识产出这个 Manifest 的构建 |
| name | Bundle 文件名 |
| hash | Bundle 内容的哈希值（用于比对是否需要更新） |
| size | Bundle 文件大小（用于下载进度和预估空间） |
| dependencies | 依赖的其他 Bundle |
| crc | 校验码（用于验证下载完整性） |

### 比对算法

比对的逻辑：

```
对于远程 Manifest 中的每一个 Bundle：
  如果本地 Manifest 中不存在 → 需要下载（新增）
  如果本地存在但 hash 不同 → 需要下载（更新）
  如果本地存在且 hash 相同 → 不需要下载（一致）

对于本地 Manifest 中有但远程没有的 Bundle → 可以删除（废弃）
```

比对的粒度是 **Bundle 级**，不是文件级。即使 Bundle 内只改了一个贴图，整个 Bundle 需要重新下载。

如果需要更细粒度的增量更新（只下载 Bundle 内变更的部分），需要实现 **差分更新**（binary diff），这比 Bundle 级比对复杂得多，通常只在包体极度敏感的项目中使用。

### 下载的可靠性

下载差异文件时需要处理的边界情况：

**断点续传**。下载到一半断网了，恢复后从断点继续，不重新下载已完成的部分。实现方式：记录每个文件的下载进度，用 HTTP Range 请求恢复。

**并行下载**。多个 Bundle 可以并行下载以提升速度。但并行数需要限制（移动网络下并行太多反而更慢），建议 2-4 个并行连接。

**重试策略**。单个文件下载失败时，重试 2-3 次。全部重试失败后提示用户重试或回退。不要无限重试——如果是 CDN 配置问题，重试不会成功。

**空间检查**。下载前检查设备剩余空间是否足够。如果空间不足，提示用户清理后再更新，不要下载一半因空间不足失败。

### 校验与回退

下载完成后必须校验文件完整性：

```
对于每个下载的文件：
  计算本地文件的 hash
  与 Manifest 中记录的 hash 比对
  如果一致 → 校验通过
  如果不一致 → 校验失败，删除本地文件，标记为需要重新下载
```

**原子性替换**。新文件不应该直接覆盖旧文件。应该先下载到临时目录，全部校验通过后再一次性替换。这样如果校验失败或下载中断，旧文件仍然完整可用。

```
1. 下载到 /temp/update/ 目录
2. 全部下载完成 + 全部校验通过
3. 将 /temp/update/ 的文件移动到 /bundles/ 目录（原子操作）
4. 更新本地 Manifest
5. 删除 /temp/update/
```

如果步骤 2 失败，用户重启后仍然使用旧版本的资源（完整可用）。不会出现"一半新一半旧"的中间状态。

## CDN 部署的工程要点

Manifest 和 Bundle 文件通常部署在 CDN 上。CDN 部署需要处理：

**缓存一致性**。CDN 有缓存层，更新 Manifest 后 CDN 节点可能仍然返回旧版本。解决方案：

- Manifest 的 URL 带版本号或时间戳后缀，确保每次请求到的是最新版本
- Bundle 文件名带 hash 后缀（`bundle_characters_abc123.bundle`），新版本和旧版本的 URL 不同，不存在缓存覆盖问题

**部署顺序**。先部署 Bundle 文件，再部署 Manifest。如果反过来，客户端拿到新 Manifest 后去下载 Bundle，但 Bundle 还没有部署上去，下载 404。

**多区域部署**。全球发行的项目需要在多个 CDN 区域部署。确保所有区域在发布前都已完成同步。

## 常见事故与排障

**事故：热更新后部分用户画面异常**。排查发现是 Manifest 更新了但 CDN 缓存返回了旧 Manifest。旧 Manifest 指向旧版 Bundle 文件名，但旧文件已被新文件覆盖——内容不匹配。

**根因**：Bundle 文件名没有带 hash 后缀。新版 Bundle 覆盖了旧版 Bundle 的 URL，但 CDN 缓存仍然返回旧 Manifest，旧 Manifest 的 hash 与新文件不匹配但文件名相同，没有触发重新下载。

**预防**：Bundle 文件名必须带内容 hash，确保不同版本的文件 URL 不同。

**事故：下载更新包时断网，恢复后资源残缺**。用户下载了 10 个 Bundle 中的 6 个后断网。重启后本地有 6 个新 Bundle + 4 个旧 Bundle，但本地 Manifest 还是旧的（因为步骤 4 未执行）。下次启动时按旧 Manifest 加载，6 个新 Bundle 的文件名和旧 Manifest 不匹配。

**预防**：下载到临时目录 + 原子性替换。未完成的更新不影响当前可用的旧版本。

## 小结与检查清单

- [ ] Manifest 是否包含版本号、hash、大小、依赖和校验码
- [ ] 比对算法是否正确处理新增、更新和废弃三种情况
- [ ] 下载是否支持断点续传和重试
- [ ] 下载前是否检查设备剩余空间
- [ ] 文件替换是否是原子操作（先下载到临时目录再整体替换）
- [ ] Bundle 文件名是否带内容 hash（避免 CDN 缓存覆盖问题）
- [ ] CDN 部署顺序是否先 Bundle 后 Manifest
- [ ] 下载中断后恢复时是否能正确回退到旧版本
