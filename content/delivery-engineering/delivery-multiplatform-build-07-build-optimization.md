## 这篇解决什么问题

构建时间直接决定交付链路内环的转速。如果一次构建要 2 小时，那从代码提交到 CI 反馈就是 2 小时——开发者的迭代速度被构建时间锁死。

## 为什么这个问题重要

构建时间对团队的影响不是线性的，而是指数级的：

| 单次构建时间 | 每日可构建次数 | 反馈延迟 | 团队行为变化 |
|------------|-------------|---------|------------|
| 10 分钟 | 40+ 次 | 即时 | 小步提交、快速验证 |
| 30 分钟 | 15 次 | 可接受 | 正常提交节奏 |
| 1 小时 | 8 次 | 明显 | 攒代码一起提交，风险增大 |
| 2 小时 | 4 次 | 严重 | 不等 CI 就继续开发，问题后移 |

构建时间从 30 分钟涨到 2 小时，不是"多等一会"，而是整个团队的工作模式退化。

## 构建时间的分解

优化前必须先度量。一次完整构建的时间通常分布在：

| 阶段 | 典型耗时占比 | 可优化空间 |
|------|------------|-----------|
| 资源导入（Import） | 20-40% | 大（缓存） |
| 脚本编译（C# → IL） | 5-10% | 中（asmdef 拆分） |
| IL2CPP 编译（IL → C++ → 原生代码） | 20-40% | 大（增量、缓存） |
| 资源打包（AssetBundle） | 10-20% | 中（SBP 增量） |
| 平台后处理（Xcode/Gradle 编译） | 10-20% | 中（缓存） |

**先度量再优化**。在 CI 中为每个阶段加入计时，产出构建时间报告。只优化耗时占比最大的阶段——优化 5% 的阶段不如优化 40% 的阶段。

## 四类优化手段

### 手段一：增量构建

增量构建只处理上次构建后发生变化的部分。

**资源导入增量**。Unity 的 Library 目录缓存了所有资源的导入结果。如果 Library 缓存完整，只有变更的资源需要重新导入。

CI 缓存策略：
```
构建前：从缓存服务恢复 Library/ 目录
构建后：将 Library/ 目录保存到缓存服务
缓存键：Unity版本 + 平台 + 项目哈希
```

**IL2CPP 增量编译**。Unity 2021+ 支持增量 IL2CPP 编译——只重新编译变更的 Assembly 对应的 C++ 文件。

前置条件：
- 构建目录不要每次清除（保留上次的 IL2CPP 中间产物）
- asmdef 划分合理（只变更的 Assembly 触发重新编译）

**AssetBundle 增量构建**。SBP 基于内容哈希做增量——只有内容真正变化的 Bundle 重新打包。Legacy Build Pipeline 基于文件时间戳，增量不可靠。

### 手段二：缓存策略

缓存是增量构建的基础。缓存管理的关键问题：

**缓存的存储位置**：
- 本地磁盘（同一台 Agent）：最快但不可跨 Agent 共享
- 网络存储（NFS / S3）：可跨 Agent 共享但有网络延迟
- 专用缓存服务（如 Unity Accelerator）：折中方案

**缓存的失效条件**：
- Unity 版本升级 → 全部失效
- 平台切换 → 该平台的缓存失效
- 资源导入设置变更 → 受影响资源的缓存失效

**缓存的容量管理**：
- Library 缓存可能有几 GB 到几十 GB
- 定期清理过期缓存（超过 N 天未使用的）
- 按 LRU 淘汰（最近最少使用的先清理）

### 手段三：并行拆分

将构建的不同部分拆分到多台 Agent 上并行执行：

**三端并行**（最简单的并行）：
```
同一个 commit → iOS Agent、Android Agent、WebGL Agent 同时构建
```

三端构建完全独立，天然可以并行。如果有三台 Agent，三端构建时间等于最慢的一端，而不是三端之和。

**AssetBundle 和 Player 分离**：
```
Agent A：构建 AssetBundle（资源密集）
Agent B：构建 Player（CPU 密集，IL2CPP 编译）
两者完成后 → 合并产物
```

如果项目的 AssetBundle 构建和 Player 构建都很耗时，拆分到不同 Agent 可以显著缩短总时间。

**AssetBundle 分片构建**：
将 AssetBundle 按分组拆分到多台 Agent 上并行打包。但需要处理跨分片的依赖关系——增加了复杂度。

### 手段四：构建产物归档与复用

某些构建步骤的产物可以跨构建复用：

**Shader 编译缓存**。Shader 变体的编译结果可以缓存。如果 Shader 源码没有变化，直接复用上次的编译结果。

**IL2CPP 中间产物**。IL2CPP 的 C++ 代码生成结果可以缓存。如果 Assembly 没有变化，直接复用上次的 C++ 文件。

**Native 编译缓存**。C++ 编译器的缓存（如 ccache / sccache）可以缓存编译对象文件。

## 构建时间的监控

优化不是一次性的。随着项目增长，构建时间会持续增长。需要持续监控：

**构建时间看板**：
- 每次构建的总时间和各阶段时间
- 构建时间趋势（按天 / 按周的平均值和 P95）
- 时间异常检测（某次构建时间突然翻倍 → 告警）

**时间预算**：
- 为构建时间设定预算（如"完整构建不超过 45 分钟"）
- 超预算时触发告警并排查原因

## 常见错误做法

**每次构建都清理 Library**。"为了确保干净"每次删除 Library 目录。结果每次都要全量导入资源，构建时间翻 3-5 倍。Library 缓存是增量构建的核心，不应该常规性清理。正确做法是在怀疑缓存污染时才清理，或定期（每周）做一次全量构建。

**不区分快速构建和完整构建**。每次提交都触发完整的三端 Release 构建。应该分层：提交时触发快速构建（编译检查 + 冒烟测试，5 分钟），Release 分支才触发完整构建（全量打包 + 签名，30-60 分钟）。

**只优化单台 Agent**。一台 Agent 的构建时间从 2 小时优化到 1 小时。但如果有三台 Agent 可以并行，三端并行后总时间直接从 6 小时降到 1 小时——加机器的收益可能大于优化单机。

## 小结与检查清单

- [ ] 是否有构建时间的分阶段度量
- [ ] Library 缓存是否在 CI 中复用（不每次清理）
- [ ] IL2CPP 增量编译是否启用
- [ ] AssetBundle 是否使用 SBP 的内容哈希增量
- [ ] 三端是否并行构建
- [ ] 构建时间是否有预算和监控
- [ ] 是否区分快速构建和完整构建
