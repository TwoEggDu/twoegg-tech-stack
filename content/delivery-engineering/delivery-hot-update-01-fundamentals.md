## 这篇解决什么问题

V06-07 讲了资源热更新——不重新安装就更新贴图、模型、配置等资源。脚本热更新更进一步：不重新安装就更新代码逻辑本身。

脚本热更新在国内游戏行业几乎是标配需求，但它的工程复杂度常常被低估。这一篇先把"为什么需要、边界在哪、风险是什么"说清楚，再在后续文章中讲具体方案。

## 为什么需要脚本热更新

核心驱动力只有一个：**绕过平台审核的周期。**

| 更新方式 | 周期 | 适用 |
|---------|------|------|
| 版本更新（提审） | iOS 1-3 天，Android 数小时-1 天，微信 1-3 天 | 新功能、大改动 |
| 资源热更新 | 分钟级（CDN 部署后立即生效） | 资源替换、配置修改 |
| 脚本热更新 | 分钟级 | **代码逻辑修改** |

如果线上发现一个代码 Bug（不是配置问题、不是资源问题），没有脚本热更新就只能：
1. 修复 Bug → 重新构建 → 提交平台审核 → 等 1-3 天 → 审核通过 → 发布
2. 在等待期间，线上用户持续遇到这个 Bug

有脚本热更新：修复 Bug → 构建热更包 → 部署到 CDN → 分钟级生效。

**这就是脚本热更新的全部价值：把代码修复的生效时间从天级压缩到分钟级。**

## 脚本热更新能做什么、不能做什么

### 能做的

- 修复代码逻辑 Bug（条件判断错误、算法 Bug、空引用等）
- 新增简单功能（用已有框架实现的新活动玩法、新 UI 面板）
- 修改游戏行为（技能效果调整、AI 行为修改、流程变更）

### 不能做的

- 修改引擎原生代码（Unity C++ 层、IL2CPP 运行时本身）
- 新增原生插件（新的 iOS Framework 或 Android AAR）
- 修改 App 启动流程（启动屏、权限请求等原生层面的行为）
- 突破平台 API 限制（iOS 禁止的能力不会因为热更新就能用）

**边界原则**：脚本热更新只能修改在热更新框架管辖范围内的代码。框架管辖范围之外的东西（引擎层、原生层、平台层）必须通过版本更新。

## 脚本热更新的风险

脚本热更新的价值很大，但风险也很大。以下是三类核心风险：

### 风险一：运行时兼容性

热更新的代码运行在首包的运行时环境中。如果热更代码调用了首包运行时没有的类型或方法，运行时崩溃。

```
首包（v1.0）的运行时包含：ClassA, ClassB
热更包（v1.0.1）的代码引用了：ClassA, ClassB, ClassC
→ ClassC 在首包运行时中不存在 → 运行时 TypeLoadException
```

这类问题在编译期不会报错（热更代码在构建时是用完整的代码库编译的），只在运行时加载热更程序集时才暴露。

### 风险二：元数据一致性

IL2CPP 的 AOT 编译在构建时生成泛型方法的实例化代码。如果热更代码使用了首包构建时不存在的泛型实例化，运行时会找不到对应的原生代码。

```
首包构建时存在：List<int>.Add(), List<string>.Add()
热更代码调用：List<MyNewClass>.Add()
→ List<MyNewClass>.Add() 的 AOT 代码不存在 → 运行时 ExecutionEngineException
```

HybridCLR 通过补充元数据（Supplementary Metadata）解决这个问题，但需要在构建时正确生成补充元数据。

### 风险三：安全性

热更新包通过网络下载，存在被篡改的风险。如果攻击者替换了 CDN 上的热更包或劫持了下载链路，可以在用户设备上执行任意代码。

安全措施：
- 热更包必须签名验证（下载后校验签名）
- 传输必须使用 HTTPS
- 可选：热更包加密（防止逆向分析）

## 脚本热更新与平台合规

### iOS

Apple 的审核指南禁止"下载可执行代码"，但有例外：
- WebKit/JavaScriptCore 执行的脚本代码不受限制
- 解释执行的脚本（如 Lua）在实践中被广泛接受
- HybridCLR 的解释器模式在技术上是解释执行 IL，和 Lua 解释器类似

**工程建议**：使用 HybridCLR 的解释器模式时，技术上是合规的（解释执行而非下载原生代码）。但 Apple 的审核有主观判断空间，建议不在审核说明中主动提及"热更新代码"。

### Android 和微信

Google Play 和微信对脚本热更新没有明确的限制。国内 Android 渠道更是完全接受热更新。

## 脚本热更新在交付链路中的位置

```
V07 构建 → 首包构建（包含 AOT 编译的基础代码）
             ↓
V08 热更新构建 → 热更包构建（只包含变更的程序集）
             ↓
V06 CDN 部署 → 热更包部署到 CDN
             ↓
客户端启动 → 检查热更版本 → 下载 → 校验 → 加载 → 生效
```

脚本热更新的构建和部署复用 V06（CDN 部署）和 V07（构建管线）的基础设施，但增加了额外的构建步骤和验证要求。

## 小结与检查清单

- [ ] 是否清楚脚本热更新的能力边界（能改什么、不能改什么）
- [ ] 是否理解运行时兼容性风险（首包运行时 vs 热更代码）
- [ ] 是否理解元数据一致性风险（AOT 泛型实例化）
- [ ] 热更包是否有签名验证和 HTTPS 传输
- [ ] 是否评估过平台合规风险（特别是 iOS）
- [ ] 热更新构建是否复用了首包的构建环境
