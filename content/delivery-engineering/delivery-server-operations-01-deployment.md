## 这篇解决什么问题

服务端构建产物（Docker 镜像）已经就绪，怎么把它部署到线上？直接停掉旧版本、启动新版本——对游戏服务器来说这意味着所有在线玩家断线。本篇讲三种部署策略的选型和零停机设计。

## 游戏服务器的特殊性

游戏服务器和 Web 后端不同。Web 请求是短连接、无状态的——换一个实例处理没有区别。游戏服务器通常具备以下特征：

- **有状态**：游戏世界/房间/战斗的状态保存在内存中
- **长连接**：玩家通过 TCP/WebSocket 维持持久连接
- **实时性**：服务端每帧 tick 一次，中断即卡顿

这三个特征决定了游戏服务端的部署不能简单照搬 Web 后端的方案。

## 三种部署策略对比

| 维度 | 滚动更新 | 蓝绿部署 | 金丝雀发布 |
|------|---------|---------|-----------|
| 原理 | 逐个替换实例 | 两套环境整体切换 | 小流量验证后逐步放量 |
| 资源开销 | 低（不需要额外环境） | 高（需要双倍资源） | 中（只需少量额外实例） |
| 回滚速度 | 慢（逐个回退） | 快（切回旧环境） | 快（停止放量即可） |
| 验证能力 | 弱（替换后才知道有问题） | 弱（全量切换后才知道） | 强（小流量验证） |
| 适合场景 | 常规更新、低风险变更 | 关键更新、需要即时回滚 | 高风险变更、需要线上验证 |
| 游戏服务端适用性 | 最常用 | 无状态服务适用 | 需要流量分流能力 |

## 零停机更新设计

不管选哪种策略，零停机的核心都是同一个问题：**怎么处理正在进行的游戏会话**。

### 连接排水（Connection Draining）

更新前，先让旧实例停止接受新连接，但继续服务已有连接：

```
1. 标记旧实例为"排水中"——负载均衡器不再将新玩家路由到此实例
2. 等待已有会话自然结束（对局结束、玩家退出）
3. 超时后强制断开剩余连接（给予提示）
4. 停止旧实例
```

### 会话迁移

对于持续时间长的游戏会话（MMO 的世界服务器），排水等待可能太久。此时需要会话迁移：

- 服务端将玩家状态序列化并保存到外部存储（Redis/数据库）
- 通知客户端短暂断线重连
- 客户端连接到新实例，新实例从外部存储恢复状态

### 优雅关闭（Graceful Shutdown）

服务进程收到终止信号后，不是立即退出，而是：

1. 停止接受新连接
2. 完成正在处理的请求
3. 保存必要的状态
4. 关闭数据库连接和网络连接
5. 退出进程

## 滚动更新

滚动更新是最常用的策略。原理是逐个替换实例——始终保持足够的实例在线。

**执行步骤**：

1. 从负载均衡器摘除一个旧实例
2. 对该实例执行连接排水
3. 停止旧实例，启动新实例
4. 新实例通过健康检查后加入负载均衡器
5. 对下一个实例重复以上步骤

**关键参数**：

| 参数 | 含义 | 游戏服务端建议 |
|------|------|---------------|
| maxUnavailable | 同时不可用的实例数 | 1（保守策略） |
| maxSurge | 同时多出的实例数 | 1 |
| 健康检查间隔 | 多久检查一次新实例状态 | 5-10 秒 |
| 排水等待时间 | 等待已有会话结束的最大时间 | 取决于平均对局时长 |

## 蓝绿部署

蓝绿部署维护两套完全相同的环境（蓝和绿）。当前线上跑的是蓝环境，新版本部署到绿环境，验证通过后将流量从蓝切到绿。

**优势**：切换是原子操作——瞬间完成，回滚也是瞬间（切回蓝）。

**局限**：游戏服务端的有状态性使蓝绿部署面临挑战——切换时在线玩家的状态怎么办？因此蓝绿部署更适合**无状态的游戏后端服务**（登录服务、匹配服务、排行榜服务），而不适合有状态的游戏世界服务器。

## 金丝雀发布

金丝雀发布先将一小部分流量（如 5%）路由到新版本，观察指标正常后逐步增加比例。

**游戏服务端的实现方式**：

- 匹配系统将部分新创建的房间/对局分配到新版本服务器
- 已有的游戏会话不受影响
- 通过对比新旧版本的关键指标（tick 时间、崩溃率、玩家投诉率）判断新版本是否健康

**分流维度**：

| 维度 | 适用场景 |
|------|---------|
| 按服务器 ID | 指定某几台服务器跑新版本 |
| 按区服 | 指定某个区服先更新 |
| 按玩家白名单 | 内部测试人员先验证 |
| 按新房间比例 | 新创建的房间有 N% 分配到新版本 |

## 决策框架

选择部署策略的简单决策树：

1. **常规更新、低风险** → 滚动更新（成本低、流程简单）
2. **关键更新、需要即时回滚** → 蓝绿部署（限无状态服务）
3. **高风险变更、需要线上验证** → 金丝雀发布（需要分流基础设施）
4. **大版本更新** → 停服维护（传统做法，MMO 常用，用公告提前通知玩家）

大多数游戏团队的实际做法：常规更新用滚动更新，大版本更新用停服维护。金丝雀发布需要额外的基础设施投入，适合规模较大的项目。

## 小结

- [ ] 是否根据变更风险选择了合适的部署策略
- [ ] 是否实现了连接排水（旧实例停止接受新连接）
- [ ] 是否实现了优雅关闭（进程收到终止信号后有序退出）
- [ ] 滚动更新是否在健康检查通过后才路由流量
- [ ] 高风险变更是否有金丝雀发布或分区灰度能力
- [ ] 是否有分钟级回滚方案
