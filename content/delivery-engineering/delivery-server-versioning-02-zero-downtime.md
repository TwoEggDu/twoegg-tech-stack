## 这篇解决什么问题

V11-01 讲了三种部署策略（滚动/蓝绿/金丝雀）。本篇深入零停机更新的工程设计——不是"选哪种策略"，而是"选完之后怎么落地"。重点解决游戏服务器的有状态难题。

## 为什么游戏服务端必须零停机

| 场景 | 停机影响 |
|------|---------|
| 玩家正在 PVP 对局 | 对局中断，胜负无法判定 |
| 玩家正在 Boss 战 | 进度丢失，消耗品已使用但奖励未发放 |
| 公会战/GVG 进行中 | 大规模断线，活动结果争议 |
| 玩家正在交易 | 交易中断，可能出现物品/货币异常 |

任何一个场景出现都是线上事故。零停机不是加分项，而是必选项。

## 连接排水（Connection Draining）

零停机的第一步：让旧实例"逐渐退出"而非直接停止。

### 排水流程

1. 通知负载均衡器不再将新连接路由到该实例
2. 通知匹配系统不再在该实例创建新房间
3. 已有连接继续正常服务，定时检查剩余连接数
4. 超过最大等待时间后，通知剩余玩家即将维护，给予 30-60 秒缓冲
5. 强制断开剩余连接

### 排水参数设计

| 参数 | 含义 | 建议值 |
|------|------|--------|
| drain_timeout | 最大等待时间 | 对局类：最大对局时长 + 缓冲；MMO：15-30 分钟 |
| check_interval | 检查剩余连接间隔 | 5-10 秒 |
| force_disconnect_notice | 强制断开前通知时间 | 30-60 秒 |
| min_drain_time | 最短排水时间 | 10 秒（防止健康检查延迟） |

## 优雅关闭序列

连接排水完成后，服务进程需要有序关闭——顺序很重要，乱序可能导致数据丢失。

```
1. 接收终止信号（SIGTERM）
2. 停止接受新连接
3. 执行连接排水
4. 保存所有内存中的状态到持久层（玩家数据、房间状态、消息队列 flush）
5. 关闭数据库和缓存连接
6. 从服务注册中心注销
7. 退出进程（exit code 0）
```

| 常见错误 | 后果 | 正确做法 |
|---------|------|---------|
| 收到 SIGTERM 后立即退出 | 内存数据丢失 | 注册 SIGTERM 处理器 |
| 先关数据库再保存数据 | 数据写入失败 | 先保存再关连接 |
| 没有关闭超时 | 进程永远无法退出 | 设置最大关闭时间（如 60 秒） |

## 健康检查集成

| 类型 | 检查内容 | 失败后果 |
|------|---------|---------|
| Liveness | 进程是否存活 | 重启实例 |
| Readiness | 是否准备好接收流量 | 从负载均衡器摘除 |
| Startup | 初始化是否完成 | 等待（不重启） |

Readiness 返回 true 的前提：数据库连接可用、缓存连接可用、配置数据已加载、游戏逻辑初始化完成、服务注册成功。**不要只检查端口是否监听**——端口开了不代表服务就绪。

## 部署速度与安全的权衡

| 方案 | 速度 | 安全性 | 适用 |
|------|------|--------|------|
| 逐台滚动，等排水完成 | 慢 | 最高 | 有状态游戏服务器 |
| 分批滚动（每批 N 台） | 中等 | 高 | 无状态服务 |
| 蓝绿切换 | 快 | 中等 | 无状态服务、短连接 |
| 停服更新 | 快 | 最高 | 大版本更新 |

## 维护窗口：何时不值得做零停机

零停机增加工程复杂度。以下场景可以考虑计划停机：

| 场景 | 原因 |
|------|------|
| 数据库 Schema 大变更 | 迁移可能耗时且影响性能 |
| 底层架构调整 | 无法在旧架构上做排水 |
| 玩家数量极少的凌晨时段 | 影响面小，工程收益不高 |

维护窗口最佳实践：提前 24 小时游戏内公告，选择在线最低时段（通常凌晨 3:00-6:00），预估时长留 50% 缓冲，准备回滚方案。

## 小结

- [ ] 是否实现了连接排水（旧实例停止接受新连接）
- [ ] 排水超时时间是否匹配业务场景
- [ ] 进程是否注册了 SIGTERM 处理器执行优雅关闭
- [ ] 关闭序列是否正确（先保存数据，后关闭连接）
- [ ] Readiness 检查是否验证了所有依赖就绪
- [ ] 是否根据变更风险选择了合适的更新策略
- [ ] 大版本更新是否有计划维护窗口方案
