## 这篇解决什么问题

前两篇讲了美术资源线。这一篇切换到配置线——策划填表、程序导表、运行时读表的完整链路。

配置管线的问题往往比资源管线更隐蔽：一个配置错误不会导致画面异常或加载失败，但可能导致数值失衡、逻辑错误甚至经济系统崩溃。

## 为什么这个问题重要

配置是游戏逻辑的数据驱动层。一个中型项目可能有 200-500 张配置表，覆盖技能、掉落、商城、关卡、AI、任务等所有系统。

配置出错的典型代价：

- 技能冷却时间填成 0，玩家无限释放技能（逻辑错误）
- 掉落概率填成 100%，经济系统在几小时内崩溃（经济失衡）
- 物品 ID 引用了一个被删除的资源，进入背包就崩溃（引用断裂）
- 配置表新增了一列但导表工具没有更新解析逻辑，整张表读取失败（工具链断裂）

这些问题的共同特征：**发布前的功能测试通常覆盖不到，因为测试用的配置和实际发布的配置可能不同。**

## 本质是什么

一条完整的配置管线包含六个环节：

```
编辑 → 导出 → 验证 → 序列化 → 加载 → 热更新
```

### 编辑：策划怎么填表

大多数游戏项目的配置编辑工具是 Excel 或 Google Sheets。也有项目使用自研的配置编辑器或在线工具。

编辑环节的核心问题：

**Schema 定义**。每张表的每一列应该有明确的类型定义：整数、浮点数、字符串、枚举、外键引用。Schema 不明确，验证就无从下手。

**多人协作**。多个策划同时编辑同一张表时的冲突管理。Excel 文件的合并冲突几乎无法自动解决——这是很多团队改用 Google Sheets 或自研工具的主要原因。

**版本追踪**。配置变更需要能追溯到"谁在什么时候改了什么"。Excel 文件在 Git 中的 diff 不可读——需要工具把变更提取为可读的文本格式。

### 导出：从编辑格式到中间格式

导出环节把 Excel / Sheets 转换为程序可处理的中间格式（通常是 JSON / CSV / 自定义文本格式）。

导出环节的设计要点：

**确定性**。同一份 Excel 在不同机器上导出的结果必须完全一致（字节级一致）。否则 CI 和本地的构建产物会不同。

**增量感知**。导出工具应该能识别哪些表发生了变更，只导出变更的部分。全量导出在表数量多时速度很慢。

**错误报告**。导出过程中发现的格式错误（类型不匹配、缺失必填列）应该以清晰的报告形式输出，而不是在控制台里打一堆日志。

### 验证：导出后的自动化检查

验证是配置管线中最关键也最容易被忽视的环节。下一篇文章会详细展开验证的设计，这里先列出需要验证的维度：

- 类型检查：每个字段的值是否符合 Schema 定义的类型
- 范围检查：数值是否在合理范围内（攻击力不能为负，概率不能超过 100%）
- 引用完整性：外键引用的目标是否存在（物品 ID 是否在物品表中，资源 ID 是否在资源清单中）
- 唯一性检查：主键是否唯一
- 一致性检查：跨表引用是否双向一致

### 序列化：从中间格式到运行时格式

验证通过的中间格式需要序列化为运行时可高效读取的二进制格式。

序列化格式的选型考量：

| 格式 | 读取速度 | 文件大小 | 可读性 | 热更新友好 |
|------|---------|---------|--------|-----------|
| JSON | 慢 | 大 | 好 | 好（文本 diff） |
| MessagePack | 快 | 小 | 差 | 好 |
| FlatBuffers | 最快（零拷贝） | 小 | 差 | 中 |
| Protobuf | 快 | 小 | 差 | 好 |
| ScriptableObject（Unity） | 快 | 中 | 中 | 差（二进制序列化） |

选型的核心原则：**开发期用可读格式（方便调试），发布期用高效格式（减少加载时间和包体大小）。**

### 加载：运行时读取配置

配置在运行时的加载策略：

**全量加载**。游戏启动时一次性读取所有配置到内存。简单直接，但配置量大时会影响启动时间和内存占用。

**按需加载**。进入某个系统时才加载对应的配置。减少内存占用，但增加了加载时机的管理复杂度。

**分层缓存**。频繁访问的配置常驻内存，低频配置按需加载并缓存。平衡了内存和加载时间。

### 热更新：不重新安装的配置推送

配置热更新是交付效率的关键能力。相比资源热更新和脚本热更新，配置热更新通常是最轻量的：

- 配置文件体积小，下载快
- 配置变更通常不涉及代码变更，验证范围可控
- 配置回退简单——推送上一个版本的配置文件即可

配置热更新的设计要点：
- 客户端启动时检查配置版本，如有新版本则下载替换
- 必须有配置版本号管理，不能用文件时间戳判断（时间戳在不同时区和设备上可能不一致）
- 必须有本地回退机制——如果新配置导致崩溃，下次启动时恢复为首包内的配置

## 常见错误做法

**导表工具只有一个人会维护**。导表工具通常是项目早期某个程序写的脚本，没有文档，没有测试，没有人接手。工具出了问题只能找这个人——这是单点故障。

**配置和代码的变更不同步**。代码增加了一个新的配置字段，但配置表还没有加上这一列。或者反过来，配置表加了新列，但代码还没有更新解析逻辑。必须在提交流程中建立配置和代码的联动检查。

**开发期和发布期用同一套配置**。开发期为了调试方便，配置里可能有测试数据、GM 命令开关、超高倍率。发布前忘记切回正式配置是一个高频事故源。开发配置和发布配置应该通过构建参数或环境变量切换，不应该靠手动替换。

## 小结与检查清单

- [ ] 每张配置表是否有明确的 Schema 定义（字段名、类型、范围、是否必填）
- [ ] 导出工具是否确定性（同一份表不同机器导出结果一致）
- [ ] 导出后是否有自动化验证（至少覆盖类型检查和引用完整性）
- [ ] 序列化格式是否区分开发期和发布期
- [ ] 配置热更新是否有版本号管理和本地回退机制
- [ ] 导表工具是否有文档和至少两个人能维护
- [ ] 开发配置和发布配置是否通过构建参数切换
