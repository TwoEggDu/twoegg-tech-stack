## 这篇解决什么问题

RACI 矩阵定义了谁该做什么，审批流程规定了怎么走，但如果团队文化是"出了问题先找人背锅"、"复盘就是开会走形式"、"自动化太麻烦下次再说"，再好的流程也会被绕过。文化是流程能否真正执行的底层土壤——本篇讨论交付体系最需要的三根文化支柱。

## 文化的定义

文化不是贴在墙上的口号，也不是年会上的价值观宣讲。工程文化的实操定义：

> **文化 = 没人监督时团队的默认行为模式**

测试方法——回答这三个问题：
1. 上线出了 Bug，第一反应是找"谁写的代码"还是"为什么自动化没拦住"？
2. 上一次线上事故后，复盘的 Action Item 现在完成了几项？
3. 团队里有没有人在手动做一件每周重复三次的事情？

这三个问题分别检验质量文化、复盘文化和自动化文化的真实水平。

## 支柱一：质量文化

### 核心转变

从"谁的 Bug"到"系统的问题"。

| 旧模式 | 新模式 |
|--------|--------|
| "这个 Crash 是 A 同学写的代码" | "这个 Crash 为什么没被测试拦住" |
| "A 同学应该更仔细" | "代码评审规则是否覆盖了这类模式" |
| "下次注意" | "加一条自动化检查" |

**为什么要去掉"人的责任"**：不是说个人不需要负责，而是把改进方向从"人要更小心"（不可控）转向"系统要更完善"（可控）。人会犯错是确定的事实，系统能不能拦住错误才是可改进的变量。

### 质量文化的可观测行为

| 行为 | 健康信号 | 不健康信号 |
|------|---------|-----------|
| Bug 讨论 | "这类问题怎么预防" | "这个 Bug 谁写的" |
| 代码评审 | 关注设计和风险 | 只关注格式和命名 |
| 测试投入 | 每个功能都有验证计划 | "赶进度，测试后面补" |
| 质量指标 | 团队主动看 Crash 趋势 | 只有 QA 看质量数据 |
| 缺陷修复 | 修完 Bug 顺手加防护 | 修完 Bug 就结束 |

### 质量文化不是口号：可观测的行为差异

怎么判断一个团队的质量文化是真的还是口号？不用看墙上贴的标语，看这几个场景下团队的实际反应就行了：

**场景一：线上出了一个 P1 Bug，团队群里的第一条消息是什么？**
- 口号型团队："这个是谁改的？"（找人）
- 文化型团队："这个 Crash 的调用栈是什么？影响范围多大？"（找问题）

**场景二：Code Review 中发现了一个潜在的空指针问题，Reviewer 怎么写评论？**
- 口号型团队："这里可能空指针，加个判断"（指出问题但不讨论为什么）
- 文化型团队："这里的数据来源是异步回调，异步场景下对象可能已销毁。建议加空判断 + 在回调注册时加弱引用。类似的模式在 BattleManager 里也有，是不是需要统一处理？"（讨论模式而不是个案）

**场景三：QA 报了一个 Bug，开发修完后的动作是什么？**
- 口号型团队：修完 → 标记已修复 → 等 QA 验证 → 关工单
- 文化型团队：修完 → 加了一条防止同类问题的自动化测试 → 检查其他模块有没有类似写法 → 标记已修复

**场景四：Sprint 最后两天进度紧张，测试还没跑完。**
- 口号型团队："测试先放放，下个 Sprint 补上"（从来补不上）
- 文化型团队："砍掉一个低优先级功能，保住测试时间"（质量优先于范围）

这些场景中的行为差异不是培训能教出来的——它需要 Leader 在每一次实际发生时做出正确的示范和决策。质量文化的建设周期通常是 6-12 个月，没有捷径。

### 建设方法

1. **Leader 先做到**——主程在 Code Review 中示范"讨论问题而不是讨论人"
2. **把 Blame 从流程里去掉**——Bug 报告模板不设"责任人"字段，改为"受影响模块"
3. **奖励防御性行为**——表扬加检查、加测试、加防护的人，不只表扬修 Bug 快的人
4. **可视化质量趋势**——让团队看到"我们的 Crash 率在持续下降"，建立集体成就感
5. **把质量行为纳入绩效评估**——不是"谁引入的 Bug 多"这种负面指标，而是"谁主动加了多少防护措施""谁的代码 Bug 密度在持续下降"这种正向指标

## 支柱二：复盘文化

### 核心原则

每次事故必须产出可执行的改进项，改进项必须跟踪到完成。

| 要素 | 要求 |
|------|------|
| 触发条件 | 每次线上事故都触发复盘，不分大小 |
| 时间要求 | 事故解决后 48h 内完成复盘 |
| 参与者 | 相关角色全员参与，不只是出问题的团队 |
| 产出格式 | 标准化的 Postmortem 文档（V15-06 模板） |
| Action Item | 每项 AI 有明确 Owner、Deadline、验收标准 |
| 跟踪闭环 | Action Item 纳入下一个 Sprint/迭代的待办 |

### 复盘的三个常见失败模式

**失败模式一：只开会不记录**

开了两小时复盘会，大家讨论了很多，但没人写 Postmortem。一个月后同类问题再次发生，没人记得上次讨论了什么。

**修复**：复盘会的组织者必须在会议结束前完成 Postmortem 文档的框架，24h 内补充完整。

**失败模式二：记录了但 Action Item 没人跟**

Postmortem 写了三条 Action Item，但没有排进迭代计划。下一次复盘发现上次的 AI 一条都没做。

**修复**：Action Item 必须创建工单（不是 Postmortem 文档里的文字），纳入 Sprint Backlog，和功能需求一样有优先级和 Deadline。

**失败模式三：复盘变成追责会**

复盘会上领导追问"谁干的"，团队成员开始互相甩锅。之后没人愿意主动报告问题。

**修复**：复盘的第一条规则——**不讨论人，只讨论系统**。如果领导在复盘会上追责，复盘主持人有权叫停并重新引导。

### 复盘质量的衡量

| 指标 | 健康值 | 说明 |
|------|--------|------|
| 复盘覆盖率 | 100% | 每次事故都有 Postmortem |
| AI 完成率 | > 80% | Action Item 在 Deadline 前完成的比例 |
| 复发率 | < 10% | 同类问题在 90 天内再次发生的比例 |
| 复盘时效 | < 48h | 事故解决到 Postmortem 完成的时间 |

## 支柱三：自动化文化

### 核心原则

如果一个人手动做了两次同样的事情，第三次应该交给机器。

### 自动化成熟度

| 级别 | 特征 | 示例 |
|------|------|------|
| L0 纯手动 | 全靠人记忆和手工操作 | 手动打包、手动跑测试 |
| L1 脚本化 | 有脚本但需要人触发 | `build.sh`、`run_test.sh` |
| L2 管线化 | 脚本串成管线自动执行 | CI 提交触发构建 + 测试 + 部署 |
| L3 自愈化 | 异常自动处理，人只处理例外 | 构建失败自动重试、自动回滚 |

**大多数团队卡在 L1 → L2 的跨越**：有脚本但没串起来，人还是需要记住"先跑这个脚本再跑那个"。V16 CI/CD 管线系列详细讨论了这个跨越。

### 自动化文化的可观测行为

| 行为 | 健康信号 | 不健康信号 |
|------|---------|-----------|
| 重复操作 | "我来写个脚本" | "反正就几分钟，手动做吧" |
| 脚本维护 | 脚本有文档、有测试 | 脚本只有作者能跑 |
| 新流程引入 | "新流程的自动化方案是什么" | "先手动跑着，以后再自动化" |
| 工具投入 | 工具链有专人维护 | 脚本坏了等原作者修 |

### 自动化文化的一个真实转折点

某个项目在引入自动化文化时有一个转折点特别值得分享。当时团队每次发版需要手动执行 17 个步骤（配置检查、版本号修改、构建触发、签名验证、提审操作等），每次耗时约 45 分钟，而且每次都有人漏掉某个步骤。

有人提出"写个脚本自动化这 17 步"，但一直没人动手——因为大家觉得"45 分钟也不是很久""脚本写起来麻烦""万一脚本出错了更麻烦"。

转折点是一次发版事故：有人在第 11 步（修改版本号）时写错了数字，导致新包的版本号比线上版本低，Google Play 拒绝了提审。排查和修复花了 3 小时，加上重新构建和提审等待，那次发版延误了一天半。

事故后有人算了一笔账：
- 手动发版每次 45 分钟，每月发 4 次 = 每月 3 小时
- 加上每季度平均 1 次因手动失误导致的事故（平均 4 小时修复）= 每季度 4 小时
- 年化成本：3 * 12 + 4 * 4 = 52 小时

而写一个发版自动化脚本预计需要 16-24 小时。**投资回收期不到半年，之后每年节省 50+ 小时**——还没算上减少事故带来的隐性收益。

这笔账一算清楚，团队内部对自动化的态度立刻转变了。经验教训：推动自动化文化最有效的方式不是说教，而是**量化手动操作的成本**，让所有人看到数字。

### 建设方法

1. **给自动化留时间**——每个 Sprint 预留 10-15% 的容量用于工具和自动化改进
2. **内部 Demo**——每月一次工具 Demo，让团队看到自动化带来的效率提升
3. **降低自动化门槛**——提供脚本模板、CI 模板、文档模板，让写自动化比手动操作更容易
4. **量化手动操作**——统计每个月花在重复手动操作上的工时，让成本可见
5. **"自动化债务"看板**——维护一个列表记录所有"还在手动做的重复操作"，每个 Sprint 从中选 1-2 个自动化掉

## 三根支柱的关系

三种文化不是独立的，而是互相加强的：

```
质量文化 → 发现问题后不甩锅 → 复盘才能产出真正的改进
复盘文化 → AI 要求加防护 → 自动化检查落地
自动化文化 → 自动化拦住问题 → 质量提升
```

如果只建其中一个：
- 只有质量文化但没有复盘文化——发现了系统问题但改进不落地
- 只有复盘文化但没有自动化文化——Action Item 写了"加检查"但没人实现
- 只有自动化文化但没有质量文化——自动化了很多东西但检查的方向不对

## 文化健康度快速评估

对团队做以下 6 个问题的快速调研（匿名，1-5 分），平均分反映文化健康度：

| 问题 | 测量维度 |
|------|---------|
| 报告 Bug 时我不担心被追责 | 质量文化-安全感 |
| 代码评审中会讨论设计和风险 | 质量文化-深度 |
| 每次线上事故都有 Postmortem | 复盘文化-覆盖 |
| 上次复盘的 Action Item 我知道完成了多少 | 复盘文化-闭环 |
| 我最近一个月写过/改过自动化脚本 | 自动化文化-参与 |
| 团队会给工具改进预留时间 | 自动化文化-投入 |

**评分标准**：平均分 > 4.0 = 文化健康；3.0-4.0 = 有基础但需强化；< 3.0 = 需要专项改进。

## 检查清单

- [ ] 质量文化：Bug 讨论聚焦系统而非个人
- [ ] 质量文化：Bug 报告模板没有"责任人"字段
- [ ] 复盘文化：每次事故都有 Postmortem，覆盖率 100%
- [ ] 复盘文化：Action Item 创建为工单并纳入 Sprint
- [ ] 自动化文化：每个 Sprint 预留 10-15% 容量用于工具改进
- [ ] 自动化文化：重复手动操作的工时有统计
- [ ] 文化健康度快速评估已执行且有改进计划
