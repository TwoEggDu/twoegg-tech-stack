## 这篇解决什么问题

版本号看起来是最简单的工程决策之一——不就是 1.0.0 改成 1.1.0 吗？

但在多端交付场景下，版本号的设计直接影响：热更新能不能匹配到正确的资源包、客户端和服务端能不能兼容、平台商店能不能正确提示用户更新、出了问题能不能追溯到准确的构建产物。

## 为什么这个问题重要

版本号设计出问题的典型场景：

- 热更新系统用版本号判断"是否需要下载新资源"，但 iOS 和 Android 的版本号不一致，导致 Android 用户跳过了一次关键的资源更新
- 服务端用客户端版本号判断协议兼容性，但版本号只有 Major.Minor 没有 Patch，无法区分"需要强更"和"可以热更"
- 出了线上问题要定位构建产物，但版本号 1.2.0 对应了三次不同的构建（因为没有 Build Number），不知道用户手上的是哪一次
- 微信小游戏的版本号格式和原生端不同，CDN 上的资源版本匹配逻辑出错

## 本质是什么

版本号不是给人看的标签，而是交付链路中多个系统共同依赖的**机器可读标识符**。它在以下环节被使用：

| 使用者 | 怎么用版本号 |
|--------|------------|
| 平台商店 | 判断是否需要提示用户更新 |
| 热更新系统 | 判断需要下载哪个版本的资源包 |
| 服务端 | 判断客户端协议是否兼容 |
| 崩溃上报 | 把 Crash 堆栈关联到正确的符号表 |
| 构建归档 | 追溯到产出这个包的具体 commit 和构建环境 |
| 灰度系统 | 控制不同版本的用户比例 |

一个版本号要同时满足这六个使用场景的需求。

## 版本号的四段结构

推荐的版本号结构：

```
Major.Minor.Patch (Build)
  1  .  2  .  3    (456)
```

### Major：大版本

**含义**：不向后兼容的重大变更。通常对应一次大版本更新或赛季更新。

**变更条件**：客户端和服务端之间的协议不兼容——旧客户端无法连接新服务端。

**对交付的影响**：Major 变更意味着强制更新。用户必须到平台商店下载新版本，不能通过热更新跳过。

### Minor：功能版本

**含义**：向后兼容的功能更新。新增了功能、内容或修改了行为，但旧版本客户端仍然可以连接服务端。

**变更条件**：有新功能或内容上线，但不需要强制更新。

**对交付的影响**：Minor 变更通常通过平台商店的常规更新推送。服务端需要同时支持 Minor 和 Minor-1 版本的客户端。

### Patch：修复版本

**含义**：不含新功能的 Bug 修复或热更新。

**变更条件**：修复了线上问题，通过热更新或小版本更新推送。

**对交付的影响**：Patch 变更通常不需要用户去商店更新——通过热更新静默推送。热更新系统用 Patch 号判断"当前客户端需不需要下载新的热更包"。

### Build：构建号

**含义**：同一个版本号可能经历多次构建（修复构建问题后重新出包）。Build Number 唯一标识每一次构建产物。

**变更条件**：每次 CI 构建自动递增。

**对交付的影响**：Build Number 不对用户可见，但对内部追溯至关重要——崩溃上报、符号表匹配、构建归档都依赖它。

## 三端版本号：统一还是独立

| 方案 | 做法 | 优点 | 缺点 |
|------|------|------|------|
| **统一** | 三端用同一个 Major.Minor.Patch | 简单、一致、服务端兼容逻辑简单 | 某端独立修复时版本号被迫联动 |
| **独立** | 每端有自己的版本号体系 | 灵活、各端独立发版 | 兼容性矩阵复杂、容易出错 |
| **统一 + 平台后缀** | 1.2.3-ios / 1.2.3-android / 1.2.3-wx | 主版本统一、平台可区分 | 需要工具支持后缀解析 |

**推荐**：**Major.Minor 统一，Patch 和 Build 可以独立。**

理由：
- Major.Minor 决定了客户端和服务端的协议兼容范围，必须三端一致
- Patch 是端内修复，某一端修了一个 Bug 不需要拖动其他端
- Build Number 天然按构建独立，每端每次构建不同

## 版本号与平台商店的对应

三个平台对版本号有不同的格式要求：

| 平台 | 展示版本 | 内部版本 | 格式约束 |
|------|---------|---------|---------|
| iOS | CFBundleShortVersionString (1.2.3) | CFBundleVersion (456) | 三段数字，递增 |
| Android | versionName (1.2.3) | versionCode (10020300) | versionCode 必须整数递增 |
| 微信小游戏 | version (1.2.3) | — | 字符串，无强制格式 |

**Android versionCode 的设计**：versionCode 是一个整数，必须每次发版递增。推荐的编码方式：

```
versionCode = Major * 10000000 + Minor * 100000 + Patch * 100 + Build
例：1.2.3 (Build 1) → 10200301
```

这样 versionCode 既能递增，又能反向解析出 Major.Minor.Patch.Build。

## 常见错误做法

**版本号手动维护**。每次发版前有人手动改 PlayerSettings 里的版本号。手动操作容易忘记、容易改错。版本号应该由 CI 自动生成。

**没有 Build Number**。版本号只有 1.2.3，没有构建号。同一个 1.2.3 可能构建了三次（第一次构建失败、第二次配置有误、第三次成功），但线上崩溃日志里的 1.2.3 无法确定是哪一次的产物。

**热更新不带版本号**。热更新包的文件名是 `update.zip`，没有版本标记。多次热更新后客户端不知道自己当前的热更版本是什么，出了问题无法追溯。

## 小结与检查清单

- [ ] 版本号是否有 Major.Minor.Patch.Build 四段结构
- [ ] Major.Minor 是否三端统一
- [ ] Build Number 是否由 CI 自动生成
- [ ] Android versionCode 是否有规范的编码方式
- [ ] 热更新包是否带版本标识
- [ ] 崩溃上报是否能通过版本号 + Build Number 定位到具体的构建产物
