前三篇定义了交付的概念、画出了链路全景、拆开了质量维度。这一篇把这些内容收成一个方法论模型——交付飞轮。

飞轮不是一个理论框架，而是一个可操作的思维工具。它的作用是：当你面对交付链路上的任何问题时，能快速定位"这个问题在哪个环节"、"该由哪道门拦住"、"属于哪个环的职责"。

## 为什么是飞轮，不是流水线

很多团队把交付理解成一条流水线：

```
代码 → 编译 → 打包 → 测试 → 上传 → 审核 → 上线 → 完成
```

流水线模型有一个根本缺陷：**它是单向的，到"完成"就结束了。**

但真实的交付不是到"上线"就结束的。上线后会发现新问题，新问题需要定位根因，根因需要在上游修复，修复需要重新走完构建和验证，新版本需要再次发布。

交付是一个环，不是一条线。

飞轮模型的核心特征是：**每一轮转动的产出不只是"把这个版本送达用户"，还包括"让下一轮转得更快、更稳"。** 这就是飞轮效应——前期推动阻力大，但每一轮积累的改进会让后续的转动越来越轻。

## 八个环节

交付飞轮由八个环节组成，首尾相接形成闭环：

```
生产 → 集成 → 构建 → 验证 → 发布 → 监控 → 闭环 → 改进 → (下一轮生产)
```

每个环节的职责：

| 环节 | 职责 | 产出 |
|------|------|------|
| **生产** | 内容、资源、配置、代码四条线的合规产出 | 可合入的变更集 |
| **集成** | 多条线的产出合并到统一代码库 | 可构建的代码库状态 |
| **构建** | 从代码库生成各平台可执行产物 | 签名并归档的构建产物 |
| **验证** | 确认产物满足功能/性能/稳定性/兼容性标准 | 发布/不发布的判断 |
| **发布** | 把产物送达用户，管控上线风险 | 版本到达用户设备 |
| **监控** | 观察版本健康指标，检测异常 | 版本健康报告或告警 |
| **闭环** | 问题分类 → 根因定位 → 修复 → 回归 → 防复发 | 修复方案 + 防御措施 |
| **改进** | 复盘 → 流程变更 → 自动化沉淀 | 更好的流程和工具 |

八个环节中，**没有一个是可选的**。每个团队都在做这八件事，区别只在于做到了什么程度——是有意识地设计了工程方案，还是依赖个人经验和临时操作。

## 三道门

八个环节之间不是自由流动的。在三个关键的阶段转换点上，应该设置质量门——不满足条件，不允许进入下一阶段。

### 质量门：构建 → 验证

**位置**：构建完成后、进入验证前。

**检查项**：
- 编译是否全部通过（零编译错误，零链接错误）
- 资源是否通过规范检查（格式、大小、命名、引用完整性）
- Shader 变体数量是否在阈值内
- 构建产物大小是否在包体预算内
- 构建是否可重复（同一代码库状态再构建一次，产物一致）

**设计原则**：质量门的检查应该尽可能自动化。人工判断的门禁会成为瓶颈，自动化的门禁可以在分钟内给出结果。

### 验证门：验证 → 发布

**位置**：验证完成后、进入发布前。

**检查项**：
- 冒烟测试全部通过
- 功能回归测试通过率达到阈值
- 性能指标在基线范围内，无退化
- Crash 率和 ANR 率在阈值内
- 目标设备矩阵验证通过
- 已知问题已分类并确认为可接受

**设计原则**：验证门需要人工判断和自动化检查结合。自动化提供数据，人工做最终的发布决策。

### 发布门：发布 → 用户

**位置**：准备推送给用户前。

**检查项**：
- 发布审批流程完成（至少两人确认）
- 灰度策略已确认（起始比例、观察窗口、扩量条件）
- 回滚预案已验证（回滚后的版本确认可用）
- 热修复通道已就绪（如需紧急修复可走快速通道）
- 平台审核已通过（iOS / Android / 微信各自的审核状态）

**设计原则**：发布门是最后一道防线。它不检查技术细节（那是前两道门的事），而是确认"万一出了问题，我们有退路"。

### 三道门的关系

```
生产 → 集成 → 构建 ──[质量门]──→ 验证 ──[验证门]──→ 发布 ──[发布门]──→ 用户
```

越靠前的门应该越严格、越自动化——因为越早拦住问题，修复成本越低。

越靠后的门应该越侧重风险管控——因为到了发布阶段，问题已经不是"对不对"，而是"万一出事怎么办"。

## 两个环

八个环节实际上形成了两个嵌套的循环，分别驱动开发期和运营期的交付：

### 内环：开发期循环

```
生产 → 集成 → 构建 → 验证 → (发现问题) → 修复 → 重新构建 → 重新验证
```

内环是开发期的日常循环。团队在这个环里不断迭代：写代码、合入、构建、验证、发现问题、修复、再验证。

内环转得越快，团队的迭代速度就越高。加速内环的关键手段：

- 缩短构建时间（增量构建、缓存）
- 提高验证效率（自动化测试、并行验证）
- 减少合入冲突（合理的分支策略、频繁合入）

### 外环：运营期循环

```
发布 → 监控 → (发现问题) → 闭环 → 改进 → 下一轮生产
```

外环是版本上线后的运营循环。版本发布后，通过监控发现问题，通过闭环修复问题，通过改进防止同类问题复发。

外环的质量决定了团队的运营能力。外环强的团队：

- 问题在监控阶段被发现，而不是被用户投诉发现
- 修复在小时级完成，而不是天级
- 同类问题不会在下一个版本重现

### 两环的衔接

内环和外环通过**改进段**衔接。每一轮外环转完后，改进段的产出（流程变更、自动化工具、检查项增补）会进入下一轮内环的生产段，让内环的下一轮转得更顺畅。

这就是飞轮效应的来源：外环每转一圈，内环就更高效一点。

## 一个轴心：持续改进

飞轮的轴心是持续改进。没有这个轴心，飞轮就只是在原地重复同样的动作，不会变得更好。

持续改进不是"写复盘文档"。它的实质是：**每一次交付完成后，至少有一件事变得不需要手动做了，或者有一类问题变得不可能再发生了。**

改进的典型形态：

**从人工到自动化**。上一个版本的资源检查是人工做的，这个版本把检查规则变成了 CI 里的自动化脚本。

**从事后到预防**。上一个版本的 Crash 是上线后被用户发现的，这个版本增加了构建后的稳定性冒烟测试。

**从个人到流程**。上一个版本的回滚是靠一个人的经验操作的，这个版本把回滚步骤变成了可重复的检查清单和脚本。

**从单次到沉淀**。上一个版本的事故复盘结论停留在会议纪要里，这个版本把结论变成了 CI 门禁规则。

如果每个版本的改进段都能沉淀 1-2 项自动化或流程改进，10 个版本后，团队的交付能力会和一年前完全不同。

## 飞轮模型的使用方法

这个模型的使用不是"背下来"，而是在遇到具体问题时用来定位：

**定位问题在哪个环节**。"这次上线 Crash 率飙升"——问题暴露在监控段，但根因可能在生产段（代码质量）、构建段（编译配置）或验证段（测试覆盖不足）。沿着飞轮往上游追，直到找到根因所在的环节。

**判断该由哪道门拦住**。"这类 Shader 变体爆炸的问题怎么预防"——应该在质量门拦住，加入变体数量的自动化检查。

**区分是内环问题还是外环问题**。"构建时间太长影响迭代速度"——这是内环问题，需要优化构建缓存和增量构建。"上线后发现问题响应太慢"——这是外环问题，需要优化监控告警和热修复通道。

**评估改进的价值**。"这次改进应该投入在哪里"——看飞轮的哪个环节最慢或最脆弱，改进那个环节对整体转速的提升最大。
