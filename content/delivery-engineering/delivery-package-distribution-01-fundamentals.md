## 这篇解决什么问题

V05 讲了资源怎么打包、怎么加载、怎么管理版本。但还有一个更上层的架构决策没有回答：**什么资源放在首包里、什么资源按需下载、什么资源通过热更新推送？**

这个决策不是打完包以后再看"包大了就砍一些"，而是一开始就应该设计好的架构。

## 为什么这个问题重要

包体架构影响三个方面：

**用户获取成本**。首包越大，下载时间越长，放弃率越高。移动端用户在 Wi-Fi 下载超过 200MB 会犹豫，蜂窝网络下载超过平台限制会被阻止。

**运营灵活度**。热更新可以覆盖的内容越多，不需要提审就能修复的问题越多。但热更新能力取决于架构设计——如果所有资源都打进首包且不支持替换，热更新就不存在。

**平台合规**。三个平台各有包体限制：iOS 蜂窝网络下载限制、Android AAB 基础模块上限、微信小游戏主包限制。超出限制意味着提审不通过或用户体验受损。

## 本质是什么

包体架构的本质是把项目的全部资源分成三层：

```
┌─────────────────────────────────────┐
│  首包（First Install）               │
│  用户安装后立刻可用的内容             │
│  体积受平台限制约束                   │
├─────────────────────────────────────┤
│  追加下载（On-Demand Download）       │
│  用户需要时才下载的内容               │
│  在游戏内提示下载                     │
├─────────────────────────────────────┤
│  热更新（Hot Update）                │
│  不需要用户主动操作的静默更新          │
│  启动时或后台自动下载                 │
└─────────────────────────────────────┘
```

### 首包：用户安装后立刻能做什么

首包包含用户安装游戏后**不需要额外下载就能开始体验**的所有内容。

首包应该包含：
- 引擎运行时和核心框架代码
- 启动流程需要的 UI 资源（登录、主界面）
- 新手引导的全部资源（让用户第一次体验完整）
- 基础 Shader 和通用材质
- 配置框架和启动配置

首包不应该包含：
- 所有角色、所有关卡（除非总量很小）
- 高清过场动画（体积大、使用频率低）
- 多语言的全部语音包（按用户语言按需下载）

**首包的目标**：让用户尽快开始玩。不是"下载所有内容"，而是"下载够玩第一次的内容"。

### 追加下载：用户需要时再给

追加下载的内容在用户需要时提示下载：

- 进入新关卡前提示"下载关卡资源（50MB）"
- 抽到新角色后提示"下载角色资源（20MB）"
- 切换语言时提示"下载语音包（100MB）"

追加下载的设计要点：

**时机透明**。用户应该知道为什么要下载、下载多大、需要多久。不要在不可预期的时候弹出下载提示。

**可中断可恢复**。下载中断后重新进入同一内容时应该从断点继续，不要重新下载。

**离线可用**。已下载的内容在断网后仍然可用，不需要每次联网验证。

### 热更新：用户无感知的更新

热更新是在用户启动游戏时或游戏运行中静默下载的更新包。用户不需要去商店更新，不需要手动确认。

热更新通常用于：
- Bug 修复（修复了一个配置错误或资源问题）
- 内容更新（新增活动资源、调整数值配置）
- 紧急修复（线上崩溃的紧急补丁）

热更新的设计要点：

**包体小**。热更新包通常在启动时强制下载，用户等待。包体应该控制在 10-50MB 以内。如果超过 100MB，应该考虑是否改为追加下载或版本更新。

**可回退**。热更新失败时必须能回退到上一个可用版本。V05-06 已经讲了原子性替换的机制。

**版本兼容**。旧版首包 + 新版热更包必须能正确工作。如果热更包依赖了新版首包才有的功能，旧版用户会崩溃。

## 三层的比例设计

三层的比例没有标准答案，取决于项目类型：

| 项目类型 | 首包 | 追加 | 热更 | 特征 |
|---------|------|------|------|------|
| 轻度休闲 | 95% | 0% | 5% | 内容少，全放首包 |
| 中度游戏 | 40% | 40% | 20% | 首包放核心内容，关卡按需下载 |
| 重度 MMO | 20% | 60% | 20% | 首包只放登录和新手，大部分按需下载 |
| 微信小游戏 | 5% | 80% | 15% | 主包极小，几乎全部按需 |

比例设计的原则：**首包越小用户获取成本越低，但首次体验越依赖网络。**平衡点取决于目标用户的网络环境和耐心。

## 包体预算的制定

包体预算应该从平台限制和用户体验倒推：

| 约束 | 数值 | 来源 |
|------|------|------|
| iOS 蜂窝网络下载限制 | 200MB（2024 年标准） | Apple 政策 |
| Android AAB 基础模块上限 | 150MB | Google Play 政策 |
| 微信小游戏主包上限 | 4MB / 8MB / 12MB（按类别） | 微信政策 |
| 用户 Wi-Fi 下载耐心阈值 | ~500MB | 行业经验 |
| 用户蜂窝下载耐心阈值 | ~100MB | 行业经验 |
| 设备存储空间安全线 | 安装大小 < 2GB | 低端设备约束 |

基于这些约束，典型的包体预算：

```
首包（安装包）：150-300MB
  其中代码：20-40MB
  其中基础资源：100-200MB
  其中引擎运行时：10-30MB

追加下载总量：500MB-2GB
  按需下载，不计入安装包

热更新单次包体：10-50MB
  启动时强制下载
```

## 与交付链路的关系

包体架构是交付链路中连接"构建段"和"发布段"的关键决策：

```
V05 资源管线（怎么打包）
    ↓
V06 包体管理（什么放首包、什么按需、什么热更）
    ↓
V07 多端构建（按包体架构构建三端产物）
    ↓
V09 平台发布（按平台限制提交审核）
```

包体架构决定了构建管线需要产出哪些产物（首包 + 追加下载包 + 热更新包），发布管线需要上传哪些文件到哪里（商店 + CDN）。

## 常见错误做法

**先打包再看大小**。先把所有资源打进首包，发现超了再砍。这种做法导致每次发版前都在紧急删资源，质量无法保证。应该先定预算再生产。

**追加下载没有预加载策略**。用户进入新关卡时才开始下载，等待时间长。应该在用户还在上一个关卡时预测下一步可能去哪，后台预下载。

**热更新包没有大小限制**。一次热更新推了 200MB，用户在启动时等待下载 5 分钟。热更新包应该有大小上限，超出时考虑拆成多次推送或改为版本更新。

## 小结与检查清单

- [ ] 是否有明确的首包 / 追加下载 / 热更新三层架构设计
- [ ] 首包大小是否在平台限制和用户体验阈值内
- [ ] 首包是否包含完整的首次体验（不需要额外下载就能开始玩）
- [ ] 追加下载是否有预加载策略
- [ ] 热更新单次包体是否有大小上限
- [ ] 包体预算是否从平台限制和用户体验倒推
