## 这篇解决什么问题

一个团队的时候，分支策略简单、发布节奏自主、资源冲突靠喊一嗓子就能解决。当项目增长到 3+ 个团队并行开发，协调开销急剧上升——分支冲突变频繁、谁先合入谁后合入变成政治问题、发布窗口变成稀缺资源。这篇从分支策略、资源冲突、发布列车、沟通协议四个维度解决多团队协作问题。

## 协调开销的二次方增长

团队间的沟通链路数 = n×(n-1)/2：

| 团队数 | 沟通链路数 | 协调体感 |
|--------|-----------|---------|
| 1 | 0 | 不需要协调 |
| 2 | 1 | 偶尔对齐 |
| 3 | 3 | 开始感到摩擦 |
| 5 | 10 | 需要专人协调 |
| 8 | 28 | 必须有正式流程 |

**启示**：多团队协作不是"大家多沟通"就能解决的——需要结构化的协调机制来降低每条链路上的协调成本。

## 从 1 个团队到 3 个团队的真实过渡故事

某个项目的团队规模经历了一次典型的增长。最初只有一个 12 人的全功能团队，后来因为项目扩张分成了战斗组、社交组和经济组三个团队。以下是过渡过程中遇到的具体痛点和解决方案：

### 第一阶段：分团队后的头两周（混乱期）

分团队后第一周就出了问题。战斗组修改了 `Assets/Scripts/Core/EventSystem.cs`（公共事件系统），社交组同时在改 `Assets/Scripts/Core/UIManager.cs`。两个改动都合入了 main，但它们之间有隐式依赖——EventSystem 的改动影响了 UIManager 的事件注册顺序。结果 main 分支上社交系统的 UI 全部无法响应点击。

排查这个问题花了一整天——因为两个改动分别看都没问题，只有合在一起才出问题。更糟糕的是，没人知道这个问题是什么时候引入的，因为两个改动间隔不到 2 小时。

**教训**：公共模块（Core 目录）必须有单一的 Owner。分团队后的第一件事应该是划定目录所有权，特别是公共模块的所有权。

### 第二阶段：引入目录所有权后（第 3-4 周）

划定了目录所有权后，跨目录修改需要 Review。但新问题来了：**Review 响应时间变成了瓶颈**。社交组提了一个修改 Core/NetworkManager 的 MR，架构组的 Reviewer 当天在处理别的事情，第二天才 Review。Review 来回了两轮，总共花了三天。社交组的功能因此延误了。

**解决方案**：
1. 给跨团队 Review 设了 SLA——普通 MR 24 小时内必须首次 Review，紧急 MR 4 小时内
2. 每个 Owner 团队至少指定两名 Reviewer，互为备份——一个人忙的时候另一个接上
3. 小改动（< 20 行，不涉及接口变更）可以先合入后 Review（事后审查），大改动必须先 Review 后合入

### 第三阶段：引入发布列车后（第 5-8 周）

三个团队的开发节奏不一样——战斗组做大功能（两周一个迭代），社交组做小功能（一周一个迭代），经济组需要频繁调数值（几乎每天都有改动）。如果按最快的节奏发版，战斗组的功能永远做不完就被催着发版；如果按最慢的节奏，经济组的数值调整积压严重。

最终的方案是**双轨发版**：
- 大版本发布列车：双周一班，三个团队的功能统一上车
- 热更新通道：经济组的数值调整走热更，不等大版本。社交组的小功能如果只涉及脚本层也可以走热更

这个方案的关键在于**明确哪些改动可以走热更通道**——必须有清晰的分类标准，不能每次都临时讨论。

### 三个月后的状态

过渡完成后的关键数字：

| 指标 | 分团队前 | 过渡期（第 1 个月） | 稳定后（第 3 个月） |
|------|---------|-------------------|-------------------|
| 集成冲突频率 | 无（单团队） | 每周 5-8 次 | 每周 1-2 次 |
| 冲突平均解决时间 | — | 4-6 小时 | 30-60 分钟 |
| MR Review 响应时间 | 同团队 2-4h | 跨团队 24-72h | 跨团队 4-24h |
| 发布延迟（因协调问题） | 0 次/月 | 2-3 次/月 | 0-1 次/月 |

## 多团队分支策略

### 团队分支模型

在 V04-02 的基础分支策略上扩展：每个团队有自己的长期集成分支，通过定期合入主干来同步。

```
main（主干）
├── team/combat      （战斗团队分支）
├── team/social      （社交团队分支）
├── team/economy     （经济团队分支）
└── release/2.1      （发布分支）
```

### 集成节奏

| 策略 | 集成频率 | 适用场景 | 风险 |
|------|---------|---------|------|
| 持续集成 | 每日 | 团队间依赖紧密 | 冲突频繁但每次小 |
| 周期集成 | 每周 | 团队间相对独立 | 冲突不频繁但每次大 |
| 里程碑集成 | 按阶段 | 完全独立的模块 | 集成时可能爆炸 |

**推荐**：除非团队之间完全没有代码交叉，否则至少保持**每周集成一次**。超过两周不集成，合并冲突的解决成本会急剧上升。

### 集成日流程

```
集成日（每周三，举例）
├── 08:00  各团队冻结 team 分支提交
├── 09:00  按优先级顺序合入 main
│   ├── 第一优先级：combat（本版本核心功能）
│   ├── 第二优先级：economy（依赖 combat 的接口）
│   └── 第三优先级：social（独立模块）
├── 10:00  自动化集成测试
├── 11:00  冲突解决（如有）
└── 12:00  各团队从 main 同步到 team 分支
```

## 资源冲突防控

### 目录所有权

用目录划分代替文件级划分——每个团队"拥有"特定的目录树：

| 目录 | 所属团队 | 修改规则 |
|------|---------|---------|
| `Assets/Scripts/Combat/` | 战斗团队 | 其他团队修改需 Combat 团队 Review |
| `Assets/Scripts/Social/` | 社交团队 | 其他团队修改需 Social 团队 Review |
| `Assets/Scripts/Core/` | 架构组 | 任何修改需架构组 Review |
| `Assets/Art/UI/` | UI 团队 | TA 协助 Review 资源规范 |

### CODEOWNERS 配置

```
# .github/CODEOWNERS 示例
/Assets/Scripts/Combat/    @team-combat
/Assets/Scripts/Social/    @team-social
/Assets/Scripts/Core/      @arch-team
/Assets/Art/               @ta-team
/ProjectSettings/          @toolchain-team
/Packages/                 @toolchain-team
```

**关键规则**：
- 公共模块（Core、Packages）的修改需要架构组 Review——防止"偷偷改公共库然后影响所有团队"
- CODEOWNERS 配置本身也需要 Review——防止有人删除自己模块的 Owner 规则

### 跨团队 Review 机制

| 场景 | Review 要求 | 响应时间承诺 |
|------|------------|-------------|
| 修改自己的目录 | 团队内 Review | 4h 内 |
| 修改其他团队目录 | 对方团队 + 本团队 Review | 24h 内 |
| 修改公共模块 | 架构组 Review | 24h 内 |
| 紧急修复跨目录 | 对方团队 Lead 口头确认 + 后补 Review | 即时 |

## 发布列车协调

### 什么是发布列车

发布列车是固定节奏的发布模式——不管功能做完了多少，到点就发版。没赶上这班车的功能等下一班。

| 属性 | 说明 |
|------|------|
| 发车频率 | 通常双周或月度（游戏项目常见月度） |
| 上车截止时间 | 发车前 N 天（通常 3-5 天），过了截止时间不再接受新功能合入 |
| 谁决定上车 | Release Manager 根据功能完成度和测试状态决定 |
| 没赶上车 | 不加塞、不延期——等下一班。这是纪律 |

### 发布列车的决策流程

```
发车前 5 天：功能提名
├── 各团队提交本次想上车的功能列表
├── 每个功能附带：完成度、测试状态、依赖关系
│
发车前 3 天：上车确认
├── Release Mgr 评估所有提名功能
├── 确认上车列表（必须全部通过集成测试）
├── 未通过集成测试的功能 → 下车
│
发车前 1 天：封版
├── release 分支从 main 切出
├── 只接受 Bug 修复，不接受新功能
│
发车日：审批 + 发布
└── 走 V18-02 审批流程
```

### 功能开关配合

当一个功能没赶上发布列车但代码已经合入 main 时：

- **不要 revert 代码**——revert 可能影响其他功能
- **用 Feature Flag 关闭**——代码在但功能不可见，下个版本打开即可
- **Feature Flag 需要 Review**——防止遗忘的 Flag 变成永久暗角

## 跨团队沟通协议

### 日常沟通

| 机制 | 频率 | 参与人 | 目的 |
|------|------|--------|------|
| 团队内 Stand-up | 每日 | 团队全员 | 同步进展和阻塞 |
| 跨团队同步会 | 每周 | 各团队 Lead + Release Mgr | 同步依赖和风险 |
| 集成日同步 | 集成日 | 各团队代表 + Toolchain | 处理集成冲突 |

### 发布前沟通

| 机制 | 时间点 | 参与人 | 目的 |
|------|--------|--------|------|
| 发布就绪评审 | 发车前 3 天 | 全角色 | 确认上车功能列表 |
| Go/No-Go 会议 | 发车前 1 天 | Release Mgr + 各团队 Lead | 最终发布决策 |
| 发布通知 | 发车日 | 全员 | 告知发布已开始 |

### 冲突升级路径

当两个团队对合入顺序、资源使用、发布优先级有分歧时：

```
团队 Lead 直接沟通（30 min 内解决）
├── 解决 → 记录决策
└── 未解决 → 升级到 Release Mgr
    ├── Release Mgr 裁定（2h 内）
    └── 重大分歧 → 技术委员会裁定（24h 内）
```

**原则**：升级不是甩锅，是把决策交给有更多上下文的人。升级路径必须有时间承诺——不能"回去讨论讨论"然后无限期挂起。

### 多团队协作中最被低估的问题：配表冲突

代码冲突大家都会关注，但实际操作中最频繁、最难解决的冲突往往是**配表冲突**。游戏项目的配表（Excel/JSON/CSV）通常是二进制或半结构化格式，Git 的合并能力对它们几乎无效。

某个项目战斗组和经济组同时修改了同一张道具配表——战斗组加了新的武器道具，经济组调整了现有道具的价格。两个人改的不是同一行，但 Git 无法自动合并 Excel 文件。结果每次集成日都要花 1-2 小时手动合并配表。

**最终的解决方案**：
1. 配表拆分——按功能域拆成多张表，减少多团队同时修改同一张表的概率
2. 配表用 CSV/JSON 而不是 Excel——文本格式 Git 可以行级合并
3. 如果必须用 Excel，用工具把 Excel 转成 JSON 后提交 JSON，合并在 JSON 层面做
4. 配表修改走专门的 MR 通道，不和代码变更混在一起——这样冲突时只需要处理配表，不用同时处理代码
