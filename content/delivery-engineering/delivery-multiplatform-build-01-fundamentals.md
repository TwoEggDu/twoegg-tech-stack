## 这篇解决什么问题

构建系统是交付链路的中枢：上游所有生产线（内容、资源、配置、代码）在这里汇聚，下游所有验证和发布从这里开始。构建不可靠，后面全部白做。

但很多团队对构建系统的理解停留在"点 Build 按钮"。这一篇用四段模型把构建系统拆开，让每一段的职责和失败模式都可管理。

## 本质是什么

任何构建系统都可以拆成四段：

```
输入（Input）→ 变换（Transform）→ 产物（Output）→ 验证（Verify）
```

### 输入段：构建的原料

构建的输入包括：

| 输入 | 来源 | 变化频率 |
|------|------|---------|
| 源代码 | 版本库（Git） | 高（每天多次） |
| 资源文件 | 版本库（Git LFS / Perforce） | 高 |
| 配置数据 | 导表产物 | 高 |
| 构建脚本 | 版本库 | 低 |
| 引擎版本 | 项目锁定的 Unity/Unreal 版本 | 极低（每大版本 0-1 次） |
| SDK 和插件 | 版本锁定的第三方依赖 | 低 |
| 构建参数 | CI 传入的参数（平台、配置、版本号） | 每次构建不同 |
| 构建环境 | CI Agent 的操作系统、工具链版本 | 应该固定不变 |

**输入段的工程要求：确定性。**同一组输入必须产出完全相同的产物。如果构建结果依赖于构建时间、机器名、环境变量的随机差异，构建就不可重复。

输入段的常见失败模式：
- 构建环境漂移——CI Agent 升级了 OS 或 SDK 版本，构建行为改变
- 依赖版本不锁定——构建时拉取了最新版 SDK，和上次不同
- 构建参数遗漏——漏传了某个关键参数，产物和预期不同

### 变换段：从原料到产物

变换段执行实际的编译、打包、签名等操作。对游戏项目来说，变换通常包括：

```
脚本编译（C# → IL → 原生代码）
    ↓
资源导入（源文件 → 引擎格式）
    ↓
资源打包（引擎格式 → AssetBundle / 资源包）
    ↓
平台构建（代码 + 资源 → 平台产物）
    ↓
签名（产物 + 证书 → 签名产物）
    ↓
归档（签名产物 + 元信息 → 归档包）
```

**变换段的工程要求：可追溯。**每一步的输入和输出都应该被记录，构建失败时能定位到是哪一步出了问题。

变换段的常见失败模式：
- 编译错误——某个 asmdef 引用了不存在的类型
- 资源导入失败——某个资源的导入设置和引擎版本不兼容
- 打包依赖断裂——AssetBundle 的依赖图中有环或缺失
- 签名失败——证书过期、Provisioning Profile 不匹配

### 产物段：构建的输出

构建的最终产物因平台不同：

| 平台 | 产物形态 | 主要组成 |
|------|---------|---------|
| iOS | Xcode 项目 → .ipa | 可执行文件 + 资源 + Frameworks |
| Android | Gradle 项目 → .apk / .aab | DEX + native libs + assets |
| 微信小游戏 | WebGL 目录 | .wasm + .js + 资源 |

**产物段的工程要求：可归档、可追溯。**

每次构建的产物应该：
- 标记构建号（Build Number），关联到触发这次构建的 commit
- 归档到构建服务器或对象存储，保留 N 个版本
- 可以从归档中恢复——发版用的产物必须可追溯到来源

### 验证段：产物是否合格

构建成功不等于产物正确。验证段在构建完成后自动执行检查：

| 检查项 | 方法 | 阻止级别 |
|--------|------|---------|
| 包体大小 | 与预算比对 | 超预算 20% → 阻止 |
| 编译警告 | 计数和分类 | 新增关键警告 → 阻止 |
| 资源完整性 | 检查关键资源是否存在 | 缺失 → 阻止 |
| 签名有效性 | 验证签名证书 | 无效 → 阻止 |
| 基本启动 | 在模拟器或真机上启动 | 启动崩溃 → 阻止 |

验证段是构建系统的"质量门"——V01 讲的交付飞轮中的第一道门就在这里。

## 构建的可重复性

可重复性是构建系统最重要的工程属性：**同一个 commit + 同一组构建参数，无论在哪台机器上构建多少次，产物都应该完全一致。**

破坏可重复性的常见因素：

| 因素 | 表现 | 解法 |
|------|------|------|
| 时间戳写入产物 | 每次构建的产物哈希不同 | 禁止在产物中嵌入构建时间 |
| 环境变量差异 | 不同 Agent 产出不同 | CI 环境标准化 |
| 并行编译的顺序不确定 | 偶发的产物差异 | 固定编译顺序或接受确定性差异 |
| 缓存污染 | 增量构建复用了过期产物 | 清理缓存或校验缓存一致性 |
| 依赖版本浮动 | 构建时拉取了不同版本的依赖 | 锁定所有依赖版本 |

**验证可重复性的方法**：同一个 commit 构建两次，比对产物哈希。如果不一致，排查上面的因素。

## 构建系统与交付链路的关系

```
V02 内容生产  ─┐
V03 工程基建  ─┤→ V07 构建系统 →┬→ V08 热更新构建
V04 版本管理  ─┤                ├→ V09 平台发布
V05 资源管线  ─┤                └→ V13 验证与测试
V06 包体分发  ─┘
```

构建系统是汇聚点——前面六卷的工程决策最终都在构建系统中落地。构建系统的质量决定了后面所有环节的基础。

## 常见错误做法

**构建只在本地执行**。某个程序的电脑上能打出正确的包，但其他人的环境不行。构建必须在 CI 上执行，本地构建只用于开发期调试。

**构建失败后手动修复再重试**。CI 构建失败后，有人手动到 CI Agent 上清理缓存、改配置、重新触发。这些手动操作不可追溯、不可重复。构建失败应该修复代码或配置后重新触发，不应该手动干预 CI Agent。

**不归档构建产物**。发版用的包没有归档，出了线上问题后找不到当时的构建产物来复现。每次发布的产物必须归档，保留至少 6 个月。

## 小结与检查清单

- [ ] 构建是否在 CI 上执行（不依赖个人电脑）
- [ ] 构建输入是否全部确定（依赖版本锁定、环境固定）
- [ ] 同一个 commit 的两次构建产物是否一致（可重复性）
- [ ] 构建失败是否能定位到具体步骤（可追溯性）
- [ ] 构建产物是否标记了 Build Number 并归档
- [ ] 构建后是否有自动化验证（包体、签名、基本启动）
