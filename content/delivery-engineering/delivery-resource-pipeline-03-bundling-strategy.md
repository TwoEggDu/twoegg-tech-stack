## 这篇解决什么问题

序列化解决了"单个资源怎么变成字节"。打包解决"多个资源怎么组合成包"。

打包策略是资源管线中最难标准化的环节——因为最优策略取决于项目的加载模式、更新频率和设备限制。本篇不给"标准方案"，而是给出做决策的框架。

## 为什么这个问题重要

打包策略选错的代价：

- **粒度太粗**：一个 Bundle 200MB，加载某个角色要拉整个 Bundle。热更新改了一个贴图，用户要重新下载 200MB
- **粒度太细**：一万个 Bundle 各 100KB，加载一个场景要发起上千次 IO 请求。移动设备上随机 IO 极慢
- **依赖混乱**：A Bundle 依赖 B、B 依赖 C、C 依赖 A，形成循环。加载顺序无法确定，随机崩溃
- **冗余爆炸**：同一个贴图被打进了 5 个 Bundle，包体膨胀 5 倍。内存中同时存在 5 份相同贴图

## 本质是什么

打包策略的本质是在三个相互约束的目标之间找平衡：

```
加载性能（IO 次数少）  ←──→  热更新效率（增量包小）
           ↕
     内存效率（无冗余）
```

粒度粗 → IO 少但增量大 → 冗余少但浪费加载
粒度细 → IO 多但增量小 → 冗余风险高
提取公共包 → 无冗余但增加依赖链

没有完美的平衡点。决策取决于项目的具体约束。

## 五种分组原则

### 原则一：按逻辑模块分组

把同一个游戏模块的资源放在一起：所有角色资源一个 Bundle，所有 UI 资源一个 Bundle，每个关卡一个 Bundle。

```
bundle_characters/    → 所有角色的模型、贴图、动画
bundle_ui/           → 所有 UI 的 Atlas、Prefab
bundle_level_01/     → 关卡 1 的场景、物件
bundle_level_02/     → 关卡 2 的场景、物件
```

**适用场景**：模块之间相对独立，加载以模块为单位（进入战斗时加载战斗资源，进入商城时加载商城 UI）。

**优点**：逻辑清晰，加载和卸载的单位与游戏逻辑对齐。
**缺点**：单个 Bundle 可能很大；模块间的共享资源处理复杂。

### 原则二：按更新频率分组

把更新频率相近的资源放在一起。经常变化的资源和很少变化的资源分开。

```
bundle_static/       → 基础 Shader、通用材质、系统 UI（几乎不更新）
bundle_characters/   → 角色资源（版本更新时可能新增角色）
bundle_events/       → 活动相关资源（每周活动更新）
bundle_configs/      → 配置数据（可能每天热更新）
```

**适用场景**：热更新频繁的项目。把高频更新的资源和低频更新的资源分开，减少热更包体。

**优点**：热更新包体最小化——更新活动资源时只下载 bundle_events。
**缺点**：分组维度不够直觉，需要持续维护更新频率的归类。

### 原则三：按加载时机分组

把同一时机需要加载的资源放在一起。启动时需要的资源一个 Bundle，进入战斗时需要的一个 Bundle。

```
bundle_boot/         → 启动必需：登录界面、Logo、基础 UI
bundle_lobby/        → 大厅：主界面、角色展示、商城
bundle_battle/       → 战斗：战斗场景、技能特效、HUD
bundle_cutscene/     → 过场：剧情动画、对话资源
```

**适用场景**：加载时间敏感的项目。把每个加载节点的资源打在一起，减少加载时的 IO 次数。

**优点**：加载性能最优——每次加载只拉一个或少数几个 Bundle。
**缺点**：跨时机共享的资源（基础 Shader、通用贴图）需要抽成公共包，增加依赖管理复杂度。

### 原则四：按单体资源分组

每个独立的逻辑资源一个 Bundle。每个角色一个 Bundle，每个武器一个 Bundle。

```
bundle_hero_warrior/   → 战士角色的模型+贴图+动画
bundle_hero_mage/      → 法师角色
bundle_weapon_sword/   → 剑武器
bundle_weapon_staff/   → 法杖
```

**适用场景**：资源可以按需下载的项目（类似 Gacha 游戏，抽到某个角色后才下载该角色资源）。

**优点**：按需下载粒度最细，用户只下载需要的内容。
**缺点**：Bundle 数量爆炸；共享资源（骨骼、通用材质）的处理复杂。

### 原则五：混合策略

实际项目通常混合使用多种原则：

```
顶层：按逻辑模块分（角色/场景/UI/特效/音频）
模块内：按单体或加载时机细分
公共资源：按类型抽取（公共 Shader、公共贴图、公共材质）
```

混合策略需要明确的规则文档，否则随着项目演进，分组会逐渐失控。

## 依赖管理

当资源跨 Bundle 引用时，就产生了 Bundle 间的依赖关系。

### 依赖的三种处理方式

**方式一：冗余打入**。被引用的资源复制一份打进引用方的 Bundle。

```
bundle_A 包含 texture_shared（冗余副本）
bundle_B 包含 texture_shared（冗余副本）
→ 包体增大，但 A 和 B 互不依赖
```

适用于：被少数 Bundle 引用的小资源（复制成本低）。

**方式二：提取公共包**。把被多个 Bundle 引用的资源提取到公共 Bundle。

```
bundle_shared 包含 texture_shared
bundle_A 依赖 bundle_shared
bundle_B 依赖 bundle_shared
→ 无冗余，但加载 A 前必须先加载 shared
```

适用于：被大量 Bundle 引用的大资源（提取收益高）。

**方式三：自动依赖分析**。工具自动检测跨 Bundle 引用，按阈值决定冗余还是提取。

```
如果引用者 < 3 个 → 冗余打入
如果引用者 >= 3 个 → 提取公共包
```

实际项目推荐自动分析 + 人工审核。

### 依赖链的管理原则

- **禁止循环依赖**：A→B→C→A 会导致加载死锁。构建时必须检测并报错
- **依赖深度可控**：依赖链不超过 3 层。超过 3 层意味着加载一个 Bundle 要先级联加载多个前置
- **公共包数量可控**：公共包不超过总 Bundle 数的 10-20%。过多的公共包会让依赖关系变成蜘蛛网

## 冗余控制

冗余的检测和治理：

**检测**：构建后分析所有 Bundle 的内容，统计每个资源出现在多少个 Bundle 中。出现 2 次以上的就是冗余。

**治理优先级**：
1. 大资源冗余（>100KB）→ 必须提取
2. 中资源冗余（10-100KB）× 高冗余次数（>5 次）→ 建议提取
3. 小资源冗余（<10KB）→ 通常容忍（提取的依赖管理成本高于冗余成本）

**持续监控**：在 CI 中加入冗余检测，每次构建后输出冗余报告。设定包体冗余率阈值（如 <5%），超出时告警。

## 常见错误做法

**不做依赖分析就定打包策略**。先拍脑袋定了"按场景分组"，构建后发现场景间大量共享资源导致冗余爆炸。应该先分析资源的引用关系图，再决定分组策略。

**公共包只有一个大包**。所有共享资源全部丢进一个 `bundle_common`。结果这个包 50MB，加载任何一个功能都要先加载这个巨大的公共包。公共包也应该按类型或加载时机拆分。

**打包策略定了之后不再调整**。项目初期的打包策略适合当时的资源量。两年后资源量翻了 10 倍，原来的策略早已不合适。每个大版本应该重新评估打包策略。

## 小结与检查清单

- [ ] 打包分组原则是否明确（按模块/更新频率/加载时机/单体/混合）
- [ ] 跨 Bundle 依赖是否有处理策略（冗余打入 vs 提取公共包的阈值）
- [ ] 是否禁止循环依赖（构建时检测）
- [ ] 依赖链深度是否可控（不超过 3 层）
- [ ] 冗余率是否在 CI 中持续监控
- [ ] 打包策略是否定期重新评估
