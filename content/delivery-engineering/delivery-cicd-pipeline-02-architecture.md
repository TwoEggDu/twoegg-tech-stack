## 这篇解决什么问题

V16-01 建立了五阶段管线模型。但游戏项目不是"一条管线跑完就结束"——至少要出 iOS、Android、微信三个平台的包，每个平台的构建环境和工具链都不同。这一篇讲管线架构怎么设计，重点是多端并行、扇入扇出和失败隔离。

## 单管线 vs 多端并行

### 单管线：小团队的起步方案

```
触发 → 构建 Android → 检查 → 部署
```

一条管线一次只构建一个平台。切换平台需要重新跑一遍。

**适用场景**：只出一个平台的项目，或 CI 资源极有限的小团队。

**问题**：三个平台串行构建耗时是并行的三倍。

### 多端并行：生产环境的标准方案

```
              ┌→ 构建 iOS     → 检查 iOS     ─┐
触发 → 公共阶段 ─┼→ 构建 Android → 检查 Android ─┼→ 聚合 → 部署 → 通知
              └→ 构建 微信     → 检查 微信     ─┘
```

一次触发同时启动多个平台的构建。各平台独立执行，最后汇聚结果。

## 扇出与扇入

### 扇出（Fan-out）

一个触发事件分裂为多个并行任务：

| 扇出维度 | 示例 |
|---------|------|
| 按平台 | iOS / Android / 微信 / PC |
| 按构建类型 | Debug / Release / Profile |
| 按资源配置 | 高清资源包 / 标清资源包 |

扇出的前提是各分支任务之间无依赖——iOS 构建不需要等 Android 构建的结果。

### 扇入（Fan-in）

多个并行任务完成后汇聚到一个判定点：

```
iOS 构建通过     ─┐
Android 构建通过  ─┼→ 全平台通过 → 允许发布
微信构建通过     ─┘
```

扇入的判定策略：

| 策略 | 含义 | 适用场景 |
|------|------|---------|
| 全部通过 | 所有平台都通过才允许继续 | 正式发布 |
| 部分通过 | 指定必要平台通过即可继续 | 日常开发构建 |
| 任一通过 | 至少一个平台通过即可继续 | 紧急修复只出一端 |

## 失败隔离

**核心原则：一个平台的构建失败不应阻塞其他平台。**

### 常见失败场景

| 失败类型 | 示例 | 影响范围 |
|---------|------|---------|
| 平台特有代码编译失败 | iOS 的 Objective-C 桥接报错 | 仅 iOS |
| 平台 SDK 更新导致不兼容 | Android targetSdkVersion 升级 | 仅 Android |
| 签名证书过期 | iOS Distribution Certificate 过期 | 仅 iOS |
| 构建机故障 | macOS Agent 磁盘满 | 仅 iOS（如果只有一台 Mac） |
| 公共代码编译失败 | C# 编译错误 | 全平台 |

### 隔离策略

```
公共阶段（编译 C# + 资源检查）
    ↓ 公共阶段失败 → 全平台终止，立即通知
    ↓ 公共阶段通过 → 进入平台分支
    ├→ iOS     失败 → 仅标记 iOS 失败，通知 iOS 负责人
    ├→ Android 通过 → 继续 Android 部署
    └→ 微信    通过 → 继续微信部署
```

关键设计：平台分支之间不互相等待、不互相阻塞。

## 快速通道 vs 完整通道

不是每次提交都需要跑全平台完整构建。按触发类型选择通道：

| 通道 | 触发条件 | 包含阶段 | 耗时目标 |
|------|---------|---------|---------|
| 快速通道 | 每次 push 到特性分支 | 编译 + Lint + 单元测试 | ≤5 分钟 |
| 标准通道 | PR/MR 创建或更新 | 编译 + 测试 + 资源检查 + 单平台构建 | ≤15 分钟 |
| 完整通道 | 合并到主干 / 每日定时 | 全平台构建 + 完整测试 + 性能基线 | 30-60 分钟 |
| 发布通道 | 手动触发 / Tag 创建 | 全平台构建 + 签名 + 上传商店 | 60-90 分钟 |

**快速通道的目标是让开发者在提交后 5 分钟内知道有没有破坏编译。** 这是 CI 最核心的价值——快速反馈。

## Pipeline as Code

管线定义应该作为代码存储在版本库中，而不是在 CI 工具的 Web 界面上手动配置：

| 工具 | 管线定义文件 | 位置 |
|------|------------|------|
| Jenkins | Jenkinsfile | 仓库根目录 |
| GitHub Actions | .github/workflows/*.yml | 仓库 .github 目录 |
| GitLab CI | .gitlab-ci.yml | 仓库根目录 |

### Pipeline as Code 的好处

- **版本化**：管线变更有 Git 历史，可以回滚
- **代码审查**：管线变更通过 PR 审查，避免误改
- **一致性**：所有环境用同一份管线定义
- **可移植性**：换 CI 工具时管线逻辑可参考

### 管线定义的分层

```
pipeline/
├── Jenkinsfile              # 入口：定义阶段和触发规则
├── scripts/
│   ├── build_unity.sh       # 构建脚本（引擎无关的通用逻辑）
│   ├── build_ios.sh         # iOS 特有步骤（Xcode 构建、签名）
│   ├── build_android.sh     # Android 特有步骤（Gradle 构建）
│   ├── build_wechat.sh      # 微信特有步骤
│   ├── check_quality.sh     # 质量检查脚本
│   └── deploy.sh            # 部署脚本
└── config/
    ├── build_config.json    # 构建参数（版本号、环境变量）
    └── quality_thresholds.json  # 质量门阈值
```

管线入口文件编排阶段和流程，具体操作委托给独立脚本——这样脚本可以在本地测试，不必每次都通过 CI 触发来调试。

## 管线元数据

每次管线执行应该记录元数据，便于事后追溯：

| 元数据 | 用途 |
|--------|------|
| Build Number | 唯一标识这次构建 |
| Commit Hash | 这次构建对应的代码版本 |
| 触发者 | 谁触发了这次构建 |
| 触发原因 | push / PR / 定时 / 手动 |
| 各阶段耗时 | 识别构建变慢的阶段 |
| 各阶段结果 | 通过/失败/跳过 |
| 产物路径 | 构建产物存储在哪里 |

## 小结与检查清单

- [ ] 是否采用多端并行而非串行构建
- [ ] 扇出维度是否覆盖所有目标平台
- [ ] 扇入判定策略是否区分日常构建和正式发布
- [ ] 平台构建失败是否实现了隔离（不互相阻塞）
- [ ] 是否区分了快速/标准/完整/发布四种通道
- [ ] 管线是否以代码形式存储在版本库中
- [ ] 管线脚本是否可以本地独立测试
