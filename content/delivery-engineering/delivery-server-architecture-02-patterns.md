## 这篇解决什么问题

游戏服务端架构选型通常从功能角度讨论。这一篇从交付视角看：四种架构模式对构建、部署、扩展和更新的影响各是什么。

## 四种架构模式的交付特征

### 单体架构（Monolith）

所有功能在一个进程中运行。

| 交付维度 | 表现 |
|---------|------|
| 构建 | 一次构建一个产物，简单 |
| 部署 | 整体部署，停机更新或滚动更新 |
| 扩展 | 整体水平扩展（加更多实例），无法按功能独立扩展 |
| 更新 | 任何改动都需要重新部署整个服务 |
| 回滚 | 整体回滚，简单但粒度粗 |

**适用**：小型项目、原型阶段、团队人数少（<10 人）。
**交付风险**：单体变大后，一个模块的 Bug 修复需要重新部署整个服务。构建和部署时间随代码量线性增长。

### 微服务架构（Microservices）

功能拆分为多个独立服务，各自独立部署。

| 交付维度 | 表现 |
|---------|------|
| 构建 | 每个服务独立构建，可以并行 |
| 部署 | 每个服务独立部署，改了哪个部署哪个 |
| 扩展 | 按服务独立扩展（登录服务加实例不影响战斗服务） |
| 更新 | 只更新变更的服务，影响范围可控 |
| 回滚 | 按服务回滚，粒度细 |

**适用**：大型项目、多团队协作、需要独立扩展不同功能。
**交付风险**：服务间调用的版本兼容、分布式事务、服务发现和负载均衡都增加了运维复杂度。构建简单了但部署和监控变复杂了。

### Actor 模型

每个游戏实体（玩家、房间、NPC）是一个 Actor，通过消息通信。

| 交付维度 | 表现 |
|---------|------|
| 构建 | 通常单体构建（Actor 框架在一个进程中） |
| 部署 | 整体部署或按 Actor 类型分布 |
| 扩展 | Actor 可以在节点间迁移（理论上可以动态扩展） |
| 更新 | 可以逐类型更新 Actor 的行为（如果框架支持） |
| 回滚 | 取决于状态持久化策略 |

**适用**：需要高并发、有状态、消息驱动的游戏服务（MMO、社交游戏）。ET Framework 就是 Actor 模型。
**交付风险**：Actor 的状态持久化和迁移是交付中的难点——更新时正在运行的 Actor 的状态怎么处理。

### Server ECS

游戏状态存储在 Component 中，System 负责处理逻辑。数据和逻辑分离。

| 交付维度 | 表现 |
|---------|------|
| 构建 | 通常单体构建（ECS 框架在一个进程中） |
| 部署 | 整体部署 |
| 扩展 | 可以按 World 分片（每个 World 一个进程） |
| 更新 | System 可以热替换（如果框架支持） |
| 回滚 | Component 数据可以序列化回滚 |

**适用**：高性能游戏服务（RTS、大规模模拟、物理密集）。
**交付风险**：ECS 的 Schema 变更（新增/删除 Component）影响数据序列化格式——需要 Schema 迁移策略。

## 选型的交付视角决策框架

| 约束 | 推荐 | 理由 |
|------|------|------|
| 小团队、快速迭代 | 单体 | 构建部署最简单 |
| 多团队、功能独立 | 微服务 | 独立构建部署，互不阻塞 |
| 有状态、高并发 | Actor | 状态管理和扩展能力好 |
| 高性能计算密集 | Server ECS | 数据局部性好，CPU 利用率高 |
| 混合需求 | Actor + 微服务网关 | 核心游戏逻辑用 Actor，辅助服务用微服务 |

## 小结

- [ ] 架构选型是否考虑了交付维度（不只是功能维度）
- [ ] 构建和部署的复杂度是否在团队能力范围内
- [ ] 有状态服务的更新策略是否有设计
- [ ] 扩展方式是否满足预期的用户规模
