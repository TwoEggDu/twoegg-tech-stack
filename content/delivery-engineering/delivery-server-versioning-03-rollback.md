## 这篇解决什么问题

服务端部署出问题后回滚，通常几分钟就能完成——重新部署旧版本镜像即可。但如果新版本修改了数据库 Schema 或数据格式，回滚就不只是换个镜像的事了。本篇讲服务端回滚中最难的部分：数据兼容性。

## 代码回滚 vs 数据回滚

| 维度 | 代码回滚 | 数据回滚 |
|------|---------|---------|
| 速度 | 分钟级 | 小时级 |
| 风险 | 低（旧代码已验证） | 高（可能丢失新数据） |
| 可逆性 | 随时可以再回滚 | 恢复后丢失的数据无法找回 |
| 自动化程度 | 高（一键回滚） | 低（需要人工判断） |

**核心原则**：代码可以随时回滚，但数据不能随意回滚。设计时要让代码回滚不依赖数据回滚。

## 前向兼容的 Schema 迁移

### 不安全的变更

| 变更类型 | 回滚影响 |
|---------|---------|
| 删除列 | 旧代码读取已删除的列会报错 |
| 重命名列 | 旧代码用旧名查询失败 |
| 缩小列类型范围 | 旧代码写入超出范围的值 |
| 添加非空列且无默认值 | 旧代码插入数据时缺少该列 |

### 安全的变更

| 变更类型 | 为什么安全 |
|---------|-----------|
| 添加可空列 | 旧代码不知道新列，不影响读写 |
| 添加有默认值的列 | 旧代码插入时自动填默认值 |
| 扩大列类型范围 | 旧代码写入的值在新范围内 |
| 添加索引 | 不影响数据读写 |

## Expand-Migrate-Contract 模式

当需要做不兼容的 Schema 变更时（比如重命名列），拆成三步：

| 阶段 | 动作 | 回滚安全性 |
|------|------|-----------|
| Expand | 添加新列，代码同时写入新旧两列 | 安全——旧列数据仍在 |
| Migrate | 后台任务将旧列数据迁移到新列 | 安全——旧列数据仍在 |
| Contract | 删除旧列，代码只读写新列 | 不安全——旧列已删除 |

**关键**：Expand 和 Migrate 阶段可以安全回滚。Contract 执行后就不能回滚了——所以必须在确认稳定后才执行 Contract。

### 示例：重命名 `player_name` 为 `display_name`

1. Expand：添加 `display_name` 列，写入时两列都写
2. Migrate：`UPDATE ... SET display_name = player_name WHERE display_name IS NULL`
3. Contract（确认稳定后）：删除 `player_name`，只用 `display_name`

## 回滚安全检测

每次部署前验证是否可以安全回滚到上一版本：

| 检查项 | 方法 |
|--------|------|
| Schema 变更是否向后兼容 | 自动化工具扫描 migration 文件 |
| 新增数据格式是否可被旧版本忽略 | Code review + 集成测试 |
| 缓存 Key/Value 格式变更 | 旧版本能否解析新格式缓存 |

CI 中加入回滚测试：部署新版本 → 执行 Schema 迁移 → 回滚到旧版本（Schema 不回滚）→ 运行旧版本集成测试。这步验证回滚安全性。

## Point-in-Time Recovery

当回滚必须伴随数据回滚时（极端情况），使用数据库的时间点恢复。前提是数据库开启了持续备份（WAL 归档 / binlog），恢复时间通常 30 分钟到数小时，恢复时间点之后的所有变更都会丢失。

### 数据问题的分级处理

| 问题范围 | 处理方式 |
|---------|---------|
| 个别玩家 | 人工修复特定数据 |
| 部分玩家 | 编写修复脚本定向修复 |
| 全部玩家 | Point-in-Time Recovery + 补偿正常操作 |

## 回滚演练

| 演练项目 | 频率 | 验证目标 |
|---------|------|---------|
| 代码回滚 | 每次部署前在 staging 验证 | 旧版本能否在新 Schema 上运行 |
| 数据库恢复 | 每月一次 | 恢复流程可执行，时长可接受 |
| 全链路回滚 | 每季度一次 | 从发现问题到完成回滚的总时长 |

## 小结

- [ ] Schema 变更是否遵循"只做向后兼容变更"的规则
- [ ] 不兼容变更是否拆成 Expand-Migrate-Contract 三步
- [ ] CI 是否包含回滚安全性测试
- [ ] 数据库是否开启持续备份（支持 Point-in-Time Recovery）
- [ ] 是否定义了数据问题的分级修复策略
- [ ] 是否定期演练回滚流程
