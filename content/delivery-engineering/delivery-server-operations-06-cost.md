## 这篇解决什么问题

服务端上云后，成本不再是"买服务器"的一次性投入，而是每月持续的支出。不做成本管理的团队往往在项目后期才发现：服务器费用比预期高出数倍，但不知道钱花在了哪里。本篇讲云成本管理的核心策略。

## 资源规划：按实际用量选型

### Right-Sizing

最常见的浪费是实例选型过大——按峰值配置但大部分时间跑在 10% 负载。

**正确做法**：

1. 先用中等规格上线
2. 观察 1-2 周的实际 CPU/内存使用率
3. 根据实际数据调整实例规格
4. 定期（每月/每季度）复查

| 使用率 | 动作 |
|--------|------|
| 平均 CPU < 20% | 降规格 |
| 平均 CPU 20%-60% | 合适 |
| 平均 CPU > 70% | 升规格或扩容 |

### 游戏服务器的规格选择

游戏服务器通常是 CPU 密集型（逻辑计算、物理模拟）或内存密集型（大量玩家状态），较少是 IO 密集型。根据实际瓶颈选择对应类型的实例：

| 瓶颈 | 推荐实例类型 | 典型场景 |
|------|------------|---------|
| CPU | 计算优化型（C 系列） | 战斗服务器、物理模拟 |
| 内存 | 内存优化型（R 系列） | MMO 世界服务器、大量在线状态 |
| 网络 | 网络优化型 | 高并发网关、实时同步 |

## 三种计费模式对比

| 维度 | 按需实例（On-Demand） | 预留实例（Reserved） | 竞价实例（Spot） |
|------|---------------------|--------------------|--------------------|
| 价格 | 标准价 | 标准价的 30%-60% | 标准价的 10%-30% |
| 承诺 | 无 | 1 年或 3 年 | 无 |
| 可用性 | 保证 | 保证 | 可能被中断（提前 2 分钟通知） |
| 适合 | 测试环境、临时扩容 | 基准负载（7x24 运行的服务） | 无状态批处理、可中断的任务 |

### 游戏服务端怎么选

| 服务类型 | 推荐计费模式 | 原因 |
|---------|------------|------|
| 基础设施（数据库、缓存、网关） | 预留实例 | 7x24 运行，负载稳定 |
| 基准游戏服务器 | 预留实例 | 低谷期也需要的最低实例数 |
| 高峰弹性扩容部分 | 按需实例 | 只在高峰时段运行 |
| 构建/测试 CI | 竞价实例 | 中断可重试，不影响在线服务 |
| 数据分析/离线计算 | 竞价实例 | 任务可中断可重试 |

### Spot 实例的禁区

**游戏世界服务器不能用 Spot 实例。** Spot 实例可能在任意时刻被云厂商回收（提前 2 分钟通知）。对于持有玩家游戏状态的服务器，被回收 = 所有在线玩家断线 + 可能丢数据。

Spot 实例只适合可中断的工作负载：CI/CD 构建、日志分析、数据聚合、负载测试。

## 弹性伸缩匹配玩家模式

游戏的在线人数有鲜明的时间规律。弹性伸缩的目标是让服务器数量跟着玩家数量走。

### 典型的在线人数曲线

```
      在线人数
        │
   8000 │          ┌──┐
        │         ╱    ╲
   5000 │       ╱        ╲
        │     ╱            ╲
   2000 │───╱                ╲───
        │
        └────────────────────────
        6:00  12:00  20:00  24:00
```

### 伸缩策略

| 时段 | 策略 | 实例类型 |
|------|------|---------|
| 低谷（2:00-10:00） | 缩容到基准线 | 预留实例 |
| 平峰（10:00-18:00） | 基准线 + 少量弹性 | 预留 + 少量按需 |
| 高峰（18:00-24:00） | 基准线 + 大量弹性 | 预留 + 按需 |
| 活动/节假日 | 提前预扩容 | 预留 + 按需 |

**预扩容比响应式扩容更重要**——当玩家已经涌入时再扩容，启动和预热可能需要几分钟，这几分钟的体验就已经损失了。

## 成本监控

### 按服务归因

每个服务/组件的成本必须可追踪。通过云厂商的标签（Tag）功能实现：

| 标签 | 值 | 用途 |
|------|---|------|
| service | battle-server / match-server / gateway | 按服务统计成本 |
| environment | prod / staging / dev | 按环境统计成本 |
| team | server / client / data | 按团队统计成本 |

### 预算告警

| 告警类型 | 阈值 | 动作 |
|---------|------|------|
| 月度预算告警 | 达到预算的 80% | 通知负责人 |
| 日增长异常 | 单日费用超过月均的 150% | 立即调查 |
| 单服务异常 | 某服务费用突增 50% | 检查是否有配置错误或流量异常 |

日增长异常特别重要——常见事故是配置错误导致实例数暴涨，一天烧掉一个月的预算。

## 常见浪费

| 浪费类型 | 现象 | 解决方案 |
|---------|------|---------|
| 全天候高配 | 8 小时高峰的游戏用 24 小时高配 | 弹性伸缩 |
| 僵尸资源 | 已不使用的实例/负载均衡器/EIP | 定期清理 + 未使用资源告警 |
| 过大的数据库实例 | 数据库配了 64GB 内存但只用了 8GB | 按实际使用 right-sizing |
| 未清理的测试环境 | Staging 环境跑了和生产一样的规格 | 测试环境用最小规格 + 定时关停 |
| 未优化的数据传输 | 跨区域/跨 AZ 的大量数据传输 | 同区域部署、减少跨区域调用 |
| 日志存储膨胀 | DEBUG 日志保留 90 天 | 分级保留策略（见 03 监控体系） |

## 小结

- [ ] 实例规格是否基于实际使用数据选择（而非拍脑袋）
- [ ] 基准负载是否使用预留实例（而非全部按需）
- [ ] 有状态游戏服务器是否避免使用 Spot 实例
- [ ] 是否有弹性伸缩策略匹配玩家在线人数曲线
- [ ] 每个服务的成本是否可追踪（标签/归因）
- [ ] 是否有预算告警和日增长异常检测
- [ ] 是否定期清理僵尸资源和过大的实例
