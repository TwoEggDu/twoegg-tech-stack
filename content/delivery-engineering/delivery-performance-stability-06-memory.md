## 这篇解决什么问题

CPU 超预算的后果是卡顿，GPU 超预算的后果是掉帧。内存超预算的后果比这两者都严重——**OOM 闪退**。用户看到的是"游戏崩了"，不是"游戏卡了"。

这一篇讲内存治理的完整链路：预算怎么按类目分配、OOM 怎么主动防护、资源瘦身怎么系统化执行。

## 内存是崩溃的边界

### 各平台的 OOM 阈值

| 平台 | OOM 行为 | 说明 |
|------|---------|------|
| iOS | 系统 Jetsam 直接杀进程 | 不会给任何警告就杀（内存警告只是"建议"） |
| Android | 系统 LMK 杀后台 → 杀前台 | 低内存设备先杀后台进程，极端情况杀前台 |
| WebGL | 浏览器 Tab 崩溃 | 受限于浏览器内存管理 |

**iOS 特别危险**：iOS 的 Jetsam 机制会根据系统整体内存压力杀进程，你的 App 可能只用了 1.2 GB 就被杀——因为其他 App 也在占内存。

### 安全内存预算

安全内存预算 = 设备物理内存 * 安全比例 - 系统占用。

| 设备档位 | 物理内存 | 安全预算 | 峰值红线 |
|---------|---------|---------|---------|
| 高档（≥6 GB） | 6-8 GB | ≤1.5 GB | 1.8 GB |
| 中档（4 GB） | 4 GB | ≤1.0 GB | 1.2 GB |
| 低档（2-3 GB） | 2-3 GB | ≤600 MB | 750 MB |

**峰值红线**：短暂允许超过安全预算但不允许超过红线。超过红线 → 触发紧急释放。

## 内存预算按类目分配

### 分配比例

| 类目 | 占比 | 低档设备预算（600 MB） | 高档设备预算（1.5 GB） |
|------|------|---------------------|---------------------|
| 纹理 | 40-60% | 240-360 MB | 600-900 MB |
| Mesh | 10-20% | 60-120 MB | 150-300 MB |
| 音频 | 5-10% | 30-60 MB | 75-150 MB |
| 脚本堆 | 10-20% | 60-120 MB | 150-300 MB |
| Shader | 3-5% | 18-30 MB | 45-75 MB |
| 引擎内部 | 10-15% | 60-90 MB | 150-225 MB |

### 纹理——最大的内存消耗项

纹理通常占总内存的 40-60%，是内存治理的首要目标。

| 治理方向 | 说明 |
|---------|------|
| 分辨率上限 | 按设备档位限制最大纹理分辨率（低档 1024、中档 2048、高档 4096） |
| 压缩格式 | ASTC 6x6 比 RGBA32 节省 90% 内存（V14-05 已覆盖） |
| Mipmap | 3D 场景必须开启（减少远处纹理的采样量），UI 纹理不需要 |
| 纹理流式加载 | Unity 的 Texture Streaming 按需加载 Mipmap 层级 |
| Atlas 合并 | UI 纹理打 Atlas，减少纹理对象数量和内存碎片 |

### 脚本堆——增长不可控的隐患

托管堆（Mono/IL2CPP）的特点是**只增不减**——堆扩容后即使对象被 GC 回收，堆大小也不会缩小。

| 问题 | 说明 |
|------|------|
| 堆持续增长 | 每次 GC 不够时堆扩容，长时间运行后堆越来越大 |
| 峰值冲高 | 某些操作（加载大场景、反序列化大数据）瞬间分配大量内存 |
| 碎片化 | 堆中充满碎片，实际可用空间远小于堆大小 |

**治理方式**：减少分配（V14-04 的 GC 压力管理）+ 控制峰值（大操作分帧执行）+ 监控堆大小趋势。

## OOM 防护机制

被动等 OOM 发生再修是不够的——需要主动防护。

### 内存警告回调

| 平台 | 回调方式 | 说明 |
|------|---------|------|
| Unity | `Application.lowMemory` | 系统内存不足时触发 |
| iOS | `didReceiveMemoryWarning` | 在 Jetsam 杀进程前的最后机会 |
| Android | `onTrimMemory` | 分级别的内存告警 |

### 紧急释放策略

收到内存警告后执行分级释放：

```
Level 1（轻度）：
  - 清理未使用的 Asset 缓存（Resources.UnloadUnusedAssets）
  - 清理对象池中超出最小数量的实例
  - 降低纹理流式加载的内存预算

Level 2（中度）：
  - 强制 GC（GC.Collect）
  - 卸载非当前场景的所有 AssetBundle
  - 降低纹理分辨率到最低档

Level 3（重度）：
  - 卸载所有可卸载的资源
  - 切换到最低画质模式
  - 如果仍在危险区域 → 跳转到低内存安全场景
```

### 持续监控

| 监控项 | 采集频率 | 告警条件 |
|--------|---------|---------|
| 总内存使用 | 每秒 | 超过安全预算 |
| 纹理内存 | 每 5 秒 | 超过纹理预算 |
| 托管堆大小 | 每 5 秒 | 持续增长无回落（疑似泄漏） |
| 内存增长速率 | 每分钟 | 增长率异常（疑似泄漏） |

## 包体大小治理

包体大小与内存密切相关——包里的资源最终都要加载到内存中。V06 已完整覆盖包体分发策略，这里补充治理维度。

### 包体审计

| 类目 | 常见占比 | 瘦身方向 |
|------|---------|---------|
| 纹理 | 50-70% | 压缩格式、分辨率、按需下载 |
| 音频 | 10-20% | 采样率、压缩格式（Vorbis/ADPCM） |
| Shader | 5-15% | 变体裁剪（V14-05 已覆盖） |
| 代码（IL2CPP） | 5-10% | Stripping 级别、减少第三方库 |
| Mesh | 5-10% | 压缩、LOD 级别数量 |

### 包体预算的 CI 检查

| 检查项 | 方式 | 拦截条件 |
|--------|------|---------|
| 总包体大小 | Build Report | 超过渠道限制 |
| 单资源大小 | 资源扫描 | 单张纹理 > 阈值 |
| 未压缩资源 | 格式检查 | 存在 RGBA32 纹理 → 告警 |
| 重复资源 | 哈希比对 | 相同内容的资源出现在多个 Bundle 中 |

## 内存回归检测

内存问题的特点是**缓慢积累**——每个版本增加一点，几个版本后就超标了。CI 中的回归检测至关重要。

| 检测方式 | 说明 |
|---------|------|
| 固定场景内存快照 | 每次构建后跑固定路径，对比内存快照 |
| 内存峰值趋势 | 记录每次构建的内存峰值，画趋势线 |
| 泄漏检测 | 反复进出同一场景 N 次，检查内存是否回落到基线 |

## 小结与检查清单

- [ ] 是否有按设备档位的内存安全预算
- [ ] 内存预算是否按类目分配（纹理/Mesh/音频/脚本堆）
- [ ] 纹理是否有分辨率上限和压缩格式要求
- [ ] 托管堆大小是否有监控（防止只增不减）
- [ ] 是否注册了平台内存警告回调
- [ ] 紧急释放策略是否分级设计并经过验证
- [ ] 包体大小是否有渠道级别的预算
- [ ] CI 是否有内存回归检测和包体大小检查
- [ ] 是否有泄漏检测机制（反复进出场景测试）
