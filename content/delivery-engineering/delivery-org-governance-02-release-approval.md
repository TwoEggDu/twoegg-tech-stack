## 这篇解决什么问题

CI 全绿、测试通过、质量门达标——但这些都是自动化检查。发布审批是上线前的最后一道**人工**关卡，检查的是自动化无法覆盖的判断：版本是否真的准备好了？回滚方案是否可执行？监控是否配置到位？On-call 是否安排好了？这篇从审批清单、变更管理、双人审批、紧急通道、审计追踪五个维度搭建发布审批流程。

## 审批不是走形式

先破一个误区：审批不是"签字盖章"仪式。如果审批只是确认"CI 绿了"，那它没有任何附加价值——CI 已经检查过了。

审批的价值在于**人的判断**：

- CI 绿了，但有一个 known issue 还没修——这个版本能上吗？
- 回归测试通过了，但这个版本改了支付模块——额外的手动验证做了吗？
- 所有检查都过了，但明天是周五下午——真的要在周末前上线吗？

## 发布审批清单

审批清单分为四个类别，每个类别都必须显式确认：

### 一、构建与验证

| 检查项 | 确认人 | 检查方式 |
|-------|--------|---------|
| CI 构建成功，产物 Hash 一致 | Toolchain | CI 报告自动关联 |
| 功能测试全量通过 | QA | 测试报告链接 |
| 性能指标在预算范围内 | QA / TA | 性能测试报告 |
| 兼容性测试覆盖目标设备 | QA | 兼容性矩阵 |
| 已知缺陷已评估并记录 | QA / Dev | Bug 列表 + 风险评估 |

### 二、回滚准备

| 检查项 | 确认人 | 检查方式 |
|-------|--------|---------|
| 回滚方案已文档化 | Release Mgr | 回滚文档链接 |
| 回滚操作已在预发布环境验证 | Toolchain / Ops | 回滚演练记录 |
| 服务端兼容新旧版本客户端 | Dev (Server) | 接口兼容性测试 |
| CDN 缓存清理方案就绪 | Ops | CDN 配置确认 |

### 三、监控与值守

| 检查项 | 确认人 | 检查方式 |
|-------|--------|---------|
| 版本监控看板已配置 | Ops / Toolchain | 看板 URL |
| 告警规则已更新为当前版本 | Ops | 告警配置截图 |
| 灰度各阶段指标阈值已定义 | Release Mgr / QA | 灰度计划文档 |
| On-call 人员已安排 | Release Mgr | 值班表 |

### 四、外部依赖

| 检查项 | 确认人 | 检查方式 |
|-------|--------|---------|
| 平台审核已通过（iOS/Android/微信） | Release Mgr | 审核状态截图 |
| 第三方 SDK 版本已确认兼容 | Dev | SDK 兼容性记录 |
| 数据库变更已执行（如有） | Ops / Dev | 迁移脚本执行记录 |

## 变更管理——哪些变更需要审批

不是所有变更都需要走完整审批流程。按影响范围分级：

| 变更级别 | 定义 | 审批要求 | 示例 |
|---------|------|---------|------|
| L1 常规 | 功能迭代、Bug 修复，不涉及核心模块 | 标准审批流程 | 新活动、UI 调整、非核心 Bug 修复 |
| L2 重要 | 涉及核心模块、支付、数据迁移 | 标准审批 + 额外技术评审 | 支付流程修改、数据库结构变更、SDK 升级 |
| L3 关键 | 架构变更、引擎升级、跨端改动 | 技术委员会评审 + 双人审批 | 引擎版本升级、热更框架切换、协议版本变更 |
| L4 紧急 | 线上事故修复 | 紧急通道（见下文） | Crash 修复、安全漏洞修补 |

**分级标准**：影响用户范围 × 影响严重程度 × 可回滚难度。三个维度任一达到高危，整体级别上调。

## 双人审批规则

为什么需要双人审批：

1. **防止单点失误**——一个人容易遗漏检查项，两个人交叉确认降低风险
2. **分散责任压力**——一个人拍板的心理压力远大于两人共同确认
3. **知识冗余**——确保至少两个人理解这个版本改了什么

### 双人审批的实施方式

| 审批角色 | 关注维度 | 典型人选 |
|---------|---------|---------|
| 技术审批人 | 代码变更影响、技术风险评估 | 主程 / Tech Lead |
| 流程审批人 | 清单完整性、流程合规性 | Release Manager |

**规则**：两人都同意才能放行。任一人否决，必须说明原因并给出修复建议。否决不是终止——是暂停并修复。

## 紧急审批通道

线上出了致命问题，不能等完整审批流程走完。紧急通道是标准流程的压缩版，不是跳过审批。

### 紧急通道流程

```
线上致命事故确认
├── 触发条件：Crash 率 > 阈值 / 核心功能不可用 / 安全漏洞
│
├── Step 1：Release Mgr 宣布启用紧急通道（群通知 + 工单记录）
├── Step 2：Dev 提交修复 + QA 执行最小回归（仅核心路径）
├── Step 3：单人审批（Release Mgr 或其授权代理人）
├── Step 4：上线 + 监控观察
│
└── Step 5（24h 内必须）：补齐审计记录 + 触发复盘
```

### 紧急通道的约束

| 约束 | 说明 |
|------|------|
| 缩减审批但不跳过 | 单人审批代替双人，但审批动作必须完成 |
| 最小回归必须做 | 核心路径的冒烟测试不能省 |
| 24h 内补齐记录 | 紧急通道的所有决策必须有审计追踪 |
| 触发复盘 | 每次使用紧急通道都必须触发 Postmortem |
| 月度统计 | 紧急通道使用频率纳入交付健康度指标 |

**红线**：如果一个月用了超过 2 次紧急通道，说明常规流程有问题——不是紧急通道太严，而是常规发布的质量不够。

## 审计追踪

每次发布审批必须留下可追溯的记录：

| 记录字段 | 说明 |
|---------|------|
| 版本号 | 精确到构建号 |
| 审批时间 | 精确到分钟 |
| 审批人 | 技术审批人 + 流程审批人 |
| 审批结果 | 通过 / 否决 / 条件通过 |
| 审批清单完成情况 | 每项检查的确认状态 |
| 否决原因（如有） | 具体问题 + 修复建议 |
| 紧急通道标记（如有） | 触发原因 + 缩减了哪些步骤 |
| 关联工单 | 对应的发布工单 / 事故工单 |

**存储方式**：不要用 IM 消息作为审计记录——消息会被淹没。审批记录应该存在发布系统或工单系统中，可检索、可统计。

## 审批流程嵌入 CI/CD 管线

审批不应该是管线之外的独立流程，而应该嵌入管线成为一个 Stage：

```
Build → Test → Quality Gate → [Approval Gate] → Deploy → Canary
```

| 集成方式 | 说明 |
|---------|------|
| 审批 Stage | CI/CD 管线中插入人工审批节点，不点"Approve"流程不往下走 |
| 自动预填 | 审批表单自动从 CI 报告、测试报告中提取数据，减少人工填写 |
| 超时提醒 | 审批超过 2h 未处理，自动通知审批人和 Release Mgr |
| 审批记录 | 审批结果自动写入发布系统，不需要手动复制 |

## 检查清单

- [ ] 发布审批清单已覆盖构建、回滚、监控、外部依赖四个类别
- [ ] 变更分级标准已定义（L1-L4）
- [ ] 双人审批规则已明确（技术审批人 + 流程审批人）
- [ ] 紧急通道的触发条件和流程已文档化
- [ ] 审计追踪的字段和存储方式已确认
- [ ] 审批流程已嵌入 CI/CD 管线（而不是管线外的独立流程）
- [ ] 紧急通道使用频率纳入月度交付健康度指标
