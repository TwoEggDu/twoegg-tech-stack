## 这篇解决什么问题

上一篇建立了四层验证模型。但验证不是在一个时间点全做完——不同阶段适合拦截不同类型的问题。验证门定义了每个阶段的检查内容和通过标准。本篇解决三个问题：门怎么分、每道门的标准怎么定、自动化门和人工门怎么配合。

## V01 飞轮模型中的三道门

| 门 | 阶段 | 核心职责 | 执行方式 |
|----|------|---------|---------|
| 质量门 | 代码合入前 | 防止低质量代码进入主干 | 自动化为主 |
| 验证门 | 构建完成后 | 确认构建产物满足交付标准 | 自动化 + 人工 |
| 发布门 | 发布前 | 最终确认、审批放行 | 人工为主 |

三道门形成逐级过滤：质量门拦代码问题，验证门拦产品问题，发布门拦流程问题。

## 质量门：代码合入前

质量门在 Merge Request / Pull Request 阶段触发，目标是保证合入主干的代码不降低代码库质量。

| 检查项 | 类型 | 通过标准 |
|--------|------|---------|
| 编译通过 | 必须通过 | 0 编译错误 |
| 静态分析 | 必须通过 | 0 新增严重/高危告警 |
| 单元测试 | 必须通过 | 通过率 100%，新代码覆盖率 ≥ 60% |
| 代码审查 | 必须通过 | 至少一位 Reviewer 批准 |
| 包体增量 | 信息性 | 记录增量，不阻断 |

质量门全部自动化执行，只有代码审查需要人工参与。

### 质量门的真实配置示例

以下是某个 Unity 手游项目实际使用的质量门配置，供参考：

**单元测试阈值**：新代码行覆盖率 ≥ 60% 这个数字是怎么来的？经验上来看，游戏项目的单元测试覆盖率普遍比 Web 项目低——因为大量逻辑和表现紧密耦合，很难做纯单元测试。60% 是一个务实的起点。如果团队有良好的架构分层（逻辑层和表现层分离），可以逐步提高到 70-80%。但一上来就要求 90%，大概率的结果是大家写大量无意义的测试来凑覆盖率。

**Warning 预算的真实演进**：
- 项目初期：500+ 条 Warning（历史债务）
- 第一个季度：设基线 500，新代码不允许增加
- 第二个季度：降到 400（每个 Sprint 各团队认领 20-30 条）
- 第三个季度：降到 250
- 半年后：降到 100 以内

关键点是**不要试图一次性清零**。我见过一个团队下了死命令"一周内消灭所有 Warning"，结果开发为了赶工大量使用 `#pragma warning disable`——Warning 是消失了，但问题被藏起来了。正确做法是逐步降低，同时在 CI 里检查 `#pragma warning disable` 的使用不能增长。

**静态分析规则的选择**：不要一次性开启所有规则。先从确定性最高、误报最少的规则开始（空引用检查、未释放资源、线程安全），让团队先接受"静态分析能拦住真问题"这个事实。误报率一旦超过 30%，团队就会开始无脑忽略所有告警。

## 验证门：构建完成后

验证门是四层验证模型的主战场。构建完成后触发，覆盖功能、性能、稳定性、兼容性四个维度。

| 检查项 | 判定级别 | 通过标准 |
|--------|---------|---------|
| 冒烟测试 | 必须通过 | 核心链路 100% 通过 |
| 功能回归 | 必须通过 | 通过率 ≥ 95%，P0/P1 缺陷为 0 |
| 性能基线比对 | 必须通过 | 关键指标退化 ≤ 5% |
| 稳定性长跑 | 应当通过 | Crash 率 < 0.1%，无内存持续增长 |
| 兼容性覆盖 | 应当通过 | 核心设备通过率 ≥ 90% |
| 包体大小 | 信息性 | 记录变化趋势 |

### 三级判定标准

| 级别 | 含义 | 不通过时 |
|------|------|---------|
| 必须通过（Must Pass） | 不通过则流程阻断 | 构建标记为失败，不能进入下一阶段 |
| 应当通过（Should Pass） | 不通过需要审批 | 可以通过人工审批豁免，但记录在案 |
| 信息性（Info Only） | 不阻断，仅记录 | 数据记录用于趋势分析和决策参考 |

## 发布门：发布前

发布门是最后的人工确认环节，确保所有前置条件都已满足。

| 检查项 | 执行者 | 确认内容 |
|--------|--------|---------|
| 验证门结果审核 | QA 负责人 | 所有必须通过项是否通过，豁免项是否合理 |
| 发布内容确认 | 产品负责人 | 发布内容是否符合预期 |
| 回滚方案确认 | 技术负责人 | 回滚方案是否就绪、回滚条件是否明确 |
| 运维就绪确认 | 运维/SRE | 监控告警、值班人员是否到位 |
| 最终审批 | 发布审批人 | 签字放行 |

## 自动化门 vs 人工门

| 维度 | 自动化门 | 人工门 |
|------|---------|--------|
| 执行速度 | 分钟级 | 小时到天级 |
| 一致性 | 完全一致 | 依赖个人判断 |
| 适用场景 | 可量化的判定标准 | 需要主观判断的场景 |
| 成本 | 前期高（建设），后期低 | 持续人力成本 |
| 典型例子 | 编译、测试、性能比对 | 代码审查、发布审批 |

原则：能自动化的一定自动化，只有真正需要人工判断的才留给人工门。

## 门的所有权

| 角色 | 职责 |
|------|------|
| 定义者 | 制定每道门的检查项和通过标准（通常是技术负责人 + QA 负责人） |
| 执行者 | 执行检查并记录结果（自动化系统 + 测试工程师） |
| 审批者 | 审核结果、决定是否放行（门的 Owner） |

## 紧急绕过流程

生产环境出了紧急 bug，需要快速修复上线——验证门能不能跳过？

可以，但必须有规范的绕过流程：

1. **发起申请**：说明紧急原因和影响范围
2. **最小验证**：至少跑冒烟测试，确保修复不引入新问题
3. **双人审批**：至少两位有权限的负责人批准
4. **补充验证**：上线后 24 小时内补齐完整验证
5. **事后复盘**：记录绕过原因，分析是否可以避免

绕过不是无记录地跳过，而是有审批、有记录、有补偿的快速通道。

### 一次绕过引发的事故

某个项目在一次紧急修复中绕过了验证门——理由是"只改了一行配置，不可能出问题"。那一行配置确实修复了原始问题，但同时无意中修改了另一个配置项的默认值，导致特定服务器的匹配逻辑出错。因为跳过了冒烟测试，这个副作用在上线后 4 小时才被发现。

事后复盘发现，如果跑了冒烟测试（约 15 分钟），这个问题一定会被发现——因为匹配功能是冒烟测试的核心链路之一。为了省 15 分钟的冒烟时间，付出了 4 小时的线上故障。

**经验结论**：紧急绕过可以跳过很多东西，但**冒烟测试是底线中的底线**——无论多紧急，至少跑一次核心链路冒烟。如果连冒烟都要跳过，那说明冒烟测试本身太慢了，需要优化到 10 分钟以内。

### 绕过频率的警示意义

我更倾向于把绕过频率当作一个**流程健康度的反向指标**。如果一个季度内紧急绕过超过 3 次，问题不在于"这些紧急情况是否合理"，而在于**为什么会有这么多需要紧急修复的情况**。

某个项目的绕过频率一度达到每月 2 次（年化 24 次），分析后发现根因是热更新通道不稳定——每次热更失败就只能走紧急发版通道。修复热更通道后，紧急发版频率降到了每季度 1 次以内。

## 门的度量指标

| 指标 | 含义 | 健康范围 |
|------|------|---------|
| 门通过率 | 每道门的首次通过率 | > 80%（过低说明前置环节质量差） |
| 门拦截率 | 每道门拦住了多少问题 | 质量门 > 验证门 > 发布门（越早拦住越好） |
| 误报率 | 自动化门的误判比例 | < 5%（过高会导致团队忽视门） |
| 门耗时 | 每道门的平均执行时间 | 持续优化，不阻塞交付节奏 |
| 绕过频率 | 紧急绕过的发生频率 | < 10%（过高说明流程设计有问题） |

## 小结

- [ ] 交付流程中是否定义了质量门、验证门、发布门
- [ ] 每道门是否有明确的检查项和判定标准
- [ ] 判定标准是否分为必须通过、应当通过、信息性三级
- [ ] 能自动化的门是否已经自动化
- [ ] 是否有规范的紧急绕过流程
- [ ] 是否在跟踪门通过率、误报率等度量指标
