上一篇定义了交付和发布的区别。这一篇把"交付是完整链路"这句话拆开，画出完整的地图。

## 交付链路的完整形态

一个游戏版本从"可以开始收口"到"稳定运行在用户设备上"，至少经过以下八个环节：

```
生产 → 集成 → 构建 → 验证 → 发布 → 监控 → 闭环 → 改进
```

这八个环节不是某个引擎或技术栈的产物，而是任何需要把软件送到用户手上的项目都会经历的过程。区别只在于每个团队在每个环节投入了多少工程能力。

下面逐段拆开。

## 生产段：四条生产线

交付链路的第一段是生产。游戏项目的生产不是只有"写代码"一条线，而是至少四条并行的生产线：

**内容线**。美术资源、关卡、剧情、音频——这些内容由美术和策划产出，经过导入、格式化、规范验证后进入引擎。

**配置线**。数值表、技能配置、掉落表、商城定价——这些数据通常在 Excel 或自研工具中编辑，通过导表工具转化为运行时可读的格式。

**代码线**。客户端逻辑、服务端逻辑、工具链脚本——这是程序团队的产出，通过版本控制系统管理。

**基础设施线**。构建脚本、CI 配置、部署脚本、环境配置——这些"看不见的基础设施"由工具链团队维护。

四条线的产出最终汇聚到同一个构建管线。如果任何一条线的产出不合规，下游的构建和验证就会出现问题。

这就是为什么交付问题往往不出在发布环节——根因常常在这四条生产线里。

## 集成段：从分散到统一

四条生产线的产出需要通过集成段汇聚到一起。集成段解决的核心问题是：**多人、多团队、多条线的产出，怎样可控地合并到同一个代码库里。**

集成段的关键机制：

**版本管理**。分支策略决定了代码怎样从开发流入发布。Main / Develop / Release / Hotfix 各有各的角色，选错策略会直接制造冲突和混乱。

**内容冻结**。在某个时间点停止接受新功能，只允许 Bug 修复进入发布分支。冻结窗口设计不合理，要么新内容来不及验证，要么修复来不及进入。

**合入门禁**。代码合入主干前需要通过哪些检查——编译是否通过、自动化测试是否通过、资源规范是否达标。这些门禁越早建立，下游的问题就越少。

集成段的产出是一个"可构建的代码库状态"——所有生产线的最新合规产出已经合入，可以触发构建。

## 构建段：从源码到产物

构建段把代码库状态转换成各平台的可执行产物。对游戏项目来说，构建通常包括：

**脚本编译**。C# 代码通过 Mono 或 IL2CPP 编译成目标平台的可执行代码。编译配置（Debug / Release / Development）影响产物的行为和性能。

**资源打包**。美术资源、配置数据、Shader 等被打包成各平台的资源格式。打包策略决定了首包大小、加载性能和热更新粒度。

**平台构建**。针对 iOS、Android、微信小游戏各自生成平台专属的构建产物——Xcode 工程、Gradle 工程、WebGL 目录。

**签名与归档**。构建产物经过签名后归档，标记构建号和版本号，供后续验证和发布使用。

构建段有一个关键特性：**它应该是完全自动化和可重复的。**如果同一个代码库状态在不同机器上构建出不同的产物，那构建系统本身就是风险源。

## 验证段：交付的质量门

构建产物出来后，不能直接交给用户。它需要经过验证段的检查，确认是否满足交付标准。

验证段通常覆盖四个维度：

**功能验证**。核心流程是否可走通，已修复的问题是否回归，新功能是否按预期工作。

**性能验证**。帧率、内存、加载时间、包体大小是否在预算基线内，是否存在性能退化。

**稳定性验证**。长时间运行是否稳定，Crash 率是否在阈值内，弱网和断网是否能正常恢复。

**兼容性验证**。在目标设备矩阵上是否能正常运行，不同 OS 版本和 GPU 型号是否存在兼容问题。

四个维度各自有不同的验证方法和通过标准。关于每个维度的详细设计，在本专栏的 V13（验证与测试）和 V14（性能与稳定性工程）中展开。

验证段的产出是一个判断：**这个构建产物是否满足发布条件。**

## 发布段：从产物到用户

通过验证的构建产物，进入发布段。发布段的职责是把产物送达用户，同时管控上线风险。

**平台提审**。iOS 需要 App Store 审核，Android 需要 Google Play 审核，微信小游戏需要微信审核。三个平台各有各的审核规则和时间周期。

**灰度发布**。不是一次性推给所有用户，而是先推给 1%，观察指标，再扩大到 10%，最终全量。灰度的意义不是"慢慢上线"，而是"可控地暴露风险"。

**回滚预案**。在灰度观察期或全量发布后，如果发现关键问题，是否能快速回到上一个稳定版本。回滚预案需要提前验证，不是"到时候再想办法"。

发布段的产出是：**新版本到达用户设备，并且有回退通道。**

## 监控段和闭环段：交付不是上线就结束

版本上线后，交付链路还有两个环节。

**监控段**。上线后持续观察版本健康指标——Crash 率、ANR 率、帧率分布、内存分布、加载时间、留存率。监控的目的不是"看看数据"，而是建立告警机制：什么样的指标变化应该触发响应。

**闭环段**。当监控或用户反馈发现问题时，启动缺陷闭环：分类 → 定位根因 → 修复 → 回归验证 → 防止复发。闭环段的关键不是"修掉这个 Bug"，而是"这类问题以后怎么在上游拦住"。

这两个环节的存在，是"交付"和"发布"的根本区别。发布在"版本上线"那一刻就结束了；交付在"版本被确认健康"那一刻才算完成。

## 改进段：让飞轮转起来

交付链路的最后一个环节是改进。每一次版本交付完成后，应该有一个回顾：

- 这次交付中暴露了什么问题？
- 问题的根因在链路的哪个环节？
- 下次怎样在更上游的地方拦住它？
- 有没有重复性操作应该被自动化？

改进段的产出不是文档，而是流程变更和自动化沉淀。它让下一轮交付的链路更短、更稳。

关于改进段的方法论，在下一篇文章（交付飞轮模型）中展开。

## 客户端和服务端：两条平行链路

上面描述的链路主要以客户端视角展开。服务端有自己的交付链路，在结构上平行但在细节上有重要差异：

| 环节 | 客户端 | 服务端 |
|------|--------|--------|
| 构建 | 多端构建（iOS/Android/WebGL） | 单一构建（Docker 镜像） |
| 发布 | 平台审核 → 用户主动更新 | 滚动更新 → 服务端直接替换 |
| 回滚 | 用户设备上的旧版本无法强制回退 | 可以在分钟级回到上一个版本 |
| 热更新 | 受平台限制，需要绕过审核 | 可以直接替换运行中的代码或配置 |
| 版本共存 | 不同用户可能运行不同版本 | 服务端通常只有一个版本在线 |

两条链路在**协议兼容性**和**版本协调**上必须同步。客户端发了新版本，服务端必须同时支持新老协议；服务端更新了逻辑，必须确保存量客户端不会崩溃。

本专栏在 V07–V09 覆盖客户端构建和发布链路，在 V10–V12 覆盖服务端交付链路。

```
         客户端链路                                    服务端链路
┌─────────────────────────────────┐     ┌─────────────────────────────┐
│                                 │     │                             │

│  客户端代码 ┤                    │     │  基础设施   ─┘               │
│  基础设施  ─┘                    │     │                             │
│                                 │     │                             │
│  → iOS 发布  ─┐                  │     │  → 滚动部署 ─┐              │
│  → Android  ──┤→ 灰度 → 监控    │     │  → 金丝雀   ─┤→ 监控       │

                    闭环 → 改进 → 下一轮生产
```

这就是交付链路的完整形态。
