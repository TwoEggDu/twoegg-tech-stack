## 这篇解决什么问题

上一篇讲了美术资源的四阶段管线。这一篇聚焦管线中"验证"环节的核心输入：资源规范。

规范回答的问题是：一个资源要满足什么条件才能进入构建管线？

## 为什么这个问题重要

没有规范的生产线，等于没有准入条件的管线。任何资源都能进入构建，意味着任何资源都可能制造问题。

常见的代价：

- 包体超预算 3 个月后才发现，回头压缩已经来不及（美术不可能重做几百个已经量产的资源）
- 低端设备大面积闪退，排查发现是某批角色模型的骨骼数超出了设备限制
- 构建时间从 20 分钟涨到 2 小时，因为大量未压缩的源贴图被提交到了项目里

这些问题的共同特征：**如果在资源提交时就有规范检查，不会走到后面才发现。**

## 本质是什么

资源规范的本质是**用预算约束反推生产标准**。

不是先做完资源再看超不超预算，而是先定义预算，再从预算推导出每种资源类型的具体限制。

```
交付预算（包体/内存/性能）
      ↓
分系统预算（角色/场景/UI/特效/音频）
      ↓
单资源标准（面数/贴图尺寸/骨骼数/格式/压缩）
```

这条推导链的起点是 V14（性能与稳定性工程）中定义的预算体系。本篇不展开预算怎么定，只讲预算确定后，资源规范怎么落地。

## 五类核心规范

### 1. 命名规范

命名规范解决的不是"好看"，而是**可检索、可分类、可自动化处理**。

好的命名规范应该让工具能从文件名中提取出关键信息：

```
{类型}_{名称}_{变体}_{尺寸标记}
例：tex_hero_warrior_d_1024    (角色贴图，漫反射，1024x1024)
    mesh_weapon_sword_lod0     (武器模型，LOD0)
    sfx_ui_button_click        (UI 按钮点击音效)
```

命名规范的实施要点：
- 全小写 + 下划线分隔（避免跨平台文件系统问题）
- 禁止中文和特殊字符（编码问题是跨平台的重灾区）
- 类型前缀必须标准化（tex / mesh / mat / sfx / anim）
- 在提交时用钩子自动检查，不靠人肉审核

### 2. 格式规范

每种资源类型应该有明确的格式白名单：

| 资源类型 | 允许的源格式 | 禁止的格式 | 理由 |
|---------|------------|-----------|------|
| 模型 | FBX | .max / .blend / .mb | DCC 源文件不应进入项目 |
| 贴图 | TGA / PNG / EXR | PSD / TIFF | PSD 带图层信息会导致体积膨胀 |
| 音频 | WAV（源）/ OGG（运行时） | MP3 | MP3 有专利和精度问题 |
| 动画 | FBX | BVH | 统一格式降低导入复杂度 |

格式规范的核心原则：**源格式和运行时格式分开管理**。版本库里存源格式，构建管线负责转换为运行时格式。

### 3. 尺寸和面数规范

这是与性能预算直接挂钩的规范：

| 资源类别 | 贴图最大尺寸 | 模型最大面数 | 骨骼上限 |
|---------|------------|-----------|---------|
| 主角（近景） | 2048x2048 | 30K 三角面 | 80 骨骼 |
| NPC / 怪物 | 1024x1024 | 15K 三角面 | 50 骨骼 |
| 场景物件（大） | 1024x1024 | 10K 三角面 | — |
| 场景物件（小） | 512x512 | 3K 三角面 | — |
| UI 图标 | 256x256 | — | — |
| 特效贴图 | 512x512 | — | — |

这些数字不是通用标准——它们必须从你的项目的具体预算推导出来。不同项目、不同平台、不同设备分档的数字会完全不同。

关键是这些数字必须**明确写下来、工具可检查、超标能自动拦住**。

### 4. 压缩规范

压缩规范是多端交付中最容易出问题的地方，因为不同平台的最优压缩格式不同：

| 贴图类型 | iOS | Android | 微信小游戏 |
|---------|-----|---------|-----------|
| 漫反射 / Albedo | ASTC 6x6 | ASTC 6x6 或 ETC2 | ASTC 6x6（WebGL 2.0+） |
| 法线贴图 | ASTC 4x4（需要高精度） | ASTC 4x4 或 ETC2 | ASTC 4x4 |
| UI 贴图 | ASTC 4x4（需要清晰度） | ASTC 4x4 | ASTC 4x4 |
| 光照贴图 | ASTC 6x6 | ASTC 6x6 | ASTC 8x8 |

压缩规范不应该由美术逐个手动设置。它应该通过**导入预设**或**自动化脚本**根据资源类型和目标平台自动应用。

### 5. 目录结构规范

目录结构规范解决的是**可归类、可查找、可自动化处理**：

```
Assets/
├── Art/
│   ├── Characters/
│   │   ├── Hero/
│   │   └── NPC/
│   ├── Environments/
│   ├── UI/
│   ├── VFX/
│   └── Audio/
├── Config/
├── Scripts/
└── Plugins/
```

目录结构的实施要点：
- 美术资源和代码资源分开（不同的变更频率和审核流程）
- 资源按功能模块组织，不按文件类型组织（角色的贴图和模型放在一起，不是所有贴图放一个文件夹）
- 目录边界与 AssetBundle 的打包分组对齐（目录结构影响打包策略）

## 规范怎么落地

规范写在文档里没有用。规范只有变成自动化检查才有意义。

落地的三个层次：

**第一层：提交时轻量检查**。命名、格式、文件大小——在 pre-commit hook 或 CI 的提交后检查中执行。速度要求秒级。

**第二层：导入时中量检查**。导入设置一致性、压缩格式——在 Unity 的 AssetPostprocessor 或自动化脚本中执行。

**第三层：CI 中重量检查**。面数统计、贴图内存估算、包体贡献度——在 CI 构建后的分析步骤中执行。

三层检查的结果应该合成一份资源质量报告，区分：
- **阻止**（必须修复才能合入或构建）
- **警告**（记录但不阻止）
- **信息**（仅供参考）

## 常见事故与排障

**事故：贴图尺寸规范只检查了宽高，没检查实际内存**。一个 1024x1024 的贴图如果是 RGBA 32bit 未压缩，占用 4MB 内存；如果是 ASTC 6x6 压缩，占用约 170KB。只检查分辨率不检查压缩格式，内存预算会严重失控。

**事故：规范只对新资源生效，存量资源不管**。项目早期没有规范，积累了大量不合规资源。后来加了规范但只对新提交的资源检查。结果构建产物里 60% 的资源都是不合规的存量。必须有存量资源的清理计划。

**事故：规范太严导致美术绕过流程**。所有检查都设为阻止级别，美术频繁被拦住后开始找其他路径提交资源（直接复制到项目目录、绕过版本库）。规范的严格程度必须和问题的严重程度匹配。

## 小结与检查清单

- [ ] 命名规范是否有自动化检查（而非靠人审核）
- [ ] 格式白名单是否明确，DCC 源文件是否被阻止提交
- [ ] 面数和贴图尺寸限制是否从性能预算推导，而非凭经验拍脑袋
- [ ] 压缩规范是否通过导入预设自动应用
- [ ] 目录结构是否与 AssetBundle 打包策略对齐
- [ ] 检查结果是否区分阻止/警告/信息三级
- [ ] 存量资源是否有清理计划
