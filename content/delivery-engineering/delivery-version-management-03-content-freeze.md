## 这篇解决什么问题

分支策略定义了代码怎么流动。但还有一个更上层的问题：**什么时候停止往 Release 分支里塞东西？**

内容冻结（Content Freeze）是交付链路中把"开发态"切换到"发布态"的关键节点。冻结设计不合理，要么验证时间不够，要么紧急修复进不去。

## 为什么这个问题重要

没有冻结机制的项目，每次发版前都会上演同一幕：

- 发版前一天策划还在改配置表："就改个数值，很小的"
- 发版前两小时程序还在合代码："这个 Bug 不修上线会被投诉"
- 每次"最后一刻的变更"都没有经过完整验证，上线后出问题的概率显著高于计划内变更

冻结不是限制团队的生产力，而是为验证环节争取足够的时间。

## 本质是什么

内容冻结的本质是**把变更的接受窗口和验证的执行窗口分开**。

```
                        冻结点
开发窗口 ─────────────────┤
                         ├──── 验证窗口 ────┤
                                           ├── 发布
```

冻结点之前：接受所有类型的变更（新功能、Bug 修复、资源更新、配置调整）
冻结点之后：只接受 Bug 修复，且必须经过审批

### 三层冻结

实际项目中，不同类型的变更应该有不同的冻结时间：

| 冻结层 | 冻结时间 | 允许的变更 | 理由 |
|--------|---------|-----------|------|
| **功能冻结** | 发布前 5-7 天 | 不接受新功能 | 新功能需要完整的功能测试 |
| **代码冻结** | 发布前 2-3 天 | 只接受 P0/P1 Bug 修复 | Bug 修复需要回归测试 |
| **配置冻结** | 发布前 1 天 | 只接受经过审批的配置修改 | 配置变更需要验证 |

三层冻结的时间差给了不同类型的变更不同的验证窗口：

```
功能冻结 ─────┤
              ├── 代码冻结 ──┤
                            ├── 配置冻结 ─┤
                                         ├── 发布
```

### 冻结期间的例外处理

冻结不是绝对的。但例外必须有明确的流程：

**P0 问题**（阻断性 Bug：崩溃、数据丢失、核心流程不可用）：
- 可以在任何冻结阶段合入
- 必须有至少一个人批准
- 修复后必须重新跑冒烟测试

**P1 问题**（严重但不阻断）：
- 在代码冻结前可以合入
- 代码冻结后需要两人批准
- 配置冻结后原则上不接受

**P2 及以下**：
- 冻结后不接受，延到下一个版本

**例外流程的关键**：不是"能不能破例"，而是"破例的门槛有多高"。门槛太低等于没有冻结，门槛太高会导致紧急问题无法修复。

## 发布列车模型

发布列车（Release Train）是把冻结和发布节奏固定下来的运作模式。

### 时间驱动 vs 内容驱动

| 模式 | 做法 | 优点 | 缺点 |
|------|------|------|------|
| **时间驱动** | 每两周一趟列车，到时间就发 | 节奏可预期，验证时间固定 | 功能可能赶不上这趟车 |
| **内容驱动** | 功能做完了才发 | 功能完整性好 | 节奏不可预期，验证时间被压缩 |

**推荐：时间驱动**。赶不上这趟车的功能等下一趟。这比"为了等一个功能延迟发版"更安全——因为延迟意味着验证窗口被压缩。

### 发布列车的典型节奏

以双周发布为例：

```
第 1 周：开发窗口
├── 周一-周四：Feature 合入 Develop
├── 周五：从 Develop 拉 Release 分支（功能冻结）

第 2 周：验证窗口
├── 周一-周二：功能验证 + 性能验证（代码冻结）
├── 周三：兼容性验证 + 最终回归（配置冻结）
├── 周四：提审（iOS / Android / 微信）
├── 周五：审核通过后灰度发布
```

这个节奏的核心是：**开发窗口和验证窗口不重叠**。团队要么在开发下一个版本，要么在验证当前版本，不会出现"边开发边验证"的混乱状态。

### 多端发布列车的协调

三端的审核周期不同：
- iOS：1-3 天（不可预测）
- Android：几小时到 1 天
- 微信小游戏：1-3 天

协调策略：
- 三端从同一个 Release 分支、同一个 commit 构建
- iOS 最先提审（因为审核时间最长）
- Android 和微信在 iOS 审核期间并行提审
- 全部审核通过后统一上线（或按优先级先发某一端）

## 常见错误做法

**只冻结代码不冻结配置**。代码冻结了但策划还在改表，最终发版的配置没有经过完整验证。配置变更对游戏行为的影响可能比代码更大。

**冻结期间不记录例外**。破例合入的变更没有记录，版本复盘时不知道"为什么这个版本的线上问题比上一个多"——因为没有追踪到冻结期间的例外变更比往常多。

**发布列车没有固定节奏**。"差不多了就发"——每次发版时间不确定，团队无法提前规划验证资源，验证窗口被反复压缩。

## 小结与检查清单

- [ ] 是否有明确的功能冻结 / 代码冻结 / 配置冻结时间点
- [ ] 冻结期间的例外流程是否有审批和记录
- [ ] 发布节奏是否固定（时间驱动）
- [ ] 开发窗口和验证窗口是否不重叠
- [ ] 三端提审是否有协调策略（先提审核最慢的平台）
- [ ] 例外变更是否纳入版本复盘
