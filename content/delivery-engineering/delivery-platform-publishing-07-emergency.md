## 这篇解决什么问题

线上发现严重问题（崩溃率飙升、数据丢失、安全漏洞），需要在最短时间内修复或回撤。三个平台各自有不同的应急能力和时间约束。这一篇把应急选项和决策框架讲清楚。

## 应急手段的优先级

遇到线上严重问题时，按以下优先级选择应急手段：

```
1. 配置热推（分钟级）→ 能通过关闭 Feature Flag 解决吗？
2. 资源/脚本热更新（分钟级）→ 能通过热更修复吗？
3. 版本回撤（小时级）→ 能回到上一个稳定版本吗？
4. 紧急发版（天级）→ 必须发新版本通过审核才能修复？
```

**优先选择最快的手段**。配置热推最快（改一个 Flag 就能关闭有问题的功能），热更新次之，版本回撤再次之，紧急发版最慢。

### 配置热推（分钟级）

**前置条件**：有问题的功能通过 Feature Flag 控制。

**操作**：在远程配置服务中关闭该功能的 Flag → 客户端下次启动或定时刷新时生效。

**优点**：最快，不需要构建任何东西。
**限制**：只能关闭功能，不能修复代码 Bug。

### 热更新（分钟-小时级）

**前置条件**：项目有脚本热更新能力（V08），问题在热更可修复范围内。

**操作**：修复代码 → 构建热更包 → 部署到 CDN → 客户端启动时下载。

**优点**：可以修复代码逻辑 Bug，不需要平台审核。
**限制**：只能修复管理代码，不能修复原生层问题。需要构建和部署时间。

### 版本回撤（小时级）

三端的回撤能力差异很大：

| 平台 | 回撤能力 | 时间 | 对已更新用户的影响 |
|------|---------|------|------------------|
| **iOS** | 停止分发新版本，恢复旧版本 | 即时 | 已更新用户无法回退 |
| **Android (Google Play)** | 暂停 Staged Rollout / 回退版本 | 即时 | 已更新用户无法回退 |
| **Android (国内渠道)** | 各渠道分别操作 | 小时级（逐渠道） | 已更新用户无法回退 |
| **微信小游戏** | 后台直接回退到历史版本 | 即时 | **已有用户下次打开自动回退** |

**微信小游戏是唯一支持"强制所有用户回退"的平台**——这是一个重要的安全优势。

**iOS 和 Android 的回撤限制**：已经更新到新版本的用户无法回退。只能阻止新用户继续获取问题版本。已更新用户只能等热更新修复或新版本发布。

### 紧急发版（天级）

当问题无法通过热更新修复（原生层问题、引擎 Bug、安全漏洞），需要发新版本：

| 平台 | 正常审核 | 加急审核 |
|------|---------|---------|
| iOS | 1-3 天 | 24h 内（申请 Expedited Review） |
| Android | 数小时-1 天 | 通常不需要加急（审核本身快） |
| 微信 | 1-3 天 | 可联系对接人加急 |

**紧急发版的流程**：

```
1. 从上一个稳定版本的 tag 拉 hotfix 分支
2. 只修复关键问题（不夹带其他变更）
3. 精简验证：冒烟 + 问题修复验证 + Release 配置检查
4. 三端并行提审（iOS 申请加急）
5. 审核通过后立即发布（不走常规灰度，或只灰度到 10% 后快速全量）
```

## 一次真实的紧急响应时间线

以下是某次线上事故的完整响应时间线，从发现到修复。当时版本刚全量发布 6 小时，Crash 率从基线的 0.08% 飙到了 4.7%。

| 时间（相对 T0） | 事件 | 执行人 | 决策 |
|----------------|------|--------|------|
| T+0min | Crashlytics 触发 Critical 告警：Crash-free rate 跌破 99% | 自动化系统 | — |
| T+2min | On-call 确认告警，打开 Crash 面板查看 Top Crash | On-call 值班 | 初步判断是新增 Crash |
| T+5min | 确认 Top 1 Crash 是空指针异常，调用栈指向新增的活动模块 | On-call + 开发 A | 判断：不是基础设施问题，是代码 Bug |
| T+8min | 检查该功能是否有 Feature Flag → 有 | 开发 A | **决策：先关 Flag 止血** |
| T+10min | 远程配置平台关闭活动模块 Flag | 开发 A | 等待客户端刷新配置 |
| T+25min | 观察 Crash 率开始下降，从 4.7% 降到 1.2% | On-call | 已更新用户还在触发老缓存 |
| T+40min | Crash 率降到 0.3%，接近但仍高于基线 | On-call | 还有用户的配置缓存没刷新 |
| T+60min | 确认 Crash 率回到 0.1% 附近 | On-call | **止血完成** |
| T+90min | 开发 A 定位到根因：新活动页面在特定网络条件下数据回调为 null | 开发 A | 准备热更修复 |
| T+3h | 热更包构建完成，部署到 CDN | 开发 A + 运维 | 走快速灰度通道 |
| T+4h | 灰度 5% 验证热更修复有效 | QA | Crash 率正常 |
| T+5h | 全量推送热更，重新打开活动模块 Flag | 开发 A | **事故彻底修复** |
| T+24h | 完成 Postmortem | 团队 | 补充自动化测试 + 活动模块必须有空数据兜底 |

几个关键的经验教训：

1. **Feature Flag 是救命的**。如果这个活动功能没有 Flag 控制，唯一的止血手段就是回撤版本或紧急热更——前者需要 iOS 重新提审，后者至少要 2-3 小时。Flag 让止血时间从小时级压缩到了 10 分钟
2. **配置缓存刷新有延迟**。关了 Flag 不是立刻生效的——客户端有配置缓存，通常 5-15 分钟刷新一次。紧急情况下可以考虑推送一个"强制刷新配置"的指令，但这个能力需要提前建好
3. **Crash 率 4.7% 听起来不高，但算一下绝对数字**。如果日活 50 万，4.7% 意味着每小时有约 1000 个用户崩溃。用户不会看百分比，他们只知道"游戏闪退了"

## 应急决策框架

```
线上严重问题发现
    ↓
评估：Feature Flag 能解决吗？
  → 能 → 配置热推（分钟级解决）
  → 不能 ↓

评估：热更新能修复吗？
  → 能 → 构建热更包 → 部署 → 灰度验证（小时级解决）
  → 不能 ↓

评估：回撤到旧版本可接受吗？
  → 微信小游戏：直接回撤（分钟级解决）
  → iOS/Android：回撤只能阻止新用户（已更新用户需要额外修复）
  → 回撤 + 热更新组合：先回撤阻止扩散，同时准备热更修复已更新用户
  → 回撤不可接受（旧版本也有其他问题）↓

紧急发版：
  → 拉 hotfix 分支 → 精简验证 → 加急审核 → 快速发布
```

### 决策框架中的隐藏陷阱

实际操作中，这个决策框架看起来很清晰，但真正出事时最容易犯的错误是**评估时间过长**。某个项目曾经花了 45 分钟讨论"到底用热更还是紧急发版"，而这 45 分钟里 Crash 在持续影响用户。

我的建议是设定一个**硬性时间盒**：

- **止血决策**：发现问题后 15 分钟内必须做出第一个止血动作（哪怕是先关 Flag）
- **根因定位**：止血后 2 小时内必须定位到根因
- **彻底修复**：根因定位后 4 小时内必须完成修复方案并开始执行

如果在时间盒内无法完成，自动升级到下一级决策者。不要陷入"再看看""再等等"的循环。

## 应急预案的准备

不要等到出了问题才想应急方案。以下准备应该在日常就完成：

| 准备项 | 日常维护 |
|--------|---------|
| Feature Flag 系统 | 关键功能都通过 Flag 控制 |
| 热更新通道 | 每个版本发布后验证热更通道可用 |
| 回撤脚本 | 预设一键回撤脚本（每个平台） |
| Hotfix 分支模板 | 从最新 release tag 拉分支的脚本 |
| 加急审核额度 | 知道 Apple 加急审核的申请入口和剩余额度 |
| 值班制度 | 发版后 48 小时内有值班人员监控 |

**演练**：每季度做一次应急演练——模拟"线上 Crash 率飙升"场景，走一遍从发现到修复的完整流程。确认每个环节的负责人、工具和时间。

### 演练中发现的真实问题

做过三次应急演练后，我发现演练暴露的问题比想象中多得多。以下是实际演练中发现的典型问题：

| 演练次数 | 发现的问题 | 影响 | 修复措施 |
|---------|-----------|------|---------|
| 第 1 次 | 热更新通道的签名证书已过期 3 个月 | 热更包无法部署 | 加证书有效期监控，过期前 30 天告警 |
| 第 1 次 | 只有 1 个人知道 Apple 加急审核的入口 | 该人不在就没人能操作 | 写操作手册，至少 3 人会操作 |
| 第 2 次 | 回撤脚本调用了已废弃的 API | 脚本执行失败 | 回撤脚本纳入 CI 定期验证 |
| 第 2 次 | On-call 电话通知系统的手机号有 2 个已换号 | 电话打不通 | 每月验证联系方式有效性 |
| 第 3 次 | 从 release tag 拉 hotfix 分支的脚本，在最新 tag 上跑会报错 | 需要手动处理 | 修复脚本，增加 tag 格式兼容 |

**经验结论**：如果你的应急演练从来没发现过问题，那很可能是演练做得不够真实。真正有价值的演练应该是让值班人员在不提前通知的情况下，用生产环境的工具走完全流程。

### Apple 加急审核的注意事项

实际操作中，Apple 的 Expedited Review 有几个容易忽略的点：

1. **不是无限次使用的**。Apple 没有公开的额度限制，但频繁申请加急会降低通过率。经验上一年用 2-3 次比较安全，用太多次后续申请可能被拒绝
2. **申请理由要写清楚**。"Bug fix" 两个字不够——要写清楚影响范围（多少用户受影响）、问题严重程度（崩溃/数据丢失/安全漏洞）
3. **加急不保证一定快**。申请加急后平均 24 小时内出结果，但偶尔也会 48 小时。不要把加急审核当作确定性的时间承诺来做排期
4. **周末和美国节假日不要指望加急**。Apple 审核团队在这些时段人力减少，加急效果打折

## 常见错误做法

**紧急发版时夹带其他变更**。"反正要发版，顺便把这个功能也带上"——结果新功能引入了新问题，紧急发版变成了新事故的起点。Hotfix 分支必须只包含修复。

**回撤后不通知用户**。iOS/Android 回撤只能阻止新下载，已更新用户不知道发生了什么。应该通过公告或推送告知用户"已知问题正在修复"。

**不做应急演练**。"不会出那么严重的问题"——直到真的出了，发现值班人员不知道回撤脚本在哪、加急审核怎么申请、热更新通道的密钥过期了。

## 小结与检查清单

- [ ] 应急手段的优先级是否团队知晓（热推 → 热更 → 回撤 → 发版）
- [ ] Feature Flag 是否覆盖了关键功能
- [ ] 热更新通道是否在每次版本发布后验证
- [ ] 三端回撤脚本是否预设且验证过
- [ ] Apple 加急审核的申请入口和额度是否知晓
- [ ] 发版后 48 小时是否有值班人员
- [ ] 是否每季度做一次应急演练
