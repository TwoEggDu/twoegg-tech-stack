## 这篇解决什么问题

很多团队对 CI/CD 的理解停留在"自动打包"。实际上 CI 和 CD 解决的问题不同，游戏项目的 CD 边界也和 Web 项目有本质差异。这一篇把 CI 和 CD 的概念拆清楚，建立五阶段管线模型。

## CI vs CD：两个不同的问题

| 维度 | CI（持续集成） | CD（持续交付/部署） |
|------|--------------|-------------------|
| 解决的问题 | 代码集成后有没有问题 | 集成通过的代码怎么到用户手上 |
| 触发时机 | 每次提交/合并 | 发布分支就绪时 |
| 核心目标 | 快速反馈——5 分钟内知道提交有没有破坏构建 | 可靠路径——从代码到生产环境的每一步都可重复 |
| 失败代价 | 低（开发环境回滚即可） | 高（用户可见的故障） |
| 执行频率 | 一天几十次 | 一周/一月几次 |

### Continuous Delivery vs Continuous Deployment

CD 有两层含义，游戏项目通常只做到 Delivery 层：

- **Continuous Delivery**（持续交付）：每次通过 CI 的构建都处于"随时可发布"状态，但发布动作仍需人工审批
- **Continuous Deployment**（持续部署）：每次通过 CI 的构建自动发布到生产环境，无需人工干预

游戏项目几乎不做 Continuous Deployment——因为 iOS 有审核周期、Android 有渠道包、微信有审核、主机平台有 TRC/Lotcheck。最后一步"发布到用户"必然需要人工判断和平台审批。

## 五阶段管线模型

```
触发（Trigger）→ 构建（Build）→ 检查（Check）→ 部署（Deploy）→ 通知（Notify）
```

### 阶段一：触发

**什么驱动管线启动？**

| 触发类型 | 场景 | 执行什么管线 |
|---------|------|-------------|
| 代码提交 | 开发者 push 到特性分支 | 快速通道（编译+lint） |
| PR/MR 创建 | 代码请求合并到主干 | 标准通道（编译+测试+资源检查） |
| 定时触发 | 每晚凌晨 | 完整通道（全平台构建+完整测试） |
| 手动触发 | 发布管理员点击"发布" | 发布通道（构建+签名+上传商店） |
| Tag 创建 | 打版本 Tag | 发布归档通道 |

### 阶段二：构建

**把源码变成可运行的产物。**

- 编译代码（C# → IL → IL2CPP/Mono）
- 处理资源（AssetBundle 打包、纹理压缩、图集生成）
- 生成平台产物（IPA、APK/AAB、微信小游戏包）
- 归档产物（附带 Build Number、Commit Hash、环境信息）

### 阶段三：检查

**验证产物是否满足质量标准。**

- 编译结果：零 Error，Warning 在预算内
- 资源合规：命名规范、格式正确、尺寸合规（V02-03 标准）
- 性能基线：包体大小、Shader 变体数、帧时间（V14 预算）
- 自动化测试：单元测试、集成测试、冒烟测试（V13 体系）

### 阶段四：部署

**把通过检查的产物送达目标环境。**

- 开发环境：自动部署到测试服/测试设备
- 预发布环境：部署到 staging 供 QA 验证
- 生产环境：上传到 App Store / Google Play / CDN

### 阶段五：通知

**把管线结果告知相关人。**

- 成功：通知触发者"构建已就绪"
- 失败：通知责任人具体哪个阶段失败、失败原因
- 部署完成：通知 QA"新版本已部署到 staging"

## 游戏项目的 CI/CD 特殊性

| 特殊点 | Web 项目 | 游戏项目 |
|--------|---------|---------|
| 构建时间 | 分钟级 | 30-60 分钟（IL2CPP + 资源打包） |
| 构建产物大小 | MB 级 | GB 级（包含资源） |
| 平台数量 | 通常 1 个 | 3-5 个（iOS/Android/微信/PC/主机） |
| 发布审核 | 无 | 平台审核 1-7 天 |
| 热更新 | 重新部署即可 | 需要独立的资源/代码热更新管线 |
| 许可证 | 开源工具链 | Unity/Unreal 需要 CI 专用许可证 |

这些特殊性决定了游戏项目的 CI/CD 不能照搬 Web 项目的方案——需要分快速通道和完整通道、需要处理大型资源缓存、需要管理引擎许可证。

## CI 能自动化什么、不能自动化什么

### 可以且应该自动化的

- 编译检查（每次提交）
- 资源格式和命名检查（每次提交）
- 单元测试和集成测试（每次提交）
- 全平台构建（每日或每次 PR）
- 包体大小和 Shader 变体数检查（每次构建）
- 部署到测试环境和 CDN（每次构建通过）

### 不能完全自动化的

- 平台审核提交（可自动上传，审核需人工触发）
- 游戏性测试（需要人工判断"好不好玩"）
- 灰度比例决策（需要运营数据+人工判断）
- 版本发布审批（需要多角色签字）

## 管线作为交付飞轮的执行层

V01 讲了交付飞轮模型：生产→集成→构建→验证→发布→监控→闭环→改进。CI/CD 管线覆盖其中"集成→构建→验证→发布"四个环节——不是全部，但是最核心、最值得自动化的部分。

```
交付飞轮：  生产 → 集成 → 构建 → 验证 → 发布 → 监控 → 闭环 → 改进
                   ↑                          ↑
CI/CD 管线：       触发  →  构建  →  检查  →  部署  →  通知
```

- [ ] 是否理解游戏项目通常做到 Continuous Delivery 而非 Deployment

- [ ] 是否识别了游戏项目 CI/CD 的特殊性（构建时间、产物大小、平台审核）
