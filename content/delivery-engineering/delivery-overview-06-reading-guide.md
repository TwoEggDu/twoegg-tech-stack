## 五条阅读线

这套专栏设计了五条阅读线，每条线穿过不同的卷，服务不同的读者：

### L1：交付工程原理线

**适合谁**：使用 Unreal / Godot / 自研引擎的技术负责人，或想学方法论但不关心 Unity 具体实现的人。

**读什么**：每篇文章的原理层段落。跳过"Unity 实践"和"平台专项"部分。

**核心卷**：V01（总论）→ V04（版本管理）→ V13（验证）→ V15（缺陷闭环）→ V16（CI/CD）→ V18（组织治理）

**预计阅读量**：约 40 篇，每篇只读原理层约 60% 的内容。

### L2：Unity 客户端交付线

**适合谁**：Unity 客户端开发者，这是本专栏的主力路径。

**读什么**：原理层 + Unity 实践层 + 三端平台层。跳过服务端卷（V10-V12）。

**核心卷**：V01 → V03（工程基建）→ V05（资源管线）→ V06（包体管理）→ V07（多端构建）→ V08（热更新）→ V09（平台发布）→ V13（验证）→ V14（性能）→ V16（CI/CD）→ V17（灰度上线）

**预计阅读量**：约 70 篇，完整阅读。

### L3：服务端交付线

**适合谁**：C# 服务端开发者，或需要理解前后端交付协调的全栈工程师。

**读什么**：V10-V12 完整读 + 其他卷的服务端相关章节。

**核心卷**：V01 → V04（版本管理）→ V10（服务端架构）→ V11（服务端部署）→ V12（服务端版本）→ V13（验证）→ V16（CI/CD）→ V17（灰度上线）→ V18（组织治理）

**预计阅读量**：约 45 篇。

### L4：质量与治理线

**适合谁**：QA 负责人、版本负责人、技术管理者。

**读什么**：聚焦质量体系和组织流程，跳过具体的技术实现细节。

**核心卷**：V01 → V13（验证与测试）→ V14（性能与稳定性）→ V15（缺陷闭环）→ V16（CI/CD）→ V17（灰度上线）→ V18（组织治理）→ V19（案例与模板）

**选读**：V04（版本管理的流程部分）、V09（平台审核部分）

**预计阅读量**：约 40 篇。

### L5：平台发布专项线

**适合谁**：负责 iOS / Android / 微信小游戏平台对接的工程师。

**读什么**：聚焦构建、发布和平台专项内容。

**核心卷**：V06（包体管理与分发）→ V07（多端构建）→ V09（平台发布）→ V17（灰度上线的平台差异部分）

**选读**：V04（版本号和分支策略的平台相关部分）

**预计阅读量**：约 25 篇。

## 按角色推荐

如果你不确定自己属于哪条阅读线，按角色选择：

| 角色 | 推荐阅读线 | 优先读的卷 |
|------|-----------|-----------|
| 客户端程序员（3-5 年） | L2 Unity 客户端 | V01 → V05 → V07 → V08 |
| 高级客户端 / 引擎程序员 | L2 Unity 客户端 | V01 → V03 → V05 → V14 → V16 |
| 技术负责人 / 主程 | L1 原理 + L4 质量治理 | V01 → V04 → V16 → V18 |
| 版本 / 发布负责人 | L4 质量治理 + L5 平台 | V01 → V09 → V17 → V18 |
| QA / 测试负责人 | L4 质量治理 | V01 → V13 → V15 → V19 |
| 工具链工程师 | L2 Unity 客户端 | V01 → V03 → V07 → V16 |
| 服务端开发者 | L3 服务端 | V01 → V10 → V11 → V12 |
| 技术美术 | L2 部分 + L5 部分 | V01 → V02 → V06 → V14 |

## 按成熟度推荐

如果你已经在上一篇做了成熟度自评，按当前级别选择优先读的卷：

| 当前级别 | 目标 | 优先卷 | 理由 |
|---------|------|--------|------|
| L1 手动发布 | → L2 | V03 工程基建 + V07 多端构建 | 先让构建可重复、不依赖特定的人 |
| L2 可重复构建 | → L3 | V13 验证与测试 + V14 性能稳定性 | 加入质量门禁 |
| L3 质量门禁 | → L4 | V16 CI/CD + V17 灰度上线 | 连成完整管线 |
| L4 交付管线 | → L5 | V15 缺陷闭环 + V18 组织治理 | 建立持续改进循环 |

## 与现有专栏的关系

本站的其他栏目（渲染管线、引擎工具链、性能工程、系统设计、代码质量）并没有被取代。交付工程专栏与它们的关系是：

```
交付工程专栏                 其他栏目
──────────                ──────
交付链路的工程设计             技术深度的知识体系
"怎样把产品稳定送到用户手上"    "怎样把某个技术方向做深"
```

交付工程专栏在很多地方会链接到其他栏目的文章。例如：

- V05（资源管线）会链接到引擎工具链栏的 AssetBundle 和 Addressables 系列
- V08（脚本热更新）会链接到引擎工具链栏的 HybridCLR 系列
- V14（性能与稳定性）会链接到性能工程栏的预算管理和设备分档系列
- V13（验证与测试）会链接到代码质量栏的质量保障体系系列

这些链接的方向是：交付专栏提供"这个环节在交付链路中的位置和工程设计"，其他栏目提供"这个技术方向的深度知识"。

如果你是第一次来这个站，交付工程专栏是最适合的起点——它会告诉你整条链路的全景，然后指引你去对应的栏目深入。
