## 这篇解决什么问题

大多数团队的性能工作模式是：上线后用户反馈卡顿 → 查 Profiler → 找到瓶颈 → 优化 → 验证 → 上线。这是性能优化，不是性能工程。

性能优化是被动的——问题暴露了才去修。性能工程是主动的——问题还没暴露就被拦住。这一篇把两者的区别说清楚，建立四阶段循环模型。

## 性能优化 vs 性能工程

| 维度 | 性能优化 | 性能工程 |
|------|---------|---------|
| 触发时机 | 问题出现后 | 问题出现前 |
| 工作模式 | 被动救火 | 主动防火 |
| 目标定义 | "不卡就行" | 量化预算 |
| 度量方式 | 手动 Profiler | 自动化采集 |
| 拦截机制 | 无（靠人肉发现） | CI 门禁自动拦截 |
| 覆盖范围 | 单次、局部 | 持续、全局 |

性能优化是性能工程的子集——工程循环中的"治理"阶段包含优化，但工程循环远不止于此。

## 四阶段循环

性能工程的核心是一个持续运转的四阶段循环：

```
Budget（预算）→ Measure（度量）→ Govern（治理）→ Verify（验证）
     ↑                                                    ↓
     ← ← ← ← ← 反馈修正预算 ← ← ← ← ← ← ← ←
```

### 阶段一：Budget——定预算

**核心问题**：每个系统在每个场景下允许消耗多少资源？

- 按系统拆分：角色、场景、UI、特效、音频各自的帧时间和内存配额
- 按场景拆分：大厅、战斗、过场、加载各自的总预算不同
- 按设备拆分：高/中/低档设备的预算不同

**负责人**：技术负责人定总预算，各系统 Owner 分配子预算。

**工具**：预算表（文档或配置文件），设备分档配置。

**常见问题**：大多数团队跳过这一步，直接进入"优化"。没有预算就没有标准，没有标准就无法判定"够不够好"。

### 阶段二：Measure——采数据

**核心问题**：当前各系统的实际消耗是多少？

- 离线度量：Profiler 抓帧、内存快照、Build Report
- CI 度量：每次构建后自动化性能跑测，采集帧时间、内存峰值、包体大小
- 线上度量：真实设备的帧率分布、内存分布、Crash 率

**负责人**：工具链工程师建采集基础设施，各系统 Owner 关注自己的指标。

**工具**：Unity Profiler, Memory Profiler, Build Report, 自定义 CI 性能跑测脚本, 线上 APM。

### 阶段三：Govern——守红线

**核心问题**：实际消耗超出预算时怎么办？

- CI 门禁：帧时间超预算 → 构建标记为不通过
- 自动告警：线上指标劣化 → 通知到责任人
- 技术手段：对象池、LOD、Shader 简化、纹理压缩、代码优化

**负责人**：CI 门禁由 DevOps 维护，治理手段由各系统 Owner 执行。

**常见问题**：治理手段只在"出了问题才执行"——这就退回了"性能优化"模式。工程化的治理是持续的：每次提交都检查、每次构建都验证。

### 阶段四：Verify——判放行

**核心问题**：这个版本的性能是否满足发布标准？

- 发布前跑完整性能测试矩阵（V13 已覆盖性能测试方法）
- 对比历史基线——不只看绝对值，还看趋势
- 设备覆盖——至少在每个设备档位的代表设备上验证

**负责人**：QA 执行测试，技术负责人判定是否放行。

**工具**：性能基线数据库，趋势对比报告。

## 为什么大多数团队停在"优化"阶段

| 阶段 | 跳过的原因 | 后果 |
|------|-----------|------|
| Budget | "目标不好定"、"需求还在变" | 没有标准，无法判定好坏 |
| Measure | "手动 Profile 就够了" | 退化在积累期被忽略 |
| Govern | "CI 跑性能测试太慢" | 退化进入发布版本 |
| Verify | "测试部门会测的" | 测试覆盖不足，线上出问题 |

完整的四阶段循环需要基础设施投入。但这个投入的回报是：**性能问题在暴露给用户之前就被拦住。**

## 四阶段循环在交付链路中的位置

```
V14-02 预算体系 ← Budget
V14-03 设备分档 ← Budget + Govern
V14-04 CPU 治理 ← Measure + Govern
V14-05 GPU 治理 ← Measure + Govern
V14-06 内存治理 ← Measure + Govern
V14-07 Crash 治理 ← Measure + Govern + Verify
V16 CI/CD ← 把 Budget/Measure/Govern/Verify 嵌入构建管线
```

V14 后续文章展开每个阶段的具体实践。V16 CI/CD 将讲怎么把这四个阶段自动化到构建管线中。

## 小结与检查清单

- [ ] 是否区分了性能优化（被动）和性能工程（主动）
- [ ] 是否有明确的性能预算（不是"不卡就行"）
- [ ] 是否有持续的度量基础设施（不是手动 Profiler）
- [ ] 是否有 CI 门禁拦截性能退化
- [ ] 是否有发布前的性能验证流程
- [ ] 四阶段循环是否形成了闭环（验证结果反馈回预算调整）
