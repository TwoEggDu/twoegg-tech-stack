## 这篇解决什么问题

功能测试通过了，性能达标了，但上线后玩家反馈"玩了一会就闪退"。稳定性问题的特点是不立即暴露——需要时间、需要特定条件、需要反复操作才会触发。本篇讲稳定性测试的四个维度和具体方法。

## 稳定性问题的分类

| 类型 | 表现 | 触发条件 | 检测难度 |
|------|------|---------|---------|
| Crash | 应用闪退 | 空指针、越界、未捕获异常 | 中——有堆栈可追 |
| ANR / Watchdog Kill | 应用无响应被系统杀死 | 主线程阻塞超时 | 高——现场信息少 |
| OOM | 内存溢出被系统杀死 | 内存持续增长或峰值过高 | 高——需要长时间运行 |
| Freeze | 画面卡死但未崩溃 | 死锁、无限循环、资源等待 | 高——不会产生崩溃日志 |

四类问题的共同特点：功能测试很难覆盖，需要专门的稳定性测试手段。

## 长跑测试

长跑测试（Long-running Test / Soak Test）是稳定性测试的基础——让应用持续运行数小时，观察是否出现异常。

### 长跑测试的设计

| 参数 | 大版本 | 功能迭代 | 热更新 |
|------|--------|---------|--------|
| 运行时长 | 24 小时 | 8 小时 | 4 小时 |
| 操作模式 | 自动化脚本循环核心玩法 | 自动化脚本循环变更功能 | 自动化脚本循环热更内容 |
| 监控频率 | 每 5 分钟采集一次 | 每 5 分钟采集一次 | 每 5 分钟采集一次 |

### 长跑期间监控的指标

| 指标 | 正常 | 异常信号 |
|------|------|---------|
| 内存占用 | 波动但无趋势性增长 | 持续上升（内存泄漏） |
| 帧率 | 稳定在预算范围内 | 随时间持续下降 |
| CPU 温度 | 正常波动 | 持续升高（热降频） |
| 线程数 | 稳定 | 持续增加（线程泄漏） |
| 文件句柄数 | 稳定 | 持续增加（文件句柄泄漏） |
| GC 频率 | 稳定 | 越来越频繁 |

内存趋势是最关键的指标——如果 8 小时内存持续上升且不回落，几乎可以确定存在内存泄漏。

## Monkey 测试

Monkey 测试通过随机输入发现崩溃——不需要测试用例，不需要预期结果，只要不崩溃就算通过。

### Monkey 测试的配置

| 参数 | 推荐值 | 说明 |
|------|--------|------|
| 事件类型 | 触摸 60%、手势 20%、导航 10%、系统 10% | 模拟真实用户的操作分布 |
| 执行时长 | 2-4 小时 | 短时间内大量随机操作 |
| 事件间隔 | 100-300ms | 太快设备跟不上，太慢覆盖不够 |
| 种子固定 | 记录随机种子 | 发现崩溃后可以用相同种子复现 |

### 平台工具

| 平台 | 工具 | 说明 |
|------|------|------|
| Android | `adb shell monkey` | 系统自带，基础但够用 |
| Android | App Crawler (Firebase) | 基于 UI 结构的智能遍历 |
| iOS | Xcode UI Testing + 随机化 | 需要自己封装随机逻辑 |
| 跨平台 | 自定义输入框架 | 在游戏引擎层面注入随机输入 |

## 弱网测试

游戏在弱网环境下的表现直接影响玩家体验——高延迟、丢包、断线重连是稳定性的重要维度。

### 弱网场景矩阵

| 场景 | 参数 | 验证重点 |
|------|------|---------|
| 高延迟 | RTT 500ms-2000ms | UI 是否假死、操作是否堆积 |
| 丢包 | 丢包率 10%-30% | 协议重传是否正常、数据是否丢失 |
| 带宽限制 | 下行 100Kbps-500Kbps | 资源加载是否超时、降级策略是否生效 |
| 断线重连 | 断开 5s/15s/60s 后恢复 | 重连是否成功、状态是否恢复 |
| 网络切换 | WiFi ↔ 4G/5G | 切换过程是否断线、是否自动恢复 |
| 完全断网 | 无网络 | 离线提示是否友好、本地数据是否安全 |

### 弱网模拟工具

| 工具 | 平台 | 特点 |
|------|------|------|
| Network Link Conditioner | macOS/iOS | 系统级，配置简单 |
| Charles Proxy | 跨平台 | 可精确控制延迟和丢包 |
| tc (traffic control) | Linux | 服务端测试环境常用 |
| Clumsy | Windows | Windows 开发环境调试用 |

## 后台前台切换测试

移动端特有的稳定性场景：

| 场景 | 验证重点 |
|------|---------|
| 切到后台再切回 | 游戏状态是否正确恢复 |
| 后台停留 5 分钟以上 | 长连接是否断开、重连是否正常 |
| 接电话/短信打断 | 音频是否正确恢复 |
| 后台被系统回收后重启 | 是否从正确的状态恢复 |
| 分屏/画中画模式 | 渲染和输入是否正常 |

## 低存储空间处理

存储空间不足时应用的行为：

- [ ] 存储空间不足时是否有友好提示
- [ ] 下载资源时空间不足是否正确处理
- [ ] 写入存档时空间不足是否保护已有数据
- [ ] 清理缓存后功能是否正常恢复

## 稳定性指标与阈值

| 指标 | 定义 | 目标阈值 |
|------|------|---------|
| Crash Rate | 崩溃次数 / 总会话数 | < 0.1% |
| ANR Rate | ANR 次数 / 总会话数 | < 0.05% |
| OOM Rate | OOM 次数 / 总会话数 | < 0.02% |
| Crash-Free Users | 无崩溃体验的用户占比 | > 99.5% |
| MTBF | 平均无故障运行时间 | > 8 小时 |

这些指标在测试阶段采集用于验证，上线后通过 SDK 上报用于监控。

## 小结

- [ ] 是否区分了 Crash、ANR、OOM、Freeze 四类稳定性问题
- [ ] 是否有长跑测试（大版本 24h，迭代 8h）
- [ ] 长跑测试是否监控内存趋势、帧率趋势
- [ ] 是否定期执行 Monkey 测试
- [ ] 是否有弱网测试覆盖（高延迟、丢包、断线重连）
- [ ] 是否测试了后台前台切换和低存储空间场景
- [ ] 稳定性指标是否有明确的目标阈值
