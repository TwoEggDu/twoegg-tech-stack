## 这篇解决什么问题

上一篇讲了脚本热更新的本质和风险。这一篇回答选型问题：四种主流架构各自的原理、优劣和适用场景。

## 四种架构的核心差异

| 架构 | 执行方式 | 热更代码语言 | 性能 | 和首包代码的关系 |
|------|---------|------------|------|----------------|
| **Lua 方案** | Lua VM 解释执行 | Lua | 低-中 | 两套语言，需要桥接层 |
| **纯 C# 解释** | IL 解释器 | C# | 低 | 同一语言，但全量解释执行 |
| **HybridCLR** | AOT + 解释器混合 | C# | 中-高 | 同一语言，首包 AOT + 热更解释 |
| **DHE** | AOT + 差分替换 | C# | 高 | 同一语言，函数级差分替换 |

### Lua 方案

**原理**：在游戏中嵌入 Lua VM（如 xLua、toLua），热更代码用 Lua 编写。Lua 代码通过桥接层调用 C# 接口。

**优点**：
- 技术成熟，行业使用超过 10 年
- Lua 语言轻量，VM 体积小
- iOS 合规无争议（Lua 解释器和 JavaScriptCore 类似）

**缺点**：
- 两套语言——团队需要同时维护 C# 和 Lua 代码
- 桥接层的性能开销——每次 Lua 调用 C# 或反向调用都有序列化成本
- 调试体验差——Lua 的调试工具不如 C# 成熟
- 类型安全弱——Lua 是动态类型，编译期无法检查类型错误

**适用场景**：
- 团队已有 Lua 方案的积累和经验
- 热更代码量不大（主要是 UI 逻辑和活动玩法）
- 不追求热更代码的极致性能

### 纯 C# 解释方案

**原理**：在 IL2CPP 运行时中嵌入 IL 解释器，热更代码编译为 IL 后由解释器执行。

**优点**：
- 同一语言——开发和热更代码都是 C#
- 无桥接层开销（同一个类型系统）

**缺点**：
- 解释执行性能显著低于 AOT 编译（通常慢 5-50 倍，取决于代码类型）
- 热更代码的所有逻辑都通过解释器执行，性能瓶颈明显
- 对计算密集型逻辑不友好

**适用场景**：
- 热更代码以 UI 和流程控制为主（不涉及计算密集逻辑）
- 团队希望统一语言但对性能要求不高

### HybridCLR

**原理**：在 IL2CPP 运行时中补充了完整的 IL 解释器和元数据系统。首包代码通过 IL2CPP 的 AOT 编译执行（原生性能），热更代码通过解释器执行。两者在同一个运行时中互操作。

**优点**：
- 同一语言、同一类型系统、无桥接层
- 首包代码保持原生 AOT 性能
- 热更代码可以直接继承、调用、实现首包中的任何类型
- 支持完整的 C# 特性（泛型、LINQ、async/await）
- 开源，社区活跃

**缺点**：
- 热更代码仍然是解释执行（性能低于 AOT）
- AOT 泛型补充需要额外的构建步骤（GenerateAll）
- 元数据桥接有工程复杂度
- 相对较新，生态不如 Lua 成熟

**适用场景**：
- Unity C# 项目，希望统一语言
- 热更代码有一定的逻辑复杂度（不只是 UI）
- 团队能接受 HybridCLR 的构建集成成本

### DHE（Differential Hybrid Execution）

**原理**：HybridCLR 的进阶模式。不是把整个程序集改为解释执行，而是**只把变更的函数**改为解释执行，未变更的函数继续使用 AOT 代码。

```
首包（AOT）：ClassA.Method1() [AOT], ClassA.Method2() [AOT]
热更后：     ClassA.Method1() [AOT], ClassA.Method2() [解释器]（已修改）
```

**优点**：
- 热更后的整体性能远高于全量解释——只有变更的函数走解释器
- 适合"修复少量 Bug"场景——大部分代码仍然是 AOT 性能

**缺点**：
- 需要精确的函数级差分计算
- 工程复杂度高于标准 HybridCLR
- 商业版本（非开源核心）

**适用场景**：
- 性能敏感的项目（热更后不想牺牲整体性能）
- 热更主要用于 Bug 修复而非大量新增功能
- 愿意投入 DHE 的工程化成本

## 选型决策框架

选型不是"哪个最新"，而是根据项目约束选择：

| 约束 | 推荐 | 理由 |
|------|------|------|
| 已有 Lua 方案且运行良好 | 继续 Lua | 迁移成本高于收益 |
| 新项目，Unity C# | HybridCLR | 统一语言，社区活跃 |
| 性能极度敏感 | DHE | 函数级差分，最小性能损失 |
| 热更代码量极小（只修 Bug） | DHE 或 配置热更 | 可能不需要完整的脚本热更能力 |
| 微信小游戏（WebGL） | 需要特殊处理 | HybridCLR 对 WebGL 的支持需确认 |

### 不推荐的做法

**项目中期从 Lua 迁移到 HybridCLR**。迁移成本极高——所有 Lua 代码需要重写为 C#，桥接层需要重新设计，测试需要全量回归。除非 Lua 方案遇到了不可解决的问题，否则不建议中期迁移。

**同一个项目混用两套热更方案**。例如 Lua 负责 UI 热更、HybridCLR 负责逻辑热更。两套方案的运行时、调试和构建管线互相干扰，维护成本翻倍。

## 与交付链路的关系

热更新架构的选型影响交付链路的多个环节：

| 环节 | 影响 |
|------|------|
| V03 工程基建 | 编译域需要按热更边界划分 |
| V04 版本管理 | 热更版本号和首包版本号的兼容矩阵 |
| V07 构建 | 热更包的构建步骤和 CI 集成 |
| V13 验证 | 热更后的兼容性测试策略 |
| V17 运营 | 热更发布和回滚流程 |

选型在项目初期决定，中期更换的成本极高。

## 小结与检查清单

- [ ] 是否明确了热更新架构的选型及理由
- [ ] 是否在项目初期完成了选型（而非中期更换）
- [ ] 是否只使用一套热更方案（不混用）
- [ ] 选型是否考虑了团队技术栈（Lua 经验 vs C# 经验）
- [ ] 选型是否考虑了性能要求（全解释 vs 差分）
- [ ] 选型是否考虑了平台限制（iOS 合规、微信 WebGL）
