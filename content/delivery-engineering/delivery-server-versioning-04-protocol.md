## 这篇解决什么问题

服务端和客户端的协议是两端交互的契约。服务端可以随时改，但客户端改了要过审核、要等玩家更新。怎么在这种不对称性下安全地演进协议——这是本篇的核心问题。

## Protobuf / FlatBuffers 的字段规则

| 规则 | 原因 |
|------|------|
| 永远不要复用已删除字段的编号 | 旧客户端可能仍在用该编号发送数据 |
| 新增字段必须使用新编号 | 保证旧客户端能正常解析 |
| 新增字段使用 optional | 旧客户端不会发送该字段 |
| 不要修改已有字段的类型 | 类型不匹配导致解析错误 |
| 使用 reserved 标记已废弃字段 | 防止未来不小心复用 |

### 安全变更 vs 危险变更

| 变更类型 | 安全性 | 说明 |
|---------|--------|------|
| 新增 optional 字段 | 安全 | 旧客户端忽略，新客户端使用 |
| 新增消息类型 | 安全 | 旧客户端不会发送该消息 |
| 新增枚举值 | 注意 | 旧客户端收到未知枚举值可能崩溃——需要 default 处理 |
| 删除字段 | 危险 | 旧客户端仍在发送该字段 |
| 修改字段类型 | 危险 | 解析错误 |
| 修改字段编号 | 致命 | 数据完全错乱 |

## 服务端同时支持新旧协议

客户端在握手消息中携带 `protocol_version`，Gateway 据此路由到对应的处理逻辑，或经过适配层转换后再路由。

### 适配层的维护成本

| 同时支持的版本数 | 复杂度 | 建议 |
|----------------|--------|------|
| 2 个版本 | 低 | 常见且可接受 |
| 3 个版本 | 中 | 需要明确淘汰时间 |
| 4+ 个版本 | 高 | 强烈不建议，用强制更新 |

## 客户端版本检测与路由

### 版本协商流程

1. 客户端发起连接，握手消息中携带 `client_version`、`protocol_version`、`platform`
2. 服务端检查：`protocol_version >= min_supported` → 允许连接；否则拒绝并返回强制更新提示
3. 服务端记录该连接的协议版本，后续通信使用对应版本的序列化/反序列化

### 版本路由表

| 协议版本 | 状态 | 支持截止日期 | 在线占比 |
|---------|------|------------|---------|
| v3 | 当前版本 | - | 65% |
| v2 | 兼容中 | 2026-06-01 | 30% |
| v1 | 即将淘汰 | 2026-05-01 | 5% |

## 灰度协调：服务端必须先行

前后端协议变更时，部署顺序至关重要。

```
T0  服务端部署新版本（同时支持 v2 和 v3 协议）
T1  验证 v2 客户端正常连接 ✓ / v3 内部测试客户端正常 ✓
T2  客户端提交审核
T3  审核通过，玩家逐步更新
T4  v2 占比 < 5%，启动 v2 淘汰
T5  服务端移除 v2 支持
```

| 部署顺序 | 结果 |
|---------|------|
| 服务端先 | 新旧客户端都正常 |
| 客户端先 | 新客户端连旧服务端——协议不兼容，直接报错 |

**没有例外**：包含协议变更的更新，服务端必须先部署且同时支持新旧协议。

## 协议废弃生命周期

| 阶段 | 持续时间 | 动作 |
|------|---------|------|
| Current | 不定 | 所有新客户端使用该版本 |
| Supported | 2-4 周 | 新版本已发布，旧版本仍支持 |
| Deprecated | 1-2 周 | 登录提示"请更新"，功能不受限 |
| Sunset | 3-7 天 | 部分功能受限 |
| Removed | - | 拒绝连接，强制更新 |

## 紧急协议变更

线上发现严重协议 bug 时：评估影响范围 → 服务端紧急修复协议处理逻辑（不改协议本身）→ 如果必须改协议，服务端做适配层兼容 + 推送客户端热更新 → 如果热更新无法修复，走紧急审核通道。

## 小结

- [ ] 协议变更是否遵循 Protobuf/FlatBuffers 字段兼容规则
- [ ] 是否用 reserved 标记了已废弃的字段编号
- [ ] 新增枚举值时客户端是否有 default 处理
- [ ] 服务端是否先于客户端部署（同时兼容新旧协议）
- [ ] 是否有协议版本淘汰生命周期和占比监控
- [ ] 是否有紧急协议变更的应急流程
