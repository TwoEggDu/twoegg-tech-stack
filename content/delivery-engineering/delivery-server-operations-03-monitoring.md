## 这篇解决什么问题

服务端部署上线后，怎么知道它是否健康？出问题时怎么快速定位？本篇讲可观测性的四根支柱和游戏服务端的特有监控指标。

## 四支柱概览

| 支柱 | 回答的问题 | 核心工具 |
|------|-----------|---------|
| 日志（Logs） | 发生了什么？ | ELK / Loki / CloudWatch Logs |
| 指标（Metrics） | 系统状态如何？ | Prometheus + Grafana |
| 追踪（Traces） | 一个请求经过了哪些服务？ | Jaeger / Zipkin / OpenTelemetry |
| 告警（Alerts） | 什么时候需要人介入？ | Alertmanager / PagerDuty |

四根支柱互补。单看日志能定位具体问题但缺乏全局视角，单看指标能发现异常但不知道具体原因，单看追踪能跟踪调用链但不知道是否普遍，告警把前三者的发现转化为行动。

## 支柱一：日志

### 结构化日志

游戏服务端的日志必须是结构化的（JSON 格式），而不是自由文本。

```json
{
  "timestamp": "2026-04-14T10:30:00Z",
  "level": "WARN",
  "service": "battle-server",
  "instance": "battle-server-03",
  "message": "player disconnect during match",
  "player_id": "12345",
  "room_id": "room-678",
  "reason": "timeout"
}
```

结构化日志的好处是可以被日志系统索引和查询——"查找所有 battle-server-03 在过去一小时内的 timeout 断连"变成一条查询语句。

### 日志级别

| 级别 | 含义 | 游戏服务端示例 |
|------|------|---------------|
| ERROR | 影响功能的错误 | 数据库写入失败、玩家数据丢失 |
| WARN | 潜在问题 | 玩家断连率异常、tick 超时 |
| INFO | 关键业务事件 | 玩家登录、对局开始/结束 |
| DEBUG | 调试信息 | 帧同步数据、状态快照 |

生产环境通常只开到 INFO 级别。DEBUG 级别的日志量可能极大（每帧一条），只在排查问题时临时开启。

### 日志管理

| 环节 | 方案 | 要点 |
|------|------|------|
| 采集 | Filebeat / Fluentd / Vector | 从容器 stdout 收集 |
| 存储 | Elasticsearch / Loki | Loki 成本更低，ES 查询更强 |
| 查询 | Kibana / Grafana | 按时间、服务、级别筛选 |
| 保留 | 分级保留策略 | ERROR 保留 90 天，INFO 保留 30 天，DEBUG 保留 7 天 |

## 支柱二：指标

### 通用服务端指标

| 指标 | 含义 | 健康阈值参考 |
|------|------|-------------|
| QPS | 每秒请求数 | 视业务而定 |
| 延迟 P50/P95/P99 | 请求响应时间分位数 | P99 < 100ms（游戏后端） |
| 错误率 | 请求失败的百分比 | < 0.1% |
| CPU 使用率 | 进程 CPU 占用 | < 70%（留余量应对峰值） |
| 内存使用率 | 进程内存占用 | < 80% |
| 连接数 | 当前活跃连接 | 不超过设计上限 |

### 游戏服务端特有指标

| 指标 | 含义 | 为什么重要 |
|------|------|-----------|
| 在线人数 | 当前在线玩家数 | 最核心的业务指标 |
| Tick 时间 | 服务端每帧处理时间 | 超过帧间隔就会造成卡顿 |
| 匹配队列时长 | 玩家等待匹配的时间 | 直接影响玩家体验 |
| 房间数量 | 当前活跃房间/对局数 | 反映服务器负载 |
| 帧同步偏差率 | 客户端和服务端状态差异 | 偏差大意味着外挂或 Bug |
| 断连率 | 每分钟断连次数 | 突增可能意味着网络或服务端问题 |

### 指标采集架构

```
游戏服务端 → Prometheus（拉取指标） → Grafana（可视化面板）
```

Prometheus 通过 HTTP 端点定时拉取指标。游戏服务端暴露 `/metrics` 端点，以 Prometheus 格式输出指标。

## 支柱三：追踪

### 分布式追踪

游戏后端通常由多个服务组成（网关、登录、匹配、战斗、存档）。一个玩家操作可能跨越多个服务。分布式追踪将这些跨服务的调用串联成一条链路。

**核心概念**：

| 概念 | 含义 |
|------|------|
| Trace ID | 一个请求的全局唯一标识 |
| Span | 一个服务内的处理单元 |
| 父子关系 | Span 之间的调用关系 |

**Request ID 传播**：每个请求在入口处生成 Trace ID，之后每次服务间调用都通过 Header 传递此 ID。日志中也记录 Trace ID——这样就能通过一个 ID 查到整条调用链的所有日志。

### 游戏服务端的追踪重点

不是所有操作都需要追踪。重点追踪跨服务调用的关键路径：

- 玩家登录 → 认证 → 加载存档 → 进入游戏
- 匹配请求 → 匹配服务 → 房间创建 → 通知玩家
- 购买道具 → 扣款 → 发货 → 记录日志

实时战斗内的帧同步通常不做分布式追踪（频率太高、开销太大）。

## 支柱四：告警

### 告警设计原则

| 原则 | 含义 | 反面 |
|------|------|------|
| 可操作 | 收到告警后知道该做什么 | "CPU 高了"——然后呢？ |
| 分级 | 不同严重程度不同响应方式 | 所有告警都发短信 |
| 防疲劳 | 避免重复、低价值告警 | 一天收到 200 条告警 |

### 告警分级

| 级别 | 含义 | 响应方式 | 示例 |
|------|------|---------|------|
| P0 | 服务不可用 | 立即电话通知 on-call | 全服断线、数据库不可用 |
| P1 | 功能受损 | 15 分钟内响应 | 匹配服务异常、充值失败 |
| P2 | 性能下降 | 工作时间处理 | P99 延迟升高、CPU 持续 > 80% |
| P3 | 需要关注 | 下个工作日处理 | 磁盘使用率 > 70%、日志量异常 |

### 阈值告警 vs 趋势告警

**阈值告警**：指标超过固定阈值触发（如 CPU > 90%）。简单直接，但容易漏报渐变问题。

**趋势告警**：指标的变化趋势异常触发（如在线人数 10 分钟内下降 30%）。能发现阈值告警捕捉不到的异常，但配置更复杂。

建议两者结合——阈值告警兜底，趋势告警捕捉渐变异常。

## 版本健康面板

部署新版本后，需要一个专门的面板来对比新旧版本的关键指标。核心关注：

| 对比项 | 关注点 |
|--------|-------|
| 错误率对比 | 新版本的错误率是否高于旧版本 |
| 延迟对比 | 新版本的 P95/P99 是否劣化 |
| 断连率对比 | 新版本是否导致更多断连 |
| 资源使用对比 | 新版本的 CPU/内存是否异常 |

这个面板是金丝雀发布的决策依据——如果新版本指标劣化就停止放量并回滚。

## 游戏服务端监控面板布局示例

以下是一个实际使用的 Grafana 面板布局设计，分三行展示。这个布局经过多次迭代，目标是让值班人员在 10 秒内判断"服务是否健康"。

### 第一行：全局概览（红黄绿灯 + 核心数字）

```
┌──────────────────┬──────────────────┬──────────────────┬──────────────────┐
│   在线人数         │   错误率           │   P99 延迟         │   断连率           │
│   12,345          │   0.03%          │   45ms           │   0.8/min        │
│   (↑5% vs 昨天)    │   (✓ < 0.1%)     │   (✓ < 100ms)    │   (✓ < 2/min)    │
│   [绿灯]           │   [绿灯]          │   [绿灯]          │   [绿灯]          │
└──────────────────┴──────────────────┴──────────────────┴──────────────────┘
```

每个面板用红黄绿三色背景：绿 = 健康，黄 = 接近阈值需关注，红 = 超标需立即处理。值班人员只需要看颜色，10 秒内就能判断全局状态。

### 第二行：按服务分的关键指标（时间序列图）

```
┌────────────────────────────┬────────────────────────────┐
│ CPU/内存 使用率（按服务分层）    │  请求延迟分布（P50/P95/P99）     │
│                            │                            │
│ gateway ████████░░ 78%     │  ----  P50 (15ms)          │
│ battle  ██████░░░░ 58%     │  ----  P95 (38ms)          │
│ match   ████░░░░░░ 42%     │  ----  P99 (45ms)          │
│ save    ███░░░░░░░ 35%     │                            │
└────────────────────────────┴────────────────────────────┘
┌────────────────────────────┬────────────────────────────┐
│ 在线人数趋势（24h）           │  房间数 / 匹配队列长度          │
│                            │                            │
│ ───────/\───────           │  活跃房间：456               │
│                            │  匹配队列：23（avg等待 8s）     │
└────────────────────────────┴────────────────────────────┘
```

### 第三行：游戏特有指标

```
┌────────────────────────────┬────────────────────────────┐
│ Tick 时间分布                │  帧同步偏差率                  │
│                            │                            │
│ < 33ms (30fps): 98.5%      │  偏差 < 1%: 99.2%          │
│ 33-50ms: 1.2%              │  偏差 1-5%: 0.6%           │
│ > 50ms (卡顿): 0.3%        │  偏差 > 5% (疑似外挂): 0.2%  │
└────────────────────────────┴────────────────────────────┘
```

### 面板设计的几个实操经验

**1. 面板上的数据刷新间隔很重要**。概览层 10-15 秒刷新一次（让值班人员看到实时状态），趋势层 1 分钟刷新一次（不需要太频繁），明细层 5 分钟刷新一次（查询量大，频繁刷新会给 Prometheus 带来压力）。

**2. 面板链接（drill-down）比堆砌面板更有效**。某次我们为了"全面"，在一个 Dashboard 上放了 40+ 个面板——结果页面加载要 15 秒，没人愿意打开。后来精简到 12 个面板，每个面板点击可以跳转到详细页面，加载时间降到 3 秒以内。

**3. 面板上必须有"版本标记"**。在时间序列图上标记每次版本部署的时间点（Grafana 的 Annotations 功能），这样一眼就能看出"指标变化是不是和某次部署有关"。某次 P99 延迟突增，值班人员花了 30 分钟排查才发现是 20 分钟前的一次部署引起的——如果面板上有部署标记，这个关联在 5 秒内就能建立。

### 游戏服务端特有的告警规则

除了通用的服务端告警（CPU/内存/错误率），游戏服务端需要一些特有的告警规则：

```yaml
# Tick 超时告警：服务端帧处理时间持续超标
- alert: TickTimeout
  expr: |
    histogram_quantile(0.95,
      rate(game_tick_duration_seconds_bucket[5m])
    ) > 0.05
  for: 3m
  annotations:
    summary: "战斗服务器 Tick P95 超过 50ms，玩家会感到卡顿"

# 匹配等待时间过长
- alert: MatchQueueTooLong
  expr: |
    histogram_quantile(0.95,
      rate(match_wait_duration_seconds_bucket[5m])
    ) > 30
  for: 5m
  annotations:
    summary: "匹配等待 P95 超过 30 秒，检查匹配服务状态和在线人数"

# 在线人数异常下降（可能是服务端故障导致大量断连）
- alert: OnlinePlayerDrop
  expr: |
    (
      max_over_time(online_players[10m])
      - online_players
    ) / max_over_time(online_players[10m]) > 0.2
  for: 2m
  annotations:
    summary: "在线人数 10 分钟内下降超过 20%，可能发生大规模断连"

# 帧同步偏差率异常（可能是外挂或服务端 Bug）
- alert: FrameSyncDeviation
  expr: |
    sum(rate(frame_sync_deviation_total{level="high"}[5m]))
    /
    sum(rate(frame_sync_deviation_total[5m]))
    > 0.01
  for: 10m
  annotations:
    summary: "帧同步高偏差率超过 1%，检查是否有外挂或状态同步 Bug"
```

**经验上来看**，游戏服务端最容易被忽略的告警是**在线人数异常下降**。很多团队只监控了"服务是否还在运行"（健康检查），但没有监控"服务虽然在运行但玩家在大量掉线"。服务进程还活着不代表它是正常的——可能内部状态已经异常，只是还没完全挂掉。在线人数是最直接的"用户可用性"指标。

## 小结

- [ ] 日志是否为结构化 JSON 格式
- [ ] 是否采集了游戏服务端特有指标（在线人数、tick 时间等）
- [ ] 关键路径是否有分布式追踪（登录、匹配、购买）
- [ ] 告警是否分级（P0-P3），且每条告警都有对应的操作手册
- [ ] 是否有版本健康对比面板
- [ ] 日志是否有分级保留策略
