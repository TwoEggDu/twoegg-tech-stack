写到这里，这条系列已经把资源系统的几条主线都铺出来了：

- 资产是什么，引用怎么成立
- `Scene / Prefab` 和对象恢复链站在哪
- `AssetBundle` 为什么解决的是交付，不是单纯加载
- `Addressables` 为什么更像建立在交付层之上的管理层
- 工程治理、案例、框架选型各自该怎么看

这时候项目里最自然会冒出来的一个问题就是：

`那 Resources、StreamingAssets、AssetBundle、Addressables 到底各自该在什么场景下用？`

这个问题之所以值得单独写，是因为它特别容易在项目里被聊歪。

很多讨论会变成：

- 哪个更先进
- 哪个已经过时
- 哪个性能更好
- 哪个更适合大项目

这些问题不完全没意义，但如果不先把它们放回资源系统分层里，最后很容易得到一些过粗甚至误导的结论，比如：

- “有了 Addressables 就不要再碰 Resources”
- “StreamingAssets 就是低配版 AssetBundle”
- “AssetBundle 才是专业做法，所以所有内容都应该进 bundle”

这些说法都沾一点边，但都不够稳。

所以这篇我就只做一件事：

`把这四条资源路径放回同一张决策图里。`

## 先给一句总判断

如果把整件事压成一句话，我会这样说：

`Resources、StreamingAssets、AssetBundle 和 Addressables，最不该被理解成四个同层替代品；更稳的理解是：Resources 和 StreamingAssets 更像内容归属方式，AssetBundle 更像交付容器，Addressables 更像建立在交付层之上的管理层。`

这句话里最关键的是：

- `内容归属方式`
- `交付容器`
- `管理层`

只要这三层先分开，很多选择就会自然清楚很多。

## 一、先别问“哪个好”，先问你要解决哪一类问题

这一步最重要。

因为这四条路径真正不同的，不在“能不能把资源加载出来”，而在：

`你到底要解决哪一类问题。`

## 1. 你是在解决“程序内置带什么”

这更接近：

- `Resources`
- `StreamingAssets`

## 2. 你是在解决“内容怎样被独立构建和分发”

这更接近：

- `AssetBundle`

## 3. 你是在解决“运行时怎样统一定位、依赖调度和生命周期管理”

这更接近：

- `Addressables`

如果这三类问题没先拆开，后面就很容易变成：

`拿管理层去替代归属方式，或者拿交付容器去替代运行时资源系统。`

## 二、Resources：适合少量稳定内置内容，不适合承担复杂交付治理

`Resources` 这条路最容易被极端化。

一边有人把它当成“新手方案”，一边有人会因为它简单就一路用到底。

这两种都不太稳。

## 1. 它最适合什么场景

`Resources` 最适合的，其实是下面这类内容：

- 数量不大
- 稳定内置
- 不需要独立更新
- 路径语义简单明确

例如：

- 很小量的启动期基础内容
- 极少量必须和程序强绑定的默认资源
- 某些演示、工具、原型期内容

它的最大价值是：

`简单。`

也就是：

- 不需要额外构建交付链
- 不需要额外 catalog
- 不需要额外 bundle 依赖管理

## 2. 它最不适合什么场景

只要项目开始碰到下面这些需求，`Resources` 通常就不再稳：

- 内容量持续增长
- 需要远端更新
- 需要按入口分发
- 需要精细控制首包和非首包
- 需要显式治理依赖、缓存和回滚

因为 `Resources` 的核心问题不是“能不能用”，而是：

`它几乎不给你显式交付边界。`

## 3. 它常见的误用

最常见的误用是：

- 一开始只是放一点点
- 后来因为“方便”不断往里加
- 最后整个项目形成一个事实上的隐式大包

到这一步，问题通常不是“加载 API 不优雅”，而是：

`你已经失去了内容交付的解释能力。`

这背后有一个具体的引擎层面原因。构建时，Unity 会把项目里所有 `Resources/` 文件夹下的资产统一打入 `resources.assets`（以及溢出文件 `resources.assets.resS`）。启动时，引擎内部的 `ResourceManager` 会为 `resources.assets` 里的全部路径建一棵完整的内存查找树。这个启动开销和 Resources 里的资产总量线性相关——即使某些资产在运行期间从不被加载，开销一样会被支付。而 `Resources.Load` 本身是一个主线程同步调用，在这棵查找树上做字符串匹配。所以 Resources 不适合大规模使用的原因不仅仅是"没有显式交付边界"，更是可以实测的启动开销和只有同步加载这一条路，且两者都会随资产数量增长而恶化。

所以更稳的定位是：

`Resources 适合少量稳定内置内容，不适合承担中大型项目的资源交付治理。`

## 三、StreamingAssets：适合“原样带文件”，不适合承担 Unity 对象级资源管理

`StreamingAssets` 也经常被误读。

很多团队会把它和 `AssetBundle` 混在一起，觉得只是“另一种放文件的位置”。

其实它们解决的问题并不一样。

## 1. 它最适合什么场景

`StreamingAssets` 更适合下面这类内容：

- 你就是想把某个文件原样带进安装包
- 它不需要 Unity 对象级反序列化语义
- 它更像原生文件，而不是 Unity 资产对象图

例如：

- 配置文件
- 原生数据库
- 视频、文本、二进制表
- 某些第三方 SDK 要求按原始文件访问的内容

它解决的是：

`文件怎么原样进程序。`

不是：

`Unity 资源对象怎么被交付和恢复。`

## 2. 它最不适合什么场景

它不适合直接承担这些责任：

- Unity 对象级依赖管理
- 场景、Prefab、Material 这类资产对象的交付边界治理
- 资源按需分发和版本更新
- bundle 级缓存、依赖与回滚治理

因为它根本不是围绕 Unity 资产系统设计的交付层。

## 3. 它常见的误用

最常见的误用是：

- 为了“可直接读文件”，把大量本应进入资源交付系统的内容塞进来
- 结果后面远端更新、依赖治理、平台差异和包体控制都越来越难做

还有一个在生产中极容易踩到的坑：`Application.streamingAssetsPath` 在不同平台指向的位置和可用的文件访问方式差异很大。**Android** 上 StreamingAssets 被打在 APK 内部（压缩包），`System.IO.File.ReadAllBytes()` 和 `File.Exists()` 都无法使用，必须走 `UnityWebRequest.Get()` 或者 `AndroidJNI` 来读取——这是跨平台时最常见的报错来源。**iOS** 上路径在 app bundle 内部，可以用普通文件 API 读取，但严格只读。**WebGL** 上 StreamingAssets 由服务器通过 HTTP 提供，所有读取都是异步的，不存在同步文件访问。**Windows / macOS / Linux** 上则是普通文件系统路径，完全可读。很多团队直到 QA 阶段才发现 Android 上的限制，不得不临时改架构。如果项目目标平台包含移动端，最稳的做法是统一使用 `UnityWebRequest` 读取 StreamingAssets，以确保跨平台安全。

所以更稳的定位是：

`StreamingAssets 适合原生文件，不适合替代完整的 Unity 资源交付系统。`

## 四、AssetBundle：适合解决“独立构建和分发”，但它本身不是高层管理层

`AssetBundle` 在这四者里，位置最容易被抬高也最容易被误用。

## 1. 它最适合什么场景

只要你开始碰到下面这些需求，`AssetBundle` 就会自然变得重要：

- 内容需要独立于 Player 构建
- 内容需要远端分发
- 内容需要按包更新
- 需要显式管理依赖、压缩、缓存和回滚

因为 `AssetBundle` 最核心解决的是：

`Unity 资产对象怎样被输出成可独立交付单元。`

这也是为什么它站在交付层，而不是简单的”加载路径”。

从具体性能机制来看，`Resources.Load` 在引擎内部做的是：在启动时构建的内存路径树上执行字符串查找，然后在主线程上同步反序列化——没有异步变体。而 `AssetBundle.LoadAssetAsync` 使用的是 bundle 内部 SerializedFile 对象表中的预计算索引，支持通过加载线程进行异步反序列化，对于 LZ4 压缩的 bundle 还能按需做块解压。性能差异不只是”交付灵活性”的区别，而是可以实测的：异步加载避免了主线程卡帧，预计算索引也比字符串树查找更快。这也是即使资源只在本地使用、不需要远端更新，从 Resources 迁移到 AssetBundle / Addressables 仍然有技术收益的具体原因之一。

## 2. 它最不擅长什么

它最不擅长的不是加载，而是：

`直接给你一套好用的高层运行时资源管理语义。`

如果你直接裸用 `AssetBundle API`，团队很快就要自己面对：

- 地址定位
- 依赖拉起
- 句柄生命周期
- 目录和清单管理
- 版本切换
- 回滚和校验

所以它非常适合做交付容器，不一定适合直接拿来当完整资源系统。

## 3. 它常见的误用

最常见的误用是：

- 觉得 “专业项目都用 AssetBundle”
- 然后直接裸 API 起一整套运行时管理层

最后常常会在几年里把：

- 依赖调度
- 清单系统
- 缓存系统
- 回滚系统

全都自己补一遍。

所以更稳的定位是：

`AssetBundle 适合承载交付，但不应该自动被等同于完整资源管理方案。`

## 五、Addressables：适合解决“高层定位与调度”，但它不是新的资源本体

`Addressables` 最容易被误读成“新版 AssetBundle”。

其实不是。

## 1. 它最适合什么场景

当项目已经明确需要：

- 动态内容
- 远端或本地统一定位
- 依赖自动拉起
- 生命周期和句柄管理

同时团队又不想自己从裸 bundle 开始补整套管理层时，`Addressables` 通常是很稳的路线。

它最适合的，是：

`给 AssetBundle 这层再包一层高层资源管理语义。`

## 2. 它最不适合什么期待

它不适合被期待自动解决这些事：

- 分组边界为什么不合理
- 重复资源为什么越来越多
- shader variant 为什么缺
- 首载路径为什么卡
- 回滚快照为什么不一致

这些问题依然还在，只是被移动到了更高层接口后面。

## 3. 它常见的误用

最常见的误用是：

- 以为上了 Addressables 就可以不再理解 AssetBundle
- 或者以为上了 Addressables 以后，资源交付边界就会自动正确

这通常不成立。

所以更稳的定位是：

`Addressables 适合做高层管理层，但它不会替团队定义正确交付边界。`

## 六、可以把四者理解成一条逐层递进的资源路径图

如果把它们放回同一张图里，我更建议这样记：

## 1. Resources

解决：

`少量稳定内置内容怎么简单带进程序。`

## 2. StreamingAssets

解决：

`某些原生文件怎么按原样带进程序。`

## 3. AssetBundle

解决：

`Unity 资产对象怎么被构建成交付单元。`

## 4. Addressables

解决：

`运行时怎样统一定位这些交付单元并管理依赖和生命周期。`

一旦这样看，很多“哪个好”的问题都会自然降维成：

`你现在缺的是哪一层能力。`

## 七、如果只给一套最小决策法，可以这样选

如果你现在就要在项目里做第一轮判断，我会建议先按下面这个顺序问。

## 1. 这批内容是不是少量、稳定、必须跟程序一起内置

如果是：

- 可以优先评估 `Resources`

但前提是：

- 真的少量
- 真的稳定
- 真的不打算独立治理

## 2. 这批内容是不是原生文件，且你明确需要按文件方式访问

如果是：

- 可以优先评估 `StreamingAssets`

不要为了“方便带文件”把 Unity 对象级内容也一起往里塞。

## 3. 这批内容是不是需要独立构建、独立分发、独立缓存和独立回滚

如果是：

- 必须回到 `AssetBundle`

因为这是交付容器层的问题。

## 4. 你们是不是还想让运行时的定位、依赖和生命周期都更高层统一起来

如果是：

- 再往上叠 `Addressables`

也就是：

`AssetBundle 负责交付，Addressables 负责管理。`

## 八、最常见的四种误判

最后把几个最常见的误判也压一下。

## 1. 误判一：Resources 过时了，所以任何项目都不该用

更准确的说法应该是：

`Resources 不适合承担复杂交付治理，但对少量稳定内置内容仍然有价值。`

## 2. 误判二：StreamingAssets 就是简陋版 AssetBundle

更准确的说法应该是：

`StreamingAssets 更像原生文件归属路径，不是 Unity 对象交付容器。`

## 3. 误判三：只要用了 AssetBundle，就已经有了一套成熟资源系统

更准确的说法应该是：

`AssetBundle 先给你的是交付单元，不是高层管理层。`

## 4. 误判四：Addressables 能代替前面三条所有路径

更准确的说法应该是：

`Addressables 主要站在管理层，它不会让“原生文件怎么带”“少量内置资源怎么放”“交付容器怎么构建”这些问题自动消失。`

## 最后收成一句话

如果把这篇最后再压回一句话，我会这样说：

`Resources、StreamingAssets、AssetBundle 和 Addressables，真正的区别不在“谁更新、谁高级”，而在“谁负责把什么内容带进程序、谁负责把什么内容做成交付物、谁负责在运行时管理这些交付物”；先分清层，再选路径，才不容易把短期方便写成长期债务。`
