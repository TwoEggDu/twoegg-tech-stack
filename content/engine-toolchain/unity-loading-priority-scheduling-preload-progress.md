上一篇把异步加载管线拆开了：AsyncOperation、协程和 UniTask 都是调度层工具，底层哪些阶段在主线程、哪些在后台，由引擎决定。但知道了"加载链长什么样"以后，紧接着就会碰到一个更实际的问题：

`这么多资源要加载，先加载谁、后加载谁、什么时候偷偷提前加载、用户看到的进度条到底在反映什么？`

这就是加载优先级调度。它不是一个 API 问题，而是一个运行时资源调度的工程设计问题。

## 先给一句总判断

`加载优先级调度的核心不是"怎么让单次加载更快"，而是"怎么让用户在最需要资源的那一刻，资源已经准备好了"。它要解决的是时序问题，不是吞吐量问题。`

## 一、关键路径和非关键路径，先分清楚再谈调度

在讨论优先级之前，先定义一个基本概念：什么叫"关键路径资源"。

### 1. 关键路径资源

阻塞用户当前操作的资源。如果它没准备好，用户要么看到黑屏，要么看到占位符，要么操作卡住。

典型例子：

- 当前场景的地形、主角模型、主相机需要的 Shader
- 战斗开始前的技能特效 Prefab
- UI 界面切换时的目标页面资源
- 登录流程中的必要配置表和基础贴图

### 2. 非关键路径资源

不影响用户当前操作，但后续会用到的资源。它可以晚一点到，用户不会感知。

典型例子：

- 下一个场景的资源（用户还在当前场景）
- 角色列表里尚未选中的角色模型
- 成就系统、图鉴系统等低频功能的资源
- 远处 LOD 层级的高清模型

### 3. 为什么要先分这个

因为如果不分，所有加载请求在引擎内部排成一条队列，先到先服务。这意味着一个后台预加载的低优先级请求，可能排在用户即将看到的关键资源前面，导致关键资源反而延迟到达。

`调度的第一步不是写代码，而是画出当前状态下的关键路径。`

## 二、预加载该放在什么时机

预加载的核心思想很朴素：在用户还不需要的时候，把资源提前加载到内存。但执行起来有很多容易踩的坑。

### 1. 太早预加载的问题

如果在游戏启动时就把大量资源预加载进来，会出现：

- 启动时间变长，用户在 Loading 界面等得更久
- 内存被大量占用，后续真正需要的资源可能因为内存不够被挤掉
- 预加载的资源可能根本用不到（用户不一定会进入那个场景）

### 2. 太晚预加载的问题

如果等到用户快要进入目标场景才开始预加载，和不预加载几乎没有区别——加载时间仍然会落在用户的等待时间里。

### 3. 合理的预加载时机

通常的做法是利用用户已经在做其他事情的间隙：

- 场景过渡的 Loading Screen 期间：除了加载当前场景，同时把下一步高概率需要的资源也拉进来
- 用户在菜单界面停留时：后台静默加载下一场战斗的核心资源
- 战斗结束的结算界面：后台开始加载大厅或下一关资源
- 新手引导的固定流程：每一步引导切换时，预加载下一步需要的资源

关键原则是：

`预加载应该发生在用户注意力被其他事情占据的时段，而不是在用户等待加载完成的时段。`

### 4. 预加载不等于"加载完就放着不管"

预加载完成后，资源在内存里。但如果后续一直没有使用，它就变成了内存浪费。所以预加载必须和资源生命周期管理配合：

- 预加载的资源如果超过一定时间未被使用，应该考虑释放
- 预加载的资源如果被内存压力触发的清理机制回收了，下次使用时需要重新加载
- 预加载的优先级应该低于当前关键路径的加载

## 三、并发控制：不是提交越多加载请求越好

很多项目在实现预加载时，会一次性提交几十上百个异步加载请求。这看起来很"并发"，实际上会带来问题。

### 1. Unity 内部的加载队列有容量边界

Unity 的异步加载并不是每个请求都开一个线程。引擎内部有自己的调度队列，真正并行执行的 I/O 和反序列化操作是有限的。过多的请求只会在队列里排队，不会真正并行。

### 2. 大量并发请求会放大内存峰值

如果同时有 50 个 bundle 在加载，每个都在进行解压和反序列化，中间态的内存占用会叠加。这在内存有限的移动设备上很容易触发内存警告甚至 OOM。

### 3. 并发请求会争抢主线程时间

即使 I/O 和解压在后台，完成回调、Managed Binding 和部分反序列化仍然在主线程。如果一帧内有大量异步操作完成，回调会挤占渲染时间，造成帧率波动。

### 4. 实践中的并发控制

常见的做法是维护一个加载队列管理器：

- 设定最大并发数（通常 3–8 个，取决于目标设备性能）
- 关键路径请求插队，优先执行
- 非关键路径请求排队，按优先级依次执行
- 监控当前帧时间，如果主线程压力大，暂停提交新请求

这不需要多复杂的系统，一个带优先级的请求队列加上并发数限制就够了。但如果不做，加载行为在低端设备上会变得不可预测。

## 四、进度反馈的真实语义

用户看到的加载进度条，和真实的加载进度之间往往有很大差距。这不是因为故意欺骗用户，而是因为"进度"这个概念在加载链里不是线性的。

### 1. AsyncOperation.progress 反映的是什么

Unity 的 `AsyncOperation.progress` 返回 0 到 0.9 之间的值（0.9 表示加载完成但未激活，1.0 表示激活完毕）。但这个进度不是线性的——它反映的是引擎内部的处理阶段，不是用户感知的"还要等多久"。

可能前 80% 的进度在 0.5 秒内完成（I/O 很快），最后 20% 在 2 秒内完成（反序列化和实例化很慢）。用户看到的就是进度条飞到 80% 然后卡住不动。

### 2. 多资源加载时的进度聚合

如果一次场景切换需要加载 20 个 bundle，进度条应该反映的是所有 bundle 的综合进度。但每个 bundle 的大小和加载时间不同，简单地用"完成个数 / 总个数"会导致进度条跳跃——加载一个大 bundle 时进度条长时间不动，加载一批小 bundle 时进度条快速跳动。

### 3. 更合理的进度反馈设计

- 按资源权重计算进度：大资源占更高的权重，小资源占较低权重
- 分阶段显示：比如"下载中 → 解压中 → 加载中"，让用户知道系统在做什么
- 进度条平滑插值：即使底层进度跳跃，前端用插值让进度条匀速前进
- 设定最低显示时间：防止 Loading 界面一闪而过，用户反而觉得不稳定

核心原则：

`进度反馈是用户体验设计，不是系统状态的忠实映射。它的目标是让用户觉得"还在加载，没有卡死"，而不是精确显示底层每个字节的状态。`

## 五、常见误判

### "给每个加载请求都设成高优先级"

如果所有请求都是高优先级，等于没有优先级。关键路径资源的"优先"必须建立在有东西被"延后"的基础上。

### "预加载做得越多越好"

预加载是用内存换时间。如果预加载的资源超过了设备的承受能力，反而会触发内存清理，导致已加载的资源被回收，需要重新加载——等于白加载了。

### "进度条到 100% 就可以切场景了"

加载完成不等于可以立即使用。Shader 可能还没预热，首次可见时仍然会卡。大量 Instantiate 可能需要分帧。进度条到 100% 以后，通常还需要一个初始化阶段。

### "异步加载不需要管顺序"

如果依赖关系没处理好，可能出现 B 依赖 A，但 B 先加载完、A 还没到，导致 B 的资源引用丢失。框架层（Addressables / YooAsset）通常会处理依赖顺序，但自研加载系统必须自己管。

## 六、最小检查表

- 是否明确区分了关键路径资源和非关键路径资源？
- 关键路径资源是否有插队机制，保证优先加载？
- 预加载的时机是否合理？是在用户空闲时段还是在用户等待时段？
- 异步加载的并发数是否有上限控制？
- 进度反馈是否考虑了加权和平滑？
- 预加载的资源是否有超时释放机制？
- 加载完成后是否还有 Shader 预热和分帧实例化的初始化阶段？

## 结语

加载优先级调度不是一个高深的算法问题，而是一个时序设计问题。核心就是三件事：先加载用户马上要看到的，利用间隙偷偷加载后面要用的，控制好并发别把设备压垮。做好这三件事，加载体验就不会差。
