> DHE 不是把 DLL 整个替换，而是在函数粒度上决定每个方法走 AOT 还是走解释器——这件事的实现比听起来复杂得多。

这是 HybridCLR 系列第 25 篇。

HCLR-11 已经把 DHE 的定位立住了：它不是"解释器更快一点"，而是对已打进包体的 AOT 程序集做函数级差分执行。那篇文章回答的是"DHE 到底在解决什么"，但刻意没有往实现细节下探。

这篇接着往下走，回答的是另一组问题：

`dhao 文件里到底记录了什么？差分算法怎么判定一个方法"变了"？运行时的函数级分流在代码层面到底怎么实现？`

> 本文实现分析基于 HybridCLR 官方公开文档（v6.x ~ v7.x）。DHE 属于商业版能力，runtime 源码不公开。凡是落到具体实现推断的部分，都会标注"架构推断"字样，不会把推测写成源码事实。

## 从 HCLR-11 接续

HCLR-11 确立了几个关键判断：

- DHE 的差分粒度是函数级，不是程序集级
- 被标记为 DHE 的程序集同时存在于包体（AOT 版）和热更版本（最新 DLL）
- `dhao` 文件是离线计算出来的变化集，运行时按它做分流
- DHE 对加载顺序和边界一致性的要求比普通热更主线更强

但 HCLR-11 没有回答的问题还有很多：

1. dhao 文件的生成依赖哪些输入？构建管线的每一步具体做什么？
2. "逐条比对 IL"到底意味着什么？有哪些边界情况？
3. 运行时加载 dhao 之后，MethodInfo 上的 methodPointer 和 invoker 到底怎么被替换？
4. 间接脏函数注入是怎么回事？为什么短函数需要特殊处理？
5. DHAO 工作流和后来的 MetaVersion 工作流有什么区别？

这篇逐一回答。

## DHE 的构建流程

DHE 的构建管线可以拆成三个阶段，每个阶段的输入和输出都很明确。

### 阶段一：主包构建与 AOT Snapshot 创建

第一步和普通 HybridCLR 项目没有区别：在 `HybridCLRSettings` 中把需要 DHE 的程序集加入 `differentialHybridAssemblies` 列表，然后正常构建主包。

构建完成后，调用 `DhaoWorkflow.CreateAotSnapshot(BuildTarget target, string outputSnapshotDir)` 创建 AOT Snapshot。

AOT Snapshot 本质上是一组文件的集合：

- 构建时生成的 AOT DLL（这些 DLL 必须与主包二进制中的代码精确一致）
- `InjectRules`（函数注入策略的配置）
- `manifest.json`（快照的元信息）

这里有一个容易踩的坑：AOT Snapshot 必须在构建主包之后创建，不能在 `HybridCLR/Generate/All` 之后就创建。原因是 `Generate/All` 生成的 DLL 和实际构建产出的 DLL 可能不完全一致——构建过程中 IL2CPP 还会做额外的处理。如果 Snapshot 中的 DLL 和包体中的不一致，后续生成的 dhao 文件就会把本来没变的方法误判为已变化。

### 阶段二：热更 DLL 编译与 dhao 生成

当需要发布热更新时：

1. 编译最新的热更 DLL（这些 DLL 包含了对 DHE 程序集的修改）
2. 调用 `DhaoWorkflow.GenerateDhaoFiles(string aotSnapshotDir, string hotUpdateSnapshotDir, string dhaoOutputDir)` 生成 dhao 文件

这个函数的输入是两个目录：AOT Snapshot 目录（包含打包时的 DLL）和最新热更 DLL 目录。输出是一组 dhao 文件，每个 DHE 程序集对应一个。

如果需要加密，则改用 `HybridCLR.Editor.DHE.BuildUtils.EncryptDllAndGenerateDHAODatas`，它会同时生成加密后的 DLL 和对应的 dhao 文件。

### 阶段三：资源打包与分发

把热更 DLL 和 dhao 文件一起放入热更资源系统（Addressables、YooAsset 或自定义方案），随版本下发到客户端。

整条管线可以压成一句话：

`打包时的 AOT DLL + 最新热更 DLL → 离线差分 → dhao 文件 → 随热更资源下发。`

### 首包的特殊情况

首包发布时，因为代码没有任何变化，不需要生成 dhao 文件，也不需要携带额外的热更 DLL。运行时直接用 `LoadOriginalDifferentialHybridAssembly` 加载即可，只需要传程序集名称，不需要传任何数据。

这个设计很合理：首包的 DHE 程序集本来就全是 AOT 代码，没有任何方法需要走解释器，所以连变化集都不需要计算。

## dhao 文件到底记录了什么

官方文档对 dhao 文件的描述非常直接：它包含离线计算出来的"变化了哪些类型、哪些函数"的信息。

但如果只停在这一句，很多工程问题就没法回答。基于官方文档已经公开的行为，可以进一步分析 dhao 文件至少需要记录什么。

### 从运行时需求倒推内容

运行时加载 dhao 文件后，需要做到一件事：对 DHE 程序集中的每个方法，判定它应该走 AOT 路径还是解释器路径。

要支撑这个判定，dhao 文件至少需要包含：

**方法级变化标记**：每个方法是 unchanged（未变化，走 AOT）、changed（已变化，走解释器）还是 new（新增，走解释器）。这是 dhao 的核心数据。

**类型级变化信息**：如果一个类型新增了字段或改变了布局，仅靠方法级标记不够——类型本身的 metadata 也需要更新。

**方法匹配索引**：新旧两个 DLL 中的方法需要能正确对应。方法的 metadata token 在重新编译后可能会变化（比如新增了一个类型，导致后续所有 token 偏移），所以 dhao 不能只存 token，还需要某种匹配策略。

### 官方已确认的内容 vs 架构推断

| 内容 | 来源 |
|------|------|
| 包含变化的类型和函数信息 | 官方文档明确说明 |
| 运行时用它决定走 AOT 还是解释器 | 官方文档明确说明 |
| 离线计算，不在运行时做差分 | 官方文档明确说明 |
| 具体的二进制格式 | 未公开（商业版实现） |
| 方法匹配策略的具体实现 | 未公开（架构推断） |

关键的一点是：dhao 文件的存在，是为了避免运行时做差分计算。官方文档明确说过，如果没有 dhao 文件，就需要额外携带原始 AOT DLL，并且在运行时计算函数变化，成本非常高。这说明 dhao 文件本质上是一个预计算的查找表，而不是一个需要运行时解析的差分补丁。

## 差分算法：怎么判断一个方法"变了"

这是 DHE 实现中最有技术含量的部分之一。官方文档对差分算法有一句非常关键的描述：

`工具逐条比对 IL 来判定等价性。`

这句话看起来简单，但展开之后会发现里面有很多值得分析的边界情况。

### 逐条 IL 比对的含义

两个版本的同一个方法，如果它们的 IL 方法体（method body）中的指令序列完全一致，就判定为未变化；否则判定为已变化。

这里的"完全一致"不是指 IL 字节流的逐字节比较（因为 metadata token 会变），而更可能是指令级的语义比对：每条 IL 指令的 opcode 相同，操作数在语义上等价（比如同一个方法引用，即使 token 值不同，只要指向的方法签名相同就算匹配）。

### 为什么不能直接比字节流

考虑一个简单的场景：原始 DLL 中有 100 个类型，新版 DLL 在中间插入了一个新类型。这会导致：

- TypeDef 表中后续所有类型的 token 偏移
- 引用这些类型的 IL 指令中的 metadata token 全部改变
- 方法体的原始字节流因此不同

但这些方法的逻辑并没有变。如果用字节流直接比较，就会把大量未修改的方法误判为已变化，DHE 的性能优势就没了。

所以差分工具必须做 token 重映射：先建立新旧 DLL 之间的类型/方法/字段映射关系，然后在这个映射的基础上比较 IL 指令序列。

### 误判与 UnchangedAttribute

官方文档坦率地承认了一个事实：逐条 IL 比对并不能判断逻辑等价性。

举一个具体例子：如果你只是调换了两行不相关代码的顺序，编译器生成的 IL 序列会不同，但逻辑是等价的。差分工具会把这种情况判定为"已变化"，该方法会被切换到解释执行。

官方对此的态度是务实的：不存在完美的代码能判断逻辑等价性。对于大多数场景，这种误判不会造成功能问题（方法仍然能正确执行，只是走了解释器而不是 AOT），只会略微降低性能。

但如果某个方法对性能极度敏感，开发者可以手动用 `UnchangedAttribute` 来覆盖自动判定：

- `[Unchanged]` 或 `[Unchanged(true)]`：手动标记为未变化，强制走 AOT
- `[Unchanged(false)]`：手动标记为已变化，强制走解释器
- 不加标注：由工具自动计算

这个设计很聪明：自动判定覆盖 99% 的场景，手动标注处理剩下 1% 的边界情况。

### 需要注意的边界情况

**新增方法**：新 DLL 中存在但旧 DLL 中不存在的方法，只能走解释执行。这不需要差分比对，只需要在方法匹配阶段发现"无法匹配"即可。

**新增类型**：整个类型都是新增的，其中所有方法都走解释执行。

**方法体不变但引用的类型变了**：比如方法 A 调用了方法 B，方法 A 的 IL 没变，但方法 B 的签名或行为变了。这种情况下方法 A 的 IL 比对结果是"未变化"，仍然走 AOT。这通常是正确的，因为 AOT 代码调用的是方法 B 的入口地址，而方法 B 的入口地址会被 DHE 运行时正确地重定向。

**字段增减对值类型的影响**：如果一个值类型（struct）增加或删除了字段，它的内存布局会改变。所有直接操作这个值类型的方法，即使自身 IL 没变，也可能需要重新判定。这是 DHE 工程边界中比较棘手的一类情况。

## 运行时函数级分流

构建期把 dhao 文件准备好之后，真正的分流发生在运行时。这部分涉及 DHE 程序集的加载和方法路由两个环节。

### 加载 API

官方提供了三个加载 API，对应不同的场景：

**`LoadOriginalDifferentialHybridAssembly(assemblyName)`**：用于首包或完全没有变化的情况。只传程序集名称，不需要传 DLL 数据或 dhao 数据。运行时直接使用包体中的 AOT 代码，所有方法都走 AOT 路径。

**`LoadDifferentialHybridAssemblyWithDHAO(dllBytes, dhaoBytes)`**：用于有热更新发布的情况。传入最新的热更 DLL 和对应的 dhao 文件。运行时根据 dhao 中的变化信息做函数级分流。

**`LoadDifferentialHybridAssemblyWithMetaVersion(dllBytes, metaVersionBytes)`**：v7.7.0 之后新增的加载方式，使用 MetaVersion 工作流替代 DHAO 工作流。

需要注意的是：DHE 程序集不能用标准的 `Assembly.Load` 加载。因为 DHE 程序集已经以 AOT 形式存在于包体中，运行时需要额外的参数来决定如何做分流。`Assembly.Load` 的语义是"加载一个全新的程序集"，这与 DHE 的"在已有 AOT 程序集上做差分覆盖"完全不同。

### 分流的实现层（架构推断）

> 以下内容基于 IL2CPP 运行时已知的方法调用机制和 DHE 的公开行为进行架构推断。

在 IL2CPP 运行时中，每个方法的 `MethodInfo` 结构体上有两个关键字段：

- `methodPointer`：指向正常调用时的函数实现
- `invoker_method`：指向反射调用时的函数实现

在社区版中，热更方法被加载后，这两个字段会被设置为指向解释器的入口（`Interpreter::Execute` 的某个包装）。在 AOT 方法中，它们指向 il2cpp 生成的原生函数。

DHE 的分流本质上就是在加载 dhao 文件后，对每个方法做一次判定：

- 如果 dhao 标记该方法为 unchanged：保持 methodPointer 指向原始 AOT 函数，不做任何修改
- 如果 dhao 标记该方法为 changed 或 new：将 methodPointer 替换为指向解释器入口，解释器执行的是最新热更 DLL 中的方法体

这就是为什么 DHE 能在同一个程序集内部实现函数级的执行路径分化：每个方法的 methodPointer 可以独立地指向 AOT 或解释器。

### 间接脏函数与注入检查

但上面的简单模型有一个问题：如果方法 A（AOT）调用了方法 B（已变化，走解释器），在 AOT 代码中方法 A 对 B 的调用是直接编码的函数地址跳转——它不会经过任何分流检查。

为了解决这个问题，DHE 在 il2cpp 代码生成阶段，会在 DHE 程序集中所有函数的开头注入一段检查代码。这段代码的逻辑大致是：

```
if (method->isInterpreterImpl) {
    // 跳转到解释器执行
    goto interpreter_entry;
}
// 继续执行原始 AOT 代码
```

这样，即使 AOT 代码通过直接地址跳转调用了一个已变化的方法，被调用方法开头的注入检查也会把执行流重定向到解释器。

这个设计解决了所谓的"间接脏函数"问题：方法 A 本身没变，但它调用的方法 B 变了。如果不注入检查，方法 A 会用旧的地址直接调用方法 B 的 AOT 实现，而不是最新的解释器版本。

### 注入的性能代价

官方文档对注入的性能影响有明确说明：

对于大多数方法，注入的 `if` 检查只增加一次条件判断，对整体性能影响可以忽略。

但对于短函数（例如 `int Age { get; set; }` 这种属性访问器），情况不同。短函数本身的代码量极少，注入的检查代码相对占比很高。更关键的是，注入代码可能阻止编译器对短函数做内联优化。官方数据显示，这种情况下性能退化可以达到 10% 甚至更多。

此外，注入代码会使 DHE 程序集生成的代码文件整体增大约 30%，对包体大小有不可忽视的影响。

### 函数注入策略

为了缓解短函数的性能问题，HybridCLR 提供了函数注入策略（Function Injection Strategy）配置。开发者可以把确定不会变化的短函数配置为"不注入"，这些函数就不会有开头的检查代码，编译器可以正常内联。

但这里有一个 trade-off：如果一个被标记为"不注入"的函数在后续版本中确实发生了变化，DHE 将无法正确地把它切换到解释器路径，因为调用它的代码不会经过分流检查。所以"不注入"的配置需要开发者自己保证这些函数在项目生命周期内确实不会被修改。

## 与热更主链的关系

理解 DHE 实现之后，需要明确一个容易混淆的区分：DHE 和普通热更主线的加载模型完全不同。

### 普通热更主线

普通热更主线的模型是：

1. 把新的程序集通过 `Assembly.Load` 动态加载
2. 这个程序集在运行时是一个独立的程序集实例
3. 它里面的所有方法都走解释执行
4. 需要补 metadata、桥接等

### DHE 模型

DHE 的模型是：

1. 程序集已经以 AOT 形式存在于包体中
2. 运行时不是"加载一个新程序集"，而是"在已有程序集上做差分覆盖"
3. 同一个程序集内部，不同方法可以走不同的执行路径
4. 未变化的方法继续用原生 AOT 代码，不经过解释器

这意味着 DHE 处理的 MethodInfo 是包体中已有的 MethodInfo，而不是新创建的。分流操作是对已有 MethodInfo 的 methodPointer 做选择性替换，而不是创建新的方法入口。

这也解释了为什么 DHE 程序集不能用 `Assembly.Load`：它需要的不是"装载新程序集"的语义，而是"修改现有程序集中部分方法的执行路径"的语义。

## DHE 的工程边界

DHE 的能力边界不只是"能不能改代码"，还涉及什么样的改动能被正确处理。

### 能稳定处理的变更

- **方法体修改**：这是 DHE 最核心的场景。只要方法签名不变，修改方法内部的逻辑，dhao 文件就能正确标记变化，运行时切换到解释器执行最新逻辑。
- **新增方法**：新 DLL 中新增的方法会被标记为 new，走解释执行。
- **新增类型**：完全新增的类型，其所有方法都走解释执行。

### 需要谨慎处理的变更

- **值类型的字段变更**：给 struct 增加或删除字段会改变内存布局。所有直接操作这个 struct 的 AOT 代码，如果还按旧的内存布局来访问，就会出问题。这类变更可能需要把相关方法也标记为 changed。
- **修改类型继承关系**：改变一个类型的基类或接口实现，影响范围可能远超单个方法。虚方法表的布局会变化，所有通过虚调用访问该类型的代码都可能受影响。

### 已知的限制

HCLR-11 已经列过一部分限制，这里从实现角度补充说明：

- **DHE 程序集代码不能在加载 DHE 热更代码之前执行**：因为加载 dhao 和替换 methodPointer 必须在第一次调用 DHE 方法之前完成。如果某个 DHE 方法在加载之前就被调用了，它会执行包体中的旧 AOT 代码，且无法被后续的 dhao 加载覆盖。
- **不支持 `[InitializeOnLoadMethod]` 和 Script Execution Order**：这些机制会在引擎初始化阶段就触发代码执行，此时 DHE 还没有完成加载。
- **DHE 脚本不能挂到打进包体的资源上**：包括 `Resources` 目录下的资源。因为这些资源的反序列化可能发生在 DHE 加载之前。
- **不能通过热更给 DHE 程序集新增 `extern` 函数**：extern 函数需要链接到原生库，这在解释器中无法实现。

## DHAO 工作流与 MetaVersion 工作流

DHAO 工作流是 DHE 最早的工作流，在 v7.6.0 之前是唯一的选择。但它有一个管理上的痛点。

### DHAO 工作流的局限

dhao 文件是通过比对"打包时的 AOT DLL"和"最新热更 DLL"生成的。这意味着：如果你正式上线后发了多个主包版本（比如 1.0、1.1、1.2 三个版本，每个版本的 AOT DLL 都不同），那么每次发热更新时，你需要为每个主包版本分别生成对应的 dhao 文件。

当主包版本很多时，这个管理成本会迅速上升。

### MetaVersion 工作流

从 v7.7.0 开始，HybridCLR 引入了 MetaVersion 工作流来解决这个问题。

MetaVersion 的核心思路是：给 DLL 中的每个类型和函数分配一个唯一的版本号。运行时通过比较版本号来判定变化，而不是依赖与特定 AOT Snapshot 的逐条比对。

这意味着一份 MetaVersion 文件可以同时适配多个主包版本，不再需要为每个主包单独生成差分文件。

对应的加载 API 是 `LoadDifferentialHybridAssemblyWithMetaVersion`。

如果你使用的是 v7.7.0 或更高版本，官方推荐使用 MetaVersion 工作流。

## 自实现思路

对于想更深入理解 DHE 的读者，可以从这个角度思考：如果你要自己实现一个最小的 DHE 系统，需要哪些组件？

**DLL 差分工具**：读取两个 .NET DLL（旧版和新版），建立类型和方法的映射关系，逐方法比对 IL 方法体，输出变化集。这是构建期工具，可以用 Mono.Cecil 或 System.Reflection.Metadata 来解析 DLL。

**dhao 生成器**：把差分工具的输出序列化成一个运行时可以高效读取的二进制格式。格式设计需要考虑：方法索引的查找效率（可能需要 hash 或排序后二分查找）、文件大小（变化集通常很稀疏，可以只记录已变化的方法）。

**运行时加载器**：读取 dhao 文件，解析出变化集，然后遍历 DHE 程序集中的所有 MethodInfo，对标记为 changed 的方法替换 methodPointer。这需要对 IL2CPP 运行时的 MethodInfo 结构有深入了解。

**函数注入（可选但推荐）**：修改 il2cpp 代码生成器，在 DHE 函数开头注入分流检查。没有这个注入，AOT 代码内部的直接函数调用将无法被正确重定向。

这四个组件中，前两个是构建期工具，后两个需要修改 IL2CPP 运行时。DHE 之所以是商业版能力，很大程度上是因为后两个组件需要对 IL2CPP 代码生成器和运行时做深度修改。

## 收束

DHE 的内部机制可以用三句话收住：

1. 构建期：把打包时的 AOT DLL 和最新热更 DLL 做逐条 IL 比对，把每个方法标记为 unchanged/changed/new，结果保存为 dhao 文件。
2. 运行时：加载 dhao 文件后，对 changed 和 new 方法替换 methodPointer 指向解释器入口，unchanged 方法保持原始 AOT 函数指针不变。
3. 安全网：在所有 DHE 函数开头注入 `isInterpreterImpl` 检查，确保即使 AOT 代码直接调用已变化的方法，也能正确跳转到解释器。

整条链路的设计哲学是：把差分计算的成本推到构建期，运行时只做查表和指针替换——这正是 dhao 文件存在的意义。
