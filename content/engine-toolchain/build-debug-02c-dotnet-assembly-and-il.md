> C# 编译器的输出不是机器码。你写的 C# 代码，先变成一种中间语言（`IL`），打包进程序集（`.dll`），再由运行时在合适的时机把 `IL` 转成真正能跑的机器码。理解这一层，是读懂 Unity 编译管线的前提。

## 这篇要回答什么

1. `IL` 是什么，C# 编译器到底输出了什么东西？
2. `.NET` 程序集（Assembly）是一个什么结构的容器，里面装了什么？
3. `JIT` 和 `AOT` 分别如何把 `IL` 变成机器码，各有什么代价？
4. Debug / Release 配置在这张地图上分别动的是哪一层？

---

## IL 是什么

C# 编译器（Roslyn）读取你的 `.cs` 源文件，输出的**不是**目标平台的机器码，而是一种平台无关的字节码：**IL**（Intermediate Language，也叫 MSIL 或 CIL）。

IL 是一种基于栈的虚拟指令集。写一个最简单的加法方法：

```csharp
public static int Add(int a, int b)
{
    return a + b;
}
```

对应的 IL 大致如下（简化版，用 `ildasm` / `dnSpy` 可以看到真实版本）：

```
.method public static int32 Add(int32 a, int32 b)
{
    ldarg.0       // 把参数 a 压栈
    ldarg.1       // 把参数 b 压栈
    add           // 弹出两个值相加，结果压栈
    ret           // 返回栈顶值
}
```

IL 指令操作的是一个虚拟栈，没有"寄存器"的概念，也不知道自己将来跑在 x86、ARM 还是别的架构上。

**IL 这一层的好处：**

- 平台无关：同一份 `IL` 可以在 Windows / macOS / Linux / iOS / Android 上运行，由各平台的运行时负责最后一步翻译。
- 可以被运行时做自适应优化：运行时知道实际硬件特性，`JIT` 可以在翻译时利用这些信息。
- 工具链友好：反编译、静态分析、代码注入（ILPP）都在这一层操作，比直接操作机器码容易得多。

## .NET 程序集（Assembly）是什么

C# 编译器最终产出的文件是 `.dll`（或 `.exe`）。这个文件不是一段裸的 `IL` 字节流，而是一个**结构化容器**，称为 **.NET 程序集（Assembly）**。

程序集内部的主要组成部分：

| 区域 | 内容 | 作用 |
|---|---|---|
| IL 代码段 | 所有方法的 `IL` 字节码 | 运行时执行的实际逻辑 |
| 元数据（Metadata） | 类型定义、方法签名、字段、特性、接口实现表…… | 反射、类型加载、跨程序集引用的基础 |
| Manifest | 程序集名称、版本、依赖的其他程序集列表 | 运行时定位和加载依赖 |
| 资源（可选） | 嵌入的图片、字符串表等 | 随程序集分发的静态资源 |

元数据和 `IL` 代码是分开存放的。这就是为什么你可以用反射在运行时枚举所有类型、方法——元数据表完整地描述了程序集的结构，不需要执行任何 `IL`。

### 为什么 Unity 项目里有那么多 .dll

Unity 项目里的 `Assembly Definition`（`.asmdef`）文件，每一个都对应一个独立的 `.dll`。Unity 按照 `asmdef` 的依赖关系分批编译，每批输出一个程序集。

这样做的好处是增量编译粒度更细：改了某个 `asmdef` 范围内的代码，只需要重新编译这个程序集，不必从头编译整个项目。

项目里的 `Assembly-CSharp.dll`、`UnityEngine.dll`、`DOTween.dll`……本质上都是同一种东西：装着 `IL` 和元数据的 `.NET` 程序集。

## JIT 路径

**JIT（Just-In-Time）** 是运行时（CLR / Mono）的默认工作方式：

1. 运行时加载程序集，读取元数据，但暂时不翻译 `IL`。
2. 某个方法**第一次被调用**时，JIT 编译器把这个方法的 `IL` 翻译成当前硬件平台的机器码，缓存起来。
3. 之后再调用这个方法，直接执行已编译的机器码。

```
程序集（.dll）
    └── IL
         └── [第一次调用时] JIT 编译
                  └── 机器码（缓存在内存中，进程退出即消失）
```

**JIT 的优点：**

- 运行时能感知实际 CPU 特性（指令集扩展、缓存大小），做出比静态编译更激进的优化。
- 可以做内联、去虚化等依赖运行时类型信息的优化。

**JIT 的缺点：**

- 每个方法第一次调用有延迟（编译开销）。
- 程序启动阶段如果大量方法被首次触发，会有明显的"冷启动"卡顿。
- JIT 编译器本身需要在内存中常驻，占用额外资源。

---

## AOT 路径

**AOT（Ahead-Of-Time）** 在程序**运行之前**就把 `IL` 编译成目标平台的机器码，产出原生二进制文件（`.so`、`.dylib`、可执行文件等）。

```
程序集（.dll）
    └── IL
         └── [构建阶段] AOT 编译器
                  └── 原生机器码（.so / .dylib / 可执行文件）
```

**为什么移动平台和主机平台常用 AOT：**

| 原因 | 说明 |
|---|---|
| App Store 政策 | iOS 不允许在运行时生成可执行代码，`JIT` 的本质就是在内存里生成机器码，违反沙盒规则 |
| 启动性能 | 所有代码已经是机器码，没有 `JIT` 冷启动开销 |
| 包体可预测 | 最终分发的是原生代码，不需要运行时 `JIT` 编译器 |
| 主机平台限制 | 部分主机平台同样禁止运行时代码生成 |

`AOT` 之后，程序和普通的原生 C++ 程序没有本质区别：符号表、调试信息（`DWARF`/`PDB`）、编译器优化级别（`-O0` 到 `-O3`）这些原生层的概念都变得重要。这也是为什么调试 `IL2CPP` 打出来的包，需要同时关心托管层符号（`.pdb`）和原生层符号（`libil2cpp.so` 的符号）。

## Debug / Release 在这张地图上落在哪

这一层并不是一个开关，而是三个不同层面的配置叠加。

| 层 | Debug 配置 | Release 配置 |
|---|---|---|
| IL 层（Roslyn 编译器） | 保留 `nop` 指令作为断点占位；生成完整序列点（source → IL 偏移映射）；不做内联 | 省略 `nop`；序列点稀疏；允许内联 |
| JIT 层（运行时） | JIT 不做激进优化，变量不被优化掉，栈帧完整 | JIT 激进优化，变量可能被寄存器化或消除，调试器看到的值可能不直观 |
| AOT 层（原生编译器） | 相当于 `-O0`，保留所有调试信息 | 相当于 `-O2`/`-O3`，大量内联、循环展开、死代码消除 |

调试器能不能在某一行打断点、能不能看到局部变量的正确值，取决于这三层配置是否都对齐到"调试友好"状态。只改其中一层，可能出现"断点能停但看不到变量"或"变量能看但断点位置偏移"的情况。

## 这张地图和 Unity 的关系

Unity 项目的编译路径，前半段就是本篇讲的内容：

```
你写的 .cs 源码
    → [Roslyn] →
程序集（.dll），里面装着 IL + 元数据
```

但 Unity 在这条路上还加了两个节点：

1. **ILPP（IL Post Processing）**：在程序集生成后、运行之前，Unity 或第三方工具（如 Burst、HybridCLR）可以读取程序集、修改 `IL`、再写回。这一步发生在 `IL` 层。

2. **运行时选择（Mono vs IL2CPP）**：
   - 选 **Mono**：走 `JIT` 路径，程序集原样发布，运行时解释/JIT 执行。
   - 选 **IL2CPP**：走 `AOT` 路径，Unity 用 `IL2CPP` 工具把 `IL` 转成 C++ 源码，再用目标平台的原生编译器编译成机器码。

这两个节点是后续系列的主题，本篇不展开。本篇的任务只是把 `IL → 程序集 → JIT/AOT → 机器码` 这张基础地图立起来。

---

## 小结

1. C# 编译器（Roslyn）输出的是 `IL`（中间语言），不是机器码；`IL` 是平台无关的基于栈的字节码。
2. `.NET` 程序集（`.dll`）是一个结构化容器，包含 `IL` 代码段、元数据（类型/方法信息）和 Manifest；元数据是反射和跨程序集引用的基础。
3. Unity 项目里每个 `Assembly Definition` 对应一个 `.dll`，拆分程序集的核心目的是细化增量编译粒度。
4. `JIT` 在运行时按需编译 `IL`，可做自适应优化，但有冷启动开销；`AOT` 在构建阶段预先编译，消除运行时编译开销，是移动/主机平台的常见选择。
5. Debug / Release 不是一个开关，而是 IL 层（序列点/nop）、JIT 层（优化激进程度）、AOT 层（原生编译器优化级别）三层配置的叠加。
