> 热更 DLL 一旦被抓包拿到，如果没有加密，用 dnSpy 打开就是完整的 C# 源码。加密要解决的不是"能不能混淆"，而是"就算 DLL 被截获，攻击者拿到的也只是一段无法直接解析的二进制流"。

这是 HybridCLR 系列第 29 篇。

前面的文章已经把 runtime 主链、AOT 泛型、工具链、性能优化和指令层都拆过了。但到目前为止，有一个问题一直没正面回答：

`热更 DLL 作为标准 PE 格式文件下发到客户端，它的安全性到底怎么保证？`

这个问题在社区版主线里不存在——社区版的 `Assembly::LoadFromBytes` 直接解析标准 metadata 流，没有加密层。但一旦进入商业版，加密就成了一个独立的工程层面。

这篇文章把加密和访问控制这两件事拆开讲清楚。

> 本文加密体系描述基于 HybridCLR 商业版官方公开文档。社区版源码分析基于 v6.x（main 分支，2024-2025）。商业版 runtime 实现细节不公开，本文不做猜测。

## 热更 DLL 在未加密状态下有多容易被反编译

先把威胁模型立住。

热更 DLL 是标准的 .NET PE 文件。它的 metadata 结构完全遵循 ECMA-335 规范：`#Strings` 流存类型名和方法名，`#Blob` 流存方法签名，`#US` 流存用户字符串字面量，`#~` 表存类型定义和方法定义，method body 区存完整的 CIL 指令。

这意味着：

- 用 dnSpy 或 ILSpy 打开，可以直接看到完整的类型结构、方法签名和 CIL 反编译后的 C# 代码
- 方法体内的逻辑——包括算法、协议解析、数值公式——全部可读
- 字符串字面量——包括 URL、密钥前缀、配置路径——直接暴露在 `#US` 流里

对于单机游戏，这可能只是"不太好"。但对于竞技类、内购类、平台类游戏，这就是一个实际的安全缺口：外挂开发者可以直接读到伤害计算、协议校验、道具逻辑。

所以加密要做的不是"让代码更难读"，而是让 DLL 文件在离开可信环境之后，不再是一个有效的 PE 文件。

## 加密的 7 层体系

HybridCLR 商业版提供的不是单一加密方案，而是一套分层递进的体系。每一层解决不同等级的威胁，同时带来不同的性能和工程代价。

| 层级 | 名称 | 安全等级 | 核心机制 |
|------|------|----------|----------|
| 1 | global-metadata.dat 加密 | 1 星 | 对 Unity 的全局元数据文件做整体加密，防止直接用 Il2CppDumper 等工具提取 AOT 部分的类型信息 |
| 2 | Metadata 混淆 | 2 星 | 裁剪或重命名类型名、字段名、函数名，让反编译输出丧失可读性 |
| 3 | Metadata 流加密 | 2-4 星 | 对 DLL 内的 `#Strings`、`#Blob`、`#US`、`#~`、method body 等流分别加密，粒度可控 |
| 4 | 结构虚拟化 | 3 星 | 每次构建随机化 metadata 的物理结构布局，由专属的"结构虚拟机"在运行时解析 |
| 5 | 加密虚拟化 | 3 星 | 每次构建随机化加密算法本身，由专属的"加密虚拟机"在运行时执行解密 |
| 6 | 延迟解密 | 3 星 | 数据不在加载时一次性全解，而是在首次访问时才解密，防止内存全量 dump |
| 7 | 指令虚拟化 | 5 星 | 将 IL 指令转换为自定义寄存器 VM 操作码，编码方式每次构建随机化 |

几个需要立住的判断：

**层级不是互斥的，而是叠加的。** 你可以只开第 1-3 层做基础保护，也可以全部叠满。每多开一层，逆向成本都在指数级增加。

**第 4、5 层的"虚拟化"是真正的门槛跳变。** 结构虚拟化意味着每次构建产出的 DLL 二进制布局都不同，通用的 metadata 解析工具直接失效。加密虚拟化意味着即使逆向出某一次构建的解密算法，下一次构建的算法已经变了。

**第 7 层指令虚拟化是最高级别。** 它不只是加密 metadata，而是把 IL 指令本身替换成一套自定义的寄存器 VM 指令集，编码方式随机生成。这意味着攻击者即使拿到解密后的 metadata，面对的也不再是标准 CIL，而是一套私有指令集。

## 运行时解密链路

先看社区版。打开社区版源码，`Assembly::LoadFromBytes` 的链路非常直白：

```
Assembly::LoadFromBytes
  → RawImage::Load (或 RawImageBase 子类)
    → 按 ECMA-335 标准解析 PE header
    → 读取 CLI header
    → 定位 metadata root
    → 逐个解析 #~、#Strings、#Blob、#US、#GUID 流
    → 建立 metadata 内存表
```

这条链路里没有任何加密相关的代码。`RawImage` 直接按标准偏移读字节、按标准结构解析表。这也是社区版必然如此的原因——它是开源的，加密逻辑放在开源代码里等于没有加密。

商业版替换的正是这一层。当 runtime 是商业版时，`RawImageBase` 的实际子类不再是标准解析器，而是一个包含解密能力的版本。具体行为：

- 加密后的 DLL 字节流传入后，native C++ 层的解密模块先介入
- 解密所需的密钥和算法已经在构建时编译进了 native binary（`GameAssembly.dll` / `libil2cpp.so`）
- 解密完成后，后续的 metadata 解析、transform、解释执行链路和社区版一致

**C# 侧的调用方式完全不变。** 业务代码依然是 `Assembly.Load(encryptedBytes)`，不需要自己做任何解密操作。也没有提供用户空间的 hook 点来插入自定义解密逻辑——这是刻意的设计，因为如果解密逻辑暴露在 C# 层，它自己就成了可被反编译的目标。

## vmSeed 为什么必须与主包一致

商业版加密涉及几个关键参数，它们的作用域和更新策略各不相同：

| 参数 | 作用 | 编译位置 | 能否热更时变更 |
|------|------|----------|----------------|
| `vmSeed` | 控制结构虚拟化和加密虚拟化的随机种子 | 编译进 native binary | 不能，必须与主包一致 |
| `metadataSeed` / `metadataKey` | 控制 metadata 流加密的密钥 | 随加密 DLL 一起下发 | 可以，每次热更可轮换 |
| `xxEncCodeLength` | 控制加密代码段的长度 | 构建时配置 | 安全性与性能的权衡旋钮 |

`vmSeed` 是最容易出问题的参数。

它在构建主包时被编译进 `GameAssembly.dll`（或 `libil2cpp.so`），成为 native binary 的一部分。运行时的结构虚拟机和加密虚拟机依赖这个种子来确定"当前这次构建用的是哪套随机结构和哪套随机加密算法"。

如果热更 DLL 用的加密种子和主包 native binary 里编译进去的种子不一致，解密虚拟机根本无法正确解读 DLL 的物理结构。结果不是"解密失败报错"，而是解析出垃圾数据，后续 metadata 加载直接崩溃。

所以 `vmSeed` 的管理规则很简单：**它跟着主包走，不跟着热更走。** 如果你要换 `vmSeed`，必须发一个新的主包版本。

`metadataSeed` / `metadataKey` 则灵活得多。它可以每次热更时轮换，因为解密所需的信息跟着加密后的 DLL 一起下发。这给安全运营留了一个旋钮：即使某一次的 `metadataKey` 被逆向出来，下一次热更直接换掉就行。

`xxEncCodeLength` 是安全性和性能之间的权衡。更长的加密代码段意味着更高的安全性，但也意味着解密时间增加。对于不同敏感度的程序集，可以配置不同的长度。

## 延迟解密：为什么不是一次性全解

如果加密 DLL 在 `Assembly.Load` 时一次性全部解密到内存，那攻击者只需要在加载完成后做一次内存 dump，就能拿到完整的明文 metadata。加密做得再强，也只能防"静态分析"，防不了"运行时 dump"。

延迟解密解决的就是这个问题。

它的核心行为是：DLL 加载时只做最小限度的结构解析（够建立 metadata 索引即可），具体的流数据——类型名、方法签名、方法体、字符串字面量——在第一次被真正访问时才触发解密。

这意味着：

- 在任意时刻做内存 dump，拿到的只是"截至目前已经被访问过的那部分"的明文，加上大量仍然处于加密状态的数据
- 没有一个时间点是"所有 metadata 都以明文形式同时存在于内存中"
- 攻击者想拿到完整明文，必须触发所有类型和方法的首次访问——这在实际游戏运行中很难做到，因为大量代码路径在正常游戏流程中根本不会被执行

延迟解密和第 4、5 层的虚拟化叠加后，防 dump 效果进一步增强：即使攻击者设法触发了大量方法的首次访问，解密后的数据在内存中的布局也是虚拟化后的私有结构，不是标准 ECMA-335 格式，通用工具依然无法直接解析。

## 加密在构建链的哪一步

加密不是一个独立的离线操作，而是嵌在 HybridCLR 构建管线里的一个环节。它的位置很明确：

```
CompileDll (Roslyn 编译热更程序集)
  ↓
产出标准 .dll 文件
  ↓
EncryptionUtil.EncryptDll()        ← 加密在这里
  或
BuildUtils.EncryptDllAndGenerateDHAODatas()  ← 如果同时用 DHE
  ↓
产出加密后的 .dll 文件
  ↓
部署到 CDN / 资源服务器
```

几个工程上需要注意的点：

**加密必须在 CompileDll 之后。** 因为加密操作的输入是标准 DLL，输出是加密后的 DLL。如果你先加密再编译，编译器会报格式错误。

**加密必须在部署之前。** 这看起来是废话，但在实际 CI 流程中，很容易因为 pipeline 步骤顺序错误导致下发了未加密的 DLL。

**如果同时使用 DHE，加密和 DHAO 数据生成是一个联合步骤。** `BuildUtils.EncryptDllAndGenerateDHAODatas()` 同时完成加密和差分数据计算，不能拆成两个独立步骤分别执行。这是因为 DHAO 的差分计算依赖加密前的原始 DLL 内容，但产出物是加密后的 DLL 加上 DHAO 数据文件。

**CI 流程建议。** 把加密步骤做成 CI pipeline 的显式阶段，加密完成后做一次校验：用商业版 runtime 的验证工具确认加密后的 DLL 能被正确加载。不要等到线上才发现加密参数不匹配。

## 访问控制策略

加密解决的是"DLL 被截获后能不能读"。但还有一类问题加密不覆盖：

`即使 DLL 能正常加载和执行，某些类或方法是否应该被特定的调用者访问？`

这是访问控制要解决的问题。它属于 HybridCLR Ultimate Edition 的独占功能。

### 使用场景

访问控制最直接的场景是 UGC（User Generated Content）和平台型游戏。

在这类游戏中，第三方开发者或玩家创作的代码以热更 DLL 的形式加载到 runtime 中。平台方需要一种机制来限制这些第三方代码能访问哪些系统 API——比如允许调用游戏逻辑 API，但禁止直接访问文件系统、网络或反射。

### 策略定义

访问控制策略通过 XML 定义，在构建时编译为二进制格式：

```xml
<AccessPolicy>
  <Assembly name="ThirdPartyMod">
    <AllowClass name="GameSDK.ItemAPI" />
    <AllowClass name="GameSDK.PlayerAPI" />
    <DenyClass name="System.IO.*" />
    <DenyClass name="System.Net.*" />
    <DenyClass name="System.Reflection.*" />
  </Assembly>
</AccessPolicy>
```

粒度是类级别。可以对特定程序集声明"允许访问哪些类"和"禁止访问哪些类"。

### 运行时行为

策略在运行时通过 `RuntimeApi.LoadAccessPolicy()` 加载。加载后，检查发生在方法首次调用时——和 transform 阶段在同一个入口：

```
方法首次被调用
  → transform 阶段
    → 检查当前调用者程序集是否命中访问控制策略
    → 如果命中且调用目标类被禁止：抛出异常，阻止执行
    → 如果未命中或允许：正常继续 transform 和解释执行
```

这个设计有一个重要特征：**检查只发生在首次调用。** 一旦某个方法通过了访问控制检查并完成 transform，后续的重复调用不会再次检查。这意味着访问控制的性能开销集中在冷启动路径上，不影响热路径的执行效率。

### 与加密的关系

访问控制和加密是两个正交的能力：

- 加密防的是"DLL 被截获后能被读懂"
- 访问控制防的是"DLL 能正常运行，但不应该碰某些 API"

在 UGC 场景中，通常两者都需要：加密保护平台方自己的核心代码，访问控制限制第三方代码的能力边界。

## 加密与 Obfuz 的互补关系

HybridCLR 的加密体系工作在 metadata 流和结构层面——它改的是 DLL 文件的物理格式，让标准工具无法解析。

Obfuz 是 HybridCLR 生态中的另一个伴生工具，它工作在 IL 指令层面。两者的关系不是替代，而是互补。

| 维度 | HybridCLR 加密 | Obfuz |
|------|----------------|-------|
| 工作层面 | metadata 流、物理结构、加密算法 | IL 指令序列 |
| 核心手段 | 流加密、结构虚拟化、加密虚拟化 | 控制流混淆、常量加密、方法内联/拆分 |
| 防护目标 | 让 DLL 文件无法被标准工具打开 | 让反编译后的代码逻辑难以理解 |
| 生效阶段 | runtime 加载时由 native 层解密 | 编译后对 IL 做变换，和 runtime 无关 |

如果只用 Obfuz 不用 HybridCLR 加密：DLL 文件仍然是标准 PE 格式，dnSpy 能打开。虽然反编译后的代码可读性很差，但有经验的逆向工程师仍然可以通过模式识别和符号分析恢复逻辑。

如果只用 HybridCLR 加密不用 Obfuz：标准工具打不开 DLL，但如果攻击者设法在内存中 dump 到解密后的 metadata（比如延迟解密未覆盖的部分），拿到的就是清晰可读的 IL 指令。

两者叠加的效果是：文件级打不开，即使内存 dump 到部分明文，IL 指令本身也是混淆过的。

在构建管线中的顺序是：**先 Obfuz（IL 层混淆），再 HybridCLR 加密（metadata 流加密）。** 因为 Obfuz 需要操作标准的 IL 指令，加密后的 DLL 它无法处理。

## 工程判断

把加密和访问控制放回项目决策里，需要回答三个问题。

### 什么时候需要加密

**需要的场景：**

- 游戏有实际的外挂或私服威胁，热更代码包含核心战斗逻辑、协议处理、数值计算
- 游戏有内购系统，热更代码包含道具验证、支付回调处理
- 合规要求代码保护（某些地区的上线审核要求）

**不一定需要的场景：**

- 纯单机、无内购、无竞技的游戏，逆向收益低于逆向成本
- 热更代码只包含 UI 逻辑和配置读取，核心逻辑全在 AOT 层
- 项目还在早期验证阶段，安全问题不是当前优先级

### 什么时候需要访问控制

访问控制的场景比加密窄得多。只有当你的 runtime 会加载不完全受你控制的第三方代码时，才需要考虑：

- UGC 平台：玩家或第三方开发者提交的 mod/插件
- 小游戏平台：第三方小游戏以 DLL 形式运行在你的宿主环境中
- 企业内部多团队共用一个 runtime 基座，需要隔离不同团队代码的系统访问权限

如果你的热更 DLL 全部由自己团队开发和部署，访问控制通常是多余的。

### 加密层级怎么选

不是每个项目都需要开到 7 层满。一个实用的判断框架：

- **基础保护（1-3 层）：** 适合大多数商业项目。global-metadata 加密 + metadata 混淆 + 流加密，已经让通用工具完全失效，逆向成本显著提升。
- **高安全（4-5 层）：** 适合有定向逆向威胁的项目。虚拟化让每次构建的二进制布局都不同，逆向某一次构建的成果无法复用到下一次。
- **最高安全（6-7 层）：** 适合竞技类头部项目或有强合规要求的场景。延迟解密 + 指令虚拟化的组合防护，逆向成本接近从零实现一个自定义 VM 解释器。

层级越高，需要关注的工程代价也越高：

- `xxEncCodeLength` 需要调优，过长会影响加载性能
- 指令虚拟化会引入额外的执行开销（自定义 VM 解释），对性能敏感的热路径需要评估
- 测试矩阵变大，需要确认加密后的 DLL 在所有目标平台上都能正确加载和执行

## 收束

HybridCLR 的加密不是一个简单的"开关"，而是一套从 metadata 流到物理结构到指令集的分层体系。社区版完全没有加密能力——它的 `RawImage` 直接按 ECMA-335 标准解析字节流。商业版在 native C++ 层替换了 metadata 解析入口，让解密对 C# 业务代码完全透明。

`vmSeed` 跟着主包走，`metadataKey` 可以随热更轮换。延迟解密让内存 dump 拿不到完整明文。访问控制在首次调用的 transform 入口拦截，粒度是类级别。Obfuz 做 IL 层混淆，和 HybridCLR 的流级加密正交互补。

加密层级的选择不是越高越好，而是要匹配项目的实际威胁模型和性能预算。
