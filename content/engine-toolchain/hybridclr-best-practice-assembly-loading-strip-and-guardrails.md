> 真正让 HybridCLR 在项目里长期稳定的，不是“把原理看懂了”，而是把程序集边界、加载顺序、生成物一致性和问题诊断顺序，变成团队共同遵守的硬约束。

这是 HybridCLR 系列第 7 篇，也是第一轮落地篇。  
前面几篇已经把主链、AOT 泛型、工具链、MonoBehaviour 资源挂载和边界拆开了；这一篇不再追单条源码，而是把这些认识收回项目层，回答一个更现实的问题：

`如果团队决定长期用 HybridCLR，工程上到底该怎么组织，才能不把它用成一套脆弱流程？`

## 这篇要回答什么

这篇主要回答 5 个问题：

1. 热更程序集和 AOT 程序集应该怎么拆，哪些边界最不能碰。
2. `LoadMetadataForAOTAssembly`、热更 DLL 加载、资源世界启动，这三件事的顺序该怎么固化。
3. `CompileDll / Generate/All / Copy AOT & HotUpdate DLL` 这些步骤怎么落成稳定流程，而不是靠人记。
4. `MonoBehaviour` 资源挂载、AOT 泛型、MethodBridge 这些高频坑，应该怎么前移风险。
5. 线上或测试环境真出问题时，应该按什么顺序诊断，而不是一上来就怀疑 “HybridCLR 有 bug”。

## 先给一句总判断

如果先把这篇压成一句话，我的判断是：

`HybridCLR 不是“加一个热更新包”这么简单，它更像是给项目引入了一套新的运行时契约；最佳实践的核心，不是多背几个菜单名，而是让这套契约在构建期、启动期和诊断期都被明确执行。`

也就是说，真正稳定的项目，靠的不是“某个同学很懂 HybridCLR”，而是：

- 边界先设计好
- 顺序先固定好
- 风险先显式化
- 回归检查先自动化

## 第一条：先把程序集边界定死，尤其别让 AOT 反向引用热更

我觉得最重要的一条 best practice，不是加载顺序，而是程序集拆分。

因为只要边界先拆错，后面很多问题都会被放大：

- 热更 DLL 频繁向底层扩散
- AOT 程序集开始反向依赖热更程序集
- 泛型实例和桥接签名越来越不可控
- 资源挂载、程序集身份和工具链产物变得更难稳定

### 最稳的拆分原则

如果只保留一条硬规则，我会写成：

`AOT 做底座，HotUpdate 做变化层；依赖方向只能从热更层指向 AOT 层，不能反过来。`

更具体一点：

- 基础框架、第三方库封装、平台桥接、资源系统、存档、网络底层，更适合留在 AOT。
- 游戏玩法、活动、任务、UI 组织、剧情流程、配置驱动逻辑，更适合放进 HotUpdate。
- 频繁改动但强依赖 Unity 原生工作流的业务脚本，可以放进热更程序集。
- 高频数值核心循环、重度跨 native 边界逻辑、性能极敏感模块，不要默认扔进热更。

### 最该防的不是“热更依赖 AOT”，而是“AOT 反依赖热更”

热更依赖 AOT，本来就是设计目标。  
真正危险的是 AOT 反向引用热更程序集。

因为一旦出现这种依赖关系，你就会同时失去几样东西：

- 边界不再清晰
- 构建图会变脆
- 热更程序集变成底层契约的一部分
- 调试时很难判断问题到底属于程序集装载、metadata、还是普通引用耦合

你这个工程里其实已经有一个很好的团队级做法：`AOTHotUpdateRefChecker.cs`。  
它会扫描非热更 DLL 是否引用热更程序集，并导出报告。

我很建议把这一类检查当成 best practice，而不是“项目私有脚本”：

`凡是 AOT 反向引用热更，都不要靠 code review 肉眼抓，直接做成自动检查。`

## 第二条：把加载顺序写成流程，不要靠人记

HybridCLR 第二个最容易出错的地方，是加载顺序。

很多项目第一次接起来能跑，靠的是“当前这套代码顺序刚好没出事”；  
但真正稳定的项目，不该让顺序靠隐含约定。

我更推荐把顺序压成固定的 4 步：

1. 先准备 AOT 补充 metadata
2. 再加载热更程序集
3. 再进入资源和业务世界
4. 最后才让资源反序列化和游戏主入口全面展开

### 为什么这个顺序不能反

前面几篇其实已经把原因讲过了，这里只收工程判断：

- `LoadMetadataForAOTAssembly` 不是加载热更 DLL，它是在给 AOT 世界补可查询 metadata。
- `Assembly.Load(bytes)` 才是在把热更程序集接进 runtime。
- `MonoBehaviour` 资源挂载依赖的是程序集身份链和真实程序集对象已经接上。
- 一旦资源世界先启动，Prefab/Scene/AssetBundle 里的脚本引用可能会比你想象得更早触发。

所以顺序如果反了，最坏情况不是“多一次首帧卡顿”，而是：

- 资源反序列化先发生
- 程序集身份链还没闭合
- AOT metadata 还没补齐
- 结果你看到的是一个表象非常乱的错误现场

### 你这个工程里的顺序其实就很接近正确答案

`ProcedureLoadAssembly.cs` 里已经体现了比较稳的顺序：

- 先处理 `LoadMetadataForAOTAssembly`
- 再逐个加载热更 `TextAsset` 并 `Assembly.Load`
- 等 metadata 和热更程序集都完成后，再 `Invoke` 主入口

这个设计的价值不在于“代码写得优雅”，而在于：

`它把“metadata 补充完成”和“热更程序集装载完成”明确变成了进入主业务世界前的两道门槛。`

我很建议把这类门槛保留下来，而不是后面为了“简化流程”又把三件事揉成一坨异步逻辑。

## 第三条：把工具链固定成标准产物链，不要让生成物靠口口相传

只讲 runtime 是不够的。  
HybridCLR 的稳定性，很大一部分其实来自 build-time。

最坏的用法是：

- 某个同学记得点过 `Generate/All`
- 某个同学手工拷过 DLL
- 某次发包时 `Development Build` 和生成物不一致
- 最后线上只留下一个“为什么这次 MethodBridge 不对”的谜题

这类问题不该靠经验记忆解决。

### 最稳的做法：把工具链看成产物流水线

我建议团队内部只传播这一条顺序：

`CompileDll -> Il2CppDef -> LinkXml -> AOTDlls -> MethodBridge -> AOTGenericReference`

这不是“推荐菜单顺序”，而是运行时产物依赖顺序。

一旦决定上 HybridCLR，这条链就应该被明确成：

- 标准本地流程
- 标准构建机流程
- 标准发版前检查流程

### 你项目里已经有一个很实用的落地点

`BuildDLLCommand.cs` 里把两件工程上很重要的事收在一起了：

- `CompileDll`
- 把 HotUpdate DLL 和 AOT metadata DLL 拷到资源路径

这类封装的价值很现实：

`它不是替代官方工具链，而是把“你们项目真正需要交付的那组产物”固化成一个入口。`

我的建议是：

- 继续保留这种项目级封装
- 但不要隐藏掉它依赖的官方生成链
- 最好让 CI 也跑同一套入口，而不是让本地和打包机各走一套

### 生成物一致性必须当成契约，而不是建议

HybridCLR 最怕的不是生成物多，而是生成物和当前构建参数不一致。

尤其要盯紧这几类一致性：

- `BuildTarget`
- `Development Build`
- 裁剪条件
- 本地 `libil2cpp` 版本
- 热更 DLL 输出目录和最终资源目录

一旦这里错位，症状经常不是“直接报错提示很清楚”，而是 runtime 在另一个层上表现异常。

所以 best practice 不是“勤快一点多生成”，而是：

`每次能影响生成物语义的参数变了，就把全链重新生成。`

## 第四条：MonoBehaviour 资源挂载要用，但不要把它当成默认解

HybridCLR 很强的一点，是它确实能支持资源上直接挂热更 `MonoBehaviour`。  
但我不建议把这件事理解成：

`既然能挂，那所有热更逻辑都应该优先挂在资源上。`

这会把问题想简单。

### 更稳的工程判断

我更推荐这样用：

- 强依赖资源工作流、Prefab 组织、Scene 直观搭建的业务脚本，适合走资源挂载。
- 变动频率高、逻辑实验性强、生命周期很短的脚本，可以优先走代码动态加组件或更显式的业务装配。

原因很简单。  
资源挂载除了普通热更逻辑本身，还会额外依赖：

- `ScriptingAssemblies` 列表补丁
- placeholder assembly 身份链
- 真实热更 DLL 装载时对 placeholder 的回填

这条链不是不能用，而是它毕竟更重。

所以最稳的策略不是“能挂就挂”，而是：

`只把真正需要 Unity 资源工作流收益的脚本放进这条链。`

### 还有一个常被忽略的实践点

如果你的项目资源启动很早，那就要更克制。

因为一旦资源实例化发生在热更程序集真正装载之前，脚本身份链再漂亮也没有意义。  
所以：

- 热更主程序集和关键资源脚本程序集，要保证在资源世界全面展开前就绪
- 对启动期会立刻实例化的大资源块，最好避免让它们依赖“稍后才加载”的热更程序集

这不是 HybridCLR 特有的限制，而是资源反序列化链天生比业务逻辑更早、更隐蔽。

## 第五条：AOT 泛型不要赌，风险一定要前移

前面 AOT 泛型那篇已经把边界讲清楚了。  
落到 best practice，我只想保留一句更硬的判断：

`补充 metadata 是兜底能力，不是鼓励你对 AOT 泛型覆盖率掉以轻心。`

### 最稳的做法不是“出错了再补”，而是“先显式暴露风险”

我建议至少保留这几件事：

- 每次关键热更版本前都重新生成 `AOTGenericReference`
- 对核心泛型链路做一次人工审阅，不要把注释清单当成功能保证
- 对常见高风险泛型场景建立最小 smoke case

高风险场景通常包括：

- `async/await`
- 泛型值类型
- 委托和接口回调里的泛型实例
- 容器类型嵌套和值类型组合

### 对 “补 metadata 就完了” 保持警惕

项目里一旦出现下面这种症状：

- 缺的是具体实例化后的方法指针
- 报的是 `AOT generic method not instantiated in aot`
- 问题发生在跨 interpreter / AOT / native 边界时

那就不要再从“是不是 metadata 没补齐”开始想了。  
这时候更该看的，是：

- 具体泛型实例有没有被保进 AOT 世界
- `MethodBridge` 对应签名有没有产出
- 当前生成物是不是和构建参数一致

## 第六条：先把诊断顺序固定下来，别遇到问题就全链路乱翻

HybridCLR 真出问题时，最耗时间的通常不是修，而是找层。

所以我非常建议团队内部固定一个诊断顺序。  
我自己会按下面 5 层查：

### 1. 先查程序集边界和装载顺序

先问：

- 这是热更程序集，还是 AOT 程序集
- 是 `Assembly.Load` 没发生，还是发生太晚
- 主入口是不是在 metadata 和 DLL 都没就绪前就启动了

### 2. 再查资源身份链

如果是资源上挂脚本的问题，先问：

- `ScriptingAssemblies` 列表有没有补回
- placeholder assembly 名字对不对
- 真实 DLL 进来时有没有复用 placeholder

### 3. 再查 metadata 和泛型边界

如果是 transform、method body、泛型解析失败，先问：

- AOT metadata 有没有补
- `Consistent / SuperSet` 选得对不对
- 是“看不懂”还是“根本没实例”

### 4. 再查桥接和 ABI 边界

如果调用已经跨到 delegate、P/Invoke、reverse P/Invoke、function pointer，那就优先怀疑：

- `MethodBridge`
- `ReversePInvokeWrapper`
- 签名一致性

### 5. 最后再怀疑业务代码

很多时候问题当然也可能只是业务层自己写错。  
但如果前面 4 层没先分开，你很难高效证明“这次真的是业务 bug，不是 HybridCLR 链路问题”。

这条诊断顺序的价值就在于：

`它把一锅粥拆回几层可判断的问题。`

## 第七条：把团队协作防线做在“默认正确”，不是做在“高手兜底”

我很不喜欢一种团队状态：

- 只有一两个人知道 HybridCLR 全链怎么工作
- 其他人只知道“坏了找他们”
- 每次发版都靠口头确认“这些按钮你点了吧”

这不是技术深度高，这是系统脆。

### 更稳的团队做法

我建议至少把下面这些事做成默认流程：

- 有统一的 HybridCLR 构建入口，不让每个人自己拼步骤
- 有 AOT 反向引用热更检查，不靠 code review 碰运气
- 有最小 smoke case，覆盖热更装载、AOT metadata、资源挂载、关键泛型链路
- 有清晰的“哪些目录能改、哪些生成物不能手工编辑”的约定

### 哪些东西最适合进 CI

如果条件允许，我最建议自动化的不是“所有构建都完整跑一遍游戏”，而是先把这几件事自动化：

- 生成链是否完整执行
- AOT 是否反向引用热更
- 关键热更 DLL 和 AOT metadata DLL 是否进入最终资源目录
- 必要的 `MethodBridge` 和 `AOTGenericReference` 是否已刷新

这些检查不一定昂贵，但它们能提前挡住很多“明明是流程错了，却要等到 runtime 才爆”的问题。

## 最后压一句话

如果只允许我用一句话收这篇文章，我会写成：

`HybridCLR 的最佳实践，本质上不是“把热更新接起来”，而是把运行时依赖的那几条隐含前提显式化：程序集边界不能乱、加载顺序不能反、生成物必须一致、风险要前移、诊断要分层。`

这几条一旦立住，HybridCLR 在项目里就会更像一套可管理的工程系统，而不是一组偶尔有效的技巧。
