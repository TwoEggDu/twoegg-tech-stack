> 这组文章想先做一件事：把 `Unity DOTS`、`Unreal Mass` 和“自己手搓一套 ECS”放到同一张问题地图里。  
> 真正值得看的，不是它们各自有多少 API，而是为什么现代引擎会反复长出一块“数据导向孤岛”，以及如果你自己做，也很可能被同一批问题逼到类似结构。

很多人第一次接触这类系统时，会天然带着三种直觉：

- `ECS 就是更快的 OOP`
- `数据导向的核心就是并行`
- `上了这套系统，GameObject / Actor 这种老世界就该退场`

这三种直觉都不够准。

因为现代引擎做数据导向，真正要解决的不是“语法风格问题”，而是更硬的几类成本：

- 大量同构对象怎样低成本更新
- 内存布局怎样服务访问模式
- 结构变化怎样不把执行链打断
- 仿真层和表示层怎样避免互相污染
- 构建期怎样替运行期吃掉一部分复杂度

如果这些问题不先拆开，`DOTS` 会看起来像一组包名，`Mass` 会看起来像一个 Unreal 插件，自研 ECS 也会很容易滑成玩具框架。

## 这篇要回答什么

这篇总论主要回答 4 个问题：

1. 为什么现在值得把 `Unity DOTS`、`Unreal Mass` 和自研 ECS 放在一起看
2. Unity 和 Unreal 明明都在做数据导向，为什么最后长成了两种很不一样的系统
3. 为什么自己手搓一版，反而最容易看清这类系统的真正不变量
4. 这组文章后面准备沿着什么顺序往下拆

## 先把问题压成一张图

如果把这类系统面对的问题先压成一张最小地图，我会这样记：

`高规模对象 -> 数据布局 -> 查询 -> 调度 -> 结构变化 -> 表示层 -> 构建期转换`

这条线里，每一层都不是可有可无的“实现细节”。

它们分别在回答不同问题：

- `数据布局` 回答的是“这些对象怎样存，访问才便宜”
- `查询` 回答的是“哪些处理逻辑该看到哪些对象”
- `调度` 回答的是“这些处理逻辑按什么顺序、在什么依赖下执行”
- `结构变化` 回答的是“对象的组成变了，世界怎样还能稳定运转”
- `表示层` 回答的是“仿真数据怎样被渲染、动画、声音、特效消费”
- `构建期转换` 回答的是“哪些运行时成本应该提前吃掉”

真正的数据导向系统，差别往往不在“有没有实体和组件”，而在它们对这 6 个问题分别给了什么答案。

## 为什么现代引擎会长出“数据导向孤岛”

如果把问题说得再直接一点：

`现代引擎做数据导向，不是因为面向对象突然过时了，而是因为在高规模同构更新问题上，传统对象世界的表达方式和运行时代价越来越不匹配。`

面向对象当然仍然好用。

它擅长的事情很明确：

- 描述单个对象的职责
- 承载编辑器工作流
- 把复杂业务分给可理解的对象和组件
- 服务内容生产、关卡搭建和交互表达

但一旦问题变成下面这种形状，它就开始吃力：

- 同一帧要更新几千到几十万个对象
- 大量对象做的事情非常相似
- 访问模式不是“沿对象走”，而是“沿数据列走”
- 性能瓶颈开始被 cache miss、结构变化、调度打断主导

这时候系统就会被逼着往另一个方向长：

- 把相同结构的对象聚到一起
- 让查询尽量落在连续内存上
- 让处理逻辑按读写边界调度
- 把结构变化延迟提交
- 把表示层从仿真层旁边拆开

这就是为什么我更愿意把它叫成“数据导向孤岛”。

它并不意味着原有世界无效了。
它更像是在一个原本偏对象化、偏内容驱动的引擎里，专门划出一块区域，让高规模仿真问题按另一套成本模型运转。

## Unity DOTS 在做什么

如果把 `Unity DOTS` 压成一句判断，我会这样说：

`它更像是在尝试把一大块运行时重新建成 ECS 世界。`

Unity 这条路的特点很鲜明：

- 它不只是在运行时加一个 ECS 容器
- 它会把 `Burst`、`Jobs`、`Entities`、`Baking` 这些东西一起拉进来
- 它倾向于把 authoring world 和 runtime world 分开
- 它希望很多成本在构建期就被前移和整理

也就是说，Unity 的思路不是“在现有对象世界旁边加一点数据导向味道”。

它更像是在说：

`如果高规模仿真真的是第一问题，那运行时世界本身就应该按数据导向组织。`

所以你看 DOTS，不能只看 `Entity`、`Component`、`System` 这三个名词。
真正重要的是它把整条链都一起改了：

- 数据怎么表达
- 逻辑怎么调度
- 哪些代码能进 `Burst`
- 运行时纯数据从哪里来
- authoring 和 runtime 怎么切

这也是为什么 DOTS 文章如果只写 API，很容易把最值钱的部分写丢。

## Unreal Mass 在做什么

如果把 `Unreal Mass` 压成一句判断，我会这样说：

`它更像是在 Actor 引擎内部，划出一块高规模数据导向仿真区。`

Unreal 这条路的气质明显不同。

它没有试图把整个引擎世界都翻成 ECS。

它更现实，也更克制：

- `Actor` 世界依然是 Unreal 的主世界
- `MassEntity` 更像是给大规模群体、LOD、复制、表示切换这类问题准备的
- `MassGameplay` 继续在上面补 representation、LOD、replication、StateTree 这些子系统

这件事很关键。

它意味着 Unreal 对这类系统的核心判断是：

`高规模仿真需要数据导向，但内容表达和引擎总世界不一定要彻底改宗。`

所以 Unreal Mass 给人的感觉，不是“旧世界要退场”，而是“旧世界之外，需要一套对高规模对象更便宜的运行时规则”。

这也是为什么拿 Unity DOTS 的想法去套 Unreal，或者反过来拿 Unreal 的混合路线去评价 DOTS，都会失真。

它们在解决相似问题，但对“要改多大范围”这件事，态度根本不同。

## 把 Unity 和 Unreal 的差异压成一句话

如果只能用一句话概括两者的差异，我会这样说：

`Unity 更像重建运行时，Unreal 更像划出特区。`

这句话背后，其实压着 3 个更具体的差异。

第一，`数据导向世界`在整台引擎里占多大比重不同。

- Unity 的路线更激进
- Unreal 的路线更混合

第二，`表示层`和`仿真层`的边界处理方式不同。

- Unity 更愿意把纯数据运行时拉得更远
- Unreal 更强调数据导向仿真如何和 Actor 表示体系协作

第三，`改造成本`的分布不同。

- Unity 往往要求你更早接受一整套新的 authoring / runtime 心智模型
- Unreal 则更像在原有系统旁边，补一层专门服务规模问题的机制

这里没有谁更“先进”的问题。

更准确的说法是：

`它们各自在不同历史包袱、工具链现实和内容生产方式下，给出了不同的工程答案。`

## 为什么自己手搓一版反而很值钱

很多人会觉得，自研一版 ECS 没必要，因为引擎里已经有现成系统。

我反而觉得，恰恰因为现成系统太大，自己手搓一版才有价值。

因为一旦把引擎名词剥掉，你会更容易看见真正不会变的几件事：

- 你需要一个稳定的 `EntityId`
- 你需要一种按“组件组合”聚类的方式
- 你需要一个让查询能命中连续数据的布局
- 你需要一种不立刻打断世界的结构变化方式
- 你需要一个显式声明读写依赖的调度模型
- 你迟早要面对表示层桥接

这几件事，才是这类系统真正的硬核部分。

反过来说，很多人一上来就想做这些东西：

- 自动代码生成
- 超复杂调度器
- 网络复制
- 事件总线
- 反射驱动的万能序列化
- “自己也做一个 Burst”

这些基本都不该是第一版的问题。

第一版真正该证明的事情只有一个：

`当对象规模上来之后，这套系统能不能稳定地把存储、查询、调度和结构变化这四件事管住。`

如果这件事都还没证明，后面的高级能力大多只是在复杂度上继续加码。

## 这组文章后面准备怎么拆

这组系列后面不会按“Unity 篇、Unreal 篇、自研篇”三条线平铺。

更稳的顺序是按问题拆：

1. `Unity DOTS、Unreal Mass 与自研 ECS：问题空间怎么对齐`
2. `Archetype、Chunk、Fragment：性能到底建在什么地方`
3. `Structural Change、Command Buffer 与同步点：为什么改结构总是贵`
4. `调度怎么做：Burst/Jobs、Mass Processor、自己手搓执行图`
5. `构建期前移怎么做：Baking、Traits / Templates / Spawn、离线转换`
6. `表示层边界怎么切：GameObject、Actor、ISM 与 ECS 世界`
7. `如果自己手搓，第一版最小系统应该做到哪里`
8. `什么时候值得上这套东西，什么时候不值得`

这样拆的好处是：

- 同一个问题下可以直接比较 Unity 和 Unreal
- 自研部分可以自然收回成工程动作
- 读者不会被各家术语拖着走

## 常见误解

### 误解一：ECS 的核心就是并行

不是。

并行只是结果之一。
更底层的核心，是你先把数据布局、查询边界和结构变化成本处理对了。

如果这些没处理好，多线程只会把系统变得更难看。

### 误解二：做数据导向就意味着 GameObject / Actor 该被淘汰

也不是。

现实里更常见的答案，恰恰是混合架构。

因为内容生产、编辑器工作流、动画、特效、关卡表达、脚本驱动这些东西，并不会因为你有了 ECS 就自动变简单。

### 误解三：自己手搓第一版，应该先追求完整和高级

不应该。

第一版最重要的不是“功能像不像 DOTS / Mass”，而是有没有抓住这类系统最硬的 5 个不变量：

- 聚类存储
- 查询缓存
- 延迟结构变化
- 显式调度
- 表示桥接

## 我的结论

如果一定要把这篇总论压成一句话，我会这样写：

> 现代引擎做数据导向，不是因为它们突然讨厌对象，而是因为在高规模同构仿真问题上，数据布局、结构变化、调度边界和表示层分离，迟早会把系统推向某种“数据导向孤岛”。

后面这组文章，就是准备顺着这条线，把 Unity DOTS、Unreal Mass 和自研 ECS 一层层拆开。

不是为了列知识点。
而是为了让你最后拿到一张更稳定的地图：

- 为什么这类系统会存在
- 它们真正贵在哪里
- 它们为什么长成不同样子
- 如果自己做，第一步到底该从哪里下手
