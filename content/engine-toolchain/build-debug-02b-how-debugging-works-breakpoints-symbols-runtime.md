> 如果这篇只先记一句话，我建议记这个：`调试不是一个模式名，而是"停得住、对得上、看得见、还能继续跑"这四件事同时成立。`

前面两篇文章先把 `Debug / Release` 的工程取舍，以及 `C++` 和 `C#/.NET` 两条编译链拆开了。

但读者真正继续往下卡住的地方，通常不是：

`到底该叫 Debug 还是 Release？`

而是更底层的一句：

`为什么我明明点了断点、开了调试器、也有源码，结果还是停不住、对不上、看不清？`

如果这个问题不先补出来，后面再看 Unity 里的 `Script Debugging`、`Development Build`、`Debug.isDebugBuild` 和 `Deep Profile`，很容易继续把不同层的能力混成一个词。

## 这篇要回答什么？

这篇只回答五个问题：

1. 断点为什么能让程序停住？
2. 为什么"源码上的一行"最后能和执行点对上？
3. 为什么调用栈、函数名、行号要靠符号和源码映射？
4. 为什么变量有时能看，有时会显示成"优化掉了"？
5. 为什么托管调试一定多一层运行时协作？

## 先给一张调试地图

先把"能调试"压成最小模型，它其实至少依赖四层：

| 调试层 | 它真正回答什么 | 缺了会看到什么 |
| --- | --- | --- |
| `可停住执行` | 程序能不能在一个可执行点被挂住 | 断点发灰、不命中、附加后也停不下来 |
| `符号与源码映射` | 当前执行点能不能被还原成函数名、文件名、行号 | 栈只有地址，源码行对不上 |
| `状态可观察` | 当前帧、局部变量、对象状态能不能被正确读取 | 变量丢失、值很怪、局部看不到 |
| `调试器协作通道` | 调试器能不能和 OS、运行时或引擎建立控制关系 | 进程能跑，但 IDE 挂不上或托管对象读不出来 |

这四层里，最容易被误解的是第一层。

因为我们平时看见的是"在源码第 38 行停住了"，但程序真正执行的，从来不是"源码行"，而是某个已经可执行的代码位置。

## 一、断点为什么能停住：它停在可执行点，不是停在源码行本身

断点不是直接打在源码上。

更接近事实的过程是：

```text
源码上的断点
  -> 通过源码映射找到一个可执行位置
    -> 调试器或运行时在这个位置建立"命中就暂停"的条件
      -> 程序执行到这里时，线程被挂起
```

在原生程序里，这个"可执行位置"更像机器码地址或指令边界。

在托管程序里，它常常还要先经过一层：

`源码行 -> IL 偏移 -> 运行时/JIT/AOT 后的可执行位置`

这也是为什么你在 IDE 里打下一个断点后，通常会先看到一个"绑定"过程。

只有当调试器真的把这行源码对到某个可执行点上，这个断点才算成立。

所以如果你遇到这些现象：

- 断点发灰
- 断点始终显示未绑定
- 同一份源码明明在跑，却怎么都打不中

优先要怀疑的，不是"调试器坏了"，而是：

- 当前二进制和源码不是同一次构建
- 可执行代码没有对应到这行源码
- 这行代码被优化、内联或重排后，已经不再对应一个稳定停点

## 二、单步为什么经常不像"下一行"：执行顺序和书写顺序本来就不完全相同

很多人第一次真正被调试打击信心，不是断点停不住，而是：

`为什么我点了"下一步"，却跳去了看起来不像下一行的地方？`

因为单步前进跟着的，仍然不是"文本顺序"，而是：

`当前可执行点之后，下一段被调试器认为仍可映射回源码的执行路径。`

这中间只要有下面几种情况，体感就会开始变怪：

- 编译器或运行时做了内联
- 一行源码拆成了多个可执行点
- 多行源码被合并成了更少的执行点
- 局部变量生命周期被压缩
- 分支重排后，机器码顺序已经和源码顺序不同

所以"单步"真正依赖的是两件事一起成立：

- 程序还有足够稳定的可执行点可以停
- 这些可执行点还有足够可靠的源码映射可以解释

一旦优化更激进，这两件事都会开始变差。

这就是为什么大家会直观感觉：

`一开优化，调试就开始变得不顺手。`

## 三、为什么调用栈、函数名和行号要靠符号与源码映射

程序跑起来以后，调用栈本身并不会因为"没有符号"就不存在。

真正发生的事是：

- 栈帧还在
- 返回地址还在
- 但你已经很难把这些地址翻译回函数名、文件名和行号

所以"有没有调用栈"和"调用栈能不能看懂"本来就是两件事。

这时起作用的，就是前面几篇一直在讲的那条线：

`符号与源码映射。`

它做的不是让程序"更能运行"，而是让调试器、崩溃分析工具和 IDE 能回答这些问题：

- 这个地址属于哪个函数
- 这个函数大概对应哪段源码
- 现在停住的位置靠近哪一行

这也是为什么没有符号时，你仍然可能看到一串地址；
但只有符号和映射都足够时，你才会看到一张真正可读的栈。

放到工程里，这个区别非常重要：

`没有符号，问题不一定不存在；只是你失去了把问题翻译成人类可读线索的能力。`

## 四、为什么变量有时能看，有时会显示"优化掉了"

"看变量"这件事，本质上不是 IDE 的 UI 问题，而是状态能不能被重建的问题。

调试器想把一个局部变量展示给你，至少要知道三件事：

- 这个变量在当前执行点是否还活着
- 它现在落在寄存器、栈还是已经被消掉了
- 当前这份值还能不能稳定映射回源码语义

只要优化开始积极，下面这些事就会更常发生：

- 变量被直接常量折叠
- 变量只短暂存在于寄存器里
- 多个源码变量被合并
- 某个中间值根本没有稳定存储位置
- 一部分逻辑被内联后，不再保留你熟悉的局部边界

这时调试器不是"不想显示"，而是它已经没有一份稳定、可解释、可重复读取的状态给你看。

所以"变量看不到"往往不是单独的问题，而是这三条线叠加的结果：

- 优化更重了
- 调试信息更少了
- 当前执行点和源码边界更松了

## 五、为什么托管调试一定多一层：调试器不是直接读 C# 语义

到了托管世界，事情还会再多一层。

因为调试器通常并不是直接拿着 C# 源码去读进程内存。

它更像是在和运行时协作，让运行时帮它回答这些问题：

- 现在有哪些托管线程
- 当前托管栈帧是什么
- 这一帧有哪些局部变量和参数
- 托管对象当前是什么类型、有什么字段
- 现在抛出的异常落在哪条托管调用链上

也就是说，在托管调试里，调试器经常不是"直接看见一切"，而是：

`调试器 -> 运行时调试接口 -> 托管世界的线程 / 栈 / 对象 / 异常`

这正是为什么你在 Unity 里会看到一个单独的 `Script Debugging`。

它真正想打开的，不是"Unity 的某种 Debug 模式"，而是：

`让外部调试器有机会和托管脚本运行时建立这条协作通道。`

所以这里也能解释两个常见误会：

### 1. 有 `Development Build`，不等于托管调试链已经打开

`Development Build` 更像开发态 Player 语义。

但"能不能挂托管调试器"，还取决于托管调试通道有没有被一起带进去。

### 2. 有托管断点，不等于原生层就会像 C++ Debug 那样好调

因为你现在调到的，首先是托管语义。

一旦再往下落到 `IL2CPP`、原生二进制、原生符号和原生优化，问题又会重新变成另一条链。

## 六、看现象时，先问自己是缺了哪一层

如果把前面这些原理压回工程里，最有用的不是背术语，而是看到现象时先定位缺哪一层。

| 现象 | 更可能先查哪一层 |
| --- | --- |
| 断点发灰、始终未绑定 | `可停住执行` + `源码映射` |
| 能停，但单步乱跳 | `源码映射` + `优化` |
| 栈只有地址，没有函数名 | `符号与源码映射` |
| 局部变量显示不完整或值很怪 | `状态可观察` + `优化` |
| C# Player 能跑，但 IDE 挂不上托管断点 | `调试器协作通道` |
| Development Build 和 Release 行为差异很大 | 先分开看 `开发态 Player 语义`、`调试链接入`、`Profiler/插桩` |

只要这个问题先问对，后面很多讨论会立刻变简单。

因为你终于不再把"能不能停""能不能看懂栈""能不能看变量""能不能挂上调试器"都压成一句"debug 不好使"。

## 七、带着这条原理线再进入 Unity

如果你准备继续往 Unity 看，先带走这几句：

- `Development Build` 回答的是"这是不是开发态 Player"
- `Script Debugging` 回答的是"托管调试协作通道有没有打开"
- `符号` 回答的是"函数名、行号和可读栈能不能还原出来"
- `Deep Profile` 回答的是"要不要为了看得更细，接受更重的观测成本"

这几条一旦拆开，Unity 里的很多按钮就不会再像"一个总开关的不同别名"。

## 小结

- 调试不是一个模式名，而是"停住执行、对上源码、看见状态、继续运行"这几件事同时成立。
- 断点停住的不是源码行本身，而是某个能够被映射出来的可执行点。
- 调用栈能不能看懂，靠的是符号和源码映射，不是"是不是 Debug 包"这一个词。
- 变量之所以会"优化掉"，通常不是 IDE 的问题，而是当前已经没有稳定、可重建的状态边界。
- 托管调试一定多一层运行时协作，这正是 Unity 里 `Script Debugging` 这类选项存在的背景。
