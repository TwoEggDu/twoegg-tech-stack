前面几篇把加载链从异步管线拆到了优先级调度。但加载链走完以后，问题并没有结束——资源到了内存里，什么时候释放、怎么控制峰值、缓存和卸载之间怎么平衡，这些问题才刚刚开始。

项目里最常见的内存问题，往往不是"某个资源太大"，而是：

- "加载新场景的时候内存飙上去了，旧场景还没卸干净"
- "Unload 了，但内存没下来"
- "缓存了一堆资源怕下次加载慢，结果内存一直涨"

这些问题的根源都是同一个：

`内存峰值不是某一个资源的问题，而是多份资源的生命周期在同一个时间窗口内重叠。`

## 先给一句总判断

`加载时的内存控制，核心不是"单个资源占多大"，而是"在同一时间段内，有多少资源的多少种形态同时存在于内存中"。缓存池控制的是复用，引用计数控制的是生命周期边界，卸载时机控制的是重叠窗口。三者配合，才能把峰值从不可预测变成可管理。`

## 一、内存峰值真正可怕的地方：不是单个资源大，而是中间态和重叠期

### 1. 一个资源在内存里可能同时有多份形态

考虑一个从 LZMA bundle 加载的贴图资源。在加载链的某个瞬间，以下内容可能同时存在于内存中：

- LZMA 压缩包的原始数据（正在被读取）
- 解压后的 LZ4 临时数据（正在被写入缓存）
- 反序列化后的 Native Texture Object（已创建但尚未上传 GPU）
- GPU 端的贴图数据（正在上传）

这四份数据在某个短暂的时间窗口内同时存在。单独看每一份都不算大，但叠在一起可能就触发了内存警告。

### 2. 场景切换是峰值最危险的时刻

从场景 A 切到场景 B 时，如果处理不当：

- 场景 B 的资源开始加载，新的 bundle 被打开、解压、反序列化
- 场景 A 的资源还没来得及卸载，旧的对象和 bundle 仍然占着内存
- 两个场景的资源同时存在于内存中，峰值可能是单场景常驻内存的两倍

这就是为什么很多项目在场景切换时会先进一个"空场景"或"过渡场景"——目的不是为了好看，而是为了在中间清理旧场景的内存，再开始加载新场景。

### 3. 所以内存问题要先分"常驻占用"和"短时峰值"

- 常驻占用：长期在内存中不会释放的资源，比如主角模型、全局 UI、基础 Shader。这部分相对可预测
- 短时峰值：加载和卸载过程中临时出现的内存高点。这部分是问题的主要来源

优化常驻占用靠的是资源本身的优化（贴图尺寸、压缩格式、Mesh 精度）。控制短时峰值靠的是加载和卸载的时序管理。

## 二、缓存池：不是"缓存越多越好"

缓存的基本思路是：加载过的资源不立即释放，留在内存里，下次需要时直接用，避免重复加载的 I/O 和反序列化开销。

### 1. 缓存的收益

- 避免重复 I/O：同一个 Prefab 第二次 Instantiate 时，不需要再从磁盘读取
- 避免重复反序列化：Native Object 已经在内存里，直接引用
- 降低帧间抖动：频繁加载卸载的资源如果命中缓存，帧时间更稳定

### 2. 缓存的代价

- 内存占用：缓存的资源一直在内存里，即使当前没有任何对象引用它
- 内存压力：在移动设备上，系统会在内存紧张时杀后台进程。如果你的缓存占用过高，可能导致应用本身被系统回收
- 缓存失效的成本：如果缓存的资源版本和热更后的版本不一致，需要清除缓存重新加载

### 3. 缓存策略的选择

常见的缓存策略：

- LRU（最近最少使用）：缓存满时淘汰最久没用过的资源。简单有效，但不区分资源的加载成本
- 按资源类型分池：贴图一个池、Prefab 一个池、音频一个池，每个池有独立的上限。好处是大类资源不会互相挤占
- 按场景/功能分区：进入某个场景时加载该场景的缓存，离开时批量清理。好处是生命周期清晰

实际项目中通常是混合策略：核心资源常驻不淘汰，场景资源按场景生命周期管理，临时资源用 LRU 淘汰。

## 三、引用计数：解决"谁在用、能不能释放"

引用计数的核心问题是：一个资源可能被多个对象引用，只有所有引用者都不再需要它时，才能安全释放。

### 1. 为什么需要引用计数

如果没有引用计数，会出现两种问题：

- 提前释放：A 和 B 都引用了贴图 C。A 不再需要 C 了，把 C 释放掉。但 B 还在用 C——B 看到的就是丢失的粉红色贴图
- 泄漏：谁都不敢释放，因为不知道还有没有人在用。结果资源一直堆在内存里

### 2. Unity 的引用计数边界

Unity 引擎内部对 Native Object 有自己的引用管理。但对于 AssetBundle 层面，引用关系更复杂：

- bundle 之间有依赖关系：bundle A 依赖 bundle B 的 Shader，如果 B 先被 Unload(true)，A 里的材质就丢引用了
- `Instantiate` 创建的实例和原始资源之间是引用关系，不是拷贝关系（对于贴图和 Mesh 等资源）
- `Unload(false)` 只释放 bundle 的 SerializedFile 和元数据，已经 Load 出来的 Asset 仍然在内存中
- `Unload(true)` 会强制释放 bundle 和所有从中 Load 出来的 Asset，即使有对象还在引用它们

### 3. 框架层的引用计数

Addressables 和 YooAsset 在 bundle 层之上实现了自己的引用计数：

- 每次 `LoadAssetAsync` 增加引用计数
- 每次 `Release` 减少引用计数
- 引用计数归零时，框架决定是否释放底层 bundle

但框架的引用计数只追踪通过框架 API 进行的操作。如果你绕过框架直接操作底层 bundle，引用计数就会不准确。

### 4. 引用计数最容易出错的地方

- 忘记 Release：Load 了但忘记 Release，引用计数永远不归零，资源永远不释放。这是最常见的内存泄漏来源
- 多次 Release：Release 次数超过 Load 次数，引用计数变负，行为未定义
- 跨系统引用：系统 A 加载了资源，把引用传给了系统 B。系统 A 觉得自己用完了就 Release 了，但系统 B 还在用。如果引用传递时没有增加计数，就会提前释放

## 四、卸载时机：不是"用完就卸"，也不是"留着不管"

### 1. Unload(false) vs Unload(true) 的选择

这是 Unity AssetBundle 层最经典的决策点：

- `Unload(false)`：释放 bundle 的文件句柄和元数据，但已经 Load 出来的 Asset 保留在内存中。好处是不会打断正在使用的资源。坏处是这些 Asset 变成了"孤儿"——bundle 已经不在了，Asset 还在内存里，如果再次 Load 同一个 bundle 的同一个 Asset，会在内存里产生两份
- `Unload(true)`：释放一切，包括所有从这个 bundle 里 Load 出来的 Asset。如果有对象还在引用这些 Asset，引用会变成 missing（粉红色贴图、丢失的 Mesh）

### 2. 实践中的卸载策略

- 场景 bundle：在场景切换时使用 `Unload(true)`，因为旧场景的资源不会再被引用
- 共享 bundle（Shader、公共材质）：不要轻易 Unload，因为很多 bundle 都依赖它
- 临时 bundle（一次性剧情、活动资源）：使用后通过引用计数确认无引用，再 `Unload(true)`
- 如果用 Addressables / YooAsset，框架会帮你管理大部分 Unload 时机，但前提是你的 Load / Release 配对正确

### 3. 延迟卸载

一种常见的折中策略是延迟卸载：引用计数归零后不立即释放，而是标记为"可释放"，过一段时间（比如 30 秒）后如果仍然没有新的引用，再真正释放。

好处是：如果用户短时间内切换界面再切回来，资源还在内存里，不需要重新加载。

坏处是：延迟期间内存一直被占着。如果延迟释放的资源太多，内存压力会上升。

## 五、怎么在 Profiler 里看内存问题

### 1. Memory Profiler

Unity Memory Profiler（Package Manager 里安装）可以对内存做完整快照。关注：

- Native Objects 按类型排序：Texture2D、Mesh、AnimationClip 通常是大户
- 重复资源：同一个贴图出现多份，说明 bundle 管理或引用计数有问题
- "No Reference" 的大资源：已经没有对象引用但仍然在内存里，说明泄漏

### 2. 关键时刻对比

在场景切换前后各做一次快照，对比差异：

- 旧场景的资源是否在切换后被完整释放
- 新场景加载后的内存是否在预期范围内
- 峰值（切换过程中的最高点）和稳态（切换完成后的水平）差多少

### 3. Profiler CPU 模块

关注 `GC.Alloc`、`Unload` 相关标记。如果 `Resources.UnloadUnusedAssets` 被频繁调用，每次都耗时很长，说明有大量需要清理的孤儿资源——这通常意味着 `Unload(false)` 被过度使用了。

## 六、常见误判

### "内存高就是资源太大"

不一定。可能是同一个资源在内存里有多份（重复加载），或者旧资源和新资源的生命周期重叠。先排查重复和泄漏，再优化资源本身。

### "Unload(true) 太危险，全用 Unload(false)"

`Unload(false)` 看起来安全，但它制造的"孤儿资源"是隐性内存泄漏的温床。如果大量使用 `Unload(false)` 而不配合 `Resources.UnloadUnusedAssets`，内存会持续增长。

### "Resources.UnloadUnusedAssets 应该每帧调用"

这个调用会遍历所有资源检查引用状态，开销很大（几十到几百毫秒）。不应该频繁调用。通常在场景切换完成后调用一次就够了。

### "用了 Addressables 就不用管内存了"

Addressables 帮你管理引用计数和卸载时机，但前提是你的 Load / Release 调用配对正确。一个忘记 Release 的 Handle 就能导致整个 bundle 无法释放。

## 七、最小检查表

- 场景切换时是否有清理旧场景资源的环节（过渡场景或显式 Unload）？
- 是否存在重复加载同一资源的情况？Memory Profiler 快照里有没有同名重复对象？
- 引用计数（Load / Release）是否严格配对？有没有忘记 Release 的路径？
- 缓存策略是否有上限？缓存满时的淘汰机制是什么？
- `Unload(false)` 的使用是否配合了 `Resources.UnloadUnusedAssets`？
- 是否监控了场景切换时的内存峰值（而不是只看稳态）？

## 这一篇真正想立住的判断

`内存峰值的根源通常不是"某个资源太大"，而是"多份资源的多种形态在同一时间窗口内重叠"。控制峰值靠的不是压缩资源，而是管理加载和卸载的时序——让新资源进来之前旧资源已经出去，让中间态的存活时间尽可能短。`

`缓存、引用计数和卸载时机不是三个独立问题，而是同一个生命周期管理系统的三个切面。缓存决定"留不留"，引用计数决定"能不能放"，卸载时机决定"什么时候放"。`
