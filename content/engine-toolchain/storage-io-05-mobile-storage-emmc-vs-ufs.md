> 如果只先记一句话，我建议记这个：手机上的存储问题从来不只是“UFS 快不快”，而是“随机访问、系统争用、工作集、热约束和后处理成本会不会在同一个时间窗里一起爆出来”。

很多人在 PC 上建立了一个直觉：

- HDD 特别怕碎访问
- SSD 好很多

然后到了手机上，常见的下一步推理是：

`手机反正没有机械硬盘，所以碎访问的危险性应该小很多。`

这个判断只对了一半。

手机确实没有 HDD。  
但这不代表碎访问问题自动消失。  
它只是换了一种更移动端的方式继续出现：

- 低端机上是 eMMC 随机 I/O 先扛不住
- 主流和高端机上是 UFS 很快，但应用自己把请求打碎
- 热机后，I/O 回来的后处理又开始比纯读更危险
- 系统服务、前后台活动和更紧的工作集，会让行为变得更不稳定

所以这篇文章想讲透的是：

`为什么 eMMC 和 UFS 的差异会真实改变项目行为，以及为什么碎访问在手机上依然是一个工程风险。`

前面的设备类型篇已经把 eMMC 和 UFS 的设备侧差异立过一遍。  
所以这一篇我不会再把协议名词和设备分类完整重讲，而是专门看：

- 它们在手机系统条件下会被什么继续放大
- 为什么移动端的 SoC 预算、工作集、热和生命周期会让问题长出不同外观

## 第一层：移动端的 I/O 直觉为什么不能照搬 PC

手机和 PC 在 I/O 问题上最大的区别，不只是设备名字不一样，  
而是整台机器的工作条件都变了。

移动端更常见的是：

- CPU、GPU、内存控制器、存储控制器都在同一颗 SoC 预算里
- 可用内存更紧
- 系统前后台切换更频繁
- 热和功耗会影响持续性能
- 应用通常更难把 I/O 喂成高而稳的队列深度

这意味着即使底层存储已经很快，  
项目也仍然会在下面这些地方出问题：

- 首载第一次命中
- 运行时零散补读
- 热机后解压 / 反序列化变慢
- 缓存被系统回收后重新进场

所以手机上的 I/O 判断应该看的不是：

`宣传页写了 UFS 4.0。`

而是：

`这台机器在低队列深度、小粒度、冷热波动、系统争用时，行为到底怎样。`

## 第二层：eMMC 和 UFS 的根差别到底在哪

把复杂细节先都压平，可以先把它们理解成：

| 类型 | 典型定位 | 你最该记住的差别 |
| --- | --- | --- |
| `eMMC` | 更老、更便宜、常见于低端或旧设备 | 随机 I/O、尾延迟和持续表现更容易先暴露问题 |
| `UFS` | 更新一代、面向更高性能移动设备 | 命令队列和并行能力更强，顺序和随机表现都更稳 |

如果再往工程化一点说，UFS 通常会比 eMMC 更强在：

- 请求并行
- 队列能力
- 尾延迟控制
- 顺序与随机综合表现

而 eMMC 更容易在下面这些条件下迅速塌下去：

- 小块随机访问
- 读写混合
- 大量独立事务
- 冷启动首次命中

### 为什么“顺序读带宽”在手机上特别容易骗人

很多资料喜欢把重点放在：

- UFS 3.1 有多快
- UFS 4.0 峰值多高

这些数字当然有参考价值。  
但项目里更容易卡你的，往往不是“连续拷一个超大文件”。

真正更像真实游戏场景的是：

- 很多小 chunk
- 一点元数据
- 一点 bundle 头
- 一点依赖
- 一些并不整齐的补读

这些请求更接近：

`低 QD、小粒度、局部性不稳定的随机或半随机访问。`

而这恰恰是 eMMC 和 UFS 拉开真实体验差距的地方。

## 第三层：为什么 eMMC 更容易在低端机上先暴露问题

低端机如果用的是 eMMC，  
项目最常见到的不是“每次都特别慢”，而是：

- 平均还能接受
- 但尾延迟和抖动更明显
- 一旦访问模式变碎，就很容易突然跌下去

这是因为 eMMC 往往同时处在一个更不利的系统环境里：

- 存储本身随机 I/O 更弱
- CPU 更弱，解压和解析更慢
- RAM 更小，页缓存更容易被顶掉
- 热预算更小，持续性能更差

所以低端机上的 I/O 问题，常常不是单点问题，  
而是：

`弱存储 + 弱 CPU + 小内存 + 更容易热` 一起放大。

这也是为什么很多低端机体验会表现成：

- 第一次进场明显慢
- 进入复杂区域时会有一跳一跳的顿挫
- 资源反复重进场时特别不稳

## 第四层：UFS 很快，为什么碎访问还是危险

这件事特别值得单独讲。

UFS 确实比 eMMC 强得多。  
但“UFS 很快”并不等于：

- 文件系统成本消失
- 小文件事务变便宜
- 页缓存永远热
- 热机后 CPU 后处理不再重要

项目里真正危险的还是这种模式：

```text
open A -> read 8 KB -> close
open B -> read 32 KB -> close
open C -> read 4 KB -> close
```

因为即使底层介质足够快，你仍然要付：

- 路径解析
- 元数据查找
- open / close
- 页缓存 miss
- 主线程或关键线程上的后续收口

换句话说：

`UFS 把“设备太慢”的问题缓解了，但没有把“应用把访问模式做碎了”这层问题抹掉。`

## 第五层：手机上为什么特别容易出现“读回来了，但体验还是抖”

这是移动端最常见的误判之一。

团队会看到：

- 真机上底层读速不差
- 单独 benchmark 看着也还行

但玩家体感仍然会出现：

- 首载卡一下
- 进场抖一下
- 热机后更不稳

原因往往在于：

### 1. SoC 上所有账都在抢同一套预算

在手机上，I/O 回来之后：

- 解压要 CPU
- 反序列化要 CPU
- 上传 GPU 前还要准备数据

而这些动作并不发生在一个“资源无限的后台世界”。  
它们和游戏主循环、渲染、系统服务在抢同一颗 SoC 的预算。

### 2. 页缓存更容易失效

移动端 RAM 更紧，系统又更积极管理工作集。  
所以开发机上很常见的“第二次读就热了”，在真机上并不总是稳定成立。

尤其是：

- 长时间运行后
- 切过前后台后
- 复杂场景来回切换后

页缓存和应用自己的资源缓存都可能已经发生变化。

### 3. 热机会把瓶颈位置悄悄改掉

冷机时你可能真的主要卡在 I/O。  
热机以后，存储还没一定显著掉，但 CPU 解析、解压和收口更可能先掉。

于是体感上就会变成：

- “读速看着还行”
- 但整体加载体验变差了

这也是为什么移动端判断不能只看纯 I/O benchmark。

## 第六层：Android 上尤其容易被哪些现实放大

Android 设备跨度特别大，所以 I/O 行为也更容易出现“机型分层感”。

最常见的放大器有这些：

- 从 eMMC 到 UFS 4.x 的设备跨度很大
- 文件系统可能是 `ext4` 或 `F2FS`
- 包体、资源包、热更新和校验链路可能叠额外层次
- 系统服务和 OEM 定制会带来不同程度的后台干扰

这意味着同一套资源组织方式，在不同 Android 机型上的表现差异可能非常明显。

所以如果项目只在一两台高端 UFS 真机上看过，  
判断通常是不够的。

## 第七层：iPhone 很快，为什么这套直觉也还是成立

苹果高端设备的存储通常非常强。  
如果只看“底层设备速度”，很多 Android 低端机根本不在一个量级。

但这并不意味着这篇文章对 iPhone 就不成立。  
因为你依然会遇到：

- 文件系统和页缓存的冷热差异
- 应用线程上的后处理收口
- 热和功耗导致的持续性能变化
- 小事务太碎时的固定成本

也就是说：

`更快的移动存储，能显著抬高上限，但它不会把“访问模式”和“加载链后半段”从问题空间里删除。`

## 项目上最该怎么用这层判断

### 如果目标设备包含 eMMC 或弱 UFS 机型

更稳的策略通常是：

- 减少热路径独立文件数
- 更激进地合批和局部化
- 把首次成本尽量往 Loading 界面收
- 减少运行时零散补读次数

### 如果目标设备主要是主流和高端 UFS

你当然可以更大胆地做：

- 分 chunk streaming
- 更频繁的预取
- 更细粒度的动态资源调度

但前提仍然是：

- 请求粒度不要过碎
- 依赖不要发散
- 后处理不要收口在关键帧

### 测试时最容易漏掉的三件事

1. 只在冷机、空闲、刚启动时测试。
2. 只在一台高端 UFS 设备上测试。
3. 只看顺序带宽，不看低 QD 随机读和长时间运行后的尾延迟。

## 这一篇最该带走的结论

`eMMC 和 UFS 的差别会真实改变项目里的加载体验，但“手机没有 HDD”并不等于“碎访问已经安全”。`

移动端更危险的地方恰恰在于：

`随机 I/O、工作集、热约束、系统争用和加载链后半段，会在更紧的预算里一起放大。`
