> DHE 的理想是"只有改了的函数走解释器"，但没有注入规则时，现实是"改了一个函数，调用它的一整片都会被拖进解释器"。

这是 HybridCLR 系列第 26 篇。

前一篇（HCLR-25）从 dhao 文件格式和差分算法的角度，拆了 DHE 在离线阶段怎么算出变更集。但离线算出"谁变了"只是前半段，运行时怎么让"没变的函数继续走 AOT"才是后半段。这篇要讲的就是这后半段里最关键的问题：

`当一个函数被标记为变更，调用它的函数是不是也要一起掉进解释器？如果是，怎么止住？`

## DHE 的理想与现实

DHE 的核心承诺很清楚：未改动的函数继续走 AOT，只有真正发生变化的函数走解释执行。

这个承诺如果严格成立，性能模型就非常干净：改了 3 个函数，就只有这 3 个函数付出解释执行的代价，其余几千个函数的 AOT 性能完全不受影响。

但现实比这复杂。

举一个最小场景。假设一个 DHE 程序集里有四个函数，调用关系如下：

```
A() → B() → C() → D()
```

其中只有 D 的方法体发生了变化。按 DHE 的理想模型，应该只有 D 走解释执行，A、B、C 继续走 AOT。

但如果不做任何额外处理，实际情况是：D 变了 → C 调用了 D → C 也被标记为脏 → B 调用了 C → B 也被标记为脏 → A 调用了 B → A 也被标记为脏。

最终结果：改了 1 个函数，4 个函数全部掉进解释器。

这就是官方文档里说的"脏函数链式传染"。如果变化的函数恰好是一个被大量调用的基础函数，传染范围会迅速扩大到整个程序集。

## 传染的机制

要理解传染为什么会发生，需要回到 il2cpp 生成的 C++ 代码层面。

il2cpp 在 AOT 编译阶段，会把 C# 方法编译成 C++ 函数。对于非虚函数调用，il2cpp 生成的代码通常是直接调用：编译器在代码生成阶段就已经确定了被调用函数的地址，生成的 C++ 代码直接以函数名调用目标方法。

这意味着 C 的 AOT 代码在编译时就已经绑定了 D 的 AOT 入口地址。当 D 因为热更新而变更时，D 的执行需要切换到解释器版本。但 C 的 AOT 代码仍然在直接调用 D 的旧 AOT 入口——这个入口指向的已经不是正确的逻辑了。

解决这个问题有两种思路：

第一种：把 C 也标记为脏函数，让 C 也走解释执行。解释器版本的 C 在调用 D 时，会通过 runtime 的方法分发机制找到 D 的最新版本（解释器版本），调用链就正确了。但 C 本身的逻辑没有任何变化，让它从 AOT 退回解释执行纯粹是被 D 拖累的。

第二种：在 C 的 AOT 代码里做手脚——不要直接硬编码 D 的地址，而是在调用前先检查 D 是否已经变更。但这等于要修改 il2cpp 的代码生成逻辑。

DHE 选择的是一种更精巧的变体：不改调用方的代码，而是在被调用方的函数头部注入一段检查代码。

## 直接传染 vs 间接传染

在拆注入机制之前，先把两种传染模式分清楚。

**直接传染（直接脏函数）**：函数自身的方法体发生了变化。D 的 IL 代码和原版不同，dhao 文件会把 D 标记为变更函数。D 必须走解释执行，这没有任何争议。

**间接传染（间接脏函数）**：函数自身的方法体没有任何变化，但它调用了一个已经变更（或间接变更）的函数。C 的代码一行没改，但因为 C 调用了 D，而 D 变了，C 就被连带标记为脏。

间接传染的爆炸性在于它是递归传播的。不只是直接调用变更函数的那一层会被拖下水，调用链上每一层都会被递归标记。

回到前面的例子：

```
D: 直接脏函数（方法体变了）
C: 间接脏函数（调用了 D）
B: 间接脏函数（调用了 C，C 是脏的）
A: 间接脏函数（调用了 B，B 是脏的）
```

官方文档用了一个更直白的描述：如果未对 il2cpp 生成的原始代码作任何修改，对于非虚函数调用，存在脏函数链式传染的问题。在实际项目中，修改常用的基础函数可能导致大量代码被标记为脏函数。

这里有一个细节值得注意：官方明确指出传染问题主要存在于"非虚函数调用"。虚函数调用本身就通过 vtable 间接寻址，被调用方的实际地址在运行时才确定，所以虚函数调用天然不会产生编译期地址绑定导致的传染问题。非虚函数调用则是在编译期直接绑定目标地址，这才是传染链的传播媒介。

## 注入规则：在函数头部插检查代码

DHE 解决间接传染的方式，不是在调用方插代码，而是在被调用方的函数入口插一段检查跳转代码。

官方文档的描述是：默认会在所有函数头部注入一小段检查跳转代码。

具体来说，当 il2cpp 为 DHE 程序集生成 C++ 代码时，会在每个函数的最前面插入一个条件分支。这个分支检查当前方法的 `isInterpterImpl` 标志位：如果为 true，说明这个函数已经在运行时被标记为变更，需要跳转到解释器版本执行；如果为 false，继续走后面的原生 AOT 代码。

用伪代码表示：

```cpp
int32_t SomeFunction(int32_t x, const RuntimeMethod* method)
{
    // --- 注入的检查代码 ---
    if (method->isInterpterImpl)
    {
        // 跳转到解释器版本
        typedef int32_t (*RedirectFunc)(int32_t, const RuntimeMethod*);
        return ((RedirectFunc)method->methodPointerCallByInterp)(x, method);
    }
    // --- 原始 AOT 代码 ---
    // ... 函数原本的逻辑 ...
}
```

这段代码的关键在于：它改的不是调用方，而是被调用方自己。

回到 A → B → C → D 的例子。如果 D 的函数头部被注入了检查代码，那么 C 的 AOT 代码照常直接调用 D。调用到达 D 的入口时，D 自己的检查代码会先执行：发现 `isInterpterImpl` 为 true，直接跳转到解释器版本。C 完全不需要知道 D 变了，C 的 AOT 代码不需要做任何修改，C 也就不会被标记为脏函数。

传染链在 D 这里就被切断了。B 不脏，A 也不脏，整条调用链上只有 D 走解释器。

这就是官方说的"避免间接脏函数传染"——通过在函数入口注入检查代码，让每个函数自己负责判断该走 AOT 还是解释器，而不是把这个判断推给调用方。

注入代码实际上包含两个部分：

1. **metadata 初始化块**：在函数头部增加当前函数对应 metadata 的初始化代码。
2. **运行时检查分支**：检查 `isInterpterImpl` 标志并决定执行路径。

对于较长的函数，这两个部分的开销可以忽略不计。但对于非常短的函数，问题就来了。

## 注入的代价

注入不是免费的。官方文档明确列出了两项成本。

**代码体积膨胀**：对于 DHE 程序集，注入检查代码会导致 il2cpp 生成的 C++ 代码总量增加约 30%。每个函数的头部都多了 metadata 初始化块和条件分支，这些代码在编译后会直接增加最终二进制的体积。

**短函数性能下降**：对于 `int Age { get; set; }` 这样的极短函数，注入的影响尤其明显。官方文档的原话是：这种插入可能会造成可观测的性能下降，甚至能达到 10%。

原因很具体。极短的 getter/setter 函数本身可能只有一两条指令，在不注入的情况下，C++ 编译器大概率会把它 inline 到调用方。但注入之后，函数头部多了两次额外检查（metadata 初始化 + `isInterpterImpl` 判断），C++ 编译器很可能放弃 inline。一个原本零开销的属性访问，变成了一次真实的函数调用加两次条件判断。

这就引出了一个工程决策：是不是所有函数都需要注入？

答案是否定的。官方提供了 `InjectRules`（函数注入策略）机制，允许开发者通过 XML 配置文件精确控制哪些函数注入、哪些不注入。

配置文件的基本结构如下：

```xml
<rules>
  <assembly fullname="GameLogic">
    <type fullname="MyNamespace.MathUtils">
      <!-- 这些短函数不需要注入 -->
      <method name="get_*" mode="none"/>
      <method name="set_*" mode="none"/>
    </type>
    <type fullname="MyNamespace.PluginInterface">
      <!-- 接口方法需要注入 -->
      <method name="*" mode="proxy"/>
    </type>
  </assembly>
</rules>
```

两种 mode 值：

- **none**：不注入。函数头部没有检查代码，如果该函数的被调用方发生变更，传染会正常传播。
- **proxy**：注入标准检查代码。函数头部插入 `isInterpterImpl` 检查，切断传染链。

默认行为是：如果没有配置任何注入策略文件，DHE 会对所有函数进行注入。这是最安全的策略——不会发生任何间接传染，但要承受 30% 的代码膨胀和短函数性能下降。

匹配规则支持通配符，assembly、type、method 层级都可以用 `*` 匹配。如果某个函数满足多条规则，以最后一条规则为准，这允许先用通配符设置大范围策略，再用具体规则覆盖个别函数。

有一个容易被忽略的工作流约束：注入策略文件必须和构建的主包保持一致。每次独立发布的主包都必须备份当时使用的注入策略文件。生成 dhao 文件时也必须使用对应主包的注入策略文件，否则可能导致运行了错误的逻辑甚至崩溃。这一点和 dhao 文件本身的一致性要求是同一级别的。

## 项目实践

把注入策略落到项目里，需要回答一个核心问题：哪些函数该注入，哪些不该？

**建议注入的函数（mode="proxy"）**：

- **公共 API 边界**：模块对外暴露的接口方法。这些方法是跨模块调用的入口点，如果不注入，一个模块内部的变更会沿着调用链传染到所有使用方。
- **频繁变更的模块**：已知每次热更都可能改动的功能模块。对这些模块的函数做注入，确保变更不会波及调用方。
- **插件接口**：SDK 或框架对外提供的扩展点。注入保证框架代码和插件代码之间的隔离。

**建议不注入的函数（mode="none"）**：

- **极短的 getter/setter**：`int get_Health()` 这类单行属性访问器。它们几乎不会变更，注入只会白白损失 inline 机会。
- **数学工具函数**：`Vector3.Dot`、`Mathf.Lerp` 等纯计算函数。这类函数调用频率极高、方法体极短，注入的相对开销最大。
- **内层循环中的辅助函数**：在性能热点路径上被高频调用的短函数。

**建议直接放进热更程序集的函数**：

如果某些函数几乎每次补丁都会改动，与其反复在 DHE 里走注入检查 → 跳转解释器这条路径，不如直接把它放进独立的热更程序集。这样它从一开始就走解释执行，省去了注入检查的开销，也避免了传染问题。

官方还提到了一个自动化方案：可以用工具（如 dnlib）遍历 DHE 程序集中的所有方法，按指令数设定阈值（例如方法体指令数小于 10 的函数不注入），自动生成注入规则文件。这比手动配置每个函数要实际得多。

## 收束

DHE 的脏函数传染问题，本质上是 AOT 编译中直接调用绑定和运行时函数替换之间的矛盾。

il2cpp 在 AOT 阶段把非虚函数调用编译成直接调用，被调用方的地址在编译期就已经固化。当 DHE 需要在运行时把某个函数从 AOT 切换到解释执行时，所有直接调用该函数的 AOT 代码都面临地址失效的问题。最简单的处理方式是把调用方也标记为脏函数，但这会引发链式传染。

注入机制的核心思路是：不改调用方，改被调用方自己。在每个 DHE 函数的入口插入一段 `isInterpterImpl` 检查，让函数自己负责判断该走 AOT 还是解释器。这样调用方的直接调用仍然有效——它调用的地址没变，只是到达该地址后，函数入口的检查代码会在必要时把执行重定向到解释器。

代价是代码膨胀和短函数性能损失。收益是彻底切断间接传染链，让 DHE "只有改了的函数走解释器"这个承诺真正成立。

在项目配置上，全量注入是最安全的默认策略，但对性能敏感的短函数应该通过 InjectRules 排除。注入策略文件和 dhao 文件一样，是构建产物链中必须严格版本管理的契约文件。
