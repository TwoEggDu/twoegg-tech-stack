> 这篇不讲"哪些资源该放哪条交付路径"，那一层请先看 [Resources、StreamingAssets、AssetBundle、Addressables 到底各自该在什么场景下用]({{< relref "engine-toolchain/unity-resources-streamingassets-assetbundle-addressables-when-to-use.md" >}})。  
> 这篇只讲另一件事：`为什么你已经知道这些路该怎么选了，资源还是会留在 base。`

如果你已经在 Google Play 的 AAB 报错里看到 `base` 超过限制，第一反应通常是"是不是没拆对"。这个判断只对了一半。

更常见的现实是：

`不是所有东西都能被"拆出去"；有些内容本来就属于首启、属于执行底座，或者属于 Unity / Android 的基础交付层。`

这也是为什么很多团队一边在讨论 `Resources`、`StreamingAssets`、`AssetBundle` 和 `Addressables`，一边还是会在 AAB 校验里撞上同一个结果：

- 首场景太重，第一屏必须的内容全在 base
- 原生库太重，`libil2cpp.so`、`libunity.so` 和第三方 `.so` 一起把 native 层撑大
- ABI 还留着 `ARMv7`，同一份能力在 32 位和 64 位各来一套
- 纹理格式虽然做了 targeting，但首场景和共享贴图仍然绕不开 base

所以本篇要回答的不是"怎么搬走"，而是：

`为什么这些内容会留在 base。`

## 先把 base 这层想清楚

`base module` 不是一个"随便装内容的大桶"。  
它更像是应用启动时必须先具备的那一层：进程要起来，Unity 运行时要起来，第一帧要出来，脚本绑定和关键资源要能找到。

这意味着，只要某个内容满足下面任一条件，它就很容易留在 base：

- 它属于首启必需
- 它属于引擎执行底座
- 它属于 Unity 运行时默认可直接找到的入口
- 它属于平台安装阶段就必须解包到位的本地代码或配置

这也是为什么把 base 看成"以后再瘦身"的地方，最后会很容易失真。  
真正该问的是：

`它是不是必须在应用启动前就已经到位？`

如果答案是肯定的，它就很难被当成"后置交付内容"处理。

## 首场景为什么最容易把问题带进 base

`first scene` 不是普通场景，它往往是整个应用的启动边界。

在很多 Unity 项目里，首场景要负责的并不只是"显示一个 UI"，而是：

- 加载启动图和首屏交互
- 读取本地配置和版本信息
- 初始化热更入口
- 拉起登录、公告、服务器列表
- 触发最早的一批脚本绑定和资源查找

只要这些东西还停留在首启链路里，它们就必须在应用第一次打开时可用。  
这意味着，和首场景强绑定的资源，天然更容易留在 base。

这里最容易出现的误解是：

`只要资源后面能热更，首场景里就可以随便塞。`

这句话通常只对"后续版本可更新"成立，对"首启时必须出现"不成立。  
首场景决定的不是"之后能不能更新"，而是"应用能不能先跑起来"。

如果第一屏的 UI、引导图、配置文件、启动流程脚本和关键贴图都要靠它，那它们就是 base 的天然候选。

## Resources 和 StreamingAssets 为什么会粘住 base

这一段最容易和上一篇混在一起，所以先把边界说清楚。

上一篇讨论的是：

`这些资源路应该在什么场景下用。`

这一篇讨论的是：

`为什么你一旦用了它们，内容还是会留在 base。`

### Resources：它解决的是"运行时查找"，不是"交付边界"

`Resources` 的问题不在于"不能用"，而在于它的语义太接近"随时能找到"。  
只要项目里还在用 `Resources.Load` 做启动期查找，Unity 就会把这条路径理解成运行时必须能直接命中的资源入口。

这会带来一个非常现实的后果：

- 你以为它只是一个"方便的目录"
- 平台和引擎却把它当成"默认可达的内容面"
- 一旦它和首场景、UI、配置、热更 bootstrap 纠缠在一起，就很难再说它是"可延后的交付内容"

所以 `Resources` 的典型问题不是"它把 base 撑大了"，而是：

`它几乎没有帮你表达交付边界。`

### StreamingAssets：它更像"原样带文件"，不是 Unity 对象交付

`StreamingAssets` 常常被误解成"低配版 AssetBundle"。  
实际不是。

它更接近"把原始文件原封不动地带进包里"。  
这对配置、原生数据、视频、二进制、第三方 SDK 需要原样读取的文件很有用，但它并不天然理解 Unity 对象图、依赖关系和延迟交付。

所以 `StreamingAssets` 的典型定位是：

- 你要带一份原样文件进包
- 你不打算让 Unity 去替你理解这份文件的对象关系
- 你接受它在启动链路里可直接访问

一旦它承担的不是"原样文件"，而是"要不要首启就有的资源"，它就会很容易继续留在 base。

这也是为什么 `StreamingAssets` 不是"简单替代 AssetBundle 的低配方案"。  
它解决的是文件形态，不是交付治理。

## 为什么原生代码基本跑不掉

如果说首场景和 `Resources` 还属于"内容层"，那 `libil2cpp.so`、`libunity.so` 和第三方原生库就已经是"执行层"了。

这三类东西之所以常常留在 base，不是因为它们不大，而是因为它们不是可延后的资源：

- `libunity.so` 是 Unity 引擎运行时的一部分
- `libil2cpp.so` 是 IL2CPP 后端生成并依赖的原生执行体
- 第三方 `.so` 往往承担的是平台能力、SDK 绑定或 JNI 入口

它们要解决的问题不是"加载哪张图"，而是：

`应用怎么跑、脚本怎么执行、平台能力怎么接上来。`

所以它们和资源文件的边界完全不同。  
资源可以讨论"是否后置加载"，原生库通常讨论的是"有没有它应用就起不来"。

这就是很多团队在看 base 报错时最容易掉进去的坑：

- 他们把 `.so` 当成普通资源
- 他们把原生插件当成"后面再想办法搬"
- 他们以为只要内容够大，就应该能进 asset pack

但 base 里真正难搬的，往往不是内容本身，而是执行链条。

## ARMv7 为什么会让 base 一下子变重

`ARMv7` 不是一个"多兼容一点"的小选项，它会直接改变 native 层的体积结构。

只要项目还在同时保留 `ARM64` 和 `ARMv7`，很多原生库就不是一份，而是两份：

- 同一个插件要有两套二进制
- 同一个引擎底座要覆盖两种 ABI
- 同一批 native 能力会被复制到不同架构里

这也是为什么很多人误以为自己"只是加了几个 SDK"，但最终 base 却突然变大。  
从工程视角看，实际发生的往往不是"加了几个文件"，而是：

`把整个 native 层按 ABI 又复制了一遍。`

如果项目并不需要继续服务很老的 32 位设备，`ARMv7` 往往就是最先该被怀疑的对象。  
它不是唯一的体积来源，但它经常是最明显的乘数项。

## 纹理格式 targeting 为什么不能当成"自动离开 base"的按钮

`Texture Compression Targeting` 的价值是真实的，但它经常被理解过头。

它真正做的是：

- 让不同设备拿到更合适的纹理压缩格式
- 减少某些设备不必要下载的纹理变体
- 避免把所有纹理格式硬塞给所有设备

但它没有做的事情是：

- 把首场景必须用到的纹理从 base 里自动搬走
- 把启动期依赖的共享贴图从交付边界里抹掉
- 把原本就属于 install-time 的内容变成 on-demand

换句话说，纹理格式 targeting 解决的是"同一份内容如何按设备分发"，不是"这些内容从哪里来、能不能晚点来"。

这就是为什么它常常能让结果变好，但不能单独解决 base 超限。  
如果首场景、启动 UI、共享图集和关键材质还在启动链上，它们照样会留在 base。

## 一个更实用的判断顺序

如果你现在只想先判断"这块东西为什么在 base 里"，可以按这个顺序看：

| 先问什么 | 结论更像什么 |
|---|---|
| 它是不是首场景必需 | 是的话，优先留在 base |
| 它是不是 Unity 运行时默认直接可达 | 是的话，容易被当成基础内容 |
| 它是不是原生执行体或第三方 `.so` | 是的话，基本属于执行层 |
| 它是不是 32/64 位各一份 | 是的话，体积会被 ABI 放大 |
| 它是不是只做了设备分发优化 | 是的话，不代表它会离开 base |

这张表的核心不是让你马上拆包，而是让你先别把所有内容都归因成"没开某个开关"。

## 收口

`base` 留下来的内容，通常不是"没想办法拆"，而是"不能晚于首启到达"。

它可能是首场景必须出现的资源，可能是 `Resources` 和 `StreamingAssets` 这种默认路径，可能是 `libil2cpp.so` 和 `libunity.so` 这种执行底座，也可能是 ABI 复制和纹理 targeting 这种让体积变大的平台层机制。

所以遇到 `base` 超限时，最先要改的不是心态，而是判断方式：

`先分清这是内容层问题，还是执行层问题；先分清这是交付层问题，还是启动层问题。`
