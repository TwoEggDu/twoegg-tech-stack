> 这一篇最重要的判断是：`最怕 Strip 的，不是"用了反射"这件事本身，而是你的真实依赖图只在运行时才出现；越能在构建期把依赖写实、列明、生成出来，越适合被裁剪。`

前两篇我们先把 `managed stripping` 的地图和档位拆开了，第三篇又把 Unity 当前会自动识别的五条保留通道讲清楚了：

- 场景里的 managed type
- 序列化图里的类型
- `UnityEvent.PersistentCall`
- `Assets` 下的 `link.xml`
- `[Preserve]` 及其派生保留属性

讲到这里，很多项目里真正会追问的问题其实已经不再是"Unity 有没有 strip"，而是：

`那我平时到底哪些代码写法最危险？又该怎么改成更适合裁剪的样子？`

这一篇就只回答这件事。

它不是一篇"劝你不要用反射"的文章，也不是一篇"所有动态系统都不该存在"的文章。

它想回答的是更工程化的判断：

`什么写法会让依赖关系只在运行时才暴露出来，从而天然更难被 UnityLinker 正确保留；而什么写法能把依赖提前变成构建期可见信息。`

## 这篇要回答什么

这篇文章主要回答五个问题：

1. 哪几类 Unity 常见代码模式最容易被 strip 误伤。
2. 为什么"危险"不在于 API 名字吓人，而在于依赖对构建期不可见。
3. `nameof`、`typeof`、`[Preserve]`、`link.xml` 分别能帮到什么，帮不到什么。
4. 什么叫真正更适合裁剪的写法。
5. 如果项目已经很动态，工程上应该按什么顺序改。

如果先压成一句短答案，可以写成这样：

- 越依赖字符串、扫描、运行时拼装的代码，越容易让真实依赖落到 Unity 视野外。
- 越能把入口改成直接引用、显式注册、编辑器期生成、最小粒度保留，越适合 strip。
- Unity 当前源码已经给出一个很明确的信号：`它保的不是"任意动态语义"，而是"几条明确收集到的根"和"你显式声明别删的目标"。`

## 先给一张风险表

如果把常见写法按"构建期可见性"排一遍，大概可以先得到这样一张表：

| 写法 | 构建期能看见什么 | strip 风险 | 更适合裁剪的替代 |
| --- | --- | --- | --- |
| 直接代码引用、接口调用、显式字段引用 | 类型和调用边通常直接可见 | 低 | 保持直接引用 |
| 编辑器期生成注册表、显式注册清单 | 生成后的注册代码是静态可见的 | 低到中 | 推荐用于插件、消息、工厂 |
| 方法级 / 类型级 `[Preserve]`、精确 `link.xml` | 例外目标被显式声明 | 可控 | 推荐用于少量动态例外 |
| 字符串驱动反射 | linker 往往只能看见"你在调反射 API" | 高 | 改成显式映射、委托表、工厂 |
| 全程序集扫描、自动发现 | linker 能看见扫描器，但看不见最后会发现谁 | 高 | 编辑器期扫描并生成注册表 |
| 运行时拼泛型、配置驱动类名/方法名 | 真正目标直到运行时才确定 | 高 | 显式列出闭包组合和入口 ID |

这张表里最该看的不是 API 名字，而是第二列：

`构建期到底能不能看见你真正依赖的类型、方法和入口。`

只要这个问题答不出来，后面谈 strip-friendly 基本就会停留在"感觉上好像更安全"。

## Unity 当前源码，其实已经把方向写得很清楚了

这一篇后面很多工程建议，并不是拍脑袋来的，而是从 Unity 当前源码里能直接看见两条非常明确的实现信号。

第一条信号是：

`Unity 不会尝试理解任意运行时动态行为，它更倾向于先把几类已知入口整理成显式保留信息，再交给 linker。`

这些入口大致包括：

- 场景里已经出现的类型
- 序列化系统已经记录到的类型
- 需要额外保留的方法
- 用户显式声明的 `link.xml`

这说明 Unity 的工作方式更像是：

`collect known roots`

而不是：

`guess arbitrary runtime intent`

第二条信号是：

显式保留属性的粒度其实非常细。

它可以细到：

- `Method`
- `Class`
- `Field`
- `Property`
- `Constructor`
- `Interface`
- `Delegate`
- `Event`
- `Struct`
- `Assembly`
- `Enum`

而像 `[RuntimeInitializeOnLoadMethod]` 这类启动入口，本质上也带着显式保留语义。

这件事很值得记一下，因为它传达的设计信号非常直接：

`当 Unity 自己需要一个动态入口安全时，它也不是指望 linker 猜出来，而是给它显式保留语义。`

所以这一篇后面的工程判断，可以压成一句话：

`越像"已知根 + 显式契约"的写法，越适合裁剪；越像"运行时才临时发现目标"的写法，越危险。`

## 一、字符串驱动的反射，最危险的不是 `Invoke`，而是"目标只存在于字符串里"

先说最典型、也最容易出事故的一类：

- `Type.GetType(string)`
- `GetMethod(string)`
- `MethodInfo.Invoke`
- `Activator.CreateInstance(type)`

很多人一看到这些 API，就会本能地把"危险"归到 `reflection` 这个词本身上。

但从 strip 的角度看，更准确的说法其实是：

`危险的不是"用了反射"，而是你真正依赖的目标只存在于字符串或运行时变量里。`

比如下面这种路径：

```csharp
var type = Type.GetType(config.TypeName);
var method = type.GetMethod(config.MethodName);
method.Invoke(target, null);
```

对运行时来说，这段逻辑完全能说得通。

但对构建期来说，它通常只能看见：

- 你会调用 `Type.GetType`
- 你会调用 `GetMethod`
- 你会调用 `Invoke`

至于最后到底是哪一个类型、哪一个方法会被打中，目标要到运行时读完配置之后才知道。

这就意味着：

`真实依赖图直到运行时才形成。`

这类路径天然就是 strip 高危区。

### `typeof` 和 `nameof` 能帮一点，但别高估

很多人会把下面这种写法当成"已经安全了"：

```csharp
var method = typeof(Foo).GetMethod(nameof(Foo.Bar));
```

它确实比纯字符串更好，至少：

- `typeof(Foo)` 让类型名变成了静态可见信息
- `nameof(Foo.Bar)` 让重构时不容易把字符串写坏

但这里要非常小心一个常见误解：

`nameof` 主要提升的是重构安全，不等于自动提升 strip 安全。`

因为对 linker 来说，`nameof(Foo.Bar)` 本质上还是一个字符串常量。

`typeof(Foo)` 往往只能说明"Foo 这个类型你确实提到了"，却不自动等于"Foo.Bar 这个成员一定会被当成真实调用边保下来"。`

所以如果你真正依赖的是"某个只通过反射命中的成员"，那更稳的方式通常不是把字符串写得漂亮一点，而是：

- 直接调用它
- 或显式保留它
- 或把它变成一个可见的委托 / 注册表项

### 更适合裁剪的替代写法

如果这条动态路径其实是业务入口、命令派发、模块工厂，那我更建议把它改成：

- `ID -> 委托 / 工厂` 的显式映射
- `enum / 常量 -> 已知处理器` 的注册表
- `interface / abstract class` 的直接调用边

例如，与其让配置直接携带 CLR 类型名和方法名，不如让配置只携带稳定业务 ID：

```csharp
public static class CommandRegistry
{
    private static readonly Dictionary<string, Func<ICommand>> s_Factories = new()
    {
        ["login"] = static () => new LoginCommand(),
        ["pay"] = static () => new PayCommand(),
    };

    public static ICommand Create(string id) => s_Factories[id]();
}
```

这段代码不一定"更高级"，但它有一个对 strip 非常关键的好处：

`真实会被实例化的类型，已经提前在静态代码里列出来了。`

如果你就是必须走反射，那也别让"可反射目标集合"散在全项目各处，而应该：

- 集中放到一处注册表
- 给最小必要粒度加 `[Preserve]` 或 `link.xml`
- 明确这是一条"例外保留通道"，而不是默认机制

## 二、全程序集扫描和自动注册，本质是把"发现依赖"推迟到运行时

第二类高危写法，经常出现在所谓"优雅架构"里：

- `AppDomain.CurrentDomain.GetAssemblies()`
- `assembly.GetTypes()`
- 扫描特性后自动注册
- 容器自动装配
- 消息处理器自动发现
- serializer / binder 运行时扫描类型

这类系统的设计目标通常很合理：

- 少写样板代码
- 模块接入更方便
- 新类型只要打个特性就自动生效

但从 strip 的角度看，它的问题也非常集中：

`你把"谁会被用到"这件事，推迟到了运行时扫描阶段才决定。`

构建期能清楚看见的是：

- 有一个扫描器
- 有一些反射 API
- 也许有某个自定义特性类型

可它往往看不见的是：

- 扫描结果到底会包含哪些具体类型
- 这些类型里的哪个构造函数、哪个入口方法会被调用

于是就会出现一种很常见的错觉：

"类型明明就在工程里，为什么 build 后没有了？"

答案通常不是"Unity 莫名其妙发疯"，而是：

`这个类型的唯一使用方式，就是运行时被扫描器发现；而扫描器发现它之前，它可能已经先被判定为不可达了。`

### 更适合裁剪的替代写法

这类系统最稳的改法，不是完全放弃自动化，而是把：

`运行时发现`

改成：

`编辑器期发现，构建前生成静态注册代码`

也就是：

- 在编辑器工具里扫描一次
- 产出一个普通的 C# 注册表文件
- 构建和运行时只消费这个生成结果

这样做的好处是：

- 开发体验仍然可以保留"自动发现"
- 最终进入 player 的却是一份静态可见的注册表

从 linker 视角看，这就友好多了。

如果你暂时不想上代码生成，那至少也应该退一步，改成：

- 每个模块显式调用一次 `Register(...)`
- 或维护一个手写的 `Type[]` / 工厂列表

它看起来更笨，但在 strip 这个问题上，很多时候：

`笨一点，反而更稳。`

## 三、运行时拼泛型，危险点在于"闭包组合只有运行时知道"

第三类高危写法，是很多框架代码里很常见的一组 API：

- `MakeGenericType`
- `MakeGenericMethod`
- 基于运行时类型参数拼装闭包泛型

这类路径的问题，和前面两类其实是一回事，只是换了一种表现形式：

`真正会被用到的 closed generic 组合，不是在构建期确定的。`

例如某个 serializer、事件总线、命令系统，运行时才根据数据类型去拼：

- `Handler<PlayerState>`
- `Deserializer<MailItem>`
- `Invoker<SomeMessage>.Execute`

如果这些闭包组合从来没有以静态方式出现过，那构建期就很难把它们当成明确依赖。

这里要额外提醒一个边界：

`这一篇先只谈 strip 可见性，不展开 AOT 代码生成问题。`

也就是说，即使先不讨论 `IL2CPP` 下更复杂的泛型/AOT约束，单从"linker 能不能在构建期看见它"这个问题出发，运行时拼泛型本身就已经属于高危写法。

### 更适合裁剪的替代写法

如果业务上支持的泛型闭包组合其实是有限集，那更稳的做法是：

- 明确列出允许的组合
- 把组合写进静态注册表
- 或在编辑器期生成这些闭包实例的引用代码

例如，不要让运行时随便拿任意 `Type` 去拼：

```csharp
public static class MessageInvokerRegistry
{
    public static readonly Dictionary<int, Action<object>> Invokers = new()
    {
        [MessageIds.Login] = static payload => Invoker<LoginMessage>.Execute((LoginMessage)payload),
        [MessageIds.Mail] = static payload => Invoker<MailMessage>.Execute((MailMessage)payload),
    };
}
```

这类写法的核心价值不是"代码更短"，而是：

`所有真正会落到运行时的闭包类型，已经被提前摊开了。`

## 四、配置驱动入口和名字协议，表面像数据驱动，实质还是隐藏依赖

第四类高危写法，很多时候会披着"数据驱动"的外衣出现：

- JSON 里存完整类名
- 表里存方法名
- Lua / JS / 热更桥接层按字符串找托管入口
- 插件回调表按约定命名自动拼函数
- 某个系统规定"只要实现某名字的方法，就会自动生效"

它和第一类字符串反射很像，但工程危害通常更大。

因为第一类可能只影响一个函数调用，而这一类经常会影响：

- 初始化入口
- 插件扩展点
- 跨模块协议
- 线上配置热切换

一旦被 strip 误伤，问题经常不会立刻在主流程炸出来，而是：

- 某个开关打开后才出错
- 某张表下发某种配置才出错
- 某个插件包接入后才出错

这类问题最烦人的地方在于：

`表面上它们是"数据问题"，本质上却还是"依赖关系没被构建期看见"。`

### 更适合裁剪的替代写法

更稳的方向通常是把"配置里存类型名"改成"配置里存稳定 ID"，再由代码层做显式映射。

也就是：

- 配置侧存 `enum`、短 ID、资源键
- 代码侧维护 `ID -> handler`、`ID -> ScriptableObject`、`ID -> prefab`、`ID -> factory`

如果是 Unity 项目里很常见的资源驱动入口，那更适合裁剪的方式往往是：

- 让配置指向资源 key
- 让资源再直接引用脚本、Prefab、`ScriptableObject`

这样真实依赖更容易进入：

- 场景引用
- 序列化图
- 显式注册表

总之，真正要避免的是：

`把业务入口直接编码成 CLR 类型名和方法名字串。`

## 五、真正更适合裁剪的写法，通常有这五条

前面几类危险模式如果压缩一下，最后其实都会落回同一组工程动作。

### 1. 直接引用胜过字符串查找

能直接调用，就不要先转成字符串再找回来。

能通过：

- 接口
- 抽象类
- 委托
- 显式工厂

表达的依赖，就不要优先选：

- `Type.GetType`
- `GetMethod`
- `Invoke`

因为对 strip 来说：

`直接引用不仅更快读，更关键的是它真的形成了构建期可见的调用边。`

### 2. 编辑器期生成注册表，胜过运行时发现

如果你需要"自动注册"的开发体验，优先考虑把自动发现放到编辑器期。

这类场景包括：

- 消息处理器
- 插件模块
- 命令工厂
- serializer binder
- 某些初始化入口

最终进入 player 的最好是一份：

`已经展开的静态注册结果`

而不是：

`启动后再扫程序集、再猜哪些类型需要接入`

这里顺便提一句很实用的替代思路：

如果你的目的只是"项目启动时自动跑一段明确的方法"，那很多时候直接用 `[RuntimeInitializeOnLoadMethod]` 这种显式入口，会比"扫描自定义特性再反射调用"更稳。

原因很简单：

`前者在 Unity 当前源码里本来就是 Preserve 语义的一部分，后者通常不是。`

### 3. 最小粒度保留，胜过整程序集粗暴保活

很多团队一遇到 strip 问题，第一反应就是：

- 整个程序集写进 `link.xml`
- 或把 stripping level 一路往回调

这当然有时能止血，但它的副作用通常也很明显：

- 包体回涨
- 真实依赖边界继续模糊
- 后面没人知道到底哪些东西真的需要保

既然 `[Preserve]` 的目标粒度可以细到方法、字段、构造函数、属性，那更推荐的做法通常是：

- 小例外用方法级 / 类型级 `[Preserve]`
- 不能改源码的第三方库，用精确 `link.xml`
- 尽量不要一上来就整程序集保活

这条原则的核心不是"省几个 KB"，而是：

`让"为什么要保"保持清晰。`

### 4. 把例外集中管理，不要让保留规则散落

即使必须使用 `[Preserve]` 或 `link.xml`，也别把这件事做成灾难现场。

最糟糕的状态通常是：

- 某些方法上零散挂 `[Preserve]`
- 某些第三方类型藏在多个 `link.xml`
- 某些系统又靠降低 stripping level 兜底

最后团队只知道"这里不能动，一动就炸"，却不知道：

- 为啥要保
- 保到了什么粒度
- 还有没有更小、更清楚的做法

更稳的管理方式通常是：

- 给动态入口统一设一个注册表或 manifest
- 给 `link.xml` 统一存放位置
- 给保留规则写明对应系统和原因
- 把"显式保留"当成架构例外，而不是默认习惯

### 5. 在目标 stripping level 上做 smoke test

如果项目最后发布用的是：

- `Low`
- `Medium`
- 或更激进的配置

那验证就必须在这个级别上做，而不是平时一直用更松的配置开发，出包前再切过去碰碰运气。

最该优先覆盖的 smoke test 通常是：

- 启动初始化链
- 配置驱动入口
- 插件注册
- 反射调用最重的系统
- 运行时拼泛型的关键业务

因为这些地方最容易出现一种错觉：

编辑器里没问题，开发包里没问题，结果正式构建一上来就缺类型、缺方法、缺入口。

很多时候不是"正式环境更玄学"，而只是：

`你终于第一次在真实 stripping 条件下跑到了那条动态依赖。`

## 六、如果项目已经写得很动态，推荐按这个顺序改

真实项目里，当然不太可能一下把所有动态系统全推翻。

如果只能逐步改，我更建议按下面这条顺序来：

1. 先找"只有字符串名、没有显式根"的入口。
2. 再找"运行时扫描才能发现类型"的系统。
3. 然后梳理"运行时拼泛型"的闭包集合。
4. 对暂时改不掉的地方，加最小粒度 `[Preserve]` 或精确 `link.xml`。
5. 最后把验证固化到目标 stripping level 的构建和 smoke test 里。

这套顺序背后的原因很简单：

- 先处理最不可见的依赖
- 再处理最分散的发现机制
- 最后才处理保留补丁和验证

如果一开始就只会"先加大面积保留"，那项目虽然可能暂时不炸，但你其实是在把问题往后延。

## 这一篇最该带走的三句话

如果把这篇文章最后压成三句话，我建议记住这三句：

- 最怕 strip 的，不是"用了反射"，而是你的真实依赖直到运行时字符串、扫描结果或泛型拼装后才出现。
- 真正更适合裁剪的写法，不一定更炫，但一定更早把依赖变成静态可见信息，比如直接引用、显式注册、编辑器期生成注册表和最小粒度保留。
- Unity 当前源码本身就在提示同一个方向：它保的是已知根和显式契约，而不是替你猜任意动态行为。

下一篇，我们就把最后一层也补齐：

`Strip Engine Code 到底在裁什么，以及它为什么不是"对 libunity 再跑一次普通 strip"。`
