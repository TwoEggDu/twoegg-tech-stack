在 Unity 的资源体系里，AssetBundle 加密通常不是“防止泄露”的最终答案，而是“提高提取成本”的工程手段。它解决的是一个很具体的问题：当资源已经必须通过网络分发、必须落到客户端、并且其中又包含不能轻易被直接导出的内容时，如何让直接扒包、替换包、静态分析和批量提取变得更困难。

这篇文章只讨论边界清晰的工程问题：加密到底挡得住谁、放在哪一层、怎么做密钥分发、运行时解密会带来什么代价，以及什么时候这笔成本值得付。

## 先定义威胁模型

讨论加密之前，先把攻击者分层。否则很容易把“安全”理解成一个没有边界的抽象目标。

### 1. 被动抓包者

这类攻击者只拿得到网络流量，或者能下载到 CDN 上的文件。对于公开资源，如果没有额外保护，直接下载后就能离线分析、解包、重命名和提取。

对这类威胁，HTTPS 只是传输保护，不等于资源不可提取。传输层加密只能防中间人窃听，不能阻止客户端拿到明文资源。

### 2. 静态提取者

这类攻击者能拿到安装包、缓存目录、热更包和本地持久化文件，但不一定会运行你的游戏逻辑。AssetBundle 如果以明文形式落地，本质上就是可逆的文件格式。

对这类威胁，Bundle 加密能显著提高门槛，因为攻击者不能再直接靠工具链扫描文件头、读取资源表或批量导出。

### 3. 运行时对抗者

这类攻击者会运行客户端、挂 hook、截内存、改函数返回值、替换解密流程，甚至直接在解密后从内存里抓内容。

这也是很多人误判的地方：只要资源最终要被 Unity 使用，它就必须在某个时刻以明文出现。只要明文出现，理论上就存在被提取的可能。加密只能推迟和抬高成本，不能消灭风险。

### 4. 完整逆向者

这类攻击者会分析你的密钥来源、解密算法、资源加载链路、缓存策略和校验逻辑。他们关心的不只是“怎么解包”，而是“怎么稳定复现整条链路”。

如果目标是这一类攻击者，单独做资源加密远远不够，必须把完整性校验、密钥轮换、反篡改和发布治理放在一起考虑。

## AssetBundle 加密在加载链路中的位置

从工程上看，加密不是资源系统的起点，也不是终点，而是发生在“构建产物”和“Unity 可消费格式”之间的一层包装。

一个典型链路是：

1. 资源在打包阶段生成正常的 AssetBundle。
2. 构建后对 Bundle 文件做对称加密或混淆处理。
3. 客户端从 CDN 下载密文文件。
4. 客户端在运行时解密，恢复出可供 Unity 读取的 Bundle 数据。
5. Bundle 被加载进内存，后续按常规方式访问。

这意味着一个核心事实：如果 Unity 最终要用这个 Bundle，客户端要么拿到明文，要么自己先解密到明文。所谓“加密 AssetBundle”，本质是在“可分发文件格式”和“可加载数据格式”之间插入一道门。

## 常见加密与解密模式

不同方案的差别，不只是算法本身，而是解密发生在哪里、是否落盘、是否可缓存、是否可增量更新。

### 1. 整包加密，运行时一次性解密

这是最直观的方案。Bundle 以密文形式存储，加载前完整解密成一个临时字节数组或临时文件，再交给 Unity 读取。

优点很直接：

- 逻辑简单，容易接入现有资源流程。
- 对静态提取最有效，至少挡住“直接打开文件就能看”的路径。
- 和现有 CDN、版本号、hash 校验配合方便。

缺点也很明显：

- 峰值内存高，特别是大包会产生双份占用。
- 首次加载慢，解密耗时会直接体现在启动或首进地图。
- 如果落盘成明文临时文件，攻击面只是从“直接下载”变成“抓临时文件”。

适合中小型高价值资源包，不适合超大体积、强实时加载的包。

### 2. 流式解密，边下边解

这个方案通常更像管道处理：网络下载密文流，边接收边解密，再写入临时目标或直接供读取。

优点：

- 峰值内存更低。
- 大包更适合流式处理。
- 可以把下载和解密的部分成本重叠。

缺点：

- 实现复杂度高。
- 需要非常小心读取边界、块对齐、异常恢复和断点续传。
- 调试成本高，问题往往出在“文件完整但内容不可读”的中间态。

这个方案适合对性能比较敏感、且资源体积较大的项目，但对团队实现能力要求更高。

#### 落到 Unity API：LoadFromStream 与 CryptoStream

上面描述的"流式解密"在 Unity 里有一条明确的引擎级路径。Unity 2020.1+ 提供了 `AssetBundle.LoadFromStream(Stream)`，它接受一个 `System.IO.Stream` 子类实例。你可以实现一个自定义 Stream，内部包装 `CryptoStream`（通常配合 AES-CTR 或带块对齐 seeking 的 AES-CBC），然后把这个 Stream 直接传给 `LoadFromStream`。引擎按需从 Stream 中读取数据，解密只发生在实际被访问的块上，不需要预先解密整个文件。

这条路径把前面讨论的"流式逐块解密"从概念变成了可落地的实现：你不需要自己写文件管道或管理临时缓冲，引擎的读取调度直接驱动解密节奏。

把三种加载 API 放在一起看，它们对加密 Bundle 的处理方式完全不同：

- **`LoadFromFile`**：要求磁盘上的文件已经是可读的 Bundle 格式。如果 Bundle 被加密，必须先解密写回磁盘，再调用这个 API。这是"先解密落盘再加载"的模式，磁盘上会出现一份明文副本。
- **`LoadFromMemory`**：接受一个 `byte[]`，意味着你需要先把整个密文读进内存、解密成完整的明文 `byte[]`，再传给引擎。峰值内存里会同时存在密文和明文两份数据，对大包来说成本很高。
- **`LoadFromStream`**：按需读取，每次 Read 调用只解密当前请求的字节。内存占用最低，但每次读取都要经过一次托管层到原生层的跨越，有一定的调用开销。适合内存敏感的移动端大包场景。

选哪个 API，本质上是在"磁盘安全风险"、"内存峰值"和"调用开销"之间做取舍。

### 3. 只加密关键 Bundle

很多项目最后都会走向这个折中方案：不是所有资源都加密，只加密最值得保护的部分，例如：

- 稀有美术资源
- 角色立绘、皮肤、特效素材
- 活动内容
- 未公开版本的敏感资源
- 需要降低脚本化批量提取风险的高价值包

优点：

- 成本可控。
- 对性能和缓存影响小于全量加密。
- 能把安全投入集中在真正有价值的资产上。

缺点：

- 资源边界需要设计得很干净。
- 一旦高价值资源被拆得过碎，管理复杂度会迅速上升。
- 攻击者仍然可以从未加密资源里推断部分内容。

这是大多数项目更现实的落点。

### 4. 加密 + 签名双层方案

只加密不校验，容易让你忽略“资源是否被改过”。更稳妥的做法是把完整性校验放进发布链路里：

- 加密保证内容不可直接读。
- 签名或哈希保证内容没被篡改。
- 启动时或加载前做验证。

这个组合更接近工程上的“可信分发”而不是单纯“隐藏内容”。

## 密钥管理怎么选

资源加密真正难的地方往往不是算法，而是密钥。

### 1. 写死在客户端

最简单，也最危险。优点是开发快、依赖少、离线可用；缺点是密钥迟早会被逆向拿到。

如果你把写死密钥当成绝对保护，基本等于没有保护。它更像是提高一点自动化提取门槛，而不是提供真正的秘钥安全。

### 2. 启动时远程下发

客户端启动后向服务端请求密钥，或请求派生密钥参数。这样做能把密钥生命周期和版本绑定起来，也便于轮换。

优点：

- 可轮换、可回收。
- 能跟账号、区服、版本、活动期绑定。

缺点：

- 必须处理离线失败和弱网。
- 需要额外的鉴权、重试、缓存和容错。
- 一旦运行时抓到内存中的密钥，安全收益仍然有限。

### 3. 分段派生

不直接下发完整密钥，而是下发盐、版本因子、设备因子、账号因子等，再在客户端派生。这样能减少单点泄漏的直接价值。

但要注意：派生只是在“泄漏后是否还能复用”上做文章，不会改变客户端最终仍要持有可用密钥的事实。

### 4. 平台安全存储

如果目标平台支持安全存储，可以考虑把部分密钥材料放入系统 Keychain、Keystore 或等价能力中。

这类方案适合作为补强，不适合作为唯一依赖。它能提高提取门槛，但不能替代完整的资源链路设计。

### 5. 一包一密钥，或一版本一密钥

这是比较常见的工程折中：

- 一包一密钥，便于局部轮换和撤销。
- 一版本一密钥，便于版本隔离和发布治理。

它们都比“全项目一把钥匙”更稳妥，但也会增加密钥分发、缓存和调试成本。

## 运行时解密的工程代价

加密方案是否能落地，最终看代价是否可控。

### 性能代价

解密一定要花 CPU。区别只在于花多少，以及花在什么时候：

- 启动时一次性解密，首包体验会变差。
- 按需解密，峰值更平滑，但实现更复杂。
- 流式解密，延迟可控，但对 IO 和调度要求更高。

如果项目本来就受启动时间、首帧时间或切场景卡顿影响，资源加密会把问题放大，必须配合预加载和异步化一起设计。

还有一个容易被忽略的硬件因素：移动端的 AES 硬件加速并不总是可用。ARMv8 Crypto Extensions（硬件 AES 指令集）大约从 2016 年之后的芯片才开始普及，但大量低端 Android 设备仍然不支持。在没有硬件加速的设备上，纯软件 AES 解密一个 50MB 的 Bundle 可能需要 200-500ms，这个延迟在加载界面里会非常明显。iOS 方面压力更小——从 iPhone 5s（A7 芯片，2013 年）开始就支持硬件 AES，所以 iOS 设备基本不用担心这个问题。建议在项目最低规格目标机上实际跑一次 `System.Security.Cryptography.Aes` 的基准测试，或者通过 `SystemInfo` 检查设备能力，拿到真实数据再决定加密粒度。

### 内存代价

最容易被忽略的是双份占用：

- 密文缓存一份
- 解密后的明文再一份
- Unity 读取期间还可能有额外临时缓冲

大包或高并发加载场景里，这个成本会非常真实。你不能只看加密前后的文件大小，要看加载时的峰值内存。

### 磁盘与缓存代价

如果你把解密结果落盘，本地缓存就会出现两个问题：

- 缓存明文文件是否可被再次提取
- 缓存失效、版本回滚、清理策略是否和加密逻辑一致

很多资源加密最后失败，不是因为算法不行，而是因为缓存路径和版本策略没同步，导致明文临时文件、旧密钥文件和旧包并存。

### 调试代价

一旦 Bundle 被加密，很多传统排错方式会失效：

- 不能直接拿文件工具看内容
- 不能轻松复用现成提取器
- 出问题时需要先确认是“文件坏了”还是“解密坏了”

这会拉高排查时间。对大团队来说，必须准备专门的调试开关、日志和内部解密工具，不然加密会直接变成研发效率负债。

### 加密模式与 Unity API 对照

把前面讨论的加密模式、对应的 Unity 加载 API 和各自的代价放在一张表里，方便选型时直接对照：

| 加密模式 | Unity 加载 API | 内存特征 | 适用场景 |
|----------|---------------|---------|---------|
| 全文件加密，先解密再加载 | `LoadFromFile`（解密后文件） | 磁盘多一份解密副本 | 对安全要求不高、可接受临时明文落盘 |
| 全文件加密，内存解密 | `LoadFromMemory`（解密后 byte[]） | 临时双倍内存 | Bundle 较小、内存预算充足 |
| 流式逐块解密 | `LoadFromStream`（CryptoStream） | 最省内存，按需解密 | 大 Bundle、移动端内存敏感场景 |

实际项目中这三种模式不一定互斥。常见做法是对小包用 `LoadFromMemory` 快速解密加载，对大包用 `LoadFromStream` 控制内存峰值，对安全要求不高的通用资源直接 `LoadFromFile` 不加密。选型的核心变量是包体积、目标设备内存预算和安全等级。

## 这件事解决不了什么

这一节比“怎么做”更重要。

### 1. 不能阻止运行时提取

只要资源最终被客户端使用，它就有机会在运行时被读到。内存抓取、hook 解密函数、替换读取入口，都是现实存在的路径。

所以加密的真实目标应该是“提高提取成本”，不是“彻底不可提取”。

### 2. 不能替代签名和完整性校验

加密解决的是保密，不是可信。攻击者可以尝试：

- 替换资源包
- 篡改解密入口
- 注入伪造数据
- 利用旧版本包做回滚攻击

如果没有完整性校验，你可能只是把明文隐藏起来，但仍然接受了被篡改的数据。

### 3. 不能保护所有内容

图集、预制体、材质、贴图、音频、文本、脚本配置，各自的风险不同。某些资源即使加密了，最终在客户端还是会以较容易推断的形式出现。

更现实的做法是分级保护，而不是一刀切。

### 4. 不能替代发布治理

如果版本号、hash、CDN 缓存、回滚和密钥轮换没有一套统一治理，加密只会让问题更复杂。

资源安全不是孤立功能，它必须和版本发布、灰度、回滚、缓存清理、密钥轮换一起设计。

## 什么时候值得做

是否上资源加密，核心看价值密度，而不是看“能不能做”。

### 适合上加密的场景

- 资源有明确商业价值，且被直接提取会造成实际损失。
- 你面对的是大量自动化扒包，而不是顶级对抗者。
- 资源包数量可控，能做分级加密。
- 团队能接受额外的加载、缓存和调试复杂度。
- 资源更新链路本来就比较成熟，能同步处理版本、校验和回滚。

### 不适合上加密的场景

- 资源没有明显价值，或泄露影响有限。
- 项目对启动速度和内存极其敏感。
- 团队没有能力维护密钥体系和解密工具链。
- 资源更新频率太高，且包粒度太细，导致加密运维成本爆炸。
- 你希望它“绝对防提取”，而不是“降低提取率”。

如果目标是绝对保密，客户端分发模型本身就不支持这个目标。该接受这个现实。

## 一个可执行的决策框架

实际选型时，可以按下面四个问题判断：

### 1. 资源是否足够值钱

如果被提取后会直接影响商业收益、活动公平性或未发布内容泄露，那么值得考虑。

### 2. 攻击面主要是什么

如果主要威胁是静态下载和批量提取，加密收益很高。  
如果主要威胁是运行时逆向和内存抓取，加密收益有限，必须叠加反篡改和运行时防护。

### 3. 代价是否能被接受

检查四个成本：

- CPU
- 内存
- 启动和加载延迟
- 调试与运维复杂度

其中任何一个不可接受，都不要强行全量加密。

### 4. 能否分级落地

优先考虑：

- 高价值包加密
- 普通包明文
- 关键包签名校验
- 敏感版本轮换密钥

这比全量加密更符合多数项目的真实边界。

## 落地建议

如果你准备上 AssetBundle 加密，建议按这个顺序推进：

1. 先做好 HTTPS、版本治理、Hash 校验和回滚机制。
2. 只挑高价值资源做加密，不要一开始就全量覆盖。
3. 明确解密位置，是内存流、临时文件还是流式管道。
4. 把密钥轮换、失效和离线容错写进方案，而不是上线后补。
5. 准备内部调试工具，不然排障成本会很高。
6. 把“能防什么”写进安全边界文档，避免团队误判能力。

## 结论

AssetBundle 加密是资源分发治理的一部分，不是独立的安全答案。它能有效抬高静态提取和批量扒包的成本，但不能阻止客户端在运行时暴露明文，也不能替代签名、版本控制、缓存治理和回滚机制。

把它放回整体资产交付链路里看，最合理的策略通常不是“全都加密”，而是“对高价值资源做分级保护，并把密钥、校验和发布治理一起设计好”。这样你得到的是可维护的安全收益，而不是一个看起来很强、实际上很脆的技术幻觉。
