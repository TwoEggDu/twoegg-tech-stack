GPU 预算最容易被简化成一句话：

`画面太重了。`

这句话的问题和“主线程别超”一样，它太粗。

因为 GPU 预算真正处理的不是抽象的“重”，而是几个完全不同的成本桶：

- 像素成本
- 带宽成本
- RT 和中间缓冲
- 阴影
- 后处理
- 分辨率

如果这些桶没有拆开，团队就很容易在错误方向上反复试错。

## GPU 预算首先在管什么

我更愿意把 GPU 预算理解成：

`在目标分辨率、目标帧率和目标设备上，这一帧允许你花多少图形成本，以及这些成本主要应该花在什么地方。`

这至少要同时回答三个问题：

1. 哪部分成本和分辨率强相关
2. 哪部分成本和带宽强相关
3. 哪部分成本更像内容对象本身的复杂度

## 判断依据是什么

### 1. 帧预算是硬线

如果目标是 60fps，GPU 侧能吃掉的时间通常就不能长期超过 `16.67ms` 的总体预算。  
在真实项目里，GPU 预算还要和 CPU 预算一起共存，所以常常更紧。

### 2. 移动端首先受带宽和热约束

桌面 GPU 和移动 GPU 最大的区别，不只是强弱，而是：

- 移动端内存带宽更紧
- 功耗和热会持续收缩预算
- 全屏 RT 和后处理更容易变贵

所以很多 PC 上能随手开的效果，在手机上很快会变成持续掉帧和发热问题。

### 3. 分辨率是最大乘数

分辨率一上去，很多 GPU 成本是一起乘上去的：

- 全屏着色
- 后处理
- RT 读写
- 带宽

这也是为什么 `Render Scale` 经常是移动端最值钱的单个旋钮。

## 一套更稳的 GPU 成本模型

### A. 像素成本

这部分主要受：

- 分辨率
- 全屏 Pass
- 透明叠层
- Overdraw

影响。

典型例子是：

- 大面积透明特效
- UI 全屏叠层
- Bloom / SSAO / DOF

### B. 带宽成本

这部分主要受：

- RT 格式
- HDR
- 频繁读写中间缓冲
- 未压缩或不合适的纹理格式

影响。

移动端上很多“看起来只是多一个 Pass”的操作，实际首先吃的是带宽。

### C. 几何和提交相关成本

这部分更容易和：

- Draw Call
- 实例数
- 可见物数量
- 阴影投射体

相关。

它不一定像分辨率那样线性变化，但在复杂场景里很容易形成稳定负担。

### D. 特色系统成本

像下面这些通常应该单独记账：

- 阴影
- 反射
- 后处理
- 特效
- UI

因为它们往往是项目里“最贵且最可控”的几类视觉预算。

## 最常见的误判

### 1. “GPU 慢就是 Shader 太复杂”

不一定。  
很多 GPU 问题首先是：

- 分辨率太高
- RT 太多
- 阴影距离太长
- 后处理链太长
- 带宽被打穿

### 2. “只要 Draw Call 降了，GPU 就会好”

Draw Call 当然重要，但它不等于全部 GPU 问题。  
很多时候真正贵的是透明 overdraw 和全屏后处理。

### 3. “把效果都关掉就一定是对的”

不稳。  
GPU 预算不只在管“省”，还在管“怎么把最值钱的画面保住”。

例如：

- Color Grading 代价低但感知价值高
- Bloom 常常更适合降分辨率而不是直接关
- 阴影很多时候更适合换方案，而不是一刀砍掉

### 4. “高端机能跑，低端机掉帧只是正常现象”

如果目标设备池本来就包含这些低端机，那不是“正常”，而是预算和规格没有真正落地。

## GPU 预算怎样落到规则上

真正可执行的 GPU 预算，至少要写清：

- 目标分辨率和 Render Scale
- 阴影距离、级联、分辨率
- HDR / RT 格式策略
- Bloom、SSAO、DOF 等后处理的档位规则
- 透明特效和全屏效果的屏占比规则
- 材质分支和高价 Shader 的启用边界
