预算最容易死在最后一步。

很多团队的问题不是完全不知道该怎么立预算，而是：

- 预算写在文档里
- 例会上也讲过
- 大家口头都同意

但再过几周，它就开始失真：

- 新内容上线没人再回看规则
- 资源规格超了也没有自动提醒
- 低档配置和高档资源边界越来越乱
- 发版前才发现包体和性能一起涨了

所以预算要想活下来，就必须进入流程。

## 预算为什么一定会失真

如果预算没有进规则和门禁，它几乎一定会走向下面这条路径：

1. 立项期比较清楚
2. 中期开始出现例外
3. 例外没有回写规则
4. 规则慢慢变成参考，不再是约束
5. 预算退回到少数人脑子里

所以真正值钱的问题不是“预算文档写没写”，而是：

`它有没有进入团队每天都会经过的那几道关。`

## 预算至少要进入哪四层

### 1. 规格表

这是最靠近内容生产的一层。

至少应该有：

- 角色规格表
- 场景规格表
- 特效规格表
- 贴图与材质规格表

这层解决的是：

`预算怎样从总账变成日常可执行规则。`

### 2. 设备分档

如果项目本来就覆盖不同性能档位，那预算必须继续落实成：

- 高 / 中 / 低 / 极低档的差异
- 哪些内容跟档位联动
- 哪些内容全档保留

否则预算会天然偏向高配开发环境。

### 3. Baseline

规格表解决的是“该怎么做”。  
Baseline 解决的是“现在是不是已经偏了”。

至少要固定：

- 样本场景
- 设备组
- 构建模式
- 预算阈值
- 超线动作

### 4. CI / 发版门禁

只有到了这层，预算才真正变成约束。

例如：

- 包体涨幅超过阈值触发告警
- 关键场景帧时间退化触发阻断
- 低档机型专项样本失败不能过发版门

## 判断依据是什么

### 1. 工程治理事实

没有人能长期靠记忆维护预算。  
项目复杂度一上来，规则不进流程，就一定会失真。

### 2. 预算天然会被例外侵蚀

预算不是写完就结束，它会持续遇到：

- 活动需求
- 商业需求
- 技术债
- 档期压力

所以必须要有可追踪的例外机制，而不是口头放行。

### 3. 分档和门禁本质上是在保护预算

分档解决的是“不同设备允许花多少钱”。  
门禁解决的是“版本有没有偷偷多花钱”。

## 一套更稳的落地顺序

如果项目现在还没有这套体系，我会建议按下面顺序落：

1. 先把总预算和分账预算写清
2. 再补规格表
3. 再补设备分档
4. 再补 Baseline
5. 最后进 CI / Release gate

因为如果一上来就写 CI 阻断，而前面的规则和样本都不清，最后很容易得到一条没人信任的红线。

## 最常见的误判

### 1. “我们有文档，所以预算已经落地”

不对。  
文档只是第一步。

### 2. “我们有 QA 测试，所以不需要 Baseline”

也不对。  
人工测试能发现问题，但不能稳定发现趋势。

### 3. “分档是渲染配置的事，和预算门禁没关系”

不稳。  
如果分档没和资源规格、包体分层、特效边界一起走，预算最后还是会漏掉最大头。
