游戏开发者在做设备分级时，通常会把设备分成旗舰、高端、主流、低端四档，然后配置不同的画质预设。但"低端机用低画质"背后的原因是什么？低端机到底低在哪里？本篇从硬件层面逐项拆解。

---

## 制程工艺：能效比的根基

SoC 的制程工艺决定了在同等功耗下能集成多少晶体管，直接影响 GPU 核心数量和 CPU 频率上限。

```
制程工艺对比（以 2023-2024 年主流 SoC 为例）：

旗舰：台积电 3nm / 4nm（骁龙 8 Gen 3、Apple A17 Pro、Dimensity 9300）
  → 晶体管密度极高，同等面积集成更多 GPU Shader Core
  → 相同计算量下功耗更低（低发热 = 可以维持更高频率）

高端：台积电 4nm / 三星 4nm（骁龙 8 Gen 1/2、Dimensity 9200）
  → 性能接近旗舰，但功耗效率略低（尤其三星 4nm 的功耗问题在 Gen 1 上很明显）

主流：台积电 6nm / 三星 6nm（骁龙 7 Gen 1、Dimensity 8200）
  → GPU 核心数量减少，频率上限降低

低端：台积电 8-12nm / 三星 8nm（骁龙 6xx 系列、Dimensity 700）
  → 老制程，相同性能需要更高功耗，发热更严重，可维持时间更短
```

**关键推论**：低端 SoC 不只是"性能低"，还因为制程落后，导致相同功耗能做的事更少，更快触发降频。

## 内存规格：为什么有的内存快、有的慢

移动端内存使用 LPDDR 标准（Low Power Double Data Rate），数字越大、字母越新，带宽越高。

### LPDDR 各代规格对比

| 规格 | 频率（等效） | 单通道带宽 | 双通道带宽 | 工作电压 | 主要应用档次 |
|------|-----------|-----------|-----------|---------|------------|
| LPDDR4X | 4266 MHz | ~17 GB/s | ~34 GB/s | 0.6V | 低端 / 主流 |
| LPDDR5 | 6400 MHz | ~25 GB/s | ~51 GB/s | 0.5V | 主流 / 高端 |
| LPDDR5X | 8533 MHz | ~38 GB/s | ~77 GB/s | 0.5V | 高端 / 旗舰 |

### 带宽差距是如何产生的

LPDDR 的带宽由三个因素决定：

```
带宽 = 频率 × 总线位宽 × 通道数

以 LPDDR5X 为例（骁龙 8 Gen 3）：
  等效频率：8533 MHz
  总线位宽：32-bit（每通道）
  通道数：2（双通道）
  带宽 = 8533 × 10^6 × (32/8) × 2 = 约 68 GB/s
  （标称 77 GB/s 含效率修正系数）

以 LPDDR4X 为例（低端机）：
  等效频率：4266 MHz
  通道数：2
  带宽 = 4266 × 10^6 × 4 × 2 = 约 34 GB/s
  （部分低端机单通道，带宽减半至 17 GB/s）
```

**低端机的双重劣势**：频率低（LPDDR4X）+ 有时只有单通道，带宽可以低到 17 GB/s，是旗舰的 1/4。

### 内存规格为什么和散热有关

高频内存的代价是发热：

<!-- DATA-TODO: 内存功耗数值（0.3-0.5W / 0.5-0.8W）需要核对来源。建议：查 Micron / Samsung / SK Hynix LPDDR4X / LPDDR5X 的官方 datasheet（Active Power + IDD4R/IDD4W），在表下加一行"数据来源：Micron LPDDR5X MT62F datasheet 等厂商规格书，具体值随颗粒和电压略有差异"。 -->
```
内存功耗与频率的关系：
  LPDDR4X @ 4266 MHz：内存功耗约 0.3-0.5W
  LPDDR5X @ 8533 MHz：内存功耗约 0.5-0.8W（频率翻倍，功耗约增加 60%）

旗舰机的应对：
  → 更先进制程降低整体发热量
  → 更大面积的散热结构（均热板 / 石墨烯）为内存提供散热路径
  → 整机热设计功耗（TDP）更高，可以容纳更高的内存功耗

低端机：
  → 使用 LPDDR4X 本身也是为了控制功耗和成本
  → 即使换用高频内存，整机散热结构也无法支撑
```

## 内存容量：容量不足时系统会发生什么

内存规格决定了传输速度，内存容量决定了系统能同时在内存里放多少东西。

### 各档次的内存容量分布

<!-- DATA-TODO: "系统占用"和"留给游戏的可用内存"需要实测校准。建议：在各档代表机上跑 `adb shell dumpsys meminfo` 或 Android Studio Memory Profiler，启动系统后不打开任何 App 截一次，再打开 3-5 个后台常驻 App（微信、QQ、微博）后再截一次，取差值。截图存 static/images/perf/memory-baseline-by-tier/。暂未实测前建议在表下标"数据来源：基于 2024-2025 主流 Android 机型观察估算"。 -->
| 档次 | 典型内存容量 | 系统占用（Android + 后台） | 留给游戏的可用内存 |
|------|-----------|------------------------|----------------|
| 旗舰 | 12-16 GB | 约 3-4 GB | **8-12 GB** |
| 高端 | 8-12 GB | 约 3-4 GB | **4-8 GB** |
| 主流 | 6-8 GB | 约 2.5-3 GB | **3-5 GB** |
| 低端 | 3-4 GB | 约 2-2.5 GB | **0.5-2 GB** |

### Android OOM Killer：内存不足时的系统行为

Android 使用 OOM Killer（Out-Of-Memory Killer）管理内存压力，优先级从低到高依次终止进程：

```
Android 内存压力处理顺序（OOM Score 由低到高）：

1. 缓存进程（最先被杀）：已关闭的 App，在内存中保留以加快重新打开速度
2. 空进程：没有任何活动的后台进程
3. 后台服务进程：后台播放音乐、下载等
4. 可见进程：不在前台但用户能看到的（如分屏的另一个 App）
5. 前台进程（最后被杀）：用户正在使用的 App，即游戏本身

触发条件：当可用内存 < 系统设定的低内存阈值时，从 1 开始逐级终止
```

**对游戏的影响**：

```
场景：低内存机型（可用内存 ~1 GB），运行重度游戏

游戏启动后：
  → 加载场景，内存使用增长至 1.2 GB
  → 超出可用内存 → OOM Killer 开始杀后台进程

进入大场景时：
  → 需要额外加载 400MB 贴图
  → 系统内存不足 → OOM Killer 触发
  → 如果杀后台已无法释放足够内存 → 游戏本身被杀 → 闪退

旗舰机（可用内存 8 GB）：
  → 同场景内存使用 1.2 GB，剩余 6.8 GB
  → OOM Killer 不会被触发
  → 后台 App（微信、音乐）也不会被杀
```

### 内存容量影响纹理驻留

内存不足时，引擎会主动卸载不在视野内的资源，再次进入时重新加载：

<!-- DATA-TODO: "UFS 3.1 @ 1.5 GB/s 下 1 GB 贴图 = 约 0.7 秒"这个计算忽略了 I/O 调度、解压和 GPU 上传开销，实际会更慢。"黑屏 1-2 秒"、"瞬间切换"需要实测。建议：准备两个场景（各 ~800MB 贴图预算），分别在 4GB 和 12GB 内存机型上测切场景耗时（Unity Profiler 的 AsyncLoadingThread + GPU Upload），录一段视频存 static/videos/perf/scene-switch-low-vs-high-ram/。 -->
```
低内存机型的纹理流式加载问题：

进入新场景：
  → 引擎先卸载上一个场景的纹理
  → 加载新场景纹理（磁盘 → 内存 → GPU）
  → 加载时间：UFS 3.1 @ 1.5 GB/s，1 GB 贴图 = 约 0.7 秒

高内存机型：
  → 上一个场景的纹理可以保留在内存
  → 切换场景时只加载差异部分
  → 无明显卡顿

表现差异：
  低内存机：切场景时黑屏 1-2 秒，或出现贴图闪烁（低 Mip → 高 Mip 渐显）
  高内存机：瞬间切换，无感知
```

## CPU：大核数量与频率差异

| 档次 | 超大核（Prime） | 大核（Gold） | 小核（Silver） | Prime Core 最高频率 |
|------|--------------|------------|--------------|-------------------|
| 旗舰 | 1 | 3-4 | 4 | 3.2-3.4 GHz |
| 高端 | 1 | 3 | 4 | 2.8-3.2 GHz |
| 主流 | 0（用大核替代） | 4 | 4 | 2.4-2.6 GHz |
| 低端 | 0 | 2 | 6 | 1.8-2.2 GHz |

**游戏主要用 Prime Core 跑 GameThread**。主流和低端机没有超大核，GameThread 跑在频率更低的大核上，同样的逻辑帧预算（16.7ms）更容易超时。

## 存储：影响加载时间

<!-- DATA-TODO: 存储读取速度是厂商标称峰值，实际应用层速度通常只有 50-70%。建议：(1) 在表下标注"数据来源：JEDEC UFS 4.0 / 3.1 / 2.2 规范 + eMMC 5.1 规范，为顺序读取峰值，随机读写会显著降低"；(2) "2GB 安装包 eMMC 5-7 秒 / UFS 4.0 约 0.5 秒"需要实测——用一个真实的 2GB APK/OBB 包在各档机型上做 `adb install` 或 `cp` 实测，对比 adb logcat 里的安装耗时。截图存 static/images/perf/install-time-by-storage/。 -->
| 规格 | 顺序读取速度 | 典型应用档次 |
|------|-----------|------------|
| UFS 4.0 | ~4.2 GB/s | 旗舰 |
| UFS 3.1 | ~1.5-2 GB/s | 高端 / 主流 |
| UFS 2.2 | ~1.2 GB/s | 主流 / 低端 |
| eMMC 5.1 | ~300-400 MB/s | 低端 |

eMMC 和 UFS 4.0 的差距约 **10 倍**。一个 2 GB 的游戏安装包：
- eMMC：解压写入约需 5-7 秒
- UFS 4.0：约 0.5 秒

对于有大量 AssetBundle 动态加载的游戏，低端机的存储速度是加载卡顿的直接原因之一。

## 显示：刷新率与游戏帧率的关系

| 档次 | 典型屏幕刷新率 | 游戏可用帧率上限 |
|------|-------------|--------------|
| 旗舰 | 120Hz / 144Hz | 120fps（需游戏和引擎支持） |
| 高端 | 90-120Hz | 90fps 或 120fps |
| 主流 | 60-90Hz | 60fps |
| 低端 | 60Hz | 60fps |

**注意**：120Hz 屏幕只有在游戏实际能跑到 120fps 时才有意义。如果 GPU 只能维持 60fps，120Hz 和 60Hz 屏幕的体验没有区别。旗舰机的高刷新率需要 GPU 算力同步跟上，这也是为什么只有旗舰才有意义开 120fps 模式。

## 汇总：各档次硬件差距一览

| 维度 | 旗舰 | 高端 | 主流 | 低端 |
|------|------|------|------|------|
| SoC 制程 | 3-4nm | 4nm | 6nm | 8-12nm |
| GPU 算力 | ~2 TFLOPS | 1-2 TFLOPS | 0.5-1 TFLOPS | <0.5 TFLOPS |
| 内存规格 | LPDDR5X | LPDDR5/5X | LPDDR5 | LPDDR4X |
| 内存带宽 | 68-77 GB/s | 51-68 GB/s | 34-51 GB/s | 17-34 GB/s |
| 内存容量 | 12-16 GB | 8-12 GB | 6-8 GB | 3-4 GB |
| 存储 | UFS 4.0 | UFS 3.1 | UFS 3.1 / 2.2 | eMMC 5.1 |
| 屏幕刷新率 | 120-144 Hz | 90-120 Hz | 60-90 Hz | 60 Hz |
| Prime Core | 1 个 @ 3.3 GHz | 1 个 @ 3.0 GHz | 无 | 无 |

## 对开发者的意义

```
设备分级配置的正确思路：

不要只看 GPU TFLOPS：
  低端机的性能瓶颈可能是内存带宽（17 GB/s），而不是 GPU 算力
  同一个 Shader 在低端机上慢，原因可能是纹理采样触发了内存带宽上限

内存容量决定资源预算上限：
  低端机（可用内存 ~1 GB）→ 单场景贴图总量严格控制在 600 MB 以内
  旗舰机（可用内存 8 GB）→ 可以预加载相邻场景资源

存储速度影响加载策略：
  低端机用 eMMC → 减少动态加载频率，尽量在 Loading 界面一次性加载完
  旗舰机用 UFS 4.0 → 可以更激进地使用流式加载

刷新率分级：
  仅对确认能跑到 120fps 的设备开放 120fps 模式
  中端机默认 60fps，避免开了高刷新率反而因 GPU 不足导致帧率不稳定
```
