特效预算是最容易“做着做着就炸了”的那一类预算。

原因很简单：

它几乎天生就是多预算耦合对象。

一个技能特效，可能同时吃掉：

- CPU 模拟和发射管理
- GPU 透明 overdraw
- 纹理和 Flipbook 内存
- RT / Distortion / Bloom
- 包体和热更资源

所以特效预算不能只写一句：

`粒子别太多。`

## 特效预算首先在管什么

我更愿意把特效预算定义成：

`在一个技能或效果允许的时间窗口里，这个效果能占多少 CPU、GPU、内存和交付成本，以及它能占多大的屏幕和多长的时机。`

这至少要同时看：

- 发射器数量
- 最大活跃粒子数
- 屏占比
- 透明叠层
- 是否使用 RT / Distortion / 后处理
- 特效贴图和 Flipbook 规格
- 首次命中与常态命中的差异

## 判断依据是什么

### 1. 透明和屏占比是最大乘数

很多特效不是因为粒子数绝对值大，而是因为：

- 覆盖了大面积屏幕
- 多层透明叠在一起
- 还带 Bloom、扭曲或全屏特征

### 2. 技能时机比单帧截图更重要

特效很多时候不会单独出现。  
真正危险的是：

- 连招高峰
- Boss 大招
- 多人同屏
- 首次播放

所以特效预算不能只在编辑器里看一个孤立预览。

### 3. 特效常常既吃 CPU 又吃 GPU

例如：

- 发射器和脚本控制吃 CPU
- 透明叠层和屏占比吃 GPU
- Flipbook 和噪声贴图吃内存和包体

## 一套更稳的特效账本

### A. 结构预算

至少看：

- 发射器数量
- 子系统数量
- 脚本驱动量

### B. 渲染预算

至少看：

- 最大活跃粒子数
- 屏占比
- 透明叠层
- 是否使用 Distortion / RT / 全屏 Pass

### C. 资源预算

至少看：

- 特效贴图尺寸
- Flipbook 分辨率和帧数
- 材质变体数量
- 是否进首包

### D. 时机预算

至少看：

- 首次播放是否需要预热
- 技能高峰时同屏允许几个高规格特效
- 多人同屏时是否自动降档

## 最常见的误判

### 1. “活跃粒子数不过线，所以特效应该没问题”

不一定。  
如果屏占比太大、透明叠层太厚、Bloom 和扭曲还一起开，照样会炸。

### 2. “只要把粒子砍少就行”

很多时候更值钱的动作其实是：

- 降屏占比
- 减透明层
- 减 RT 和扭曲
- 换更便宜的表现方案

### 3. “特效质量只影响 GPU”

不对。  
发射管理、脚本控制、对象池、首次资源命中都可能先伤 CPU 和内存。

## 特效预算应该落成什么规则

至少要有：

- 技能级特效预算表
- 发射器和活跃粒子上限
- 屏占比和透明层级规则
- 高规格特效的同屏数量限制
- 特效贴图和 Flipbook 规格
- 首次播放预热与低档替代方案

很多项目最后真正稳定下来，不是因为特效都做小了，而是因为特效终于从“凭感觉做酷”变成了“在边界内做酷”。

## 结尾

所以如果只用一句话收束这篇，我会这样说：

`特效预算真正要管理的，不是单纯的粒子数量，而是发射结构、屏占比、透明叠层、资源规格和技能时机的组合成本；只有把这些因素一起放进账本，特效才不会在最想发力的时刻反过来打穿整帧预算。`
