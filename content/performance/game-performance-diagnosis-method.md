> 如果只用一句话概括这篇文章，我会这样说：性能优化里最贵的错误，不是“改得不够快”，而是“根本没判断清楚慢在哪，就开始动手改了”。

前面这个系列已经把几条最重要的线铺开了：

- 为什么某些操作会慢
- 一帧到底是怎么完成的
- 什么事不能在什么时候做
- 为什么大整文件通常比很多小散文件更稳
- 为什么读盘完成不等于资源可用
- 为什么内存不是够不够，而是行为稳不稳
- 这些问题在 Unity 和 Unreal 里通常怎么显形

到了这一步，最自然的下一篇就不该再补一个零散原理点，而应该把前面的内容收束成一套更能落地的东西：

`当你面对一个真实卡顿、掉帧、首次抖动、长玩后变差的问题时，到底该怎么判断。`

因为在开发现场，最危险的情况通常不是“不知道任何原理”，而是：

- 看见掉帧就先怪 GPU
- 看见切场景卡顿就只怪磁盘
- 看见 GC 尖峰就把一切都归结成内存不够
- 看见线程等待就立刻重写任务系统

这些动作都可能对，但如果顺序不对，最后就会变成一种很贵的假勤奋：

`改了很多，问题还在。`

所以这篇文章只做一件事：

`把 CPU bound、GPU bound、I/O bound、memory bound、sync bound、thermal / power bound 变成一套能用来排查的判断方法。`

## 为什么诊断永远比优化动作更值钱

优化动作本身当然重要。  
但在大多数项目里，真正稀缺的不是“会不会写快一点的代码”，而是：

- 能不能先把问题分对类
- 能不能先确定瓶颈在哪条链上
- 能不能先区分稳定瓶颈和偶发尖峰
- 能不能先判断这是代码问题、资源问题、时机问题，还是平台预算问题

很多人一上来就进入“解决问题”的状态。  
但在性能问题里，一个常见现实是：

`你以为自己在解决问题，其实只是换了一个地方继续花预算。`

比如：

- 以为自己在做异步，结果只是把阻塞从读盘换成了等待后台结果
- 以为自己在做内存优化，结果只是压低了总量，却让资源重进场更频繁
- 以为自己在做 GPU 优化，结果真正的问题其实是首次上传和首次命中

所以这篇文章的第一原则非常简单：

`先诊断，再优化。`

而“诊断”不是一句空话，它要能落到具体判断上。

## 先学会判断：你面对的是哪一类“慢”

在进入六类 bound 之前，先做一个更基本的分类。

很多项目一上来就问：

`这是 CPU 还是 GPU？`

这当然重要，但还不够。  
更早一步的问题其实是：

- 这是稳定慢，还是偶发抖
- 这是首次慢，还是每次都慢
- 这是长玩后变差，还是冷启动就差
- 这是某个交互点才出现，还是整场景都慢

这几个维度会直接改变你后面的诊断方向。

### 稳定慢

如果每次进入场景都慢、每帧都偏慢、整体表现长期压不住预算，这更像稳定瓶颈。

更可能优先落在：

- CPU bound
- GPU bound
- 持续的 memory / bandwidth 压力

### 偶发抖

如果大多数时候正常，只有个别帧突然爆，这更像尖峰问题。

更可能优先落在：

- 首次命中
- 生命周期尖峰
- sync bound
- GC / 分配器尖峰
- 某段后台工作在错误帧收口

### 首次慢

如果第一次进场景、第一次开 UI、第一次放技能慢，第二次明显变好，优先怀疑：

- I/O 链
- 依赖展开
- GPU 上传
- shader / PSO 首次命中

### 长玩后变差

如果刚开始还好，玩一段时间以后变差，优先怀疑：

- thermal / power bound
- 工作集波动
- 资源淘汰与重进场策略
- 长时间运行导致的分配行为劣化

这一步看起来很朴素，但非常关键。  
因为它能帮你避免一种常见浪费：

`明明是首次尖峰，结果你按稳定瓶颈去优化。`

## 一套更实用的总排查顺序

如果你现在面对一个真实问题，我建议先按这个顺序来，而不是直接扑向某个具体系统。

### 第一步：先固定复现场景

不要在模糊印象上诊断。

至少先把问题压缩成一句话：

- 什么时候发生
- 多稳定
- 第一次还是每次
- 在 PC 还是手机
- 是战斗、切场景、UI、流送边界，还是长玩后

没有稳定复现场景，后面的“分析”大概率只是猜。

### 第二步：先区分稳定瓶颈和偶发尖峰

这一步会直接决定后面的工具和思路。

- 稳定瓶颈更适合先看持续时间和整体预算分布
- 偶发尖峰更适合先找首次命中、同步边界和生命周期波动

### 第三步：先分 CPU 和 GPU 的主矛盾

很多问题最后都要落回 CPU / GPU 谁在主导预算。

先回答这些：

- CPU frame time 更长，还是 GPU frame time 更长
- 降分辨率以后有没有明显改善
- 降画质和关后处理以后有没有明显改善
- 问题是不是仍然长在主线程或 Game Thread

这里不是要求你立刻锁死结论，而是先建立第一层主矛盾。

### 第四步：再看是不是 I/O、memory、sync、thermal 在主导

很多问题不是纯 CPU 或纯 GPU，而是：

- I/O 链让 CPU 在关键帧收口
- memory 行为让帧时间波动
- sync 边界打断了原本的流水
- thermal / power 把原本勉强安全的方案放大成事故

所以前面的 CPU / GPU 分类，只是第一层，不是终点。

## CPU bound 怎么判断

### 典型现象

CPU bound 最典型的表现是：

- 主线程或 Game Thread 明显长
- 降分辨率帮助不大
- 关闭后处理或降低渲染质量帮助有限
- 某些交互点、对象激活点、系统更新点容易出尖峰

它常见的外观包括：

- 一整段逻辑、UI、对象激活拉长
- 切场景收口时 CPU 明显爆
- 不是 GPU 忙，而是 CPU 根本没来得及把这一帧准备好

### 优先看什么

先看这些问题：

- 主线程 / Game Thread 时间是不是稳定偏长
- 哪段 update、tick、生命周期回调在拉长
- 是否有大量对象在同帧被创建、激活或销毁
- 是否有后台工作最终回到这一帧收口
- 数据访问模式是否很碎，缓存局部性是否很差

### 常见误判

CPU bound 最常见的误判是：

`看见主线程长，就以为一定是某段纯逻辑算法太慢。`

很多时候不是。  
它也可能是：

- 生命周期尖峰
- UI 刷新
- 资源激活
- 后台任务收口
- 大量短命对象导致的后续成本

所以 CPU bound 不是“只看代码”，而是看：

`CPU 这一帧到底在替谁结账。`

### Unity / Unreal 里常见落点

在 Unity 里，更常见的是：

- Main Thread 尖峰
- `Update / LateUpdate`
- `Instantiate / Destroy`
- UI 刷新和生命周期回调

在 Unreal 里，更常见的是：

- Game Thread 压力
- 对象激活链
- Tick 体系
- Streaming 收口时的主线程工作

## GPU bound 怎么判断

### 典型现象

GPU bound 最典型的表现是：

- GPU frame time 明显长
- 降分辨率有效
- 降低特效、阴影、后处理、透明度或画质后明显改善
- 某些视角、某些镜头、某些高分辨率路径特别容易出问题

### 优先看什么

先看：

- GPU 时间是不是长期高于 CPU
- 降分辨率是否立刻有效
- 问题是否和画面复杂度、像素成本、后处理强相关
- 是否有高开销 pass、overdraw、带宽压力、复杂 shader
- 是否有大规模 GPU 上传或首次资源准备撞到关键帧

### 常见误判

GPU bound 最常见的误判是：

`看起来像渲染问题，就一定是场景画得太重。`

很多时候也未必。  
它还可能是：

- 首次纹理 / mesh 上传
- 首次 shader / PSO 命中
- CPU 提交和 GPU 执行节奏没对齐
- 某个关键帧把“资源准备 + 渲染执行”一起堆给了 GPU

所以 GPU bound 也要继续分：

- 是稳定的渲染负担过高
- 还是首次成本在关键时刻显形

### Unity / Unreal 里常见落点

在 Unity 里，更常见的是：

- 某些 pass 很重
- overdraw
- shader variant 首次命中
- 资源首次上传

在 Unreal 里，更常见的是：

- pass 和材质开销
- Render / RHI 路径压力
- PSO 首次命中
- GPU 侧资源准备撞上首次使用

## I/O bound 怎么判断

### 典型现象

I/O bound 最典型的表现是：

- 首次进场景慢
- 切场景或传送边界抖
- 第一次开 UI、第一次放技能、第一次看到某类资源时卡
- 第二次进入明显变好

这类问题经常带着一种很强的“首次感”。

### 优先看什么

先看：

- 这是第一次慢，还是每次都慢
- 问题是否和资源首次读取或 streaming 边界强相关
- 是大整块资源读，还是海量小文件碎访问
- 读盘之后是否还叠着解压、反序列化、依赖解析

### 常见误判

I/O bound 最常见的误判是：

`把所有首次卡顿都粗暴地叫成磁盘慢。`

这通常不够。  
更准确的判断要继续往后拆：

- 真的慢在读盘吗
- 还是慢在读盘之后的解压和解析
- 还是慢在对象创建和 GPU 上传

所以 I/O bound 在游戏里最好理解成：

`资源准备链前半段在拖你。`

### Unity / Unreal 里常见落点

在 Unity 里，更常见的是：

- Addressables / AssetBundle 首次卡顿
- UI 首开
- 首次技能和首次场景进入

在 Unreal 里，更常见的是：

- Asset Streaming 边界
- 首次进入 streaming 区域
- 首次资源使用前的准备链

## memory bound 怎么判断

### 典型现象

memory bound 最典型的表现是：

- 总量不一定立刻爆，但帧时间越来越不稳
- 切场景、开大 UI、进战斗时明显抖
- GC 尖峰
- 长玩后变差
- 同一批资源反复进出场

### 优先看什么

先看：

- 是容量逼近极限，还是行为不稳定
- 有没有高频临时对象
- 有没有生命周期尖峰
- GC、分配器尖峰是不是踩在关键帧
- 常驻与淘汰策略是不是来回抽搐
- 工作集变化是不是和体验恶化强相关

### 常见误判

memory bound 最常见的误判是：

`总量没爆，所以内存没问题。`

很多时候真正的问题不是“放不下”，而是：

- 分配过碎
- 释放过猛
- 工作集来回波动
- 生命周期变化集中发生

也就是说，memory bound 经常不是“仓库满了”，而是“工作台太乱了”。

### Unity / Unreal 里常见落点

在 Unity 里，更常见的是：

- GC
- 高频 `Instantiate / Destroy`
- UI 首开和切页抖动
- 临时对象风暴

在 Unreal 里，更常见的是：

- 对象激活链上的工作集波动
- streaming 内容进出场
- 长玩后的资源重进场和初始化波动

## sync bound 怎么判断

### 典型现象

sync bound 最典型的表现不是“谁特别忙”，而是：

`谁在等谁。`

常见现象包括：

- 某个线程本来不重，但某些帧会突然停住
- CPU、GPU 都没明显长期过载，但帧还是会爆
- Present、fence、readback 或某个等待点附近异常明显

### 优先看什么

先看：

- 某一帧是不是在等待后台任务结果
- CPU 是否在等 GPU
- GPU 是否在等资源准备
- 有没有主线程必须同帧拿到的异步结果
- 有没有不必要的同步点把原本可并行的链条串起来

### 常见误判

sync bound 最常见的误判是：

`只看哪个线程长，却不看它是不是在等。`

等待和执行是两回事。  
有些尖峰不是“做了很多”，而是“被迫停住了”。

这类问题如果被误当成纯 CPU 或纯 GPU 负担去优化，常常会越改越偏。

### Unity / Unreal 里常见落点

在 Unity 里，更常见的是：

- 主线程等异步结果
- 资源准备和首次显示撞在一起
- 看起来像主线程尖峰，但实际是等待边界

在 Unreal 里，更常见的是：

- Game Thread / Render Thread / RHI 节奏被打断
- Streaming 与首次使用之间的等待
- CPU / GPU 流水被同步边界串起来

## thermal / power bound 怎么判断

### 典型现象

thermal / power bound 最典型的表现是：

- 刚开始正常，玩一会儿以后变差
- 冷机和热机表现差异明显
- 同样场景，刚启动时能稳住，持续运行后开始掉帧和抖动

它在手机上最常见，在 PC 上也并非完全没有。

### 优先看什么

先看：

- 问题是否和运行时长强相关
- 是否只在热机状态下明显
- 频率、功耗、温度是否在收缩预算
- 是否存在“原本勉强安全”的路径，被热限制放大成明显卡顿

### 常见误判

thermal / power bound 最常见的误判是：

`代码没变，所以问题应该不在平台预算。`

但这类问题的本质恰恰是：

- 代码没变
- 预算变了

所以你看到的是同一段逻辑在不同温度、功耗和持续运行状态下，突然显得完全不像一回事。

### Unity / Unreal 里常见落点

在两个引擎里都一样，最常见的现象是：

- 手机上前几分钟正常，后面越来越不稳
- 首次成本、工作集波动、资源上传在热机状态下被明显放大

差别更多不在引擎，而在平台预算。

## 不要只诊断单一 bound，要学会看主矛盾和次矛盾

真实项目里，纯净的单一瓶颈并没有想象中那么多。

更多时候你看到的是这种结构：

- 主矛盾是 I/O / 首次资源准备
- 但它最后显在主线程尖峰

或者：

- 主矛盾是内存工作集波动
- 但它最后显在 streaming 边界的 CPU 和 GPU 同时抖动

又或者：

- 原本只是轻度 GPU 压力
- 但在热机状态下被 thermal / power 放大成稳定掉帧

所以诊断时要尽量形成两层结论：

1. 现在帧爆掉时，表面上是谁在超预算
2. 更底层的主导因素到底是什么

这两层不分开，结论就很容易半对半错。

## Unity 和 Unreal 的诊断入口为什么不一样，但方法论还是同一套

你前面让我把 Unity 和 Unreal 分成两篇写，原因其实就在这里。

两个引擎的现场入口确实不一样：

- Unity 往往更容易先看到主线程、GC、Instantiate / Destroy、Addressables、UI
- Unreal 往往更容易先看到 Game Thread、Render Thread、RHI、Streaming、PSO

但真正的诊断方法没有变。

你仍然要回答这些：

- 稳定慢还是偶发抖
- 第一次还是每次
- 谁在超预算
- 谁在等待
- 工作集是否在波动
- 平台预算是否在收缩

也就是说，引擎只是显形方式不同，不会改变底层分类本身。

## 一张更实用的快速判断表

如果你想把这篇文章压成一张脑内清单，我建议记下面这些“第一反应”。

### 看起来像 CPU bound 时

先问：

- 是纯逻辑，还是主线程在收系统总账
- 有没有生命周期尖峰
- 有没有后台任务这一帧收口

### 看起来像 GPU bound 时

先问：

- 降分辨率有没有用
- 是稳定渲染压力，还是首次上传 / 首次命中
- 是场景真的重，还是关键帧把准备工作堆给了 GPU

### 看起来像 I/O bound 时

先问：

- 是第一次慢，还是每次都慢
- 真慢在读盘，还是慢在读盘后的链条
- 是大块顺序访问，还是碎访问

### 看起来像 memory bound 时

先问：

- 是容量问题，还是行为问题
- 有没有短命对象风暴
- 有没有工作集来回抽搐

### 看起来像 sync bound 时

先问：

- 谁在等谁
- 等待是必要的吗
- 原本可并行的链条是不是被串起来了

### 看起来像 thermal / power bound 时

先问：

- 热机和冷机差异多大
- 是代码变了，还是预算缩了
- 哪条原本勉强安全的链在高温下最先失控

## 我的结论

如果把这篇文章压缩成一句话，我会这样总结：

> 性能诊断真正值钱的地方，不是背出 CPU / GPU / I/O / Memory / Sync / Thermal 这些名字，而是能在一个真实问题面前，先区分稳定瓶颈和偶发尖峰，再找出谁在超预算、谁在等待、谁在放大问题，以及这条链为什么偏偏在那个时机收口。 

理解这一点之后，你再去做优化，很多动作就不会再是“凭经验盲改”，而会变成一套更有胜率的工程判断。

你会更容易知道：

- 这个问题先该抓哪条链
- 这个现象更像哪一类 bound
- 这个尖峰到底是执行问题，还是等待问题
- 这个退化到底是代码问题，还是平台预算问题

当这些判断开始稳定下来之后，优化才真正从“救火”变成了“有秩序地减灾”。
