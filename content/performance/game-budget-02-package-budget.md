很多团队一提包体预算，第一反应都是一个数字：

- Android 不能超过多少
- iOS 首包不能太大
- 这一版比上一版涨了多少

这些数字当然需要。  
但如果包体预算只剩一个总量，它通常是不够用的。

因为交付成本真正包含的，从来不只是"安装包总共有多少 MB"，而是至少四笔不同的账：

1. `首包`
2. `全量包`
3. `热更 / 增量包`
4. `常驻资源`

把这四笔账混成一个数字，是包体预算最常见的起点错误。

## 包体预算首先在管什么

如果把整件事压成一句话，我会这样定义：

`包体预算真正要管理的，不是"每个文件都再小一点"，而是"不同类型的内容应该待在正确的交付层"。`

这就是为什么包体预算不该只由"瘦身技巧"组成。

它至少要先区分四个对象。

### 1. 首包预算

这笔账决定的是：

- 玩家第一次下载有多重
- 商店转化、更新接受度和首次进入门槛

它最适合承载：

- 核心逻辑
- 必要运行时
- 主入口所需资源
- 低档也必须存在的基础内容

### 2. 全量预算

这笔账管理的是：

- 项目总体资源规模
- 长期维护成本
- 版本演进后的总膨胀程度

它不会直接决定首日转化，但会决定后续内容生产会不会越来越重。

### 3. 热更 / 增量预算

这笔账处理的是：

- 每次更新玩家到底要再拿多少内容
- 活动、版本内容能不能按需下发
- 发布节奏会不会被补丁体积拖慢

如果没有这条线，项目很容易在活动期被增量包拖垮。

### 4. 常驻资源预算

这是最容易被漏掉的一笔账。

很多资源虽然不在首包里，但一旦下载后长期驻留、频繁复用、无法清理，它一样会持续占交付和磁盘成本。

所以"首包不大"并不天然等于"包体预算健康"。

## 为什么不能混成一个数字

如果你只看总量，会有三种典型误判。

### 1. 首包健康，但增量越来越重

这通常说明项目把压力延后了，没有真正解决分层问题。

### 2. 首包降下来了，但常驻内容仍然过重

这说明团队只是把内容从安装时挪到了运行后，并没有真正处理"谁该长期存在"。

### 3. 总量控制住了，但高档资源仍然普发

这说明分层边界没切对，低档用户还在为不需要的内容一起买单。

## 包体预算的判断依据是什么

包体预算不是拍脑袋的"别太大"，它至少有三类依据。

### 1. 交付事实

玩家真正感知到的通常不是"项目总共有多少资源"，而是：

- 第一次下载重不重
- 更新频率高时补丁重不重
- 存储空间紧张时是否容易被清理

所以首包和增量通常比全量更敏感。

### 2. 内容事实

不同资源类型天然适合待在不同交付层：

- 核心 UI、主角基础资源、主入口场景更适合首包
- 活动资源、地区差异资源、高档资源更适合差异包或按需下载

### 3. 工程事实

如果一个资源经常被改、版本耦合高、回滚复杂，它就不适合被混进一个过大的统一包里。

所以包体预算不仅是容量问题，也是版本治理问题。

## 一套更稳的包体分层法

我更建议把包体预算按这四层来拆：

### A. Managed 代码层

这层主要看：

- 托管程序集
- 反射和保留规则
- 热更和序列化保留需求

典型动作是：

- `Managed Stripping`
- 依赖裁剪
- 模块拆分

### B. 原生库与引擎层

这层主要看：

- ABI
- 引擎模块
- 第三方 SDK 的架构和功能模块

典型动作是：

- 平台架构裁剪
- 原生插件精简
- 不必要模块退回依赖层

### C. 资源层

这层主要看：

- 贴图、音频、视频、Mesh、场景内容
- 是否误进首包
- 是否有重复依赖和错误分组

典型动作是：

- 资源分组治理
- 压缩和规格治理
- 去重和共享边界调整

### D. 分发层

这层主要看：

- AAB / Split APK
- iOS On-Demand Resources
- Addressables / CDN / 差异包

典型动作是：

- 按平台、地区、档位分发
- 按入口和频率按需下载

## 包体预算最常见的假优化

### 1. 只在发版前做一轮全局清理

这不是预算管理，这是阶段性扫除。  
没有持续监控和分层规则，下一版通常会长回来。

### 2. 把所有高档内容继续留在统一首包

这不是"还没来得及分"，而是在让所有用户替少数高档设备买单。

### 3. 只盯压缩率，不看交付层

如果一个资源根本就不该进首包，你把它从 20MB 压成 10MB，也只是"错误地更小了一点"。

### 4. 用激进裁剪换运行时风险

Stripping 很有价值，但如果没有保留契约、回归验证和异常回写，它会把包体优化变成线上事故源。

## 预算怎样落到规则上

真正能执行的包体预算，至少要落成下面这些规则：

- 首包白名单和黑名单
- 高档资源是否允许进入首包
- 活动资源默认走首包还是差异包
- 资源改动超过阈值时是否触发审核
- 每版增量包允许涨多少
- 哪些模块必须按平台、语言、档位拆分

也就是从"一个数字"变成：

- 资源进入规则
- 例外审批规则
- 增量控制规则
- 构建告警规则

## 结尾

所以如果只用一句话收束这篇，我会这样说：

`包体预算真正要解决的，不是把所有内容都压得更小，而是先把首包、全量、热更和常驻这几笔账拆清，再让代码、资源和分发层各自只承载它们该承载的内容；只有这样，包体才不会每一版都重新失控。`
