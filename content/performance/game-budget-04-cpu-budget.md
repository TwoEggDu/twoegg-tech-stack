CPU 预算最容易被说成一句话：

`主线程别超。`

这句话并没有错。  
问题在于，它通常太粗了。

因为 CPU 预算真正要处理的，不只是“哪段代码最慢”，而是：

- 哪个线程在结关键账
- 这笔账是稳定超，还是偶发尖峰
- 它是不是落在玩家最敏感的时机
- 所谓“异步”到底是把工作挪走了，还是把等待挪走了

## CPU 预算首先在管什么

我更愿意把 CPU 预算理解成：

`在一帧允许的时间里，哪些 CPU 侧工作必须按时完成，以及它们能接受多大的平均值、峰值和等待成本。`

这至少要同时看三件事：

### 1. 关键线程时间

在 Unity 项目里，最先要看的通常还是：

- Main Thread
- Render Thread / 提交线程相关成本
- Job Worker 的并行结算情况

### 2. 尖峰时间

平均值看起来健康，不代表 CPU 预算真的健康。  
很多体验事故首先来自尖峰：

- 首次技能
- 切场景最后一拍
- UI 首开
- 大量对象激活
- GC

### 3. 等待和收口时间

很多 CPU 问题不是纯计算太重，而是：

- 等加载结果
- 等 GPU
- 等同步点
- 等主线程统一收口

所以“逻辑不重”不等于“CPU 没问题”。

## 判断依据是什么

### 1. 帧预算是硬约束

60fps 只有 `16.67ms`。  
30fps 也只有 `33.33ms`。

这意味着 CPU 预算不是“尽量快一点”，而是：

- 关键线程能不能稳定在预算内
- 尖峰是否会打穿帧边界

### 2. 玩家高敏时机更严格

同样是 `3ms` 的工作：

- 放在加载过场里也许可以接受
- 放在玩家输入反馈关键帧就可能很贵

所以 CPU 预算不能只看总量，还要看时机。

### 3. 异步不等于免费

后台线程当然能分担一部分工作。  
但如果主线程最终还要：

- 等结果
- 做对象激活
- 做资源挂接
- 做统一收口

那它依然会在关键时刻结账。

## 一套更稳的 CPU 预算拆法

我建议至少拆成四笔：

### A. 稳态逻辑预算

每帧稳定发生的：

- Update
- 动画
- AI
- UI 更新
- 常规系统推进

### B. 危险时机预算

只在特定节点出现，但最容易伤体验的：

- 首次资源命中
- 场景切换
- 批量实例化
- 生命周期剧烈变化

### C. 异步收口预算

即使后台做了，主线程最后仍要处理的：

- 结果挂接
- 状态切换
- 对象激活
- 可见性和渲染列表更新

### D. GC / 管理性预算

不是业务逻辑本身，却会稳定吃掉预算的：

- GC.Alloc
- 字典/列表扩容
- 空 Update
- 过多的桥接调用

## 最常见的误判

### 1. “平均帧率可以，所以 CPU 预算应该没问题”

很多项目的玩家感知问题不是稳定慢，而是尖峰抖。  
只看平均值，会把最伤体验的部分完全吃掉。

### 2. “已经异步了，所以主线程不该再有问题”

后台做完不等于主线程不用结账。  
很多所谓的异步，真正的问题是最后的收口时机不对。

### 3. “Profile 看到最贵函数，就是 CPU 预算的根因”

不一定。  
有时候最贵的是入口，不是主因；有时候真正的问题是对象生命周期和等待链。

### 4. “CPU 优化就是写更快的代码”

写更快当然重要。  
但很多项目更值钱的动作其实是：

- 减少每帧必须做的事
- 让工作换时机
- 拆批和分帧
- 去掉不必要的常驻更新

## 落地时应该有什么规则

CPU 预算真正要变成规则，至少要写清：

- 目标帧率下关键线程预算
- 高风险时机允许的尖峰上限
- 哪些工作必须异步，哪些不能落在交互关键帧
- Update / Tick / 常驻系统的注册规则
- 批量实例化、UI 首开、首次技能的预热或分帧策略
