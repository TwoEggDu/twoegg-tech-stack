很多团队会说：

`这一帧超了。`

这句话当然成立。  
但如果项目里只剩这句话，通常就还不够。

因为从预算管理角度看，一帧不是一个数字，而是一串不同的人在结的不同账：

- 逻辑时间
- 渲染提交时间
- GPU 时间
- 加载尖峰
- 同步等待

把它们混在一起，会直接导致责任不清和误判。

## 一帧为什么一定要拆账

如果你不拆，一帧所有的问题最后都会被翻译成同一句空话：

- "优化一下 CPU"
- "GPU 太满了"
- "加载有点卡"

但真实项目里更像下面这样：

- 主线程逻辑本身还行，真正超的是首次挂接和对象激活
- GPU 稳态也还行，真正出事的是切场景后的上传和 RT 重建
- 加载线程已经异步了，但主线程等结果的同步点把一帧打穿了

所以帧预算真正值钱的地方，不是告诉你"这一帧超了"，而是告诉你：

`谁在超，在哪超，为什么总是在这个时刻超。`

## 一帧账本最少要拆哪几项

### 1. 逻辑时间

这笔账管的是：

- Update / Tick
- AI
- 动画
- 物理
- UI 更新
- 常规系统推进

它更偏"世界状态推进"。

### 2. 渲染提交时间

这笔账不是 GPU 真正在画，而是 CPU 这边在：

- 做可见性和列表构建
- 设置状态
- 提交 Draw
- 收口渲染相关工作

很多人把它和 GPU 时间混成一件事，其实不是。

### 3. GPU 时间

这笔账才是 GPU 真正执行的成本：

- 几何
- 光照
- 阴影
- 后处理
- 透明
- RT 读写

### 4. 加载尖峰

这笔账平时可能不明显，但在下面这些节点会突然出现：

- 首次资源命中
- 场景切换
- 热更解压
- 后台切回

### 5. 同步等待

这笔账最容易被漏掉。

因为它看起来不像"工作"，更像"空转"。

但它同样吃掉帧时间：

- 等后台结果
- 等 GPU
- 等资源准备
- 等统一收口

## 判断依据是什么

### 1. 一帧是链路，不是单点

前面那篇 [一帧到底是怎么完成的：游戏里一个 Frame 到底在做什么]({{< relref "performance/game-performance-frame-breakdown.md" >}}) 已经讲过，一帧从来不是 `Update -> Render -> Present` 这么简单。

预算视角下，它更像一条接力链。

### 2. 不同账的修法完全不同

- 逻辑时间超了，可能要减常驻系统、分帧、换时机
- GPU 时间超了，可能要调分辨率、阴影、RT、后处理
- 加载尖峰超了，可能要预热、拆阶段、换交互时机
- 同步等待超了，可能要改结果收口和调度方式

所以不拆账，修法一定会乱。

### 3. 玩家感知的是整帧结果，不是内部归因

玩家不会关心你是逻辑超了还是等待超了。  
但团队必须关心，因为不同的超法需要不同责任人来处理。

## 最常见的误判

### 1. "这帧卡了，所以一定是 CPU 太重"

有时候并不是算得太多，而是在等。

### 2. "GPU 没超，所以渲染不是问题"

如果提交侧和资源准备侧很重，渲染相关问题仍然可能是主因。

### 3. "已经异步了，所以加载不该再伤帧"

很多时候真正伤帧的是主线程最后的挂接和激活。

## 帧预算怎样进入日常规则

真正能用的帧预算，不该只停在一次 Profiler 观察。

它至少要落实成：

- 关键场景固定口径
- 逻辑 / 提交 / GPU / 等待的分项记录
- 首次技能、切场景、首开 UI 的专项样本
- 冷机 / 热机、低档 / 高档的对照

这样你才能回答：

- 稳态是谁超
- 尖峰是谁超
- 哪笔账最容易在版本里反复回来

## 结尾

所以如果只用一句话收束这篇，我会这样说：

`帧预算真正值钱的地方，不是告诉你"这一帧慢了"，而是把逻辑、提交、GPU、加载尖峰和同步等待分别记账；只有这样，团队才可能知道该由谁处理、该怎么处理，以及为什么某些问题总会在同一种时机里反复出现。`
