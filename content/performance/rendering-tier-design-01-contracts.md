很多团队一开始做分档，写出来的都是配置表：

- High 开 HDR，Low 关 HDR
- High 开更多阴影，Low 少一些
- High 开后处理，Low 关后处理

这类表格的最大问题，不是写得不够细，而是它们没有回答一个更关键的问题：

`我们到底想让不同档位的玩家感受到什么？`

如果没有这句话，分档就会退化成“看预算哪里不够就砍哪里”，结果往往是高档机做出来的只是“更亮、更清晰、更贵”，低档机做出来的则变成“另一个版本的同一套资源”，最后谁都不满意。

这篇文章不讲具体的 URP 参数表，而是先把渲染系统分档最重要的两份合同写清楚：

- `体验合同`：每一档到底保什么
- `预算合同`：每一档到底花什么

这两份合同写完，后面的 URP 配置、Shader 门禁、内容分层和验证才有落点。

## 为什么先写合同

分档不是“尽量平均地删减画面”，而是“把体验目标换成可执行的边界”。

在项目里，最常见的错误有两个：

1. 只写预算，不写体验
2. 只写效果，不写代价

前者会让低档机被动挨砍，后者会让高档特性变成系统负担。真正可维护的分档体系，必须把这两件事绑在一起：

- 哪些体验不允许动
- 哪些特性可以加，但不能成为低档依赖
- 哪些地方可以降级
- 降级之后玩家还能不能认得出这是同一款游戏

这也是为什么我更愿意把它叫做“合同”，而不是“参数表”。

## 体验合同要先回答什么

体验合同的核心不是“高档更漂亮”，而是“低档也不能变味”。

你可以把它拆成三层：

### 1. 不可动体验

这部分是底线，任何档位都不能破坏。通常包括：

- 核心玩法的可读性
- 主角和关键敌人的识别度
- 伤害、受击、命中、躲避这些关键反馈
- 主流程 UI 的可读性和响应顺序
- 关键信息提示，比如危险区域、任务目标、可交互对象

这些东西一旦动了，低档机就不再是“画质低一点”，而是“理解规则都更难了”。

### 2. 可协商体验

这部分允许按档位缩放，但要保证“退下去以后还是同一款游戏”。

常见项包括：

- 远景细节
- 装饰层密度
- 局部阴影质量
- 次级粒子和环境特效
- 远处角色和场景的几何复杂度
- 次级后处理效果

这类体验可以降，但不能把原本的视觉语义一起拿走。

### 3. 档位增益

高档机新增的内容，应该是加分项，不是必须项。

例如：

- 更稳定的阴影边缘
- 更高的分辨率和更好的清晰度
- 更丰富的环境光和反射
- 更完整的后处理链
- 更高的粒子和材质质量

高档特性只能让体验更好，不能成为“低档缺了就看不懂”的前置条件。

## 一份体验合同应该长什么样

最实用的写法，不是把合同写成长篇说明，而是写成一个可执行的结构。

```yaml
experience_contract:
  low_tier:
    must_keep:
      - 核心玩法可读性
      - 主角/关键敌人识别度
      - 命中/受击/危险提示
      - 主流程 UI 可读性
    may_reduce:
      - 远景细节
      - 装饰密度
      - 局部阴影质量
      - 次级特效数量
    must_not_change:
      - 角色身份
      - 战斗反馈节奏
      - 关键信息层级
  high_tier:
    add:
      - 更高分辨率
      - 更完整后处理
      - 更稳定的阴影
      - 更丰富的反射和氛围
```

这份结构的价值在于，它能直接被拿去做验收和回归，而不是只在文档里看起来正确。

如果你已经在项目里讨论过“低档机不能变味”，可以顺手回看 {{< relref "performance/device-tier-experience-consistency.md" >}}；如果你想把这份合同变成可验收的边界，可以继续看 {{< relref "performance/game-performance-design-tier-acceptance.md" >}}。这两篇更偏外层分档治理，而这篇是在写渲染系统内部应该怎么签约。

## 预算合同要回答什么

体验合同决定“保什么”，预算合同决定“花什么”。

预算合同至少要覆盖五类对象：

| 预算对象 | 它控制什么 | 常见负责人 |
|---|---|---|
| 帧预算 | 目标帧率和单帧时长 | 渲染、TA、客户端 |
| GPU 预算 | Shader、RT、后处理、透明开销 | 渲染 |
| 带宽预算 | RT 读写、纹理采样、Resolve、Blit | 渲染、技术美术 |
| 内存预算 | 贴图、RT、变体、常驻资源 | 客户端、TA |
| 热预算 | 长时间运行后的稳定帧率 | 客户端、性能、QA |

如果只写“目标 60fps”，但没写预算对象，那这份目标是没法落地的。因为 60fps 可以靠不同方式达到，也可以靠不同方式被破坏。

## 预算合同应该怎么写

一个够用的预算合同，最好同时写清楚四件事：

1. `目标设备`
2. `目标帧率`
3. `目标分辨率`
4. `允许的降级方式`

可以把它写成这样的模板：

```yaml
budget_contract:
  target_devices:
    low: "代表低档机型"
    mid: "代表主力机型"
    high: "代表高端机型"
  target_fps:
    low: 30
    mid: 45
    high: 60
  target_resolution:
    low: "动态分辨率 + 低内部分辨率"
    mid: "接近原生分辨率"
    high: "原生分辨率或轻度超采样"
  allowed_degrade:
    - "降低阴影精度"
    - "降低后处理层数"
    - "减少装饰特效"
    - "关闭次级屏幕空间效果"
  forbidden_degrade:
    - "破坏核心反馈"
    - "破坏 UI 可读性"
    - "破坏角色识别"
```

这里最重要的不是数字，而是“允许怎么退”和“绝对不能退什么”。

## 低档保底，高档增强，分别应该保什么

这件事最容易被误解成“低档只保性能，高档只加效果”。实际上，低档和高档的职责完全不同。

| 档位 | 必须保住的东西 | 可以优先降低的东西 | 不该发生的事 |
|---|---|---|---|
| 高档 | 完整风格、完整反馈、完整层级 | 少量极端场景下的局部开销 | 主效果被明显砍掉 |
| 中档 | 视觉身份、核心反馈、主要信息 | 背景密度、次级动效、部分后处理 | 核心玩法不清楚 |
| 低档 | 可玩性、可读性、关键反馈、角色身份 | 装饰层、远景、次级特效、部分阴影 | 看起来像另一款游戏 |

低档不是“什么都能砍”，而是“先砍装饰层，不砍身份层和可玩层”。

这也是为什么很多项目在低档上最容易翻车：他们把“画面贵不贵”当成第一优先级，结果把身份信息和可读性一起砍掉了。

如果你担心“档位越来越多，维护成本反而更高”，可以再回看 {{< relref "performance/device-tier-hidden-cost.md" >}}。那篇讲的是分档本身的治理成本，这篇讲的是分档之前必须先写清楚的合同。

## 预算合同如何反向约束 URP

这份合同写完以后，URP 的配置就不应该再是“凭感觉开关”，而应该反向服从合同。

最直接的几个约束是：

- `URP Asset` 按档位拆开，而不是运行时拼命改单个参数，参考 {{< relref "rendering/urp-platform-02-quality.md" >}}
- `Depth Priming`、`Native RenderPass`、`Intermediate Texture` 是否开启，要看预算合同里写的是“保带宽”还是“保兼容”
- `Renderer Feature` 不能默认全开，必须先问它是否属于不可动体验
- `Shader Keyword` 和 `Variant` 不能无限扩张，必须和档位绑定，参考 {{< relref "performance/gpu-opt-06-urp-pipeline-config.md" >}} 和 {{< relref "rendering/unity-shader-variant-full-lifecycle-overview.md" >}}
- 内容侧的 LOD、贴图、包体、粒子密度，也必须和合同对齐，参考 {{< relref "rendering/urp-platform-05-content-tiering.md" >}}

换句话说，合同不是写给策划看的，也不是写给 QA 看的，而是写给整个渲染链看的。

## 一份可以直接拿去开会的最小合同

如果你现在就要在项目里落地，我建议先写一页纸，不要先写十页。

```text
项目目标
  - 高档机：完整体验 + 更高质量特性
  - 中档机：核心体验完整，次级效果可缩放
  - 低档机：可玩、可读、可认得出是同一款游戏

不可动体验
  - 核心玩法可读性
  - 关键角色识别
  - 命中 / 受击 / 危险提示
  - 主流程 UI 可读性

预算对象
  - 帧预算
  - GPU 预算
  - 带宽预算
  - 内存预算
  - 热预算

允许降级
  - 阴影精度
  - 远景密度
  - 次级粒子
  - 部分后处理

禁止降级
  - 身份
  - 反馈
  - 可读性
```

这页纸的作用，是让团队从一开始就知道：哪些东西是可以变的，哪些东西一旦变了就不再是同一个项目。

## 下一步怎么接

写完体验合同和预算合同之后，才轮得到下面这些工作：

- 用哪些指标判断这份合同是否成立
- 一条渲染链应该怎么组织
- 哪些特性进入高档，哪些特性只留给中高端
- 材质和 Shader 怎么受控
- 长期验证和线上回写怎么做

下一篇要解决的就是第一个问题：`怎么评价一套渲染系统到底健不健康`。
