> 这个入口面向美术。目标不是让你独自扛性能责任，而是帮你在提交前先完成第一轮自检，尽量把最容易返工的内容挡在联调之前。

如果你现在最关心的是贴图规格、材质数量、LOD 是否切得太晚、UI 是否过度叠层、特效是不是一上来就把预算吃光，这页比直接去啃底层渲染原理更合适。

先把边界说清楚：美术第一轮不需要先搞懂 `CPU bound`、`GPU bound`、`Draw Call`、`Overdraw`、`Culling` 这些术语。你更应该先回答 4 个问题：

- 这份资源会不会明显超出当前档位预算。
- 这份资源的风险，是我自己就能改掉，还是需要 TA / 客户端协作。
- 这份资源是不是在“看起来不重”的地方偷偷变贵了。
- 这份资源进项目后，会不会把返工留到联调、提测甚至上线阶段。

## 提交前先问自己 4 个问题

如果你只能先做一轮筛查，我建议按这个顺序问：

1. 这份资源最贵的地方是什么。
2. 这份资源最容易在哪一层出问题。
3. 哪些问题我能直接改掉。
4. 哪些问题必须找 TA 或客户端确认。

这 4 个问题比“看起来好不好看”更重要。好不好看是结果，能不能稳进项目才是提交判断。

## 贴图先查什么

贴图最容易出的问题，不是“有没有贴图”，而是规格和用途是否匹配。

先看这几件事：

- 主贴图是不是超了当前档位的常规规格。
- 法线贴图是不是比实际需要的还大。
- 远景、次级细节、遮罩图是不是被做得过细。
- UI 贴图是不是被错误地当成可降级资源处理了。
- 特效贴图是不是一口气塞了太多细节和透明边。

如果你手上这张图只是“看起来精致”，但实际上只会在远处出现，那它通常是最先该降的内容之一。反过来，角色主视觉、核心 UI、关键武器高光这些地方，往往更值得保住。

你自己先能做的动作是：

- 重新确认这张图在镜头里的实际占比。
- 把远景图、近景图、细节图分开，不要混在同一规格里。
- 主动收敛无意义的细节，减少“只是在源文件里看着很爽”的信息量。

这类问题如果涉及整套贴图分层、档位规格、平台压缩格式，就该找 TA 或客户端一起看。可以直接回到 [每档资产规格清单]({{< relref "performance/device-tier-asset-spec-texture-and-package.md" >}}) 对照。

## 材质先查什么

材质的问题，表面上常常只是“换了个效果”，实际上可能已经把渲染成本拉高了一截。

先查这几件事：

- 同一类资源是不是用了太多不同材质。
- 一个效果是不是拆成了很多层透明和混合。
- Shader 关键词是不是变得越来越多。
- 材质参数是不是在不同对象上反复复制。
- 这个材质是不是要求了太高的阴影、后处理或实时光照参与。

美术自己最容易先改的是：

- 合并能合并的材质变体。
- 减少不必要的透明层数。
- 不把“为了好看”变成“每个对象都单独一套材质”。

如果一个材质改动已经开始影响 shader 变体、渲染链路或分档策略，就不要只靠美术单人判断了，最好让 TA 一起看。很多问题不是不能做，而是不能在默认预算里做。

## LOD 和场景资源先查什么

场景、植被、角色、建筑这类资源，最怕的是“近看很好，离远也没降下来”。

先查这几件事：

- 这个资源有没有合理的 LOD 层级。
- LOD 切换是不是太晚了。
- 远景有没有继续使用近景同级别的网格和贴图。
- 植被和密集装饰是不是在远处还保留了过多密度。
- 这个资源是不是会在镜头里长期占屏，但其实没必要一直保留高细节。

你自己先能做的动作是：

- 把近景、中景、远景的视觉重点分开。
- 远景优先保轮廓，不要死保细节。
- 对反复出现的大量资源，先想“能不能按档位缩下去”，再想“能不能再加细节”。

如果资源已经接到具体的 LODGroup、剔除规则、分档规则里，就需要和客户端程序一起确认，因为这已经不是单纯的美术资产问题了，而是资源和运行时边界一起在变。可以回看 [LOD 与 Culling 系统]({{< relref "rendering/unity-rendering-supp-c-lod-culling.md" >}})。

## UI 先查什么

UI 最容易出问题的地方，不是单个图标，而是整个页面的层次、刷新和叠加。

先查这几件事：

- 这个页面是不是把太多元素塞进了同一个 Canvas。
- 有没有频繁变化的内容和静态内容混在一起。
- 有没有全屏半透明遮罩、阴影、描边、模糊层反复叠加。
- 图标和小图是不是已经进图集，还是还在单独出图。
- 这个界面是不是一打开就要重建很多内容。

你自己先能做的动作是：

- 把背景、框架、动态内容分层。
- 控制无意义的半透明叠层。
- 让同类图标尽量共享图集。
- 不要让一个功能页因为“好看”就堆出三层以上的重叠视觉。

UI 问题经常不是美术一个人能独立收口的，因为它往往会牵涉 Canvas 拆分、动态刷新逻辑和界面结构。这个时候就该和客户端程序一起定边界。具体原理可以对照 [UI 渲染：Canvas 合批、Rebuild、Overdraw、Atlas]({{< relref "rendering/unity-rendering-supp-d-ui-rendering.md" >}})。

## 特效先查什么

特效最常见的误判是“粒子看起来不多，所以应该不贵”。实际项目里，真正贵的常常不是粒子数量本身，而是透明层叠、阴影、后处理和屏幕占比。

先查这几件事：

- 粒子是不是太大、太亮、太密。
- 透明叠加是不是太多。
- 特效是不是同时开了阴影、软粒子、后处理或者屏幕空间效果。
- 这个特效是不是被做成了“近看很好，但任何一档都很重”。
- 是否有大量同类型特效可以共用一套规则或图集。

你自己先能做的动作是：

- 先减叠层，再减面积，再减细节。
- 能用简化版本表达的，不要一上来就堆复杂版本。
- 重要技能保主视觉，低优先级装饰先砍。

如果一个特效已经牵涉到粒子 Mesh、投射阴影、透明排序、后处理链路或者自动检查，就不要只靠感觉判断了。像 [特效性能检查器案例]({{< relref "projects/vfx-checker-case.md" >}}) 这种场景，本质上已经是资源门禁，不是单点美术效果。

## 这些问题我自己先改，哪些找别人

为了减少来回沟通，我建议你把问题分成两类。

### 我自己先改

- 贴图分辨率明显过高。
- 法线、遮罩、细节图没有必要那么大。
- 材质层数过多，透明叠加太重。
- LOD 明显切得太晚。
- UI 里不必要的层级和遮罩太多。
- 特效叠层过厚、面积过大、信息量过多。

### 需要找 TA / 客户端一起看

- 材质开始影响 shader 关键词、渲染路径或分档策略。
- UI 结构需要调整 Canvas、刷新逻辑或合批方式。
- LOD 和剔除规则需要和运行时联动。
- 特效需要接入统一的检查器、质量门禁或分档规则。
- 资源需要按平台、档位或包体分层交付。

这个划分很重要。美术最容易吃亏的地方，不是做不出效果，而是把所有问题都当成“我再调一下就好了”。有些问题确实可以自己改，但有些问题已经不是单张图、单个材质、单个 Prefab 能解决的。

## 提交前最小清单

如果你只想在提交前快速过一遍，我建议用下面这 6 条做最小清单：

1. 这份资源的主成本点是什么。
2. 这份资源在当前档位下是否合理。
3. 贴图、材质、LOD、UI、特效里有没有明显过重的地方。
4. 哪些问题我能自己先改完。
5. 哪些问题必须拉 TA 或客户端一起确认。
6. 这份资源会不会在联调时被反复打回。

如果这 6 条里有 2 条答不上来，就先别急着交，先把问题收口。

## 美术入口的边界

- 这页不要求你独自负责最终瓶颈定位。
- 它更关心提交前自检和高风险效果识别，而不是线程调度或运行时证据链。
- 第一轮先不要硬啃 `Draw Call`、`Culling`、`Pass`、`Shader` 这些术语；等你先知道“自己该保什么、该砍什么、哪里最危险”之后，再回来看它们会顺得多。
