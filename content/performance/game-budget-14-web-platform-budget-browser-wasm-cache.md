Web 平台最容易被误写成：

`就是把手游预算再压小一点。`

这不够准确。因为 Web 的真正容器不是操作系统直接交给你的运行时，而是：

- 浏览器页签
- JS 运行时
- WASM / 线性内存
- 浏览器缓存和 GPU 上传链

所以 Web 预算更像：

`下载预算 + 内存预算 + 浏览器容器行为` 的组合题。

## Web 预算首先在管什么

### 1. 首屏预算

玩家能不能尽快看到第一帧、第一屏、第一段可交互内容。

### 2. 运行时预算

- JS Heap
- WASM / 线性内存
- GPU 资源与纹理上传

### 3. 生命周期预算

后台页签、冻结、刷新、恢复时，项目还能不能稳。

## 为什么 Web 不能照搬原生移动端

### 1. 页签不是前台 App

浏览器会：

- 限速
- 冻结
- 回收
- 丢失上下文

### 2. 内存不只是一种

Web 项目至少要分清：

- JS Heap
- WASM / 线性内存
- GPU / 纹理资源
- 浏览器缓存

### 3. 首屏下载和缓存命中更关键

原生 App 可以假设已经安装完成；Web 往往要先过首屏这一关。

## 最常见的误判

### 1. 把下载优化等同于运行时优化

首包小不代表页内稳，页内稳也不代表首屏好。

### 2. 只看 JS Heap

很多项目真正贵的是：

- WASM 内存
- GPU 纹理
- 上传过程和中间缓冲

### 3. 忽略页签生命周期

只在“持续前台运行”的理想路径测，很容易漏掉：

- 切标签页
- 锁屏后恢复
- 浏览器回收后重进

## 一套更稳的 Web 预算拆法

| 预算桶 | 主要对象 | 典型风险 |
|-------|---------|---------|
| 首屏下载 | 首包、首资源批次、关键脚本 | 首开太慢、弱网直接流失 |
| JS Heap | 逻辑对象、UI、脚本运行时 | 长时膨胀、GC 抖动 |
| WASM / 线性内存 | 引擎内存、资源解码、中间缓冲 | 峰值顶穿、恢复失败 |
| GPU 资源 | 纹理、RT、几何、上传链 | 大纹理和 RT 直接拉高页内成本 |
| 生命周期 | 后台冻结、恢复、刷新 | 恢复后状态错乱或重建过慢 |

## 怎么落地

1. 单独写 `首屏预算`，不要和页内预算混表。
2. 把 `JS Heap / WASM / GPU` 分开观察。
3. 固定测：
   - 冷启动首屏
   - 热启动缓存命中
   - 切后台恢复
   - 弱网首进
4. 把浏览器和设备组合写进验证矩阵。

## 最短结论

Web 预算真正要面对的不是“网页比原生弱一点”，而是：

`浏览器本身就是预算系统的一部分，首屏、缓存、JS、WASM、GPU 和页签生命周期必须一起立线。`
