大多数关于机型分档的讨论都在讲怎么做得更细——更多档位、更精确的判断、更丰富的配置差异。这篇讲相反的问题：**分档分得越细，隐藏成本越高，什么时候停下来才是对的？**

---

## 每多一个档位，团队多付多少代价

分档不是免费的。每个档位都会在多个地方产生持续性的工程和运营成本：

### QA 设备矩阵

最直接的成本。三档需要测试高 / 中 / 低三类设备，四档要加极低档，五档再加 Ultra。

每个档位至少需要：
- 覆盖该档位的 2–3 台代表机型（iOS + Android 各一）
- 每次版本发布前全矩阵回归
- 每次配置变更后针对该档位的专项验证
- 线上问题时该档位的复现设备

一个四档方案的完整设备矩阵，在 iOS + Android + 华为设备下，合理配置大约需要 **12–16 台测试设备**，并且这批设备随着时间推移需要更新换代。

这不只是采购成本，更重要的是 **QA 工时的持续投入**。

### 美术资产变体

每多一个档位，部分资产需要独立的低档版本。

以角色主贴图为例：
- 三档：高档（2048）/ 中低档共用（1024）
- 四档：高档（2048）/ 中档（1024）/ 低档（1024 低质量）/ 极低档（512）

从三档升级到四档，不是"加一档配置"，而是**美术 pipeline 多一个出口**。这意味着：
- 每次角色更新，需要导出并验证额外一套贴图
- 打包流程要多处理一个变体
- 线上问题排查时，"是哪档出问题"的定位成本上升

当游戏有大量角色、频繁出新角色时，这个成本乘数效应非常显著。

### 线上问题的档位定位成本

当玩家报告"我的游戏画面有问题"时，客服和研发需要确认：这个玩家属于哪个档位？

三档时，这个判断相对简单——玩家说是小米 10，查一下是中档，缩小范围。

档位越多，这个定位越复杂。极低档和低档的边界有时候并不清晰（比如一台 Adreno 610 的设备判到低档还是极低档？），玩家投诉的画质问题如果跨档位边界，定位成本会大幅上升。

### 分支维护成本

代码里的档位分支会随时间增长。一开始可能只有几处 `switch (tier)`，半年后可能遍布整个渲染配置、资源加载和 UI 模块。

每个档位分支都是一个维护点：
- 新功能上线时，要确认在所有档位下的行为
- Bug 修复时，要确认修复在所有档位下有效
- 配置变更时，要确认不意外破坏某个档位的表现

从经验上看，**档位数量每增加一个，新功能的测试矩阵大约扩大 30–40%**，这个成本随着游戏功能复杂度增长而放大。

### 边界设备的识别抖动

这是一个经常被忽略的成本维度：**档位越多，边界越多，判档抖动的设备也越多**。

一台 Adreno 610 的机型，在三档方案里稳定被判成"低档"；但在四档方案里，它可能在"低档"和"极低档"之间反复横跳——因为不同启动时的 `SystemInfo` 返回值会有细微差异（驱动版本、系统剩余内存、当前温度），判档逻辑里任何基于这些信号的补充判断都会放大抖动。

抖动会带来两类隐藏成本：

1. **玩家感知**：同一台设备上，今天游戏"流畅"，明天"卡"，玩家会把这个体验归因为"游戏不稳定"而不是"判档抖动"
2. **定位成本**：客服收到"我的设备以前能开某个效果，现在不能了"的反馈时，很难快速判断原因——是玩家主动改了设置？是档位判定抖动？是服务器下发配置变更？

工程上常见的应对是给判档逻辑加 hysteresis（迟滞带）——一旦判到某档，短期内（比如 7 天或一个版本周期）不再因为边缘信号变化而切档。但这又增加了新的代码复杂度，也让"需要时主动升档"的路径变复杂。

**结论**：档位数超过 3 档后，边界数至少是 N-1 条；每新增一条边界，就多一批可能抖动的设备。评估"要不要加这一档"时，不应该只看"能覆盖多少设备"，也应该看"会引入多少新的边界设备"。

## 边际收益递减规律

增加档位的收益和成本不是线性的：

| 从 N 档升级到 N+1 档 | 收益 | 成本 |
|---------------------|------|------|
| 1 档 → 2 档（加低档）| 极高（覆盖了低端设备）| 低 |
| 2 档 → 3 档（加中档）| 高（中端机体验大幅提升）| 中 |
| 3 档 → 4 档（加极低档）| 中（覆盖边缘设备）| 高 |
| 4 档 → 5 档（加 Ultra 档）| 低（旗舰机画质提升）| 高 |

**前两次加档收益明显，之后边际收益快速下降，但成本持续累积。**

"加一档"的决策应该被当作一个投资决策：这个档位能覆盖多少真实用户、能提升多少体验、愿意为此付出多少持续的工程代价？

## 什么时候加档是合理的

加档有合理的条件：

**极低档（从三档扩展到四档）**

合理条件：
- 底线设备和低档的性能差距超过 2 倍（比如 Adreno 505 vs. Adreno 610）
- 这批极低档设备的真实用户占比超过 5%（东南亚、印度市场常见）
- 极低档和低档在相同配置下，稳定帧率差异超过 10fps

不合理条件：
- 极低档用户只占 1–2%，但产品要求"能运行"
- 这种情况更好的做法是：低档配置做保守一点，而不是增加一个专门的极低档

**Ultra 档（从三档扩展到高端以上）**

合理条件：
- 旗舰机用户占比高（比如日韩市场），且他们的 GPU 利用率在高档配置下确实很低（< 50%）
- 有明确的视觉提升空间（实时反射、软体积光、光追），且这些效果对游戏的核心体验有贡献

不合理条件：
- "旗舰机应该有最好的体验"是理由，但没有具体的功能支撑它
- Ultra 档的差异只是"更高分辨率的 Bloom"，这种差异不值得一个独立档位

## 什么时候应该合并档位

合并也是一个有效的策略，在以下情况下考虑：

**中低档合并**：如果你的中档和低档配置差异主要只是 Render Scale 和 Shadow Distance 的数值差异（没有功能开关的差异），它们可以合并为一档，用运行时动态调整数值来覆盖。这比维护两套配置更经济。

**极低档并入低档**：如果极低档用户占比 < 2%，且这批用户的设备在低档配置下帧率稳定在 25fps 以上，直接在低档配置里加入"30fps 锁帧"逻辑，不需要独立维护极低档。

**档位 → 运行时动态调整**：对于粒子密度、LOD Distance 这类连续值，不需要用"档位"来管理，用**运行时基于帧率的动态调整**更合理。帧率持续低于目标时自动降密度，高于目标时恢复——这消灭了一部分"需要中间档"的场景。

## 团队规模与档位合理数量

档位数量应该匹配团队的持续维护能力，而不是追求覆盖最多设备：

| 团队规模 | 客户端工程师 | 建议档位数 | 理由 |
|---------|------------|---------|------|
| 小团队 | 1–3 人 | 2–3 档 | 维护成本是主要约束 |
| 中型团队 | 5–10 人 | 3–4 档 | 有能力维护完整矩阵 |
| 大型团队 | 10+ 人，有专职平台组 | 4–5 档 | 可以承担复杂分支和设备矩阵 |

"团队有多少人专门负责平台适配和设备兼容"比"玩家有多少种设备"更能决定合理的档位数。档位数超出团队维护能力，长期后果是配置腐化——每次改动都只在高档测试，中低档配置逐渐失效，直到出了线上问题才被发现。

---

## 配置腐化：最隐蔽但最贵的一类成本

上一节末尾提到"配置腐化"，这个问题值得单独拆开。它不是一次爆发的事故，而是一条慢慢烂掉的时间线，在你发现之前代价已经积累了很久。

### 腐化是怎么发生的

大多数项目不是"决定让低档配置烂掉"，而是每次做功能决策时都选了更便利的那条路，累积下来就烂了：

- 新功能做完，只在开发机（高档）上验证通过就上了
- 某个低档配置下的崩溃 Bug 被标为 P2，排了几个版本都没修
- 美术换了一组新的材质，但低档的材质 fallback 没有同步更新
- 某台低档代表机坏了，QA 同事临时用一台中低档边界机替代
- 某次版本赶工，低档专项回归被跳过了一次

每一件单独看都不严重，但半年后回头看，项目的低档配置已经不在真实受控状态里。

### 腐化的典型表现

| 现象 | 背后的事实 |
|------|----------|
| "我们低档有专门配置，但最近没玩过低档机"| 低档配置已经进入"纸面存在"状态 |
| 低档配置的 `git blame` 上次改动在半年以上 | 没人维护 |
| 线上低档玩家的留存率明显低于中高档 | 体验已经显著变差，只是没人定期看数据 |
| 新美术上手项目时"不知道低档配置在哪里"| 低档配置没有 owner |
| 某次改动后才发现低档机崩溃，回头排查发现三个版本前就坏了 | 回归链没覆盖低档 |

这些信号单独出现都容易被当成"偶发问题"，叠在一起就是系统性腐化。

### 怎么检测

最简单的做法是季度级扫一遍，用一张检查单：

1. 每档代表机上都能正常启动并玩进核心玩法吗？
2. 各档位的关键场景截图和三个月前比有没有显著视觉差异？
3. 各档位的崩溃率、帧率中位数在最近三个月是平稳还是劣化？
4. 低档配置的代码和资产分支上，最近一次有意义的改动是什么时候？

如果四个问题里有两个答案不理想，这档的配置已经在腐化了。

### 怎么止损

发现腐化后，通常有两条路：

- **收紧档位数量**：把维护不动的档位合并进相邻档位（详见下一节）
- **指定 owner 并定期 ceremony**：给每一档指定明确的维护人，每月或每版本跑一遍"档位健康检查"，把低档当成一等公民来维护

两条路都要选一条——不选的话，腐化只会继续深化，直到出了线上事故才不得不重建。

## 一个快速评估框架

在决定"要不要增加一个档位"时，用这五个问题过滤：

1. **覆盖多少用户？** 这个档位能覆盖的真实用户 DAU 占比是多少？如果 < 3%，通常不值得独立维护。

2. **体验差距有多大？** 如果不加这个档，这批用户的体验是"偶尔卡"还是"根本玩不了"？后者才是加档的理由。

3. **有多少配置差异？** 新档位和相邻档位的配置差异有几个功能开关？如果只有数值差异（Render Scale 0.7 vs. 0.75），用运行时动态调整解决，不需要新档位。

4. **谁来维护？** 有没有人明确负责这个档位的日常验证和问题排查？没有负责人的档位，等于没有档位。

5. **代价是多少？** 增加这个档位后，每个版本的 QA 矩阵增加多少设备 × 多少时间？这个代价能接受多久？

如果五个问题里有超过两个答案是否定的，不加。

## 已经做多了的退路：从 N 档优雅回退到 N-1 档

评估后发现"不应该加"是一种情况。更现实的另一种情况是：**项目已经做到了 N 档，但运行下来发现维护不动，想缩回 N-1 档**。

这件事不能一键搞定，因为缩档会影响真实的线上玩家。按步骤来：

### 1. 先拿数据说服自己该缩

在动手之前，先看三组数据：

- **被合并档的实际玩家占比**：如果这一档只占 DAU 的 2% 以内，通常可以直接合并
- **被合并档合并到相邻档后的帧率预测**：用黄金场景在这一批设备上用相邻档配置跑一遍，看帧率是否仍然可接受
- **被合并档玩家的关键指标**：次留、付费率、Crash Free Rate——如果这批玩家本来就在流失，合并后"掉得再多一点"也可接受；如果这批玩家其实是稳定的，合并要更谨慎

如果三组数据都支持合并，继续。如果某一组数据不支持（比如"低档其实占 DAU 10%"），问题不在档位数，而在维护投入不够，不能通过合并解决。

### 2. 设计合并方案：升一档还是降一档

把原来的 N+1 档玩家合并到相邻档位时，有两个方向：

- **升一档**：原极低档玩家合并到低档。好处是他们能看到更完整的游戏体验；风险是部分设备可能扛不住，帧率会退化
- **降一档**：原中档玩家合并到低档。好处是肯定跑得动；风险是这批玩家会明显感觉"画质变差了"，流失风险高

大多数情况下，**优雅的合并方向是升一档**——低端设备少量帧率波动比中端玩家画质下降容易被接受得多。但需要提前用真实设备验证"升档后能不能稳定跑"。

### 3. 分阶段灰度，不要一次切换

合并上线不应该是一次发布完成的。推荐节奏：

- **阶段 1（5% 灰度，一周）**：只对被合并档的玩家生效新配置，观察崩溃率、帧率、投诉数量
- **阶段 2（30% 灰度，两周）**：如果阶段 1 无异常，扩大到 30%，继续观察留存和付费指标
- **阶段 3（100%）**：确认没有长尾问题后全量

灰度期间要保留回滚开关——一键能把这批玩家恢复回原档。

### 4. 代码层的合并不是删代码

档位合并不是删掉一个 `case`。建议的做法是：

- 判档逻辑里保留被合并档的识别（返回被合并到的档）
- 配置层的 `tier_config[N+1]` 文件保留存档，不要直接删除
- 只在实际消费档位的地方合并（渲染配置、资产选择、UI 判断）

这样如果未来需要回档（比如下一代低端设备又拉开了差距，需要重新做极低档），是把保留的 config 恢复使用，而不是从头重写。

### 5. 沟通而不是隐瞒

如果合并会让部分玩家的画质有感知变化，**最好在版本更新日志里说明**。玩家对"画质变差"的容忍度低，但对"因为某某原因做了调整"的容忍度明显高。隐瞒只会让玩家在论坛上自己揣测原因，结论往往比真相难看。

## 结论

分档的本质是一个资源分配决策，不是一个技术精度竞赛。

档位越多，覆盖越精准，但维护成本也越高。当维护成本超过了覆盖精准带来的收益时，多的那个档位就是负收益。

大多数中型游戏项目的合理档位数是 **3–4 个**，只有在以下两种情况下才值得做到 5 个：覆盖全球超低端市场的同时追求旗舰机极致表现，且有专职团队长期维护。

如果没有这两个条件，做好 3 档并让它们持续稳定，远比做了 5 档但只有 2 档在正常工作更有价值。
