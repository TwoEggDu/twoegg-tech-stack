小程序 / 小游戏平台最容易被误判成：

`就是更小、更弱一点的手游平台。`

这不够稳。

更准确的理解应该是：

`它们是宿主容器平台。`

也就是说，项目不仅要对自己的内容预算负责，还要对：

- 主包 / 首包
- 分包 / 子包
- 宿主容器运行时
- 桥接和平台 API 开销
- 后台切换和恢复策略

一起负责。

## 这类平台为什么要单独写预算

### 1. 首包和分包是第一道硬边界

很多小程序 / 小游戏平台首先不是死在运行时，而是死在：

- 主包过大
- 分包策略错误
- 首进下载链太长

### 2. 宿主容器会吞掉一部分现实余量

这意味着你看到的“设备有多少 RAM”，并不等于项目真能稳定占多少。

### 3. 生命周期更受宿主控制

后台切换、恢复、资源回收和上下文丢失，往往比原生更容易成为风险点。

## 预算至少要拆哪几笔

| 预算桶 | 主要对象 | 典型风险 |
|-------|---------|---------|
| 主包 / 首包 | 首次下载必须内容 | 首进门槛太高 |
| 分包 / 子包 | 按玩法或档位拆分内容 | 边界错误导致多次重下 |
| 运行时内存 | 逻辑、资源、纹理、缓存 | 宿主容器下余量更紧 |
| 桥接与平台能力 | 宿主 API、桥接通信、平台能力差异 | 高频调用和能力碎片化 |
| 生命周期 | 前后台、恢复、热更新、重进 | 状态恢复失败 |

## 常见误判

### 1. 只关心主包大小

主包过线当然会死，但分包策略错了也会让体验越来越差。

### 2. 只复用手游资源，不重切边界

如果基础资源、立绘、UI、特效都按手游默认值搬过来，通常会在首包和运行时双重失真。

### 3. 把容器当透明空气

宿主不是空气。它会带来：

- 额外运行时成本
- 生命周期差异
- 能力限制

## 一套更稳的写法

1. 先把 `主包 / 分包 / 运行时 / 生命周期` 分开建表。
2. 共同底线内容优先按主包和最低运行时来切。
3. 高档资源、低频玩法、活动资源尽量走分包。
4. 验证时固定测：
   - 冷启动首进
   - 分包首次下载
   - 前后台切换
   - 热更新后重进

## 最短结论

小程序 / 小游戏平台预算真正要处理的，不是“把手游再压小一点”，而是：

`在宿主容器里重新切首包、分包、运行时和生命周期边界；如果这一步不重做，原生移动端那套预算表通常会直接失真。`
