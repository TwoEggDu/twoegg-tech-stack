> 如果只用一句话概括这篇文章，我会这样说：资源做完了，不等于它已经可以安全进项目；美术最有价值的性能工作，往往不是事后救火，而是在提交前先把危险资源拦下来。

这篇不是写给 Profiler 工程师的，也不是写给想直接看 `CPU bound`、`GPU bound`、`Draw Call` 的读者。

它更像一张提交前自检表，目标很直接：

- 让美术先知道什么东西最容易把项目推到危险区。
- 让你在资源还没进主线前，就能识别明显的高风险信号。
- 让你知道哪些问题可以自己先改，哪些问题应该立刻去找 TA 或客户端程序确认。

我不会在这篇里展开精确的档位规格表，也不会解释底层渲染机制。  
这篇先回答一个更实用的问题：

`资源提交前先查哪几项，最能避免性能翻车？`

## 为什么美术需要一张提交前自检表

很多性能事故，在资源侧的最初形态其实并不吓人。

它们往往长这样：

- 一张看起来很正常的贴图，只是分辨率稍高一点。
- 一个看起来没几面的模型，但用了太多材质槽位。
- 一组看起来不算夸张的特效，但刚好会在战斗高峰叠在一起。
- 一个看起来很漂亮的 UI 弹窗，但首开时要一起拉起太多节点、贴图和动画。

单看其中任何一个，你都可能会觉得：

`这也没多重吧。`

真正的问题通常不在"单个资源绝对有多大"，而在：

- 它会不会和别的东西一起出现。
- 它会不会在玩家最敏感的时刻第一次出现。
- 它会不会把本来还安全的预算，一下子推到危险区。

所以美术要做的第一层判断，不是自己去定最终瓶颈，而是先做两件事：

1. 识别高风险资源。
2. 把危险信号尽量前移到提交前，而不是留到联调或线上才暴露。

这也是为什么我更建议美术先养成"自检表"习惯，而不是先啃底层术语。

## 贴图和材质先看什么

### 先看什么

这一组先看四件事：

- 贴图分辨率是不是和它在玩家视线里的重要性匹配。
- 透明、半透明、发光、软阴影这些效果是不是被滥用了。
- 一个资源用了多少材质槽位，是否真的有必要拆这么多。
- 主角、核心怪、关键 UI 和背景资源，预算优先级有没有搞反。

如果你只记一条，我建议记这句：

`高关注资源可以贵一点，低关注资源不能靠"都先拉满"来偷懒。`

### 危险信号

- 远景背景、地面、装饰物和主角贴图差不多大。
- 一个模型本身不复杂，但挂了很多材质球。
- 透明面、描边、叠层光效被当成默认做法，而不是特殊效果。
- 为了"保险"，把法线、遮罩、细节贴图都按高规格交了上来。
- 同一类资源在近景和远景没有明显的规格差异。

### 常见误判

最常见的误判是：

`文件不大，所以应该不贵。`

这不稳。因为资源进项目之后，真正影响运行时的不是只有磁盘文件大小，还包括：

- 运行时占用的纹理内存。
- 是否会触发更多材质切换。
- 是否会把透明叠层、阴影和后处理成本放大。
- 是否会在首进场景或首放特效时一起上传和初始化。

另一个误判是：

`单个资源没多夸张，所以应该没问题。`

也不稳。因为很多事故不是被"一个特别大的东西"打爆的，而是被"一堆看起来都不算大，但刚好会叠在一起的东西"打爆的。

### 该找谁确认

遇到下面几种情况，不要自己猜，直接拉 TA 或客户端程序一起确认：

- 透明、半透明、描边、软阴影、发光边缘要不要常驻。
- 一个角色或场景件为什么需要这么多材质槽位。
- 主角资源和背景资源的分辨率优先级怎么定。
- 这张贴图到底是该保分辨率，还是该换压缩规格。

精确的分档数字和压缩表，下一步可以接着看 [每档资产规格清单：贴图压缩、LOD 与包体分层]({{< relref "performance/device-tier-asset-spec-texture-and-package.md" >}})。

## 模型、LOD 和场景资源先看什么

### 先看什么

这一组先看的是"距离感"和"堆叠感"，而不是只看单个模型。

重点看：

- 这个模型主要给谁看，是近景焦点，还是中远景陪衬。
- 轮廓是不是复杂到远处也要保留高成本。
- 有没有为远近距离准备明确的 LOD 思路。
- 场景是不是靠大量重复小件、植被、装饰件和 crowd 堆出来的。

很多场景资源的问题，不是"单体太重"，而是：

`数量一上来，密度一上来，远近关系一混，就开始失控。`

### 危险信号

- 远景物体和近景物体在精度上几乎没区别。
- 植被、碎石、栏杆、装饰小件大量重复，但没有明显的降级思路。
- 近景看起来细节不错，远景看起来也还是同一套精度。
- 一个场景件为了局部花纹，做了很碎的轮廓和细节，但玩家根本不会近看。
- 场景里大量资产都"单看不重"，但组合后会让画面非常密。

### 常见误判

最常见的误判是：

`这个模型面数不高。`

这句话只回答了单体问题，没有回答项目问题。

项目里更常见的危险是：

- 一个不高的模型被摆了很多次。
- 一类不高的装饰件叠成了高密度场景。
- 一套本来适合近景的细节，被扩散到了整张地图。

另一个误判是：

`LOD 以后再补。`

这通常会让后面很被动。因为如果资源一开始就没有按远近关系和视觉优先级去设计，等场景搭完再补 LOD，很多时候已经不是补一个版本的事，而是要回头改整套资产组织。

### 该找谁确认

- 这类资源是不是必须近景保真，还是应该把预算留给主角和交互核心。
- 哪些场景组块、植被、crowd、细碎装饰一定要有明确的 LOD 和密度策略。
- 当前这类资源是"单体风险"，还是"数量风险"。
- 这个场景的预算问题该落到资源规格，还是该落到摆放密度和分档配置。

如果你已经明确问题落在规格和分档上，下一步读 [每档资产规格清单：贴图压缩、LOD 与包体分层]({{< relref "performance/device-tier-asset-spec-texture-and-package.md" >}}) 会更顺。  
如果问题更像"这东西为什么一多就贵"，那通常已经在往后面的机制解释文靠了。

## UI、特效和全屏效果先看什么

### 先看什么

这一组最重要的不是静态重量，而是：

`它们在什么时机一起出现。`

先看：

- UI 是不是会在首开、切页、叠层时突然拉起很多内容。
- 特效是不是会在战斗高峰、Boss 登场、结算瞬间集中爆发。
- 有没有大量透明全屏层、模糊背景、全屏特效常驻。
- 一个页面或一波技能，是否同时触发了 UI、特效、数字飘字和屏幕效果。

### 危险信号

- 首开 UI 时明显抖一下，第二次就好很多。
- 弹窗层数一多，界面就开始发闷、掉帧或发热。
- 特效单看不大，但一叠就立刻炸。
- 某些发光、模糊、屏幕遮罩、命中闪白看起来很轻，结果只要常驻就明显拖预算。
- 结算、进战斗、开技能、受击反馈这些时刻，所有东西都想一起抢镜头。

### 常见误判

常见误判之一是：

`UI 不是 3D 场景，应该没那么危险。`

这不对。很多 UI 问题伤体验非常直接，因为它们正好发生在高频交互时刻，而且首开、切页、重建、动画常常会叠在一起。

另一个误判是：

`粒子数量看起来不多。`

特效的风险从来不只看"有多少个粒子"，还要看：

- 它是不是透明叠层。
- 它是不是覆盖大面积屏幕。
- 它是不是会和别的特效、后处理、受击反馈一起出现。
- 它是不是首放时还要做一批首次准备。

### 该找谁确认

- 复杂 UI 的首开时机和展示顺序要不要拆阶段。
- 某些模糊背景、透明遮罩、屏幕特效能不能降级或换方案。
- 战斗特效、命中反馈、后处理是不是叠得太满。
- 某个效果是"必须保"，还是"可以换更便宜的表达"。

这一组如果已经出现"首开卡一下""首放卡一下"的信号，建议顺着看 [什么事不能在什么时候做：游戏开发里最危险的时机管理]({{< relref "performance/game-performance-dangerous-operations-timing.md" >}})。

## 哪些东西"看起来不重"，其实最容易出事

这部分最关键，因为它决定美术能不能建立正确的成本直觉。

很多资源之所以危险，不是因为它一眼看上去就很大，而是因为它们特别容易伪装成"应该没问题"。

### 透明不是天然高贵，它只是很容易被滥用

透明、半透明、柔边、叠层光效很容易让画面立刻变漂亮，所以团队最容易在这里过量。

危险不在"用了透明"，而在：

- 透明被当成默认做法。
- 透明会大面积铺屏。
- 透明会和别的透明叠在一起。
- 它偏偏还出现在特效高峰或复杂 UI 叠层时。

### 阴影和后处理最容易被低估成"就一个开关"

很多美术会天然把阴影、Bloom、模糊、景深、屏幕空间效果理解成：

`开了更好看，不开差一点。`

但在工程上，它们往往不是一个"轻量装饰项"，而是很容易吃掉整块预算的东西。

更关键的是，它们经常会和别的东西一起出现，所以成本不是独立增加，而是叠加放大。

### "单个不重，组合很重"是项目里最常见的真实风险

一个特效不重。  
一个装饰件不重。  
一个透明框不重。  
一个角色外发光也不重。

但真实项目的危险常常是：

`同一时刻它们一起出现。`

所以美术在自检时，最好别只问：

`这个资源本身贵不贵？`

而要多问一句：

`它会不会出现在最容易叠层、首发、首开、同屏最满的时刻？`

### 首次出现，比持续出现更危险

有些资源平时并不贵，但第一次出现特别危险。

比如：

- 第一次打开某个重 UI。
- 第一次播放某个技能。
- 第一次见到某种怪和某种特效。
- 第一次进某个带完整氛围效果的场景。

这类问题最容易被误会成"偶发卡顿""怎么有时候卡有时候不卡"，实际上它常常是在提醒你：

`这个资源的第一次成本，被放在了错误时机。`

## 提交前 10 项自检表

如果你只想把这篇文章收成一张能直接用的表，我建议至少在提交前过下面这 10 项：

| 自检项 | 看到什么信号要警惕 | 建议动作 |
|------|----------------|---------|
| 1. 主角和背景的贴图优先级有没有搞反 | 背景、地面、装饰件和主角差不多规格 | 先保主角和核心交互资源，背景资源回到分档表重看 |
| 2. 透明是不是被当成默认做法 | 柔边、叠层、透明装饰大量出现 | 先问清哪些必须透明，哪些能换更便宜表达 |
| 3. 材质槽位是不是过多 | 一个模型拆了很多材质，但视觉收益不大 | 先确认是否能合并，或是否真有分离必要 |
| 4. 远景资源是不是做得像近景 | 远处也保留高细节和复杂轮廓 | 提前确认 LOD 和距离策略，不要等场景搭完再补 |
| 5. 场景是不是靠大量细碎资产堆出来 | 单体都不大，但整体密度很高 | 重点看数量风险，不只看单体风险 |
| 6. UI 首开是不是会拉起太多内容 | 首开明显抖，第二次好很多 | 和程序、TA 确认能否拆阶段、预热或减少首开负担 |
| 7. 特效是不是会在高峰时刻叠满 | Boss、结算、受击、技能同时抢预算 | 先保核心反馈，背景装饰类效果优先降级 |
| 8. 全屏效果是不是常驻太多 | 模糊、Bloom、屏幕遮罩、闪白叠得很满 | 先确认哪些是风格核心，哪些只是锦上添花 |
| 9. 这个资源是不是第一次出现最危险 | 首放、首开、首见时才卡 | 优先把它当"时机问题"处理，不要只当静态规格问题 |
| 10. 这件事该不该自己拍脑袋定 | 已经涉及材质规则、分档、渲染治理 | 不要独自猜，立刻拉 TA 或客户端程序一起定边界 |

## 结论

这篇真正想留下来的，不是一堆术语，而是一种更稳的自检顺序：

- 先看这份资源是不是在错误的地方用了高规格。
- 再看它是不是特别容易和别的东西叠在一起。
- 再看它是不是会在首开、首放、首见这种危险时机第一次出现。
- 最后再判断这件事该自己先改，还是该立刻去找 TA 或客户端程序确认。

如果你把这套顺序建立起来，很多问题会在资源提交前就被看见，而不是拖到联调时才变成一句：

`怎么这个东西一进项目就贵了？`
