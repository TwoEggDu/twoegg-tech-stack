很多团队一开始做移动端优化，最自然的动作是：

- 看 Profiler
- 降参数
- 砍后处理
- 删特效
- 试着把某几个热点函数改快

这些动作当然不一定错。

但如果你还没先建立移动端硬件的基本地图，它们很容易变成一种低效循环：

- 你知道哪里贵，却不知道为什么它会在手机上特别贵
- 你知道某台机器掉帧，却不知道它是冷机结论还是热机结论
- 你知道某个效果在 PC 上没问题，却不知道为什么到移动端就会突然失控

所以这篇入门文不想先教你“怎么调某个选项”，而是想先做一件更重要的事：

`把移动端硬件最值得先懂的五个直觉，收成一张最小地图。`

## 为什么移动端不是“低配 PC”

很多误判都从这里开始。

PC 的默认直觉经常是：

- 显存和系统内存是分开的
- 散热余量比较大
- 峰值性能通常更容易长期维持
- 用户配置虽然复杂，但单机硬件上限通常比较高

而移动端的现实是另一套约束：

- CPU、GPU、NPU、ISP 往往共享同一块 SoC
- CPU 和 GPU 共用同一份内存带宽预算
- 没有风扇，热预算比峰值跑分更早决定体验
- 同一个 Android 设备池里的差异可以非常大

也就是说，移动端真正难的地方不只是“性能低”，而是：

`预算是共享的、会收缩的，而且设备池是离散分层的。`

这也是为什么很多在 PC 上看起来还合理的方案，一旦搬到移动端，就会开始出现：

- 带宽不够
- 发热过快
- 长玩掉频
- 部分机型表现特别差

## 先立住五个移动端硬件直觉

### 1. SoC 不是一个 CPU 名字，而是一整套共享预算系统

移动端 SoC 上通常同时集成：

- CPU
- GPU
- 内存控制器
- 存储控制器
- ISP / NPU / DSP

这件事最重要的工程含义不是“集成度高”，而是：

`它们在争同一份资源。`

所以你在移动端看到的很多问题，最后都不是单点性能问题，而更像：

- GPU 在抢带宽
- CPU 在热机后降频
- 存储读写和资源解压在挤压主循环余量

如果你只把手机理解成“处理器频率低一点”，后面很多判断都会歪。

继续往下最适合看的，是 [移动端硬件 01｜SoC 总览]({{< relref "performance/mobile-hardware-01-soc-overview.md" >}})。

### 2. TBDR 决定了移动端为什么特别怕带宽浪费

移动端 GPU 和桌面 GPU 很大的不同，不只是算力数字，而是执行模型。

桌面端很多路线默认更能承受：

- 多次全屏读写
- 更重的中间 Render Target
- 更长的后处理链
- 更高的透明叠加成本

移动端很多 GPU 则建立在 TBDR / TBR 这类更强调片上处理、减少外部内存往返的思路上。

这会直接影响你怎么看这些东西：

- Bloom / SSAO / Blur 为什么会突然变贵
- 透明 overdraw 为什么会比直觉更伤
- 某些 PC 上常见的 deferred 思路为什么在移动端要更谨慎

如果你只先记一句话，我建议记这句：

`移动端很多“渲染贵”，本质上都更接近“带宽贵”。`

继续往下最适合看的，是 [移动端硬件 02｜TBDR 架构详解]({{< relref "rendering/hardware-02-tbdr.md" >}})。

### 3. 机型差异不是噪声，而是移动端项目的基本现实

在移动端，设备池从来不是一条平滑坐标轴。

你真正面对的通常是：

- 不同 SoC 代际
- 不同 GPU 家族
- 不同内存规格和容量
- 不同存储速度
- 不同散热设计

这意味着“同样一套内容在所有机器上都该差不多”往往不是默认现实，而是需要项目主动去实现的目标。

所以机型分档不是附加能力，而很接近移动端项目的基本工程能力：

- 为什么有的项目做三档，有的做四档
- 为什么质量预设不该只停在几个渲染开关
- 为什么某些机型必须单独做回退或灰度治理

继续往下最适合看的，是 [移动端硬件 02｜设备档次]({{< relref "performance/mobile-hardware-02-device-tiers.md" >}})。

### 4. 峰值性能不等于真实游戏体验

很多人第一次看移动端硬件，最容易被跑分带偏。

但对游戏来说，真正更重要的问题通常不是：

`这台机子第一分钟能跑多高。`

而是：

`它在 20 分钟、30 分钟之后还能剩多少。`

这会直接把你从“峰值性能”带到“持续性能”和“热约束”上。

也就是说，移动端里的很多高档和低档差异，真正值钱的不只是峰值，而是：

- 留存率
- 降频速度
- 热饱和后的稳定性

继续往下最适合看的，是 [移动端硬件 02b｜持续性能]({{< relref "performance/mobile-hardware-02b-sustained-performance.md" >}}) 和 [移动端硬件 03｜功耗与发热]({{< relref "performance/mobile-hardware-03-power-thermal.md" >}})。

### 5. 平台碎片化会把很多问题从“优化”变成“治理”

移动端世界不仅是硬件差异大，平台行为本身也会继续放大问题：

- Android 厂商定制
- 驱动版本差异
- 图形 API 支持不一致
- 系统回收和前后台切换

这意味着很多问题到最后不会只落在“某个 Pass 太重”，而会继续演化成：

- 某些机型上 Vulkan 路径异常
- 某些驱动版本上 shader 行为不一致
- 某些设备上系统回收和热限制更激进

所以移动端后面一定会慢慢牵涉：

- 平台专项
- 真机工具
- 驱动与系统碎片化治理

如果你已经开始碰到“只在部分安卓机上出问题”，后面一定要继续补平台专项。

## 把这五个直觉翻成项目语言

如果你想把上面这些概念马上映射回项目现场，我建议先这样对应：

| 你现在看到的现象 | 更该先想到哪层硬件现实 |
|---|---|
| 开头满帧，玩一会开始掉 | 持续性能 / 热约束 |
| PC 上没事，移动端一开效果就炸 | 带宽 / TBDR / 统一内存 |
| 同一场景在不同安卓机差异特别大 | 设备档次 / 平台碎片化 |
| 后处理、透明、阴影特别容易失控 | 外部内存读写成本 / 带宽预算 |
| 某些机型经常被判错档 | 设备能力判断 / 分档策略 / 线上治理 |

你会发现，很多“优化技巧”如果不先放回这些硬件现实里，最后都会变成：

- 头痛医头
- 某台机器有效、换一台又失效
- 第一次好使，长玩之后又出问题

## 常见误解

### 误解一：移动端硬件就是“算力低一点”

不够。真正更关键的是共享预算、带宽上限、持续性能和热收缩。

### 误解二：做优化时再补硬件知识也来得及

能做，但代价通常更高。因为你会先在错误直觉上做很多试错。

### 误解三：硬件文章只是科普，和项目落地关系不大

也不对。后面的资源分档、渲染配置、动态降档和真机验证，最后都建立在这些硬件现实上。
