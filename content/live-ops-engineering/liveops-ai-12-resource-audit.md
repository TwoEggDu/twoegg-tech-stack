## 这篇解决什么运营问题

运营期资源与包体管理（V12）解决的核心问题是：内容是产品命脉不能不加，但资源总量翻了几倍之后怎么管。在 12.1-12.5 中，我们建立了资源增长曲线模型、淘汰与归档机制、增量更新包大小管理、CDN 成本控制和版本兼容性管理。

这些机制的运转有一个前提：**能及时发现资源层面的问题**——哪些资源是冗余的、哪些资源违反了规范、哪些资源的增长速度异常。在项目初期，资源量小、美术团队稳定，人工审查可以覆盖。但运营两年后，典型的资源状况是：

- 资源总条目 10 万+
- 每个版本新增资源 500-2000 条
- 美术外包团队轮换，规范执行力度参差不齐
- 没人能说清楚哪些资源还在被使用、哪些已经废弃

人工审查在这个规模下已经不可持续。本篇聚焦：**怎么用 AI 实现资源审计的自动化，覆盖冗余检测、规范违规识别和异常膨胀预警。**

## 传统做法与痛点

### 传统资源审查方式

**方式一：人工巡检。** 每个版本发布前，负责人抽查新增资源——看贴图尺寸是否合规、模型面数是否达标、命名是否符合规范。问题是抽查覆盖率低（通常 < 20%），而且随资源量增长边际效果递减。

**方式二：规则脚本。** 用脚本检查硬性规则（贴图最大尺寸、模型最大面数、文件命名格式）。这能覆盖可以用规则表达的违规，但覆盖不了"规则之外"的问题——比如一张用于 UI 图标的贴图用了 2048x2048（符合最大尺寸规则，但图标根本不需要这么大）。

**方式三：包体分析工具。** 在打包后分析最终包体的资源分布。能发现"哪些资源占用最多"，但发现不了"哪些资源是不必要的"。

### 三个长线运营中的深层痛点

**痛点一：冗余资源累积。**

运营两年后，资源库中存在大量冗余：
- **废弃资源**：活动结束后的临时资源没有清理，角色返工后旧版本资源保留
- **重复资源**：不同美术人员制作了功能相同但文件不同的资源（不同命名的同款特效、略有差异的同类 UI 元素）
- **过度规格资源**：为低端机准备的 UI 图标用了高清贴图规格，背景噪声贴图用了无损格式

这些冗余资源占用存储、增加包体、拖慢加载，但没有人有精力逐一排查十万条资源。

**痛点二：规范违规的"灰色地带"。**

硬性规则能抓住明确违规（贴图超过 4096）。但很多违规是上下文相关的：

- 一个 NPC 角色模型 5000 面——如果是主角级别可以接受，如果是路人 NPC 就超标了
- 一张贴图 1024x1024——如果是角色皮肤合理，如果是地面裂缝细节纹理就浪费了
- 一个音频文件 3MB——如果是背景音乐合理，如果是按钮点击音效就异常了

这些判断需要理解资源的用途上下文，简单规则做不到。

**痛点三：增长趋势的异常难以感知。**

每个版本资源量增长 2% 看起来正常。但如果某个模块的资源增长速度是其他模块的 3 倍，可能意味着该模块的美术制作流程有问题（过度制作、返工率高、规格失控）。人工看月度报表很难注意到这种相对增长速率的异常。

## AI 方案设计

### 核心思路：从规则审查到模式审查

传统审查基于预定义规则：if 贴图尺寸 > 阈值 then 报错。

AI 审查基于模式学习：学习"同类资源通常是什么规格"，然后检测偏离同类模式的异常个体。

这意味着系统不需要为每种资源类型写一条规则，而是从历史数据中学习"什么是正常的"，自动发现不正常的。

### 架构设计

```
┌──────────────────────────────────────────────────────┐
│  资源元数据采集层                                       │
│  资源文件属性：大小 / 格式 / 尺寸 / 面数 / 骨骼数 / 压缩率│
│  资源上下文：所属模块 / 用途标签 / 引用关系 / 创建时间    │
│  版本信息：哪个版本引入 / 最后修改版本 / 引用版本范围     │
├──────────────────────────────────────────────────────┤
│  规则审查层（传统，保留）                                │
│  硬性规则检查：格式 / 尺寸上限 / 命名规范                │
│  → 输出：确定违规列表                                   │
├──────────────────────────────────────────────────────┤
│  AI 审查层                                             │
│  冗余检测：内容相似度 + 引用分析 → 可能冗余的资源对       │
│  规格异常检测：同类资源的统计分布 → 偏离正常规格的个体     │
│  增长趋势分析：按模块 / 类型的增长速率 → 异常膨胀预警     │
├──────────────────────────────────────────────────────┤
│  审计报告层                                            │
│  分类汇总：确定违规 / 高度可疑 / 建议关注                │
│  按模块归属：方便对应责任人                              │
│  按影响排序：按包体影响 / 内存影响排优先级                │
└──────────────────────────────────────────────────────┘
```

### 三个 AI 审查能力

**能力一：智能冗余检测。**

冗余不只是"文件完全一样"。更常见的冗余形态：

- **近似重复**：两张贴图内容 90% 相同，只有细微色差或尺寸差异
- **功能等价**：两个特效的最终视觉效果几乎一样，但使用了不同的实现方式
- **用途重叠**：同一个 UI 元素在不同活动中被重复制作（换了个名字但内容一样）
- **废弃但保留**：已经不被任何场景引用的资源仍然留在资源库中

检测方法：
- 贴图内容相似度：感知哈希（perceptual hash）+ 结构相似度（SSIM），而不是简单的文件 hash
- 引用分析：从打包配置和代码引用反向追踪，找到零引用资源
- 名称语义分析：识别命名模式相似但内容可能冗余的资源组

**能力二：上下文感知的规格审查。**

传统规则："贴图尺寸不超过 2048x2048"。AI 审查："UI 图标类贴图通常是 128x128 或 256x256，这个 UI 图标用了 1024x1024 是异常的"。

实现方式：
1. 根据资源的用途标签（UI / 角色 / 场景 / 特效 / 音频）和子类型（图标 / 立绘 / 地面 / 天空盒）对资源分组
2. 在每个分组内统计规格分布（尺寸、面数、文件大小的分布）
3. 检测显著偏离组内分布的个体

示例输出：

```
规格异常检测结果：
  资源：ui_icon_shop_sale_badge.png
  类别：UI/图标
  实际尺寸：1024x1024（614 KB）
  同类正常范围：128x128 - 256x256（5-25 KB）
  偏离程度：尺寸 4 倍，文件大小 25 倍
  建议：降规格至 256x256，预计节省 ~590 KB
```

**能力三：模块级增长趋势分析。**

按模块追踪资源增长曲线：

- 每个版本每个模块的资源总量、总大小、新增数量
- 与全项目平均增长率对比，发现增长异常快的模块
- 与历史同期（上线后同阶段的其他活动模块）对比

示例输出：

```
增长趋势预警：
  模块：社交系统/表情包
  最近 6 个版本增长率：15% / 版本
  项目平均增长率：3% / 版本
  当前资源总大小：128 MB
  如果趋势持续：6 个版本后达到 300 MB
  对比参考：战斗系统（资源总大小 340 MB）运营 2 年才达到
  建议：排查表情包资源的制作流程，可能存在过度制作或规格失控
```

### 数据要求

| 要求 | 最低标准 | 建议标准 |
|------|---------|---------|
| 资源元数据 | 文件名、大小、格式、尺寸/面数 | + 用途标签 + 模块归属 + 创建版本 |
| 引用关系 | 打包后的依赖关系 | 代码引用 + 配置引用 + 场景引用 |
| 版本历史 | 当前版本快照 | 至少 10 个版本的资源列表快照 |
| 用途分类 | 文件路径暗示的类别 | 人工标注的用途标签体系 |

## 对比：传统审查 vs AI 审查

| 维度 | 人工抽查 | 规则脚本 | AI 审查 |
|------|---------|---------|--------|
| 覆盖率 | < 20% | 100%（规则覆盖范围内） | 100% |
| 冗余检测 | 偶尔发现 | 只能查完全相同的文件 | 近似重复 + 功能等价 |
| 规格判断 | 依赖经验 | 只能用全局阈值 | 上下文感知（按用途分类判断） |
| 趋势发现 | 基本不可能 | 可以做简单统计 | 多维度趋势分析 |
| 误报率 | 低（但覆盖也低） | 低 | 中（需要调优） |
| 对规范文档的依赖 | 高 | 高（规则来源于规范） | 低（从数据中学习模式） |
| 处理新类型资源 | 需要更新知识 | 需要新增规则 | 自动适应（有同类数据即可） |
| 维护成本 | 人力时间 | 规则维护 | 模型更新 + 标签维护 |

## 局限性与不适用场景

### 不适用场景一：资源量太少

资源总量低于 5000 条时，统计学习没有意义（样本不够建立"正常分布"）。此阶段人工审查 + 规则脚本完全够用。

### 不适用场景二：资源用途分类缺失

上下文感知审查依赖资源用途分类。如果资源没有任何分类信息（既没有用途标签，文件路径也没有规律），AI 只能退化为全局统计——和规则脚本差别不大。需要先建立最基础的资源分类体系。

### 不适用场景三：审美和艺术判断

AI 能判断"这个贴图尺寸比同类大 4 倍"，但不能判断"这个贴图的视觉质量是否值得这个尺寸"。最终决策需要美术负责人确认——AI 只是帮助发现候选项，不能替代审美判断。

### 局限性一：误报处理成本

AI 审查覆盖率高的代价是误报量也大。如果一次审查输出 500 条"疑似冗余"但实际只有 100 条真冗余，处理这 500 条的人工成本可能超过了收益。必须按影响排序、分优先级处理，而不是一次性全部处理。

### 局限性二：引用关系分析的不完整性

资源引用关系的追踪在 Unity 项目中本身是个难题：

- 代码中通过字符串动态加载的资源（`Resources.Load("path")`）静态分析可能遗漏
- 跨 AssetBundle 的间接引用关系需要完整的依赖图
- 运行时条件加载的资源（根据设备档位、地区选择不同资源）需要全条件覆盖

零引用不等于真的没用。一个资源可能只在特定活动期间被引用，非活动期看起来是零引用但不能删。

### 局限性三：近似重复判断的边界模糊

两张贴图 SSIM 相似度 0.85，是"近似重复"还是"有意为之的变体"？同一角色的不同皮肤配色，从像素相似度看可能很高，但它们是有意义的不同资源。边界判断需要人工确认。

## 落地路径

### 第一阶段：资源元数据收集和分类（4-6 周）

- 建立资源元数据自动采集管线（每次构建时导出资源列表 + 属性）
- 建立基础用途分类体系（至少区分 UI / 角色 / 场景 / 特效 / 音频 / 配置）
- 建立引用关系追踪（打包时输出依赖图）
- 保存版本级资源快照（每个版本的完整资源列表和元数据）

### 第二阶段：规则审查加强 + 零引用检测（2-4 周）

在上 AI 之前，先把传统方法做扎实：
- 完善规则脚本，覆盖所有硬性规范
- 实现零引用资源检测（静态分析 + 动态采集补充）
- 与构建管线集成：违规报告作为构建产物输出
- 这一步本身就能发现大量低垂果实

### 第三阶段：AI 审查上线（6-8 周）

- 贴图近似重复检测（感知哈希 + SSIM）
- 同类资源规格异常检测（基于用途分类的统计分布检测）
- 模块级增长趋势分析
- 审计报告自动生成：按影响排序、按模块归属分组

### 第四阶段：闭环和优化（持续）

- 人工审核 AI 发现的问题 → 确认 / 标记为误报 → 反馈给模型
- 按季度生成资源健康度报告
- 将高频出现的 AI 发现固化为规则（AI 发现了某类问题，确认是规律后加入规则脚本）

## 度量与可见性

| 度量指标 | 计算方式 | 目标 |
|---------|---------|------|
| 冗余资源发现量 | AI 发现并确认的冗余资源条数和总大小 | 首次审计通常能发现总量 10-20% |
| 规格违规发现量 | AI 发现的上下文相关规格异常条数 | 补充规则脚本遗漏的部分 |
| 误报率 | AI 标记的问题中人工确认为误报的比例 | < 30% |
| 包体影响 | 清理 AI 发现的问题后包体大小减少量 | 可量化的直接收益 |
| 增长趋势预警准确率 | 预警后确认存在问题的模块比例 | > 60% |
| 审计覆盖率 | AI 审查覆盖的资源条目 / 总资源条目 | > 95% |

## 小结与检查清单

AI 资源审计解决的核心问题是：资源规模增长到人工审查不可持续时，怎么保持审查覆盖率和发现能力。它不替代规则脚本（硬性规范仍然用规则），而是补充规则覆盖不到的"上下文相关"和"趋势分析"能力。

- [ ] 资源元数据自动采集管线是否建立？
- [ ] 资源用途分类体系是否至少覆盖一级分类？
- [ ] 引用关系追踪是否覆盖静态引用和主要动态引用？
- [ ] 版本级资源快照是否持续保存？
- [ ] 规则脚本是否已经覆盖所有硬性规范？
- [ ] AI 审查结果是否按影响排序、按模块归属分组？
- [ ] 是否有人工确认机制避免误删"看似冗余但实际有用"的资源？
- [ ] 是否按季度生成资源健康度趋势报告？
