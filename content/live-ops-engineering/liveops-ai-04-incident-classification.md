## 这篇解决什么运营问题

线上问题分级与响应（V04）解决的核心问题是：运营期的问题远多于能修的，怎么分级、谁决定修不修、修的节奏怎么控制。在 4.1-4.5 中，我们建立了 P0-P3 分级体系、响应 SLA、决策框架和事故指挥官机制。

但这套体系的运转依赖两个人工环节：**定级**和**根因定位**。定级需要值班工程师根据影响面、严重度快速判断——这在事故量不大时没问题，但一个运营两年的项目，每周可能收到 50-100 条需要分类的异常报告。根因定位需要"这个问题之前见过没有"的记忆——但当事故库有上千条记录、团队成员轮换了大半之后，这种记忆就不可靠了。

本篇聚焦：**怎么用 AI 辅助事故分类和相似事故匹配，加速从"发现"到"定级"和"定位根因"的过程。**

## 传统做法与痛点

### 人工分类的工作流

典型的事故分类流程：

1. 监控系统或用户反馈触发告警
2. 值班工程师查看告警详情（堆栈、日志、影响面）
3. 根据经验判断 P0-P3 等级
4. 查看是否是已知问题（翻事故库、问同事）
5. 如果是新问题，开始排查根因

### 三个随时间恶化的痛点

**痛点一：定级一致性下降。**

不同值班工程师对同一事故的定级可能不同。老手可能根据经验判断"这种堆栈通常是 P2"，新人可能因为不确定而升级为 P1。当团队轮换率超过 30%（长线运营项目的常态），定级一致性显著下降。不一致的定级导致响应资源分配失衡——P1 太多时团队麻木，P1 太少时真正的紧急事故被延误。

**痛点二：相似事故识别依赖个人记忆。**

一个运营两年的项目可能积累了 2000+ 条事故记录。值班工程师不可能记住所有历史事故。当一个新事故进来时，判断"这是不是之前那个 XX 问题的变体"完全靠个人记忆。问题是——最了解历史事故的人可能已经离开团队了。

**痛点三：事故分类的文本描述不标准。**

不同工程师写的事故描述风格差异大：有人写长篇分析，有人只贴堆栈。搜索"内存泄漏"找不到描述为"OOM 崩溃"的同类事故。关键字搜索在非标准化文本上的召回率很低。

## AI 方案设计

### 核心思路：向量化事故特征 + 语义匹配

传统搜索基于关键字精确匹配。AI 方案的核心是把事故的多模态特征（堆栈、日志、描述、影响面指标）转化为向量表示，然后用向量相似度做匹配。

这意味着：
- "内存泄漏导致 OOM" 和 "Texture 未释放引起内存持续上涨" 可以被识别为同类问题
- 即使堆栈不完全一样，如果关键调用路径相似，也能匹配到

### 架构设计

```
┌──────────────────────────────────────────────────────┐
│  事故输入层                                            │
│  新事故 → 结构化提取：堆栈 / 日志片段 / 指标快照 / 描述  │
├──────────────────────────────────────────────────────┤
│  特征工程层                                            │
│  堆栈特征：调用链关键帧提取 + 去噪（去掉框架代码帧）      │
│  日志特征：错误模式提取（正则 + NLP）                    │
│  指标特征：事故发生前后的指标变化向量                     │
│  描述特征：文本语义向量                                  │
│  → 拼接为事故特征向量                                   │
├──────────────────────────────────────────────────────┤
│  分类模块                                              │
│  输入：事故特征向量                                     │
│  输出：P0-P3 等级建议 + 置信度 + 分类依据               │
│  方法：基于历史定级数据的监督学习                        │
├──────────────────────────────────────────────────────┤
│  相似事故匹配模块                                       │
│  输入：事故特征向量                                     │
│  方法：在事故向量库中做最近邻检索                        │
│  输出：Top-5 相似事故 + 相似度分数 + 差异说明            │
├──────────────────────────────────────────────────────┤
│  辅助决策层                                            │
│  将分类建议和相似事故呈现给值班工程师                     │
│  工程师确认 / 修正 → 反馈回模型                         │
└──────────────────────────────────────────────────────┘
```

### 分类模块的关键设计

**输入特征的优先级**：堆栈调用链 > 影响面指标（影响用户数、影响设备型号范围） > 日志错误模式 > 文本描述。堆栈是最客观的特征，文本描述最容易有噪声。

**分类不是替代人工决定，而是给出建议**。系统输出示例：

```
建议等级：P2
置信度：0.82
分类依据：
  - 堆栈匹配"UI 渲染异常"类别（历史 47 例，85% 被定为 P2）
  - 影响面：<0.1% 用户，单一设备型号
  - 无内存/Crash 关联指标异常
最相似历史事故：#1847（P2，UI 列表渲染越界，2026-02-15）
```

**置信度低于阈值时，系统应该明确说"我不确定"**，而不是硬给一个分类。建议阈值：置信度 < 0.6 时标记为"需要人工判断"。

### 相似事故匹配的关键设计

**堆栈相似度计算**不能用简单的字符串比较。需要：
1. 去掉框架代码帧（Unity 引擎内部帧、系统库帧），保留业务代码帧
2. 对调用链做序列对齐（允许中间插入 / 删除个别帧）
3. 对函数名做语义归一化（不同版本的函数重命名不影响匹配）

**多维度加权匹配**：单纯的堆栈匹配会遗漏"堆栈不同但根因相同"的情况。需要综合堆栈相似度、指标模式相似度、时间上下文相似度。

**匹配结果必须包含差异说明**：不只告诉工程师"和 #1847 类似"，还要说"区别在于：#1847 只影响 iOS 低端机，本次影响 Android 全机型"。差异信息对根因定位同样重要。

### 数据要求

| 要求 | 最低标准 | 建议标准 |
|------|---------|---------|
| 历史事故记录 | 200+ 条已分级的事故 | 500+ 条，覆盖 P0-P3 各等级 |
| 堆栈信息 | 符号化后的堆栈 | 符号化 + 业务帧标注 |
| 事故描述 | 有文本描述 | 结构化描述（影响面、现象、初步分析） |
| 定级标签 | 人工确认的 P0-P3 | 有最终确认等级（不只是初始定级） |
| 根因标签 | 非必需 | 有根因分类标签可提升匹配质量 |

## 对比：传统人工流程 vs AI 辅助流程

| 维度 | 纯人工流程 | AI 辅助流程 |
|------|-----------|-----------|
| 定级速度 | 取决于值班工程师经验，5-30 分钟 | 秒级建议 + 人工确认，通常 < 5 分钟 |
| 定级一致性 | 不同工程师差异大 | 系统建议提供一致基线，人工可覆盖 |
| 相似事故发现 | 依赖个人记忆和关键字搜索 | 语义匹配，不依赖记忆 |
| 历史事故回溯 | 线性翻阅，效率低 | 自动排序 Top-N 相似事故 |
| 新人上手成本 | 高（需要熟悉历史事故） | 低（系统提供历史上下文） |
| 处理高峰应对 | 容量受限于人数 | 系统可并行处理，人工只需确认 |
| 建设成本 | 低（流程 + 文档） | 中（向量化 + 检索基建 + 标注） |
| 维护成本 | 低（更新文档） | 中（模型更新 + 特征工程调优） |

## 局限性与不适用场景

### 不适用场景一：全新类型的事故

AI 分类和匹配基于历史模式。如果遇到一种从未出现过的事故类型（例如新接入的 SDK 引发的全新崩溃模式），AI 的分类置信度会很低，匹配不到有用的历史事故。这时候系统应该明确告知"这是一个新模式"，而不是强行匹配。

### 不适用场景二：需要业务上下文的定级

有些事故的严重程度取决于业务上下文而非技术特征。例如"付费页面加载慢 2 秒"在普通日是 P2，在限时活动期间可能是 P0。AI 模型很难自动感知"今天在做大型促销活动"。

### 不适用场景三：事故数据质量差

如果历史事故记录不完整（缺堆栈、缺影响面数据）、定级不一致（同类事故有人定 P1 有人定 P2）、描述模糊，模型学到的只是噪声。**垃圾进垃圾出——AI 不能修复数据质量问题。**

### 局限性：过度依赖会削弱团队能力

如果团队完全依赖 AI 建议而不做独立判断，长期来看团队的事故分析能力会退化。AI 辅助的目标是"加速"而非"替代"——值班工程师需要理解 AI 为什么这样分类，而不是无脑接受。

## 落地路径

### 第一阶段：事故数据标准化（4-8 周）

这是最重要也最容易被跳过的一步。

- 统一事故记录模板：必填字段（堆栈、影响面、现象描述、最终定级、根因分类）
- 对历史事故做补标注：至少把最终确认的定级和根因分类补齐
- 堆栈符号化管线验证：确保线上 Crash 堆栈能正确符号化
- 建立事故根因分类体系（代码 / 配置 / 资源 / 平台 / 第三方 / 流程）

### 第二阶段：相似事故匹配先行（4-6 周）

相似事故匹配比自动分类更容易落地、更容易体现价值。

- 对历史事故做向量化（堆栈特征 + 文本语义向量）
- 建立向量检索索引
- 在事故处理界面加入"相似事故"推荐面板
- 收集工程师反馈：推荐的事故是否有用

### 第三阶段：分类建议（4-8 周）

- 基于带标注的历史数据训练分类模型
- 影子模式运行：模型给出建议但不影响实际定级
- 统计分类建议与人工定级的一致率
- 一致率 > 80% 且误分级的 P0/P1 为 0 时，开启建议模式

### 第四阶段：反馈闭环（持续）

- 工程师对 AI 建议的确认 / 修正自动回流为训练数据
- 每月评估模型准确率变化趋势
- 新的事故类型出现时，及时更新分类体系和模型

## 度量与可见性

| 度量指标 | 计算方式 | 目标 |
|---------|---------|------|
| 定级建议一致率 | AI 建议等级与最终确认等级一致的比例 | > 80% |
| P0/P1 误判率 | P0/P1 事故被建议为更低等级的比例 | 0%（绝对底线） |
| 相似事故命中率 | 推荐的 Top-5 中至少一条被工程师采纳的比例 | > 60% |
| 定级耗时 | 从事故触发到完成定级的平均时间 | 引入后下降 > 40% |
| 根因定位耗时 | 从定级完成到确认根因的平均时间 | 引入后下降 > 30% |
| 新人首次值班响应时间 | 新工程师首次独立值班的平均响应时间 | 应与老手差距缩小 |

## 小结与检查清单

AI 辅助事故分类的价值在于把定级和根因定位中"查历史、比对、判断"的工作从人工记忆转移到系统检索。落地的前提是事故数据的标准化和充分积累。

- [ ] 事故记录模板是否标准化？必填字段是否齐全？
- [ ] 历史事故是否有最终确认的定级和根因分类标签？
- [ ] 堆栈符号化管线是否正常工作？
- [ ] 相似事故匹配是否先于自动分类上线？
- [ ] 分类建议是否经过影子模式验证？
- [ ] P0/P1 误判率是否严格为 0？
- [ ] 工程师反馈是否形成闭环回流到模型训练？
- [ ] 是否保留人工覆盖 AI 建议的能力，且覆盖操作有记录？
