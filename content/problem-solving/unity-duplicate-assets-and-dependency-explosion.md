这篇不再讲新的主线原理，而是把前面那套地图落回一个最常见、也最容易长期失控的现场：

`重复资源和依赖爆炸。`

几乎所有认真做过 `AssetBundle / Addressables / 热更资源交付` 的项目，最后都会碰到它。

而且它特别容易以一种“看起来没什么”的方式开局：

- 某个 `Prefab` 在两个包里各带一份，先这样吧
- 这张贴图很多地方都用，抽个公共包吧
- 这个材质也会被别的入口复用，再抽一下吧
- 这个特效链还不稳定，先多带一点，后面再收

刚开始时，每一个局部决定看起来都合理。

但项目一旦做大，现场就会慢慢变成另一种样子：

- 包体越来越难解释
- 依赖链越来越长
- 构建结果一改就牵一串
- 首包和活动包的边界越来越模糊
- 回滚时根本说不清“到底是哪份共享内容变了”

最后团队会进入一个很典型的状态：

`不是切不出包，而是再也说不清这些包为什么要这样切。`

这就是这篇想讲的东西。

## 先给一句总判断

如果把整件事压成一句话，我会这样说：

`重复资源和依赖爆炸，通常不是两个独立问题，而是同一张交付图失去稳定边界后的两种表现：该共享的没有被正确共享，会变成重复资源；不该共享的被过度抽离，会变成依赖爆炸。`

这句话里最关键的是：

- `稳定边界`
- `该共享`
- `不该共享`

只要这三件事一乱，后面的包体、缓存、加载顺序、回滚和回归都会一起乱。

## 一、问题通常不是从“错误切包”开始的，而是从“局部合理”开始的

这点很重要。

因为如果你只把它理解成“某次切包设计失误”，很容易低估它为什么这么普遍。

现实里，它通常不是一开始就很离谱，而是：

## 1. 团队先顺着最自然的业务入口切第一版

例如：

- 登录一包
- 主城一包
- 战斗一包
- 活动一包

这一步通常没什么问题，甚至是对的。

因为一开始大家最关心的是：

- 能不能先跑起来
- 能不能先把内容按入口分发
- 能不能把首包和非首包分开

## 2. 然后共享内容开始出现

这时候很自然会发现：

- 登录和主城都用同一套 UI 图集
- 主城和战斗都用同一组角色材质
- 多个活动都共用某些特效、字体、公共配置

于是团队开始做第二轮局部优化：

- 抽公共图集包
- 抽公共材质包
- 抽公共特效包

这一步本身也不一定错。

## 3. 真正的问题，是这些共享边界往往不是“长期稳定边界”

项目最常见的失控点在这里。

因为很多被抽出去的“共享内容”，其实只是：

- 当前两个入口恰好共用
- 某个阶段暂时看起来会复用
- 某个活动周期里短期共用

它们并不一定是：

`长期稳定、跨版本稳定、跨入口稳定、值得被抽成公共交付单元的共享边界。`

一旦团队没有把这件事想清，重复资源和依赖爆炸就会从这里一起长出来。

## 二、重复资源怎么长出来：因为“不知道该在哪共享”，最后只能多带一份

很多团队第一次碰到重复资源时，直觉会觉得：

`这不就是包里多塞了一份东西吗。`

但工程上它远比这更麻烦。

## 1. 最常见的起点是“不敢共享”

项目里最容易重复的，不一定是最不重要的资源，反而常常是最关键的那批：

- 角色公共材质
- 常用特效贴图
- UI 共用图集
- 通用 `Shader` / `SVC`

原因也很现实。

团队一旦发现“抽公共包以后依赖太复杂”，最自然的回摆就是：

`那我先不要共享了，直接每个包自己带一份。`

这会立刻带来一个短期好处：

- 当前入口变得更独立
- 依赖链暂时变短
- 某些加载顺序问题立刻消失

但它只是把一个问题换成了另一个问题。

## 2. 重复资源的真正代价，不只是磁盘多一份

它至少会放大四笔账：

### 1. 包体账

同样的贴图、材质、配置在多个包里重复出现，体积会直接膨胀。

### 2. 缓存账

哪怕逻辑上是“同一份内容”，一旦它以不同 bundle 身份重复出现，缓存也更难有效复用。

### 3. 回滚账

你以为只改了一份公共资源，实际上可能有多个包都各带了一版，结果回滚时谁新谁旧很难说清。

### 4. 回归账

任何一次资源改动，都可能要在多个入口重复验证，因为谁也不知道哪些包里还藏着副本。

所以重复资源真正危险的地方不是“浪费了点空间”，而是：

`它会让内容身份开始分裂。`

## 3. 一旦内容身份分裂，问题就很难收

这时最常见的现场现象是：

- 明明是同一张贴图，但线上玩家在不同入口看到的版本不一致
- 明明修了一次材质，却只有部分内容路径生效
- 某些活动修好了，主线入口还用着旧副本

这类问题表面上看像“发布不一致”，但更早的根因其实是：

`本来该是同一个内容身份的东西，已经被复制成了多个交付身份。`

## 三、依赖爆炸怎么长出来：因为“不敢重复”，于是一路往外抽

另一边也同样会失控。

## 1. 项目发现重复资源太多后，最自然的修正就是“继续抽公共包”

于是结构会慢慢变成：

- 登录入口依赖公共 UI
- 公共 UI 又依赖公共字体
- 公共字体又依赖公共材质
- 某些材质又统一依赖公共 shader 包
- 某些特效又统一依赖公共纹理包

到某个阶段，团队会忽然发现：

`每个业务包都变瘦了，但所有入口都开始依赖一长串公共包。`

这就是依赖爆炸最典型的开始。

## 2. 问题不是“有公共包”，而是“公共边界被不断上推”

共享本身不是错。

真正危险的是：

`为了消灭局部重复，团队会不断把共享边界往更高层抽。`

于是会出现这些现象：

- 原本只在两三个入口共用的资源，被抽成“全局公共包”
- 原本生命周期不同的内容，被迫绑进同一个共享边界
- 原本该随着某个活动一起更新的内容，被放进了长期常驻公共包

这时候公共包越多，项目反而越难理解：

- 哪些公共包是真的稳定公共
- 哪些只是历史上被抽出去、但现在已经不该继续公共
- 哪些依赖其实本来应该回收到入口包

## 3. 依赖爆炸最贵的代价，不在“包多”，而在“入口解释能力消失”

项目后期最可怕的一点不是包数量本身，而是：

`再也没人能一眼说清一个入口到底为什么要依赖这些包。`

一旦走到这里，后面的所有动作都会变慢：

- 加一个新活动前不敢轻易复用旧公共包
- 删一个旧公共包时不知道会伤到谁
- 做首包优化时不敢收，因为牵一串依赖
- 做回滚时也不敢切，因为公共边界已经混进太多历史责任

这才是依赖爆炸真正让项目难收的地方。

## 四、重复资源和依赖爆炸为什么总是一起出现

因为它们经常不是“你选一个就不会有另一个”，而是：

`你在两种不稳定策略之间来回摆。`

## 1. 不敢共享时，就会重复

路径会是：

- 公共边界太复杂
- 团队先让入口自己带一份
- 重复资源变多

## 2. 不敢重复时，就会过度抽公共

路径会是：

- 包体或回归成本太高
- 团队再把内容往外抽
- 依赖链继续拉长

## 3. 来回摆久了，项目会失去稳定边界

这时你看到的，就不再是单点问题，而是：

- 某些内容一半共享、一半重复
- 某些公共包既像基础能力，又混着活动内容
- 某些入口看似独立，其实背后还是拖着一串长依赖

所以重复资源和依赖爆炸最核心的共同根因，其实不是技术细节，而是：

`团队没有持续维护一套稳定的共享边界。`

## 五、项目现场怎么判断：这是重复资源问题，还是共享依赖设计问题

这两类问题很像，但第一刀还是可以先切。

## 1. 更像重复资源问题的信号

- 同一类资源在多个 bundle 里反复出现
- 入口之间几乎没有明确共享边界，更多是在各自带副本
- 改一次公共材质，要在多个入口重复发包和回归
- 缓存命中率看起来不差，但总包体和总下载量偏大

这类问题更像：

`共享不足，内容身份分裂。`

## 2. 更像共享依赖设计问题的信号

- 每个入口都挂着一长串公共依赖
- 入口本身已经不大，但公共包越来越多
- 首包优化卡在“删不动公共依赖”
- 回滚和活动发布经常被公共边界拖住

这类问题更像：

`共享过度，入口解释能力消失。`

## 3. 两类信号同时出现，其实最常见

项目里更真实的状态通常不是纯一种，而是：

- 有些内容重复得太多
- 有些内容又被抽得太狠

所以更稳的判断不是问：

`我们到底是重复多，还是依赖多？`

而是问：

`哪些内容真的应该共享，哪些内容应该回收回入口边界？`

## 六、真正该怎么收：不是继续抽，也不是继续复制，而是重新立共享边界

如果只是继续“多抽一点”或者“多复制一点”，问题只会换个方向继续长。

更稳的做法通常是重新回到三条边界：

## 1. 更新边界

这批内容是否真的会长期一起更新、一起回滚。

如果不是，就别硬抽成长期公共包。

## 2. 共享边界

它到底是高频跨入口复用，还是当前阶段刚好共用。

如果只是局部短期复用，不一定值得抽成全局共享。

## 3. 生命周期边界

它应该常驻、首包、按场景、按活动还是按补丁一起走。

如果生命周期不同，就不该为了消灭一点重复把它们绑成一个公共单元。

真正稳的共享边界，通常同时满足这三条，而不是只满足其中一条。

## 七、一个更实际的收束策略

如果项目已经进入“重复资源和依赖爆炸同时存在”的阶段，我更建议用一套更保守的收法：

1. 先把 truly global 的基础内容挑出来
2. 再把明显只属于单入口或单活动的内容收回去
3. 对剩下那批模糊共享内容，按真实更新节奏和生命周期重分
4. 不急着一次性消灭所有重复，先消灭最贵的重复
5. 不急着一次性缩短所有依赖，先缩短首包和高频入口依赖

这套做法的重点不是“最优”，而是：

`先让边界重新变得可以解释。`

因为项目一旦重新能解释：

- 这个公共包为什么存在
- 这个副本为什么暂时保留
- 这个入口为什么要依赖那几包

后面的治理动作才会重新变得有把握。

## 最后收成一句话

如果把这篇最后再压回一句话，我会这样说：

`重复资源和依赖爆炸真正难收，不是因为团队不会切包，而是因为项目在“多带一份更稳”与“继续抽公共更省”之间来回摆太久，最后失去了稳定的共享边界；一旦边界失去解释能力，包体、缓存、加载链、回滚和回归都会一起变难。`
