## 问题空间

游戏服新版本上线前两周，工程师说"本地测了没问题"，QA 说"功能测试通过"，但上线后同时在线 5 万人时服务器崩溃，玩家无法登录。这不是低概率事件，这是每个没做好压测的游戏都会经历的必然故事。

游戏服务器压测与 Web API 压测的本质区别在于：HTTP 服务的一次请求是无状态的短连接，用工具模拟 10000 个并发请求相对容易。而游戏服务是长连接、有状态的——每个虚拟用户需要维持一条 TCP/UDP 长连接，有独立的游戏状态（角色位置、背包、技能 CD），并按照符合真实行为模式的时序发送消息。

更困难的是，游戏协议通常是**自定义二进制协议**或经过编码的 Protobuf，Wireshark 可以抓包但标准压测工具（JMeter、Gatling）没有对应的协议插件。游戏压测工具几乎全部需要自研或深度定制。

## 抽象模型

### 压测的三个目标层次

**层次一：可用性验证**
在预期最大并发下，服务是否能正常运行？这是最基础的目标，测试结果是通过/失败。

**层次二：性能基准建立**
在不同并发规模下，TPS、延迟分布（P50/P95/P99）、CPU/内存/带宽占用各是多少？建立一条"性能曲线"，为后续优化提供对比基线。

**层次三：容量规划**
根据性能曲线，推算不同在线规模需要多少机器，预留多少弹性余量，为上线资源准备提供量化依据。

三个目标依次递进，前者是后者的前提。大多数游戏公司止步于层次一，但层次二和三才能真正降低上线风险。

### 用户行为模型

真实玩家在服务器上的行为不是均匀分布的。典型 FPS 游戏的用户行为链：

```
登录（1次） → 认证（1次） → 大厅等待（30-120s） → 匹配（10-60s）
→ 游戏中（10-30 分钟，高频操作）→ 结算（1次）→ 返回大厅 / 退出
```

每个阶段对服务器产生的负载截然不同：
- 登录/认证：短时间大量并发，数据库查询密集
- 大厅等待：轻量长连接，少量心跳包
- 匹配：匹配服计算密集，延迟敏感
- 游戏中：高频状态同步，带宽和 CPU 密集
- 结算：数据库写入密集

压测场景建模的第一步，是把这条行为链拆解为各阶段的**时间占比和消息频率**，而不是简单地"模拟 N 个并发用户发消息"。

## 具体实现

### Bot 客户端架构

游戏压测 bot 不是简单的 HTTP 请求生成器，它需要：
1. 维持长连接（TCP 或 WebSocket）
2. 处理服务端推送消息（不只是请求-响应模式）
3. 根据接收到的消息做出状态转换（如：收到匹配成功消息后，切换到游戏状态并开始发送操作包）
4. 模拟真实的用户操作时序（带随机抖动，不能所有 bot 同步发包）

#### 状态机模型

每个 bot 实例是一个状态机：

```python
class GameBot:
    def __init__(self):
        self.state = 'DISCONNECTED'
        self.conn = None
        self.player_id = None

    async def run(self):
        await self.connect()            # -> CONNECTED
        await self.login()              # -> AUTHENTICATED
        await self.enter_lobby()        # -> IN_LOBBY
        await self.request_match()      # -> MATCHMAKING
        await self.wait_for_match()     # -> IN_GAME
        await self.play_game(duration=random.uniform(600, 1800))
        await self.submit_result()      # -> SETTLING
        await self.return_to_lobby()    # -> IN_LOBBY

    async def play_game(self, duration: float):
        end_time = time.time() + duration
        while time.time() < end_time:
            # 模拟玩家操作：移动 + 射击
            await self.send_move(random_direction())
            await asyncio.sleep(random.uniform(0.05, 0.2))  # 50-200ms 间隔
            if random.random() < 0.3:  # 30% 概率射击
                await self.send_fire()
```

关键细节：`asyncio.sleep` 的随机抖动避免了所有 bot 在同一时刻发包（惊群效应），这在真实压测中非常重要。

#### 协议层封装

```python
class GameProtocolClient:
    def __init__(self, host: str, port: int):
        self.host = host
        self.port = port
        self.reader = None
        self.writer = None

    async def send(self, msg_type: int, payload: bytes):
        # 自定义二进制协议：4字节长度 + 2字节消息类型 + payload
        header = struct.pack('>IH', len(payload) + 6, msg_type)
        self.writer.write(header + payload)
        await self.writer.drain()

    async def recv(self) -> Tuple[int, bytes]:
        header = await self.reader.readexactly(6)
        length, msg_type = struct.unpack('>IH', header)
        payload = await self.reader.readexactly(length - 6)
        return msg_type, payload

    async def send_proto(self, msg_type: int, proto_msg):
        # Protobuf 序列化
        await self.send(msg_type, proto_msg.SerializeToString())
```

协议层需要与服务端代码保持同步。建议将 Protobuf 定义文件作为共享库，服务端和压测 bot 都依赖同一份定义，避免协议版本不一致导致的假阴性结果（bot 发包格式错误，被服务端拒绝，但误以为是服务端性能问题）。

### 流量回放 vs Bot 客户端

另一种方案是**流量回放**：在测试环境中录制真实用户流量，然后在压测时重放，并等比例放大。

```
录制阶段：
  生产环境 → 流量镜像 → 录制存储（去除敏感信息）

回放阶段：
  录制文件 → 流量回放工具 → 测试环境（N倍速放大）
```

**流量回放的优势**：真实反映用户行为分布，无需手写 bot 逻辑。
**流量回放的劣势**：
- 有状态协议的回放极其复杂——session token、角色 ID、序列号等状态数据在回放时需要重新建立，否则服务端会拒绝请求。
- 游戏服务端的大量消息是服务端主动推送的，不是请求-响应，回放只能覆盖"客户端发送"侧，无法验证"服务端推送"侧的行为。
- 协议加密/签名使回放更复杂。

**结论**：对于游戏服务器，bot 客户端方案通常比流量回放更可靠，但两者可以结合使用（用回放数据校准 bot 的行为分布）。

### 压测执行架构

单台机器无法模拟足够多的并发 bot（每个 bot 维持长连接，5 万 bot 需要 5 万个 socket）。需要分布式压测架构：

```
压测控制节点（Master）
  │
  ├── 压测执行节点 1（Worker，运行 1000 bots）
  ├── 压测执行节点 2（Worker，运行 1000 bots）
  │   ...
  └── 压测执行节点 N（Worker，运行 1000 bots）

Master 负责：
  - 协调所有 Worker 同时开始（避免不同节点启动时间差导致负载不均）
  - 收集所有 Worker 的指标数据，聚合计算全局 P99
  - 控制负载爬坡（每 30s 增加 500 并发）

Worker 负责：
  - 运行指定数量的 bot 实例
  - 记录本地延迟样本（不要在压测时实时传输大量原始数据到 Master）
  - 定期（如每 10s）向 Master 汇报聚合统计（均值、P99、错误率）
```

### 关键指标

#### 必测指标

| 指标 | 含义 | 告警阈值示例 |
|------|------|------------|
| 登录成功率 | 并发登录成功的比例 | < 99.5% 告警 |
| 匹配延迟 P99 | 99% 玩家的匹配等待时间 | > 30s 告警 |
| 游戏帧同步延迟 P99 | 服务端帧处理延迟 | > 50ms 告警 |
| 服务端 CPU 利用率 | 单核/总体 | > 70% 告警 |
| 服务端内存增长率 | 是否存在内存泄漏 | > 1GB/hour 告警 |
| 服务端出口带宽 | 网络发送量 | 接近带宽上限的 80% |
| 错误日志增长率 | 异常日志数/分钟 | 基准的 3× 告警 |

#### 延迟分布的正确解读

**不要只看平均值**。平均延迟 20ms 听起来很好，但如果 P99 是 500ms，意味着每 100 个玩家里有 1 个感受到严重卡顿。对于竞技游戏，P99 甚至 P999 才是真正重要的指标。

```
典型延迟分布示例：
  P50:   8ms   ← 大多数请求很快
  P75:  15ms
  P90:  35ms
  P95:  80ms   ← 开始出现"长尾"
  P99: 320ms   ← 1% 的请求严重超时
  P999: 2000ms ← 千分之一出现极端延迟（可能是 GC Pause）
```

长尾延迟的常见来源：JVM/Go GC 停顿、数据库慢查询、锁竞争。压测时必须在高并发下稳定运行足够长时间（建议 30 分钟以上），让 GC 充分触发，才能在数据中看到长尾。

### 容量规划

#### 从压测数据推算机器数量

假设压测数据显示：
- 单台 8 核游戏逻辑服，CPU 70% 时，稳定支撑 500 并发游戏中玩家
- 预期上线峰值：8000 并发游戏中玩家

基础计算：
```
需要逻辑服数量 = 8000 / 500 = 16 台（70% CPU 负载基准）
```

但这只是基础计算，实际需要叠加以下系数：

**弹性余量系数**：峰值通常比预期高 20-30%，保留 30% 余量：
```
= 16 × 1.3 = 20.8 ≈ 21 台
```

**故障容错系数**：N+1 冗余，允许 1 台故障时不影响服务：
```
最终配置：22-24 台
```

**新服务热身时间**：扩容需要时间（镜像拉取、进程启动），高峰来临前必须提前扩容。如果高峰增长曲线斜率大于扩容速度，静态预配比自动扩容更可靠。

#### 各层服务的容量比例

不同角色的服务器，性能瓶颈不同，容量规划要分开计算：

```
架构角色        瓶颈类型      基准（每台每核）
登录/认证服    数据库 I/O    约 1000 TPS（依赖 DB 性能）
匹配服         CPU 计算      约 2000 并发等待玩家
游戏逻辑服     CPU + 内存    约 500 并发游戏中玩家
状态同步服     网络带宽      取决于每玩家带宽用量
数据库层       磁盘 I/O      需要单独压测
```

### 常见压测陷阱

#### 陷阱一：测试环境与生产网络差异

测试环境通常在同机房甚至同机器内，bot 和服务端之间的网络延迟接近 0。生产环境中，玩家通过公网连接，RTT 50-100ms 很正常。

某些 bug 只在有网络延迟时暴露（如乱序包处理、心跳超时判断）。建议在压测中使用流量控制工具（Linux `tc netem`）人为注入延迟和丢包，模拟真实网络条件：

```bash
# 模拟 50ms 延迟，1% 丢包
tc qdisc add dev eth0 root netem delay 50ms loss 1%
```

#### 陷阱二：冷启动效应

服务端刚启动时，JIT 编译（Java/C#）、缓存预热（Redis 冷缓存）、数据库连接池建立都会导致初始性能低于稳定运行时。

如果压测在冷启动阶段就施加满负载，会看到大量超时，误认为服务器性能不达标，但其实稳定后完全没问题。

**正确做法**：压测分两阶段——先用 10-20% 负载运行 5 分钟预热，再爬坡到目标负载。P99 指标从预热结束后才开始记录。

#### 陷阱三：bot 行为与真实用户行为偏差

如果 bot 只发送移动包而不发送射击包，漏测了命中判定逻辑（可能是 CPU 瓶颈）。如果 bot 的消息频率远高于真实玩家，会在非关键路径上制造虚假瓶颈。

建议对生产日志做统计分析，提取真实的消息类型分布和频率作为 bot 行为的基准：

```python
# 从日志分析真实消息频率
message_stats = {
    'MOVE': {'freq_per_sec': 10, 'pct': 0.65},
    'FIRE': {'freq_per_sec': 2,  'pct': 0.15},
    'SKILL': {'freq_per_sec': 1, 'pct': 0.08},
    'CHAT': {'freq_per_sec': 0.1,'pct': 0.02},
    # ...
}
```

#### 陷阱四：忽略结算阶段的数据库写入峰值

游戏结束时，服务端要批量写入玩家数据（战绩、货币、装备）。如果一场对局 10 分钟，1000 场对局同时结束时，数据库面临 1000 × N 个并发写入。这个瞬时峰值往往比游戏进行中的峰值更高，但很容易在压测中被忽略（只测"游戏中"状态）。

## 工程边界

压测能发现"已知未知"——你知道需要测什么，测出来了。但无法覆盖"未知未知"，例如特定玩家操作序列触发的极端状态、节假日同时涌入的真实流量形态。

容量规划的数字是估算，不是保证。上线后的第一个高峰期，仍然需要值班工程师实时监控，准备随时扩容或降级。

---

## 最短结论

游戏服务器压测的核心是建好用户行为模型和实现忠实于真实协议的 bot 客户端，P99 延迟和容量规划数字才能有效指导上线决策。
