前面两篇分别讲了切包粒度和首包/分包/差量更新的边界划分。走到这一步，资源怎么切、怎么分已经有了判断框架。但无论资源怎么切包、怎么分发，最终都要面对一个绕不开的问题：这些 bundle 用什么方式压缩。

压缩不是一个孤立的构建参数。它直接影响四件事情：下载体积、磁盘占用、解压时的 CPU 开销、以及首次加载时用户感知到的延迟。选错压缩方案，后果不会在打包时暴露，而是在运行时才被用户感知。

这篇文章要解决的核心问题是：

`同一批资源，用 LZ4 还是 LZMA，这个选择到底在换什么？`

## 先给一句总判断

如果要把压缩选择压成一句话：

`压缩格式的选择不是"谁压得更小"的技术参数比较，而是下载体积、磁盘缓存、解压 CPU 开销、首次加载延迟和缓存管理复杂度之间的一次预算分配。脱离项目的网络环境、设备分布和加载时序讨论压缩格式，得出的结论一定是片面的。`

## 一、先拆清楚：资源的"大小"在不同阶段指的是不同的东西

在讨论 LZ4 和 LZMA 之前，先把一个容易混掉的事情拆清楚。

### 1. 构建产物体积

bundle 文件在磁盘上的大小，取决于压缩算法和原始内容。这是你在构建完成后直接能看到的文件大小。

### 2. 下载体积

通过网络传输的大小。通常等于构建产物体积。但 CDN 可能会再做一层传输压缩（gzip / brotli），这是网络层的事情，和 Unity 的压缩设置无关。

### 3. 磁盘缓存体积

下载到本地以后占用的空间。这一层经常被忽略，但它和下载体积不一定相等。

如果是 LZ4 bundle，缓存体积就等于下载体积——文件下到本地就是最终形态，不需要二次处理。

如果是 LZMA bundle，情况完全不同。Unity 默认会在首次加载时把 LZMA bundle 解压并以 LZ4 格式重新写入本地缓存。这意味着缓存体积比下载体积大——你用 LZMA 省下的下载量，会在磁盘缓存里"还回来"。

### 4. 运行时内存占用

资源被反序列化成 Native Object 后在内存里的大小。这和压缩格式完全无关，取决于资源本身——一张 2048x2048 ASTC 贴图解压后在 GPU 内存里的占用，不管原来是 LZ4 还是 LZMA 压缩的都一样。

### 为什么这四层容易混掉

因为日常讨论里经常出现这样的说法：

- "LZMA 更小"——对的，但只在下载体积这一层。缓存体积可能更大，首次加载更慢
- "LZ4 更快"——对的，但只在解压和加载这一层。下载体积更大，传输更慢
- "不压缩最好"——只在资源本身已经是压缩格式（ASTC、ETC2、压缩音频）的特定场景下成立

`脱离阶段讨论压缩，得出的结论一定是片面的。`

## 二、LZMA 到底在做什么，为什么首次加载特别慢

LZMA（Lempel-Ziv-Markov chain Algorithm）是一种高压缩率算法。它使用大窗口字典和范围编码，能够在比 LZ4 大得多的上下文窗口中寻找重复模式，因此压缩率显著更高，同样的资源通常能比 LZ4 再小 30%–50%。

但高压缩率的代价集中在三个地方：

### 1. 解压速度慢，CPU 开销大

LZMA 的解压需要维护和重建解码状态，计算量显著高于 LZ4。在中低端移动设备上，解压一个几十 MB 的 LZMA bundle 可能需要数百毫秒甚至几秒。而且这段解压是 CPU 密集型的，会和其他需要 CPU 的逻辑争抢。

### 2. 不支持随机访问

LZMA 是流式压缩，数据之间有前后依赖关系。要读取中间某个位置的数据，必须从头解压到那个位置。这意味着即使你只需要 bundle 里的一个资源，也必须付出整包解压的成本。

这和 LZ4 的块级随机访问形成了根本区别。

### 3. Unity 的 LZMA 首次加载行为

这是最容易造成线上问题的一环。当 Unity 在运行时首次打开一个 LZMA bundle 时（通过 `AssetBundle.LoadFromFile`、`LoadFromCacheOrDownload` 或 Addressables 的加载链），引擎会执行以下流程：

- 读取整个 LZMA 文件到内存
- 在 CPU 上执行完整解压
- 把解压后的数据以 LZ4 Chunk-Based 格式重新写入本地缓存目录（`UnityCache`）
- 关闭 LZMA 源文件
- 打开新写入的 LZ4 缓存文件
- 然后才开始正常的 bundle 打开和资源加载流程

这个过程有几个值得注意的成本：

- 临时内存峰值：解压过程中，LZMA 原件和解压后数据可能同时存在于内存中
- CPU 时间：解压本身消耗大量 CPU 时间，在中低端设备上可感知
- 磁盘 I/O：解压后的 LZ4 文件需要写回磁盘，在 eMMC 存储的低端设备上写入速度是瓶颈
- 这整个过程只发生在首次加载。一旦 LZ4 缓存建立，后续加载走缓存，速度和直接使用 LZ4 bundle 一样

所以 LZMA 的交换逻辑是：

`用更高的首次加载代价（CPU + I/O + 临时内存 + 缓存写入），换更小的下载体积。首次之后的代价和 LZ4 一样。`

## 三、LZ4 到底在做什么，为什么运行时几乎无感

LZ4 是一种以速度为优先设计的压缩算法。它使用较小的匹配窗口和简单的编码方式，压缩率不如 LZMA，但解压速度可以达到内存拷贝级别。

Unity 使用的是 LZ4 Chunk-Based Compression（也叫 LZ4HC 构建 + LZ4 运行时解压）。构建时使用 LZ4HC（High Compression）模式获得更好的压缩率，运行时使用标准 LZ4 解压，速度极快。

### 1. 块级随机访问

LZ4 Chunk-Based 的关键设计是：bundle 的内容被切成固定大小的块（Unity 内部默认 128KB），每个块独立压缩。运行时加载某个资源时，只需要解压包含该资源数据的那几个块，而不是整个文件。

这带来的直接好处是：

- 加载小资源时，只解压很小一部分数据，延迟极低
- 不需要把整个 bundle 展开到内存
- 操作系统的文件系统缓存可以更高效地工作

### 2. 不需要重新缓存

LZ4 bundle 下载到本地后就是最终形态。不存在"首次加载解压重缓存"的过程。下载体积等于缓存体积，磁盘占用可预测。

这对缓存管理来说是一个很大的简化：你看到的 bundle 文件大小就是它在设备上的最终占用，不会出现"下载了 50MB 结果磁盘占了 80MB"的情况。

### 3. 解压开销可忽略

LZ4 的解压速度通常在 GB/s 级别（取决于设备内存带宽）。对于游戏中常见的几百 KB 到几十 MB 的 bundle，解压时间在亚毫秒到几毫秒级别，在 Profiler 里几乎看不到。

所以 LZ4 的交换逻辑是：

`用更大的下载和磁盘体积，换更低的运行时延迟、更简单的缓存模型和更可预测的磁盘占用。`

## 四、为什么"下载更小"不一定代表"体验更好"

这是最容易被误判的一层。很多项目选 LZMA 的理由是"包体小，下载快"，但忽略了下载以后还有一段路要走。

### 1. 首次加载的等待链

考虑一个用户首次进入某个系统，需要加载 3 个 LZMA bundle（各 20MB 压缩，解压后各 35MB）：

- 下载：3 × 20MB = 60MB（LZMA 确实省了）
- 首次打开：每个 bundle 需要 LZMA 全量解压 + LZ4 重新写盘。假设每个耗时 2 秒，总计 6 秒
- 这 6 秒是纯等待，用户看到的是 Loading 或卡顿

如果同样的内容用 LZ4 压缩（各 30MB）：

- 下载：3 × 30MB = 90MB（多了 30MB）
- 首次打开：无需解压重缓存，直接 LZ4 块级读取，几乎无感
- 省下了 6 秒的 Loading

多下载的 30MB 在 Wi-Fi 环境下可能只多花几秒，但省下的 6 秒 Loading 是用户直接感知的体验差异。

### 2. 多个 LZMA bundle 同时首次加载的叠加问题

更严重的是：如果多个 LZMA bundle 的首次加载恰好发生在同一段时间（比如进入一个新场景需要同时加载好几个 bundle），CPU 解压开销会叠加，内存峰值会叠加，磁盘写入会争抢 I/O。这种叠加在中低端设备上可能导致严重的帧率波动甚至 ANR。

### 3. 在 Profiler 里怎么看

当你在 Unity Profiler 的 CPU 模块里看到首次加载卡顿时，关注以下标记：

- `AssetBundle.LoadFromFile` 或 `AssetBundle.LoadFromFileAsync` 的耗时：如果首次特别长、后续正常，很可能是 LZMA 解压 + 重缓存的代价
- `Decompression` 或 `LZMA` 相关的标记
- `File.Write` 或磁盘 I/O 相关的标记（重缓存写盘）

如果确认卡顿来自 LZMA 首次解压，解决方案不是"优化加载代码"，而是回到构建侧重新评估压缩策略。

## 五、Uncompressed 什么时候是正确选择

除了 LZ4 和 LZMA，还有第三个选项：不压缩。大部分情况下不推荐，但有一个场景值得认真考虑：

`当 bundle 里的资源几乎全部是已经被内容压缩过的格式时。`

- ASTC / ETC2 / BC7 贴图：这些 GPU 压缩格式本身已经大幅压缩了数据。对这类数据再做 LZ4/LZMA 压缩，压缩率极低（通常只能再省 5%–10%），但仍然要付出解压的 CPU 开销
- 压缩音频（Vorbis / AAC）：同理，已经是有损压缩后的数据
- 压缩视频：更不需要二次压缩

如果一个 bundle 里绝大部分内容都是上述格式，不压缩（`BuildAssetBundleOptions.UncompressedAssetBundle`）反而是更合理的选择——省掉了几乎没有收益的解压开销。

但如果 bundle 里混有未压缩的 Mesh、AnimationClip、ScriptableObject 等数据，这些内容对 LZ4 压缩仍然有显著收益。所以关键不是"不压缩更好"，而是要看 bundle 内容的组成。

## 六、平台差异会怎样影响压缩选择

同一种压缩方案，在不同平台上的体验差异可能远大于预期。

### Android

- 存储性能差异极大。高端旗舰用 UFS 3.1/4.0，I/O 很快；低端机用 eMMC，读写速度低一个数量级。LZMA 的缓存重写在 eMMC 设备上特别痛苦
- CPU 性能差异也很大。LZMA 解压在大核上可以接受，但如果调度到小核上执行，耗时会翻倍
- Google Play 的 AAB 会对 APK 中的资源做自己的压缩。如果你的 bundle 跟首包走（放在 `StreamingAssets`），AAB 打包时会再压一层。LZ4 bundle 被 AAB 再压效果还行，LZMA bundle 被 AAB 再压几乎没有额外收益

### iOS

- 存储性能相对一致且较快，LZMA 重缓存的磁盘 I/O 瓶颈比 Android 低端机小
- App Store 对 IPA 包体有自己的压缩和瘦身（App Thinning），实际下载体积和构建体积可能不一致
- iOS 设备的 CPU 性能通常较好，LZMA 解压的绝对耗时比同期 Android 中低端设备低

### PC

- 存储和 CPU 通常不是瓶颈。LZMA 的首次解压代价相对可接受
- 但磁盘缓存体积可能是问题——PC 用户对安装后占用空间的感知比移动端强

总的来说：

`越是低端设备分布广的平台，LZ4 的安全边际越高。越是网络条件差但设备不差的场景，LZMA 的收益越明显。`

## 七、构建配置到底怎么映射

在 Unity 的构建系统里，压缩选项最终映射到 `BuildAssetBundleOptions`：

- `ChunkBasedCompression`：LZ4（推荐默认值）
- `None`（不指定压缩相关 flag）：LZMA（Unity 的历史默认值）
- `UncompressedAssetBundle`：不压缩

Addressables 的压缩配置在 Group Schema 的 `BundledAssetGroupSchema` 里设置，选项是 `LZMA`、`LZ4` 或 `Uncompressed`。注意这是 Group 级别的，同一个 Group 里的所有 bundle 共享压缩设置。如果你需要对不同内容使用不同压缩策略，必须拆成不同 Group。

YooAsset 同样在 package 配置中指定压缩方式，最终映射到 `BuildAssetBundleOptions`。

还有一个容易忽略的配置：CDN 传输层的压缩。

- 对于 LZ4 bundle，CDN 的 gzip/brotli 传输压缩仍然有一定收益，因为 LZ4 的压缩率不算极致，传输层压缩能再省 10%–20%
- 对于 LZMA bundle，CDN 再压缩几乎没有额外收益，LZMA 已经接近理论压缩极限
- 所以在评估"最终到达用户的体积"时，不能只看 Unity 构建产物的大小，还要考虑 CDN 传输压缩的差异

## 八、常见误判

### "全用 LZMA 包最小，体验最好"

下载体积最小不等于用户体验最好。首次加载的解压 + 重缓存开销可能比多下载几十 MB 的 LZ4 更让用户感知到延迟。

### "LZ4 太大了，浪费流量"

LZ4 的压缩率虽然不如 LZMA，但差距通常在 30%–50% 之间，不是数量级的差异。而且 CDN 传输压缩会进一步缩小差距。

### "先用 LZMA 上线，后面再改"

压缩方案一旦上线，用户设备上的缓存都是按当前方案建立的。中途切换压缩方式意味着所有已缓存内容失效，用户需要重新下载全部内容。这个切换成本不小。

### "已压缩贴图也应该用 LZ4"

对 ASTC/ETC2 等 GPU 压缩格式再做 LZ4 压缩，收益通常不到 10%。如果 bundle 里全是这类贴图，不压缩反而更好——省掉解压 CPU 开销，bundle 加载速度更快。

### "所有 bundle 应该用同一种压缩"

不同内容的使用场景不同，压缩策略也应该不同。首包资源、常驻资源、低频大体积资源、高频热更资源，对下载体积和加载速度的敏感度完全不同。一刀切的压缩策略必然在某些场景上过度付出代价。

## 九、项目判断表

| 场景 | 推荐压缩 | 核心理由 |
|------|---------|---------|
| 首包基础资源 | LZ4 | 无网络下载，首帧速度优先 |
| 热更包（Wi-Fi 用户为主） | LZ4 | 解压快，缓存模型简单 |
| 热更包（弱网 / 流量敏感） | LZMA | 压缩率优先，首次等待可用 Loading Screen 消化 |
| 常驻核心包 | LZ4 | 频繁加载卸载，低延迟累积优势 |
| 低频大体积一次性内容 | LZMA | 只加载一次，带宽节省明显 |
| 几乎全是 GPU 压缩贴图/压缩音频的包 | Uncompressed | 二次压缩收益极低，省解压开销 |
| PC 平台 | LZ4 或 LZMA 均可 | CPU 和存储通常不是瓶颈，按需选择 |

## 这一篇真正想立住的判断

`压缩格式的选择，本质上是在网络带宽预算和运行时延迟预算之间做分配。LZ4 把代价留在下载侧，LZMA 把代价留在首次加载侧。没有"更好"的压缩格式，只有"在你的项目约束下代价更低"的选择。`

`如果你的项目在中低端 Android 设备上出现首次加载卡顿，不要先怀疑加载代码——先看是不是 LZMA bundle 的首次解压和重缓存在付出代价。`

下一篇会进入运行时加载层面：当你说"异步加载"的时候，到底异步了哪几段，卡顿又卡在了哪几段。
