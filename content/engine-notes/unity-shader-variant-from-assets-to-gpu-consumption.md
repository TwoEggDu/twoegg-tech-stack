前面几篇分别讲了 `Shader Variant` 的来源、保留依据、剔除层级和运行时命中。
但项目现场最容易卡住的，往往不是其中某一个局部按钮，而是中间那条没有被完整说清的桥：

`Shader / Material / SVC 这些资产，究竟是怎样一步步变成目标构建里的可用结果，最后又怎样在运行时被命中并交给 GPU 的。`

这条桥一旦没立住，团队就很容易把几种完全不同的问题混成一句话：

- `这个 shader 明明在包里`
- `这个 material 也没有丢引用`
- `SVC 里我也加了`
- `为什么真机上效果还是不对`

如果只留一句总判断，我会这样压：

> `Shader`、`Material`、`SVC` 站在资产定义层；`variant 保留与剔除` 站在构建产物层；`SetPass 命中与 WarmUp` 站在运行时消费层。GPU 真正消费的，不是某个抽象的 shader 资产，而是某个 pass 在某组条件下最终留下来的那份平台相关程序结果。

这篇就只做一件事：

`把一条 Shader Variant 从资产，到构建产物，到运行时命中，再到 GPU 消费的整条链路串起来。`

---

## 一、先把三层坐标钉住，不然后面一定会混

理解 `Shader Variant`，最怕一上来就把“资产”“编译结果”“运行时命中”混成一层。
更稳的拆法是先分成三层：

1. `资产定义层`
2. `构建产物层`
3. `运行时消费层`

这三层分别回答的是不同问题：

- `资产定义层` 回答：项目里到底声明了什么、引用了什么、显式登记了什么
- `构建产物层` 回答：哪些路径真的被保留下来，并写进了目标交付物
- `运行时消费层` 回答：当前 draw call 到底命中了哪条现成路径，GPU 最终拿到的是哪份程序

只要这三层一混，后面的判断就会连续出错：

- 把 `Material` 当成“已经编译好的 GPU 程序”
- 把 `SVC` 当成“保证一切都一定存在的保险箱”
- 把“包里有 shader”误解成“运行时一定会命中正确 variant”
- 把“首帧卡顿”误判成“变体被剔除了”

所以这篇后面所有内容，都会按这三层往前推，而不是按按钮名词堆。

---

## 二、资产定义层：项目里到底有哪些和 variant 相关的东西

### 1. Shader 决定的是理论可能空间

`Shader` 最核心的职责，不是“直接交给 GPU 跑”，而是定义一套可被后续构建系统继续处理的渲染模板。
它提供的是：

- 有哪些 `SubShader` 和 `Pass`
- 有哪些编译期分支
- 有哪些 keyword 组合可能成立
- 某些平台或管线条件下有哪些分叉

所以从资产层看，`Shader` 更像是：

`后续构建系统可以继续展开、筛选和编译的源头定义`

它还不是最终设备上那份稳定可执行的结果。

### 2. Material 决定的是“怎样使用某个 Shader”

`Material` 保存的重点也不是 GPU 程序本体，而是：

- 指向哪个 `Shader`
- 当前这份材质有哪些属性值
- 当前启用了哪些功能状态
- 某些 keyword 或等价功能路径当前处于什么状态

所以 `Material` 更接近：

`某个 Shader 的一份具体使用配置`

这也是为什么同一个 `Shader` 可以被许多 `Material` 以完全不同的方式使用，而这些使用方式最后又会反过来影响构建期对 variant 的保留判断。

### 3. SVC 决定的是“哪些路径值得被显式记住”

`ShaderVariantCollection` 很容易被误解成“把 shader 编译结果装进去”。
但它更稳的定位其实是：

`把项目显式关心的 shader / pass / keyword 组合登记出来`

这件事的价值在于：

- 给构建期提供额外的保留依据
- 给运行时 WarmUp 提供可操作的清单
- 给团队提供一份可回归、可审计的关键路径名单

所以 `SVC` 更像“清单”，不是“平台程序容器”。

### 4. Graphics Settings、URP Asset、Renderer Feature 也是资产输入的一部分

很多团队会把 `Shader`、`Material`、`SVC` 当成资产输入的全部，但这其实不够。
从 Unity 实际构建链路看，下面这些配置同样会改变一条 variant 的命运：

- `Graphics Settings`
- `Always Included Shaders`
- 当前生效的 `URP Asset`
- 当前生效的 `Renderer`
- 某些全局图形开关和质量档

因为它们会直接影响：

- 哪些路径被视为“本次构建有可能发生”
- 哪些路径会被提早过滤
- 某份 shader 最后由谁负责交付

所以只看 `Material` 和 `SVC`，通常是不够解释现场问题的。

---

## 三、从资产到候选集：不是项目里有，就等于这次构建会考虑它

到了构建期，Unity 面对的第一个核心问题不是：

`这个项目里理论上有多少 variant`

而是：

`这次构建里，哪些 shader / pass / keyword 路径真的值得进入正式候选集`

这里最容易被忽略的一点是：

`项目里存在` 不等于 `参与了本次构建输入`

真正会给这次构建提供使用面的，通常是这些东西：

- 参与当前 `Player` 构建的场景和对象
- 被当前构建链收集到的 `Material`
- 当前参与构建的 `AssetBundle / Addressables` 输入
- 显式登记的 `SVC`
- 某些全局图形设置带入的 shader 边界

这一步的工程含义非常大。
它意味着：

- 某个 `Material` 明明存在于项目里，但如果不在这次构建输入中，它未必能为这次构建贡献使用面
- 某条运行时一定会走到的路径，如果静态输入看不见，就需要 `SVC` 这种显式补充
- 某些全局 shader 虽然不是由当前场景自然带出来，但会因为全局策略进入另一条交付路径

所以从资产走到构建，不是“一份 shader 被整体搬运”，而是先经历一次：

`由本次构建输入决定的候选收缩`

---

## 四、构建产物层：候选路径怎样变成真正留下来的平台结果

有了候选集，接下来才轮到真正影响“活下来多少”的那几步。

### 1. 先不是编译全部，而是继续过滤“这次构建不可能发生的路径”

尤其在 SRP，特别是 URP 里，很多路径不是先完整展开再慢慢删，而是会更早被配置过滤。
这层过滤看的不是“项目里有没有人关心它”，而是：

`当前管线配置是否承认它在这次构建里真的可能发生`

所以你会遇到那种很典型的情况：

- 某条 keyword 路径在 `SVC` 里
- 项目里似乎也有材料在用
- 但当前 `URP Asset / Renderer Feature / Graphics API` 并不承认这条路径
- 它于是会在更前面就消失

这也是为什么很多人以为是自定义 `OnProcessShader` 把东西删了，实际上问题发生得更早。

### 2. 然后才轮到内置 stripping、SRP stripping 和项目 stripping

只有那些仍然活着的候选路径，才会继续进入后面的剔除链。
这些剔除通常包括：

- Unity 内置的通用 stripping
- SRP 自己的 stripping 规则
- 项目自定义的 `IPreprocessShaders`

这一步真正决定的是：

`前面还活着的路径里，哪些仍然值得被带进目标构建`

所以构建产物层看到的，不是一份“完整理论空间”，而是不断过滤后的剩余结果。

### 3. 最终留下来的，不是抽象 keyword，而是平台相关的程序结果

这一步是整条桥里最值得钉住的一句。

很多讨论还停留在：

`这个 shader 的这个 keyword 有没有`

但设备最终不会消费“keyword 概念”。
它最终消费的，是某个 `Pass` 在某组条件下留下来的平台相关程序结果。

更工程一点地说，最后真正能被设备使用的，更接近于：

- 某个 shader 路径对应的目标平台编译结果
- 某个 pass 下可用的子程序集合
- 某份与图形 API、平台和当前配置相关的程序片段

所以一条 variant 真正“留下来”，不是停留在抽象清单里，而是：

`最终生成了目标平台可用的那份结果，并被写进正确的交付边界`

---

## 五、交付边界：活下来的结果最后是写进 Player，还是写进 Bundle

到这里，很多团队会以为问题已经结束了。
其实还差一层特别容易踩坑的边界：

`这些活下来的结果，最后到底由谁负责交付`

### 1. Player 视角下的“留下来”

如果一条路径最终由 `Player` 全局负责，那它的含义更接近：

`目标包体的全局图形环境里存在这份 shader 能力`

这时运行时很多对象只是在引用它，而不是自己再单独携带一整份可用结果。

### 2. AssetBundle 视角下的“留下来”

一旦进入 `AssetBundle`、热更新包或多包协作场景，“留下来”的含义就会变得更具体：

`这次由哪个交付物实际带着它，谁在承担它的存在责任`

这时候你必须区分：

- bundle 只是引用某份全局 shader 能力
- 还是 bundle 自己负责带齐对应路径

这也是为什么同一条 shader 路径，在纯 `Player` 场景里正常，换到 bundle 场景里问题就会突然爆出来。

### 3. Always Included 改变的是“谁来负责交付”

`Always Included` 在这条桥里最关键的意义，不是“更大的保留名单”，而是：

`把某份 shader 的交付责任提升到更全局的边界`

所以它解决的问题往往不是：

`这组具体 variant 清不清楚`

而是：

`这份 shader 到底应该由局部内容自证，还是由全局 Player 兜底`

一旦把这层理解清楚，很多“为什么一加 Always Included 就好了”的现象就不再神秘。

---

## 六、运行时消费层：Material 是怎样命中一条现成路径的

真正到了运行时，系统回答的问题已经不再是“保不保留”，而是：

`当前 draw call 到底会命中哪条已经存在的可用路径`

### 1. Material 先把 Shader 引用和状态接回来

当一个 `Material` 在运行时被加载出来时，它首先会恢复：

- 自己引用的是哪个 `Shader`
- 当前属性值是什么
- 当前启用了哪些功能状态

如果这一层就断了，现场通常更像是：

- shader 引用直接丢失
- 材质退回错误状态
- 对象明显缺失可用 shader

### 2. 渲染时再根据 pass 和当前状态去命中一条可用路径

真正决定“这次 draw call 用哪条 variant”的，是运行时命中过程。
系统会结合：

- 当前 `Material`
- 当前 `Pass`
- 当前 keyword 状态
- 当前平台与图形 API 条件

去找一条当前可接受的路径。

这一步最重要的结论是：

`运行时不是在重新发明一条 variant，而是在已经留下来的结果里做命中`

所以如果构建期根本没留下来，运行时再聪明也变不出来。

### 3. 命中正确路径，和拿到正确显示结果，不是两句重复的话

这一点很值得单独强调。
运行时如果没命中预期路径，可能出现三种完全不同的现象：

- 没有任何可接受结果：直接粉或错误 shader
- 有可接受但不精确的退化路径：材质不粉，但效果不对
- 路径存在，只是第一次用时还没准备好：首帧卡顿

这也是为什么“显示不对”“材质变粉”“第一次卡一下”不能都粗暴归结成一句“shader 出问题了”。

---

## 七、GPU 真正消费的是什么：不是 shader 资产，而是最终命中的那份平台程序

把整件事走到最后，GPU 侧真正接到的，并不是：

- 某个 `.shader` 资产
- 某个 `Material` 文件
- 某份 `SVC` 清单

GPU 真正能消费的，是：

`当前 draw call 最终命中的那份平台相关程序结果`

你可以把前面的层级压成这条更接近设备侧的链：

```text
Shader / Material / SVC
-> 本次构建输入形成使用面
-> 过滤与 stripping 留下可用路径
-> 写进 Player 或 Bundle 的正确交付边界
-> 运行时按 pass 与状态命中一条已存在结果
-> 驱动 / GPU 使用那份平台程序
```

这条链最值得记住的工程结论有三个：

1. GPU 不关心你项目里“理论上声明了什么”，它只关心当前最终拿到的是哪份可执行结果。
2. `Material` 负责表达使用意图，不负责保证目标平台程序一定存在。
3. `SVC` 能帮助保留和预热关键路径，但它本身不是 GPU 消费对象。

---

## 八、WarmUp 站在这条桥的最后端，它解决的不是“存在性”，而是“准备时机”

`WarmUp` 特别容易被误放到前面。
但如果按这篇的整条链看，它其实站在很后面。

它回答的不是：

`这条路径有没有留下来`

而是：

`这条已经存在的路径，要不要在真正首用之前先准备好`

所以 `WarmUp` 的前提一定是：

- 这条路径已经被正确保留
- 它已经被正确写进目标交付边界
- 运行时也确实会走到它

如果这三个前提里任何一个不成立，`WarmUp` 都不是解法。

这也是为什么很多项目现场会出现这种误判：

- 看到首帧卡顿，就去加 `SVC`
- 看到效果不对，也去做 `WarmUp`

但如果根因是构建期就没留下正确路径，这些动作都只是绕错层。

---

## 九、为什么“包里有资产”仍然可能不等于“GPU 用到了正确 variant”

这句话几乎是整篇最关键的落点。

很多现场争论，最后都可以被压成一个事实：

`资产存在`、`引用存在`、`正确平台结果存在`、`运行时命中正确路径`，是四件不同的事。

它们之间任何一层断掉，最后都可能表现为“画面不对”。

最常见的断裂点是这几种：

### 1. 资产层就没提供完整保留依据

比如：

- 当前构建输入根本没覆盖到那份 `Material`
- 运行时关键路径静态输入看不见，但 `SVC` 也没补

### 2. 构建层把它提早判成“不可能发生”

比如：

- 当前 `URP Asset`
- 当前 `Renderer Feature`
- 当前图形 API

并不承认这条路径。

### 3. 路径还活着，但最后没进正确交付边界

比如：

- 你以为 bundle 自己负责带齐
- 实际却依赖 `Player` 全局兜底
- 或反过来

### 4. 路径存在，但运行时没有命中你以为的那一条

这类问题往往最难看出来。
因为它表面上不是“完全坏了”，而是：

- 不粉
- 也能渲染
- 但效果总差一点

本质上，GPU 消费的是另一条可接受但不精确的路径。

---

## 十、以后遇到问题，先沿着这条桥追，而不是直接追某个按钮

如果以后你再遇到类似：

- `为什么这个 material 在编辑器对，真机不对`
- `为什么 SVC 里有，最终包里还是没有`
- `为什么 shader 明明在 bundle 里，效果还是不对`

更稳的排查顺序不是直接问某个按钮开没开，而是按这条链追：

1. 这条路径在 `Shader` 里到底有没有被声明出来
2. 这次构建输入里，谁在为它提供保留依据
3. 当前管线配置是否承认它在本次构建里可能发生
4. 后续 stripping 有没有把它裁掉
5. 它最终由 `Player` 还是 `Bundle` 负责交付
6. 运行时是否真的命中了你预期的那条路径
7. 如果路径存在，是否只是首用时机没准备好

只要按这七步问，很多看起来很“玄学”的 shader 问题，都会从一句模糊的：

`GPU 怎么没吃到这个 shader`

变成一句可以落地定位的话：

`这条路径到底死在资产输入、构建过滤、交付边界，还是运行时命中`

---

## 结论

最后把整篇压成几句最值得记住的话：

1. `Shader / Material / SVC` 是资产定义，不是 GPU 最终消费对象。
2. 真正决定设备侧能不能用到某条路径的，是构建期保留下来的平台相关程序结果。
3. `Always Included` 解决的是交付责任边界，`SVC` 解决的是显式保留与预热清单，它们不在同一层。
4. 运行时只能命中“已经留下来的东西”，不能凭空救回构建期已经消失的 variant。
5. GPU 消费链的正确心智模型，不是“shader 在不在包里”，而是：

`这条路径有没有被资产层表达出来，有没有被构建层留下来，有没有被交付层带到目标环境里，最后有没有被运行时正确命中。`
