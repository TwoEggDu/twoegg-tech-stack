引入 DOTS 之后，最容易出现的错觉是“跑起来了，就说明值了”。这在原型阶段还勉强说得过去，但在项目里不行。项目里真正需要的，不是一次成功演示，而是一条能重复、能对比、能回归的验证链。

如果没有验证链，DOTS 迁移就会很快退化成争论：有人觉得更快，有人觉得更复杂，有人觉得编译慢了。最后谁都没有错，但项目也没法继续决策。

---

## 为什么验证比实现更重要

DOTS 迁移的收益，通常不会只体现在一个数值上。它可能同时影响：

- 主线程时间。
- 批量仿真的稳定性。
- GC 压力。
- 结构边界是否更清楚。
- 构建链和回归链是否更可控。

这意味着你不能只盯着“帧率高没高一点”。如果只看一个数，很多有价值的变化会被忽略；如果什么都不看，迁移又会变成主观争论。

所以验证的目标不是“证明 DOTS 一定更快”，而是证明：**在你定义的问题空间里，它是否真的改善了你关心的工程指标。**

## 最小可重复验证长什么样

一个合格的验证，至少要满足四件事：

1. 场景固定。
2. 输入固定。
3. 指标固定。
4. 运行方式固定。

如果这四项都不固定，你测出来的任何结果都只能算偶然值。

最小验证通常不是整项目压测，而是一个可复现的 Pilot 场景，比如：

- 1000 个单位移动。
- 5000 个子弹生命周期。
- 某个技能链重复触发。
- 某个简单仿真层的固定步长回放。

重点不在规模大小，而在于它能不能稳定复现迁移前后的差异。

## A/B 基线怎么做

A/B 基线不是“旧版和新版都跑一下”，而是要把对照条件写死。

```text
A 组:
  旧实现
  同一场景
  同一输入
  同一帧数
  同一平台

B 组:
  DOTS 试点实现
  同一场景
  同一输入
  同一帧数
  同一平台

比较项:
  主线程时间
  GC 分配
  关键系统耗时
  回归结果一致性
```

这里最重要的不是“跑两次”，而是保证两次对照真的只差一个变量。只要变量不干净，验证结果就没有意义。

## 阈值应该怎么定

阈值不是越细越好，而是要能支持决策。

一条实用的做法，是把阈值分成三层：

- 通过阈值：达到这个值，说明试点值得继续。
- 警戒阈值：结果不稳定，但还可以再测一轮。
- 退出阈值：说明这条路径不值得继续投入。

举例来说，阈值可以不是绝对帧率，而是某个更贴近项目的指标，比如：

- 主线程时间下降是否稳定。
- 目标场景下是否减少明显的 GC 峰值。
- 桥接层是否没有被扩大成第二套业务核心。
- 迁移后的维护成本是否仍在团队可承受范围内。

没有阈值的验证，只能告诉你“有变化”，不能告诉你“要不要继续”。

## 证据链要分层

DOTS 验证最容易漏掉的一点，是证据链不能只看一个窗口。

建议至少分四层看：

### 1. 编译证据

先确认代码能稳定编过，AOT 和目标平台构建也能过。没有这一步，后面的运行时数据没有意义。

### 2. 运行证据

看目标场景的真实表现，包括主线程、系统耗时、批量更新结果和重复执行稳定性。

### 3. 产物证据

看 Baking 产物、桥接数据、资源引用和运行时快照是否符合预期。很多问题不是跑不起来，而是产物已经悄悄偏了。

### 4. 回归证据

看这个验证能不能在下个版本、下个平台、下次重跑时仍然成立。不能重现的收益，不算工程收益。

## 一个简单的验证流程

```csharp
ValidationResult ValidateDotsMigration()
{
    var baseline = RunScenario(useDots: false);
    var dots = RunScenario(useDots: true);

    if (!baseline.IsStable || !dots.IsStable)
        return ValidationResult.Fail("scenario unstable");

    if (!WithinThreshold(baseline.MainThreadMs, dots.MainThreadMs))
        return ValidationResult.Fail("performance threshold not met");

    if (!SameFunctionalOutcome(baseline, dots))
        return ValidationResult.Fail("behavior mismatch");

    return ValidationResult.Pass();
}
```

这个流程只说明一件事：验证不是为了“看上去更快”，而是为了在功能一致的前提下判断收益是否成立。

## 验证里最容易犯的错

第一个错，是只看一次结果。一次压测很容易被缓存、调度、编辑器状态和场景随机性误导。

第二个错，是只看性能，不看行为一致性。DOTS 迁移如果引入了边界偏移，性能提升也不能算真正通过。

第三个错，是把验证做成“高级版截图”。如果验证不能指导下一步决策，它就只是展示，不是工程判断。

## 什么时候说明这条路值得继续

当下面几件事同时成立时，基本就可以继续扩张试点：

- A/B 基线稳定。
- 目标场景的性能收益能重复出现。
- 行为结果和旧实现一致，或者偏差被解释清楚。
- 团队能继续维护验证链，而不是只靠某个人手工跑。

这时，DOTS 就不再只是一个尝试，而是一个可以进入第二阶段的工程方案。

## 小结

DOTS 值不值，不能靠“跑起来了”的感觉判断，只能靠最小可重复验证、A/B 基线、阈值和证据链判断。
如果验证不能稳定复现收益，或者不能说明行为一致性，那就说明这次迁移还没到可继续扩张的程度。
