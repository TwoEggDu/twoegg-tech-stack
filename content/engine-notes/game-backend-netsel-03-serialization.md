## 问题空间：协议栈里那层被忽视的位置

传输层选好了（UDP/KCP），接下来的问题是：你的游戏消息以什么**格式**在网络上传输？

这个问题通常在立项初期被轻描淡写地解决："先用 JSON，后面再换"。然后后面就没有"后面"了，因为消息格式一旦定下来、客户端开始接入，改起来的工程量会大到没人愿意动。

序列化层坐落在协议栈的应用层，在传输层之上：

```
应用逻辑（游戏帧、状态机）
    ↓ 序列化
应用层（消息格式：JSON / Protobuf / FlatBuffers / 自定义二进制）
    ↓ 加密/压缩
传输层（KCP / TCP / UDP）
    ↓
网络
```

它的职责是：把内存中的结构体（玩家位置、技能数据）变成可以传输的字节流，以及把收到的字节流还原回结构体。这个过程叫**序列化（Serialize）**和**反序列化（Deserialize）**。

序列化的性能直接影响服务端的 CPU 消耗和每帧延迟。格式的设计影响包体大小，进而影响带宽成本。Schema 的版本兼容策略影响你能不能在不停服的情况下做热更新。

---

## JSON 的代价：文本解析是隐形税

很多游戏团队在原型阶段用 JSON，理由充分：可读性好、调试方便、工具链成熟。但当游戏进入每秒 30 次状态同步的实时阶段，JSON 的代价就变得不可忽视。

### 包体大小

```json
{"x":103.5,"y":0.0,"z":88.2,"facing":180.0,"health":95,"seq":12345}
```

这条玩家状态消息，JSON 编码后是 64 字节。

等效的 Protobuf 二进制编码：约 18-22 字节。

差距 3 倍。对于一个 100 人 MOBA，服务端每帧广播所有玩家状态，每帧的带宽消耗直接乘以 3。

### 解析速度

JSON 是文本格式，解析器需要：
1. 按字节扫描，识别键名、冒号、值、引号、转义字符
2. 把字符串 `"103.5"` 解析成浮点数
3. 查找键名并赋值给对应字段

这个过程在现代 CPU 上也需要几微秒到几十微秒，取决于消息大小和解析器实现。在 60Hz 的服务端（每帧 16ms），如果有 1000 个连接，每帧序列化+反序列化的总开销可以轻松达到几毫秒量级。

**JSON 的定位**：调试工具、配置文件、低频 API、热更配置读取。不适合高频游戏消息。

## Protobuf：游戏消息的工业标准

### 工作原理

Protocol Buffers（Protobuf）是 Google 开发的二进制序列化格式。它的设计分两层：

**Schema 定义层**：用 `.proto` 文件描述消息结构

```protobuf
message PlayerState {
    int32  seq     = 1;
    float  x       = 2;
    float  y       = 3;
    float  z       = 4;
    float  facing  = 5;
    int32  health  = 6;
}
```

**编码层**：字段 Tag + 编码值

每个字段在二进制流中的编码格式是 `[Tag][Value]`。Tag 包含字段编号和线型（Wire Type，标识值的编码方式）。值根据类型用不同编码：
- `int32`、`int64`：VarInt 编码，小数值用更少字节（1 用 1 字节，1000 用 2 字节，不是固定 4 字节）
- `float`、`double`：固定 4/8 字节
- `string`、`bytes`：长度前缀 + 数据

VarInt 是 Protobuf 包体小的关键——序列号、血量、小坐标值都能压缩表示，不像 C++ 的 `int` 固定 4 字节。

### Schema 演进规则

Protobuf 的版本兼容是有明确规则的，这是它最重要的工程特性之一：

```protobuf
// v1 版本
message PlayerState {
    int32 seq    = 1;
    float x      = 2;
    float y      = 3;
}

// v2 版本：新增字段
message PlayerState {
    int32 seq    = 1;
    float x      = 2;
    float y      = 3;
    float z      = 4;  // 新增，字段编号 4
    int32 health = 5;  // 新增，字段编号 5
}
```

规则是：
- **已有字段的编号不能改变**（编号是二进制流中的实际 key）
- **新字段用新编号**
- **旧 Parser 遇到未知字段编号时跳过**（不报错，兼容新格式）
- **新 Parser 遇到 v1 没有 z 字段时**，使用默认值（0）

这意味着服务端先升级到 v2，老客户端发来的 v1 消息仍然可以正确解析——只是 z 字段为 0。客户端陆续升级后，v2 的完整数据自然流入。这个特性对手游来说至关重要，因为你不能强制所有玩家同时更新客户端。

### Protobuf 的适用范围

- 大多数游戏的消息格式（95% 的场景够用）
- 跨语言（C++、Java、C#、Go、Python 都有官方代码生成）
- 长期维护的项目（Schema 演进有规范）

### Protobuf 的局限

Protobuf 序列化需要**内存分配和拷贝**。从字节流还原消息，需要分配一个新的对象，把字段逐一填入。在极高频率（每秒数千条消息）的服务端，这些内存分配产生的 GC 压力是真实可测量的。

## FlatBuffers：零拷贝读取的设计哲学

### 根本差异

FlatBuffers 和 Protobuf 的差异不是"哪个更快"的简单比较，而是**两种完全不同的序列化模型**。

Protobuf 的工作模式：

```
字节流 → 解析（内存分配 + 字段填充）→ 内存对象 → 读取字段
```

FlatBuffers 的工作模式：

```
字节流 → 直接读取（无解析步骤）
```

FlatBuffers 把数据组织成一种特殊的内存布局，使得字节缓冲区本身就可以直接按偏移量访问字段，不需要任何解析步骤。

```cpp
// FlatBuffers 读取：直接从缓冲区指针访问
auto player = GetPlayerState(buffer_ptr);
float x = player->x();          // 直接读偏移量，无解析
float y = player->y();          // 同上
int health = player->health();  // 同上
```

这意味着**零内存分配、零解析延迟**。在需要高频读取消息但不需要修改其内容的场景下（比如服务端转发位置更新给其他玩家），FlatBuffers 的性能优势非常显著。

### FlatBuffers 的代价

**写入比 Protobuf 复杂**：构建 FlatBuffers 消息需要使用 Builder API，按特定顺序逆向构建（嵌套对象必须先构建子节点），代码比 Protobuf 冗长。

**Schema 演进更受限**：FlatBuffers 对字段顺序和偏移量有严格要求，Schema 修改需要非常谨慎，文档中有明确的演进限制列表。

**调试困难**：二进制缓冲区没有自描述性，调试时需要借助 flatc 工具或自行实现 dump 工具。

**适合场景**：需要大量读取、很少修改的消息（比如配置数据加载、批量状态广播）；有 GC 压力问题的服务端（Java/C# 游戏服务端）；数据通道中间件（只转发不解析的场景）。

## 自定义二进制格式：适合谁

有一类场景，KCP + Protobuf 的组合依然满足不了需求：极致低延迟的格斗游戏或赛车游戏，每帧需要处理数千条消息，Protobuf 的解析开销累积起来也需要优化。这时候，一些团队会选择完全自定义的二进制格式。

自定义格式的典型设计：

```
[消息类型: 1字节][消息长度: 2字节][消息体: N字节]

消息体（位置更新）:
[序列号: 2字节][x: 4字节 float][y: 4字节 float][z: 4字节 float]
[facing: 2字节 fixed16]   <- 角度压缩成 uint16，不用 float
[health: 1字节 uint8]     <- 血量 0-255，1字节够
```

通过字段大小的精心设计，同等数据量可以再压缩 20-30%，且解析直接是指针偏移，无任何分配开销。

### 自定义格式的前提条件

- 团队有能力维护 Schema 版本管理工具
- 有足够的测试覆盖，字段偏移错误不会被轻易发现
- 客户端和服务端都在内部控制，不需要跨团队/跨公司对接
- 有长期维护的意愿（这个格式会存在整个产品生命周期）

**不推荐自定义格式的情况**：小团队、快速迭代阶段、需要外部合作（SDK 集成、联运平台对接）、服务端有多种语言。

---

## 版本兼容：一个常被忽视的关键工程约束

版本兼容不是锦上添花，它是游戏服务端的基础工程约束。

手游的版本分发有天然的不一致性：苹果审核周期 1-3 天、Android 分发有梯度放量、部分玩家不开自动更新。这意味着服务端升级后，线上同时存在多个客户端版本是常态，可能持续 1-4 周。

如果你的序列化格式不支持向前/向后兼容，你有两个选项：
1. 强制踢掉旧版本客户端（需要客户端强更新弹窗，玩家体验差）
2. 同时维护多个版本的服务端（运维成本极高）

Protobuf 的演进规则设计就是为了解决这个问题。正确使用 Protobuf，你可以做到：
- 服务端先升级，旧客户端继续工作（新字段被忽略或使用默认值）
- 客户端逐步升级，新字段自动开始生效
- 不需要停服、不需要强更新

**关键纪律**：
- 永远不要复用字段编号（即使删除了一个字段，它的编号也要保留在注释里）
- 新字段必须有合理的默认值
- 重命名字段名是安全的（二进制格式只用编号，不用名字）
- 改变字段类型是危险的（不同 Wire Type 的解析不兼容）

## 选型建议总结

| 格式 | 适用场景 | 不适用场景 | 版本兼容 |
|------|---------|---------|--------|
| JSON | 调试、配置文件、低频 API | 高频游戏消息 | 弱（手动管理） |
| Protobuf | 大多数游戏消息 | 超高频零拷贝需求 | 强（有规范演进规则） |
| FlatBuffers | 大量读取、GC 敏感 | 高频写入、跨语言混合团队 | 中（有限制） |
| 自定义二进制 | 极致性能、单一语言栈 | 快速迭代、外部对接 | 需自行实现 |

**一句话建议**：除非有明确的性能测量数据证明 Protobuf 是瓶颈，否则用 Protobuf。它的版本兼容机制和多语言支持在整个项目周期内会节省大量工程时间，远超那点序列化开销。

---

## 最短结论

JSON 的文本解析开销和包体膨胀在实时游戏场景是不可接受的，它是调试工具，不是协议格式。

Protobuf 是大多数游戏的正确选择，不仅因为性能，更因为它的 Schema 演进规则解决了一个长期困扰游戏团队的版本兼容问题。

FlatBuffers 在零拷贝读取方面有不可替代的优势，适合作为 Protobuf 的专项补充，而不是全面替代。

自定义二进制是终极性能优化手段，但工程成本高，只有在有明确测量数据支撑时才值得投入。
