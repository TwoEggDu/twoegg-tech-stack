如果前面几篇已经把“怎么设计渲染系统”讲清楚了，那还差最后一步：

`怎么证明这套系统长期没坏。`

这一步很容易被忽略。很多团队会在冷机、首帧、单次跑图里拿到一组漂亮结果，然后默认系统已经稳定了。

但真正把项目推到线上以后，问题往往出在另一条线上：

- 热机以后档位是否还成立
- 长时间运行后是否仍然保持预算
- 新内容上线后，旧的黄金场景有没有被重新打穿
- 某台边界机是不是被误分到更高档
- 用户手动改过设置后，系统有没有继续按预期工作

所以这篇不再讲“怎么配”，而是讲一条更长的链：

`设计 -> 验证 -> 上线 -> 回写 -> 回归 -> 再验证`

## 质量闭环不是最后补一组测试

渲染系统的长期质量，不是简单补几个性能测试就能兜住的。

如果没有闭环，最常见的情况是：

- 版本 A 验证过了，版本 B 换了一组材质就开始漂
- 白天测过了，晚上长时间运行以后开始降频
- 某个机型本地识别没错，但线上流量里它的真实表现已经变了
- 某个特性在高档没有问题，但低档 fallback 在边界场景里不可读

所以质量闭环要看的，不只是“有没有测试”，而是：

`测试是否覆盖了系统真正会变的那部分。`

这也是为什么 [机型分档怎么验证：设备矩阵、Baseline、截图回归与错配排查]({{< relref "code-quality/device-tier-validation-matrix-baseline-and-visual-regression.md" >}}) 要和这篇一起看。
前者讲的是验证方法，这篇讲的是如何把验证方法变成持续运转的工程机制。

## 设备矩阵要围绕边界，不要只围绕销量

很多设备矩阵看起来很完整，但其实只是“常规兼容性测试”。

对分档系统来说，这还不够。设备矩阵至少要覆盖四类样本：

| 样本类型 | 目的 |
|---|---|
| 每档代表机 | 验证这档的预期行为是否成立 |
| 档位边界机 | 验证最容易误判的临界点 |
| 异常机 / 特殊驱动机 | 验证最容易失真和回归的组合 |
| 长尾机 | 验证线上真实分布里真正会撞到的问题 |

这里最重要的不是“机器越多越好”，而是“每一档至少要有能代表边界的样本”。

建议矩阵里至少记录这些字段：

- 机型名
- `SoC / GPU`
- 图形 API
- `OS` 版本和补丁版本
- 分辨率与刷新率
- 内存和可用存储
- 当前档位来源

这样，测试结果才知道是在回答哪类问题：

`是档位错了，还是这类设备本来就不稳定。`

## 黄金场景要围绕体验边界，不要只围绕平均帧

如果只测空场景和普通巡游，系统很容易看起来“挺健康”。

但真正容易暴露问题的，通常是这些黄金场景：

| 场景 | 为什么必须测 |
|---|---|
| 主城 / 大厅 / 活动入口 | 最容易暴露 UI、导航和常驻对象负载 |
| 战斗峰值 | 最容易暴露 fragment、overdraw 和特效压力 |
| 高价值 UI 页 | 最容易暴露切页、首开和层级反馈 |
| 首次进入 / 首次触发 | 最容易暴露冷启动和首次资源代价 |
| 热机后长时运行 | 最容易暴露频率收缩和动态降档 |

这几类场景的共同点是：它们不是“平均状态”，而是“体验边界”。

分档系统要证明的，也不是平均跑得多快，而是：

`在边界上，这一档仍然保留了应有的体验语义。`

## Baseline 不是一组数字，而是一套快照

很多团队把 baseline 只理解成一张性能表，这通常不够。

对分档系统来说，baseline 至少要分四层：

### 1. 性能 baseline

记录：

- 平均帧时间
- `p95 / p99`
- 关键场景帧稳定性
- 长时运行后的退化幅度

### 2. 视觉 baseline

记录：

- 关键场景截图
- 关键 UI 页截图
- 高 / 中 / 低档差异
- fallback 是否还保留识别度

### 3. 热机 baseline

记录：

- 冷机到热机的时间曲线
- 降频前后档位变化
- 长时运行 10 分钟、20 分钟、30 分钟后的表现

### 4. 稳定性 baseline

记录：

- 是否误分档
- 是否异常回退
- 是否有崩溃、黑屏、卡死、资源缺失

如果这四层没有一起存下来，后面回归时你只能看到“掉了多少帧”，看不到“掉的是哪一层”。

## 热机和长时运行，决定了很多冷机结论都不成立

`mobile-hardware-02b` 已经讲过一个很关键的事实：冷机时判断对了，不代表热机后仍然成立。  
见 [为什么高端机的游戏体验更持久：散热设计、持续性能与内存稳定性]({{< relref "engine-notes/mobile-hardware-02b-sustained-performance.md" >}})。

对分档系统来说，这意味着：

- 冷机判定只能作为起点
- 长时运行会改变 CPU / GPU 预算
- 某些资源首开代价只会在热机后更危险
- 动态降档必须看热机状态，而不是只看一张静态表

所以热机验证最好直接覆盖这些动作：

- 长时间站城 / 主菜单停留
- 战斗循环反复触发
- 进出前后台
- 切换场景和加载大资源
- 充电 / 非充电、低电量 / 正常电量、性能模式 / 省电模式

只要这些条件没有进矩阵，热机后的质量就很难被真正验证。

## 回归链要贯穿开发、CI 和线上

这套系统最后能不能长期稳定，取决于回归链是不是连起来了。

我建议至少分成三层：

### 1. 开发层

本地开发阶段，最少要能快速看到：

- 当前档位是否变化
- 当前黄金场景是否已经跑偏
- 新材质、新 Shader、新 `Renderer Feature` 是否打穿预算

### 2. CI / Nightly 层

CI 和 nightly 至少要负责：

- baseline 对比
- 视觉回归
- 构建门禁
- 关键场景自动采样

这部分和 [baseline budgets in CI]({{< relref "code-quality/baseline-budgets-in-ci.md" >}}) 的思路是一致的：  
不是让 CI 代替工程判断，而是让 CI 提前拦住明显越界的变化。

### 3. 线上层

线上层要负责：

- 远程回写
- 灰度发布
- 错配纠正
- 强制回退
- 异常时自动降档

这部分和 [机型分档怎么接线上：遥测回写、Remote Config、灰度与回滚]({{< relref "engine-notes/urp-platform-03-online-governance.md" >}}) 是同一个闭环，只是这里更强调验证结果如何持续反哺系统。

## 真正好用的闭环，应该满足哪些条件

如果一套分档验证体系是健康的，它至少应该满足这几个条件：

| 条件 | 含义 |
|---|---|
| 可重复 | 同一机型、同一场景、同一配置能得到可比结果 |
| 可解释 | 掉帧、错档、视觉漂移都能定位到一层原因 |
| 可阻断 | 明显越界的版本能在发布前被拦住 |
| 可回滚 | 线上发现问题后能迅速回退 |
| 可扩展 | 新内容、新设备、新特性可以继续接入 |

只要这五条里有一条断掉，分档系统就很容易重新退化成“靠经验盯着看”。

## 结语

渲染系统分档真正难的地方，不是“第一次把它配对”，而是“让它一年以后还对”。

所以这套系统必须同时回答两件事：

- 设计层：哪些特性该给高档，哪些必须给低档保底
- 验证层：这套设计在真实设备、真实场景、真实时间里有没有继续成立

如果前面的几篇解决了“怎么设计一条渲染系统”，那这篇解决的就是：

`怎么让这条系统不是一次性工程，而是可长期维护的质量体系。`
