> 最短的一句话是：`技能系统的回归不是“多写几个单元测试”，而是围绕同一条技能执行链，建立一套能证明“这次改动没有悄悄改坏旧真相”的验证体系。`

写到 `11｜多人同步` 和 `12｜编辑器与配置工具` 之后，团队下一句几乎一定会问：
`现在这套系统看得见了，但怎么证明下一轮修改不会把它悄悄打坏？`

这个问题在技能系统里尤其尖锐，因为它几乎不在“编不过”时暴露，而更常在这些场景里暴露：

- 某个 Buff 叠层规则被顺手改了，三天后才有人发现老技能伤害不对
- 动画关键帧往后挪了 0.1 秒，表现看着顺，但真实生效点已经偏了
- 命中判定逻辑修了一处投射物，范围技和连段却跟着一起变了
- 工具侧能预览、能配置，但进战斗后的结果和预览不再一致

所以这一篇真正要讲的，不是测试框架语法，而是：
`技能系统到底该在哪些层留下“可被证明”的回归面。`

---

## 先承认一个现实：技能系统最容易坏的，往往不是最终结果，而是中间过程

很多团队一说回归，第一反应就是比最终数值：

- 伤害是不是 120
- 冷却是不是 5 秒
- 有没有挂上 Burn
- 最后有没有播完特效

这些检查不是没用，但不够。

因为技能系统是一条长链：

```text
输入意图
-> CastRequest
-> 校验与约束
-> 生命周期推进
-> 目标选择 / 命中确认
-> ExecuteEffect
-> ApplyBuff / Refresh / Remove
-> 表现与日志对齐
-> Completed
```

只看最终结果，会把中间所有失真都压扁成一个“最后对不对”。
问题是，很多坏边界根本不会立刻反映到最终 HP 上，它们更早地体现在：

- 什么时候进入 Windup / Active / Recovery
- 命中是在本地预测成立，还是权威确认成立
- Buff 是没有挂上，还是挂上后立刻被互斥规则移除了
- 冷却是 Accepted 时开始，还是 HitConfirmed 时开始
- 表现没播，是资源没配，还是逻辑事件根本没发

所以技能系统的回归，不能只验证“结果值”，还要验证“事件链”。

---

## 回归面不要按测试框架分，而要按执行链分

我更建议沿着整条执行链去布置回归，而不是先问：
`这里写单元测试，那里写集成测试。`

更稳的分法是这六层。

### 1. 配置层：先证明这条链能被合法地配出来

这一层最适合承接上一篇 `12｜编辑器与配置工具`。

它回答的不是“运行后对不对”，而是：
`有没有明显不可能正确运行的配置被放进系统。`

这里至少要拦住三类问题：

- 引用完整性：缺资源、缺模板、死引用、循环依赖
- 配置合法性：被动技能却要求手动选目标、关键帧超出动作时长、Buff 互斥组自相矛盾
- 链路一致性：宣称会发射投射物却没有 Spell，宣称会附加 Buff 却没有 ApplyBuffEffect，时间线有生效点却没有逻辑事件

这一层越早做，越能减少“进战斗才发现配错”的低级返工。
它本质上是最便宜的一层回归。

### 2. 公式层：把容易悄悄漂移的数值真相钉住

技能系统里最常见的慢性事故，不是技能完全失效，而是：
`它还能跑，但结果悄悄变了。`

比如：

- 伤害加成顺序改了
- 暴击和易伤的乘区合并方式变了
- Buff 快照和实时重算的边界被改混了
- 同一目标身上的多层 Modifier 叠加优先级漂移了

这类问题最适合做成确定性的公式夹具。

例如：

```text
Case: Fireball_WithBurnAndAtkBuff
Input:
- caster.atk = 100
- skill.coeff = 1.2
- atkBuff = +20%
- target.fireVuln = +10%
Expected:
- directDamage = 132
- burnTick = 18
- burnDuration = 3.0s
```

这里的关键不是测试框架，而是两点：

- 输入必须可重现，不能依赖实时场景状态
- 断言必须落在业务真相上，别拿一整个大对象快照糊过去

如果断言对象是一大坨运行时状态，测试很快就会退化成“改一次就全量更新 snapshot”。
公式回归要钉住的是规则，不是对象内存形状。

### 3. 生命周期层：验证时序，不只验证结局

很多技能系统真正难维护的地方，都藏在 `03｜生命周期` 讲的时序层里。

要回归的至少包括：

- 吟唱能否被打断
- 连段窗口什么时候打开和关闭
- Channel 技能在资源不足、目标丢失、控制打断时怎样停
- Toggle 技能在失去条件时是立即关闭，还是延迟收尾
- 冷却在请求被接受、技能开始、生效命中、正常完成中的哪一刻起算

这一层不能只看“技能最后有没有完成”，而要看时间线是不是对。

例如：

```text
CastRequest
-> ValidationPassed
-> EnterWindup
-> EnterActive
-> HitConfirmed
-> StartCooldown
-> EnterRecovery
-> Completed
```

如果团队已经有事件流或时间线记录器，这些就应该直接成为回归基线。
没有的话，这正是最值得补的基础设施之一。

### 4. 命中层：给单体、范围、投射物、锁定分别留样例

很多团队会把命中系统回归偷懒成一句：
`放个技能，看怪有没有掉血。`

这远远不够。

命中层至少要拆出几类固定样例：

- 单体锁定：目标切换、死亡、失去可见性时怎样处理
- 范围命中：边缘单位是否命中、地面技能落点偏移是否一致
- 投射物：发射、飞行、碰撞、穿透、反弹、多段命中的边界
- 近战判定：前方扇形、矩形盒、受击窗口与表现关键帧是否一致

这类样例的价值，在于把“命中逻辑本身”从整场战斗里拆出来。
否则每次回归都依赖完整对局，最后没人愿意跑。

### 5. Buff 层：叠层、刷新、覆盖、驱散要有矩阵，不要只测一条 happy path

Buff 系统是最容易“局部改动，系统级外溢”的地方。

最常见的坑包括：

- 同名 Buff 是叠层还是刷新
- 高等级覆盖低等级，还是拒绝覆盖
- 驱散只移除可驱散层，还是整组一起移除
- 快照值是否在刷新时重采样
- 来源消失时 Buff 是否同步失效

这一层最适合用“规则矩阵”而不是零散案例。

例如：

```text
Burn vs Burn:
- sameSource + sameLevel -> refresh
- sameSource + higherLevel -> replace
- differentSource -> stack(2)

Shield vs Shield:
- lowerValue -> ignore
- higherValue -> replace
- dispel(type=magic) -> remove
```

只要这类矩阵不在，Buff 回归最后一定会退化成：
“这个英雄的这个技能现在看起来还正常。”
这不是体系，只是碰运气。

### 6. 日志与时间线层：把“最终结果”还原成“过程证据”

到了技能系统后期，真正值钱的回归资产，往往不是更多 UI 自动化，而是稳定的事件日志和技能时间线。

因为它能同时服务三件事：

- 测试：能断言这次施法到底经历了什么
- 调试：能快速定位卡在哪个阶段
- 协作：策划、程序、QA 可以对着同一条事件链说话

一个足够实用的技能事件流，至少应该长这样：

```text
[12:31:08.121] CastRequest(skill=Fireball, caster=PlayerA)
[12:31:08.122] ValidationPassed
[12:31:08.123] EnterWindup
[12:31:08.356] ProjectileSpawned(id=3007)
[12:31:08.601] HitConfirmed(target=Monster12)
[12:31:08.602] ExecuteEffect(type=Damage, value=120)
[12:31:08.603] ApplyBuff(name=Burn, duration=3.0)
[12:31:08.604] CooldownStarted(5.0s)
[12:31:08.980] Completed
```

一旦这条链能被稳定采集，你就不再只能问“这次为什么没掉血”，而能问：

- 请求没发出？
- 校验没过？
- 生命周期没推进？
- 命中没成立？
- Effect 没执行？
- Buff 被互斥规则吃掉了？
- 冷却起点和预期不一致？

这就是回归真正能落地的前提。

## 不要把所有回归都压成“打一场完整战斗”

技能系统测试最常见的组织性错误，是所有东西都依赖整场战斗场景。

表面上它最真实，实际上它最脆弱：

- 启动慢
- 定位差
- 环境噪音大
- 改一个无关系统也可能把它带红

我更建议把回归切成三档。

### 第一档：内容侧快检

这类回归不进战斗，直接在保存、导出或 CI 上跑。

目标是快速干掉：

- 缺引用
- 越界值
- 互斥冲突
- 生命周期断裂
- 资源缺失
- 配置与链路不一致

这一档最适合高频跑，甚至每次提交都跑。

### 第二档：局部执行链回归

这类回归只搭最小运行时，不进完整战斗。

它重点验证：

- 公式
- 生命周期
- 命中
- Buff 规则
- 事件流

这一档是技能系统最核心的“稳定盘”。
大部分逻辑回归都应该落在这。

### 第三档：整场景冒烟

这类回归才去验证：

- 真实输入接线
- 资源加载
- 动画与表现挂接
- UI 展示
- 网络消息接线

它当然要有，但不应该承担全部责任。
如果团队把所有正确性都寄托在整场景冒烟上，回归成本会越来越高，覆盖率却越来越低。

## 技能系统最容易写坏的，不是测试代码，而是断言边界

很多回归最后失效，不是因为没人写，而是因为断言写错了地方。

### 1. 只断言最终 HP，不断言中间事件

这样会导致你知道“它错了”，却不知道“它在哪里开始错”。

### 2. 把表现结果当逻辑真相

比如特效播了，就默认命中了；受击动画播了，就默认结算了。
这会把 `09｜动画与表现解耦` 里好不容易拆开的边界又重新粘回去。

### 3. Snapshot 一整个运行时对象

这类测试最开始很快，后面最难维护。
因为对象里混着太多和业务真相无关的字段，任何重构都会让它大面积漂红。

更稳的做法是 snapshot 归一化后的业务事件流、关键数值表或状态转移序列。

### 4. 校验器和回归完全脱节

如果配置校验器能发现的问题，还要等集成测试报红才知道，那说明层次没立住。
能前置的错误，就前置；把战斗回归留给真正需要运行时真相的问题。

### 5. 多人同步只对最终伤害，不对权威事件链

到了联网场景，如果只比较最后血量是否一致，而不比较：

- Request 是否被接受
- Accepted 后何时进入 Active
- 命中确认何时广播
- 冷却从哪一刻开始

那很多同步错误仍然会长期潜伏。
这也是为什么 `11｜多人同步` 和这一篇其实是连着的。

## 如果让我从零补一套最小回归栈，我会按这个顺序来

资源有限时，我不会一开始就追求“覆盖一切”的自动化平台。
更实际的顺序是：

### 第一步：先把配置校验器和公式夹具立住

因为它们成本最低，收益最高。
能最快减少“配错”和“规则悄悄漂移”。

### 第二步：把技能事件流和时间线记录下来

不是为了日志好看，而是为了让后面的时序回归、命中回归、多人对账有共同证据。

### 第三步：给生命周期、命中、Buff 建一套边界场景库

不是追求海量 case，而是优先覆盖最容易出事故的边界：

- 打断
- 刷新 / 覆盖
- 边缘命中
- 多段命中
- 目标丢失
- 冷却起点
- 权威确认

### 第四步：最后再接整场景冒烟和跨系统回放

到这一步时，冒烟测试才真正有意义。
因为它验证的是“接线没断”，而不是被迫替所有底层逻辑兜底。

换句话说，`12` 让技能系统更容易被看见，`13` 则让它更容易被证明没坏。
这两篇本来就是一前一后的同一件事。

## 这一篇真正想留下来的结论

技能系统的测试与回归，最容易被误解成：

- 写几个伤害单测
- 跑一场自动战斗
- 看看特效和血量对不对

但如果只停在这一步，你得到的仍然只是“结果检查”，不是“系统级回归”。

更稳的回归体系，应该沿着同一条执行链把真相一层层钉住：

- 配置层拦住不合法数据
- 公式层钉住数值规则
- 生命周期层钉住时序
- 命中层钉住结算边界
- Buff 层钉住持续规则
- 日志与时间线层把整个过程变成可对账证据

所以这一篇最短的结论可以记成：

`技能系统的回归，不是给“最终结果”补几道题，而是让 CastRequest、状态推进、命中确认、Effect 执行、Buff 变化和 Completed 这些关键真相，都留下可被重复验证的证据链。`
