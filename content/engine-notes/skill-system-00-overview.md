> 如果这篇只记一句话，我建议记这个：`技能系统真正要解决的，不是“把一个技能放出来”，而是“把一次能力请求安全、稳定、可扩展地执行完”。`

很多团队一开始谈“技能系统”，脑子里浮现的往往是这些东西：

- 一个技能按钮
- 一个图标和冷却遮罩
- 一段伤害公式
- 一个播动画、放特效的函数

这些东西都和技能有关，但它们还不是“技能系统”。

因为只要项目稍微复杂一点，问题马上就会从“能不能放出来”升级成：

- 这个技能现在为什么不能放
- 这个技能为什么会被打断
- 这个技能的命中时机到底跟动画哪一帧对齐
- 这个技能给目标上的 Burn、Slow、Shield 到底谁先生效
- 这个技能为什么在本地看起来命中了，服务器上却没命中
- 这个技能为什么策划一改配置，别的技能也跟着出问题

也就是说，真正的难点从来不在“做一个技能”，而在：

`当技能数量、角色数量、效果种类、约束条件、表现联动和多人同步一起上来时，系统还能不能稳住。`

---

## 技能系统真正要处理的是什么

我更愿意把技能系统理解成一条执行链。

最短可以压成下面这条：

```text
输入意图
-> 技能请求
-> 校验能否释放
-> 选目标/定范围
-> 进入施法生命周期
-> 生成并执行效果
-> 改变世界状态
-> 触发表现
-> 收尾并进入冷却
```

这条链里，技能系统真正负责的是“中间这段执行框架”。

它需要回答六类问题。

### 1. 技能到底是什么

你必须先回答：

- 技能和 Buff 是一回事吗
- 技能和效果是一回事吗
- 被动技能和主动技能是不是同一套抽象
- 资源、冷却、标签、状态限制到底站在哪一层

如果这些边界不先拆开，后面所有实现都会慢慢塌成一个大 if-else。

### 2. 一次技能请求怎么走完

技能系统不是“调用一个 UseSkill()”就结束了。

一次请求往往要经过：

- 输入缓冲
- 前置校验
- 目标选择
- 消耗扣除
- 前摇/吟唱/引导
- 生效
- 后摇/恢复
- 冷却

这其实是一个小型状态机。

### 3. 效果如何真正改变世界

大多数技能最后都会落到几类状态变化上：

- 改血量
- 改属性
- 上 Buff
- 做位移
- 召唤实体
- 打断、驱散、控制

技能系统如果没有一层独立的 Effect System，很快就会变成“每个技能自己改世界”，最后完全收不住。

### 4. 约束条件怎么表达

技能为什么不能放，往往不是一个布尔值，而是一堆条件组合：

- 在不在冷却
- 资源够不够
- 有没有目标
- 目标是否可见
- 当前状态是否允许施法
- 身上有没有禁止标签
- 当前是否处于别的技能生命周期里

这些条件不是“额外判断”，而是技能系统本体的一部分。

### 5. 技能和表现怎么对齐

玩家看到的是：

- 动画什么时候抬手
- 特效什么时候出来
- 伤害数字什么时候跳
- 受击什么时候僵直

程序处理的是：

- 技能什么时候进入生效窗口
- 碰撞什么时候结算
- Buff 什么时候挂上
- 冷却什么时候开始

如果这两条时序线绑死，系统就会变脆；如果完全脱开，手感又会散。

### 6. 系统如何在项目里长期存活

一个真正的技能系统，不只服务“当前这 3 个技能”。

它得面对：

- 角色增多
- 技能种类增多
- 策划不断加规则
- 表现联动越来越多
- AI 要调它
- 网络同步要接它
- 测试和回归要覆盖它

所以技能系统的最后一个目标其实是：

`让变化被局部吸收，而不是每次变化都改核心骨架。`

## 技能系统不该和什么混为一谈

这件事特别重要，因为很多项目失败，不是因为不会写代码，而是因为边界从一开始就混了。

### 技能系统不等于 AI

AI 负责的是：

- 什么时候放
- 对谁放
- 当前该选哪个技能

技能系统负责的是：

- 收到请求后，怎样合法地执行完整条链路

AI 是调用方，不是执行框架本身。

### 技能系统不等于动画系统

动画系统负责的是动作表现、混合、状态切换和通知点。

技能系统负责的是技能生命周期和生效时机。

两者必须联动，但不应该互相吞掉。

### 技能系统不等于战斗系统

战斗系统一般是更大的外层概念，它还包括：

- 属性系统
- 仇恨系统
- 伤害公式
- 目标选择
- 命中判定
- AI
- 网络同步

技能系统是战斗系统里最重要的一块，但它不是整个战斗系统。

### 技能系统不等于网络同步

单机项目也需要技能系统。

网络只是给它增加了：

- 权威性
- 预测
- 回滚
- 重同步

不要一上来就把问题写成“技能系统 = 网络技能系统”，否则文章会从第一篇开始就变形。

---

## 先把技能系统拆成最小分层

如果只为了先把骨架立住，我建议最少拆成下面五层：

```text
输入层
  玩家或 AI 提交“我要用这个技能”

请求与生命周期层
  校验、排队、施法、打断、收尾、冷却

目标与命中层
  选目标、范围、碰撞、投射物、命中时机

效果层
  伤害、治疗、位移、Buff、驱散、召唤

状态与约束层
  属性、标签、资源、控制状态、共享冷却、互斥关系
```

这五层不是“完美架构”，但它能先帮你避免一个常见灾难：

`让每个技能自己从输入一路管到动画、伤害、Buff、冷却和 UI。`

一旦这样写，技能数量上来之后，系统就不再是系统，而是技能脚本堆。

## 用一个最小例子感受一下问题空间

假设有一个技能：`Fireball`。

如果只看表面，它像是这句代码：

```csharp
UseSkill("Fireball");
```

但真正发生的事可能是：

```text
玩家按下按键
-> 输入系统识别到 Fireball 槽位
-> 技能系统收到 Cast Request
-> 检查沉默、眩晕、公共冷却、法力值、距离
-> 锁定目标或确定发射方向
-> 进入 0.4 秒前摇
-> 在动画事件点生成投射物
-> 投射物命中后结算伤害
-> 额外挂一个 3 秒 Burn Buff
-> 触发命中特效和受击反馈
-> 技能进入 8 秒冷却
```

注意这条链里没有任何一步是“可有可无的细节”。

它们合在一起，才叫技能系统。

## 一个坏技能系统通常长什么样

很多项目早期的技能代码都很像这样：

```csharp
public void UseSkill(Character caster, Character target, SkillData data)
{
    if (caster.IsSilenced) return;
    if (caster.Mana < data.cost) return;

    caster.Mana -= data.cost;

    PlayAnimation(data.animationName);
    SpawnEffect(data.vfxName, target.transform.position);

    if (data.skillType == SkillType.Damage)
        target.HP -= data.damage;
    else if (data.skillType == SkillType.Heal)
        target.HP += data.heal;
    else if (data.skillType == SkillType.Buff)
        target.AddBuff(data.buffId);

    StartCooldown(data.id, data.cooldown);
}
```

这类代码在“前 3 个技能”阶段很好用，但很快就会出现这些问题：

- 技能类型一增多，`if/else` 爆炸
- 同一段代码同时在处理输入、约束、表现和效果
- 无法表达前摇、引导、投射物、打断、共享冷却
- Buff、属性、命中、动画通知只能硬塞进去
- 任何新规则都得改核心函数

这不是“代码风格不好”，而是问题空间本身已经超出了这种写法的承载能力。

## 一个成型技能系统应该具备什么气质

我不会用“功能全不全”来判断一个技能系统成不成型。

我更看下面这几件事。

### 技能定义和技能执行是分开的

配置描述“这个技能想做什么”，运行时实例负责“这次施法现在做到哪一步了”。

### 效果是独立层，而不是每个技能各写各的

伤害、治疗、位移、上 Buff 这些应该是可组合的效果单元，不是写死在某个技能脚本里。

### 生命周期是显式存在的

Instant、Cast、Channel、Charge、Toggle、Passive 至少能被同一套时序模型描述。

### 约束是可表达、可查询、可复用的

技能为什么不能放，应该是系统能解释的，不是埋在 17 个 if 里的。

### 表现联动是对齐的，但没有互相吞掉

技能能驱动动画和特效，但技能逻辑不应该被动画状态机完全接管。

### 能支撑策划和测试长期迭代

你能不能新增技能、加规则、做回归，而不重写一遍核心骨架，这件事比“今天能不能放出火球”更重要。

## 这整个系列后面会往哪走

这一篇只做一件事：先把问题空间立住。

后面的文章，会顺着这条骨架继续往里拆：

- 技能、效果、Buff、属性、标签的边界怎么切
- SkillDef 和运行时实例怎么分
- 生命周期怎样统一 Instant、Cast、Channel、Charge
- Effect System 怎样成为真正的执行引擎
- Buff / Modifier 怎样避免互相覆盖和结算错乱

如果没有这一篇，后面的所有实现文章都很容易自动滑回“写一个技能脚本”的局部视角。

但如果先把这一层看清楚，后面的所有设计都会更稳：

`技能不是一个按钮，也不是一段伤害代码。技能系统本质上是一套把能力请求变成世界状态变化的执行框架。`
