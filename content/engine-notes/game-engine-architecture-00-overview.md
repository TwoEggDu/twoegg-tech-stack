> 这篇总论只做一件事：先把现代游戏引擎放回同一张分层地图里。  
> 它不展开 GameObject / Actor 的内部差异，不拆 GC、DOTS、Mass 的机制，也不讲 Build / Cook / Package 的实现细节，而是先回答一个更上游的问题: `如果按架构理解，现代游戏引擎最稳定的第一张地图到底该怎么画。`

先说明这篇的证据边界。

当前这版首稿只使用官方文档证据。`docs/engine-source-roots.md` 里 Unity 和 Unreal 的源码根路径都还不是 `READY`，所以下文里凡是“事实”，都只落在官方资料明确写出来的范围；凡是“判断”，都会明确写成工程判断，而不是伪装成源码结论。

## 这篇要回答什么

很多人理解游戏引擎时，脑子里最先浮出来的往往是一串功能名词：

- 渲染
- 物理
- 音频
- UI

这套记法当然不算完全错，但它有一个很大的问题：它太像教材目录，不像工程地图。

因为只按这四五类功能记，你会直接漏掉几件决定引擎形状的大事：

- 内容是谁在生产，靠什么工作流生产
- 游戏世界里的对象到底按什么模型组织
- 脚本、反射、GC、任务系统这类运行时机制站在哪
- 资源为什么要重新经过导入、构建、打包和交付链
- 数据导向系统为什么看起来像“长在引擎旁边的一块新大陆”

所以这篇真正要回答的问题只有一句：

`如果不按零散功能背词，而按架构理解，现代游戏引擎最稳定的分层方式是什么。`

这个问题值得先回答，是因为后面很多常见争论都依赖它。

比如：

- 为什么 `GC` 不该和渲染、物理并排摆成同一行
- 为什么 `DOTS / Mass` 不像“再多一个模块”
- 为什么编辑器和构建链不能被当成外围工具
- 为什么 Unity 和 Unreal 的差异，不该只用 API 风格来总结

如果第一张地图就画错，后面每个专题都会互相串台。

## 这一层负责什么

先看事实。

从 Unity 官方资料里，你能同时看到 Package Manager、GameObject、IL2CPP、URP、Build Settings、Entities 这些正式入口；从 Unreal 官方资料里，你也能同时看到 Editor Interface、GameFramework、Reflection、Packaging、Mass 这些正式入口。

这至少说明一件事：

`现代游戏引擎从来不只是一个运行时播放器。`

如果先把问题压成一张最小地图，我会把它画成六层：

| 层 | 它真正回答的问题 | 在总论里先挂哪些代表词 |
| --- | --- | --- |
| 内容生产层 | 内容怎么被制作、组织、复用、协作 | Editor、Package、Plugin、Prefab、Blueprint |
| 世界模型层 | 世界里有哪些对象，它们怎样存在、更新、协作 | Scene、GameObject、Component、World、Actor |
| 运行时底座层 | 执行模型靠什么站住 | C#、IL2CPP、UObject、Reflection、GC、Task |
| 专业子系统层 | 渲染、物理、动画、音频、UI 为什么各自像半台小引擎 | URP、Renderer、Physics、Animation、UI |
| 资产与发布层 | 内容怎样从编辑状态变成可交付产品 | Import、Serialization、Build、Cook、Package |
| 数据导向扩展层 | 当规模和执行成本失控时，引擎怎样长出另一套组织方式 | DOTS、Entities、Mass |

这张图最重要的，不是名字好不好听，而是它把不同问题分到了不同层级。

### 内容生产层负责什么

这一层负责的不是“有一个编辑器窗口”这么简单，而是：

- 团队怎样组织内容生产
- 工具链怎样承载配置、预制、脚本挂接和资源编辑
- 包、插件、模板怎样影响引擎的扩展边界

我的判断是：

`如果一个系统同时决定内容怎么被制作、修改、复用和交给团队协作，它就不是外围工具，而是引擎本体的一层。`

### 世界模型层负责什么

这一层负责的是：

- 世界里到底有什么
- 对象怎样挂接行为
- 对象和对象之间靠什么协作
- “游戏框架”到底压在什么默认组织方式上

我的判断是：

`对象如何存在，本身就是引擎的大问题，不能和渲染、物理、音频这些专业系统混成同一层。`

### 运行时底座层负责什么

这一层负责的是：

- 脚本或原生逻辑怎么进入执行链
- 对象元信息、反射、GC、序列化怎样支撑引擎运转
- 任务系统或执行框架怎样托住整台引擎的调度

我的判断是：

`GC、脚本 VM、反射、任务系统更接近执行地基，而不是平铺在功能列表上的业务模块。`

### 专业子系统层负责什么

这一层负责的是：

- 渲染、物理、动画、音频、UI 这些高复杂度系统怎样各自成立
- 它们怎样带着自己的数据结构、调度规则和工具链运转

我的判断是：

`这些系统当然重要，但它们不是整台引擎的全部，而是长在整机上的重型子系统。`

### 资产与发布层负责什么

这一层负责的是：

- 资源怎样被导入、序列化、转换
- 项目怎样被重新组织成可运行、可部署、可发布的产物
- 构建链怎样把“编辑期内容”翻译成“交付期产品”

我的判断是：

`资源导入、Build、Cook、Package 不只是最后一步按钮，它们是引擎把内容变成产品的正式能力。`

### 数据导向扩展层负责什么

这一层负责的是：

- 当对象规模、访问模式和调度成本改变时，引擎怎样长出新的数据组织方式
- 数据导向系统怎样对原有世界做重构、扩展或特区化处理

我的判断是：

`DOTS 和 Mass 之所以值得单独放一层，不是因为它们功能多，而是因为它们在重新定义对象组织和执行成本。`

把六层收成一句话，就是：

`先生产内容，再组织世界，再提供运行时底座，再挂载专业子系统，再完成资产与发布；当默认对象世界撑不住规模和成本时，再长出数据导向扩展层。`

## 这一层不负责什么

总论最容易犯的错，就是一上来什么都想讲。

但这篇必须克制，因为它只负责搭地图，不负责替代后面的专题。

所以这篇明确不做下面几件事：

- 不教你怎么点 Unity 或 Unreal 的编辑器按钮
- 不罗列 Unity 有什么功能、Unreal 有什么功能
- 不把 GameObject / Actor、Scene / World 的内部差异一次讲透
- 不把 GC、Job System、Task Graph、Blueprint VM、Burst 的机制塞进同一篇
- 不把 DOTS、Mass、Build、Cook、Package、RHI、Target Platform 的细节一次写完
- 不做“谁更强、谁更先进、谁更适合谁”的产品优劣比较

这里顺手说清一个边界问题。

平台抽象当然很重要，但它在这张首层地图里先不单独展开。原因不是它不重要，而是它会同时穿过运行时底座、专业子系统和资产与发布链。第 `06` 篇应该单独回答它，而不是在总论里抢戏。

也正因为如此，这篇只做一件事：

`把后面各篇要处理的问题先摆回它们该在的层级里。`

## Unity 怎么做

先看事实。

从 Unity 官方文档能直接看到几条很清楚的线：

- Package Manager 和 packages 文档把 editor tools、runtime tools、asset collections、templates 都放进正式 package 体系
- GameObject 文档把 GameObject 说明为核心概念，并明确行为依赖 Components
- IL2CPP 文档把 scripting backend、IL 到 C++ 的转换、原生二进制生成放进正式运行时链路
- URP 文档把渲染能力写成正式 render pipeline 包
- Build Settings 文档把 target platform、Scenes in Build、构建设置放进正式手册
- Entities 文档把 ECS implementation 明确写成官方能力入口

如果只看这些公开材料，不碰源码，我认为至少可以先落下一个稳妥判断：

`Unity 不是“一个运行时 + 一堆附加功能”，而是一台把包化生产体系、组件化对象模型、脚本执行底座、专业子系统、构建交付链和数据导向扩展并列组织起来的引擎。`

进一步说，Unity 在这张地图上的气质也已经能看出一些轮廓：

- 在内容生产层，它明显带有包化和工具化的组织方式
- 在世界模型层，它默认强调 `Scene -> GameObject -> Component`
- 在运行时底座层，它把脚本后端和执行链作为正式基础设施
- 在专业子系统层，它倾向于把渲染等能力包装成可挂接、可升级的系统块
- 在资产与发布层，它把构建和目标平台放进正式交付流程
- 在数据导向扩展层，它又单独长出了一套不再按默认对象世界思考的路线

这里最重要的，不是马上给 Unity 贴一个大标签，而是看清它的层级分布确实存在。

## Unreal 怎么做

再看 Unreal 官方资料，逻辑同样很清楚。

能直接看到的事实包括：

- Editor Interface 和相关导航文档把编辑器工作流当成正式能力面
- GameFramework API 和术语文档把 `UObject`、`Actor`、`Pawn`、`Game State` 这些类型放进基础世界组织能力
- Reflection System 文档明确把反射与 editor/runtime 行为连在一起说明
- Packaging 和 building multi-platform projects 文档把 Build、Cook、Stage、Package 解释成正式 build operations
- MassEntity 和 MassGameplay 文档把 data-oriented framework 作为独立入口给出

基于这些事实，我能先做出的判断是：

`Unreal 也不是“Actor 加 Renderer”的二元结构，而是一台把编辑器工作流、世界与游戏框架、运行时对象系统、重型专业子系统、构建发布链和数据导向框架并列组织起来的引擎。`

继续往下压，你也会看到 Unreal 的默认组织方式更强调另一种重心：

- 在内容生产层，它把编辑器和工作流本身摆得更靠前
- 在世界模型层，它把对象组织和 gameplay framework 绑得更紧
- 在运行时底座层，`UObject`、反射、GC、元数据和序列化明显是底层共同依赖
- 在专业子系统层，渲染不是全部，但确实是重型系统之一
- 在资产与发布层，Build / Cook / Package 明显是体系内交付能力
- 在数据导向扩展层，Mass 也不是一个普通功能开关，而是一块单独成立的区域

同样，这里不急着下“谁更合理”的结论。

总论里更有价值的，是先承认：

`Unreal 的文档结构本身，也已经在告诉你这台引擎必须按层理解。`

## 这些差异为什么不是表面 API 差异

如果把前面两节再压缩一次，最关键的结论其实不是“Unity 有 Package、Unreal 有 Plugin”，也不是“Unity 有 GameObject、Unreal 有 Actor”。

更关键的是：

`两台引擎把复杂度放在了不同组织方式里。`

这句话为什么重要？

因为很多看似像同类词的东西，实际并不在同一层回答问题。

例如：

- `GameObject` 和 `Actor` 都是“对象”，但背后默认世界组织方式并不一样
- `GC` 和 `DOTS / Mass` 都会影响性能，但它们不在同一层
- `Package`、`Plugin`、`Build`、`Cook` 都会影响项目结构，但它们分别站在生产体系和交付体系的不同位置

所以把 Unity 和 Unreal 的差异理解成“一个 API 写法更像这样，另一个 UI 菜单更像那样”，就会非常表面。

真正稳的看法是：

- 先问这个问题属于哪一层
- 再问这一层里两边各自怎么组织复杂度
- 最后才看具体功能名词和接口形式

这样做的好处，是后面写 `02`、`03`、`07`、`05`、`06` 时不会互相串题。

你不会再把 `GC` 和渲染摆在一行，也不会把 `DOTS / Mass` 当成“多了一组 API”，更不会把 Build / Cook / Package 当成项目末尾的流水线脚本。

## 常见误解

### 误解一：游戏引擎就是渲染、物理、音频、UI 的打包

这会直接漏掉编辑器、对象模型、运行时底座和交付链。

专业子系统当然重要，但它们只是大机器上的几个重模块，不是整台机器的全部。

### 误解二：编辑器和构建链只是外围工具

如果官方文档把它们写进正式能力面，它们就不是外围。

编辑器决定内容怎样被制作，构建链决定内容怎样变成产品。两者都在塑造引擎本体。

### 误解三：GC、DOTS、Mass 都只是“优化项”

这也是把层级压平了。

`GC` 更接近运行时机制，`DOTS / Mass` 更接近世界模型和执行方式的结构性扩展。它们都会影响性能，但它们不是同类问题。

### 误解四：Unity 和 Unreal 的差异主要是 API 风格不同

API 风格只是最表面的结果。

真正的差异更早就出现了：内容怎么生产、对象怎么组织、底层机制怎么站住、交付链怎么组织，这些地方才是架构气质真正分叉的地方。

### 误解五：总论应该一次把所有专题讲透

不是。

总论的任务是先搭地图，让每个问题回到自己的层里。地图如果一开始就塌成百科，后面所有文章都会一起塌。

## 我的结论

先重申这篇能直接成立的事实。

- Unity 和 Unreal 的官方资料，都不只覆盖渲染或运行时，而是同时覆盖内容生产、对象组织、运行时机制、专业子系统、构建交付和数据导向框架
- 本地源码路径目前还没有 `READY` 标记，所以这篇不能声称自己完成了源码级验证

基于这些事实，我在这篇总论里愿意先给出的工程判断是：

`现代游戏引擎最稳的第一张地图，不是功能清单，而是内容生产层、世界模型层、运行时底座层、专业子系统层、资产与发布层、数据导向扩展层这六层结构图。`

这张图的价值，不是把每层一次写尽，而是先把后面最容易混掉的问题摆回正确位置：

- `02` 去回答世界模型层
- `03` 去回答运行时底座层
- `07` 去回答数据导向扩展层
- `05` 去回答资产与发布层
- `06` 去回答平台抽象跨过哪些层

如果后面源码路径补齐，这组文章当然可以继续把关键判断往源码级证据上压实。

但在当前这版首稿里，我更想先把一件事做对：

> 游戏引擎不是一堆功能模块的堆叠，而是一台按层组织复杂度的系统机器。  
> 只有先拿到这张地图，后面关于 Unity、Unreal、GC、DOTS、Mass、Build、Cook、Package 的讨论，才不会一直混线。
