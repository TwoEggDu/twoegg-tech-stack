很多团队并不是完全没有预算意识。

他们也会说这些话：

- 首包不能再大了
- 低端机内存要小心
- 主线程别超 16.67ms
- Bloom 太贵了，先关一点
- 技能特效不能做太满

这些话单独看都没错。  
问题在于，它们经常只是零散提醒，不是一套系统。

所以项目最后最常见的，不是“完全没人做优化”，而是：

- 每个人都在优化自己眼前的一块
- 但没人对总账负责
- 某一层省下来的成本，会在另一层被加倍花掉
- 预算一旦没有规则、告警和超线动作，就会慢慢退化成口头约定

如果把这件事压成一句话，我会这样说：

`很多性能和交付问题，本质上不是技巧不够，而是预算管理失败。`

## 预算不是一个数字，而是一套边界系统

很多人一提预算，第一反应就是“给我一个数”。

比如：

- 包体 1.5GB
- 内存 1.2GB
- CPU 8ms
- GPU 8ms

这些数字当然重要。  
但如果预算只剩这几个数字，它其实还不够用。

一个真正可执行的预算，至少要同时包含四样东西：

1. `预算对象`
这笔钱到底在管什么。

2. `预算线`
硬线、告警线、观察线分别在哪里。

3. `分账方式`
总预算是怎样分给场景、角色、特效、UI、加载链和不同档位的。

4. `超线动作`
一旦超了，是返工、降档、拆包、阻断构建，还是走例外审批。

如果缺了后面三样，单个数字就只是一个愿望。

## 为什么游戏特别需要预算语言

游戏和很多别的应用有一个根本区别：

`它不是“能做完就行”，而是“必须在有限预算里按时做完”。`

这个约束至少同时来自四个方向。

### 1. 帧时间是硬约束

60fps 只有 `16.67ms`，30fps 也只有 `33.33ms`。

这意味着：

- 逻辑推进不能无限长
- 主线程不能随意结算大工作
- GPU 不能把一切都画到最好
- 同步等待不能想等多久就等多久

### 2. 内存不是仓库，是工作台

运行时真正危险的，很多时候不是“总量立刻爆了”，而是：

- 常驻太高
- 峰值太尖
- 工作集波动太剧烈
- CPU 副本、GPU 副本、临时缓冲、缓存叠在一起

这也是为什么移动端常见的死法不是代码里抛出 `OutOfMemoryException`，而是被 `LMK / jetsam` 一把干掉。

### 3. 移动端预算会动态收缩

PC 上很多时候你面对的是“这台机器能不能跑”。  
移动端经常面对的是“这台机器在玩了 20 分钟以后，还剩多少预算”。

热、功耗、后台环境和设备碎片化会让预算本身变化。

也就是说，预算不是一个静态天花板，而是一个会收缩的天花板。

### 4. 交付成本也是预算

很多团队只把 CPU / GPU / 内存看作预算，却把下载、安装、更新和包体当成另一类问题。

其实不是。

如果一个资源根本不该进首包，你把它压小一点，不叫真正解决问题。  
如果一个高档内容被所有用户一起下载，再怎么说“运行时没问题”，交付账也已经花错了。

## 预算真正要回答的五个问题

我更愿意把预算看成一套固定问题，而不是一份静态表格。

每次立预算，都应该先回答这五个问题：

### 1. 成本对象是什么

是首包？热更包？  
是稳态内存？峰值内存？  
是主线程时间？GPU 时间？  
是场景级成本？角色级成本？技能级成本？

对象不清，预算一定会乱。

### 2. 为什么这条线立在这里

这条线不能只来自“经验感觉”。

它至少应该有一种明确依据：

- 帧时间上限
- 目标设备档位
- 平台强杀和热收缩现实
- 下载与安装体验
- 团队可维护能力

### 3. 这笔账分给谁

如果总预算没有继续拆给：

- 场景
- 角色
- 特效
- UI
- 包体分层
- 档位规格

那最后它永远只存在于主程脑子里。

### 4. 超线先显形在哪里

不同预算超线，显形方式不同：

- 包体超线先显成下载和更新负担
- 内存超线先显成切场景抖、长玩后差、回前台死
- CPU 超线先显成主线程尖峰、输入迟滞、逻辑拖尾
- GPU 超线先显成稳定掉帧、发热、降档和功耗问题

如果你不知道它会先在哪里显形，团队就会一直在错误位置找原因。

### 5. 超了以后怎么办

没有超线动作的预算，是最容易失真的预算。

至少要明确：

- 谁能触发阻断
- 什么情况下允许例外
- 例外是否需要回写计划
- 哪些预算超线必须返工，哪些可以先降档兜底

## 为什么很多优化最后都成了假优化

如果项目没有预算系统，就很容易出现“看起来做了很多事，其实只是转移了账”的假优化。

最典型的几种：

### 1. 用包体换运行时稳定，却没有交付分层

预烘焙、预热、预下载当然都可能有价值。  
但如果你把所有高档内容、低频内容、活动内容都塞进首包，这其实是在把运行时账转嫁成下载账。

### 2. 用对象池省 GC，却抬高常驻内存

对象池不是白送性能，它是典型的“用内存换尖峰”的方案。  
如果池子没有上限、分场景和回收策略，它就会把瞬时对象变成长期常驻。

### 3. 用更低分辨率换 GPU，却伤了视觉身份

Render Scale、Bloom 迭代、阴影距离当然能降。  
但如果你把 Color Grading、角色核心贴图、技能关键识别信号一起砍掉，最后省下来的不只是预算，也把产品辨识度一起砍了。

### 4. 用热路径优化掩盖错误时机

有些卡顿不是因为工作本身太重，而是因为工作发生在错误时机。  
比如首次技能、首次开 UI、切场景最后一拍、后台切回前台。

这类问题如果只盯“让代码更快”，往往会很贵。

## 预算应该长成什么样

如果你问我一套真正能用的预算系统最短应该长什么样，我会给出这六层：

1. `总预算`
包体、内存、CPU、GPU 的上限和目标档位。

2. `分账预算`
帧链路、场景、角色、特效、UI 各自拿多少。

3. `规则表`
面数、骨骼数、贴图尺寸、压缩格式、粒子上限、屏占比、同屏数量。

4. `证据链`
Profiler、Build Report、Memory Profiler、GPU Counter、包体报告、线上看板。

5. `超线动作`
阻断、返工、降档、拆层、补规则。

6. `回归门禁`
Baseline、CI、设备矩阵、截图回归、版本健康。

只要这六层里缺两层以上，项目最后就很容易退回“靠经验维持”。

## 这组文章接下来会怎么写

这个系列不会按“工具大全”来展开，而会按预算系统本身的层次展开：

- 先讲总论和体检
- 再讲包体、内存、CPU、GPU 这四条总预算主线
- 再讲帧链路、场景、角色、特效这些分账对象
- 最后把预算接回规格、分档、Baseline 和 CI

因为预算真正值钱的地方，从来不是单篇技巧，而是：

`它能不能让同一个项目里的不同角色，开始用同一种语言谈边界。`

## 结尾

所以如果只用一句话收束这篇，我会这样说：

`预算管理不是在超线以后补救，而是在问题还没长成事故之前，先把成本对象、边界、分账和超线动作都说清楚；只有这样，优化才不会永远停留在救火。`
