很多团队遇到的问题，其实并不是“完全没有优化动作”。

更常见的是下面这些状态：

- 包体每一版都在涨，但每次都只在发版前临时瘦身
- 明明编辑器里看起来还能跑，低端机长玩后还是开始抖、热、死
- 帧率一不稳，团队就开始在 CPU、GPU、加载和网络之间互相甩锅
- 新内容一上量，场景、角色、特效和 UI 就开始一起越线
- 文档里有很多“建议值”，但没人知道超线以后谁拍板、谁返工、谁兜底

所以这组文章想解决的，根本不是“再补一轮优化技巧”。

它更想先回答一件更前面的事：

`为什么很多项目看起来一直在优化，最后还是会反复掉进同一类包体、内存、性能和交付问题里？`

我的判断是：

`很多问题的本质，不是团队不会优化，而是项目从一开始就没有建立预算管理。`

## 预算管理到底在管什么

如果只用一句话定义，我会这样说：

`预算管理不是超线之后再去救火，而是先把交付成本和运行时成本拆清，再让内容、代码、资源和流程共同待在正确边界里。`

这意味着预算管理管的，从来不只是一个“指标数字”。

它至少同时在管五件事：

1. `成本对象`
   到底在花什么钱。是首包、热更、稳态内存、峰值内存、CPU 帧时间、GPU 帧时间，还是某一类场景 / 角色 / 特效的内容成本。
2. `预算边界`
   不是一句“别太大”“别太慢”，而是明确的观察线、告警线、硬线，知道什么算正常波动，什么已经不该再放行。
3. `显形位置`
   包体超线不会先显形成 FPS 掉，内存超线也不会总先显形成分配失败。预算管理要知道一条线先坏掉时，现场会最先出现什么症状。
4. `责任分发`
   谁对哪一层负责。哪些是客户端责任，哪些是 TA / 美术 / 资源分发 / 构建链责任，不能把一切都堆回主程。
5. `超线动作`
   超了以后是返工、降档、拆分、阻断，还是走例外审批。没有动作的预算，只是意见，不是边界。

所以预算管理和“抽象 KPI”最大的区别就在这里：

`KPI 只告诉你结果好不好看，预算管理要告诉你账怎么拆、线怎么立、问题先坏在哪、以及超线以后谁该动。`

## 为什么很多问题本质上是预算管理失败

因为游戏项目里的成本，从来不是单线的。

你以为自己在做一个局部优化动作，很多时候其实只是把成本从一个桶挪到了另一个桶。

常见的转嫁关系大概是这样：

- 为了压 `包体`，把更多内容改成运行时下载，结果把压力转成了 `首帧 / 加载 / 缓存 / 常驻资源`
- 为了压 `GC`，把更多对象池和缓存常驻，结果把压力转成了 `稳态内存`
- 为了救 `CPU`，把更多事情并行、异步、分帧，结果把问题转成了 `危险时机 / 同步等待 / 尖峰抖动`
- 为了救 `GPU`，降低分辨率和画质，结果把问题转成了 `内容规格分裂` 和 `高低档资源边界失控`
- 为了让场景“看起来更满”，不断叠角色、灯光、特效和 Streaming，结果同一份内容同时打爆 `CPU / GPU / 内存`

如果项目里没有一套共同预算语言，这些动作就会变成一种非常熟悉的循环：

1. 某一条线先出事
2. 团队做局部止血
3. 止血动作把压力转嫁到另一条线
4. 另一条线过一阵子再出事
5. 团队误以为“问题很多很散”，其实本质上只是没有总账和分账

所以我越来越不愿意把“优化”理解成一组孤立技巧。

更准确的理解应该是：

`优化是局部动作，预算管理是总账系统。没有总账，局部动作就很容易彼此打架。`

## 为什么“只有一个总数”几乎一定会失真

很多项目最容易犯的错，就是把预算理解成“给每条线写一个总数”。

但只要只剩总数，预算就很快会失真。

### 1. 只有包体总量，没有交付分层

你会看到首包压下来了，但补丁越来越重；  
或者总量没爆，但高档资源还在给所有用户一起买单。

### 2. 只有内存总量，没有生命周期分账

你会看到“明明还有很多 RAM”，但低端机长玩后还是卡、切场景还是抖、回前台还是死。  
因为真正先坏掉的，常常不是“总量不够”，而是稳态、峰值、工作集、显存和系统强杀边界混在一起了。

### 3. 只有 FPS，没有帧链路拆账

你会看到“平均帧率还行”，但手感就是不稳。  
因为一帧里真正该拆开的，至少还有逻辑时间、渲染提交、GPU 时间、加载尖峰和同步等待。

### 4. 只有项目总预算，没有内容对象分账

你会看到场景、角色、特效、UI 都在慢慢变重，但没人说得清到底是谁在买单。  
因为没有把总账继续回写成对象预算，预算就永远停在“知道有问题”，进不了日常生产。

所以总量当然重要，但它不够。

`总量只能告诉你贵不贵，分账才能告诉你为什么贵、贵在哪里、以及该让谁先收。`

## 一套预算语言至少要先拆开哪几笔账

如果把这组文章先收成最核心的一层，我会把它拆成下面六笔。

### 1. 包体预算

它回答的是下载与交付成本怎么立线。  
核心不是“包越小越好”，而是首包、全量包、热更和常驻资源为什么不能混成一个数字。

这一块后面展开见 [包体预算怎么定：首包、全量包、热更和常驻资源为什么不能混成一个数字]({{< relref "engine-notes/game-budget-02-package-budget.md" >}})。

### 2. 内存预算

它回答的是运行时内存的账怎么立。  
核心不是“内存够不够”，而是稳态、峰值、工作集、显存和 OOM 风险为什么要分开管理。

这一块后面展开见 [内存预算怎么定：常驻、峰值、工作集、显存和 OOM 风险要分开管理]({{< relref "engine-notes/game-budget-03-memory-budget.md" >}})。

### 3. CPU 预算

它回答的是主线程时间、尖峰时机和异步任务怎么立线。  
核心不是“平均 CPU 占用高不高”，而是关键帧、危险时机和主线程压力如何被稳定约束。

这一块后面展开见 [CPU 预算怎么定：帧时间、主线程尖峰、异步任务和危险时机]({{< relref "engine-notes/game-budget-04-cpu-budget.md" >}})。

### 4. GPU 预算

它回答的是像素、带宽、RT、阴影、后处理和分辨率缩放怎么立线。  
核心不是“把画质关低一点”，而是 GPU 成本应该按什么模型拆，才能回到可执行动作。

这一块后面展开见 [GPU 预算怎么定：像素、带宽、RT、阴影、后处理和分辨率缩放]({{< relref "engine-notes/game-budget-05-gpu-budget.md" >}})。

### 5. 帧预算拆账

它回答的是一帧内部到底谁在结账。  
核心不是“这一帧有 16.6ms”，而是逻辑时间、渲染提交、GPU 时间、加载尖峰和同步等待为什么必须分别记账。

这一块后面展开见 [帧预算拆账：逻辑时间、渲染提交、GPU 时间、加载尖峰和同步等待怎么分账]({{< relref "engine-notes/game-budget-06-frame-budget-breakdown.md" >}})。

### 6. 内容对象预算

它回答的是场景、角色、特效这些内容对象，到底各自在吃哪几笔总预算。  
如果这层不补，项目就会永远停在“知道总线超了”，却不能把预算回写成规格。

这一块后面展开见 [场景预算怎么定]({{< relref "engine-notes/game-budget-07-scene-budget.md" >}})、[角色预算怎么定]({{< relref "engine-notes/game-budget-08-character-budget.md" >}}) 和 [特效预算怎么定]({{< relref "engine-notes/game-budget-09-vfx-budget.md" >}})。

## 预算管理为什么一定要进设备、内容和流程

预算如果只留在文档里，很快就会退化成口头约定。

因为项目真正每天都在发生变化的地方，不在文档，而在下面几层：

### 1. 设备分档

如果预算不进入高 / 中 / 低 / 极低档设备的现实差异，它就会天然偏向开发机和旗舰机。

### 2. 内容规格

如果预算不进入场景、角色、特效、贴图、UI 的规则表，它就永远只是“主程知道一点、TA 知道一点、美术再猜一点”。

### 3. Baseline

如果预算不进入固定样本、固定设备组、固定构建模式，它就没法判断“现在是不是已经偏了”。

### 4. CI / Release gate

如果预算不进入告警、阻断、例外审批和回归，它就会在版本压力下被一点点侵蚀掉。

所以预算管理的真正落点，不是“写出一篇大家都认同的文章”，而是：

`把预算变成规格、分档、Baseline、CI 和例外机制里都能执行的边界。`

这也是为什么这条线最后一定会接到 [预算怎样进入规格、分档、Baseline 和 CI：从规则表到门禁和回归]({{< relref "engine-notes/game-budget-10-rules-tiering-baseline-ci.md" >}})，以及更偏流程视角的 [Baseline：性能 / 包体 / 加载 / Crash 预算怎样立线并进 CI]({{< relref "code-quality/baseline-budgets-in-ci.md" >}})。

## 结尾

所以如果只用一句话收住这篇，我会这样说：

`很多性能和交付问题之所以会一遍遍重来，不是因为团队永远缺少下一条优化技巧，而是因为项目从一开始就没有把包体、内存、CPU、GPU、帧链路和内容规格收成同一套预算语言；没有这套语言，优化动作就会彼此打架，预算也迟早会退化成临近发版时的集体救火。`
