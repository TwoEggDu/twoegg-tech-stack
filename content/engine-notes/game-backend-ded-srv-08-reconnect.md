## 问题空间：断线是常态，不是异常

在理想的实验室网络环境里，TCP 连接可以维持数小时不断。但在真实的移动网络环境下——地铁进隧道、电梯关门、4G 切换 WiFi——短暂的网络中断是玩家体验中无法回避的现实。

一个没有设计断线重连的多人游戏，等于把"偶发的网络抖动"变成了"强制踢出游戏"。玩家会感受到：明明只是切换了一下网络，却要重新排队、失去本局进度，甚至被判负。这是体验和留存的双重伤害。

断线重连本质上需要回答两个问题：

**问题一：服务端怎么判断"这个人是断线了"还是"这个人主动离开了"？**

这两种情况在协议层面的表现是不同的。主动离开通常有显式的断开包（TCP FIN）或应用层的离开消息；而网络中断可能只是连接突然沉默，没有任何通知。服务端需要一套机制来区分这两种情况，并对断线情形给予宽限期。

**问题二：玩家重连后，服务端怎么把"游戏状态"交还给他？**

重连不只是重新建立连接，还需要让客户端的游戏状态与服务端同步。这中间涉及认证（确认是同一个玩家）、状态恢复（把错过的游戏进度补给客户端）两个子问题。

## 断线的类型与服务端识别方式

### 断线的两种性质

**主动断线**（Graceful Disconnect）：玩家主动关闭游戏，操作系统发送 TCP FIN 包，或应用层发送显式的离开消息。服务端收到后立即知道玩家已离开。

**网络中断**（Network Disruption）：网络链路故障，TCP 连接不再有数据流，但 OS 并不立即关闭连接——TCP keepalive 机制需要数分钟才能判定连接断开。对于游戏来说，等待 OS 的 keepalive 超时太慢了。

### 心跳机制：游戏层面的连接检测

大多数游戏服务端不依赖 TCP keepalive，而是在应用层实现心跳：

- 客户端每隔 N 秒（通常 1-5 秒）发送一个心跳包
- 服务端记录每个连接的最后心跳时间
- 服务端定期检查所有连接，将超过心跳超时阈值（通常 10-30 秒）的连接标记为"疑似断线"

心跳超时触发后，服务端**不会立即清理玩家状态**，而是进入一个"保活宽限期"，等待玩家重连。

### 主动断线 vs 网络中断的识别

```
收到客户端的显式离开消息？
  ├─ 是 → 主动离开，直接执行离开流程
  └─ 否 → 心跳超时？
           ├─ 是 → 标记为"断线中"，启动重连宽限期计时器
           └─ 否 → 连接正常
```

部分游戏会在客户端发送"主动离开"消息时，在服务端记录一个特殊标记，以区分"玩家主动退出"和"程序崩溃导致的非正常断开"。程序崩溃也属于无显式通知的断线，需要重连宽限期。

## 会话保活策略：超时时间的设计哲学

### 宽限期的本质权衡

重连宽限期（Reconnect Grace Period）是服务端在判定断线后，为玩家预留的重连时间窗口。这个时间的选择是一个权衡：

- **太短**：玩家刚切换网络、还没来得及重连就被踢出，体验极差
- **太长**：房间里有玩家的"幽灵"长时间占位，其他玩家面对一个不能操作的游戏实体，体验也差

### 不同游戏类型的超时容忍度

不同游戏类型对断线的容忍度差异很大：

| 游戏类型 | 推荐宽限期 | 原因 |
|---------|----------|------|
| MOBA（5v5） | 60-120 秒 | 断线期间 AI 接管，对局仍可继续；玩家损失高，有强烈重连意愿 |
| FPS（竞技） | 30-60 秒 | 断线玩家会导致团队劣势，但太长会让对局不平衡 |
| 卡牌/回合制 | 180-300 秒 | 回合制允许暂停，对实时性要求低 |
| 实时 ARPG | 30-60 秒 | 时间敏感，但玩家数据不可丢 |
| 房间派对游戏 | 60-180 秒 | 休闲属性，容忍度较高 |

这些数字不是规范，而是参考起点。实际数值需要根据玩家数据（历史断线时长分布）和产品决策来确定。

### 宽限期内的游戏状态处理

宽限期内，断线玩家的游戏实体通常有三种处理策略：

1. **原地静止**：玩家实体停在断线位置不动。简单，但在高强度对战中会造成影响（玩家成为可被攻击的靶子）
2. **AI 接管**：用简单 AI 控制玩家行为，避免影响对局平衡。实现复杂，需要一套最低限度的 AI 逻辑
3. **无敌/隐藏**：断线期间玩家实体获得无敌或进入隐藏状态。对于高竞技性游戏，这可能被认为不公平

大多数 MOBA 类游戏选择 AI 接管，FPS 类游戏倾向于原地静止。

## 重连认证流程

### 重连 Token 的设计

重连时，服务端需要确认"重连的人是原来那个玩家"。这不能依赖普通的登录 Token，因为：

- 普通 Session Token 通常有 15-60 分钟的有效期，断线 30 秒后仍有效，可以直接复用
- 但需要额外绑定到特定的 session_id，防止用旧 Token 重入其他会话

实践中有两种做法：

**方案 A：复用登录 Session Token + session_id 校验**

玩家重连时携带原有的登录 Token 和目标 session_id，服务端校验：
1. Token 有效性（签名、过期时间）
2. Token 中的 player_id 是否在此 session 的玩家列表中
3. 此 player_id 是否处于"断线中"状态

**方案 B：发放专用重连 Token**

玩家首次加入 DS 时，DS 下发一个专用的重连 Token（短有效期，如 5 分钟，包含 player_id + session_id + 时间戳的 HMAC 签名）。重连时客户端携带此 Token，DS 直接验证，无需查询中央鉴权服务。

方案 B 的优势是 DS 可以独立验证（离线验证），重连速度更快；劣势是需要客户端本地持久化这个 Token。

### 重连的完整认证流程

```
客户端（断线后尝试重连）
    │
    ├─ 重连到 DS（携带重连 Token / 登录 Token）
    │
    ▼
DS 验证 Token
    ├─ Token 无效 / 过期 → 拒绝，通知客户端重新排队
    ├─ session 已进入 SETTLING 或之后阶段 → 拒绝，通知结算结果
    └─ 验证通过 → 将玩家状态从"断线中"改为"已连接"
                   → 进入状态恢复流程
```

## 重连后的状态恢复

重连的最后一步也是最复杂的一步：把错过的游戏进度还给客户端。

### 方案 A：发送当前完整快照（Full State Snapshot）

DS 将当前游戏世界的完整状态序列化后发给重连客户端：所有玩家的位置、血量、状态，当前游戏时间，所有活跃实体等。

优点：实现简单，客户端收到后直接覆盖本地状态即可。
缺点：对于复杂游戏世界，快照可能很大（几百 KB），对于高频更新的 FPS 游戏，快照信息在传输过程中就可能过时。

适用场景：状态变化较慢的游戏（MOBA、回合制 RPG）。

### 方案 B：增量事件重放（Event Replay）

DS 保存一段时间内的所有游戏事件日志（比如最近 120 秒内的所有状态变更事件）。玩家重连后，DS 找到玩家断线时间点，将断线期间的所有事件打包发给客户端，客户端按序重放。

优点：传输量更精确（只传玩家错过的部分），客户端可以忠实还原断线期间发生的一切。
缺点：客户端需要具备事件重放能力；对于高频 FPS 游戏，120 秒的事件日志量依然很大；重放速度需要加速以快速追上当前时间点。

适用场景：需要客户端完整还原对局历史的游戏，例如 MOBA。

### 实践中的混合方案

许多游戏采用混合方案：

1. 发送当前时刻的完整状态快照（让客户端立刻进入可玩状态）
2. 发送断线期间的关键事件日志（补充快照无法表达的瞬态信息，如技能施放记录、死亡事件）

客户端收到快照后立即渲染当前状态，同时在后台处理补充事件，无需重放历史动画。

## 其他玩家视角：断线期间的体验设计

断线重连不只是断线玩家自己的问题，也影响房间内其他玩家的体验。

**断线通知**：断线发生后立即广播"玩家 X 断线，等待重连"，让其他玩家知情。不要沉默处理。

**AI 接管的质量**：接管 AI 的行为质量直接影响对局平衡。过于弱的 AI 会让队友劣势明显；过强的 AI 则破坏游戏性。大多数游戏选择"防御性 AI"——只做最低限度的操作（不让玩家死亡、不进攻），降低对对局的影响。

**是否允许继续对局**：对于 5v5 MOBA，1 人断线通常不暂停游戏（否则容易被滥用）。但对于 2v2 或更小规模的对局，1 人断线会导致严重的不平衡，部分游戏会在此情况下暂停对局等待重连。

## 重连失败的降级策略

当宽限期内玩家未能重连，或玩家尝试重连时 session 已超出可重连阶段，需要有明确的降级处理：

**会话已结束（SETTLING 或之后）**：告知客户端对局已结束，展示结算结果（如果可用）。

**会话仍在进行但宽限期已过**：
- 玩家被标记为"永久离开"
- 根据游戏规则决定是否触发对局提前结束
- 将相应的处罚（排位分惩罚、信用分扣除）写入玩家记录

**玩家希望继续游玩**：重定向到匹配流程，加入新的对局。不要直接断开客户端不给任何反馈。

**技术层面的降级**：如果 DS 本身在重连尝试期间崩溃，客户端应识别到连接失败，通知玩家"服务器异常"，触发联系客服 / 补偿流程，而不是让玩家陷入无限重连循环。

---

## 最短结论

断线识别依赖应用层心跳而非 TCP keepalive，超时后进入保活宽限期而不是立刻清理玩家状态。宽限期长度根据游戏类型设定（MOBA 60-120 秒，FPS 30-60 秒）。重连认证可复用登录 Token 加 session_id 校验，或发放短期专用重连 Token。状态恢复根据游戏类型选择完整快照（简单慢游戏）或事件重放（复杂快游戏），实践中常用混合方案。宽限期超时后，降级到永久离开处理并给玩家明确反馈。
