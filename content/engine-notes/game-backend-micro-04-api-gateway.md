## 没有网关的后端长什么样

想象一个没有 API 网关的微服务游戏后端：客户端需要知道账号服务的地址（用来登录）、匹配服务的地址（用来排队）、背包服务的地址（用来查物品）……每次服务部署迁移，客户端都要跟着更新配置。鉴权逻辑散落在每个服务里，每个团队自己实现一套 Token 验证，出现一致性漏洞只是时间问题。限流策略也散落各处，某个接口被刷接口时，要在 5 个服务里分别找到对应的限流配置修改。

这种状态不是极端假设，而是很多早期微服务项目的真实状态。API 网关的出现是为了解决这个问题：**所有外部流量通过统一的入口进入系统，统一处理横切关注点**。

但网关的职责一旦边界模糊，它又会成为另一个问题——一个承载了业务逻辑的、难以维护的大型中间件。

---

## 网关的核心职责：接入层应该做什么

API 网关是客户端（游戏客户端、Web 控制台、GM 工具）和内部服务之间的唯一入口层。它的职责必须严格限定在**接入层关注点**，而不是业务逻辑。

四个核心职责：

### 1. 鉴权（Authentication & Authorization）

鉴权是接入层最核心的职责，也是最值得放在网关层统一处理的地方。

**Token 验证**：客户端每次请求都携带 Token（通常是 JWT 或服务端生成的 Session Token）。网关负责验证这个 Token 的有效性——签名是否正确、是否过期、是否被撤销。

- JWT 方案：网关本地验证 JWT 签名（RSA/HMAC），无需查库，延迟极低。缺点是 Token 一旦签发，在过期前无法强制失效（踢号场景需要额外处理，通常是维护一个 Token 黑名单）。
- Session Token 方案：网关调用账号服务或查询 Redis 验证 Token，可以立即失效，但增加了一次网络调用（通常是 0.5ms 以内的 Redis 查询，可以接受）。

**权限检查（Authorization）**：确认当前玩家是否有权限访问这个接口。粗粒度的权限检查（比如：普通玩家不能访问 GM 接口）适合在网关层做。细粒度的业务权限（比如：玩家是否有权领取某个活动奖励）不应该在网关层做，因为这需要业务上下文，应该由目标服务自行判断。

### 2. 限流（Rate Limiting）

限流是防止服务被恶意刷接口或突发流量压垮的关键机制。网关层的限流是粗粒度的防护层，按维度分：

**按 IP 限流**：每个 IP 每分钟最多请求 N 次，防止未登录阶段的爆破攻击（比如暴力猜账号密码）。

**按玩家 ID 限流**：登录后按 PlayerID 限流，每个玩家每秒最多触发 M 次操作，防止外挂和脚本刷接口。

**按接口限流**：某些接口（比如支付、排行榜查询）本身有容量上限，独立设置接口级别的 QPS 上限，超过则返回 429（Too Many Requests）。

**算法选择**：
- **令牌桶（Token Bucket）**：允许短暂突发流量，适合大多数游戏接口。桶的容量决定了突发上限，填充速率决定了平均速率。
- **漏桶（Leaky Bucket）**：严格匀速输出，适合对下游服务有严格保护需求的场景（比如支付接口，不希望突发流量打穿支付服）。
- **滑动窗口**：比固定窗口精确，适合 "每分钟 N 次" 这类语义明确的限制。

限流状态存储在 Redis（共享状态，支持多网关实例）中，使用 Lua 脚本保证原子性。

### 3. 路由（Routing）

网关根据请求的 URL 路径、Header 或其他特征，将请求转发到对应的后端服务。

**基础路由**：`/api/account/*` 转发到账号服，`/api/match/*` 转发到匹配服，`/api/inventory/*` 转发到背包服。这是最基础的路径前缀路由。

**灰度分流**：新版本服务上线时，可以通过网关按比例分流（比如 5% 流量走新版本，95% 走旧版本）。分流依据可以是随机比例，也可以是特定的玩家群体（内测资格、特定渠道玩家）。

**版本路由**：客户端通过 Header 携带 App 版本号（`X-App-Version: 2.5.1`），网关根据版本号路由到对应的服务版本，实现不同客户端版本共存。

### 4. 协议转换（Protocol Translation）

游戏客户端和内部服务之间存在协议差异，网关负责转换：

**HTTP → 内部 gRPC**：客户端发送标准 HTTPS 请求，网关将其转换为内部的 gRPC 调用。客户端不需要知道内部服务使用什么协议。

**外部 JSON → 内部 Protobuf**：外部接口用 JSON（可读性好，易于调试），内部服务间用 Protobuf（高效），网关做序列化格式的转换。

## 游戏网关的特殊需求：长连接路由

普通 Web 后端的 API 网关处理的都是短连接 HTTP 请求，每次请求独立处理。但游戏后端有大量长连接场景，这给网关设计带来了额外的复杂度。

**WebSocket 路由**：游戏客户端通常通过 WebSocket 保持长连接，用于接收实时推送（战斗状态同步、好友上线通知、实时聊天）。WebSocket 的路由和 HTTP 不同——连接建立时（握手阶段）网关做鉴权和路由决策，之后的消息在同一条连接上流转，不再做逐包路由。

**连接迁移**：战斗过程中玩家断线重连，需要把新的 WebSocket 连接路由到同一个战斗服实例（有状态路由）。这不是无状态 HTTP 路由能处理的，需要网关维护 `PlayerID → 战斗服实例` 的映射，或者由战斗服把映射写入 Redis，网关查询后路由。

**多协议接入**：有些游戏客户端用 TCP 自定义协议（比如旧平台客户端），有些用 WebSocket，有些用 HTTP。网关层可以统一接入不同协议，将其转换为统一的内部协议后转发。这避免了内部服务需要支持多种外部协议。

## 常见网关方案对比

### Kong

Kong 是基于 Nginx + OpenResty 的开源 API 网关，有丰富的插件生态（鉴权、限流、监控、链路追踪）。优点是功能完善、社区成熟、支持声明式配置（通过 KongAdmin API 或 Kubernetes CRD 管理路由规则）。缺点是插件开发需要用 Lua，学习曲线陡，部署依赖 PostgreSQL 或 Cassandra。

**适用场景**：需要快速接入网关能力、团队没有自研网关精力的中型项目。业务定制化要求不高，标准插件能满足大部分需求。

### Nginx + Lua（OpenResty）

OpenResty 是 Nginx 嵌入 LuaJIT 的扩展，可以在 Nginx 内部用 Lua 实现自定义逻辑。性能极高（接近 C 原生），支持 ngx.shared.DICT 做限流状态存储，支持 cosocket 实现异步 Redis 查询。

**适用场景**：团队有 Lua 能力，需要高性能且高度定制化的网关逻辑（比如自定义鉴权逻辑、特殊协议处理）。国内大型游戏公司很多用这个方案做接入层。

### 自研网关

用 Go（高并发、低内存）或 Java（Spring Cloud Gateway / Zuul）自研网关，完全控制所有逻辑。优点是和内部系统深度集成（统一鉴权模型、内部服务发现、自定义监控指标）。缺点是研发和维护成本高。

**适用场景**：大型游戏公司，有专门的基础设施团队，网关的定制化需求多（比如需要和自研游戏 SDK 深度集成，或者需要自定义的多协议接入）。

### 云厂商 API 网关

AWS API Gateway、阿里云 API 网关、腾讯云 API 网关等。优点是免运维，直接集成云平台的鉴权（IAM）、监控（CloudWatch）体系。缺点是定制化能力有限，成本随流量线性增长，且对 WebSocket 长连接的支持通常有特殊限制。

**适用场景**：流量规模不大、运维资源有限的小型团队，或者需要快速验证 MVP 的早期阶段。

## 网关的职责边界：不该做什么

网关职责边界放宽的常见表现：

**在网关里实现业务逻辑**：比如在网关里判断"玩家等级不足 10 级不能进入竞技场"。这是业务规则，不是接入层关注点。业务规则可能每个版本都变，如果放在网关里，每次版本更新都需要修改网关配置，网关的发布节奏被业务拖着走。

**在网关里做数据聚合**：有些场景（比如个人主页需要账号信息 + 排行榜数据 + 好友列表）需要聚合多个服务的数据，有人把这个聚合逻辑放进网关。这会让网关变成一个 BFF 层，承担了本不属于它的职责。BFF 应该是独立的服务，网关只负责转发，不负责聚合。

**在网关里做格式转换和数据清洗**：如果网关开始做复杂的请求/响应数据变换（重映射字段、剔除敏感字段、格式归一化），它就变成了一个数据管道，而不是路由层。复杂的数据转换应该由下游服务或专门的适配器处理。

**在网关里存储业务状态**：网关应该是无状态的（每个实例处理请求的方式完全相同），只有限流状态和路由规则这类基础设施状态可以存储（在 Redis 或配置中心）。业务状态（玩家会话、游戏进度）一律不进网关。

判断一个逻辑是否应该放在网关的原则：**这个逻辑如果不存在，对业务功能有影响吗？如果没有影响，那它属于接入层关注点；如果有影响，那它属于业务逻辑，不应该在网关里。**

## 网关成为单点的问题

单个网关实例是系统的单点故障。所有流量都经过网关，网关挂了，整个后端对外不可用。解决方案：

**多实例 + 负载均衡**：部署多个网关实例，前面挂 DNS 负载均衡（AWS Route53、阿里云 SLB）或 L4 负载均衡（Nginx 作为前置 LB）。任何一个网关实例挂掉，流量自动切到其他实例。

**网关本身的无状态化**：网关实例之间不共享内存状态，所有需要共享的状态（限流计数、路由规则）存在 Redis 或配置中心。这样网关实例可以任意水平扩展，任意重启，不影响系统状态。

**健康检查和自动摘除**：负载均衡层对网关实例做定期健康检查（HTTP /health 或 TCP check），不健康的实例自动摘除出负载均衡池。配合 K8s 的 Readiness Probe，可以在版本发布时做滚动更新，避免发布瞬间流量打到未就绪的实例。

**容量规划**：网关是所有流量的必经之路，它的 QPS 上限决定了整个系统的接入上限。要根据游戏的峰值 QPS（在线人数 × 每人每秒请求数）提前规划网关集群的容量，预留 30%~50% 的冗余容量，避免峰值时满载。

**跨地域容灾**：大型游戏通常需要在多个地域（华东、华南、华北）分别部署网关集群，玩家就近接入。地域级故障时（机房断电、运营商故障），可以将玩家路由切到其他地域的网关集群。

## 工程边界：哪些关注点不该进网关

有些横切关注点看起来适合放网关，但实际上不应该：

**日志记录**：网关可以记录接入层的访问日志（IP、请求路径、响应时间、HTTP 状态码），但业务日志（某个玩家做了什么操作，产生了什么业务事件）必须在业务服务里记录，不能依赖网关层的访问日志来重建业务状态。

**加解密**：如果需要对请求体/响应体做端到端加密（防抓包），可以在网关层做 TLS 终止（HTTPS → 内部 HTTP），但应用层的自定义加密逻辑（比如游戏常见的自定义二进制协议加密）应该由专门的安全层或 SDK 处理，不要污染网关。

**链路追踪**：网关应该在每个请求入口生成 TraceID，并在响应头中返回（`X-Trace-Id`），但链路追踪的采集、存储、展示是独立的可观察性基础设施，不是网关的职责。

## 最短结论

API 网关的价值在于统一接入层的鉴权、限流、路由，但它的边界必须严格限定在接入层关注点——一旦开始在网关里写业务逻辑，网关就从基础设施变成了所有团队都要维护的共享代码库，维护成本会快速失控。
