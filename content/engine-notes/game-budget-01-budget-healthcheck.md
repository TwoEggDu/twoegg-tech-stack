很多项目的问题不是“我们完全没有预算意识”。

更常见的是下面这几种：

- 预算写过，但只存在于去年的一份文档里
- 主程和 TA 心里有数，但没有变成团队共同规则
- 包体、内存、CPU、GPU 各自有数字，但彼此互不相认
- 低档机、活动场景、长玩场景、热更新场景根本没进入预算口径
- 超线以后没有固定动作，所以每次都靠临场沟通

所以在重做预算之前，更值钱的第一步通常不是立刻开表，而是先判断：

`我们现在到底算不算“有预算体系”？`

这篇文章就是一份预算体检框架。

## 先分清三种病

预算问题最容易被混成一句话：“我们预算做得不够好。”

但更准确地说，常见的其实是三种不同的病：

### 1. 没预算

表现是：

- 只有模糊愿望，没有明确数字
- 只有总量要求，没有分账
- 没有人知道谁对哪一层负责
- 线上出了问题，团队才临时讨论该怎么收

### 2. 预算失真

表现是：

- 文档里有数字，但已经不能反映真实项目
- 首包线写的是旧版本目标
- 角色规格写的是理想值，真实资源早已超标
- 目标设备已经变化，但预算仍按旧设备池立

### 3. 预算没接进执行

表现是：

- 有文档，有数字，甚至有模板
- 但资源导入、日常提审、CI、发版门禁都不看它
- 超线以后没有自动告警，也没有固定审批动作

这三种病的解法不同。  
如果不先分清，团队就会很容易用错误方式下药。

## 体检不看“写没写文档”，而看能不能回答 8 个问题

我会用 8 个问题判断一个项目的预算体系是不是站住了。

### 1. 项目有没有明确区分不同预算桶

至少应该能分清：

- 首包 / 全量包 / 热更包
- 稳态内存 / 峰值内存 / 红线
- CPU 帧时间 / GPU 帧时间
- 场景 / 角色 / 特效 / UI 的内容规则

如果这些东西还混成“不要太大”“不要太慢”，那通常还不算真正有预算。

### 2. 有没有“硬线、告警线、观察线”

只有一个总数字，通常不够。

更稳的做法是三条线：

- `观察线`
开始关注趋势，但暂不阻断。

- `告警线`
必须解释原因和处理计划。

- `硬线`
超过就返工、阻断或走例外审批。

如果没有层次，团队就会在“到底算不算超”这件事上无限拉扯。

### 3. 预算是不是只存在于少数人脑子里

一个很危险的状态是：

- 主程知道 CPU 预算
- TA 知道贴图和 Shader 预算
- 美术知道角色面数目标

但这些知识没有互相连起来，也没有写成共享规则。

这种项目在关键人忙、换人或版本压力大的时候最容易失控。

### 4. 内容规则有没有真正落地

我会直接看四种东西：

- 角色规格表
- 场景规格表
- 特效规格表
- 包体 / 资源分层规则

如果预算还没有继续落到：

- 面数
- 骨骼数
- 材质球数
- 贴图尺寸和格式
- 粒子上限
- 屏占比
- 同屏数量

那它通常还停在口号层。

### 5. 预算是不是只在开发机和旗舰机上成立

很多项目看起来“没问题”，其实只是因为：

- 场景在编辑器里看
- 测试机配得太高
- 没测长玩
- 没测热机
- 没测活动期内容叠加

我会直接问：

- 最低支持设备是谁
- 最长游玩时长按多久看
- 有没有冷机和热机对照
- 有没有活动 / 大世界 / 高特效战斗场景的独立口径

如果这些没有，预算很可能只是“开发环境预算”。

### 6. 超线以后有没有固定动作

这是我最看重的一条。

因为很多团队真正的问题不是“不知道超了”，而是“超了也不知道该怎么办”。

至少应该明确：

- 谁能决定阻断
- 谁能批准例外
- 什么情况走降档兜底
- 什么情况必须返工资源
- 什么情况需要回写规格表

如果每次都重新开会讨论，这套预算就还不稳定。

### 7. 有没有证据链，而不是只靠感觉

预算体检时我会看这些证据是否存在：

- 包体报告 / Bundle 贡献度
- Memory Profiler 或 memreport
- CPU / GPU 真机采样
- 关键场景 Baseline
- 设备矩阵
- 线上崩溃、卡顿、热、掉档数据

没有证据链，预算很快就会退回成“我感觉差不多”。

### 8. 有没有回归门禁

最后要看的是：

- 它有没有进 CI
- 有没有夜构对比
- 有没有发版前门禁
- 有没有超线告警

没有门禁的预算，最后通常只会在复盘 PPT 里存在。

## 体检时最常见的误判

### 1. “我们有优化动作，所以我们有预算”

不对。  
优化动作是局部行为，预算体系是边界系统。

### 2. “我们有一份规则文档，所以我们有预算”

也不对。  
如果规则没有证据链、没有责任人、没有超线动作，它只是静态资料。

### 3. “我们线上没大问题，所以预算应该合理”

这同样不稳。

很多预算问题不会先显成严重事故，而会先显成：

- 更新越来越重
- 低端机体验越来越差
- 活动版本越来越难发
- 每次做新内容都越来越费劲

### 4. “我们只有总预算，不需要分账”

这通常是最容易拖垮团队协作的一种错觉。

因为没有分账，就没有办法把责任真正分发到：

- 场景
- 角色
- 特效
- UI
- 包体分层

## 体检结论怎样落地

体检不是为了得出一句“项目不健康”，而是为了决定先修哪一层。

我会按下面顺序修：

1. 先把预算桶分清
先分清首包、热更、稳态内存、峰值内存、CPU、GPU。

2. 再立三条线
观察线、告警线、硬线。

3. 再补内容分账
把场景、角色、特效、UI 这些对象接回规则表，而不是继续只盯总量。

4. 再补证据链
如果没有真机、包体报告、内存快照、Baseline、设备矩阵和线上数据，预算很快还会再次失真。

5. 最后才是门禁和审批
把真正已经稳定的预算接进 CI、发布门禁和例外审批，不要一开始就把“模糊数字”硬塞进流水线。

## 体检结论可以分成四档

为了避免最后只落成一句“还行”或者“有问题”，我建议直接把结论分成四档。

### A 档：预算体系已站住

特征：

- 总预算和分账预算都明确
- 有证据链
- 有责任人
- 有超线动作
- 已经进入规则和门禁

这类项目下一步最该做的，不是再补总论，而是持续回写模板和附录。

### B 档：预算已建立，但执行层还不稳

特征：

- 有总预算和部分内容规则
- 也有一些证据
- 但执行层不稳定，超线动作还不够固定

这类项目最该先补的是模板和验收口径，而不是重写整套预算体系。

### C 档：预算失真，已经不能指导当前项目

特征：

- 旧文档还在
- 旧数字还在
- 但目标设备、内容规模和上线节奏都已经变了

这类项目最该先做“预算整编”，把旧规则收成当前版本能用的新表。

### D 档：还没有真正的预算体系

特征：

- 只有模糊愿望
- 没有分账
- 没有证据链
- 没有责任边界
- 没有超线动作

这类项目不要先谈工具和门禁，应该先回去看 `预算-00`，再把最小体检表和总预算总表立出来。

## 什么时候该去看模板文

如果你在体检过程中出现下面这些状态，说明已经不该只读概念文了，而应该立刻切到模板文：

- 团队能说出问题，但说不清字段怎么填
- 预算写过几版，但每次审计口径都不一样
- 大家都认同要体检，但没人知道体检表长什么样

这时最适合继续看的就是：

- [预算体检模板：怎样用一张表审项目有没有预算体系]({{< relref "engine-notes/game-budget-17-budget-healthcheck-template.md" >}})

- 你知道问题在哪，但不知道表格该怎么开
- 你知道角色 / 场景 / 特效该收规则，但不知道字段该怎么写
- 你知道 CI 应该拦预算，但不知道“拦哪几个字段”才合理

## 什么时候该开“预算整改单”

我建议至少满足下面两条中的一条，就应该开一张正式整改单，而不是继续靠会议口头跟：

- 同一条预算已经连续两版超线
- 同一个问题已经在两个以上对象上重复出现，例如角色、特效和场景都在为同一条 GPU 预算买单

整改单最少应该包含：

- 问题对象
- 当前证据
- 责任人
- 计划动作
- 截止版本
- 回归验证点

## 结尾

所以我现在更愿意把预算体检理解成：

`不是检查“有没有一份预算文档”，而是检查这个项目到底有没有一套可分账、可证明、可执行、可回归的边界系统；如果没有，下一步该补的是总表、对象模板和整改单，而不是继续靠经验硬撑。`
