## 问题空间：多人游戏的"一致性困局"

多人游戏的本质难题不是"数据传输"，而是**世界状态的一致性**。

每个玩家的机器各自运行一份游戏逻辑，操作延迟不同、帧率不同、硬件浮点行为可能不同。如果什么都不做，每台机器的世界很快就会分叉——A 玩家看到敌人在左边，B 玩家看到同一个敌人在右边，游戏就完了。

解决这个问题的思路在历史上走过两条截然不同的路：

1. **帧同步（Lockstep）**：不同步结果，同步过程。所有客户端收到完全相同的输入序列，各自独立计算，如果逻辑完全确定，结果自然一致。
2. **状态同步（State Sync）**：不依赖客户端计算，服务端维护权威状态，周期性向所有客户端推送世界快照。

这两条路的分歧点在于：**谁来计算游戏逻辑，计算结果由谁说了算**。搞清楚这个问题，选型才有依据。

## 抽象模型：帧同步的"确定性世界"

### 核心思路

帧同步的理论基础极其简洁：**给定相同的初始状态和相同的输入序列，确定性系统必然产生相同的输出**。

```
初始状态 S0 + 输入序列 [I1, I2, I3, ...] → 状态序列 [S1, S2, S3, ...]
```

服务端（或中继服务器）的职责只有一件事：**收集所有玩家的输入，打上帧编号，广播给所有人**。每个客户端拿到相同的输入序列后，独立跑游戏逻辑，结果理论上完全一致。

这意味着什么？带宽消耗极低。N 个玩家的游戏，服务端每帧只需传输 N 个玩家的输入包，而非整个世界的状态。一个 RTS 游戏有几百个单位，帧同步传输的数据量可能只有几十字节——服务端甚至不需要跑任何游戏逻辑。

### 帧同步的根本约束：完全确定性

这里有一个非常严苛的前提：**游戏逻辑必须完全确定（Deterministic）**。

什么叫完全确定？给相同输入，永远输出完全相同的结果——不是"差不多一样"，是**逐 bit 相同**。

这个约束直接带来三类工程禁区：

**1. 浮点数运算**

IEEE 754 浮点数在不同平台、不同编译器优化级别下，可能产生不同的计算结果。一个看上去微不足道的 `float` 精度差异，经过几百帧累积，会让两台机器的游戏世界完全分叉。

解决方案：用**定点数（Fixed Point）**替代浮点数。定点数用整数运算模拟小数，结果完全一致，代价是运算性能略低，且需要专门的定点数数学库（向量、三角函数、物理模拟都要重写）。

**2. 随机数**

如果游戏逻辑用到随机数，必须保证所有客户端使用**相同的伪随机数生成器（PRNG），并从相同的种子出发**。C# 的 `System.Random`、Unity 的 `UnityEngine.Random` 在不同平台下未必产生相同序列，必须使用自实现的确定性 PRNG。

**3. 集合遍历顺序**

`Dictionary`、`HashSet` 等集合在不同语言实现或平台上，遍历顺序可能不同。帧同步代码中，所有集合操作必须使用确定性顺序——通常用排序后的列表，或专门的确定性容器。

**4. 多线程并发**

多线程执行顺序不确定，帧同步逻辑必须单线程化（或严格序列化执行路径）。

### 帧同步的同步点与回放

由于所有客户端各自独立计算，理论上任何一点微小的不一致都会随时间放大。为了检测和纠正这个问题，工业实践中通常加入**哈希校验（Hash Check）**：每隔若干帧，客户端计算当前世界状态的哈希值并上报服务端，如果某个客户端的哈希值与其他人不一致，说明出现了"desync"，需要回滚重算或重新同步。

这套机制还附带一个额外能力：**完美回放**。只要保存输入序列，就能在任何时刻完整重现整场对局，这对录像、观战、赛后分析都极其有价值。

## 抽象模型：状态同步的"权威服务端"

### 核心思路

状态同步的出发点完全不同：**不依赖客户端逻辑的正确性，由服务端维护唯一权威状态，定期向客户端推送**。

```
服务端 World State (权威) → 周期性快照 → 所有客户端
```

客户端收到快照后，用插值（Interpolation）和预测（Prediction）让画面看起来流畅，但这些都是"展示层"的手段，不影响服务端的权威判断。

这个模型的最大优点是**容错性高**。客户端的逻辑随便出 bug、随便作弊，都不影响服务端的状态。服务端说你死了你就死了，服务端说你有多少血就是多少血。

### 状态同步的带宽模型

状态同步的代价是带宽随实体数量线性增长。

假设服务端每帧（20ms = 50fps）推送所有动态实体的状态，每个实体包含位置（Vector3，12 bytes）、旋转（4 bytes Quaternion）、血量（4 bytes int）等字段，约 32 bytes/实体：

- 10 个实体：640 bytes/s 下行（可忽略不计）
- 100 个实体：6.4 KB/s（完全可接受）
- 1000 个实体：64 KB/s（开始需要优化）
- 10000 个实体：640 KB/s（需要强力裁剪策略）

真实的 MMO 或大型战场游戏不可能每帧推送所有实体，必须引入 **Interest Management**：只向玩家推送其"可感知范围"内的实体状态。常见实现是空间划分（九宫格、AOI 圆形区域），将推送量控制在合理范围。

此外，状态同步还需要对字段做**差量编码（Delta Encoding）**——只发生变化的字段才发送，以及**位域压缩（Bitpack）**——把多个小数值打包到同一字节，进一步降低带宽。

## 具体实现对比

| 维度 | 帧同步（Lockstep） | 状态同步（State Sync） |
|------|--------------------|-----------------------|
| 服务端职责 | 只做中继，不跑游戏逻辑 | 跑权威游戏逻辑，推送状态 |
| 带宽特征 | 随玩家数增长（输入数据小） | 随实体数量增长（状态数据大） |
| 客户端复杂度 | 必须完全确定，逻辑更严苛 | 主要做渲染与插值，逻辑较简单 |
| 作弊难度 | 较难防作弊（客户端跑逻辑） | 天然防作弊（服务端权威） |
| 断线重连 | 需要快速回放追帧，耗时 | 直接拿最新快照，恢复快 |
| 延迟感知 | 低（本地先行，无需等服务端） | 高（每次操作需服务端确认） |
| 回放支持 | 天然支持（存输入序列即可） | 需额外记录快照流 |
| 适用规模 | 小到中规模（4-8 人为主） | 无上限（有 Interest Management） |

## 适用场景与典型案例

### 帧同步的主场：RTS 与格斗游戏

帧同步天然适合以下场景：

- **RTS（实时策略）**：《星际争霸》系列、《魔兽争霸》系列都是帧同步的经典案例。游戏单位数量多（几百个），但玩家操作输入量不大，带宽优势明显；同时游戏单位的物理和 AI 需要在所有客户端完全一致。
- **格斗游戏**：GGPO 是格斗游戏界最著名的帧同步网络库，用于《街头霸王》系列的网络对战实现。格斗游戏帧精度要求极高，帧同步能保证判定一致性。
- **MOBA（多人在线竞技）**：《王者荣耀》采用帧同步，解决了 5v5 中较多单位的带宽问题，同时保证技能判定一致。

### 状态同步的主场：FPS 与 MMO

状态同步天然适合：

- **FPS（第一人称射击）**：CS:GO、Valorant、Apex Legends 都是状态同步。射击游戏对作弊敏感，服务端权威判定子弹是否命中是必须的。
- **MMO（大型多人在线）**：《魔兽世界》、《最终幻想 XIV》等，在线玩家数量巨大，必须用 Interest Management 管理状态推送范围。
- **Battle Royale**：大逃杀类游戏（PUBG、Fortnite）玩家分散在大地图，天然适合 AOI 裁剪的状态同步。

### 混合方案：实用主义的选择

现实中不少游戏采用混合方案：

- **部分逻辑帧同步 + 部分逻辑状态同步**：比如技能判定用服务端权威（状态同步），但单位寻路和移动用帧同步逻辑在客户端运行，减少服务端计算压力。
- **客户端帧同步 + 服务端仲裁**：客户端之间帧同步保证一致，服务端定期做哈希校验和仲裁，发现 desync 时以服务端状态为准。

这类混合方案的代价是架构复杂度翻倍，调试难度极高，通常只在有足够工程积累的大型团队中出现。

## 工程边界：选型时的决定性问题

在做技术选型之前，有一个问题必须先回答清楚：

**你的游戏逻辑能否做到完全确定性（Full Determinism）？**

如果答案是"否"或者"不确定"，帧同步的路基本就关闭了。原因很简单：不确定性系统一旦用帧同步，desync 问题会让你的工程师花掉大量时间追查跨平台浮点差异，代价极高。

几个判断标准：

1. **技术栈**：如果你用 Unity + C# 的内置物理引擎（PhysX），PhysX 在不同平台的确定性存疑——这意味着你需要完全绕过内置物理，自己实现或引入第三方确定性物理库（如 FixedMath.Net + 自定义 2D/3D 物理）。
2. **游戏类型**：游戏是否有大量随机元素、复杂物理交互？这些都需要确定性化，工作量可观。
3. **团队规模与经验**：帧同步的 desync 排查需要专门的工具链（回放、逐帧状态比对），没有足够经验的团队往往低估这部分成本。
4. **目标玩家规模**：如果单局玩家数超过 16 人，帧同步的同步点控制（所有人必须在同一帧推进）会带来显著的掉帧风险，状态同步往往更适合。

另一个维度是**防作弊需求**。帧同步的客户端本地跑逻辑，天然对外挂友好——客户端修改了判断条件，服务端看不出来。如果你的游戏有强竞技性（如射击、MOBA），需要在帧同步框架外额外做行为检测，而状态同步的服务端权威在这方面有天然优势。

## 最短结论

- **帧同步**：低带宽，高逻辑一致性要求，适合小规模、强确定性场景（RTS、格斗、MOBA）；选它之前先确认你能接受定点数、确定性 PRNG、单线程逻辑的全套约束。
- **状态同步**：高带宽，服务端权威，天然防作弊，适合大规模、高作弊风险场景（FPS、MMO）；选它意味着你需要认真处理带宽优化、Interest Management 和延迟补偿。
- **核心判断标准**：你的游戏逻辑能做到完全确定性吗？能 → 帧同步可行；不能 → 默认选状态同步，必要时做混合方案。

两种方案都没有对错，只有适不适合。错误的选型会在项目中期带来巨大的重构成本。在立项阶段花两天做这个决策，远比上线前六个月返工来得值。
