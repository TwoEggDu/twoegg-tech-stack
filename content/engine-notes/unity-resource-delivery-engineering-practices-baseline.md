写到这里，这条系列的原理主线其实已经铺得比较完整了：

- 资产是什么，文件和资产对象不是一回事
- `Scene / Prefab / ScriptableObject` 这些序列化资产各自站在哪
- 序列化数据怎样恢复成运行时对象
- AssetBundle 为什么解决的是交付，不是单纯加载
- Addressables 为什么更像建立在交付层之上的管理层

这时候项目里真正会冒出来的问题，通常就不再是：

`Unity 底层到底怎么做。`

而是：

`团队现在到底该先把哪些规则定下来，资源交付这件事才不会越做越乱？`

这也是很多项目最容易反过来做的地方。

一上来先争论的是：

- 用不用 `Addressables`
- 要不要上 `YooAsset`
- 还能不能继续用裸 `AssetBundle API`
- 要不要做远端更新、分包、预下载

但这些问题真正排优先级时，通常都排在第二层。

因为如果更前面的工程基线没立住，换哪套框架，最后都只是：

`用不同名字承载同一团混乱。`

## 先给一句总判断

如果把这篇的核心结论压成一句话，我会这样说：

`Unity 资源交付的最佳实践，核心不是先选哪套框架，而是先把交付单元、逻辑命名、内容身份、缓存复用、回滚路径和自动校验做成显式规则，然后再让 Addressables、YooAsset 或自研管理层去承载这些规则。`

这句话里最关键的，是两个词：

- `显式规则`
- `承载`

前者在说，团队不能再靠经验和口头约定维护资源系统。
后者在说，框架更像规则的承载层，不是规则本身。

## 一、先立六条基线，再谈工具选型

如果要把资源交付这件事压成一套最小工程基线，我建议至少先立下面六条：

1. `分组边界`
2. `逻辑命名规则`
3. `内容身份规则`
4. `缓存与回滚规则`
5. `构建校验规则`
6. `烟测入口`

这六条里，最常被低估的不是技术难度，而是它们必须被当成团队级规则，而不是“打包同学自己的习惯”。

因为只要资源交付进入多人协作，很多问题都不再是某一次构建命令写错了，而是下面这些边界从来没有被正式定义：

- 什么东西应该放在同一个交付单元里
- 什么名字代表稳定语义，什么名字只是发布产物
- 版本号、`Hash` 和缓存副本之间到底谁说了算
- 回滚时切回的是哪个层级
- 哪些风险必须在上线前自动被挡住

不把这些边界讲清，资源系统很快就会进入一种很典型的状态：

`能跑，但没人敢动。`

## 二、分组先按更新边界、共享边界和运行时局部性切，不要按资源类型切

资源交付里最先决定后面复杂度的，通常不是加载 API，而是：

`你最初怎么切组。`

### 1. 不要优先按资源类型切

很多项目最早会直觉地按下面这些方式切：

- 所有 `Texture` 一组
- 所有 `Material` 一组
- 所有 `Prefab` 一组
- 所有 `AudioClip` 一组

这类切法的直观感很强，但工程上通常不稳。

因为资源交付真正要服务的，不是编辑器里“它看起来是什么类型”，而是：

- 它跟谁一起更新
- 它跟谁一起被加载
- 它跟谁一起被复用
- 它跟谁一起应该被缓存或淘汰

按资源类型切，最容易带来的后果就是：

- 一个场景首包需要横跨很多组才能凑齐
- 共享依赖被过度抬升成“全局公共包”
- 加载链变长，但每个组又没有稳定业务边界

### 2. 也不要轻易走到另外两个极端

另两个常见极端同样不稳：

- `一资源一包`
- `一功能一个超级包`

前者的问题是包数爆炸、依赖调度复杂、缓存碎片化严重。
后者的问题是更新粒度粗、首包膨胀、明明只改了很小的内容却要整体重发。

所以更稳的切法通常不是最“细”或最“粗”，而是更贴近三条真实边界：

1. `更新边界`
同一批内容是否大概率一起发布、一起回滚。

2. `共享边界`
它到底是被两个入口偶尔共用，还是被全项目高频复用。

3. `运行时局部性`
它是不是会在同一段时序里一起准备、一起驻留、一起释放。

### 3. 比较稳的默认规则

如果项目还没有更成熟的数据支撑，可以先用一套更保守的默认法则：

- 同一业务入口、同一更新节奏、同一生命周期的内容优先放同组
- 真正高频共享的基础资源才抽成共享组，不要为了“看起来优雅”提前全局抽公共包
- 首包、常驻包、按需下载包、活动包要有明确边界，不要混在一套模糊分组里
- `Shader`、脚本绑定和场景首屏链路上的关键依赖，要单独被显式关注，不要默认它们只是普通资源

切组这件事如果一开始就想一步到位，通常会失败。
但如果一开始没有任何规则，后面只会更难收敛。

## 三、逻辑命名要稳定，物理产物可以变化，内容身份必须独立出来

很多项目到后期开始难治理，不是因为没有版本号，而是因为把三种不同身份混在了一起：

- 逻辑名
- 发布名
- 内容身份

### 1. 逻辑命名负责稳定语义

我更建议把 bundle 或 group 的逻辑名理解成：

`这组内容在项目里的长期语义。`

比如它代表：

- 哪个场景入口
- 哪个功能域
- 哪类常驻资源
- 哪条活动线

这层命名应该尽量稳定，因为它服务的是工程协作、依赖分析、告警阅读和问题排查。

如果逻辑命名频繁跟着目录、临时活动、构建脚本细节一起波动，后面很多分析结果都会迅速失去可比性。

### 2. 物理产物名可以服务发布，但不要反过来污染逻辑语义

真正发到 CDN、磁盘缓存或远端对象存储上的文件名，常常会携带：

- 平台
- 压缩形式
- `Hash`
- 发布批次

这层是物理交付层，可以变，也经常应该变。

但比较稳的做法，是让它围绕内容身份演化，而不是让它回头定义逻辑边界。

更直白一点说：

`不要让“这次构建出来叫什么文件”反过来决定“这组内容在项目里是谁”。`

### 3. 内容身份最好单独落在不可变指纹上

项目里真正跟缓存复用、CDN 命中、回滚验证强相关的，通常都不是“人类可读版本号”，而是：

`这份内容字节上到底是不是同一份东西。`

所以比较稳的实践是：

- 逻辑名负责长期语义
- 发布快照负责一整套内容映射
- `Hash` 之类的内容指纹负责识别具体产物

只要这三层不混，很多治理动作才容易成立：

- 客户端知道自己兼容哪套内容
- 发布系统知道当前指向哪份快照
- 缓存系统知道本地持有的是不是目标内容

## 四、版本、缓存和回滚必须围绕“内容快照”设计，不要围绕口头描述设计

真正的资源治理，到这里就会进入另一条很关键的分叉：

`你们回滚和缓存，究竟围绕的是完整内容快照，还是围绕“大家都知道这是哪一版”的口头共识。`

工程上，后者通常不可靠。

### 1. 先把三层身份拆开

至少要明确区分三种东西：

1. `客户端版本`
它回答的是：这份程序是否兼容这批内容。

2. `发布版本`
它回答的是：线上当前声明的一整套资源快照是什么。

3. `内容 Hash`
它回答的是：某个具体交付物是不是那份目标内容。

只要这三层没拆开，就很容易反复出现这些现场问题：

- 客户端是新包，但资源快照还是旧的
- 发布说明切回去了，本地缓存却还命中旧内容
- 逻辑版本没变，物理内容已经换了
- CDN 路径一样，但字节已经不是同一份东西

### 2. 回滚应该尽量切快照，不要手工拼目录

比较稳的回滚动作，不应该是：

- 手动删几个远端文件
- 把几个 bundle 名字改回去
- 人工记忆“上个稳定版本是哪一组文件”

而应该是：

`把发布入口切回一份可验证的历史快照。`

这样做的价值很直接：

- 依赖链是一整套一起回去的
- 内容身份是一整套一起回去的
- 客户端和发布系统对“当前线上内容”有同一份答案

### 3. 缓存命中应该优先围绕内容身份

缓存这层最常见的误判是：

`本地有文件，所以应该可以复用。`

但工程上真正该问的是：

`本地是不是已经持有目标快照里的那份内容身份。`

所以更稳的缓存规则通常会倾向于：

- 用内容指纹决定复用，而不是只看逻辑名
- 尽量避免“同一路径覆盖不同字节内容”的发布方式
- 让小而明确的入口文件去更新快照映射，而不是让大量历史 bundle 反复被覆盖

这层一旦站稳，缓存和回滚就不再是两套零散技巧，而会自然收进同一张治理图里。

## 五、构建校验和烟测不要追求全能，先守住最值钱的高风险项

很多团队一提自动化校验，就会立刻滑进两个极端：

- 什么都不做
- 试图一次把所有问题都自动发现

这两个方向都不太现实。

更稳的做法是先守住最值钱、最容易在上线后造成高成本的问题。

### 1. 构建校验至少先覆盖这几类风险

如果只能先做一批最低成本、最高收益的校验，我会优先放这些：

- 重复资源报告
- 共享依赖异常膨胀
- 漏包、错包或依赖闭包不完整
- `missing script`、断引用、关键资源缺失
- 关键 `Shader` / `Material` / Variant 链路异常
- 包体大小、首包大小、组数量和关键入口依赖数量的回归

这批检查的共同点是：

`它们都不是“理论上可能有风险”，而是极容易在真实项目里反复出现，并且一旦漏过就很难靠人工补救。`

### 2. 烟测入口至少要覆盖真实玩家第一条链路

烟测如果只是“能下载 catalog”或者“能打开一个空场景”，价值很有限。

更稳的烟测入口，应该至少覆盖一条真实玩家链路，例如：

1. 启动后完成远端配置拉取或本地快照读取
2. 命中一组首包或首屏资源
3. 成功加载关键场景、UI 或核心 `Prefab`
4. 完成一次真实实例化和激活
5. 验证关键材质、贴图、脚本绑定和常见依赖没有断

如果项目还有灰度、活动包、地区服或热修复链路，这些分支最好也有最小化的专项烟测入口。

### 3. 不要把校验写成“看构建是否成功”

构建成功，通常只能说明：

`脚本没有在那一步直接崩掉。`

它并不说明：

- 依赖结构是对的
- 缓存映射是对的
- 运行时恢复链是对的
- 玩家第一条资源路径是对的

所以真正有价值的资源自动化，最好始终围绕一个问题设计：

`这次构建出来的交付快照，是否仍然符合我们对上线内容的结构预期。`

## 六、框架选型放在最后：先看你想把哪一层交给框架

写到这里，再回头看“到底该选什么框架”，就会稳很多。

因为框架选型真正该回答的，不是：

`谁更流行。`

而是：

`你想把哪些责任交给框架，你们团队又准备自己承担哪些责任。`

### 1. 默认优先看 Addressables

如果团队没有非常强的定制交付需求，也不打算自己长期维护一整套资源调度层，那么默认先看 `Addressables` 通常是最稳的。

它的优势通常在这些地方：

- 跟 Unity 生态的接缝更自然
- 地址、标签、依赖调度、生命周期管理有更统一的语义
- 对大多数“我不想直接裸用 AssetBundle API”的团队来说，门槛更合理

但它并不会替你解决前面那六条基线。

如果分组规则、版本映射、缓存策略和校验入口本来就混乱，换成 `Addressables` 以后，混乱只会被包进另一层接口里。

### 2. 需要更强交付控制感时，可以认真评估 YooAsset

如果项目更强调下面这些条件：

- 热更新链路更重
- 远端分发和下载控制更重
- 团队愿意显式管理资源系统工程化
- 希望对 bundle 层、补丁层、清单层有更直接的掌控感

那 `YooAsset` 往往是值得认真评估的一条路线。

它的价值通常不在“比 Addressables 更先进”，而在：

`它更明确地把资源交付当成一套工程系统来组织。`

这类路线更适合已经知道自己在资源交付上要解决什么问题的团队。

### 3. 裸 AssetBundle API 只适合两种情况

我不太建议普通团队从裸 `AssetBundle API` 直接起一整套管理层。

除非你们满足下面两种情况之一：

1. 已经有一套成熟、长期维护中的资源交付基础设施
2. 业务约束非常特殊，现成管理层确实接不住

否则团队最后往往会在这些地方重复造轮子：

- 清单映射
- 缓存复用
- 依赖调度
- 引用计数或生命周期管理
- 发布回滚
- 构建校验和烟测

这些恰恰又是最容易在长期里变成成本黑洞的部分。

### 4. 第三方产品化方案可以评估，但不要让它替你定义边界

如果团队还会看其他第三方打包框架或产品化方案，我更建议先问同一个问题：

`它能不能承载我们已经定义好的分组、身份、缓存、回滚和校验规则？`

而不是先问：

`它的编辑器面板是不是更方便。`

因为面板体验能提升效率，但不能替你决定正确的交付边界。

## 最后收成一套最小落地顺序

如果你现在就要在项目里推进资源交付治理，我会建议按下面这个顺序落：

1. 先定义分组边界：更新、共享、运行时局部性
2. 再定义三层身份：逻辑名、发布快照、内容指纹
3. 把缓存和回滚收进同一套快照规则
4. 为构建结果加上最小高价值校验
5. 为真实玩家第一条链路补上烟测
6. 最后再决定让 `Addressables`、`YooAsset` 或自研管理层承载哪一层责任

这套顺序的价值在于，它会逼着团队先解决“规则是不是清楚”，再解决“工具是不是顺手”。

而资源交付这件事里，前者几乎总比后者更值钱。
