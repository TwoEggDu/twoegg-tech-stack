## 这篇要回答什么

这篇文章主要回答五个问题：

1. `player` 默认会自动保留哪些东西。
2. `bundle` 和 `player` 在“保代码”这件事上分别负责什么。
3. 什么场景更适合用 `[Preserve]`。
4. 什么场景更适合用 `link.xml`。
5. 工程上怎样最大化裁剪收益，同时把误伤风险压到最低。

如果先把结论压成一个最短版，可以先记成这样：

- 能改写法，就优先把依赖改成构建期可见。
- 你能改源码、目标又很小的时候，优先用 `[Preserve]`。
- 不能改源码、涉及第三方库、跨程序集或远端内容时，优先用精确 `link.xml`。
- `bundle` 不会替你把托管代码“带走”；真正能不能用，最终还是取决于 `player` 里的程序集有没有把对应类型留下来。

## 先给一张决策表

如果把最常见的几类情况压成一张表，大概可以先这样判断：

| 场景 | Unity 默认能自动保多少 | 更稳的做法 |
| --- | --- | --- |
| 类型直接挂在场景、Prefab、`ScriptableObject` 或其它序列化图里 | 通常类型级更容易被保住 | 先不急着加保留，先确认真实依赖是不是成员级反射 |
| 你自己代码里的少量入口方法、回调方法、反射目标 | 取决于它是否进入已知根 | 目标很小时优先 `[Preserve]` |
| 第三方库、包代码、没法改源码的插件 | Unity 不会因为“你迟早会反射到它”就自动保 | 用精确 `link.xml` |
| 本地已知的 `AssetBundle` 所依赖的脚本类型 | 如果 `player build` 知道这些 bundle，通常可以一并纳入保留集 | 尽量同工程、同构建链构建 `player + bundle` |
| 远端未知 bundle、热更、配置驱动入口 | 构建期通常看不见 | 默认按“不会自动保”处理，用 `link.xml` / `[Preserve]` 明确声明 |
| 字符串驱动反射、运行时扫描、运行时拼泛型 | 默认高风险 | 优先改写法；保留规则只是兜底，不是首选设计 |

这张表里最关键的不是记住哪一列更“危险”，而是记住一个判断顺序：

`先问依赖是不是构建期可见，再问它该不该用显式保留。`

## 一、先分清：player 会留下什么，bundle 又不会替你留下什么

这一节如果不先讲清楚，后面 `link.xml` 和 `[Preserve]` 的决策很容易混乱。

### player 默认会留下什么

从前面几篇拆出来的结论看，`player` 默认更容易保住的，主要是这些“构建期已知根”：

- 场景里已经出现过的类型
- 序列化图里已经记录到的类型
- `UnityEvent.PersistentCall` 这类已知持久化调用
- 你显式写出来的 `link.xml`
- 你显式标出来的 `[Preserve]` 或带保留语义的入口属性
- 构建时已经纳入视野的本地 bundle 类型信息

也就是说，`player` 自动保留的不是“所有以后可能被用到的代码”，而是：

`构建期已经有证据表明必须保的那部分代码。`

所以如果某个依赖关系只有运行时配置、字符串、热更脚本或者远端 bundle 才知道，那它默认就不在这批已知根里。

### bundle 不会替你带走托管代码

这个点特别容易被误会。

普通 `AssetBundle` 更像是：

- 资源数据
- 序列化对象
- 对脚本类型的引用关系

而不是：

- 一份会独立执行的托管程序集

所以从“代码能不能活下来”这个角度看，更准确的理解是：

`bundle 需要 player 里已经存在并且没有被裁掉的脚本类型。`

换句话说，很多时候并不是：

`代码跟着 bundle 进来`

而是：

`bundle 里的资源和序列化数据，运行时要回头去 player 里找对应脚本类型。`

这就是为什么“bundle 能下载”和“bundle 依赖的代码没被裁”是两回事。

### 本地已知 bundle 和远端未知 bundle，要分开看

如果 bundle 是和 `player` 一起在同一个工程、同一条构建链里产出的，Unity 通常更容易知道：

- 这些 bundle 里出现了哪些脚本类型
- 哪些原生类、序列化类型需要额外保住

这种情况下，`player build` 可以把 bundle 相关类型一并纳入保留集。

### 这里的“保留”到底发生在什么时候

这件事最容易被误解成：

`先把 bundle 打出来，运行时再回头让 player 知道它依赖哪些代码。`

更接近实际的流程其实是：

1. `player build` 先建立一份“本次构建已知会被用到什么”的保留根集合。
2. 场景、`Resources`、序列化图、显式保留规则会先进入这份集合。
3. 如果这次构建同时知道本地 bundle 的 manifest，构建机会再把 bundle 相关类型信息并进同一份集合。
4. 后面的脚本裁剪和原生构建，再统一基于这份更大的集合继续往下走。

也就是说，真正发生的不是：

`bundle 自己把代码带进 player`

而是：

`player 在裁剪前，先把“本地已知 bundle 会用到什么”并进自己的保留根。`

### 它具体会把什么并进来

如果压成最短结论，可以先记这三类：

- bundle 里实际出现过的脚本类型
- bundle 里涉及到的原生类 / 引擎类型
- bundle 里 `SerializeReference` 这类额外序列化类型标识

这三类信息的共同点是：

`它们都还是“构建期可收集到的类型级证据”。`

所以这条链路最擅长解决的是：

- 某个 Prefab / `ScriptableObject` 进了本地 bundle
- 这个 bundle 里真的用了某个脚本类型
- player 又恰好知道这份 bundle manifest

在这种情况下，Unity 更容易在裁剪前就把对应类型留下来。

### 它最后保到哪里去了

这里的“保留”，最终落到的不是 bundle 文件本身，而是 `player` 侧：

- 托管侧：相关类型不容易在 player 程序集里被裁掉
- 原生侧：相关原生类、模块和注册信息更容易被保住

所以从工程结果上看，更准确的说法是：

`bundle 负责带资源和序列化数据，player 负责把这些资源真正需要的代码和注册留住。`

### 这条链路的边界也要看清

这条自动保留链路不是万能的，它更像“类型级保留增强”，而不是“任意运行时语义保留增强”。

换句话说，它通常不自动替你解决这些问题：

- 某个 bundle 里的脚本类型虽然活着，但你运行时还要反射它的私有方法
- 某个入口完全靠字符串方法名、名字协议或配置表驱动
- 某个远端 bundle 根本没有在 `player build` 时进入视野

这些情况仍然要回到前面的规则：

- 能改写法，先改成构建期可见
- 不能改写法，再用 `[Preserve]` 或 `link.xml`

但如果是下面这些情况，就不能再假设 Unity 会自动保：

- `player` 构建时根本不知道后面会下发哪些远端 bundle
- bundle 来自另一条工程线或另一份程序集
- 入口完全靠热更脚本、配置表、名字协议来驱动

这时最稳的默认假设就是：

`Unity 看不见它，所以也不会替你自动保。`

## 二、什么时候优先用 `[Preserve]`

先说 `[Preserve]`。

我更建议把它当成一种：

`局部、贴身、可读性高的显式保留`

它最适合下面几类场景：

- 你能改源码
- 需要保留的目标粒度很小
- 你希望“为什么要保”直接写在代码旁边
- 这个类型、方法、字段本来就是系统明确入口

例如这些情况就很适合：

- 运行时通过反射命中的一个具体方法
- 某个序列化框架需要的无参构造函数
- 某个桥接层会回调的固定入口
- 某个只会被少量动态路径命中的类型

这种做法的主要优点是：

- 离代码近，维护时更容易看懂
- 粒度可以很小，不容易一保就是一大片
- 适合“我明确知道这里就是例外入口”的场景

如果压成一个简单判断，可以写成这样：

```csharp
if (can_edit_source && target_is_small_and_local)
    use_preserve();
```

### `[Preserve]` 不是默认策略

虽然 `[Preserve]` 很实用，但我不建议把它用成“哪里炸了就贴一个”的默认手段。

因为一旦团队开始这么用，最后很容易变成：

- 到处都是 `[Preserve]`
- 但没人知道为什么保
- 也没人知道保到哪个粒度才刚好

所以更稳的原则是：

- 小而明确的例外，用 `[Preserve]`
- 能改写法的，先改写法
- 涉及第三方和远端内容的，不要硬用 `[Preserve]` 去覆盖所有问题

## 三、什么时候优先用 `link.xml`

和 `[Preserve]` 相比，`link.xml` 更适合用在：

- 你不能改源码
- 目标不在你直接维护的代码里
- 你需要跨程序集集中管理保留规则
- 你要为第三方库、远端 bundle、热更入口补保留

最典型的几类场景包括：

- 第三方插件内部靠反射、binder、serializer 才会命中的类型
- 包代码里你不能直接加 `[Preserve]` 的成员
- 远端 bundle 才会用到的脚本类型
- 某些跨模块约定入口，需要集中登记

如果把判断压成最短版，可以先记这个：

```csharp
if (!can_edit_source || target_lives_in_third_party_or_remote_content)
    use_precise_link_xml();
```

### `link.xml` 更像“集中保留清单”

`link.xml` 的优势不在于它“比 `[Preserve]` 更高级”，而在于它更适合做这些事：

- 给第三方程序集补保留
- 把一组动态例外集中放在一处
- 做跨程序集的保留策略

一个极简例子大概像这样：

```xml
<linker>
  <assembly fullname="ThirdParty.Plugin">
    <type fullname="ThirdParty.Plugin.MessageCodec" preserve="all" />
  </assembly>
  <assembly fullname="Game.Runtime">
    <type fullname="Game.Runtime.RemoteBundleEntry">
      <method name=".ctor" />
      <method name="Execute" />
    </type>
  </assembly>
</linker>
```

这里最重要的不是语法，而是两个习惯：

- 尽量精确到最小必要粒度
- 把它当成“例外清单”，而不是“整程序集先保活再说”

## 四、什么时候先改代码，不要急着补保留

很多团队一遇到 strip 问题，第一反应就是：

- 先贴 `[Preserve]`
- 先写 `link.xml`
- 先把 `Managed Stripping Level` 往回调

这些做法有时能止血，但经常治标不治本。

如果你真正的问题是：

- `Type.GetType("...")`
- `GetMethod("...")`
- 扫程序集后自动注册
- 运行时才拼泛型闭包
- 配置表里直接写 CLR 类型名

那更大的问题通常不是“少一条保留规则”，而是：

`你的真实依赖图直到运行时才出现。`

这时候更稳的方向通常是：

- 把字符串入口改成稳定业务 ID
- 把运行时扫描改成编辑器期扫描 + 生成注册表
- 把运行时拼泛型改成显式列出闭包组合
- 把反射目标集中到一处例外清单里

如果把这件事压成决策树，我建议按这个顺序问：

```csharp
if (dependency_can_become_build_time_visible)
    refactor_first();
else if (can_edit_source && target_is_small)
    use_preserve();
else
    use_precise_link_xml();
```

这条顺序的核心价值就在于：

`先减少“看不见的依赖”，再减少“过宽的保留”。`

## 五、和 AssetBundle 一起工作时，最稳的一套做法是什么

如果项目里有 `AssetBundle`、Addressables 或其它内容分发链，我更建议把下面这几条当成工程基线。

### 1. 尽量用同一个工程、同一套程序集、同一条构建链产出 player 和 bundle

这是最稳的一条。

原因不是“Unity 强制你这么做”，而是这样最容易保证：

- `player` 能知道 bundle 用到哪些类型
- 程序集名字和类型身份完全对得上
- 构建期保留视图是一致的

如果分工程、分流水线甚至分程序集去做，也不是绝对不行，但你就得自己承担更多显式保留工作。

### 2. 对本地已知 bundle，让 player build 明确把它们纳入视野

也就是说，不要让 `player` 在一种“完全不知道 bundle 内容”的前提下去裁剪。

越能让 `player build` 在构建时就知道：

- bundle manifest
- bundle 用到的脚本类型
- bundle 用到的序列化类型

你后面就越少需要靠大面积保留去兜底。

### 3. 对远端未知 bundle，默认按“Unity 看不见”处理

这一条最好直接当默认规则，而不是“也许没问题”。

只要这部分内容在 `player build` 时还不存在，或者没有进入构建期已知根，你就应该直接假设：

- 它不会被自动保
- 它需要显式保留

这里常见做法就是：

- 给远端入口类型写 `link.xml`
- 或在自己可控的类型上加 `[Preserve]`
- 或把真正的动态入口收敛成一小组稳定桥接点

## 六、如果目标是“尽量多裁，又别误伤”，我建议按这个顺序做

最后把前面几节合起来，真正能落地的一套顺序大概是这样：

1. 先在目标 `stripping level` 上做验证，不要一直在更松的配置里自我安慰。
2. 先把字符串驱动、扫描驱动、运行时拼装驱动的依赖改成构建期可见。
3. 你能改源码、目标又小的时候，用 `[Preserve]`。
4. 你改不了源码、涉及第三方、跨程序集或远端内容时，用精确 `link.xml`。
5. `player` 和本地 bundle 尽量同工程、同构建链构建。
6. 对远端未知 bundle、热更内容，默认按“不会自动保”处理。
7. 除非只是临时止血，否则尽量别用“整程序集全保活”这种粗暴做法。

这套顺序看起来没什么花活，但它背后其实只有一个非常稳定的原则：

`先让真实依赖更可见，再让显式保留更精确。`

## 这篇最该带走的三句话

如果把这篇文章最后压成三句话，我建议记住这三句：

- `player` 默认保的是构建期已知根；`bundle` 不会替你把已经裁掉的托管代码再带回来。
- 你能改源码、目标又小的时候，优先 `[Preserve]`；你改不了源码、涉及第三方、跨程序集或远端内容时，优先精确 `link.xml`。
- 真想做到“尽量多裁又不误伤”，关键不是越保越多，而是先把依赖变成构建期可见，再只给少量例外补最小粒度保留。
