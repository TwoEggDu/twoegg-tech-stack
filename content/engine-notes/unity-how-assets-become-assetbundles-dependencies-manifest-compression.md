上一篇我先把 AssetBundle 的位置掰正了：

`AssetBundle 首先不是“怎么 Load 一个包”的运行时 API 话题，而是 Unity 在构建期把资产系统输出成交付物的一种格式。`

如果这个判断站住，下一步最自然就该问：

`Unity 到底是怎么把项目里的 prefab、material、shader、texture 这些资产，编成最后磁盘上的 bundle 文件的？`

这个问题如果不拆开，后面很多运行时现象都会看起来像黑盒子：

- 为什么同一批资源切包方式一变，依赖关系就变了
- 为什么 Manifest 能决定运行时先加载谁
- 为什么有时资源“明明在 bundle 里”，运行时还是不对
- 为什么压缩方式会直接影响首载成本和随机访问
- 为什么 Shader、脚本、共享资源问题经常在构建期就已经埋下了

所以这一篇，我不打算从 `BuildPipeline.BuildAssetBundles` 的参数表写起，而是先把构建链本身拆开。

## 先给一句总判断

我会把 AssetBundle 构建压成一句话：

`Unity 构建 AssetBundle，不是把一批文件简单压成一个包，而是先按你给出的边界定义输入集合，再沿资产引用图收集依赖、生成序列化内容和资源数据，最后连同 Manifest 和压缩策略一起输出成交付物。`

这句话里最关键的不是“打包”，而是四个动作：

- `定义边界`
- `收集依赖`
- `生成可恢复内容`
- `输出交付物`

这四个动作只要有一个理解偏了，后面运行时就会开始出问题。

## 一、构建 AssetBundle 的起点，不是“文件夹”，而是“输入边界”

这是最该先钉住的一点。

### 1. Unity 不会自己替你决定交付边界

项目里有很多资产，并不代表 Unity 会自动知道：

- 哪些该在同一个 bundle
- 哪些该拆开
- 哪些该共享
- 哪些该跟主包走

它需要你先给出一层“这次构建的输入边界”。

### 2. 这层边界通常来自 AssetBundle Name 或 Build Map

无论你是通过：

- 给资产打 `AssetBundle Name`
- 还是显式传一份 build map / bundle build 列表

本质上都在做同一件事：

`先声明这次要把哪些资产视为某个 bundle 的根输入。`

所以构建的真正起点，不是“Assets 目录里有哪些文件”，而是：

`哪些资产被你声明成了这次交付物的入口。`

### 3. 这也是为什么切包策略一变，后面所有结果都可能跟着变

很多人第一次接触 AssetBundle，会低估这层。

但只要输入边界变了，后面至少都会跟着变：

- 依赖收集结果
- 重复资源分布
- Manifest 关系
- 运行时先后加载顺序
- 包体大小和首载成本

所以从工程视角看，切包策略不是“输出优化细节”，而是整个 bundle 世界的第一层语义。

## 二、Unity 真正收集的不是“你点到的文件”，而是“资产依赖图”

一旦输入边界确定，构建系统接下来干的就不是简单“拷文件”。

### 1. 它会沿着资产引用关系继续往下收集

如果你把一个 Prefab 作为 bundle 输入，Unity 真正要关心的显然不只是这个 Prefab 文件本身。

它还要继续看：

- 这个 Prefab 引了哪些 `Material`
- 这些 `Material` 又引了哪些 `Shader` 和贴图
- 脚本、动画、音频、配置、子资源依赖有没有接进来

也就是说，构建看到的不是平面文件表，而是：

`从一批根资产出发向外展开的依赖图`

### 2. 所以构建面对的对象，从一开始就是资产系统里的对象，不是原始文件路径

这和前面整条主线是对得上的。

AssetBundle 构建处理的不是：

`磁盘上某几个路径下有哪些文件`

更接近：

`这组 Unity 资产对象彼此怎样引用，哪些对象最终应当跟这次交付物一起走。`

### 3. 这也是为什么“同一份资源为什么被重复打进多个 bundle”是结构问题

如果某个资源同时被多个 bundle 的根资产依赖，而你又没有把它显式提成共享边界，那么构建系统通常就只能在多个输出里各自带上它。

所以重复资源问题，本质上不是 Unity “打错了包”，而是：

`你的输入边界和共享边界没有被显式组织出来。`

## 三、依赖收集之后，Unity 真正要生成的是什么

到了这里，很多人脑子里会自然出现一个误解：

`是不是把依赖都收齐，然后原样塞进一个压缩文件就完了？`

不是。

### 1. 它要输出的不只是源文件，而是可恢复的资产内容

前面几篇已经讲过，运行时真正恢复的不是原始文件本身，而是：

- 序列化对象图
- 资源对象数据
- 引用关系和外部依赖线索

所以构建期真正生成的，也不是简单的“文件归档”，而是：

`符合运行时恢复链要求的资产内容表达`

### 2. 对象图资产和普通资源资产都会被放进交付语义里重新组织

例如：

- `Scene / Prefab` 更偏对象图内容
- `Texture / Mesh / AudioClip` 更偏资源数据
- `Material / ScriptableObject` 更偏可引用对象和字段语义

这些资产的结构气质不同，但到了 bundle 构建阶段，Unity 要做的是把它们统一组织成：

`运行时可被解析、引用和恢复的一份交付结果`

### 3. 所以“打进 bundle”不等于“运行时已经是对象”

构建期只是把恢复所需的内容和边界准备好。

真正的对象恢复，还要等到运行时的：

- 依赖满足
- 容器打开
- 资产取出
- 反序列化和对象创建

这也是为什么前面那篇 `C02` 一直强调：

`AssetBundle 是交付容器，不是最终还原单位。`

## 四、Manifest 到底在解决什么问题

Manifest 是 AssetBundle 体系里最容易被低估的一层。

很多人会把它理解成“顺手生成的附带文件”，但它其实很关键。

### 1. 它首先在回答“构建到底产出了什么”

当一批 bundle 被构建出来，系统需要一个稳定地方去描述：

- 这次有哪些 bundle
- 它们各自的身份是什么
- 每个 bundle 的 hash / 版本线索是什么

没有这层，后面的缓存和版本治理都会失去抓手。

### 2. 它还在回答“这些 bundle 彼此依赖谁”

如果 bundle A 里引用了 bundle B 里的共享资产，那运行时想正确恢复 A，就必须知道：

`A 不是一个自足容器，它前面还有依赖要先满足。`

Manifest 的一大价值，就是把这种交付单元之间的关系显式化。

这也是为什么运行时加载 bundle 时，经常不是“拿一个就完”，而是先沿 Manifest 去满足依赖链。

### 3. 它让缓存、校验和回滚真正有对象可落

只要 bundle 有了 Manifest 层，你才能继续谈：

- 客户端本地缓存的是哪一版
- 服务端最新的是哪一版
- 哪些 bundle 变了
- 哪些不需要重新下载
- 出问题时回滚哪一个 hash

所以 Manifest 不是附带说明书，而是：

`交付单元身份表 + 依赖表`

## 五、压缩到底改变了什么

压缩这一层，很多人也会误会成“只是包更大还是更小”。

其实它直接改的是交付和运行时表现。

### 1. 它先改变磁盘和分发体积

这当然是最直观的一层。

不同压缩策略会影响：

- 输出包大小
- CDN 传输成本
- 下载体积

但这只是表面。

### 2. 它还会改变运行时读取和解压方式

例如常见的 `LZMA / LZ4` 差异，本质上不是“哪个压缩率更高”这么简单，而是：

- 什么时候解压
- 是否更适合随机访问
- 首次加载成本落在哪
- CPU 和 I/O 的账怎么分

所以压缩策略不是纯发布参数，而是：

`交付成本和运行时成本的交换点`

### 3. 它会直接影响首载体验和内存峰值

只要你把压缩看成运行时链的一部分，很多现象就会变直白：

- 下载很快，但第一次打开很慢
- 包体很小，但解压时主线程成本高
- 随机读取和局部取用不友好

这些都不是“资源系统突然变慢”，而是构建期压缩策略在运行时的延迟后果。

## 六、一个 AssetBundle 文件里大致装了什么

这一层不需要先掉进底层格式考古，但结构上值得先有一个稳定印象。

### 1. 它不是“若干原始文件的 zip”

更准确的说法是：

`它是为 Unity 运行时准备的一份内容容器。`

里面大致会包含：

- 资源对象和对象图恢复需要的序列化内容
- 资源数据块
- 目录和定位信息
- 与压缩和读取相关的组织信息

所以它天然更接近：

`运行时可解析内容容器`

而不是：

`项目原文件备份包`

### 2. 对象图内容和资源数据会在里面共同存在

如果一个 bundle 里装了 Prefab、Material、Texture、AnimationClip，它并不是在保存“Prefab 文件 + 贴图文件 + 材质文件”的概念。

它是在保存：

- 哪些可恢复对象会被运行时取出来
- 这些对象怎样引用彼此或引用外部内容
- 对应资源数据块怎样随容器一起交付

这也解释了为什么 bundle 文件结构和项目目录结构并不一一对应。

### 3. 运行时恢复时看到的是“bundle 内容”，不是“项目原始文件世界”

一旦内容被构建成 AssetBundle，运行时面对的已经不再是编辑器里的项目文件树，而是：

`被重组过的一份交付内容世界`

这也是为什么很多编辑器里看似理所当然的关系，到 bundle 路径上会显得更敏感。

## 七、为什么很多运行时问题，其实是构建期切分策略的延迟后果

这篇最该钉住的结论之一就是这个。

### 1. 运行时看到的是结果，不是切分决策过程

当运行时出现：

- 缺依赖
- 重复下载
- 首次卡顿
- Shader 或脚本问题

它显形的位置通常在运行时。

但根因很多时候更早，在于：

- 根输入怎么定义
- 共享资源怎么切
- 哪些东西被打到了哪里
- Manifest 怎么记录
- 压缩怎么选

### 2. 所以很多“加载问题”其实是“构建问题”

这也是为什么我前面一直不想把 AssetBundle 写成 API 教程。

如果只盯着运行时 `LoadAsset`，你会漏掉最关键的一层：

`加载路径正在消费的，其实是构建期已经被决定好的交付物结构。`

### 3. 这也是 AssetBundle 最像工程治理问题的地方

它真正难的不是调用某个函数，而是：

- 构建边界怎么定
- 共享依赖怎么治理
- 输出身份怎么稳定
- 版本和缓存怎么对齐

一旦这几件事没定稳，后面的运行时就只能在坏边界上挣扎。

## 八、工程里该怎么用这条构建链排查问题

以后你在项目里看到某个 AssetBundle 问题，我建议先别问：

`是不是运行时加载顺序写错了？`

而先问下面这些问题。

### 1. 这次构建的输入边界到底是什么

先明确根资产是谁，不要用“应该会被打进去吧”这种模糊语气。

### 2. 依赖是怎样被收集和切分的

重点看：

- 哪些依赖跟着根资产进了同一个 bundle
- 哪些被切成共享 bundle
- 哪些被重复打包了

### 3. Manifest 到底记录了什么依赖关系

如果 Manifest 关系没看清，运行时先加载谁、为什么缺依赖，都会变得模糊。

### 4. 压缩策略是不是在把运行时成本推迟爆出来

特别是首载慢、随机访问差、内存峰值奇怪这类问题，不要只怪加载 API。

### 5. 这到底是 bundle 容器层问题，还是 bundle 里具体资产类型的问题

比如：

- 配置资产更像引用和类型边界问题
- Material 更像 shader 边界问题
- Prefab 更像模板恢复和脚本身份问题

这一步如果不分开，所有问题都会被压成“bundle 不对”。

## 总结

最后把这篇压成三句话：

1. Unity 构建 AssetBundle，不是把几份文件简单压到一起，而是先按输入边界定义交付根，再沿资产依赖图收集内容，组织成运行时可恢复的交付物。
2. Manifest 解决的是交付单元身份和依赖关系，压缩解决的是分发成本和运行时成本怎么交换，它们都不是附属细节。
3. 很多看起来像运行时加载异常的问题，根因其实早在构建期的切分策略、共享边界、Manifest 关系和压缩选择上就已经决定了。
