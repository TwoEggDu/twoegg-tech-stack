上一篇我已经把 `ShaderVariantCollection` 的定位压清了：

`SVC` 最重要的价值，不是神奇地修复 shader 问题，而是把项目真正关心的 shader variant 显式沉淀成资产，再让这些路径可以被预热、被治理、被回归。

如果这个判断站住，下一步最自然的问题就不是：

`SVC 是什么？`

而是：

`项目里到底该怎么把它做出来、管起来，并且不让它很快失真？`

这也是很多团队真正卡住的地方。

因为知道 `SVC` 有用，不等于真的会落地。  
项目一旦上了 `AssetBundle`、`Addressables`、热更内容、运行时 keyword 切换和多平台构建，`SVC` 很快就会遇到三类现实问题：

- 从哪里收集，才不会漏关键路径
- 怎么分组，才不会变成一坨巨大的“全项目 SVC”
- 怎么接回归，才不会越积越旧、最后没人信

所以这篇我就只回答这三件事。

## 先给一句总判断

如果把整件事压成一句话，我会这样说：

`SVC 的正确落地方式，不是一次性导出一份大清单扔进项目，而是围绕”真实入口、真实内容边界、真实回归路径”持续收集、分组和校验。`

这句话里最关键的是三个”真实”：

- `真实入口`
- `真实内容边界`
- `真实回归路径`

只要这三个点没立住，`SVC` 很容易变成两种没有长期价值的东西：

- 一个只在某次排障里短暂有用的临时资产
- 一份越积越大、没人敢删、也没人知道它还对不对的历史包袱

## 零、先把 SVC 在构建流程里的位置说清楚

很多团队在落地 SVC 前，对它在构建期的行为有一个关键误解：

`SVC 是不是一种”覆盖材质收集”的机制？`

正确的理解是：

**SVC 作为构建资产（放在 AssetBundle / Resources / 项目中被构建收集到）时**，Unity 在构建期的 `ComputeBuildUsageTagOnObjects` 阶段会把 SVC 里登记的 keyword 组合**并入** `usedKeywords`，与材质收集的 keyword 组合一起参与枚举。

这意味着：

- SVC 不会覆盖材质贡献的 `usedKeywords`，两者是合并关系
- 如果某个 keyword 组合只在 SVC 里登记、没有材质使用，它依然会被枚举进构建
- 这就是为什么”加了 SVC 就能修复缺变体”：材质不在构建的 `allObjects` 集合里时（比如热更包），SVC 可以替代材质把关键 keyword 组合显式注入构建

**SVC 作为 Preloaded Shaders（在 Graphics Settings - Preloaded Shaders 里挂载）时，行为完全不同**：它会在 ShaderWriter 枚举阶段通过 `g_VariantCollectionOverride` 完全替代 `usedKeywords`，只有 SVC 里登记的变体才会被生成。这是一种激进的全量覆盖，而不是补充。

这两种模式的差别后面有专文讲，这里先记住：

`作为普通构建资产的 SVC，解决的是”显式补充 keyword 组合进入枚举”，而不是”替代整个变体生成决策”。`

---

## 一、先收集：不要只靠一次导出，也不要只靠手填

`SVC` 的第一步不是创建资产文件，而是决定：

`你准备从哪些证据里收集真正关心的 variant。`

这一步如果太懒，后面一定会漏。
这一步如果太贪，后面一定会膨胀。

所以更稳的做法不是只信一种来源，而是把收集来源分层。

## 1. 第一层来源：真实运行路径

这是最值钱的一层。

更适合被优先收集的，通常不是“项目理论上可能用到哪些 variant”，而是：

`真实入口、真实内容、真实设备路径下，真的走到过哪些 variant。`

这类来源通常包括：

- 主场景和首屏入口
- 核心战斗或主循环场景
- 高频活动入口
- 热更内容的首批关键资源
- 真机上真实触发过的材质和特效链

这层最重要，因为它最接近玩家真的会看到的世界。

## 2. 第二层来源：构建期记录

如果项目已经接了 `IPreprocessShaders` 或类似构建期 variant 审计，那么它会是非常有价值的第二来源。

这层的意义不是直接替代运行时收集，而是帮你回答：

- 这次构建到底保留了哪些 variant
- 某条路径是没被运行时走到，还是构建期根本没生成
- 某个 `SVC` 想保护的 variant，在目标构建里到底还在不在

所以构建期记录更像“编译世界的证据”，和运行时命中证据是互补的。

## 3. 第三层来源：显式人工补充

有一部分路径，天然不容易被自动收集完整，必须人工补：

- 某些只在运营活动里临时出现的 keyword 组合
- 某些需要特定战斗条件、天气、时间段才触发的效果
- 某些运行时脚本显式切 keyword 的路径
- 某些平台特有的渲染路径

这类路径如果坚持“等真实跑到再收集”，通常会太晚。

所以更稳的策略是：

`自动收集负责覆盖高频真实面，人工补充负责兜住高风险稀有面。`

## 4. 不建议只靠一种收集方式

只靠一次自动导出，问题通常是：

- 你不知道它采样了哪些场景
- 你不知道它遗漏了哪些动态路径
- 你也不知道它有没有把无关路径一并带大

只靠人工手填，问题则更明显：

- 很难全
- 很快过时
- 没法随着项目演化稳定更新

所以更稳的工程实践通常是：

`真实运行路径 + 构建期记录 + 人工高风险补充`

这三层一起收。

## 二、再分组：不要做“一个全项目大 SVC”

很多团队第一次认真做 `SVC`，最容易犯的错误就是：

`既然这东西重要，那就把全项目的 variant 都收成一个大 collection。`

这在短期排障里也许有效，但长期通常不稳。

原因很简单。

## 1. 一个全项目大 SVC 会同时失去三种边界

它会把这些边界全部抹平：

- 入口边界
- 内容边界
- 生命周期边界

结果就是：

- 你不知道哪些 variant 属于首屏关键路径
- 你不知道哪些只属于某个活动或某个 DLC
- 你也不知道哪些应该首载预热，哪些应该按需准备

而 `SVC` 一旦失去这些边界，就只剩下一个结果：

`越来越大，越来越难信。`

## 2. 更稳的默认分组法：按入口和生命周期分

如果项目还没有更成熟的数据系统，我更建议先按下面两层去分：

### 1. 按入口分

例如：

- 启动与首屏一组
- 主场景一组
- 战斗入口一组
- 活动入口一组

因为 shader 卡顿和错效果，很多时候都是“第一次走到某个入口”才暴露出来的。

按入口分，最接近真实首载路径。

### 2. 按生命周期分

例如：

- 常驻集合
- 跟场景走的集合
- 跟活动内容走的集合
- 跟热更补丁走的集合

因为 `SVC` 不只是名单，它往往还会影响预热时机和加载边界。

如果生命周期没分开，后面很容易变成：

- 该常驻的不敢常驻
- 不该首载的却被首载一起预热

## 3. 场景大于内容域时，优先按场景入口分

有些项目会纠结：

`到底按 shader 类型分，还是按场景分，还是按业务模块分？`

如果没有特别强的特殊约束，我更推荐优先按：

`真实入口 / 场景入口`

而不是按 shader 名字、材质类型或渲染技术域分。

原因是 `SVC` 最终服务的，是：

- 哪条路径先被玩家看到
- 哪批 variant 需要先准备
- 哪个入口回归必须覆盖

这些判断都更接近入口，而不是技术分类。

## 4. 少量 truly global 的基础路径才适合单独抽“全局 SVC”

当然不是说永远不能有公共集合。

更稳的做法是：

- 只有那些几乎所有入口都会稳定命中的基础路径
- 而且确实应该由 Player 或常驻初始化统一准备

才单独抽成一个小而明确的全局 `SVC`。

重点是：

`小而明确。`

不是把“懒得分类”的内容都塞进去。

## 三、再决定预热：SVC 的加载时机必须和入口时机对齐

很多团队在 `SVC` 上的第二个常见误用是：

`只管把 collection 做出来，不管它什么时候被加载和 WarmUp。`

这会直接让分组价值大幅下降。

因为 `SVC` 的工程意义，很大一部分就在于：

`它应该在哪个时机，为哪个入口的第一次命中做准备。`

## 1. 首屏关键路径的 SVC 要早于首屏真正渲染

这类通常包括：

- 启动 Logo 后马上会出现的 UI 材质
- 首屏角色、主场景天空盒、关键后处理
- 第一个战斗场景马上会看到的角色和特效材质

如果这些集合直到场景已经开始真实渲染后才 WarmUp，很多首载卡顿已经来不及避免了。

## 2. 跟内容一起下载的 SVC，要跟内容边界一起走

如果项目本身是 `AssetBundle` / `Addressables` / 热更内容交付，那么 `SVC` 更稳的归属通常不是全都进主包，而是：

`跟它保护的那批内容边界一起走。`

例如：

- 活动资源包带自己的活动 `SVC`
- 某个 DLC 入口带自己的 `SVC`
- 热更场景跟自己的 shader 集合一起更新

这样它才不会在交付边界上重新失真。

## 3. WarmUp 时机要和“第一次真的会命中”对齐

这里最重要的不是“尽量早”，而是：

`刚好早于真实第一次命中。`

太晚了，效果来不及。  
太早了，则可能带来不必要的启动时间、内存和 GPU 准备成本。

所以 `SVC` 的 WarmUp，本质上也应该被视为入口调度问题，而不是一个孤立 API 调用。

## 四、最后接回归：没有回归，SVC 很快就会失真

这一步其实最重要。

因为 `SVC` 最大的敌人不是“不会建”，而是：

`建出来以后，半年后谁也不知道它还对不对。`

如果没有回归，它很快就会变成一份历史文件，而不是一份可被信任的名单。

## 1. 回归先分三类，不要只做一种

更稳的回归，至少应该分成三层：

### 1. 构建回归

回答的是：

- 这次构建保留了哪些 variant
- 某些关键 `SVC` 里的路径是不是被 strip 掉了
- 构建结果和上个稳定版本相比，哪些关键集合发生了异常变化

这层更像“编译世界的回归”。

### 2. 运行时缺口回归

回答的是：

- 当前真机或目标平台运行时还缺哪些 variant
- 哪些 `SVC` 实际没有覆盖首载路径
- 哪些路径虽然在 `SVC` 里，但首次命中仍然冷

这层更像“真实命中世界的回归”。

### 3. 入口烟测回归

回答的是：

- 主场景首载是不是还稳
- 核心战斗入口第一次渲染有没有卡顿或错效果
- 活动入口切进来时关键材质是否正常

这层更像“玩家体验世界的回归”。

三层缺一不可。

## 2. 不要只回归“collection 文件还在不在”

`SVC` 回归最没价值的一种做法，就是只检查：

- 这个 `.shadervariants` 文件还在
- 引用没丢
- 程序调用了 `WarmUp()`

这些最多只能说明：

`形式上还存在。`

它并不能说明：

- 里面那批 variant 还是对的
- 它覆盖的还是当前真实入口
- 构建期没有把关键路径裁掉
- 真机上第一次命中已经不再卡

所以 `SVC` 回归必须围绕真实路径，而不是只围绕资产文件本身。

## 3. 最值得固定回归的，是“高价值小集合”

不要试图一开始就对全项目所有 `SVC` 做最重的真机回归。

更稳的策略通常是：

- 先定义几组高价值集合
- 对它们做最硬的回归

例如：

- 启动首屏集合
- 主场景集合
- 主战斗集合
- 当前线上活动集合

这些集合最值得被长期盯住，因为它们最直接对应玩家第一感知。

## 4. 允许集合演化，但要让变化可解释

`SVC` 不是一旦建好就不再变化的文物。

项目内容在变，材质在变，keyword 使用面在变，入口路径也在变。

所以更合理的目标不是：

`让 SVC 永远不变。`

而是：

`让每一次变化都能被解释。`

例如：

- 为什么这次多了这批 variant
- 为什么这次删了那批旧路径
- 为什么活动集合拆出来了
- 为什么某个全局集合变小了

只要变化可解释，`SVC` 就还是治理资产。  
一旦变化不可解释，它就会重新退回成历史包袱。

## 五、可以直接落地的一套最小实践

如果你现在就要开始把 `SVC` 真正接进项目，我会建议先用一套更保守的最小法：

1. 先定义 3-4 个高价值入口
2. 针对这些入口做真实运行路径收集
3. 用构建期记录补一层编译证据
4. 允许人工补少量高风险动态路径
5. 按入口和生命周期拆成小集合，而不是一个大集合
6. 把首屏、主场景、主战斗三组集合接入固定烟测
7. 每次大版本只看关键集合 diff，不要一开始就盯全量

这套做法的价值在于：

- 收集面不会太虚
- 资产规模不会一开始就失控
- 回归也能先从最值钱的入口做起

## 最后收成一句话

如果把这篇最后再压回一句话，我会这样说：

`SVC 真正的工程落地，不是“导出一份 variant 文件”这么简单，而是围绕真实入口去收集，围绕内容和生命周期去分组，再围绕关键路径去回归；只有这样，它才会一直是一份可信的 shader 治理资产。`
