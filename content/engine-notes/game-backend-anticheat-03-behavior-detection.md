## 当外挂在协议层是合法的

有一类外挂特别难处理：它们发出的每一条数据包都是协议合法的，执行的每一个操作都在游戏规则允许的范围内。Aimbot 就是最典型的例子——玩家确实按下了射击键，准心确实对准了头部，服务端收到一个完全正常的射击请求。

单看任何一条指令，没有任何异常。但如果你把这个玩家过去 100 场比赛的数据放在一起看，会发现他在 100 米外对移动目标的爆头率是 94%，而最顶尖的职业选手平均是 40%。

**行为异常检测的核心思路：单条指令无法判断，统计分布可以。**

---

## 统计异常检测的基本框架

### 正态分布假设

对于大多数玩家行为指标，其分布近似于正态分布（或对数正态分布）。以命中率为例：

- 新手玩家：5%-15%
- 普通玩家：20%-40%
- 高水平玩家：40%-60%
- 职业选手：55%-75%（特定武器/距离）
- Aimbot：80%-99%

建立正态分布模型后，可以计算某个玩家在该指标上处于第几个标准差之外：

```
z = (x - μ) / σ

z > 3  → 超过 99.7% 的玩家，需要关注
z > 4  → 超过 99.994% 的玩家，高度可疑
z > 5  → 几乎不可能是正常分布内的值
```

**置信区间的实际含义**：如果 z > 3 就封号，意味着有 0.3% 的正常玩家会被误封。对于百万级用户的游戏，这是 3000 个无辜玩家。

### 样本量的重要性

统计判断需要足够的样本量。单场比赛的命中率波动很大（运气因素），但 1000 场比赛的平均值就稳定得多。

实践上，通常需要：
- 至少 200-500 次相关操作才开始纳入统计
- 跨时间段的数据（防止短时间突刺带来的误判）
- 按游戏场景分层（不同地图、武器、对手水平）

## 命中率异常检测的具体实现

### 分层命中率模型

不能用一个全局命中率阈值判断所有玩家，需要按维度分层：

| 维度 | 原因 |
|---|---|
| 武器类型 | 霰弹枪和狙击步枪的正常命中率差异巨大 |
| 目标距离 | 近战命中率天然高于远距离 |
| 目标移动状态 | 静止目标命中率高于移动目标 |
| 玩家等级/段位 | 高段位玩家基准命中率更高 |
| 头部命中比例 | 爆头率是更敏感的 Aimbot 指标 |

分层之后，对比同层玩家的分布，异常才有意义。

### 反应时间检测

Aimbot 和 Triggerbot 的另一个特征是反应时间分布异常：

- 正常人类反应时间：150ms-300ms，分布有自然波动
- Triggerbot：5ms-20ms，且方差极小（几乎固定延迟）
- 高级 Triggerbot：会人为加入随机延迟来模拟人类，但方差和分布形态仍然与真实人类不同

检测方法：计算"敌人进入准心到射击指令发出"的时间分布，拟合分布模型，检测其是否符合人类反应时间的生理学特征（下限约束、正态性检验）。

### 瞄准轨迹分析

Aimbot 的鼠标移动轨迹有机械特征：
- 直线插值向目标移动（真实人类会有微小过冲和修正）
- 目标切换时瞬间角速度过大（超过人类手腕运动极限）
- 准心"吸附"特征（突然的方向修正指向敌人头部）

这类特征的检测需要客户端上报鼠标移动数据，服务端进行模式分析。客户端数据的可信度是个问题，但配合其他信号可以提高整体置信度。

## 移动速度异常检测

速度外挂（Speedhack）和传送外挂是服务端最容易检测的类型之一。

### 基本速度验证

服务端记录玩家上一帧位置和时间戳，收到新的位置数据时计算：

```
实际位移 = distance(新坐标, 旧坐标)
允许位移 = max_speed × elapsed_time × tolerance_factor
if 实际位移 > 允许位移:
    拒绝本次移动，发送位置纠正
```

`tolerance_factor` 是容错系数，通常取 1.1-1.3，用于吸收网络延迟、Tick 率误差带来的合理波动。

### 延迟补偿场景下的复杂性

延迟补偿（Lag Compensation）允许服务端在验证射击时"回溯"到玩家发送指令时的游戏状态。这带来了一个问题：如何在考虑延迟的情况下验证位移合法性？

实践上的处理方式：
- 对每个连接维护一个位置历史缓冲区（通常保存最近 200-500ms）
- 按玩家上报的 Timestamp 在历史中找到对应状态
- 基于历史状态做合法性验证，而非当前服务端时间

高延迟玩家（200ms+）会有更大的合法位移容差，这本身也是作弊者的攻击面——伪造高延迟状态来为异常位移争取容差。对 RTT 异常波动的检测是附加的防护层。

## 资源获取速率异常

MMO、刷怪类游戏中，机器人的主要检测维度是资源/经验的获取速率：

### 速率基准建立

统计真实玩家在相同场景下的资源获取速率分布：
- 平均每小时金币获取量（按区域/职业分层）
- 击杀速率（每分钟击杀数）
- 操作间隔分布（点击时序）

机器人的典型特征：
- **速率异常高**：24小时不间断运行，速率超出人类可持续时间
- **速率异常稳定**：人类玩家会有注意力波动，速率有自然波动；机器人速率方差极小
- **路径重复性**：在相同路径上来回刷怪，GPS 轨迹几乎完全重叠
- **无休眠**：正常玩家会离开游戏或挂机，机器人在线时长分布异常

### 操作时序分析

机器人的键鼠操作时序具有机械特征：
- 操作间隔为固定值或固定值加小随机数
- 操作序列有明显的周期性
- 对"陷阱"触发无反应（正常玩家会对突然事件做出非脚本化反应）

## 误报率：行为检测的核心工程挑战

所有行为检测都面临同一个根本问题：**正常玩家的行为分布与作弊玩家有重叠区域**。

```
          正常玩家分布          作弊玩家分布
               ██████
            ████████████         ████
          ████████████████     ████████
        ████████████████████  ████████████
─────────────────────────────────────────────→ 命中率
0%      20%     40%     60%     80%    100%
                         ↑
                       阈值？
```

无论阈值放在哪里，都存在：
- **假阳性（False Positive）**：正常玩家被误判为作弊
- **假阴性（False Negative）**：作弊玩家漏过检测

对于游戏运营，这两种错误的代价不对称：**误封一个正常玩家的声誉损失，远大于放过一个作弊者**。因此行为检测的阈值通常被设置得相当保守，单独的行为检测很少直接触发封禁，而是作为人工审核的触发条件或多信号融合模型的输入。

### 多信号融合

实践中的做法是把多个弱信号融合：

```
总分 = w1×命中率得分 + w2×反应时间得分 + w3×移动异常得分 
       + w4×客户端检测信号 + w5×举报信号

总分 > 高阈值 → 自动封禁（极高置信度）
总分 > 中阈值 → 加入人工审核队列
总分 > 低阈值 → 加强监控（增加采样频率）
```

## 工程边界

- 行为检测数据量大，需要专门的流式处理管道（Kafka + 实时聚合），不能在游戏服务器主循环里做
- 模型需要持续更新：外挂作者会分析被封号的玩家的行为特征，针对性地调整外挂参数以规避统计检测
- 高段位玩家的合法行为与外挂的重叠区间天然更大，对顶级玩家的误封风险更高
- 举报数据可以辅助提高优先级，但举报本身也可能被滥用（针对正常玩家的恶意举报）

---

## 结论

行为异常检测的本质是在正常与异常的统计分布重叠区间内做决策，单一信号无法可靠判断，多维度融合配合保守阈值和人工复核才是可落地的工程方案。
