上一篇我主要在讲：

`为什么 Unity 会有 Shader Variants，以及为什么它最后很容易变成工程治理问题。`

但真到项目里，大家更常卡住的其实不是原理，而是下面这三句很具体的话：

- 我怎么知道项目到底用了哪些 `Shader Variant`
- 我怎么知道运行时丢了哪些 variant，或者丢了哪些 `SVC`
- 我怎么把不需要的 shader variant 剔掉，又不把线上内容删坏

这三个问题经常会被混在一起讲，但它们其实是三个不同层次的问题。

如果不先拆开，后面就很容易乱：

- 你以为自己在查“项目用了哪些 variant”，结果只是在数 `.shadervariants` 文件
- 你以为自己在查“运行时缺了哪些”，结果看到的只是编辑器里没问题
- 你以为自己在做“variant 裁剪”，结果实际只是在删 `SVC`

所以这篇我想把边界先摆正，再讲方法。

## 一、先把三个问题分开

### 1. “项目用了哪些 Shader Variant”是在问什么

这句话真正想问的，通常不是“项目目录里有多少可能的 variant”，而是：

`在真实构建和真实内容下，哪些 shader keyword 组合、pass、平台路径真的会被项目用到。`

它更接近“真实使用面”。

### 2. “运行时丢了哪些”是在问什么

这句话真正想查的是：

- 某些运行时会走到的 variant 根本没被编进包
- 或者 variant 编进去了，但没有被提前预热
- 或者某份 `Shader Variant Collection` 没被带上、没被加载、没被 `WarmUp`

它更接近“线上缺口”。

### 3. “怎么剔除不需要的 Shader Variant”是在问什么

这句话问的是：

`在不影响真实运行路径的前提下，怎样把构建时那些根本不会用到的 variant 排除掉。`

它更接近“构建治理”。

这三件事的关系可以粗略理解成：

- 先知道项目真的用了什么
- 再知道现在到底缺了什么
- 最后才有资格去删不需要的东西

顺序反了，通常就会出事故。

## 二、怎么知道项目到底用了哪些 Shader Variant

这个问题最容易掉进一个坑：

`不要从“可能性”开始，要从“真实运行证据”开始。`

因为 shader variant 的理论组合永远比项目真实使用面大得多。

### 1. 先看材质、渲染配置和平台边界

一个 variant 会不会真的被项目用到，通常取决于几层东西一起作用：

- Shader 本身有哪些 keyword
- 材质到底启用了哪些 keyword 组合
- 场景和资源里到底引用了哪些 shader / material
- 当前 Render Pipeline 开了哪些 feature
- 平台、图形 API、质量档位会走哪些分支

所以第一步不是看 SVC，而是先认清：

`真实使用的 variant，首先来自真实资源和真实配置。`

也就是说，如果一个 shader 有十个 keyword，不代表项目真的用了所有组合。很多组合只是理论上能编，但线上内容根本不会走到。

### 2. 再看构建日志和编译报告

如果项目已经有正式构建流程，那么最值得看的不是编辑器里“看起来会用到什么”，而是：

- Build 过程里的 shader variant 统计
- stripping 前后的数量变化
- 哪些 shader / pass / keyword 组合保留了
- 哪些被 strip 掉了

这一步的价值在于，它开始把“理论可能”缩成“本次构建实际保留”。

但要注意：

`构建里保留了，不代表运行时一定会用到；构建里没保留，则运行时一定不可能用到。`

这是后面排查缺失问题的重要前提。

### 3. 用 `IPreprocessShaders` 把编译经过的 variant 打出来

如果团队要认真治理 variant，最稳的办法通常不是肉眼看日志，而是自己在构建阶段接住这条链。

最常见的做法是实现：

- `IPreprocessShaders`

在 `OnProcessShader` 里拿到：

- `Shader`
- `ShaderSnippetData`
- `IList<ShaderCompilerData>`

这里的 `ShaderCompilerData` 本质上就是“这次构建正在处理的 variant 数据”。

这样你就可以把信息打出来，或者落成报告，比如：

- 是哪个 shader
- 是哪个 pass
- 是哪个平台
- 带了哪些 keyword
- 最终保留了多少条

从工程治理角度看，这一步非常值钱，因为它第一次把“variant 问题”从体感问题变成了可统计问题。

### 4. 再把运行时命中证据补上

仅靠构建报告，你知道的是“编进去了哪些”。

但如果你想知道“项目实际用了哪些”，还差一步：

`运行时真正命中过哪些路径。`

这一层常见的证据通常来自：

- 关键场景首次进入时是否触发 shader warmup
- 某些特效第一次出现时是否卡顿
- 某类材质第一次出现时是否有明显加载或准备成本
- Frame Debugger / 渲染调试里实际出现了哪些 pass 和 shader

这一步不一定能把每一个 variant 精准枚举出来，但它能帮你把注意力从“全宇宙可能性”拉回“真实高风险路径”。

### 5. SVC 只能回答一部分问题

`Shader Variant Collection` 当然也要看，但它回答的问题只是：

- 你预期哪些 variant 值得提前准备
- 你为哪些路径做了显式预热

它回答不了：

- 项目真实会不会命中全部这些 variant
- 还有没有别的 variant 没进 SVC 但运行时照样会用到
- 当前构建是不是已经把某些关键 variant strip 掉了

所以如果你的问题是“项目用了哪些 shader variant”，那 `SVC` 只是证据之一，绝对不是全部答案。

## 三、怎么知道运行时丢了哪些 variant，或者丢了哪些 SVC

这个问题本质上是在找“构建结果”和“运行时需求”之间的缺口。

我一般会先把“丢了哪些”拆成两类。

### 1. 第一类：variant 根本没编进去

这种情况最典型的表现是：

- 编辑器里正常
- 真机构建异常
- 某些材质在特定平台上变粉
- 某些 pass、阴影、后处理、光照路径直接失效

这类问题通常说明：

`运行时需要的 variant，在构建时就没被保留下来。`

它常见的来源包括：

- stripping 过头
- 资源引用链没覆盖到
- 平台切换后 variant 集合变化
- URP / HDRP 某些 feature 配置导致变体没生成
- AssetBundle 构建链和 Player 构建链收集条件不一致

要查这种问题，最有效的不是盯运行时，而是回去对比：

- 运行时出问题的 shader / pass / keyword 组合
- 构建报告里它到底有没有出现

如果构建里根本没有它，那你就不是在查“运行时为什么没加载”，而是在查“构建为什么没保留”。

### 2. 第二类：variant 编进去了，但首次命中时没准备好

这种情况的典型现象不是功能错误，而是：

- 首次出现某个特效时卡一下
- 首次进入某类场景时卡一下
- 首次切换某种材质或画质路径时卡一下

这时候常见的问题是：

- variant 在包里
- 但没有被预热
- 或者本来依赖某份 SVC 来预热，但那份 SVC 没带上、没加载、没 `WarmUp`

这种问题的边界要讲清楚：

`缺 SVC，不一定等于功能缺失；很多时候它表现为卡顿，而不是画错。`

### 3. 怎么判断是“缺 variant”还是“缺 SVC”

可以用一个很朴素的判断法：

- 如果表现是粉材质、pass 丢失、功能直接不对，优先怀疑缺 variant
- 如果表现是首次卡顿、之后正常，优先怀疑缺 SVC 或缺预热

当然这不是百分之百，但作为工程排查的第一刀很有用。

### 4. 先盘 SVC 的静态引用链

如果怀疑丢的是 SVC，而不是 variant 本身，那么先查：

- `Graphics Settings -> Preloaded Shaders`
- 代码里有没有 `ShaderVariantCollection`
- 有没有 `WarmUp`
- 有没有 `Resources.Load` / Addressables / AssetBundle 加载 SVC
- 哪些配置、表、ScriptableObject 间接引用了 `.shadervariants`

你真正要确认的是：

- 这份 SVC 有没有进包
- 有没有在目标平台那条链路里被带上
- 有没有在正确时机被加载
- 加载后有没有真的执行预热

### 5. 再拿运行时问题回推

如果静态链上看着都在，但运行时还是有问题，就要反过来问：

- 出问题的是哪个场景
- 第一次出现的是哪类材质或特效
- 是哪个平台、哪个图形 API、哪个质量档
- 这个路径是否本来就应该被那份 SVC 覆盖

这一步很重要，因为很多团队的问题不是“没有 SVC”，而是：

- SVC 存在
- 但收录的并不是现在真实会走到的 variant

也就是说：

`SVC 不是挂上就完事，它也会过时。`

## 四、怎么剔除不需要的 Shader Variant

这一步最容易做错，因为很多人一上来就想“我把不用的 `.shadervariants` 文件删了”。

但如果目标是减少真正的 shader variant 负担，那重点通常不在删 SVC，而在更前面的几层。

### 1. 第一层：先减少根源上的组合爆炸

真正放大 variant 数量的，通常是这些东西：

- `multi_compile` 用得太宽
- 本来应该是运行时参数的东西，被做成了 keyword
- 一个 shader 里堆了太多彼此并不独立的开关
- Render Pipeline feature 开得太多
- 平台、质量档、图形 API 组合太散

所以最值钱的减法往往不是“后面 strip 掉”，而是：

`前面就不要生成那么多没意义的组合。`

这也是为什么很多项目 variant 永远减不干净，因为它在源头上就一直往外冒。

### 2. 第二层：把 `multi_compile` 和 `shader_feature` 用对

这是 Unity 里非常典型的一道分水岭。

粗略讲：

- `multi_compile` 更偏“无论用没用到，都倾向保留这些组合”
- `shader_feature` 更偏“只保留真正被使用到的组合”

所以如果某个开关只是项目里少数材质会用，而且没有必要把所有组合都长期保留，那它通常更适合做成 `shader_feature`，而不是 `multi_compile`。

当然这里不能机械替换，因为还要考虑：

- 运行时会不会动态开 keyword
- AssetBundle 里的资源是否会晚于主包出现
- 某些间接引用路径会不会导致“看起来没用，实际上线上会用”

但大方向是明确的：

`很多不需要的 variant，根本不是后面 strip 掉的，而是前面声明方式就写错了。`

### 3. 第三层：用 stripping 规则做构建裁剪

当源头收敛过后，才轮到构建阶段的裁剪。

这一层通常会围绕：

- 平台
- Pass
- Render Pipeline feature
- 质量档
- 灯光 / 阴影 / 后处理相关开关

来做有边界的剔除。

更务实的思路是：

- 先根据真实项目配置确定永远不会发生的组合
- 再在构建阶段把这些组合排除掉
- 最后用真实构建回归去验证

而不是一开始就靠直觉“这个应该没人用”。

### 4. 第四层：SVC 的减法只能减少预热和管理负担

删 `SVC` 的价值当然有，但它主要减少的是：

- 启动或切场景时的预热成本
- 额外的 SVC 资源管理负担
- 某些重复收录造成的冗余

它不天然等于：

- variant 总量立刻明显下降
- 构建编译压力一定明显下降

所以如果你真正目标是“把不需要的 shader variant 剔掉”，不要把主要火力放在删 SVC 上。

删 SVC 更像是在处理：

`哪些路径值得预热，哪些历史预热名单已经过时。`

而 variant 治理本身，还得回到 shader、keyword、pipeline 和 stripping 规则。

## 五、一个更稳的执行顺序

如果让我把这件事压成一个更实操的流程，我会这样做。

### 第一步：拉“构建实际保留”的报告

先通过构建日志或 `IPreprocessShaders` 拿到：

- 哪些 shader 在编
- 哪些 pass 在编
- 哪些 keyword 组合在保留
- 每类数量有多少

先把“实际保留面”看清。

### 第二步：补“运行时真实需求”的证据

重点关注：

- 主场景
- 战斗场景
- 高频特效
- 首次进入路径
- 平台差异路径

先把“真实需求面”看清。

### 第三步：对齐缺口

把运行时异常路径和构建结果对齐，判断到底是：

- 没编进去
- 编进去了但没预热
- 预热了但预热名单已经过时

### 第四步：先改源头，再改构建，再改 SVC

优先级我会这样排：

1. 先改 shader keyword 设计
2. 再改 pipeline / feature 配置
3. 再改 stripping 规则
4. 最后才整理 SVC

因为越靠前，收益越根本。

### 第五步：每次都用真实构建和真机回归

不要只在编辑器里验证。

至少要覆盖：

- 真实目标平台
- 真实图形 API
- 首次进入关键场景
- 首次触发关键特效

否则你看到的只是“编辑器里暂时没坏”，不是“线上真的安全”。

## 六、我会怎么压缩成一句工程判断

如果只能留一句最有用的话，我会这样说：

`Shader Variant 治理不是先删，而是先把“真实使用面”“运行时缺口”“构建保留面”三张图对齐。`

只有这三张图对齐了，你才知道：

- 项目到底用了哪些 variant
- 运行时到底缺了哪些 variant 或 SVC
- 哪些东西是真的可以安全剔掉的

## 结论

最后把这篇压成三句话：

1. 想知道项目用了哪些 shader variant，不能只看 SVC 或 shader 代码，而要结合真实资源、构建报告和运行时命中证据一起看。
2. 想知道运行时丢了哪些，先区分是“variant 根本没编进去”，还是“variant 在包里但没有被 SVC / 预热覆盖”；前者更像功能缺失，后者更像首次卡顿。
3. 想剔除不需要的 shader variant，重点不只是删 SVC，而是先收敛 keyword 设计、再做 stripping、最后才整理预热名单。

所以真正该问的不是：

`项目里有多少个 variant，或者有多少份 SVC。`

而是：

`哪些是项目真实会走到的路径，哪些是运行时真的有缺口的路径，哪些是构建里根本不该继续保留的路径。`

把这三件事分开，Unity 的 Shader Variant 问题才会从“玄学”重新变回工程问题。
