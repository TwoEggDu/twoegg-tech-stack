总预算总表最常见的失败方式不是“没做”，而是做成了一个没人再回看的汇总表。

真正有用的总表，至少要同时满足三件事：

- 它能把 `包体 / 内存 / CPU / GPU` 放在一张账本里
- 它能明确 `谁负责`
- 它能明确 `超线以后怎么动作`

## 这张表为什么存在

因为如果没有总表，项目里最容易出现的状态就是：

- 每条线各自有数字
- 但没人知道哪些是硬线，哪些只是观察线
- 超线以后只剩下口头沟通

## 推荐字段

| 预算对象 | 硬线 | 告警线 | 观察线 | owner | 验证频率 | 超线动作 | 例外审批 |
|---------|-----|-------|-------|------|---------|---------|---------|
| 首包 |  |  |  |  |  |  |  |
| 全量包 |  |  |  |  |  |  |  |
| 稳态内存 |  |  |  |  |  |  |  |
| 峰值内存 |  |  |  |  |  |  |  |
| CPU 帧时间 |  |  |  |  |  |  |  |
| GPU 帧时间 |  |  |  |  |  |  |  |

## 字段为什么这样设计

### 1. `硬线`

超过就不能假装没看见，必须阻断、返工或审批。

### 2. `告警线`

还没到阻断，但已经说明趋势不对。

### 3. `观察线`

用于看长期漂移，不等于立即风险。

### 4. `owner`

没有 owner 的预算，最后就只是一句愿望。

### 5. `超线动作`

预算最怕的是超线以后没人知道下一步是什么。

## 怎么判断这张表填得对不对

- 能不能说清每个对象的测量口径
- 能不能说清谁有权修改这条线
- 超线以后是不是知道该阻断还是审批
- 这张表能不能对上实际 CI / baseline / 验证动作

## 最短结论

总预算总表的价值，不是把数字放一起，而是：

`把分散的预算线收成一张项目主账本，让每条线都有边界、责任人和超线动作。`
