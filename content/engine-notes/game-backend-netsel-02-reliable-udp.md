## 问题空间：可靠 UDP 是什么，为什么需要它

上一篇讲清楚了为什么游戏需要 UDP——TCP 的队头阻塞和拥塞控制对实时高频小包是结构性的障碍。但原始 UDP 什么都不保证，而游戏里确实有些消息需要可靠送达：开枪命中事件、道具购买、玩家死亡通知，如果这些消息丢了没有任何重传机制，游戏逻辑就会出错。

"可靠 UDP"的工程需求因此浮现：**在 UDP 之上实现可配置的可靠性，保留 UDP 的低延迟特性，同时对需要保证送达的消息提供 ACK 和重传**。

这听起来像是在重新发明 TCP，但有一个关键区别：**TCP 的可靠性是全局的、不可绕过的；可靠 UDP 的可靠性是按消息类型配置的，可以选择性地关闭**。

位置更新：不可靠，丢了就丢了，用最新的  
技能释放：可靠，必须送达  
聊天消息：可靠，甚至可以接受更高的延迟  

这种粒度的控制，正是可靠 UDP 的核心价值。

## 抽象模型：可靠 UDP 的基本设计目标

一个完整的可靠 UDP 实现需要解决以下几个问题：

**序列号管理**：每个包有一个单调递增的序列号，接收方通过序列号检测乱序和重复，丢弃过期包。

**ACK 机制**：接收方定期回复已收到哪些序列号。发送方据此知道哪些包需要重传。ACK 可以是 SACK（Selective ACK，只确认收到的包）而不是 TCP 的累积 ACK，这样重传更精确。

**重传策略**：超时没有收到 ACK 就重传。重传时间（RTO）的计算直接影响性能。太长=延迟高，太短=无效重传增加带宽。

**拥塞控制（可选）**：TCP 强制开启，可靠 UDP 库通常可以配置是否开启，或者使用更适合游戏的算法。

**有序交付（可选）**：有些消息不需要按顺序交付，只需要到达即可。可靠 UDP 可以提供"可靠无序"模式，避免不必要的队头阻塞。

市场上的三种主流选择——KCP、QUIC、ENet——在这几个维度上做出了不同的权衡。

## KCP：为游戏而生的最小可靠 ARQ

### 设计哲学

KCP（KCPX/KCP Protocol）由 skywind3000 开发，2016 年前后在国内游戏圈广泛流行。它的设计目标极其明确：**用带宽换延迟**。

KCP 不是一个完整的网络库，它甚至不负责底层的 socket 收发——你需要自己读写 UDP socket，把数据喂给 KCP，KCP 输出处理后的数据流。这种"纯算法层"的设计让它极易集成，也让它的职责边界非常清晰。

### ARQ 参数可调

KCP 的核心是一个可配置的 ARQ（Automatic Repeat reQuest）算法。关键参数：

```c
// nodelay: 是否开启无延迟模式（1=开启，直接重传不等待）
// interval: 内部更新时钟间隔，毫秒
// resend: 快重传阈值，收到几次 ACK 之后触发快重传
// nc: 是否关闭拥塞控制（1=关闭，游戏场景通常关闭）
ikcp_nodelay(kcp, 1, 10, 2, 1);  // 极速模式
```

正常模式下 KCP 的延迟约等于 TCP，但在 `nodelay=1, nc=1`（关闭拥塞控制）的极速模式下，RTT 可以比 TCP 低 30%-40%，这个差距在 60Hz 帧率下非常显著。

### KCP 的代价：带宽

关闭拥塞控制意味着 KCP 不会主动降速。当网络质量差时，KCP 会持续重传，产生显著更多的带宽消耗。一个 10% 丢包率的网络，KCP 极速模式可能产生 TCP 2 倍以上的流量。

这在 4G/5G 移动网络和 Wi-Fi 场景下是一个真实的成本。你需要在服务端实现流量控制，避免在拥塞网络下让 KCP 把客户端的带宽全部打满。

### KCP 适合的场景

- 服务端到服务端的高速数据传输，带宽充裕
- 移动游戏的实时对战，可以接受一定带宽开销换低延迟
- 已有 UDP socket 管理，需要一个纯算法层的可靠传输
- 国内网络环境（高丢包率，KCP 的快重传优势明显）

### KCP 的局限

KCP 只实现了 ARQ 算法层，没有加密、没有连接管理、没有多路复用、没有 MTU 探测。这些都需要在 KCP 之上自行实现。在团队规模较小或需要快速交付时，这些"胶水工作"的开发量不可低估。

## QUIC：HTTP/3 的传输层，目标不是游戏

### 设计目标

QUIC 最初由 Google 设计，目标是替代 TCP 作为 HTTP 的传输层，解决 HTTP/2 在 TCP 上的队头阻塞问题。它现在是 HTTP/3 的底层协议，由 IETF 标准化（RFC 9000）。

QUIC 在 UDP 上实现了：
- 多路复用（多个 Stream 独立，一个 Stream 的丢包不阻塞其他 Stream）
- 0-RTT 或 1-RTT 连接建立
- 内置 TLS 1.3 加密
- 连接迁移（IP 变化后连接不断开，手机从 Wi-Fi 切到 4G 时尤其有用）

这些特性听起来对游戏也很有用，但 QUIC 的设计目标始终是**最大化吞吐量和连接可靠性**，不是最小化延迟。

### QUIC 的适用场景

**资源下载**：游戏补丁、Asset Bundle 下载场景下，QUIC 的多路复用和 0-RTT 特性有明显优势，可以替代 HTTP/2。

**API 调用**：大厅服务、账号系统、道具查询这类接口，对延迟不极端敏感，QUIC 的连接迁移特性对移动客户端友好。

**热更新服务**：文件传输不需要极低延迟，但需要断点续传、多文件并发，QUIC 非常适合。

### QUIC 不适合游戏实时逻辑的理由

QUIC 的拥塞控制是标准 cubic/bbr，设计上以吞吐量为目标，和 KCP 的"宁愿多用带宽也要低延迟"是截然不同的哲学。

QUIC 的 Stream 语义是有序可靠的（类似 TCP 的语义），虽然 Stream 之间不互相阻塞，但每个 Stream 内部依然是队头阻塞。游戏位置更新的语义是"无序不可靠"，不适合放进 QUIC Stream。

此外，QUIC 的实现复杂度很高。主流实现（quiche、msquic、lsquic）的代码体量是 KCP 的数十倍，调试难度和维护成本显著更高。

**一句话定位**：QUIC 是服务端对服务端通信、资源分发、API 网关的好选择，不是实时游戏帧同步的选择。

## ENet：轻量嵌入式，独立游戏的实用选择

### 历史和定位

ENet（github.com/lsalzman/enet）是 Lee Salzman 开发的一个 C 语言可靠 UDP 库，被大量独立游戏和开源游戏引擎使用（包括 Godot 的早期版本）。它的特点是：**完整的连接管理 + 可靠/不可靠通道 + 单文件库**。

```c
// ENet 的 Channel 概念
// Channel 0: 可靠有序（登录、状态机事件）
// Channel 1: 不可靠（位置更新）
// Channel 2: 可靠无序（聊天消息）
enet_peer_send(peer, 0, packet); // 发到 Channel 0
```

ENet 提供了开箱即用的连接建立、心跳、断线检测和带宽控制，不需要在它之上再自己实现这些基础设施。

### ENet 的优势

- **完整度高**：连接管理、多通道、带宽限制、压缩（LZ4 可选）都有
- **轻量**：单文件 C 库，约 5000 行代码，嵌入成本极低
- **成熟**：有超过 15 年的生产使用历史，稳定可靠
- **跨平台**：Windows/Linux/macOS，移动端也支持

### ENet 的局限

ENet 没有加密层，需要自行叠加 DTLS 或应用层加密。在商业游戏中，明文传输是不可接受的安全风险。

ENet 的连接数上限和吞吐量在 C10K 以上场景下有性能瓶颈，不适合万人以上同服的 MMO。

ENet 的维护者活跃度近年下降，社区有几个 fork 版本（比如 cubed ENet），在选用时需要考虑长期维护风险。

**适合场景**：独立游戏、小型多人对战、快速原型、不超过几千并发连接的场景。

## 工程边界：选型决策依据

把三种协议放在同一个决策维度上比较：

| 维度 | KCP | QUIC | ENet |
|------|-----|------|------|
| 核心目标 | 最低延迟 | 最高吞吐/连接可靠 | 完整轻量嵌入 |
| 延迟特性 | 优秀（极速模式） | 普通（类 TCP） | 中等 |
| 带宽消耗 | 高（换延迟的代价） | 中 | 中 |
| 加密支持 | 无（需自行叠加） | 内置 TLS 1.3 | 无（需自行叠加） |
| 连接管理 | 无（算法层） | 完整 | 完整 |
| 实现复杂度 | 低（纯算法） | 非常高 | 低 |
| 适合场景 | FPS/MOBA 实时逻辑 | 资源下载/API 网关 | 独立游戏/小型对战 |
| 团队要求 | 需要自建基础设施 | 需要专项维护能力 | 开箱即用 |

### 决策流程

**第一问：你的场景是资源传输还是实时逻辑？**
- 资源传输（补丁下载、Asset Bundle）→ 考虑 QUIC 或 HTTP/2
- 实时逻辑 → 进入下一问

**第二问：团队是否有能力维护自建网络基础设施？**
- 有 → KCP + 自建连接管理（性能最优）
- 没有 → ENet（快速集成，工程成本低）

**第三问：并发连接数上限是多少？**
- 几千以内 → ENet 够用
- 万级以上 → KCP 或自研

## 自研可靠 UDP 层的基本代价

当 KCP/ENet 都不满足需求时（比如需要深度定制拥塞控制算法、专有加密方案、特殊的多路复用语义），一些大型游戏公司会自研可靠 UDP 层。

自研的最小代价清单：

- **序列号和 ACK**：SACK 实现，通常需要一个滑动窗口
- **RTO 计算**：至少实现 RFC 6298 的 RTO 计算，避免在不同 RTT 下出现大量不必要重传
- **快重传**：连续收到 N 个 ACK 就触发重传，不等超时
- **Jitter Buffer**：处理乱序到达的包，在消费之前按序列号排队
- **连接状态机**：建立、保活、断开
- **加密层**：DTLS 或等价方案
- **防护**：防止 UDP 放大攻击和伪造来源

这个列表的工程量大约是 2-3 名有经验工程师 2-4 个月的工作量，不计后续维护。在项目早期，强烈建议用 KCP 或 ENet 先跑起来，等有了真实的性能数据和瓶颈，再决定是否值得自研。

## 最短结论

KCP 是游戏实时逻辑的首选，代价是带宽和自建基础设施的工程量。

QUIC 适合资源分发和 API 网关，不适合高频实时帧数据。

ENet 是独立游戏和小型团队的工程捷径，完整度够用，嵌入成本低。

三者之间不是竞争关系，一个游戏服务端可以同时用 QUIC 做资源下载、KCP 做实时对战逻辑——选型是按场景做的，不是全局统一的。
