## 问题空间：一个看起来无害的顺序问题

假设有一个玩家背包系统。玩家消耗道具时，后端要做两件事：更新 MySQL 里的道具数量，以及更新 Redis 里的玩家背包缓存。这两件事的执行顺序，在低并发时不会有任何问题，但在高并发下会导致数据撕裂——玩家消耗了道具，但缓存里还显示道具存在，或者反过来，缓存已经删掉，但数据库里道具还没扣减。

这个问题在游戏项目里反复出现的原因不是程序员不够细心，而是**缓存和数据库是两个独立系统，任何跨系统的写操作天然不是原子的**。没有人会保证你的两条写操作之间不会插入一个并发请求。

更深层的原因是游戏后端在这个问题上的需求比大多数业务系统更苛刻：货币和道具的错误直接影响玩家付费体验，会导致客诉和投诉；但同时游戏后端又有极高的并发压力，不能为了一致性把所有写操作都加全局锁。这个矛盾是游戏后端缓存设计最核心的张力。

## 抽象模型：三种一致性策略

在解决具体问题之前，先把常见的缓存一致性策略抽象出来，理解它们各自的设计前提。

### Cache-Aside（旁路缓存）

这是游戏后端最常用的策略。核心规则只有两条：

**读操作**：先读缓存，命中则返回；未命中则读数据库，将结果写入缓存，返回数据。

**写操作**：先写数据库，成功后删除（不是更新）缓存。

为什么是"删除缓存"而不是"更新缓存"？原因有两个。第一，删除是幂等操作，失败后重试没有副作用；而更新缓存需要知道最新值，在并发写的情况下，后到的写操作可能把早到的写操作的结果覆盖，形成最终不一致。第二，如果写操作需要复杂计算才能得到最新缓存值（比如背包总价值需要聚合计算），删除缓存后让下一次读操作重新计算更简单，也更符合"缓存是可重建快照"的语义。

Cache-Aside 的**核心假设**是：缓存未命中的代价可以接受。如果某个数据的缓存命中率极高，偶发未命中会打一次数据库，这是可以容忍的。但如果缓存失效风暴来了（大量 key 同时过期），这个假设就不成立了——这是雪崩问题，在第四篇文章会详细讲。

### Write-Through（写穿）

Write-Through 要求每次写操作必须**同时更新数据库和缓存**，两者都成功才算写入成功。

这个策略的好处是缓存始终和数据库保持一致，读操作永远命中缓存，性能最优。代价是：写操作的延迟翻倍（要等数据库和缓存都写完），而且对于很多数据，写入缓存后不一定会被立即读到（比如某个玩家的数据写入了缓存，但那个玩家并没有立刻上线读取），这叫**写放大**——做了没必要的缓存更新。

游戏后端的典型问题是：游戏中有大量"写多读少"的数据，比如战斗日志、行为埋点。这类数据如果用 Write-Through，缓存里塞满了几乎不会被读到的数据，浪费内存。

Write-Through 在游戏后端适用于**读密集型的热数据**：玩家基础属性（等级、昵称、头像）在每次战斗、组队、社交界面都会被读到，写频率相对低，非常适合 Write-Through。

### Write-Behind（异步写回，Write-Back）

Write-Behind 只写缓存，异步延迟批量写数据库。这个策略写性能最高，但宕机时数据库和缓存的差距就是丢失的数据，风险极高。游戏后端**几乎不在货币、道具等关键数据上使用这个策略**。但它有一个合法的使用场景：实时更新的排行榜积分可以先写 Redis ZSet，每隔几分钟批量持久化到数据库。

## 具体实现：Cache-Aside 的踩坑细节

Cache-Aside 看起来简单，但有几个细节是项目里反复踩的坑。

### 踩坑 1：先删缓存再写数据库

很多人会这样写：先删 Redis 缓存，再更新 MySQL。逻辑上看起来没问题，但并发下必然出问题：

1. 线程 A 删除缓存
2. 线程 B 读取数据，缓存未命中，读到数据库中的**旧值**，写入缓存
3. 线程 A 更新数据库，写入新值
4. 此后所有读操作读到的都是缓存里的旧值，直到 TTL 过期

结果是：缓存里留着一个过期的旧值，持续时间等于 TTL。如果 TTL 是 10 分钟，玩家的背包状态会错误 10 分钟。

**正确顺序是：先写数据库，再删缓存。**

### 踩坑 2：先写数据库再删缓存——并发下仍有窗口

即使用了正确顺序，也存在理论上的并发窗口：

1. 线程 A 读缓存，未命中
2. 线程 A 读数据库，得到旧值 V1（还没写入缓存）
3. 线程 B 写数据库，V2，删除缓存（此时缓存本来就没有，删除是空操作）
4. 线程 A 把 V1 写入缓存
5. 此后读到的是旧值 V1

这个窗口实际上非常短（步骤 1-4 全部发生在毫秒级），在大多数游戏场景下可以接受。但对于绝对不能容忍错误的数据（货币、道具），不能只依赖这个"小概率不出错"的假设。

### 双删策略：第一次删和第二次延迟删

为了缩短上述窗口，工程上有个"双删"方案：

1. 写操作时，先删缓存（让并发读操作去数据库读）
2. 更新数据库
3. 再次删除缓存（清除步骤 2 之前可能被写入的旧值）

步骤 3 通常加一个短延迟（500ms 到 1s），因为需要等步骤 2 的数据库写入在主从复制中传播完成。如果你用的是读写分离，读操作可能打到从库，主从延迟期间从库还是旧值，延迟双删的延迟时间要大于主从复制延迟。

**双删策略的局限**：它把一致性窗口压缩到了"数据库写入到第二次删除之间"，但不能彻底消除窗口。而且延迟删除需要异步实现（通常通过消息队列），增加了架构复杂度。

### 延迟双删的工程实现

在游戏后端，延迟双删通常这样实现：

```python
def update_player_gold(uid: str, delta: int):
    # 第一次删缓存
    redis.delete(f"player:cache:{uid}")
    
    # 更新数据库
    db.execute("UPDATE player SET gold = gold + ? WHERE uid = ?", delta, uid)
    
    # 发送延迟消息（500ms 后执行第二次删除）
    mq.send_delayed(
        topic="cache_invalidate",
        payload={"key": f"player:cache:{uid}"},
        delay_ms=500
    )
```

消费者收到消息后再执行一次 `redis.delete`。这个方案把延迟删除的可靠性交给了消息队列，即使删除失败也可以重试，比在业务代码里 sleep 然后删除更健壮。

## 游戏场景的一致性分级

不同的游戏数据对一致性的容忍程度是不同的。把数据分级，对不同级别用不同策略，是游戏后端缓存设计的核心方法。

### 不能容忍短暂不一致

- **货币（钻石、金币）**：玩家消费、充值相关，错误会导致客诉和财务损失。这类数据建议：写操作直接走数据库，Redis 只做读缓存且 TTL 极短（5-30 秒），或者对这些关键数值完全不缓存，直接读库（配合数据库连接池和读写分离承载并发）。
- **道具数量**：和货币类似，错误会影响玩家体验。部分游戏选择在 Redis 上用 Lua 脚本做原子操作（检查+扣减在一个事务里），以 Redis 为准，异步写回数据库（这是一种特殊的 Write-Behind，只在确认 Redis 高可用时使用）。
- **登录 Token**：Token 验证错误会导致玩家被误踢下线或账号安全问题，需要保证 Redis 与授权系统的一致性。

### 可以容忍短暂不一致（最终一致性）

- **排行榜**：玩家分数更新后，排行榜展示延迟几秒到几分钟都是可以接受的。通常 ZSet 直接在 Redis 里更新，异步写数据库即可。
- **公告、活动配置**：这类数据变化频率低，可以接受几分钟的缓存延迟，设置合理的 TTL 即可。
- **玩家在线状态**：其他玩家看到某个玩家的在线状态有几秒延迟是完全正常的游戏体验。
- **聊天记录（非关键）**：聊天消息允许偶发的短暂不一致，不影响核心玩法。

### 分级的工程含义

把数据分为"强一致"和"最终一致"两类，在代码层面对应不同的写入路径。强一致数据的写入路径要经过写确认（等待数据库写成功），并配套缓存失效机制；最终一致数据可以异步写，允许短暂的数据库和缓存不同步。

不要试图把所有数据都做成强一致——代价会体现在写入延迟上，进而影响战斗、交互的响应速度，玩家会直接感知到卡顿。

## 分布式场景下的额外复杂性

单机 Redis 的一致性问题已经足够复杂，分布式环境（Redis Cluster、主从复制）会进一步引入新的不确定性。

**主从复制延迟**：在主从架构下，写操作到 master，读操作可能路由到 slave。主从同步有延迟（通常几毫秒到几十毫秒），这意味着删缓存后立即读缓存，仍可能从 slave 读到旧值。这是延迟双删里"延迟"的来源之一。

**缓存删除失败**：网络抖动导致 `DEL` 命令没有执行成功。解决方案是把缓存失效操作写入可靠消息队列，由消费者保障最终执行。

**Redis Cluster 的 key 分片**：在 Cluster 模式下，不同 key 可能落在不同节点。涉及多个 key 的原子操作（Lua 脚本、事务）要求所有 key 在同一 slot，否则会报错。这个约束在设计数据结构时就要考虑，不是上线后能轻松改的。

## 工程边界：一致性不是 Redis 该独自解决的问题

这里有一个重要的认知边界：缓存一致性是**应用层的责任**，而不是 Redis 的责任。Redis 只保证单个命令的原子性，不保证你的多步业务操作的原子性。

企图在 Redis 层面解决所有一致性问题（比如把 Redis 变成分布式事务的参与者）是架构上的错误方向。游戏后端中，强一致数据的一致性应该由**数据库事务**保证，Redis 在这里只是加速读操作的一层缓存，逻辑上可以随时被清空并从数据库重建。

如果你发现自己在 Redis 里存储了"删了就不可恢复的数据"，这通常意味着系统设计出了问题。

---

## 最短结论

缓存一致性没有银弹——选对策略的前提是先把数据分级，货币道具要强一致，排行榜公告容忍最终一致，然后对每类数据选不同的写入路径。
