写到这里，这条系列其实已经把很多“为什么会坏”的问题讲得比较完整了：

- 资源系统有哪些层
- 哪些层最容易被混
- `AssetBundle / Addressables / Shader / SVC` 分别站在哪
- 重复资源、脚本引用丢失、首载卡顿和缓存失效这些事故为什么会发生

但项目真正上线前，最容易把团队拖进长期成本的，往往不是不知道这些原理，而是：

`没有一套稳定的烟测和回归体系，能在问题上线前先把它挡住。`

这也是资源系统特别容易吃亏的地方。

很多团队对“回归”的理解还停留在：

- 构建成功
- 场景能打开
- 核心流程能跑一遍

这些当然有价值，但对资源系统来说，远远不够。

因为 Unity 资源问题真正高发的地方，常常都不在“程序还能不能启动”这种粗层面，而在：

- 关键入口第一次实例化时才暴露的对象恢复问题
- 关键 `Shader` 路径第一次命中时才暴露的首载或错效果
- 远端内容切换时才暴露的缓存与回滚问题
- `Scene / Prefab / 脚本` 身份链只在真机构建下才断的情况

所以这篇我就只讲一件事：

`Unity 资源系统到底该怎么做一套最低可落地的烟测和回归防线。`

## 先给一句总判断

如果把整件事压成一句话，我会这样说：

`Unity 资源系统的烟测和回归，最不该被理解成“构建完打开一个场景看看”，而更应该被理解成一条验证链：先验证构建产物，再验证入口实例化，再验证 Shader 首载与关键路径，最后验证内容身份、缓存复用和回滚快照。`

这句话里最关键的是：

- `验证链`
- `入口实例化`
- `内容身份`

只要把这三件事先立住，很多“上线后才知道坏了”的资源事故都能明显往前挡。

## 一、为什么资源系统的烟测不能只做“能不能打开场景”

这是最常见、也最贵的误判。

## 1. 场景能打开，不代表资源链就是对的

一个场景打开成功，顶多只能说明：

- 主入口没在最前面直接崩掉
- 至少有一部分资源和脚本边界还对着

它并不说明：

- 所有关键 `Prefab` 都能正确实例化
- 所有关键脚本身份链都还在
- 所有关键 `Shader` 路径都存在且首载稳定
- 当前内容快照和缓存副本完全一致

所以只做“场景能开”这种烟测，通常只能挡住最粗的事故。

## 2. 资源系统的问题往往在“第一次真正使用”时才显形

这也是为什么资源系统的烟测必须更贴近真实入口。

例如：

- `Prefab` 要等 `Instantiate` 才真正展开对象图
- 脚本要等生命周期接入后才真正跑起来
- `Shader` 要等某条关键路径第一次命中才暴露缺口
- 缓存问题要等第二次进入同一内容世界才真正显影

所以资源系统验证的重点不该只是：

`能不能 load。`

而应该是：

`第一次真的用起来时，会不会坏。`

## 二、最小验证链应该先分成五层

如果项目现在还没有成体系的资源回归，我更建议先把验证链拆成下面五层：

1. `构建校验`
2. `入口实例化烟测`
3. `Shader 首载与关键路径验证`
4. `缓存复用验证`
5. `回滚和快照一致性验证`

这五层不要求一开始都做得很重，但一定要先分层。

因为只要不分层，资源回归很快就会变成一句模糊的话：

`我们已经测过了。`

但谁也说不清到底测了哪一层。

## 三、第一层：构建校验先挡住“产物世界已经错了”的问题

这层是最便宜、也最应该最先做的。

因为如果构建产物本身就已经明显不对，后面再跑真机入口只是在浪费时间。

## 1. 最低限度应该校验什么

如果只能先做一批最低成本、最高收益的构建校验，我会优先放这些：

- bundle 数量、总大小、关键 group 大小是否异常跳变
- 重复资源是否明显增加
- 关键 bundle 的依赖数量是否异常膨胀
- 漏包、错包、依赖闭包不完整
- 关键 `SVC` 中的 variant 是否还存在于目标构建
- 关键 `Shader` / `Material` / built-in bundle 是否出现异常变化

这层的重点不是“构建成功”，而是：

`构建出来的内容世界还像不像我们预期的那一份。`

## 2. 哪些证据最值得用

这层最值得结合的，通常就是前面那篇讲过的几类构建产物：

- `Manifest`
- `BuildLayout`
- `Catalog`

其中：

- `BuildLayout` 最适合看大小、隐式依赖、重复和 built-in
- `Manifest` 最适合看依赖关系和 bundle 图
- `Catalog` 最适合看运行时映射有没有切错

## 3. 这层最常见的误用

最常见的误用是：

- 只看构建有没有报错
- 或者只看总包体有没有变大

这样往往会漏掉最值钱的结构变化：

- 某个关键入口依赖突然变多了
- 某个 built-in 资源被重新带进来了
- 某个关键 `Shader` 路径虽然没报错，但已经不在目标构建里

## 四、第二层：入口实例化烟测，先挡住对象恢复链和脚本身份链

这层是资源系统烟测里最值钱、也最容易被省掉的一层。

## 1. 为什么必须测“实例化”，不能只测“加载”

因为很多高频事故都要等到：

- `Prefab` 真正 `Instantiate`
- `Scene` 真正接入世界
- 脚本真正进入生命周期

之后才会显形。

只测 `LoadAsset`，常常只能说明：

`对象模板能拿到。`

它说明不了：

`对象世界真的能稳定展开。`

## 2. 最低限度应该覆盖哪些入口

如果只选几条最值钱的入口，我会优先覆盖：

- 首屏入口
- 主场景入口
- 主战斗入口
- 当前线上活动入口

这些入口最值钱，不是因为它们最复杂，而是因为：

`它们最直接对应玩家第一感知。`

## 3. 实例化烟测要测到什么程度

更稳的做法通常不是“把对象生成出来就算过”，而是至少要确认：

- 关键 `Prefab` 能成功实例化
- 关键脚本没有 missing 或关键字段断引用
- 场景或 UI 关键对象能进入基本生命周期
- 首批必须看到的材质和特效没有明显异常

这层的目标不是替代完整玩法测试，而是：

`先挡住资源和对象恢复链层面的高频事故。`

## 五、第三层：Shader 首载和关键路径验证，先挡住“第一次才坏”的问题

这层如果不单列，资源系统烟测几乎一定不够。

## 1. 为什么 Shader 验证必须单独拉出来

因为 `Shader` 问题最典型的特点就是：

- 构建时不一定报错
- 编辑器里不一定显影
- 对象实例化也不一定立刻报错
- 但第一次真正命中某条路径时才炸

所以如果没有单独的 shader 首载验证，很多线上问题会自然漏过去。

## 2. 最低限度应该测什么

如果只能做一版最低成本的 shader 验证，我会优先放这些：

- 首屏关键材质和后处理是否正确显示
- 主战斗首批高价值特效是否正确显示
- 关键 `SVC` 是否在预期时机 `WarmUp`
- 第一次进入与第二次进入同一入口，首载体验差异是否异常

## 3. 这层为什么必须和入口绑定

因为 shader 问题最容易被测偏的一点是：

`你在测试场景里手动点一下，和玩家在真实入口里第一次命中，往往不是同一条路径。`

所以更稳的做法是：

`让 shader 验证跟入口走。`

而不是单独做一个脱离入口的图形小场景就算结束。

## 六、第四层：缓存复用验证，专门回答“第二次为什么还像第一次”

这层是很多项目最容易完全没有的。

## 1. 为什么必须把第一次和第二次进入分开测

因为很多资源问题只有在这个对比里才看得出来：

- 第一次进入该慢一点，第二次应该明显更稳
- 如果第二次还几乎和第一次一样慢，就要怀疑缓存复用或准备链调度

如果只测第一次进入，你很难区分：

- 本来就是冷启动成本
- 还是本该复用却没复用

## 2. 这层最低限度要验证什么

更稳的最低限度通常是：

- 同一入口第一次进入的耗时和关键节点
- 同一入口第二次进入的耗时和关键节点
- 当前内容快照不变时，第二次是否真正复用了缓存

重点不是追求非常精确的基准，而是：

`有没有明显失去复用收益。`

## 3. 这层为什么不能只翻缓存目录

因为缓存目录只能说明：

`本地有什么副本。`

它说明不了：

- 当前 catalog 到底指向谁
- 本地副本是不是目标 hash
- 运行时是不是又重新走了一遍准备链

所以缓存验证一定要同时看：

- 本地副本
- 当前运行时映射
- 入口第二次进入表现

## 七、第五层：回滚和快照一致性验证，专门回答“线上切版本会不会乱”

这一层最容易被认为“以后再补”，但其实很值钱。

## 1. 为什么资源系统必须单独测回滚

因为资源系统的回滚不是只换一个版本号那么简单。

它至少同时牵涉：

- 发布快照
- catalog
- bundle hash
- 本地缓存副本

任何一层没对齐，现场都可能会变成：

- 某些玩家是新内容
- 某些玩家是旧内容
- 某些玩家是混合世界

## 2. 最低限度应该验证什么

如果只能做一版最小回滚验证，我会优先放这些：

- 当前版本 -> 新快照 -> 回退快照 这条链是否可重复
- 回退后入口内容是否和旧快照一致
- 本地缓存不会在回滚后错误复用新副本

这层的目标不是把所有线上策略都模拟出来，而是：

`先证明这套快照体系不是单向的。`

## 八、把五层收成一套最低可落地体系

如果你现在就要在项目里把资源系统烟测和回归补起来，我更建议先用一套更保守的最小法：

1. 每次构建先出一版结构校验
2. 固定 3-4 个高价值入口做实例化烟测
3. 让这些入口同时覆盖关键 shader 首载路径
4. 每个入口至少测第一次和第二次进入
5. 每个大版本至少做一次快照切换和回滚验证

这套做法的重点不是“一次做全”，而是：

`先把资源系统最贵的几类事故挡在上线前。`

## 九、最常见的三个误判

最后把最常见的误判也压一下。

## 1. 误判一：构建成功就说明资源世界是对的

更准确的说法应该是：

`构建成功只说明脚本没在那一步直接崩掉，不说明交付世界是对的。`

## 2. 误判二：场景能打开就说明资源回归够了

更准确的说法应该是：

`场景能开只能挡住最粗问题，挡不住实例化、脚本身份链、shader 首载和缓存复用问题。`

## 3. 误判三：只要有自动化，就已经有资源回归体系

更准确的说法应该是：

`资源系统真正有价值的自动化，不在“有没有脚本”，而在“是不是沿着构建产物、入口实例化、shader 首载、缓存复用和回滚快照这条链去挡风险”。`

## 最后收成一句话

如果把这篇最后再压回一句话，我会这样说：

`Unity 资源系统的烟测和回归，真正该做的不是“能不能跑起来”这种粗验证，而是沿着构建产物、入口实例化、Shader 首载、缓存复用和回滚快照，一层层把“第一次真正进入内容世界”最容易出事的地方往前挡。`
