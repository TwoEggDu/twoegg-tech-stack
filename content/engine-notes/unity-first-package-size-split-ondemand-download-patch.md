上一篇讲了 AssetBundle 的切包粒度，核心结论是：粒度设计不是按目录随便分，而是按变化频率、复用范围和依赖闭合来画边界。但粒度讲完以后，紧接着就会碰到一个更具体的问题：这些 bundle 切好了，哪些应该跟安装包一起发出去，哪些应该留在服务器上等用户需要时再下载。

这就是首包问题。它不是一个体积压缩问题，而是一个交付边界问题。

## 为什么首包问题不是"越小越好"

最直觉的想法是：首包越小越好，商店下载越快，用户流失越少。这个想法对了一半。

对的那一半是：移动端商店对包体有硬性限制。Google Play 的 AAB 基础 APK 有 200 MB 的门槛，超过以后需要走 Play Asset Delivery。iOS 有蜂窝网络下载限制，超过阈值用户会被提示切换到 Wi-Fi。包体过大直接增加商店转化漏斗的流失率。

错的那一半是：把首包压到极小，换来的是安装后的等待。用户装完 App，点进去，看到一个进度条，等几分钟甚至十几分钟下载"必要资源"，体验比首包大几十 MB 但能直接进游戏更差。

所以正确的问题不是"首包怎么压到最小"，而是：

`首包应该装到什么程度，才能让用户在安装后尽快进入一次完整的首进体验，同时不超过商店和网络环境的硬限制？`

这里面有两条边界同时存在：

- 下限：装得太少，首进体验断裂，要么黑屏等下载，要么进去以后什么都没有
- 上限：装得太多，商店门槛过不去，或者下载弃装率太高

首包设计就是在这两条边界之间，找到那个"刚好能支撑完整首进"的位置。

## 哪些内容必须跟首包走

首包的最小集合，不是按"重不重要"排的，而是按"首进链路会不会断"排的。

用户从安装到第一次真正"在玩"，中间会经过几个必经节点：启动画面、登录流程、首屏 UI、第一场战斗或第一个关卡。如果这条链上任何一个节点缺资源，要么卡在下载进度条上，要么直接出异常。

以下这些内容，通常必须跟首包走：

- 启动和登录流程所需的 UI 资源、配置表和脚本
- 首屏主界面的 UI 和基础图标
- 第一场战斗或第一个关卡的场景、角色模型、动画和特效
- 基础 Shader 集合——特别是首帧可见的那些，否则会触发运行时 Shader 编译卡顿
- 基础音频：BGM、关键音效
- 客户端核心配置和必要的数据表

这里面最容易被忽略的是 Shader。很多项目把 Shader 放进了热更包，导致首次进入时 GPU 要在运行时编译变体，出现明显的帧卡顿甚至白屏。Shader 的首包策略应该和首进视觉体验直接绑定，而不是按"大小"来决定放不放。

还有一类容易遗漏的是本地化基础资源。如果项目多语言上线，至少默认语言的文本和字体必须进首包，否则首屏就会出现缺字或 fallback 字体。

## 哪些内容适合分包和按需下载

跟首包逻辑相反，分包的核心判断是：`用户在首进链路完成之前，用不到这些内容。`

典型的分包候选：

- 非首进的后续关卡和场景
- 活动、赛季、节日等限时内容
- 角色皮肤、时装等可选外观内容
- 高清 CG 视频和过场动画
- 非默认语言的本地化资源包
- 低频系统的资源，比如成就图鉴、收藏系统的高清图
- 特定地区的合规内容或地区专属素材

分包之后，这些内容在运行时按需下载。下载时机通常有几种模式：

- 首进结束后后台静默下载：用户在首屏停留或做新手引导时，后台把次要资源拉下来
- 进入对应功能时触发下载：用户点进某个系统，如果资源不在本地，弹出下载提示
- 版本更新时整批拉取：大版本更新时一次性拉完，用户明确知道"在更新"

无论哪种模式，都需要在客户端实现一套完整的资源状态管理：知道哪些资源在本地、哪些需要拉取、拉取失败怎么重试、版本不一致怎么处理。这不是一个简单的 if-else，而是一个需要和资源管理框架深度配合的运行时逻辑。

## 差量更新和热更 patch 真正在解决什么

分包解决的是"首包和次包的边界"，而差量更新解决的是另一个问题：`已经发出去的内容，改了一点以后怎么让用户最小代价拿到变更。`

最朴素的做法是全量替换：改了一个 bundle 里的一张贴图，就重新发一整个 bundle 让用户重新下载。这个方案在 bundle 很小时能接受，但 bundle 一旦超过几十 MB，用户就会开始抱怨"什么都没变，怎么又要下载这么多"。

差量更新的核心思路是：只发变化的部分。但"变化的部分"可以站在不同层面：

- 文件级差量：整个 bundle 文件做二进制 diff，生成 patch，用户端合并。这要求客户端保留旧版 bundle 原件用来合并，磁盘成本高
- Bundle 级重建：把变化的资源重新打成一个 bundle 或若干 bundle，只下载这些新 bundle。这要求切包粒度合理——粒度太粗，一个资源改动牵动整个大包；粒度太细，bundle 数量爆炸
- 资源级热更：框架层面跟踪每个资源的版本，只更新变化的资源条目。Addressables 的 Content Update 和 YooAsset 的补丁包走的大致是这个方向

这里面最关键的一层判断是：

`差量更新的代价不是在更新时才决定的，而是在切包时就已经决定了。`

如果打包粒度设计做得好——变化频率高的内容和稳定内容分在不同 bundle 里——那差量更新时只需要重发少量 bundle。如果粒度设计做得差——一个 bundle 里混着稳定资源和频繁变动资源——那每次更新都会牵动不该动的内容。

所以首包、分包和差量更新，本质上不是三个独立问题，而是同一条交付链上的三个切面：

`首包定"第一次装什么"，分包定"后面什么时候补什么"，差量定"改了以后怎么最小代价送到"。`

## Addressables / YooAsset 在这条链里分别站在哪里

当讨论首包和热更时，很容易把"框架名字"当成"解决方案本身"。但 Addressables 和 YooAsset 在这条链里的角色是管理层和调度层，不是交付格式本身。

底层的交付格式仍然是 AssetBundle。无论用 Addressables 还是 YooAsset，最终生成的产物都是 bundle 文件加上一套描述内容映射的元数据（Addressables 叫 Catalog，YooAsset 叫 Package Manifest）。

框架真正帮你做的事情是：

- 把 Group / Package 的划分和 bundle 的生成关系管理起来
- 提供 Remote / Local 的边界配置，决定哪些 bundle 跟首包走、哪些放 CDN
- 在运行时根据 Catalog 或 Manifest 判断本地版本和远端版本的差异
- 提供更新检查、增量下载和版本回退的接口

但框架不会替你回答这些问题：

- 首包到底该装哪些 Group
- Bundle 粒度是按场景拆还是按模块拆
- 差量更新时哪些 bundle 会被牵动
- 版本回滚到底回滚到哪个状态

这些仍然是工程判断，需要结合项目的内容规模、更新频率和运营节奏来定。框架提供的是执行能力，不是决策依据。

## 项目落地检查表

以下是围绕首包、分包和更新边界做项目决策时，应该先回答的问题。

首包边界：

- 首进链路上有哪些必经节点？这些节点的资源是否全部进了首包？
- 首帧可见的 Shader 是否进了首包？是否会触发运行时编译？
- 默认语言的文本和字体是否进了首包？
- 首包体积是否在目标商店和网络环境的限制之内？
- 安装后不联网的情况下，用户能否完成首进链路？

分包边界：

- 非首进内容是否有明确的分包归属？
- 分包下载的触发时机是否设计过？是后台静默、功能入口触发还是版本更新时拉取？
- 资源不在本地时，客户端是否有合理的降级或等待体验？
- 分包和首包之间是否存在未声明的依赖？

更新边界：

- 打包粒度是否支持按变化频率独立更新？
- 一次典型的资源改动，会牵动多少 bundle 重新发布？
- 框架层的版本检查和增量下载是否和实际的 bundle 粒度匹配？
- 回滚能力是否在设计时考虑过？回滚到上一个版本需要什么条件？

## 结语

首包问题看起来是一个体积问题，实际上是一个交付边界问题。它不是孤立的——首包边界决定了用户的首进体验，分包边界决定了后续内容的交付节奏，差量更新的代价在切包时就已经决定了。这三个切面加在一起，才是"资源怎么发出去"这件事的完整地图。
