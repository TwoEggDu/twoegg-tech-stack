> 如果只先记一句话，我建议记这个：`fio`、`iostat`、Perfetto 和 Instruments 不是四个“谁更高级”的替代品，而是在回答四个不同问题；真正稳定的 I/O 排查，不是挑一个工具猛看，而是把它们串成一条判断链。

前面几篇我们已经把底图立住了：

- 存储设备在怕什么
- 文件系统会在哪些地方替你先结账
- OS I/O 请求怎样流动
- 为什么移动端会把同一个问题换一种方式放大

再往下走，就该进入很多团队真正会卡住的一步：

`那我到底怎么测？`

因为项目里“像 I/O 的症状”其实很容易长得像同一件事：

- 首载卡一下
- streaming 边界抖一下
- 某些低端机就是慢
- 明明异步读了，还是在关键帧掉下来

但这些外观看起来很像的现象，背后可能完全不是同一个根因：

- 真的是底层设备太慢
- 是访问模式把请求打得太碎
- 是 `open / close`、元数据和页缓存先在前面结账
- 是 I/O 回来后解压、反序列化、依赖解析和 GPU 上传在后面收口

所以这篇文章想解决的，不是“教你背几个命令”，而是先把这件事说清楚：

`当你怀疑游戏 I/O 有问题时，哪一个工具该先上，它到底回答什么，不回答什么。`

## 先把四个工具放回它们真正的位置

如果把这四个工具压成一张判断表，我会先这样记：

| 工具 | 它最擅长回答什么 | 它不适合回答什么 |
| --- | --- | --- |
| `fio` | 底层设备在顺序 / 随机、不同块大小、不同队列深度下的大致能力边界 | 你的游戏线程为什么在某一帧卡住 |
| `iostat` | 真实运行时设备有没有忙、请求粒度大不大、等待和队列有没有拉高 | 具体是哪一个资源、哪一段代码在发 I/O |
| Perfetto | Android 上一条加载链是怎样跨 app、线程、内核和系统资源流动的 | 设备理论上应该跑多快 |
| Instruments | Apple 平台上 app 侧文件活动、读写节奏、目录 / 属性操作和 CPU 收口时序 | Android 内核和系统调度细节 |

这张表最值钱的地方不是“工具介绍”，而是：

`不要让一个工具回答它根本不该回答的问题。`

比如：

- 拿 `fio` 跑得很快，就说“项目 I/O 没问题”
- 看见 `iostat` 没打满，就说“肯定不是 I/O”
- 在 Perfetto 里看到线程阻塞，就直接认定“底层盘不行”
- 在 Instruments 里看到很多文件操作，就立刻断言“必须合包”

这些判断都太快了。

## 开始测之前，先把实验条件锁死

I/O 分析最容易被污染的，不是工具本身，而是实验条件。

正式开始前，我更建议至少先锁住这几件事：

### 1. 分清冷缓存和热缓存

要明确这次测的是：

- 冷启动第一次进场
- 第二次热命中
- 长时间运行后的重进场

如果这三种情况混在一起，后面任何数据都很难解释。

### 2. 固定触发动作

尽量让每次都触发同一条加载链，比如：

- 首次进入同一个场景
- 同一个角色第一次出现
- 同一个 streaming chunk 进视野

不要一边测一边改内容路径，否则 trace 只会越来越花。

### 3. 固定设备状态

尤其在移动端，要尽量知道当前是：

- 冷机还是热机
- 前后台切过没有
- 是否有下载、同步、录屏这类后台活动

### 4. 先定义你到底想证伪什么

更稳的问法通常不是：

`是不是 I/O 慢？`

而是：

- 这是不是设备本身的极限问题
- 还是访问模式问题
- 还是后处理问题
- 还是只有某个平台 / 档位 / 热状态才会爆

只要问题先问准，后面的工具顺序就会清楚很多。

## 第一层：用 `fio` 先量“设备能力边界”，不是量你的游戏

`fio` 最适合做的，是先回答一个非常基础但很重要的问题：

`这块设备在某种 I/O 模式下，大概能到什么量级？`

这一步的价值在于先把“设备极限”和“应用层路径”拆开。

### `fio` 最适合什么时候上

通常是你想先知道：

- 这台机器顺序读大概有多少上限
- 小块随机读会不会非常差
- 队列深度拉起来后设备有没有明显改善
- 同样一套资源方案，为什么某些机型天然更危险

### `fio` 不适合回答什么

它不适合直接回答：

- 你的游戏线程为什么在某一帧卡住
- 哪个 bundle 或哪个对象在发 I/O
- 为什么读完以后还是卡

因为 `fio` 测的是“设备和 I/O 模式”，不是“你的加载链”。

### 跑 `fio` 时最重要的一个意识

如果你想测设备本身，不要不小心把页缓存也一起测进来。

`fio` 文档里明确把 `direct=1` 定义成 non-buffered I/O，通常就是 `O_DIRECT`。  
这意味着：

`你是在尽量绕开 page cache，去量更接近设备本体的行为。`

如果你不这么做，很容易测到的是：

- 热缓存
- 内存拷贝
- 文件系统缓存效应

而不是设备本身。

### 一组够用的 `fio` 最小实验矩阵

如果我只想先建立最小直觉，我通常不会一上来跑很多 case，  
而是先测三种：

1. 顺序读，大块，比如 `128k`
2. 随机读，小块，比如 `4k`
3. 随机读，中等块，比如 `64k`

如果你还怀疑设备是不是特别吃队列深度，再加一组：

4. 同样的随机读，对比 `iodepth=1` 和更高 `iodepth`

可以先用类似这样的 job file 思路：

```ini
[global]
filename=/path/to/fio-test.bin
size=2G
runtime=20
time_based=1
direct=1
group_reporting=1

[seqread_128k]
rw=read
bs=128k

[randread_4k]
rw=randread
bs=4k

[randread_64k]
rw=randread
bs=64k
```

如果你要专门比较队列深度，再补一组异步 engine 下的 case。  
`fio` 文档对 `iodepth` 说得很直接：同步 ioengine 下把 `iodepth` 拉高，并不一定真的会生效；在 Linux 上，buffered I/O 也可能让你拿不到想要的 async depth。

也就是说：

`不要看到自己写了 iodepth=32，就默认设备真的被你喂到了 32。`

### 看 `fio` 输出时，先别贪多，先盯三件事

#### 1. `bw`

它更适合回答：

- 顺序大读大概到什么上限
- 大块流式读取有没有明显天花板

#### 2. `IOPS`

它更适合回答：

- 小事务 storm 时设备大概能扛多少请求
- 小随机读是不是天然就弱

#### 3. 延迟，尤其尾延迟

平均值当然有用，  
但如果你关心的是运行时抖动，我更建议看：

- `clat` / `lat`
- 以及更高分位的尾延迟

因为游戏最怕的不是“平均有点慢”，而是：

`偶尔特别慢，而且刚好发生在关键时刻。`

### 怎么用 `fio` 结果做第一轮判断

如果你看到：

- 顺序读很好
- 但 `4k randread` 很差

那通常意味着：

`这块设备不怕大块连续读，但会被海量小事务迅速打回延迟主导。`

如果你看到：

- `iodepth=1` 很一般
- 队列拉高后明显变好

那通常意味着：

`设备更吃并发和队列，不喜欢应用层一个请求一个请求慢慢喂。`

如果你看到：

- 顺序和随机都很差

那就要先怀疑：

- 设备档位本身就低
- 当前环境有争用
- 或者你的测试路径本身不干净

如果你看到：

- `fio` 跑得很好
- 但游戏里还是卡

那也别急着开心。  
这通常只能说明：

`设备不是你唯一的问题。`

## 第二层：用 `iostat` 看“真实运行时设备是不是正在付账”

如果说 `fio` 是“量天花板”，  
那 `iostat` 更像是在问：

`我真实复现问题的时候，设备到底忙不忙，排队重不重，请求长什么样。`

### 最常用的一条命令

在 Linux 上，最常见的起手通常就是：

```bash
iostat -x -y 1
```

这里最值钱的两个点是：

- `-x`：看 extended stats
- `-y`：跳过“开机以来平均值”那一屏，只看当前采样窗口

`iostat` 手册也明确写了这一点：默认第一屏是 since boot，`-y` 才会把它省掉。  
如果不加 `-y`，很多人第一眼就会被旧历史值带偏。

### 真正值得盯的列，不需要太多

不同 distro / kernel / `sysstat` 版本，列名会有一点变化。  
但如果只想做判断，先盯下面这些就够了：

- `r/s`、`w/s`
- `rkB/s`、`wkB/s` 或对应单位列
- `areq-sz`
- `await`
- `aqu-sz`
- `%util`

### 这些列各自更像在回答什么

#### `r/s` / `w/s`

看的是：

`设备每秒到底收到了多少读写请求。`

如果请求数很高，但吞吐不大，  
通常就要怀疑：

`是不是很多小请求在 storm。`

#### `areq-sz`

看的是：

`平均请求到底有多大。`

如果它长期很小，  
而你又在复现首载 / streaming 抖动，通常不是个好信号。

#### `await`

`iostat` 手册对 `await` 的定义很清楚：  
它是 request 被服务的平均时间，里面既包括排队时间，也包括真正被设备处理的时间。

也就是说：

`await` 高，不一定只是设备慢，也可能是队列先堆起来了。`

#### `aqu-sz`

它更接近在看：

`当前平均排队有多深。`

如果它跟 `await` 一起高，通常就是强信号。

#### `%util`

很多人会把它直接理解成“盘满了”。  
这也不完全对。

`iostat` 手册里对 `%util` 的解释是：  
设备在采样窗口里有 I/O request 被发给它的时间占比。  
而它也专门提醒了：对可以并行服务请求的现代 SSD / RAID，这个数字并不总是准确代表极限。

所以：

- 在 HDD、单设备 SATA 盘上，它更接近饱和提示
- 在现代并行 SSD 上，它更像一个参考信号，不是绝对上限

### `iostat` 最容易帮你看出来的三种图景

#### 图景一：真设备压力

你会看到：

- `await` 高
- `aqu-sz` 高
- `%util` 很高

这时更像：

`设备和队列本身正在吃满。`

#### 图景二：小请求 storm

你会看到：

- `r/s` 很高
- 吞吐不一定高
- `areq-sz` 很小

这时更像：

`请求粒度有问题，不是大块顺序流。`

#### 图景三：看起来像 I/O，但设备并不忙

你会看到：

- 游戏卡
- 但 `%util` 不高
- `await` 也不高

这时就要高度怀疑：

- 不是底层设备瓶颈
- 或者 I/O 不是主要瓶颈
- 真正卡的是 I/O 回来后的 CPU / 同步点 / 主线程收口

### 一个特别容易踩坑的地方：别把 `%iowait` 当磁盘利用率

`iostat` 手册对 `%iowait` 的解释是：

`CPU 空闲、同时系统存在 outstanding disk I/O request 的时间占比。`

所以它说的是 CPU 状态，不是设备状态。  
它不能直接替代 `%util`、`await` 或队列指标。

## 第三层：Android 上，用 Perfetto 把“线程时间线”和“系统时间线”接起来

如果你已经知道：

- 设备档位大概怎样
- 真实运行时设备有没有忙起来

下一步最值钱的事情，通常就是把问题放回时间线上。

Perfetto 在 Android 上最有价值的地方，就是它能把：

- app 自己的切片
- 线程调度
- CPU 状态
- 系统 trace

收进同一张时间图里。

Perfetto 官方文档把它定位成：

`collect, analyse and visualize` 客户端 / Android 系统 trace 的统一工具。

### Perfetto 最适合回答什么

它最擅长回答的是：

- 这一帧或这一段加载期里，线程到底在干什么
- 主线程是在等 I/O、等 future，还是在自己做解压 / 反序列化
- worker thread 真正在什么时候读
- I/O 完成以后，后处理又拖了多久

### 最低成本的起步方式

如果你先不想写复杂 config，  
最简单的路径通常是：

1. 打开 `ui.perfetto.dev`
2. 录一段 Android system trace
3. 在问题发生的那 10-20 秒里只做一个固定动作

如果后面要做脚本化或 CI，再走 command line config。

Perfetto 官方文档也提供了两条路径：

- 用 UI 录 Android system trace
- 用 `perfetto` CLI 和 trace config 录更可控的 trace

### 真正让 Perfetto 变得好用的一步：给加载阶段打 marker

如果你能控制 Android app 或 native 层，  
我非常建议给关键阶段补 trace marker，比如：

- `RequestBundle`
- `ReadChunk`
- `Decompress`
- `Deserialize`
- `UploadTexture`

Perfetto 官方文档里也把 `android.os.Trace` 和 `ATrace_beginSection()` 作为 app / native 侧的标准 trace point 入口。

因为只要没有 marker，  
你看到的就更像“线程在忙”，  
但有了 marker，才会更容易回答：

`它到底在忙哪一笔账。`

### 在 Perfetto 里，我最先会看什么

#### 1. 主线程 / 渲染线程是不是在等

如果主线程的大段空白后面跟着等待，  
而 worker thread 同时在发 I/O 或忙于解压，  
那就说明：

`不是设备 alone 的问题，而是等待点和收口点出了问题。`

#### 2. 读什么时候结束，CPU 什么时候才真正放过你

很多项目里，  
你会看到真正的 I/O 活动并没有那么长，  
后面真正拖时间的是：

- 解压
- 反序列化
- 依赖解析
- 激活

这时候如果还继续只盯“磁盘快不快”，就已经偏了。

#### 3. 问题是不是只在某种状态下才出现

比如：

- 冷机没事，热机后开始糟
- 冷缓存第一次差，第二次明显好
- 前后台切回来以后路径变样

Perfetto 的时间线特别适合用来对比这种状态差异。

### Perfetto 最常帮你排除掉的误判

最常见的就是：

`我们怀疑是 I/O，结果 trace 里真正很长的是 I/O 之后那一段 CPU。`

这时候优化方向就会完全变掉：

- 不是继续改读盘
- 而是去拆解压、反序列化、依赖收口和主线程同步

## 第四层：Apple 平台上，用 Instruments 看 app 侧文件活动和 CPU 收口

在 Apple 生态里，我更建议把 Instruments 理解成：

`看 app 自己到底在怎样碰文件系统，以及这些文件活动是不是刚好和 CPU 卡顿重叠。`

### Instruments 最适合从哪几个模板切

不同 Xcode 版本、目标平台和模板命名会有些差异。  
但如果目标是排 I/O，我最常见的组合通常是：

- `File Activity`
- `Time Profiler`
- `Activity Monitor`

Apple 的文档里对 `File Activity` 的描述很直白：  
它会记录 file opens / closes / stat operations；  
还会把 `Reads/Writes`、`File Attributes`、`Directory I/O` 这些活动拆开给你看。

这对定位“小文件 storm”和“名字 / 目录 / 属性操作过多”特别有用。

### 我在 Instruments 里最先看什么

#### 1. `open / close / stat` 是不是在狂刷

如果你看到：

- bytes 并不夸张
- 但 file open / close / stat 非常多

那通常更像：

`不是盘带宽不够，而是文件事务太碎。`

#### 2. `Directory I/O` 是不是异常多

这通常是在提醒你：

- 路径和目录活动很重
- 名字查找可能在前面已经先结账

这和前面讲的路径解析、目录项和元数据那一层正好能对上。

#### 3. `Reads/Writes` 和 `Time Profiler` 的时间关系

最值钱的不是单看 file activity，  
而是把它和 `Time Profiler` 的 CPU 时间线一起看。

如果你看到：

- 文件活动已经结束
- 但 CPU 还在主线程或 worker 上烧很久

那问题就更像：

`I/O 已经回来了，但后处理还没完。`

### Instruments 最适合帮你确认什么

它特别适合帮你确认：

- 问题是不是“大量独立文件事务”
- 是不是目录和属性操作被低估了
- 读取行为是不是过碎
- app 卡顿是不是主要发生在文件活动之后

Apple 旧的性能文档里也有一句很实用的建议：

> 用 Instruments 建立 baseline，再一轮一轮修，确认改动是不是真的让文件活动更有效率。

这其实就是 I/O 排查最该有的习惯。

## 最后把四个工具串成一条真正能用的排查链

如果你问我现实项目里最稳的一条顺序是什么，我通常会这样走：

### 1. 先用 `fio` 知道设备能力边界

先回答：

- 这台机器顺序读怎样
- 小块随机读怎样
- 它是不是天然就对碎访问敏感

### 2. 再用 `iostat` 看真实复现场景时设备有没有真的在忙

先回答：

- 设备有没有打满
- 请求是不是很碎
- 队列有没有明显拉起来

### 3. 然后用 Perfetto 或 Instruments 看时间线

再回答：

- 主线程在等什么
- I/O 回来后是不是 CPU 还在继续结账
- 问题到底是设备、访问模式，还是后处理

### 4. 最后再决定怎么改

这时你的优化动作才会更稳地落到正确层：

- 真设备极限问题：
  更偏向减少随机读、增大块、提高局部性、预取、分档

- 小事务和访问模式问题：
  更偏向合包、chunk 化、减少热路径独立文件数、减少 `open / close`

- I/O 后处理问题：
  更偏向解压、反序列化、依赖收口、主线程同步、GPU upload

## 几种很常见的结论模式

### 模式一：`fio` 随机读弱，`iostat` 的 `await` / 队列也高

这更像：

`设备和访问模式正在一起把问题放大。`

更稳的动作通常是：

- 减少热路径随机小读
- 合并请求
- 提高局部性
- 在弱设备档位上降低 streaming 激进度

### 模式二：`fio` 其实不差，但 `iostat` 显示大量小请求

这更像：

`不是设备不行，是应用把请求切得太碎。`

更稳的动作通常是：

- 重做资源组织
- 降低文件数
- 减少名字查找和 `open / close`

### 模式三：`iostat` 不忙，但 Perfetto / Instruments 里 CPU 很长

这更像：

`你以为在排 I/O，实际上在排加载链后半段。`

更稳的动作通常是：

- 优化解压
- 优化反序列化
- 拆依赖收口
- 把后处理移出关键帧

### 模式四：只在低端机或热机后恶化

这更像：

`移动端系统条件正在放大原本就脆弱的访问模式。`

更稳的动作通常是：

- 对 `eMMC / 弱 UFS` 档位减少运行时碎补读
- 增大 preload
- 减小工作集
- 用冷机 / 热机 / 长时运行三套 case 分别验证

## 这一篇最该带走的结论

`I/O 分析真正值钱的，不是“会不会用 fio / iostat / Perfetto / Instruments”，而是知道每个工具只应该回答哪一层。`

一旦你把这四个工具串起来，很多“看起来像加载慢”的问题就会开始自动分层：

- 是底层设备极限
- 是访问模式太碎
- 是文件系统和缓存行为
- 还是 I/O 后面的那条加载链

而只要层分对了，后面的优化方向通常就不会差太远。
