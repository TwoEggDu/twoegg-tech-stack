> 如果只用一句话概括这篇文章，我会这样说：很多游戏里的“内存问题”并不是因为总量立刻爆了，而是因为系统一直在用一种对帧时间很不友好的方式使用内存。

前面几篇文章，我一直在往同一个方向收：

- 为什么某些操作会慢
- 一帧到底是怎么完成的
- 什么事不能在什么时候做
- 为什么大整文件通常比很多小散文件更稳
- 为什么读盘完成不等于资源可用

如果顺着这条线继续往下走，接下来最该讲的一层，就是内存。

但这里有一个很常见的误区：

很多团队一提到内存，脑子里第一反应只有一个问题：

`够不够。`

这个问题当然重要。  
如果真的不够，进程会被系统回收、崩溃、OOM，或者被迫进入非常糟糕的回收状态。

但在大量真实项目里，真正更常见、也更折磨人的问题不是“绝对不够”，而是：

- 明明总量看起来还行，为什么还是会卡
- 明明场景不是特别大，为什么切场景时总会抖
- 明明一次分配不大，为什么长玩后越来越不稳
- 明明资源压下来了，为什么首次打开 UI、首次进战斗还是会掉帧

这些问题的共同点是：

`它们更像内存行为问题，而不是单纯的内存容量问题。`

所以这篇文章我想讲透的是：

`内存不是仓库面积，而是运行时工作集。`

真正影响体验的，往往不是“仓库有没有装满”，而是你是不是在用一种混乱、波动很大、会打断帧预算的方式搬运和管理这个工作区。

## 先把一个概念摆正：RAM 更像工作台，不像仓库

磁盘、SSD、UFS 更像仓库。  
它们擅长长期存放数据。

RAM 在游戏运行时扮演的角色更像工作台：

- 当前帧要访问的对象在这里
- 正在运行的系统状态在这里
- 资源被激活后的工作集在这里
- GPU 上传前的很多准备数据也在这里

工作台和仓库最大的差别在于：

- 仓库更怕装不下
- 工作台更怕一直乱

如果工作台总是在这些状态里来回切换：

- 突然堆很多东西上来
- 刚放上来又立刻扔掉
- 一会儿清空一大片，一会儿又塞满
- 刚准备好的内容下一秒又被换走

那即使“总面积还够”，整个系统也会变得很不稳定。

这就是为什么很多游戏里的内存问题，不会先表现成 OOM，而会先表现成：

- 帧时间抖动
- 场景切换尖峰
- UI 首开卡顿
- 长玩后变差
- 某些设备上明显更容易出事

## 很多卡顿，真正伤人的不是“占了多少”，而是“怎么进、怎么出”

如果只看内存占用截图，很容易有一种错觉：

`内存总量没爆，应该没啥问题。`

但运行时更敏感的常常是下面这些事：

- 分配发生得太频繁
- 释放发生得太集中
- 生命周期边界太剧烈
- 临时对象太多
- 资源常驻和淘汰策略来回抖
- 页面缺失和缓存失效在关键时刻出现

也就是说，内存问题最危险的地方，很多时候不是“占用高”，而是：

`行为不平滑。`

如果每帧都在做很多小而碎的分配与释放，或者每到关键节点就突然来一波大规模对象激活和销毁，那系统对帧预算的破坏会非常明显。

这也是为什么我更愿意把很多项目里的 memory bound 理解成：

`工作集管理不稳定。`

## 临时对象为什么这么容易变成性能问题

很多程序员第一次踩内存坑，通常不是踩在“超大资源”上，而是踩在看起来不起眼的临时对象上。

因为临时对象从业务语义看太无害了：

- 临时拼一段字符串
- 临时组一份列表
- 临时构建一棵 UI 数据
- 临时生成一批特效参数

单次看起来都不大。  
但一旦这些事发生在高频路径里，比如：

- 每帧
- 每次输入
- 每次角色更新
- 每次 UI 刷新

它就会慢慢变成一类特别典型的问题：

- 分配频率高
- 回收压力高
- 缓存局部性差
- 很容易触发 GC 或分配器尖峰

这类问题最容易骗过人的地方在于：

`每一处都不大，但总量和节奏很糟。`

所以很多时候，真正危险的不是“大对象”，而是：

- 高频临时对象
- 短命对象风暴
- 热路径里习惯性地产生中间结果

对运行时来说，这就像你在工作台上不停地扔下小纸片、揉掉、再扔、再揉。  
每张纸都不大，但整个台面会变得非常乱。

## 生命周期尖峰，才是很多内存卡顿真正的源头

这件事和前面“什么事不能在什么时候做”那篇是连着的。

很多内存问题，根子都不在“单个对象太大”，而在：

`很多对象在同一个时刻一起活过来，或者一起死掉。`

比如这些高风险时刻：

- 进战斗
- 切场景
- 打开大 UI
- 一波敌人同时生成
- 大量 streaming 内容一起进场

这些时刻里，系统常常会同时发生：

- 分配内存
- 初始化对象
- 建立引用
- 激活组件
- 触发生命周期回调
- 将来再在另一个节点集中释放

这类问题特别难受，因为从业务角度看你只是做了一个“切换状态”的动作。  
但从运行时角度看，你其实在一个很短的时间里制造了一整波内存行为尖峰。

所以很多切场景卡顿、进副本顿一下、大 UI 首开抖一下，表面上像加载问题，底层其实都和生命周期尖峰紧密绑定。

## GC 真正可怕的，不是“会回收”，而是“什么时候回收”

一提到内存，很多团队第一反应会直接想到 GC。  
这没错，但 GC 也常常被说得太粗。

GC 本身不是问题。  
真正的问题是：

- 你是不是制造了太多短命垃圾
- 回收是不是恰好落在关键帧
- 回收是不是和别的重操作撞在一起

如果一套系统平时就在疯狂制造短命对象，那 GC 最后只是替你结账而已。

所以看见 GC 尖峰时，最不该做的事是只抱怨：

`GC 又卡了。`

更应该问的是：

- 为什么会在这个时刻积累这么多垃圾
- 这些垃圾是哪些路径制造出来的
- 它们是不是本来就不该出现
- 回收是不是刚好和对象激活、资源加载、UI 切换这些事情撞上了

也就是说，GC 不是无缘无故来找你麻烦。  
它经常只是把前面那些“看起来不大的临时对象习惯”放大成一次真正的体验问题。

## 页面缺失和工作集抖动，为什么长玩后会更明显

有一类问题特别典型：

- 刚进游戏还好
- 玩了一会儿开始抖
- 切几次场景后明显变差

这类问题不一定是热，也不一定是 GPU。

它常常和工作集变化有关。

简单说就是：

- 活跃内存不断变化
- 某些资源刚进来又被换走
- 某些页面不再常驻
- 需要时又得重新拉回来

于是系统就容易在关键时刻出现：

- 页面缺失
- 资源重新进场
- 局部缓存命中变差
- 整体工作集越来越不稳定

这类问题最容易被误判成“偶发卡顿”或者“某次加载慢”。  
实际上，它常常反映的是：

`系统对常驻、淘汰和重进场的策略不够平滑。`

这也是为什么长玩后的问题，很难只靠“压缩资源体积”解决。  
如果工作集管理本身不稳定，压下去一点总量并不会自动让行为变稳。

## 常驻与淘汰，不是越省越好，而是要避免来回抽搐

很多团队讲内存策略时，容易把目标说成：

`尽量少常驻。`

这句话对一半。

因为如果你什么都常驻，内存压力肯定会越来越大。  
但如果你什么都不敢常驻，系统又会在关键时刻不停追资源、追对象、追上传。

真正的问题不是“常驻多还是少”，而是：

`常驻和淘汰是不是在服务实际访问模式。`

最怕的是一种很常见的策略失败：

- 这个资源刚被淘汰
- 过几秒又要用
- 用完又被淘汰
- 再过一会儿又得重新拉回来

这种来回抽搐式的管理，往往比单纯“占用略高一点”更伤体验。

因为它意味着：

- I/O 反复发生
- 解析反复发生
- 对象和 GPU 资源准备反复发生
- 帧时间波动被放大

所以从游戏体验角度看，更成熟的目标通常不是“极限省内存”，而是：

`在可接受总量内，让工作集尽量稳定。`

## CPU 缓存局部性和 RAM 行为，为什么会一起变成帧时间问题

很多人一说内存问题，就只想到 RAM 总量。  
但 CPU 视角下，还有另一个很现实的层面：

`你访问内存的方式，决定了 CPU 跑这段逻辑到底顺不顺。`

如果数据布局很乱、访问模式跳来跳去、对象分散得很碎，那即使总量不大，CPU 也会：

- 更难命中缓存
- 更频繁地去更慢的层级取数据
- 更容易在热路径上浪费时间

这类问题为什么重要？

因为它会让很多人产生一种错觉：

`逻辑明明不复杂，为什么 CPU 还是会抖？`

答案常常不是算法本身太重，而是：

- 数据太碎
- 布局太乱
- 访问太散
- 生命周期太频繁变化，导致局部性持续被破坏

所以内存行为问题并不只属于“资源管理”，它也深深影响着纯 CPU 侧的帧时间。

## 用一个大 UI 首开的例子，把内存行为问题串起来

这类场景特别适合说明问题。

玩家看到的现象是：

`第一次打开背包或商城时，会顿一下。`

真实系统里可能同时发生的是：

1. UI 资源第一次被拉进工作集
2. 图集、字体、配置、脚本数据被解析
3. 节点树和运行时对象被创建
4. 布局和绑定开始计算
5. 一批临时对象和字符串处理中间结果被制造出来
6. 某些纹理和材质状态第一次上传 GPU
7. 回调、动画、音效和脚本逻辑一起启动

如果只把这件事叫成“UI 有点重”，你就会什么也优化不明白。

只有把它翻译成：

- 首次资源进场
- 生命周期尖峰
- 临时对象风暴
- GPU 首次成本

这几层，你才知道该怎么拆：

- 哪些资源可以提前准备
- 哪些节点可以延迟创建
- 哪些子页面可以按需展开
- 哪些字符串和列表不该在热路径里反复构造

## Unity 和 Unreal 里，这些问题通常怎么显形

### Unity 里更常见的表现

Unity 项目里，内存行为问题常常会长成下面这些样子：

- GC 尖峰
- `Instantiate / Destroy` 高频触发
- UI 首开、切页、列表刷新时明显抖动
- 某些逻辑路径每帧产生大量短命对象
- Addressables / AssetBundle 读完以后，激活和运行时构建阶段卡顿

这些现象表面上各不相同，但底层往往都指向：

- 临时对象多
- 生命周期波动大
- 收口时机太集中
- 工作集进出场不稳定

### Unreal 里更常见的表现

Unreal 里，这类问题通常会更明显地长在线程和对象生命周期边界上：

- Game Thread 在对象激活和初始化阶段有尖峰
- 流送、激活和渲染准备在同一时间窗口收口
- 某些对象链条第一次进入场景时成本很高
- 常驻和 streaming 内容切换时工作集波动明显

它们看起来和 Unity 不一样，但本质仍然是：

`运行时内存行为不够平滑。`

也就是说，不是“哪一个引擎更有内存问题”，而是这类问题在不同引擎里会以不同形状露出来。

## PC 和手机在内存行为上的风险差异

### PC 更容易让你误以为“内存够大就没事”

PC 常常有更大的 RAM，总量压力确实没那么紧。  
但这也很容易让团队松懈，最后把一些本不健康的运行时习惯养出来：

- 高频临时对象
- 过度依赖大工作集兜底
- 觉得某些尖峰“反正开发机扛得住”

结果一到低配机器、后台程序复杂的环境、或者长时间运行场景，问题就开始暴露。

所以 PC 上最容易的误判是：

`总量够，所以内存不是问题。`

很多时候，真正的问题仍然是行为而不是容量。

### 手机更容易把不稳定的内存行为放大成真实事故

手机上的风险则更直接：

- 可用内存更紧
- 系统回收更激进
- 热和功耗会压缩 CPU/GPU 预算
- 前后台切换更频繁

这意味着不稳定的工作集管理，在手机上更容易直接表现成：

- 更频繁的资源重进场
- 更明显的生命周期尖峰
- 更高概率的回收和重建
- 更差的持续表现

所以手机上更难骗自己。  
你如果总是在让工作集抽搐，它很快就会用体验问题把你打回来。

## 一套更实用的内存行为排查问题

如果你现在面对的是“总量看起来没爆，但体验还是不稳”的问题，我建议先问下面这些问题。

### 1. 这是容量问题，还是行为问题

先分清：

- 是不是已经接近 OOM
- 还是总量还行，但帧时间在抖

### 2. 高风险时刻有没有生命周期尖峰

切场景、开 UI、进战斗、首播技能时，是否有大批量对象同时创建或销毁。

### 3. 热路径里有没有很多短命对象

如果每帧、每次输入、每次 UI 刷新都在反复分配临时对象，先盯这里。

### 4. GC、回收或分配器尖峰是不是踩在关键帧

如果是，就不要只盯“GC 花了几毫秒”，要回头找前面是谁制造了垃圾。

### 5. 常驻与淘汰策略是不是在来回抽搐

如果同一批资源不断进出场，问题通常不在“某次加载”，而在策略本身。

### 6. CPU 侧数据访问是不是太碎

如果逻辑不复杂却 CPU 抖动明显，要怀疑数据布局和局部性。

### 7. 这类成本能不能前置、池化、分帧、降波动

如果能，就不要接受“它就是会这样”。

## 我的结论

如果把这篇文章压缩成一句话，我会这样总结：

> 游戏里的很多内存问题，不是“内存立刻不够了”，而是系统正在以一种对帧时间极不友好的方式制造、移动、回收和重建工作集；真正伤体验的，往往不是总量本身，而是这些行为发生得太频繁、太集中、太不稳定。

理解这一点之后，你再去看 GC 尖峰、UI 首开抖动、切场景顿、长玩后变差、对象生命周期波动，很多问题就不再会被粗暴地归类成“内存不够”。

你会更容易开始问：

- 是谁在制造短命对象
- 是谁在关键时刻拉起大量生命周期变化
- 是谁让工作集不断来回抖动
- 是谁把可前置的准备拖到了当前帧

当这些问题开始被持续追问时，内存优化才真正从“省一点 MB”变成了“让运行时稳定下来”。
