前三篇分别看了 Unity DOTS 在 RTS/生存游戏中的落地、Unreal Mass 在开放世界 NPC 中的应用、以及 Flecs/EnTT 在跨平台 C++ 项目中的实践。读完会发现一个共同特征：没有任何一个团队是"因为 ECS 好所以用 ECS"，而是因为遇到了特定的性能或架构瓶颈，才选择了某条路。

这篇的任务是把散落在三篇案例里的判断依据收拢成一张决策地图。目标不是告诉你哪个框架更好，而是帮你在具体约束下做出更合理的选择。

---

## 先问自己的四个问题

在看任何推荐之前，先明确自己的处境。这四个问题的答案基本决定了你的选型空间。

**问题一：我用的是哪个引擎？**

引擎绑定程度是最硬的约束。如果你在用 Unity，DOTS 是最自然的选择——它和 Burst 编译器、Job System 深度集成，脱离这个生态等于放弃引擎提供的大半优化工具链。如果你在用 Unreal，Mass Framework 已经内置，不需要引入外部依赖，且和引擎的 Niagara、StateTree 有配套接口。如果你在写自研引擎或者纯 C++ 跨平台游戏，引擎绑定不是约束，反而是自由选择的机会。

**问题二：我的性能瓶颈在哪里？**

ECS 解决的核心问题是大量同构对象的批量处理——通过内存连续布局减少 cache miss，通过 Archetype 分组实现 SIMD 友好的并行计算。如果你的瓶颈是"5000 个士兵每帧更新位置和 AI 状态"，ECS 是对症的。如果你的瓶颈是"一个 BOSS 的状态机逻辑太复杂"，ECS 不是解法，你需要的是更好的状态机设计。

**问题三：是否需要服务端部署或跨平台？**

服务端 ECS 和客户端 ECS 的约束完全不同。服务端需要多 World 隔离（每个游戏房间独立运行），需要剥离渲染层，需要考虑内存占用和 GC 压力。跨平台 C++ 项目需要框架足迹小、无运行时依赖。这两个需求都会把选型往 Flecs 或 EnTT 方向推。

**问题四：团队有多少 ECS 经验？**

DOTS 的学习曲线比很多团队预期的陡。Entities 1.x 的 Baking 系统、Safety System、Structural Change 的限制，每一个都需要时间消化。Mass Framework 更陡——工具链不成熟，官方文档稀少，很多东西需要读源码。EnTT 相对平缓，API 设计直觉友好。如果团队完全没有 ECS 经验，在原型阶段就上完整 DOTS 往往意味着大量时间花在框架调试上而不是游戏逻辑上。

## 按引擎绑定程度分叉

把四个问题的答案组合起来，主要路径如下：

```
用 Unity          → DOTS（Entities 1.x + Burst + Job System）
用 Unreal         → Mass Framework（内置，无需额外依赖）
自研 / 跨平台 C++ → Flecs 或 EnTT
游戏服务端        → Flecs / EnTT / Bevy ECS（Rust）/ Arch（C#）
```

这是第一层分叉。每条路内部还有进一步的判断。

---

## Unity 项目：什么时候上 DOTS，什么时候不上

**适合上 DOTS 的场景：**

同类对象数量超过 100，且这些对象每帧都需要高频更新——位置、速度、AI 状态、碰撞检测。RTS 的单位系统、生存游戏的僵尸群、弹幕游戏的子弹池，都是典型场景。DOTS 在这类场景能把每帧更新从毫秒级压到微秒级。另一个强信号是物理仿真密集型需求，Unity Physics（基于 DOTS）在大量物体同时参与仿真时有明显优势。

**不适合上 DOTS 的场景：**

UI 逻辑、单例系统、玩家角色控制、复杂的叙事状态机。这些系统对象数量少、逻辑异构、和 Unity 传统 API（UnityEngine.Object、协程、事件系统）深度耦合。强行用 ECS 表达这些逻辑会引入不必要的间接层，代码可读性下降，收益为零。

**混合架构是现实选择：**

没有大型商业项目是纯 DOTS 的。常见分层是：ECS 做仿真层（战斗单位、弹道、物理），MonoBehaviour 做表现层（角色动画、特效触发、UI 绑定）。两层之间通过 Entity 的 Component 数据单向流动，渲染层从 ECS 读位置和状态，不反向写入。这个架构在 Megacity Metro 等 Unity 官方示例里有完整展示。

## Unreal 项目：Mass vs 传统 Actor

**适合上 Mass 的场景：**

NPC 数量超过 500，需要 LOD 分级仿真（远处 NPC 只更新位置，近处才跑完整 AI），或者在做人群系统、交通系统这类天然适合批量处理的功能。Matrix Awakens 技术演示里的城市人群就是 Mass 的标志性案例。

**不适合上 Mass 的场景：**

NPC 数量少于 100，或者 NPC 逻辑高度异构（每个 NPC 的行为树差异很大）。更重要的是，Mass 和 GAS（Gameplay Ability System）、BehaviorTree 的配合存在架构边界问题——GAS 和 BT 都是为 Actor 设计的，把它们嫁接到 Mass Entity 上需要额外的桥接层，维护成本不低。如果你的 AI 逻辑已经深度依赖 BehaviorTree，上 Mass 反而会制造麻烦。

**关于学习成本：**

Mass 的学习曲线比 DOTS 更陡。DOTS 至少有相对完整的官方文档和大量社区资源。Mass Framework 的官方文档截至目前仍然稀少，很多核心机制（Processor 执行顺序、Fragment 的生命周期、Shared Fragment 的使用边界）需要读 Unreal 源码才能搞清楚。团队评估时需要把这个学习成本计入。

---

## 自研引擎 / 跨平台：Flecs vs EnTT

这两个都是成熟的纯 C++ ECS 库，无运行时依赖，可以嵌入任何项目。选择依据在于你需要框架提供多少功能。

**选 Flecs 的理由：**

需要内置调度器来管理 System 执行顺序和并行依赖时，Flecs 的 Pipeline 系统开箱即用。需要 Observer 机制（监听 Component 添加/删除事件）时，Flecs 原生支持。需要多 World 隔离（服务端每个房间一个 World）时，Flecs 的 World 是一等公民。此外 Flecs 提供 REST API 和 Web Explorer，可以在运行时查询和调试 ECS 状态，这在大型项目中是很实用的工具。

**选 EnTT 的理由：**

需要最小框架足迹、完全控制执行顺序时，EnTT 不强加任何调度抽象——你用 Registry，自己决定什么时候调用什么。已有调度系统（引擎主循环、任务图）不想引入另一套调度器时，EnTT 的单文件头文件库特别合适。Minecraft 的 C++ 服务端曾使用 EnTT，正是看中了它的轻量和可控性。

两者都不是"更好"，是"在什么约束下更合理"的关系。

## 服务端 ECS 的特殊考虑

服务端 ECS 和客户端 ECS 面对的问题有本质差异，值得单独列出。

**渲染层的处理：** DOTS 本身可以 headless 运行，剥离 Unity.Rendering 和 Unity.Physics 的渲染部分后是可以在服务端部署的。但这个剥离过程需要手动管理，且 Unity 对服务端授权的商业策略需要提前确认。

**多 World 隔离：** 服务端通常需要每个游戏房间运行在独立的 World 里，便于内存隔离和横向扩展。Flecs 的多 World 支持是内置的，DOTS 需要手动管理多个 World 实例，EnTT 的 Registry 天然就是轻量的 World 容器。

**Rust 生态：** Bevy ECS 在服务端有越来越多的实际使用，特别是对安全性和并发正确性有高要求的场景。Bevy ECS 可以独立于 Bevy 渲染层使用，作为纯服务端 ECS 运行时。

**C# 生态：** Arch 是一个 C# ECS 库，在不想引入 DOTS 完整依赖但需要 C# 服务端 ECS 时是一个选项。

服务端 ECS 的完整讨论涉及网络同步、状态快照、回滚机制等更多问题，这些留给「高性能游戏服务端 ECS」系列展开。

## 综合决策矩阵

| 场景 | 推荐方案 | 次选 | 回避 |
|------|----------|------|------|
| Unity 大量同构 NPC（100+） | DOTS + Burst | - | 纯 MonoBehaviour |
| Unreal 大量 NPC（500+） | Mass Framework | - | 全 Actor |
| 跨平台 C++ 游戏（需要调度器） | Flecs | EnTT | 引擎绑定方案 |
| 跨平台 C++ 游戏（极简嵌入） | EnTT | Flecs | 引擎绑定方案 |
| 游戏服务端（多 World） | Flecs | Bevy ECS | DOTS（渲染依赖复杂）|
| 自研引擎渲染批处理 | 手工 Job Graph | Flecs | 引擎绑定方案 |
| 小团队 / 原型阶段 | 传统 OOP | Managed ECS | 完整 DOTS（学习成本）|
| Unity NPC < 100 | MonoBehaviour | - | DOTS（收益不覆盖成本）|

矩阵不是教条，是起点。实际项目里经常出现"场景 A + 场景 B 混合"的情况，这时候通常的答案是混合架构，而不是二选一。

## 值得上 ECS 的最小对象数量

这是一个实际问题，却很少有人直接给出数字参考。

以下是经验值，基于 x86/ARM 现代处理器的 L1/L2 cache 行为：

- **< 100 个同类对象：** 传统 OOP 的 cache miss 通常不会成为性能瓶颈。对象数量太少，即便内存布局不理想，每帧的总访问量也在 cache 容量以内。引入 ECS 的架构成本远大于收益。

- **100 ～ 1000 个同类对象：** 收益开始显现，特别是当每个对象有多个 Component 需要同时访问时。如果帧预算紧张，这个区间值得做性能剖析后再决定是否引入 ECS。

- **> 1000 个同类对象：** ECS 的批量处理优势明显。内存连续布局 + Burst/SIMD 能把这个规模的更新从数毫秒压到数十微秒。这个量级下不用 ECS 基本是在给自己制造困难。

这个阈值不是绝对的。如果同类对象每帧只做极简计算（比如只移动位置），100 个对象用传统 OOP 也没问题。如果每帧需要做视野检测 + 路径代价计算 + 状态机更新，50 个对象就可能值得考虑批量处理。性能剖析永远优先于经验值。

## 行业对比四篇的完整图景

至此，DOD 行业横向对比的四篇到这里收尾。回看整个系列覆盖的问题空间：

第一篇（Unity RTS/生存）确认了 DOTS 在同构对象批处理上的实际收益，以及混合架构在商业项目中的必然性。第二篇（Unreal 开放世界）展示了 Mass Framework 在人群仿真中的上限，同时直面了它的工具链不成熟问题。第三篇（跨平台 C++）对比了 Flecs 和 EnTT 在无引擎约束下的差异，服务端场景的特殊需求开始浮现。这一篇把判断依据收拢成可操作的决策框架。

DOD 这个问题空间的全貌是：它不是一种"更先进的编程范式"，而是在特定性能约束下，对内存布局和执行模型做出的一组有意义的权衡。权衡有代价——架构复杂度上升、学习曲线变陡、和既有系统的集成需要桥接层。权衡也有收益——在正确的场景下，性能提升是数量级的。

选型的本质是：在你当前的约束下，哪条路的权衡比更高。希望这四篇给了你做这个判断的足够依据。
