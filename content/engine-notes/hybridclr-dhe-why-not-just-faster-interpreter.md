> 这篇同样不写“商业版实现揭秘”。能确定的部分，只来自官方公开文档已经明说的行为；凡是落到 runtime 细部怎么拼起来，我都会按“高可信推断”来写，而不是把猜测写成源码事实。

这是 HybridCLR 系列第 11 篇，也是第二篇继续往高级能力边界下探的文章。  
如果说上一篇 [HybridCLR Full Generic Sharing｜为什么它不是补充 metadata 的升级版]({{< relref "engine-notes/hybridclr-full-generic-sharing-why-not-metadata-upgrade.md" >}}) 解决的是 generic 覆盖率上限，那这一篇要讲的 `DHE`，解决的就是另一类更工程化的问题：

`当你不只是新增热更逻辑，而是要直接增删改一个已经 AOT 进包的程序集时，能不能别让整块逻辑都退回纯解释执行。`

这件事如果不单独拆开，读者很容易把 `DHE` 理解成一句模糊的话：

`解释器更快一点。`

但官方公开定义其实比这精确得多。

## 这篇要回答什么

这篇主要回答 6 个问题：

1. `DHE` 解决的真实问题是什么，为什么它不是“解释器更快一点”。
2. 普通热更程序集解释执行，和 `DHE` 的函数级差分执行到底差在哪。
3. `dhao` 文件为什么会成为 `DHE` 的核心概念。
4. 如果只按公开资料和现有 runtime 边界推断，`DHE` 大概率碰到哪些运行时层。
5. 它在项目里真正换来了什么，又额外带来了哪些限制和代价。
6. 什么时候应该认真考虑 `DHE`，什么时候反而该克制。

## 先给一句总判断

如果先把这篇压成一句话，我的判断是：

`DHE` 解决的不是“把 interpreter 调得更快”，而是“让同一个已经 AOT 进包的程序集，在运行时按函数是否发生变化，分别走原生 AOT 路径或最新解释执行路径”。`

只要这句话成立，后面很多行为就都顺了：

- 为什么它强调的是对 AOT dll 的增删改
- 为什么它会说未修改函数继续走 AOT
- 为什么它需要一个离线计算出来的 `dhao` 文件
- 为什么它有一组看起来很奇怪、但其实很合理的加载顺序和资源限制

也就是说，`DHE` 真正改的不是“解释器速度曲线”，而是：

`AOT 与 interpreter 之间的分流策略。`

## 先把真正的问题立住：普通热更主线并不直接回答“修改现有 AOT 程序集”这件事

前面整条社区版主线，默认回答的问题更像是：

- 新增一个热更程序集，能不能在 `IL2CPP` runtime 里装起来
- 这个程序集里的方法能不能解释执行
- 缺的 AOT metadata 能不能补回来
- 跨 AOT / native 边界时桥接能不能补齐

这套主线非常强，但它天然更偏向：

`新增热更程序集，把变化组织到热更模块里。`

一旦你真正想做的是下面这类事情：

- 直接修改一个已经进主包的 AOT 程序集
- 只改动其中一小部分类型和函数
- 又不想让没动的部分也一起掉进解释器慢路径

那你关心的已经不是“热更程序集能不能跑”，而是：

`同一个程序集内部，变更部分和未变更部分能不能走不同执行路径。`

这就是 `DHE` 想解决的那一层。

## 官方已经明确公开了什么

这部分尽量只说官方文档已经明说的内容，不做推断。

### 0. `DHE` 是更高层的商业版能力

官方文档已经明确标注：  
`DHE` 不属于社区版主线，而是更高层的商业版能力边界。

### 1. `DHE` 的完整名字就是 Differential Hybrid Execution

官方定义已经把核心语义写出来了：  
它允许对 AOT dll 做任意增删改，并且在运行时：

- 新增或变更的类、函数走解释执行
- 未改动的类、函数继续走 AOT

这已经足够说明一件事：

`DHE` 的重点从来不是“让所有热更代码都更快”，而是“按变更粒度做混合执行”。`

### 2. `DHE` 对应的程序集，同时存在于包体和热更版本里

官方文档明确说过：

- 被标记为 `DHE` 的程序集也会打进主包
- 运行时还会再加载最新热更 dll

这就意味着 `DHE` 不是普通意义上的“只加载一个新的热更程序集”，而是天然带着一组“双版本”关系：

- 包体里有一个 AOT 版
- 运行时又来了一个最新热更版

后面的“函数级差分选择”，就是在这两者之间建立起来的。

### 3. `dhao` 文件是 `DHE` 的核心产物

这件事官方讲得非常直白。  
`dhao` 文件里保存的是离线计算出来的“变更了哪些类型、哪些函数”的结果。  
运行时执行 DHE 程序集函数时，会直接依赖这个文件来决定：

- 继续调原来的 AOT 函数
- 还是改走最新代码的解释执行

这说明 `dhao` 不是附属文件，而是 `DHE` 这条路线的判断依据本身。

### 4. `dhao` 依赖打包时 AOT dll 和最新热更 dll 的准确配对

官方还明确说了另一件很关键的事：

- `dhao` 是把“打包时生成的 AOT dll”和“最新热更 dll”做离线比对后算出来的
- 如果没有 `dhao`，运行时要么得额外带原始 AOT dll，要么在线计算变化，成本会非常高

所以 `DHE` 的本质并不是“运行时临时猜谁变了”，而是：

`离线先算好变化集，运行时只负责按这个变化集分流。`

### 5. 官方也公开写了它的限制边界

至少这些限制是文档已经明说的：

- 对应的 DHE AOT 程序集代码，不能在加载 DHE 热更代码之前执行
- 因此前置基础库一类场景不适合 DHE
- `[InitializeOnLoadMethod]` 和 `Script Execution Order` 不支持
- DHE 脚本不能挂到打进包体的资源上，包括 `Resources`
- 不能通过热更给 DHE 程序集新增 `extern` 函数

这些限制看起来零散，实际上都在指向同一件事：

`DHE` 对“加载顺序”和“边界一致性”的要求，比普通热更主线更强。`

## 为什么它不是“普通解释执行更快一点”

到这里，已经可以把最常见的误解正面打掉了。

普通热更主线的默认模型更像这样：

- 新程序集加载进来
- 这个程序集里的方法进入 interpreter 主链
- 需要 metadata 的地方补 metadata
- 需要桥接的地方补 bridge

它当然也可以做性能治理，但前提通常还是：

`这个热更程序集里的代码，本质上还是一整块解释执行语义。`

而 `DHE` 不是这个意思。  
官方公开语义里，它更像这样：

- 同一个程序集，包体里已经有一份 AOT 代码
- 运行时再加载最新 dll
- 每次进到 DHE 程序集的方法时，先判断它是不是变化过
- 没变化就继续调原 AOT 实现
- 有变化才切到最新解释执行路径

你会发现，这里最核心的词不是“更快”，而是：

`按函数做执行路径选择。`

所以最准确的判断其实是：

`DHE` 不是普通解释执行的性能优化补丁，而是“对已进包程序集做函数级混合执行”的 runtime 分流机制。`

## `dhao` 为什么会成为核心概念

如果你只从“性能更好”这个角度看 `DHE`，很容易忽略 `dhao`。  
但如果你从 runtime 约束看，它反而是最关键的那个部件。

原因很简单。  
`DHE` 要成立，运行时至少得回答三个问题：

1. 当前调用的这个类型或函数，和打包时相比到底变没变。
2. 如果变了，应该去哪里拿最新逻辑。
3. 如果没变，怎样继续安全地落回原来的 native AOT 路径。

这些判断如果都放到运行时现算，代价会很高。  
而且它还会让运行时不得不持有更多对照信息。

所以官方才会把 `dhao` 设计成离线产物：

- 打包时的 AOT dll 是一个基准面
- 最新热更 dll 是新的目标面
- 二者差分后的变化集被固化进 `dhao`
- 运行时只吃这个变化集结果

把这层想明白之后，你对 `DHE` 的理解就会很稳：

`DHE` 的关键，不是“解释器更能打”，而是“离线差分 + 运行时按差分分流”。`

## 如果只按公开资料和现有边界推断，它大概率碰到哪几层

下面这部分开始进入“高可信推断”，不把猜测写成事实。

### 高可信推断一：它一定碰方法身份映射和分发入口

既然 `DHE` 的核心语义是“同一个程序集内部，不同函数走不同执行路径”，那 runtime 就必须先稳定识别：

- 现在进来的到底是哪个类型
- 到底是哪个方法
- 这个方法在 `dhao` 里是不是被标记为变化过

这说明它一定会碰“方法身份映射”这层，而不是只碰执行器本身。

### 高可信推断二：它一定要同时维护两条合法路径

`DHE` 不是把 AOT 替掉，而是让 AOT 和 interpreter 在同一程序集里共存。  
因此它几乎不可能不维护两条合法路径：

- 未变化函数继续走原来的 native `methodPointer`
- 变化函数改走最新代码对应的解释执行入口

也就是说，它不是“把 interpreter 跑快”，而是“尽量别让没变化的函数掉进 interpreter”。  
这两件事看起来接近，runtime 代价却完全不同。

### 高可信推断三：它需要比普通热更更强的加载顺序约束

官方文档已经明说：  
对应的 DHE AOT 程序集代码，不能在加载 DHE 热更代码之前执行。

这背后的 runtime 逻辑其实很好理解。  
如果某段代码在 DHE 热更 dll 和 `dhao` 都还没加载时就已经执行了，那系统根本还不知道：

- 这个函数是不是已经变了
- 当前到底该走 AOT 还是 interpreter

所以它只能先把 “任何可能抢跑的入口” 都收紧。  
这也是为什么 `[InitializeOnLoadMethod]`、`Script Execution Order` 这些早期入口会被直接列进不支持清单。

### 中可信推断四：资源挂载限制，本质上也是身份和时序问题

官方文档明确写了：  
DHE 脚本不能挂到打进包体的资源上，包括 `Resources`。

如果按我们前面 MonoBehaviour 身份链那篇的逻辑来理解，这条限制并不奇怪。  
因为打进资源的脚本引用，本来就依赖于稳定的程序集与类型身份。  
而 `DHE` 又要求“同一程序集的执行路径要等到热更 dll 和 `dhao` 就绪后才能判定”，这两条约束放在一起，很容易把资源反序列化前置阶段变成高风险区。

所以这条限制大概率不是“资源系统刚好没支持”，而是：

`在资源反序列化和 DHE 装载时序之间，官方选择了更保守、更稳定的边界。`

### 中可信推断五：`extern` 限制说明 native 边界仍然不能被随意热改

官方写了不能通过热更给 DHE 程序集新增 `extern` 函数。  
这条限制也很能说明问题。

`extern` 不是纯托管逻辑，它天然连着 native 侧导出、wrapper、绑定和调用约定。  
既然原包里的 native 世界已经定型，运行时再给同一个 DHE 程序集临时塞一个新的 `extern` 声明，边界契约就会立刻变得不稳定。

所以 `DHE` 虽然大幅放宽了“已进包程序集可不可以热改”的边界，但它并没有打算把 native 边界也一起做成完全动态的。

## 它真正换来了什么

如果把收益放到工程视角里看，我会这样总结。

### 收益一：未改动逻辑保住 AOT 性能

这是 `DHE` 最核心的收益，也是官方公开宣传的重点。  
真正值钱的不只是“更快”，而是：

`同一个程序集里，没动的那部分逻辑不必因为少量热修就整体退回解释执行。`

这意味着你可以把性能预算更集中地花在“真正发生变化的函数”上。

### 收益二：对现有 AOT 模块的改动成本更低

普通热更路线经常逼着项目做一件事：

- 把未来可能变的逻辑尽量前置拆到热更程序集
- 把一些 extern 或边界逻辑留在 AOT

这当然是稳的，但一旦历史包袱很重，模块切分成本也会很高。  
而 `DHE` 的价值就在于，它让“直接对现有 AOT 模块做差分热更”变得现实很多。

### 收益三：包体里的 native 代码完整度更高

官方还明确把这一点当成优势提过：  
native 代码完整打进包体，有助于降低某些平台审核风险，尤其是 iOS 场景。

这一点并不是 runtime 原理本身，但确实是很现实的项目收益。

## 它额外付出的代价是什么

如果只讲收益，这篇又会变成宣传文。  
真正工程化的判断，必须把代价和限制一起摆出来。

### 代价一：构建产物链会比普通热更更严格

普通热更已经要求你对：

- 热更 dll
- AOT 快照
- metadata
- bridge 生成物

保持一致。

而 `DHE` 额外再引入了一个很硬的契约文件：`dhao`。  
它必须和：

- 打包时那一版 AOT dll
- 当前这次下发的最新热更 dll

严格配对。

只要这里有一处错配，运行时就可能把“该走解释执行”的函数误判成“继续走 AOT”，或者反过来。  
也就是说，`DHE` 的风险不是“功能失效这么简单”，而是“分流判断失真”。

### 代价二：加载顺序会更苛刻

这条其实官方已经用不支持清单替你说明了。  
只要某个 DHE 程序集可能在热更加载前就被执行，它就不再适合走 `DHE`。

这意味着：

- 启动前期入口要非常克制
- 初始化链路要非常清楚
- 哪些程序集能标记成 DHE，不能只看“想不想热改”，还要看“会不会抢跑”

所以 `DHE` 不是单纯的执行器能力，它会反过来约束你的启动架构。

### 代价三：资源和脚本挂载边界会更保守

不能把 DHE 脚本挂到打进包体的资源上，这对很多 Unity 项目不是小限制。  
它意味着你不能简单把 DHE 当成“现有 MonoBehaviour 热更的全面替代”。

换句话说：

- 普通热更主线强在“动态程序集 + 资源挂载身份链”
- `DHE` 强在“已进包 AOT 程序集的函数级差分执行”

这两条能力不是互相覆盖，而是各自擅长不同问题。

### 代价四：它解决的是执行路径分流，不是所有动态问题的总开关

就算用了 `DHE`，你也不能把这些判断全部删掉：

- 某段逻辑是不是本来就更适合做成普通热更程序集
- 某些高频跨边界调用是不是仍然该收缩
- 某些需要资源挂载的脚本是不是根本不该进 DHE
- 某些 native 边界是不是仍然必须留在稳定 AOT 模块

它确实很强，但它强的是一条非常具体的轴：

`已进包程序集的函数级混合执行。`

## 项目里什么时候该认真考虑它

如果你的项目已经出现下面这些信号，那 `DHE` 就值得认真进入方案表了：

- 你想热改的是现有 AOT 模块本身，而不是单独追加一个热更模块
- 实际改动只占模块的一小部分，但未改动部分又明显是性能热点
- 历史包袱让你很难把逻辑优雅地重新拆到“纯热更程序集 / 纯 AOT 程序集”
- 你能严格控制 DHE 模块的加载顺序，不会在热更前抢跑
- 你能把 `dhao` 产物链纳入稳定的构建和发布流程

反过来，如果你的项目特征更接近下面这些情况：

- 主要是新增业务逻辑，普通热更程序集已经够用
- 资源上挂脚本是核心工作流
- 启动期很难保证 DHE 程序集绝不抢跑
- 你无法稳定维护“打包 AOT dll / 最新热更 dll / dhao”三者一致性

那 `DHE` 就不一定是最先该上的能力。

## 最后压一句话

如果只允许我用一句话收这篇文章，我会写成：

`DHE` 真正抬高的，不是解释器本身的速度，而是“修改现有 AOT 程序集”这件事的工程上限；它靠离线差分得到的 `dhao` 文件，在运行时把“未变函数继续走 AOT、变更函数切到最新解释执行”这套分流机制立起来，因此换来了接近原生的整体性能，也换来了更严格的加载顺序、资源边界和构建一致性要求。`
