很多 DOTS 项目不是死在“第一次迁移”，而是死在“后续升级”。前面那批代码已经跑起来了，团队也建立了自己的 Hybrid 边界，但一到包升级、引擎升级、平台切换或者补丁回收窗口，问题就会重新浮出来：编译不过、Baking 失效、行为变了、性能抖了、某个旧的 workaround 失去意义。

这也是为什么版本升级比运行时 bug 更难处理。运行时 bug 通常只在一个确定场景里暴露，而升级问题会同时穿过包管理、代码生成、构建链、Baking、验证链和团队习惯。看起来只是“换个版本”，实际更像一次小规模的架构重审。

---

## 为什么升级风险总在维护期爆发

维护期的特点，是项目已经有资产、有代码、有发布压力，但又没有新功能阶段那种“随时可以重来”的自由度。这个阶段最容易出现的不是单点失败，而是连锁失败。

DOTS 的连锁失败通常长这样：

- 包升级后，某些 API 或行为变化，需要同步改一批 System、Baker 和桥接层。
- 代码能编译，但 Baking 输出和运行时预期不一致。
- 运行时还能跑，但性能曲线变了，原来的阈值不再成立。
- 团队为了赶窗口，把修复压成临时补丁，结果下一次升级又重新踩一次。

所以，升级问题不是“某个包坏了”，而是“项目对这套包的依赖关系没有被显式管理”。只要依赖关系是隐式的，维护窗口就会把所有旧债一起结算。

## 版本升级不是单包动作

最常见的误解，是把 `Entities`、`Burst`、`Collections`、`Jobs` 看成可以独立升级的普通库。实际项目里，这些包是绑在一起工作的：

- `Entities` 决定世界、系统、Baking 和组件组织方式。
- `Burst` 决定哪些热路径能被编译成更高效的机器码。
- `Collections` 影响 Native 容器、内存生命周期和 Job 侧数据结构。
- `Jobs` 决定调度、并行边界和依赖声明。

如果其中任意一层变化，其他层的工作方式都可能跟着变。比如你以为自己只是换了一个容器实现，实际上变的是 Job 可见性、序列化行为或者构建时的失败模式。

这意味着升级不能按“单包替换”思路做，而要按“依赖矩阵”做。你需要知道哪些包是核心依赖，哪些是验证依赖，哪些是构建依赖，哪些只是工具依赖。

## 风险分层

DOTS 升级风险通常可以拆成四层。

### 1. 编译层

这一层最直观，表现为 API 名称变化、命名空间移动、约束收紧、generated code 报错。它容易被发现，但不一定最危险，因为很多问题能在编译期直接暴露。

### 2. 语义层

代码能编过，但行为变了。比如系统执行时机变化、Baker 输出结构变化、某些缓存失效、查询结果边界变动。这一层最麻烦，因为它会让团队误以为“版本升级没问题”，直到回归场景才发现偏了。

### 3. 构建层

构建链出问题，通常体现为 Build 变慢、AOT 失败、Burst 产物不稳定、某些平台包不可用。维护期里这类问题最讨厌，因为它直接挡住发布。

### 4. 组织层

这层不是技术报错，而是团队是否能持续维护。升级工作如果每次都依赖某一个人记得某些 workaround，那就是组织层风险。它会把技术债变成个人经验债。

## 依赖矩阵怎么做

真正可维护的做法，不是“看到新版本就升级”，而是先把依赖关系写清楚。

```text
核心包:
  Entities -> 世界、系统、Baking
  Burst -> 热路径编译
  Collections -> Native 数据结构
  Jobs -> 调度和并行边界

验证包:
  Hybrid 桥接层
  调试工具
  Profiler / 构建脚本

平台依赖:
  Mono / IL2CPP
  目标平台 SDK
  AOT / Headless / CI 镜像
```

这个矩阵的作用，是让每次升级都能回答三个问题：

1. 哪一层变了。
2. 哪一层会被波及。
3. 哪一层需要回归验证。

如果做不到这一点，升级就只能靠“试试能不能跑”。

## 为什么很多风险只在维护窗口出现

维护窗口的危险，不在于技术本身更复杂，而在于项目已经积累了固定行为。你过去用过的 workaround、编译器配置、临时桥接、版本锁定，平时都看起来没问题，一到升级就一起失效。

最典型的情况有三种：

- 之前靠临时补丁绕过的问题，在新版本里变成正式报错。
- 旧版本的行为依赖被团队当成了“事实”，新版本一改就会破坏业务假设。
- 旧的验证脚本只覆盖旧行为，没有人重新定义升级后的通过标准。

所以维护期的本质，不是“修 bug”，而是“重新确认这套工程链是不是还能成立”。

## 一个稳一点的升级流程

```csharp
bool UpgradeDotsPackages()
{
    // 1. 锁定升级范围，只改一组相关包，不做顺手重构。
    // 2. 先跑编译和静态检查，再跑最小场景。
    // 3. 对比 Baking 产物、运行时行为和性能基线。
    // 4. 任何一层异常，都先回滚升级，不先扩大改动面。
    return Compile() && Bake() && RunSmoke() && CompareBaseline();
}
```

这段逻辑看起来简单，但原则很硬：升级不是“先合并再看”，而是“先证明升级仍然守住边界，再继续往前走”。

## 什么时候该停

下面几种情况出现任意一种，就应该暂停升级，而不是继续向前推：

- 编译问题已经不是单点修复，而是成批波及多个包。
- 运行时行为和升级前差异太大，但没有足够的对照数据解释。
- 团队开始靠经验绕过问题，而不是靠矩阵和基线判断问题。
- 修复升级问题的成本，已经明显高过这次升级带来的收益。

DOTS 升级不是“必须追最新”的任务。对多数项目来说，稳定地维持在一个可验证、可回归、可重复的版本带宽里，比追求每次都上最新版本更重要。

## 小结

DOTS 的升级风险不是单包风险，而是包依赖、构建链、Baking 和团队维护方式一起叠出来的风险。
真正可靠的做法，是先写依赖矩阵，再做分层验证；如果维护窗口里连最小回归链都守不住，那就先别升级。
