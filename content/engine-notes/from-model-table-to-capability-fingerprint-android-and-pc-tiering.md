很多团队第一次做机型分档时，最自然的实现方式会是：

- 看到一台机器
- 查到型号名
- 在表里找它应该进哪一档

这套思路在一些平台上并不是完全错的。

例如：

- 硬件代际非常稳定的平台
- 团队手里只有少量固定设备池
- 产品只面向非常窄的设备范围

但只要场景一放到 `Android` 和 `PC`，这套做法很快就会变得越来越脆。

因为真正决定体验的，经常不是机型名本身，而是：

- `GPU` 家族
- 图形 API 路径
- 分辨率和刷新率
- 内存和存储状态
- 驱动版本
- 热状态
- 玩家自己设置的目标帧率与画面偏好

所以这篇文章真正要讲的，不是“型号表还有没有用”，而是：

`在 Android 和 PC 这种开放环境里，分档判断怎样从型号名单升级成能力指纹。`

如果说 [手机和 PC 为什么要用不同的性能直觉]({{< relref "engine-notes/game-performance-mobile-vs-pc-intuition.md" >}}) 解释的是“为什么两个平台的问题长得不一样”，那这篇更像是在把那个判断框架压回到一个具体工程动作里：

`默认分档规则到底该看哪些信号。`

## 为什么型号表在 Android 和 PC 上很快会失真

先把问题说清楚：

型号表最大的问题不是“维护麻烦”，而是它默认了一个前提：

`型号名 = 设备能力。`

这个前提在 Android 和 PC 上都很容易失真。

### 在 Android 上，问题是同名设备不等于同样表现

常见原因包括：

- 同一代际芯片的散热设计差异很大
- 同品牌不同市场版本的存储、内存和屏幕规格不同
- 系统版本与驱动更新会改变图形路径表现
- 用户后台环境、电池状态和老化程度差异大

也就是说，型号名能提供线索，但它远远不等于真实预算。

### 在 PC 上，问题是你甚至没有一个稳定的“机型名”

用户真正带来的差异通常是：

- 不同 `GPU`
- 不同 `VRAM`
- 不同显示分辨率和刷新率
- 不同驱动
- 不同电源模式
- 不同后台环境

在这种环境里，继续沿用“型号表思维”，很容易把一套分档系统做成一堆难以解释的特殊判断。

所以更稳的目标不是完全抛弃型号信息，而是把它降级成：

- 少量黑白名单
- 少量已知风险覆盖

而把真正的默认判定，交给能力指纹。

## 什么叫能力指纹

如果把这件事压成一句话，我会这样定义：

`能力指纹不是设备名字，而是一组真正决定当前运行预算和风险面的信号组合。`

对分档系统来说，一套更实用的能力指纹通常至少包括三层。

### 第一层：硬能力门槛

这层解决的是“能不能”：

- 是否支持目标图形 API
- 是否满足最小内存 / 显存
- 是否支持某些关键格式或特性

如果这层不满足，后面不必再做精细评分，直接走更保守路径。

### 第二层：性能与预算信号

这层解决的是“跑哪档更稳”：

- `GPU` 家族与代际
- 分辨率与刷新率
- 可用内存与存储速度
- 热状态或长时运行表现
- 目标帧率

### 第三层：风险修正信号

这层解决的是“理论上可以，但现在要小心”：

- 某驱动版本已知有问题
- 某 API 路径稳定性差
- 某组机型在历史版本里反复出现异常

这样一来，分档系统就不再是：

- 看到型号表，直接给档

而会变成：

- 先看硬门槛
- 再看预算信号
- 最后再做风险修正

## Android 分档最值得看的信号是什么

Android 是最容易让型号表失真的地方，所以也最适合先讲。

我更建议 Android 的默认分档至少看下面几类信号。

### 1. GPU 家族与代际

这通常仍然是最值钱的第一信号。

原因不是它万能，而是它最接近图形预算本身。

例如：

- `Adreno` 不同代际
- `Mali` 不同代际
- 某些长尾 `GPU` 的 API 支持差异

它比单看机型名更接近真实图形能力，也更适合和线上风险聚类对齐。

### 2. 图形 API 路径

同一设备在：

- `Vulkan`
- `OpenGL ES`

两条路径上的表现与稳定性，可能完全不是一回事。

所以默认分档至少要知道：

- 当前实际跑的是哪条 API
- 该 `GPU` 在这条 API 上是否稳定

有些时候，你真正需要降的不是“画质档位”，而是“API 路径风险”。

### 3. 分辨率与刷新率

同一个 `GPU`，在下面两种目标下预算完全不同：

- `1080p @ 60`
- 高分屏 + 高刷新

所以 Android 分档不能只看芯片型号，还要看当前的输出负担。

### 4. 内存与存储状态

很多机型不是纯 GPU 不够，而是：

- 内存紧
- 存储慢
- 可用空间低

这些信号会更直接地影响：

- 首载
- 切场景
- 热更资源挂载

### 5. 热状态与持续运行表现

Android 上同型号设备的热差异很大。

所以更稳的系统通常会把：

- 首次 Benchmark
- 长时运行帧时间
- 热后掉速情况

当成模型之外的校正信号。

## PC 分档为什么更像能力指纹，而不是“机器识别”

如果说 Android 还能勉强靠“机型名 + `GPU`”维持一层近似，那 PC 基本就不适合继续走这条路。

因为 PC 环境的本质更像：

`开放硬件组合 + 开放用户目标 + 开放后台环境。`

所以 PC 分档更适合围绕下面几类信号。

### 1. GPU 与 VRAM

这通常是第一层。

但即便如此，也不应只问“显卡型号是什么”，还要结合：

- `VRAM`
- 驱动版本
- 是否为移动版 / 低功耗版

### 2. 目标分辨率与刷新率

PC 最容易被低估的一点，是：

- 同一块显卡
- 不同显示目标

预算可以差非常多。

所以 PC 默认分档最好不是“这块卡永远高档”，而是：

- 这块卡在当前目标分辨率和帧率下，更适合哪档

### 3. CPU 与后台环境

有些 PC 问题并不是图形档位本身，而是：

- 后台录屏
- 浏览器标签页
- 杀毒或同步工具
- 笔记本电源模式

这些因素经常会把“理论上应该高档”的设备，拉回中档体验。

### 4. 玩家目标与手动覆盖

PC 和移动端另一个明显差别是，PC 用户通常更能接受：

- 自己选画面与帧率取舍
- 自己关某些昂贵效果

所以 PC 分档的意义通常不是替玩家做最终决定，而是：

`给出一个更稳的默认起点，再允许玩家手动调。`

## 一个更实用的规则模型：硬门槛 + 评分 + 黑白名单

如果把 Android 和 PC 的默认分档规则统一抽象，我更喜欢下面这种结构：

1. **硬门槛**
例如：
- 不支持目标 API
- 显存 / 内存不足
- 已知高风险驱动

2. **能力评分**
例如：
- `GPU` 家族得分
- 分辨率修正
- 刷新率修正
- 热状态修正
- 存储 / 内存修正

3. **黑白名单修正**
例如：
- 某驱动强制降档
- 某已知稳定组合允许提一档

4. **玩家手动覆盖**
作为最终体验选择的一部分保留

伪代码可以像这样：

```csharp
public static TierDecision Decide(DeviceFingerprint fp)
{
    if (!fp.SupportsRequiredApi)
        return TierDecision.Low("api_hard_gate");

    if (RiskTable.IsHardBlocked(fp))
        return TierDecision.Low("risk_hard_block");

    int score = 0;
    score += ScoreGpu(fp.GpuFamily, fp.GpuModel);
    score += ScoreResolution(fp.ScreenWidth, fp.ScreenHeight, fp.RefreshRate);
    score += ScoreMemory(fp.SystemMemoryMb, fp.GraphicsMemoryMb);
    score += ScoreThermal(fp.IsThermallyConstrained);

    var tier = score >= 80 ? QualityTier.High
             : score >= 50 ? QualityTier.Mid
             : QualityTier.Low;

    tier = RiskTable.ApplySoftAdjustments(fp, tier);
    return new TierDecision(tier, "capability_fingerprint");
}
```

这里最重要的不是分数本身，而是：

`默认判定由能力信号驱动，型号表只做少量风险修正。`

## 怎么避免规则表越写越脏

只要分档系统真的上线，规则一定会增长。

所以真正需要设计的，不是“如何永远不加规则”，而是：

`如何让新增规则不会把系统拖进一团无法解释的例外。`

我更建议至少守住下面三条。

### 1. 型号表只做例外，不做主体

默认判断一定优先走能力指纹。

型号表应该只承担：

- 已知严重风险黑名单
- 已知稳定白名单
- 少量市场特殊机型修正

### 2. 规则必须版本化并可回滚

只要某条规则会影响线上档位，就必须：

- 有版本号
- 能灰度
- 能回滚

否则你很快就会遇到这种状态：

- 大家都知道“最近好像加过一条规则”
- 但没人能说清到底是哪条导致了线上变化

### 3. 规则来源必须能回溯到证据

更稳的规则，不该来自：

- “感觉这机子不太行”

而应该来自：

- 遥测数据
- 崩溃与卡顿聚类
- 固定 baseline
- 边界机复核

也就是说，规则的来源最好是证据，而不是印象。

## 一条最小可执行路径

如果项目现在还停留在型号名单分档，我会建议先从下面这五步升级：

1. 先把当前规则拆成“主体规则”和“例外规则”
2. 先把主体规则改成 `GPU / API / 分辨率 / 内存` 这些能力信号
3. 先保留少量黑白名单，只处理已知高风险例外
4. 先把线上遥测和问题归因切到能力指纹维度
5. 先让玩家手动覆盖和规则版本都进入日志

只要这五步立住，分档系统就已经不再是一个型号表，而是一套更接近真实预算与风险面的能力判断系统。

## 小结

如果把这篇文章压成一句话，我会这样收：

`在 Android 和 PC 上，型号名最多只是线索，不该再是分档系统的主体；更稳的默认判断，应该由 GPU、API、分辨率、内存、热状态和驱动风险这些能力信号组成，再让型号表退回少量黑白名单和例外修正。`
