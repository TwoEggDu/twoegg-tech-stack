> 最短的一句话是：`多人技能系统不是“把单机技能加一层同步”，而是把施法请求、权威执行、命中确认和表现反馈重新拆回不同层。`

很多团队第一次把技能系统接进多人时，最自然的写法通常长这样：

```csharp
public void TryCastFireball()
{
    if (!manaEnough || cooldownActive)
        return;

    StartCooldown();
    SpawnProjectile();
    PlayCastAnimation();
    RpcNotifyOthersFireball();
}
```

这在单机里可能没问题，因为：

- 输入来源只有本地
- 命中判定只发生一次
- 扣蓝、进冷却和播表现都在同一份状态上

但只要进到真实多人环境，问题会立刻出现：

- 本地已经进冷却了，服务器却判这次请求无效
- 本地已经生成投射物了，服务器晚一拍才确认
- 本地看起来打中了，服务器却算没中
- 远端只收到了一个“掉血结果”，却不知道技能到底什么时候开始、什么时候命中

也就是说，多人里真正难的不是“再加一个 RPC”，而是：

`技能系统里“已经放出去了”这句话，在多人场景下不再只代表一件事。`

这一篇就专门把这件事拆开。

---

## 先别急着谈 RPC，先承认“技能已经发生”至少有五种不同含义

单机讨论里最容易偷懒的一句话，就是：

`这个技能已经放出去了。`

但在多人里，这句话至少可能指下面五件完全不同的事。

### 1. 本地输入已经产生

玩家按下了按钮，客户端决定发起一次施法请求。

这时发生的只是：

- 输入被采样
- 技能系统本地生成了 `CastRequest`
- 请求准备发给服务器

它还不等于技能成立。

### 2. 服务器已经接受这次请求

服务器完成了最基本的合法性判断，例如：

- 资源是否足够
- 冷却是否结束
- 当前状态是否允许施法
- 角色是否仍然具备释放这个技能的前提

到这一步，技能才真正从“想放”进入“被允许放”。

### 3. 技能已经进入服务器权威生命周期

即便服务器接受了请求，也不等于技能已经完成。

很多技能还要继续经历：

- Windup
- Active
- Recovery
- Cooldown

所以“请求被接受”和“技能进入 Active”也不是同一个时刻。

### 4. 命中已经被权威确认

尤其是投射物、近战窗口、范围判定这类技能，命中确认通常比开始施法更晚。

这时才真正回答：

- 有没有打到目标
- 打到了谁
- 伤害、治疗、Buff、位移是否成立

### 5. 结果已经被各端看见并对齐

最后还有一层经常被忽略：

- 本地玩家看到了什么
- 远端玩家看到了什么
- 旁观者看到了什么

掉血、飘字、受击特效、击退表现，可能都比服务器内部的结果传播更晚。

所以多人里更准确的说法不该是“技能已经放了”，而应该明确成：

- 请求已发出
- 请求已接受
- 技能已开始
- 命中已确认
- 结果已广播

只要这几个状态不拆开，后面所有“为什么和预期不一致”的讨论都会变模糊。

## 多人里最危险的不是延迟，而是把单机执行链原样搬上网

很多人一谈多人技能系统，第一反应是：

`单机技能逻辑基本不变，再给关键地方加同步就行。`

这条路最大的问题不是不优雅，而是它会直接让边界失真。

### 1. 客户端自己先扣资源、先开冷却

这样写的后果是：

- 本地手感立刻成立
- 但服务器一旦拒绝，这次资源和冷却就要回滚

回滚本身不是不能做，而是如果你一开始没把“本地预测状态”和“权威状态”分开，后面代码会变成：

- 一会儿按本地算
- 一会儿按服务器改
- 谁都不知道当前 UI 展示的是哪份真相

### 2. 客户端自己先判命中

单机里“刀碰到了就算打中”很自然。

但多人里一旦位移、网络抖动、插值、服务器碰撞快照都进来之后，本地看到的接触并不自动等于权威命中。

如果这时客户端已经先：

- 扣血
- 播受击
- 挂 Buff

那后面就只能靠纠错把结果硬扳回来。

### 3. 表现时序被误当成权威时序

比如本地角色一抬手就播火球发射，玩家会自然觉得：

`火球已经真的飞出去了。`

但服务器也许还没接受请求，或者还在等 Windup 完成。

如果你没先定义“哪些表现是预测表现，哪些结果必须等权威确认”，视觉直觉就会不断和服务器真相打架。

### 4. 远端只能收到最终结果，看不到中间过程

很多粗糙实现只同步：

- 扣了多少血
- 挂了什么 Buff

但不同客户端对这个技能“怎么放出来的”完全没有一致时间线。

结果就是：

- 命中反馈突然出现
- 动作和结果脱节
- 旁观者看起来像被空气打中

所以多人技能系统真正要做的，不是给单机技能“包一层网络”，而是重新切清：

`谁拥有权威，谁只做预测，谁负责确认，谁只消费结果。`

---

## 先切权威边界：谁决定什么

我建议先把多人技能系统里的权责切成最硬的几句话。

### 客户端负责什么

- 采样玩家输入
- 生成 `CastRequest`
- 做有限度的本地预测
- 播放本地预期表现
- 接收服务器纠正并对齐

### 服务器负责什么

- 最终合法性校验
- 权威生命周期推进
- 权威命中判定
- 权威效果结算
- 权威冷却和资源状态
- 把关键结果广播给相关客户端

### 其他客户端负责什么

- 消费服务器广播的技能开始/命中/结算结果
- 在自己的表现系统里重建可见时间线

如果把这一层再压缩成最短的话，就是：

- 客户端拥有 `Intent`
- 服务器拥有 `Authority`
- 各端消费 `Confirmed Result`

这一步非常重要，因为它直接决定后面：

- 什么能预测
- 什么必须确认
- 什么要回滚
- 什么只需要表现补偿

## 最稳的入口不是“直接施法”，而是带标识的网络化 Cast Request

在单机里，`TryCast(skillId)` 可能已经够用了。

但一旦进多人，这个入口最好升级成带上下文和身份的请求对象。

例如：

```csharp
public class NetCastRequest
{
    public int localSequence;
    public EntityId casterId;
    public SkillId skillId;
    public EntityId targetId;
    public Vector3 aimDirection;
    public Vector3 aimPoint;
    public int clientTick;
}
```

这里最关键的不是字段多少，而是它明确表达了：

- 这是一次“请求”
- 不是一次已经成立的施法结果
- 这次请求有自己的本地序号和时间上下文

为什么这个对象很重要？

### 1. 它把“输入产生”和“技能成立”拆开

客户端只负责提交请求，不直接宣布技能成功。

### 2. 它让服务器能清楚返回哪一次请求的结果

服务器可以围绕同一个 `localSequence` 或映射出来的 `castId` 返回：

- `Rejected`
- `Accepted`
- `Started`
- `HitConfirmed`
- `Completed`

### 3. 它让预测和对账有锚点

如果没有这个锚点，客户端经常不知道：

- 服务器拒绝的是哪一次请求
- 当前显示的 Windup 到底对应哪次施法
- 本地投射物应该跟哪条权威事件对齐

所以多人里的第一原则不是“把技能同步出去”，而是：

`先把施法请求对象化，再让服务器围绕这次请求给出权威反馈。`

---

## 预测不是把整条链都提前执行，而是挑能预测的阶段

很多多人技能系统一开始最容易走两个极端：

- 什么都不预测，结果手感很差
- 什么都预测，结果纠错爆炸

更稳的做法不是二选一，而是显式划出预测边界。

### 1. 适合本地预测的部分

通常可以预测的是：

- 输入缓冲是否接收
- 本地起手表现
- 朝向调整
- 预热动画
- 非结果性的轻量特效

这些东西即便后面被服务器拒绝，也比较容易取消或平滑回收。

### 2. 最好由服务器确认的部分

通常不该让客户端拍板的，是：

- 资源是否真的扣除
- 冷却是否真的开始
- 命中是否成立
- 伤害是否生效
- Buff 是否挂上
- 控制、击退、位移是否结算

这些东西一旦先本地拍板，后面回滚的成本会远大于前面省下的等待感。

### 3. 需要“预测 + 对账”的部分

还有一类状态既不能完全不预测，也不能只看本地。

典型例子是：

- 投射物发射时机
- 近战挥刀的命中窗口
- 技能是否进入 Active

这类状态更适合做成：

- 本地先按预期展示
- 服务器返回权威 `castId` 和关键时刻
- 客户端把本地预测对象重新绑定到权威事件上

也就是说，预测的正确打开方式不是“本地先把全部结果算完”，而是：

`本地先把手感成立，再把真正的世界结果留给服务器。`

## 命中确认要和表现播出分开，尤其不能把“看起来碰到了”直接当命中

命中是多人技能系统里最容易偷懒、也最容易出事故的地方。

因为本地视觉上“碰到了”，往往只代表：

- 本地位置是这样插值出来的
- 本地动画当前播到这个姿态
- 本地镜头刚好站在这个角度

它不自动等于服务器权威命中。

### 近战技能

近战最常见的错误是：

- 本地看到武器擦到目标
- 就直接触发扣血和受击

更稳的做法是分成两层：

- 本地可以预开命中窗口和预播打击感
- 真正伤害、受击控制、Buff 施加仍等服务器确认

否则只要位置回滚或目标状态变了，就会出现：

- 本地明显砍中了，但服务器判空
- 本地已经出硬直了，服务器又要撤销

### 投射物技能

投射物的核心不是“本地能不能生成一颗球”，而是：

`这颗球到底只是视觉预测物，还是带权威身份的命中载体。`

更稳的落法通常是：

- 本地先生成一个预测投射物，保证手感
- 服务器接受请求后，生成权威投射物或权威飞行事件
- 客户端把预测投射物和权威投射物做绑定或替换
- 最终命中以服务器权威结果为准

### 范围与瞬发技能

这类技能最容易误写成：

`本地特效一播，就顺手把附近目标都结算了。`

但更稳的逻辑应该是：

- 本地可以先播范围提示和施法表现
- 服务器用自己的快照决定谁在范围内
- 真正命中名单由服务器广播回来

只要你把“视觉接触”和“权威命中”拆开，多人技能系统里一大半的错位都会立刻变得可解释。

## 冷却最好跟“按钮按下”脱钩，跟权威时刻绑定

冷却同步看起来像小问题，实际上是多人技能系统里最容易把 UI 和真相撕开的地方。

单机里，按钮一按下就进冷却通常没问题。

但多人里至少有三种常见时刻：

- 请求发出时
- 服务器接受时
- 技能真正进入 Active 或 Commit 时

如果每端默认用的不是同一个时刻，后面就会出现：

- 本地显示还在冷却，服务器其实已经恢复
- 本地以为没进冷却，服务器那边早就占掉资源
- 被打断时，不知道要不要退冷却

我更建议先设计一个显式状态机，而不是只设计一个 `cooldownRemaining` 数字。

例如：

- `PendingRequest`
- `Rejected`
- `Accepted`
- `Committed`
- `Interrupted`
- `Completed`

然后明确规定：

- 冷却在 `Accepted` 开始，还是在 `Committed` 开始
- 被打断后是否退还全部冷却、部分冷却，还是不退
- UI 展示的是本地预测冷却，还是服务器确认冷却

这里没有唯一答案，但有一个很硬的原则：

`冷却起点必须是服务器定义的统一规则，而不是客户端各自“差不多看着办”。`

## 用一个火球术走一遍多人执行链

假设玩家 A 对玩家 B 释放 `Fireball`。

更稳的一条链可以是这样。

### 第一步：本地生成请求

玩家 A 按下技能键：

- 客户端 A 采样输入
- 生成 `NetCastRequest(Fireball, target=B, sequence=41)`
- 本地进入 `PendingRequest`
- 播放起手动画和预热表现

这时候还没有真正扣蓝，也没有最终命中。

### 第二步：服务器做合法性校验

服务器收到请求后检查：

- 当前是否可施法
- 资源是否足够
- 冷却是否结束
- 目标是否仍有效

如果失败：

- 返回 `Rejected(sequence=41, reason=NotEnoughMana)`
- 客户端 A 取消本地预测表现
- UI 恢复到未释放状态

如果成功：

- 分配权威 `castId`
- 返回 `Accepted(sequence=41, castId=9001)`
- 服务器进入权威 Windup

### 第三步：客户端对账

客户端 A 收到 `Accepted` 后：

- 把本地预测的那次起手和 `castId=9001` 绑定
- 如果服务器修正了朝向、起手时间或目标，就在这一刻对齐

其他客户端收到 `CastStarted(castId=9001)` 后：

- 开始重建远端可见表现

### 第四步：服务器推进到 Active 并发射投射物

Windup 完成后，服务器进入 Active：

- 生成权威投射物
- 广播 `ProjectileSpawned(castId=9001, projectileId=3007)`

客户端 A 可以：

- 让自己的预测投射物吸附到 `projectileId=3007`
- 或直接用权威投射物替换本地预测物

### 第五步：服务器确认命中和结算

投射物飞行后命中 B：

- 服务器确认命中成立
- 服务器计算伤害和 Buff
- 广播 `HitConfirmed(castId=9001, target=B, damage=120, buff=Burn)`

这时各端才真正：

- 扣血
- 播受击
- 跳数字
- 挂 Burn

### 第六步：客户端完成收尾

收到 `Completed(castId=9001)` 后：

- 客户端关闭本次施法状态
- UI 冷却进入稳定显示
- 战斗日志记录完整一条权威链路

这条链最关键的价值在于：

`请求、开始、发射、命中、结算、收尾都有自己的状态，不再被“按钮按下”和“伤害出现”硬压成一个瞬间。`

## 一个坏边界通常怎么坏

### 1. 客户端本地先把资源和冷却都扣了

服务器一拒绝，请求虽然没成立，但 UI、日志和状态全都要倒着修。

### 2. 客户端本地视觉命中直接等于真实命中

只要位置插值和服务器快照不一致，后面一定会出现“我明明打中了”的争论。

### 3. 服务器只同步最终伤害，不同步关键时刻

远端玩家就会看到：

- 没看见施法开始
- 没看见投射物生成
- 直接突然掉血

### 4. 冷却没有显式状态，只剩一个剩余时间

项目后期一旦加入：

- 施法打断
- 部分返还冷却
- 服务器拒绝请求

这个数字就很难解释它当前到底代表哪种真相。

## 最好给多人技能链一个清晰的对账反馈

多人技能系统要稳定，除了 `CastRequest`，还最好有一套显式反馈。

例如：

- `Rejected`
- `Accepted`
- `Started`
- `ProjectileSpawned`
- `HitConfirmed`
- `Interrupted`
- `Completed`

这样客户端、AI、表现系统、战斗日志都能清楚知道：

- 这次请求是不是被接受了
- 技能到底开始没有
- 命中是不是权威成立了
- 是正常完成，还是中途被打断

如果没有这层反馈，多人技能系统里的每一层都会开始自己猜：

- UI 猜冷却是否开始
- 动画猜技能是否已进入 Active
- AI 猜这次施法是不是成功了
- 战斗日志猜这次伤害对应哪次请求

而一旦每层都在猜，系统就再也不会稳定。

---

## 这一篇真正想留下来的结论

多人技能系统最容易掉进去的坑，不是“少写了一个同步函数”，而是没有先把真相分层。

客户端可以先有：

- 输入
- 请求
- 预测
- 表现

但服务器必须拥有：

- 合法性
- 生命周期推进
- 命中确认
- 结果结算
- 冷却与资源真相

所以这一篇最短的结论可以记成：

`多人技能系统里，客户端负责发起和预测，服务器负责确认和结算；真正要同步的不是“一个技能按钮”，而是一条从 Cast Request 到 Hit Confirmed 再到 Completed 的权威执行链。`
