内存带宽是现代高性能代码最容易被忽视的瓶颈。CPU 的算力每年在增长，但内存带宽的增速明显慢得多。当你写出一段看起来"计算不多"的代码，它却跑得很慢，原因往往不是 CPU 不够快，而是内存带宽撑不住数据流量。移动端的情况比桌面更严峻：CPU 和 GPU 共享同一条内存通道，带宽是真正的公共资源。

---

## 两个不同的指标：带宽与延迟

内存性能由两个完全不同的维度描述，经常被混淆。

**延迟（Latency）**：发出一个内存读取请求，到收到第一个字节，中间经过的时间。单位是纳秒（ns）或 CPU 周期（cycle）。

**带宽（Bandwidth）**：单位时间内能够传输的数据总量。单位是 GB/s。

用水管来类比：延迟是水源到水龙头的管道长度，带宽是水管的直径。管道很长，水从源头流到你手里要等一段时间——这是延迟。水管很细，即便等到了，每秒流过的水量也有上限——这是带宽。

对游戏代码的影响截然不同：

- **延迟**决定单次 cache miss 的代价。CPU 发出访问请求后，必须等内存响应，期间流水线停顿（stall）。DRAM 延迟通常在 50～100 ns，折算成 3 GHz CPU 的 cycle 数，是 150～300 cycle 的停顿。
- **带宽**决定大量连续数据处理的吞吐量上限。批量遍历粒子位置、更新物理模拟、拷贝顶点缓冲，这类操作的真正瓶颈往往是带宽而非计算。

## 主流内存规格：真实数字

| 规格 | 典型场景 | 理论带宽 | 访问延迟 |
|------|---------|---------|---------|
| DDR4-3200（双通道） | 桌面 PC | ~51 GB/s | ~14 ns |
| DDR5-6000（双通道） | 高端桌面 / 主流主机 | ~96 GB/s | ~14 ns |
| LPDDR5（双通道） | 主流旗舰手机 | ~44 GB/s | ~14 ns |
| LPDDR5X（双通道） | 高端旗舰手机 | ~68 GB/s | ~12 ns |
| GDDR6（独立显卡） | 桌面 GPU 显存 | ~400～600 GB/s | ~10 ns |
| HBM2e | 高端 GPU / 服务器 | ~900 GB/s | ~7 ns |
| Apple M3 Pro 统一内存 | Mac 移动工作站 | ~150 GB/s | ~10 ns |

几个关键观察：

1. **延迟差距比带宽差距小得多**。DDR4 和 LPDDR5X 的延迟几乎相同，但带宽差了将近一倍。延迟主要由物理距离和 DRAM 电气特性决定，相对稳定。
2. **移动端 LPDDR5 的带宽约等于桌面 DDR4 双通道，但这条带宽要同时供 CPU 和 GPU 使用**（见下一节）。
3. **独立显卡的 GDDR6 带宽是移动端的 8～10 倍**，这不是优化技巧的差距，是物理规格的差距。

## UMA：移动端的内存共享现实

这是移动端和桌面最根本的内存架构差异。

### 桌面架构：各有专属通道

```
桌面 PC / 主机 架构
─────────────────────────────────────────────
         ┌──────────────┐
         │     CPU      │
         │  (多核心)     │
         └──────┬───────┘
                │ 内存控制器
                │
         ┌──────┴───────┐
         │  DDR5 系统内存 │   ~96 GB/s
         │  (CPU 专用)   │
         └──────────────┘

         ┌──────────────┐
         │     GPU      │
         │  (独立显卡)   │
         └──────┬───────┘
                │ 显存控制器（PCIe 连接主板）
                │
         ┌──────┴───────┐
         │  GDDR6 显存  │   ~400～600 GB/s
         │  (GPU 专用)  │
         └──────────────┘
─────────────────────────────────────────────
CPU 内存通道和 GPU 显存通道完全独立，互不干扰
```

### 移动端架构：UMA 共享

```
移动端 SoC（UMA 架构）
─────────────────────────────────────────────
  ┌────────────────────────────────────────┐
  │               SoC 芯片                  │
  │                                        │
  │  ┌──────────┐      ┌──────────────┐   │
  │  │  CPU 核心 │      │    GPU 核心   │   │
  │  │ (大核+小核) │      │ (移动端 GPU)  │   │
  │  └────┬─────┘      └──────┬───────┘   │
  │       │                   │           │
  │       └─────────┬─────────┘           │
  │                 │  片内总线              │
  │        ┌────────┴────────┐            │
  │        │   统一内存控制器  │            │
  │        └────────┬────────┘            │
  └─────────────────┼─────────────────────┘
                    │
           ┌────────┴────────┐
           │  LPDDR5 系统内存 │   ~44～68 GB/s
           │  CPU + GPU 共享  │
           └─────────────────┘
─────────────────────────────────────────────
CPU 和 GPU 争用同一条带宽，总量固定
```

**后果**：GPU 在渲染时，它消耗的带宽会从总量里扣除，CPU 可用的带宽随之减少；反过来一样。移动游戏的一帧里，GPU 在渲染上一帧的同时，CPU 在处理当前帧的逻辑——两者在同一时刻竞争同一条 44～68 GB/s 的内存通道。

这不是软件设计的问题，是物理架构的约束。

**Apple Silicon 的特殊位置**：M 系列芯片同样是 UMA，但带宽规格要高得多（M3 Pro 150 GB/s，M3 Max 300 GB/s）。这是 Apple 用更激进的封装工艺和更宽的内存总线买来的，不是 UMA 架构本身的优势。普通移动 SoC 的带宽远没有这个水平。

## 带宽瓶颈如何出现：Roofline 模型

判断一段代码是不是带宽瓶颈，有一个经典工具：**Roofline Model**（屋顶线模型）。

核心指标是**算术密度（Arithmetic Intensity）**：每字节内存传输对应多少次浮点运算（FLOP/byte）。

```
峰值性能（FLOP/s）
        │  ← compute-bound 区域（算力是瓶颈）
        │  ╱
        │ ╱   屋顶线
        │╱─────────────────────────
       ╱│  ← memory-bound 区域（带宽是瓶颈）
      ╱ │
─────╱──┼─────────────────────────→ 算术密度（FLOP/byte）
     转折点
```

- **转折点左侧**（低算术密度）：带宽利用率先达到上限，增加算力没有用，这是 memory-bound。
- **转折点右侧**（高算术密度）：计算能力先达到上限，增加带宽没有用，这是 compute-bound。

典型游戏代码的算术密度：

| 代码类型 | 算术密度 | 通常是？ |
|---------|---------|---------|
| 位置 / 速度更新（ECS 遍历） | 极低（2～4 FLOP/byte） | memory-bound |
| 碰撞检测（AABB 遍历） | 低（4～8 FLOP/byte） | memory-bound |
| 蒙皮动画（顶点变换） | 中等 | 接近转折点 |
| 物理模拟（约束求解） | 中等 | 视具体实现 |
| 复杂 PBR Shader | 高（50+ FLOP/byte） | compute-bound |
| FFT / 卷积（后处理） | 高 | compute-bound |

游戏逻辑部分——ECS 遍历、AI 状态更新、物理宽相——普遍是 memory-bound。这意味着优化这些代码，增加 SIMD 或多线程并不能带来对应的提升，真正有效的优化是**减少内存访问量**或**提高访问局部性**。

## 数据导向在带宽受限下的实际意义

这里用一个具体的数字来说明 SoA（结构体数组）和 AoS（数组的结构体）在带宽上的差异。

假设一个 Entity 结构体：

```cpp
// AoS（Array of Structures）— 传统布局
struct Entity {
    Vector3 position;    // 12 bytes
    Vector3 velocity;    // 12 bytes
    Quaternion rotation; // 16 bytes
    float health;        //  4 bytes
    float energy;        //  4 bytes
    int   flags;         //  4 bytes
    // ... 更多字段
    char  padding[204];  // 假设总大小凑到 256 bytes
};

Entity entities[10000]; // 总计 2.5 MB
```

现在有一个系统只需要更新位置（读 position + velocity，写 position）：

```cpp
// AoS 访问模式
for (int i = 0; i < count; ++i) {
    entities[i].position += entities[i].velocity * dt;
}
```

每次循环迭代，CPU 加载一个 cache line（64 bytes）。这 64 bytes 里包含了 position（12 bytes）和 velocity（12 bytes），但同时也加载了 rotation、health、energy、flags 和 padding。**实际需要的数据是 24 bytes，但加载了 64 bytes，有效数据利用率 37.5%**。

如果 Entity 更大（256 bytes），每次迭代需要加载多个 cache line，有效数据利用率降到 9.4%。

```cpp
// SoA（Structure of Arrays）— 数据导向布局
struct EntityPositions { Vector3 data[10000]; }; //  120 KB
struct EntityVelocities { Vector3 data[10000]; }; // 120 KB
// ... 其他数据分开存储

// SoA 访问模式
for (int i = 0; i < count; ++i) {
    positions[i] += velocities[i] * dt;
}
```

每次加载 cache line（64 bytes），包含约 5 个 Vector3。**全部都是需要的数据，有效数据利用率接近 100%**。

| 布局 | 每次迭代加载字节 | 有效数据 | 带宽利用率 |
|------|--------------|---------|---------|
| AoS（256 bytes/Entity） | 256 bytes | 24 bytes | 9.4% |
| AoS（64 bytes/Entity） | 64 bytes | 24 bytes | 37.5% |
| SoA | 24 bytes | 24 bytes | 100% |

在移动端，这个优化的价值更高：当总带宽只有 44 GB/s 且要和 GPU 共享时，把带宽利用率从 9.4% 提升到 100%，等效于把 CPU 可用的有效内存带宽扩大了 10 倍——而这不需要任何硬件升级。

## NUMA：多路服务器的带宽陷阱

NUMA（Non-Uniform Memory Access，非均匀内存访问）是服务端需要关注的问题，游戏客户端一般不涉及，但游戏服务器往往运行在多路服务器上。

```
双路服务器 NUMA 拓扑
─────────────────────────────────────────────
  Socket 0                    Socket 1
  ┌──────────────┐            ┌──────────────┐
  │  CPU 0       │───QPI/UPI──│  CPU 1       │
  │  (16核心)    │            │  (16核心)    │
  └──────┬───────┘            └──────┬───────┘
         │ 本地内存控制器              │ 本地内存控制器
  ┌──────┴───────┐            ┌──────┴───────┐
  │  DDR5 内存   │            │  DDR5 内存   │
  │  NUMA 节点 0  │            │  NUMA 节点 1  │
  └──────────────┘            └──────────────┘
─────────────────────────────────────────────
CPU 0 访问节点 0：延迟 ~80 ns，带宽 ~96 GB/s（正常）
CPU 0 访问节点 1：延迟 ~160 ns，带宽 ~48 GB/s（远端，代价加倍）
```

如果游戏服务器用多线程 ECS，线程被操作系统调度到 Socket 1，但数据在 Socket 0 的内存里，每次访问都走跨 NUMA 节点的远端链路，延迟翻倍、带宽减半。

实践建议：
- 游戏服务器用多路服务器时，显式配置**线程亲和性**（Thread Affinity），把同一个 World 的所有线程绑定到同一 NUMA 节点
- 在 Linux 上使用 `numactl` 或 `mbind` 控制内存分配的 NUMA 节点
- Multi-World ECS（如多个游戏房间并行）：每个 World 独占一个 NUMA 节点，避免跨节点竞争

## Non-Temporal 写入：绕过 Cache 的特殊场景

普通内存写入的执行流程：

```
普通写入（Read-For-Ownership）
CPU 要写地址 X
→ 检查 cache：未命中（cache miss）
→ 从内存读取 X 所在的整个 cache line（64 bytes）到 cache
→ 修改 cache line 里的目标字节
→ cache line 标记为 dirty，等待写回内存
```

即便只写 4 bytes，也要先读 64 bytes——这叫 **Read-For-Ownership（RFO）**。如果你要初始化一个 100 MB 的缓冲区，RFO 会让写操作消耗双倍带宽（读 + 写）。

**Non-Temporal（NT）写入**绕过了这个机制：

```
NT 写入（movntps / VSTMIA）
CPU 要写地址 X
→ 直接写入内存，不读 cache line
→ 数据不进入 cache（避免污染 cache）
→ 直接 Store 到内存
```

适用场景：
- 初始化大型缓冲区（粒子系统 reset、ECS Component 数组清零）
- 写入后短期内不会再读的数据（如帧尾的结果输出）
- 流式写入（streaming store）大型纹理或网格数据

在 ARM 架构（移动端）对应的指令是 `VSTMIA`（NEON）或 `ST1` with non-temporal hint；x86 上是 `movntps`/`movntdq`。Unity Job System 在某些批量写入场景下会自动使用 NT 写入，不需要手动干预。

## 回到原点：移动端 DOD 的带宽压力更大还是更小

结论是：**移动端的带宽压力显著更大，数据导向设计（DOD/ECS）在移动端的收益也更高。**

原因可以从三个层面说明：

**第一层：绝对带宽更小。** 桌面 DDR5 双通道提供 ~96 GB/s，移动端 LPDDR5 提供 ~44 GB/s，不到桌面的一半。

**第二层：UMA 竞争。** 移动端这 44 GB/s 要同时供 CPU 逻辑和 GPU 渲染使用。游戏负载高峰期，GPU 渲染复杂场景可能消耗 20～30 GB/s，留给 CPU 的有效带宽可能只剩 10～20 GB/s。桌面的 CPU 和 GPU 各有专属通道，不存在这种竞争。

**第三层：功耗收益叠加。** 移动端是电池供电，内存带宽每 GB/s 都对应实际的功耗。AoS 布局造成的无效 cache line 加载，不只是浪费带宽，还直接消耗电量、加剧发热，触发热降频。

所以在桌面，DDR5 的带宽相对充裕，AoS 布局引起的带宽浪费可能在测试中完全感知不到（被充裕的带宽掩盖）；但在移动端，同样的代码会明显拉高带宽占用率，在 UMA 竞争的背景下，效果放大。

**数据导向不只是 ECS 的架构哲学，在移动端它是切实有效的带宽优化手段**——在带宽这个公共资源最紧张的地方，把有效数据密度从个位数百分比拉到接近 100%，等效于给 CPU 端减少了大量不必要的内存流量，把有限的带宽留给真正需要的部分。

## 小结

- **延迟**（ns）和**带宽**（GB/s）是独立的两个指标，分别影响单次 miss 代价和大量连续数据的处理上限
- 移动端 LPDDR5 ~44 GB/s，桌面 DDR5 双通道 ~96 GB/s，差距约一倍
- **UMA 架构**：移动端 CPU 和 GPU 共享同一条内存通道，带宽是零和竞争的公共资源
- **Roofline Model**：算术密度低的代码（ECS 遍历、碰撞检测）通常是 memory-bound，真正的瓶颈是带宽而非算力
- **SoA vs AoS**：SoA 把有效数据密度从个位数提升到接近 100%，在带宽受限的移动端收益更大
- **NUMA**：多路服务器上跨节点访问延迟翻倍、带宽减半，线程亲和性配置是服务端的必要实践
- **NT 写入**：大块初始化场景用 Non-Temporal Store，省掉 Read-For-Ownership 的额外读带宽
