如果前面几篇解决的是“渲染系统要花多少钱”，那这篇要解决的就是另一个更难的问题：

`钱到底是花在谁身上，谁又有资格决定这笔钱是不是值得花。`

很多项目的分档最后失效，不是因为 `Quality Level` 没配对，而是因为材质和 Shader 没有被纳入同一套治理规则：

- 关键词谁都能加，结果同一个效果被拆成多套写法
- 变体谁都能开，结果构建里留下了一堆没人敢删的路径
- `Renderer Feature` 谁都能挂，结果高档能看见，低档却没有明确 fallback
- 运行时看起来都正常，但真正重的路径只在少数场景、少数设备上暴露出来

所以这篇不打算再讲一遍 Shader 语法，而是把它收成一条治理链：

`Shader 不是代码文件，Shader 是质量合同。`

## 先把治理边界定清楚

材质与 Shader 的治理，至少要拆成四层。

| 层级 | 负责什么 | 典型问题 |
|---|---|---|
| 模板层 | 决定项目默认怎么写材质、默认保留哪些能力 | 同类材质写法不一致、默认路径过重 |
| 关键词层 | 决定哪些能力是显式开关，哪些能力不能再靠随机拼接 | 关键词爆炸、逻辑重复、同一效果多套实现 |
| 变体层 | 决定构建里到底保留哪些可执行路径 | `variant` 过多、剔除时机不清、构建后才发现缺路径 |
| 运行时层 | 决定最终命中的路径是不是符合档位预算 | 高档能跑、低档掉特性、主路径和 fallback 不一致 |

这四层不是并列的检查项，而是一条链。

更具体地说，治理要先回答这些问题：

- 哪些效果必须进入默认模板，不能要求每个美术自己拼
- 哪些效果必须显式关键词化，不能混在一个“大一统 Shader”里
- 哪些 `variant` 在理论上存在，但不应该进入当前构建
- 哪些 `Renderer Feature` 是质量档的一部分，不能只靠场景里“开没开”判断

如果这几件事没有先定好，后面的 stripping、Shader Variant Collection（SVC）、`OnProcessShader`、构建门禁都只能是补丁。

## 先管模板，不要先管 keyword

很多团队一上来就想“把 keyword 收口”，但顺序通常反了。

真正更稳的起点是：先把材质模板标准化。

模板层要定义的不是“这个 Shader 能干什么”，而是“项目里默认该怎么用它”：

- 默认材质是偏 `Lit` 还是偏 `Unlit`
- 法线、遮罩、发光、细节贴图是不是默认保留
- 透明和半透明是不是强制走独立模板
- 高价特性是不是必须通过专门模板才能开启

这件事的意义在于，它会把“能力”从“随手开关”变成“明确契约”。

举个简单的分层方式：

| 模板 | 目标 | 默认行为 |
|---|---|---|
| Base Lit | 大部分角色、场景、道具 | 保留主光照、法线、基础高光 |
| Effect Lit | 需要特殊视觉反馈的内容 | 允许额外发光、边缘、局部特效 |
| Transparent Lit | 透明、玻璃、粒子 | 强制分离排序和混合路径 |
| Low Tier Lit | 低档保底 | 只保留最关键的可读性和识别度 |

这样做不是为了“模板更多”，而是为了让后面的关键词和变体都有归属。

## 关键词要有所有权

关键词最容易失控的地方，不是它本身多，而是它没有所有权。

我的建议是把关键词按来源分三类：

1. `渲染系统关键词`
   - 由 `URP Asset`、`Renderer Feature`、平台能力决定
   - 例如阴影、SSAO、额外光源、某些特定渲染路径

2. `内容关键词`
   - 由具体美术效果决定
   - 例如描边、溶解、特殊溶液、局部高亮、命中特效

3. `档位关键词`
   - 由分档策略决定
   - 例如高档保留高质量路径，中低档走简化路径

关键词一旦分了所有权，治理就会简单很多：

- 系统关键词只能由渲染系统改
- 内容关键词必须对照模板和预算
- 档位关键词必须能回收成明确 fallback

这会直接解决一个常见问题：同一套功能被写成三四种不同 keyword 组合，最后构建里谁都不敢删。

## 变体治理不是“删掉越多越好”

`Shader Variant` 的治理目标，不是把数字压到最小，而是把“本次构建真正可能用到的路径”保留下来。

这点在 [Unity Shader Variant 全流程总览]({{< relref "engine-notes/unity-shader-variant-full-lifecycle-overview.md" >}}) 里已经有完整链路：从理论空间到构建候选集，再到过滤、剔除、写入交付物、最终被 GPU 使用。

这篇要补的是治理视角：

- 变体不是先天资产，而是有成本的交付对象
- 变体不是“在 `OnProcessShader` 看见就算处理过”
- 变体不是“构建里有一个路径，运行时就一定会命中”

更实用的做法，是给每个 Shader 家族设预算：

| 预算对象 | 约束方式 | 目的 |
|---|---|---|
| 关键词数量 | 每个家族限定关键开关总数 | 避免无边界增长 |
| 变体候选数 | 每次构建前后可比 | 避免一版比一版胖 |
| 高价路径数 | 必须有明确 owner | 避免没人敢删 |
| fallback 数 | 必须能回到低档可玩路径 | 避免低档断路 |

`OnProcessShader` 的作用，是最后一道门，不是全部门禁。
它能说明“这次构建走到了什么地方”，但不能代替前面的模板约束、关键词所有权和 `Renderer Feature` 设计。

这点和 [Unity Shader Variant 在哪几层被剔除]({{< relref "engine-notes/unity-shader-variant-stripping-stages-prefiltering-builtin-custom.md" >}}) 是同一件事的两面：

- 前面的层决定“谁有资格进场”
- 中间的层决定“哪些路径值得被留下”
- 最后的层决定“这次构建到底还剩什么”

## `Renderer Feature` 必须进入质量合同

很多项目只把 `Renderer Feature` 当“功能开关”，这会直接破坏分档。

原因很简单：`Renderer Feature` 不是纯装饰，它往往改的是整条路径。

所以对于高 / 中 / 低档，`Renderer Feature` 至少要回答三件事：

1. 这个 feature 在哪一档是必须的
2. 这个 feature 在哪一档是可选的
3. 这个 feature 关闭时，内容该怎么 fallback

如果一个 feature 只在高档存在，那它就不能偷偷承载核心信息。

如果一个 feature 在低档必须存在，那它就不能依赖只在高档才编译进去的 shader 路径。

最典型的治理方式，是把 feature 和材质模板一起写入契约：

| 档位 | 允许的 feature | 必须有的 fallback |
|---|---|---|
| 高档 | 全量 feature，但仍受预算约束 | 复杂路径可回退但不丢核心反馈 |
| 中档 | 只保留对体验影响最大的 feature | 去掉高价装饰，不去掉可读性 |
| 低档 | 只保留生存型 feature | 保证识别度、命中反馈和 UI 可读性 |

这里的关键不是“关掉多少”，而是“关掉以后还算不算同一款游戏”。

## 一条可执行的质量门禁链

如果要把治理落到工程里，我建议最少放四道门：

### 1. 设计门

新材质、新 keyword、新 `Renderer Feature` 必须先回答：

- 它属于哪个模板层
- 它的 owner 是谁
- 它的 fallback 是什么
- 它在哪些档位允许出现

### 2. 构建门

构建阶段必须能看见：

- 变体预算有没有超
- 某条路径是不是只有高档会留下
- 某个 feature 关闭后，是否仍有合法替代路径

### 3. 发布门

发布前要抽查：

- 高 / 中 / 低档实际命中的是否是预期路径
- 核心表现是不是都能在低档找到对应语义
- 视觉反馈有没有因为 fallback 变得不可读

### 4. 运行门

运行时要持续观察：

- `SetPass` 和 material 切换是不是异常高
- 低档是否误命中了高价路径
- 高档是否因 feature 组合过多而反而命中不了最优路径

## 结语

材质与 Shader 的治理，本质上不是“少写几个 keyword”。

它真正要做的，是把项目里所有视觉能力都变成可管理对象：

- 模板能约束默认写法
- 关键词能标明能力边界
- 变体能被预算和剔除策略管理
- `Renderer Feature` 能和质量档绑定
- 运行时能证明命中的路径符合预期

如果前面的分档文章解决的是“系统要花多少钱”，那这篇解决的是：

`这笔钱到底花在了哪条路径上，谁来为它负责。`
