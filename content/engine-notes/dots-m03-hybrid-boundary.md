M01 解决的是“要不要上 DOTS”，M02 解决的是“第一刀切哪里”。到了 M03，问题就变成另一种：**混合架构到底怎么长期活下去**。

这个问题之所以重要，是因为大多数项目不会变成纯 ECS。UI、输入、动画、音频、资源系统、编辑器工具，这些东西短期内都不应该被幻想成“完全 Unmanaged 化”。真正能活得久的，是边界稳定的混合架构，而不是一味追求把所有东西挪进 ECS。

---

## 先给一张边界表

如果只想先看结论，可以先看这张表：

| 主题 | 权威通常站哪边 | Managed 层负责什么 | ECS 层负责什么 | 最该避免的坏写法 |
|------|----------------|--------------------|----------------|------------------|
| 输入 | Managed 采样，ECS 消费 | 读平台输入、写桥接数据 | 读取输入并推进仿真 | System 直接读 `Input` API |
| UI | Managed | 读结果、做展示 | 输出 UI 可消费状态 | UI 每帧扫描实体查状态 |
| 动画 / 音频 / 相机 | Managed | 把仿真结果翻译成表现 | 输出姿态、状态标记、事件 | ECS 直接持有 Animator / AudioSource |
| 资源 / 配置 | 编辑期 Managed，运行时 ECS 数据 | 维护 Authoring、ScriptableObject、资源引用 | 消费烘焙后的运行时数据 | 运行时 System 直接抓 Unity Object |
| 调试 | 两边都可见，但不应双权威 | 读状态、发调试命令 | 暴露快照和合法入口 | 调试器直接篡改仿真核心状态 |
| 数据桥 | 中立桥接层 | 写入或消费桥接数据 | 读取或写出桥接数据 | 把桥接层做成万能业务中心 |

这张表压成一句话，就是：**Managed 层负责平台、表现和编辑器世界，ECS 层负责仿真和批量状态推进，中间只通过明确的桥接数据交接。**

## 为什么 Hybrid 不是临时补丁

很多团队一开始把 Hybrid 看成“迁移中间态”，意思是：先让 MonoBehaviour 和 ECS 共存，等以后都迁完就结束了。实际项目里通常不是这样。越往后走，越会发现两侧各有各的擅长区。

ECS 适合高密度仿真、批量状态推进和可并行的数据处理。MonoBehaviour / Managed 层则更适合承载 UI、输入、编辑器兼容、资源引用和各类 Unity Object。只要项目还在使用 Unity 引擎，Hybrid 就几乎一定会长期存在。

所以要解决的不是“怎么尽快消灭一侧”，而是“怎么让两侧各自待在该待的位置上”。

## 输入站哪边

输入通常站在 Managed 侧，但输入结果不应该留在 Managed 侧。

这是最稳的做法：`MonoBehaviour` 或输入系统读取平台输入，把结果写进一个明确的桥接数据层，比如 Singleton Component、CommandData，或者一个短生命周期的桥接结构。ECS 仿真层只读这个数据，不直接去碰输入 API。

这样做的原因很简单。输入本质上是外部事件入口，和仿真结果不是同一种权威。把输入留在 Managed 层，能保持对平台和 UI 的兼容；把输入结果交给 ECS，能让后续仿真保持可并行、可重放、可验证。

最怕的是反过来：系统里直接去读 `Input.GetAxis`，或者让多个系统各自读输入并各自解释。那样会把“谁定义了这帧输入”变成一团。

## UI 站哪边

UI 也应该留在 Managed 侧，但 UI 读取的数据应该来自 ECS 的明确输出。

UI 的职责不是参与仿真，而是展示状态。血条、技能冷却、任务进度、单位选中框，这些都应该从 ECS 输出一份稳定的可读数据，再由 UI 读取。如果 UI 直接回头扫描实体、逐个查组件，就会把展示层变成仿真层的影子。

比较稳的方式有两类：

- ECS 写入一个专用的 UI Singleton 或事件通道，UI 只读它
- ECS 维护一份展示用快照，UI 在自己的节奏里读取

这两种方式都遵守同一个原则：**UI 只消费结果，不参与权威裁决**。

## 动画、音频和其他 Managed 对象站哪边

动画、音频、相机和很多平台对象，本质上都还在 Managed 侧。它们短期内不适合强行迁移成 ECS 的“核心仿真对象”，因为它们通常有大量 Unity Object 引用、回调、资源依赖和编辑器耦合。

这时候最合理的边界是：ECS 负责输出状态，Managed 侧负责把状态翻译成表现。比如角色移动系统只负责推进位置和状态标记，动画层只负责根据这些标记设置 Animator 参数；音频系统只负责读取事件并触发播放；相机只负责根据目标状态做跟随。

如果某个 Managed 对象必须和实体保持一对一关系，可以借助桥接方案或 CompanionObject 机制，但那仍然不是让 ECS 去“拥有”这个对象，而是让它作为表现层挂靠在仿真结果上。

## 资源、配置和烘焙数据站哪边

资源和配置也不应该和运行时仿真搅在一起。

项目里最常见的稳妥路径，是把设计期的数据保留在 ScriptableObject、Authoring 或配置文件里，再在 Baking 阶段转成 ECS 可消费的数据。运行时系统只看烘焙后的结果，不直接依赖原始编辑器对象。

这件事的意义不是“技术洁癖”，而是长期维护。只要运行时还直接抓着 Unity Object，不管你是查配置还是查资源，最终都会重新把热路径拖回 Managed 世界。

## 数据桥该站哪边

真正让 Hybrid 长期可维护的，不是“大家都有边界意识”，而是你有没有把桥接层单独立出来。

桥接层的职责应该很窄，只做三件事：

- 把 Managed 世界的输入整理成 ECS 可消费的数据
- 把 ECS 世界的状态或事件整理成表现层可消费的数据
- 固定两边的同步点，防止谁都能随手改核心状态

桥接层最容易写坏的地方，是把它变成一个“什么都能管”的中间层。只要桥接层开始存业务规则、开始代替仿真决策、开始代替 UI 做状态推导，它就不再是桥，而会变成新的耦合中心。

## 调试站哪边

调试工具可以两边都站，但观察点必须明确。

最稳的做法不是让调试器直接篡改仿真，而是让它读取明确的桥接数据。比如：

- ECS 写一份状态快照给调试器
- 调试器发出命令后，由桥接层转换成合法的仿真输入
- 需要记录时，记录的是状态变化，不是随手去改任意对象

这样做的好处是，调试器不会成为新的权威源。它只是观察和触发入口，不是第二套隐藏规则。

## 长期边界的三条原则

第一条，**单向依赖**。ECS 仿真层不应该持有 MonoBehaviour 引用，也不应该直接调用 Managed 对象的方法。依赖方向只能是 Managed -> ECS 的写入，或者 ECS -> Managed 的读取和表现。

第二条，**数据权威唯一**。同一份数据在同一时刻只能有一个权威源。输入归输入，仿真归仿真，展示归展示。只要一个数值允许两边同时改，bug 就会很快变成常态。

第三条，**同步点清晰**。Managed 与 ECS 的交接点必须固定，不能每天换位置。你要知道输入什么时候写进去，ECS 什么时候读，ECS 什么时候写回表现层，UI 什么时候取快照。

## 常见反模式

最危险的反模式，是每帧在 MonoBehaviour 里扫描所有实体，再用 `GetComponent` 式思维去查 ECS 数据。这个写法表面上只是“桥接”，实际上是在把 ECS 重新拉回对象遍历模型。

第二个反模式，是把 UI、输入、动画都塞进一个“万能桥接系统”。这种系统短期看起来省事，长期会变成混合架构里的单点失控。

第三个反模式，是让 ECS 系统直接依赖 Managed 的生命周期。只要一旦这么做，你就会发现系统组顺序、加载顺序、销毁顺序全都开始纠缠。

## 小结

Hybrid 能长期活下去，靠的不是“迁完就好”，而是边界清楚。

输入、UI、动画、资源、调试各自都有适合待的位置：Managed 层负责平台和表现，ECS 层负责仿真和批量状态推进，中间用明确的桥接数据传递，不要让双向依赖和每帧扫描把边界磨平。

下一步应读：`DOTS-M02｜第一阶段怎么迁移：别从 UI 开始，先切高密度仿真层`

理由：先把长期边界切清，再讨论第一阶段到底该先切哪层，迁移动作才不会反过来破坏边界。
