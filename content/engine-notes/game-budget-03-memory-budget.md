很多项目一提内存，第一反应就是一个问题：

`够不够。`

这个问题当然重要。  
但如果你只问这一句，最后经常会错过真正更值钱的判断：

- 为什么总量看着还行，长玩后还是不稳
- 为什么切场景最后一下总会抖
- 为什么热更解压、后台切回、首进大场景时更容易死
- 为什么同一个资源文件不大，进内存后却比想象中更贵

所以内存预算最容易被说错的一点就是：

`它不是仓库容量，而是运行时工作台的稳定性。`

## 内存预算先要分哪几笔账

如果只给一个“项目总内存上限”，通常不够。  
更稳的拆法至少要有下面五笔。

### 1. 稳态线

玩家连续跑图、战斗、切 UI 一段时间后的常驻占用。

这条线处理的是：

- 长时间运行是否稳定
- 是否给系统和后台环境留下余量

### 2. 峰值线

首进大场景、切场景、热更解压、批量实例化、回前台重建时的瞬时高点。

很多项目不是死在稳态，而是死在峰值。

### 3. 红线

这不是“理论还能申请到多少”，而是：

- 再往上就进入 `LMK / jetsam` 高风险区
- 再往上就明显失去安全余量

### 4. 工作集

真正这一段时间在被访问、被保持活跃的那部分内存。

很多卡顿首先来自工作集不稳，而不是总量立刻爆。

### 5. 显存相关成本

很多团队把内存只看成 CPU 侧 RAM。  
但对 Unity 项目来说，纹理、RT、ShadowMap、上传缓冲、Read/Write 副本这些都会把“同一份内容”的真实代价抬高。

## 判断依据是什么

内存预算的线不能只靠“这台机器还有很多 RAM”来定。

更稳的依据至少有三类。

### 1. 系统事实

移动端真正危险的不是能不能继续 `new`，而是：

- Android 的 `LMK`
- iOS 的 `jetsam`

系统关心的是当前进程占住了多少、系统是否还有余量，而不是你的项目内部觉得“看起来还可以”。

### 2. 运行时事实

同一份资源，常常不只存在一份：

- 磁盘压缩包一份
- CPU 解压副本一份
- GPU 资源一份
- 上传或过渡缓冲一份

如果贴图或 Mesh 开了 `Read/Write Enabled`，就更容易出现双份驻留。

### 3. 时机事实

很多内存事故不是因为内容本身非法，而是因为：

- 旧场景还没退完，新场景已经进来了
- 下载、解压、反序列化和实例化挤在同一时刻
- 对象池没有分场景回收

所以内存预算必须和生命周期时机一起看。

## 一套更稳的内存分桶

以 Unity 项目为例，我更建议直接按运行时驻留位置分桶：

- `代码与运行时`
  IL2CPP、托管堆、线程栈、插件、运行时数据结构
- `Texture / RT / Shadow`
  最常见的大头
- `Mesh / Animation`
  顶点索引、动画片段、蒙皮相关数据
- `Shader / Variant / Driver`
  不一定最大，但经常被低估
- `Object / Pool`
  经常把瞬时对象变成常驻对象
- `Bundle / Cache / Temp`
  切场景、下载、解压时最容易抬高峰值

如果这几桶没有分开看，体检时很容易只盯贴图尺寸，而漏掉真正让峰值失控的缓存和双驻留。

## 最常见的误判

### 1. “总量没爆，所以内存不是问题”

不对。  
很多 memory bound 问题先显成：

- GC 和释放尖峰
- 场景切换抖动
- 热更和下载期峰值
- 长玩后越来越差

### 2. “对象池一定更省性能”

对象池是在用常驻内存换实例化和 GC 尖峰。  
如果池容量没有上限、回收点和分场景策略，它反而会把稳态线抬高。

### 3. “资源文件不大，内存里应该也不贵”

这是典型误判。  
压缩包大小、解压后大小、CPU 副本大小、GPU 占用从来不是同一个数。

### 4. “只要压贴图就算做了内存预算”

贴图当然通常是大头，但真正让项目失控的经常是“很多个次大头一起出现”：

- RT
- Shadow
- 对象池
- 缓存
- 临时缓冲

## 预算应该怎样落地

真正可执行的内存预算，至少要落成下面这些产物：

- 稳态线、峰值线、红线
- 各桶分配比例
- 切场景与回前台的峰值口径
- 对象池上限和回收规则
- RT / Shadow / Texture Streaming 规则
- 设备档位对应的内存配置

如果这些没有进入规则表，内存预算就仍然只是“主程自己心里知道”。

## 和旧文的关系

如果你已经看过 [内存不是够不够，而是行为稳不稳]({{< relref "engine-notes/game-performance-memory-behavior.md" >}}) 和现有的 [CPU 性能优化 05｜内存预算管理：按系统分配上限、Texture Streaming 与 OOM/LMK 防护]({{< relref "engine-notes/cpu-opt-05-memory-budget.md" >}})，可以把它们理解成这篇的上游背景和细节支撑。

这篇更想明确的是：

`内存预算不是 CPU 子问题，也不是一句“够不够”；它是一套要同时管理稳态、峰值、工作集和系统强杀边界的独立预算系统。`

## 结尾

所以如果只用一句话收束这篇，我会这样说：

`内存预算真正要做的，不是只给一个总量上限，而是先把稳态、峰值、红线、工作集和运行时分桶拆清，再把对象池、缓存、贴图、RT 和生命周期尖峰都放回正确边界里；只有这样，项目才不会在“明明内存看起来还够”的状态下继续失控。`
