## 游戏代码的三种特殊压力

### 压力一：实时循环

普通的 Web 应用是**请求-响应**模型：用户发请求，服务器处理，返回结果，然后等待下一个请求。

游戏是**实时循环**模型：每秒 60 次，不管有没有用户输入，都要更新全部游戏状态并渲染一帧。

```
Web 应用：
请求 → [处理] → 响应
请求 → [处理] → 响应
（等待中...）

游戏主循环（每帧 16ms）：
Update() → FixedUpdate() → LateUpdate() → Render()
Update() → FixedUpdate() → LateUpdate() → Render()
Update() → FixedUpdate() → LateUpdate() → Render()
...
```

这个区别的影响是深远的：

**时序耦合**：游戏里所有系统都运行在同一个时间轴上。`PlayerController.Update()` 和 `EnemyAI.Update()` 在同一帧里执行，它们的执行顺序会影响结果。这是 Web 应用里几乎不存在的问题。

**帧预算**：每帧只有 16ms（60fps）。你的代码不能"想花多久就花多久"，必须在预算内完成。一段逻辑上正确但耗时过长的代码，在游戏里是不可接受的。

**状态在帧间持续**：每帧结束时，所有状态保留到下一帧。这不像请求处理完就销毁上下文——游戏的"上下文"是永续的。

### 压力二：状态爆炸

一个敌人 AI 可能有多少种状态？

```
基础状态：待机、巡逻、追击、攻击、受伤、死亡
组合条件：血量高低 × 与玩家距离 × 玩家状态 × 时间（白天/黑夜）× 同伴数量
特殊状态：眩晕、燃烧、冰冻、恐惧、狂暴
行为变体：普通攻击、重击、范围技、逃跑、呼叫支援

理论上的状态组合数：数百到数千
```

如果用 `if/else if` 来管理所有这些状态转换，会得到一团没有人能完全理解的条件逻辑。状态机、行为树、黑板系统——这些都是专门解决"状态爆炸"的工具。

### 压力三：性能敏感

游戏里的性能不只是"运行得快"，而是**帧时间一致**。

一个偶尔花 100ms 才完成的操作，在服务端可以接受（单次请求慢一点），在游戏里是一次明显的卡顿，破坏游戏体验。

这意味着：

**对象分配有代价**：`new SomeObject()` 在服务端代码里随手就写，在游戏里每次分配都要付出 GC 代价，在错误的时机触发 GC 会导致卡帧。这是 Object Pool 模式存在的直接原因。

**虚函数调用有代价**：OOP 里的多态依赖虚函数表查找，CPU 无法预测跳转地址，会打断流水线。对于每帧执行数万次的逻辑（粒子、子弹），这个开销是可测量的。

**内存访问模式有代价**：CPU 缓存是现代性能的关键。一个对象数组（AoS：Array of Structures）和一个数组对象（SoA：Structure of Arrays）在功能上等价，但缓存行为完全不同。游戏引擎（特别是 DOTS）对此有专门的优化策略。

## 设计模式在游戏里的定位

GoF 定义了 23 种设计模式。它们来自企业软件的最佳实践，在游戏里的适用性参差不齐。

**直接适用的**：Observer（事件系统）、Command（操作对象化）、State（状态机）、Strategy（OCP 的实现手段）、Flyweight（共享不可变数据）。

**需要性能考量的**：Decorator、Composite——在逻辑上很优雅，但如果每帧对大量对象调用，虚函数链的开销需要权衡。

**游戏专有的**：Object Pool（GoF 里没有，但游戏里极其常用）、Game Loop（整个主循环本身就是一种模式）、Component（Unity 的架构基础）、Data-Oriented Design（DOTS 的理论基础，完全超出 GoF 的范畴）。

这个系列只讲**游戏里真正常用、真正解决实际问题的模式**，不追求 23 种全覆盖。

## "模式"不是目标，解决问题才是

一个常见的误区是把设计模式当成展示技术深度的工具——"我用了 Factory、Decorator、Chain of Responsibility，设计很优雅"。

但模式只是工具，用工具的目的是**解决问题**。如果一个模式让代码更难理解，而它解决的问题其实不存在，那就不应该用它。

```csharp
// 这个函数只会被调用一次，在一个地方，永远不会有多种实现
// 不需要工厂、不需要接口、不需要抽象类
string GetPlayerName()
{
    return PlayerPrefs.GetString("player_name", "Player");
}
```

YAGNI——不需要的模式现在不要加。

正确的使用时机：当你**识别出了一个具体的问题**（重复 switch、无法扩展、依赖难以测试），并且某个模式**确实解决了这个问题**，此时引入该模式。

## 小结

游戏代码面临三种普通软件少有的压力：

| 压力 | 具体表现 | 对应模式 |
|---|---|---|
| 实时循环 | 时序耦合、帧预算、永续状态 | Game Loop、Command（延迟执行）|
| 状态爆炸 | AI 状态组合爆炸、条件逻辑失控 | State、FSM、行为树 |
| 性能敏感 | GC 压力、虚函数开销、缓存命中率 | Object Pool、DOD/ECS |

理解这三种压力，就理解了为什么每种模式被设计成现在的样子，以及在什么情况下它真正有价值。
