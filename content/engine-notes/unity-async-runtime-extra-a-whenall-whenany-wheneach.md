> 如果只用一句话概括这篇文章，我会这样说：`WhenAll、WhenAny、WhenEach 在 UniTask 里不是“并发等待的三种写法”，而是三种不同的收口协议：一种把多条链压成一个整体终局，一种把竞争压成第一个完成者，一种则把每个完成事件重新变成流。`

主线前面几篇已经把 UniTask 的基础地形铺开了：

- 外层只是薄句柄，真正的完成真相活在 source 协议层
- continuation 的时间位置由 `PlayerLoop` 骨架决定
- `builder` 很薄，真正承重的是 runner/source/completion 协议
- `Version/token`、单次消费和未观察异常共同定义了一整套更贴 Unity 的运行时边界

走到这里，再看并发组合，就不该再把它们理解成“写法差异”。

因为只要把多个异步链凑到一起，真正的问题马上就会变成：

- 这几条链的完成真相由谁来汇总
- 异常是立刻决定总结果，还是先作为数据保留
- 谁先完成时，其他链还会不会继续跑
- 聚合结果是一次性交付，还是按完成顺序流式交付
- 最终返回的是一个“整体结果对象”，还是一个“逐项消费通道”

也就是说，并发组合真正关心的不是“并发”本身，而是：

`多条异步链最终怎样被重新折叠成一个你能继续 await、继续 foreach、继续处理的上层协议。`

这篇文章只做这一件事：

- 把 `WhenAll`、`WhenAny`、`WhenEach` 看成三种不同的收口模型
- 解释它们各自怎样处理结果、异常、先后顺序和后续链路
- 说明它们为什么和很多人脑子里的 `Task.WhenAll / WhenAny` 很像，但在 UniTask / Unity 现场里会产生额外工程后果

## 第一层：并发组合真正难的不是“怎么一起等”，而是“怎么重新收口”

很多人第一次使用并发组合时，脑子里默认的问题是：

- 我怎样同时发起几条异步链
- 我怎样在它们都完成后往下走
- 或者怎样拿到最先完成的一个

这些当然没错，但如果只停在这一层，你会很容易把组合 API 理解成语法糖。

更准确的看法应该是：

`组合 API 的本质，是把 N 条独立异步链重新压缩成 1 条上层协议。`

而这条新协议至少要重新定义四件事：

1. 上层何时被视为完成
2. 上层如何对待单个子任务的异常或取消
3. 上层交付的是一次性结果，还是按顺序流式交付
4. 子任务在上层已经决定结果之后，是否仍会继续推进并被消费

只要你问的是这四个问题，`WhenAll`、`WhenAny`、`WhenEach` 的差异就会立刻变得非常大。

## 第二层：WhenAll 是“整体终局协议”，不是“并发跑一下再收数组”

先看最常见的 `WhenAll`。

表面看，它表达的是：

- 所有子任务都完成以后
- 再往下走

但如果回到 UniTask 源码，这件事其实更精确：

- `WhenAllPromise<T>` 或 `WhenAllPromise` 会为每个子任务拿 awaiter
- 对每个 awaiter 注册 continuation
- 每个 continuation 自己调用 `GetResult()` 消费子任务终局
- 成功则填入结果槽位或递增完成计数
- 失败则立刻 `core.TrySetException(ex)`
- 当完成计数达到总长度，整体 `TrySetResult(...)`

这说明 `WhenAll` 的真正协议不是“拿数组”，而是：

`只要所有子项都成功，整体成功；只要任一子项在被消费时抛出异常，整体立即转入失败终局。`

### 1. `WhenAll` 关心的是整体是否已经形成一个单一终局

`WhenAll` 和普通 `await` 很像的一点在于：

- 上层最终还是只想得到一个结果
- 这个结果要么是所有项的成功聚合
- 要么是一个已经足够判定整体失败的异常终局

所以 `WhenAll` 的核心不是“并发”，而是“整体性”。

这也是为什么它非常适合表达：

- 这一组资源都准备好后才能继续
- 这批依赖都到齐后才能进入下一阶段
- 这几个前置异步步骤缺一不可

换句话说，它表达的是：

`这一批任务共同构成一个 gate。`

### 2. `WhenAll` 的异常语义不是“等所有失败都凑齐再讨论”，而是“谁先在消费路径上失败，谁先决定整体失败”

如果直接看 `WhenAllPromise<T>.TryInvokeContinuation`，逻辑其实非常直白：

- 先 `awaiter.GetResult()`
- 如果抛异常，`core.TrySetException(ex)` 并返回
- 成功才继续记录结果和递增计数

这意味着在 UniTask 的实现里，`WhenAll` 的聚合并不是“先收齐所有异常再一次性打包”，而是：

`谁先在聚合消费路径上抛出异常，谁就先决定整体 promise 的失败终局。`

这点很重要，因为很多开发者脑子里对“all”的直觉是：

- 那就应该把所有结果、所有异常都收完
- 然后统一交出来

UniTask 的实现没有走这个方向。它更像是在说：

- 我需要一个上层整体终局
- 任一失败已经足以判定“整体不成功”
- 后续子任务仍可能继续完成，但它们不再改变这个上层终局

### 3. 但这不等于“剩余任务就不再被消费了”

这里有一个很容易误判的点。

虽然 `WhenAll` 只要有一个子任务失败，就已经足以把上层 promise 设为 faulted，但其他子任务此前已经注册好的 continuation 仍会继续跑。那些 continuation 仍然会：

- 调用各自 awaiter 的 `GetResult()`
- 消费自己的成功值或异常
- 只是后续的 `TrySetResult / TrySetException` 会因为 core 已完成而不起决定作用

这意味着什么？

意味着 `WhenAll` 并没有在“第一处失败后就不再观察其他子任务”，而是在内部继续把它们的终局消费掉。工程后果是：

- 它不会因为其余子任务后续完成而天然造成一串未观察异常泄漏
- 但它也不会自动替你停止这些任务

也就是说，`WhenAll` 决定的是“上层结果”，不是“下层执行是否终止”。

### 4. 在 Unity 项目里，`WhenAll` 最容易被误用成“并发提速万能药”

`WhenAll` 很适合表达“这批都到齐才继续”，但它不天然意味着：

- 更好的分帧收口
- 更少的主线程峰值
- 更合理的对象生命周期边界

因为它只是在等待语义上并发，并不自动重排：

- continuation 落在哪个帧阶段
- 结果最终在哪一帧集中收口
- 实例化、反序列化、UI 刷新是否被同时压回关键帧

所以在 Unity 现场里，`WhenAll` 解决的是“整体完成条件”，不是“性能调度”。

## 第三层：WhenAny 是“第一个完成者决定上层”，不是“第一个成功者获胜”

再看 `WhenAny`。

如果只看名字，很多人会自动脑补成：

- 谁先成功，我就拿谁
- 剩下的就无所谓了

但 UniTask 源码真正表达的语义更严格：

`第一个把自己的终局带到组合 promise 上的子任务，决定上层结果。`

这里的“终局”包括：

- 成功
- 异常
- 取消

### 1. `WhenAny` 判断的是“第一个完成”，不是“第一个成功”

源码里的 `WhenAnyPromise<T>.TryInvokeContinuation` 和 `WhenAnyPromise.TryInvokeContinuation` 都遵循同一个顺序：

- 先 `awaiter.GetResult()`
- 如果这里抛异常，就 `core.TrySetException(ex)`
- 成功则 `Interlocked.Increment(ref completedCount)`，第一个成功写入结果

这意味着：

- 如果最先结束的是一个 faulted 子任务，上层 `WhenAny` 会直接 fault
- 它不会继续等“第一个成功者”
- 也不会把失败默默跳过，然后拿下一个成功结果

这和很多业务直觉不一样，但它是更一致的协议：

`WhenAny` 关心的是谁最先给出了终局，而不是谁最先给出了你喜欢的终局。`

### 2. `WhenAny` 特别适合表达“竞争条件”，但不适合假装自己在做“容错竞赛”

适合的场景是：

- 多个数据源抢一个最快响应
- 超时任务和业务任务竞争
- 左右分支谁先完成就走谁

不适合的场景是：

- 一堆可能失败的候选源，想“谁先成功就用谁，失败的都忽略”

后者如果直接用 `WhenAny`，就会遇到语义错位：

- 第一个失败者先完成，整个组合就 fault 了
- 你真正想表达的“第一个成功者获胜”其实根本不是 `WhenAny` 默认语义

这时候你应该显式写自己的容错竞赛策略，而不是期待 `WhenAny` 替你把失败当成没发生。

### 3. `WhenAny` 也不会自动取消“输掉的任务”

这是另一个非常高频的误判。

源码里 `WhenAny` 做的是：

- 给所有子任务注册 continuation
- 第一个完成者决定 core 结果
- 其他 continuation 后续如果仍然到来，只会因为 core 已完成而失去决定权

它没有做的事是：

- 自动给其余任务发取消
- 自动停止网络请求
- 自动撤销剩余加载链

这意味着一旦你把 `WhenAny` 用在：

- 请求竞速
- 资源竞速
- 首个条件达成即继续的场景

你就必须自己回答：

`那些没赢的任务，后面怎么办？`

不然你只是在“上层逻辑已经继续了”的同时，保留了一堆仍在后台推进的子任务。

## 第四层：WhenEach 根本不是“另一个 WhenAll”，它是在把并发结果重新变成流

`WhenEach` 是这三个里最容易被低估的一种。

因为它表面上也能“处理很多任务”，但底层心智完全不同。

源码里它不是返回一个单一 promise，而是返回 `IUniTaskAsyncEnumerable<WhenEachResult<T>>`。内部做法是：

- 创建一个单消费者无界 `Channel<WhenEachResult<T>>`
- 给每个子任务启动一个 `RunWhenEachTask(...).Forget()`
- 子任务完成后，把成功值或异常包装成 `WhenEachResult<T>` 写进 channel
- 全部完成时 `TryComplete()`
- 调用方通过异步枚举按完成顺序逐项消费

这说明 `WhenEach` 的真正协议不是：

- “把多任务压成一个总结果”

而是：

- “把多任务的完成事件重新展开成一条流”

### 1. `WhenEach` 保留的是完成顺序，而不是原始索引顺序

这是它和 `WhenAll` 最大的不同。

`WhenAll` 最终给你的是：

- 一个按原始索引对齐的结果数组

`WhenEach` 给你的则是：

- 谁先完成，谁先吐出来

这个差异在 Unity 项目里非常关键。因为很多场景真正关心的是：

- 谁先回来我先渲染谁
- 谁先下载完我先填哪个槽位
- 谁先准备好我就先把哪个条目展示出来

也就是说，`WhenEach` 更适合渐进式收口，而不是整体 gate。

### 2. `WhenEach` 把异常也包装成数据项，而不是默认让整个组合立刻 fault

`WhenEachResult<T>` 本身带两个通道：

- `Result`
- `Exception`

并提供：

- `IsCompletedSuccessfully`
- `IsFaulted`
- `TryThrow()` / `GetResult()`

这意味着 `WhenEach` 的异常哲学和 `WhenAll/WhenAny` 完全不同：

- `WhenAll/WhenAny` 里，异常是上层 promise 的立即终局候选
- `WhenEach` 里，异常首先会被包装成“这一项的完成结果”，再由消费者决定何时重新抛出或怎样处理

这就是把“整体终局协议”改成“流式结果协议”之后最直接的连带变化。

### 3. `WhenEach` 更像“异步版 merge 流”，不是“等待工具”

一旦这样理解，它的适用场景就很清楚了：

- 多个独立加载项谁先好就先显示
- 多个后台探测结果按完成顺序逐个处理
- 并发查询后希望逐条刷新 UI，而不是等全部结束

而它不适合表达：

- “这一批必须全部成功才能继续”
- “第一个完成就立刻选定，其他都不算”

所以 `WhenEach` 的价值不是“多一种组合 API”，而是：

`它把并发结果重新转换成一种流式消费模型。`

## 第五层：从异常与观察路径看，这三者其实代表三种不同的责任分配

把这三者并排放在一起，最容易看清的不是返回类型，而是责任边界。

### 1. `WhenAll`：整体责任

- 上层只关心“整体是否成功形成结果”
- 任一子项失败，整体失败
- 其余子项仍会在内部继续被消费，但不再改变整体终局

这是一种“总负责人”模型。

### 2. `WhenAny`：竞争责任

- 第一个完成者定义上层结果
- 失败也算完成
- 其他子项并不会被自动停止

这是一种“裁判先吹哨，赛道并没有自动清空”的模型。

### 3. `WhenEach`：消费者责任

- 框架只负责把每个完成事件流出来
- 成功还是失败都先包装成交付项
- 最终怎么处理每一项，由上层消费者自己决定

这是一种“框架只搬运结果，消费者自己决定每条消息怎么解释”的模型。

把它们写成一张判断表会更清楚：

- `WhenAll`：全部到齐才过闸
- `WhenAny`：第一个终局者决定走向
- `WhenEach`：每个完成者都成为一条独立消息

## 第六层：为什么它们在 Unity 里会有额外工程后果

如果这是纯后端世界，上面这些差异已经够用了。Unity 里还要再多看两件事。

### 1. 收口时机仍然活在 PlayerLoop 里

这三个组合 API 都没有取消前面主线已经讲过的事实：

- continuation 最终仍然要落在某个 `PlayerLoopTiming`
- 子任务结果的消费仍会发生在 Unity 的帧驱动环境里

所以即使你在等待语义上写对了，工程上仍然要考虑：

- 结果会不会在同一帧集中收口
- 渐进式 UI 刷新是否更适合 `WhenEach`
- `WhenAll` 是否会把一批主线程后处理压到同一时刻

### 2. 组合 API 不负责替你解决生命周期和撤销策略

前面 `06` 已经说过，真正难的往往不是 await 本身，而是谁负责终止 continuation。这里同样成立：

- `WhenAll` 不会因为某项失败就自动撤销所有其他链
- `WhenAny` 不会因为已经选出赢家就自动停掉输家
- `WhenEach` 更不会替你决定“这一项失败后整个页面要不要收口”

所以组合 API 只解决收口模型，不解决生命周期策略。

## 常见误解

### 误解一：`WhenAll` 就是更快地并发跑任务

不对。

`WhenAll` 表达的是整体完成条件，不自动保证更好的分帧、更少的主线程峰值，也不替你解决收口性能问题。

### 误解二：`WhenAny` 等价于“谁先成功就用谁”

不对。

它表达的是“谁先完成谁决定上层终局”，失败和取消同样可能先决定结果。

### 误解三：`WhenEach` 只是 `WhenAll` 的逐个回调版

不对。

`WhenEach` 在协议上已经不是“整体 promise”，而是把并发结果重新变成异步流；异常也首先以数据项的形式出现。

### 误解四：组合 API 会自动替你取消没用的子任务

不对。

源码里这三者都不提供这种自动撤销策略；如果业务需要，必须由你自己的取消与生命周期模型来接管。

## 最后把这件事压成一句话

`WhenAll、WhenAny、WhenEach 在 UniTask 里代表的是三种完全不同的收口哲学：WhenAll 把多条链压成一个整体终局，WhenAny 让第一个完成者决定走向，WhenEach 则把每个完成事件重新展开成流。它们看上去都叫“并发组合”，但真正决定工程后果的，从来不是名字，而是结果何时形成、异常怎样进入上层协议、以及其余子任务在上层已经决定之后是否仍然继续推进。`
