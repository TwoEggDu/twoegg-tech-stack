上一篇我先把最通用的恢复链单独立了出来：

`Serialized Data -> object records -> 引用恢复 -> Native Object -> Managed Binding -> 实例化 / 激活 / 子系统接入`

那一篇的重点，是把“Unity 的加载不是把文件读出来”这件事先讲清。

但如果只停在那里，很多人还是会有一个自然追问：

`既然 Scene、Prefab、AssetBundle 都会走到反序列化和恢复链，它们到底哪里一样，哪里又不一样？`

这件事如果不继续拆，项目现场就很容易把三种东西混成一团：

- 把 `AssetBundle Load` 误当成“资源已经完全恢复”
- 把 `Prefab Load` 误当成“实例已经在世界里”
- 把 `Scene Load` 误当成“只是读了一个大文件”

所以这一篇，我想专门把三者放在同一张图里比较一次。

## 先给一句总判断

如果要把这三者的差别压成一句话，我会这样说：

`Scene` 是整张世界对象图直接恢复并接入场景世界，`Prefab` 是模板对象图先被恢复，再通过 `Instantiate` 变成实例，`AssetBundle` 则只是把这些序列化内容和资源数据交付到运行时，它本身不是最终还原单位。`

这句话已经包含了这篇最核心的三点：

- 三者共享一大段通用恢复链
- 它们分叉的位置不同
- `AssetBundle` 不是和 `Scene / Prefab` 同层级的“对象图语义”，它更偏交付容器

## 一、先把三者放回同一条总链

为了避免一开始就掉进细节，我建议先看下面这条粗链：

`交付容器 / 资产文件 -> 序列化内容 -> 对象记录 -> 引用恢复 -> Native Object -> Managed Binding -> 实例化 / 激活 / 世界接入`

三者的共同点是：

- 都会碰到序列化内容
- 都会碰到对象记录和引用恢复
- 都会碰到 Native Object 创建
- 只要涉及脚本，都会碰到 Managed Binding

三者的分歧点则在于：

- 谁负责把序列化内容交到运行时
- 谁本身就是对象图
- 谁还需要额外一步 `Instantiate`
- 谁会直接进入场景世界

理解这几件事之后，很多“明明都叫加载，为什么行为完全不同”的问题就会变得很直白。

## 二、三者真正共享的是哪一段

先不要急着看差异，先看公共部分。

### 1. 都要先面对序列化描述，而不是现成运行时对象

无论你拿到的是：

- 一份 `Scene`
- 一份 `Prefab`
- 一个 `AssetBundle` 里装着的序列化资产

起点都不是“活着的场景对象已经在内存里等你”，而是：

`有一段序列化描述，等待被恢复。`

### 2. 都要走对象记录和引用恢复

只要里面装的是对象图或可引用资产，Unity 就要知道：

- 这里有哪些对象
- 这些对象彼此怎么连
- 哪些引用指向当前容器内部
- 哪些引用指向外部资产

这一段公共链，在 `Scene`、`Prefab`、`AssetBundle` 承载的序列化内容上都存在。

### 3. 都要继续走到引擎对象创建

Unity 不是把序列化字段扔给一堆 C# POCO 就完了。

它还要把这些描述恢复成引擎真正能用的对象。

这也是为什么三者后面经常都会牵扯：

- 对象创建
- 子系统注册
- 脚本绑定

### 4. 只要带脚本，都会撞到 Managed Binding

一旦资源里挂着脚本组件，这条链就会继续走到：

- `MonoScript`
- 程序集身份
- 类型解析
- 托管包装和脚本绑定

所以脚本问题会同时出现在：

- 场景加载
- Prefab 实例化
- AssetBundle 资源恢复

它们不是三类完全无关的问题，而是共享后半段恢复链。

## 三、Scene 是怎样还原成运行时对象的

在三者里，`Scene` 最接近“整张世界直接被恢复出来”。

### 1. Scene 的起点是一张世界快照对象图

上一篇已经讲过，Scene 文件保存的不是资源表，而是：

- `GameObject / Transform / Component`
- 场景内对象之间的引用
- 场景对象连向外部资产的引用
- 场景级上下文和配置

所以 Scene 的恢复，从一开始就是“整张图”的恢复。

### 2. Scene 不需要先变成模板再实例化

这是它和 Prefab 最大的差别之一。

Scene 加载时，目标不是先恢复一个“模板资产形态”，而是：

`把这张世界图直接放进当前场景世界。`

所以它恢复出来的对象，本来就是场景对象，不是模板对象。

### 3. Scene 恢复后会直接进入场景上下文

场景对象恢复完，不只是“对象存在了”，它们还会进一步进入：

- 当前 Scene 的层级和根对象集合
- 渲染、物理、动画、音频、UI 等系统
- 激活和生命周期阶段

也就是说，Scene 这条链更像：

`Scene 序列化图 -> 关系恢复 -> 场景对象创建 -> 场景上下文接入 -> 激活`

### 4. 这也是为什么 Scene 问题经常在进场景时集中爆

因为 Scene 恢复不是单个资源变可用，而是：

- 整批对象一起恢复
- 整批引用一起接回
- 整批脚本一起绑定
- 整批场景状态一起接入系统

所以 missing、引用断裂、激活异常、生命周期问题都容易在这一步一起显形。

## 四、Prefab 是怎样还原成运行时对象的

`Prefab` 和 Scene 很像，因为它也是对象图。
但它恢复链的关键分叉点在于：

`Prefab 先恢复的是模板图，不是世界实例。`

### 1. Prefab 的起点是一张模板对象图

Prefab 保存的是：

- 节点骨架
- 组件组合
- 模板默认字段
- 图内引用
- 图外资产引用

这和 Scene 一样也是对象图，但它的语义不是“当前世界长这样”，而是：

`如果以后要造一个这样的对象结构，模板长这样。`

### 2. 加载 Prefab Asset，不等于运行时对象已经进世界

这一步特别值得钉住。

很多人会把：

`Load Prefab`

理解成：

`对象已经被造出来了`

但更准确地说，先被恢复出来的是：

`Prefab Asset 对应的模板图表示`

这时它还不是最终进入场景或运行时世界的实例对象。

### 3. Prefab 还要再经过 `Instantiate`

Prefab 和 Scene 的关键分叉就在这里。

Prefab 恢复出模板图之后，通常还要继续走一段：

- 克隆模板图
- 重映射图内对象关系
- 生成当前实例自己的对象身份
- 把实例接入某个场景或运行时上下文

所以它的链更接近：

`Prefab 序列化图 -> 模板恢复 -> Instantiate -> 实例图生成 -> 激活 / 接入世界`

### 4. 为什么 Prefab 问题经常既像加载问题，又像实例问题

因为它天然横跨两段：

- 前半段是模板恢复
- 后半段是实例展开

于是很多问题就会看起来很绕：

- 到底是模板本身有问题
- 还是实例化时才出问题
- 到底是图没恢复好
- 还是实例层差异、override、脚本绑定才让它出事

这也是为什么 Prefab 话题总显得比普通资源更复杂。

## 五、AssetBundle 是怎样参与这条恢复链的

`AssetBundle` 最容易被误解，因为它经常和“加载”这个词绑得最紧。

但从结构上讲，它和 `Scene / Prefab` 不是同一层概念。

### 1. AssetBundle 先是交付容器，不是最终对象图语义

AssetBundle 最先解决的是：

- 内容怎样分发
- 怎样缓存
- 怎样按依赖组织交付物
- 怎样把序列化内容和资源数据带到运行时

所以它首先是：

`运行时可交付容器`

而不是“天然等于某个最终对象图”。

### 2. `LoadAssetBundle` 只是在把容器准备好

这一点必须和“资源已经可用”分开。

当你做：

- 下载 bundle
- 读取 bundle
- 解压或映射 bundle 内容
- 建立 bundle 容器对象

你得到的首先不是最终运行时对象，而是：

`一个已经就绪的资源容器`

它让后续的 `LoadAsset`、`LoadScene`、依赖解析和内容恢复成为可能。

### 3. 真正的恢复，还要看 bundle 里装的是什么

如果 bundle 里装的是：

- 普通资源对象
- `Prefab`
- `Scene`
- 材质、贴图、脚本可见资产

那么真正的后续恢复链，要按这些内容自己的语义继续走。

也就是说：

- bundle 里装 `Prefab`，后面仍然要走 Prefab 的模板恢复和可能的实例化
- bundle 里装 `Scene`，后面仍然要走 Scene 的世界图恢复
- bundle 里装普通资源，后面则停在资源对象恢复那一层

所以 `AssetBundle` 更像：

`把“谁能被恢复”带到运行时`

而不是：

`自己直接等于最终对象已经恢复`

### 4. 这也是为什么 AssetBundle 问题总像“多了一层”

因为它在通用恢复链前面又加了一段交付链：

`下载 / 缓存 / 依赖 / 容器打开 -> 具体资产恢复链`

所以你在 bundle 路径上看到的问题，常常有两层来源：

- 前半段是交付和容器层问题
- 后半段是具体资产内容自己的恢复问题

如果这两层不分开，就很容易把所有异常都粗暴归成“Bundle 加载失败”。

## 六、把三条链并排放一次

如果把三者并排压成最粗的版本，可以这样看。

### 1. Scene

`Scene 序列化对象图 -> 对象关系恢复 -> 场景对象创建 -> 场景上下文接入 -> 激活`

关键词是：

`整张世界直接恢复`

### 2. Prefab

`Prefab 序列化模板图 -> 模板恢复 -> Instantiate -> 实例图生成 -> 激活 / 接入世界`

关键词是：

`模板先恢复，再展开成实例`

### 3. AssetBundle

`Bundle 交付容器 -> 依赖满足 / 容器可读 -> 取出具体资产内容 -> 按内容类型继续走 Scene / Prefab / 普通资源的恢复链`

关键词是：

`先交付，再把具体内容交给各自的恢复链`

只要这三条链分清，很多名词混乱会立刻消失。

## 七、为什么项目里最容易把这三条链混在一起

### 1. 因为它们都被口头叫成“加载”

但工程上：

- `LoadBundle`
- `LoadAsset`
- `LoadScene`
- `Instantiate`
- `Activate`

根本不是一回事。

### 2. 因为它们共享中段恢复链

对象记录、引用恢复、Native Object、Managed Binding 这段本来就高度共用。

所以它们表面上不同，故障脸却很像。

### 3. 因为 AssetBundle 又在前面多包了一层交付语义

于是本来已经复杂的恢复链，前面又叠了：

- 下载
- 缓存
- 依赖
- 容器打开

看起来就更像一团。

## 八、工程里该怎么用这三条链排查问题

以后你在项目里看到 `Scene / Prefab / AssetBundle` 相关异常，我建议先别问：

`为什么这个资源又没加载出来？`

而先问下面四个问题。

### 1. 我现在处理的是对象图、模板图，还是交付容器

先确定你看的到底是：

- `Scene`
- `Prefab`
- `AssetBundle`

不要一上来就默认三者是同一种“资源”。

### 2. 问题发生在容器准备阶段，还是具体内容恢复阶段

如果是 bundle 路径，这一步尤其关键。

先分清：

- bundle 本身有没有准备好
- 依赖有没有满足
- 还是具体资产内容恢复时才出问题

### 3. 如果是 Prefab，问题停在模板恢复，还是停在实例展开

Prefab 这条链天生有两段。

如果这一步不分开，排查就会乱。

### 4. 如果是 Scene，问题是对象图没恢复好，还是已经恢复但没接入世界好

Scene 的问题也常常不是“文件没读到”，而是：

- 图内关系没接好
- 脚本绑定没接好
- 激活和子系统接入阶段才真正暴露

## 总结

最后把这篇压成三句话：

1. `Scene`、`Prefab`、`AssetBundle` 都会共享序列化内容解析、引用恢复、对象创建和脚本绑定这段公共恢复链。
2. `Scene` 的目标是整张世界图直接恢复，`Prefab` 的目标是模板图先恢复再实例化，`AssetBundle` 的目标则是把这些内容交付到运行时，并把后续恢复交给具体资产类型。
3. 很多所谓“加载问题”之所以难排查，不是因为名字太多，而是因为容器层、模板层、对象图层和实例层本来就站在不同位置，却经常被口头压成了同一个词。
