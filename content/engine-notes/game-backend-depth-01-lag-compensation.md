## 问题空间

玩家 A 网络延迟 80ms，对着屏幕上的玩家 B 开枪，子弹理应命中。但服务端收到射击包的时刻，B 已经移动了 80ms——服务端判定未命中。

这不是 bug，这是物理现实：信息传输需要时间。但玩家感知里，"我明明打中了"。如何让物理真实的网络延迟在游戏体验层面透明化，是延迟补偿要解决的核心问题。

问题的本质是**时间对齐**：客户端在 T 时刻看到的世界快照，是服务端 T-latency 时刻的状态。玩家在客户端 T 时刻瞄准并射击，这一行为在服务端验证时，对应的"世界时刻"应该是哪一帧？

不同的游戏给出了截然不同的答案，背后是完全不同的工程哲学。

---

## 抽象模型

### 延迟补偿的三种立场

**立场一：客户端权威**
客户端说打中了就打中了，服务端只做广播。极低延迟感，极易被外挂利用，几乎只出现在早期局域网游戏或单机模拟中。

**立场二：服务端时间回滚（Lag Compensation）**
服务端保存过去 N 帧的世界快照。收到客户端射击包时，根据客户端延迟，将服务端世界回滚到"客户端射击时看到的那一刻"，在该历史快照上做命中判定，然后将结果应用到当前帧。Overwatch、CS:GO、Quake 走这条路。

**立场三：严格服务端权威**
服务端持有完整世界状态，客户端的位置输入只作为"意图"，服务端重新计算最终位置。射击命中判定完全在服务端当前帧进行，不回滚历史。代价是感知延迟更高，但反作弊边界极其清晰。Valorant 的核心思想更接近这一极端。

现实中绝大多数竞技游戏是立场二的变体，但实现细节差异巨大。

## 具体实现

### Overwatch：hitbox 历史回滚

#### 位置历史缓冲区

服务端为每个玩家维护一个循环缓冲区，存储过去约 **200ms** 的骨骼/hitbox 快照。Overwatch 以 **63Hz** 的服务端 tickrate 运行（后期部分模式提升），每帧约 16ms，200ms 对应约 12-13 帧的历史深度。

```
PositionBuffer[player] = [
  { tick: T-13, hitboxes: [...] },
  { tick: T-12, hitboxes: [...] },
  ...
  { tick: T,    hitboxes: [...] }  // 当前帧
]
```

缓冲区是滚动覆盖的，旧于 200ms 的数据被丢弃。

#### 射击验证流程

1. 服务端收到客户端射击包，包中携带 **客户端发包时刻的本地时间戳**（或通过 RTT 推算）。
2. 服务端计算该射击包对应的"客户端视角时刻"：`client_fire_tick = current_server_tick - (rtt / 2 / tick_interval)`。
3. 从历史缓冲区中取出 `client_fire_tick` 对应的所有玩家 hitbox 快照。
4. 在该历史快照上执行射线检测（raycast）。
5. 命中结果应用到**当前帧**（扣血、死亡判定等）。
6. 恢复当前帧世界状态，继续正常推进。

这个过程在服务端是**瞬间完成**的，不是真正的"暂停时间"，而是读取历史数据进行一次离线查询。

#### 200ms 窗口的边界原因

为什么回滚窗口是约 200ms，而不是 500ms 或 50ms？

- **太短**：高延迟玩家（100ms+ RTT）的历史数据超出窗口，无法回滚，命中判定降级为当前帧验证，高延迟玩家感觉"打不中"。
- **太长**：存储和计算成本线性增加（每个玩家每帧都要存完整 hitbox），更严重的是"时间旅行"窗口扩大，公平性争议加剧。

200ms 是工程上的经验值，对应约 P95 的玩家延迟分布——覆盖大多数玩家而不为极端情况无限牺牲公平性。

#### 具体数据结构示意

```c
// 伪代码，非真实 OW 代码
struct PlayerSnapshot {
    uint32_t tick;
    Bone bones[BONE_COUNT];  // 骨骼变换
    AABB hitboxes[HB_COUNT]; // 命中体积
};

// 每个玩家的历史缓冲
PlayerSnapshot history[MAX_PLAYERS][HISTORY_DEPTH];

```

每帧结束时，当前帧快照写入缓冲区，指针推进。射击验证时按 tick 索引查找，O(1) 访问。

### Valorant：服务端权威的工程实现

#### 设计哲学的根本差异

Valorant 的反作弊目标极高。Riot 明确表示服务端不信任客户端发来的**任何位置数据**——玩家位置完全由服务端根据输入重新计算，客户端发送的是"我按了哪些键、朝哪个方向看"，而不是"我现在在哪里"。

这意味着：
- 服务端有完整的权威世界状态。
- 客户端的命中预测只是视觉反馈，最终裁决在服务端。
- 服务端 tickrate 提升到 **128Hz**，减少因帧率差异导致的判定误差。

#### 延迟补偿在 Valorant 中的存在形式

Valorant **并非完全不做延迟补偿**，但实现更为保守：

1. 服务端仍然保存位置历史，但回滚窗口更小（约等于服务端判定延迟，通常小于 Overwatch 的 200ms）。
2. 服务端对客户端延迟有更严格的上限限制——超过阈值的玩家可能被拒绝验证或被判定为异常。
3. 命中包必须携带服务端可验证的时序信息，服务端独立重建弹道，而不仅仅依赖客户端射线结果。

#### 服务端弹道重建

```
客户端发送：
  { fire_tick: 1024, yaw: 45.2, pitch: -3.1, weapon_id: 7, spread_seed: 0xABCD }

服务端执行：
  1. 取 tick=1024 时的权威玩家位置（服务端记录，非客户端上报）
  2. 用 yaw/pitch 计算射线方向
  3. 用 spread_seed 还原武器散布（客户端和服务端共享同一个随机数种子）
  4. 执行 raycast，取 tick=1024 附近的世界快照
  5. 返回命中或未命中
```

散布随机数种子同步是个细节：如果服务端用自己的随机数，两端散布不一致，玩家会看到命中但服务端判无效。Valorant 的方案是客户端上报种子，服务端用相同种子重现，但服务端会验证种子的合法性（防止客户端上报"永远不散布"的特殊种子）。

### 两种模型的本质对比

| 维度 | Overwatch 回滚模型 | Valorant 权威模型 |
|------|-------------------|-----------------|
| 命中判定基准 | 历史快照（客户端视角时刻） | 当前帧（服务端权威时刻）|
| 位置数据来源 | 服务端推算 + 历史存储 | 服务端完全重算，不信任客户端 |
| 延迟感知 | 低（客户端预测命中立即反馈） | 较高（需等待服务端确认） |
| 反作弊强度 | 中（hitbox 可被篡改的面较窄） | 高（位置完全由服务端计算） |
| Tickrate 要求 | 63Hz 基本够用 | 128Hz 以减少时序误差 |
| 存储开销 | 较高（历史快照 * 玩家数） | 中（历史窗口较小） |

## 工程边界

### "时间旅行"争议：躲进掩体后被打死

这是延迟补偿中最难解释给玩家的现象。

场景：玩家 B 延迟 80ms，玩家 A 延迟 20ms。B 在 T 时刻向右移动躲进掩体，A 在 T 时刻看到 B 刚开始移动（因为 A 看到的是 T-20ms 的 B）。A 开枪，射击包在 T+20ms 到达服务端。服务端回滚到 T 时刻，在该历史快照里 B 确实还没完全躲进掩体，判定命中。

B 的客户端显示自己已经在掩体后面，却收到了死亡通知。

**这是功能，不是 bug**：从 A 的视角，B 在掩体外，这是对 A 的公平保障。从 B 的视角，明明躲了却死了，这是对 B 的"惩罚"。

**工程上能改善但无法消除**：
- 降低 tickrate 误差（提高服务器频率）。
- 减小回滚窗口（但损害高延迟玩家体验）。
- 匹配时优先匹配延迟相近的玩家（减少两端延迟差异）。
- 对"掩体保护"做特殊处理（检测玩家是否在掩体遮挡物后，减少回滚判定的优先级）——部分游戏有实验性实现，但引入新的边界问题。

### 回滚窗口与反作弊的矛盾

回滚窗口越大，高延迟玩家的命中感越好，但外挂作者的操作空间也越大。恶意客户端可以伪造延迟（上报假时间戳），让服务端回滚到更早的历史帧，实现"预判"命中。

防御手段：
- 服务端独立测量客户端真实 RTT（通过心跳包，不依赖客户端自报）。
- 回滚时间不超过服务端测量 RTT 的合理上限（例如：RTT 20ms 的客户端最多回滚 30ms）。
- 滑动窗口异常检测：连续多帧要求超大回滚的客户端触发告警。

### 性能开销

每帧需要为所有玩家存储快照：

```
存储开销 = 玩家数 × 历史帧数 × 单帧快照大小
         = 12 × 13帧 × ~2KB
         = ~312KB（12 玩家，200ms，每帧 2KB）
```

对服务端来说不算大，但历史 raycast 的 CPU 开销需要注意：在一帧内可能需要验证多个射击事件，每次都要反序列化历史快照、重建 BVH 或类似加速结构。高并发房间数的服务器需要提前 benchmark。

## 最短结论

延迟补偿的本质是在"谁的时钟为准"这个问题上做政治决策，Overwatch 选择了体验优先，Valorant 选择了反作弊优先，两者都没有错，只有取舍。
