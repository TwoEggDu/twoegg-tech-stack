上一篇我先从构建期把 AssetBundle 拆开了：
它不是把一批项目文件简单压成一个包，而是沿着输入边界、依赖图、序列化内容和压缩策略，最后输出成运行时可交付的内容单元。

如果那一篇回答的是：
`资源怎么被编成 bundle`

这一篇要回答的就是：
`bundle 到了运行时之后，到底还要经历哪些步骤，资源才会真正变成可用对象？`

这个问题如果不拆开，项目里很容易把很多不同层的问题混成一句模糊的话：

- “bundle 已经加载了，为什么资源还不能用”
- “文件已经下载到了，为什么实例还没出来”
- “明明 `LoadAsset` 成功了，为什么场景里还是不对”
- “明明已经 `Unload` 了，为什么内存还没下来”

这些现象之所以总是纠缠在一起，就是因为运行时口中的“加载一个 AssetBundle”，从来都不是一步。

## 先给一句总判断

如果要把 AssetBundle 的运行时加载链压成一句话，我会这样说：

`运行时加载一个 AssetBundle，不是把一个“资源对象”直接读进内存，而是先取得交付容器，再满足依赖和缓存条件，然后从容器里取出具体资产，沿着反序列化与恢复链把它变成可用对象，最后再决定是否实例化、激活以及何时卸载容器。`

把这句话展开，大致就是下面这条链：

`定位内容 -> 下载 / 本地读取 -> 缓存命中 -> 依赖满足 -> 打开 bundle 容器 -> LoadAsset / LoadScene -> 反序列化 -> Native Object / Managed Binding -> Instantiate / Activate -> 可选 Unload`

这条链里最重要的不是 API 名字，而是层次边界：

- `下载 / 本地读取` 解决的是内容有没有拿到
- `缓存` 解决的是这次要不要重新拿
- `依赖满足` 解决的是恢复时缺不缺前置内容
- `LoadAsset / LoadScene` 才真正开始进入资产恢复
- `Instantiate / Activate` 决定对象有没有进入运行时世界
- `Unload` 处理的是交付容器和已加载对象的生命周期边界

只要把这些边界混了，排查就会立刻失真。

## 一、先拿到 bundle，不等于资源已经可用

很多人第一次接触 AssetBundle，最容易把“拿到 bundle 文件”和“拿到可用资源”当成同一件事。

但从运行时视角看，前者只是交付层动作，后者才是内容层动作。

### 1. 下载或本地读取，只是在取得交付容器

无论你是：

- 从 CDN 下载 bundle
- 从磁盘 `LoadFromFile`
- 从内存数据 `LoadFromMemory`

本质上都还停留在同一层：
`先把某个交付容器带到当前进程可访问的位置`

这一步解决的是“包在不在”，不是“对象活没活”。

所以即便 bundle 文件已经拿到了，下面这些事也都还没发生：

- 具体要用哪一个资产还没确定
- 资产对象还没开始恢复
- Prefab 还没实例化
- Scene 还没进入场景世界
- 挂脚本的对象也还没完成托管绑定

这也是为什么项目现场常见一种错觉：
“下载成功了，所以资源加载应该没问题。”

实际上，下载成功只说明交付链前半段通了。

### 2. 缓存命中，解决的是“这次还要不要再拿”

缓存这一层也经常被误解。

缓存解决的不是对象恢复，而是：

- 这次是否要重新下载
- 当前版本是否已经有可复用的本地副本
- 这份内容身份是不是和运行时期望一致

所以缓存命中之后，运行时得到的依然不是现成对象，而是：
`一个可以被重复打开的交付单元`

也就是说，缓存站在交付优化层，不站在资产恢复层。

## 二、打开 bundle 容器，不等于具体资产已经恢复

这一层是第二个最容易混的地方。

很多时候项目里会把“load bundle”和“load asset”说成一句话，但这两个动作在语义上并不处于同一层。

### 1. 打开 bundle，是让内容目录对运行时可见

当运行时把一个 bundle 打开之后，发生的事情更接近于：

- 识别这个容器里的内容结构
- 建立访问其中资产内容的入口
- 让后续 `LoadAsset`、`LoadAllAssets`、`LoadScene` 有路径可走

这里最关键的判断是：
`bundle 打开之后，容器可见了；但具体资产还没有必然变成运行时对象。`

它更像是“把仓库门打开”，而不是“把仓库里的东西已经摆到场景里”。

### 2. 真正开始恢复资产，是从 LoadAsset / LoadScene 这一层开始的

只有当你进一步去取具体内容时，运行时才会进入前面几篇文章已经铺好的恢复链：

`serialized data -> object records -> 引用恢复 -> native object -> managed binding`

也就是说，AssetBundle 在运行时的职责不是直接替代这条链，而是：
`把这条链所需的内容交付到运行时，并在需要时把入口交给它。`

所以从结构上看：

- `AssetBundle` 站在容器层
- `LoadAsset / LoadScene` 站在内容提取层
- `反序列化 / 恢复` 站在对象恢复层

这三层不是一回事。

## 三、依赖必须先满足，恢复链才站得住

上一篇从构建期讲过，AssetBundle 输出的不是孤立文件，而是一组有依赖关系的交付单元。

到了运行时，这件事不会消失，只会变得更明显。

### 1. 依赖 bundle 没满足，很多“加载成功”仍然可能是假的成功

如果一个 Prefab 所依赖的：

- 共享贴图
- 共享材质
- 共享 Shader
- 共享配置对象

落在别的 bundle 里，那当前 bundle 即使已经成功打开，也不代表这份内容就能完整恢复。

因为恢复链里一直都存在“外部引用接回去”这一步。

只要依赖没先满足，后面就可能出现这些现象：

- 资源能取到，但引用是空的
- 材质在，Shader 不对
- Prefab 在，实例效果不对
- 脚本组件挂着，但绑定失败

所以运行时真正需要建立的，不只是“我拿到了哪个 bundle”，而是：
`当前要恢复的这份内容，它所需的 bundle 依赖是否已经进入可用状态。`

### 2. 依赖问题经常伪装成渲染问题、脚本问题、实例问题

这也是 AssetBundle 现场最麻烦的一点。

表面上看，问题往往爆在：

- 粉材质
- 丢贴图
- missing script
- 实例化后对象不完整

但这些并不自动说明根因在渲染层、脚本层或者 Instantiate 本身。

更早的一层原因，常常是：
`恢复链需要的依赖内容压根没有先到位。`

所以排查时一定要先问：
这究竟是对象内容本身坏了，还是它引用的外部依赖还没被接回来？

## 四、Prefab、Scene、普通资产，在 bundle 里取出来后并不会走同一条后半程

前面 `C02` 已经讲过，`Scene / Prefab / AssetBundle` 共用一大段恢复链，但分叉点不一样。

放到运行时加载链里，这个差异会更直观。

### 1. 普通资产，很多时候到“对象可用”就停了

像：

- `Texture`
- `AudioClip`
- `Material`
- `AnimationClip`
- `ScriptableObject`

这类内容从 bundle 里取出来之后，往往主要目标是：
`把资产对象恢复到可用状态`

它们不一定还需要额外的实例化步骤。

### 2. Prefab 取出来后，通常还只是模板对象图

如果你从 bundle 里 `LoadAsset` 出来的是一个 Prefab，那么到这一步时，更准确的理解应该是：

`你拿到的是一个已经可被实例化的模板对象图，而不是场景里的活对象。`

后面还要再走一步：
`Instantiate`

只有到了这一层，对象才会真正变成运行时实例，进入层级关系、生命周期和场景世界。

这也是为什么很多项目里会出现一句误判：
“Prefab 已经 load 出来了，所以它应该已经在场景里了。”

并不是。

### 3. Scene 取出来后，目标是把整张世界图接入运行时世界

Scene 的情况又不一样。

如果你从交付层拿到的是场景内容，那么运行时真正要做的不是“拿到一个模板”，而是：

- 恢复整张场景对象图
- 接回对象之间的关系
- 接入场景级系统
- 进入激活与生命周期阶段

所以从运行时语义上看：

- `Prefab` 更偏向“模板恢复 + 实例化”
- `Scene` 更偏向“世界图恢复 + 接管激活”
- `AssetBundle` 只是把这两类内容运到运行时

把这三个词放在同一层说，基本都会出偏差。

## 五、Instantiate 不是加载附属步骤，而是对象真正进场的分界线

这一层值得单独拎出来，因为项目里非常容易低估它。

很多时候大家把注意力都放在：

- bundle 有没有下到本地
- bundle 有没有成功打开
- `LoadAsset` 有没有返回对象

但对 Prefab 来说，真正决定“它有没有进游戏里”的，恰恰不是前面这几步，而是：
`Instantiate 之后，这个模板对象图有没有变成运行时实例。`

也就是说：

- `LoadAsset<Prefab>` 解决的是“模板可不可取到”
- `Instantiate` 解决的是“实例有没有被创建出来”

很多看上去像“加载问题”的东西，实际上是实例阶段的问题，比如：

- 父子层级挂接不对
- 激活顺序不对
- 初始化依赖没准备好
- 组件脚本绑定后在生命周期里再出错

所以运行时一定要分清：
`取到资产` 和 `把资产变成场景里的对象` 不是同一步。

## 六、Unload 处理的是生命周期边界，不是“用完就一定全清掉”

`Unload` 是 AssetBundle 运行时链里另一个高误用点。

因为很多人会把它理解成一个非常朴素的动作：
“包用完了，卸掉就完了。”

但真正的难点在于，AssetBundle 是容器，而对象一旦从容器里恢复出来，就可能继续在运行时世界里存活。

### 1. `Unload(false)` 更接近“卸交付容器，不碰已加载对象”

高层理解上，可以把 `Unload(false)` 看成：
`当前不再保留这个 bundle 容器本身，但已经从中加载出来并被运行时持有的对象继续活着。`

它解决的是容器生命周期，不是对象生命周期。

所以做完这一步之后，项目里常见的现象是：

- bundle 已经卸了
- 但贴图、材质、Prefab 资产对象或实例仍然还在
- 场景里看起来也还正常

这不是“卸载失效”，而是因为你卸的是容器，不是所有对象。

### 2. `Unload(true)` 则会把“从这个 bundle 来的对象”也纳入卸载目标

高层理解上，`Unload(true)` 的危险在于：
它不只是关闭容器，还会把已经从这个 bundle 取出来的对象一起视作清理目标。

所以它经常不是一个简单的“省点内存”开关，而是一个很容易把后续引用链打断的动作。

项目里一旦对它的语义判断太粗，就会出现这些问题：

- 对象后面还要用，但来源容器和对象一起被清了
- 某些引用表面还在，实际内容已经不可用
- 场景过渡、对象池、共享资源在后续帧才暴露异常

这也是为什么 `Unload(true)` 和 `Unload(false)` 经常不是性能优化问题，而是生命周期设计问题。

### 3. 所以真正该问的不是“要不要 Unload”，而是“谁该活到什么时候”

从工程角度看，`Unload` 最该服务的不是“看见 API 就顺手调一下”，而是一个更前置的问题：

`这个 bundle 容器该活多久，这些从容器里恢复出来的对象又该活多久？`

只要这两个生命周期没有先设计清楚，`Unload` 大概率会变成延迟爆炸点。

## 七、很多 AssetBundle 运行时问题，本质上都是“把容器层和内容层混了”

如果把前面几节压缩成排查语言，最值得反复问的其实只有一件事：

`现在出问题的，到底是交付容器层，还是内容恢复层，还是实例/生命周期层？`

可以直接按下面这组问题去切：

- 是不是 bundle 根本没拿到
- 是不是缓存身份不对，拿到的是旧内容
- 是不是依赖 bundle 没先满足
- 是不是 bundle 打开了，但具体资产还没真正 load
- 是不是资产 load 出来了，但恢复链上的外部引用没接回去
- 是不是 Prefab 已经取到，但 `Instantiate` 后半段才出错
- 是不是对象正在使用中，却过早做了 `Unload`

只要这几层不拆开，现场讨论就会持续陷在这些模糊表述里：

- “加载应该没问题”
- “包已经有了”
- “资源已经出来了”
- “卸载过了，为什么还不对”

这些话最大的问题，不是信息少，而是层次不明。

## 八、这一篇真正想立住的判断

写到这里，其实最核心的判断只有两个。

第一个是：
`AssetBundle 在运行时首先是交付容器，不是现成对象。`

第二个是：
`运行时所谓“加载一个 bundle”，实际上横跨了交付、缓存、依赖、内容提取、对象恢复、实例化和生命周期管理多层边界。`

只要这两个判断站住，很多现场问题都会立刻变得可定位：

- 你是在拿包，还是在拿资产
- 你是在恢复对象，还是在创建实例
- 你是在卸交付容器，还是在动对象生命周期

这比背几个 `Load` 和 `Unload` API 更重要。
