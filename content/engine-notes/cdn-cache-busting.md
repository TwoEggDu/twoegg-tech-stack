做过热更新的人，迟早都会遇到一个很让人抓狂的问题：

CDN 上的文件明明已经更新了，客户端却还是拿到旧内容。

更让人困惑的是，上传的时候明明加了 `--cache-control no-cache`，理论上应该不缓存。但问题还是出现了。

要真正搞清楚这件事，需要先把 CDN 的工作方式说清楚，然后再来看这两种防缓存写法到底各自解决的是哪一层问题。

---

## CDN 是什么

CDN 的全称是 Content Delivery Network，内容分发网络。

很多人第一反应是"加速"，但加速只是结果，原因是：**把内容复制到离用户更近的地方**。

一个典型的 CDN 架构长这样：

```
用户设备
  │
  ▼
CDN 边缘节点（离用户最近的机房）
  │   ← 缓存命中：直接返回，不回源
  │   ← 缓存未命中：继续向上
  ▼
CDN 中间层（可选）
  │
  ▼
源站（S3 / 你自己的服务器）
```

当用户第一次请求一个文件，CDN 边缘节点本地没有，它会向源站取回来，同时把内容缓存在本地。下一个用户请求同一个 URL，CDN 直接从本地缓存返回，不再访问源站。

这就是 CDN 的核心逻辑：**用地理分布和缓存副本，减少每次请求的延迟和源站压力**。

---

## CDN 如何决定"要不要缓存"

CDN 主要看响应头里的 `Cache-Control`。

| 响应头 | 含义 |
|--------|------|
| `Cache-Control: max-age=3600` | 缓存 1 小时，时间内直接命中 |
| `Cache-Control: no-cache` | 可以缓存，但每次使用前必须向源站验证 |
| `Cache-Control: no-store` | 不缓存，每次都回源 |
| `ETag` / `Last-Modified` | 内容指纹，用于条件请求（304 Not Modified） |

注意 **`no-cache` 不是"不缓存"**。它的真实含义是：你可以存，但每次用之前要先问一下源站这个内容还算不算最新。如果源站说"没变"（304），才使用缓存内容。

所以当你用 `--cache-control no-cache` 上传 `versions.json` 到 S3 时，对 CDN 的效果是：CDN 每次收到请求都要回源验证，不会直接用旧缓存。**CDN 这一层的问题算是解决了**。

## 但还有另一层缓存

这是很多人忽略的地方。

HTTP 缓存不只存在于 CDN，**客户端本身也有 HTTP 缓存**。

在 Android 上，`UnityWebRequest` 底层走的是系统的 HTTP 栈（Java `HttpURLConnection` 或 OkHttp），它维护着一套本地磁盘缓存，行为遵循 HTTP 缓存规范。

这两层缓存是完全独立的：

```
CDN 边缘节点缓存
  ← 由响应头 Cache-Control 控制
  ← 你上传时加 --cache-control no-cache 影响的是这里

Android 客户端本地缓存
  ← 由 HTTP 响应头控制
  ← 也受 UnityWebRequest 发出的请求头影响
  ← 但 Unity 内部对这层缓存的控制是不透明的
```

当客户端发出请求时，数据流是这样的：

```
UnityWebRequest.Get(url)
  → Android HTTP 缓存：命中了吗？
    ↳ 命中且未过期 → 直接返回缓存内容，请求根本不发出去
    ↳ 未命中或已过期 → 真正发出网络请求
      → CDN 边缘节点：命中了吗？
        ↳ 命中且 no-cache → 回源验证 → 304 或 200
        ↳ 未命中 → 回源取 → 缓存并返回
```

你在 S3 上加的 `--cache-control no-cache` 作用在 CDN 这一层，但 **Android 客户端本地缓存**可能在请求发出之前就已经截断了整个流程。

## 两种修法，各自解决哪一层

### 方案 A：在请求头加 `Cache-Control: no-cache`

```csharp
using var request = UnityWebRequest.Get(url);
request.SetRequestHeader("Cache-Control", "no-cache");
request.SetRequestHeader("Pragma", "no-cache");
```

这是在**请求头**里加，而不是响应头。

请求头里的 `Cache-Control: no-cache` 的含义是：告诉沿途所有缓存层（包括 Android 本地缓存、CDN），这次请求不要直接用缓存，必须向源站验证。

听起来很完美，但有几个实际问题：

**第一，依赖实现的完整度。** HTTP 协议要求缓存层遵守请求头中的指令，但这是"应该"而不是"一定"。不同 CDN 的实现对请求头 Cache-Control 的遵守程度不完全一致。

**第二，Android 系统 HTTP 缓存的行为不透明。** Unity 在不同版本和不同 Android 版本上对 `UnityWebRequest` 的底层实现有差异，请求头是否被正确传递、系统缓存是否严格遵守，没有官方保证。

**第三，即使 CDN 遵守，也只是触发了"条件请求"。** no-cache 并不意味着回源取新内容，而是回源验证。如果内容没变，服务器返回 304，CDN 和客户端仍然使用缓存内容（虽然这次是正确的）。

总结：方案 A 是**向缓存层发出请求，要求它别用缓存**。能不能做到，要看对方是否配合。

---

### 方案 B：URL 追加时间戳

```csharp
string url = $"{baseUrl}?{DateTime.UtcNow.Ticks}";
```

这是 YooAsset 在请求 CDN 版本控制文件时的做法。

原理完全不同。

HTTP 缓存的 key 是 URL。同一个 URL，才可能命中缓存。一旦 URL 变了，缓存系统根本认不出这是"同一个资源"，必须重新发请求、重新回源、重新获取。

不是"请求缓存层别用缓存"，而是**让缓存层根本就认不出来这是它缓存过的资源**。

不依赖 CDN 的实现，不依赖 Android 系统的行为，不依赖 Unity 的内部处理，直接从语义上绕开所有缓存层。

代价是：CDN 边缘节点的缓存完全失效。`versions.json` 每个用户每次启动的 URL 都不一样，CDN 节点的缓存永远无法被别的用户复用，相当于把 CDN 降级成了一个纯代理。CDN 日志里也会出现大量碎片化的 URL，不利于分析。

但对 `versions.json` 这个场景来说，这个代价是完全可以接受的：

- 文件很小（几 KB），回源拿一次的代价几乎可以忽略
- 本来就配置了 `Cache-Control: no-cache`，CDN 缓存对它的价值本就不大
- 最重要的是：每次启动必须拿到最新值，绝对不能读旧的

---

## 为什么游戏里更常用方案 B

游戏热更新场景有一个特殊性：**版本控制文件错了，代价很大**。

如果 `versions.json` 读到了旧版本，客户端会认为不需要更新，继续用旧资源，出现数值表现错误、逻辑问题，甚至崩溃。这类问题复现困难、影响范围广、玩家感知强。

方案 B 是绝对可靠的：逻辑简单，没有依赖，不会出错。它的问题（CDN 缓存失效、URL 碎片）在这个场景里都不是真正的问题。

方案 A 则是"应该没问题"——HTTP 规范这么写，大部分情况下能跑通，但边界场景里可能悄悄失效，而且失效的时候非常难排查。

所以从工程角度看，**对于必须拿到最新值的版本文件，方案 B 更可靠**。

---

## 最后说一个容易混淆的点

很多人在上传 `versions.json` 时加 `--cache-control no-cache`，然后认为"我已经处理缓存问题了"。

这句话对了一半。

`--cache-control no-cache` 写的是**响应头**，控制的是 CDN 节点的行为和客户端收到响应后的缓存策略。

客户端**发请求时的缓存行为**——也就是"要不要把请求真的发出去"——是另一回事，它受客户端本地已有的缓存状态影响，需要用请求头或者 URL 变化来干预。

这是两件不同的事，夹在中间被大多数人忽略掉了。
