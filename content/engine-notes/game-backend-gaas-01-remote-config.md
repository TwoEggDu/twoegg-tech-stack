## 发版的成本

想象这样一个场景：游戏上线后，运营团队发现某个副本掉落率太高，市场上的稀有道具价格崩盘，需要紧急降低掉落率。

如果参数硬编码在客户端：
- 提交修改、测试、打包、上传各平台（iOS 审核可能需要 1-3 天）
- 等待玩家更新客户端（强制更新影响体验，可选更新则部分玩家继续在旧参数下游戏）
- 总计响应时间：1 天到数天

如果参数通过远程配置下发：
- 运营人员在后台修改参数值，点击保存
- 新客户端启动时拉取最新配置，或者服务端推送更新
- 总计响应时间：分钟级

这个差距，在 Live Service 游戏（持续运营的服务型游戏，即 GaaS）中意味着实质性的商业价值差异。

## 远程配置的本质

**远程配置（Remote Config）** 和热更新是两个经常被混淆的概念，需要先做区分。

**远程配置：** 修改游戏运行时的**参数值**，不改变代码逻辑。例如：副本掉落率从 5% 改为 3%，玩家登录奖励从 100 金币改为 200 金币，某功能的开关从关闭改为开启。客户端代码的逻辑分支不变，只是分支的判断依据（参数值）来自服务端。

**热更新：** 修改游戏的**代码逻辑或资产**，不发整包版本。例如通过 Lua 热更新修改战斗逻辑，通过 AssetBundle 更新 UI 界面。这需要客户端支持脚本解释执行或资产热加载机制。

两者的边界：
- 远程配置 = 改参数，代码预先写好了所有分支，只是读取不同的值
- 热更新 = 改代码/资产，改变的是程序行为本身

本文讨论的是远程配置，不涉及热更新架构。

### 远程配置的工作原理

```
服务端配置存储（数据库 / KV Store）
    ↓
配置分发服务（Config Service）
    ↓ HTTP 或 WebSocket
客户端拉取配置（启动时 + 定期轮询 + 服务端推送）
    ↓
客户端本地缓存配置
    ↓
游戏逻辑读取配置值（替代硬编码常量）
```

**客户端侧的配置使用模式：**

```csharp
// 不好的写法：硬编码常量
const float DROP_RATE = 0.05f;

// 好的写法：从配置管理器读取
float dropRate = RemoteConfig.GetFloat("dungeon_drop_rate", defaultValue: 0.05f);
```

`defaultValue` 参数很重要：当远程配置拉取失败时（网络问题、服务异常），游戏应该回退到合理的默认值，而不是崩溃或显示异常行为。

## Feature Flag：灰度发布的工程工具

Feature Flag（功能开关，也叫 Feature Toggle）是远程配置的一种特殊应用——配置值是布尔型（或枚举型），用来控制某个功能是否启用。

### Feature Flag 的四种主要用途

**1. 灰度发布（Gradual Rollout）**

新功能先对 5% 的用户开放，观察数据和崩溃率，再逐步扩大到 20%、50%、100%。如果新功能出现问题，立即关闭 Flag，影响范围控制在 5% 以内，而不是所有用户。

```json
{
  "feature_new_battle_system": {
    "enabled": true,
    "rollout_percentage": 5
  }
}
```

**2. A/B 测试**

同时向不同用户组展示不同版本的功能（或参数），对比关键指标（留存率、付费率），选择更好的版本全量推出。

```json
{
  "shop_discount_display": {
    "variants": {
      "control": {"show_discount": false},
      "treatment": {"show_discount": true}
    }
  }
}
```

**3. 紧急功能关闭（Kill Switch）**

生产环境发现某功能有严重 Bug（比如某个副本奖励配置错误，玩家可以无限刷稀有道具），立即通过 Feature Flag 关闭该功能，不需要发版。这是 Live Service 游戏的紧急响应标准工具。

**4. 环境差异化配置**

开发环境、测试环境、线上环境使用不同的配置值，通过远程配置统一管理，不需要在代码中写 `if (env == "dev")` 这类判断。

## A/B 测试的分组策略

A/B 测试的核心工程问题是：**如何确保同一个用户在多次访问中始终看到同一个版本？**

如果用户每次请求都随机分组，他可能第一次看到版本 A，第二次看到版本 B，体验不一致，也无法准确归因数据。

### 用户 ID 哈希分组

最简单且稳定的分组方案：对用户 ID 取哈希，再按哈希值的余数分组。

```python
def get_user_variant(user_id: str, experiment_name: str, num_variants: int) -> int:
    # 将实验名称加入哈希，使同一用户在不同实验中有不同分组
    hash_input = f"{experiment_name}:{user_id}"
    hash_value = int(hashlib.md5(hash_input.encode()).hexdigest(), 16)
    return hash_value % num_variants
```

**特点：**
- 确定性：同一用户 ID 在同一实验中始终得到相同分组
- 无状态：不需要在数据库存储用户分组，服务端无状态计算
- 均匀分布：哈希函数保证用户均匀分配到各组

**注意多个实验的交叉污染：** 如果用户同时参与实验 A（测试新付费界面）和实验 B（测试新副本奖励），两个实验的效果可能相互影响，导致数据分析困难。专业的 A/B 测试系统需要管理实验之间的互斥关系。

### 分组维度的选择

| 分组维度 | 适用场景 |
|----------|----------|
| 用户 ID | 最常用，测试用户感知变化（UI、奖励） |
| 设备 ID | 测试未登录场景 |
| 服务器 / 分区 | 测试服务端逻辑变化，同一服务器内体验一致 |
| 地区 | 针对特定地区的功能测试 |

## 参数热更新的实现

客户端获取最新配置有两种模式：

### 客户端轮询（Polling）

客户端每隔固定时间（如每 5 分钟）向服务端请求最新配置。

**优点：** 实现简单，服务端是纯 HTTP，无状态。
**缺点：** 更新有延迟（最多等待一个轮询周期），服务端承受持续的轮询请求压力。
**适用场景：** 不需要实时生效的参数（如活动配置、数值平衡参数）。

**优化手段：**
- 客户端携带本地配置的版本号（或 ETag），服务端对比后只在有更新时返回完整数据（304 Not Modified），减少数据传输量。
- 将配置文件托管到 CDN，减少源服务器压力。

### 服务端推送（Server Push）

服务端主动向客户端推送配置更新，通常通过 WebSocket 或长轮询（Long Polling）实现。

**优点：** 实时性高，配置修改后秒级生效。
**缺点：** 需要维护持久连接，服务端连接数管理复杂。
**适用场景：** 需要实时生效的紧急开关（Kill Switch、活动开始/结束）。

### 两层缓存架构

实践中通常组合使用两种模式：

```
配置管理后台
    ↓ 修改配置
服务端配置存储（数据库）
    ↓ 变更事件
消息队列（Kafka / Redis Pub/Sub）
    ↓
推送服务（Push Service）→ WebSocket → 在线客户端（实时）
    +
配置 CDN 缓存（HTTP 轮询）← 客户端定期拉取（兜底）
```

推送保证实时性，轮询保证可靠性（即使推送失败，下次轮询也能拿到最新配置）。

## 配置版本管理和回滚

配置本身需要版本管理，原因和代码一样：修改可能引入问题，需要有能力快速回滚到上一个已知正常的版本。

**最小配置版本管理方案：**

```
配置变更时：
  1. 保存当前配置为历史版本（版本号 + 时间戳 + 操作人）
  2. 写入新配置，分配新版本号
  3. 客户端下次拉取时获取新版本配置

回滚时：
  1. 从历史版本列表选择目标版本
  2. 将目标版本的配置内容写为当前配置
  3. 客户端获取到"新"配置（实际是旧版本内容）
```

回滚操作的响应时间，就是配置系统从修改到全量客户端生效的时间。这应该是分钟级别，不应该是小时级别。

## 常见滥用：配置系统的过度使用

远程配置系统的设计初衷是管理**参数值**，但在实践中经常被滥用：

**滥用 1：把游戏逻辑写入配置**

把复杂的业务规则（如奖励计算公式、战斗平衡算法）完全写入远程配置的 JSON 中，客户端解释执行。

问题：
- 失去了代码编译时的类型检查，配置写错就是运行时 Bug
- 无法用常规的代码调试工具调试
- 配置文件体积膨胀，客户端解析成本上升

**原则：** 配置管的是参数（数字、字符串、布尔值），不管的是逻辑（算法、流程）。

**滥用 2：所有功能都加 Feature Flag**

每个新功能都添加一个永久的 Feature Flag，导致代码里充满了 `if (flagEnabled("xxx"))` 的判断，代码路径爆炸，维护困难。

**原则：** Feature Flag 是临时的工具，灰度发布完成后应该移除旧路径代码，不应该长期维护两套逻辑。

## 工程边界

- 远程配置的生效时间不是"立即"：客户端从服务端拉取配置有延迟（推送延迟、客户端缓存 TTL）。对于需要立即生效的配置变更（如服务器紧急下线、关键功能 Kill Switch），需要评估实际生效时间是否满足要求。
- 配置系统本身也是基础设施，它的可用性影响游戏功能：配置拉取失败时，客户端必须能用默认值正常运行，不能强依赖配置服务可用。

---

## 最短结论

远程配置的本质是将参数的控制权从客户端发版流程中解耦，交给可以运行时修改的服务端存储。Feature Flag 是其特殊化应用——用布尔配置控制功能开关，支撑灰度发布和 A/B 测试。

**远程配置解决的是"速度"问题，不是"逻辑"问题**。把游戏逻辑塞进配置只会增加复杂度，不是正确的使用姿势。
