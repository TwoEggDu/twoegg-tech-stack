上一篇我刚把 `Scene` 立成了一张对象图：

`Scene 文件不是一个大资源，而是一份描述场景世界快照的序列化对象图。`

这一层一旦站稳，下一件最该做的事，就是把和它最容易混在一起的 `Prefab` 拆开。

因为项目里关于 Prefab 的很多误解，本质上都来自同一件事：

`大家会把 Prefab 同时当成资产、场景对象和运行时实例去想。`

于是很多问题就会开始打结：

- 为什么 `Prefab Asset` 和场景里的那个对象不是一回事
- 为什么有些改动看起来已经改了对象，但最后只是一个 override
- 为什么 nested prefab、variant、override 看起来都像“继承”或“嵌套”，其实不是同一种关系
- 为什么很多 missing、引用异常、实例恢复异常会集中爆在 Prefab 上

如果这层边界不拆清，后面你再去讲序列化恢复链、脚本挂载链、AssetBundle 里的对象恢复，就会一直悬空。

所以这一篇，我想专门回答一个问题：

`Unity 的 Prefab 文件本质上到底是什么？`

## 先给一句总判断

我会把 Prefab 压成一句话：

`Prefab Asset 不是场景里的对象实例，而是一份可被反复展开的模板对象图。`

这里最关键的不是“模板”两个字本身，而是它隐含了三件事：

- 它保存的是对象图，不是单一节点
- 它站在资产层，不站在场景层
- 它的核心价值不是“现在就在世界里”，而是“以后可以被重复展开成实例”

所以如果 Scene 更像：

`一张世界快照`

那 Prefab 更像：

`一张可复用的对象模板`

这两者都和对象图有关，但站位完全不同。

## 一、为什么 Prefab 不能只被理解成“可复用 GameObject 文件”

这是最常见，也最容易把人带偏的说法。

它不算完全错，但太粗了。

### 1. Prefab 不是单一 GameObject，而是一组对象关系

一个看起来很普通的 Prefab，序列化层面往往不只是一个节点。

它通常同时包含：

- 根 `GameObject`
- 整个 `Transform` 层级
- 每个节点上挂着的 `Component`
- 组件字段里的本地引用
- 指向外部资源的引用

所以 Prefab 和上一篇的 Scene 一样，本质上也不是“一个对象”，而是一张对象图。

### 2. Prefab 的重点不是“保存一个当前世界状态”，而是“保存一个可复用结构”

Scene 保存的是：

`这个世界当前是怎样的`

Prefab 保存的更接近：

`以后如果我要再造一个这种对象结构，应该按什么模板造`

这也是为什么同样都是对象图，Scene 和 Prefab 的语义完全不同。

### 3. Prefab 的真正价值是“反复实例化”

Prefab 之所以成为独立资产，不是因为它恰好也能保存一组对象关系，而是因为它可以被：

- 拖进多个场景
- 在一个场景里放很多次
- 运行时被加载并 `Instantiate`
- 被别的 Prefab 复用

一旦你从“实例化模板”这个角度看，Prefab 的很多复杂性就会开始变得合理。

## 二、Prefab 这张模板对象图里到底装了什么

如果先不碰 nested、variant、override 这些更高层概念，Prefab 最基础的结构，依然可以先抓三组东西。

### 1. 节点骨架：`GameObject / Transform / Component`

这和 Scene 一样，是 Prefab 最核心的骨架。

Prefab 保存的不是某个模型资源本身，而是：

- 哪些对象节点存在
- 它们的层级怎样组织
- 每个节点挂了哪些组件
- 组件上有哪些初始字段和值

所以从结构气质看，Prefab 更像“对象组合定义”，而不是“单资源文件”。

### 2. 图内引用

Prefab 内部对象之间也会互相引用。

比如：

- 子节点组件引用同一 Prefab 里的其他对象
- `Transform` 指向父节点
- 脚本字段指向图内别的组件

这意味着 Prefab 不是若干对象的并列存放，而是一张内部连通的图。

### 3. 图外引用

Prefab 还会连接到外部资产系统。

比如：

- `Material`
- `Mesh`
- `AnimationClip`
- `ScriptableObject`
- 音频、粒子、UI 相关资源

于是 Prefab 一样同时拥有：

- `图内边`：模板内部对象关系
- `图外边`：模板对象对外部资产的依赖关系

这也解释了为什么 Prefab 在构建、加载、恢复时都不可能被看成一个孤立节点。

## 三、Prefab Asset、Scene Instance、Runtime Instance 不是一回事

这一层是整个 Prefab 话题里最该钉死的边界。

### 1. `Prefab Asset` 是模板对象图

它站在资产层。

它的职责是保存：

- 对象结构
- 组件组合
- 默认字段值
- 内外部引用关系

它本身不是已经进入场景世界的对象。

### 2. `Scene Instance` 是模板在某个场景上下文里的展开

当你把一个 Prefab 拖进 Scene，或者在编辑器里在场景中放出一个 Prefab 实例，场景里出现的已经不是 `Prefab Asset` 本身，而是：

`这个模板在当前场景里的一个实例图`

它一方面保留着和源 Prefab 的关联，另一方面又已经进入当前场景上下文。

所以它天然会开始拥有一些只有实例层才有的东西：

- 自己在场景里的位置和父子关系
- 自己在当前场景里的引用关系
- 相对模板的局部改动

### 3. `Runtime Instance` 是模板在运行时世界里的实际对象

运行时 `Instantiate` 之后，真正参与渲染、逻辑、物理、生命周期的是运行时对象实例。

这和 `Prefab Asset` 也不是一回事。

所以如果强行压成一条链，Prefab 最该被这样看：

`Prefab Asset -> Scene Instance 或 Runtime Instantiate -> 运行时对象图`

一旦这三层混在一起，后面的 override、丢引用、实例异常就很容易看错。

## 四、为什么 Prefab 和 Scene 很像，但又不是同一种对象图

这两个概念最容易在脑子里打架，所以最好正面比较一次。

### 1. Scene 更像世界快照

它回答的是：

`这个场景当前世界怎样组织起来`

里面天然包含场景上下文、场景级配置、对象激活状态和世界入口。

### 2. Prefab 更像局部模板

它回答的是：

`如果我要复用这一组对象关系，模板长什么样`

它不是整张世界快照，而是世界里某类对象结构的可复用定义。

### 3. Scene 关心“当前在场”，Prefab 关心“以后能展开”

这就是两者最稳定的区分法。

只要你发现自己在把 Prefab 想成“当前这个场景里就应该长这样”，就已经开始朝 Scene 的思维滑过去了。

## 五、Nested Prefab 到底站在哪

`Nested Prefab` 很容易被误解成“Prefab 里再套一个普通子对象”。

它比这个更具体。

### 1. 它不是单纯层级嵌套，而是模板复用进入模板

如果说普通 Prefab 是一张模板图，那 nested prefab 表达的是：

`这张模板图里，有一部分子图来自另一张 Prefab 模板。`

也就是说，这不是简单的对象包含，而是：

`模板引用模板`

### 2. 它解决的是组合，而不是继承

Nested prefab 的重点是把多个模板图拼成更大的模板图。

它表达的更像：

- 这个角色 Prefab 里嵌了一个武器 Prefab
- 这个 UI 模板里嵌了一个通用按钮模板
- 这个关卡模块里嵌了另一个可复用模块

所以 nested prefab 的关键词不是“变种”，而是“组合”。

### 3. 它为什么容易让问题变复杂

一旦模板图里再挂模板图，很多问题都会变成多层来源的问题。

比如：

- 这个对象到底来自当前 Prefab，还是来自某个被嵌套的 Prefab
- 某个改动是改了当前模板，还是改了嵌套模板实例上的局部层
- 某个引用是在本模板图内，还是跨进了嵌套图

这也是为什么 nested prefab 一上来就会把“对象图边界”变得更难看清。

## 六、Variant 到底站在哪

`Variant` 和 nested prefab 很像，但它们解决的不是同一种关系。

### 1. Variant 不是组合，而是“基于模板的派生模板”

如果 nested prefab 是：

`模板 A 里放入模板 B`

那 variant 更接近：

`模板 B 继承模板 A 的结构，再加一层自己的差异`

所以 variant 的关键词不是“把别的模板放进来”，而是：

`在同一条模板谱系里派生`

### 2. Variant 站在资产层，不站在实例层

这一点非常关键。

Variant 自己仍然是 `Prefab Asset`，不是场景里的实例。

它不是“某个实例被改了点东西”，而是：

`一个新的模板资产，以另一个模板资产为基底。`

这也是为什么 variant 应该和 override 分开看。

### 3. Variant 解决的是“模板家族”，不是“单实例修改”

当你发现多个对象结构大体一致，但又需要一组稳定差异时，variant 就比复制 Prefab 再各改各的更有意义。

因为它表达的是：

`这是一类模板的稳定派生关系`

而不是一次临时实例改动。

## 七、Override 到底站在哪

很多 Prefab 相关混乱，最后都落在 override 身上。

因为 override 最容易被误读成“已经改了 Prefab 本体”。

### 1. Override 不是模板本体，而是“实例相对模板的差异层”

它的核心语义是：

`这个实例和它的源模板相比，哪些地方不一样`

所以 override 先天就站在实例侧，不站在资产侧。

### 2. 它可以出现在场景实例，也可以出现在更复杂的模板组合上下文

只要某个实例仍然和源模板保持关联，override 这层差异就有意义。

所以 override 讨论的不是“对象有没有被改”，而是：

`这个改动是写回源模板，还是只挂在当前实例这一层`

### 3. 它为什么经常成为问题来源

因为很多人看到 Inspector 已经变了，就默认：

`Prefab 本体被改了`

但工程上真实发生的可能只是：

- 当前场景实例挂了一层差异
- 某个嵌套实例挂了一层差异
- 某个 variant 相对基底模板再叠了一层关系

如果不把“模板资产”和“实例差异层”分开，后面一出问题就很难判断：

- 该修模板，还是修实例
- 该 apply，还是 revert
- 某个异常到底来自源模板，还是来自实例侧 override

## 八、为什么 Prefab 问题总显得特别复杂

现在把前面的边界拼起来，你会发现 Prefab 复杂，不是因为 Unity 故意把一个文件做难，而是因为它同时站在几条关系上。

### 1. 它既是对象图，又是模板

这已经比普通“资源文件”复杂一层了。

### 2. 它既能被场景实例化，也能被运行时实例化

所以它天然跨资产层、场景层和运行时恢复层。

### 3. 它还会叠加组合、派生和差异层

一旦把 nested prefab、variant、override 都叠上去，Prefab 讨论的就不再只是“一张模板图”，而是：

- 模板与模板的组合关系
- 模板与模板的派生关系
- 模板与实例的差异关系

这三种关系只要有一种没分清，问题就会显得像玄学。

## 九、工程里该怎么用这套边界排查 Prefab 问题

以后你在项目里看到 Prefab 相关异常，我建议先别问：

`这个 Prefab 怎么又坏了？`

而先问下面四个问题。

### 1. 我现在看的到底是模板、场景实例，还是运行时实例

这是第一问。

如果这一层没分清，后面所有判断都会偏。

### 2. 这个问题发生在模板图内部，还是实例展开之后

有些问题是模板本身的结构、引用、脚本绑定就不对。
有些问题则是实例进入场景或运行时之后才暴露。

这两类问题的排查入口完全不同。

### 3. 这是组合问题、派生问题，还是差异层问题

也就是先分清：

- `Nested Prefab`
- `Variant`
- `Override`

到底是哪一类关系在出问题。

### 4. 这是引用值丢了，还是恢复链没有把模板图正确接回运行时世界

很多 Prefab 问题表面上像“引用没了”。
但真实根因可能是：

- 模板图本身有问题
- 实例差异层改坏了关系
- 脚本身份链没接回去
- 运行时恢复阶段才真正暴露

所以 Prefab 问题往往不是一眼看 Inspector 就能定性的。

## 总结

最后把这篇压成三句话：

1. `Prefab Asset` 不是场景里的对象实例，而是一份可被反复展开的模板对象图。
2. `Nested Prefab` 解决的是模板组合，`Variant` 解决的是模板派生，`Override` 解决的是实例相对模板的差异层，它们不是同一种关系。
3. 很多 Prefab 问题之所以复杂，不是因为 Unity 把一个文件做复杂了，而是因为 Prefab 同时站在模板、实例、组合、派生和恢复链这几层交界处。
