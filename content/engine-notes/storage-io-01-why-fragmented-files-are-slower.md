很多项目在遇到加载抖动、首载卡顿或者 streaming 边界波动时，最容易冒出来的一句判断是：

`是不是硬盘太慢了？`

这句话有时对，但它经常还不够精确。

因为真正的问题，往往不是“盘慢”这么简单，而是这几层东西叠在一起：

- 存储设备本身的随机读写特性
- 文件系统处理路径、目录项和元数据的方式
- 操作系统怎样做页缓存、预读和 I/O 调度
- 游戏自己又怎样把读取、解压、反序列化和依赖展开连在一起

所以这篇文章想解决的，不是“劝大家一律合包”，而是先把一个更底层的问题讲清楚：

`为什么总字节数差不多时，海量碎文件往往比一个大整包更慢、更抖，也更容易在游戏里出事。`

---

## 这篇要回答什么

如果把这篇的定位说得更明确一点，它不是想把每一层都讲成独立教材，而是先给你一张“总图”：

1. 为什么碎文件问题不是“硬盘慢”三个字就能解释完。
2. 这笔账到底会在哪几层出现。
3. 为什么顺序、局部、可预测访问更容易被整条系统链路优化。
4. 为什么这些底层固定成本最后会在游戏加载链里一起变成体感卡顿。
5. 如果你想继续深挖，后面各篇应该分别接哪一层。

---

## 先把一句最短结论放在前面

如果你只先记一句话，我建议记这个：

> 碎文件更慢，通常不是因为“小”本身有原罪，而是因为它更容易放大随机访问、元数据查找、`open / close`、请求调度和首次依赖展开这些固定成本；而游戏又恰好特别怕这些账在关键时刻一起结清。

理解这句话之后，后面很多现象会顺很多：

- 为什么同样 200MB，整包常常比 2 万个小文件稳
- 为什么 NVMe 上看着没问题，到了慢盘用户就开始炸
- 为什么手机明明是 UFS，也照样会在首载和热机后抖

---

## 先把“碎文件更慢”拆成四笔账

如果想把这个问题真正讲稳，我更建议把它拆成四层，而不是只问一句“盘快不快”。

### 第一笔：存储设备在为随机访问和小事务收费

不管是 HDD、SATA SSD、NVMe、eMMC 还是 UFS，  
它们都更喜欢：

- 更连续的访问
- 更稳定的局部性
- 更容易批量处理的请求

它们都更怕：

- 很碎的小读
- 低效的小事务 storm
- 本来可以合并却被应用层切碎的请求

区别只在于，不同设备为这件事收的“固定成本”不一样：

- HDD 主要怕机械寻道和旋转等待
- SSD / NVMe 主要怕小事务把延迟和协议成本放大
- eMMC / UFS 在移动端还会叠加尾延迟、并行度和持续表现差异

如果你想把这一层拆细，看 [存储与 IO 02｜存储设备类型：HDD / SSD / NVMe / eMMC / UFS 的读写特性与延迟差异]({{< relref "engine-notes/storage-io-02-device-types-hdd-ssd-nvme-emmc-ufs.md" >}})。

### 第二笔：文件系统在为名字和管理动作收费

很多人一说读取，就只想到“把多少字节搬回来”。  
但真正的 `open(path)` 背后，系统还在做：

- 路径解析
- 目录项查找
- 元数据定位
- 文件对象和文件描述符管理
- `open / close` 前后的状态维护

如果这是一个大整包，这些动作很多只做一次。  
如果是几万次碎文件访问，这些固定成本就会被重复几万次。

所以很多加载问题并不是“正文读得很慢”，  
而是：

`系统先花了很多力气确认“你到底要找谁、它在哪里、它能不能读”。`

如果你想把这一层拆细，看 [存储与 IO 03｜文件系统基础：路径、目录项、元数据和页缓存为什么会影响游戏加载]({{< relref "engine-notes/storage-io-03-filesystem-path-directory-metadata-page-cache.md" >}})。

### 第三笔：操作系统只擅长优化顺序、局部、可预测的访问

整包更稳，不只是因为设备更喜欢顺序读，  
也因为操作系统更容易替这类访问兜底。

最典型的几个优化是：

- 页缓存更容易命中
- Read-Ahead 更容易猜对
- 请求更容易合并
- 队列深度更容易被设备有效利用

而碎访问天然会破坏这些前提：

- 数据页太散
- 复用间隔太长
- 下一个请求难以预测
- 请求粒度太小又太离散

这也是为什么“异步 I/O”不是免死金牌。  
异步通常只能改变等待方式，不能替你消灭碎访问本身的固定成本。

如果你想把这一层拆细，看 [存储与 IO 04｜OS I/O 机制：同步 / 异步 I/O、DMA、mmap、Read-Ahead、队列深度]({{< relref "engine-notes/storage-io-04-os-io-sync-async-dma-mmap-read-ahead-queue-depth.md" >}})。

### 第四笔：游戏会把这些底层账在关键时刻一起结清

这也是为什么同样的 I/O 问题，一到游戏里就特别容易变成体感问题。

对游戏来说，读取很少只停在 `read()`。

更真实的一条链通常是：

```text
read -> decompress -> deserialize -> resolve dependency -> create runtime object -> upload to GPU
```

如果资源组织得更集中，这条链更容易像一条线。  
如果资源本身很碎、依赖又在运行时临时展开，这条链就更容易变成一张网。

而游戏最怕的，恰好是：

- 首次命中
- 首次进场
- streaming 边界
- 关键交互帧

也就是说，真正危险的往往不是“文件很多”这件事本身，而是：

`这些账偏偏在玩家最敏感的时候一起收口。`

如果你想把这件事继续往游戏现场映射，看 [为什么一个大整文件，往往比很多小散文件更稳]({{< relref "engine-notes/game-performance-big-files-vs-small-files.md" >}}) 和 [读盘完成，为什么还是不等于资源可用]({{< relref "engine-notes/game-performance-read-does-not-mean-ready.md" >}})。

## 为什么这件事在 PC 和手机上看起来不像同一个问题

同一个碎访问问题，在不同平台上会长出不同外观。

### 在 PC 上

PC 更常见的放大器是：

- HDD / SATA SSD / NVMe 的跨度很大
- 后台软件和安装环境差异更明显
- 有些问题只在冷缓存和慢盘用户那里才暴露

所以 PC 上更常见的体感是：

- 开发机没事，用户机器突然很差
- 某些盘型和后台环境下首次特别抖

### 在手机上

手机没有 HDD，  
但它也没有因此自动摆脱碎访问问题。

手机上的放大器更常见的是：

- eMMC / UFS 档次差异
- 工作集更紧
- 前后台切换更频繁
- 热机后 CPU 解压和解析更容易变慢

所以手机上的外观更像：

- 一开始还能撑住，后面越来越不稳
- 资源重进场后抖动更明显
- 同一条加载链在热机和冷机下结论不同

如果你想把这一层拆细，看 [存储与 IO 05｜移动端存储特性：eMMC vs UFS 随机读写差异，碎访问为什么在手机上也照样危险]({{< relref "engine-notes/storage-io-05-mobile-storage-emmc-vs-ufs.md" >}})。

## 最后怎么把它落回工程判断

如果把这些底层机制翻成项目动作，我更建议先记下面这五条：

### 1. 先优先争取局部性，不要先迷信“文件越碎越灵活”

运行时频繁一起访问的资源，优先保证：

- 被成组组织
- 读取路径集中
- 依赖边界尽量收敛

### 2. 合包不是目的，服务顺序访问和稳定性才是目的

真正值钱的不是“把东西做大”，而是：

- 让系统更容易预读
- 让请求更容易合并
- 让依赖更少在关键帧现追

### 3. 异步只是在隐藏等待，不是在消灭工作量

如果链路后半段仍然很重，或者碎访问本身仍然很多，异步并不会自动把体验问题消掉。

### 4. 关注尾延迟和首次命中，不要只看平均吞吐

游戏运行时更怕的是：

- 偶尔特别慢
- 关键时刻突然收口
- 某些环境下突然失控

### 5. 把“读盘问题”重新命名成“加载链问题”

只要你开始这样命名，很多误判都会少很多。

因为这会逼你继续问：

- 慢在硬件、文件系统、OS I/O，还是链路后半段
- 是总量问题，还是访问方式问题
- 是平均慢，还是关键时刻抖

---

## 常见误解

### 误解一：只要上了 SSD，碎文件问题就不存在了

不对。SSD 消掉了机械寻道，但没有消掉元数据、请求粒度、队列和依赖展开这些成本。

### 误解二：异步 I/O 一上，就说明读取问题已经解决

也不对。异步只能帮你重叠等待，不能替你消灭碎访问和链路后半段的工作量。

### 误解三：只要总字节数一样，加载表现就应该差不多

不现实。文件数量、访问顺序、元数据访问、依赖形态和首次命中都会改变体验。

---

## 我的结论

如果把这篇文章压缩成一句话，我会这样总结：

> 海量碎文件之所以经常比大整包更慢，不是因为“大文件天然有魔法”，而是因为存储硬件、文件系统和操作系统都更擅长优化顺序、局部、可预测的访问；一旦访问变碎、依赖变散、首次成本又刚好在关键时刻收口，这些底层固定成本就会一起浮出水面，最终表现成游戏里的首载卡顿、streaming 抖动和跨设备不稳定。
