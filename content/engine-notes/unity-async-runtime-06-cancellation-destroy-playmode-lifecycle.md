> 如果只用一句话概括这篇文章，我会这样说：`Unity 里的异步问题，最难的常常不是“怎么继续”，而是“什么时候必须停止继续”；任务语义可以还活着，但对象世界、页面上下文、场景和 PlayMode 可能早就已经不该再让这段 continuation 落地了。`

前面的几篇文章把问题空间已经逐步立住了：

- `Task` 的世界观和 Unity 的帧驱动运行时并不完全重合
- Unity 不只是一个“主线程环境”，还是一个强 `PlayerLoop` 阶段环境
- UniTask 之所以会长出自己的 `PlayerLoop` runner 和 timing 语义，是因为 Unity 里的 continuation 不只是“回主线程”就够了

但真正到了项目里，最容易把系统拖进泥潭的，往往还不是这些调度问题。

更常见的爆炸点是：

`这段异步逻辑还在继续，但它想作用的那个对象、页面、场景，已经不该再继续了。`

你会在项目里反复见到下面这些现象：

- 页面关了，异步结果回来还在刷 UI
- 物体销毁了，continuation 还在访问组件
- 场景切了，旧场景里的任务链还在落回主线程
- PlayMode 退出了，编辑器里还有未收口的异步逻辑继续跑
- 逻辑本身“没错”，但它已经跨过了对象生命周期边界

这些现象背后的共同点是：

- 任务语义只会问“是否完成、失败、取消”
- Unity 对象世界还要额外问“这个对象、页面、场景、运行期上下文是否还存在”

如果这两套边界不接起来，异步系统就很容易表面上能跑，实际上长期不可维护。

所以这篇文章要回答的问题不是：

- `CancellationToken` 的 API 怎么写

而是：

- 为什么生命周期在 Unity 异步里会成为硬边界
- 为什么取消不是一种“可选礼貌”，而是结构性要求
- 为什么销毁、场景切换、PlayMode 退出这些事件，会把 continuation 语义从“以后继续”改写成“以后必须停掉”

## 第一层：为什么 Unity 里的异步总会撞上生命周期边界

如果先不讲任何 API，只看问题本体，Unity 里的异步之所以特别容易出生命周期问题，是因为这里同时存在两套时间轴：

- **任务时间轴**：这段逻辑何时完成
- **对象时间轴**：这段逻辑想作用的对象世界何时结束

在很多一般托管应用里，这两条时间轴也会错开，但没那么频繁、也没那么强烈。

而在 Unity 里，错开几乎是常态。

### 1. Unity 里的对象不是“只要引用还在就算还活着”

这是最核心的一点。

在普通托管对象世界里，很多对象的生命期基本可以粗看成：

- 只要还有引用，或者还没被 GC 回收，它就还在

但 Unity 里的很多关键对象不是这样。

例如：

- `GameObject` 可能已经被销毁
- `Component` 可能已经失效
- `Scene` 可能已经卸载
- 某个 UI 页面虽然对象引用还在，但业务上下文已经结束

也就是说，Unity 里的“对象还在不在”，不能只看托管引用是否为空。

它还要看：

- 引擎对象语义是否还成立
- 场景归属是否还有效
- 运行时上下文是否还允许这段逻辑落地

### 2. 异步会天然拉长逻辑跨度

同步代码的问题是顺序问题。

异步代码一旦展开，就天然带来时间跨度：

- 这一段现在发起
- 过几帧、几百毫秒、几秒、甚至更久才恢复
- 恢复时运行时世界已经变了

所以异步不是“制造生命周期问题”，而是：

`把原本藏在一个短同步调用里的边界暴露出来。`

### 3. Unity 的世界特别爱在这段时间里发生结构变化

游戏项目和一般后台服务不同，最常发生的事情就是：

- 页面关闭
- 面板切换
- 场景切换
- 物体销毁与重建
- 进入 / 退出战斗态
- 编辑器 PlayMode 开始与结束

这些变化意味着：

`你刚发起异步逻辑时依赖的那套对象关系，恢复时很可能已经不存在了。`

所以生命周期问题在 Unity 里不是边角料，而是主战场。

## 第二层：为什么取消在 Unity 里不是风格问题，而是边界契约

很多团队第一次引入 `CancellationToken` 时，会把它当成一种“更认真一点的代码风格”：

- 能 cancel 更优雅
- 不 cancel 也不是不能跑

这个判断在 Unity 里通常过于乐观。

更准确的理解应该是：

`取消在 Unity 里不是“可选美德”，而是把任务时间轴和对象时间轴接起来的最基本契约。`

### 1. 没有取消，你就没有“停止继续”的统一语言

异步模型天然会回答：

- 成功
- 失败
- 继续等待

但 Unity 还必须回答第四种情况：

- 这段逻辑虽然没有自然完成，也没有业务异常，但运行时上下文已经不允许它继续了

这第四种情况，如果不通过一种统一语义表达出来，项目就只能退化成：

- 到处 if 判空
- continuation 里到处补“对象还在吗”
- 出问题再补丁式拦截

这类系统短期看似能跑，长期几乎一定会烂。

### 2. 取消的价值不只是“尽早停止后台工作”

很多人谈取消，只盯一个收益：

- 既然页面关了，就别浪费后台算力了

这当然对，但还不够。

更重要的价值其实是：

- 明确告诉 continuation：恢复回来时，这段业务已经不再合法
- 明确告诉调用链上层：这里结束不是成功，也不是普通失败，而是上下文收口
- 给异常处理和日志系统一个统一的语义出口

也就是说，取消首先是**语义工具**，其次才是资源优化工具。

### 3. Unity 里的取消经常在做“上下文撤销”，不是“任务失败”

这也是为什么取消和异常不该混写。

例如：

- 面板关闭后不再需要结果
- 场景切换后旧请求应该作废
- 物体销毁后不该再更新视觉状态

这些都不是“逻辑出错”，而是：

`原本合法的 continuation，上下文已经撤销。`

这时如果没有明确取消语义，系统很容易把“本该正常收口”写成：

- 静默吞掉
- 普通异常
- 判空后直接 return

久而久之，异步链就不再可推理。

## 第三层：Unity 项目里最常见的四种生命周期边界

现在把抽象边界压回具体运行时现场。

### 1. GameObject / Component 销毁边界

这是最常见的一类。

典型链路是：

- 某个组件发起异步逻辑
- 逻辑在等待网络、资源、下一帧或某个条件
- 期间对象被销毁或场景切换
- continuation 恢复后还想继续访问该组件或其关联对象

这类问题最烦人的地方在于：

- 代码本身经常逻辑完全正确
- 出错点却在恢复时刻，而不是发起时刻

所以它特别容易被误判成：

- 偶发空引用
- 某个 API 调用顺序不稳定
- 某次资源加载“偶尔慢了一点”

其实根因是：

`异步逻辑的时间跨度已经超过了对象的合法生命周期。`

### 2. UI 页面 / 面板边界

很多 Unity 项目里最容易失控的异步逻辑都发生在 UI。

因为 UI 有几个特点：

- 交互频繁
- 开关切换频繁
- 用户操作速度不可预测
- 一次点击经常会发起网络、资源、动画、状态刷新等多段异步链

于是就很容易出现：

- A 页面发起请求
- 用户很快跳到 B 页面
- A 的结果回来时还在更新旧页面

这里“页面销毁”未必是物理销毁，也可能只是：

- 页面已隐藏
- 业务上下文已切换
- 当前激活页已经不是原来的那一个

这说明生命周期边界不只是引擎对象边界，还是：

`业务上下文边界。`

### 3. Scene 切换边界

场景切换会把很多看似局部的问题统一放大。

因为它同时可能触发：

- 对象树整体销毁
- 资源归属变化
- 单例状态切换
- 新旧上下文并存的短暂过渡期

场景切换最危险的不是“切了以后就全没了”，而是：

- 某些系统已经开始进新场景
- 某些旧任务还没结束
- continuation 可能在主线程合法恢复，但业务目标已经属于旧场景

也就是说，场景切换把“继续执行”这件事拆成了两个问题：

- 线程上是否还能继续
- 业务语义上是否还应该继续

后者往往更容易被漏掉。

### 4. PlayMode 退出 / 编辑器状态切换边界

编辑器环境下，这类问题会更诡异。

因为 PlayMode 退出意味着：

- 一部分运行时世界开始被回收
- PlayerLoop 不再按原来的方式持续推进
- 某些回调仍可能在退出期尾部被触发
- 某些异步逻辑在编辑器世界和运行时世界之间处在尴尬过渡期

这也是为什么很多“编辑器里偶发的异步错误”很难靠普通业务直觉定位。

它真正的根因通常是：

`运行时上下文已经结束，但异步协议本身还不知道它应该终止。`

## 第四层：为什么 Unity 的生命周期问题不能只靠“恢复时判空”解决

很多项目最初都会走到一个非常常见的折中写法：

- 异步回来以后先判一下对象还在不在
- 如果不在就 return

这个写法短期当然有用，但它不是完整模型。

### 1. 判空只能拦住结果落地，不能表达调用链已经撤销

如果只是恢复时判空，你顶多做到：

- 不再去碰失效对象

但你没有表达：

- 上层是否知道这条链已经取消
- 资源是否应该回收
- 后续分支是否还会继续跑
- 日志和异常是否应该被视为正常收口

也就是说，判空是在补末端防护，不是在建立统一语义。

### 2. 判空会把“正常撤销”和“意外失效”混在一起

例如：

- 页面正常关闭后不再更新 UI
- 对象意外提前被销毁

这两件事在工程上并不等价。

如果都用“回来后判空然后 return”收掉，系统就失去区分能力了。

### 3. 判空不解决中间资源和订阅关系的收口

异步链里经常还伴随：

- 注册回调
- 监听事件
- 定时等待
- 后台任务仍在推进

光在末端判空，并不能保证这些中间状态已经正确收口。

## 第五层：为什么 UniTask 会特别强调取消扩展和生命周期接线

到这里，可以把视角收回到 UniTask 世界。

UniTask 之所以会特别强调：

- `CancellationToken`
- 与对象销毁绑定的 token
- 无捕获 ExecutionContext 的注册方式
- PlayerLoop 中的取消检查

并不是它“比 Task 更讲究”，而是 Unity 的运行时现实逼着它必须把这些边界前置。

### 1. 取消必须贴近对象世界，而不是只贴近任务世界

在一般 Task 世界里，你完全可以把取消理解成：

- 调用者想不想继续等

而在 Unity 里，更高频的现实是：

- 这个对象世界本身已经不该再承接这段 continuation

所以取消必须和：

- 对象销毁
- 页面关闭
- Scene 切换
- PlayMode 退出

这些边界接起来。

### 2. 取消检查必须足够便宜、足够统一、足够可组合

如果每条异步链都自己临时发明一套“还能不能继续”的判断：

- 代码会四分五裂
- 调试会失去统一出口
- 错误收口会高度依赖人肉习惯

这也是为什么生命周期问题最终一定会把系统逼向：

`统一 cancellation 语义 + 统一收口策略`

### 3. 取消不是和成功、异常对立，而是补上一种“上下文终止”状态

这是很多团队真正需要建立的心智模型。

在 Unity 里，一条异步链的终局至少应该被拆成：

- 正常完成
- 业务异常
- 生命周期 / 上下文终止

如果第三种状态没有被正式命名，系统就会不断把它伪装成前两种之一。

## 第六层：工程上最容易出现的失败模式

现在把这些抽象压成项目里最常见的失败模式。

### 1. “逻辑没错，但已经晚了”

这是 Unity 异步里最典型的一类问题。

发起时一切都合法，恢复时也没有明显语法错误，但业务语境已经不对了。

这类问题的本质不是代码错，而是：

`恢复时刻已经越过上下文边界。`

### 2. 取消做成了局部习惯，而不是系统契约

常见表现是：

- 有的调用链传 token
- 有的调用链不传
- 有的页面关掉时 cancel
- 有的靠判空兜底
- 有的 fire-and-forget 完全没有收口

这类系统短期能跑，长期排障极其痛苦，因为谁负责终止 continuation 已经没有一致答案。

### 3. 退出期错误被误当成“编辑器偶发 bug”

很多 PlayMode 退出期错误并不是编辑器怪，而是：

- 任务链仍在继续
- 运行时世界已经拆掉
- 你没有把这当成正式生命周期边界来处理

### 4. 正常撤销和真正异常混在一起

如果系统里取消语义不清，日志里就会出现两种很糟糕的结果：

- 正常页面关闭也在报错误
- 真正的生命周期管理缺失被埋没在“偶发取消”里

## 第七层：更稳的收口原则是什么

这篇不写 API 清单，但需要把工程判断压成几条明确原则。

### 原则一：每条会跨帧、跨页面、跨场景的异步链，都必须明确回答“谁负责终止它”

不是“理想情况下谁会记得取消”，而是：

`结构上谁负责终止。`

### 原则二：对象级逻辑、页面级逻辑、场景级逻辑要分别绑定自己的生命周期边界

不要把所有异步逻辑都寄希望于一个全局 token 兜底。

因为不同层次的结束条件根本不是同一件事。

### 原则三：把“上下文终止”当成正常终局之一，而不是异常分支里的附属品

只有这样，日志、监控、调用链和回调收口才会变得可推理。

### 原则四：不要把“恢复时判空”当成主方案，它只能做最后一道防护

真正的主方案必须是：

- 统一的取消语义
- 统一的生命周期绑定
- 统一的收口路径

否则项目越大，异步链越长，补丁式判空越会失控。

## 常见误解

### 误解一：取消只是优化，不影响正确性

不对。

在 Unity 里，取消首先影响的是正确性，因为它在表达“这段 continuation 已经不再合法”。

### 误解二：对象销毁后回来判空就够了

不对。

判空只能拦住最末端的一次落地，不会自动替你收掉中间状态、资源和语义分支。

### 误解三：PlayMode 退出期报错是编辑器特例，不算真实问题

不对。

它只是把你平时没有正式处理的生命周期边界，以更诡异的方式暴露出来。

### 误解四：异步链只要能成功完成，就说明生命周期设计没问题

也不对。

很多生命周期设计的问题只会在：

- 用户快速切页
- 低端机慢恢复
- 场景切换中断
- 编辑器退出

这些边界条件里暴露出来。

## 最后把这件事压成一句话

`Unity 里的异步问题，真正难的常常不是“怎么继续”，而是“什么时候必须停止继续”；取消在这里不是可选编码风格，而是把任务时间轴和对象时间轴接起来的边界契约。只有把 GameObject、页面、Scene 和 PlayMode 的结束条件正式接进异步模型，continuation 才不会在对象已经不该继续时继续落地。`
