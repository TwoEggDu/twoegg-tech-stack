## 为什么需要客户端检测

服务端权威能拒绝逻辑上不合理的操作，但有两类威胁它无法触碰：

一是透视外挂（ESP）——外挂读取游戏内存中的敌人坐标并在屏幕上标注，这件事完全发生在客户端，没有任何异常数据包发往服务端。

二是 Aimbot——瞄准动作在协议层面和正常玩家没有区别，服务端收到的只是一个合法的鼠标移动指令。

服务端对这两类行为是盲的。客户端完整性检测的存在，就是为了在客户端这一侧建立一条防线，检测那些服务端看不见的本地篡改行为。

---

## Ring3 vs Ring0：检测层级的本质差异

Windows 系统将权限分为 Ring0（内核态）和 Ring3（用户态）。大多数程序（包括游戏）运行在 Ring3。

### Ring3 检测

反作弊程序作为普通用户态进程或游戏内 DLL 运行，能做的事情：

- 扫描游戏自身进程的内存，检测已知外挂特征码
- 枚举运行中的进程和已加载的模块（DLL）
- 检测进程句柄（是否有外部进程打开了游戏进程）
- 监控 Windows API Hook（如 `OpenProcess`、`ReadProcessMemory` 是否被拦截）
- 检测调试器（`IsDebuggerPresent`、时序检测）

Ring3 检测的致命弱点：**外挂可以以同等权限进行对抗**。一个 Ring3 外挂可以 Hook 反作弊的 API 调用，让它看到假的进程列表、假的内存内容。

### Ring0 检测（驱动级）

驱动作为内核模块运行，具有最高权限：

- 直接访问任意进程的内存，无法被 Ring3 程序拦截或欺骗
- 监控内核回调（进程创建/销毁、模块加载、线程创建）
- 检测内核层面的 Hook 和 SSDT（系统服务描述表）篡改
- 通过 `MmGetSystemRoutineAddress` 等验证关键内核函数完整性

Ring0 检测让外挂对抗的门槛大幅提高——要绕过它，外挂也必须写成内核驱动，需要合法的代码签名（Windows 要求）或者利用系统漏洞进行驱动签名绕过（EV 证书劫持、DSE 禁用等）。

现代主流反作弊（EAC、BE、Vanguard）都使用 Ring0 驱动。

## VAC 的延迟封禁策略

VAC（Valve Anti-Cheat）有一个显著特征：检测到外挂后，不立即封号，而是延迟数天甚至数周执行封禁。

**为什么要延迟？**

原因是信息对抗。外挂作者会用"小号"测试自己的外挂是否被检测，如果封禁是即时的，测试结果非常明确——"被检测了，立刻改"。延迟封禁的策略是：

1. 检测到特征，记录但不立即封
2. 批量积累一批"被标记"的账号
3. 在某个时间点集中执行封禁（VAC Wave）

这样做的效果：
- 外挂作者无法通过测试账号的封禁时间点来判断"哪个版本暴露了"
- 在集中封禁前，作者可能已经把外挂卖给了大量用户，波及面更大
- 延迟增加了逆向分析 VAC 检测逻辑的难度（因为不知道是哪次检测触发的）

延迟封禁的代价是：在封禁执行前，被检测到的用户仍在游戏里损害其他玩家的体验。这是一个可接受的取舍：封禁的精准性和不可逆转性优先于即时性。

## EasyAntiCheat 和 BattlEye 的行为签名

EAC 和 BE 不仅依赖静态特征码扫描，还使用行为特征检测：

**静态特征码扫描**：维护已知外挂 DLL/驱动的哈希库，扫描内存中是否存在匹配的字节序列。缺点是多态外挂可以通过运行时加密绕过。

**行为签名（启发式检测）**：
- 监控异常的内存操作模式（大量跨进程读取）
- 检测典型的注入特征（`CreateRemoteThread`、`NtMapViewOfSection` 模式）
- 监控鼠标/键盘输入的时序特征（是否来自合法的 HID 驱动路径）
- 检测游戏模块的完整性（代码段 checksum，防止内存 Patch）

**进程和模块白名单/黑名单**：
- 维护已知恶意软件和外挂工具的进程名、模块名黑名单
- 对部分受保护的游戏实施白名单模式，不在白名单的模块禁止加载

BattlEye 额外特征：采用随机化的检测时机，不以固定间隔扫描，增加外挂预测检测时间窗口的难度。

## 客户端检测的架构模式

```
┌─────────────────────────────────────┐
│           游戏进程 (Ring3)           │
│  ┌──────────────────────────────┐   │
│  │   反作弊 SDK (Ring3 组件)    │   │
│  │  - API 调用监控              │   │
│  │  - 内存完整性检查            │   │
│  │  - 进程枚举                  │   │
│  └──────────────────────────────┘   │
└─────────────────────────────────────┘
          ↕ 通信（加密）
┌─────────────────────────────────────┐
│        反作弊驱动 (Ring0)            │
│  - 内核回调监控                      │
│  - 跨进程内存访问检测                │
│  - 驱动完整性验证                    │
└─────────────────────────────────────┘
          ↕ 上报
┌─────────────────────────────────────┐
│         反作弊服务端                 │
│  - 收集检测报告                      │
│  - 关联分析（同账号多端）            │
│  - 封禁决策（即时/延迟）             │
└─────────────────────────────────────┘
```

客户端检测的数据会上报到服务端进行综合判断，单纯依赖本地决策有被绕过的风险。

## 根本局限：你永远不能信任客户端

这是客户端检测最重要的哲学前提：**无论客户端检测做得多强，都存在理论上的绕过路径**。

原因很简单：检测程序和外挂程序运行在同一台物理机器上，而机器的所有者是玩家（可能是外挂用户）。在完全掌控硬件的情况下，攻击者可以：

- 使用虚拟机绕过驱动检测（Hypervisor 层劫持）
- 使用内核漏洞 (EoP) 提权绕过驱动保护
- 使用 DMA 设备（直接内存访问硬件）从另一台机器读取内存，完全绕过软件检测
- 使用物理外设（Arduino 模拟键鼠）绕过输入检测

这意味着：**客户端检测是提高攻击门槛的手段，不是构建安全防线的基础**。真正的防线必须建立在服务端。

## 工程边界

- 驱动级反作弊（Ring0）会在系统层面引入风险，曾出现反作弊驱动导致系统 BSOD 的事故（如 Vanguard 早期版本）；部署前需要充分的兼容性测试
- 反作弊 SDK 的更新频率需要跟上外挂迭代速度，静态特征库需要持续维护
- 用户隐私争议：Ring0 驱动在游戏之外仍持续运行（如 Vanguard），会引发用户反感和监管合规风险
- 多平台游戏（PC + 主机）需要不同的检测策略，主机平台由于封闭系统，内存修改类外挂难度极高

---

## 结论

客户端检测通过 Ring0 驱动提高了外挂的攻击成本，但在攻击者完全掌控硬件的前提下，它只是一道门槛而非边界，游戏安全的最终底线必须由服务端权威来保证。
