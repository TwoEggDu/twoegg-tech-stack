> 如果只用一句话概括这篇，我会这样说：模型、材质和贴图不是"拼在一起的三块东西"，而是三份数据在同一条流水线上各自扮演不同角色——模型决定形状，材质决定计算规则，贴图是被按 UV 地址查询的数据。

上一篇我们从整体上看了图形管线的分工：顶点处理、光栅化、像素着色、深度测试、混合。但那篇停在了"每一步负责什么"，还没有回答：你放进场景里的那个角色或者那面墙，它的数据具体长什么样，每一步拿到的是什么，又在往后传什么。

这篇就来补这个细节。

---

## 模型不是"立体的图"

"模型"这个词很容易让人误解，以为它是某种"立体版本的图片"。但实际上，一个 3D 模型在内存里是一组数字，大致分两部分：

**顶点数据（Vertex Buffer）**：每个顶点是一个数据包，通常包含：

- **Position**（位置）：`(x, y, z)`，这个点在模型自身坐标系里的位置
- **UV**（纹理坐标）：`(u, v)`，这个点对应到二维贴图上的哪个位置，u 和 v 通常在 0 到 1 之间
- **Normal**（法线）：`(nx, ny, nz)`，这个点所在表面朝向哪里，用于光照计算
- **Tangent**（切线）：配合法线贴图使用，定义表面切空间
- **Vertex Color**（顶点色）：有时候用来存额外信息，比如遮蔽值、区域权重

**索引数据（Index Buffer）**：一组整数，三个一组，每三个整数指向顶点缓冲里的三个顶点，组成一个三角形。整个模型的表面，就是这些三角形拼起来的。

所以一个"球"在 GPU 眼里不是一个圆，而是几百个三角形，每个三角形是三个顶点的位置坐标。一个角色脸部的皮肤细节，也不是贴在脸上的——在这个阶段，脸只是一堆三角面，什么颜色都没有。

## 材质不是"贴在表面的图"

材质也是个容易混淆的概念，特别容易被等同于"贴图"。

一个材质由两部分组成：

**Shader（着色器）**：一段在 GPU 上运行的程序，定义了"这个表面应该怎么计算颜色"。同一面墙可以用一个 Shader 算出"石头感"，也可以换一个 Shader 算出"塑料感"——贴图是一样的，但计算规则不同，结果就不同。

**参数集（Property Block）**：Shader 运行所需的输入值，包括：
- 颜色值（`_BaseColor`、自发光颜色）
- 浮点数（粗糙度 Roughness、金属度 Metalness、法线贴图强度）
- 贴图引用（`_BaseMap`、`_NormalMap`、`_MetallicGlossMap`）

材质是这两者的组合实例：同一个 Shader 可以被多个材质使用，只是参数不同。同样是 URP/Lit，一个材质金属度高、粗糙度低，另一个金属度低、粗糙度高，最终画面差别很大。

**材质本身不包含颜色数据**，它包含的是"怎么用贴图和参数算出颜色"的规则。

---

## 贴图不是"贴在表面的"

"贴图"这个名字暗示了一种错误理解——好像 GPU 把一张图片"粘"在模型表面上。实际上更接近这个比喻：

> 贴图是一个二维数组，GPU 用 UV 坐标作为地址去查询它，取出对应的值。

一张 `1024×1024` 的 BaseColor 贴图，就是 1024×1024 个颜色值。UV 坐标 `(0.25, 0.75)` 会被映射到这个数组里某个位置，取出一个 `(R, G, B)` 值，然后这个值会参与后续的光照计算。

同一个顶点的 UV 坐标在光栅化阶段会被插值到每个像素，每个像素用自己的 UV 值去贴图里取样。这就是为什么贴图上的细节能"跟着"模型表面移动。

常见的贴图类型：

| 贴图类型 | 存的是什么 | 用途 |
|---|---|---|
| BaseColor / Albedo | 表面固有色 | 物体本来是什么颜色 |
| Normal Map | 微观法线方向（编码在 RGB 通道里） | 让平面表现出细节凹凸 |
| Metallic / Roughness | 金属度和粗糙度（通常打包在同一张图） | 控制 PBR 材质的光泽感 |
| Occlusion | 环境遮蔽强度 | 让缝隙、角落偏暗 |
| Emissive | 自发光颜色 | 让某些区域"发光" |

## 它们是怎么一起流过管线的

现在可以把上面三份数据放进管线里，看它们各自在哪一步发挥作用。

### CPU：组织渲染任务

每帧开始前，CPU 要告诉 GPU "这一帧要画什么"。对于场景里每一个可见物体，引擎提交一个 Draw Call，大致包含：

- 顶点缓冲和索引缓冲的指针（从哪个 Mesh 取数据）
- 当前材质绑定的 Shader 和参数（用什么规则画）
- Transform 矩阵（这个物体在世界里的位置、旋转、缩放）

这三项内容会一起被送进图形 API（DX12、Vulkan、Metal），然后交给 GPU 流水线。

### Vertex Shader：变换坐标，传递插值数据

GPU 拿到 Mesh 的顶点数据后，对每个顶点执行一次 Vertex Shader。

这一步最核心的工作是坐标变换，也叫 MVP 矩阵变换：

```
Model Space → World Space → View Space → Clip Space → NDC → Screen Space
```

- **Model Space → World Space**：乘以 Model Matrix（就是 Transform 矩阵），把顶点从模型自身坐标系变换到世界坐标系
- **World Space → View Space**：乘以 View Matrix，把世界坐标变换到以摄像机为原点的坐标系
- **View Space → Clip Space**：乘以 Projection Matrix（透视或正交），把视锥体内的内容映射到一个标准立方体，同时产生透视效果
- **Clip Space → NDC**：硬件自动做透视除法（除以 w 分量），得到归一化设备坐标，范围 `[-1, 1]`
- **NDC → Screen Space**：硬件按视口设置映射到像素坐标

除了位置，Vertex Shader 还要把 UV、法线、切线等数据传递给后续阶段。这些数据会在光栅化时按三角形面积被插值到每个像素。

### 光栅化：三角形变成像素，插值 UV 和法线

Vertex Shader 输出了三角形的三个顶点在屏幕上的位置。光栅化器接着做两件事：

1. **覆盖测试**：判断屏幕上哪些像素落在这个三角形里面
2. **重心插值**：对于每个被覆盖的像素，按照该像素在三角形内的位置（用重心坐标计算），插值出该像素的 UV、法线、切线等值

这意味着：每个像素都得到了一组插值后的 UV 坐标——不是模型顶点上的那个值，而是按位置计算出来的"中间值"。这套 UV 坐标，就是接下来 Fragment Shader 去贴图里取样的地址。

### Fragment Shader：采样贴图，计算光照

每个像素（Fragment）运行一次 Fragment Shader。这一步是"出颜色"的地方，也是材质 Shader 程序真正运行的地方。

以一个典型的 PBR 材质为例，Fragment Shader 大致做这些事：

**1. 用 UV 采样贴图，取出材质参数**

```
albedo     = sample(BaseColorTexture, uv)          // 固有色
metallic   = sample(MetallicGlossMap, uv).r        // 金属度
roughness  = 1.0 - sample(MetallicGlossMap, uv).a  // 粗糙度（有时存在 alpha 通道）
normalTS   = decode(sample(NormalMap, uv))         // 从 RGB 解码出切空间法线
```

**2. 把法线从切空间变换到世界空间**

法线贴图存的是切空间（Tangent Space）的偏移——以顶点法线为 Z 轴的局部坐标系。Fragment Shader 用顶点传来的法线和切线向量，把它变换到世界空间，得到一个能参与光照计算的法线方向。

**3. 执行 PBR 光照计算**

拿到 `albedo`、`metallic`、`roughness`、世界空间法线、光源方向和摄像机方向之后，PBR BRDF（双向反射分布函数）计算这个像素应该反射多少光：

- **漫反射**：从 albedo 出发，用 Lambertian 或类似模型计算，金属度高的表面漫反射贡献少
- **高光反射**：取决于粗糙度（粗糙度高，高光宽而暗；粗糙度低，高光集中而亮）和菲涅尔效应（掠射角时反射更强）
- **环境光**：从 IBL（Image-Based Lighting）或 SH（球谐光照）取出环境贡献

最终混合出这个像素的 HDR 颜色值。

**这就是"材质"真正发挥作用的地方**——材质参数（albedo、metallic、roughness）和材质贴图（BaseColor、Normal、MetallicGloss）都在这一步被消费掉，产生出一个颜色。

### 深度测试和输出

Fragment Shader 输出颜色后，还要过深度测试：把这个像素的深度和深度缓冲里已有的值比较，更近的留下，更远的丢弃。

最终通过测试的像素写入颜色缓冲，一帧结束后显示到屏幕上。

## 一张图总结数据流

```
Mesh（顶点缓冲 + 索引缓冲）
  → CPU Draw Call 提交 → GPU
    → Vertex Shader
        输入：Position, UV, Normal, Tangent（per vertex）
        输出：Clip Space Position + 插值数据（UV, Normal, Tangent → per vertex）
    → 光栅化
        覆盖测试：哪些像素在三角形内
        重心插值：UV, Normal, Tangent → per pixel
    → Fragment Shader（材质 Shader 在这里运行）
        输入：插值后的 UV, Normal（per pixel）+ 材质参数（albedo, metallic, roughness）+ 贴图引用
        执行：sample(texture, uv) → 取出颜色/法线/参数
        执行：PBR 光照计算
        输出：像素颜色
    → 深度测试 + 写入颜色缓冲
```

## 几个容易误解的地方

**误解一："贴图决定模型形状"**

贴图只能影响光照计算，不影响几何形状（法线贴图产生视觉上的凹凸感，但实际上面还是平的）。要真正改变形状，要么改 Mesh 数据，要么用 Displacement Map 在 Vertex Shader 里偏移顶点位置。

**误解二："一个材质只能用一张贴图"**

一个材质可以引用任意数量的贴图，取决于 Shader 程序里绑定了多少个纹理采样器。一个完整的 PBR 材质通常至少有 4-5 张贴图。

**误解三："模型的颜色是存在顶点里的"**

大多数情况下，颜色来自 Fragment Shader 里对贴图的采样，不是顶点数据。顶点色是一个额外数据，有时候用来存遮蔽值或者区域权重，用途因 Shader 而异。

**误解四："UV 展开是给美术看的，程序不用管"**

UV 展开决定了贴图坐标如何分布在模型表面，直接影响贴图采样的精度和接缝。如果 UV 展开不合理（拉伸、重叠、接缝错误），Fragment Shader 取样的地址就会有问题，贴图在模型上的表现就会乱。这是美术和程序都需要理解的基础。

## 和下一篇的关系

这篇追踪的是"单个物体的一次 Draw Call"里数据怎么流的。但真实的一帧不是只有一个 Draw Call，通常有几百甚至几千个。

引擎怎么组织这几千个 Draw Call、决定画的顺序、哪些东西合并在一起、哪些东西先画——这是下一篇要讲的：引擎的渲染流程，和 GPU 管线之间的边界在哪里。
