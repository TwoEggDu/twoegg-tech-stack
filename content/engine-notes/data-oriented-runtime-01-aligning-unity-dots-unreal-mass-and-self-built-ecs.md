> 
> 不先把问题对齐，后面你看到的就只会是一堆术语差异：`Entity`、`Fragment`、`Processor`、`Baker`、`Trait`、`Subsystem`。这些名词当然有意义，但它们不是最稳定的那层。

在上一篇
[《数据导向运行时 00｜总论：为什么现代引擎都在做“数据导向孤岛”》]({{< relref "engine-notes/data-oriented-runtime-00-why-engines-build-data-oriented-islands.md" >}})
里，我先给了一个总判断：

> 现代引擎做数据导向，不是因为它们突然讨厌对象，而是因为高规模同构仿真会把系统逼向另一套成本模型。

如果这个判断成立，那接下来的第一步就不该是去背术语。

更值得先回答的是：

- 这三套东西到底在试图解决同一批什么问题
- 它们分别愿意改动引擎的哪一层
- 它们各自拒绝改动什么
- 如果自己手搓，应该继承哪部分结构判断，而不是照抄哪家的名词

这篇就是准备先把这几件事压清。

## 这篇要回答什么

这篇主要回答 5 个问题：

1. 为什么 `Unity DOTS`、`Unreal Mass` 和自研 `ECS` 可以放在一起讨论。
2. 它们真正应该对齐的，到底是哪些问题，而不是哪些名词。
3. Unity 和 Unreal 在同一问题上，分别愿意改到多深。
4. 自研这条线为什么不是“第三个现成框架”，而是一个对照实验。
5. 把问题对齐之后，后面几篇该怎样往下拆。

## 先给一句总判断

如果必须先把这篇压成一句话，我会这样写：

`Unity 更像试图把一大块运行时重建成 ECS 世界，Unreal 更像在 Actor 引擎内部划出一块高规模数据导向仿真区，而自研最适合把两者共同面对的问题压回最小结构约束。`

这句话比“一个是 ECS，一个是 Mass，一个是自己实现”更有用。

因为它直接告诉你：

- 它们不是同构产品
- 它们也不是互相替代关系
- 但它们确实在回答同一类运行时问题

## 不要先对齐术语，要先对齐问题

很多文章一上来就喜欢做这种表：

- `Entity` 对 `Entity`
- `Component` 对 `Fragment`
- `System` 对 `Processor`
- `Baker` 对 `Trait`

这种对照不能说完全没价值，但它太容易把人带偏。

因为真正稳定的不是这些名词，而是这些名词背后在回答的结构问题。

如果先把术语全部拿掉，只保留问题本身，我认为这三条线真正应该先对齐的是下面 7 件事：

1. `世界怎样承载`
   这套系统是想重写默认运行时世界，还是挂在默认世界旁边。

2. `高规模对象怎样存`
   同结构对象怎样聚类，查询为什么要尽量命中连续数据。

3. `处理逻辑怎样找到对象`
   是按对象引用一层层走，还是先拿到一批满足条件的数据块再批处理。

4. `结构变化怎样提交`
   为什么加减组件、迁移布局、生成销毁实体，几乎都要走延迟提交。

5. `执行怎样调度`
   读写边界怎样声明，系统之间怎样排顺序，并行是不是可证明的。

6. `仿真层和表示层怎样切`
   数据导向仿真怎样接回渲染、动画、LOD、表现对象和编辑器工作流。

7. `构建期和运行期怎样分工`
   哪些对象和配置应该在离线阶段转成更便宜的运行时数据。

后面这组文章其实也会严格按这 7 个问题往下拆。

所以这篇的任务，不是把所有答案讲完。
而是先把三套系统放到同一张问题表里。

## 先把三套系统放到一张问题表里

如果先把最关键的差异压成一张最小对照表，我会这样记：

| 对照维度 | Unity DOTS | Unreal Mass | 自研 ECS |
| --- | --- | --- | --- |
| 它想解决什么 | 更系统地重建高规模数据导向运行时 | 在 Actor 世界内部开一块高规模仿真特区 | 证明哪些结构约束是这类系统第一版必需的 |
| 和默认对象世界的关系 | 倾向于把 authoring world 与 runtime world 切开 | 倾向于挂在 `UWorld / Actor` 世界旁边协同存在 | 你自己决定，但必须明确是否混合 |
| 对表示层的态度 | 更愿意把纯数据仿真拉远，再做桥接 | 更明确接受混合表示与多层 LOD | 第一版最好显式做桥，而不是混在一起 |
| 对构建期的态度 | 更强调 baking / conversion / runtime data | 更强调模板、traits、spawn、representation 等混合工作流 | 应该尽早做 authoring/runtime separation |
| 它最像什么 | 运行时重构路线 | 数据导向仿真特区 | 去掉引擎包袱后的结构对照组 |
| 它最怕什么 | 频繁结构变化、边界穿透、双世界心智成本 | 表示和仿真协同时的复杂边界、跨系统维护成本 | 一上来做太全，结果什么都没证明 |

这张表最关键的地方，不是它有多完整。

而是它先把一个常见误区拆掉了：

`Unity DOTS 和 Unreal Mass 不是“两个同类框架的不同 API 设计”；它们更像是两种不同工程立场下，对同一批运行时问题给出的不同答案。`

## 从 Unity 这边能直接看见什么

先只看当前能比较稳地落下来的事实。

### 事实一：Unity 并不是只加了一组高性能运行时 API

从 `Entities`、`DOTS`、`SubScene`、`Baking` 和 `World` 相关资料里，至少可以直接看见下面几件事：

- `Entities` 被明确放在 `DOTS` 语境里，而不是一组彼此独立的小包
- `SubScene` 和 `Baking` 说明 authoring 数据进入 runtime 数据有单独路径
- `World`、`EntityManager`、`systems` 说明 DOTS 有自己的世界组织和执行承载
- `Burst`、`Jobs`、`Collections` 又把编译、执行和内存容器一起拉进了这条链

这些事实合在一起，已经足够说明一件事：

`Unity 想做的不是“让 GameObject 也能跑快一点”，而是给高规模仿真准备一套更完整的运行时组织方式。`

### 事实二：Unity 接受 authoring world 和 runtime world 明确分离

这一点非常关键。

一旦你承认：

- 编辑器里给内容团队操作的对象世界
- 运行时给大规模批处理服务的数据世界

可以不是同一个东西，很多后续结构就都顺了：

- 为什么需要 baking
- 为什么需要 SubScene
- 为什么 runtime world 会越来越纯
- 为什么 DOTS 学起来不像“MonoBehaviour 的升级版”

所以从 Unity 这边最先看见的，不是 ECS 术语，而是这条更硬的边界：

`作者数据可以继续按传统对象方式工作，但运行时高规模仿真不一定还沿用同一套表示。`

### 工程判断：Unity 的核心立场更接近“运行时重建”

基于上面这些事实，我认为对 Unity 最稳的判断不是“它在 Unity 里加了 ECS”。

更稳的说法是：

`它试图把高规模运行时从对象世界里剥出来，再按数据布局、查询、调度和构建期转换重新组织一次。`

所以 Unity 这条线最值得盯住的，从来不是单个 API。

而是下面这组问题：

- runtime world 想离 authoring world 多远
- data layout 想做得多纯
- system scheduling 想做得多显式
- 哪些成本愿意前移到构建期

这也是为什么后面 `02`、`03`、`04`、`05` 这些篇目，在 Unity 这边会天然连成一条主链。

## 从 Unreal 这边能直接看见什么

再看 Unreal `Mass`。

### 事实一：Mass 从一开始就不是“把 Actor 换掉”

从 `MassEntity`、`MassGameplay`、`FMassEntityManager`、`FMassEntityQuery`、`UMassProcessor` 这条文档链上，至少能直接看见：

- Unreal 把 `MassEntity` 定义成面向 data-oriented calculations 的 gameplay framework
- `Mass` 是挂在 `UWorld` 相关体系里的，而不是完全独立的世界
- `EntityManager`、`Query`、`Processor`、`Command Buffer` 说明它有一套专门的批处理仿真组织方式
- `MassGameplay` 又继续把 representation、LOD、replication、signals、StateTree 这些外围能力拉进来

这些事实合在一起，更接近的图景不是“Actor 世界即将退场”。

而是：

`Actor 世界继续存在，但当问题变成大规模仿真时，Unreal 在内部开了一块更适合数据导向处理的区域。`

### 事实二：Unreal 更早接受混合架构是长期形态

这点和 Unity 气质差别很大。

从 Unreal 对 `Mass Representation`、LOD、spawn 和 gameplay framework 的整体组织来看，它显然没有试图对外宣布一件事：

`以后大家都别用 Actor 了。`

它给人的感觉更像：

- 复杂内容、交互、传统 gameplay 依然有自己的对象世界
- 高规模仿真问题则在 `Mass` 这块数据导向区域里处理
- 两者之间靠 representation、spawn、signals、replication 等机制接起来

这件事带来的后果也非常直接：

- Unreal 的数据导向路线更现实，也更混合
- 它更少给你“世界重建”的感觉
- 但它也更早承认跨边界协作会是长期成本

### 工程判断：Unreal 的核心立场更接近“仿真特区”

所以对 Unreal Mass 更稳的判断，不是“Unreal 也有自己的 ECS”。

更稳的说法是：

`它是在默认 Actor 世界内部，给高规模对象和批处理计算单独划出一块仿真特区。`

这个判断很重要，因为它解释了为什么 Unreal 那边很多看起来“没有 Unity 那么纯”的设计，反而是合理的。

因为它一开始追求的就不是把引擎运行时全部翻成纯 ECS 世界。

它真正追求的是：

- 先把规模问题放进更便宜的数据组织里
- 再把这块区域和已有 gameplay / representation / replication 体系接起来

所以你如果拿“为什么它不彻底替代 Actor”去质疑 Mass，其实等于在拿错标尺。

## 自研这条线为什么一定要放进来

很多人会觉得，把“自研 ECS”放进这组三角对照里，有点不像同一量级的话题。

我反而觉得它必须放进来。

原因不是因为自研可以和 Unity、Unreal 一一对应。

恰恰相反，正因为它不能一一对应，所以它最适合做对照实验。

### 自研不是第三个现成框架，而是一个去包袱的实验场

一旦你自己做，就没有现成历史包袱替你做决定了。

你必须自己回答下面这些问题：

- 你的世界容器是什么
- 你的 EntityId 怎么保证稳定
- 你的组件集合怎样映射到聚类存储
- 你的查询怎么缓存
- 你的结构变化什么时候提交
- 你的系统怎样声明读写边界
- 你的表示层什么时候桥接回对象世界

也正因为没有现成引擎替你兜底，你会更容易发现：

`真正稳定的不是 Unity 的术语，也不是 Unreal 的术语，而是这类系统躲不开的结构约束。`

### 工程判断：自研的价值在于看见不变量，不在于复刻全家桶

所以这条线的价值，从来不是“第三家实现长什么样”。

而是让你在没有引擎历史包袱时，更清楚地看见：

- 哪些是第一版必须有的
- 哪些是因为大型引擎生态才会长出来的
- 哪些高级能力应该后做

如果把这个判断压得再硬一点，我会这样说：

`自研 ECS 的任务，不是复制 DOTS 或 Mass；而是证明当你只保留最小运行时问题时，系统会自然收敛到哪些核心结构。`

这句话后面会直接落到 `07｜如果自己手搓，第一版最小系统应该做到哪里`。

## 真正该对齐的，不是 API，而是三种工程立场

如果把前面的内容收一下，我认为这三条线真正应该先对齐的，是三种不同的工程立场。

### 第一种立场：愿不愿意重建运行时

这里 Unity 更激进。

它更愿意承认：

- 传统对象世界很好用
- 但它不适合承载所有高规模仿真
- 所以需要给 runtime world 单独建新规则

Unreal 则更克制。

它更像是在说：

- 传统对象世界仍然是主世界
- 但规模问题需要单独开一条更便宜的数据导向通道

### 第二种立场：愿不愿意长期接受混合架构

这里 Unreal 更坦然。

它更明确接受：

- 仿真层和表示层长期并存
- Actor 和 Mass 长期协作
- LOD、representation、replication 这些边界就是成本中心

Unity 当然也不是纯世界，现实里依然常常是混合架构。

但它的表达方式和工具链组织，更容易让人感受到一种“尽量把运行时拉纯”的方向性。

### 第三种立场：第一版到底应该证明什么

这里自研最有价值。

因为当你自己做第一版时，你最该证明的不是：

- 能不能做出完整编辑器工作流
- 能不能做出自动代码生成
- 能不能做出网络复制、热更新、动画桥接全家桶

你真正该证明的是：

- 聚类存储能不能站住
- Query 能不能稳定命中
- Structural Change 能不能延迟提交
- System 调度能不能显式声明依赖
- 表示桥能不能把对象世界和仿真世界干净地接起来

这五件事如果没站住，后面所有“高级能力”大多都会变成堆复杂度。

## 这篇对后面几篇的作用是什么

把问题空间先对齐，有一个很直接的好处：

后面几篇就不会再乱。

### `02` 为什么可以只讲存储，不急着讲调度

因为这篇已经先说明：

`三套系统先对齐的是问题，不是实现细节。`

所以到 `02｜Archetype、Chunk、Fragment：性能到底建在什么地方` 时，我们就可以专心盯住：

- 为什么同结构对象要聚类
- 为什么 chunked storage 会反复出现
- 为什么 query 会缓存 archetype 集合

而不用再回头解释“为什么 Unity 和 Unreal 可以放在一起讲”。

### `03` 为什么会自然落到 structural change

因为只要问题空间对齐了，下一件一定会浮出来的事情就是：

`如果对象不是按传统对象引用散着放，而是按结构聚类放，那结构变化为什么会贵。`

这就会自然把文章带到：

- Unity 的 structural change / sync point
- Unreal 的 command buffer / deferred commands
- 自研第一版为什么必须做延迟结构变化

### `07` 为什么不是“从零实现教程”

因为这篇已经先把自研放回“结构对照实验”的角色了。

所以到 `07`，重点就不会变成：

- 写多少代码
- API 长得像不像某个框架

而是：

- 哪些结构一定得先做
- 哪些事情第一版不该做
- 最小系统怎样证明它已经够用了

## 常见误解

### 误解一：把术语对译清楚，问题就算对齐了

不是。

术语最多只能告诉你“这里看起来像对应”。
真正决定系统站位的，还是它在世界承载、构建期转换、结构变化、调度和表示桥接上分别怎么表态。

### 误解二：Unity 和 Unreal 都在做数据导向，所以它们只是实现风格不同

也不是。

它们不只是风格不同。
更准确地说，它们对“愿意改多大范围”这件事，立场就不同。

一个更像重建运行时。
一个更像划出仿真特区。

### 误解三：把自研放进来，会把文章拉成玩具框架实现

前提是你把自研写成代码教程。

如果把自研当成“看不变量的实验场”，它反而会把 Unity 和 Unreal 身上的历史包袱剥掉，让很多工程判断更稳定。

## 我的结论

如果一定要把这篇压成一句话，我会这样写：

> Unity DOTS、Unreal Mass 与自研 ECS 真正应该先对齐的，不是 `Entity / Fragment / Processor` 这些名词，而是它们怎样回答世界承载、聚类存储、结构变化、执行调度、表示桥接和构建期转换这几类更稳定的运行时问题。

把这张问题地图先立住之后，后面再去看 `Archetype`、`Chunk`、`Query`、`Command Buffer`、`Baking`、`Representation`，很多东西才不会散。
