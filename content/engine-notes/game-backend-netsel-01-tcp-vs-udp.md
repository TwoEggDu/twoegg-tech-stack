## 问题空间：为什么"能用"的不等于"该用"的

一个做了三年 Web 后端的程序员转做游戏服务端，第一个本能反应是：用 HTTP。理由很充分——HTTP 成熟、工具链完整、调试方便、框架遍地都是。第一个版本也确实能跑。

但当游戏上线，玩家开始抱怨"操作延迟"、"打不到人"、"明明按了跳但角色没跳"，排查半天发现：服务端帧率 60Hz，每帧 16ms，但 HTTP 请求平均 80ms 往返，最坏情况 300ms。游戏逻辑根本跑不起来。

这不是服务器性能不够，也不是网络带宽不足。是协议选型错了。

Web 流量和游戏实时流量是两种完全不同的负载模型：

- **Web 流量**：请求-响应，每次交互相对独立，数据完整性至关重要，延迟可以接受到几百毫秒
- **游戏实时流量**：持续高频小包，状态更新比历史数据重要，16ms 的延迟目标，丢一两个包无所谓但不能卡住

这两种模型对传输层的要求，几乎是对立的。

---

## 抽象模型：TCP 的可靠性是怎么工作的

TCP（Transmission Control Protocol）的核心承诺是：**有序、可靠、不重复地交付所有数据**。为了兑现这个承诺，它引入了一系列机制。

### 三次握手建立连接

```
客户端 → SYN → 服务端
客户端 ← SYN-ACK ← 服务端
客户端 → ACK → 服务端
连接建立，此时已经过了 1.5 RTT
```

1.5 个 RTT（Round-Trip Time）之后连接才能开始发送数据。国内跨运营商 RTT 50-100ms 很正常，意味着每次重连需要等待 75-150ms，什么还没做就先花了两帧。

### ACK 确认和重传机制

每个 TCP 数据包都有序列号。接收方收到数据后发送 ACK 确认。发送方如果在超时时间内没有收到 ACK，会触发**重传**。

重传的代价不只是时间。TCP 把网络丢包视为"拥塞信号"，触发拥塞控制算法，主动降低发送速率。这个设计对文件传输是合理的——慢下来是对的。但对游戏来说是灾难性的——你不想因为一个包丢了，整个连接的吞吐量都被踩刹车。

### 有序交付和队头阻塞

这是 TCP 在游戏场景最致命的问题。TCP 保证数据**按顺序**交付给应用层。如果包 3 丢了，即使包 4、5、6 都已经到达，它们也会被内核缓冲区扣住，直到包 3 重传成功后，才一起交给应用层。

这叫 **Head-of-Line Blocking（队头阻塞）**。

对于游戏来说，这意味着什么？假设你在做一个 FPS，每帧发送一次位置更新。玩家 A 的第 100 帧位置包丢了。按照 TCP 的逻辑，第 101 帧、102 帧的位置更新都被压着，等第 100 帧重传回来——但这时候已经过了 50ms，玩家 A 早就跑到别的地方去了，第 100 帧的位置根本没有价值，还不如直接用第 102 帧的最新状态。

**TCP 为了保证"没有一个包被丢下"，强行让你等待一个你根本不需要的旧数据。**

## 具体实现：UDP 的"不可靠"到底意味着什么

UDP（User Datagram Protocol）的设计几乎是 TCP 的对立面：发出去就算，不 ACK，不重传，不保证顺序，不保证到达。

```c
// UDP 发送，没有握手，没有状态，直接发
sendto(sock, data, len, 0, &addr, addrlen);
```

这看起来很不可靠，但对游戏来说，这正是想要的语义：

**老数据不如没有数据。**

FPS 游戏每帧发送位置更新。如果第 100 帧丢了，应用层自己决定：要不要重传？答案是不要——因为 16ms 后第 101 帧就来了，第 100 帧的位置已经过时。与其等重传，不如用最新数据做外插值。

UDP 把这个决策权交给了应用层。你可以：
- 对位置更新：完全不可靠，丢了就丢了
- 对伤害事件：加一层应用层 ACK，确保送达
- 对聊天消息：甚至可以跑在完全可靠的模式下

这种**按消息类型配置可靠性**的能力，是 TCP 根本给不了的。

### UDP 的实际限制

UDP 本身不负责分片。以太网 MTU 通常是 1500 字节，IP 头 20 字节，UDP 头 8 字节，留给应用数据的只有 1472 字节。如果你发送的包超过这个大小，需要在应用层自己做分片和重组。游戏数据包大多在 100-500 字节，这通常不是问题。

UDP 也没有连接状态。你需要自己管理"哪些地址是活跃的客户端"，需要自己处理会话建立和销毁。

## 工程边界：不同游戏类型的协议选择矩阵

不是所有游戏都需要 UDP。协议的选择应该匹配游戏的流量特征：

| 游戏类型 | 更新频率 | 延迟容忍 | 推荐传输层 | 理由 |
|---------|---------|---------|-----------|------|
| 回合制 RPG | 极低，玩家操作触发 | 500ms-1s 可接受 | HTTP/TCP | 请求-响应模型完全匹配 |
| 卡牌游戏 | 低，轮次内操作 | 200-500ms | HTTP 或 WebSocket | 操作频率低，状态一致性优先 |
| MMORPG 普通移动 | 中，每秒 10-20 次 | 100-200ms | TCP 或可靠 UDP | 位置同步要求高但不极端 |
| MOBA | 高，每秒 30 次 | < 80ms | UDP + 应用层可靠性 | 技能和位置频繁，延迟直接影响体验 |
| FPS/TPS | 极高，每秒 30-60 次 | < 50ms | UDP 原生 | 每一帧都重要，队头阻塞不可接受 |
| 赛车/格斗 | 极高 | < 20ms | UDP 原生 | 帧同步，输入延迟直接感知 |

关键判断标准只有两个：
1. **更新是否高频**（> 每秒 20 次以上）
2. **旧数据是否有价值**（如果不，丢弃比等待更好）

只要两个答案都是"是/否"（高频、旧数据无价值），就该用 UDP。

### HTTP/TCP 在实时场景的具体瓶颈

**连接建立延迟**：HTTP/1.1 每次请求都可能需要重新建立 TCP 连接（三次握手），HTTP/2 虽然复用连接，但基于 TCP 的队头阻塞依然存在。

**拥塞控制不适合游戏流量**：TCP 的拥塞控制（CUBIC、BBR 等）是为最大化吞吐量设计的，不是为最小化延迟。游戏流量是小包、高频、低带宽，拥塞控制算法的各种窗口和慢启动机制对这种流量适配性很差。

**Nagle 算法**：TCP 默认开启 Nagle 算法，会把小包攒到一起发送以提高效率。这对游戏是毒药——你的 50 字节位置更新可能被等待 200ms 才真正发出去。虽然可以通过 `TCP_NODELAY` 关闭，但这又会增加带宽消耗。

**HTTP 的文本开销**：HTTP/1.1 的头部动辄几百字节，对于一个 50 字节的位置更新包，头部开销是数据本身的数倍。即使用二进制的 HTTP/2，帧头依然有 9 字节，对高频小包还是有意义的比例开销。

## 为什么光靠 UDP 还不够

用了 UDP 不代表解决了所有问题，只是获得了解决问题的空间。

原始 UDP 上你需要自己实现：

- **序列号**：区分包的先后顺序，丢弃过期包
- **应用层 ACK**：对需要可靠送达的消息（开枪命中、购买道具）实现确认和重传
- **去重**：网络重传可能导致同一个包到达两次，需要 dedup
- **会话管理**：记录哪些 IP:Port 是合法客户端，维护会话状态
- **MTU 探测和分片**：如果消息可能超过 1472 字节，需要应用层分片
- **连接保活**：定期发心跳，检测断线

这些不是小工作。在生产项目中，直接用原始 UDP socket 写游戏网络层的团队很少。通常的做法是在 UDP 之上使用一个可靠 UDP 库，比如 KCP、ENet 或 QUIC，它们封装了上述机制，提供更友好的 API。

---

## 最短结论

TCP 的可靠性是为"不能丢数据"的场景设计的，它的队头阻塞和拥塞控制在高频实时游戏流量下是硬性代价，不是可以调优掉的问题。

UDP 是实时游戏的正确起点，不是因为它"更快"，而是因为它把数据可靠性的决策权还给了应用层，让游戏逻辑自己决定"这个包值不值得重传"。

回合制游戏用 HTTP 是完全合理的工程决策。MOBA 和 FPS 必须走 UDP。中间有一片灰色地带，用延迟要求和更新频率做判断，不要用"大厂都这样做"做判断。
