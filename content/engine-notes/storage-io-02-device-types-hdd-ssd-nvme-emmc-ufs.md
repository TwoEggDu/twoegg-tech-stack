> 如果只先记一句话，我建议记这个：游戏加载最怕的，通常不是“峰值带宽不够高”，而是“单次请求固定成本太大、随机访问太碎、而程序自己又给不出稳定的并发和局部性”。

很多团队一谈到加载，就会先问：

`我们是 HDD、SSD 还是 UFS？`

这个问题当然重要。  
但如果后面只跟一句“谁更快”，判断还是会很容易歪掉。

因为不同存储设备的区别，不只是宣传页上那个顺序读写带宽数字，而是：

- 单次随机访问到底要付多少固定成本
- 顺序访问时能不能稳定地把带宽吃起来
- 多个 I/O 同时在飞时，控制器和协议能不能并行处理
- 写入时会不会被擦写、垃圾回收、空闲空间和温度影响

如果这几件事不分开，最后就会出现一种常见误判：

- 看见 NVMe，就以为小文件访问已经不成问题
- 看见 UFS 4.0，就以为手机上 streaming 可以随便做
- 看见 SSD 顺序读几 GB/s，就以为项目里的加载卡顿不会再是 I/O

这些判断都不稳。

这一篇会故意把视角收在“设备本身的延迟模型和读写特性”上。  
像路径解析、页缓存、异步 I/O、热机和移动端系统争用这些账，我不在这里展开，后面分别放到文件系统、OS I/O 和移动端那几篇里讲。

## 先把几个最容易混的名字分开

这几个词经常被放在一张表里，但它们并不处在同一层。

| 名词 | 它主要在说什么 | 最容易混淆的点 |
| --- | --- | --- |
| `HDD` | 机械硬盘这种存储设备 | 它的慢，很多时候来自机械寻道和旋转等待 |
| `SSD` | 基于 NAND Flash 的固态存储设备大类 | `SSD` 不等于 `NVMe`，也不等于所有固态盘都一样 |
| `SATA` | 一种总线 / 接口与协议组合 | 很多 `SATA SSD` 的瓶颈不只是闪存，也包括协议开销 |
| `NVMe` | 面向非易失性存储的协议，通常跑在 PCIe 上 | 它是协议，不是“另一种介质” |
| `eMMC` | 移动端常见的嵌入式存储标准 | 常见于低端和旧设备，随机 I/O 往往更弱 |
| `UFS` | 更新一代的移动端嵌入式存储标准 | 比 eMMC 更擅长并行和低延迟，但不是“碎访问免疫” |

所以如果把它们压成一句更工程化的话，可以这样记：

- `HDD` 和 `SSD` 更像“介质与设备形态”
- `SATA / NVMe / eMMC / UFS` 更像“这套设备怎样和主机交互”

而游戏加载真正关心的是：

`这个设备在低队列深度、小粒度、冷热混合、随机访问时，到底会变成什么样。`

## 先别急着背带宽，先看五个真正有用的指标

如果你只盯顺序读带宽，就很容易把判断做偏。  
更稳的看法通常要同时看这五件事：

### 1. 单次读取延迟

也就是：

`从请求发出去，到第一批数据真正开始回来的时间。`

这件事对游戏特别重要，因为很多运行时加载不是大块离线拷盘，而是：

- 某个对象进视野
- 某个 bundle 被命中
- 某个 streaming chunk 需要补数据

这些场景更怕延迟尖峰，而不是只怕平均吞吐。

### 2. 顺序吞吐

也就是设备在连续读取一大片数据时，能有多快。  
这当然重要，尤其是：

- 安装包解包
- 大整包首载
- 长条线性 streaming

但它只能说明“路修得够不够宽”，不说明“每次上路前要不要反复停车办手续”。

### 3. 随机访问能力

随机访问能力决定的是：

`很多零散请求飞过来时，设备会不会立刻从“高吞吐”退化成“延迟主导”。`

这往往比顺序带宽更接近真实项目的痛点。

### 4. 队列深度下的表现

有些设备在一个请求一个请求来时并不惊艳，  
但一旦你能给它稳定的并发和连续队列，它就能突然吃满带宽。

反过来，有些设备即使理论吞吐还行，只要请求模式变碎、队列变浅，性能就会迅速塌下来。

### 5. 写入时的持续性

读路径和写路径不是一回事。

闪存设备的写入往往比读取更复杂，因为它通常会涉及：

- 页写入
- 擦除块管理
- 地址映射更新
- 垃圾回收
- 空闲空间不足时的放大效应

这件事会在：

- 安装包解压
- 资源热更新
- 大版本 patch
- 首次落盘缓存

这些阶段变得特别明显。

## HDD：为什么机械寻道会把随机访问直接打回原形

HDD 的问题不在“旧”，而在“它真的需要动起来”。

一次 HDD 随机读取，通常至少要经历三件事：

- 寻道：磁头先移动到目标磁道
- 旋转等待：等待目标扇区转到磁头下面
- 传输：真正把数据连续读出来

这意味着什么？

意味着对 HDD 来说，最贵的经常不是“传输那几 KB 数据本身”，而是前面那一大串准备动作。

所以它特别怕这种访问模式：

```text
读 A 的 8 KB
跳到很远的 B 读 8 KB
再跳到 C 读 16 KB
再回到别的区域读 4 KB
```

因为这不是在“稳定地读”，而是在不停付：

- 毫秒级寻道
- 旋转等待
- 请求切换

### HDD 什么时候反而没那么差

HDD 在下面这些模式里并不一定糟糕：

- 大块顺序读取
- 日志型顺序写入
- 已经高度局部化的扫描

原因很简单：  
一旦磁头已经在正确区域附近，后续连续读就相对便宜得多。

### HDD 的缓存到底在救什么

很多机械硬盘上会带一块缓存。  
它确实有用，但它解决的是“局部连续访问”和“短暂写入缓冲”这类问题，不是把机械延迟变没。

对读路径来说，它主要帮你做的是：

- 预读邻近扇区
- 缓一小段最近刚读过的数据

对写路径来说，它主要帮你做的是：

- 短时间合并小写入
- 平滑一部分突发写入

但如果你的访问模式是大跨度随机跳转，这块缓存救不了根问题。  
因为最贵的那部分账，还是：

`磁头得重新找地方。`

所以“磁盘有缓存”从来不等于“HDD 适合海量碎文件”。

## SATA SSD：没有机械寻道了，但随机访问依然不是零成本

很多人一听到 SSD，就会自动脑补成：

`没有机械动作了，那随机读应该几乎无代价。`

这也不准确。

SSD 确实没有 HDD 的磁头和转盘，  
但一次读取仍然不是“直接魔法取数”，它仍然会涉及：

- 主机发命令
- 控制器接收并解析请求
- 地址翻译
- 找到对应的 Flash 页
- 把数据搬回主机内存

所以 SSD 相比 HDD 的优势主要是：

- 没有机械寻道和旋转等待
- 单次随机访问固定成本低很多
- 小读的延迟通常不会像 HDD 那样爆炸

但它不代表：

- 小读完全免费
- 随机访问和顺序访问没有区别
- 只要换 SSD，加载系统就可以不管局部性

### 为什么 SATA SSD 仍然有自己的“天花板”

很多消费级 SATA SSD 用的是 `AHCI + SATA` 这套组合。

这套东西的好处是成熟、普及。  
但它的设计年代更早，协议层和队列模型并不是专门为现代高并发固态存储量身做的。

所以对 SATA SSD 来说，你常见到的是：

- 顺序读带宽已经很高
- 低队列深度随机读比 HDD 强很多
- 但再往上冲时，会更早碰到协议和队列的边界

换句话说：

`SATA SSD 解决了机械硬盘最痛的那部分问题，但还没有把“高并发低延迟固态存储”这件事做到极致。`

## NVMe SSD：协议、队列和并行度一起升级

`NVMe` 真正重要的地方，不是“换了个名字”，而是它把协议模型也一起换了。

和更老的接口相比，NVMe 更强调的是：

- 更低的协议开销
- 更适合固态介质的命令模型
- 更深、更并行的队列
- 更容易把设备内部并行度吃出来

这也是为什么 NVMe 特别擅长：

- 大块高速顺序读
- 多个 I/O 同时在飞的场景
- 需要更低尾延迟的工作负载

### 但为什么项目里常常感觉不到“宣传值那么夸张”

因为很多游戏运行时并没有把设备喂到 NVMe 最擅长的模式里。

真实项目里，程序经常做的是：

- 低队列深度
- 小粒度请求
- 阻塞式逐个读
- 中间还夹着解析、解压和主线程同步

这时你虽然用的是 NVMe，  
但设备看到的并不是“理想的高并发连续流”，而是一串被应用层切碎的零散事务。

于是最终体验就会变成：

- 理论峰值很好看
- 但项目里实际提升没有想象中线性
- 小文件 storm 依旧可能把你打回“延迟主导”

### DRAM 缓存、控制器并行度对 NVMe 为什么重要

很多 NVMe 盘真正强的地方，不只是 PCIe 总线快，  
还在于控制器、映射表管理和内部并行度更强。

这会直接影响：

- 地址翻译成本
- 多请求并发时的调度效率
- 随机读下的尾延迟

所以你不能只把 NVMe 理解成“更快的线缆”。  
它是协议、控制器、队列模型和介质行为一起升级的结果。

## eMMC：为什么它在手机上常常先暴露“碎访问问题”

到了移动端，很多人会先把 HDD 从脑子里删掉，  
然后下意识觉得：

`手机反正都是闪存，那 I/O 差异不会像 PC 那么大。`

这也不对。

eMMC 的关键特点通常是：

- 控制器和并行能力相对更简单
- 随机 I/O 表现更容易下滑
- 小事务和低队列深度下更容易先暴露尾延迟问题

它的问题不只是“带宽低一些”，而是：

`当访问变碎、请求变小、冷热状态又不稳定时，随机读取和尾延迟更容易先出问题。`

## UFS：为什么它比 eMMC 更适合现代移动加载，但仍然不代表可以乱读

UFS 相比 eMMC，通常会在下面这些点上更强：

- 命令队列能力更好
- 并行度更高
- 顺序和随机表现都更强
- 尾延迟更容易压住

所以它确实更适合现代移动游戏里这些动作：

- 更激进的流式加载
- 更频繁的资源补读
- 更高分辨率素材的进场

但这里有个特别常见的误解：

`只要是 UFS，碎访问问题就不再重要。`

这不成立。

因为项目里最终看到的不是“设备理论能力”，而是：

- 应用层一次发多大请求
- 请求之间有没有局部性
- 文件系统和页缓存能不能帮上忙
- CPU 解压和依赖解析会不会把时间又吃回去

UFS 只是把“底层设备太慢”这层问题往后推了一些，  
它并没有抹掉碎访问的固定成本。

## 一张更适合游戏加载的比较表

下面这张表，不是给采购看的，而是给做加载判断的人看的。

| 类型 | 最擅长 | 最怕什么 | 对游戏最重要的启示 |
| --- | --- | --- | --- |
| `HDD` | 大块顺序读写 | 大跨度随机访问、海量小文件 | 任何碎访问都会被机械延迟放大 |
| `SATA SSD` | 稳定顺序读、比 HDD 好得多的随机读 | 协议开销、低效的小事务 storm | 已经能显著改善加载，但仍然别把请求打得太碎 |
| `NVMe SSD` | 高并发、高吞吐、低尾延迟 | 应用层只给低 QD 小请求时吃不满能力 | 别迷信宣传带宽，程序自己的访问模型更关键 |
| `eMMC` | 成本和容量平衡 | 随机 I/O、尾延迟、持续写入压力 | 低端机更需要合批、预取和减少热路径碎访问 |
| `UFS` | 移动端更强的顺序与随机表现 | 热机、系统争用、碎访问固定成本 | 可以更大胆做 streaming，但仍要守局部性和批量化 |

## 为什么“读写特性”里写入也不能不看

虽然很多运行时问题最后表现成“读取慢”，  
但项目里不少阶段其实是读写混合的：

- 下载后解压
- patch 覆盖
- shader cache / 资源缓存落盘
- 安装或首次启动时的展开

这时候闪存设备会比纯读更容易出现波动，因为写入经常要面对：

- 擦除块回收
- 垃圾回收
- 空闲空间不足
- 长时间持续写后的降速

所以“安装很慢、更新很慢、落盘很慢”不能直接按“读路径直觉”去解释。

## 最后把判断落到项目上

如果你在做的是：

- 大整包首载
- Streaming World
- 运行时 AssetBundle / Addressables 补读
- 热更新和资源缓存

那你真正该问的通常不是：

`用户是什么硬盘？`

而是下面这些更具体的问题：

1. 请求是不是被我切得太碎了。
2. 我是在吃顺序吞吐，还是在被单次延迟反复抽血。
3. 我有没有给设备稳定的局部性和合适的并发。
4. 我是不是只看了顺序读带宽，却没看低 QD 随机读和尾延迟。
5. 我是不是把读路径、写路径和解压路径混成了一笔账。

## 这一篇最该带走的结论

`HDD、SSD、NVMe、eMMC、UFS 的差别，不只是“谁更快”，而是“它们分别怎样对待随机访问、顺序吞吐、并发队列和持续写入”。`

对游戏来说，最危险的从来不是设备名字本身，而是：

`程序把访问模式做成了设备最不擅长的那一种。`
