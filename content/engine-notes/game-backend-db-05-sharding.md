## 这篇文章在解决什么问题

"单库撑不住了，要分库分表吗？"这个问题在游戏后端项目里出现的时机，往往是最不该讨论架构的时候——公测前一周，或者游戏已经在跑、数据库开始报警。

问题之所以棘手，不只是"怎么拆"，而是：

1. **判断要不要拆**：很多"撑不住"的感觉是假象，问题可能根本不在数据库容量
2. **拆了带来什么新问题**：跨库查询、跨库事务、全局唯一 ID、数据迁移，每一个都是工程成本
3. **拆错了比没拆还麻烦**：选错分片键，数据倾斜，某个分片成为热点，反而比单库还差

这篇文章不提供"最佳实践"，提供**判断框架**——什么情况下该拆，拆什么，怎么拆，以及哪些情况下应该忍住不拆。

---

## 先把问题分清楚：容量问题 vs 性能问题

"数据库撑不住"通常是两类问题之一，但这两类问题的解法完全不同：

**性能问题**：查询慢、写入延迟高、CPU 或 I/O 打满。这类问题**首先应该通过索引优化、查询重写、读写分离、缓存层来解决**，而不是分库分表。分库分表对性能问题的帮助是有限的——你把数据拆到 4 个库，写入压力分散了，但如果根本问题是查询设计不合理，拆完还是一样慢。

**容量问题**：单表行数太多（超过 5000 万行），单库数据量太大（超过 TB 级），B-Tree 索引深度增加，写入时的索引维护成本线性上升。这才是分库分表真正要解决的问题。

判断方法：先看 EXPLAIN，先看慢查询日志，先看数据库的监控（CPU、I/O、连接数）。如果问题是索引或查询设计，优先从那里下手。如果优化过后，问题的根源确实是数据规模，再考虑分表。

## 垂直拆分：按业务模块拆库

垂直拆分是把原来在一个数据库里的多个业务模块，拆到不同的数据库实例上。

例如：

```
单体数据库（所有表）
    ↓ 垂直拆分
账号库（player_account, player_auth）
游戏库（player_profile, player_currency, item_instance）
社交库（guild, friend_list, chat_log）
日志库（login_log, payment_log, battle_log）
```

垂直拆分的优点：
- 不同业务模块的数据库可以独立扩容
- 日志库（高写低读）和账号库（低写高读）可以用不同的硬件配置
- 不同模块的故障隔离（日志库宕机不影响游戏核心功能）

垂直拆分的代价：
- **跨库 JOIN 消失了**：原来 `SELECT ... JOIN guild ...` 的查询，现在要在应用层做数据合并
- **跨库事务变复杂**：如果一个业务操作跨越两个库（扣账号库的钻石 + 写游戏库的道具），需要分布式事务或最终一致性方案

垂直拆分是微服务化的前置步骤。在游戏后端里，当不同业务模块的负载差异很大时（日志写入量是游戏核心数据的 100 倍），垂直拆分有实际价值。

**什么时候做垂直拆分**：业务模块边界清晰、不同模块的负载特征差异明显时。不要为了"将来的扩展性"提前拆——拆分意味着跨模块操作变复杂，如果业务边界还没稳定，过早垂直拆分会让代码里到处是跨服务数据合并逻辑。

## 水平拆分：按数据规模拆表

水平拆分是把一张大表按某个规则分散到多张结构相同的表（或多个库）里。这才是通常意义上的"分表"。

### 分片键的选择

分片键（Sharding Key）决定数据如何分布。选错分片键，是水平拆分最常见的失败原因。

游戏后端的主要分片键选择：

**user_id 取模**
```
物品表 item_instance 按 user_id % 4 分到 4 张表：
item_instance_0：user_id % 4 = 0 的玩家数据
item_instance_1：user_id % 4 = 1 的玩家数据
item_instance_2：user_id % 4 = 2 的玩家数据
item_instance_3：user_id % 4 = 3 的玩家数据
```

优点：数据分布均匀（前提是 user_id 均匀分布），查询时只需要知道 user_id 就能路由到正确的分片。
缺点：扩容困难，从 4 张表扩到 8 张表时，需要重新计算所有数据的分片归属，数据迁移量是全量数据。

**user_id range 分段**
```
item_instance_1：user_id 1 - 10,000,000
item_instance_2：user_id 10,000,001 - 20,000,000
item_instance_3：user_id 20,000,001 - ...
```

优点：扩容简单，新增用户直接写新分片，不需要迁移历史数据。
缺点：数据分布不均匀（新服务器的分片是活跃用户聚集地，老分片是不活跃的老用户），可能出现分片冷热不均。

**一致性哈希（Consistent Hashing）**
扩容时只需要迁移部分数据，而不是全量。适合需要频繁扩容的场景，但实现复杂度高。

**游戏后端的常见选择**：中小型游戏用 user_id 取模，确定分片数量（通常是 2 的幂次：4、8、16、32），一次性设定好，不频繁扩容；大型游戏用 range 或一致性哈希，换来更好的扩容能力。

### 分片数量的预估

不要根据当前数据量决定分片数，要根据**未来 3 年的预期数据量**来决定。

规则：**单分片的 item_instance 行数控制在 5000 万以内，单库数据量控制在 500GB 以内**。

假设游戏预期峰值注册用户 2000 万，每个用户平均持有 500 件物品，item_instance 总行数约为 100 亿。单分片 5000 万行上限，需要 200 个分片。但 200 个物理表或库的运维成本极高。

实际上，大多数游戏不需要这个极端规模，而且活跃用户远少于注册用户。可以根据更保守的估算（50 万日活、平均 200 件物品）来设计初始分片数量，同时预留扩容能力。

## 分库分表后的新问题

分库分表不是免费的。每一个被解决的问题后面，都跟着一个新问题。

### 跨分片查询

```sql
-- 这个查询在单表里很简单，分表后变成问题
SELECT * FROM item_instance
WHERE item_template_id = 2001  -- 按 user_id 分片的表，无法直接路由
ORDER BY acquire_time DESC
LIMIT 100;
```

按 `user_id` 分片的 `item_instance` 表，如果查询条件里没有 `user_id`，就必须扫描所有分片，合并结果，再排序。这个操作叫做**全分片散射查询（Scatter Query）**，在高并发下对数据库集群的压力巨大。

解法：建立**辅助索引表**。把某些查询维度（如 `item_template_id`）单独维护一张映射表（存在一个不分片的 Redis 或单独的索引库里），用来快速定位目标分片。

### 跨分片事务

玩家 A（在分片 0）向玩家 B（在分片 3）交易道具，这个操作涉及两个分片，无法用单机事务保证原子性。

解法：
- **分布式事务（2PC）**：强一致性，但性能差，协调者故障会导致事务悬挂
- **Saga 模式**：最终一致性，通过补偿操作（Compensating Transaction）处理失败，复杂度高
- **设计上避免跨分片事务**：最优解。玩家之间的交易可以通过"中间账户"模式实现（A 先把道具转到中间账户，B 再从中间账户取走），把跨分片操作转化为两个独立的单分片操作。

### 全局唯一 ID

分片后，MySQL 的 `AUTO_INCREMENT` 在不同分片上会产生重复 ID。需要全局唯一 ID 方案：

- **雪花算法（Snowflake）**：64 位 ID，包含时间戳、机器 ID、序列号。Twitter 开源，广泛使用
- **数据库号段模式**：从一个专门的 ID 分发服务批量申请 ID 段（如每次申请 1000 个），减少对 ID 服务的依赖
- **UUID**：简单，但 128 位、无序，作为 B-Tree 主键性能差（随机插入导致页分裂）

游戏后端推荐雪花算法或号段模式，不推荐 UUID 作为数据库主键。

## 中间件：ShardingSphere vs Vitess

分库分表中间件的作用是：让应用层感知不到分片，用统一的 SQL 接口操作分片后的数据库集群。

### ShardingSphere（Apache）

Java 生态，提供 JDBC 驱动模式（ShardingSphere-JDBC，嵌入应用进程）和代理模式（ShardingSphere-Proxy，独立进程）。

JDBC 模式的优点：零额外网络跳转，延迟低；缺点：应用和分片规则耦合，需要修改代码依赖。

Proxy 模式的优点：对应用完全透明，支持任何语言；缺点：多一跳网络，需要运维额外的进程。

ShardingSphere 支持的功能完整（分库分表、读写分离、分布式事务、数据加密），在国内游戏公司的使用率很高。

### Vitess

MySQL 的分布式扩展层，YouTube 开源，现在是 CNCF 毕业项目。Kubernetes 原生友好。

Vitess 的核心是 VTGate（查询路由层）和 VTTablet（管理 MySQL 实例），提供连接池、查询重写、分片路由等能力。

Vitess 在大规模部署下比 ShardingSphere 更稳定，但配置复杂度高，需要较强的运维能力。适合有 SRE 团队支持的大型游戏公司。

**选择建议**：
- 中小型游戏、Java 技术栈：ShardingSphere（JDBC 模式）
- 大型游戏、有运维能力：Vitess
- 不确定：先用 ShardingSphere Proxy，语言无关，后续迁移代价低

## 什么时候不该拆

过早分库分表是游戏后端最常见的过度工程之一。以下情况**不应该分库分表**：

**单表行数不到 3000 万**：InnoDB B-Tree 在这个规模下索引效率仍然良好，配合读写分离（主库写、从库读）足以应对大多数负载。

**性能问题来自查询设计**：慢查询日志里显示的是全表扫描或低效 JOIN，这是索引和查询优化的问题，分表解决不了。

**业务还在快速迭代**：业务边界没稳定，分片键选错了代价极大（全量数据迁移）。宁可在设计上预留分表空间（`user_id` 用 BIGINT 而不是 INT，给未来的分片留余地），也不要现在就拆。

**团队没有运维能力**：分库分表的运维复杂度是单库的 N 倍。没有数据库 DBA 或有经验的后端工程师支持，分表后出问题的修复成本很高。

**游戏典型的分库时机**：
- 日活稳定在 50 万以上，且在增长
- 单表行数超过 5000 万（背包表、日志表）
- 主库 CPU 持续 > 70%，且已经做了读写分离和缓存优化
- 单次发版的 ALTER TABLE 需要超过 30 分钟（表太大）

## 工程边界

分库分表不该承担的职责：

**分库分表不是缓存的替代品**。如果热点数据（在线玩家状态、排行榜）在 Redis 里，数据库的压力本来就不大，不需要分片。先把缓存层建好，再评估是否需要分片。

**分库分表不解决数据模型问题**。把一个设计糟糕的大宽表分片，只是把问题分散了，没有解决。在分片前，应该先确认数据模型是否合理。

---

## 最短结论

分库分表是解决数据规模问题的手段，不是解决性能问题的银弹——在真正需要拆之前，读写分离、缓存、索引优化通常能多支撑一到两个数量级的规模。
