> 如果只用一句话概括这篇，我会这样说：现代渲染不是告别传统图形管线，而是在它的骨架上不断叠加新的阶段、缓存和跨帧重建逻辑。

到这里，系列已经把整套基础地形铺得差不多了：

- 引擎怎么组织一帧
- GPU 怎么执行图形任务
- API、OS、Driver 和 GPU 各站在哪一层
- 桌面与移动端的硬件现实为什么不同
- 跨平台引擎为什么要管理大量差异

但如果你回头看今天的大型游戏图形，又会发现另一个显眼事实：

`现代渲染越来越复杂了。`

从传统光栅化一路走过来，我们又叠上了：

- Compute Pass
- Deferred / Forward+
- TAA
- SSR / SSAO
- Ray Tracing
- DLSS / FSR / XeSS
- Frame Generation
- 各种缓存、历史帧和中间结果

如果对这套演化没有整体视角，就很容易形成两个极端判断：

- 要么觉得“传统图形管线已经过时了”
- 要么觉得“这些新东西都只是外挂特效”

这两种理解都不够准确。

## 这篇要回答什么

这一篇主要回答四个问题：

1. 为什么现代渲染会越来越复杂。
2. 新技术到底是在替代传统管线，还是在扩展它。
3. Compute、Ray Tracing、TAA、Upscaling 各自插在什么位置。
4. 为什么“这一帧”越来越不只由当前帧决定。

如果先压缩成一句最短判断，那就是：

`现代渲染不是把旧图形系统推翻重来，而是在传统图形骨架上不断增加新阶段、新数据和新时域依赖。`

## 为什么复杂度会持续增加

因为图形目标一直在同时抬高。

玩家和开发者想要的东西越来越多：

- 更高分辨率
- 更高帧率
- 更复杂几何
- 更真实光照
- 更稳定帧时间
- 更低输入延迟
- 更强后处理表现
- 更好的跨平台体验

这些目标彼此并不天然兼容。

分辨率升高，像素成本就涨；帧率升高，每帧预算就变短；光照和材质更复杂，单位像素成本又更高。

所以现代渲染越来越不像“把一帧老老实实从头算完”这件事，而更像：

`在有限预算里，把哪些工作放在这一帧做，哪些工作跨帧复用，哪些工作交给别的阶段或别的路径完成。`

这就是复杂度增加的根源。

## 传统图形管线并没有消失

这是最先要讲清楚的一点。

很多人一看到光追、AI 超分、Frame Generation，就会直觉觉得传统图形管线被替代了。

其实不是。

即使在今天的大型游戏里，传统光栅化与图形管线仍然是主干之一。

为什么？

因为这些基础问题依然存在：

- 几何怎么处理
- 屏幕空间覆盖怎么判断
- 材质和贴图怎么参与着色
- 深度和混合怎么控制输出

很多现代路线不是把这些基础问题取消了，而是在它们之上再加层。

所以更准确的说法是：

`传统图形管线仍然是骨架，现代技术是在骨架上加器官和神经系统。`

## Compute 为什么会越来越重要

Compute 的意义，不只是“又一种 Shader”。

它的重要性在于，它给引擎和图形系统提供了一条不完全受传统图元绘制路径约束的通用 GPU 计算能力。

这意味着很多工作不一定非要塞进传统 draw call 路径里。

于是你会看到越来越多东西转向 compute：

- 粒子模拟
- 光照剔除和分桶
- 后处理
- 体积效果
- 某些阴影和屏幕空间算法
- 数据预处理和缓存构建

这并不代表传统图形管线没用了，而是说明现代渲染开始同时依赖：

- 图形绘制路径
- 通用 GPU 计算路径

两者共同组织一帧画面。

## Deferred、Forward+ 这类架构为什么会让整帧更复杂

在前面几篇里我们已经讲过，Deferred 和 Forward 首先是整帧组织方式。

现代渲染继续复杂化，很大程度上就来自这些组织方式不断叠层。

比如一帧里可能会出现：

- Depth Prepass
- G-Buffer Pass
- Lighting Pass
- Shadow Pass
- Transparent Pass
- 屏幕空间效果 Pass
- 后处理链

也就是说，现代渲染不是单次 draw 把世界算完，而是把整帧拆成了越来越多相互依赖的阶段。

复杂度不只来自某个算法更难，而是来自：

`一帧被拆成越来越多彼此交换数据的步骤。`

## TAA 为什么让“这一帧”不再只属于这一帧

这是现代渲染复杂化的一个巨大转折点。

在更早的直觉里，我们总觉得画面是“当前帧算当前帧”。

但 TAA 和其他时域技术把这个直觉改掉了。

因为它们开始依赖：

- 历史帧颜色
- 运动矢量
- 深度历史
- 抖动采样
- 跨帧收敛

也就是说，今天屏幕上这张图，不再只是当前这 16.67ms 或 8.33ms 的纯结果，而是多个帧共同收敛出来的结果。

这件事很重要，因为它意味着“时间”本身已经变成图像质量的一部分。

后来的很多现代渲染路线，实际上都在沿着这条路继续走。

## Upscaling 为什么不是单纯“拉伸分辨率”

这也是一个常见误解。

很多人还会把 DLSS、FSR、XeSS 这类技术简单理解成“更聪明的放大器”。

但现代 upscaling 的核心，往往已经是：

- 用较低成本渲染基础帧
- 利用历史帧和运动信息重建高分辨率结果
- 通过模型或时域算法恢复细节

所以它真正改变的是图形预算分配方式。

过去的问题更像：

`怎样把目标分辨率上的每个像素都尽量算出来？`

现在的问题越来越像：

`哪些信息必须真实渲染，哪些可以跨帧重建，哪些可以交给模型推断？`

这不是简单的“把图拉大”，而是在重写图像形成路径里的预算分工。

## Ray Tracing 插在了哪

很多人会问：那光追到底是替代谁？

最准确的理解不是“光追替代整个图形管线”，而是：

`它把某些原本依赖传统近似方法的问题，改用另一种更昂贵但更物理一致的求解路径。`

比如：

- 反射
- 阴影
- 全局光照
- 某些间接光照与可见性问题

这意味着现代图形系统里，可能会同时存在：

- 传统光栅化主路径
- 光追参与的特定子路径
- 之后再叠加 denoise、reconstruction、post-process

所以 Ray Tracing 不是“一个独立新世界”，而是被插入现代渲染系统的一部分。

## AI 和神经重建为什么让图形系统继续分层

随着 AI 重建、Frame Generation、Ray Reconstruction 一类技术加入，图形系统又多了一层复杂度：

`模型开始参与图像形成。`

这意味着最终输出帧，可能不再只是：

- 几何 + 光照 + 后处理

而会变成：

- 基础渲染结果
- 历史帧信息
- 运动信息
- 神经网络重建或补帧
- 再经过显示链输出

也就是说，现代图形系统开始越来越像一套混合流程：

- 一部分工作由传统图形管线完成
- 一部分工作由 compute 和缓存完成
- 一部分工作由时域算法完成
- 一部分工作由模型重建或生成完成

这就是为什么它越来越复杂，也越来越难用单张“经典图形管线示意图”讲完。

## 为什么这件事对引擎特别重要

复杂度增加不只是显卡厂商的事，它会直接回到引擎和项目里。

引擎必须决定：

- 哪些 pass 放在哪
- 哪些历史帧数据要留
- 哪些 feature 可以在什么平台打开
- 哪些效果值得走 compute 路线
- 哪些平台需要 upscaling 或 fallback
- 哪些设备无法承担某些现代路径

所以现代渲染的复杂化，最后并不是“多几个技术名词”，而是：

`整帧调度和数据依赖关系显著变重。`

这也解释了为什么现代引擎越来越强调：

- Render Graph
- Resource Lifetime
- Async Compute
- Frame History
- Feature Toggle
- Quality Tier

因为不这么组织，整帧很快就会失控。

## 常见误解

### 误解一：现代渲染已经不需要传统图形管线了

不是。传统图形管线仍然是大部分实时渲染的重要骨架，很多现代路线都是在它之上叠加，而不是把它彻底抛弃。

### 误解二：Compute 会取代所有图形路径

也不对。Compute 非常重要，但它不是把所有绘制路径一锅端掉，而是补上了传统图形路径不最适合处理的那部分工作。

### 误解三：TAA、DLSS、FSR 只是后处理细节

太轻了。它们已经开始改变图像质量如何形成、性能预算如何分配，以及“这一帧”本身的定义。

### 误解四：现代渲染更复杂，只是因为厂商喜欢堆概念

不准确。复杂度增加的根本原因，是画质、分辨率、帧率、延迟和硬件预算之间的矛盾越来越尖锐，不加层根本无法同时逼近这些目标。

## 我的结论

如果必须用一句话概括这篇，我会这样说：

> 现代渲染的复杂化，不是传统图形系统失效了，而是它为了继续逼近更高的图像质量和实时目标，不得不在原有骨架上持续增加 compute、时域积累、光追路径和神经重建这些新层。

理解这一点之后，后面很多现象就会顺很多：

- 为什么“这一帧”越来越依赖上一帧
- 为什么图形系统越来越像多阶段混合流程
- 为什么引擎里的整帧调度和资源生命周期管理越来越重要
