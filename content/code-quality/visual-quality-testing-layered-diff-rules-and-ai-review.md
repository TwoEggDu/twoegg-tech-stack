视觉质量问题有一个很麻烦的特点：

- 它经常不是 crash
- 它也不一定直接阻断功能
- 但玩家一眼就能看出来"这个版本不对劲了"

比如：

- 材质突然变粉
- 某个关键物件消失
- 角色武器、特效或阴影没了
- 某个平台上透明排序错了
- 某个 shader variant 明明应该命中，但表现像根本没生效

这些问题最容易掉进两个极端：

- 要么团队完全靠人工看
- 要么团队试图把它们全压成一套"自动截图 diff"

前者成本高且不稳定，后者又会遇到大量噪声和误报。再往前走一步，很多人会自然想到"那就让 AI 判断画面对不对"，但如果直接把 AI 当第一道门，这件事也很容易失控。

`视觉质量测试真正要解决的，不是"谁来判断画面对不对"，而是怎样把规则检测、图像差异、AI 语义复核和人工裁决分层放到各自最值钱的位置。`

这也是为什么这一篇不写成"AI 截图审查教程"，而是先把整体分层立住。

## 视觉质量问题到底在解决什么

和 [兼容性测试策略：设备矩阵划定优先级、Device Farm（AWS / Firebase Test Lab）、驱动差异检测]({{< relref "code-quality/compatibility-testing-device-matrix-and-device-farm.md" >}}) 一样，视觉质量问题常常只会在某个具体交叉点上暴露：

- 某个平台
- 某条渲染路径
- 某个图形档位
- 某个 shader keyword 组合
- 某个资源或热更快照

更麻烦的是，这类问题并不总能被普通功能测试发现。流程走通了，不代表画面就是对的。

所以这篇真正要回答的是：

`当"功能还能用，但画面已经错了"时，团队怎样把这种问题稳定变成可自动拦截、可回归、可复核的质量信号。`

## 一上来就让 AI 判图，为什么不稳

AI 在视觉检查里当然有价值，但如果一开始就把它放在第一层，会立刻遇到几个问题：

- 它不擅长替代硬规则
- 同一张图里的关注点不总是稳定
- 没有上下文时，很难知道"这是预期变化还是异常"
- 输出可解释性通常不如规则检测和 diff 明确

比如：

- `粉色材质`
- `材质引用丢失`
- `某个关键 prefab 没实例化`
- `某个 shader keyword 压根没开`
- `某个 pass 没编进去`

这些问题不该主要依赖 AI，因为它们本来就更适合被规则和运行时状态直接证明。

所以更稳的原则应该是：

`AI 不该替代规则层，也不该替代图像差异层；它更适合做规则和 diff 之后的语义复核。`

## 更稳的四层视觉质量检测

我更愿意把视觉质量测试拆成下面四层：

1. 规则检测
2. 图像差异检测
3. AI 语义复核
4. 人工裁决

这四层不是互相替代，而是逐层收窄问题面。

## 第一层：规则检测先拦"本来就能被证明错了"的问题

这一层最重要，因为它最便宜、最稳定、最适合放进 CI。

适合在这一层拦的典型问题包括：

- 材质 / shader / texture / mesh 引用丢失
- 粉色材质或 fallback shader
- 关键 prefab / 关键 renderer / 关键特效组件缺失
- shader variant keyword 组合不合法
- 关键 pass 没编进包里
- 某些平台专用资源没有生成

这一层尤其适合处理你刚才提到的三类问题：

### 1. `粉色`

粉色本质上通常不是"画面审美异常"，而是材质链或 shader 链断了。它更适合通过：

- 材质引用检查
- shader 是否存在
- shader 是否被剔除
- 平台变体是否可用
- 运行时 fallback 统计

来直接拦，而不是让 AI 看一张截图再猜。

### 2. `丢失物件`

如果是"应该出现的关键物件根本没生成"，也不应该先靠 AI。更稳的做法通常是：

- 关键对象实例化断言
- 特定场景下的对象计数或标签计数
- 关键骨骼挂点 / 武器挂点 / VFX 挂载点存在性检查

AI 可以辅助发现"看起来像少了东西"，但对象到底存不存在，规则层往往能更直接地回答。

### 3. `variant 是否正确`

这个问题最容易被误写成"让 AI 判断变体对不对"。其实更稳的顺序应该是：

- 先检查 keyword 是否按预期组合
- 再检查对应 pass / variant 是否被保留
- 再检查目标平台或目标图形 API 下的编译结果
- 最后如果画面仍然异常，再让 AI 或人工看语义表现

也就是说：

`variant 正确性首先是编译与运行时状态问题，其次才是截图语义问题。`

## 第二层：图像差异检测负责证明"它真的和基线差太多了"

规则检测能回答"链是不是断了"，但它回答不了所有问题。很多画面异常不是引用丢了，而是：

- 光照结果变了
- 阴影整片没了
- 某个平台上透明叠加错了
- 后处理参数跑偏
- LOD、剔除或深度写入逻辑不对

这时就需要图像差异层。

这里更稳的做法通常是：

- 固定截图点位
- 固定平台 / 图形 API / 质量档位
- 固定基线图
- 用 `Perceptual Diff` 而不是纯像素差做第一轮对比

这层真正负责回答的是：

`即使资源引用都没坏，这次画面是不是已经和稳定版本偏得太远了。`

但它也有天然限制：

- 很容易被噪声影响
- 动态内容会让差异抖动
- 不一定知道"差异意味着什么"

所以它适合做"发现异常"，不适合单独做"最后裁决"。

## 第三层：AI 语义复核，负责把"差异"翻译成"像不像真实问题"

AI 最适合放在这里。

也就是说，前两层已经给了它足够多的上下文：

- 规则层没有找到硬错误，或者只找到部分信号
- Diff 层已经证明这张图和基线差异明显
- 现在需要一个更接近"人怎么看图"的判断

这时 AI 最适合做的是：

- 关键物件是否疑似消失
- UI 是否被异常遮挡
- 某个角色、武器、特效是否明显不符合预期
- variant 看起来是否没有正确生效
- 某个平台截图是否呈现了明显的异常语义，而不是单纯像素噪声

这里 AI 的角色更像：

`第二审查员，而不是第一道 gate。`

它可以帮助回答：

- 这次 diff 更像噪声还是像真实故障
- 这块区域的异常更接近"少了东西"还是"参数变了"
- 这个版本是否值得升成人工复核或阻断

但它不该直接替代规则层，因为：

- 它无法稳定替代 shader / variant / 引用链状态判断
- 它也不该直接替代最终人类裁决

## 第四层：人工裁决负责最终决定

视觉质量问题最终仍然需要有人类裁决。原因很简单：

- 游戏画面本来就会变化
- 有些变化是预期更新，不是回归
- 有些问题技术上存在，但商业上不一定值得阻断
- 有些 AI 或 diff 报警虽然"像问题"，但实际上是噪声

所以最终更稳的闭环应该是：

- 规则层拦确定性错误
- Diff 层给出异常证据
- AI 层做语义辅助判断
- 人工层做最后发布决策

这也意味着，视觉质量流程不该追求"完全无人裁决"，而应该追求：

`让人只在最值得花时间的异常上做裁决。`

## 最常见的假视觉质量测试

### 1. 全靠人工巡检

最后看图的人越少，质量越依赖个人经验；版本一多，这套机制会先被时间打垮。

### 2. 全靠像素 diff

截图里一点正常波动就会产生报警，团队很快学会忽略它。

### 3. 一上来就让 AI 当裁判

结果是规则该拦的没提前拦，AI 却在一堆噪声里反复做高成本判断。

### 4. 不区分"资源链断了"和"语义上看起来不对"

这两类问题看起来都叫"画面坏了"，但检测手段完全不同。

### 5. 只报异常，不绑定发布动作

看起来有截图、有 diff、有 AI 评论，但最后没人知道这条异常到底会不会阻断版本。

## 一条最小可执行路径

如果团队现在还没有像样的视觉质量测试，我会建议先按这条最小路径落：

1. 先挑 3 到 5 个最关键场景，固定截图机位、平台和图形档位。
2. 先把粉色、丢引用、关键对象缺失、variant / keyword 明显异常做成规则检测。
3. 先给这些固定场景建立 `Perceptual Diff` 基线，不追求一口气覆盖所有内容。
4. 再把 AI 接进来，但只用于复核"规则没拦住、diff 又明显异常"的截图。
5. 最后把报警收敛到人工只需要处理少数真正可疑的问题。

只要这五步站住，视觉质量测试就不会再是"靠谁眼尖"，而会开始变成一条能进 CI、能回归、能服务发布决策的工程链。

## 结尾

所以我现在更愿意把视觉质量测试理解成这样一条链：

`先用规则检测拦粉色、丢引用、丢物件和 variant 状态异常，再用 Perceptual Diff 证明画面偏离了稳定基线，再让 AI 只对高可疑差异做语义复核，最后由人工完成版本裁决。`

只有这样，AI 才不会被神化成"万能看图裁判"，而会被放到一个真正有工程价值的位置上。
