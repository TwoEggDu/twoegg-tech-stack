> 大多数项目的质量问题不是没人测，而是测的那一层根本不是问题最常出现的那一层。

这是游戏质量保障体系系列的索引页。
它不讲新的底层机制，只做一件事：

`给这组文章补一个稳定入口，让读者知道先读哪篇、遇到什么问题该回看哪篇。`

## 这篇要回答什么

1. 这组文章现在已经覆盖了哪些主题。
2. 如果按系统阅读走，最稳的顺序是什么。
3. 如果不是系统读，而是项目里遇到具体问题，该先跳哪几篇。
4. 这组文章和 Code Quality 系列是什么关系。

## 先给一句总判断

如果把整个系列压成一句话，我会这样描述：

`游戏质量不是 QA 测出来的，而是从配表、资源、逻辑、兼容性到线上监控的每一层都有防线，再用门禁把这些防线串成闭环，最后靠团队文化让闭环持续运转。`

所以这组文章故意不是按"怎么写测试"去展开，也不是按某个 CI 工具的功能平铺，而是按问题层拆：

- 游戏质量到底由哪些层构成
- 门禁应该设在管线的哪些位置
- 配表、资源、逻辑、兼容性各自的防线怎么建
- 线上质量怎样监控和响应
- 为什么工具和流程到位了，质量仍然可能失败

## 这组文章的核心模型

整个系列围绕两个模型展开：

**六层质量问题空间**（01 篇建立）：

```
第 1 层  配表质量    ← 策划数据：系数、引用、公式、版本
第 2 层  资源质量    ← 美术资产：尺寸、格式、LOD、引用、冗余
第 3 层  逻辑质量    ← 代码正确性：空引用、状态机、竞态、协议
第 4 层  兼容性质量  ← 多设备：GPU 驱动、OS 版本、分辨率、内存
第 5 层  性能质量    ← 帧率、内存、加载、发热（由性能工程栏目覆盖）
第 6 层  线上质量    ← 崩溃率、ANR、卡顿、内存泄漏、灰度回滚
```

**七道门禁管线**（02 篇建立）：

```
Gate 1  Pre-commit        → 格式、敏感文件、大文件
Gate 2  资源导入          → 压缩格式、面数、命名
Gate 3  PR 合并（最关键）  → 编译、测试、依赖、变体、Review
Gate 4  构建              → 全量编译、包体、Shader 变体、IL2CPP
Gate 5  冒烟测试          → 启动、核心流程、截图对比、帧率基线
Gate 6  发布审批          → 文档、签名、灰度方案、冒烟报告
Gate 7  线上监控          → 崩溃率、ANR、异常 API、自动回滚
```

03–08 篇分别展开这些层和门禁的具体实现。

## 这组文章和 Code Quality 系列是什么关系

同在 `code-quality` 栏目下，但关注点不同：

- **Code Quality 系列**按工程管线阶段组织，覆盖 Code Review、静态分析、CI 门禁、Shader 变体治理、崩溃分析等具体工具和实践。它回答的是"这个环节用什么工具、怎么配、怎么进 CI"。
- **游戏质量保障体系**按质量层级组织，先把配表、资源、逻辑、兼容性、线上拆成独立的问题层，再用门禁串起来，最后用文化收口。它回答的是"质量体系怎么设计、六层防线怎么分工、门禁怎么串成闭环"。

简单说：这个系列给架构，Code Quality 系列给零件。两个系列互相引用但不互相依赖。

## 这篇真正想留下来的结论

游戏质量最怕的，不是某个测试没覆盖到，而是团队压根不知道质量由哪些层构成。

只要没有这张地图，团队讨论就会自动混成：

- 测试
- Bug
- QA
- 上线

全都像"质量"的一部分，但谁也说不清楚防线该建在哪。

而这组系列真正想做的，就是先把六层问题空间拆开，再用门禁和文化把它们重新接成一个闭环。

如果把整篇压成最短一句话：

`先搞清楚质量在哪一层出了问题，再去想用什么工具拦。这个索引页的作用，就是先给你这张地图。`
