很多团队一说“质量工具”，脑子里先想到的还是风格检查。

但真正值钱的质量工具，从来不是帮你挑空格和命名，而是尽量把一些本来会拖到 review、测试甚至运行时才暴露的问题，前移到：

- 编译期
- 导入期
- 构建期

`只要某个问题已经能被写成稳定规则，就不该继续主要依赖人脑记忆。`

这也是为什么“代码质量工具”和“资产质量工具”其实应该放在一起看。

它们解决的都是同一个问题：

`哪些规则已经清楚到可以由机器稳定执行。`

## 先把两类工具分开

如果把这件事拆得更清楚一点，我会把它分成两条线：

### 1. 代码侧静态分析

典型工具包括：

- Roslyn Analyzer
- Clang-Tidy
- 编译器 warning as error
- 自定义危险 API 扫描

它们更适合解决：

- 明确禁止的写法
- 已知危险 API
- 层级依赖违规
- 可枚举的 async / 异常 / 空引用风险

### 2. 资产侧导入与构建校验

典型工具包括：

- 资产导入器校验
- 构建前资源规则检查
- bundle / layout / manifest 分析
- 资源命名、路径、尺寸、压缩和引用契约检查

它们更适合解决：

- 资产命名和目录边界
- 导入参数错误
- 平台压缩配置错误
- 缺引用、重复资源、错误依赖
- 构建产物与资源预算异常

这两类东西如果分开看，很容易各自退化成一堆工具清单。
但放回工程流程里看，它们其实站在同一层：

`都是在替团队前移规则。`

## Roslyn / Clang-Tidy 最适合抓什么

静态分析最常见的失败，不是能力不够，而是抓错了东西。

如果一上来就想把所有“看起来不优雅”的写法都塞进去，最后通常会得到：

- 大量误报
- 团队开始无视告警
- 真正高价值的问题淹没在噪声里

所以我更建议先抓这几类：

### 1. 明确禁止的 API 或调用方式

例如：

- 某层禁止直接访问某组单例
- 运行时逻辑禁止直接打某类编辑器 API
- 某些耗时或危险调用禁止进入热点路径
- 某类同步 I/O 禁止出现在主线程入口

这类规则的特点是边界清楚、误报少、收益高。

### 2. 层级依赖违规

这是最值钱、也最容易被低估的一类。

如果你已经知道某个模块不该直接依赖另一个模块，就应该尽量把这件事写成机器规则，而不是靠 reviewer 每次临场提醒。

### 3. 明确可枚举的 async / 异常 / 空值风险

例如：

- 忽略返回 Task
- 在不该阻塞的路径里同步等待
- 明确的空值使用风险
- 某些必须成对调用的资源释放约束

这类问题不是因为“高级”才值得静态分析，而是因为它们太适合被稳定枚举。

## 资产导入时校验，值钱在“够早”

和代码侧静态分析一样，资产质量工具最值钱的地方也不是多，而是早。

如果一个资源问题直到构建时、真机 smoke 或上线后才暴露，修复成本通常都会比导入期高一大截。

所以我更建议把资产规则尽量前移到导入期和构建前：

### 导入期最适合拦什么

- 命名规则
- 路径规则
- 贴图尺寸、压缩格式、平台 override
- 音频压缩和加载模式
- 模型导入设置
- 关键资源标签和分组归属

### 构建前最适合拦什么

- 缺引用
- 重复资源
- bundle / group 归属异常
- 关键资源预算超线
- 关键 Shader / Material / built-in 资源异常变化

这类问题一旦前移，团队会明显感受到两个变化：

- review 里少了很多重复提醒
- 真机 smoke 开始不再承担那么多“本该更早就发现”的垃圾错误

## 为什么代码和资产规则要放在一张图里

很多项目把这两件事分给两组人，最后就会出现一种典型断裂：

- 程序在守代码规范
- TA / 美术 / 资源侧在守资产规范
- 但没有人把两边当成同一条质量护栏

结果就是：

- 代码侧觉得资源问题不归自己管
- 资源侧觉得规则前移是程序工具链的事
- 最后最早的阻断点没人真正立起来

但从工程角度看，这两类规则的本质完全一致：

- 都是已知规则
- 都能稳定枚举
- 都值得前移

所以更稳的做法，是把它们都收进同一套质量机制里：

- 本地导入时先挡一层
- CI 再挡一层
- 构建产物里再看预算和趋势

## 这些规则该放在哪一层

最容易用错工具的地方，是把所有规则都堆到同一个阶段。

更稳的切法通常是这样：

| 阶段 | 更适合放什么 |
|------|--------------|
| IDE / 编译期 | Roslyn、warning、危险 API、层级依赖规则 |
| 导入期 | 资源命名、贴图 / 音频 / 模型导入参数、关键标签 |
| PR / CI | 快速静态分析、关键契约校验、禁止回归的规则 |
| 合并后 / Nightly | 更重的资源分析、layout 对比、预算趋势 |
| 发版前 | 最终产物校验、符号 / 包体 / 关键资源预算确认 |

也就是说，不要把本该在导入期就拦住的资源规则拖到 CI，也不要把特别重的 layout 分析塞进每个 PR。

## 最常见的假质量工具

看起来接了很多工具，实际上没有真正提升工程质量，通常是下面几种：

### 1. 只抓风格，不抓边界

团队看起来告警很多，但真正危险的依赖违规、危险 API 和资产契约问题没有被优先处理。

### 2. 规则太多，但没有优先级

所有问题都一个级别，最后谁也不知道哪些是必须拦、哪些只是建议。

### 3. 误报太多

误报只要高到一定程度，团队最终一定会学会忽略它。

### 4. 只在 CI 才看见

如果一个资源命名或导入参数问题必须等到 CI 才暴露，那就说明它还没有被前移到真正值钱的位置。

### 5. 只有工具，没有 owner

规则很多，但没人持续维护规则，也没人判断什么时候该升级成硬 gate。

## 一条最低可执行路径

如果从零开始，我会建议先做下面这批最小版本：

1. 先挑 3 到 5 条最明确、误报最少、最有价值的代码规则。
2. 先挑 5 条最常见、最容易回归的资产导入规则。
3. 先让代码规则在编译期或 PR 阶段就可见。
4. 先让资源规则尽量在导入期可见，不要全部拖到构建后。
5. 对关键资源预算和构建产物，再单独做一层合并后 / nightly 检查。

这一套先跑稳，再慢慢往上加。
不要一开始就试图做一套无所不包的规则平台，那样很容易把团队拖进另一套维护噪声。

## 结尾

所以我现在更愿意把这类质量工具理解成这样：

`代码侧的静态分析负责尽早挡住危险写法和边界违规，资产侧的导入与构建校验负责尽早挡住资源契约和预算问题；两者本质上都在做同一件事：把已知规则尽量前移。`

真正值钱的不是“工具很多”，而是规则开始在最早、最便宜的阶段被稳定拦住。
