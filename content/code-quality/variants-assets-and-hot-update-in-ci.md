很多团队的 CI 看起来已经很完整了：

- 代码能编译
- 单测能过
- 包能打出来

但真到了线上真正容易炸的那几条链，往往并不在这三件事里。

真正高频出事故的，常常是这些：

- 某个 shader variant 被裁掉了
- 某个资源快照和 catalog 对不上
- 某条热更链的元数据、DLL、配置或发布快照不一致

也就是说，很多 CI 实际上只验证了：

`代码世界还能不能生成一个包。`

它并没有稳定验证：

`内容世界是不是还是那一份内容世界。`

所以这篇真正要讲的，是怎样把 `变体、资源、热更` 这三条容易散落在不同系列里的工程链路，收成一条 CI 流程里的统一验证线。

## 先给一句总判断

如果把整件事压成一句话，我会这样说：

`变体、资源、热更链路接进 CI，不是给构建流程多塞几步脚本，而是让每次构建都显式产出一组能证明“内容世界仍然正确”的证据：variant 报告、资源快照、发布映射、热更产物和最小烟测结果。`

这里最关键的不是“跑了多少工具”，而是：

- 每次构建都能产出同类证据
- 这些证据能和稳定版本做对比
- 超线后有明确的阻断、告警和回查入口

## 为什么这三条链总是容易掉在 CI 之外

它们掉出去，往往不是因为团队不知道重要，而是因为它们天然跨层：

- `variant` 站在渲染、构建裁剪和运行时表现之间
- `资源` 站在内容组织、分发、缓存和回滚之间
- `热更` 站在代码产物、元数据、资源快照和发布流程之间

一旦 CI 只围绕“源代码有没有编过”，这三条链就很容易退化成：

- 本地手工看一次
- 发版前临时想起来补一次
- 出事故后再回头追一次

这也是为什么很多团队明明有 [Shader Variant 数量监控与 CI 集成：怎么把变体治理接入构建流程]({{< relref "rendering/unity-shader-variant-ci-monitoring.md" >}})、[Unity 资源交付工程实践：分组、命名、版本、缓存、回滚和烟测基线]({{< relref "engine-toolchain/unity-resource-delivery-engineering-practices-baseline.md" >}})、[HybridCLR 打包工程化｜GenerateAll 必须进 CI 流程，Development 一致性与 Launcher-only 场景]({{< relref "engine-toolchain/hybridclr-ci-pipeline-generate-all-and-development-flag.md" >}}) 这类局部实践，整体流程还是会断。

因为它们缺的不是“某个点会不会做”，而是：

`这三条链到底怎样被放进同一条 CI 视角。`

## 一、先把目标改掉：CI 不只是验证代码世界

更稳的 CI 目标，不该只是：

- 代码编译通过
- 构建脚本跑完

而应该至少覆盖三层：

1. `代码世界`
代码、测试、构建脚本本身是否稳定。

2. `内容世界`
variant、资源映射、catalog、快照、关键产物是否仍然处在可发布状态。

3. `发布世界`
这次构建对应的资源、热更、回滚抓手和版本元信息是否能一起落地。

如果 CI 只覆盖第一层，后面很多“真机构建才出、内容分发才出、热更切换才出”的事故都挡不住。

## 二、Variant 链路进 CI，核心不是“看总量”，而是让变化可追踪

variant 这条线最容易犯的错，是只在出问题时临时审计一次。

更稳的做法应该是每次构建都产出：

- 本次 variant 明细
- 本次汇总报告
- 与稳定基线的 diff

这条链最适合在 CI 里做三类事：

### 1. 构建期采集

每次构建都固定产出 variant 报告，而不是靠人工按需导出。

### 2. 基线对比

拿当前构建和最近稳定版本对比：

- 总量有没有异常抬升
- 某个关键 shader 有没有异常膨胀
- 是否出现新的 shader 或 pass

### 3. 阈值动作

把结果按风险分层：

- 绝对超线，直接阻断
- 可疑增长，给告警
- 预期变化，只留存档

variant 这条链真正进了 CI，价值不在“数字永远不变”，而在：

`每次变化都能被看见，异常变化能在合并前就有信号。`

## 三、资源链路进 CI，核心不是“包能打出来”，而是快照和映射还能成立

资源链是最容易被低估的一层。

很多团队把资源 CI 理解成：

- bundle 构建成功
- Addressables build 成功
- 远端目录有文件

这些当然重要，但都太粗。

资源链真正值得放进 CI 的，是这些证据：

- `BuildLayout`
- `Manifest`
- `Catalog`
- 资源快照版本
- 关键 group / bundle 的大小、数量、依赖变化
- 关键入口实例化或首载烟测结果

这和 [Unity 资源系统怎么做烟测和回归：从构建校验、入口实例化到 Shader 首载]({{< relref "engine-toolchain/unity-resource-system-smoketests-and-regression.md" >}}) 讲的是同一件事：资源问题真正高发的地方，不是“能不能 build”，而是“内容映射和首次真实使用时会不会坏”。

所以资源链接进 CI，至少应该回答：

- 本次快照是不是完整的
- 关键映射是不是没漂
- 大小、依赖和重复资源有没有异常跳变
- 关键入口是不是还能完成最小实例化或首载
- 回滚时能不能明确指回上一个稳定快照

也就是说：

`资源 CI 的核心不是构建动作本身，而是内容快照还能不能被证明是正确的。`

## 四、热更链路进 CI，核心不是“DLL 生成了”，而是整条兼容链一致

热更链往往最容易产生一种错觉：

- DLL 有了
- 包也打出来了
- 那应该没问题

但真正容易炸的地方，恰恰是“一半是对的，一半不一致”：

- 这次 `GenerateAll` 用的环境和最终构建不一致
- 热更 DLL、补充元数据和资源快照不是同一批
- 发布元信息能指到代码，但指不到正确资源
- Launcher-only、Development、一致性标记没有固定下来

所以热更链进 CI，最重要的不是“把某个生成命令自动跑掉”，而是把下面这些东西显式化：

- 生成命令的参数来源
- 生成产物的 hash 或版本标识
- 与最终构建一致的环境标志
- 这次发布对应的资源快照和回滚点
- 一条最小热更烟测结果

如果做不到这些，所谓“热更进 CI”最后很容易退化成：

`我们只是把本地手点菜单改成了服务器上自动点一次。`

## 五、把三条链统一进一条 CI 流程

更稳的做法，不是给每条链各自加一个孤立脚本，而是把它们放回同一条发布前流水线。

如果压成一条最小流程，我会建议至少拆成下面四段：

### 1. 合并前或 PR 阶段：轻量静态检查

这阶段重点是快：

- 命名、目录和配置约束
- 关键资源 / 配置引用合法性
- 热更元信息最小一致性

它负责尽量早地挡住明显错误。

### 2. 合并后或 nightly：完整构建与报告产出

这阶段重点是把证据产出来：

- variant 报告
- 资源构建报告
- hot-update 产物与版本元信息
- 大小、依赖、重复和快照摘要

它负责让“内容世界”有稳定可回看的报告。

### 3. 候选版本阶段：最小烟测与关键路径验证

这阶段重点是验证：

- 关键入口实例化
- 关键 shader 路径或首载
- 热更模块最小加载 / 执行
- 资源快照与回滚指针

它负责证明“这批产物不只是生成了，而且最小关键路径能活”。

### 4. 发布阶段：归档与回滚抓手

这阶段重点不是再多做一次构建，而是明确保存：

- variant 报告
- 资源快照和 catalog
- 热更 DLL / metadata / mapping
- 符号文件、构建日志和发布元信息
- 上一个稳定版本的回滚定位点

它负责让出问题后的回查和回退有抓手。

## 六、门禁不要一刀切，最好分成三档

这三条链接进 CI 后，最容易犯的另一个错，是所有异常都直接失败。

更稳的做法通常是：

### 1. 硬阻断

适合拦：

- 构建产物缺失
- 快照映射不完整
- 热更关键生成步骤失败
- 关键入口烟测失败

### 2. 软告警

适合拦：

- variant 数量异常上涨
- 关键 bundle 大小异常上涨
- 某个 shader 或 group 依赖异常膨胀

### 3. 观察指标

适合持续看：

- 历史趋势
- 波动区间
- 单模块贡献度

这样做的好处是：既不会把 CI 搞成“天天红”，也不会让真正危险的问题继续漂过去。

## 一条最小可执行路径

如果团队现在还没有把这三条链正式接进 CI，我会建议先按下面这条最小路径落：

1. 先固定每次构建必须产出的五类证据：variant 报告、资源构建报告、快照映射、热更产物、最小烟测结果。
2. 先选三条最值钱的硬阻断：关键入口烟测失败、资源快照不完整、热更关键生成步骤失败。
3. 再选两条最值钱的软告警：variant 异常增长、关键 bundle 或 group 大小异常增长。
4. 先把这些报告做成固定归档路径，不要每条链各自散在日志里。
5. 发布时把“本次代码 commit、资源快照、热更版本、回滚点”显式绑在一起。

只要这一步站住，CI 才算从“代码能不能 build”进化成“内容世界还能不能发布”。

## 结尾

所以我现在更愿意把这件事理解成：

`变体、资源、热更链路接进 CI，不是在原有构建后面多挂几个脚本，而是在每次构建里把内容世界的关键证据显式化、可对比、可归档、可阻断；只有当 variant 报告、资源快照、热更产物和最小烟测一起进入流程，CI 才真正开始对版本可发布性负责。`

只有这样，这三条过去总是各自为战的工程实践，才会真的变成同一条质量护栏。
