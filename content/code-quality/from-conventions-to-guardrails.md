很多团队一提"规范"，第一反应还是：

- 起一份命名文档
- 约一个目录结构
- 再补一页"开发约定"

这些东西当然有价值，但它们最容易卡在同一个地方：

`大家都知道有这份规范，但它既没有变成默认路径，也没有变成阻断错误的护栏。`

于是团队会进入一种很熟悉的状态：

- 文档越写越长
- 例外越来越多
- review 里重复争论同一类问题
- 新人只能靠口头传承猜"这里真正的隐含规则是什么"

所以这篇真正想回答的不是"要不要写规范"，而是：

`哪些问题适合停留在规范层，哪些问题必须升级成护栏；而命名、目录边界、配置契约和脚手架，到底各自解决什么。`

## 先给一句总判断

如果把这件事压成一句话，我会这样说：

`规范负责表达共识，护栏负责减少自由度；当某类错误已经高频、可重复、代价明确时，它就不该继续只靠文档和 review，而应该前移成命名规则、目录边界、配置契约或脚手架中的某一种。`

这句话里最关键的，是后半句：

- 不是所有规范都该自动化
- 也不是所有问题都该做成一个 linter
- 更不是"加个脚手架"就等于团队治理已经成立

真正重要的是先分清：你到底在解决哪一类协作问题。

## 为什么很多规范最后会失效

规范最常见的失效方式，不是内容写错了，而是它承担了不该承担的工作。

比如很多团队会混在一起说下面这些事：

- 变量名和目录名该怎么写
- 模块之间能不能互相引用
- 配置表里哪些字段必填、哪些字段可空
- 新功能一开始应该从哪套模板起步

这些问题都和"规范"有关，但它们显然不是同一种问题。

如果不先切开，后面就会出现两个极端：

- 要么什么都停留在文档层，结果只能靠 reviewer 和老成员记忆维持
- 要么什么都想自动化，最后做出一堆既不稳定、也不好维护的工具

所以更稳的起点不是"先写规则"，而是：

`先判断这条规则是在约束命名、约束依赖边界、约束配置契约，还是约束起手方式。`

## 一、命名规则解决的是"识别成本"，不是架构正确性

命名规则最适合解决的问题，是：

- 人能不能一眼看懂这个东西是什么
- 搜索和排查时能不能快速定位
- 跨角色协作时，大家是不是在用同一套词汇指代同一类对象

比如：

- 资源组名是不是稳定表达业务语义
- `FooConfig`、`FooService`、`FooPresenter` 这类后缀是不是有一致含义
- 热更 DLL、资源快照、构建产物的命名是不是能直接看懂版本和用途

命名规则最值钱的地方，不在"看起来整齐"，而在：

`它能降低沟通成本、排查成本和误读成本。`

但命名规则也有明显边界。它不负责解决：

- 模块之间到底能不能依赖
- 一个配置对象是不是合法
- 新模块起手时会不会漏掉必须文件

所以如果团队当前的问题是"目录一团乱、依赖互相穿透、配置字段经常漏填"，继续补命名文档通常不会有根治效果。

命名规则比较适合被升级成护栏的地方，是那些已经可以被稳定判断的名字约束，例如：

- 资源路径和 bundle 逻辑名格式
- 测试文件、编辑器代码、运行时代码的命名约定
- 发布产物、快照、回滚包、日志归档的命名模式

也就是说：

`命名护栏的价值，在于让"看一眼就知道它是谁"成为默认状态。`

## 二、目录边界解决的是"依赖方向和所有权"，不是可读性补丁

目录边界更适合解决的问题，是：

- 哪些代码、资源、配置应该属于同一个责任域
- 哪些模块允许直接引用，哪些不允许
- 谁对一块目录和其下资产拥有明确所有权

很多团队会把目录只当成"整理文件"的手段，但一旦项目进入多人协作，目录其实承担的是更重的职责：

`它在帮团队表达边界。`

例如：

- `Runtime / Editor / Tests` 不只是分类，而是不同运行环境的边界
- `Feature / Shared / Infrastructure` 不只是好看，而是在切责任方向
- `Client Config / Server Config / Generated Config` 不只是分文件夹，而是在切修改权和发布权

目录边界如果只是写在 wiki 里，最后很容易变成"大家大概都知道，但总有人例外"。

更稳的做法通常是把它和下面这些东西绑在一起：

- asmdef 或模块依赖限制
- 导入期 / 构建期的路径校验
- CI 里的 forbidden dependency 检查
- owner 归属和变更范围约束

目录边界真正不该承担的，是"补救命名不清"或"替代架构设计"。目录分得再漂亮，也不能自动证明模块职责已经正确。

它最适合解决的，其实是这类问题：

`哪些东西根本就不该站在一起，哪些引用根本就不该发生。`

## 三、配置契约解决的是"数据合法性和协作接口"，不是设计品味

配置契约在很多项目里是最容易被低估的一层。

团队常见的误判是：

- 表能配出来就行
- 字段多一点没关系
- 配错了回头靠运行时报错

这套思路在单人阶段还能撑一阵子，一旦进入多人协作、跨端发布、热更新和回滚，成本会迅速爆炸。

因为配置契约真正解决的是：

- 这个配置对象的最小合法形态是什么
- 哪些字段必填，哪些字段互斥，哪些字段必须成组出现
- 这个配置能不能被当前客户端、资源快照和服务端版本同时理解

也就是说，配置契约不是"策划表头该怎么起名"的美学问题，而是：

`跨角色协作时，数据世界的接口协议。`

这也是为什么很多高频事故都该前移到配置契约里，而不是留到运行时才发现，例如：

- 技能、资源、热更配置的引用链断了
- variant 相关开关组合非法
- 回滚时某个版本号或快照标识缺失
- 某些平台专用字段没填，但流程仍然放行

比较稳的做法，通常是把配置契约落实在：

- schema
- 静态校验器
- 导入期检查
- 构建期阻断
- 运行时最小防御性验证

这和前面的 [什么问题必须做成自动检查，不能靠人盯]({{< relref "code-quality/what-must-be-automated-checks.md" >}}) 是一致的：只要规则明确、能稳定判定、错误代价高，就不该继续只靠人盯。

## 四、脚手架解决的是"默认起手方式"，不是长期质量本身

脚手架也是最容易被神化的一类工具。

很多团队一觉得"大家写得不一致"，第一反应就是做一个模板生成器，仿佛这样一来质量问题就解决了。

脚手架真正适合解决的问题，其实更窄：

- 新模块起手时，哪些基础结构不该每次手写
- 哪些目录、测试、配置、注册代码应该默认就存在
- 哪些高频漏项应该在创建阶段直接补齐

比如：

- 新 feature 默认带上 `Runtime / Tests / Editor`
- 新配置默认带 schema、样例和校验入口
- 新资源分组默认带命名、分组和构建标签模板
- 新热更模块默认带版本元信息、校验入口和发布说明骨架

脚手架的价值，不在"生成很多代码"，而在：

`把正确的第一步做成默认第一步。`

但它不负责解决：

- 之后的依赖会不会继续腐化
- 设计边界会不会被破坏
- 生成出来的东西是不是始终被正确维护

所以脚手架适合补的是"起手一致性"，而不是替代目录边界、配置契约和后续验证链。

## 什么时候该从"规范"升级到"护栏"

我更建议用四个问题来判断：

1. 这类错误是不是高频出现？
2. 这类错误是不是可以稳定判定？
3. 这类错误的代价是不是已经高于维护护栏的成本？
4. 这类错误是不是已经不值得继续在 review 里反复争论？

如果这四个问题里，前面三个大多都是"是"，那它通常就不该再停留在口头规范层。

更直白一点说：

- 经常命名乱，但规则明确，那就做命名检查
- 经常目录穿透，那就做目录边界和依赖检查
- 经常配置漏填或组合非法，那就做契约校验
- 经常新模块漏掉固定结构，那就做脚手架

不要什么都丢给 reviewer，也不要什么都想靠培训解决。

## 一条最小可执行路径

如果团队现在想把"规范"往"护栏"推进，我会建议先按下面这条最小路径落：

1. 先盘点过去一段时间里最常重复的三类 review 问题，而不是先扩写文档。
2. 把这些问题分成命名、目录边界、配置契约和脚手架四类，不要混成一个"大规范包"。
3. 只优先升级那些规则明确、代价高、可以稳定判定的问题。
4. 命名和目录边界优先接入导入期、构建期或 CI 校验。
5. 配置问题优先做 schema 和静态校验，不要等运行时日志兜底。
6. 对高频起手动作补脚手架，但不要指望脚手架替代后续护栏。
7. 把还需要人做最后判断的问题继续留给 review，不要为了自动化而自动化。

这条路径的重点，不是一次性做很多工具，而是先把最值钱的自由度收住。

## 结尾

所以我现在更愿意把"从规范到护栏"理解成一次职责拆分：

`命名负责降低识别成本，目录边界负责收住依赖方向和所有权，配置契约负责定义数据世界的最小合法接口，脚手架负责把正确起手方式做成默认路径；真正成熟的团队，不是规范写得更长，而是知道哪类问题该停留在文档层，哪类问题必须前移成机器可执行的护栏。`

只有这样，规范才不会继续停留在"大家大概都知道"，而会变成团队长期能执行的工程机制。
