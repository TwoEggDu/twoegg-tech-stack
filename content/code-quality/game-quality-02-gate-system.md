## 一句总判断

质量门禁不是一道墙，而是一条流水线上的七道闸门。每道闸门只检查它该检查的东西，该拦的坚决拦住，该放的立刻放行。门禁的核心不是"严不严"，而是"准不准"——精准到团队愿意信任它，而不是绕过它。

## 这篇要回答什么

1. 从代码提交到线上发布，一共经过几道门
2. 每道门该检查什么
3. 哪些该阻断（block merge / build / release），哪些只需警告
4. 工具选型：每道门用什么工具实现

如果你的项目已经有 CI/CD 流水线，但总觉得"该拦的没拦住，不该拦的一直拦"，这篇的核心价值就在这里：帮你校准每道门禁的检查粒度和拦截策略。

---

## 门禁的核心原则

在拆解每道门之前，先定四条原则。这四条会贯穿后面每一道门的设计决策。

**越早拦住，成本越低。**

一个贴图格式错误在导入时被拦住，修复成本是 5 分钟。等它进了 AssetBundle、上了构建、被玩家加载后才发现内存超标，定位和修复的成本是几个小时甚至一次热更新。

**阻断要精准。**

误报是门禁的头号敌人。一旦门禁频繁误拦正常提交，团队的第一反应不是修复门禁，而是找方法绕过去。一道被绕过的门禁比没有门禁更危险——它给了一种"有人在管"的假象。

**警告要有跟踪。**

只弹一条 warning 但没有人跟进的告警等于没有。每条警告必须有归属人、有超时升级策略。包体大小增长了 3MB 是 warning，三天没有人认领就升级为 block。

**门禁本身需要维护。**

规则会过时。一年前设定的 Shader 变体数上限可能早已不适用当前的渲染管线。门禁需要定期审查，至少每季度一次。指定一个人作为门禁规则的 Owner，负责收集团队反馈、评估误报率、更新阈值。没人维护的门禁迟早变成摆设。

**门禁需要分层。**

不是所有检查都放在同一个环节。轻量的、能秒级完成的放在提交时；需要编译环境的放在 PR 或构建时；需要真机的放在冒烟测试。分层不仅是为了效率，也是为了让每道门的职责清晰，出了问题知道该去哪里找原因。

## Gate 1：提交时（Pre-commit / Pre-push）

这是最前面的一道门，成本最低，也最容易被忽视。

### 检查什么

- **代码格式。** 缩进、命名、花括号风格等。目的不是审美，而是让 diff 里只包含逻辑变化，不掺杂格式变化。格式变化越多，Code Review 的噪声越大。
- **敏感文件拦截。** `.env`、`credentials.json`、`.p12` 证书文件、含 API Key 的配置文件。一旦进入版本控制历史，即使后续删除也需要 git filter-branch 清理，成本极高。
- **大文件拦截。** 超过 10MB 的文件如果不走 Git LFS 直接提交，会让仓库体积快速膨胀，clone 速度越来越慢。

### 用什么工具

- **pre-commit framework**（Python 生态，跨语言支持好）：统一管理各语言的 hook，团队共享 `.pre-commit-config.yaml`。
- **husky + lint-staged**（Node 生态）：如果项目本身有 Node 工具链，husky 可以直接集成。
- **自定义 shell hook**：在 `.git/hooks/pre-commit` 中写检查脚本。简单项目够用，但难以版本化管理。

### 拦截策略

**全部 Block。**

理由：这些问题修复成本极低（改个格式、删个文件），但一旦漏过去，后续清理成本非线性增长。宁可让提交者多花 10 秒修复，也不要让问题进入 PR 环节。

### 注意事项

pre-commit hook 是本地执行的，开发者可以用 `--no-verify` 跳过。所以 Gate 1 不能作为唯一防线，它只是第一道过滤器。真正的兜底在 Gate 3（PR 合并检查）。

另外，hook 的执行速度很关键。如果 pre-commit 检查需要 30 秒以上，开发者的第一反应就是绕过。控制在 5 秒以内是基本要求。把耗时的检查（比如全量 lint）放到 pre-push 或 CI 阶段，pre-commit 只做最轻量的过滤。

## Gate 2：资源导入时（Asset Import）

游戏项目的一半质量问题不在代码里，在资源里。

### 检查什么

- **贴图尺寸和格式。** 非 POT（非 2 的幂次）贴图在某些平台无法压缩，导致内存翻倍。UI 贴图超过 2048 需要有明确理由。
- **贴图压缩格式。** Android 平台用 ASTC 或 ETC2，iOS 用 ASTC，PC 可以宽松一些。未压缩或用错格式的贴图是内存杀手。
- **模型面数和骨骼数。** 角色模型超过预算面数直接阻断。场景静态物件可以宽松一些但需要警告。
- **命名规范。** `tex_hero_body_d.png` 能一眼看出是角色身体的 diffuse 贴图。`texture_1_final_v2.png` 只会在三个月后让所有人困惑。
- **动画片段长度和关键帧密度。** 一个 idle 动画如果有 500 帧关键帧，要么是美术没有优化，要么是导出配置有问题。

### 用什么工具

- **Unity AssetPostprocessor。** 在资源导入时自动触发检查脚本。可以读取 TextureImporter、ModelImporter 的各项参数，不满足条件的直接 cancel 导入或修改设置。
- **自定义 Import Pipeline。** 对于更复杂的检查（如跨资源的依赖校验），可以在 Editor 脚本中注册 `AssetModificationProcessor`。
- **资源规范配置文件。** 把预算上限写在一个 ScriptableObject 或 JSON 配置里，而不是硬编码在检查脚本中。这样美术负责人可以按品类调整预算。

### 拦截策略

- **超预算：Block。** 贴图超过最大尺寸、模型超过面数上限、命名不合规——直接阻止导入完成。
- **接近预算：Warn。** 面数达到上限的 80%、贴图尺寸在 warning 区间——弹出警告但允许导入，同时记录日志。

### 注意事项

AssetPostprocessor 只在 Unity Editor 内生效。如果有人直接往仓库里提交原始资源文件绕过 Editor，Gate 3 的 PR 检查中需要再做一次服务端校验。

## Gate 3：PR 合并时（Pull Request）

这是整条门禁链中最关键的一道。它是代码和资源进入主分支前的最后防线。

### 检查什么

- **编译通过。** 看起来理所当然，但在多人并行开发的项目里，合并冲突导致的编译失败比想象中常见。
- **单元测试通过。** 所有已有的测试用例必须全部通过。新增代码如果修改了核心系统，需要有对应的测试覆盖。
- **资源依赖检查。** AB（AssetBundle）的依赖关系是否完整。新增了一个 prefab，它引用的材质、贴图、Shader 是否都在正确的 bundle 中。依赖断裂在打包阶段才会暴露，定位成本很高。
- **Shader 变体数量回归。** 一次 PR 新增了哪些 keyword，导致变体数从 1200 涨到 3800。变体爆炸是包体膨胀和运行时 Shader 编译卡顿的常见原因。
- **代码审查审批。** 至少一个有模块所有权的人 Approve。不是走形式，而是确保修改范围被相关人看过。

### 用什么工具

- **GitHub Actions / GitLab CI。** 在 PR 创建和更新时自动触发 CI 流水线，运行编译 + 测试。
- **Jenkins PR Builder。** 如果团队已有 Jenkins 基础设施，用 Generic Webhook Trigger 插件监听 PR 事件。
- **自定义 CI 脚本。** 资源依赖检查和 Shader 变体统计通常需要自己写。跑 Unity 的 `-batchmode` 调用 Editor 脚本，输出检查报告。
- **CODEOWNERS 文件。** GitHub/GitLab 原生支持。指定哪些路径的修改必须由哪些人审批。

### 拦截策略

**全部 Block。**

PR 是进入主分支的唯一入口。编译失败、测试失败、依赖断裂、审批不通过——任何一项不满足，PR 不可合入。

没有例外。如果有人说"这个很急，先合进去再修"，那说明流程有别的问题（比如分支策略不合理、紧急修复路径没打通），而不是门禁应该放水。

### 注意事项

PR 检查的运行时间直接影响团队的合并节奏。如果一次 PR 检查需要 40 分钟，开发者会倾向于攒多个改动一起提 PR，导致 PR 越来越大、Review 质量越来越低。

优化路径：编译缓存 + 增量检查 + 并行测试。目标是把 PR 检查控制在 15 分钟以内。对于特别大的项目，可以做分层检查——先跑轻量检查（编译 + 快速测试），通过后再异步跑重量检查（变体统计、AB 依赖），后者不阻断合入但结果同步到 PR 评论区。

## Gate 4：构建时（Build）

代码和资源合入主分支后，下一步是构建出可部署的产物。构建阶段是所有问题汇聚的地方——前面的门禁漏掉的，这里都会集中暴露。

### 检查什么

- **全量编译。** 不只是增量编译通过。定期（至少每日）做一次 clean build，避免增量编译掩盖的问题积累。增量编译有缓存，有些头文件变化不会触发重新编译，问题可能隐藏数天。
- **AB 打包 + 依赖校验。** 所有 AssetBundle 是否能正常打出来。bundle 之间的依赖关系是否符合预期。有没有出现重复资源打入多个 bundle。重复资源是包体膨胀的隐性杀手——同一张贴图被两个 bundle 各打了一份，内存里就存了两份。
- **Shader 变体编译 + 数量统计。** 统计本次构建的总变体数，与上一次发布版本对比。增量超过阈值需要 Review。特别关注 multi_compile 和 shader_feature 的组合爆炸。
- **包体大小检查。** 安装包 APK/IPA 的最终大小。首包大小和总资源大小分别统计。渠道方通常对首包大小有硬性要求（比如 Google Play 的 AAB 150MB 限制），超标意味着无法上架。
- **IL2CPP 产物校验。** 如果使用 IL2CPP 后端，检查转译后的 C++ 代码是否有未识别的泛型实例化缺失。这类问题只有在特定代码路径执行时才会崩溃，构建阶段不检查就会留到线上。

### 用什么工具

- **Jenkins Pipeline。** 用 Jenkinsfile 定义多阶段流水线：编译 → 打 AB → 出包 → 统计。每个阶段的产出作为下一阶段的输入。阶段间可以设置检查点，某个阶段失败后不继续执行。
- **自定义构建验证器。** 在 Unity build pipeline 的 `IPreprocessBuildWithReport` 和 `IPostprocessBuildWithReport` 中嵌入检查逻辑。
- **包体分析工具。** Unity 的 Build Report，或自建脚本解析 APK/IPA 内的文件分布，输出 Top 50 大文件列表。每次构建记录包体数据，绘制趋势图，涨了多少一目了然。
- **构建产物归档。** 每次构建的产物、日志、统计报告都归档到 Artifactory 或 S3。出了问题可以回溯任意一次构建的详细情况。

### 拦截策略

- **编译失败、AB 打包失败：Block。** 构建产物不完整，后续所有环节都没有意义。
- **包体增长超阈值：Warn。** 比如单次增长超过 5MB 发出警告，邮件通知负责人。连续三次警告未处理则升级为 Block。
- **Shader 变体数增长超阈值：Warn → 需 Review。** 增量超过 200 个变体自动 tag 负责人 Review。

### 注意事项

构建时间本身也是一个需要关注的指标。如果一次完整构建要 3 小时，那构建频率就只能是每天一次。构建慢会拖慢整条门禁链的反馈速度。可以通过构建缓存、分布式编译、增量 AB 打包等手段控制构建时间。

## Gate 5：构建后冒烟测试（Post-build Smoke）

包已经打出来了，但一个能打出来的包不代表能正常运行。

### 检查什么

- **启动不崩溃。** 安装 → 启动 → 到达登录界面，全程无 crash。听起来是最低标准，但每一次版本迭代都有可能打破这个最低标准。
- **核心流程可通行。** 登录 → 进主城 → 打开背包 → 进入战斗 → 退出。这条链路覆盖了场景加载、资源加载、UI 系统、网络通信、战斗系统五大模块。
- **截图对比。** 在固定场景的固定机位截图，与基线图做像素级或感知级对比。渲染异常、材质丢失、光照错误会被自动捕捉。
- **帧率基线对比。** 在标准场景跑 30 秒，记录平均帧率和 1% low。与上一个稳定版本对比，回退超过 10% 需要关注。

### 用什么工具

- **Unity Test Framework（PlayMode Tests）。** 在 Editor 或真机上跑自动化测试脚本，验证核心流程。
- **Appium / Airtest。** 跨平台 UI 自动化测试框架。通过图像识别或 UI 元素定位来操作游戏。
- **Device Farm。** AWS Device Farm、Firebase Test Lab，或自建的设备墙。确保在不同机型上都能通过冒烟。
- **截图对比工具。** 自建的截图 diff 脚本，或集成 Percy、Applitools 等视觉回归服务。

### 拦截策略

- **崩溃、核心流程不通：Block。** 包不能发出去。
- **帧率回退超过 10%：Warn。** 通知性能负责人。连续两个版本回退则 Block。
- **截图差异超过阈值：Warn。** 视觉差异可能是预期改动，也可能是 bug。需要人工确认。

### 注意事项

冒烟测试的关键是"快"。它不是完整的回归测试，而是一个 5-10 分钟的快速检查。如果冒烟测试要跑 2 小时，团队会选择跳过它。

## Gate 6：发布审批（Release）

技术门禁全部通过之后，还需要一道人工审批。

### 检查什么

- **发布单完整性。** 版本号、变更内容摘要、关联的 Jira/Issue 列表、回滚方案。缺任何一项不允许提交发布单。
- **审批人签字。** 至少两人审批：一个技术负责人确认技术风险可控，一个产品/运营负责人确认变更内容符合预期。
- **灰度比例设定。** 不允许一次全量发布。标准路径：1% → 10% → 50% → 100%。每个阶段观察一段时间，确认崩溃率和关键指标正常后再扩量。
- **冒烟测试报告。** Gate 5 的测试报告必须作为发布单的附件。没有报告的发布单不受理。

### 用什么工具

- **发布系统。** 自建的发布平台，或基于 Jenkins 的发布流水线。核心是固化流程、记录审批、支持灰度。
- **通知系统。** Slack / DingTalk / 飞书 webhook。发布单创建、审批、灰度扩量、全量发布、回滚——每个状态变化都发通知。
- **灰度分发。** Google Play Staged Rollout、App Store Phased Release，或自建的热更新灰度策略。

### 拦截策略

**审批不通过：Block。**

灰度期间如果崩溃率超过基线，自动暂停扩量并通知。

### 注意事项

发布审批看似是最"非技术"的一道门，但它往往是阻止事故的最后一个人工判断点。

常见的错误是把发布审批当走形式——审批人点一下"通过"，连变更内容都没看。解决方法不是加更多审批人，而是让发布单的内容结构化、关键信息醒目。版本号、影响范围、有没有数据库变更、有没有不可逆操作——这些信息必须在发布单模板里有固定字段，不是靠审批人自己去翻。

另一个常见问题是灰度流程流于形式。1% → 100% 之间每个阶段应该观察多久，每个阶段关注哪些指标，需要提前定义好。不是"观察一下没问题就扩量"，而是"崩溃率低于 X%、核心 API 错误率低于 Y%，保持 Z 小时后才允许扩量"。

## Gate 7：线上监控（Post-release）

发布不是终点，而是质量保障的最后一环。前面六道门拦不住的问题，这里是最后一次止损的机会。

### 检查什么

- **崩溃率突增。** 新版本发布后 1 小时内，崩溃率与前一版本对比。超过 0.5% 绝对值或 50% 相对增长，触发告警。
- **ANR 率。** Android 的 Application Not Responding 率。超过 0.3% 触发告警。
- **关键 API 错误率。** 登录、支付、匹配等核心接口的错误率。超过历史 P99 触发告警。
- **自动回滚触发条件。** 崩溃率超过 2%、核心 API 错误率超过 5%——自动触发回滚，不等人工决策。

### 用什么工具

- **崩溃收集。** Bugly（腾讯）、Firebase Crashlytics（Google）。接入 SDK 后自动采集崩溃堆栈、设备信息、用户操作路径。
- **监控面板。** Grafana + Prometheus / InfluxDB。自定义 dashboard 展示崩溃率、ANR 率、API 错误率的时间趋势。
- **告警系统。** PagerDuty、Opsgenie，或直接用 Grafana Alerting。告警规则与值班人直接绑定，不是发到一个群里等有人看到。
- **自动回滚。** 在灰度分发系统中配置回滚条件。满足条件后自动将灰度用户回退到前一个稳定版本。

### 拦截策略

- **超阈值：Alert → Auto-rollback。** 先告警，给 15 分钟窗口人工确认。如果 15 分钟内没有人确认"已知问题、不需要回滚"，自动执行回滚。

### 注意事项

自动回滚不是万能的。热更新的回滚相对简单（切换资源版本），但原生包的回滚意味着用户需要重新下载旧版本。所以灰度策略和 Gate 5 的冒烟测试才格外重要——尽量不要走到需要线上回滚的地步。

还有一个容易忽略的点：告警疲劳。如果线上监控每天弹 50 条告警，值班人员会开始忽略它们。告警规则需要精心调校——只有真正需要立刻处理的事件才触发告警，其他的降级为日报汇总。告警的价值不在于多，在于每一条出来，值班人员都知道"这条我必须看"。

## 门禁矩阵总表

| Gate | 触发时机 | 检查内容 | 工具 | 策略 |
|------|---------|---------|------|------|
| 1 | 代码提交时 | 格式、敏感文件、大文件 | pre-commit, husky | Block |
| 2 | 资源导入时 | 贴图/模型/动画规格 | AssetPostprocessor | Block / Warn |
| 3 | PR 合并时 | 编译、测试、依赖、变体、审批 | GitHub Actions, Jenkins | Block |
| 4 | 构建时 | 全量编译、AB、变体、包体 | Jenkins Pipeline | Block / Warn |
| 5 | 构建后 | 启动、核心流程、帧率、截图 | Unity Test, Device Farm | Block / Warn |
| 6 | 发布审批时 | 发布单、审批、灰度、报告 | 发布系统, 通知 | Block |
| 7 | 线上运行时 | 崩溃率、ANR、API 错误率 | Bugly, Grafana | Alert / Rollback |

读这张表的方式：**从左到右是一个问题被拦住的成本递增曲线。** Gate 1 拦住一个问题的成本是秒级，Gate 7 拦住的成本是小时级甚至需要一次紧急回滚。所以每道门存在的意义就是尽量不把问题留给下一道门。

## 常见的门禁反模式

在实际落地中，门禁体系最常见的失败不是"门不够多"，而是以下四种模式。

### 全部设为 Block

看起来最严格，实际效果最差。

每次提交都被拦住三五次，改完 lint 改 import，改完 import 改 naming。开发者的耐心是有限的，到第三天就会有人开始研究怎么绕过 hook。一旦养成了绕过的习惯，连真正该拦的问题也一起绕过去了。

正确做法：只有影响不可逆或修复成本高的问题才 Block，其余 Warn。

### 全部设为 Warn

看起来很友好，实际等于没有门禁。

100 条 warning 堆在 CI 报告的最底下，没有人看。美术同事导入了一张 4096 未压缩贴图，warning 了一下，然后这张贴图一路畅通无阻地到了线上玩家的手机内存里。

正确做法：Warn 必须有归属人和超时升级机制。

### 门禁规则一年不更新

一年前项目用的是 Built-in 渲染管线，Shader 变体上限设的 2000。现在切到 URP，正常的变体数量就有 3000。结果每次构建都被 Block，团队的解决方案是把阈值改成 999999——门禁彻底失效。

正确做法：每季度 Review 一次门禁规则，和当前项目阶段对齐。

### 没有 Metrics

门禁到底拦了多少问题？哪道门拦得最多？误报率多少？

如果这些数据都没有，就无法判断门禁是在帮忙还是在添乱。也无法向团队证明"这套门禁值得维护"。当领导问"这套 CI 系统值不值得续费"的时候，你需要拿出数据。

正确做法：每道门的拦截记录都入库。每月输出一份门禁健康报告：拦截数、误报数、绕过数。

### 门禁和人的关系处理不好

门禁不是用来惩罚人的。如果一个开发者的提交被 Gate 1 拦了三次，这不应该出现在他的绩效报告里。门禁拦住问题是正常工作流程的一部分，就像编译器报错一样——报错不是羞耻，忽略报错才是。

一旦门禁变成了"谁被拦的次数多谁就有问题"的考核工具，团队就会花大量时间在"怎么不触发门禁"上而不是"怎么写出好代码"上。这和门禁的初衷南辕北辙。

## 这篇真正想留下来的结论

门禁的终极目标不是"拦住所有问题"，而是"让问题在成本最低的阶段被发现"。

七道门不是七道墙。每道门的存在是为了让后面的门压力更小。Gate 1 多拦住一个格式问题，Gate 3 的 Review 就少一条噪声。Gate 5 多发现一个崩溃，Gate 7 就少一次线上回滚。

好的门禁体系有三个特征：

1. **团队信任它。** 不会习惯性绕过，因为它拦的都是真问题。
2. **它自己在演进。** 规则随项目阶段更新，不是一成不变。
3. **它有数据支撑。** 拦了多少、漏了多少、误报多少，都有数字。

如果你现在的项目还没有成体系的门禁，不需要一次性搭建七道门。先从 Gate 1（pre-commit）和 Gate 3（PR 检查）开始，这两道门的投入产出比最高。等团队习惯了门禁思维，再逐步向两端延伸——向前延伸到 Gate 2（资源导入校验），向后延伸到 Gate 5（冒烟测试）和 Gate 7（线上监控）。

最后一句话：门禁体系的设计标准不是"够不够严"，而是"团队愿不愿意用"。一套被团队信任、被持续维护、有数据支撑的门禁，比一套写在文档里但被集体绕过的"完美流程"，有价值一百倍。
