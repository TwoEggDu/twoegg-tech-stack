很多团队一谈自动化测试，就会下意识想把覆盖率先拉满。
但在游戏和客户端项目里，真正值钱的从来不是“测了多少”，而是“先把哪几类最贵的回归锁住”。

因为大多数质量问题并不平均分布。
有些问题一旦漏掉，后面每次改动都要继续付利息；有些问题 review 能看出来一半，但机器能更稳定地看另一半；还有些问题如果不在入口、构建和基线层拦住，最后就只能靠人肉回忆。

所以我更愿意把自动化验证理解成一种排序，而不是一种信仰：

`先把最值钱、最可重复、最适合机器做的那部分拎出来，再谈覆盖面。`

## 先回答结论

如果只能先做 5 类验证，我会按这个顺序排：

1. 配置表契约
2. 关键资源 smoke
3. 数值 / 规则回归
4. 构建产物校验
5. 基线阈值

这个顺序不是按“好不好写”排的，而是按三个标准排的：

- 规则是否明确
- 回归是否频繁
- 出问题后是否会持续付利息

只要同时满足这三点，就应该尽量自动化；不满足的，先交给 review 或阶段性人工验证。

## 1. 配置表契约

**本节职责：** 把“数据有没有写对”这件事从人脑里拿出来。

**对应层级：** 配置 / 契约层

配置表最容易被低估，因为它看起来像数据，实际上更像契约。
字段名、默认值、引用关系、枚举范围、表间依赖、平台差异，这些东西看起来都很小，但任意一处改动都可能在后面变成业务偏差、UI 异常、加载失败或兼容分支错误。

这类问题最适合自动化，不是因为它们高级，而是因为它们稳定、可枚举、可重复。
机器最擅长的，就是重复检查同一套规则：

- 必填字段有没有空
- 引用的资源有没有丢
- 枚举值有没有越界
- 主键 / ID 有没有重复
- 表间依赖有没有断
- 平台专属字段有没有和通用字段冲突

review 当然也能看配置，但 review 更容易看见“字段写没写”，不容易稳定看见“这条契约会不会在运行时把别的层一起拖坏”。
所以配置契约最适合做成导入时校验、构建前校验，或者直接做成 CI 门禁。

如果一个项目的配置已经开始多层复用，那配置契约就不再是可选项。
它本质上是在替团队记忆边界，而不是替团队写说明书。

## 2. 关键资源 Smoke

**本节职责：** 把“资源能不能真的被用起来”这件事提前拦住。

**对应层级：** 资源 / 启动链路层

资源系统最容易被骗过的地方，是“看起来加载成功”。
真正的问题往往要等到第一轮真实使用才出现：

- `Prefab` 不是 load 成功，而是 `Instantiate` 才暴露缺脚本、缺引用、层级异常
- 场景不是打开成功，而是进入真实流程后才暴露对象恢复问题
- `Shader` 不是编译通过，而是首帧、首个特效、首个关键路径才暴露缺 variant
- 远端资源不是拉下来了，而是切到真实入口后才暴露缓存和回滚问题

所以资源 smoke 不该只测“能不能加载”，而要测“第一次真用的时候会不会坏”。
这也是为什么它应该尽量贴近真实入口，而不是只跑一个脱离业务的空场景。

如果项目里已经有首屏、主场景、战斗入口、活动入口，那么 smoke 最应该覆盖的就是这些真实入口。
它不需要一开始就很重，但一定要先挡住“对象恢复链、脚本链、Shader 首载链、缓存链”这几类高频事故。

这类验证的价值很直接：

- 让问题在上线前暴露，而不是让玩家帮你发现
- 让回归点稳定，而不是靠人回忆上一次到底点坏了什么

## 3. 数值 / 规则回归

**本节职责：** 把“这次改动有没有悄悄改坏行为”固定下来。

**对应层级：** 纯逻辑 / 规则层

如果说配置和资源是“契约和入口”，那数值与规则回归就是“行为本身”。
技能伤害、冷却、掉落、成长、状态切换、判定逻辑、优先级决策，这些东西最适合被写成确定性的输入输出。

原因很简单：它们通常有明确输入、明确输出、明确边界。
只要规则没变，机器就能一直重复验证；一旦规则变了，测试就能第一时间告诉你：

- 是行为真的变了
- 还是你在重构时把旧真相弄丢了

这类回归最适合优先覆盖的，不是全量玩法，而是经常被改、最容易串链、最容易在版本压力下被顺手碰坏的部分。
例如：

- 技能公式
- Buff / 状态叠层
- 冷却和资源消耗
- 掉落和概率逻辑
- 关键判定和边界条件

这类测试的目标不是“覆盖率很好看”，而是让团队以后能放心改。
如果一段规则每次一动就会连带出一串隐性调用，那它就应该被测试钉住，而不是只靠 review 看一眼。

## 4. 构建产物校验

**本节职责：** 把“最后到底产出了什么”变成可见、可比对的事实。

**对应层级：** 构建 / 发布层

构建产物是最容易被忽略、但最值钱的一层。
因为它检验的不是源代码看起来对不对，而是最终出来的东西是不是我们想要的那个版本。

这层最适合自动化的检查包括：

- 该进包的资源有没有进
- 不该进包的资源有没有混进去
- 关键 bundle 的依赖有没有断
- `Manifest`、`BuildLayout`、`Catalog` 里有没有异常变化
- 关键 `Shader` / `Material` / built-in 资源有没有被错误重打
- 体积、重复、缺失、回退路径有没有异常

review 能看出一部分代码改动，但很难在同一个 diff 里稳定判断“最后产出来的包到底是不是我们想要的那个结构”。
所以构建产物校验的意义，不是补一个表面检查，而是把发布前最后一层硬事实变成门禁。

如果项目已经有资源交付、热更新、包体治理，这一层就更不能省。
它是把“代码看起来没问题”翻译成“产物真的没问题”的那道墙。

## 5. 基线阈值

**本节职责：** 把“退化了没有”变成量化判断。

**对应层级：** 性能 / 稳定性 / 成本层

基线阈值最容易被口头化。
大家都知道 FPS、内存、加载时间、首帧、崩溃率、包体大小重要，但如果不把它们量化成门槛，最后就会变成这种状态：

- 这次好像慢了一点，但不明显
- 包体大了一点，但还能发
- Crash 多了一点，但先观察

这些判断单独看都合理，叠在一起就会把系统慢慢拖坏。
基线的价值，不是追求绝对值永远不变，而是让变化有边界。
一旦超出边界，就应该有动作，而不是继续靠经验解释。

如果项目已经开始碰到版本压力、设备差异、资源膨胀和首载问题，基线阈值就不该再是“后面再说”。
它应该从一开始就和 smoke、构建校验一起进 CI。

最值得先立的基线通常是这些：

- 关键场景 FPS
- 加载时间
- 首帧时间
- 内存峰值
- 包体大小
- 崩溃率或关键错误率

## 为什么这 5 类排在前面

这 5 类之所以值得先自动化，不是因为它们最炫，而是因为它们最符合“机器比人更适合做”的条件：

- 规则清楚
- 重复频繁
- 回归代价高
- 结果可以确定
- 出问题后会持续付利息

反过来说，像“这次重构值不值得”“接口边界是不是最合理”“两个方案哪个更符合当前阶段”这种问题，不应该先变成测试。
这些更适合 review、设计判断和阶段性人工验证。

## 推荐落地顺序

如果从零开始，我会这样排：

1. 配置表契约
2. 关键资源 smoke
3. 数值 / 规则回归
4. 构建产物校验
5. 基线阈值

这个顺序的原因很简单：

- 前两类最容易把问题卡在入口
- 第三类最容易锁住行为
- 后两类最容易把“最后发出去的东西”管住

也就是说，先把“会立刻坏、会重复坏、会持续坏”的地方锁住，再逐步扩大范围。
不要一上来就追求全量自动化，那样通常只会把团队拖进一个维护成本更高的新系统里。

如果你要把这条线继续往前推，下一步最自然的方向就是：

- 哪些问题该交给 review
- 哪些问题该交给 AI 先扫一遍
- 哪些问题该继续硬拦在 CI 里

这三者分开以后，团队的质量体系才会真正开始稳定。
