很多团队已经有 review、有 CI、有一些测试，甚至也开始接 AI review 了，但发版时还是经常靠经验和临场判断。

问题通常不在于工具不够，而在于这些信号还没有真正变成 gate。

`真正的 Quality Gate，不是一张写得很完整的 checklist，而是一组会决定“版本能不能继续往前走”的阻断点。`

只要一个检查结果不会影响提交、合并、构建、发版或回滚，它就还只是观察，不是 gate。

## 先把 gate 的定义讲清楚

我现在判断一个质量检查算不算 gate，基本只看五件事：

- 它在哪个阶段执行
- 失败后是告警还是阻断
- 谁负责看结果
- 谁负责处理失败
- 在什么条件下允许临时放行

这五件事如果说不清，流程最后一定会退化成：

- 报告有人看，但没人负责
- CI 偶尔红，但照样合并
- 发版前才临时翻一遍检查项
- 出了线上问题以后，回头没人说得清哪道门应该先拦住它

所以 Quality Gate 的本质，不是“多加几个检查”，而是把检查和决策绑定起来。

## 提交前：把最便宜、最确定的规则前移

提交前最适合放的，是那些便宜、稳定、明确的规则：

- 格式化
- 编译
- lint
- 最快的静态检查
- 最轻量的契约校验

这一层的目标不是“验证整套系统”，而是尽量让明显错误不要进入 PR。

提交前 gate 有两个关键要求：

- 必须足够快，不能逼团队绕过它
- 必须足够确定，避免大量误报消耗作者耐心

如果一个检查既慢又不稳定，就不要硬塞在提交前。
它更适合放在 PR 后或 nightly，而不是把作者的每次提交都拖慢。

## PR 阶段：AI 提疑点，人来裁决，快 gate 硬拦

PR 阶段是最容易把东西堆得过重的地方。
但真正稳的做法，应该把三类职责分开：

- AI：输出第一轮风险列表
- reviewer：判断边界、取舍和例外
- CI：执行快而硬的门禁

AI 在这里最适合干的，不是做审批器，而是帮 reviewer 先把可能的风险挑出来，比如：

- 边界和依赖被打穿
- 规则重复散落
- 临时补丁混进长期层
- 接口语义或异常路径不清
- 改动缺少必要测试

reviewer 在这里真正要做的，是决定：

- 哪些 AI 疑点是真风险
- 哪些改动现在不能接受
- 哪些问题需要拆小、补验证或延期处理
- 哪些例外这次允许，但必须留追踪项

CI 在 PR 阶段最适合放的，则是快而硬的东西：

- 编译和基础静态检查
- 关键契约校验
- 最小 smoke
- 关键 regression

这一层最重要的一条原则是：

`PR gate 要快，但必须真拦。`

如果一个检查重要到应该阻断合并，它就不能只生成报告；
反过来，如果一个检查慢到团队天天等它，那它就不该继续待在 PR gate 里。

## 合并后：把重一点的构建和回归放到主干

有些检查确实重要，但它们不适合每个 PR 都同步等待。
比如：

- 更完整的资源和构建产物校验
- 更广的场景 smoke
- 跨平台构建
- 更重的 regression
- 更稳定的 baseline 采样

这些检查更适合放在合并后主干流程里执行。

这一层的关键不是“它不重要”，而是“它不适合阻塞每个 PR 的反馈速度”。
一旦它失败，团队要么自动回滚主干，要么在当天明确分诊，不要让主干长期带病。

## Nightly：把重型和趋势型检查放在这里

Nightly 的职责不是替代 PR，而是承接那些更重、更慢、更适合看趋势的检查。

最常见的包括：

- 多场景 smoke
- 更大规模的 regression
- 多设备或固定设备集的 baseline
- 包体、加载、性能的趋势报表
- 需要完整构建产物才能做的校验

Nightly 最值钱的地方，不是它比 PR 更“全面”，而是它能让团队看到趋势。
很多问题单次看不明显，但放进 nightly 连续比较就很清楚：

- 包体是不是在持续变大
- 加载时间是不是每周都在涨
- 某条关键链路是不是偶发失败开始变频繁

如果这些问题不在 nightly 里被看见，它们通常会一路滑到发版前，甚至滑到线上。

## 发版前：把“能不能发”变成硬门槛

发版前 gate 和 PR gate 最大的区别是：
它不再只是看“这次改动像不像可以合并”，而是看“这个版本到底能不能发”。

这一层更适合放的，是和 release 直接绑定的硬门槛：

- release build 是否成功
- 构建产物、符号文件、回滚包是否完整
- 关键 smoke、关键 regression 是否通过
- 包体、加载时间、性能、稳定性预算是否超线
- 监控、告警、回滚条件是否准备好

这一层最怕的，是把 release gate 做成一张“大家一起看看”的清单。
只要它不能真正阻断发版，它就还是观察，不是 gate。

## 上线后：没有监控回流，就没有闭环

很多团队把 Quality Gate 理解成发版前的事，但真正的闭环必须接到线上。

因为有一类风险，只有上线后才能真正看到：

- Crash / ANR
- 卡顿和加载长尾
- 真实设备和真实网络条件下的异常
- 灰度放量后的版本健康变化

所以更完整的流程应该是：

- 线上指标触发回滚或限流条件
- 线上异常回流到下一轮 gate 规则里
- 某条经常在线上才暴露的问题，被前移成 smoke、regression 或 baseline

如果线上问题只做事故复盘，不回写 gate，团队就会一直在同一种地方摔倒。

## 最常见的“假 gate”

看起来有 gate，实际上没有真正起作用，通常是下面几种：

### 1. 只有报告，没有阻断

CI 跑完生成一堆报告，但无论红绿都不影响合并和发版。
这种东西最多算观察面板，不算 gate。

### 2. 所有人都能习惯性点 rerun

如果一条 gate 经常不稳定，最后大家只会把它当成流程噪音。
真正的 gate 要么修到稳定，要么降级成告警，不要长期停在“习惯性重跑”。

### 3. 所有重检查都堆在 PR

这会让 PR 变得又慢又重，团队最后不是尊重 gate，而是想办法绕开 gate。

### 4. 发版前才第一次看关键预算

如果包体、加载、性能、稳定性预算只在发版前才看，通常已经来不及低成本修正。
这些东西应该在合并后和 nightly 就开始持续可见。

### 5. 线上问题不回写流程

如果一次线上事故发生后，流程没有增加新的 gate 或调整旧阈值，那这次事故实际上没有变成工程资产。

## 一条最低可执行版本

如果团队现在还没有完整的 gate，我会建议先搭这五层：

1. 提交前：格式化、编译、lint、最快契约检查。
2. PR：AI 风险扫描 + 人类 review + 快速 smoke / regression。
3. 合并后：完整构建和关键产物校验。
4. Nightly：更重的回归和 baseline 趋势。
5. 发版前 + 上线后：release gate、回滚准备和线上监控回流。

这一套的关键，不是每层都做很多，而是每层都知道自己在拦什么、失败后谁负责、什么情况下允许放行。

## 结尾

所以我现在更愿意把 Quality Gate 理解成一条连续流程：

`提交前拦便宜而确定的错误，PR 拦明显高风险改动，合并后和 nightly 观察更重的构建与趋势，发版前用硬门槛做最后裁决，上线后再把真实世界的失败回写成下一轮 gate。`

只有这样，自动化测试和 AI 评审才不是零散工具，而会真正变成团队交付流程里的一部分。
