发布不是质量链的终点，而是线上质量的起点。

很多团队在发版之前做了很多事情：Code Review、自动化测试、CI 门禁、灰度放量。但版本一旦过了全量，整个质量链就好像松了一口气。接下来会发生的，通常是这样：

- 崩溃率有看板，但没人每天盯
- ANR 有采集，但分类和归因做不了
- 卡顿有感知，但没有稳定的量化口径
- 内存有 OOM 上报，但不知道是哪里漏的

`这篇要回答的就是：线上质量应该监控哪四个维度，告警阈值怎样设计，发现问题之后的响应流程是什么，以及崩溃分析和灰度回滚怎样接成闭环。`

---

## 一、线上质量的 4 个监控维度

线上质量不是一堆指标的堆叠，而是四个维度各自回答一个不同的问题。

### 1. 崩溃率（Crash Rate）

崩溃率回答的是最基本的问题：

`这个版本还活着吗？`

#### 工具选型

国内项目通常选 Bugly，海外项目通常选 Firebase Crashlytics，跨平台或自建体系的团队也有用 Sentry 的。三者核心能力相似，差异主要在生态和数据归属：

| 工具 | 适用场景 | 关键能力 | 常见限制 |
|------|----------|----------|----------|
| Bugly | 国内 Android / iOS | 国内网络友好、微信生态打通 | 聚类粒度有限，自定义维度扩展弱 |
| Crashlytics | 海外、Google 生态 | 和 Firebase 联动好，实时看板 | 国内上报延迟，自定义上下文有限 |
| Sentry | 跨平台、自建体系 | 自定义能力强，支持自托管 | 需要自己维护符号化和聚类逻辑 |

不管选哪个，都必须做好一件事：

`构建流水线结束时，把符号文件、mapping 和构建元信息一起归档上传。`

符号化不完整的 crash 看板，和没有看板一样。

#### 配置要点

- 每次构建自动上传 dSYM / mapping / symbols
- crash 事件必须带包版本、资源快照版本、热更版本
- 自定义 key 至少包含：设备档位、渠道、区域、登录状态
- 上报采样率在灰度阶段建议 100%，全量后按量级决定

### 2. ANR 率

ANR 回答的是另一类问题：

`这个版本虽然没崩，但用户的操作是不是被冻住了？`

ANR 比 crash 更难处理，因为它的原因分布更广，而且不同原因的处理姿势完全不同。

#### ANR 的三大分类

| 类型 | 典型表现 | 常见原因 |
|------|----------|----------|
| 主线程阻塞 | UI 无响应超过 5 秒 | 同步 IO、重计算、锁等待 |
| 死锁 | 界面完全冻结 | 多线程锁顺序不一致 |
| IO 阻塞 | 启动或场景切换卡住 | 磁盘读写、网络同步调用、资源加载 |

Android 的 ANR 采集本身系统有支持，但系统提供的 traces 往往不够定位根因。更稳的做法是在采集时额外带上：

- 当前场景和入口标记
- 主线程最近 N 帧的调用采样
- 当时的内存水位和 IO 状态

`ANR 的难点从来不是"知道有 ANR"，而是"知道它为什么 ANR"。`

### 3. 卡顿 / 帧率

卡顿回答的是体验层面的问题：

`这个版本的流畅度有没有变差？`

卡顿和 crash 不同，它没有一个系统级的"发生了就上报"机制。必须自己定义采集口径。

#### 自定义帧率采集

更实际的做法是在客户端埋一个帧率采样模块：

- 每帧记录 deltaTime
- 连续 3 帧 > 33ms（即帧率低于 30fps）标记为一次卡顿事件
- 单帧 > 100ms 标记为一次冻帧事件
- 按场景、入口、设备档位聚合上报

这条规则不是唯一标准，但它比"平均帧率"有用得多。平均帧率会被大量正常帧稀释，而卡顿事件是用户真正能感知到的。

#### 卡顿阈值建议

| 级别 | 定义 | 建议阈值 |
|------|------|----------|
| 轻微卡顿 | 连续 3 帧 > 33ms | 上报，不告警 |
| 明显卡顿 | 连续 5 帧 > 50ms | 告警，P2 |
| 冻帧 | 单帧 > 100ms | 告警，P1 |
| 严重冻帧 | 单帧 > 500ms | 告警，P0 |

`卡顿监控最怕的是噪声，所以必须绑定场景、设备组和版本区间，否则很快就退化成"好像最近不太流畅"。`

### 4. 内存泄漏

内存监控回答的是长期健康问题：

`这个版本会不会越跑越吃内存，最后被系统杀掉？`

#### 内存水位线监控

不要等 OOM 发生了才去查。更稳的做法是在运行时持续采样内存水位：

- 每 30 秒采样一次 PSS（Android）或 Memory Footprint（iOS）
- 当内存超过设备物理内存的 70% 时，标记为黄色预警
- 当内存超过 85% 时，标记为红色预警并上报
- OOM 发生时，把最近 N 次采样的内存曲线和当时的场景栈一起上报

#### OOM 预警

真正有用的 OOM 预防不是等崩溃上报，而是：

- 在内存水位线持续攀升时就发出预警
- 把内存曲线和场景切换时间线叠在一起看
- 重点关注"场景退出后内存没有回落"的模式

`内存泄漏最可怕的不是一次 OOM，而是每次场景切换都漏一点，直到系统忍不住为止。`

## 二、告警阈值设计

有了监控维度还不够，还要回答一个关键问题：什么情况下应该告警，以及告警了之后是什么级别。

告警阈值如果设得太松，事故发生了才知道；设得太紧，团队每天被噪声淹没。所以阈值设计必须同时有三种维度。

### 1. 绝对阈值

绝对阈值回答的是"这个值本身是否已经不可接受"。

| 指标 | P0 阈值 | P1 阈值 | P2 阈值 |
|------|---------|---------|---------|
| 崩溃率 | > 0.5% | > 0.3% | > 0.1% |
| ANR 率 | > 0.3% | > 0.15% | > 0.05% |
| 启动失败率 | > 1% | > 0.5% | > 0.1% |
| OOM 率 | > 0.2% | > 0.1% | > 0.05% |

这些数字不是固定标准，每个项目需要根据自己的历史数据来定。但必须有一组明确的数字写在规则里，而不是每次靠人感觉。

### 2. 相对阈值

相对阈值回答的是"这次版本有没有比上次明显变差"。

- 崩溃率较上一版本上升 50% → P1
- ANR 率较上一版本上升 100% → P1
- 卡顿率较上一版本上升 30% → P2
- 内存峰值较上一版本上升 20% → P2

相对阈值的价值在于，即使绝对值还在安全区间，趋势恶化本身就是信号。

### 3. 分维度阈值

分维度阈值回答的是"有没有某个局部已经失控"。

- 新增 crash signature > 100 次/小时 → P0
- 某个机型崩溃率 > 2% → P1
- 某个渠道 ANR 率突然翻倍 → P1
- 某个场景卡顿率 > 5% → P2

`分维度阈值是最容易被忽略的一层。很多时候总体指标看起来还行，但某个渠道、某个设备组、某个场景已经烂得不行了。`

## 三、告警响应流程

告警发出来只是第一步，真正决定事故损失大小的是响应流程。

### P0：1 小时内响应，4 小时内决定回滚或 hotfix

P0 意味着版本可用性已经受到威胁。

- 15 分钟内：值班 owner 确认告警是否为真
- 30 分钟内：拉起事故响应群，启动分诊
- 1 小时内：给出初步判断——回滚、hotfix 还是降级
- 4 小时内：执行决定并验证效果

`P0 的响应速度不取决于团队多努力，而取决于回滚预案是不是早就准备好了。`

### P1：当天响应，下一版本修复

P1 意味着问题明确存在，正在影响部分用户，但还没到需要立即回滚的程度。

- 当天内确认问题根因
- 评估是否需要紧急 hotfix 还是排入下一版本
- 如果排入下一版本，必须写入该版本的发布前 checklist

### P2：跟踪观察，排入迭代

P2 意味着趋势异常但尚未到阻断程度。

- 建立追踪项，持续观察
- 如果连续两个版本 P2 不收敛，自动升级为 P1
- 纳入常规迭代排期

## 四、崩溃分析流程

告警让你知道出问题了，但真正要解决问题，还需要一条稳定的分析流程。

### 符号化 → 聚类 → Top N → 根因 → 修复 → 验证

这六步是崩溃分析的标准链路：

**第一步：符号化。** 把原始堆栈还原成可读的函数名和行号。这一步的前提是构建流水线在每次出包时都把符号文件上传到了 crash 平台。

**第二步：聚类。** 把相似堆栈的 crash 事件归成同一个 issue。聚类质量直接决定后面的优先级判断是否准确。

**第三步：Top N 排序。** 按影响用户数排序，而不是按 crash 次数。一个用户反复崩 100 次和 100 个用户各崩 1 次，后者优先级更高。

**第四步：根因分析。** 拿到 Top N 之后，结合版本上下文、设备维度和场景信息定位根因。

**第五步：修复。** 修复可以是 hotfix、下一版本修复或服务端降级。

**第六步：验证。** 修复发布后，在 crash 看板上确认对应 issue 的发生率是否回落到预期水位。

`如果这六步里任何一步断了，崩溃分析就会退化成"知道有问题但不知道怎么收"。`

## 五、灰度发布与自动回滚

线上质量监控和灰度发布是一对必须绑在一起的机制。监控发现问题，灰度决定止损。

### 灰度比例

| 阶段 | 比例 | 观察期 | 核心观察指标 |
|------|------|--------|-------------|
| 第 1 阶段 | 1% | 2-4 小时 | 崩溃率、启动失败率、致命 ANR |
| 第 2 阶段 | 10% | 12-24 小时 | 全部 P0 指标 + 卡顿率 + 内存水位 |
| 第 3 阶段 | 50% | 12-24 小时 | 全部指标 + 业务关键路径 |
| 第 4 阶段 | 100% | 持续监控 | 转入常规线上监控 |

### 每阶段的自动回滚触发条件

自动回滚不是"有人觉得不对就点按钮"，而是预先写好的规则：

- 第 1 阶段：崩溃率 > 1% 或启动失败率 > 2% → 自动回滚
- 第 2 阶段：崩溃率 > 0.5% 或任一 P0 指标超线 → 自动暂停放量，人工介入
- 第 3 阶段：崩溃率 > 0.3% 或 ANR 率较上一版本恶化 100% → 暂停放量

`自动回滚的前提是回滚版本早就准备好了。如果触发回滚的时候才开始找上一个稳定版本，那回滚机制形同虚设。`

回滚必须提前准备：

- 上一个稳定构建产物和符号文件已归档
- 热更配置可以切回
- 服务端兼容位已保留
- 回滚后的监控看板和告警规则已就绪

## 六、线上质量 Dashboard

### 该看哪些指标

一个有效的线上质量 Dashboard 应该回答三个问题：

1. **这个版本是否健康？** 看崩溃率、ANR 率、启动成功率、OOM 率。
2. **和上个版本比有没有变差？** 看版本对比趋势图。
3. **如果变差了，集中在哪里？** 看设备、渠道、场景维度的切分视图。

### 不该看哪些指标

- **不该看平均帧率。** 被大量正常帧稀释，看不出真实卡顿。
- **不该看总 crash 次数。** 一个用户反复崩会严重扭曲数字，应该看 crash-free user rate。
- **不该看不分版本的总图。** 新旧版本混在一起，看不出新版本到底是好是坏。
- **不该看太多业务指标。** Dashboard 是质量看板，不是运营看板。业务指标另开一套。

`Dashboard 最重要的设计原则是：打开 10 秒内能判断版本是否健康。如果需要翻三层才能看到关键数字，这个 Dashboard 就已经失败了。`

## 七、结论

线上质量不是发版之后的附属动作，而是整条质量链的最后一环和第一个反馈源。

如果把线上质量压成一条链，它应该是这样的：

`四个维度持续采集（崩溃、ANR、卡顿、内存），三种阈值交叉告警（绝对、相对、分维度），分级响应流程明确到人和时限，崩溃分析链路从符号化一直走到验证回落，灰度发布和自动回滚绑死在监控指标上，Dashboard 让团队 10 秒内看到版本健康。`

前面六篇讲的是怎样在发布之前把质量拦住。这篇讲的是，拦不住的那部分到了线上之后，怎样最快发现、最快止损、最快收敛。

两头接起来，才是一条完整的质量链。
