春节活动上线前一天，数值策划在掉落表里加了一行新的节日道具。导出、提交、合入主分支、构建、发版，全流程一路绿灯。

上线 40 分钟后，运营看到后台数据不对：某个稀有坐骑的产出量已经超过了预期全服一周总量的 20 倍。排查发现，策划在掉落表里新增那行时，概率列填的是 `10000`（万分比格式下代表 100%），而不是预期的 `50`（代表 0.5%）。也就是说，每个玩家打一次副本必定掉落一个本该千分之五概率才出的稀有坐骑。

40 分钟里，全服产出了超过 3 万个稀有坐骑，黑市价格从 200 金币跌到 2 金币，连带影响了绑定的成就系统和排行榜。最终停服回档 2 小时，补偿方案又花了运营团队两天。

事后复盘，锅的第一轮指向是"策划填错了数"。但真正的问题是：从 Excel 到线上，没有任何一层校验拦住了"一个概率值等于 100%"这件事。导出工具没有范围检查，CI 没有跑掉落模拟，构建流程不看数值合理性，运行时也没有对异常概率做防御。

`配表质量要解决的核心问题不是"策划会不会犯错"，而是"犯了错以后，哪一层能拦住它"。`

这篇要回答的就是这件事：从 Excel 到运行时，校验链应该有几层，每层该拦什么，以及怎样把这条链接进 CI 和发布流程。

## 策划配表出问题的 5 种常见方式

配表问题看起来五花八门，但归纳起来最高频的就是这五类。

### 1. 空引用

最典型的场景：物品表里配了一个 `icon_id = 90021`，但图标资源表里根本不存在 `90021` 这条记录。

运行时读到这行数据去查图标，返回 null，UI 层没做防御，直接 NullReferenceException。

这类问题的危险在于，它在 Excel 里看起来完全正常，只有到了运行时才会炸。

### 2. 类型错误

策划在 Excel 里手动输入了一个本该是整数的字段，但多打了一个空格或写成了中文逗号。

导出工具做了类型转换，静默地把它变成了 0 或默认值，没有报错。结果进了线上，某个道具的价格变成了 0，被玩家批量购买。

### 3. 公式溢出

数值策划调整了一个伤害公式的系数，从 `1.5` 改成了 `15`。

本地测试时没打满级怪物，看起来一切正常。上线后玩家用满级角色一刀打出了 int32 溢出的伤害值，伤害显示变成负数，后端扣血逻辑崩溃。

### 4. 版本覆盖

两个策划同时在改同一张 Excel 表。策划 A 改了第 30 行，策划 B 改了第 50 行。策划 B 后提交，直接覆盖了策划 A 的修改，但没有任何冲突提示。

等到 A 的改动被发现丢了，已经过了两个版本。

### 5. 逻辑矛盾

活动配表里，某个任务的完成条件是"击杀 Boss 3 次"，但对应的副本表里这个 Boss 每天只能打 1 次。

策划各自看自己的表都没问题，但两张表放在一起就产生了不可能完成的任务。

这五类问题有一个共同特征：在数据被填写的那一刻，没有任何系统告诉策划"这里有问题"。

## 校验链应该有 4 层

一个成熟的配表管线，不该把所有检查都堆到某一个环节。更稳的做法是分成 4 层，每层拦不同类型的问题。

### 第一层：Excel / 编辑器端

这是离策划最近的一层，也是反馈最快的一层。

适合在这层做的：

- 单元格类型约束（枚举下拉、整数范围、字符串长度）
- 同表内引用合法性（外键字段在被引用表中是否存在）
- 明显的空值和格式问题
- 命名规则和 ID 规则检查

如果团队用的是自定义配表编辑器而不是原始 Excel，这层可以做得更重，比如填写时实时校验引用完整性。

这层的价值在于：策划填完数据按保存的瞬间就知道哪里不对，不需要等到导出甚至构建才发现。

### 第二层：导出时

从 Excel 或编辑器导出为运行时格式（json / binary / scriptableobject）时做的检查。

适合在这层做的：

- 严格类型校验（string 能不能转成目标类型，数值是否在合法范围内）
- 跨表引用完整性（所有外键引用的目标记录是否都存在）
- 唯一性约束（主键是否重复）
- 必填字段是否为空
- 枚举值是否在合法集合内

这层最重要的原则是：**导出失败必须阻断**，而不是打个 warning 继续跑。

如果导出工具只打 warning 不阻断，策划很快就会习惯忽略满屏的黄色日志。

### 第三层：CI 时

数据提交到版本库后，CI 流水线里做的自动化校验。

适合在这层做的：

- 全量跨表引用完整性（不只是改动的表，而是所有表之间的引用关系）
- 公式回归测试（后面单独展开）
- 数据量变化监控（某张表突然多了 1000 行或少了 500 行）
- 与上一个稳定版本的 diff 报告
- 已知高危规则的自动扫描

这层的价值在于：就算策划本地的编辑器和导出环节都漏了，CI 依然能兜住。

### 第四层：运行时

客户端或服务端加载配表后的防御性检查。

适合在这层做的：

- 关键数据非空断言
- 引用查找失败时的降级逻辑而不是直接崩溃
- 异常数值的上下界兜底（攻击力不能为负数，价格不能为 0）
- 配表版本号和客户端版本号匹配检查

运行时校验不是替代前面三层的理由。它是最后一道保险。如果所有问题都指望运行时兜底，成本会远高于在前面三层拦住。

## 每层的具体检查项

把四层的检查项收成一张表：

| 层级 | 典型检查项 | 工具 / 手段 | 阻断策略 |
|------|-----------|------------|---------|
| 编辑器端 | 类型约束、同表引用、枚举、ID 规则 | 自定义编辑器插件 / Excel 数据验证 | 即时提示 |
| 导出时 | 严格类型转换、跨表外键、唯一性、必填 | 导出工具内置校验器 | 导出失败阻断 |
| CI | 全量引用、公式回归、数据量变化、diff | CI 脚本 + 自定义校验框架 | 合并阻断或告警 |
| 运行时 | 非空断言、范围兜底、版本匹配、降级 | 代码内防御 | 日志上报 + 降级 |

这四层不是选其中一层做好就够了。它们拦的问题类型不一样，反馈时机不一样，目标受众也不一样。

## 公式回归：改了攻击力公式以后怎么知道没有一刀秒杀

数值公式是配表里最容易出事的部分，因为它的影响面通常远大于改一个字段。

一个攻击力系数从 `1.5` 改成 `2.0`，表面上只改了一个数，但它会影响所有使用这个公式的角色、所有等级、所有装备组合下的伤害输出。

单靠人工审查根本覆盖不了这个组合空间。

更稳的做法是建一组公式回归测试：

### 1. 定义关键公式的测试用例集

每个核心公式对应一组固定的输入和预期输出。例如：

- 1 级角色 + 基础武器 vs 1 级怪物，预期伤害在 `[50, 80]` 区间
- 60 级角色 + 满级武器 vs 60 级怪物，预期伤害在 `[800, 1200]` 区间
- 60 级角色 + 满级武器 vs 1 级怪物，预期伤害不超过 `99999`（防溢出）

### 2. 在 CI 中自动运行

每次数值表有改动时，CI 自动跑这组测试用例，把当前公式的输出和预期区间做对比。

### 3. 超出预期区间时阻断

如果某个用例的输出超出了预期区间，CI 直接报红。

策划可以选择更新预期区间（说明这是有意调整），但不能让一个不在预期内的数值改动静默地合进主干。

公式回归测试的维护成本确实不低。但它覆盖的恰恰是人工审查最容易漏掉的场景：极端等级组合和边界条件。

## 版本控制：多人同时改同一张表怎么处理

配表版本控制是每个团队都会遇到的痛点。

Excel 是二进制格式，Git 无法做行级 diff 和 merge。两个人改了同一个 `.xlsx` 文件，Git 只会告诉你"有冲突"，但无法帮你合并。

常见的几种处理方式：

### 1. 导出为文本格式后入库

不直接把 `.xlsx` 入库，而是在提交前先导出为 csv / json / yaml，用文本格式做 diff 和 merge。

好处是 Git 能做行级对比，code review 时能直接看到哪行改了什么。

坏处是需要约定清楚"以哪个格式为准"。如果 Excel 和文本格式同时存在，很容易出现"改了 Excel 没导出"或者"改了文本格式没同步回 Excel"的问题。

### 2. 表级锁

在配表编辑器里加锁机制：策划 A 正在编辑"物品表"时，策划 B 无法同时编辑这张表。

好处是彻底避免了同时编辑产生的冲突。

坏处是粒度太粗。一张大表几百行，两个人改的可能是完全不相关的行，但也被锁住了。

### 3. 行级分片

把大表拆成多个分片，每个分片对应一个独立文件。不同策划负责不同分片，物理上就避免了冲突。

好处是冲突概率大幅降低。

坏处是增加了维护成本，跨分片的引用关系需要额外验证。

没有一种方案是完美的。但有一个原则是确定的：如果团队还在用"最后提交的人覆盖前面所有人"的方式管理配表，那它迟早会丢数据。

### 4. 变更日志

不管用哪种版本控制方式，每次配表改动都应该有明确的变更记录：谁改了哪张表的哪些行，改动原因是什么。

这不只是为了追责，更是为了排查。线上出了数值异常，如果没有变更日志，排查的第一步就是在几十张表里人工 diff，效率极低。

更好的做法是让 CI 自动生成每次配表提交的 diff 摘要，附在 PR 描述里。Review 的人不需要逐行看 Excel，直接看摘要就能判断这次改动的范围和影响。

## 防崩链：从校验器到灰度的完整闭环

把前面讲的所有环节串起来，一条完整的配表防崩链应该是这样的：

### 1. 策划填数据

编辑器端实时校验，填错了立刻有红色提示。

### 2. 策划导出

导出工具做严格类型检查和跨表引用检查。有错误时导出失败，不生成产物。

### 3. 策划提交

Git 提交触发 pre-commit hook，做最基本的格式检查和命名规则检查。

### 4. CI 校验

PR 合入时 CI 跑全量校验：跨表引用、公式回归、数据量变化、diff 报告。有阻断级问题时 PR 不能合入。

### 5. 构建验证

构建流水线中加载配表数据做一次冒烟：关键表能不能正常反序列化，关键引用能不能查到，关键公式输出是否在合理范围内。

### 6. 灰度验证

配表改动跟随版本灰度发布，先放小比例用户，观察关键指标（崩溃率、异常日志、数值指标）。确认无异常后再全量推送。

这六步不需要每个团队都做到满配，但至少应该覆盖到第四步。

因为如果连 CI 校验都没有，配表改动合进主干的唯一门禁就是人工 review。而人工 review 配表数据，漏检率极高。

## 反模式

### 1. 校验规则写死在导出工具代码里

规则和代码耦合在一起，策划看不到、改不了、也理解不了当前到底有哪些校验规则。

每次需要加新规则都要找程序改代码、重新编译工具。结果要么规则加得太慢，要么策划绕过工具直接手动改导出文件。

更好的做法是把校验规则做成配置文件或规则表，策划和程序都能看到当前有哪些规则，修改规则不需要改代码。

### 2. 只检查格式，不检查逻辑

导出工具检查了类型、非空、枚举值范围，但完全不检查业务逻辑。

比如：一个技能的冷却时间配成了 0 秒、一个商店物品的折扣配成了 200%、一个关卡的怪物数量配成了负数。这些格式上都合法，但逻辑上是错的。

格式校验是基础，但光有格式校验远远不够。

### 3. 校验规则和策划不同步

程序加了 20 条校验规则，但策划不知道有这些规则，也不知道规则的具体阈值。

结果策划提交数据后不断被 CI 打回，但不知道为什么被打回。策划开始觉得校验系统是"找茬工具"，想办法绕过它。

校验规则必须对策划透明。每条规则都应该有清晰的说明、合理的阈值和可理解的错误提示。

### 4. 所有校验都只在 CI 跑

策划改完数据、提交、等 CI、被打回、修改、再提交、再等 CI。

一个简单的类型错误要经过 15 分钟的 CI 流程才能发现。策划的体验极差，效率极低。

能在编辑器端和导出时拦住的问题，就不要等到 CI。CI 应该是兜底，不是唯一防线。

### 5. 没有 diff 报告

每次配表改动合入主干，没有人能快速看到"这次到底改了哪些表的哪些行"。

等到出了问题要排查时，只能一行一行比对。

CI 产出的 diff 报告是低成本但高收益的投入。

### 6. 校验环境和线上环境不一致

校验器跑在 CI 的 Python 环境里，用的是 CI 上的导出工具版本。但实际上线的数据，经过了另一套打包流程，两边的解析逻辑有微小差异。

结果 CI 校验通过了，但线上解析时还是崩了。

校验器用的导出和解析逻辑，必须和运行时一致。最好是共用同一份代码，而不是各自实现一遍。

## 一条最小可执行路径

如果团队现在的配表管线还没有系统化的校验，我会建议先按这个顺序落：

1. 先把导出工具的校验从 warning 改成 error，类型错误和空引用必须阻断导出。
2. 先把跨表引用完整性检查接进 CI，PR 合入时自动跑。
3. 先给最核心的 3 到 5 个数值公式写回归测试用例。
4. 先统一配表的提交格式（文本格式入库，或者至少有稳定的 diff 手段）。
5. 先让 CI 产出每次配表改动的 diff 报告。

这五步站住以后，再逐步扩展：加编辑器端实时校验、加构建冒烟、加灰度验证。

不要一开始就试图把所有规则都配齐。规则太多、阈值太死，策划会绕过它；规则太少、反馈太慢，策划会忽略它。

## 结论

所以我现在更愿意把配表质量这件事理解成：

`它不是"策划小心一点就不会出问题"，而是从编辑器、导出、CI 到运行时，每一层都有明确的校验规则、阻断策略和反馈机制；只有当类型检查、引用完整性、公式回归、版本控制和冒烟验证一起进入流程，配表才算从"靠人不犯错"变成"靠系统拦错误"。`

如果连导出都不阻断、CI 都不校验，那所有"策划要仔细"的提醒最后都只是心理安慰。
