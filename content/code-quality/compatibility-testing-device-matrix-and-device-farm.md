兼容性测试最常见的误区有两个方向：

- 一种是"机子越多越好"
- 另一种是"线上有问题再说"

前者通常会把测试预算打散，后者则会把问题推到玩家手里。真正能落地的兼容性策略，既不是"设备海"，也不是"完全靠线上监控补漏"，而是：

`先用设备矩阵把预算花在最值得测的风险面上，再把真机、Device Farm 和线上分维监控各自放回合适位置。`

这篇真正要回答的，不是"到底要买多少台设备"，而是下面三件事：

- 哪些设备和环境必须先覆盖
- 哪些检查适合放进 Device Farm
- 驱动和平台差异应该怎样被稳定看见

## 兼容性测试到底在解决什么

兼容性问题的难点，不在于它"偶尔出现"，而在于它常常只会出现在某个具体维度交叉上：

- 某个 GPU 家族 + 某个 OS 版本
- 某个低内存设备 + 首次资源下载
- 某个渠道包 + 某个系统权限状态
- 某个驱动版本 + 某条渲染路径

这也是为什么兼容性问题最怕被笼统描述成"安卓上偶现"或者"部分机型有问题"。因为这样的描述几乎没法驱动动作。

所以兼容性测试真正解决的是：

`怎样在有限预算下，把最有可能出问题、而且一旦出问题损失最大的设备 / 平台 / 驱动组合优先暴露出来。`

它和 [Baseline：性能 / 包体 / 加载 / Crash 预算怎样立线并进 CI]({{< relref "code-quality/baseline-budgets-in-ci.md" >}}) 以及 [线上质量监控：崩溃率 / ANR / 卡顿的实时监控告警、版本健康看板、问题响应 SLA]({{< relref "code-quality/online-quality-monitoring-and-version-health-dashboard.md" >}}) 是连着的：一个负责定义实验室里的最低健康线，一个负责在真实世界里按版本分维看异常，而兼容性策略负责回答"实验室里到底先覆盖哪些差异面"。

## 设备矩阵不是清单，而是预算分配

如果把设备矩阵理解成"越长越好的一张列表"，这件事就很容易失控。更稳的做法是先把矩阵分层。

一个最小够用的矩阵，通常至少分三层：

| 层级 | 怎么选 | 适合做什么 |
|------|--------|------------|
| 核心矩阵 | 核心收入机型、核心市场机型、主流 GPU 家族、当前主 OS 版本 | 每轮版本都要覆盖的关键路径 |
| 扩展矩阵 | 新系统版本、次主流设备、低端和极端配置、特殊渠道 | Release 前重点补洞 |
| 长尾观察矩阵 | 低占比但风险明显的设备 / 驱动 / 渠道组合 | 借助 Farm 和线上监控持续补样本 |

这三层真正解决的是预算分配问题：

- 哪些必须每次真机覆盖
- 哪些可以按版本风险抽样
- 哪些更适合借助云真机和线上数据去看

所以设备矩阵的本质不是"列出所有设备"，而是：

`把有限测试资源稳定花在最值得测的交叉点上。`

## 优先级不是按感觉排，而是按风险面排

一个设备要不要进核心矩阵，不应该只看销量。更稳的判断通常会同时看下面几类信号：

- 用户量 / 收入占比
- GPU / SoC 家族是否有代表性
- OS 主版本是否正在放量
- 内存、存储、分辨率是否代表高风险边界
- 渠道、商店、地区和网络环境是否特殊
- 过去线上是否反复在这类设备上出问题

也就是说，优先级不是"最贵的先测"，也不是"销量最高的先测"，而是：

`用户影响面 × 出问题概率 × 一旦出问题的修复成本。`

低端机为什么经常值得进核心矩阵，不是因为它"看起来弱"，而是因为：

- 它更容易暴露内存和加载边界
- 它更容易把性能和卡顿问题提前放大
- 它一旦出问题，常常不是轻微退化，而是直接不可用

## Device Farm 适合做什么，不适合做什么

`Device Farm` 最大的价值，不是替代真机，而是把"扩矩阵"和"固定 smoke"这两件事规模化。

像 `AWS Device Farm`、`Firebase Test Lab` 这类平台，最适合承接的是：

- 安装、启动、登录、进入主界面这类固定入口 smoke
- 多 OS / 多机型的权限、安装、升级、冷启动覆盖
- 固定脚本下的关键路径回归
- 大规模基础兼容冒烟

但它们并不适合代替下面这些事：

- 长时间热稳定和真实发热观察
- 细腻交互手感、触控边界、复杂网络抖动
- 高精度性能 baseline
- 很依赖本地抓帧、驱动调试和长时观察的渲染问题

所以更稳的分工应该是：

- `Device Farm` 负责把广覆盖的基础兼容 smoke 跑起来
- 核心真机负责性能、视觉、交互和高风险入口
- 线上监控负责补实验室永远覆盖不全的真实长尾

如果把 Farm 当成真机替代品，最后很容易得到一种错觉：

`机器很多，测试很多，但关键问题还是在线上才第一次看到。`

## 驱动差异为什么不能只看"安卓 / iOS"

兼容性问题里最容易被低估的一层，是驱动差异。

很多团队会把平台切得很粗：

- Android
- iOS

但真正导致问题的，往往不是这个粒度，而是更具体的交叉：

- Adreno / Mali / Apple GPU 不同家族
- Vulkan / OpenGL ES / Metal 不同后端
- 同一 OS 大版本上的不同驱动更新
- 某条渲染特性只在特定 GPU 上退化

这也是为什么兼容性测试不能只记"机型名"，还应该尽量补齐：

- GPU / SoC 家族
- 图形 API
- OS 大版本与补丁版本
- 渲染管线 / 图形设置档位
- 资源或热更快照版本

否则问题单最后会写成"某安卓机闪屏"，但这句话对修复并没有太大帮助。

兼容性策略在这里真正做的，是把驱动差异从"玄学"变成"可分维观察的问题面"。

## 最常见的假兼容性测试

### 1. 设备很多，但没有矩阵

最后不是覆盖更广，而是每轮都在随机抽查。

### 2. 只看销量，不看风险面

高收入机型测了很多，真正容易暴露边界的低端机和特殊渠道却没人测。

### 3. 把 Device Farm 当成真机替代

看起来自动化覆盖很漂亮，但关键体验问题和驱动问题仍然没人真正看。

### 4. 没有固定 smoke 路径

每次都"随便测一圈"，最终无法横向比较。

### 5. 线上发现问题后，矩阵不回写

同一类设备或驱动问题下次还会以同样方式回来。

## 一条最小可执行路径

如果团队现在还没有成型的兼容性策略，我会建议先从下面这条最小路径开始：

1. 先建立一套 `核心矩阵`，覆盖核心收入机型、低端边界机型、主流 GPU 家族和当前 / 上一主版本 OS。
2. 先固定 3 到 5 条最值钱的兼容性 smoke：安装、启动、登录、主入口、关键玩法入口。
3. 先把这几条 smoke 放进 `Device Farm`，用来补广覆盖的基础兼容检查。
4. 先保留少量核心真机，专门盯性能、视觉、交互和驱动高风险问题。
5. 先把线上告警和看板按设备、GPU、OS、渠道切维，季度回写一次矩阵。

只要这五步跑稳，兼容性测试就不再是"机器够不够多"的问题，而是"预算有没有用在最值得的差异面上"的问题。

## 结尾

所以我现在更愿意把兼容性测试策略理解成这样一条链：

`先用设备矩阵给预算排优先级，再用 Device Farm 承担广覆盖的固定 smoke，用核心真机盯高价值风险，用线上分维监控补长尾；兼容性不是穷举所有设备，而是持续收紧最值得测试的风险面。`

只有这样，团队才不会在"买更多设备"和"线上再看"之间来回摇摆，而是逐步建立一套能复用、能演化、能对准真实风险的兼容性策略。
