很多团队嘴上会同时说"烟测"和"回归"，但真正落到流程里，经常还是两种模糊状态：

- 烟测变成"包打完了大家随便点一点"
- 回归变成"出事后再临时拉一轮全量测试"

这两种做法共同的问题是：

`谁也说不清这次到底验证了哪一层。`

于是流程就会出现一种很典型的错觉：

- 明明"已经测过"
- 但线上还是坏了

原因通常不是大家没测，而是验证链没有被分层，也没有被固定。

所以这篇真正要讲的，是怎样把"构建产物验证、关键入口烟测、问题复现用例和发布前回归"收成一条最小但稳定的链路。

## 先给一句总判断

如果把整件事压成一句话，我会这样说：

`烟测负责回答"这批产物最基本地还能活着"，回归负责回答"那些我们不能再打坏一次的关键行为是否仍然成立"；两者都不该停留在口头习惯，而应该从构建产物、关键入口、历史故障和发布前候选版本这几层被固定下来。`

这句话里最关键的，是两件事：

- 烟测和回归不是一回事
- 但它们也不是彼此孤立的两堆 case

更稳的理解方式，是把它们放回同一条验证链。

## 一、先切概念：烟测在看"活不活"，回归在看"有没有重新坏"

烟测最适合解决的问题，是：

- 这次构建有没有形成一批完整产物
- 关键入口能不能最小跑通
- 最容易致命的链路有没有一上来就断

回归最适合解决的问题，是：

- 以前修过的问题会不会回来
- 一些关键业务、关键技术路径会不会被联动打坏
- 发布前的候选版本和稳定版本相比，是不是仍然站在可控范围内

所以如果把烟测写得太重，它就会失去速度和高频执行价值。
如果把回归写得太散，它又会退化成"每次想起来测点不一样的东西"。

更稳的做法是：

`烟测负责高频、窄面、快速证明；回归负责固定、可重放、覆盖那些已经证明值得被记住的风险。`

## 二、最小验证链，至少应该有五层

如果项目现在还没有成型的验证体系，我更建议先把最小链路拆成下面五层：

1. `构建产物验证`
2. `关键入口烟测`
3. `关键路径烟测`
4. `问题驱动回归`
5. `候选版本回归与回滚验证`

这五层的好处，不是让流程更复杂，而是让"这次到底测了哪一层"变得可说清。

## 三、第一层：构建产物验证，先证明"产物世界还是完整的"

这是最便宜、也最值得最早做的一层。

它负责回答的是：

- 包体、资源、热更产物是不是都生成了
- 关键 manifest、catalog、快照、符号文件是不是齐了
- 关键版本元信息是不是对得上

这层和 [游戏项目自动化构建：Jenkins / GitHub Actions 打包流水线]({{< relref "code-quality/game-project-build-pipeline-jenkins-github-actions.md" >}}) 以及 [变体、资源、热更链路怎样接进 CI]({{< relref "code-quality/variants-assets-and-hot-update-in-ci.md" >}}) 是直接相连的。

如果产物世界本身已经残缺，后面再做真机验证和玩法回归，很多时候只是更贵地发现同一件事。

所以这层最适合固定的通常包括：

- 构建是否成功
- 产物目录结构是否完整
- 关键资源快照和 catalog 是否齐全
- 热更 DLL / metadata / mapping 是否齐全
- 符号文件、日志和版本元信息是否归档

它的目标不是证明"这个版本没问题"，而是先证明：

`这次至少形成了一批可以被继续验证的完整产物。`

## 四、第二层：关键入口烟测，先看"第一脚能不能站住"

很多项目把烟测理解成"打开游戏看看"，这个粒度太粗。

更稳的烟测，应该围绕少数最值钱的入口来设计，例如：

- 启动到主界面
- 登录到主城
- 主战斗入口
- 当前线上活动入口
- 热更或资源切换后的首次进入

这层最重要的，不是覆盖全面，而是：

`只验证那些一旦坏了，版本就根本不值得继续往下看的关键入口。`

所以关键入口烟测应该尽量短、尽量稳、尽量少依赖人临场发挥。

这和 [热修复流程：紧急修复的快速验证通道、Cherry-Pick 策略、回滚预案与演练]({{< relref "code-quality/hotfix-flow-fast-verification-cherry-pick-and-rollback.md" >}}) 里的"快速验证通道"也是同一逻辑：紧急时不是不要验证，而是先把最值钱的一小段链路打透。

## 五、第三层：关键路径烟测，先挡住"入口能进，但用起来就坏"

很多高频事故并不会在启动或登录阶段马上显形，而是要等到第一次真实使用时才暴露：

- 第一个关键 prefab 实例化
- 第一段重要战斗流程
- 首次资源下载或内容切换
- 首次 shader 关键路径命中
- 第一条热更逻辑真正被执行

这也是为什么只做入口烟测通常不够。

关键路径烟测更像是在问：

`这批产物不只是能打开，而且第一段最关键的真实使用路径也还能活。`

它往往不需要覆盖整局玩法，但至少要固定少数高价值链路。

## 六、第四层：问题驱动回归，不要把回归写成"全量再测一遍"

回归最容易退化成两种东西：

- 一个永远做不完的大清单
- 或者每次临时根据记忆补一点点

更稳的做法，是把回归来源固定下来：

### 1. 历史线上事故

只要线上真出过一次问题，而且后续还可能回来，它就值得被升级成回归用例。

### 2. 高价值系统边界

例如：

- 资源快照切换
- 关键配置契约
- 灰度 / 回滚关键动作
- 热更生成和加载链
- 视觉或兼容性高风险点

### 3. 容易被联动打坏的关键路径

像登录、首包下载、主战斗、支付、关键活动入口，这类路径就算没出过事故，也往往值得常驻回归。

也就是说，回归不该来自"谁想起什么就补什么"，而应该来自：

`历史代价 + 当前风险 + 关键路径。`

## 七、第五层：候选版本回归与回滚验证，负责最后的发布把关

候选版本阶段的回归，不该只是把前面东西重复跑一遍。

它更适合承担两件事：

1. 证明候选版本相对最近稳定版本没有明显退化
2. 证明如果候选版本失败，团队仍然知道怎么退

所以这个阶段真正值钱的往往是：

- 候选版本和稳定版本的关键指标对比
- 已知高风险路径复跑
- 资源 / 热更 / 配置快照一致性确认
- 回滚指针和回滚版本可执行性确认

也就是说，发布前回归不是"再测一次"，而是：

`把最不允许带进线上、也最不允许回不来的那几类问题最后再确认一遍。`

## 八、烟测和回归分别该站在哪些节奏里

如果把这条链放回日常流程，我更建议这样分：

### 1. PR 或提交前

重点是最轻的产物检查和局部自动验证。

### 2. 合并后或 nightly

重点是构建产物验证、关键入口烟测、少量关键路径烟测。

### 3. 候选版本

重点是历史事故回归、关键路径复跑、回滚验证和版本对比。

### 4. 热修复

重点是缩窄到事故相关路径，不追求大而全，但必须保住构建完整性、目标回归和回滚抓手。

这条节奏线如果不先立住，烟测和回归最后就会永远卡在一句模糊的话里：

`大家找时间测一下。`

## 九、最常见的假烟测和假回归

最值得警惕的，通常是下面几种：

### 1. 把"构建成功"当成烟测完成

这只能说明脚本没在最前面直接炸掉。

### 2. 入口能进，就默认关键路径也没问题

很多问题恰恰要等第一次实例化、第一次下载、第一次切换、第一次执行时才暴露。

### 3. 回归清单只增不减，最后谁也不愿意跑

回归必须围绕高价值风险，而不是无限堆叠。

### 4. 修过一次的问题没有回写成回归

这样同类问题就只是在等下一次换一种形式回来。

### 5. 只验证"修复动作"，不验证"回退动作"

这会让发布和热修复都失去最后的安全绳。

## 一条最小可执行路径

如果团队现在想把烟测和回归真正立起来，我会建议先按下面这条最小路径落：

1. 先固定构建产物验证清单，保证每次构建都有完整产物和可归档证据。
2. 先选 3 到 5 条最值钱的关键入口做稳定烟测，不要一开始铺很大。
3. 再选 2 到 3 条最容易"入口能进、用起来就坏"的关键路径做烟测。
4. 把近一段时间最贵的线上问题和热修问题，各回写成至少一条固定回归。
5. 候选版本阶段固定跑一轮关键回归，并把回滚验证纳入同一批检查。
6. 每次发现某类问题重复出现时，不是再补一页文档，而是判断它该不该升级到这条验证链里。

这条路径的重点不是"一口气把测试做全"，而是先把验证链分层，先让最值钱的证据稳定下来。

## 结尾

所以我现在更愿意把烟测和回归理解成这样一条链：

`先验证构建产物完整，再验证关键入口能活，再验证关键路径第一次真实使用不会立刻坏，再把历史事故和高风险边界沉淀成固定回归，最后在候选版本阶段把版本对比和回滚抓手一起确认。`

只有这样，烟测才不是"随便点一点"，回归也不是"再测一遍所有东西"；它们才会真正变成团队发布前最小但稳定的工程护栏。
