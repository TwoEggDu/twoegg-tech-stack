很多团队一说"自动化构建"，第一反应还是：

- Jenkins 还是 GitHub Actions
- Workflow 怎么写
- 机器怎么配
- Unity 命令行参数怎么拼

这些当然都重要，但如果把"自动化构建"理解成工具脚本本身，就很容易把真正关键的东西看浅了。

`真正的自动化构建，不是把编辑器里的手工步骤搬进脚本，而是把输入、环境、生成步骤、产物和失败责任都固定下来。`

也就是说，Jenkins 和 GitHub Actions 只是承载物。
真正决定一条构建流水线有没有工程价值的，是它是否让构建变成：

- 可重复
- 可追溯
- 可归因
- 可发布

## 先把"流水线"这个词说具体

我现在更愿意把一条游戏项目构建流水线理解成下面这几个连续阶段：

```mermaid
flowchart LR
  A["代码 / 配置 / 资源输入"] --> B["环境准备"]
  B --> C["生成步骤"]
  C --> D["平台构建"]
  D --> E["产物整理"]
  E --> F["验证与门禁"]
  F --> G["发布 / 灰度 / 回滚准备"]
```

这里最容易被低估的，不是"怎么 build"，而是中间两层：

- 生成步骤
- 产物整理

因为在游戏项目里，真正容易出事故的，往往都发生在这两层：

- 代码生成、资源构建、热更生成物和最终 build 环境不一致
- 平台 SDK、证书、符号文件、bundle / manifest 没有稳定归档
- 同样的 commit 在不同机器上产出不一致

## 为什么游戏项目的构建，比普通业务项目更像一条生产线

普通应用项目的"构建"常常更接近编译 + 打包。
但游戏项目的 build 往往还要穿过更多层：

- 引擎版本
- 平台 SDK / NDK / Xcode / keystore / signing
- 资源构建和 bundle 产物
- Shader / stripping / AOT / codegen
- 热更、配置、CDN、版本号
- 符号文件、崩溃分析产物、灰度回滚产物

所以一条真正能用的游戏构建流水线，必须同时回答这几个问题：

- 这次 build 用的输入到底是什么
- 哪些生成步骤必须和最终 build 绑定
- 最终产物有哪些，放在哪里
- 后续 smoke、baseline、灰度和回滚需要的文件有没有一起产出

如果这些问题没有被流程固定住，所谓自动化构建通常只是：

`把人工点按钮的动作搬到了另一台机器上。`

## Jenkins 和 GitHub Actions 的真正区别，没你想的那么大

只从工程结构看，Jenkins 和 GitHub Actions 在这件事上的核心差异没有大到会改变文章主结论。

它们都能做这些事：

- 拉代码
- 准备环境
- 跑构建命令
- 归档产物
- 触发后续测试和发布步骤

真正影响你构建质量的，通常不是工具品牌，而是下面这些结构选择：

- 你有没有把环境做成固定版本
- 你有没有把生成步骤显式写进流程
- 你有没有把 release / development / debug 的差异写死
- 你有没有把产物归档、符号产物和回滚材料一起输出

所以比"选哪个平台"更值钱的问题是：

`这条流水线到底有没有把构建变成可重复的生产过程。`

## 一条最小可用的游戏构建流水线，至少要有哪几层

如果只能先搭最小版本，我会建议至少把下面几层立起来：

### 1. 环境准备层

这层要固定：

- 引擎版本
- 平台工具链版本
- 构建机依赖
- 证书、签名、密钥、环境变量来源

只要环境来源不固定，后面的失败就很难归因。

### 2. 生成步骤层

这层是最容易被漏掉的。

典型包括：

- 代码生成
- 热更生成物
- AOT / stripping / metadata / bridge 生成
- 配置编译
- 资源 bundle / manifest / catalog 生成

这也是为什么前面那篇 [HybridCLR 打包工程化｜GenerateAll 必须进 CI 流程，Development 一致性与 Launcher-only 场景]({{< relref "engine-toolchain/hybridclr-ci-pipeline-generate-all-and-development-flag.md" >}}) 特别重要。

它证明了一件很典型的事：

`有些生成步骤如果不和最终 build 环境一起被固定，流水线产物就会天然不可靠。`

### 3. 平台构建层

这一层才是大家通常以为的"build"本体：

- Android / iOS / Windows / macOS / Console 构建
- Development / Release 变体
- 平台开关、编译宏、证书和签名

### 4. 产物整理层

这一层特别容易被轻视，但它直接决定后续能不能发布、灰度、回滚和做 crash 分析。

至少要稳定产出：

- 安装包 / 可执行文件
- 资源产物
- 版本号与变更元信息
- 符号文件 / dSYM / symbols.zip / mapping
- manifest / build layout / catalog
- 回滚需要的上一个稳定版本引用

### 5. 验证与门禁层

构建成功不代表可以继续往前走。

这层至少应该能挂上：

- smoke
- 最小 regression
- 构建产物校验
- baseline
- release gate

也就是说，流水线的目标不是"成功生成一个包"，而是"生成一个可以继续进入发布流程的版本"。

## 为什么"生成步骤"和"最终 build 变量"必须同源

这件事在游戏项目里尤其关键。

如果生成步骤和最终 build 读的不是同一组参数，最后很容易出现一种特别难排的状态：

- 构建成功了
- 包也能装
- 但运行时某些链路在真机上坏了

典型例子包括：

- Development / Release 标志不一致
- 场景列表不一致
- 资源快照和最终包不一致
- 热更 DLL、AOT 元数据和最终裁剪产物不一致

所以更稳的做法永远是：

`生成步骤和最终 build 都由同一份请求参数驱动。`

不要一部分读命令行，一部分读编辑器面板，一部分再从机器本地状态里"猜"。

## 构建缓存很重要，但不能把错误也缓存下来

游戏项目构建通常都很重，所以缓存是必要的：

- Library cache
- 包管理缓存
- 构建中间产物
- 资源缓存

但缓存带来的风险也很明显：

- 上一次机器状态偷偷影响了这一次
- 某个过期生成物没有被刷新
- 看起来变快了，实际上只是复用了错误输入

所以缓存策略最核心的不是"缓存越多越好"，而是：

- 哪些缓存和输入强绑定
- 哪些缓存只在同一引擎版本、同一平台、同一关键参数下可复用
- 哪些生成步骤必须强制重跑

如果这三件事不清楚，缓存就会把一次偶发错误变成可重复错误。

## 产物归档不是顺手保存，而是后续流程的前提

一条真正能用的流水线，不应该在 build 结束时只留下一个 apk / ipa / exe。

因为发布、灰度、回滚、监控和 crash 分析都还要依赖这些东西：

- 版本元信息
- 资源版本信息
- 符号文件
- 构建日志
- 关键分析产物
- Smoke / baseline 结果

这些东西如果不和安装包一起归档，后面出了问题，团队很快就会陷入：

- 包还在，但符号文件没了
- crash 发生了，但当时的 build 参数找不到
- 线上版本有问题，但不知道对应的是哪一份资源快照

所以"归档产物"从来不是收尾动作，而是整条流水线的组成部分。

## 最常见的假自动化构建

看起来已经自动化，实际上工程价值不高，通常是下面几种：

### 1. 只是远程点按钮

构建过程搬到了 CI 上，但输入和机器状态依然不固定。

### 2. 生成步骤没有被显式写进流程

比如热更、代码生成、资源构建仍然依赖某次手工操作或编辑器面板当前状态。

### 3. 环境和参数不一致

同一个 commit 在不同机器、不同时间上 build 出来的东西不一样。

### 4. 只有安装包，没有上下文

包有了，但符号、资源版本、manifest、日志和回滚材料没有一起保存。

### 5. 构建成功就算通过

没有 smoke、没有产物校验、没有 baseline，流水线最终只是把"能打出来包"自动化了。

## 一条最低可执行路径

如果团队现在还没有一条真正站得住的游戏构建流水线，我会建议先这样落：

1. 先把引擎版本、平台 SDK、签名和环境变量固定住。
2. 先把所有必要生成步骤显式写进流程，不再依赖编辑器面板状态。
3. 先让 build request 同时驱动生成步骤和最终平台 build。
4. 先把安装包、资源快照、符号文件和关键日志一起归档。
5. 先把最小 smoke 和关键产物校验接在 build 后面。

这一套先跑稳以后，再往上接 baseline、灰度和更复杂的发布编排。

## 结尾

所以我现在更愿意把游戏项目自动化构建理解成这样一条链：

`不是"让一台机器替你点打包按钮"，而是把环境、输入、生成步骤、平台构建、产物整理和后续门禁固定成一条可重复、可追溯、可发布、可回滚的生产线。`

真正决定这条线值不值得信任的，从来不是 Jenkins 还是 GitHub Actions，而是它有没有让构建结果脱离"机器当前状态"和"操作者临场习惯"。
