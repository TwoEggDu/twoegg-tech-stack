很多团队说自己“很难写测试”，真正的难点往往不是不会写断言，也不是不会用测试框架，而是代码从一开始就没有可测试的切口。

最典型的状态是：

- 业务逻辑直接长在 `MonoBehaviour` 里
- 时间、随机数、输入、网络和存档全都直接写死
- 状态变化和表现、副作用混在一个方法里
- 要测一个判断，必须先把半个场景都搭起来

这时候“补单元测试”听起来像一种额外负担，其实问题不在测试，而在设计本身。

`可测试性设计真正要解决的，不是怎样让你更容易写几条测试，而是怎样让系统的决策逻辑、状态变化和副作用边界变得清楚。`

## 可测试性不是为了测试框架而抽象

很多人一听到“可测试性设计”，就会本能地担心系统被过度抽象。

这种担心并不假，因为确实有团队会为了“好测”把所有东西都包成接口，最后得到一套没人愿意维护的抽象层。

但真正值钱的可测试性设计，不是为了适配测试框架，而是为了把下面三件事拆开：

1. 决策逻辑
2. 状态变化
3. 副作用

只要这三件事混在一起，测试就会很重，重构就会很险，review 也很难准确判断“这次到底改了什么”。

所以更准确的说法应该是：

`可测试性不是额外要求，而是系统边界清楚以后自然得到的副产物。`

## 游戏代码为什么特别容易失去可测试性

和很多普通业务系统相比，游戏客户端更容易把逻辑写得不可测，通常有几个典型原因：

### 1. 引擎对象天然带很多环境依赖

`MonoBehaviour`、`Transform`、`Animator`、资源加载、事件回调、生命周期消息，这些都让逻辑很容易和运行环境绑死。

### 2. 逻辑和表现很容易长在一起

比如：

- 技能释放逻辑和动画事件混在一起
- UI 状态判断和真正的业务状态混在一起
- 战斗判定和特效播放、声音回调混在一起

### 3. 隐式全局状态太多

单例、静态入口、全局配置、当前时间、当前场景、当前玩家状态，这些如果不显式暴露出来，测试很难精准构造上下文。

### 4. 副作用特别密集

发送网络、写存档、生成对象、播特效、发事件、打日志、改 UI，这些动作一多，任何一个判断都不再是“输入 -> 输出”，而是“输入 -> 一堆环境变化”。

## 最核心的设计目标：让决策逻辑先变成可独立运行的一层

如果只能说一个最重要的原则，我会选这个：

`先让决策逻辑脱离引擎环境，变成一层可以独立运行和断言的东西。`

这通常意味着：

- 输入是显式数据
- 输出是结果、命令或状态变化
- 副作用由外层执行

举个非常简化的例子。

不要一开始就把逻辑写成这样：

```csharp
public class SkillButton : MonoBehaviour
{
    public void OnClick()
    {
        if (Time.time < player.NextCastTime)
            return;

        if (player.Mp < cost)
            return;

        player.Mp -= cost;
        player.NextCastTime = Time.time + cooldown;
        audioSource.PlayOneShot(castClip);
        battleService.Cast(skillId, targetId);
    }
}
```

这里点击、时间、资源扣减、冷却更新、音频播放和真正的施法请求全都绑在一起了。

更稳的切法，通常是先把判断层抽出来：

```csharp
public sealed class SkillCastPolicy
{
    public SkillCastResult TryCast(SkillCastInput input)
    {
        if (input.Now < input.NextCastTime)
            return SkillCastResult.Fail("Cooldown");

        if (input.CurrentMp < input.Cost)
            return SkillCastResult.Fail("NoMp");

        return SkillCastResult.Success(
            nextCastTime: input.Now + input.Cooldown,
            nextMp: input.CurrentMp - input.Cost);
    }
}
```

然后让外层去消费结果、执行副作用。

这样做的价值不只是“更好测”，而是：

- 规则边界清楚了
- 改动更容易 review
- 重构时更容易做最小验证

## 哪些依赖最该显式化

如果代码想变得更可测，最值钱的不是抽更多接口，而是把最影响判断稳定性的依赖显式化。

我会优先显式化这些：

### 1. 时间

冷却、超时、重试、Buff 持续、动画窗口、节流，这些一旦直接读全局时间，测试会非常难写。

### 2. 随机数

掉落、暴击、目标选择、随机事件，如果不能稳定注入种子或随机源，测试几乎不可重复。

### 3. 外部服务

网络、存档、资源加载、平台接口、支付、广告，这些都应该尽量作为边界能力，而不是在核心规则里直接调用。

### 4. 全局状态

如果逻辑依赖当前玩家、当前关卡、当前模式、当前配置，但这些状态都只能通过一堆全局单例拿到，测试就很难精准搭建上下文。

## 可测试性更像“切口”，不是“全量替换”

这也是很多团队容易误解的点。

可测试性设计不意味着一上来就要把整个旧系统重写成“六边形架构”。
真实项目里更稳的做法，通常是先从热点找切口：

- 先挑最常改、最容易回归的判断逻辑
- 先把这部分逻辑从引擎对象里切出来
- 先把关键输入和输出显式化
- 先让最值钱的一小块能被单独断言

这和前面 [测试保护下的最小重构路径]({{< relref "code-quality/minimal-refactoring-path-with-test-protection.md" >}}) 是一条线的：

`不要一上来追求系统全面可测，而是先把最常改、最值钱的那一小块变得可验证。`

## 什么不该为了“可测试”而做

可测试性很重要，但也不是越多抽象越好。

我会避免下面这些过度反应：

### 1. 为了测而给所有类都包接口

如果一个依赖本来就稳定、简单、不会替换，也不在核心判断层里，就没必要为“好 mock”而额外包一层。

### 2. 把所有逻辑都拆得过细

可测试的目标不是把系统拆得碎，而是把边界拆得清楚。

### 3. 把引擎胶水层也硬写成纯逻辑

有些层天生就是 orchestration 或 glue。
它们不需要假装自己完全纯净，更重要的是保证它们薄、明确、好替换。

## 一条最低可执行路径

如果一个项目现在测试基础很弱，但又想逐步把代码变得可测，我会建议先按这条路走：

1. 先挑一个高频回归的热点判断。
2. 先把时间、随机数、外部服务这几类最关键依赖显式化。
3. 先把判断结果写成明确输入 / 输出，不要夹着副作用一起做。
4. 先补最小单元测试，钉住这条规则的正常值、边界值和异常值。
5. 再让外层 glue 代码去消费结果、执行真正副作用。

这一套跑稳以后，再继续往状态机、配置解析、资源恢复链、关键服务编排这些更复杂的地方扩。

## 结尾

所以我现在更愿意把“可测试性设计”理解成这样：

`不是为了测试而额外制造抽象，而是先把决策逻辑、状态变化和副作用边界切开，让系统最关键的判断层能够独立运行、独立断言、独立重构。`

只有这样，单元测试才不是一套额外负担，而会变成系统边界清楚以后自然成立的结果。
