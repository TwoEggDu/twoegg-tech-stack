很多团队一说"自动化测试"，还是很容易把问题压成一句：

- 写点单测
- 接进 CI
- 然后等它帮我们拦问题

这套说法当然不完全错，但放到 Unity 项目里，真正的问题很快就会冒出来：

- 哪些东西适合在 `EditMode` 里测
- 哪些必须进 `Play Mode`
- 性能回归到底算测试、算基线，还是算 nightly 报告

如果这些边界不先切开，自动化测试最后很容易退化成两种样子：

- 要么全堆在 `EditMode`，结果挡不住运行时问题
- 要么什么都想放 `Play Mode`，结果变慢、变脆、没人愿意维护

所以这篇真正要回答的不是"Unity Test Framework 怎么用"，而是：

`Unity 项目里，EditMode、Play Mode 和性能回归测试到底各自解决什么，以及它们该怎样站进同一条 CI 链。`

## 先给一句总判断

如果把整件事压成一句话，我会这样说：

`Unity 自动化测试最怕的不是数量不够，而是边界不清；EditMode 负责快速证明代码和配置契约，Play Mode 负责证明关键运行时路径仍然成立，性能回归负责证明预算没有被慢慢吞掉。`

这三层不是互相替代的，而是互相收口的。

## 一、不要把"Unity 自动化测试"当成一种东西

很多团队失败，不是因为没写测试，而是因为把下面这些本来不同的问题塞进了同一个测试桶里：

- 逻辑和配置契约验证
- 场景、对象和生命周期验证
- 性能与资源预算验证

它们都能叫"自动化测试"，但显然不适合放在同一节奏、同一机器、同一 gate 上。

更稳的切法通常是：

1. `EditMode`
2. `Play Mode`
3. `Performance Regression`

只有先分开，后面才谈得上接进 CI。

## 二、EditMode 最适合拦"快、稳、规则清晰"的问题

`EditMode` 最值钱的地方，不在"它是单元测试"，而在：

`它跑得快，依赖少，最适合前移到合并前。`

它更适合处理的问题包括：

- 纯逻辑和公式验证
- 配置 schema、字段约束、引用合法性
- 代码生成、构建前校验、导入期规则
- 资源或模块命名、目录和依赖边界

这和前面 [什么问题必须做成自动检查，不能靠人盯]({{< relref "code-quality/what-must-be-automated-checks.md" >}})、[单元测试基础：游戏代码的可测试性设计，如何解耦才能写测试]({{< relref "code-quality/testability-design-for-game-code.md" >}}) 讲的是同一件事：

`只要规则稳定、代价高、执行成本低，就应该尽量前移。`

### EditMode 不适合硬扛什么

它不适合硬扛：

- 真实场景初始化
- 关键 prefab / scene 生命周期
- 资源下载、热更加载
- 渲染、输入、异步和帧驱动问题

如果团队拿 `EditMode` 去假装验证这些，最后通常只是获得一堆"看起来像自动化"的假确定性。

## 三、Play Mode 最适合证明"运行时关键路径还能活着"

`Play Mode` 的价值，不在于"它更接近真机"，而在于：

`它能真实展开对象世界、生命周期和关键入口。`

更适合放进 `Play Mode` 的通常包括：

- 关键场景或主入口初始化
- prefab / UI / 脚本组件的真实装配
- 异步流程、协程、事件驱动和一帧以上的状态变化
- 资源、配置、热更加载后的最小功能路径

这也是为什么 [烟测与回归：从构建产物验证到关键路径回归的最小链路]({{< relref "code-quality/smoke-tests-and-regression-minimal-chain.md" >}}) 里会强调：

`很多问题不是"能不能 load"，而是"第一次真的用起来会不会坏"。`

### Play Mode 不该变成什么

它不该变成：

- 一套无限膨胀的全量 UI 回归
- 一堆依赖真实网络、真实运营配置的脆弱 case
- 每个 PR 都要硬跑一小时的流程测试

更稳的思路，是把 `Play Mode` 聚焦在少数高价值关键路径，而不是追求"模拟所有真实世界"。

## 四、性能回归测试更像"预算验证"，不是另一种玩法测试

很多团队把性能回归混在 `Play Mode` 里一起跑，最后经常会遇到两个问题：

- 结果不稳定
- 超线了也没人知道该怎么解释

更稳的理解应该是：

`性能回归不是为了证明"这一帧一定没问题"，而是为了证明固定样本、固定环境下的预算没有发生明显退化。`

所以性能回归更适合围绕：

- 固定场景
- 固定设备或设备组
- 固定构建模式
- 固定统计口径

去看下面这些指标：

- 帧时间 / FPS
- 峰值内存
- 关键场景加载时间
- shader 首载或资源首载耗时

这和 [Baseline：性能 / 包体 / 加载 / Crash 预算怎样立线并进 CI]({{< relref "code-quality/baseline-budgets-in-ci.md" >}}) 是一条线：

`性能回归真正有价值，是因为它接到了预算和超线动作，不是因为它能输出很多曲线。`

## 五、这三层更适合放在哪些 CI 节奏里

如果把它们都塞在每个 PR 上，团队很快就会开始绕流程。

更稳的放法通常是：

### 1. PR / 合并前

更适合放：

- EditMode 逻辑验证
- 配置和生成规则校验
- 轻量 smoke

这里目标是快，不是全。

### 2. 合并后 / Nightly

更适合放：

- 关键 Play Mode 路径
- 关键资源 / 热更 / 入口 smoke
- 性能回归和加载基线

这里目标是稳定地产出证据，而不是卡死每个开发者。

### 3. 候选版本 / Release Gate

更适合放：

- 高价值 Play Mode 回归
- 固定场景性能预算
- 发布产物和线上前置验证

这里目标是证明候选版本仍然站在预算和可发布边界内。

## 六、最常见的假自动化测试

最容易把团队带偏的，通常是下面几种：

### 1. 只有 EditMode，没有 Play Mode

看起来自动化很多，实际上最容易坏的运行时路径仍然是空的。

### 2. 什么都堆进 Play Mode

最后结果是慢、脆、难维护，团队只会越来越少跑。

### 3. 性能回归没有固定样本和阈值

这样得到的不是预算，只是噪声。

### 4. 所有测试都试图卡在 PR

不同测试适合不同节奏，强行统一只会让流程失去可信度。

### 5. 测试失败后没有 owner

自动化不是"机器替团队负责"，失败以后仍然要有人解释、修复和回写。

## 一条最小可执行路径

如果团队现在想把 Unity 自动化测试真正接起来，我会建议先按下面这条最小路径落：

1. 先立一批最快、最稳的 EditMode 校验，优先拦逻辑、配置和生成规则。
2. 再选 3 到 5 条最值钱的 `Play Mode` 关键路径，不要一开始铺全量。
3. 固定 1 到 2 个性能回归样本场景，先跑出可比较的基线。
4. 把三层测试分别放进 PR、nightly 和候选版本，不要强行同一节奏。
5. 每次线上事故或热修问题结束后，判断它该回写到哪一层自动化里。

只要这一步站住，自动化测试才会从"写了一些 case"真正变成"持续交付里的验证链"。

## 结尾

所以我现在更愿意把 Unity 自动化测试理解成这样一条链：

`EditMode 先拦规则明确的问题，Play Mode 再证明关键运行时路径仍然成立，性能回归最后守住固定样本上的预算；只有三层各站各位，Unity Test Framework、Play Mode 测试和性能回归测试才不会互相拖累，而会一起组成真正可维护的 CI 护栏。`
