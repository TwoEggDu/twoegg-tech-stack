我现在越来越不相信一种做法：把所有本来应该机器判断的东西，最后都交给人去盯。

因为 review 再认真，也只能覆盖一部分问题。人擅长看上下文，擅长在两个方案之间做取舍，擅长在信息不完整的时候补齐意思；但人不擅长长期稳定地重复同一组判断，也不擅长在几十个 PR、几百次改动里，一次不漏地记住所有规则。

所以自动化测试真正该做的，不是“证明一切都没问题”，而是把最值钱、最可重复、最容易回归的那部分问题先锁住。

如果把这件事压成一句话，我更愿意这样说：

`测试不是把所有问题都接过来，而是优先接住那些一旦变坏、后面就会持续付利息的地方。`

## 先看什么，后看什么

自动化测试最值得先盯的，通常不是“看起来最复杂”的部分，而是“最容易反复出错、最容易悄悄退化、最容易把后续改动拖慢”的部分。

这类问题大致有五种：

1. 纯逻辑：输入明确，输出明确，规则明确。
2. 配置契约：表、字段、引用、约束、默认值。
3. 资源链路：导入、打包、加载、实例化、首帧。
4. 构建产物：最后产出到底是不是我们想要的东西。
5. 基线阈值：性能、包体、加载时间、崩溃率有没有退化。

这五类问题有一个共同点：它们都可以被定义、被重复、被自动比对。

一旦能做到这三点，就不该继续主要依赖人去记、去猜、去临场判断。

## 纯逻辑最该先自动化

纯逻辑是最适合测试的地方，因为它最接近“同样输入应该得到同样输出”的世界。

像伤害计算、数值修正、状态切换、冷却规则、资源消耗、条件分支、优先级判定，这类东西如果每次都靠人手点一遍，很快就会变成一种假稳定：

- 小改动看不出来
- 边界情况容易漏
- 以后回归时很难知道是哪里坏了

这类测试的价值不在于“覆盖率好看”，而在于它能把最容易被改坏的行为钉住。

如果一个逻辑改动只要一改就容易带出另一个分支，那这个地方就不该只靠 review 盯着。review 能看出“这段逻辑像不像”，但测试才能证明“改完以后行为是不是还是原来的行为”。

## 配置、资源、构建物，测试更像门禁

第二类最该自动化的，是配置和资源。

很多团队把配置当成“数据”，把资源当成“文件”，但实际它们更像契约。一个字段改名、一个引用断掉、一个默认值变了，影响的往往不是一个点，而是一串链路：

- 业务逻辑
- UI 表现
- 加载过程
- 兼容分支
- 平台差异

这类问题最麻烦的地方在于，它们经常不是“代码错了”，而是“规则散了”。规则一旦散到多个地方，review 就会开始失真，因为人很难在一次 diff 里同时看见所有后果。

所以配置契约、资源链路和构建产物，更适合被测试当成门禁来拦。

测试在这里看的不是“这一行写得对不对”，而是“这个变更最后有没有把系统产出的东西变坏”。

## 基线不是感觉，是门槛

性能、包体、加载时间、首帧时间、崩溃率，这些都不应该只靠感觉。

很多团队会说：

- 这次好像慢了一点，但不明显
- 包体大了一点，但还能发
- 卡顿多了一点，但先观察

这些判断单独看都合理，但堆在一起就会把系统慢慢拖垮。

所以基线测试的意义不是追求“永远不变”，而是让变化有边界。超过边界，就要有人看到；一旦超出阈值，就应该有明确动作，而不是继续靠经验解释。

如果一个项目已经开始遇到版本压力、设备差异、资源膨胀和首载问题，那基线测试就不再是可选项。它是把“觉得还行”改成“是否真的还行”的最低手段。

## 哪些更适合 review 或人工判断

自动化测试不是 review 的替代品。

这句话要说清楚，因为很多团队一旦开始上测试，就会不自觉地把 review 的职责也一起甩给测试，最后变成两边都不够用。

更适合 review 或人工判断的，通常是这些：

- 这次改动有没有打穿系统边界
- 依赖方向有没有变乱
- 这个接口语义是不是清楚
- 这次重构值不值得
- 两个方案哪个更符合当前阶段
- 临时补丁要不要先收着，还是现在就折进去

这些问题需要上下文，需要取舍，需要判断阶段目标。它们不是“测一测就能定”的问题。

测试更适合回答的是“行为有没有变坏”，review 更适合回答的是“这次改动合不合理”。

把这两个职责混在一起，最后通常会出现一种假象：测试很多，review 也很认真，但系统还是越来越难改。

## 哪些不值得写测试

不是每个问题都值得写测试。

如果一个问题本质上只是局部风格、一次性处理、或者只会在探索期变化，那把它强行变成测试，往往是在制造额外维护成本。

这类东西包括：

- 一眼就能看出来的局部整理
- 只会出现一次的临时修补
- 方案还在探索时的高频试错点
- 必须依赖业务背景才能理解的取舍

这些问题该交给人判断的时候，就不要硬塞进测试系统里。否则测试会变成一种新的负担，团队为了维护它，反而把时间花在解释工具上，而不是修正真正值钱的风险。

## 最后看什么

我现在判断一个问题值不值得自动化，基本只看三件事：

1. 它是不是会反复出现。
2. 它是不是能被明确写成规则。
3. 它是不是一旦回归，后面会持续付利息。

满足这三条的，优先自动化。

不满足的，先交给 review、设计判断或者阶段性人工验证。

这就是我理解里的工程质量：不是把所有事情都塞给机器，而是把机器最擅长的部分拿走，把人最值钱的判断留下来。
