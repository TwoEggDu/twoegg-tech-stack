一次版本迭代里，场景美术集中提交了一批新的建筑贴图。48 张，每张都是 4096x4096，格式 RGBA32 未压缩。单张 64MB，48 张合计超过 3GB 的源文件涌进了资源目录。

问题不是立刻暴露的。Editor 内一切正常——开发机有 32GB 内存，加载 3GB 贴图连卡都不卡。CI 构建也顺利通过——构建脚本没有做资源规格检查，只要不报编译错误就是绿灯。版本打出来之后，QA 在中端测试机上跑了核心流程，也没有进这个场景。

问题是在海外提审时暴露的。Google Play 的 AAB 限制是 150MB，这批贴图压缩后仍然把包体从 130MB 推到了 210MB，提审直接被拒。排查后还发现了更深层的问题：这些贴图在 Android 上没有被设置为 ASTC 压缩，运行时每张占用 64MB 显存，进入这个场景时 3GB 内存的设备直接 OOM。

回溯整个链路：导入时没有 AssetPostprocessor 强制设置压缩格式，PR 时没有资源规格检查脚本，构建时没有包体增量告警。48 张超标贴图畅通无阻地走完了从提交到构建的全部流程。

`资源质量要解决的核心问题不是"美术会不会犯错"，而是"不符合规范的资源能不能在进入主干之前被自动拦住"。`

这篇要回答的就是这件事：美术资源从导入到构建，自动化检查应该在哪几个时机介入，每个时机该检查什么，以及怎样把检查结果接进 CI 和构建门禁。

## 美术资源出问题的 6 种常见方式

资源问题的种类很多，但最高频的就是这六类。

### 1. 超预算贴图

UI 图标用了 2048 甚至 4096 的尺寸，场景贴图全是未压缩的 RGBA32，一张角色的法线贴图比角色本身的 diffuse 还大。

单张贴图超标看起来影响不大，但几百张叠在一起，包体和运行时内存就会被一起拖垮。

### 2. 缺 LOD 的高面数模型

角色模型 8 万面，场景装饰物 3 万面，但都只有一级 LOD 或者根本没有 LOD。

在中远距离渲染时，GPU 在画玩家根本看不清的细节。低端机的帧率就是这样被吃掉的。

### 3. 错误的压缩格式

Android 上用了 PVRTC，iOS 上用了 ETC2，或者某个平台的贴图根本没有设置压缩格式，用的是默认的未压缩导入。

这类问题在编辑器里看不出来，只有打包到目标平台后才会暴露成包体膨胀或加载变慢。

### 4. 引用丢失

材质球引用了一个已经被删掉的 Shader，Prefab 引用了一个已经被移动路径的贴图，动画引用了一个已经被重命名的骨骼。

这些在编辑器里可能只显示一个粉红色或 Missing 标记，但到了运行时就是崩溃或视觉异常。

### 5. 命名违规

贴图命名包含中文和空格，模型文件用了大写开头，材质球命名和对应的模型对不上。

命名问题短期看只是乱，长期看会让自动化工具失效、资源查找出错、bundle 分组策略被打破。

### 6. 重复资源

同一张贴图在三个不同目录各存了一份，名字不一样但内容完全一样。

构建时它们会被分别打进不同的 bundle，运行时加载三次、内存占三份。

这六类问题有一个共同特征：它们在美术的 DCC 工具和 Unity 编辑器里通常不会报错，只有在打包、真机运行或 Profiler 里才会暴露。

## 自动化检查的 3 个时机

资源检查不应该只在某一个环节做。更稳的做法是在 3 个时机分别设防。

### 第一个时机：导入时（AssetPostprocessor）

Unity 的 `AssetPostprocessor` 可以在资源导入到编辑器的瞬间执行检查和修正。

适合在这个时机做的：

- 强制设置压缩格式（按平台、按资源类型）
- 强制设置 maxTextureSize（按目录或命名规则）
- 检查模型面数并输出警告
- 检查命名规则
- 自动设置 AssetBundle 标签或 Addressable Group

这个时机的价值在于：美术拖一个文件进 Unity，几秒内就知道哪里不符合规范。

但它有一个明确的边界：`AssetPostprocessor` 只能处理单个资源的导入事件，无法做跨资源的引用检查或全局统计。

### 第二个时机：PR 时（自定义检查器）

美术提交 PR 后，CI 流水线里运行自定义检查脚本。

适合在这个时机做的：

- 全量引用完整性检查（材质引用的 Shader 是否存在，Prefab 引用的贴图是否存在）
- 跨资源重复检测（内容 hash 比对）
- 资源预算汇总报告（本次提交涉及的所有资源是否在预算内）
- 命名规则全量扫描
- 与上一个稳定版本的 diff 报告

这个时机的价值在于：就算 `AssetPostprocessor` 被绕过或者规则没覆盖到，PR 阶段依然能兜住。

### 第三个时机：构建时

正式构建流水线中，在打包之前和之后分别做检查。

适合在这个时机做的：

- 构建产物的体积验证（单个 bundle 不超过阈值，总包体不超过预算）
- 构建后的 bundle 内容审计（哪些资源被打进了哪个 bundle，有没有意外的重复）
- Shader variant 数量验证
- 构建报告归档

这个时机的价值在于：它验证的是真正会被交付给玩家的产物，而不只是编辑器里的状态。

## AssetPostprocessor 的检查能力和边界

Unity 的 `AssetPostprocessor` 提供了 `OnPreprocessTexture`、`OnPreprocessModel`、`OnPreprocessAudio` 等回调，可以在资源导入时自动执行逻辑。

它能做的事：

- 读取和修改导入设置（压缩格式、maxTextureSize、Mesh 压缩等级等）
- 根据资源路径判断资源类型并应用不同规则
- 输出日志或弹出对话框提醒美术

它不能做的事：

- 无法访问其他资源的状态（做不了跨资源引用检查）
- 无法做全局统计（做不了"当前项目所有贴图的总内存是多少"）
- 如果美术直接在 Inspector 里手动改回导入设置，`AssetPostprocessor` 不会再次触发
- 重导入整个目录时可能产生大量日志，需要控制输出量

所以 `AssetPostprocessor` 适合做"单个资源的导入规范化"，但不适合做"全局资源质量审计"。后者需要自定义检查器。

## 自定义检查器的规则设计

一个能长期维护的资源检查器，需要满足三个条件：可配置、可扩展、区分严重级别。

### 可配置

检查规则不要写死在代码里，而是用配置文件定义。

例如：

```
ui_texture:
  max_size: 512
  format_android: ASTC_6x6
  format_ios: ASTC_6x6
  allow_alpha: true

character_texture:
  max_size: 2048
  format_android: ASTC_4x4
  format_ios: ASTC_4x4
  allow_alpha: true

scene_texture:
  max_size: 1024
  format_android: ASTC_6x6
  format_ios: ASTC_6x6
  allow_alpha: false
```

这样美术和 TA 能直接看到当前的规范标准，修改规则不需要改代码。

### 可扩展

每条规则实现一个统一接口，新增规则只需要加一个类和一条配置。

不要把所有检查逻辑堆在一个几千行的大脚本里。那样的脚本三个月后没人敢动。

### 区分 Block 和 Warn

不是所有违规都应该阻断提交。

- **Block**：必须修复才能合入。例如：贴图超过允许的最大尺寸、材质引用了不存在的 Shader、模型面数超过硬性上限。
- **Warn**：建议修复但不阻断。例如：贴图尺寸接近上限、模型面数接近预算线、命名不完全符合规范但不影响功能。

如果所有规则都是 Block，美术会觉得提交任何东西都要过关打怪，最后会想办法绕过检查系统。

## 预算验证：每种资源类型的预算标准

资源预算不是一个全局数字，而是按资源类型分别定义的。

以下是一组常见的移动端资源预算参考：

### 贴图预算

| 类型 | 最大尺寸 | 压缩格式 | 说明 |
|------|---------|---------|------|
| UI 图标 | 512x512 | ASTC 6x6 | 大部分 UI 元素不需要超过 256 |
| 角色 Diffuse | 2048x2048 | ASTC 4x4 | 主角可放宽到 2048，NPC 建议 1024 |
| 角色法线 | 1024x1024 | ASTC 6x6 | 法线对精度敏感，但尺寸通常可以比 diffuse 小一级 |
| 场景贴图 | 1024x1024 | ASTC 6x6 | 大场景可考虑 tiling 而不是加大尺寸 |
| 特效贴图 | 256x256 | ASTC 6x6 | 特效贴图生命周期短，尺寸应该尽量小 |

### 模型预算

| 类型 | 面数上限 | LOD 要求 | 说明 |
|------|---------|---------|------|
| 主角 | 30000 面 | 至少 3 级 LOD | 近景角色预算可适当放宽 |
| NPC | 15000 面 | 至少 2 级 LOD | 非战斗 NPC 可进一步压缩 |
| 场景主体建筑 | 20000 面 | 至少 2 级 LOD | 包含碰撞体需要单独计算 |
| 场景装饰物 | 5000 面 | 至少 1 级 LOD | 批量装饰物的 LOD 收益最大 |
| 武器 / 道具 | 8000 面 | 至少 1 级 LOD | 视情况调整 |

### 动画和音频预算

- 单个动画片段不超过 60 秒（超过的应该拆分）
- 音效采样率不超过 44100Hz，大部分 SFX 用 22050Hz 就够
- 背景音乐用 Vorbis 压缩，质量 70%
- 单个音频文件不超过 10MB

这些数字不是绝对标准，每个项目应该根据自己的目标平台和性能预算调整。但一定要有明确的数字写下来，而不是靠口头约定。

口头约定的预算，三个月后就没人记得了。

## CI 集成：检查结果怎样阻断构建或生成报告

资源检查接进 CI 后，需要解决两个问题：怎样展示结果，怎样阻断流程。

### 结果展示

CI 产出的资源检查报告应该至少包含：

- 本次提交涉及哪些资源文件
- 每个文件的检查结果（Pass / Warn / Block）
- Block 级问题的具体原因和修复建议
- 与上一个稳定版本相比，新增了哪些资源、删除了哪些资源、哪些资源的体积发生了变化
- 全项目资源预算的当前占用情况

报告应该输出为可读的格式（HTML 或 Markdown），而不是埋在几千行构建日志里。

### 阻断策略

和配表校验一样，资源检查也应该分级阻断：

- **硬阻断**：存在 Block 级问题时，PR 不能合入、构建不能通过。例如：贴图超过最大尺寸上限、引用丢失、Shader 编译失败。
- **软告警**：存在 Warn 级问题时，允许合入但在报告中标注。例如：贴图接近上限、面数接近预算线、命名不完全规范。
- **趋势监控**：不阻断也不告警，但持续追踪。例如：总包体趋势、各类型资源占比变化、重复资源数量变化。

CI 的资源检查脚本通常通过 Unity 的 `-batchmode` 模式执行，调用自定义的检查器代码，输出结果文件，再由 CI 平台解析结果决定是否通过。

## 美术自检：提交前能自己验证的 checklist

不要让所有检查都发生在提交之后。美术应该有能力在提交前自己验证。

一个实用的美术自检清单：

1. **贴图尺寸**：检查当前资源类型的最大尺寸限制，在 Inspector 里确认 maxTextureSize 设置正确。
2. **压缩格式**：检查 Android 和 iOS 平台的压缩格式是否已设置，不要留默认值。
3. **模型面数**：在 DCC 工具导出前确认面数，进 Unity 后在 Inspector 里再确认一次。
4. **LOD**：确认模型是否按要求提供了足够层级的 LOD。
5. **引用完整性**：材质球上所有 slot 都有对应贴图或 Shader，没有 Missing 标记。
6. **命名规则**：文件名符合项目命名规范，没有中文、空格、特殊字符。
7. **目录位置**：资源放在了正确的目录下，没有放到临时目录或其他模块的目录里。

这个清单最好做成编辑器内的一键检查工具，美术点一下就能跑完所有检查，而不是靠记忆逐项核对。

## 反模式

### 1. 检查规则和美术规范不同步

美术规范文档写的是"UI 贴图不超过 512"，但检查器里配的是 1024。或者反过来，检查器比文档更严格，美术按文档做了还是被打回。

规范和检查器必须同源。最好的做法是检查器的配置文件本身就是规范的一部分，而不是另外维护一份文档。

### 2. 只检查新增，不检查存量

检查系统上线后只对新提交的资源生效，但项目里已经有几千个不符合规范的存量资源。

美术会觉得不公平：老资源没人管，新资源卡得这么严。

更好的做法是上线检查系统的同时，给存量资源做一次全量扫描，输出一份存量违规清单，分批治理。

### 3. Block 太多导致美术绕过

所有规则都是 Block 级别，美术提交任何东西都会被打回。

最后美术会找到绕过检查的方法：直接在服务器上改文件、提交到不受检查的分支、或者干脆不用版本控制。

Block 级规则应该只覆盖真正会导致崩溃或严重性能问题的情况。其他问题用 Warn 就够了。

### 4. 检查工具只有程序能用

检查器的运行方式是命令行脚本，美术不会用也不想用。

等到 CI 打回来，美术还是不知道具体哪里有问题。

检查工具应该有编辑器内的 GUI 入口，美术点一下就能看到结果，而不是需要打开终端敲命令。

### 5. 没有豁免机制

某些特殊资源确实需要突破预算（比如过场动画的特写角色贴图可能需要 4096）。但检查系统没有豁免机制，这些资源要么被强制打回，要么被加进全局例外导致规则形同虚设。

更好的做法是支持按路径或按标签的豁免配置，每条豁免都有明确的审批记录和过期时间。

## 一条最小可执行路径

如果团队现在还没有系统化的资源质量检查，我会建议先按这个顺序落：

1. 先写 `AssetPostprocessor`，对贴图的 maxTextureSize 和压缩格式做强制设置。
2. 先定义每种资源类型的预算数字，写进一个所有人都能看到的配置文件。
3. 先在 CI 里加一个最小资源检查脚本，检查引用完整性和 Block 级贴图超标。
4. 先给美术做一个编辑器内的一键自检工具。
5. 先对存量资源做一次全量扫描，输出违规清单，分批治理。

这五步站住以后，再逐步扩展：加模型面数检查、加重复资源检测、加构建后 bundle 审计、加趋势报告。

不要一开始就试图把所有规则都配齐。先让最关键的几条规则跑起来、让美术用起来、让 CI 能拦住最危险的问题，就已经比"什么都没有"好得多。

## 结论

所以我现在更愿意把资源质量这件事理解成：

`它不是"美术规范写得够不够细"，而是从导入、提交、构建到交付，每个环节都有自动化的检查、明确的预算数字和分级的阻断策略；只有当 AssetPostprocessor、自定义检查器、CI 门禁和美术自检工具一起进入流程，资源质量才算从"靠规范文档"变成"靠工程系统"。`

如果规范写了 50 页但没有任何自动化检查，那它最后就只是一份没人看的文档。
