很多团队一到 code review，就很容易把注意力放歪。

有人盯命名，有人盯缩进，有人盯空行，有人盯“这行是不是可以再短一点”。这些问题不是完全不重要，但它们通常不是最该拦的东西。真正该被 review 拦住的，是那些会在后面几轮迭代里持续制造协作成本、扩大回归面、让后人越来越难接手的结构问题。

所以我现在更愿意把 review 理解成一种判断顺序，而不是一种审美表态。

`代码评审的目标，不是把每个 PR 都挑得像教科书一样整齐，而是尽量在合并前，把未来会反复付利息的风险挡住。`

## 先看边界，不先看局部

review 最先该看的，不是某个变量名顺不顺眼，而是这次改动有没有碰到系统边界。

边界一旦乱了，后面很多小问题都会变成大问题。一个模块到底负责什么、不负责什么，数据从哪里进、从哪里出，哪些依赖是稳定契约，哪些只是临时穿透，这些问题如果没立住，代码写得再整齐也只是把混乱排得更整齐。

所以我会先问四个问题：

- 这个改动有没有把职责边界打穿
- 这个模块的依赖方向有没有被反过来
- 这次新增的数据流，是否让原本清楚的 ownership 变模糊了
- 这里是不是已经开始把一个“功能点”偷偷长成“半个子系统”

如果答案里有任何一个“是”，那 review 就不该继续停留在局部细节上，而应该先把边界问题讲清楚。

## 再看规则，有没有被重复散落

第二层该看的，是业务规则是不是开始四处分叉。

很多团队的代码一开始都不复杂，真正变难改，往往不是因为逻辑本身有多高深，而是因为同一条规则被拆进了多个地方。今天在 UI 里改一次，明天在业务层补一次，后天在资源或网络回调里再补一次，表面上每一处都“能跑”，实际上规则已经没有单一真相了。

这类问题 review 应该尽量拦住，因为它的代价不是一次性的。它会带来三件事：

1. 以后改规则时，你不知道要改几处。
2. 以后排 bug 时，你不知道哪一处才是“真规则”。
3. 以后做重构时，你会发现每个地方都自成一套小逻辑。

如果一次 PR 只是把临时判断复制到别处，或者为了赶进度把同一条规则又塞进了另一个长期层，那就不该把它当成“先跑起来再说”的小事。它很可能是在往系统里种长期债。

## 临时补丁，不要落进长期层

第三层该看的，是这次改动有没有把临时补丁写进了长期结构。

临时补丁本身不是原罪。问题在于，很多临时补丁会被贴进最稳定、最难改的地方，最后变成“谁都知道不优雅，但谁也不敢动”的永久结构。

review 里一旦看到这些信号，就应该提高警惕：

- 为了一个短期问题，直接改了基础抽象层
- 为了赶版本，把应急分支塞进了长期主流程
- 为了兼容一个旧场景，给新逻辑加了一个没有边界的例外
- 为了省事，在核心层留了一条“以后再清”的特殊路径

如果补丁只是落在一个局部、可替换、可回收的位置，那它还有机会被收掉；如果它已经落进长期层，后面再想清理，通常就得付两次成本，一次是当下改动，一次是未来还债。

## 对外接口，语义要先说清

第四层该看的，是对外接口是不是说清了自己到底做什么。

这里的“接口”不只是 API，也包括事件、配置、消息、回调、数据结构，甚至是某个跨模块函数的参数语义。很多 review 看起来没问题，实际上问题就藏在“名字看得懂，但语义不明确”这件事里。

最容易出问题的，不是有没有这个参数，而是这个参数到底表示条件、开关、策略、还是兼容位。最容易出问题的，也不是函数能不能跑，而是调用方到底能不能准确预期它的异常路径、默认行为和回滚路径。

如果一段接口代码让后续调用者必须靠猜，那 review 就应该拦。

因为接口语义不清，后面一定会出现三种后果：

- 调用者各自理解一套
- 新人接手时只能沿着旧调用链猜意图
- 后续修改时，没人敢确认哪条分支是“标准路径”

这类问题表面上不像 bug，实际上特别容易在多人协作里变成 bug。

## 异常和回滚，要能一眼看懂

第五层该看的，是异常路径和回滚路径是否清楚。

正常路径写得再顺，如果失败时会发生什么说不明白，系统就还是不稳。review 里我最不愿意看到的，就是“成功时很漂亮，失败时很含糊”的代码。因为真正决定系统可靠性的，往往不是主路径，而是异常时怎么退、怎么补、怎么恢复默认状态。

这类问题应该重点检查：

- 出错后是直接失败、局部失败，还是继续降级
- 回滚是否会留下半完成状态
- 默认值是不是会掩盖真实错误
- 某个异常分支是不是只在作者脑子里存在

如果一段逻辑的失败路径说不清，review 就不该放过。因为它不是“将来再补测试”的问题，而是当前行为本身就不确定。

## 功能和重构，不要混成一锅

第六层该看的，是功能改动和结构重整有没有混在同一个提交里。

这个问题很常见，也很致命。很多人会说：“我顺手重构了一下。”或者“这个问题修掉了，旁边也清理了一点。”听起来很顺，但对 review 来说，这通常意味着两件事已经混在一起了：一件是行为变化，一件是结构变化。

一旦混在一起，review 就会失去最重要的能力：判断到底是哪一步引入了风险。

所以我会优先要求：

- 能分开的，尽量分开提交
- 重构和行为变化不要在同一个 diff 里乱缠
- 如果必须混写，至少要在说明里说清哪些地方是行为变化，哪些地方只是结构调整

这不是形式主义。分开提交的价值，在于让后续排查有抓手。否则出了问题，你根本不知道是逻辑改坏了，还是结构改动顺手带偏了行为。

## 哪些问题可以后置

review 不是说什么都要当场打死。

有些问题可以后置，但前提是它们真的只是局部问题，而且没有把边界、规则、异常路径和提交边界搞乱。换句话说，后置的只能是“局部可修的小问题”，不能是“结构风险没处理完”的借口。

可以后置的，通常是这些：

- 局部命名还可以再磨，但语义已经足够明确
- 函数还能再拆，但当前改动没有让职责失控
- 注释还能再整理，但不会影响调用方理解
- 局部格式还能再修，但不会掩盖真正的结构问题

后置不等于放过。更准确的说法是：先把最贵的风险拦住，再把低风险的局部问题记下来，等下一次触碰这块代码时再顺手修掉。

## review 的顺序

如果把整套判断压成一条顺序，我会这样排：

1. 先看边界和依赖方向，有没有把系统职责打穿。
2. 再看业务规则有没有重复散落，临时补丁有没有落进长期层。
3. 再看对外接口语义、异常路径和回滚路径是否明确。
4. 再看这次是不是把功能和重构混在了一起。
5. 最后才处理命名、格式、注释、局部微调这些问题。

这个顺序很重要，因为它决定了 review 到底是在保护团队未来的修改能力，还是只是在做局部洁癖。

## 真正该拦的

我现在越来越相信，代码评审最值钱的地方，不是指出更多“看起来不对”的地方，而是尽早拦住那些会不断制造利息的结构问题。

如果一个 PR 让边界更糊、规则更散、异常更不清、提交更难拆，那它就应该被拦。反过来，如果它只是某个局部还可以再修得更漂亮，但整体风险已经被控制住，那 review 就没必要在细枝末节上耗太多精力。

`代码评审真正该保护的，不是代码长得像不像样，而是团队未来还能不能低成本、低风险地继续改。`
