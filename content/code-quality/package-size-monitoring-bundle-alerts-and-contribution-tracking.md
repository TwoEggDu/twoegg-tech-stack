很多团队对包体问题的管理方式，直到发版前都还是：

- 这次包是不是又大了
- 看看 APK / IPA 最终大小
- 如果临近提包再紧急清一轮资源

这种做法的问题不是“大家不重视包体”，而是：

`团队真正监控到的，往往只是一个太晚、太粗、也太难归因的结果数。`

所以这篇真正要讲的，不是“怎么瘦包”，而是：

`在真正开始优化之前，包体大小应该怎样被拆解、被告警、被归因、被追踪。`

## 先给一句总判断

如果把整件事压成一句话，我会这样说：

`包体大小监控真正要解决的，不是知道“最终包有多大”，而是让每一次增长都能回答三个问题：长在了哪里、是谁带进来的、是否已经超过预算。`

这也是为什么监控永远应该先于优化。

## 一、包体不是一个数字，而是几层成本叠在一起

很多团队最早只盯一个数：

- APK / AAB 大小
- IPA 大小

这个数当然重要，但它太粗，粗到很多行动都无法从它直接推出来。

更稳的拆法通常至少要分成下面几层：

1. `安装包主体`
2. `动态资源 / Bundle / Asset Pack`
3. `引擎与原生库`
4. `Managed 代码和 stripping 结果`
5. `符号、mapping、调试产物`

只有拆开以后，团队才会发现很多“包变大了”其实不是同一件事：

- 有时是资源涨了
- 有时是 shader / native lib 涨了
- 有时是符号没分离
- 有时是某个活动资源进入了错误的交付层

所以包体监控的第一步，不是看总量，而是先把成本结构显式化。

## 二、总量监控只能告诉你“变大了”，不能告诉你“为什么”

总量仍然要看，但只能算第一层信号。

更稳的包体监控通常至少要同时看：

- 总包大小
- 关键 bundle / 关键 group 大小
- 安装包内资源占比
- 原生库和 Managed 代码占比
- 关键平台差异

原因很简单：

`总量异常只是事故入口，真正有价值的是能不能快速定位变化来自哪一层。`

如果没有这一层拆解，团队每次只能重复同一种低效动作：

- 猜测是不是资源
- 猜测是不是代码
- 猜测是不是某次合并顺手带了太多东西

## 三、关键 bundle 告警比“总量超线”更早发现问题

很多时候，总包还没明显变大，但问题其实已经开始了。

例如：

- 某个首包 bundle 明显膨胀
- 某个活动资源错误地进入常驻组
- 某个共享 bundle 的依赖数量突然上涨

这也是为什么只看总量很容易太迟。

更稳的做法，通常是给下面几类对象单独立告警：

- 首包相关 bundle
- 常驻 bundle
- 共享资源 bundle
- 当前版本新增的 bundle
- 某些历史上高波动的 bundle

也就是说：

`总量看的是全局健康，关键 bundle 看的是结构退化。`

## 四、贡献度追踪才是真正能推动行动的那层

监控系统最容易停在“知道变大了”，但真正能让团队动起来的，通常是贡献度视图。

所谓贡献度追踪，本质上是在回答：

- 哪个目录、模块、资源组对本次增长贡献最大
- 哪个团队或 owner 带来的变化最明显
- 这次增长是一次性、结构性，还是长期趋势

没有贡献度追踪，包体监控最后很容易沦为一句泛泛的话：

`最近几版都在慢慢变大。`

有了贡献度追踪以后，才更容易出现可执行的动作：

- 某个 bundle 由谁负责解释
- 某类资源是否应该拆组或改成按需下载
- 某个模块是否需要做 stripping 或懒加载

所以监控真正要落地，不能只有图表，还要能落到 owner。

## 五、哪些信号最适合进 CI

包体监控不是所有东西都要在每个 PR 上阻断。

更稳的做法通常会分三档：

### 1. PR 或合并前的轻量信号

更适合看：

- 明显异常的大文件引入
- 某些禁止进入安装包的资源是否误入
- 基础命名、分组、导入规则

### 2. 合并后 / Nightly 的完整包体报告

更适合产出：

- 总包体变化
- 关键 bundle 变化
- 资源 / 代码 / 原生库占比
- 贡献度排名

### 3. Release Gate 的硬预算

更适合阻断：

- 总包超过绝对阈值
- 首包或常驻资源超过预算
- 某些关键 bundle 异常增长

这样做的好处是：

- 合并前足够轻
- Nightly 足够完整
- 发版前有明确门槛

## 六、最常见的假包体监控

最常见的“看起来在监控，实际上很难行动”的情况通常有这些：

### 1. 只看最终安装包大小

这样最多只能说明结果，不足以指导修复。

### 2. 只看一次，不看趋势

包体问题很少是一夜之间爆炸，更多时候是慢慢被吞掉的。

### 3. 没有关键 bundle 视角

总量平稳，不代表首包和关键资源结构没坏。

### 4. 没有贡献度归因

知道“变大了”但没人知道该谁处理，这个系统很快就会失去威慑力。

### 5. 没有绝对阈值和相对阈值

只看绝对值会让退化被慢慢吞掉，只看相对值又会失去长期总量控制。

## 一条最小可执行路径

如果团队现在还没有成型的包体监控，我会建议先按下面这条最小路径落：

1. 先把包体拆成安装包、关键 bundle、代码 / 原生库三层，而不是只看总数。
2. 先给首包、常驻组和共享组立单独报告。
3. 先做最近稳定版本对比，不追求一开始就有完整历史系统。
4. 先给总包和关键 bundle 同时设绝对阈值与相对涨幅阈值。
5. 再补贡献度视图，把异常增长落到目录、模块或资源组 owner。

只要这五步站住，后面的优化才会从“大家都知道该瘦包”变成“知道该瘦哪里、为什么、谁来做”。

## 结尾

所以我现在更愿意把包体大小监控理解成这样一套系统：

`先把总量拆成可解释的成本结构，再给关键 bundle 和关键交付层立单独告警，最后把每次增长追到具体模块和 owner；只有这样，包体问题才不会继续停留在发版前的最后一分钟，而会变成持续交付里能被看见、能被解释、能被追责的长期信号。`
