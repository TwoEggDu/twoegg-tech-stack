很多团队都知道包体、加载时间、性能和 Crash 很重要，但真到流程里，常见的说法却还是这些：

- 这次好像大了一点，但还能发
- 加载似乎慢了一点，先观察
- FPS 看起来掉了点，但不是每次都能复现
- Crash 好像多了点，等线上再看

这些判断单独看都合理，叠在一起就会把系统慢慢拖坏。

所以 Baseline 真正要解决的，不是“多跑几次性能测试”，而是把这些变化变成可比较、可告警、可阻断的预算。

`Baseline 不是偶尔跑一下的观察报告，而是一条明确写着“测什么、在哪测、超多少算坏、坏了以后怎么办”的门槛。`

## Baseline 不是“偶尔跑一下”

一个真正能用的 baseline，至少要把五件事说清楚：

- 指标是什么
- 样本场景是什么
- 采样环境是什么
- 阈值怎么定
- 超线以后谁处理、是否阻断

这五件事里最容易被忽略的，不是指标，而是后面四个。
因为如果样本不固定、环境不固定、阈值不固定、超线没有动作，那结果再多也只是参考，不是预算。

所以我更愿意把 baseline 看成“预算 + 采样方法 + 超线动作”的组合，而不是单次跑分。

## 哪几条线最值得先立

如果团队现在还没有 baseline，我会优先立这四类：

| 类别 | 最小样本 | 适合看的阈值 | 更适合放在哪 |
|------|----------|--------------|--------------|
| 包体 | release 包 / 关键 bundle | 绝对上限 + 相对涨幅 | 合并后 / Release gate |
| 加载 | 冷启动、主入口、关键场景切换 | 上限值 + 相对涨幅 | Nightly / Release gate |
| 性能 | 固定场景、固定机型、固定构建模式 | Frame time / FPS 预算 | Nightly / Release gate |
| 稳定性 | 自动化 smoke、符号产物、灰度版本健康 | 零 fatal + 线上阈值 | CI smoke + Release gate + 线上回流 |

这四类之所以排在前面，不是因为它们最炫，而是因为它们最容易持续付利息：

- 包体一旦失控，交付、下载、更新和存储成本都会一起上涨
- 加载一旦拉长，首日体验和转化率都会被拖累
- 性能一旦变差，后面每次修都比早拦更贵
- 稳定性一旦没有预算，团队最后就会退回“看起来还行”的主观判断

## 阈值不能拍脑袋

Baseline 最大的坑，不是完全没有线，而是线画得很随意。

如果你只说“包体不要太大”“加载不要太慢”“FPS 不要掉太多”，这还不是预算，只是愿望。

更稳的做法，是同时有两条线：

- 绝对上限
- 相对涨幅

绝对上限解决的是“再怎么解释也不能再涨了”。
相对涨幅解决的是“虽然还没撞上总上限，但这一轮已经明显退化了”。

例如：

- 包体总量不能超过某个值，同时单次涨幅不能超过某个百分比
- 主入口加载时间不能超过某个秒数，同时相对上一个稳定版本不能慢超过某个比例
- 固定场景平均帧时间不能超过某个预算，同时不能比基线版本恶化超过某个百分比

只看绝对值，退化会被慢慢吞掉。
只看相对值，长期总量又容易失控。

## 样本和环境必须固定

很多 baseline 最后失效，不是因为团队不重视，而是因为测量条件一直在变。

比如：

- 今天测 Debug，明天测 Release
- 今天测编辑器，明天测真机
- 今天测空场景，明天测活动入口
- 今天测冷启动，明天测热启动

这种结果不能横向比较，更不能拿来做门禁。

所以每条 baseline 至少要固定三件事：

- 固定场景 / 路径
- 固定设备 / 设备组
- 固定构建模式

只有这三件事固定了，趋势才有解释力。
否则所谓“这次慢了一点”，很可能只是环境变了，而不是版本真的退化了。

## 怎么把它们真正接进 CI

Baseline 真正难的地方，不是定义指标，而是接进流程。
不同指标，适合放的阶段并不一样。

### 包体：最适合尽早进构建链

包体是四类里最适合早进 CI 的。
因为它稳定、可枚举，而且和构建产物天然绑定。

更稳的做法通常是：

- 每次合并后或 nightly 构建都输出体积报告
- 对 release 包和关键 bundle 做对比
- 同时看总量和单次涨幅
- 超过阈值时直接阻断 release，必要时阻断主干合入

包体问题如果只在发版前才看，几乎总是来不及低成本修。

### 加载：固定关键入口，不要追求全量

加载时间最容易掉进一个误区：想一次把所有场景都测了。
结果往往是系统太重、结果不稳定、团队很快放弃。

更稳的方式是先固定几条最值钱的路径：

- 冷启动到可交互
- 登录到主界面
- 主界面到战斗 / 关键玩法入口

先把这几条链做成可重复采样，再逐步扩展。

加载类 baseline 更适合放在 nightly 和 release gate，而不是每个 PR 都阻塞。
因为它对环境更敏感，也更需要稳定样本。

### 性能：固定场景、固定机型、固定预算

性能 baseline 最怕两件事：

- 场景不固定
- 设备不固定

这样测出来的数据没有可比性，最后团队只会把它当背景噪音。

所以性能 baseline 要尽量收窄：

- 先选 1 到 3 个固定场景
- 先选一组固定机型或测试环境
- 先只盯最核心的预算，比如平均帧时间、长帧数、峰值内存

这类检查通常更适合 nightly 和 release gate。
把它强塞进每个 PR，只会让团队开始想办法绕过流程。

### Crash：CI 不会替你知道线上崩溃率，但它能先拦很多前置问题

Crash 是四类里最容易被误解的一类。
因为真正的线上 crash rate 当然不可能在 CI 里被完全预测出来。

但这不代表 crash 预算不能进流程。
更准确的说法应该是：

- CI 负责拦可以前移的稳定性信号
- Release gate 负责拦发版前的稳定性准备
- 线上监控负责拦真实世界里的版本健康阈值

前两层适合检查的，通常包括：

- 自动化 smoke 过程中是否出现 fatal crash
- 关键启动链和关键入口是否稳定跑通
- 符号文件、映射文件、崩溃分析产物是否齐全
- 灰度和回滚条件是否准备好

真正的线上 crash rate，则应该继续作为 release 后的健康门槛，回流到下一轮流程里。

所以 crash baseline 不是“在 CI 里算出线上崩溃率”，而是把可以前移的稳定性预算尽量前移，把不能前移的那部分继续接进 release gate 和线上监控。

## 最常见的假 baseline

Baseline 看起来在跑、实际上没什么工程意义，通常是下面几种情况：

### 1. 场景不固定

今天测 A 路径，明天测 B 路径，数据没有连续性。

### 2. 环境不固定

Debug / Release、编辑器 / 真机、热启动 / 冷启动混着测，最后谁也解释不清变化来自哪里。

### 3. 只有结果，没有阈值

报表很好看，但没有任何“超多少就要动作”的规则。
这不是预算，只是观察。

### 4. 超线没有动作

就算大家都看见退化了，如果没有明确的 owner、修复时限和阻断策略，baseline 很快就会失去威慑力。

### 5. 把所有 baseline 都塞进 PR

很多 baseline 天生更适合 nightly 或 release gate。
强行塞进 PR，只会让流程变慢、结果变噪，最后大家一起学会忽略它。

## 一条最低可执行路径

如果团队现在想从零开始，我会建议先按这个最小版本落：

1. 先立 release 包和关键 bundle 的体积预算。
2. 先立两条最关键加载路径的时间预算。
3. 先立一个固定场景、固定设备组的性能预算。
4. 先立一条关键启动 / 入口的 crash smoke，以及符号产物完整性检查。
5. 每条预算都写清 owner、阈值和超线后的动作。

这一套先跑稳以后，再逐步扩大场景、设备和指标。
不要一开始就试图覆盖一切，那样很容易把 baseline 做成另一个维护不起的大系统。

## 结尾

所以我现在更愿意把 baseline 压成一句话：

`它不是“偶尔跑一下看看有没有变差”，而是把包体、加载、性能和稳定性这些长期成本，变成固定样本、明确阈值和超线动作，再把它们分别接进 CI、nightly、release gate 和线上回流。`
