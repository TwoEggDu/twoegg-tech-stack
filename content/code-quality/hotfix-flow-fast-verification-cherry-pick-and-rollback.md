很多团队一说“热修复”，第一反应还是：

- 赶紧修
- 赶紧 `Cherry-Pick`
- 赶紧发

这套动作听起来很合理，但真正把线上版本搞坏的，往往也正是这里。因为一旦“紧急”变成主导词，团队最容易牺牲掉的不是速度，而是边界：

- 这次到底只修什么
- 哪些验证可以压缩，哪些不能跳
- 回滚抓手是否已经准备好
- 这个修复会不会只在发布分支存在，最后和主干再分叉一次

`热修复真正要解决的，不是“怎样更快发一个修”，而是“线上已经出问题时，怎样在压缩验证路径的同时，仍然保持版本可控”。`

这也是为什么它应该紧接在 [灰度发布：分批放量策略（1% → 10% → 全量）、关键指标监控窗口、自动回滚触发条件]({{< relref "code-quality/gradual-rollout-and-automatic-rollback.md" >}}) 和 [线上质量监控：崩溃率 / ANR / 卡顿的实时监控告警、版本健康看板、问题响应 SLA]({{< relref "code-quality/online-quality-monitoring-and-version-health-dashboard.md" >}}) 后面。前两篇解决“怎么发现问题”和“什么时候收回来”，这一篇解决“收回来之后怎么安全地修”。

## 热修复流程到底在解决什么

正常发布节奏解决的是“怎样有计划地交付版本”，热修复流程解决的是另一种问题：

`线上事故已经发生，或者已经足够接近事故阈值，团队怎样在不等下一个常规版本的前提下，把修复安全地送到受影响版本上。`

这里真正麻烦的地方有三层：

- 时间被压缩了
- 风险更高了
- 允许犯错的空间更小了

因为热修复本身通常就发生在版本健康已经不稳的时候。如果这时再把流程退化成“先修再说”，那团队只是把一次线上问题换成了第二次线上问题。

## 热修复的最小单位，不是“一个 commit”

很多团队把热修复理解成“把一个提交挑到发布分支”。这个视角太窄了。真正能被安全执行的热修复，最小单位至少要包括下面这些信息：

| 维度 | 必须先说清什么 |
|------|----------------|
| 问题范围 | 影响的是哪条关键路径、哪类用户、哪个版本段 |
| 目标版本 | 要修哪一个线上分支、哪些分渠道包、哪些热更快照 |
| 改动边界 | 这次只修什么，不顺手带什么 |
| 验证目标 | 哪几个入口、哪几个设备组、哪条回归链必须过 |
| 回滚抓手 | 如果修复失败，退回哪个包、哪个配置、哪个资源快照 |

没有这五件事，后面的 `Cherry-Pick`、验证、上线动作都会变成临场拍板。

所以热修复的最小单位，从来都不是“一段代码”，而是：

`一个明确问题范围上的受控修复包。`

## 快速验证通道，不等于跳过验证

热修复流程里最容易出现的误判，就是把“快速验证”理解成“少做验证”。更准确的说法应该是：

`热修复不是放弃验证，而是把验证从“尽量全”收缩成“对准这次事故最值钱的验证链”。`

一个能用的快速验证通道，至少要保住三层：

### 1. 构建与产物完整性不能跳

不管多紧急，都至少要确认：

- 目标分支能稳定构建
- 包体、资源、热更产物版本是一致的
- 符号文件、mapping、构建日志被正确归档
- 这次修复对应的发布元信息可追踪

否则就算问题修好了，后面再出问题，团队也会失去回查能力。

### 2. 目标回归不能跳

热修复不需要把全量回归重新跑一遍，但必须把“受影响链路”打透。最常见的最小集合通常包括：

- 事故入口的复现路径
- 修复后最直接的成功路径
- 一到两条最容易被联动打坏的相邻路径
- 受影响平台 / 设备 / 账号类型的针对性检查

如果这次修复的是支付崩溃，就别把时间花在一堆无关 UI 浏览上；如果这次修复的是登录链，就不要只验证主界面能不能进。

### 3. 上线动作和回滚动作也要验证

很多团队把“修复代码能跑”当成热修复验证完成，但热修复真正落地还包括：

- 配置切换是否能生效
- 热更资源是否能正确下发
- 灰度 / 白名单是否能受控放量
- 失败时能否切回上一个稳定版本

这部分如果不提前演练，线上真正出二次事故时，团队常常会发现自己只准备了“修复”，没有准备“撤回”。

## Cherry-Pick 不是“把修复抄过去”

`Cherry-Pick` 本身不是风险，它只是一个动作；真正的风险来自团队把它当成了“热修复就该这样做”的默认答案。

更稳的使用原则通常是：

- 只挑最小必要提交，不把重构、格式化、顺手优化一起带过去
- 如果一个修复依赖一长串前置改动，就不要硬挑，改成显式热修复分支整合
- 记录清楚 `主干提交 -> 热修复提交 -> 发布版本` 的映射关系
- 热修复完成后，必须把修复回灌到主干和后续发行分支
- 如果修复逻辑在两个分支上已经明显分叉，就不要假装它还是同一个 patch

很多线上问题不是被修复本身打坏的，而是被“发布分支有一套、主干又有一套”的双轨状态慢慢养出来的。

所以 `Cherry-Pick` 最重要的不是会不会敲命令，而是始终守住一句话：

`它只应该复制最小修复，不应该复制分支分裂。`

## 回滚预案必须先于热修复存在

如果一个团队只有“发布动作”，没有“回退动作”，那它的热修复流程其实并不成立。

一个可执行的回滚预案，至少应该提前准备好：

- 上一个稳定包和对应资源快照仍然可用
- 热更配置可以切回
- 服务端兼容位或开关能在修复失败时降级
- 数据迁移不会把旧版本直接卡死
- 灰度、告警、看板可以跟着切换到回滚后的版本视角

注意，这里说的不是“理论上可以回滚”，而是“现在就知道回哪一版、谁来回、多久能回、回完看什么指标”。

热修复的工程价值，很多时候恰恰体现在这点上：

`不是把问题修掉，而是先保证即使修不好，团队也还能退。`

## 最常见的假热修复

看起来在做热修复，实际上只是把线上风险继续放大，通常有下面几种：

### 1. “紧急”变成跳过验证的理由

最后的结果通常不是更快恢复，而是第二次事故来得更快。

### 2. 一个 hotfix 顺手带三四件事

只要这次顺手把别的优化、资源调整、配置改动一起塞进去，修复边界就已经失控了。

### 3. 只挑到发布分支，不回灌主干

下一轮正常发版时，老问题会用另一种形式回来。

### 4. 回滚写在文档里，但没有预演

真正触发回滚时，团队才开始现找稳定包、现确认配置、现猜监控面板。

### 5. 热修复流程没有 owner

没有发布 owner、验证 owner、回滚 owner，最后就会退化成群里一堆人临场协调。

## 一条最小可执行路径

如果团队现在还没有成型的热修复流程，我会建议先按这条最小路径落：

1. 先定义什么级别的问题会触发热修复，而不是等事故发生后再争论。
2. 先固定热修复单模板：问题范围、目标版本、改动边界、验证链、回滚抓手五项必填。
3. 先给发布分支准备一条单独的快速验证通道，只保留构建完整性、目标回归和上线 / 回滚演练。
4. 先规定 `Cherry-Pick` 的边界：只带最小修复，禁止顺手捎带其他改动。
5. 先把回滚动作做成可执行预案，而不是停留在“必要时回滚”。
6. 每次热修复结束后，必须回灌主干，并把这次事故至少回写成一条新的 gate、smoke 或 regression。

这条线一旦站住，热修复才不会只是“加急发布”，而会变成版本健康闭环的一部分。

## 结尾

所以我现在更愿意把热修复流程理解成这样一条链：

`线上问题先被监控和灰度机制发现，再由热修复流程把修复边界、快速验证通道、Cherry-Pick 策略和回滚预案固定下来；修复不是目的，可控地修、可控地退、可控地回写主干，才是这条流程真正的工程意义。`

只有这样，热修复才不是“发布节奏失控后的应急动作”，而是团队面对线上事故时仍然能保持秩序的方式。
