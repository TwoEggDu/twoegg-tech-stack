很多团队一提"瘦包"，动作都很熟：

- 删点没用资源
- 压一压图片
- 看看能不能少带几个场景

这些当然能出效果，但如果把包体优化长期理解成"发版前清垃圾"，最后通常都会回到同一种状态：

`这一版勉强瘦下来，下一版又慢慢长回去。`

所以这篇真正要讲的，不是"列一份瘦包技巧清单"，而是：

`包体优化到底该按哪些成本层来做，以及 IL2CPP Managed Stripping、分包和资产治理各自站在哪一层。`

## 先给一句总判断

如果把整件事压成一句话，我会这样说：

`包体优化真正要解决的，不是让每个文件都再小一点，而是让代码、原生库、资源和分发层都只承载它们该承载的内容。`

这也是为什么优化一定要建立在 [包体大小监控：资产分析、Bundle 大小告警、贡献度追踪]({{< relref "code-quality/package-size-monitoring-bundle-alerts-and-contribution-tracking.md" >}}) 之后。

没有归因，就没有稳定优化。

## 一、先把优化对象拆成四层

包体优化最怕的，是所有问题都拿同一种手段处理。

更稳的拆法通常是：

1. `Managed 代码层`
2. `原生库与引擎层`
3. `资源层`
4. `分发层`

同样是"包大了"，这四层的解法完全不同。

## 二、Managed 代码层：IL2CPP Managed Stripping 的价值是裁掉不用的代码，不是赌博式瘦身

很多团队对 `Managed Stripping` 的态度只有两个极端：

- 要么不敢开，怕裁坏
- 要么一把开到很激进，坏了再补白名单

更稳的理解方式应该是：

`Managed Stripping 不是魔法按钮，它本质上是在帮你收回那些明明不会被用到、却仍然被打进构建里的托管代码。`

这层更适合解决的问题包括：

- 没被真正用到的程序集或代码路径
- 由于反射、序列化、泛型保守保留带来的额外体积
- 第三方 SDK 把很多不用模块一起带进来的情况

### 这层最关键的不是"裁多少"，而是"保什么"

所以更稳的动作通常是：

- 先明确反射、序列化、热更、native 回调等保留需求
- 再逐步提升 stripping 等级
- 把缺失符号、缺失类型、运行时异常回写成保留规则

如果没有这层契约，只追求"裁得更狠"，最后很容易把包体优化变成运行时事故制造机。

## 三、原生库与引擎层：这里更像"平台裁剪"和"模块选择"

很多时候，包大并不只是因为 C# 代码，而是因为：

- 引擎模块启多了
- 某些平台 ABI 一起打进来了
- 原生 SDK 带了不需要的架构或功能

这时真正能起作用的，往往不是继续压图片，而是：

- 平台架构裁剪
- 原生库裁剪
- 引擎模块和第三方 SDK 模块化

也就是说，这一层更像：

`先把平台和依赖世界里"不该带的东西"退回去。`

## 四、资源层：真正最容易失控的，通常还是资产世界

资源层的体积问题，很多时候比代码层更容易长期失控。

常见来源包括：

- 高清纹理、音视频资源误进首包
- 活动资源进了常驻组
- 共享 bundle 过度膨胀
- 重复资源和错误依赖被慢慢养出来

这也是为什么真正稳定的优化，最后几乎一定会落回：

- 分组边界
- 首包 / 常驻 / 按需下载边界
- 压缩策略
- 资源规格治理

所以资源层优化，不该只理解成"压缩资源"，而更应该理解成：

`把资源放回正确的交付层。`

如果一个资源本来就不该在首包里，压得再小也只是错误地更小了一点。

## 五、分发层：Split APK/AAB 和 iOS On-Demand Resources 解决的不是"文件变小"，而是"别让所有人下载同一批内容"

这层特别容易被误解。

像：

- `Split APK / AAB`
- `iOS On-Demand Resources`

这些能力真正解决的，往往不是字节本身变小，而是：

`让不同设备、不同地区、不同入口不必一次性拿到同一整份内容。`

这层更适合解决的问题包括：

- 多 ABI、多分辨率、多语言资源被统一塞进同一初始包
- 某些平台特有资源让所有用户一起买单
- 某些低频内容没有必要进入首次安装体验

所以这层更像交付优化，而不是素材优化。

## 六、不要把资源瘦身、stripping 和分包混成一种动作

这三类手段都能让"包变小"，但它们解决的问题完全不同：

- `Stripping` 解决"代码和符号为什么还在"
- `资源治理` 解决"哪些内容根本就不该在这里"
- `分包 / 按需下载` 解决"不是所有用户都该一次拿全"

如果把这三者混成同一种"瘦包动作"，最后就会出现两个坏结果：

- 优化动作很散，难以复盘
- 问题很容易在下一版换一种形式回来

## 七、优化要和监控、验证、回滚绑在一起

包体优化真正难的，不是想到一批技巧，而是确保：

- 这次优化到底减少了哪一层成本
- 它有没有把下载、加载、资源命中或运行时行为打坏
- 如果策略失败，是否知道回退到哪一版配置和资源快照

所以更稳的流程通常是：

1. 先用监控证明问题在哪
2. 再对准某一层做优化
3. 再用 smoke / regression / baseline 证明没引入副作用
4. 最后把这次优化沉淀成新的长期规则

如果没有这条闭环，瘦包就很容易变成一轮轮短期行为。

## 八、最常见的假优化

最容易让团队觉得"我们在瘦包"，但其实长期收益很弱的，通常有下面几种：

### 1. 只在发版前清一轮资源

没有监控、没有结构治理，下一版几乎一定会长回来。

### 2. 一上来就把 stripping 开到最激进

短期数字可能很好看，后面补例外和修线上会更贵。

### 3. 把应该按需下载的内容继续留在首包里

这不是资源没压够，而是交付边界切错了。

### 4. 只看总包大小，不看首包和关键入口成本

很多用户真正感知到的不是总量，而是首次安装和首次进入的负担。

### 5. 优化后没有做回归验证

瘦包不是免责区，它同样可能打坏加载、variant、资源引用和热更链路。

## 一条最小可执行路径

如果团队现在想把包体优化做成长期能力，我会建议先按下面这条最小路径落：

1. 先用监控把问题拆成代码、原生库、资源和分发四层。
2. 对代码层先做保守的 stripping 提升，不要跳过保留规则梳理。
3. 对资源层先清首包 / 常驻 / 按需下载边界，而不是先做全局压缩。
4. 对分发层优先把多 ABI、多语言、多平台特有内容从统一安装包里拆出去。
5. 每轮优化后都固定跑最小 smoke、加载和关键路径回归。

只要这五步站住，包体优化才会从"发版前一轮冲刺"变成"持续交付中的长期工程动作"。

## 结尾

所以我现在更愿意把包体优化理解成：

`先分清代码、原生库、资源和分发各自的成本结构，再分别用 stripping、模块裁剪、资产治理和按需交付把不该存在的内容退回正确边界；只有这样，瘦包才不是一次性清理，而会变成可以被持续维护的交付策略。`
