很多团队做完机型分档之后，心里会自然产生一种“这事已经完成”的感觉：

- 有了高 / 中 / 低档
- 有了设备识别
- 甚至还有首次 Benchmark 和玩家手动改档

但真到版本推进时，最容易出问题的恰恰不是“有没有档位”，而是：

`这套档位到底有没有命中正确设备。`

因为只要分档错了，后面很多现象都会一起变形：

- 该进 `Mid` 的机型被判进 `High`
- 低档设备拿到了高档资源
- 某组 `GPU` 只在高档阴影下花屏
- 自动判定本身没错，但截图回归、热机验证和发布门禁根本没跟上

所以这篇真正要讲的，不是“怎么再写一套分档策略”，而是：

`怎样把机型分档本身变成一个可验证、可回归、可阻断发布的质量对象。`

前面几篇平台文解决的是：

- 档位怎么判
- 线上怎么修
- 热机后怎么降
- 内容预算怎么跟着档位一起变

而这篇要把这条线接回质量体系，回答的是：

`怎么证明这一切没有配错。`

## 分档验证到底在验证什么

如果只说“验证机型分档”，这件事太模糊了。

我更建议先把它拆成五个对象：

### 1. 判定正确性

回答的是：

- 这台设备为什么被分到这个档
- 结果是不是符合当前规则表预期

### 2. 性能稳定性

回答的是：

- 被分到这个档后，关键场景的帧时间是否站得住
- 热机后会不会持续掉出预算

### 3. 视觉一致性

回答的是：

- 这个档位是不是关掉了不该关的东西
- 同一内容在高 / 中 / 低档之间是否仍然保持可接受的一致性

### 4. 加载与资源一致性

回答的是：

- 该档位到底拿到了哪些资源
- 首载、切场景、热更快照是否和档位边界一致

### 5. 异常退化与安全兜底

回答的是：

- 错档时能不能被快速发现
- 有问题时是否有明确的降档、回滚和阻断动作

也就是说，分档验证不是在问“机器上跑起来没有”，而是在问：

`这个档位的判定、表现、资源边界和退化动作是不是都成立。`

## 设备矩阵要围绕档位来选，不要只围绕销量来选

机型分档验证最容易犯的第一个错误，是矩阵选得太像“常规兼容性测试”：

- 核心收入机型
- 几台低端机
- 几台热门旗舰

这当然有价值，但对分档验证来说还不够。

因为分档更关心的是：

`规则边界附近的设备，是否被分到了正确一侧。`

所以围绕分档设计设备矩阵时，我更建议至少有四类样本。

### 1. 每档代表机

每个档位至少一到两台你长期固定的代表机：

- `High` 代表机
- `Mid` 代表机
- `Low` 代表机

它们负责回答：

`这档的预算表本身是不是成立。`

### 2. 边界机

这是最值钱的一组。

例如：

- 刚好位于 `High / Mid` 临界线附近的机型
- 某代际高端芯片的降级市场机
- 看上去像中端，但分辨率和刷新率更高的机型

它们负责回答：

`规则有没有把最容易判错的设备压到正确一边。`

### 3. 异常机

这组不一定占比大，但一旦出事很容易让你误判：

- 驱动已知异常机
- 发热特别敏感的机型
- 存储老化或低可用空间样本机
- 某品牌系统定制很重的设备

它们负责回答：

`这套规则在现实世界最容易失真的地方，有没有被覆盖到。`

### 4. 长尾观察机

这类设备不必每轮都重跑，但应该进入季度或版本级观察样本。

它们负责回答：

`线上长尾问题有没有被逐步吸回验证体系。`

所以分档矩阵和普通兼容性矩阵最大的区别，不是机子更多，而是：

`它更强调边界机和错档高风险样本。`

## 每个档位最值得固定哪些验证

分档验证最怕做成“每轮随便看看”。

更稳的做法，是给每个档位固定一组最小验证链。

### 第一层：判定日志验证

这是最便宜也最值得最先固定的一层。

至少要让每次测试都能看到：

- 设备指纹
- 本地识别结果
- 首次 Benchmark 结果
- 当前生效档位
- 档位来源（本地 / 远程 / 手动 / 安全兜底）

否则后面看到一切问题时，第一句永远会变成：

- 它到底现在是哪个档

### 第二层：固定性能 baseline

分档验证里，性能不该只看“平均 `FPS` 还行不行”。

更稳的样本应该固定：

- 固定设备
- 固定场景
- 固定构建模式
- 固定资源快照
- 固定观察窗口

然后盯：

- `avg / p95 / p99 frame time`
- 首载到可交互
- 关键场景切换耗时
- 热机后的持续稳定性

这层和 [Baseline：性能 / 包体 / 加载 / Crash 预算怎样立线并进 CI]({{< relref "code-quality/baseline-budgets-in-ci.md" >}}) 是直接连着的。

区别只是：

- 那篇在讲“怎么立基线”
- 这篇在讲“分档验证最该怎么用这些基线”

### 第三层：关键路径 smoke

分档不该只在空场景里测。

至少要固定几条最值钱的路径：

- 启动到主界面
- 主城或主关卡入口
- 一段高特效战斗
- 首次资源切换或热更后入口

这层负责回答：

`这档不仅理论上成立，而且关键路径真的还能活。`

### 第四层：截图回归

分档验证如果没有视觉回归，很容易出现另一种假通过：

- 性能是回来了
- 但画面已经配歪了

更稳的截图回归，不是全世界随便截，而是固定：

- 关键场景
- 固定机位
- 固定时间点
- 固定设备
- 固定档位

然后比较：

- 同设备不同档位
- 同档位不同版本

这层和 [视觉质量测试：感知差异（Perceptual Diff）、渲染 Artifact 自动检测、多平台视觉一致性]({{< relref "code-quality/visual-quality-testing-layered-diff-rules-and-ai-review.md" >}}) 直接相连。

这里先把它当成分档验证的一部分，不展开感知差异算法细节。

### 第五层：热机与长时运行验证

如果项目上的是移动端，这层不能省。

因为很多错档并不会在冷机阶段暴露，而是会在：

- 连续 15~20 分钟运行
- 高负载场景循环
- 后台切回

之后才显形。

如果这层没进验证链，分档很容易在线上才第一次真正坏。

## 错档、漏档和假高配，最常见的症状长什么样

分档问题最麻烦的一点，是现象经常很像别的问题。

我更建议先把几类典型症状固定下来。

| 现象 | 更可能的问题 | 第一优先排查 |
|------|-------------|-------------|
| 某设备只在高档明显掉帧 | 规则表过于激进，或热机后没有及时降档 | 看当前档位来源、热机后帧时间趋势 |
| 某设备自动判高，玩家大量手动改中 | 默认档位偏激进 | 看玩家改档比例和改档后稳定性 |
| 低档性能正常，但画面明显配歪 | 资源分档或 `Shader` 关键字不一致 | 看资源快照与截图回归 |
| 编辑器与高端机都正常，低端机首载才坏 | 资源首载、首次命中或错档叠加 | 看首载路径和关键资源挂载 |
| 同机型不同批次表现差异很大 | 热、老化、电池、系统扰动 | 看冷机 / 热机样本与长时运行结果 |
| 线上某档崩溃率异常高 | 档位规则和特定驱动 / 特性冲突 | 看设备指纹、图形 API、规则版本 |

这张表的价值不在于穷举，而在于先让团队形成一条统一的排查顺序：

`先确认它现在到底在哪个档，再看这个档是不是应该给它，然后再去分性能、视觉和资源边界。`

## 怎样把分档验证接进 nightly 和 release gate

机型分档验证如果只靠人工抽查，很快就会失控。

更稳的方式，是把验证链按节奏拆开。

### 1. PR / 合并前

只放最轻的自动校验，不要把真机重验证全塞进来。

适合做的通常是：

- 分档日志是否带全必要字段
- 配置表和资源快照是否完整
- 明显错误的规则变更是否被静态校验拦住

### 2. Nightly

这是分档验证真正最值钱的位置。

更适合放：

- 代表机的固定 baseline
- 固定关键路径 smoke
- 固定截图回归

因为这些检查：

- 有固定环境
- 不需要每次提交都跑
- 又足够早地发现趋势性退化

### 3. Release Gate

这里更适合放：

- 边界机复核
- 高风险路径复跑
- 热机样本验证
- 当前规则表与线上稳定版本对比

这层的职责不是“再把所有东西测一遍”，而是：

`证明这次候选版本没有把最容易错档、最容易出事故的那几类边界再打坏一次。`

## 一条最小可执行路径

如果团队现在还没有成型的分档验证链，我会建议先从下面这五步开始：

1. 先固定每档至少一台代表机，再加两到三台边界机
2. 先让所有测试都能打印设备指纹、档位和档位来源
3. 先固定三条关键路径 smoke，不追求一上来就全量
4. 先给每档固定一组截图回归机位
5. 先把代表机 baseline 放进 nightly，把边界机复核放进 release gate

只要这五步稳定跑起来，分档就已经不再只是“配置存在”，而是一个真正可验证的系统。

## 小结

如果把这篇文章压成一句话，我会这样收：

`机型分档真正要验证的，不只是“这台机子现在跑在哪个档”，而是这个档位的判定、性能、视觉、资源边界和异常兜底是否都成立；只有把代表机、边界机、固定 baseline、关键路径 smoke 和截图回归一起接进 nightly 与 release gate，分档才算从经验判断变成工程事实。`
