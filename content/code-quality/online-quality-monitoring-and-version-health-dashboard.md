只要版本真正发到线上，质量判断就会进入另一种世界。

这个世界里，很多问题不再表现成“某个测试没过”，而是表现成：

- 某个机型的 crash 突然抬高
- 某个 OS 版本上的 ANR 明显恶化
- 某个渠道的冷启动卡顿变得很重
- 某个版本的核心入口成功率开始下滑

如果团队对这些变化没有稳定、实时、可分诊的观察面，所谓“发版完成”其实只是把判断权交给玩家。

`线上质量监控真正要解决的，不是把更多日志接上来，而是让团队能够持续看见版本健康、及时告警、明确响应，并把线上事故重新前移成下一轮工程护栏。`

## 线上质量监控不是单个工具，而是四层能力

我现在更愿意把线上质量监控拆成四层：

1. 采集
2. 看板
3. 告警
4. 响应

很多团队前两层有一点，后两层几乎没有。
结果就是：

- 数据在，但没人持续看
- 看板在，但没有明确阈值
- 告警在，但没有 owner
- 出了问题，只能靠群里临时拉人

所以线上质量监控不能只回答“我们有没有接 Crashlytics / Bugly / 某个监控 SDK”，而要回答：

`谁在看、看什么、超线以后谁负责、多久必须响应。`

## 版本健康看板最先该盯什么

一开始最容易把看板做得过于铺陈。
指标越多，团队越容易看不出真正决定版本是否健康的那几条线。

如果只选最核心的，我会先盯下面这几类：

### 1. 稳定性指标

- crash 率
- crash-free users / sessions
- ANR 率
- fatal 启动失败率

这类指标回答的是：

`这个版本还活着吗？`

### 2. 流畅性指标

- 卡顿率
- 长帧 / 冻帧比例
- 首帧时间
- 关键场景切换耗时

这类指标回答的是：

`这个版本虽然能用，但是否已经明显变差？`

### 3. 关键路径指标

- 启动成功率
- 登录成功率
- 主界面进入率
- 核心玩法 / 匹配 / 支付 / 活动入口成功率

这类指标回答的是：

`玩家真正关心的关键链路是否仍然成立？`

### 4. 版本分布与维度指标

- 渠道
- 区域
- 设备型号
- OS 版本
- 网络类型
- 包版本 / 热更快照 / 资源版本

这类维度不直接决定版本是否健康，但它决定你能不能定位问题到底集中在哪。

## 看板必须按版本分层

线上看板最怕的是把所有版本、所有渠道、所有人群揉成一个总图。

总图有用，但只适合看大盘。
真正做版本健康判断时，最少要同时有这三层视图：

### 1. 当前发布版本视图

这是灰度期和近期上线后最重要的一层。
它回答的是：

`这次新版本到底有没有把健康线拉坏。`

### 2. 历史稳定版本对比视图

只看当前值是不够的。
你必须知道和上一个稳定版本相比，这次是：

- 绝对变差了
- 只是样本波动
- 还是本来就处于同一个水平

### 3. 维度切分视图

一旦总图异常，就要快速切到：

- 某个机型
- 某个 OS
- 某个渠道
- 某个区域
- 某个热更快照

没有这层，团队最后只能得到一句没法行动的结论：

`好像有些用户变差了。`

## 告警不要等事故“完全成立”才触发

和 [灰度发布：分批放量策略（1% → 10% → 全量）、关键指标监控窗口、自动回滚触发条件]({{< relref "code-quality/gradual-rollout-and-automatic-rollback.md" >}}) 一样，线上告警也不该等到事故已经非常明显才响。

更稳的告警体系，通常至少会有三层：

| 级别 | 触发含义 | 典型动作 |
|------|----------|----------|
| P0 | 已经影响版本可用性或广泛用户 | 立刻响应、冻结放量、准备回滚 |
| P1 | 明显异常，正在持续恶化 | 快速分诊、限制扩散、确认修法 |
| P2 | 趋势异常，但未到阻断 | 建立追踪项、持续观察、纳入下一轮 gate |

也就是说，告警不是只分“响或不响”，而应该分“响了以后需要多快、谁来处理、是否影响版本推进”。

## 什么叫实时告警

很多团队以为“实时”就是秒级推送。
但真正重要的不是推得多快，而是：

- 能不能在版本还可控时发现
- 能不能在负责的人还来得及处理时发现

如果一条告警每次都在事故已经被玩家大面积感知以后才来，那它技术上也许是实时的，工程上却已经太晚了。

所以更实用的做法通常是：

- 对 crash / ANR / 启动失败这种硬健康指标做更快告警
- 对卡顿、加载、长尾趋势做分钟级或小时级聚合
- 对版本分布、特定设备异常做聚合后告警，而不是单点噪声直推

## 问题响应 SLA 不是管理术语，而是工程时钟

只要线上告警没有响应时限，它就很容易退化成“有人看到了再说”。

所以我会建议至少把下面几件事写成明确规则：

### 1. 谁负责确认告警真假

是值班 owner、客户端负责人、基础设施 owner，还是 QA / 发布 owner？

### 2. 多久必须确认

例如：

- P0：15 分钟内确认
- P1：30 分钟到 1 小时内确认
- P2：当天内确认

### 3. 多久必须给出动作

动作不一定是“已经修好”，但至少应该包括：

- 暂停灰度
- 限流
- 回滚
- 开启降级开关
- 确认误报后关闭告警

### 4. 谁负责把结果回写

如果一条线上问题最终没有沉淀成：

- 新阈值
- 新告警
- 新 smoke
- 新 regression
- 新 baseline

那它下次很可能还会以同一种形式回来。

## Crash、ANR、卡顿三类问题，处理姿势并不一样

这三类指标经常被放在一起，但处理方式并不相同。

### Crash

Crash 更适合快速阻断和快速符号化。

关键是：

- 符号文件是否完整
- 崩溃聚类是否稳定
- 是否能快速确认是否为新版本引入

### ANR

ANR 更像一种“交互被冻结”的长期质量信号。

它往往比 crash 更依赖：

- 设备和 OS 维度
- 场景和入口维度
- 主线程阻塞、磁盘 I/O、资源加载链的关联判断

### 卡顿

卡顿问题最怕噪声。

它必须尽量绑定：

- 固定入口
- 固定设备组
- 固定版本区间

否则最后很容易退化成“好像最近都觉得不太顺”。

## 最常见的假监控

看起来有监控，实际上工程价值很低，通常是下面几种：

### 1. 只有总量，没有版本视角

看板能看总 crash，但看不出是哪个版本开始抬头。

### 2. 只有告警，没有 owner

消息发出来了，但没人对“谁来先看、谁来拍板、谁来回滚”负责。

### 3. 只有采集，没有分诊

日志和聚类很多，但没有稳定的版本健康视图和问题优先级。

### 4. 只有监控，没有回写

线上事故每次都靠经验处理，却没有前移成新的 gate 或新的 smoke。

### 5. 只有工具，没有时钟

工具很完整，但没有明确的响应 SLA，最后依然是临时协调。

## 一条最低可执行路径

如果团队现在还没有一套真正站得住的线上质量监控，我会建议先这样落：

1. 先给每个发布版本单独做版本健康视图。
2. 先把 crash、ANR、卡顿、启动成功率、关键入口成功率做成最小看板。
3. 先给 P0 / P1 / P2 三档告警写死触发条件和 owner。
4. 先把告警确认时限和动作时限写成 SLA。
5. 每次线上事故都至少回写一条新的 gate、smoke、regression 或 baseline。

这一套先跑稳以后，再扩展到更细的业务指标和更复杂的设备矩阵。

## 结尾

所以我现在更愿意把线上质量监控理解成这样一条链：

`先把版本健康看板站住，再把 crash / ANR / 卡顿和关键路径做成实时告警，再把告警接到明确的 owner 和响应 SLA 上，最后把每次线上事故重新前移成新的工程护栏。`

只有这样，监控才不是“发版后的观察面板”，而是版本健康闭环的一部分。
