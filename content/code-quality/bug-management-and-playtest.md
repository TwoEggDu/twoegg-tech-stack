很多团队说自己"有 Bug 管理"，实际状态更接近下面这些：

- 有个工单系统
- 大家会提问题
- 有人偶尔分一下优先级
- 发版前再组织几个人玩一会儿

这些动作当然都不是错的，但它们离"闭环"还差得很远。因为真正让版本越来越稳的，不是"问题有没有被记下来"，而是：

- 问题有没有被稳定分级
- 有没有被复现到可执行
- 修完以后谁负责验证
- 验证完以后有没有回写成下次不会再漏掉的护栏

`Bug 管理真正要解决的，不是"把问题记账"，而是把问题从噪音变成可分诊、可追踪、可验证、可关闭的交付对象。`

Playtest 也是一样。它不是"多找几个人玩玩"，而是把高风险版本放进有目标、有脚本、有观察点的试运行。

## 这篇到底在解决什么

如果说 [热修复流程：紧急修复的快速验证通道、Cherry-Pick 策略、回滚预案与演练]({{< relref "code-quality/hotfix-flow-fast-verification-cherry-pick-and-rollback.md" >}}) 解决的是"线上出问题怎么安全修"，那这一篇解决的是：

`问题被发现以后，团队怎样把它接回日常的分诊、验证和发版流程，而不是只靠一次次临时协调。`

这条链一旦没有站住，团队很容易进入一种反复消耗的状态：

- 同样的问题每个版本都有人再提一次
- 严重级别总是按谁声音大来算
- "已修复"其实只是代码合了
- Playtest 做了很多，但最后没留下稳定证据

## P0 / P1 / P2 不是情绪级别，而是决策级别

很多团队表面上有严重级别，实际上定义却非常模糊。更稳的做法不是造很多层，而是把少数几档写死。

如果只保留 `P0 / P1 / P2` 三档，一个够用的判断方式通常是：

| 级别 | 判断标准 | 典型动作 |
|------|----------|----------|
| P0 | 阻断发布、影响核心可用性、造成大面积崩溃 / 登录失败 / 支付失败 / 数据错误 / 安全风险 | 立即升级、冻结放量、走热修复或回滚 |
| P1 | 核心功能明显受损，但影响面或严重性还没到必须立刻回滚 | 尽快进入修复计划，优先挤进最近版本 |
| P2 | 有明显缺陷，但存在绕行方式，或者影响范围较窄、风险可控 | 进入常规排期，持续跟踪 |

真正重要的是：严重级别必须对应动作，而不是只对应一个标签。

如果一个团队标了 `P0` 却没有升级动作、标了 `P1` 却没有修复期限、标了 `P2` 却没人跟踪，那这些级别就只剩装饰作用。

所以 `P0 / P1 / P2` 的本质不是问题有多让人烦，而是：

`这个问题会不会改变版本决策。`

## 一个能用的 Bug 生命周期，至少要固定哪些状态

Bug 生命周期不需要复杂，但必须让责任交接清楚。一个最小够用的状态流通常包括：

| 状态 | 这一步要确认什么 |
|------|------------------|
| 新建 | 问题被记录，信息是否完整 |
| 已分诊 | 严重级别、负责人、目标版本是否明确 |
| 已复现 | 复现条件、设备、版本、资源快照是否稳定 |
| 已修复待验证 | 修复进入目标构建，等待验证 |
| 已验证 | 在目标版本和目标环境中确认问题消失 |
| 已关闭 | 问题闭环完成，必要时已回写 smoke / regression / gate |

这里最容易偷懒的地方是两步：

- `已复现`
- `已验证`

因为很多团队会直接从"有人提了"跳到"程序说修了"，中间缺少稳定复现；又会从"代码合了"直接跳到"关闭"，中间缺少目标构建验证。

如果这两步不立住，Bug 管理最后就会退化成任务状态管理，而不是质量闭环。

## Playtest 不是什么都玩一遍，而是带问题清单去撞风险

Playtest 最常见的问题，不是参与的人不认真，而是目标太虚。

如果只是说"这版大家有空玩一下"，最后收回来的信息通常会很散：

- 有人注意到 UI
- 有人注意到手感
- 有人说某处怪怪的
- 但谁也说不清这一轮到底要验证什么

更稳的 Playtest 应该先回答三个问题：

1. 这一轮最想撞出哪类问题。
2. 哪几条路径最值得花时间。
3. 哪些观察点会决定版本去留。

比如：

- 大版本上线前，重点撞关键入口和新玩法主链
- 活动版本前，重点撞配置切换、入口显隐、奖励发放
- 资源热更前，重点撞首载、缓存、回退和资源版本一致性
- 高风险修复后，重点撞事故原路径和相邻路径

也就是说，Playtest 不是"大范围闲逛"，而是：

`用有限人力把最容易漏掉、但一旦漏掉代价最高的路径先撞一遍。`

## 结构化测试用例真正的价值，不是文档整齐，而是复现稳定

很多人一听"结构化测试用例"，就会想到一堆很重的模板。其实它最重要的不是形式，而是复现质量。

一个值得留下来的 Bug 或 Playtest 用例，至少要把下面这些字段固定住：

- 构建版本 / 资源版本 / 热更快照
- 设备型号 / OS 版本 / 渠道 / 网络类型
- 前置条件：账号状态、功能开关、活动配置
- 操作步骤：尽量可复现，而不是"随便点一会儿"
- 预期结果
- 实际结果
- 证据：截图、视频、日志、时间点

这些信息不是"测试文档好看"的装饰，而是为了保证：

- 程序能复现
- QA 能回归
- 后续版本能复查
- 线上问题能和线下验证链对上

没有这些维度，Bug 和 Playtest 记录就很容易变成一句模糊描述：

`好像有时候会出问题。`

## 最常见的假 Bug 管理

### 1. 严重级别按谁喊得大来定

最后团队得到的不是优先级，而是情绪排序。

### 2. 问题单没有版本、设备、快照信息

程序接到以后第一步不是修，而是重新猜上下文。

### 3. "已修复" 等于 "代码已经提交"

没有目标构建验证，这个状态没有工程意义。

### 4. Playtest 没有目标场景

大家都很忙，也都玩了，但最后没有形成稳定结论。

### 5. Bug 关闭后不回写护栏

同类问题下次仍然会从同一个入口回来。

## 一条最小可执行路径

如果团队现在还没有成型的 Bug 管理和 Playtest 机制，我会建议先这样收：

1. 先把 `P0 / P1 / P2` 三档写死，并绑定对应升级动作和处理时限。
2. 先把问题单的必填字段写死：版本、设备、前置条件、步骤、预期、实际、证据。
3. 先把生命周期收成最小状态流：新建、分诊、复现、待验证、已验证、关闭。
4. 先规定"已修复"不等于"已关闭"，关闭必须基于目标构建上的验证。
5. 先给每个高风险版本配一轮目标明确的 Playtest，不做泛泛试玩。
6. 每次 P0 / P1 事故关闭后，都至少回写一条新的 smoke、regression 或发布 gate。

只要这六步站住，Bug 管理就会从"问题仓库"变成"质量闭环"，Playtest 也会从"大家玩一下"变成版本前的风险撞击面。

## 结尾

所以我现在更愿意把 Bug 管理与 Playtest 理解成这样一条链：

`问题先被稳定记录，再按严重级别决定版本动作；修复不是结束，复现、验证和关闭都必须有明确责任；Playtest 负责在版本真正上线前，带着明确目标去撞那些最值钱的高风险路径。`

只有这样，团队才不是"不断发现问题"，而是在逐步建立"问题怎样被稳定消化"的能力。
