很多人重新打开 ET9 的公开仓库，第一反应都差不多：这还是以前那个 ET 吗，怎么主仓库里看不到多少核心代码了？这个反应很正常，因为 ET9 真正变的，不是“还能不能做 MMO”这种问题，而是它的交付形态、阅读入口和工程边界都变了。

如果这一点不先讲清，后面的源码阅读会从第一步就歪掉。读者会一边在主仓库里找“核心实现”，一边拿 ET7/ET8 时代的资料套 ET9，最后得到一个很混乱的印象：框架好像还在，但又像只剩下一个壳。这个判断不对。

## ET9 当然还是 ET，但它不再以“单仓库就是全部框架”的方式出现

从公开信息看，ET9 关心的问题并没有根本变化。它仍然在做一套 Unity 客户端和 C# 服务端统一技术栈的框架，仍然强调 `Entity`、`Fiber`、Actor 消息、Location、分布式和工程化能力。你在主仓库 README 里依然能看到这些关键词：新的 Fiber 机制、机器人测试框架、服务发现、云原生支持、Package 化、HybridCLR、YooAssets 这些线都还在。

真正变化的是交付方式。ET9 不再默认把“框架本体”全部摊平在主仓库里，而是改成了更明显的 Package 模式。README 直接写了 `ET9 改成了 Package 模式`，并且把 `ET Packages` 当成真正的能力中心之一。也就是说，主仓库开始更像“入口工程 + 初始化工程 + 文档入口”，而不是“单仓库就是全体源码”。

这一步变化，会直接改变你阅读 ET 的姿势。以前你可以默认“顺着一个仓库往下翻就是主线”；现在你必须接受“主仓库只给你入口，核心能力分散在若干 `cn.etetet.*` 包里”。

## 主仓库为什么看起来像个入口工程

看公开仓库里的结构就能发现，这个仓库承担了几件很明确的事：
- 提供运行指南和 Book 文档。
- 提供初始化工程与包接入入口。
- 提供一部分初始化相关代码，例如 `com.etetet.init`。
- 指向 Package 中心，让你去按需接入 `cn.etetet.*` 能力包。

这也是为什么只盯着主仓库会产生错觉。因为主仓库当然有代码，但它的角色已经明显不只是“框架实现所在处”，而更像“把一整组包接成可运行工程的总入口”。

一个很直接的证据来自初始化工程里的依赖解析器。`Packages/com.etetet.init/Editor/GitDependencyResolver/DependencyResolver.cs` 在包注册完成后，会扫描 `cn.etetet.*` 包，并把 `Library/PackageCache` 里的相关包移动到工程中继续使用。这说明 ET9 的公开工程不是简单依赖几个 Unity Package 就完了，而是把包获取、包落地和工程装配都纳入了启动流程。这里的主仓库，更像装配现场。

## Package 化之后，ET9 的“真实框架”在哪里

要回答这个问题，最直接的地方不是猜，而是看公开文档里的包目录。`Book/8.2ET Package目录.md` 已经把包地图给得很清楚：
- `cn.etetet.core` 是核心，里面放 Fiber、网络、Entity 等基础能力。
- `cn.etetet.loader` 负责加载 `model / modelview / hotfix / hotfixview`。
- `cn.etetet.actorlocation` 承接 Actor Location 机制。
- `cn.etetet.login` 给出前后端登录流程样板。
- 后面还有 `move`、`unit`、`db`、`yooassets`、`router`、`watcher`、`statesync`、`lockstep` 等能力包。

这张图一出来，ET9 的阅读入口就清楚了。你不该再问“核心代码为什么不在主仓库”，而要问“在这一组包里，哪几块才是当前公开可证的主脊梁”。对我们这组系列来说，答案就是 `core + loader + actorlocation + login` 这四条主线先行。

也正因为这样，ET9 的源码阅读顺序必须重排。先看主仓库的运行指南与初始化逻辑，再转入公开包建立运行时地图，最后才沿着登录链路、ActorLocation、Loader、Entity 这些能力往下拆。用旧时代“从仓库根目录逐层读代码”的方法，效率会很低。

## 这不是单纯的目录变化，而是工程边界被前置了

很多人看到 Package 化，会把它理解成“仓库组织方式变化”。这个理解不够。对 ET9 来说，Package 化同时意味着三件事：
- 模块边界被显式写出来了。
- 版本和分发边界被显式写出来了。
- 阅读边界和证据边界也被显式写出来了。

模块边界很直观。不同能力不再默认卷在一个代码树里，而是按包拆开。版本与分发边界也很直观。Package 中心成为真实分发入口，哪些包公开、哪些包收费、哪些包只是生态扩展，边界都比过去更明确。最容易被忽略的是阅读边界：公开仓库、公开包、课程版、付费包之间的证据层级已经明显不同，读者不能再把它们混成同一强度的“源码事实”。

这点对写 ET 文章尤其重要。我们后面会持续区分三种东西：公开源码可证、公开文档可证、生态层可知但不直接做源码断言。ET9 之所以必须这样写，正是因为它的 Package 化把边界摆到了台面上。

## ET9 变了什么，又没变什么

如果非要把 ET9 的变化压成一句话，可以这么说：它没有放弃自己原来的问题空间，但它换了一种更工程化、更模块化的交付方式。

没变的是：
- 它依然是 Unity 客户端和 C# 服务端统一栈的框架。
- 它依然在强调 `Entity / Fiber / Actor / Location` 这一套核心抽象。
- 它依然面向 MMO、强联网、分布式和长线工程能力。

变了的是：
- 主仓库不再等于全体框架源码。
- 真正的能力中心转向 `cn.etetet.*` 包。
- 初始化工程与包装配本身成了架构的一部分。
- 公开版、课程版、生态包之间的边界比旧版本更显式。

这组变化的后果很直接：以后再谈 ET9，先问“它现在如何被交付和装配”，比先问“它支持哪些功能”更重要。因为如果入口判断错了，后面所有源码判断都会跟着错。

## 这件事对读者的实际影响是什么

最大的影响不是“资料少了”，而是“阅读方法必须升级”。你不能再把 ET9 当成一个可以靠仓库树平铺阅读的项目，而要把它看成一个由入口工程和一组能力包组成的框架系统。只有先接受这一点，后面去看 `World`、`CodeLoader`、`Entity`、`Session`、`Mailbox`、`Location` 才有正确地基。

第二个影响是版本意识必须更强。现在网上很多对 ET 的讨论来自 ET7、ET8 时代。如果不先把“单仓库时代”和“Package 化时代”分开，读者很容易一边看旧资料，一边读 ET9 包，最后把不同版本的结构硬拼在一起。这也是本系列第一篇要先做“形态纠偏”的原因。

第三个影响是工程判断会更清楚。Package 化之后，你更容易看见 ET9 到底把什么当核心、把什么当扩展、把什么交给生态和课程版。这反而让我们更容易建立稳定主线，而不是在一个巨大仓库里四处捞重点。

## 这一篇真正想留下来的结论

`ET9 当然还是 ET，但它已经不是“单仓库即框架全貌”的 ET 了。` 它的核心问题空间没变，变的是交付方式、工程边界和阅读入口。主仓库现在更像初始化工程与文档入口，真正的框架骨架分散在一组 `cn.etetet.*` 包里。

所以后面的阅读姿势应该很明确：先把主仓库当入口，再把公开包当骨架，再沿着一条可证的业务链路去读。下一篇会继续把这个问题再推进一步：既然 ET9 真正的能力已经分散到包里，那为什么主仓库里还是让很多人觉得“几乎看不到核心代码”？
