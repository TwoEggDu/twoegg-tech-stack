到这里，ET9 的形态、入口和包化边界已经基本立住了。接下来最重要的事，不是继续看更多目录，而是把这些抽象第一次收成一张图。对于这套框架来说，最适合拿来收图的入口不是战斗，也不是移动，而是登录链路。

原因很简单：登录链路会同时经过网络、会话、纤程、消息分发、对象挂接和位置寻址。也就是说，它天然会把 `World / Fiber / Session / Mailbox / Location` 这几个最核心的抽象串起来。只要这一条链路看懂了，后面的源码篇就不再是单个名词，而是一组协作关系。

## 为什么第一张全局图要从登录链路看

如果先去看某个单独系统，比如 `World`、`Session` 或 `ActorLocation`，读者很容易只看到局部定义，看不见它们之间是怎么互相依赖的。登录链路不一样，它是一个完整的过河点：客户端先进入网络世界，再进入账号与网关世界，最后进入真正承载玩家状态的运行时边界。

这条链路的价值不在于“登录”本身，而在于它能把分散的概念第一次放回同一张图里。你会看到：
- 客户端并不是直接跟最终世界对象对话，而是通过自己的网络纤程和会话对象进入系统。
- 服务端并不是一个单点处理全部逻辑，而是分成 Realm、Gate 等不同边界。
- 玩家对象也不是直接写在某个 Socket 回调里，而是挂到 Actor 和 Location 的运行时结构上。

## 先把这条链路的最小结构讲清

客户端侧可以先把它理解成两段：一段是把连接和请求交给独立的网络纤程，一段是拿登录结果继续往后走。`ClientSenderComponentSystem.cs` 和 `Main2NetClient_LoginHandler.cs` 体现的就是这个思路：客户端先通过自己的发送器和网络处理器把登录请求送出去，再根据返回结果切换后续接入路径。

服务端侧则更明显地分成 Realm 和 Gate 两层。`C2R_LoginHandler.cs` 负责的是登录前半段的接入与转交，`C2G_LoginGateHandler.cs` 负责的是进入 Gate 之后的玩家接纳与继续分发。这里最重要的不是每一步业务字段，而是边界：Realm 更像前置分诊，Gate 更像真正接住玩家后续运行时入口。

把这几段拼起来，登录链路就不再是“一个协议”，而是“把玩家从外部连接收进框架世界”的一条运行时通道。

## `World / Fiber / Session / Mailbox / Location` 各自站在哪一层

`Session` 解决的是连接与会话边界。它不是底层 Socket 本身，而是把“这条连接还能不能继续说话”这件事变成可管理对象。

`Fiber` 解决的是调度与隔离边界。登录链路里，网络不应该无边界地和主逻辑混跑，所以它会被纳入独立的运行时上下文。

`Mailbox` 解决的是消息进入 Actor 的入口边界。登录流程并不是把消息随便丢给某个对象，而是进入一个具备串行处理纪律的对象消息入口。

`Location` 解决的是对象寻址边界。玩家真正进入后，不是直接依赖某个物理进程的地址，而是依赖逻辑对象与位置服务，把“我在找谁”与“它现在在哪”分开。

`World` 则是这一整套边界的总容器。它不是单独的一层业务对象，而是把这些系统挂起来、组织起来、收起来的运行时骨架。

## 这条链路真正说明了什么

最关键的结论不是“ET 有登录模块”，而是“ET 会把一条业务链同时当作运行时链、对象链和分布式链来处理”。登录链路把这些层级第一次串起来，读者才会真正意识到：
- 网络不是只负责发包。
- Fiber 不是只负责异步。
- Actor 不是只负责消息。
- Location 不是只负责找对象。
- World 也不是只负责挂全局变量。

它们合起来才是 ET 的全局图。

## 一个坏理解通常怎么坏

最常见的误解是把登录流程当成普通业务流程，只盯协议字段和返回码。这样写出来的内容会很像接口说明书，但不会让读者理解框架脊梁骨。

第二种误解是把 Realm、Gate、Player 这些角色写成孤立模块，不解释它们如何衔接。这样会让登录链路变成分散名词，读者知道名字，却不知道交接点。

第三种误解是把 `World / Fiber / Session / Mailbox / Location` 五个概念塞成一段“综合介绍”，结果每个都讲了一点，但没有任何一条协作关系被讲透。

## 这一篇真正想留下来的结论

`ET 的第一张全局图，不是某个单独类的说明，而是一条登录链路把运行时、消息、会话和对象寻址串成了一体。`

所以后面再去看 `World`、`Fiber`、`Session`、`ActorLocation`、`Login` 包时，你已经不该把它们当成互不相干的名词了。它们是同一张地图上的不同站点。
