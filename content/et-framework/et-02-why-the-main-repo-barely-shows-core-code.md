很多人把 ET9 的公开仓库拉下来之后，马上会有第二个疑问：就算我已经接受它变成了 Package 化框架，为什么主仓库里还是几乎看不到核心代码？这个疑问比上一篇更具体，也更重要。因为如果这一步不讲透，读者后面会反复陷入一种错觉：主仓库代码这么少，是不是 ET9 公开版其实已经不剩什么了。

这个判断还是不对。主仓库“看起来不像框架本体”，并不等于 ET9 没有骨架，而是说明骨架已经不再用旧时代那种“所有能力摊平在一个仓库里”的方式交付。上一篇我们只做了形态纠偏，这一篇要把边界继续钉死：主仓库、Package 中心、公开包、课程版和生态包分别是什么角色，证据强度又分别到哪里为止。

## 读者为什么会在主仓库里产生“几乎没代码”的感觉

原因其实很简单，读者脑子里默认的阅读模型还是旧框架模型。

旧模型是这样的：拉一个仓库下来，核心目录就在根目录下面，顺着目录树往下翻，基本就能把主要实现读出来。所以当读者打开 ET9 主仓库，看到的是 `README`、`Book`、初始化相关包、若干工程入口，却没有在仓库根部直接看到一整片 `core`、`network`、`actor`、`login` 这种熟悉的大块能力时，第一反应自然就是“是不是核心没开”。

但 ET9 公开仓库从一开始就没有把自己伪装成“一个仓库就是全部框架源码”。`README.md` 反复把读者往几个入口引：
- `ET Packages`
- `运行指南`
- `ET Packages 制作指南`
- 论坛、视频、分析器说明

这套导航本身就在暗示一个事实：主仓库承担的是“入口、说明、装配、初始化”的职责，而不是把所有能力直接平铺给你读。

再看 `Book/1.1运行指南.md`，这个意图就更明显了。运行指南不是让你“打开根目录，直接跑框架本体”，而是要求你：
- 配置 `ET-Packages` 的 scope registry。
- 准备 GitHub token。
- 从 `ET-Packages` 中安装 demo 包。
- 安装后再观察所有 `ET` 开头的包会变成 `Custom` 包。
- 如果没有正确变成 `Custom`，还要执行 `ET -> Init -> RepairDependencies`。

这已经不是传统意义上的“克隆仓库即可读完整源码”流程了，而是一个标准的“入口工程 + 包装配”流程。读者会在主仓库里觉得代码少，不是因为代码真的消失了，而是因为它们被放到了另一个交付层级里。

## 主仓库、Package 中心、公开包，各自到底在干什么

把这三者拆开，ET9 的结构就很清楚。

第一层是主仓库。它负责三件事：
- 给出框架的总入口和版本说明。
- 给出运行指南、包目录、制作指南等文档。
- 提供初始化相关代码，把包真正装配进工程。

第二层是 `ET Packages` 中心。它负责“分发”。也就是说，哪些能力被做成包、哪些 demo 可以安装、哪些生态能力存在、哪些包是公开的、哪些包带价格信息，这一层都会显式出现。它不是单纯的文档目录，而是 ET9 的实际能力清单。

第三层才是公开包源码本身。比如从 `Book/8.2ET Package目录.md` 能看到的这些能力：
- `cn.etetet.core`
- `cn.etetet.loader`
- `cn.etetet.actorlocation`
- `cn.etetet.login`
- `cn.etetet.statesync`
- `cn.etetet.lockstep`
- `cn.etetet.router`
- `cn.etetet.watcher`
- `cn.etetet.memorypack`
- `cn.etetet.hybridclr`

这些包里，才是 ET9 真正的框架肌肉和骨架。主仓库的职责更像“把你带到能力中心，并把能力装进来”，而不是自己把每块实现都展开给你。

如果你把这三层混在一起，就会得出很多错误结论。比如把主仓库没有某个系统的完整实现，误判成 ET9 没有这个系统；或者把包目录里列出的收费包、课程包、生态包，误判成当前公开仓库里已经能逐文件验证的源码事实。ET9 真正难的，不是“代码藏起来了”，而是它把不同层级的东西摆得更近了，读者必须自己学会分层。

## 一个很关键的证据：初始化工程本身就在做“包装配”

如果只停在 README 和运行指南层面，很多读者还是会觉得这是“文档说法”。真正把事情坐实的是初始化工程里的依赖解析器。

`Packages/com.etetet.init/Editor/GitDependencyResolver/DependencyResolver.cs` 做的事情很直接：
- 监听 Unity Package Manager 的注册事件。
- 当新增或变更的包名字以 `cn.etetet.` 开头时，调用 `MoveToPackage`。
- 从 `Library/PackageCache` 中找到对应包。
- 再通过 `MoveToPackages*.ps1` 把这些包移入工程继续使用。
- 如果包状态不对，还提供 `RepairDependencies` 重新修复。

这段逻辑说明了两个问题。

第一，ET9 的主仓库不是“把核心代码省略掉”，而是把“如何把核心包拿进来、修复好、落成本地工程结构”也纳入了框架入口的一部分。也就是说，主仓库少的不是能力，而是“平铺出来给你看的能力实现”。

第二，ET9 的公开工程不是单纯依赖 Unity Package Manager 就结束了。它明显希望把包装配、包落地、包修复都控制在自己定义的流程里。所以运行指南里才会反复强调 `Manage scope registries`、`RepairDependencies`、`Custom 包` 这些步骤。主仓库在这里更像一个装配台，而不是一个源码展示厅。

这也是为什么很多人第一次打开 ET9 会有违和感。因为他们看到的是装配现场，却用看“框架全景剖面图”的期待去读它。

## 公开证据层级必须分开，不然整组文章都会写乱

写 ET9 相关文章时，最危险的不是看不懂，而是证据边界混淆。这里至少要分四层。

第一层是公开源码可证。

这一层最硬。比如主仓库里的 `DependencyResolver.cs`，或者我们已经拉下来的公开包源码：`cn.etetet.core`、`cn.etetet.loader`、`cn.etetet.actorlocation`、`cn.etetet.login`。这类内容可以直接按文件、按调用链、按类关系来做断言。

第二层是公开文档和包目录可证。

比如 `README.md`、`Book/1.1运行指南.md`、`Book/8.2ET Package目录.md`。这一层可以用来判断框架如何交付、安装流程如何设计、官方公开列出了哪些能力包、哪些能力明显属于生态扩展。但这一层不适合直接做“内部实现细节”断言。

第三层是生态层可知，但不能假装等同于当前公开源码。

`Book/8.2ET Package目录.md` 里不只列了核心包，也列了很多生态包、收费包和扩展包，例如：
- `cn.etetet.webgl`
- `cn.etetet.mongodb`
- `cn.etetet.yiuiluban`
- `cn.etetet.yiuinumeric`
- 一整组 `YIUI`、`XET` 相关包

这些信息说明生态边界是公开的，甚至价格边界都是公开的。但这只说明“官方在这个体系里承认它们存在，并把它们放进包地图”，不说明我们现在就应该把这些包当成与 `core` 同强度的源码事实去讲。

第四层是课程版和持续演进版信息。

`README.md` 里其实写得很直白：课程在持续更新，而开源版本不再持续同步更新。这个信息很重要，但它的作用是提醒读者“版本边界不同”，不是授权我们去把课程版的内部实现细节当作公开源码来拆。

这四层如果不分，后面每一篇都可能出问题。最典型的坏写法有两种：
- 把课程版、收费包、生态包的能力描述，写成“当前公开源码就是这么实现的”。
- 把公开包目录里的存在性信息，写成已经逐文件验证的源码事实。

所以这一篇真正要立住的不是“主仓库少代码”这句抱怨，而是“ET9 必须按证据层级来读”这个方法。

## ET9 的阅读路径为什么必须重排

到了这里，阅读路径其实已经很清楚了。

第一步，先读主仓库，但只把它当入口。

这一层该看的，是 `README`、运行指南、包目录、初始化工程。目的是先回答：
- 它怎么交付
- 它怎么装配
- 它把哪些能力显式包化
- 它公开把哪些边界摆到了台面上

第二步，转入公开包，建立主骨架。

对我们这组系列来说，当前最值得做源码主线的就是四块：
- `cn.etetet.core`
- `cn.etetet.loader`
- `cn.etetet.actorlocation`
- `cn.etetet.login`

因为这四块已经足够支撑起 ET9 的第一层全景：运行时容器、代码加载、Fiber 与 Entity、ActorLocation、再加一条真实登录链路。

第三步，再沿着一条业务链路把抽象串起来。

这也是为什么我们后面会把登录链专门提出来讲。因为如果没有一条链路，读者很容易把 `World`、`Fiber`、`Session`、`Mailbox`、`Location` 看成几块彼此并列的概念板。只有走一遍“客户端登录 -> Realm -> Gate -> Player”的过程，读者才会知道这些抽象不是摆设，而是在同一条路径里协作。

第四步，最后才看外围与扩展。

比如 `statesync`、`lockstep`、`watcher`、`router`、`ai`、`yooassets`、`hybridclr` 这些能力，当然重要，但它们不该在读者还没建立核心运行时地图之前就压上来。否则系列会从第二篇开始失控。

## 所以，主仓库“代码少”到底是问题还是设计结果

更准确地说，这是设计结果，不是意外缺失。

ET9 改成 Package 模式之后，主仓库如果还试图承担“展示全部核心实现”的职责，反而会把入口工程、包分发、包修复、文档说明这些新的架构角色又重新卷回一个大仓库里，那就等于形态已经变了，但阅读模型没变。

现在的 ET9 更像这样一套系统：
- 主仓库负责入口、初始化、说明书。
- Package 中心负责分发和能力地图。
- 公开包负责可验证的框架骨架。
- 生态包和课程版则处在另一层边界上。

一旦这样看，主仓库“几乎看不到核心代码”就不再是异常，而是一个非常明确的信号：你已经不该再把主仓库当作旧时代那种“框架本体全展开”的阅读入口了。

## 这一篇真正想留下来的判断

`ET9 主仓库代码少，不是因为框架骨架不见了，而是因为骨架被包化、分发化、装配化了。`

真正要读的，不只是主仓库目录树，而是“主仓库如何把读者带到包地图，再把包装配进工程”这一整套过程。只要这个判断立住，后面很多看似奇怪的地方都会变得合理：
- 为什么运行指南核心步骤是配置包源和安装 demo 包。
- 为什么会有 `RepairDependencies` 这种修复入口。
- 为什么包目录里同时出现核心包、扩展包、收费包和生态包。
- 为什么系列写作必须把“公开源码可证”和“公开生态可知”分开。

下一篇就接着往前走，不再停留在“仓库边界”本身，而是去看一个更具体的问题：为什么 ET9 的安装要求本身，就已经像一份架构说明书。
