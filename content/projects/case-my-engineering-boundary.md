## 一句话总结

知道自己的边界在哪，比声称自己什么都能做更有价值。

我写过 770 多篇技术文章，涵盖 Unity 性能、渲染管线、构建工具链、热更新、CI/CD 等等方向。
但写文章和独立交付是两回事。
能讲清楚一个概念，不等于能在生产环境里扛住这个方向。
这篇的目的就是把这两个概念拆开，让看到的人能直接判断：
我能帮你做什么，不能帮你做什么，以及中间地带该怎么配合。

这不是一篇炫技的文章，也不是一篇谦虚的文章。
它就是一份清单，帮你快速评估我和你的需求是否匹配。
不绕弯子，直接分三层讲。

---

## 能独立交付的（不需要外部协助）

这一层是我最核心的交付能力。
如果你的需求完全落在这个范围内，我一个人就能从 0 到 1 产出结果，不需要额外配合资源。
这不是说我喜欢单干，而是说这些方向我有足够的经验和判断力，交给我不会出意外。
你给我一个目标和时间线，我能自己拆任务、自己推进、自己交付，中间不需要你反复对齐。

### 客户端性能诊断与治理

这是我做得最多、也最有信心的方向。

- **全链路 Profiling**：Unity Profiler、RenderDoc、GPU Inspector，从 CPU 帧耗时到 GPU 管线瓶颈，我能自己完成完整的性能诊断链路。不是点开 Profiler 截个图，而是能追到具体哪个 DrawCall 贵、贵在哪一步、该砍什么。Profiler 里的 GC Alloc 飙了，我能判断是逻辑层的装箱问题还是引擎内部的临时分配，该改代码还是该换 API。
- **分层诊断和治理方案**：帧率问题、内存问题、加载问题，这三类我能分别给出诊断路径和治理方案。不是笼统地说"内存高了要优化"，而是能定位到具体是 Texture 格式问题、Mesh Read/Write 没关、还是 AB 依赖导致的冗余加载。每一类问题我都有成型的排查清单和治理优先级。
- **设备分档策略设计与落地**：高中低档设备该跑什么画质、加载什么 LOD、开哪些后处理，这套策略我能设计并落地到代码里。包括运行时的动态降级逻辑和离线的设备数据库维护。我做过的设备分档方案覆盖过数百款 Android 机型，不是理论上的分类，是实际跑过验证的。

举个具体的例子。
一个项目在低端机上 loading 超过 15 秒，团队的第一反应是"资源太大了，压一压"。
但实际 profiling 发现，瓶颈不在资源大小，而在同步加载阻塞了主线程 -- 几十个 AB 包被串行加载，每个都在等前一个完成。
解决方案不是压资源，而是改加载策略：异步化、分优先级、预加载关键资源。
这种判断不是靠猜，是靠 Profiler 数据和对加载机制的理解。

做这些事情的前提是我对 Unity 运行时的内存模型、渲染管线流程和资源加载机制有足够的理解，不是靠试，是靠知道底层在干什么。
当一个性能问题出现时，我能判断它属于哪一层、该用什么工具看、看到数据后该怎么解读。
这个判断力是写了几百篇分析文章积累出来的，不是背了几个优化技巧。

### 构建与工具链

这是我第二个能独立交付的方向。

- **Unity 构建管线优化**：增量构建、缓存策略、构建产物分析。我经历过从 40 分钟构建缩到 12 分钟的完整优化过程，知道哪些环节能省、哪些不能动。Library 缓存的边界在哪、SBP 的增量构建什么时候靠谱什么时候会坑，这些我有实际的经验判断。
- **AssetBundle 打包策略**：依赖治理、冗余检测、Shader 变体收集与裁剪。这不只是配个打包规则，而是能覆盖从打包到加载到运行时卸载的完整生命周期。我能分析一个 AB 包的依赖图，找出哪些资源被重复打包了、哪些 Shader 变体是多余的、卸载时机该怎么设计才不会内存泄漏。
- **HybridCLR 热更新集成与排障**：从接入到 AOT 泛型补充元数据、到 CI 里自动化 generate all，到线上崩溃排查。我在这个方向写了一整个系列的文章，踩过的坑比大多数人多。ValueTuple 丢方法、async 导致的 native crash、DHE 和社区版的能力差异，这些问题我都写过完整的分析。
- **CI/CD 管线搭建**：Jenkins 和 GitHub Actions 都做过。不只是写个 pipeline 跑通，而是包括构建缓存、产物归档、自动化测试集成、失败通知和回滚机制。我搭的 CI 管线能做到：代码提交后自动跑检查，构建产物自动归档带版本号，构建失败自动通知到对应的人。

构建工具链这个方向，很多团队是"能跑就行"的状态。
出了问题临时修，修完继续跑。
但我做这个方向的思路是把它当成一个产品来设计 -- 有明确的输入输出、有可观测性、有降级方案。
构建挂了不只是"重跑一次"，而是要知道为什么挂、下次怎么避免。

这个方向的核心能力不是会写脚本，而是理解构建系统的全局。
知道改一个打包规则会影响什么、加一个热更模块会引入什么风险、CI 里多加一步检查值不值得。
这种全局判断力是工具链方向最稀缺的能力。
大部分人能写脚本，但不是大部分人能判断一个构建策略的改动会对下游产生什么连锁影响。

### 渲染管线配置

注意这里说的是"配置"，不是"开发"。
我不写渲染引擎，也不写自定义的 SRP。
但我能把 Unity 的渲染管线配置到项目需要的状态，并且知道每个配置选项背后的成本是什么。

- **URP 管线配置**：Quality 分级、Renderer Feature 选择与调参、渲染路径选择。我知道每个选项背后的渲染成本是什么，不是盲调。Forward 和 Forward+ 什么时候该切、SSAO 在低端机上的成本到底多大、额外的 Renderer Feature 会多几个 RT 切换，这些我能给出准确的评估。
- **Shader 变体生命周期管理**：从收集、编译、打包到运行时加载，这条链路我能全程管控。包括变体膨胀的诊断和裁剪策略。一个项目的 Shader 变体从几千膨胀到几万，我能定位到是哪个 keyword 组合导致的，然后制定裁剪方案。
- **多平台渲染适配**：Android、iOS、PC 的渲染差异我清楚，能针对不同平台做 Quality 分级和 Fallback 策略。Mali 和 Adreno 的 Tile-Based 差异、iOS Metal 的特殊限制、PC 上的宽松预算该怎么利用，这些实际的平台差异我在做适配方案时都会考虑。

这个方向我最常做的事情是：拿到一个渲染效果需求，评估它在目标设备上的可行性，然后在 URP 框架内找到性价比最高的实现路径。
如果效果需求本身不合理（比如低端机上要求全屏 Bloom + SSAO + 体积光），我会直接给出替代方案而不是硬做。

说白了，渲染管线配置这个方向，我的角色更像是一个"渲染预算管理者"。
美术和策划提出效果诉求，我负责在目标设备的预算内找到最优解。
能做就做，做不到就给出替代方案和取舍建议。
这个能力需要同时理解渲染原理和性能约束，缺一不可。

### 工程治理

这一块是偏"基建"的工作，不直接产出游戏功能，但决定了项目能不能长期健康运转。
很多项目前期跑得快，后期崩得也快，根本原因就是工程治理没跟上。

- **代码质量门禁**：代码规范检查、静态分析集成、MR 自动检查。不是写个文档说"请大家注意代码风格"，而是用工具把规则落到 CI 里，不合规的提交过不了。规则的设计也讲究分寸 -- 太松等于没有，太严团队会绕过。我能根据团队的实际情况设定合理的门禁线。
- **资源检查器**：特效粒子数检查、Texture 格式检查、Mesh 顶点数检查、Prefab 引用完整性检查。这些我都做过，能根据项目需求定制规则。检查结果能集成到 CI 流程里，也能在编辑器内提供即时反馈。关键是规则不能只是一个数字，要配上具体的原因和修改建议，不然美术不知道为什么被打回。
- **自动化测试框架**：Unity Test Framework 的集成、PlayMode 测试的设计、CI 里的自动化运行。我知道游戏项目做自动化测试的现实约束，不会设计一套理论上完美但实际没人跑的方案。重点是找到投入产出比最高的测试点 -- 核心系统的回归测试、资源加载的 Smoke Test、关键流程的端到端验证。
- **分支策略和发布流程**：Git Flow、Trunk Based、Feature Flag，哪种适合什么规模的团队我有判断。10 人团队和 50 人团队的分支策略应该完全不同，我不会套模板。包括发布前的检查清单、灰度发布策略和回滚机制，这些都是发布流程的一部分。
- **崩溃分析与线上质量监控**：Bugly、Firebase Crashlytics 这类工具的接入和日常监控，崩溃堆栈的解读和定位，我能独立完成。从一条 native crash 堆栈反推到具体的 C# 代码路径，这个过程我做过很多次。线上问题的响应不只是看堆栈，还要判断影响面、优先级和修复策略。

工程治理这个方向，最大的挑战不是技术，是推动。
工具写好了没人用，规则定好了没人守，这才是现实。
所以我做工程治理的原则是：能自动化的不靠人、能卡流程的不靠自觉、能给反馈的不只给数字。
让团队感受到这些基建在帮他们省时间，而不是在添麻烦，这是工程治理能推得动的前提。

## 需要协作的（能参与但不能独立完成）

这一层是我有知识、有判断力、能参与设计和讨论，但不能一个人扛到底的领域。
如果你的需求落在这个范围，我需要对应方向的同事配合。

把这层单独列出来很重要。
因为很多合作的失败不是因为"不会"，而是因为没搞清楚谁负责什么。
我在这些方向上能做 40-60% 的工作，但剩下的部分必须有人补。
如果你以为我能全部搞定而没有配备相应的人，项目到中期一定会出问题。

反过来说，如果你有对应的配合资源，我在这些方向上能起到很好的桥接作用 --
因为我既理解客户端的技术约束，又能理解对方领域的基本逻辑，沟通成本会比纯客户端程序员低很多。

### 服务端架构

- **能做的**：我理解常见的服务端架构模式（微服务、网关、消息队列），能看懂服务端的接口设计，能参与客户端-服务端的协议定义和联调。我写过客户端同步方案的文章，对帧同步和状态同步的差异有实际理解。在联调阶段，我能快速定位问题出在客户端还是服务端，不会盲目甩锅。
- **不能独立做的**：数据库 Schema 设计、高并发场景下的性能调优、分布式事务的处理。这些我有概念但没有生产环境的深度经验。我能和后端同事讨论方案的合理性，但不能替代他们做技术选型和实现决策。MySQL 和 Redis 的基本用法我懂，但让我设计一个支撑百万 DAU 的存储方案，那不是我的强项。
- **协作方式**：我负责客户端这一侧的接口对接和数据结构设计，服务端同事负责内部实现。接口定义的环节我们一起做，避免两边的数据模型对不上。协议文档我能写，也能 Review 后端同事写的。

在实际项目里，客户端-服务端联调阶段出问题的概率非常高。
很多时候不是某一边写错了，而是两边对同一个字段的理解不一致。
我在联调阶段的价值是：能快速判断问题出在哪一侧，能用后端同事听得懂的语言描述客户端的需求，减少来回扯皮的时间。

### 技能/战斗系统

- **能做的**：技能系统的架构设计我有经验。数据模型怎么定义、执行链怎么组织、Effect 和 Buff 怎么分层、网络同步方案怎么选，这些我能给出完整的技术方案。技能编辑器的工具设计也在我的能力范围内 -- 怎么让策划能用可视化工具配置技能，怎么支持热更新的技能数据。
- **不能独立做的**：战斗数值平衡。技能的数值体系、DPS 曲线、经济系统的平衡性，这些是策划的专业领域，我不会越界。我能提供技术实现上的约束和建议（比如"这个机制在帧同步下会导致同步压力过大"或者"这种 Buff 叠加规则会让计算复杂度指数增长"），但最终的数值决策不在我这里。
- **协作方式**：我出技术方案和框架实现，策划出数值表和规则定义。技能编辑器的设计我们一起定需求，我负责实现。上线后的数值调优由策划主导，我提供数据埋点和分析工具的支持。

技能系统是游戏项目里技术和策划交叉最密集的地方。
很多冲突来自于策划想要的灵活性和技术能支撑的复杂度之间的矛盾。
我在这个方向的价值是：能在两者之间找到平衡点，让策划拿到够用的自由度，同时技术实现不会失控。

### 美术管线

- **能做的**：资源规范的定义（Texture 尺寸上限、Mesh 面数预算、特效粒子数限制），资源检查工具的开发和 CI 集成，美术资源导入流程的自动化。我能搭建一套完整的美术资源质量门禁：从导入设置的自动矫正、到打包前的规范检查、到 CI 里的自动化验证。
- **不能独立做的**：美术制作本身。建模、贴图、特效制作、动画制作，这些不是我的技能。我能告诉美术"这个模型面数超了"或者"这个特效在低端机上会卡"，但我不能帮他们改模型或重做特效。Shader 的视觉效果调参我能做基础的，但复杂的视觉效果开发需要图形程序员或者 TA。
- **协作方式**：我定义规范和检查规则，美术 TA 负责规范的落地指导和制作流程的优化，美术制作人员负责在规范内创作。资源入库时自动检查，不合规的打回修改。这套流程的关键是规范要合理 -- 太死板会影响美术发挥，太宽松等于没有。这个度需要我和 TA、美术主管一起定。

美术管线协作里最常见的问题是：程序定了规范但美术不理解为什么。
比如"Texture 最大 1024"，美术会问"为什么不能 2048"。
如果我只给一个数字，推不动。
但如果我能解释"2048 在低端机上会导致单张纹理占 4MB 显存，一个场景 50 张就是 200MB，超过目标设备的显存预算"，美术就能理解并配合。
技术规范的推动力来自解释能力，不是权力。

## 不做的（明确不在我的能力范围）

以下这些方向，我没有专业能力，也不会假装有。
如果你的需求核心在这些方向，你需要找对应领域的专业人士。
列在这里不是谦虚，是为了节省你的时间。

- **游戏策划 / 关卡设计 / 数值策划**：我能理解策划案，能评估技术可行性，但我不做策划设计本身。关卡节奏、数值模型、付费点设计、玩家心理分析，这些是策划的专业能力。我能告诉策划"这个设计实现成本很高"，但我不会替策划决定"这个设计好不好玩"。
- **美术创作（概念、建模、动画制作）**：我管的是美术资源的技术规范和管线效率，不是美术创作。概念设计、角色建模、场景制作、动画制作、动作捕捉后处理，这些我完全不涉及。让我评价一个角色模型的美术质量，我给不出专业意见。
- **市场运营 / 用户增长 / 商业化策略**：我做的是产品的技术底座，不是商业层。用户获取、留存策略、变现模型、广告投放优化，这些我没有专业知识。项目赚不赚钱不取决于技术底座好不好，虽然技术底座不好确实会让项目亏钱。
- **后端运维的深度 DBA 工作**：大规模分布式系统调优、数据库分库分表、高可用架构设计。我理解这些概念在做什么，在讨论中能听懂并提出合理的问题，但我没有做过线上大规模系统的运维，不能承担这个职责。这个方向的经验必须是实战积累的，光看文章没有用。
- **移动端原生开发（iOS / Android native app）**：我做的是 Unity 跨平台开发，不是原生 iOS 或 Android 应用。Swift、Kotlin、原生 UI 框架、系统级 API，这些不是我的技术栈。偶尔处理 Unity 和原生的桥接层可以（比如接 SDK、处理权限申请），但完整的原生应用开发不行。

列这个清单的时候我也想过，要不要措辞委婉一点。
但想了想还是直接写。
如果一个人看到这个清单觉得"这人不行"，那说明他要找的不是我这个方向的人，提前知道比合作到一半发现好。
如果一个人看到这个清单觉得"挺好，至少知道边界在哪"，那说明我们对合作的期望是对齐的。

这个"不做"的清单可能会让人觉得范围太窄。
但反过来想：一个人能把上面"能独立交付"的部分全部做好，已经覆盖了客户端工程从性能到构建到工具链到质量保障的核心链路。
游戏开发本来就是协作的事，不存在一个人什么都做的角色。
把自己的部分做到足够深，比什么都沾一点但都做不深要有价值得多。

如果你需要的是一个"什么都能干一点"的通才，我不是最好的选择。
但如果你需要的是一个在客户端工程方向上能把事情做透的人，我的能力范围刚好覆盖你需要的链路。

## 为什么要写这篇

大多数技术博客都在展示"我会什么"，很少有人主动画边界。
但对于合作方来说，知道"这个人不做什么"比"这个人会什么"更有用。

原因很简单：

**"会什么"是模糊的**。
写过一篇文章算"会"吗？做过一个 Demo 算"会"吗？在项目里用过一次算"会"吗？
每个人对"会"的标准不一样，这个信息的信噪比很低。
你看到简历上写"熟悉 Shader 开发"，不知道是写过完整的渲染管线还是改过一个参数。

**"不做什么"是精确的**。
它直接告诉你：不要对我有这个预期。
这比任何吹嘘都有用，因为它能帮你在合作前就排除掉不匹配的场景。
省下来的不只是你的时间，也是我的时间。

**主动画边界是一种信号**。
它说明这个人对自己的能力有清醒的认知，不会为了接活而夸大范围。
合作中最怕的不是对方说"这个我不会"，而是对方说"我会"但做不到交付标准。
前者损失的是一次机会，后者损失的是时间、信任和项目进度。

我见过太多这样的情况：一个人什么都接，前期看起来很能干，到了中后期开始频繁延期、质量下滑、甩锅说"这个方向我其实不太熟"。
这种合作体验比一开始就说"不行"要差得多。
所以我宁愿在这里把边界画清楚，哪怕因此失去一些机会。
留下来的机会，匹配度会高很多。
对我来说，做一个高匹配度的项目，远比做三个勉强凑合的项目有价值。

如果你在评估是否和我合作：

- 看"能独立交付"那一列 -- 判断我能不能直接产出你需要的东西。如果你的核心需求在这个范围内，我可以快速启动，不需要额外的上下文铺垫。
- 看"需要协作"那一列 -- 判断你是否有对应的配合资源。如果你团队里有后端、有策划、有美术，我能和他们配合得很好。如果你希望我一个人把这些也扛了，那预期会错位。
- 看"不做"那一列 -- 确认你的需求不在这个范围。如果在，我会直接告诉你，不会接了再说做不了。

三列都看完，你对我的能力边界就有了完整的画像。
这比看一份罗列了 50 个技能关键词的简历有用得多。

## 这篇真正想留下来的结论

写 770 篇文章不代表什么都能做。
但它代表我在自己的范围内，有足够的深度和密度。

我的核心价值不是"什么都会一点"，而是在客户端工程这个方向上，从性能到构建到工具链到工程治理，我能形成一个完整的闭环。
这个闭环里的每一环我都有实际的生产经验，不是停留在概念层面。
那 770 多篇文章不是为了凑数量，是在记录和沉淀这些经验的过程。
每一篇文章背后都有一个具体的问题、一个排查过程、一个解决方案。

对合作方来说，最危险的不是对方能力不够，而是对方不知道自己能力不够。
能力不够可以协作补齐，认知不够会导致方向性的错误。
一个人说"这个我搞不定，你需要找专门的人"，这不是示弱，这是在帮你避开一个坑。

我写这篇的目的，就是确保你在和我合作之前，已经知道：

- 我能做什么 -- 这些事情我能快速、独立、高质量地交付。不需要你操心过程，只需要对齐目标。
- 我需要什么配合 -- 这些事情我能贡献技术判断和部分实现，但需要其他角色补齐。提前安排好配合资源，合作效率会高很多。
- 我做不了什么 -- 这些事情请找对的人，我不会勉强自己也不会耽误你的进度。与其到中途发现不匹配，不如一开始就说清楚。

画清楚边界，不是示弱。
是让合作从一开始就建立在准确的预期上。
这比任何技术炫技都更值钱。

如果你的需求和我的能力对得上，我们聊起来会很快。
因为不需要试探、不需要吹嘘、不需要到中途才发现不匹配。
你知道我能做什么，我知道你需要什么，直接聊怎么做就行。

如果对不上，我也愿意帮你想想谁更合适。
这就是画边界的好处 -- 它让每一次合作都从诚实开始。
