## 一句话总结

这个案例证明的不是"我能接手复杂项目"，而是：面对一个信息不对称严重的工程环境，我有一套可复用的诊断方法论，能在不破坏现有节奏的前提下，快速建立工程全景图并找到安全的切入点。

## 背景：接手的是什么样的项目

去年我加入了一个已经跑了两年多的 Unity 手游项目。项目大概是这样的状态：

- Unity 2021 LTS，团队 12 个人左右，客户端 5 个，服务端 3 个，策划和美术各两个。
- 同时出 Android、iOS 和 PC 三端，Android 是主力，iOS 和 PC 各有各的坑。
- 用了 HybridCLR 做热更新，版本大概是 4.x，中间自己改过一些配置。
- 资源管线是自研的 AssetBundle 方案，打包逻辑散落在好几个编辑器脚本里。
- CI 有一条 Jenkins 流水线，但只覆盖了 Android 包，iOS 和 PC 还是手动打的。
- 线上有性能投诉，QA 也报了不少崩溃，但没有人系统地看过这些数据。

我的角色是接替之前离职的工具链工程师。交接文档只有半页 Confluence，上面写了几个目录路径和一句"打包直接跑 Jenkins 就行"。

说实话，前三天我基本是懵的。代码仓库有 1200 多个 .cs 文件，EditorTools 目录下有 40 多个编辑器脚本，光看文件名根本猜不出哪些还在用、哪些已经废弃。git log 里有些脚本的最后修改时间是一年多以前，但 `using` 引用还在，删不删都不好说。

我第一天做的事情是跑 `git log --since="30 days ago" --stat`，看最近一个月谁在改哪些文件。结果发现活跃修改集中在 `HotUpdate` 和 `HotUpdate.Battle` 两个程序集里，EditorTools 基本没人碰。这个信息很重要——它告诉我工具链这块确实处于"维护真空"状态。

但我没有急着改任何东西。之前踩过的教训告诉我：越复杂的项目，前两周的动作越需要克制。动一个你不理解的环节，可能引发三个你看不到的问题。我之前在另一个项目里犯过这种错：入职第三天就"顺手"修了一个打包脚本里的路径问题，结果触发了资源依赖链的变化，导致了一个晚上才被发现的线上资源丢失。那次之后我学到了一个教训——在你能画出完整地图之前，别碰地图上任何一个节点。

## 第一周：不改代码，只画地图

第一周的目标只有一个：画地图。搞清楚这个项目的工程全景长什么样，哪些链路是健康的，哪些链路是带病运行的，哪些链路根本没有人看。

我给自己定了一个规则：这一周不开任何 PR，不改任何文件，只输出文档。所有发现都记录在一个私人的 Notion 页面里（后来后悔了，应该一开始就用团队可见的方式记录）。

### 构建链路诊断

第一件事，我把三端的完整构建链路走了一遍。不是看脚本，是真的从 `git clone` 开始，一步一步跑到最终出包。

Android 端相对好一些，Jenkins 上有一条 pipeline，触发方式是手动点按钮加一个 webhook。我计时了一下：从触发到出包，整个流程大概 47 分钟。其中 `Generate All` 占了 8 分钟左右，il2cpp 编译占了 14 分钟，AssetBundle 打包占了 11 分钟，剩下是签名、上传和通知。

iOS 端要麻烦很多。没有 CI，全靠一台 Mac Mini 手动操作。我找负责 iOS 的同事要了操作步骤，他给我发了一个微信聊天截图，上面是 6 条命令，中间夹着两句"这里要等一下"和"如果报错就重新来一遍"。整个流程大概 70 分钟，但因为中间需要人盯着，实际消耗的人力时间更长。

PC 端更离谱。打包脚本是从 Android 端复制过来改的，里面有好几个 `#if UNITY_ANDROID` 的宏没删干净，导致偶尔会打出带 Android 配置的 PC 包。比如 `PlayerSettings.Android.targetArchitectures` 的设置代码在 PC 构建时也会执行，虽然不报错但会写入无意义的配置。之前的同事是靠"打完之后检查一下 log 里有没有异常"来发现这个问题的。我问他"你怎么知道哪些 log 是异常的"，他说"看多了就知道了"。这种依赖个人经验的流程，换一个人就会出问题。

我把这些信息画成了一张表：

| 维度 | Android | iOS | PC |
|------|---------|-----|-----|
| CI 覆盖 | Jenkins pipeline | 无 | 无 |
| 出包时间 | ~47 min | ~70 min (手动) | ~55 min (手动) |
| 自动化程度 | 高 | 低 | 低 |
| 已知问题 | 无 | 签名偶尔失败 | 宏污染 |
| 人力依赖 | 低 | 高 | 中 |
| 增量构建 | 支持（~18 min） | 不支持 | 不支持 |

这张表后来成了我向 lead 汇报优先级的基础。不需要解释太多，一看就知道 iOS 和 PC 的构建链路是短板。

另外我还做了一件事：在三端各跑了一次 clean build 和 incremental build，记录差异。Android 的增量构建大概 18 分钟，比全量快了 60% 多。但 iOS 和 PC 因为是手动流程，根本没有增量构建的概念——每次都是全量。这意味着即使不考虑自动化，光是加上增量构建的支持，iOS 出包时间就能从 70 分钟降到 30 分钟左右。

### 资源管线诊断

第二件事是看资源管线。项目用的是自研 AssetBundle 方案，没有用 Addressables。

我先在编辑器里跑了一遍完整的 AB 打包，然后用 `AssetBundle Browser` 和自己写的一个小脚本把依赖关系导出来。结果发现几个问题：

第一，打包粒度不统一。UI 图集是按 Atlas 名分的，场景资源是按场景分的，但角色和特效是按目录分的。这导致有些 bundle 只有 200KB，有些超过了 15MB。粒度差异大意味着更新效率低——改一个角色的贴图，玩家可能要下 15MB 的 bundle。

第二，Shader 变体膨胀。我统计了一下，项目里有 37 个自定义 Shader，通过 `shader_feature` 和 `multi_compile` 产生的变体组合超过了 4200 个。但实际运行时用到的大概只有 800 个左右。没有人做过变体收集，也没有 ShaderVariantCollection，全靠 Unity 默认的保留策略。这直接导致了两个后果：包体比预期大了约 120MB，首次进入某些场景时有明显的 Shader 编译卡顿。

其中最严重的一个 Shader 是角色渲染用的 `Character_PBR.shader`，它一个就产生了 680 多个变体。原因是里面同时用了 `multi_compile_fog`、`multi_compile_instancing`，再加上自己定义的 5 个 `shader_feature`，排列组合就爆炸了。这种情况下正确的做法应该是把低频的分支改成 `shader_feature_local`，或者提前做好 Warmup 收集，但之前没有人管这个。

第三，命名和目录结构有隐性规则但没有文档。比如 `Assets/Art/Character` 下面按角色 ID 分目录，但 `Assets/Art/Effect` 下面是按功能分的（Attack、Buff、UI 等）。这些规则全在老同事的脑子里，新来的美术同事经常放错位置，然后打出来的 bundle 依赖关系就乱了。

我还用 `Android Studio` 的 APK Analyzer 看了一下最终包体的构成。整包大小 1.1GB，其中 AssetBundle 资源占了 680MB，il2cpp 的 libil2cpp.so 占了 78MB，Shader 相关的数据大概 130MB。对比同类项目，Shader 这一块明显偏大，印证了前面变体膨胀的判断。

### 热更新链路诊断

热更新用的是 HybridCLR 4.x，Community 版本。我花了大概一天半的时间把整条链路从头到尾走了一遍。

先看配置。`HybridCLRSettings` 里的 hotUpdateAssemblies 配了 3 个程序集：`HotUpdate`、`HotUpdate.UI` 和 `HotUpdate.Battle`。补充元数据用的是 `HomologousImageMode.SuperSet`，生成策略是每次打包时跑 `Generate All`。`HybridCLR/GeneratedData` 目录下的产物我仔细对比了 Git 历史，发现 `MethodBridge` 的桩函数最后一次更新也是三个月前，和 AOTGenericReferences 同期。

然后看线上热更的发布流程。这部分让我比较意外——热更新包的发布居然是手动上传到一个 CDN 后台，然后在运营后台改一个版本号。没有灰度，没有回滚机制，没有校验。也就是说，如果某次热更新包有问题，唯一的补救方式是再发一个新包覆盖。

我问了一下之前出过几次热更事故，得到的回答是"两三次吧，都是赶版本的时候忘了跑 Generate All"。这个回答直接印证了一个判断：热更新发布流程里的人工步骤太多，而人工步骤就是事故的温床。

我还检查了 `AOTGenericReferences.cs` 的生成情况。发现上一次生成的时间戳是三个月前，中间加的新泛型实例根本没有补进去。这意味着如果有某个新加的泛型组合第一次在热更新程序集里被调用，可能会在低端机上崩溃。

这个发现我当天就反馈给了 lead，因为它是一个已经埋在线上的定时炸弹。lead 的反应是"之前打包的人走了之后就没人管这块了"。这也侧面验证了我的判断——工具链的维护不是可选的，它只是在有人管和没人管之间切换，而没人管的后果会累积。

另外一个细节：项目配置了 `HomologousImageMode.SuperSet`，但我检查了 `AOTGenericReferences.cs` 里的泛型列表，发现有几个常用的泛型组合缺失——比如 `Dictionary<int, ValueTuple<string, int>>` 和 `List<UniTask>`。这些组合在热更新程序集里被频繁使用，但补充元数据里没有对应条目。在大部分机型上这不会崩溃（因为 il2cpp 有 fallback），但在某些内存紧张的低端机上，fallback 失败就是 Native Crash。

## 第二周：建基线，定优先级

### 性能基线

第一周画完地图之后，第二周开始建基线。

性能基线的思路很简单：用有限的设备和场景，建立一个可重复测量的参考点。以后任何优化或改动，都可以和这个基线对比。

我选了 3 台代表机型：

- 低端机：Redmi Note 11（Snapdragon 680，4GB RAM）
- 中端机：OPPO Reno8（Dimensity 1300，8GB RAM）
- 高端机：iPhone 13（A15，4GB RAM）

场景选了 5 个最典型的：

1. 登录 → 主界面的加载流程
2. 主界面静置 30 秒
3. 战斗场景（5v5 中等特效密度）
4. 打开背包界面（2000 个道具）
5. 场景切换（从主城到副本）

每个场景在每台设备上跑 3 次取平均值。Android 用 Unity Profiler 通过 USB 连接抓帧，同时开 Android Studio 的 Memory Profiler 监控系统级内存。iOS 用 Xcode Instruments 抓 Time Profiler 和 Allocations，Metal System Trace 看 GPU 侧的瓶颈。

有一个经验分享：不要只看 Unity Profiler 的 Mono/IL2CPP Heap，要同时看操作系统层面报的 PSS（Proportional Set Size）。两者的差值往往是 Native 插件和驱动层的内存消耗，这部分在 Unity Profiler 里看不到，但它也会触发 OOM。

跑出来的数据是这样的：

| 场景 | Redmi Note 11 | OPPO Reno8 | iPhone 13 |
|------|--------------|------------|-----------|
| 登录→主界面 | 12.3s | 7.1s | 5.8s |
| 主界面静置 | 24fps / 380MB | 42fps / 410MB | 58fps / 390MB |
| 战斗场景 | 16fps / 520MB | 31fps / 560MB | 45fps / 480MB |
| 背包界面 | 打开耗时 3.2s | 1.4s | 0.9s |
| 场景切换 | 8.7s | 4.2s | 3.1s |

几个关键发现：

第一，低端机的战斗帧率只有 16fps，已经严重影响体验了。用 Profiler 看，瓶颈在渲染侧，Overdraw 严重，半透明特效叠了好几层。

第二，背包界面在低端机上打开要 3 秒多。追下去发现是 ScrollView 里一次性实例化了 2000 个 item，没有做虚拟列表。

第三，内存在所有设备上都偏高。主界面 380-410MB，进战斗直接飙到 520-560MB。在低端机上已经接近系统杀进程的阈值。用 Memory Profiler 抓了一个 snapshot，发现最大的几块是：贴图未释放（战斗结束后角色贴图没有 Unload）、UI 图集常驻内存（主界面的 Atlas 在战斗中也没卸载）、以及 Lua 虚拟机的内存占用（项目里有一部分历史遗留的 Lua 脚本还在跑）。

这三个发现后来成了第三周开始动手的核心依据。有了具体数字，和 lead 讨论优先级的时候就不需要"我觉得这里有问题"这种表达，而是"低端机战斗帧率 16fps、OOM 占崩溃 26%、背包打开 3.2 秒"——这些数字本身就在帮你争取资源和排期。

### 崩溃分布

项目接了 Bugly，但之前没有人系统地看过崩溃数据。我拉了最近 30 天的数据，做了一轮分类。

总崩溃量：日均约 340 次，影响用户约 1200 人/天。崩溃率大概在 1.8% 左右，对于一个日活 6 万多的项目来说，这个数字不算低。行业一般认为崩溃率 1% 以下是健康的，1-2% 是需要关注的，超过 2% 就应该紧急处理。

Bugly 的后台数据按 Android 和 iOS 分开看差异也很大。Android 的崩溃率在 2.1%，iOS 只有 0.9%。这和设备碎片化程度直接相关——Android 低端机型多、GPU 型号杂、系统版本分裂严重。

按原因分类之后，Top 5 是：

| 排名 | 崩溃类型 | 日均次数 | 占比 |
|------|---------|---------|------|
| 1 | OOM（内存不足被杀） | 89 | 26% |
| 2 | Native Crash（il2cpp 层） | 72 | 21% |
| 3 | Java 层空指针（Android） | 58 | 17% |
| 4 | Shader 编译失败（Mali GPU） | 41 | 12% |
| 5 | 热更新加载失败 | 33 | 10% |

几个值得注意的地方：

OOM 排第一不意外，因为前面性能基线已经看到内存偏高了。进一步拆分发现，OOM 有 70% 集中在 4GB RAM 以下的设备，而且大部分发生在战斗场景。

Native Crash 里有一部分是 HybridCLR 相关的。堆栈里出现了 `RuntimeApi::LoadMetadataForAOTAssembly` 和 `InterpreterModule::Execute` 的崩溃。结合前面发现的 AOTGenericReferences 三个月没更新的问题，这两个很可能是同一个根因。

Shader 编译失败集中在 Mali GPU 设备上，主要是几个 Shader 用了 `half` 精度在 Mali 上行为不一致的 feature。这个问题在 Adreno 和 Apple GPU 上完全不存在。按设备分布看，Samsung Galaxy A 系列和部分华为中低端机型是重灾区，这些机型恰好是项目用户画像里占比最高的一档。

热更新加载失败那 10%，追下去发现主要是 CDN 下载超时和文件校验失败。有一部分用户的网络环境比较差（弱网、地铁场景），下载到一半断了但客户端没有做断点续传，重新下载时又因为本地有不完整的文件导致校验失败。这个问题不复杂，但修复需要改客户端的资源下载模块，牵涉面比较广，放到了"计划做"的象限里。

### 优先级矩阵

把上面两周的信息汇总之后，我画了一个四象限矩阵：

**立刻做（风险低 x 收益高）：**
- 补全 AOTGenericReferences 生成并加入 CI（一天搞定，消除线上崩溃隐患）
- 给 CI 加 iOS 和 PC 的构建 job（减少人力，降低出错率）
- 背包虚拟列表改造（改动范围小，体验提升明显）

**计划做（风险中 x 收益高）：**
- 内存优化（需要系统性排查，但收益大）
- Shader 变体收集和裁剪（需要测试覆盖，但能减包体和卡顿）
- AssetBundle 粒度重新规划（牵涉面广，但长期收益高）

**观察（风险高 x 收益中）：**
- 热更新发布流程改造（需要运维配合，改动链路长）
- 资源目录结构重构（牵涉美术工作流，需要充分对齐）

**先别动（风险高 x 收益低）：**
- 战斗系统的渲染管线（核心业务逻辑，不熟悉时动风险极大）
- 自研 AB 方案迁移到 Addressables（收益不确定，迁移成本高）

这个矩阵我打印出来贴在了工位上。后来每次有人说"这个也该优化一下"的时候，我都会让他在矩阵上找到位置。不是所有问题都值得立刻解决，关键是顺序。

## 第三周开始：从最安全的地方切入

有了前两周的诊断地图，第三周开始动手。但我给自己定了一个原则：前三刀必须是低风险、高可见的改进。因为这时候我还是新人，需要用可见的成果建立团队信任。

这里有一个很容易犯的错误：拿到诊断结果之后，直接去改最严重的问题。但"最严重"往往意味着"最复杂"、"牵涉面最广"。作为新人去碰最复杂的部分，出问题的概率远高于出成果的概率。正确的策略是：先从确定性最高的小改动开始，积累信任之后再推大动作。

### 第一刀：CI 补全

最先做的是把 iOS 和 PC 的构建接入 Jenkins。

Android 的 pipeline 已经有了，我参照它的结构，为 iOS 和 PC 各写了一条。核心差异主要在三个地方：

1. iOS 需要 Mac 机器上跑 xcodebuild，我把那台 Mac Mini 配成了 Jenkins agent。
2. PC 端的打包脚本里那几个 Android 宏污染，我加了一个预处理步骤来检测并报错。
3. 三端共用的 HybridCLR `Generate All` 步骤，我抽成了一个 shared library step，避免重复维护。

同时，我在 pipeline 里加了两个检查：

- AOTGenericReferences 生成时间戳检查：如果距离上次生成超过 7 天就发 warning。
- Build Report 里的 Shader 变体数量检查：如果超过 5000 就阻断构建。

这一刀花了大概 4 天。中间有半天花在了配置 Mac Mini 的 Jenkins agent 上——那台机器上跑着一个很老版本的 Xcode，需要先升级到和 Unity 版本兼容的 Xcode 14。

效果很直接：iOS 出包时间从 70 分钟（手动）降到 52 分钟（自动），PC 端不再出现宏污染的包，团队不用专门安排人盯 iOS 和 PC 的打包了。更重要的是，三端构建的触发和产物现在在一个统一的 Jenkins Dashboard 上可见，谁都能看到最近一次构建的状态和耗时。之前"iOS 包今天出了没有"这种问题要在群里问人，现在看一眼 Dashboard 就知道了。

### 第二刀：资源检查器

第二刀是加资源检查的门禁，不改任何业务逻辑，只在 CI 里加一道检查。

我写了一个编辑器脚本，在 BatchMode 下跑，检查以下几项：

- AssetBundle 粒度异常：单个 bundle 超过 10MB 或小于 50KB 的，输出 warning。
- Shader 变体膨胀：单个 Shader 产生超过 200 个变体的，输出 warning。
- 贴图尺寸超标：超过 2048x2048 的非背景贴图，输出 error。
- 命名规范违规：不符合 `[类型]_[功能]_[编号]` 格式的资源，输出 warning。

这个检查器不阻断构建（第一版只报不拦），但结果会被发到项目的飞书群里。效果比我预期的好——美术同事看到每天的报告之后，主动来问我命名规范应该怎么改。这比我自己写一个规范文档然后要求大家遵守，效果好得多。

第一周跑下来，发现了 23 张超标贴图、4 个膨胀严重的 Shader 和 11 个粒度异常的 bundle。

这些数字让团队意识到资源管线确实需要治理，而不是"差不多就行"。23 张超标贴图如果全部从 4096 降到 2048，仅贴图这一块就能省大约 90MB 的内存和包体。4 个膨胀 Shader 如果做好变体裁剪，保守估计能减少 60-80MB 的 Shader 数据。这些数字不是我推算的，是检查器直接输出的——有了工具做支撑，对话就从"我觉得资源有问题"变成了"这 23 张贴图分别是哪些、谁负责的、什么时候改"。

### 第三刀：性能热点的最小修复

第三刀选了性能基线里最明确、改动最小的一个：背包虚拟列表。

原来的实现是在 ScrollView 的 Content 下面直接挂了 2000 个 item GameObject，打开背包时一次性全部实例化。我把它改成了一个简单的对象池 + 虚拟列表，可见区域外的 item 回收复用。

改动涉及 2 个文件，大概 180 行代码。核心逻辑是维护一个 `visibleRange`，根据 ScrollView 的 `normalizedPosition` 计算当前应该显示哪些 index，超出范围的 item 回收到对象池，新进入范围的从池里取。测试也很简单：打开背包、快速滑动、检查边界 item 的数据正确性、检查滑到最底部再滑回来数据有没有错乱。

结果：低端机上背包打开时间从 3.2 秒降到 0.4 秒。中端机从 1.4 秒降到 0.2 秒。内存方面，同一时刻 active 的 item 从 2000 个降到了可见区域的 15-20 个。

这个改动虽然小，但在团队里的效果很好。策划和 QA 每天都要打开背包，他们是最直接感受到变化的人。有个 QA 在群里说"背包终于不卡了"，这比任何技术汇报都管用。

有一个选择值得说一下：为什么第三刀选了背包虚拟列表，而不是战斗帧率或者内存优化？因为后两个的改动范围大、验证周期长、牵涉的模块多。背包虚拟列表是一个独立的 UI 模块，改完之后只需要 QA 过一轮背包相关的用例就行。作为新人的第三个改动，选一个"改坏了也只影响一个界面"的方向，远比选一个"改对了很牛但改坏了全盘崩"的方向安全。

这不是保守，是对风险的尊重。前三刀的目的不是展示技术深度，而是建立"这个人改的东西是可信的"这个印象。有了这个基础，后面再推大改动的时候，团队的接受度会高很多。

事实上到了第二个月，当我提出要做 Shader 变体裁剪和 AssetBundle 粒度重规划的时候，lead 几乎没有犹豫就批了排期。他后来跟我说："你前面三个改动都很稳，所以我相信你对风险的判断。"这就是前两周克制的回报。

## 反思：哪些做对了，哪些可以更好

做完第一个月之后，我回头看了看这套接手方法论，梳理一下哪些判断和动作是对的，哪些下次可以做得更好。这种复盘不是为了自我表扬，而是让方法论变得可迭代——每接手一个新项目，方法论就应该比上一次更成熟一点。

**做对了：前两周不碰业务代码，先建信任。**

这是最重要的一条。新加入一个团队，你对代码的理解一定不如在场的人。如果一上来就改业务逻辑，出了问题会直接摧毁信任。前两周只做诊断、只建基线、只输出数据，这些动作的共同特点是：不影响线上，不影响其他人的工作节奏，但能产出可见的信息增量。

**做对了：用数据说话而不是凭感觉。**

"内存太高了"和"低端机战斗场景内存 520MB，4GB 设备 OOM 占崩溃 26%"，说的是同一件事，但在团队里的说服力完全不同。前两周建的性能基线和崩溃分布数据，后来成了所有优化决策的基础。没有人能反驳数字。

后来有一次周会上 lead 要和老板争取一个迭代周期专门做性能优化，用的就是我建的基线数据。他说"有了这些数据，我和老板谈的时候底气完全不一样了"。这让我意识到，工程诊断的价值不止是技术层面的，它还是管理决策的弹药。

**做对了：第一批改动选低风险高可见的。**

CI 补全、资源检查、背包虚拟列表，这三个改动都有一个共同特点：改动范围小，不涉及核心业务逻辑，但效果对团队可见。这让我在第一个月结束时已经建立了基本的信任——大家知道我改的东西是靠谱的，后面推更大的优化项时阻力就小很多。

**可以更好：应该更早和策划/美术对齐资源规范。**

我在第二周才开始看资源管线，第三周才开始输出检查报告。但资源规范的问题本质上是协作问题，不是技术问题。如果我在第一周就约策划和美术负责人聊一轮，了解他们的工作流和痛点，后面推资源门禁的时候会更顺畅。

第二周出检查报告的时候，有一个美术同事的反应是"为什么之前没人告诉我贴图不能超过 2048"。这个反应说明规范缺失的问题早就存在了，只是没有人从工具侧去补这个环。我应该更早发现这个需求。

更深层的教训是：工具链工程师不能只和程序员打交道。美术、策划、QA 都是你的用户，他们的工作流和痛点决定了你做的工具有没有人用、好不好用。技术再好的门禁系统，如果美术不理解为什么要遵守，执行效果一定打折扣。第一周就应该约每个职能的负责人聊 30 分钟，了解他们眼里的"项目痛点"是什么。很多时候，程序员觉得的痛点和美术觉得的痛点完全不一样。

**可以更好：崩溃分析应该第一天就开始拉数据。**

我是第二周才开始看 Bugly 数据的。但实际上，崩溃数据是不需要理解代码就能看的——它是纯粹的线上事实。如果我第一天就拉了 30 天的崩溃数据，HybridCLR 相关的 Native Crash 问题可能能提前一周被识别出来。

后来我自己总结了一条规则：加入新项目的第一天，先做三件事——拉崩溃数据、跑一遍完整构建、看一遍 git log 最近 30 天的提交分布。这三件事不需要任何前置知识，但能让你在第一天就对项目的"健康状况"有一个粗略的判断。

具体来说：崩溃数据告诉你线上最痛的点在哪，完整构建告诉你工程链路的健康程度，git log 告诉你团队最近在忙什么、哪些模块在高频变动。三者结合，你大概就能画出一张粗略的"项目风险热力图"。

比如这个项目，如果我第一天就拉了 git log，我会发现 `HotUpdate.Battle` 的提交频率远高于其他模块，同时崩溃数据也会告诉我战斗场景是 OOM 重灾区。两条信息交叉验证，第一天就能锁定"战斗模块"是需要重点关注的区域，而不用等到第二周跑完性能基线才得出这个结论。

**可以更好：应该更早建立一个共享的问题追踪看板。**

前两周发现的问题，我是记在自己的笔记里的。后来向 lead 汇报的时候才整理成文档。如果一开始就建一个团队可见的看板（哪怕是一个简单的飞书文档），其他人也能看到诊断的进展，能更早地参与到优先级讨论中来。单独闷头做两周再一次性输出，效率没问题，但错过了"让团队一起参与诊断过程"的机会。工程诊断不应该是一个人的事。

## 这篇真正想留下来的结论

接手复杂项目不是一场考试，不需要在前两周证明自己有多厉害；需要证明的是你有一套系统化的诊断方法，能在不制造新问题的前提下，把已有问题变得可见、可量化、可排序，然后从最安全的地方开始切入——先赢得信任，再推动改变。
