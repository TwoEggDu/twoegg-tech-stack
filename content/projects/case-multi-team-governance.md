## 一句话总结

把一个 30+ 人、3 个团队并行开发的项目，从"每周构建失败 3-5 次、资源互相覆盖"的状态，用分支策略 + CI 门禁 + 资源目录归属 + 发布审批四刀，收敛到"每周构建失败 0-1 次、资源覆盖事故每月不到 1 次"。

## 背景：从 1 个团队到 3 个团队

项目最早是 1 个 10 人团队，所有人共享一条 main 分支，日常开发靠口头沟通和 IM 群里喊一嗓子。那个阶段其实没什么大问题，大家互相知道谁在改什么，冲突了当场解决，构建也比较稳。

后来项目进入功能扩展期，团队拆分成了三个方向：

- 玩法团队：约 12 人，负责核心战斗、关卡、任务系统
- 系统团队：约 10 人，负责背包、社交、商城、活动
- 美术技术团队：约 8 人，负责特效、UI 资源、场景制作和资源规范

三个团队加起来超过 30 人，同时在一个仓库里开发。每个团队有自己的迭代节奏和版本目标，但最终都要合到一起出包。

团队规模翻了三倍，但开发流程还停留在 10 人团队的阶段——没有分支策略、没有 CI 门禁、没有资源目录规范、没有发布审批。用一句话概括就是：团队扩张了，治理没跟上。

问题很快就爆发了。

每周构建失败 3-5 次，原因五花八门：有人推了编译不过的代码，有人覆盖了别人的 prefab，有人改了公共模块没通知其他团队。资源覆盖更头疼，美术改了一个 UI prefab，程序那边引用的组件丢了；玩法团队加了一个特效，打包的时候发现和系统团队的同名资源冲突了。

最离谱的一次，三个团队各自在本地构建都没问题，合到一起直接崩溃，查了大半天才发现是两个人改了同一个 ScriptableObject 但 merge 的时候 YAML 冲突没解干净。

那段时间每周至少有半天在处理合并冲突和构建修复，主程和主美都快被磨崩溃了。

更隐蔽的问题是团队之间的信任开始出裂痕。玩法团队觉得"又是系统团队改坏了构建"，系统团队觉得"美术推了不规范的资源"，美术觉得"程序改了公共接口也不说一声"。大家开始互相甩锅，每次构建失败的第一反应不是"我们一起查"，而是"这次不是我的问题"。

这个时候我意识到，技术问题只是表象，真正的危机是协作机制跟不上团队规模的增长。

## 第一个月：诊断治理缺口

我没有一上来就推方案。先花了两周时间，把现有流程从头到尾走了一遍，记下来所有痛点。

**分支策略：基本没有。**

所有人直接推 main，没有 feature branch 的概念。偶尔有人自己建个分支，但合并时间不确定，经常是攒了一大堆改动一次性合进来，冲突量巨大。

**CI：形同虚设。**

Jenkins 上有一个构建 job，但是手动触发的，而且只做编译。没有资源校验，没有自动化测试，更没有和分支合并挂钩。构建失败了也没有通知，经常是打包的人自己发现然后在群里喊。

**资源管理：自由发挥。**

没有目录归属的概念，美术和程序的资源混在一起。prefab 放在哪里全靠个人习惯，同一个功能的贴图可能散布在三四个目录。最要命的是 Shared 类的公共资源，谁都能改，改完也不需要审核。

**发布：谁都能触发。**

打包入口是一个 Jenkins job，任何人都可以手动点。没有审批流程，没有冒烟测试，打完包直接发。出了问题回滚也没有预案，只能现场扒 git log 找上一个能用的 commit。

诊断完之后，我整理了一份治理缺口清单，按影响面排了优先级。构建失败排第一，因为它直接阻塞所有人的日常开发；资源覆盖排第二；发布风险排第三。

排优先级这件事本身就需要克制。第一个月的冲动是想一次性把所有问题都解决，但我很快意识到如果同时推太多改变，团队消化不了。30 多个人同时适应新分支策略、新 CI、新目录规范和新发布流程，只会让大家觉得"又来折腾了"。所以我决定一刀一刀来，每刀之间留出至少两到三周的缓冲期，让大家适应了再推下一个。

然后找三个团队的 lead 开了一次碰头会，不讲大道理，就把上两周的事故清单摆出来：这次构建失败耽误了多少人时，那次资源覆盖导致回滚花了多久，那次发布事故周末加了几个人的班。数据摆出来之后，大家都同意需要改。

碰头会上还达成了一个重要共识：治理规则的推进由我来牵头，但各团队 lead 负责在自己团队内推动落地。我一个人不可能盯着 30 多个人，必须借助每个团队的管理力量。这个分工后来被证明是正确的。

## 第一刀：分支策略

这是改动最小但见效最快的一刀。

我没有引入完整的 GitFlow，那套模型对游戏项目太重了——release branch、support branch 一堆，光给大家解释就要半天。最终采用的是一个简化版，只保留最核心的三层：

- `main`：只接受从 develop 合入的经过验证的代码，对应可发布状态
- `develop`：日常集成分支，所有 feature 合入的目标
- `feature/*`：每个功能或任务一个分支，命名规范 `feature/team-name/brief-description`

规则很简单：

1. 任何改动必须在 feature branch 上开发
2. feature branch 必须通过 CI 检查才能发起 PR
3. PR 需要至少 1 个同团队 + 1 个跨团队 reviewer 审核
4. develop 到 main 的合并走 weekly release train，每周四下午统一合入

分支命名里带团队名这个细节很重要。`feature/gameplay/new-boss-fight` 和 `feature/system/shop-refactor` 一眼就能看出是谁的，出问题的时候排查也快。

落地的时候遇到的最大阻力不是技术问题，而是习惯。有人觉得开 feature branch 太麻烦，"就改一行代码还要开分支？"

我的做法是先不强制，先把 develop 分支保护规则加上——不接受直接 push，只接受 PR。这样大家自然就被"逼"着走 feature branch 了。两周之后大家适应了，也就不抱怨了。

还有一个实际落地中的小问题：weekly release train 的节奏需要和版本计划对齐。我们定的是每周四下午合 develop 到 main，但如果某一周有紧急 hotfix 需求，就走单独的 `hotfix/*` 分支，从 main 切出来，修完直接合回 main 和 develop。hotfix 流程一个月大概用到 1-2 次，不多，但必须有，否则遇到线上问题就会手忙脚乱。

## 第二刀：CI 门禁

分支策略落地之后，CI 门禁是顺理成章的事。有了 PR 流程之后，没有自动化检查，review 就只能靠人眼看代码和资源改动，效率低而且容易漏。特别是资源相关的问题——贴图尺寸超标、命名不规范、AB 依赖错误——光看 diff 根本发现不了。

我在 Jenkins 上搭了一条分级 pipeline，每次 PR 创建或更新时自动触发。之所以选 Jenkins 而不是 GitHub Actions，是因为 Unity 构建需要本地 agent 和 License，Jenkins 在这方面更灵活：

**第一级：快速检查（目标 < 5 分钟）**

- 编译检查：Unity BatchMode 编译，确保代码能过
- 基础资源校验：贴图尺寸是否超标、命名是否符合规范
- 语法和格式检查：C# 代码规范

这一级的设计原则是快。开发者推了代码之后 5 分钟内就能看到结果，如果挂了可以马上修。

**第二级：完整检查（目标 < 20 分钟）**

- AssetBundle 依赖检查：是否引用了不该引用的包、是否有循环依赖
- 单元测试：核心系统（背包、商城、战斗数值）的测试用例
- 资源完整性检查：prefab 引用是否完整、材质引用是否丢失

第二级只在 PR 合入 develop 之前强制通过，日常推 feature branch 时不阻塞。这样开发者在 feature branch 上可以快速迭代，但合入主干之前必须过完整检查。

**构建优化**

原来的全量构建要 45 分钟，这对 CI 来说完全不可接受。
45 分钟的反馈周期意味着开发者推完代码就去做别的了，等 CI 结果出来早就忘了上下文。我做了几个优化：

- Library 缓存：Jenkins agent 上保留 Unity Library 目录，避免每次全量导入
- 增量编译：只重新编译改动的 assembly
- 资源校验并行化：把贴图检查、命名检查、依赖检查拆成并行任务

优化之后全量构建降到 18 分钟，增量构建 8 分钟左右。

这里还有一个容易踩的坑：Jenkins agent 的 Library 缓存如果长时间不清理，可能会出现缓存和实际资源不一致的问题。我设了一个定时任务，每周日凌晨全量清理一次 Library 缓存然后做一次干净的全量构建，保证周一大家用的缓存是干净的。这个代价可以接受，但如果不做，偶尔会遇到"我本地没问题但 CI 就是编译不过"的灵异事件。

**通知**

CI 结果通过 DingTalk webhook 推送到对应团队的群。格式很简单：谁的 PR、检查结果、失败原因、链接。失败的消息会 @ 提交者，避免漏看。

这里有一个细节：我一开始把所有 CI 消息都推到一个大群里，结果太吵了，大家开始忽略。后来改成按团队分群推送，只有跨团队的 PR 才同时推送到公共群。噪音降下去之后，大家反而更关注 CI 结果了。

另外一个容易被忽略的点是 CI 失败后的处理流程。光告诉你"挂了"没用，还得让人快速知道"为什么挂了"。我在 Jenkins pipeline 里加了失败分类标签：编译错误、资源校验失败、测试用例失败、超时。DingTalk 通知里会带上分类和关键错误信息的前几行，开发者点进去之前就大概知道问题方向了。这个小改动让 CI 失败的平均修复时间从 40 分钟降到了 15 分钟左右。

## 第三刀：资源目录归属

资源覆盖的根因是没有归属。谁都能改任何目录下的任何文件，改之前也不需要知会任何人。

我的方案是按团队划分资源目录，同时引入 code owner 机制：

```
Assets/
├── Art/                    # 美术技术团队
│   ├── Effects/
│   ├── UI/
│   └── Scenes/
├── Gameplay/               # 玩法团队
│   ├── Characters/
│   ├── Levels/
│   └── Combat/
├── System/                 # 系统团队
│   ├── Shop/
│   ├── Inventory/
│   └── Social/
└── Shared/                 # 公共资源（需要审批）
    ├── CommonUI/
    ├── Audio/
    └── Configs/
```

规则：

1. 每个团队只能自由修改自己目录下的资源
2. 修改其他团队的目录需要在 PR 中说明原因，并获得对方 lead 审核
3. Shared 目录的修改必须有至少 2 个不同团队的 reviewer 批准

这个规则不是拍脑袋定的，而是根据过去一个月的资源覆盖事故统计出来的。90% 以上的覆盖事故都发生在跨团队改动上——A 团队改了 B 团队的文件。把归属定清楚，把跨团队改动加审核，就能挡住绝大部分问题。

落地工具是 GitHub 的 CODEOWNERS 文件：

```
# 美术技术团队
/Assets/Art/                @art-tech-team
# 玩法团队
/Assets/Gameplay/           @gameplay-team
# 系统团队
/Assets/System/             @system-team
# 公共资源 - 需要多团队审批
/Assets/Shared/             @tech-leads
```

这样 PR 创建之后，GitHub 会自动根据改动文件分配 reviewer。改了 Art 目录就自动拉美术技术团队的人来看，改了 Shared 目录就自动拉所有 tech lead。

迁移过程比预想中痛苦。存量的资源目录结构很乱，有些 prefab 散落在各处，移动的时候要小心处理引用关系。我分了三批做：

- 第一批：新增资源全部按新规范放
- 第二批：各团队自行整理自己明确归属的资源
- 第三批：处理模糊地带，把"说不清归谁"的资源统一收进 Shared

整个迁移前后花了差不多两个月，这也是我后来反思中觉得应该更早动手的一件事。

顺便说一下 Shared 目录的管理后来又演化了一次。最初我只是要求多团队审批，但实际执行中发现 Shared 目录里的改动频率比预期高很多，审批堆积导致 PR 阻塞。后来我把 Shared 进一步细分了子目录归属：CommonUI 归美术技术团队维护，Configs 归系统团队维护，Audio 由专人负责。只有涉及子目录归属之外的新增或结构性改动才需要跨团队审批。这样既保持了管控力度，又不至于把日常开发卡住。

## 第四刀：发布审批链

前三刀解决的是日常开发中的问题，第四刀对准的是发布环节的风险。

之前的发布流程说白了就是没有流程——谁想打包就去 Jenkins 上点一下，打完了就发。出了问题回滚全靠经验和运气。有一次周五下午有人随手发了个版本，结果线上出了问题，周末三个人加班排查到半夜。从那次之后，所有人都认同发布环节需要管控了。

我设计的发布审批链：

**1. 发布单**

每次发布前必须填写发布单，内容包括：

- 本次发布包含的功能列表（关联的 PR 或 JIRA ticket）
- 影响范围评估（哪些系统可能受影响）
- 回滚方案（如果出问题，具体回退到哪个版本，需要什么操作）
- 发布负责人

回滚方案这一条是强制的。一开始有人敷衍填"回退上个版本"，我要求必须写清楚具体的 commit hash 和操作步骤。理由很简单：出问题的时候大家都慌，提前写好方案比临时想靠谱得多。

**2. 审批**

发布触发需要至少 2 人审批：发布负责人 + 至少 1 个其他团队的 lead。周末或节假日前的发布需要 3 人审批。

审批通过后，Jenkins job 才会解锁，允许执行。

**3. 自动化冒烟测试**

构建完成后不直接发布，先跑一轮自动化冒烟测试：

- 客户端能否正常启动
- 核心流程是否跑通（登录 → 进入主城 → 打开背包 → 进入战斗）
- 关键 UI 是否正常渲染

冒烟测试通过后才会推进到正式发布环节。如果失败，自动通知发布负责人和审批人，流程暂停。

**4. 发布后监控**

发布完成后 30 分钟内，由发布负责人确认线上状态，在 DingTalk 群里发确认消息。如果发现问题，按发布单中的回滚方案执行。回滚操作也走 Jenkins job，一键执行，不需要手动操作 Git，降低紧急情况下出错的概率。

这整套流程看起来步骤不少，但实际执行下来，从填发布单到完成发布大概 1-1.5 小时。和之前"随手一打"相比多了流程成本，但换来的是发布事故率大幅下降。

有一点我觉得特别值得提：回滚方案的强制填写看起来是个小事，但它改变了大家对发布的心态。以前大家觉得发布就是"往前走"，出问题再说。现在填发布单的时候被迫想一遍"如果失败了怎么办"，这个思考过程本身就在降低风险。好几次我看到有人在填回滚方案的时候发现自己的改动缺少向后兼容，主动改了方案再发。这种效果是流程设计时没预料到的。

## 落地数据

治理前后的关键指标对比：

| 指标 | 治理前 | 治理后 |
|------|--------|--------|
| 每周构建失败次数 | 3-5 | 0-1 |
| 资源覆盖事故 | 每周 2-3 次 | 每月 < 1 次 |
| 合并冲突平均解决时间 | 2-4 小时 | 20-30 分钟 |
| 发布回滚次数 | 每月 2-3 次 | 每季度 < 1 次 |
| CI 平均反馈时间 | 无（手动构建 45min） | 快速检查 4min / 完整检查 17min |

数据不是一步到位的。整个治理从开始到数据稳定下来，前后经历了大约四个月。

分支策略落地后第一周构建失败就从 5 次降到了 2 次，这是见效最快的一刀。CI 门禁上线后又降了一档，基本稳定在每周 0-1 次。但资源覆盖问题到目录归属完全落地之后才真正收敛，中间又经历了两个月的阵痛期。发布回滚的改善最慢，因为发布审批链是最后上的，而且需要冒烟测试积累足够的用例覆盖才能真正发挥作用。

还有一组数据没写进表格但我自己很看重：团队在合并冲突和构建修复上花费的时间，从每周平均 12-15 人时降到了 2-3 人时。省下来的时间回到了功能开发上，这才是治理真正的价值。

还有一个不太好量化但能明显感知到的变化：团队之间的互相甩锅变少了。以前构建挂了，群里第一条消息通常是"谁推的？"现在 CI 会明确告诉你哪个 PR 引入了问题，DingTalk 通知直接 @ 到人，不需要互相猜。问题定位从"社交行为"变成了"自动化行为"，团队关系也缓和了不少。

## 反思

**做对了的事：**

先解决最痛的问题。构建失败是所有人都感受到的痛点，先把分支策略和 CI 门禁落地，大家看到效果之后就愿意配合后面的改动了。如果一上来就推资源目录迁移或者发布审批链，阻力会大很多——大家还没有切身感受到治理的好处，凭什么花时间配合你改？

CI 门禁分级的设计也很关键。如果所有检查都堆在一起要跑 20 分钟，开发者推一行代码就要等半天，体验太差会导致大家想办法绕过去。快速检查 5 分钟以内出结果，日常开发不阻塞，完整检查只在合入 develop 时才强制，这个节奏是对的。

通知分群推送。把 CI 消息按团队分流，减少噪音，提高关注度。这个改动很小但效果很好。

**可以做得更好的事：**

资源目录归属应该更早定。这件事我拖了差不多两个月才开始推，原因是觉得存量资源迁移工作量大，想等"合适的时机"。但事实是永远不会有合适的时机，越早定规则越好，存量的慢慢迁就行。这两个月里因为资源覆盖引发的问题完全可以避免。

面向策划和美术的培训做得不够。我把分支策略和 CI 规则写成了文档发到 Wiki 上，觉得大家看了就能理解。但实际上策划和美术同事对 Git 操作并不熟练，文档里的概念他们看不太懂。后来补做了几次面对面的工作坊，手把手带着操作一遍，效果比文档好太多。早知道应该一开始就做这件事。后来我把工作坊的内容整理成了一份 Git 操作速查卡，贴在每个人工位旁边，新人入职的时候也发一份，这个做法后来成了团队的标准流程。

回滚演练应该定期做。虽然发布单里要求写回滚方案，但如果从来没演练过，真出问题的时候方案执行也会打折。后来我推动每月做一次回滚演练，把回滚当成常规操作而不是紧急事件，团队的应对能力提升了不少。

监控和告警链路可以更完善。发布后的监控主要靠人盯，缺少自动化的健康检查和告警。如果发布后 5 分钟内某些关键指标（崩溃率、报错率）超过阈值就自动告警甚至自动回滚，会更安全。这是我离开时没来得及做的事。

跨团队 PR 的 review 效率可以再提高。规则要求至少一个跨团队 reviewer，但不同团队的工作节奏不一样，有时候 reviewer 响应慢导致 PR 挂了一两天。后来试过设置 review SLA（12 小时内必须响应），配合 DingTalk 定时提醒超时未审核的 PR，效果有改善但还不够理想。这件事的根本解法可能不是流程规则，而是需要更好的跨团队沟通文化。

## 这篇真正想留下来的结论

工程治理不是一个技术问题，是一个信任建设问题。

技术方案其实都不复杂。分支策略、CI pipeline、CODEOWNERS、发布审批，这些东西网上搜一搜都能找到现成方案。难的是让 30 多个人愿意改变习惯，接受约束，遵守规则。

我的经验是不要一上来就推"完美方案"。先找到大家都痛的点，解决它，让团队看到效果。然后再一步步加规则。每一步都要有数据支撑，能说清楚"加了这个规则之后，构建失败减少了 X 次，每周省了 Y 小时"。人不是不讲道理，只是需要看到改变确实有效。

还有一点：治理的目标不是加规则，是减少摩擦。如果一个规则加上去之后大家每天多花 30 分钟在流程上，但只省了 10 分钟的事故处理时间，那这个规则就不值得加。好的治理应该像基础设施一样，平时感觉不到它的存在，但它一直在默默保护着整个团队的开发节奏。

我还想强调一点：治理方案不能一成不变。团队规模会变、项目阶段会变、痛点也会变。当初设计的规则，过半年可能就不合适了。所以我在每个月的技术复盘会上都会专门留 15 分钟回顾治理规则——哪些规则大家觉得有用，哪些觉得多余，哪些地方还有新的痛点。保持这个反馈循环，治理才能活下去，而不是变成一堆没人遵守的死规矩。

如果让我用一句话总结整个治理过程中最重要的原则，那就是：每加一条规则，先想清楚它保护了什么、付出了什么代价。保护的价值大于代价，就加；否则就不加。听起来简单，但实际操作中很容易忘记——尤其是当你已经尝到了治理的甜头之后，会本能地想加更多规则。克制这种冲动，是做治理的人最需要练的功夫。

最后说一件小事。治理完全落地大概三个月之后，有一次新来了一个同事，他问"你们这里怎么从来不出构建问题？"老员工说"不是从来不出，是以前天天出，后来有人一刀一刀给治好了。"

听到这句话的时候我觉得，这件事做得值。
