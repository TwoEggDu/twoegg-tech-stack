## 一句话总结

一个 500 行的 EditorWindow 脚本，经过两次关键判断，变成了跨 3 个项目复用的资源检查平台。这个过程里最难的不是写代码，而是每次都要回答同一个问题：现在该动它吗？

## 阶段一：项目内的脚本（该不该动它？）

故事的起点很普通。

项目里美术资源频繁出问题：贴图尺寸超标、prefab 命名不规范、依赖关系混乱。每次出问题都是提测阶段才发现，改一个经常连带改三四个。最严重的一次，一张 4096 的贴图混进了 UI 图集，直接导致低端机内存爆了，线上紧急出了一个热更。

于是我写了一个 EditorWindow 脚本，大概 500 行，功能很直接：扫描指定目录下的 prefab，按写死的规则逐条检查，结果输出到 Console。

它解决了一个真实问题：美术同学提交前能自己跑一次，至少不会带着明显的规范问题进版本。TA 不用再一个个去盯着看，提测前的资源问题少了不少。

但它的问题也很明显：

- 路径是写死的，换个目录结构就不能用
- 检查规则硬编码在代码里，改规则就是改脚本
- 输出格式是 Debug.Log，没法做持久化，也没法和 CI 串
- 完全没有配置化的概念，一个项目一份代码
- 新增一条检查规则，要在一个巨大的 switch-case 里加分支

当时有一瞬间我想"要不要重构一下，做得更通用"。甚至已经在脑子里画了一个规则注册 + 配置驱动的架构图。

我最后的判断是：不动。

原因很简单：它只服务一个项目，这个项目的美术同学已经习惯了现有的使用方式，改了反而要重新教。而且这个阶段我手上还有别的事情，不应该把精力花在一个"够用"的东西上。

还有一个更现实的原因：我当时对"通用化"应该做到什么程度其实没有信心。没有第二个使用场景做参照，你很难知道哪些东西该抽象、哪些该保留。凭空做出来的抽象，大概率是错的。

这是第一个判断点。回头看，我觉得这个判断是对的。

很多工具链的过度设计，都是在这个阶段发生的——东西还只服务一个场景，就急着做抽象。结果抽象层做出来了，真正的第二个场景出现时，发现抽象的方向完全不对，还得重来。

我后来跟几个做工具链的朋友聊过，发现大家踩的坑出奇一致：第一个版本做太多了。不是功能做太多，而是"为将来做"的部分做太多了。

## 阶段二：第二个项目需要类似功能（该抽吗？）

这个阶段是整个过程中判断难度最大的。因为第一阶段的答案很明显（不动），第三阶段的答案也很明显（既然决定抽了，就按最小可用来做），但第二阶段是一个真正的分叉口：抽和不抽都有合理的理由。

大概半年后，公司开了第二个项目。新项目的 TA 找到我，说他们也需要类似的资源检查功能。他发了一条消息，大意是"听说你们项目有个资源检查的工具，能不能给我们也来一份"。

我的第一反应是：把脚本复制一份，改改路径和规则就行了。

这个直觉不是没有道理。复制一份，快，不影响老项目，新项目也能立刻用。半天就能搞定，还不用碰老项目的代码。

但我多想了一步：如果复制了，一年后会怎样？

我之前在另一个团队见过这种情况。同一个打包脚本，复制到三个项目里之后，各自加了不同的参数、不同的异常处理、不同的输出路径。半年后有人发现一个 bug，修了自己项目里的版本，另外两个项目根本不知道这个 bug 存在。等到他们也踩到的时候，已经是线上问题了。

所以答案很清楚：两边各自演化，各自加规则，各自改输出格式，各自踩各自的坑。一年后这两个脚本除了最初的骨架，可能已经完全不一样了。到时候如果有第三个项目，又是从头来一遍。

所以问题变成了：这两个项目的检查逻辑，到底是"同构"的，还是"本质不同"的？

我当时的判断标准是这样的：

**如果规则和执行逻辑是同构的，只是配置不同，就该抽。**

什么叫同构？就是两个项目虽然检查的东西不完全一样，但它们做的事情本质上是一回事：遍历资源 -> 对每个资源执行一组规则 -> 输出结果。区别只在于遍历哪些目录、执行哪些规则、结果输出到哪里。

**如果两个项目的检查逻辑本质不同，就不该抽。**

什么叫本质不同？比如一个项目是规则化的静态检查，另一个是运行时的性能采样。虽然都叫"资源检查"，但执行模型完全不一样，强行统一只会让两边都难受。

这次的情况是：两个项目都是静态检查，都是扫描 prefab 和贴图，规则类型一致（贴图大小、命名规范、依赖关系），只是具体阈值和目录结构不同。

判断结论：同构，该抽。

做这个判断的时候，我还额外确认了一件事：第二个项目的需求不是"今天要用"，而是"下个月要用"。这意味着我有一个月的窗口期来做抽象，不需要赶工。如果新项目说"明天就要"，我可能还是会先复制一份顶上，再慢慢做抽象。时间压力会影响判断，这一点也要考虑进去。

这是第二个判断点。我觉得关键不在于"要不要复用"这个问题本身，而在于你得有一个判断框架。不是所有"看起来相似"的东西都该合并，也不是所有"分开维护"都是浪费。

这里还有一个容易忽略的点：判断同构性的时候，不要看表面的功能描述，要看执行模型。"资源检查"这四个字可以涵盖的东西太多了，从静态扫描到运行时 Profiler 都算。只有当两个项目的执行流程是"同一个模板、不同的参数"的时候，才是真正的同构。

## 阶段三：平台化（怎么抽？）

决定抽之后，下一个问题是怎么抽。

这里最容易犯的错误是做成一个大而全的框架。我见过不少团队，一旦决定做平台化，就开始设计插件系统、规则注册机制、UI 配置面板、权限管理、历史记录数据库。结果平台还没落地，光架构讨论就花了两个月。

我的做法是反过来：先想清楚"最小可用"是什么，再一步步加。

当时我给自己定了一个约束：第一版只做到"两个项目都能跑起来"就行。任何不影响这两个项目当前检查流程的功能，全部放到后面再说。UI 面板不做，历史记录不做，自动修复不做。先让核心流程跑通。

这个约束后来被证明非常重要。因为第一版跑起来之后，我们才发现一些真正的需求是什么。

比如一开始没想到的：不同项目对"同一条规则失败"的严重程度定义不一样。A 项目觉得贴图超过 2048 是 Error，必须修；B 项目觉得是 Warning，提示一下就行。这种需求不是提前设计能想到的，是跑起来之后项目组自己提出来的。

还有一个需求是检查结果的输出格式。一开始只输出到 Console，但 A 项目想要 CSV 给策划看，B 项目想接到 CI 的报告里。如果一开始就设计输出系统，我可能会做一个很复杂的插件化方案。但因为先跑起来了，我知道真正需要的只是两种格式，写两个输出器就够了。

### 架构决策

最终的架构思路只有三条：

**第一，把"规则"和"执行器"分离。**

原来的脚本里，规则和执行逻辑是混在一起的。一个方法里既有"遍历目录"的代码，也有"判断贴图大小"的逻辑。抽出来之后，每条规则变成一个独立的类，实现一个统一的接口：输入是一个资源引用，输出是检查结果。

执行器只负责：拿到规则列表 -> 遍历资源 -> 逐条执行 -> 收集结果。它不需要知道具体在检查什么，只需要知道怎么跑。

**第二，规则用配置驱动。**

最初考虑过 JSON，后来选了 ScriptableObject。原因是 Unity 项目里 ScriptableObject 更自然，美术和 TA 可以直接在 Inspector 里看和改，不需要理解 JSON 格式。而且 ScriptableObject 可以直接序列化 Unity 对象引用，比如你可以配一条规则说"检查这个目录下的所有 prefab"，直接拖文件夹进去就行。

每个项目维护自己的一组规则配置文件，放在项目目录里，不进公共 Package。这一点很重要，因为它保证了项目之间的配置完全隔离，A 项目的规则变更不会意外影响 B 项目。

**第三，执行器作为 Unity Package 分发。**

执行器、规则接口、基础规则实现，这些公共部分打成一个 UPM Package。每个项目通过 manifest.json 引用，版本号锁死。

这样每个项目只需要关心两件事：我要检查什么（规则配置），我用哪个版本的检查器（Package 版本）。

回过头看，这三条决策没有一条是多么精妙的设计。但正是因为简单，它们才容易被执行、容易被理解、容易被新人接手。我见过太多"精心设计"的平台，最后死在"只有写它的人才懂"上面。

### 维护边界

架构定了之后，比架构更重要的是维护边界。

技术方案好画，但如果三个项目都能直接改 Package 源码，用不了两个月就会乱。

我们定下来的规则是：

**Package 由工具组维护，项目组提需求但不直接改源码。**

这一条听起来很官僚，但实际上是保护所有人。项目组如果直接改 Package，他们的改动可能影响其他项目；工具组如果不收口，就没有人对 Package 的稳定性负责。

**版本锁定：项目组锁定 Package 版本，不被迫升级。**

Package 新版本发布后，不会强制所有项目升级。每个项目自己决定什么时候升级到新版本。这样即使新版本有问题，也只影响选择升级的项目。

**回退方案：任何项目可以随时回退到上一版本。**

manifest.json 里改一行版本号就能回退。这不是什么高深的设计，但它保证了一个基本的安全感：升级不是单行道。这个方案后来真的用上了一次——某个新版本的规则引擎在特定条件下会误报，影响了一个项目的提测流程。项目组当天就回退到上一版本，业务完全没受影响。

这三条规则看起来都很简单，但我后来发现，绝大多数平台化失败的案例，问题都不在技术方案上，而是在维护边界上。谁能改、谁审核、谁决定升级时机，这些事情不提前定好，技术再好也会乱。

举一个反面例子。我知道一个团队做了一个很漂亮的通用构建系统，架构设计得非常干净。但他们没有定好维护边界，任何项目都可以直接往主仓库提 PR。结果三个月后，主仓库里塞满了各种项目特有的 workaround，每个项目都在里面加自己的分支逻辑。最后这个"通用"系统变成了一个谁都不敢碰的庞然大物。

所以我的经验是：在开始写第一行平台代码之前，先把维护规则写下来。哪怕只是一个简单的文档，列清楚谁负责、怎么提需求、怎么升级、怎么回退。这份文档的价值不低于代码本身。

## 抽出来之后的真实代价

做完之后有人问我：值吗？

这个问题在平台化之前很难回答，因为你只能估算。但做完之后可以回头算一笔实际的账。我把成本分成四类，尽量如实记录。

**初始成本：大约 2 人周。**

包括规则引擎的接口设计和基础实现、配置系统（ScriptableObject + 序列化）、Package 打包和分发管线搭建、已有规则从老脚本迁移过来、以及两个项目各自的配置文件初始化和验证。两个人干了一周，不算轻松，但也没有超出预期。

其中最花时间的不是写新代码，而是把老脚本里硬编码的规则一条条拆出来。有些规则看起来是独立的，实际上和上下文耦合得很深，比如某条规则里直接引用了项目特定的 Shader 名字。这些都得一个个处理。

还有一块意外的工作量是测试。原来的脚本没有测试，跑一下看结果就行。但 Package 化之后，你得保证新版本不会破坏已有功能。我们后来补了一组基础的单元测试，虽然覆盖率不高，但至少保证核心的规则匹配逻辑是对的。

**持续成本：每月大约 2-3 天。**

主要花在处理跨项目的需求差异上。比如 A 项目要加一条新规则，B 项目觉得这条规则没用不想要，C 项目想要但阈值不一样。这些事情靠配置化能解决一部分，但总有一些边界情况需要人去判断和协调。还有一类成本是 bug 修复：公共 Package 里的 bug 一旦被发现，三个项目都需要验证修复结果，测试量直接乘三。

**隐性成本：三个项目的需求优先级冲突。**

这是我事前没有充分估计的。三个项目都觉得自己的需求最紧急，工具组只有两个人，排期经常冲突。后来不得不建立一个简单的需求池和优先级机制，每两周对一次齐。有一次 A 项目急着上线，要求加一条特殊规则，但这条规则的实现方式会影响 Package 的公共接口。如果直接加，B 和 C 项目都要受影响；如果走配置化方案，至少多花三天。这种权衡在单项目时代是不存在的。

**收益：新项目接入从 2 周降到 2 天。**

第三个项目接入的时候，只花了两天就跑起来了：引入 Package、创建自己的规则配置、跑通一轮检查。对比第一个项目当初从零写脚本花了将近两周，这个效率提升是实实在在的。而且这个收益会随着项目数量增加而放大。如果有第四个、第五个项目，接入成本基本就是半天的事。

除了效率收益，还有一个不太好量化但很实在的好处：三个项目的检查规则虽然不完全一样，但它们共享同一套规则引擎。这意味着任何一个项目发现的规则引擎 bug，修复后所有项目都受益。规则的写法也趋于统一，项目之间可以互相参考，新人上手也更容易。

还有一个很容易被忽略的成本：文档和沟通。Package 做出来之后，不是丢给项目组就行了。你得写接入文档、录使用视频、回答各种群里的问题。第一个月光是回答"怎么加自定义规则"这一个问题，我大概回复了十几次。后来不得不写了一个 FAQ 页面。

所以值不值？我的结论是：如果你确定有三个以上的项目要用，并且逻辑同构，那值。如果只有一个项目，或者逻辑不同构，那大概率不值。

这不是一个纯技术问题，它本质上是一个投入产出的判断。而且投入产出的比值会随着时间变化：短期来看，平台化一定是亏的（初始成本 > 短期收益）；中长期来看，如果项目数量确实在增长，平台化才会变得划算。所以做这个决策的时候，你对公司未来半年到一年的项目规划要有一个基本的判断。

## 什么情况下不该抽

做完这件事之后，我反而更清楚什么时候不该抽了。这其实是整个经历中我觉得最有价值的部分。因为"该抽"的情况相对好判断，信号也比较明确；"不该抽"的情况才是坑最多的地方，而且一旦做了错误的决定，回头的成本非常高。

**只有 1 个项目用，且没有第二个项目的确定计划。**

注意是"确定计划"，不是"可能以后会有"。如果只是"可能"，那就等真的有了再说。提前抽象的代价几乎总是被低估的。你花了两周做出来一个平台，结果第二个项目永远没来，这两周就是纯粹的浪费。更糟的是，你为了"通用"而引入的抽象层，会让当前项目的代码变得更难理解、更难改。

**两个项目的核心逻辑不同构。**

前面说过，如果执行模型不一样，强行统一只会让两边都变复杂。宁可各自维护两套简单的，也不要维护一套复杂的。判断同构性的时候，不要被功能名称迷惑。两个都叫"资源检查"的东西，底层可能完全不一样。一个是遍历文件做规则匹配，另一个是 Runtime 采样做性能分析，这两个东西不应该共享一个框架。

**团队没有独立的工具维护人力。**

平台化之后，Package 需要有人持续维护。如果没有专门的工具组或者至少一个固定的人负责，那平台化之后的维护成本会落到所有项目头上，反而比各自维护一份脚本更累。我见过一种更糟糕的情况：平台做出来了，但维护的人调走了，Package 变成了没有人敢碰的黑盒。三个项目锁在旧版本上不敢升级，新需求也没法加。这比没有平台更糟。

**"以后可能有用"不是抽的理由。**

这句话我见过太多次了。每次有人提议做平台化，理由都是"以后可能有别的项目也会用"。但"以后可能"是最不可靠的需求来源。真正可靠的信号是：现在已经有第二个项目在问你要了。

还有一种常见的情况值得单独说：**领导说"做成通用的"。**

这种情况我遇到过不止一次。领导的出发点通常是好的，但"做成通用的"这句话本身不构成需求。你需要追问：通用给谁用？有几个项目确定要用？它们的执行模型一样吗？如果答案是"目前就这一个项目，但以后可能有别的"，那我的建议是先把当前项目做好，等真的有第二个项目的时候再谈通用化。

当然这不是说要和领导对着干。更好的做法是在当前项目里有意识地把代码写得干净一点、耦合松一点，为将来可能的抽取留一些空间。但不要现在就动手抽。这和过度设计之间的区别是：你没有引入额外的抽象层，你只是在写更干净的代码。

我后来给自己总结了一条很简单的经验法则：如果你犹豫该不该抽，那就别抽。等你不犹豫的时候再抽。犹豫本身就说明信号还不够强。

## 反思

回过头看这整个过程，前后大概跨了一年多的时间。我觉得最有价值的不是最终那个 Package，而是这三次判断本身。

第一次判断：脚本够用，不动。这个判断帮我避免了过早抽象。

第二次判断：逻辑同构，该抽。这个判断让我在正确的时机做了正确的事。

第三次判断：最小可用，先跑起来再迭代。这个判断让我没有陷入过度设计。

这三次判断的间隔分别是半年和三个月。不是坐下来开个会就能一次性想清楚的，是在实际项目推进中，等到真实信号出现了，才做的决定。

三次判断，每次的核心问题都不是"技术上怎么做"，而是"现在该不该做"。

我越来越觉得，工具链工程师和其他工程师最大的区别不在于代码能力，而在于判断能力。写代码的部分，大部分人都能做到。但什么时候该做、什么时候不该做、做到什么程度就够了，这些判断才是真正拉开差距的地方。

而且这种判断能力没办法通过看书或者上课学到。它来自于你真的在生产环境里做过决策，承受过错误决策的后果，然后慢慢形成自己的判断框架。每个人的框架可能不一样，但一定要有。没有框架的判断就是拍脑袋。

还有一点我觉得值得说。做平台化不是一个技术决策，它是一个组织决策。你不只是在决定代码怎么组织，你同时在决定谁来维护、谁来提需求、谁来排优先级、升级策略是什么、出了问题谁兜底。如果你只想好了技术方案但没想好这些，平台化大概率会失败。

如果让我重新做一次，我会在更早的阶段就把维护边界和升级策略写成文档，而不是等到三个项目都接入之后才补。很多后来的冲突和返工，根源都是最初没有把规则说清楚。技术债好还，流程债难还。

另外一个我会做不同的事情是：更早地建立一个跨项目的沟通渠道。平台化之后，三个项目的 TA 和程序其实是一个隐性的社区，他们会有共同的问题、共同的经验、共同的吐槽。如果有一个群或者一个定期的碰头会，很多问题在变成需求之前就能解决。我们后来建了一个企业微信群，效果比想象中好得多。很多问题是项目之间互相回答的，根本不需要工具组介入。

## 这篇真正想留下来的结论

第一，不要在只有一个使用场景的时候做抽象。脚本够用就让它够用。过早抽象的代价比你想象的大，因为你不知道真正的抽象边界在哪里。

第二，判断该不该抽的核心标准是"同构性"。执行模型一样、只是配置不同，就该抽；执行模型不一样，再相似的表象也不该合并。同构性要看执行流程，不要看功能名字。

第三，平台化的难点不在技术方案，在维护边界。谁改、谁审、谁决定升级，这些比代码架构重要。技术债好还，流程混乱难收拾。

第四，真实代价要提前算，尤其是隐性成本。初始开发成本好估，持续维护、跨项目协调、文档沟通这些成本很容易漏掉。

第五，"以后可能有用"永远不是动手的理由。等到有人真的来问你要的时候，才是动手的信号。真正的需求不需要你猜，它会来找你。

最后补一句：这篇文章讲的是资源检查脚本的案例，但判断框架本身是通用的。无论是构建管线、配表工具、热更方案，还是任何一个可能被多个项目用到的东西，决策过程都一样：先问够不够用，再问是不是同构，最后问谁来维护。把这三个问题回答清楚，大多数情况下答案就出来了。
