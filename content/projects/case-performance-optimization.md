## 一句话总结

接手一个中低端机帧率只有 22fps 的 ARPG 项目，按 GPU → CPU → 内存 → 加载四层拆解，每层只抓 Top 1 瓶颈逐层治理，三周后中低端稳定 45fps，内存从 1.8GB 降到 1.2GB。

## 背景

这个项目是一款 ARPG 手游，目标平台是 Android 和 iOS，核心适配档位是中低端 Android（骁龙 680 级别、Mali-G57 级别的 GPU）。

项目在我接手性能优化之前的状态：

- 主城场景平均帧率 22fps，玩家密集区域会掉到 18fps，肉眼可感的卡顿
- 战斗场景在技能特效密集时掉到 15fps，大招阶段甚至到 12fps
- 内存占用 1.8GB（4GB 设备），3GB 设备上频繁 OOM 闪退
- 首次进主城加载 12 秒，切场景 8 秒
- 线上 crash rate 里有 30% 以上和内存相关

团队之前也做过"优化"——减了一些粒子、合了一些 mesh、压了一些贴图——但都是零散的点状操作，做完之后帧率从 20fps 提到 22fps，效果不明显，而且没有人能说清楚到底是什么在拖性能。

这是我接手时最大的问题：没有人知道瓶颈在哪里。

## 第一步：建基线，不猜

我做的第一件事不是优化，是建测量体系。

选了 3 台代表机型覆盖三个档位：

| 档位 | 设备 | SoC | 内存 |
|------|------|-----|------|
| 低端 | Redmi Note 11 | 骁龙 680 | 4GB |
| 中端 | OPPO Reno8 | 天玑 1300 | 8GB |
| 高端 | 小米 13 | 骁龙 8 Gen 2 | 12GB |

选了 5 个场景覆盖主要玩法路径：

1. 主城（玩家密集）
2. 主城（空闲时段）
3. 副本战斗（3v3 BOSS）
4. 大世界探索
5. UI 主界面 + 背包

每台设备每个场景录 3 个指标：帧率（平均 + P95）、内存占用（PSS）、加载耗时。

这一步花了两天。数据出来之后第一个发现就颠覆了团队之前的假设：

**主城场景是 GPU bound，不是 CPU bound。**

之前团队一直在 CPU 侧做优化——合批、减 GC、降 Update 频率——方向完全跑偏了。Unity Profiler 的 CPU Usage 面板会显示 `Gfx.WaitForPresent` 占了很大比例，很多人以为这是"CPU 太慢了 GPU 在等"，但实际上这恰恰说明 CPU 在等 GPU 完成当前帧的渲染。

用 RenderDoc 抓帧确认，主城场景单帧 GPU 耗时 28ms，而 CPU main thread 是 18ms。瓶颈明确在 GPU 侧。

这就是为什么"先 profile 再动手"这件事值得反复强调——你凭直觉猜的优化方向，可能从一开始就是反的。

## 第二步：GPU 层 — 找到 Top 1

确定 GPU bound 之后，下一步是找 GPU 侧的 Top 1 瓶颈。

### 现象

用 Android GPU Inspector（AGI）分析 GPU 的时间分布，同时配合 RenderDoc 看每个 draw call 的开销。

几个关键数字：

- 主城 Overdraw 达到 4.5x 平均，部分区域（喷泉、NPC 密集区）超过 6x
- 半透明粒子渲染占 GPU 总时间的 35%
- Shadow map 在所有档位都用的 2048 分辨率，低端机上单 shadow pass 就吃了 4ms
- 后处理链（Bloom + Color Grading + FXAA）在低端机上占 3.5ms

用 Xcode Metal System Trace 在 iOS 侧做了交叉验证，结论一致：**半透明粒子和 overdraw 是 GPU 侧最大的单一瓶颈。**

### 治理

GPU 层的治理按开销排序，优先处理收益最大的项：

**1. 粒子系统整治**

这是 Top 1。主城里有大量装饰性粒子特效——喷泉水雾、NPC 头顶光效、建筑环境光、地面灰尘——其中 6 个粒子系统的 screen coverage 超过 30%，相当于每个都在画大半个屏幕的半透明像素。

处理方式：
- 砍掉 3 个纯装饰性全屏粒子（玩家反馈数据显示没有人注意到它们的存在）
- 剩余 3 个改为低开销方案：降粒子数（从数百降到数十）、缩小粒子尺寸、改用预乘 Alpha 混合减少 overdraw
- 所有粒子增加 Max Particles 上限和 screen-space 尺寸限制

单项收益：GPU 粒子渲染时间从 ~9.8ms 降到 ~4.2ms。

**2. 半透明排序与裁剪**

远距离半透明物体（树叶半透明部分、远处水面反射）对画面贡献极小但 overdraw 开销很大。

处理方式：
- 对距离相机超过 50m 的半透明物体做 LOD 切换，替换为不透明近似材质
- 中低端机额外把半透明裁剪距离缩到 35m

单项收益：Overdraw 从 4.5x 降到 2.8x。

**3. Shadow Map 分档**

之前所有档位统一 2048 shadow map，低端机完全没必要。

处理方式：
- 低端机降到 512 分辨率
- 中端机降到 1024 分辨率
- 高端机保持 2048
- 同时把 shadow distance 在低端机从 80m 缩到 40m

单项收益：低端机 shadow pass 从 4ms 降到 1.2ms。

**4. 后处理链精简**

低端机关闭 FXAA（用 MSAA 2x 替代，在 tile-based GPU 上开销更低），关闭 Bloom 的高精度模式。

单项收益：后处理从 3.5ms 降到 1.8ms。

### GPU 层结果

主城场景 GPU time：28ms → 16ms。

这一步完成后，主城帧率从 22fps 提到了接近 35fps。但还没到目标的 45fps——因为 CPU 侧的 18ms 现在变成了新瓶颈。

## 第三步：CPU 层 — 找到 Top 1

GPU 治理完之后再回头看 CPU，Profiler 数据变得干净了很多（之前 GPU bound 时 CPU 数据受等待帧影响会失真）。

### 现象

Unity Profiler 的 CPU 时序分析：

- `Animator.Update`：3.2ms（主城同屏 40+ 角色，每个都在播放完整骨骼动画）
- `Physics.Simulate`：2.1ms（大量 NPC 挂了碰撞体，即使玩家看不到也在参与物理模拟）
- `UI.Bindding`相关：1.8ms（主城 HUD 每帧刷新大量文字）
- GC Alloc：每帧 ~2.4KB，偶发帧 spike 到 40KB+（字符串拼接、LINQ 临时分配）

Top 1 是 Animator.Update 的 3.2ms。40+ 个同屏角色，每个都在用完整的 Humanoid Animator 播放 idle/walk 动画，骨骼数 50-70 根，每帧都在算 IK 和骨骼矩阵。

### 治理

**1. 动画 LOD 系统**

这是 CPU 层的 Top 1 治理项。

核心思路：不在屏幕中心的角色不需要完整动画精度。

实现方式：
- 屏幕占比 < 5% 的角色切换到 simplified animation（只保留 spine 和四肢主骨骼，骨骼数从 60+ 降到 12）
- 屏幕外但仍在加载范围的角色直接暂停 Animator，用位移插值模拟移动
- Animator 的 `cullingMode` 统一设为 `CullCompletely`，确保不可见角色零开销

单项收益：`Animator.Update` 从 3.2ms 降到 1.4ms。

**2. 物理碰撞体管理**

远处 NPC 不需要碰撞交互，但一直在参与 `Physics.Simulate`。

处理方式：
- 距离玩家 > 20m 的 NPC 禁用 Collider 组件
- 距离 > 30m 的直接禁用 Rigidbody
- 用一个全局管理器按距离分帧启用/禁用，避免同帧大量开关引起 spike

单项收益：`Physics.Simulate` 从 2.1ms 降到 0.8ms。

**3. UI 更新频率控制**

主城 HUD 上的玩家名字、血条、小地图标记每帧都在 rebuild。

处理方式：
- 非关键 UI 元素（名字、称号）改为 0.5 秒刷新一次
- 血条只在数值变化时刷新
- 小地图标记用脏标记模式，只在位置变化超过阈值时更新

单项收益：UI 相关开销从 1.8ms 降到 0.6ms。

**4. GC 优化**

排查高频 GC Alloc 来源，用 Profiler 的 GC Alloc 列排序定位。

- 战斗日志的字符串拼接改为 StringBuilder 复用
- LINQ 查询替换为 for 循环
- Delegate 回调缓存避免每帧 new

单项收益：每帧 GC 从 ~2.4KB 降到 ~0.3KB，偶发 spike 消除。

### CPU 层结果

CPU main thread：18ms → 12ms。

配合 GPU 的 16ms，此时帧率已经到了主城 42fps 左右。还差一点到 45fps——但此时帧率已经不是最紧迫的问题了，内存才是。3GB 设备还在频繁闪退。

## 第四步：内存层 — 止血

内存问题的可怕之处在于它不会直接表现为卡顿，而是突然闪退。用户感知是"玩着玩着突然没了"，在 crash report 里看到的是 `LowMemoryKiller` 或 `jetsam`。

### 现象

用 Memory Profiler 做完整快照：

| 类别 | 占用 |
|------|------|
| Texture | 680MB |
| Mesh | 220MB |
| AudioClip | 85MB |
| AssetBundle 常驻 | 150MB |
| Mono Heap | 120MB |
| 其他（Shader, AnimClip 等） | 545MB |
| **总计 PSS** | **~1.8GB** |

几个突出问题：

- **Texture 680MB**：大量角色贴图在低端机上仍然使用 2048 甚至 4096 分辨率，没有按设备档位做降级
- **Mesh 220MB**：所有静态 Mesh 都是 uncompressed，包括很多只用一次的装饰物
- **AB 常驻 150MB**：AssetBundle 加载后没有正确 unload unused assets，引用计数逻辑有 bug 导致部分 bundle 永远不会释放

### 治理

**1. Texture 按档位自动降级**

这是内存层的 Top 1。680MB 的 Texture 内存在低端机上完全不可接受。

处理方式：
- 建立三档 Texture Quality 配置：高端 max 2048、中端 max 1024、低端 max 512
- 通过 `QualitySettings` 结合启动时的设备检测自动选档
- 对 UI 贴图单独处理：UI atlas 在所有档位保持 1024（降太低影响清晰度）
- 角色贴图在低端档用 ETC2 4bit 替代 8bit，进一步压缩 50%

单项收益：Texture 内存从 680MB 降到 低端 320MB / 中端 420MB。

**2. Mesh Compression**

静态场景 Mesh 开启 Mesh Compression（Medium 档）。对视觉影响极小，但内存收益明显。

处理方式：
- 场景静态 Mesh 统一开启 Medium compression
- 角色 Mesh 保持不压缩（压缩会导致 skinning 精度下降）
- 通过 AssetPostprocessor 在导入时自动设置，避免人工遗漏

单项收益：Mesh 内存从 220MB 降到 145MB。

**3. AssetBundle 引用计数修复**

这个 bug 是最隐蔽的。AB 加载管理器里的引用计数在异步加载 + 同步 unload 的交叉场景下会出现计数不归零的情况，导致 bundle 永远不会 unload。

排查方式：
- 在 AB 管理器里加了引用计数的调试面板，实时显示每个 bundle 的引用数
- 发现有 23 个 bundle 在切出场景后引用数仍然 > 0
- 根因：异步加载的回调里持有了对 bundle 的引用，但取消加载时没有清理回调

处理方式：
- 修复取消加载时的回调清理逻辑
- 增加场景切换时的强制 sanity check：如果一个 bundle 不属于当前场景但引用数 > 0，输出警告并强制释放
- 切场景后增加 `Resources.UnloadUnusedAssets()` + `GC.Collect()` 调用

单项收益：AB 常驻内存从 150MB 降到 60MB。

**4. AudioClip 懒加载**

85MB 的 AudioClip 里有一半是战斗音效，在主城场景不需要常驻。

处理方式：
- 战斗音效改为进入战斗时才加载，退出后卸载
- 背景音乐改为 streaming 模式，不占运行时内存

单项收益：AudioClip 常驻从 85MB 降到 40MB。

### 内存层结果

总内存（4GB 设备 PSS）：1.8GB → 1.2GB。

3GB 设备上的 OOM 闪退率从每日 12% 降到 < 1%。

## 第五步：加载层 — 体验优化

到这一步，帧率和内存已经达标。但加载体验还很差——首次进主城 12 秒、切场景 8 秒——在玩家视角这就是"卡了"。

### 现象

用 Profiler 分析加载阶段的时间分布：

- AB 文件 I/O：4.5 秒（大量小文件随机读取）
- Texture 上传 GPU：2.8 秒（大贴图同步上传）
- Shader 编译 / warmup：2.2 秒（首次遇到的 shader variant 运行时编译）
- 场景初始化（Awake/OnEnable）：2.5 秒

### 治理

**1. 高频资源异步预加载**

核心思路：玩家还在看 loading 界面的时候，就把下个场景的高频资源加载完。

处理方式：
- 统计每个场景最先使用的 Top 20 资源（按首次访问时间排序）
- 在 loading 界面阶段异步预加载这些资源
- 实现优先级队列：阻塞性资源（地形、天空盒）> 视觉资源（角色、建筑）> 装饰资源（粒子、小物件）

**2. Shader Warmup**

首次进入游戏时集中 warmup 所有 shader variant，避免运行时首次编译的卡顿。

处理方式：
- 构建时收集所有使用到的 shader variant 组合到 `ShaderVariantCollection`
- 在 splash 界面阶段调用 `WarmUp()` 预编译
- 低端机上把 warmup 拆成多帧执行，避免一次性卡死

**3. AB 加载合并**

之前很多小 bundle（< 50KB）导致大量 I/O 开销。

处理方式：
- 调整 AB 分包策略，合并同场景的小 bundle
- 主城相关 bundle 从 87 个合并到 34 个
- 减少 I/O 次数的同时也降低了 bundle 依赖管理的复杂度

### 加载层结果

| 指标 | 治理前 | 治理后 |
|------|--------|--------|
| 首次进主城 | 12s | 5s |
| 切场景 | 8s | 3s |
| Shader 首次卡顿 | 明显 | 无感知 |

## 最终结果

三周治理完成后的对照数据：

| 指标 | 治理前 | 治理后 | 变化 |
|------|--------|--------|------|
| 主城帧率（中端） | 22fps | 45fps | +104% |
| 战斗帧率（中端） | 15fps | 38fps | +153% |
| 主城帧率（高端） | 38fps | 60fps | +58% |
| 内存 PSS（4GB 设备） | 1.8GB | 1.2GB | -33% |
| OOM 闪退率（3GB 设备） | 12%/天 | <1%/天 | — |
| 首次进主城 | 12s | 5s | -58% |
| 切场景 | 8s | 3s | -63% |

每层治理的贡献分布：

| 治理层 | 帧率贡献 | 关键指标 |
|--------|----------|----------|
| GPU 层 | +13fps | GPU time 28ms → 16ms |
| CPU 层 | +10fps | CPU time 18ms → 12ms |
| 内存层 | 稳定性 | PSS 1.8GB → 1.2GB |
| 加载层 | 体验 | 进城 12s → 5s |

## 反思

做对了的事情：

- **先 profile 再动手，不猜。** 如果按团队之前的直觉继续优化 CPU，可能再做三周也看不到效果——因为瓶颈在 GPU。两天的基线测量避免了几周的无效劳动。

- **每层只找 Top 1，不贪多。** GPU 层最想去做的事情很多——shader 优化、mesh LOD、遮挡剔除——但粒子就占了 35%，把它搞定就够了。如果什么都想做，反而会分散精力，每个都做一点但每个都不到位。

- **每次改动前后都有对照数据。** 不是"做完了跑一圈感觉快了一点"，而是修改前录基线、修改后录对照、数字放到表格里。这让每个决定都有据可依，也让团队里其他人能看懂优化的逻辑。

- **分档治理，不一刀切。** 高端机不需要砍画质，低端机不需要开全特效。通过设备检测自动分档，让不同设备都在自己的能力范围内跑到最优。

可以做得更好的地方：

- **应该更早建立自动化性能回归测试。** 优化是做完了，但如果后续有人加了一个新特效或者改了一个 shader，帧率可能悄悄回退。手动测试没法每天跑，应该在 CI 上挂性能回归，每次构建自动跑基准场景，帧率掉了就报警。这是后来才补上的。

- **Texture 档位切换应该在构建时处理，而不是运行时。** 当时做的方案是运行时根据设备档位设置 `QualitySettings.masterTextureLimit`，这意味着高分辨率贴图还是会被打进包里、加载到内存里、然后再降采样。正确的做法是在构建时就按档位生成不同的 texture asset，从源头控制包体和内存。

- **AB 引用计数的 bug 不应该存在那么久。** 如果一开始就有 AB 加载的调试面板和 sanity check，这种问题在开发阶段就能发现，不会拖到需要专项治理才暴露。

## 这篇真正想留下来的结论

性能优化不是一种直觉，是一种工程方法。

"帧率低了就砍粒子""内存高了就压贴图"——这种思路偶尔能撞到对的答案，但更多时候是在浪费时间。

真正有效的做法是：

1. **建基线**：在代表设备上录可对比的数据
2. **分层定位**：GPU / CPU / 内存 / 加载，先确定瓶颈在哪层
3. **层内找 Top 1**：每层只抓最大的那个瓶颈
4. **治理并验证**：改完之后用同样的方法验证
5. **推进到下一层**：一层 OK 之后再看下一层

每层只做 Top 1 这个原则，不是懒，是效率。一个项目在任意时刻真正拖性能的瓶颈通常只有 1-2 个。找到它、解决它、验证它，比做十个"可能有用"的优化有效得多。

工具也好、技巧也好，都只是手段。真正决定优化成败的，是"先测后动"这四个字。
