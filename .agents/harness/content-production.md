# TechStackShow 内容生产 Harness v0

> 这份文件回答的是：AI 参与一篇 TechStackShow 文章时，应该怎样读取上下文、推进流程、触发验证、沉淀反馈。

## 1. 定位

内容生产 Harness 不是新的写作规范，也不是替代现有 `col-*` 技能的总 prompt。

它的职责是把分散的规则和技能串成一条可复用流水线，让 AI 在明确边界内完成以下动作：

- 知道先读哪些上下文
- 知道每个阶段该调用哪类能力
- 知道什么时候只输出工作材料，不能直接写正文
- 知道哪些检查必须跑
- 知道失败经验应该沉淀到哪里

第一版目标不是追求全自动发文，而是让“选题到可审稿初稿”这条链路稳定、可复盘、可逐步提高自动化比例。

## 1.1 跨 Agent 原则

本 Harness 的核心规程必须保持 agent-neutral。

也就是说：

- 核心流程写在普通 Markdown 文档里
- 阶段输入、阶段输出、停止点、验证口径不绑定某一个 AI 产品
- Codex 的 `col-*` skills 只是当前仓库里的一个适配层
- Claude / DeepSeek / 其他 Agent 不支持 skills 时，应直接读取对应 `SKILL.md`，把它当成阶段提示词使用

跨 Agent 适配细则见：

- `.agents/harness/agent-compatibility.md`

## 2. 第一版适用范围

v0 只覆盖 TechStackShow 的内容生产，不覆盖代码仓库的 bug fix / feature / PR 流程。

适用任务：

- 新文章选题定位
- 技术主题深挖
- 工程验证设计
- 初稿骨架生成
- 文章风险审查
- 解释桥接审查
- 系列一致性审查
- 最小必要改写
- Hugo 构建前检查

暂不覆盖：

- 自动生成整篇最终正文
- 自动发布
- 自动改写系列规划
- 从私仓源码直接同步到公仓正文

## 3. 上下文入口

每次执行内容生产任务前，AI 默认按任务深度读取上下文。

### 3.1 必读上下文

- `AGENTS.md` 或当前会话中的项目规范
- `docs/article-production-workflow.md`
- `docs/article-writing-method.md`
- 根 `doc-plan.md`，只用于系列级路由
- 目标系列的 `docs/*-series-plan.md`

### 3.2 按需读取

- `docs/article-outline-template.md`：起 outline 或初稿骨架时读取
- `docs/*-execution-plan.md` / `docs/*-outline.md` / `docs/*-workorders.md`：已有执行层材料存在时读取
- `content/<section>/<slug>.md`：需要判断相邻文章、重复风险、文风或链接时读取
- `kb/`：需要知识沉淀或历史决策时读取，不能替代 `docs/` 的前置规划职责

### 3.3 禁区

- 不把 `GameEngineDev` 私仓 C++ 源码直接复制进本仓正文
- 不把篇级清单回填到根 `doc-plan.md`
- 不绕过系列 `*-series-plan.md` 直接新增孤立文章

## 4. 标准流水线

默认流水线如下：

```text
选题输入
-> 编辑定位
-> 技术深挖
-> 验证设计
-> 初稿骨架
-> 人工确认主线
-> 正文起草
-> 风险审查
-> 解释桥接
-> 系列一致性
-> 最小改写
-> Hugo 构建
-> 回写系列状态
-> 反馈沉淀
```

不是每篇都必须完整跑完。跳过某个阶段时，需要能说清原因。

## 4.1 Agent 交接状态

如果一次任务可能在 Codex、Claude、DeepSeek 之间切换，每个阶段结束时都应输出一个简短的 `Harness State`：

```text
Harness State
- Task: [当前文章或任务]
- Stage: [当前阶段]
- Context read: [已读取的关键文档]
- Decisions: [已经确认的边界和判断]
- Open questions: [需要作者或下一个 agent 确认的问题]
- Next stage: [建议下一步]
- Changed files: [已修改文件，没有则写 none]
- Verification: [已运行检查，没有则写 not run + 原因]
```

这个状态块是跨 Agent 交接的最小协议。

## 5. 阶段定义

### 5.1 编辑定位

触发条件：

- 新选题
- 老选题边界不清
- 系列位置不确定

使用能力：

- `col-editor`

输出：

- 编辑工作单
- 本篇唯一职责
- 与前后文章的重复风险
- 推荐二级标题结构
- 明确不展开内容

停止点：

- 如果系列位置不清，先停下让作者确认，不直接进入深挖或写作。

### 5.2 技术深挖

触发条件：

- 选题涉及机制、架构、引擎行为、工程取舍
- 文章不是纯经验随笔

使用能力：

- `col-drilldown`

输出：

- 七层展开分析稿
- 机制、版本条件、工程代价、误解、项目决策
- 不确定处标注 `[待核查]`

停止点：

- 如果关键结论缺版本、平台或源码/文档依据，不进入正文，只进入验证设计或资料补查。

### 5.3 验证设计

触发条件：

- 文章包含可实验验证的技术结论
- 文章出现具体数字、性能判断、版本差异、平台差异

使用能力：

- `col-verify`

输出：

- 要验证的结论
- 最小测试场景
- 控制变量
- 观测指标与工具
- 结果不符时的排查方向

停止点：

- 如果验证需要真实项目数据或截图，使用 `DATA-TODO` / `EXPERIENCE-TODO` 标记，不能编造结果。

### 5.4 初稿骨架

触发条件：

- 已有编辑工作单
- 至少已有深挖分析稿或验证设计稿之一

使用能力：

- `col-draft`

输出：

- 文章信息
- 开头段策略
- 各 H2 的职责、核心论点、术语、示例位置、边界提醒
- 结尾策略与文末导读

停止点：

- 骨架生成后默认等待作者确认主线，不直接扩写最终正文。

### 5.5 正文起草

触发条件：

- 作者确认骨架
- 文章目标、系列位置、边界已明确

执行规则：

- 遵守 `docs/article-writing-method.md`
- 第一屏先立问题，不先讲 API
- 中间必须有抽象模型
- 最后落回具体实现或工程决策
- 需要真实数据处使用 `DATA-TODO`
- 需要项目经验处使用 `EXPERIENCE-TODO`

停止点：

- 如果正文会依赖私仓 C++ 源码细节，先确认目标仓库与公开边界。

### 5.6 审查与最小改写

触发条件：

- 正文初稿完成
- 或作者粘贴外部文章 / 旧文要求审查

推荐顺序：

```text
col-risk
-> col-bridge
-> col-consistency
-> col-rewrite
```

各阶段职责：

- `col-risk`：只找高风险句，不润色全文
- `col-bridge`：只找读者卡点和最小补桥方案
- `col-consistency`：只查系列一致性、重复和导航问题
- `col-rewrite`：只修改已确认有问题的地方

停止点：

- 风险、桥接、一致性问题还没确认前，不做大范围重写。

### 5.7 构建验证

触发条件：

- 修改了 `content/`
- 修改了系列索引页
- 修改了 shortcode / frontmatter / relref

必须执行：

```bash
hugo
```

验收口径：

- `ERROR` 为零才可提交
- `REF_NOT_FOUND` 指向计划中未写文章时，按项目规范可单独说明，不视为同类构建错误
- YAML 引号和 shortcode 引号是优先排查项

### 5.8 回写状态

触发条件：

- 新文章完成
- 系列索引或计划状态变化

回写顺序：

1. 目标系列 `docs/*-series-plan.md`
2. 相关系列索引页
3. 根 `doc-plan.md`，仅当系列级路由变化时修改

禁止事项：

- 不把篇级清单回填到根 `doc-plan.md`
- 不用执行层文档反向覆盖 canonical series plan

## 6. 反馈沉淀

每次出现可复用问题时，优先沉淀到对应位置。

| 问题类型 | 沉淀位置 |
|---------|---------|
| 项目级规则遗漏 | `AGENTS.md` |
| 内容生产流程遗漏 | 本文件 |
| 写法判断问题 | `docs/article-writing-method.md` |
| 单篇推进问题 | `docs/article-production-workflow.md` |
| 系列结构变化 | 对应 `docs/*-series-plan.md` |
| 需要真实数据 | 正文内 `DATA-TODO` |
| 需要项目经验 | 正文内 `EXPERIENCE-TODO` |

反馈沉淀原则：

- 只沉淀会复现的问题
- 规则必须可执行，不能只写口号
- 新规则应说明触发条件和处理动作
- 不把一次性偏好写成全局规则

## 7. v0 指标

每次完整跑一篇文章，记录以下 5 个指标：

| 指标 | 记录方式 |
|------|---------|
| 一次交付可用率 | 作者无需大改即可进入下一阶段的输出比例 |
| 人工返工点 | 作者主要改了结构、事实、风格、边界还是链接 |
| 规则违例数 | frontmatter、relref、系列规划、私仓隔离等违规次数 |
| 验证通过率 | `hugo` 是否一次通过，若失败记录原因 |
| 反馈沉淀率 | 本次失败是否沉淀为规则、检查项或 TODO |

第一版不追求复杂量化，只追踪问题是否重复出现。

## 8. 一周落地计划

### Day 1：固定试点任务

只选择一篇真实选题，目标是走通“选题到初稿骨架”。

产出：

- 编辑工作单
- 深挖分析稿
- 验证设计稿
- 初稿骨架

### Day 2：补上下文缺口

检查 Day 1 中 AI 缺了哪些上下文。

产出：

- 补充本文件的上下文入口
- 必要时补系列 plan 或执行层文档

### Day 3：扩写正文

基于确认后的骨架扩写正文。

产出：

- `content/<section>/<slug>.md`
- `DATA-TODO` / `EXPERIENCE-TODO` 标记

### Day 4：审查

按顺序执行风险、桥接、一致性审查。

产出：

- 高风险句清单
- 解释卡点清单
- 系列一致性问题清单

### Day 5：最小改写与构建

只改确认有问题的位置，然后执行 Hugo 构建。

产出：

- 最小改写后的文章
- 构建结果
- 需要人工补充的数据清单

### Day 6：回写状态

更新 canonical series plan 和必要索引页。

产出：

- 系列状态更新
- 文内 relref 检查

### Day 7：复盘 Harness

复盘不是复盘文章，而是复盘 Harness。

产出：

- 本文件的修订
- 新增检查项
- 重复问题记录

## 9. 最短执行口令

当作者给出一个新选题时，AI 默认按下面顺序推进：

```text
先读项目规范和系列 plan
-> 输出编辑工作单
-> 等作者确认边界
-> 做技术深挖和验证设计
-> 生成初稿骨架
-> 等作者确认主线
-> 扩写正文
-> 审查、最小改写、hugo 构建、回写状态
```

最短结论：

`内容生产 Harness 的第一版，不是让 AI 自动写完所有文章，而是让 AI 每一步都知道该读什么、该产出什么、该停在哪里、该怎么验证。`
