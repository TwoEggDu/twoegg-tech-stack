# 技术底座优先的写作计划

## 目标

这一阶段的目标不是继续扩专栏边界，而是先把公开站点的技术底座补到“闭环感成立”。

这里的“技术底座”不是指 [doc-plan.md](E:/NHT/workspace/TechStackShow/doc-plan.md) 全部 `600+` 规划项全部写完，而是指当前公开站点已经承诺的四条主线先彻底站稳：

- 项目交付
- 引擎链路
- 渲染性能
- 工程判断

在这个前提下，允许穿插少量管理价值观文章，但不能让它们打断技术闭环节奏。

## 当前判断

按现有公开内容看，下面三条已经很强：

- `系统理解`
  `游戏图形系统`、`游戏引擎架构地图`、`数据导向运行时`、`DOTS`、`Mass`、`Unreal` 这些主干已经足够支撑深度。
- `运行时 / 资源 / 工程链路`
  `Unity 资产系统与序列化`、`Shader Variant`、`Unity 裁剪`、`脚本编译管线`、`HybridCLR`、`CrashAnalysis` 已经形成明显辨识度。
- `工程质量`
  `code-quality` 已经从代码质量一路接到工程质量、CI、质量门禁、灰度、回滚和线上观察。

真正还没闭上的，是下面三处：

1. `Projects 太薄`
2. `Problem Solving 现场分布不均`
3. `性能链还缺收口文`

所以这一阶段真正要补的，不是新技术大陆，而是最后一圈高价值补洞文。

## 这阶段不优先扩的方向

以下方向可以保留在总规划里，但当前阶段不优先开写：

- 美术生产管理全体系
- 项目管理、制作人方法、团队组织
- 商业化、发行、买量、商店运营
- 游戏策划方法论全体系
- 关卡设计与内容生产全体系
- 更大的后端 / 服务端体系扩写
- 再继续扩很多基础百科型技术系列

原因不是它们不重要，而是现在先把技术底座最后一圈补实，性价比更高。

## 当前阶段的主任务

这一阶段只做两件事：

1. `补技术闭环`
2. `少量穿插管理价值观文章`

管理价值观文章只能作为侧线，不能抢主线。

## 技术闭环最小补洞包

如果只按“让公开站点形成技术闭环感”来算，最小需要补下面 6 篇：

1. `真机问题排查流程：从闪退到黑屏到画面异常的系统性方法`
2. `性能诊断工具选择指南：什么问题该用哪种工具`
3. `一篇新的崩溃 / IL2CPP / 运行时问题拆解`
4. `一篇新的性能 / 真机 / GPU-CPU 问题拆解`
5. `一篇偏性能 / 渲染治理的项目案例`
6. `一篇偏运行时 / 崩溃 / 资源交付的项目案例`

如果希望收口更稳，再追加 3 篇：

7. `一次资源 / 构建 / 热更事故的完整复盘`
8. `我怎样定义一条能发布、能回滚、能追责的研发生产链`
9. `接手一个复杂 Unity 项目的前两周：我怎么诊断、分层、收口`

也就是说，这一阶段的技术闭环任务，合理规模是 `6 到 9 篇`。

## 管理价值观文章的写法原则

这类文章可以写，但必须满足三条：

1. 从真实工程现场长出来。
2. 能挂回已有技术证据。
3. 帮读者理解你怎样做判断、怎样带节奏、怎样看组织。

不建议写成空泛管理感悟。更适合写成：

- 为什么我越来越重视 `可回滚、可验证、可追责`
- 为什么流程不是为了管人，而是为了降低协作成本
- 团队从 `10 人到 50 人` 时，哪些事情必须从默契变成机制

## 这一阶段的写作比例

建议采用下面的节奏：

- `3 篇技术文`
- `1 篇管理价值观文`

这样既不会打断技术闭环，也能照顾那批更喜欢看判断和管理表达的读者。

## 推荐写作顺序

### 第一段：先把性能与现场诊断收口

1. `真机问题排查流程：从闪退到黑屏到画面异常的系统性方法`
2. `性能诊断工具选择指南：什么问题该用哪种工具`
3. `管理价值观：技术负责人真正管理的不是人，而是不确定性`

这一段的目标是把“性能 / 真机 / 工具链”从很多专题，收成稳定工作流。

### 第二段：补两类关键现场

4. `一篇新的崩溃 / IL2CPP / 运行时问题拆解`
5. `一篇新的性能 / 真机 / GPU-CPU 问题拆解`
6. `管理价值观：流程不是为了管人，而是为了降低协作成本`

这一段的目标是让 `problem-solving` 不再只偏资源 / 工具链 / AssetBundle / 引用链，而是把崩溃现场和性能现场都补上。

### 第三段：补项目证据

7. `一篇偏性能 / 渲染治理的项目案例`
8. `一篇偏运行时 / 崩溃 / 资源交付的项目案例`
9. `管理价值观：团队变大后，哪些事情必须从默契变成机制`

这一段的目标是把 `projects` 从两篇扩到足够支撑多技术轴交付能力。

### 第四段：做技术收口

10. `一次资源 / 构建 / 热更事故的完整复盘`
11. `我怎样定义一条能发布、能回滚、能追责的研发生产链`
12. `接手一个复杂 Unity 项目的前两周：我怎么诊断、分层、收口`

这一段完成后，公开站点会从“技术深文很多”变成“有完整方法、证据链和工作流的技术作品集”。

## 这一阶段结束后的判断标准

如果下面这些条件都满足，就可以认为“技术底座这一阶段已经闭环”，然后再进入更上层的体系：

- `projects` 至少新增 2 篇高质量技术案例
- `problem-solving` 至少补上 1 篇崩溃现场和 1 篇性能现场
- `性能链` 至少补上 2 篇收口文
- `工程治理` 至少补上 1 到 2 篇收口型技术文
- 已经穿插完成 2 到 3 篇能代表你管理判断的价值观文章

## 这阶段之后再开的方向

技术底座这一阶段闭环后，再按下面顺序扩展更稳：

1. `项目管理、制作人方法、团队组织`
2. `美术生产管理全体系`
3. `游戏策划方法论全体系`
4. `关卡设计与内容生产全体系`
5. `商业化、发行、买量、商店运营`

原因是：

- `1` 和 `2` 与当前工作、当前证据、当前站点气质最接近；
- `3`、`4`、`5` 更适合作为“系统理解过”的上层体系，在技术底座彻底站稳后再写会更自然。

## 一句话总结

当前阶段最该做的不是继续扩边界，而是：

> 先用 `6 到 9 篇` 技术补洞文把技术底座补到闭环感成立，再用 `2 到 3 篇` 管理价值观文章把你的判断方式显性化。

等这一步完成，再进入更高层的生产、组织、策划和商业化体系。
