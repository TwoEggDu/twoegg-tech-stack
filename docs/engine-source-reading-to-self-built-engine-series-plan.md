# 从引擎源码阅读到自研引擎：专题计划

## 定位

这组专题不写成 `Unity 源码观光`，也不写成 `Unreal 功能导览`，更不写成“从零手搓一个玩具引擎”的实现连载。

它真正要解决的问题是：

`先让读者建立现代游戏引擎的稳定直觉，再把这套直觉落到 Unity 和 Unreal 的实现上，最后收回成“如果自己规划一台引擎，第一版该怎样裁剪”。`

这组专题最适合放在几条现有主线之间：

- 比 `游戏引擎架构地图` 更靠近源码阅读
- 比 `Unreal Engine 架构与系统` 更偏“问题导向的总入口”
- 比 `Unity 资产系统与序列化`、`Unity 脚本编译管线` 更上层，不只讲单条链
- 比 `数据导向运行时` 更总览，不只讲 ECS / DOTS / Mass

它承担的角色是：

`给“想从会用引擎，走向能读懂引擎源码，最后能规划自己引擎”的读者，一条连续主线。`

## 为什么要这样拆

如果直接按 `Unity 一套`、`Unreal 一套` 平铺，文章很容易滑成：

- 文件夹导航
- API / 类名导览
- “这个引擎有这个模块”的功能清单

如果直接从 `自研引擎` 开始写，又很容易滑成：

- 抄别人模块表
- 玩具框架实现
- 第一版就想做完整编辑器、完整渲染图、完整资源系统

更稳的拆法是三段式：

1. `先建立直觉`
2. `再看 Unity / Unreal 的实现`
3. `最后收回到自己的实现`

这样做的好处是：

- 读者先知道自己到底在看什么问题，不会被术语牵着走
- Unity 和 Unreal 可以放在同一问题下对照，而不是各说各话
- 最后一段“自研”不是抄作业，而是从共性约束里做取舍

## 目标读者

- 已经有 Unity 客户端经验，想进入引擎开发方向的人
- 已经能读业务代码，但读大型引擎源码还缺地图的人
- 想建立 Unity / Unreal 共性认知，而不是只会一个引擎用法的人
- 对“以后自己规划一台引擎”有兴趣，但不想先走玩具引擎路线的人

## 读者前置

这组专题默认读者不是从零开始，但也不能假设读者已经具备完整的引擎背景。

### 必须先补的前置

- C++ 阅读能力：指针、引用、对象生命周期、虚函数、模板、RAII
- 大型工程阅读能力：目录、模块、头源分离、全局搜索、调用链跟踪
- C# / IL / AOT 基础：托管代码、程序集、IL、JIT/AOT、代码生成
- 序列化基础：对象图、引用恢复、版本兼容、元数据
- 线程基础：线程、任务、锁、原子、可见性、同步点
- 图形基础：图形 API、命令缓冲、资源状态、渲染阶段
- 构建与调试基础：Debug/Release、符号、断点、调用栈、构建链分层

### 仓库里可直接复用的前置入口

- `doc-plan.md` 里的 `C++-01 ~ C++-07`
- `doc-plan.md` 里的 `零C-01`、`DS-21 ~ DS-23`
- `content/engine-notes/build-debug-prereqs-series-index.md`
- `content/engine-notes/graphics-api-series-index.md`
- `content/engine-notes/game-engine-architecture-series-index.md`

## 证据与材料边界

这组专题默认分三层证据。

### 1. 架构层证据

来自现有的架构地图、系列索引和稳定的引擎结构文档。

这一层负责回答：

- 问题怎么分层
- 哪些模块属于同一条主链
- 哪些内容不该混写

### 2. 引擎实现层证据

- `Unity`：允许按“完整引擎源码 + 包源码 + 公开文档”来写
- `Unreal`：允许按“引擎源码 + 官方文档 + API”来写

这一层负责回答：

- 同一类问题在 Unity 和 Unreal 里分别落到哪
- 哪些结论是源码直接可见的
- 哪些差异来自世界模型、类型系统、构建链和运行时选择

### 3. 自研层证据

这一层不追求“完整开源引擎复刻”，只讨论第一版真正值得做的最小闭环：

- 世界模型怎么选
- 资源系统做到哪里
- 执行模型做到哪里
- 渲染抽象做到哪里
- 工具链最小集是什么

## 统一写法约束

- 每篇只回答一个主问题，不把世界模型、渲染、构建、工具链塞进同一篇
- 先给问题地图，再进源码细节
- 先讲“为什么会长成这样”，再讲“类和函数在哪”
- 明确区分“源码事实”和“工程判断”
- Unity 和 Unreal 不做生硬的一一映射
- 每篇结尾都要收回到“如果自己做，应该学到什么”

固定骨架建议如下：

1. 这篇要回答什么
2. 先给问题地图
3. 这个问题在 Unity / Unreal 里分别落在哪
4. 关键主链或关键数据结构是什么
5. 从源码里直接能看见什么
6. 为什么会这样设计
7. 如果自己规划，第一版该怎样裁剪
8. 常见误解
9. 最后压成一句话

## 总主线

这组专题不按“引擎 A 篇、引擎 B 篇、自研篇”三条完全平行的线来写。

更稳的总主线是：

`先建地图 -> 建立对象与类型直觉 -> 建立资源与序列化直觉 -> 建立执行与线程直觉 -> 建立渲染与平台抽象直觉 -> 看 Unity / Unreal 如何各自实现 -> 最后收回自研最小系统`

这个主线的好处是：

- 先解决“我在看什么”
- 再解决“这台引擎为什么这样实现”
- 最后解决“如果轮到我自己做，该怎样裁剪”

## 专题结构

建议做成 `1 篇总入口 + 3 个阶段 + 若干补篇`。

---

## 阶段 0：总入口

### 00｜读 Unity / Unreal 源码前，先建立哪几张地图

核心问题：
如果还没进入源码，脑中最先要立住的几张地图是什么。

职责：

- 给整组专题做总入口
- 先把内容生产、世界模型、运行时底座、资源构建、平台抽象放回同一张图
- 告诉读者后面该按什么顺序进入

这一篇必须明确：

- 源码阅读不是从“找入口文件”开始
- 而是从“我在追哪条问题链”开始

---

## 阶段 1：先建立直觉

### 01｜现代游戏引擎到底在解什么题：不要先背模块表

核心问题：
现代游戏引擎如果不按功能列表记忆，最稳定的理解方式是什么。

本篇负责：

- 建立引擎的总问题空间
- 说明为什么“渲染 / 物理 / 音频 / UI”这类平铺分类不够
- 把“内容生产、运行时、构建交付、平台抽象”放到一张图里

### 02｜世界模型直觉：对象图、组件树、数据导向孤岛分别在解决什么

核心问题：
一台引擎到底如何组织“世界里有哪些对象，它们怎样被更新和管理”。

本篇负责：

- 建立 `Scene / Object Graph / Component` 的直觉
- 解释为什么现代引擎还会长出 `DOTS / Mass` 这类数据导向特区
- 给后面的 Unity / Unreal 世界模型对照打底

### 03｜类型系统直觉：反射、代码生成、绑定、序列化为什么总是一起出现

核心问题：
为什么引擎里的代码生成、类型元数据、反射、绑定和序列化几乎总是连在一起。

本篇负责：

- 建立“引擎不是只靠语言原生类型系统运行”的直觉
- 说明 editor、runtime、serialization、network、inspector 为什么都需要这套东西
- 给 Unity 的绑定/代码生成与 Unreal 的 UHT/UProperty 打共同底

### 04｜执行模型直觉：帧循环、主线程、渲染线程、任务系统到底怎样分工

核心问题：
为什么引擎不会只有一个“主线程跑到底”的简单执行模型。

本篇负责：

- 建立 PlayerLoop / Tick / Phase / Task / Job 的关系
- 解释为什么跨线程访问总是高风险
- 给后面的 Unity Job / Unreal TaskGraph / RenderThread 打底

### 05｜资源与序列化直觉：为什么文件进引擎后就不只是文件

核心问题：
为什么资源导入、引用身份、序列化和对象恢复必须连成一条链。

本篇负责：

- 建立 `源文件 -> 中间产物 -> 引用身份 -> 运行时对象` 的直觉
- 说明为什么 Prefab / Scene / Asset / Package 不能混看
- 给 Unity 资产系统和 Unreal Package / Asset 链做前置

### 06｜渲染与平台抽象直觉：为什么上层逻辑和 GPU 提交之间总隔着好几层

核心问题：
为什么渲染代码不会直接等于图形 API 调用。

本篇负责：

- 建立场景数据、渲染抽象、命令记录、平台后端之间的层次感
- 解释 RHI / backend / render graph 这类结构为什么会出现
- 给 Unity 渲染后端和 Unreal RHI / RDG 做前置

### 07｜构建与发布直觉：为什么源码、脚本、资源到最后都要换一种形态

核心问题：
为什么引擎的 build、cook、package、player build 不是外围脚本，而是引擎本体的一部分。

本篇负责：

- 建立“构建会重排代码、资源和运行时边界”的直觉
- 解释代码生成、剥离、打包、平台目标为什么连成一条链
- 给 Unity BuildPipeline / IL2CPP 和 Unreal UBT / Cook / Package 做前置

---

## 阶段 2：再看 Unity 和 Unreal 的实现

### 10｜Unity 的世界模型：GameObject、Component、Native Object、Managed Binding 怎么接起来

核心问题：
Unity 的世界模型到底是怎样被引擎内部组织起来的。

### 11｜Unreal 的世界模型：UObject、UClass、CDO、Actor Framework 怎么接起来

核心问题：
Unreal 的世界模型和对象系统为什么会长成现在这样。

### 12｜Unity 的类型与代码生成：序列化元数据、脚本身份链、绑定生成、IL2CPP 各站在哪

核心问题：
Unity 引擎里的类型系统、绑定生成和运行时边界到底怎样配合。

### 13｜Unreal 的类型与代码生成：UHT、UPROPERTY、UFUNCTION、反射与序列化链

核心问题：
Unreal 的反射系统为什么不是“几个宏”，而是一整条正式主链。

### 14｜Unity 的执行模型：PlayerLoop、主线程、渲染线程、Job System、Burst 怎样协作

核心问题：
Unity 的执行模型到底在哪些层分叉，又在哪里重新汇合。

### 15｜Unreal 的执行模型：GameThread、RenderThread、RHI Thread、TaskGraph 怎样协作

核心问题：
Unreal 的多线程模型和渲染分层为什么会如此严格。

### 16｜Unity 的资源与构建主链：Importer、Asset Database、Serialized Data、BuildPipeline、AssetBundle、Player Build

核心问题：
Unity 里从源文件到运行时产物，中间真正经过了哪些层。

### 17｜Unreal 的资源与构建主链：Package、Asset Registry、Asset Manager、Cook、Package、Build

核心问题：
Unreal 里资源、内容索引、构建转换和最终发布是怎么接起来的。

### 18｜Unity 的渲染实现：Render Loop、SRP、backend 抽象和图形 API 的衔接

核心问题：
Unity 的渲染实现到底怎样从场景走到平台 API。

### 19｜Unreal 的渲染实现：FScene、SceneProxy、RHI、RDG 怎样把场景数据送到 GPU

核心问题：
Unreal 的渲染架构为什么一定要有这么多层代理和抽象。

### 20｜同一类问题，Unity 和 Unreal 为什么会给出两种不同实现

核心问题：
Unity 和 Unreal 的差异，到底是“模块不同”，还是“约束不同”。

本篇负责：

- 把前面几篇的世界模型、类型系统、执行模型、资源构建、渲染抽象重新放回同一张对照图
- 收束成“问题相同，约束不同，气质不同”

## 阶段 3：最后收回到自己的实现

### 30｜如果自己规划一台引擎，第一步不是列模块，而是先选约束

核心问题：
自研引擎为什么不能从“我也做个渲染器、资源系统、编辑器”开始。

本篇负责：

- 先定目标产品、团队规模、平台、内容类型
- 先定“不做什么”
- 先定要借多少现成系统

### 31｜第一版世界模型怎么选：Scene Graph、Component、ECS、Hybrid 到底怎么取舍

核心问题：
如果自己规划第一版运行时，世界模型应该怎样选。

本篇负责：

- 对比对象图、组件树、数据导向特区、混合方案
- 明确哪些团队适合什么方案
- 不假装第一版就该做完整 DOTS / Mass

### 32｜第一版类型与序列化系统怎么收：对象身份、引用恢复、版本演化做到哪里

核心问题：
自己做引擎时，最小可用的类型系统和序列化系统应该做到哪里。

本篇负责：

- 明确第一版必须解决的几件事：类型身份、字段元数据、对象引用、版本兼容
- 解释哪些“高级反射能力”可以先不做

### 33｜第一版资源系统怎么闭环：导入、缓存、中间产物、构建产物做到哪里

核心问题：
自己做引擎时，资源系统最小闭环应该是什么。

本篇负责：

- 明确 `source -> import -> artifact -> runtime asset` 的最小主链
- 解释为什么资源系统不该只做“运行时加载”

### 34｜第一版执行模型怎么演化：单线程主循环、分阶段调度、任务系统分别该何时引入

核心问题：
自己做引擎时，执行模型应该怎样渐进演化，而不是一开始就过度设计。

本篇负责：

- 给出从单线程到多阶段再到任务系统的演化顺序
- 明确什么时候不该急着上 job system

### 35｜第一版渲染层怎么收：先做 Render Queue，还是先做 Render Graph

核心问题：
自己做引擎时，渲染抽象第一版到底该做到多深。

本篇负责：

- 明确第一版更该追求稳定边界，而不是完整现代名词
- 解释 Render Queue、Pass、Resource Lifetime、Graph 各自该在哪一阶段引入

### 36｜第一版工具链必须有什么：日志、Profiler、资源检查、构建诊断、最小编辑能力

核心问题：
如果缺工具链，为什么引擎很快就会失控。

本篇负责：

- 明确自研第一版最小工具清单
- 把“可观测性”提到和运行时同等重要的位置

### 37｜什么时候它算一台引擎，什么时候它还只是项目运行时

核心问题：
怎么判断自己做的东西已经进入“引擎”阶段，而不是一堆项目私有脚本。

本篇负责：

- 给出边界判断：复用性、工具化、资源主链、构建主链、平台抽象、可观测性
- 作为整组专题的收束
