# 专栏架构

## 一句话定位

这个专栏不是游戏开发百科，也不是零散技术笔记。

它要长期解释的是：

`一款游戏是怎样从立项、内容生产、客户端与服务端实现、质量治理，到上线运营，被稳定生产出来的。`

更具体一点，你的专栏主定位应该固定成：

`游戏研发生产线 / 游戏工程系统`

它以引擎和客户端基础架构为技术主轴，但必须持续打通：

- 内容制作管线
- 客户端 / 服务端边界
- 质量治理
- 发布与上线流程
- 线上运营与运维协同

## 这份文档的作用

这不是对外文案，而是你的内部约束文档。

以后每写一篇新文章，都先用这份文档判断三件事：

1. 这篇是不是还在专栏主线上。
2. 这篇是在补地图，还是在制造岔路。
3. 这篇能不能增强你的统一形象，而不是让人觉得方向发散。

## 统一形象

你最该避免的风险，不是内容不够多，而是内容越来越散。

以后不管写引擎、客户端、服务端、内容制作、质量管理还是协作流程，都要服务同一个统一形象：

`能从游戏研发生产线视角，理解系统边界、工程成本、质量门禁和跨团队交付的人。`

不是：

- 只会写 Unity 技巧的人
- 什么都懂一点的技术杂谈作者
- 纯管理经验输出者
- 纯后端、纯客户端或纯美术流程作者

## 专栏主线

整个专栏固定为 6 条主线。

### 1. 立项与技术选型

回答的问题：

- 游戏立项时，技术方案到底该怎么选。
- 客户端、服务端、引擎、工具链、内容生产方案各自影响什么。
- 技术选型真正锁定的长期成本是什么。

典型题目：

- 游戏立项阶段，客户端和服务端到底该怎么做技术选型
- 技术选型真正决定的不是框架喜好，而是未来两年的生产成本
- 为什么游戏研发的技术选型必须把内容生产和上线方式一起考虑

### 2. 内容制作管线

回答的问题：

- 美术资产如何从 DCC 工具进入引擎。
- 资源规范为什么必须进入工具、检查器和门禁。
- 内容生产如何影响包体、内存、渲染和上线稳定性。

典型题目：

- 一张贴图从 DCC 到屏幕，中间到底经过了哪些环节
- 为什么内容制作管线是游戏工程系统的一部分
- 美术流程为什么不能只靠规范文档，而必须工具化

### 3. 引擎与客户端基础架构

回答的问题：

- 引擎、运行时、渲染、资源系统、构建系统分别在什么位置。
- 客户端基础架构怎样把工具链、性能、热更、构建、平台适配放进同一套判断里。
- 一个看似局部的客户端问题，为什么会回到运行时和工程边界。

典型题目：

- 现代游戏引擎到底该怎么分层
- 为什么客户端基础架构必须同时懂工具链、运行时和渲染
- 资源系统、构建系统和运行时为什么不该被拆开理解

### 4. 服务端与协作边界

回答的问题：

- 客户端和服务端分别负责什么。
- 协议、共享数据层、版本兼容、配置同步为什么是共同边界。
- 为什么客户端负责人也必须理解服务端架构。

典型题目：

- 一次登录从按钮到 Logic 回包，中间到底发生了什么
- 共享数据层、协议生成和客户端 / 服务端协作边界
- 为什么真正值钱的不是会写某个服务，而是能讲清系统边界

### 5. 质量治理与发布工程

回答的问题：

- 质量怎样被定义、度量、门禁化和回滚化。
- Crash、ANR、卡顿、资源错误、联调事故如何前移。
- 发布质量为什么不是测试团队单独承担的事。

典型题目：

- 质量管理不是多测几轮，而是把风险前移成工程系统
- 性能回归为什么本质上是质量问题
- 一条游戏版本链路上，哪些地方必须设置质量门禁

### 6. 线上运营与研发协同

回答的问题：

- 策划、美术、客户端、服务端、PM、运营、运维围绕什么工件协作。
- 灰度、开关、热修、回滚和监控为什么不是上线后才出现。
- 研发交付怎样和线上运营能力连成闭环。

典型题目：

- 为什么运营和运维应该更早进入研发流程
- 灰度、回滚、开关和热修，为什么本质上是研发协作问题
- PM 在游戏研发里推动的不是排期，而是边界和节奏对齐

## 一张总地图

以后所有内容，都尽量能放进下面这张链路图里：

`立项 -> 技术选型 -> 内容生产 -> 客户端 / 服务端实现 -> 联调 -> 质量门禁 -> 构建发布 -> 灰度上线 -> 运营反馈 -> 持续迭代`

一篇文章最好明确站在这条链路上的某个位置。

如果一篇文章根本放不进这条链路，它大概率就不是这个专栏的核心内容。

## 每篇文章必须回答的 5 个问题

以后起题前，先强制写下这 5 行。

1. 这篇文章属于哪一条主线。
2. 这篇文章站在研发链路的哪个环节。
3. 这篇文章主要解释哪个边界或哪个成本。
4. 这篇文章会牵涉哪些角色。
5. 这篇文章最后要收回成什么工程判断。

如果这 5 个问题回答不出来，不要开写。

## 选题准入规则

一个题目只有同时满足下面 4 条，才值得进入主专栏。

### 1. 它不是纯知识点

不要只讲：

- 某个 API
- 某个工具怎么用
- 某个模块怎么配

必须回答：

`这个东西在游戏研发生产线里到底解决什么问题。`

### 2. 它必须落到边界或成本

每篇都要至少落到下面之一：

- 系统边界
- 协作边界
- 质量门禁
- 运行时成本
- 交付成本
- 返工成本

### 3. 它最好跨至少两层

最有价值的文章，通常不是单层文章，而是跨层文章。

例如：

- 内容制作 -> 客户端运行时
- 客户端 -> 服务端
- 引擎 -> 操作系统 / 硬件
- 研发流程 -> 发布质量

### 4. 它最后必须能收回到动作

文章最后不能停在“知道了”。

应该能收回到：

- 怎么建规则
- 怎么做门禁
- 怎么改流程
- 怎么做工具
- 怎么做质量基线
- 怎么做版本兜底

## 明确不写什么

为了防跑偏，下面这些内容默认不作为主专栏重点。

- 纯教程口吻的 Unity / Unreal 入门
- 只讲单个 API 用法的技巧文
- 脱离项目上下文的通识科普
- 没有工程边界和成本讨论的管理心得
- 纯粹为扩覆盖面而写的后端八股或操作系统八股

这些内容不是永远不能写，而是即使写，也必须重新挂回主线。

## 最稳的文章类型

你后面最好长期复用下面 5 种文章模版。

### 1. 总论地图

作用：

- 建立读者脑中的全局坐标系。

适合写：

- 一款游戏从立项到上线，到底经过哪些生产链路
- 现代游戏引擎到底该怎么分层

### 2. 单链路拆解

作用：

- 讲穿一条真实路径。

适合写：

- 一次资源加载从磁盘到屏幕发生了什么
- 一次登录从客户端到服务端发生了什么

### 3. 边界与协作

作用：

- 讲清楚角色之间如何分工，哪里最容易出错。

适合写：

- 策划、客户端、服务端为什么会在配置边界上反复出问题
- 运营、运维为什么不能只在上线后接手

### 4. 质量与门禁

作用：

- 把“问题复盘”升级成“如何前移和治理”。

适合写：

- 为什么性能回归必须进入 CI
- 资源检查为什么要在导入阶段拦住

### 5. 案例复盘

作用：

- 用真实问题证明你的工程判断。

适合写：

- 一次 AssetBundle 构建后 Shader Variant 丢失问题的定位与修复
- 特效性能检查器案例

## 防跑偏检查表

每篇文章开写前，先过一遍下面这张检查表。

- 这篇是不是服务于“游戏研发生产线 / 游戏工程系统”这个总定位。
- 这篇是不是能放回 6 条主线之一。
- 这篇是不是讲清了一个真实边界，而不只是一个技术点。
- 这篇是不是能连接至少两个角色、两层系统或两个阶段。
- 这篇最后是不是能落到工程动作，而不只是观点。
- 这篇发出去之后，会不会让别人更清楚你的统一形象，而不是觉得你方向飘了。

只要有两条答不上来，这篇就先别写。

## 发布配比

为了防止内容重心继续只堆在引擎和渲染侧，建议按阶段做配比控制。

### 第一阶段

先补齐短板，配比建议：

- 引擎与客户端基础架构：`30%`
- 内容制作管线：`20%`
- 服务端与协作边界：`15%`
- 质量治理与发布工程：`20%`
- 立项与技术选型：`5%`
- 线上运营与研发协同：`10%`

### 第二阶段

短板补齐后，再恢复为更均衡的结构：

- 引擎与客户端基础架构
- 内容制作管线
- 服务端与协作边界
- 质量治理与发布工程
- 线上运营与研发协同

尽量不要连续 5 篇都落在同一条主线。

## 题目池应该怎么管理

建议你把选题池分成 3 层。

### P0：立形象

这些文章负责定义你是谁。

例如：

- 一款游戏从立项到上线，到底经过哪些生产链路
- 为什么内容制作管线是游戏工程系统的一部分
- 质量管理不是多测几轮，而是把风险前移成工程系统
- 为什么客户端负责人也必须理解服务端架构

### P1：补地图

这些文章负责把主线补完整。

例如：

- 一次登录链路
- 一次资源加载链路
- 一次构建发布链路
- 一次灰度回滚链路

### P2：做案例

这些文章负责证明你不是只会讲概念。

例如：

- Shader Variant 丢失问题
- 特效性能检查器
- 资源格式门禁
- 补丁包差分分析

## 最小执行规则

如果你以后不想每次都重新思考专栏结构，可以只记住这 4 句。

1. 先写链路，不先写知识点。
2. 先写边界和成本，不先写 API。
3. 先写如何被稳定生产出来，不先写某个人怎么做。
4. 先服务统一形象，不先追求覆盖面。

## 每季度复盘一次

每季度回头看一次，至少检查这 4 件事：

1. 6 条主线里，哪条又断更了。
2. 有没有连续太多文章都挤在引擎 / 渲染侧。
3. 有没有文章越来越像教程、杂谈或经验感想。
4. 现在的内容，能不能让人一句话说出你的专栏到底是写什么的。

如果第四条说不清，就说明专栏又开始跑偏了。

## 最后压成一句话

以后这个专栏最稳定的内部定义应该是：

`这不是一个分散的技术专题集合，而是一套围绕游戏研发生产线展开的工程系统写作；所有文章都应该帮助读者理解，一款游戏是如何被设计、制作、实现、验证、发布和运营起来的。`
