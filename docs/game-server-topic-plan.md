# 游戏服务端专题规划（基于系列十三重组）

## 专题定位

这条线不建议再额外开一个和 `系列十三：游戏后端基础` 平行的新一级系列。

更稳的做法是：

`保留系列十三·A~G 作为知识库分层，再单独整理一个“游戏服务端专题”作为阅读主线、写作主线和入口页。`

这样做的原因很简单：

- 你现在已经有 `数据库 / 缓存 / 后端架构 / Dedicated Server / 安全 / Live Service / 服务端 ECS` 这些块
- 真正缺的不是“有没有后端文章名目”，而是“读者能不能沿着一条服务端主线读进去”
- 如果再新开一套平行系列，很容易和 `十三·A~G` 重复，后面维护成本会越来越高

这条专题真正要解决的问题是：

`一局在线游戏在服务端到底经过哪些层，状态怎样被承载，仿真怎样推进，结果怎样同步、持久化、扩缩容和治理。`

一句话说：

`它不是泛互联网后端教程，而是游戏运行时在服务器侧的完整地图。`

---

## 为什么值得单独做成专题

虽然 `系列十三` 已经有完整框架，但从读者视角看，当前还存在三个明显问题：

### 1. 已有内容更像“后端模块库”，还不像“服务端主线”

`十三·A~F` 按基础设施切得很清楚，但对客户端/引擎读者来说，还缺一条从：

`权威服务器 -> 房间 / World -> Tick -> 状态同步 -> 持久化 -> 扩缩容 -> 线上治理`

一路往下的阅读链。

### 2. `SV-ECS` 已经有深度，但偏“进阶补充”

当前已落地的 `SV-ECS` 正文已经有：

- `SV-ECS-01`
- `SV-ECS-02`
- `SV-ECS-04`
- `SV-ECS-05`
- `SV-ECS-06`
- `SV-ECS-08`

这些文章已经很有价值，但它们默认读者已经接受“服务端仿真内核值得单独讨论”这个前提。对于第一次进服务端线的读者，还缺更上层的入口和桥梁。

### 3. 站点整体受众决定了这条线必须“游戏化”，不能写成通用后端课

你的核心读者仍然是：

- 有 Unity 经验的客户端程序
- 想往引擎 / 网络 / 服务端延伸的人
- 需要做技术选型或系统判断的人

所以这条线最该强调的不是：

- Spring Boot 怎么配
- 云原生术语大全
- 通用微服务八股

而是：

- 权威服务器为什么会改变游戏逻辑分层
- 房间 / World / Dedicated Server 是什么关系
- Tick、异步 I/O、状态广播、持久化的边界怎么切
- 游戏服务端和普通互联网后端到底哪里不同

## 目标读者

- 已经做过客户端开发，但脑子里还没有“游戏服务端完整地图”的工程师
- 做过网络同步或联机功能，但对服务端架构只停留在局部概念的人
- 在做技术选型，需要判断 `Unity Headless / Unreal DS / 独立服务端 / ECS / Actor` 路线的技术负责人
- 做过传统后端，但刚进入游戏领域，需要快速建立“游戏特有约束”心智的人

## 专题边界

### 属于这个专题的内容

- 游戏服务端在整体游戏运行时中的位置
- 权威服务器、房间、World、Dedicated Server 的关系
- 网关、匹配、大厅、逻辑服、数据服、推送/消息系统的职责边界
- Tick、状态同步、AOI、广播、异步 I/O 接缝
- 仿真内核的组织方式：OOP / Actor / ECS 各自适合什么问题
- 持久化、断线重连、快照、回放、冷热数据边界
- 游戏服务器编排、扩缩容、负载治理、线上观测
- 服务端安全、运营配置、活动系统和监控在游戏场景下的特殊性

### 不属于这个专题的内容

- 完整的数据库教程
- 完整的 Redis 教程
- 完整的 Kubernetes / 云原生教程
- 完整的网络协议基础教程
- 完整的客户端预测 / 回滚 NetCode 系列
- 完整的反作弊或数据分析大系列

这些内容只覆盖与“游戏服务端主线”直接相交的部分，不吞掉它们各自的完整体系。

## 当前真正缺口

真正要补的不是更多零散标题，而是下面五类文章：

### 1. 入口文

需要一篇把整条服务端链一次讲清的总论，让读者知道：

`我接下来会读到哪些层，它们各自解决什么问题。`

### 2. 桥梁文

需要几篇把：

- 客户端熟悉的联机概念
- 引擎里的 Dedicated Server
- 后端里的网关 / 房间 / 数据 / 编排

翻译到同一套语言里。

### 3. 主循环文

你现在有 `异步 I/O` 和 `Tick 预算` 的局部点，但还缺一篇更总的：

`一帧服务端逻辑是怎样从收包、排队、仿真、广播、持久化一路走完的。`

### 4. 状态恢复文

断线重连、快照、回放、冷热数据边界是游戏服务端的典型高频问题，但目前在规划里还没有一个足够像“专题骨架”的入口文。

### 5. 线上治理文

你已经有 `GaaS` 和 `安全` 的系列位置，但游戏服务端专题还缺一个更靠近运行时的治理视角：

`怎么知道某个房间 / 某台逻辑服 / 某组 Tick 已经过载，怎样观测和降级。`

## 公开参照系也应该成为专题的一部分

你刚才补的这个方向很对。

这条专题如果只写“应该怎么设计”，会有一个明显短板：

`读者不知道这些判断到底只是理论推演，还是行业里真的有人这样做。`

所以这条线建议显式加入一层：

`典型做法 + 开源/公开引擎 + 国际大厂公开案例`

但这里要注意一个边界：

这部分不是要写成“公司八卦”或“引擎史料堆积”，而是要回答：

`公开方案里，哪些做法反复出现；它们分别解决了什么问题；为什么不同团队会落在不同解。`

### 1. 典型做法层，比“品牌名堆叠”更重要

我建议先把公开案例压成下面几类稳定做法：

#### 做法 A：引擎内 Dedicated Server / Headless 路线

适合：

- 游戏逻辑和客户端强共享
- 想沿用现有引擎运行时
- 先把“能跑服务端”做出来，再谈外围拆分

典型参照：

- `Unity Dedicated Server`
- `Unreal Dedicated Server`
- `Godot headless / dedicated server export`

核心问题：

- 怎么裁掉渲染、音频和客户端专属逻辑
- 怎么处理客户端 / 服务端代码共享边界
- 怎么从 listen server 走向真正 dedicated server

#### 做法 B：引擎多人框架 / 复制框架路线

适合：

- 引擎已经内置网络复制模型
- 重点是实体复制、RPC、输入、预测和回滚

典型参照：

- `O3DE Multiplayer Gem`
- `Unreal Replication / NetMode / MultiServer 扩展`

核心问题：

- 哪些状态走属性复制，哪些走 RPC
- 预测和回滚站在哪一层
- 单服内复制和多服通信边界怎么切

#### 做法 C：独立仿真内核路线

适合：

- 不想把服务端绑定在完整引擎运行时里
- 更重吞吐、调度控制和部署弹性

典型参照：

- `Flecs`
- `EnTT`
- `Bevy ECS`
- `独立 C# ECS`

核心问题：

- Tick 主循环怎样组织
- 异步 I/O 怎样接进同步仿真
- 多 World、状态广播、持久化怎么做

#### 做法 D：权威服务器 + 快照复制路线

适合：

- 实时对战
- 服务端权威强
- 客户端只做预测、插值和表现

典型参照：

- `Valve Source`
- 以及很多 FPS / 对战类公开架构

核心问题：

- 输入采样、服务器 Tick、快照、Delta 压缩
- 插值、预测、Lag Compensation
- 兴趣管理和网络带宽预算

#### 做法 E：比赛服 / 房间服执行体 + 平台外围路线

适合：

- 一局一进程 / 一局一执行体的匹配制游戏
- 需要在平台层做匹配、调度、鉴权、观战、战绩、回放

典型参照：

- `VALORANT`
- 很多现代竞技游戏的“game server executable + platform”分层

核心问题：

- 一局游戏的服务端执行体和平台微服务怎样分工
- 如何扩容到海量同时对局
- 运行时权威逻辑和平台服务怎样解耦

### 2. 开源/公开引擎参照，最适合承担“典型做法的证据”

这部分不一定每个都单开大文，但至少应该在专题中反复作为对照材料出现。

#### 开源/公开引擎与框架建议优先看这几类

- `Godot`：重点看 dedicated server export 和 `--headless` 运行方式，代表“轻量引擎内服务端”
- `O3DE`：重点看 Multiplayer Gem、AzNetworking、ServerLauncher，代表“引擎内建多人框架”
- `Bevy`：重点看 MinimalPlugins / ScheduleRunnerPlugin 这类 headless 运行方式，代表“最小运行时内核”
- `Flecs`：重点看 REST API / Explorer / pipeline，代表“ECS 仿真内核 + 调试工具链”

### 3. 国际大厂公开案例，最适合承担“为什么这样分层”

这部分最值钱的不是“谁用了什么技术名词”，而是他们公开暴露出来的结构选择。

#### 我建议优先拿这几类大厂公开材料做参照

- `Valve Source`
  代表做法：权威服务器、固定 Tick、快照广播、Delta 压缩、客户端插值/预测
- `Riot VALORANT`
  代表做法：一局一个 game server executable，外围再挂平台微服务
- `Riot League of Legends`
  代表做法：长期运营下的确定性、统一时钟、输入与时间系统治理
- `Unreal / Unity 官方文档`
  代表做法：商业引擎怎样把 dedicated server/headless 变成正式产品能力

### 4. 这层内容最适合的写法

这部分建议遵守三个写法原则：

- 不写成“某公司技术大全”，只抽服务端主线相关结构
- 不追求罗列更多名字，而是归纳“这其实是哪一种稳定做法”
- 每次引用案例，都要落回“它解决了什么问题，不适合什么项目”

## 专题内部结构

建议把这个专题整理成 6 个模块。

### 模块一：入口与问题空间

职责：

`让读者先建立游戏服务端和普通后端不一样的地方，以及整条服务端主线的大地图。`

建议承担的问题：

- 为什么游戏服务端不是普通互联网后端的简单变体
- 权威服务器到底改变了什么
- 一局游戏在服务端经过哪些层

### 模块二：服务形态与职责划分

职责：

`把网关、匹配、大厅、房间、逻辑服、数据服、Dedicated Server 放进同一张图。`

建议承担的问题：

- 哪些服务是接入层，哪些是仿真层，哪些是数据层
- Dedicated Server 和独立后端服务是什么关系
- 什么时候一个房间就是一个进程，什么时候一个进程跑多个房间

### 模块三：仿真内核

职责：

`讲清服务端真正的“游戏运行时”是怎样推进的。`

建议承担的问题：

- Tick 怎样组织
- OOP / Actor / ECS 的选择依据
- Multi-World、异步 I/O 接缝、状态广播、Tick Budget

这一层直接复用并继续扩写 `SV-ECS`。

### 模块四：状态与数据边界

职责：

`讲清哪些状态该留在 World，哪些该广播，哪些该持久化，哪些只该缓存。`

建议承担的问题：

- AOI、Delta、Snapshot 的角色
- 断线重连的数据最小集是什么
- World 快照和数据库行式存储怎样映射

### 模块五：扩缩容与线上运行

职责：

`讲清服务端不是“跑起来就完了”，而是要能扩、能看、能回收、能降级。`

建议承担的问题：

- 服务器池、房间调度、编排
- 过载检测、排队深度、Tick 超时、负载迁移
- 日志、指标、告警和回滚

### 模块六：安全、运营与选型收束

职责：

`把服务端作为长期运营系统，而不是一次性仿真程序来看。`

建议承担的问题：

- 鉴权、限流、重放攻击、防刷
- 活动系统、远程配置、在线指标
- 不同团队和项目规模下的技术选型地图

## 这条专题最适合的写法

为了避免这条线写回“泛后端知识点堆积”，建议统一坚持下面四个写法原则。

### 1. 始终从“游戏运行时链路”出发

每篇都优先回答：

`这个东西在一局游戏从开始到结束的哪一段介入。`

而不是先从技术名词定义讲起。

### 2. 每篇都要明确客户端视角和服务端视角的分界

你的读者很多来自客户端，所以要经常提醒：

- 哪些状态在客户端只是表现
- 哪些状态必须由服务端权威维护
- 哪些逻辑可以共享，哪些边界不能共享

### 3. 每篇都尽量有“什么情况下不要这么做”

服务端专题如果只讲“该怎么做”，很容易变成玄学最佳实践。

更有价值的是明确：

- 什么项目规模不值得上 ECS
- 什么团队不适合过早拆服务
- 什么情况下 Redis 只是临时补丁
- 什么情况下 DS 不如独立逻辑服

### 4. 每篇都尽量把下一篇导向写清楚

因为这条专题本质上是重组现有系列，不是另起一套完全独立体系，所以文章结尾应该显式导向：

- 继续读本专题哪篇
- 或者回看 `系列十三` / `一·D 网络` / `十八系列 ECS` 的哪篇
