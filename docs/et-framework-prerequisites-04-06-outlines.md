# ET-Pre-04｜详细提纲：Unity Package、asmdef、代码装配

## 文章主问题与边界

- 这篇只回答：`为什么 ET9 要改成 Package 化结构，以及 Unity Package、asmdef、代码装配边界为什么会直接影响框架形态。`

- 这篇不展开：`所有 Unity 包管理实践细节`
- 本篇允许落下的判断强度：`先把 ET9 的工程形态解释清楚，再让读者知道为什么主仓库不再等于全部框架。`

## 一句话中心判断

- `ET9 改成 Package 模式，不是目录风格变化，而是把“模块边界、装配边界、升级边界”正式抬到框架设计层。`

## 详细结构

### 1. 这篇要回答什么

- 开篇先写读者最常见的困惑：
  - 为什么 ET9 仓库里看不到成规模核心代码
  - 为什么 README 讲了很多能力，但主仓库源码很少
  - 为什么一打开工程，反而像个“入口壳子”
- 直接点出主问题：
  - `这不是仓库不完整这么简单，而是 ET9 的框架形态发生了变化。`

### 2. 为什么 ET9 会让人产生“代码不见了”的错觉

- 先给一个总判断：
  - 旧直觉默认“Git 仓库 = 框架主体”
  - ET9 则变成“主仓库 + 包中心 + 本地装配”的组合
- 这里要拆出三层对象：
  - 主仓库：初始化工程与文档入口
  - 包中心：真正的模块分发仓储
  - 本地工程：把包装进来后的运行现场
- 本节结论：
  - `ET9 的真正阅读入口已经从单仓库转成了包化装配体系。`

### 3. 先把三个关键对象放回原位

#### 3.1 Unity Package 在解决什么

- Package 不是“把文件放另一个目录”
- 它定义的是模块分发和依赖边界
- 一旦框架采用 Package 交付，就意味着升级、安装、依赖链都被正式化了

#### 3.2 asmdef 在解决什么

- asmdef 不是单纯加快编译
- 更重要的是定义程序集边界、引用关系、编译分层
- 对大框架来说，它决定“哪些代码属于同一装配层”

#### 3.3 代码装配边界在解决什么

- 模块化更新
- 不同层级代码的职责分离
- 工程扩展与调试边界
- 让热更、分析器、工具链有稳定落点

### 4. 一个更稳的最小工程模型

- 建议画一个最小图：
  - `ET 主仓库` 负责入口、文档、初始依赖、运行指南
  - `cn.etetet.core / loader / actorlocation / login ...` 负责真实模块能力
  - 本地工程把这些包组合成最终运行现场
- 本节必须压出的判断：
  - `ET9 不是代码消失了，而是代码被切成了可装配模块。`

### 5. 放回 ET 现场：为什么主仓库只是入口，不是全部

- 从当前公开可证事实出发：
  - 主仓库里 `com.etetet.init` 更像引导层
  - README 与 Book 持续把读者引向 `ET-Packages`
  - 公开核心能力分散在 `cn.etetet.*` 包里
- 要明确讲清的三个边界：
  - 主仓库边界
  - 公开包边界
  - 课程版/未公开部分边界
- 本节最后收成一句：
  - `只有先接受 ET9 是 Package 化框架，后面读运行时和工程化篇才不会一直用旧直觉误判。`

### 6. 一个坏理解通常怎么坏

- 坏法 1：拿 ET8 的“单仓库直读法”硬套 ET9
- 坏法 2：以为主仓库源码少就等于框架空心化
- 坏法 3：把 Package 化只理解成分发手段，看不到它其实改变了升级和协作模型
- 坏法 4：忽略 asmdef 和装配边界，把 ET9 误读成普通 Unity 工程拆目录

### 7. 工程边界与取舍

- 这种组织方式的收益：
  - 模块可装配
  - 升级路径更清晰
  - 框架扩展层更自然
  - 生态化能力更强
- 这种组织方式的代价：
  - 阅读门槛更高
  - 主仓库不再能一眼看全
  - 包中心、版本、依赖、装配会显著增加理解成本

### 8. 这篇真正想留下来的结论

- 最短结论：
  - `ET9 的 Package 化不是表层整理，而是把模块边界正式抬到框架设计层；不理解这一点，后面几乎所有 ET9 文章都会读歪。`
- 明确导向后续：
  - 正文对应 `ET-01`、`ET-02`、`ET-04`、`ET-05`、`ET-27`

# ET-Pre-05｜详细提纲：会话、请求响应、超时、心跳

## 文章主问题与边界

- 这篇只回答：`为什么游戏网络层最终总会长出 Session / Request / Response / Timeout / Heartbeat 这套稳定形态，以及这如何给 ET 的 Session 设计铺路。`
- 这篇不展开：`KCP/TCP/WebSocket 的底层协议细节`
- 这篇不展开：`ET Session 源码逐行拆解`，这部分留给正文 `ET-14` 和 `ET-15`
- 这篇不展开：`完整网络同步问题`
- 本篇允许落下的判断强度：`先把会话与请求响应模型的最小必要语义立住，再进入 ET 的网络对象化设计。`

## 一句话中心判断

- `ET 的 Session 不是“把 socket 包一层”，而是把连接状态、请求闭环、超时和生命周期收口到一个运行时对象里。`

## 行文顺序与字数预算

| 正文部分 | 目标字数 | 本段任务 |
| --- | --- | --- |
| 1. 这篇要回答什么 | 300 - 450 | 点明为什么很多人会把 Session 误解成连接句柄 |
| 2. 为什么游戏网络 API 总会长得很像 | 500 - 700 | 解释请求响应、超时、心跳为何总是成组出现 |
| 3. 先把最小会话模型讲清 | 900 - 1200 | Session、Request、Response、Timeout、Heartbeat 各自负责什么 |
| 4. 一个更稳的最小链路骨架 | 500 - 700 | 用最小请求闭环图说明“请求发出到结果回来”中到底要保存什么 |
| 5. 放回 ET 现场 | 650 - 900 | 解释为什么 ET 要把 Session 做成 Entity 风格对象，并暴露 Call/Send |
| 6. 一个坏理解通常怎么坏 | 350 - 500 | 集中拆掉几种最常见误解 |
| 7. 工程边界与取舍 | 350 - 550 | 收回到为什么框架一定会做高层网络对象，而不是让业务直面裸 socket |
| 8. 最后结论 | 200 - 300 | 给出最短判断并导正文 |

## 详细结构

### 1. 这篇要回答什么

- 开篇先列几个熟悉但容易粗糙的说法：
  - Session 就是一条连接
  - RPC 就是发消息等回包
  - 心跳就是每隔几秒发个 ping
- 然后指出：
  - 这些说法不算错，但都没抓住为什么框架最终会把它们收成一层对象
- 本节主问题：
  - `为什么游戏网络 API 无论用什么语言，最后都很容易长出 Session / Call / Timeout 这套形态。`

### 2. 为什么游戏网络 API 总会长得很像

- 先给总判断：
  - 因为业务真正关心的不是收发字节，而是“一个请求是否闭环、何时失败、谁来收口状态”
- 这一段要拆出四个现实压力：
  - 连接状态要保存
  - 请求和响应要对上号
  - 超时和取消要能回收现场
  - 心跳和断线检测要能收口生命周期
- 结论：
  - `会话对象不是语法喜好，而是状态收口需求。`

### 3. 先把最小会话模型讲清

#### 3.1 Session 是什么

- 连接相关状态的最小收口对象
- 不只是 socket，还包括：
  - 远端身份
  - 最近收发时间
  - 当前错误状态
  - 正在等待的请求回调

#### 3.2 Request / Response 是什么

- 它们不是协议名词而已
- 在运行时里，它们意味着：
  - 发出去时要记录一次待完成操作
  - 收回来时要能找到对应等待者
  - 失败时要有统一失败出口

#### 3.3 Timeout 在解决什么

- 防止请求永久悬挂
- 给业务层一个明确失败语义
- 让网络异常和业务失败可以被区分

#### 3.4 Heartbeat 在解决什么

- 它不是“证明网络永远没问题”
- 它是帮助会话层判断连接活性与异常状态收口

### 4. 一个更稳的最小链路骨架

- 建议用一个最小请求闭环图来写：
  - 创建请求
  - 分配 request id / rpc id
  - 记录等待表
  - 发送
  - 超时计时
  - 收到响应后匹配并完成
  - 连接断开时批量失败收口
- 本节要形成一个清晰判断：
  - `会话层真正维护的是“连接 + 待完成请求集合 + 生命周期状态”。`

### 5. 放回 ET 现场

- 这里先不拆源码细节，只建立直觉：
  - ET 的 Session 挂在 Entity 体系里
  - 它内部保存请求回调、最近收发时间、错误码、远端地址
  - `Call` / `Send` 不是普通帮助方法，而是会话语义的一部分
- 应该讲清：
  - 为什么 ET 不让业务直接围绕裸连接写逻辑
  - 为什么后面 `RpcInfo`、超时、取消、断开时统一失败都是自然结果

### 6. 一个坏理解通常怎么坏

- 坏法 1：把 Session 当成 socket wrapper
- 坏法 2：把超时当成“网络底层自己的事”
- 坏法 3：把心跳和业务消息混为一谈
- 坏法 4：看见 `Call` 就以为只是“同步风格写法”而看不到其运行时闭环意义

### 7. 工程边界与取舍

- 框架为什么一定要做高层会话对象：
  - 状态集中
  - 故障集中
  - 请求生命周期集中
  - 更方便把网络层接入业务层
- 代价：
  - API 会显得比裸 socket 更“重”
  - 读者容易忽略底层网络真实成本
  - 一旦不理解 request table / timeout / cancellation，就会误判实现复杂度

### 8. 这篇真正想留下来的结论

- 最短结论：
  - `Session 的本质不是连接句柄，而是一次网络会话里所有待完成请求和连接状态的统一收口点；理解这一点，后面看 ET 的 Call/Send 才不会停在 API 表面。`
- 明确导向后续：
  - 正文对应 `ET-14`、`ET-15`、`ET-19`、`ET-20`

# ET-Pre-06｜详细提纲：Actor 模型最小桥接

## 文章主问题与边界

- 这篇只回答：`为什么 Actor 不是一个“分布式黑话”，而是一种把对象、邮箱、串行处理和位置透明绑在一起的运行时模型。`
- 这篇不展开：`Erlang / Orleans / Akka 的系统化比较`
- 这篇不展开：`ET MailBoxComponent 和 MessageDispatcher 的源码细节`，这部分留给正文 `ET-16`、`ET-17`
- 这篇不展开：`ActorLocation 的具体实现`，那是正文 `ET-21 ~ ET-23` 的职责
- 本篇允许落下的判断强度：`先让读者知道 Actor 到底在工程上解决什么问题，再进入 ET 的对象化 Actor 实现。`

## 一句话中心判断

- `Actor 的本质不是“一个会收消息的类”，而是“一个有身份、有邮箱、按串行语义处理消息、并且最好能位置透明地被访问的运行时单元”。`

## 行文顺序与字数预算

| 正文部分 | 目标字数 | 本段任务 |
| --- | --- | --- |
| 1. 这篇要回答什么 | 300 - 450 | 点出为什么 Actor 总被写空、写玄 |
| 2. 为什么 Actor 经常会被说歪 | 450 - 650 | 解释“会发消息”这种说法为什么不够 |
| 3. 先把 Actor 的最小模型讲清 | 950 - 1250 | 身份、邮箱、串行处理、位置透明四件事分别负责什么 |
| 4. 一个更稳的最小消息模型 | 500 - 700 | 用最小消息投递图说明 Actor 到底比普通对象多了什么 |
| 5. 放回 ET 现场 | 650 - 900 | 解释为什么 ET 要通过 MailBoxComponent、MessageDispatcher、ActorId 去落地 Actor |
| 6. 一个坏理解通常怎么坏 | 350 - 500 | 拆掉几种常见误解 |
| 7. 工程边界与取舍 | 350 - 550 | 收回到为什么 Actor 模型有价值，也为什么它会显著提高理解门槛 |
| 8. 最后结论 | 200 - 300 | 给出最短判断并导正文 |

## 详细结构

### 1. 这篇要回答什么

- 开篇先列常见说法：
  - Actor 就是一个对象加消息队列
  - Actor 就是分布式对象
  - Actor 就是不加锁并发
- 然后指出这些说法的问题：
  - 它们各自只抓到一小部分现象
  - 但没有回答 Actor 为什么能成为架构模型
- 收成主问题：
  - `一个 Actor 相比普通对象，到底多了哪些架构层语义？`

### 2. 为什么 Actor 经常会被说歪

- 先给出总判断：
  - Actor 最容易被写空，是因为大家往往只看见“消息”而没看见“处理顺序和访问边界”
- 拆出三类常见误写：
  - 把 Actor 讲成“消息 API”
  - 把 Actor 讲成“线程模型”
  - 把 Actor 讲成“远程对象调用”
- 本节收束：
  - `Actor 真正重要的是它怎样约束对象与消息的关系。`

### 3. 先把 Actor 的最小模型讲清

#### 3.1 身份在解决什么

- Actor 必须有稳定身份
- 没有身份，就谈不上定向发消息、跟踪归属、做位置映射

#### 3.2 邮箱在解决什么

- 邮箱不是“一个队列”这么简单
- 它定义的是：外部如何把工作交给这个 Actor，而不直接闯入内部状态

#### 3.3 串行处理在解决什么

- Actor 最大价值之一是把对象内部状态修改收拢成串行语义
- 这不是说系统里永远单线程，而是说“对于某个 Actor，自身状态修改按一条受控顺序进行”

#### 3.4 位置透明在解决什么

- 最理想的 Actor 访问方式：发送方不必关心接收方具体在哪个进程或节点
- 这正是很多分布式框架愿意引入 Actor 的关键原因

#### 3.5 小结：四者缺一不可

- 没身份，无法路由
- 没邮箱，无法隔离访问
- 没串行语义，状态模型会散掉
- 没位置透明，分布式价值会大幅下降

### 4. 一个更稳的最小消息模型

- 建议画一个最小图：
  - 发送方持有 Actor 身份
  - 消息先入邮箱
  - 调度器决定何时取出
  - Actor 串行处理
  - 必要时再回复或转发
- 本节必须压出的判断：
  - `Actor 不只是“能收消息”，而是“消息成为唯一正规入口”。`

### 5. 放回 ET 现场

- 先建立最小桥接：
  - ET 里的 Actor 不是单独一类特殊对象，而是 Entity 挂上 MailBox 后获得 Actor 语义
  - MessageDispatcher 负责把消息投给正确处理链
  - ActorId/Address 让消息寻址和位置映射成为可能
- 要明确讲清：
  - ET 的 Actor 是“对象化 + 消息化 + 运行时归属化”的实现
  - 这解释了为什么后面 Mailbox、MessageDispatcher、ActorLocation 必须一起看

### 6. 一个坏理解通常怎么坏

- 坏法 1：把 Actor 当成“加了收消息方法的对象”
- 坏法 2：把 Actor 误等同于线程
- 坏法 3：把消息队列存在就当成 Actor 架构成立
- 坏法 4：忽略位置透明和身份映射，导致看不懂 ActorLocation

### 7. 工程边界与取舍

- Actor 模型的收益：
  - 更容易收口对象内部状态修改
  - 更方便做分布式对象寻址
  - 更适合作为游戏服务端的对象执行边界
- 代价：
  - 概念成本高
  - 消息边界一旦乱掉，系统会很快变成“假 Actor”
  - 不先立模型，后面看框架实现会觉得全是“绕路”

### 8. 这篇真正想留下来的结论

- 最短结论：
  - `Actor 的价值不在“消息”两个字，而在“对象身份 + 邮箱入口 + 串行语义 + 位置透明”这四件事一起成立；理解这一点，后面看 ET 的 Mailbox 和 ActorLocation 才不会空。`
- 明确导向后续：
  - 正文对应 `ET-16`、`ET-17`、`ET-18`、`ET-21`、`ET-22`
