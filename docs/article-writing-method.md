# TwoEgg 文章写法规范

> 这份文件是仓库内关于“怎么写文章”的唯一真相源。它回答的是 `怎么写`，不是 `写什么`。

## 1. 适用范围

这份规范默认适用于本仓库的大多数技术文章，尤其是：

- 原理篇
- 架构篇
- 系列主线文章
- 引擎映射篇

它不直接替代以下内容：

- `doc-plan` 这类总计划
- 某个系列的专属 plan
- 单篇 outline / evidence card

也就是说：

- `plan` 解决的是：写什么
- `method` 解决的是：怎么写

## 2. 核心目标

我们的文章默认不是：

- API 教程
- 菜单说明书
- 工具清单
- 版本差异备忘录

我们的文章默认要做的是：

1. 先把问题空间立住，让读者知道“为什么这个问题会反复出现”。
2. 再把抽象模型立住，让读者知道“这个问题真正应该怎样分层、怎样命名、怎样切边界”。
3. 最后再落到具体实现，让读者知道“在 Unity / Unreal / 自研框架里，这个抽象到底怎么落地”。

最短的一句话是：

`先写问题空间，再写抽象模型，最后落到具体实现。`

## 3. 默认结构

大多数技术文章默认按下面这条主结构写。

### 3.1 问题空间

先回答：

- 这篇到底在解决什么问题
- 这个问题为什么会在项目里反复出现
- 常见但粗糙的理解是什么
- 为什么“表面现象”不是问题本体

这一段的目标不是煽情，也不是铺垫篇幅，而是先把讨论对象定义清楚。

### 3.2 抽象模型

再回答：

- 这个问题真正的结构是什么
- 应该拆成哪几层、哪几个对象、哪几段链路
- 哪些概念最容易被混掉
- 更稳的最小模型应该长什么样

这一段通常是文章的中心。

如果没有这一层，后面的具体实现很容易退化成“讲一套 API 用法”。

### 3.3 具体实现

最后回答：

- 在 Unity 里通常怎样落
- 在 Unreal / GAS / 自研框架里对应什么
- 项目里常见的实现误区是什么
- 哪些部分是引擎能力，哪些部分是团队自己的系统设计

注意这里的“具体实现”不是简单列 API。

它应该回答的是：

`这个抽象在具体引擎里怎样站住。`

### 3.4 工程边界与取舍

如果文章主题适合，再补最后一层：

- 为什么某种做法在原型期可行，但在中后期会出问题
- 哪些能力值得做成独立系统
- 哪些能力不应该被这个系统吞掉
- 多人、工具链、回归测试、性能等会在哪些点改变设计

不是每篇都必须有这段，但很多架构篇都应该有。

## 4. 默认写作顺序

写作时建议按下面顺序组织，不要一上来先讲引擎 API。

1. 先写一句最短判断。
2. 先列项目里会遇到的真实问题。
3. 再拆抽象边界和最小模型。
4. 最后才落引擎实现。
5. 末尾用一句最短结论收口。

如果一篇文章的第一屏直接出现：

- 包名
- API 名
- Inspector 设置
- 某个引擎版本差异

那大概率写法已经偏了。

## 5. 引擎相关内容怎么写

### 5.1 默认先抽象，后落引擎

只要主题不是纯引擎专属机制，默认都先写引擎无关的问题空间和模型，再落具体实现。

例如：

- 技能系统
- 资源系统
- 动画系统
- 渲染架构
- 工具链边界

都应该先问：

`这个问题在任何项目里为什么会出现？`

而不是先问：

`Unity 里这个 API 怎么调？`

### 5.2 即使是引擎专属篇，也先说清它在解决什么通用问题

例如：

- URP Renderer Feature
- GAS
- Addressables
- HybridCLR

这类文章虽然会强依赖具体引擎或框架，但仍然应该先写：

- 它在解决哪一类通用问题
- 它站在整条系统链的哪一层
- 为什么团队容易把它和别的层混掉

### 5.3 具体引擎实现要服务于抽象，而不是替代抽象

引擎实现段的目的不是证明“我知道很多类名”，而是：

`把前面建立的抽象，落回真实工程语境。`

## 6. 文章类型与变体

不是所有文章都完全使用同一模板，但大多数都应从下面四种类型里选一种。

### 6.1 原理篇

默认结构：

- 问题空间
- 抽象模型
- 具体实现
- 工程边界

适用：

- 架构
- 系统分层
- 运行时机制

### 6.2 映射篇

默认结构：

- 通用问题
- 抽象模型
- Unity / Unreal / 自研的对应关系
- 取舍与边界

适用：

- Unity vs Unreal
- 自研方案 vs 框架方案
- 某类系统在不同引擎中的落法

### 6.3 案例篇

默认结构：

- 现象
- 为什么表面现象会误导
- 分层定位
- 根因
- 修法
- 如何预防

适用：

- 排障
- 故障诊断
- 事故复盘

案例篇不是按“问题空间 -> 抽象模型 -> 具体实现”硬套，但内部仍然应该保留同样的思想：先拆层，再落地。

### 6.4 索引篇

默认结构：

- 这组文章在解决什么
- 推荐阅读顺序
- 按问题跳转路径
- 接下来还会写什么

适用：

- 系列索引
- 阅读地图

## 7. 一篇好文章至少要回答哪些问题

如果一篇文章属于主线技术文，通常至少要回答下面这些问题里的大部分：

- 这篇在解决什么问题
- 为什么这个问题会在项目里反复出现
- 这个问题最容易被混掉的边界是什么
- 更稳的抽象模型是什么
- 在具体引擎里，这个模型怎么落
- 项目里最常见的坏写法是什么
- 最后应该记住的一句话是什么

如果这些问题大多没回答，那文章大概率只是“知道一些实现细节”，还没形成真正的方法文。

## 8. 明确反模式

下面这些写法，默认都应该避免。

### 8.1 一上来就讲 API

错误方向：

- 从包名开始
- 从 Inspector 开关开始
- 从菜单路径开始

这会让文章退化成工具说明书。

### 8.2 只有概念，没有落回工程

如果整篇都停留在抽象名词，没有回到：

- Unity 怎么落
- Unreal / GAS 怎么对应
- 项目里为什么会踩坑

那文章会显得空。

### 8.3 只有实现，没有先立问题空间

这会让读者只记住一套用法，却不知道它到底在解决什么问题。

### 8.4 把某个引擎实现误写成问题本体

例如把：

- Addressables
- GAS
- Renderer Feature

直接写成问题本体，而不先解释它们各自在解决哪一类通用问题。

### 8.5 把跨系统边界写糊

例如：

- 技能系统吞掉 AI
- 动画系统吞掉技能生命周期
- 网络同步吞掉战斗逻辑

文章要主动把边界拆出来，而不是默认接受它们混在一起。

## 9. 写作检查表

写完一篇之前，至少自查下面这些问题：

- 第一屏有没有先把问题说清楚
- 有没有一段明确的抽象模型
- 有没有至少一处把“常见误解”拆开
- 有没有落到具体引擎或工程实现
- 有没有说明这个系统不该吞掉什么
- 结尾有没有一句最短结论

如果是案例篇，再额外检查：

- 现象和根因有没有明确分开
- 排查过程是不是按层推进

如果是索引篇，再额外检查：

- 是否能回答“先读哪篇”
- 是否能回答“遇到问题回看哪篇”

## 10. 维护规则

这份文件是写法规范的源头。

如果以后团队调整文章套路：

1. 先改这份文件。
2. 再改模板文件。
3. 最后只让 AI skill 引用这两份文档，不要在 skill 里重复抄一遍规则。

最重要的一条是：

`规范只保留高价值、可复用、跨系列稳定的写法约束。`

不要把某个具体系列的目录、某篇文章的独特结构、一次性写作偏好塞进这里。
