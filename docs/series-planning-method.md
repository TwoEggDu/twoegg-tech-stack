# TwoEgg 系列规划方法

> 这份文件回答的是：`一个想法怎样长成一个可持续维护的系列。`

## 1. 适用范围

这份方法适用于：

- 新系列立项
- 旧系列重构
- `doc-plan` 的系列级维护
- 某个大系列的专属 plan 编写

它不直接解决：

- 单篇文章怎么写
- 单篇 outline 怎么做
- 某篇文章的证据卡怎么整理

这些属于文章生产流程和单篇写法规范。

## 2. 目标

一个好的系列 plan 不只是“列标题”。

它至少要解决五件事：

1. 先说清这组文章在解决什么问题。
2. 说清它和别的系列的边界。
3. 把目录拆到合适粒度，而不是忽粗忽细。
4. 让读者和作者都知道推荐阅读顺序。
5. 能持续维护“已写 / 待写 / 待修订”状态。

最短的一句话是：

`系列 plan 的价值，不在标题数量，而在于它能不能稳定地定义边界、结构和进度。`

## 3. 系列规划的默认产物

默认分三层维护。

### 3.1 总 plan

例如：

- `doc-plan.md`

这一层只保留：

- 系列名称
- 系列定位
- 系列篇数
- 系列依赖关系
- 当前大致完成度

不要把总 plan 写成每篇文章细节说明书。

### 3.2 系列专属 plan

例如：

- `xxx-series-plan.md`

这一层才详细展开：

- 系列目标
- 读者定位
- 系列边界
- 文章目录
- 推荐写作顺序
- 当前已完成 / 待写 / 待修订

### 3.3 单篇 outline / evidence card

这一层只在具体开写某篇时补。

也就是说：

- 总 plan 管地图
- 系列 plan 管结构
- 单篇 outline 管执行

## 4. 什么时候值得单独开系列

我建议至少满足下面三条中的两条，再单独立系列：

- 这个主题不是一篇文章能讲完
- 这个主题有稳定的内部主线和边界
- 这个主题会持续扩写，而不是一次性专题

反过来说，如果一个主题只是：

- 零散补充
- 某个系列中的一小段
- 只有 2 到 3 篇文章的支线

那更适合先作为子系列或专题，不要急着立成一级系列。

## 5. 一个系列 plan 至少要写清哪些东西

每个系列 plan 建议至少包含下面这些块。

### 5.1 系列定位

先回答：

- 这组文章到底在解决什么问题
- 它的目标读者是谁
- 它和仓库里别的系列边界在哪

### 5.2 系列边界

明确写：

- 属于这个系列的内容
- 不属于这个系列的内容
- 只覆盖交叉点、不吞掉完整体系的内容

这一步非常重要。

很多系列后期变形，不是因为写得不够多，而是因为边界从一开始就没切清。

### 5.3 目录结构

目录最好先按“问题层”或“系统层”拆，而不是先按 API 或工具名拆。

优先顺序建议是：

1. 问题空间
2. 抽象骨架
3. 主执行链
4. 跨系统边界
5. 工程化落地

### 5.4 阅读依赖

要明确：

- 哪几篇是前置
- 哪几篇是主线
- 哪几篇是进阶
- 哪几篇是按需回看

### 5.5 进度状态

系列 plan 应显式维护三种状态：

- `已写`
- `待写`
- `待补充 / 待修订`

不要只写一个“总篇数”，不维护真实状态。

## 6. 目录粒度怎么控制

系列目录最容易出问题的地方，是粒度忽粗忽细。

常见坏法：

- 前三篇都很粗，后面突然细成十几个子点
- 一部分是“系统级标题”，另一部分是“具体 API 标题”
- 一部分是“问题”，另一部分是“工具”

更稳的做法是：

- 同一层目录尽量保持同一抽象层级
- 先按问题拆，再按实现补
- 把“具体引擎落地”留到文章内部，而不是目录一开始就写成 API 清单

## 7. 推荐写作顺序怎么定

系列目录顺序不一定等于写作顺序。

默认建议这样排：

1. 先写骨架性文章
2. 再写执行主链
3. 再写高复杂度交叉边界
4. 最后写工程化与对照收束

例如：

- 总论
- 边界
- 数据模型
- 生命周期
- 输入 / 校验 / 命中
- Effect / Buff
- 表现 / AI / 网络
- 工具 / 测试 / 对照

这样比“想到哪写到哪”更稳。

## 8. 系列和系列之间的关系怎么写

一个成熟仓库里，系列之间一定会交叉。

不要用“把所有相关内容都吞进来”的方式解决。

更稳的写法是明确区分：

- 上游前置系列
- 本系列主线
- 交叉系列
- 后续延伸系列

例如：

- 技能系统系列会碰到 AI 和网络
- 但 AI 不属于技能系统
- 完整网络同步也不属于技能系统
- 技能系统系列只覆盖和这些主题的交点

## 9. 维护规则

### 9.1 总 plan 只保留系列级信息

不要把每篇文章的长说明都塞回 `doc-plan.md`。

### 9.2 系列状态要及时回写

如果某个系列的专属 plan 已存在，那么文章写完后应优先更新系列 plan，而不是只改总 plan。

### 9.3 已经过期的状态必须清理

如果 plan 里出现：

- 实际已写，但标记仍是待写
- 文章标题和真实内容已经不再对应
- 系列边界已经变化，但 plan 还停留在旧结构

就要优先清理。

过期状态会直接削弱整个计划系统的可信度。

## 10. 系列规划检查表

建一个新系列或重构旧系列之前，至少自查下面这些问题：

- 这组文章到底在解决什么问题
- 它和别的系列边界有没有写清楚
- 目录是不是按问题层拆，而不是按 API 平铺
- 目录粒度是否一致
- 推荐阅读顺序是否明确
- 当前状态是否可维护
- 是否值得单独开系列，而不是留在父系列里

## 11. 最短结论

`一个好的系列 plan 不是标题清单，而是一张可维护的结构地图：先定义问题，再切清边界，再拆目录，再维护状态。`
