# ET 框架源码解析系列规划

> 这组专题不写成 `ET 功能清单导览`，也不写成 `按 README 复述的上手教程`。
> 它真正要解决的问题是：
>
> `先让读者建立 ET9 的运行时地图，再把这张地图落到公开源码和公开包上，最后收回成“这套框架到底适合什么项目、该怎样继续深入”。`

---

## 系列定位

ET 不是普通意义上的 Unity 客户端框架。

从公开仓库和公开包来看，ET9 更接近一套：

- `Unity 客户端 + C# 服务端` 的统一技术栈
- 以 `Entity / Fiber / Actor / Location / Package` 为核心抽象的运行时体系
- 面向 MMO / 强联网 / 分布式服务端的框架化底座

所以这组专题的目标不是“教你把 ET 跑起来”，而是讲清楚：

1. ET9 的真实形态是什么
2. 它的核心抽象是怎样协作的
3. 为什么它会长成这种架构
4. 这套架构适合什么问题，不适合什么问题

---

## 为什么要单开一组专题

如果直接按 ET 的包目录平铺，文章很容易滑成：

- 包名说明书
- 功能点罗列
- “这个框架支持什么”的 README 翻译

如果直接按 MMO 业务功能写，又会很快把框架主线打散：

- 登录讲一点
- 移动讲一点
- 帧同步讲一点
- AI 再讲一点

最后读者知道 ET 很大，但不知道 ET 的脊梁骨在哪里。

更稳的拆法应该是：

1. `先讲框架形态`
2. `再讲运行时骨架`
3. `再讲网络与 Actor`
4. `再讲分布式机制`
5. `最后讲工程化与 Demo 承载`

## 前置知识补充建议

如果目标是“尽可能细致地写 ET”，那么前置知识**需要补**，而且最好显式分成两层：

### A. 必须补的前置层

这部分不补，后面很多 ET 文章会不断回头解释基础概念，导致主线被打断。

| 前置层 | 为什么必须补 |
|------|-------------|
| C# 并发语义 | 不先拆清 `线程 / 任务 / async / continuation / SynchronizationContext`，就很难真正讲透 `ETTask` 和 `Fiber` |
| C# 程序集与代码装配 | 不理解 `程序集 / DLL / 反射 / 动态加载 / 代码生成`，就很难讲 `CodeLoader`、热更、SourceGenerator |
| 网络与 RPC 基础 | 不先讲 `请求响应 / 超时 / 心跳 / 会话 / 可靠性`，后面 `Session`、`Call`、`MessageDispatcher` 会变成 API 说明书 |
| Actor 与消息模型 | 不把 `Actor` 当成“理论名词”，而要先建立“串行处理、邮箱、消息边界、位置透明”的直觉 |
| 序列化与对象图 | 不理解 `对象树 / 身份 / 引用恢复 / 跟随父节点序列化`，很难讲 ET 的 `Entity` 模型 |
| 游戏服务端进程模型 | 不补 `Realm / Gate / Scene / Watcher / StartConfig` 这一层，分布式章节会非常跳 |
| Unity 工程装配链 | 不理解 `UPM Package / asmdef / HybridCLR / YooAssets`，会误解 ET9 的工程化设计 |

### B. 建议补的桥接层

这部分不是所有读者都必须先读完，但如果要把 ET 写得足够深，最好提前准备。

| 桥接层 | 作用 |
|------|------|
| MMO 权威同步基础 | 给 `statesync`、`lockstep`、`router`、`AOI` 提前埋地基 |
| MongoDB / BSON 与游戏服数据模型 | 给 `db`、`mongodb`、`Entity` 序列化篇提供持久化视角 |
| 进程守护、服务发现、动态扩缩容 | 给 `watcher`、`router`、云原生相关内容预留落点 |
| SourceGenerator 与分析器最小原理 | 给 ET 的“约束前移”写法提供语言层解释 |

### 当前仍然缺的 ET 直连前置

仓库里虽然有很多基础规划，但从“直接服务 ET”这个角度看，仍然缺 5 组桥接前置：

1. `线程 / async / ETTask / Fiber` 之间的桥接篇
2. `程序集 / 动态加载 / 热更 DLL / CodeLoader` 之间的桥接篇
3. `Session / RPC / Message / Mailbox / Actor` 之间的桥接篇

### 最稳的补法

不建议把这些前置散落到 36 篇正文里临时解释。

更稳的方案是：

- 在 ET 正文之前，单独增加一组 `ET-Pre` 预备篇
- 数量控制在 `8~12 篇`
- 只做桥接，不重写完整通识系列
- 读者缺什么，就先补到能进入 ET 主线为止

## 证据边界与写作约束

### 证据层级

| 证据层 | 允许使用 | 主要用途 |
|------|---------|---------|
| A. 公开源码可证 | `cn.etetet.core`、`cn.etetet.loader`、`cn.etetet.actorlocation`、`cn.etetet.login` 等公开包源码 | 用来写运行时骨架、消息链路、登录链路、ActorLocation 等主干篇 |
| B. 公开目录/公开文档可证 | 主仓库 `Book`、`README`、公开包目录 | 用来建立 ET9 的整体地图、模块边界、安装运行与包化组织 |
| C. 外部生态可证 | `ET-Packages` 公开组织、公开 Readme、公开仓库 | 用来补充包生态、插件层、扩展层 |
| D. 不直接断言的范围 | 课程版、付费包、未公开源码、群内资料 | 可以注明“存在此模块/此能力”，但不把它写成源码拆解结论 |

### 固定写法约束

- 每篇只回答一个主问题，不把多个层级混写
- 先给问题地图，再进源码主链
- 明确区分 `源码事实`、`文档事实`、`工程判断`
- 默认以 `ET9` 为主，只有在比较版本演进时才回提 ET7/ET8
- 每篇结尾都必须回答：`这个机制解决了什么问题，又引入了什么复杂度`
