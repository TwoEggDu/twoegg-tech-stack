# 《为什么某些操作会慢：给游戏开发的性能判断框架》修订问题清单

这份清单只保留会明显影响下一版质量和方向的问题，避免后续讨论被低影响细节打散。

## 1. 这篇文章在整套内容里的位置，要不要明确

需要你确认：

- 它是独立长文
- 它是大文章中的核心章节
- 它是未来系列文章的总论

为什么这题重要：

这会直接影响下一版的结构密度。如果它是总论，我会保留更多判断框架，少下沉到单点机制；如果它是核心章节，我会把存储、内存和加载链写得更深。

## 2. 客户端程序、引擎程序、TA 三类读者的比重要不要区分

需要你确认：

- 三类读者平均覆盖
- 更偏客户端程序
- 更偏引擎程序
- 更偏 TA

为什么这题重要：

当前初稿是三类读者共用一套判断框架。下一版如果要更偏某一类，举例、术语密度和篇幅分配都要调整。

## 3. 手机和 PC 的比重，要不要故意失衡

需要你确认：

- 手机和 PC 保持大致对半
- 更偏手机
- 更偏 PC

为什么这题重要：

如果更偏手机，我会把热、功耗、UFS、前后台、内存回收写得更具体；如果更偏 PC，我会强化驱动、硬件分档、安装环境和兼容性治理。

## 4. Unity 和 Unreal 的映射要写到什么深度

需要你确认：

- 只保留现有这种“原理到现象”的轻映射
- 增加一节更具体的 Unity 对照
- 增加一节更具体的 Unreal 对照
- 两个引擎都各补一个更具体的小节

为什么这题重要：

现在的写法更稳，也更耐久；但如果你希望它更贴近开发现场，我可以把 `Addressables / AssetBundle / Shader Variant` 和 `Streaming / PSO / Game Thread / RHI Thread` 展开得更明确。

## 5. 你要不要在下一版里加入更强的“加载链”章节

需要你确认：

- 保持当前篇幅
- 单独加深“为什么大文件更稳、小文件更抖”
- 单独加深“读盘不等于加载完成”
- 两者都加深

为什么这题重要：

这是当前初稿里最适合继续下潜的部分，也是最能服务游戏体验优化的一块。

## 6. 内存章节要不要更靠近运行时工程

需要你确认：

- 保持当前解释级别
- 更强调 GC、对象池、生命周期尖峰
- 更强调资源常驻/淘汰和峰值管理
- 两者都加深

为什么这题重要：

这决定下一版是更偏客户端直觉，还是更偏引擎和资源系统设计。

## 7. 要不要补一节更明确的“危险操作清单”

需要你确认：

- 维持现在分散写法
- 新增一节集中列出“不该在什么时候做什么”

为什么这题重要：

如果你的目标读者是开发现场的人，这一节会非常实用，也更容易被拿来做团队共识材料。

## 8. 要不要加入更具体的伪代码和流程图

需要你确认：

- 维持现在少量伪代码
- 增加 3 到 5 段伪代码
- 增加流程图或链路图
- 两者都加

为什么这题重要：

当前初稿更偏长文解释。下一版如果你想让它更像工程文章，适当加入伪代码和图，会明显提升可读性和复用性。

## 9. 要不要在下一版里加入工具视角

需要你确认：

- 暂时不要
- 加一小节讲 Unity Profiler / Unreal Insights / RenderDoc 这类工具如何映射到本文判断框架

为什么这题重要：

这会把文章从“原理判断”进一步推进到“如何在现场验证判断”。

## 10. 语气要不要再收得更像“工程手册”

需要你确认：

- 保持当前长文解释风格
- 略微收紧，增加判断句和结论框
- 明显收紧，偏工程手册

为什么这题重要：

这直接决定下一版是更适合公开长文阅读，还是更适合团队内部培训和协作对齐。

## 11. 要不要补真实案例

需要你确认：

- 暂时不加
- 加匿名化案例
- 加典型抽象案例

为什么这题重要：

案例能显著提高说服力，但也会改变文章节奏。真实项目经验如果能适当匿名化加入，会让这篇文章更像从开发现场长出来的。

## 12. 下一版最优先加深哪三个点

这题不需要你长答，只需要给我三个优先项。

可选方向建议：

- 一帧预算和尾延迟
- 危险操作时机
- 大文件和小文件
- 读盘、解压、反序列化、上传 GPU 的完整链路
- 内存生命周期
- CPU 缓存局部性和任务系统
- GPU 上传和首次命中
- Unity 映射
- Unreal 映射
- 手机和 PC 的差异

为什么这题重要：

你一旦给出三个优先项，我就可以直接把下一版重点收窄，不用平均发力。
