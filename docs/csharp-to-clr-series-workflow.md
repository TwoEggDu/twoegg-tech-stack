# 从 C# 到 CLR：执行计划与多 Agent 工作流


> 这份文档回答的是：在 `CCLR` 系列正式开写后，一篇文章应该怎样从选题走到交付，以及主控、调研、代码、审稿这几类 agent 应该怎么并行协作。

## 一、目标

这份执行计划只解决三件事：

1. 把 `CCLR` 的单篇生产流程固定下来
2. 让多 agent 协作可重复、可交接、可审计
3. 把“正文、审稿、返工、交付”从临场发挥变成固定动作

它不替代：

- `docs/csharp-to-clr-series-plan.md` 的选题权威清单
- `docs/csharp-to-clr-codex-prompt.md` 的写作协议
- 仓库里已有的深水 runtime 系列

## 二、总原则

- 批次内并行，批次间串行
- 先定题和大纲，再写正文
- 所有稿件都必须经过两轮自审
- 以“可交付成品”为唯一出口，不接受半成品
- 当前环境默认按 `1 名主控 + 4~5 个并行 agent` 设计，不假设无限并发

## 三、角色分工

### 1. 主控 Agent

职责：

- 核对题目是否在 18 篇权威清单内
- 指派调研 / 代码 / 正文 / 审稿任务
- 收拢不同 agent 产出
- 判断是否进入下一阶段

### 2. Research Agent

职责：

- 盘点本篇相关上游 / 下游 / 对照文
- 抽出可直接引用的规范、源码、版本、文件路径
- 标出潜在冲突点、边界风险和待核实点

交付物：

- `已有内容盘点`
- `关键引用清单`
- `潜在冲突点`

### 3. Code Agent

职责：

- 写最小可运行 C# 示例
- 尽量做本地可编译验证
- 无法验证时显式写 `<!-- VERIFY: ... -->`

交付物：

- `代码片段`
- `验证结果`
- `未验证原因`

### 4. Draft Agent

职责：

- 基于协议和大纲写正文初稿
- 按入口主线颗粒控制内容边界
- 保证三类链接齐全

交付物：

- `正文初稿`
- `DATA-TODO / EXPERIENCE-TODO / VERIFY` 标记清单

### 5. Checklist Agent

职责：

- 只按硬指标检查，不负责润色
- 逐项打勾验收清单
- 标出 `✅ / ⚠️ / ❌`

交付物：

- `Checklist Report`

### 6. Reader Agent

职责：

- 代入目标读者视角审稿
- 只找卡点、冗余、失衡
- 单独识别“读着跳不跳”：术语首次出现是否有桥、概念切换是否跨层、样例到机制是否有过渡
交付物：


- `Reader Review`

## 四、工作流总览

```text
选题冻结
-> 并行调研
-> 大纲确认
-> 正文写作
-> 第一轮自审（Checklist）
-> 第二轮自审（Reader）
-> 返工
-> 最终交付
```

## 五、分阶段执行

### Stage 0：选题冻结

输入：

- 用户指定的 `CCLR-XX` 题号与标题
- 特殊约束
- 预设桥接组合（如果已有）

动作：

- 核对题目是否在 `docs/csharp-to-clr-series-plan.md` 第五节权威清单内
- 核对是否触碰本系列明确不覆盖的边界
- 确认本篇属于 `A / B / C / D` 哪一层

通过条件：

- 题目合法
- 边界清楚
- 相关上游 / 下游 / 对照文可确定

### Stage 1：并行调研

推荐并行度：

- `Research Agent x2`
- `Code Agent x1`
- 主控本地同步做目录与计划核对

Research Agent A 负责：

- 上游前置知识
- 本系列计划中的桥接组合
- 同级 CCLR 已有文章（如果存在）

Research Agent B 负责：

- 下游深水文
- 规范章节
- 源码结构与版本信息

Code Agent 负责：

- 先行准备最小代码示例
- 判断能否本地验证

通过条件：

- 关键引用可落到文件、结构、章节
- 至少有一个清晰的最小代码示例主题
- 没发现与已有文章严重冲突的事实

### Stage 2：大纲确认

主控整合调研结果，生成一份不超过 30 行的大纲，必须包含：

- 1 个核心洞察
- 3~5 个主小节标题
- 1 个最小代码示例主题
- 1 组“直觉 vs 真相”
- 3 个准备导流的下游链接
- 预估字数
- 初步风险点

通过条件：

- 用户确认大纲
- 本篇边界没有越界到深水文

### Stage 3：正文写作

Draft Agent 根据协议写正文。

此阶段要求：

- 开头 3 段内写清楚“本文不展开什么”
- 至少一段可运行代码
- 至少一组“直觉 vs 真相”
- 结尾三类链接齐全
- 不重写 B3 / G2 / G3 主体

通过条件：

- 初稿结构完整
- 所有引用都有明确落点
- 已显式标出 `DATA-TODO / VERIFY`

### Stage 4：第一轮自审（Checklist）

Checklist Agent 只看初稿与清单，逐项打勾。

重点检查：

- 开场是否模板化
- 是否写清“不展开内容”
- 代码、对比、三类链接是否齐全
- 术语和引用格式是否合规
- 是否越界写成深水文

通过条件：

- 没有 `❌`
- `⚠️` 项已可解释

### Stage 5：第二轮自审（Reader）

Reader Agent 代入目标读者视角，检查：

- 卡点
- 冗余
- 失衡
- 跳读点：前 2~3 段是否丢主线、术语是否裸奔、概念是否跨层跳转、样例到机制是否有过渡

通过条件：

- 没有致命卡点
- 核心洞察没有被埋掉
- 阅读路径清楚

### Stage 6：返工与定稿

主控统一整理两轮审稿意见，只做最小必要修改。

返工规则：

- 事实错误：直接返工
- 结构失控：重排段落
- 越界写深：删减并改为导流
- 链接缺失：补齐三类链接
- 桥接不足：补一句最小桥句，不重写整节

### Stage 7：交付

最终交付物固定为三件：

1. 正文 markdown
2. `Review Report`
3. `DATA-TODO / EXPERIENCE-TODO / VERIFY` 清单

## 六、关键清单

### 1. 选题清单

- 是否在 18 篇权威清单内
- 是否有明确边界
- 是否能找到上游 / 下游 / 对照文

### 2. 调研清单

- 是否有规范章节
- 是否有源码文件或结构名
- 是否有版本信息
- 是否有冲突点说明

### 3. 大纲清单

- 是否有核心洞察
- 是否有代码主题
- 是否有“直觉 vs 真相”
- 是否有 3 个下游链接

### 4. 正文清单

- 结构是否符合协议
- 是否存在越界展开
- 是否有最小代码
- 是否有三类链接

### 5. 自审清单

- Checklist 是否清零
- Reader 卡点与跳读点是否可接受
- 是否仍有待核实断言

### 6. 交付清单

- 正文完整
- Review Report 完整
- TODO / VERIFY 清单完整
- 必要时提示作者执行 `hugo`

## 七、里程碑

- `M1` 选题冻结
- `M2` 调研完成
- `M3` 大纲确认
- `M4` 正文初稿完成
- `M5` 第一轮自审完成
- `M6` 第二轮自审完成
- `M7` 最终交付

## 八、失败回滚规则

- 题目越界：退回 Stage 0
- 调研证据不足：退回 Stage 1
- 大纲失控：退回 Stage 2
- 正文越界或事实错误：退回 Stage 3
- 审稿发现结构问题：退回 Stage 3 或 6

## 九、Definition of Done

一篇 CCLR 文章只有在下面条件全部满足时，才算完成：

- 题目合法且边界清楚
- 正文符合协议结构
- 关键事实有出处
- 三类链接齐全
- 两轮自审完成
- 返工项已清零或显式说明
- 交付物三件套齐全

## 十、建议的并行策略

### 单篇最小并行配置

- 主控 `x1`
- Research `x2`
- Code `x1`
- Draft `x1`
- 审稿 `x2`

### 单篇推荐推进方式

1. 主控冻结题目
2. 并行开 `Research x2 + Code x1`
3. 主控汇总后出大纲
4. 用户确认后开 `Draft`
5. 初稿完成后并行开 `Checklist + Reader`
6. 主控收束返工并交付

### 批次推进建议

- `Batch C-A` 可拆成 2~3 篇一组并行推进
- `Batch C-B / C-C / C-D` 逐批推进，避免跨层乱跳
- 跨 runtime 层文章优先留到前面入口层立稳后再开
