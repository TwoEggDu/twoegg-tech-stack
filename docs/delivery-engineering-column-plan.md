## 一、专栏定位

### 名称

- 正式标题：**《Unity 多端交付工程》**
- 副标题：**从内容冻结到三端上线的完整闭环**
- 传播品牌名：**《交付的纪律》**

### 一句话定位

> 讲清楚一个 Unity 项目从内容冻结到稳定运行在 iOS、Android 和微信小游戏三端设备上的完整工程链路，以及这条链路上每一环出了问题会怎样。

### 核心问题

这套专栏解决的核心问题：**为什么很多团队"能开发"但"不能交付"，以及如何把发布能力升级为交付工程体系。**

### 与普通教程的本质区别

| 维度 | 普通发布教程 | 本专栏 |
|------|------------|--------|
| 范围 | 怎么打 iOS 包 | iOS 构建在交付链路中的位置、上下游依赖、失败模式 |
| 视角 | 客户端程序员 | 版本负责人 + QA + 工具链 + 运维多角色 |
| 深度 | 菜单路径 + 配置项 | 架构设计 + 流程设计 + 验证设计 |
| 覆盖 | 成功路径 | 成功路径 + 事故视角 + 排障路径 + 防复发 |
| 组织 | 独立知识点 | 闭环体系，章节有依赖关系 |

### 目标读者分层

| 读者层 | 核心诉求 | 专栏提供的价值 |
|--------|---------|---------------|
| 客户端程序员（3-5 年） | 理解自己的工作在链路中的位置 | 建立全链路认知 |
| 高级客户端 / 引擎程序员 | 能独立负责某一环的交付工程 | 资源管线、构建系统、热更新的工程化方法 |
| 技术负责人 / 主程 | 设计并落地整条交付管线 | 体系设计、质量门、成熟度模型、组织协作 |
| 版本 / 发布负责人 | 做发布决策、管控风险 | 灰度策略、回滚机制、发布审批、事故响应 |
| QA / 测试负责人 | 设计验证策略、建立质量闭环 | 四层验证模型、缺陷闭环、回归策略 |
| 工具链工程师 | 建设自动化交付基础设施 | CI/CD 管线设计、质量门自动化、部署自动化 |
| 技术美术 | 理解资源生产如何影响交付质量 | 资源规范、多端差异、性能预算对美术的约束 |

## 二、设计原则

### 长青原则

```
原则一：原理不绑引擎——每个主题先讲"任何项目都要解决的问题"
原则二：实践用案例——Unity / C# 服务端作为当前实践层，可替换
原则三：平台是分支——iOS / Android / 微信小游戏是平台专项，不是主干
原则四：已有文章能接入——不重写，而是在交付主线上为已有内容开入口
```

### 三层内容模型

每个主题按以下三层组织，保证原理层长青、实践层可替换、平台层可更新：

```
┌─────────────────────────────────────────┐
│  原理层（长青）                           │
│  "为什么任何项目都要解决这个问题"           │
│  引擎无关，10 年后仍然成立                 │
├─────────────────────────────────────────┤
│  实践层（当前技术栈）                      │
│  "用 Unity / C# 怎么具体落地"             │
│  绑定当前选型，未来可替换或追加              │
├─────────────────────────────────────────┤
│  平台层（平台专项）                        │
│  "iOS / Android / 微信小游戏各自的差异"     │
│  平台规则变了就更新这一层                   │
└─────────────────────────────────────────┘
```

### 写作原则

- 先讲本质，再讲做法
- 先讲共性，再讲平台差异
- 先讲系统，再讲局部
- 必须强调工程边界、职责边界、流程边界
- 必须体现质量门、验证门、发布门、回归门等闭环意识
- 必须覆盖"发现问题 → 分析问题 → 修复问题 → 防止复发"
- 少写菜单路径，多写架构设计、流程设计、工程设计、验证设计
- 不能把专栏写成碎片知识点堆砌
- 不能只从客户端程序员视角写，要有版本负责人 / QA / TA / 工具链 / 平台对接 / 运维 / 组织流程视角
- 输出必须有层级、有结构、有章节依赖关系

## 三、主问题集

以下 15 个问题是本专栏真正要回答的核心问题：

1. **发布和交付的本质区别是什么？** 为什么很多团队"会发版"但"不会交付"？
2. **为什么交付问题往往不出在发布环节？** 大多数事故的根因在内容、资源、配置和流程。
3. **同一个 Unity 项目，如何稳定地同时交付三端？** iOS / Android / 微信小游戏的结构性差异如何管理？
4. **内容、资源、配置、代码四条生产线，如何保持一致性？** 上游不一致，下游不可能稳定。
5. **如何把"人盯人"的发布流程变成有质量门的工程体系？**
6. **功能验证、性能测试、稳定性测试、兼容性测试分别该拦什么，不该拦什么？**
7. **问题发现后，如何快速判断是内容 / 资源 / 配置 / 代码 / 平台问题？**
8. **如何建立缺陷闭环，让每个问题追溯到根因并防止复发？**
9. **CI/CD 管线应该承担哪些职责？** 哪些必须自动化，哪些必须靠人？
10. **灰度发布、回滚、热修复各自的边界是什么？**
11. **线上监控该看什么指标？什么变化应该触发告警、回滚或热修？**
12. **微信小游戏与原生 App 有哪些不可忽视的结构性差异？**
13. **多团队协作时，版本管理和发布审批怎样设计才不会成为瓶颈？**
14. **如何建立"复盘 → 改进 → 沉淀"的持续改进循环？**
15. **一个交付体系从零开始，应该按什么顺序搭建？**

## 四、主线与副主线

### 主线：交付闭环

从内容冻结到产品稳定运行在三端用户设备上的完整链路。

**主线链路**：内容生产 → 工程集成 → 版本管理 → 资源打包 → 多端构建 → 平台发布 → 灰度上线 → 线上运营

### 副主线

| 副主线 | 负责什么 | 贯穿方式 |
|--------|---------|---------|
| **质量闭环** | 每个环节的质量定义、验证方法、质量门 | 每一卷都有"本环节的质量门"小节 |
| **平台分支** | iOS / Android / 微信小游戏在每个环节的差异 | 每一卷先讲共性，再开平台差异专节 |
| **缺陷闭环** | 发现 → 分类 → 根因 → 修复 → 回归 → 防复发 | V15 集中展开，其他卷有"本环节常见故障"小节 |
| **自动化演进** | 从手动到自动化到智能化 | V16 集中展开，其他卷有"本环节的自动化边界"小节 |
| **组织协作** | 角色分工、流程设计、决策机制 | V18 集中展开，其他卷有"本环节的职责边界"小节 |

### 为什么不会散

- 主线是时序线——沿着交付链路从左到右
- 五条副主线是横切面——每条独立贯穿所有卷
- 每一卷既属于主线某个环节，又被五条副主线穿过，形成"经线 + 纬线"的网格结构

## 八、交付飞轮方法论模型

### 模型名称：交付飞轮（Delivery Flywheel）

### 八个环节

```
生产 → 集成 → 构建 → 验证 → 发布 → 监控 → 闭环 → 改进
  ↑                                                    |
  └────────────────────────────────────────────────────┘
```

### 三道门

- **质量门**（构建→验证）：编译通过 + 资源合规 + 变体可控
- **验证门**（验证→发布）：功能回归 + 性能基线 + 稳定性达标
- **发布门**（发布→用户）：审批通过 + 灰度观察 + 回滚预案就绪

### 两个环

- **内环**（开发期）：生产 → 集成 → 构建 → 验证 → 修复 → 回验证
- **外环**（运营期）：发布 → 监控 → 发现 → 闭环 → 改进 → 下次生产

### 一个轴心

**持续改进**——每一轮飞轮转动都应该让下一轮更快、更稳。

## 九、统一写作模板

```markdown
# 标题

## 这篇解决什么问题
一句话说清本文要解决的交付问题。不是技术问题，是交付问题。

## 为什么这个问题重要
不解决会怎样？真实项目中的代价是什么？

## 本质是什么（原理层，长青内容）
引擎无关的原理性解释。任何技术栈都适用。
这一段是 L1 读者唯一需要读的部分。

## 常见错误做法
列出 2-3 种典型的错误做法，说明为什么错。

## 工程化做法（实践层）
用当前技术栈（Unity / C# 服务端）展示具体实现。

## 平台差异（平台层，如适用）
iOS / Android / 微信小游戏各自的差异处理。

## 如何验证
怎么确认做对了？提供可执行的验证步骤或检查清单。

## 常见事故与排障
这个环节出过什么事故？怎么定位？怎么防复发？

## 小结与检查清单
3-5 条要点 + 可复用的 Checklist。
```

### 模板使用规则

- 不是每篇都要写满所有节——没有平台差异就不写，没有事故就不编
- "本质是什么"必须能独立于技术栈阅读
- "常见错误做法"和"常见事故"必须来自真实观察
- "如何验证"必须是可执行的，不是空话

## 十、核心术语表

| 术语 | 定义 | 边界 |
|------|------|------|
| **发布** | 把构建产物上传到平台并使用户可获取的动作 | 发布是交付链路的一个环节，不是终点 |
| **交付** | 从内容冻结到产品稳定运行在用户设备上的完整过程 | 包含发布但远大于发布 |
| **构建** | 把源码 + 资源 + 配置转换成平台可执行产物的过程 | 不含发布、不含验证 |
| **首包** | 用户首次安装时下载的包体 | 不含后续热更新内容 |
| **热更新** | 不重新安装 App 的情况下更新资源或代码 | 分资源热更和脚本热更 |
| **内容冻结** | 停止接受新功能变更，只允许 Bug 修复进入 | 不等于代码冻结 |
| **质量门** | 阶段转换的准入条件，不满足则不允许进入下一阶段 | 可自动化，也可人工 |
| **验证门** | 发布前的验证准入条件 | 质量门的子集，专指发布前 |
| **发布门** | 发布决策的最终审批条件 | 包含技术 + 业务双重判断 |
| **回归** | 验证修复没有引入新问题 | 不只是重新跑一遍测试 |
| **回滚** | 回到上一个已知稳定版本 | 客户端回滚 ≠ 服务端回滚 |
| **灰度** | 向部分用户发布新版本以观察效果 | 三端灰度机制各不相同 |
| **根因分析** | 找到问题的根本原因而非表面症状 | 根因可能在上游而非出问题的环节 |
| **提审** | 向平台提交版本审核申请 | 不等于发布，审核通过后才能发布 |
| **分包** | 将内容拆分为多个包以满足平台限制或按需下载 | Android PAD / iOS ODR / 微信分包各自不同 |

## 十一、质量标准

### 一篇合格文章的标准

- 原理层可以脱离具体引擎独立阅读
- 实践层有具体代码、配置或流程
- "如何验证"是可执行的
- 术语使用与术语表一致

### 一篇顶级文章的标准

- 读完后读者能立刻用在自己的项目上
- "常见事故"来自真实案例且有根因分析
- 原理层的解释能让第一次接触的人理解"为什么"
- 能被其他文章引用为权威参考

### 必须避免的写法

- ~~"打开 Build Settings，选择 iOS，点击 Build"~~ → 菜单路径不是工程知识
- ~~"建议进行充分的测试"~~ → 空话，必须说清楚测什么、怎么判断够了
- ~~"这个问题很复杂，需要根据项目情况决定"~~ → 必须给出决策框架
- ~~只写成功路径~~ → 必须覆盖失败模式和排障思路
- ~~只从一个角色视角写~~ → 必须标注涉及哪些角色的协作
