## 一、专栏定位

### 名称

- 正式标题：**《长线运营工程》**
- 副标题：**从第 1 个版本到第 500 个版本的技术可持续性**

### 一句话定位

> 讲清楚一个长线运营游戏在技术层面如何跨越数年保持健康——不是怎么发一个版本，而是怎么在发了几百个版本之后系统还能扛、团队还能转。

### 核心问题

这套专栏解决的核心问题：**为什么很多团队"能上线"但"运营不到一年技术就塌了"，以及如何把运营期的技术工作从被动救火升级为主动治理体系。**

### 与交付工程的关系

```
┌────────────────────────────────────────────────────┐
│  交付工程（Delivery Engineering）                    │
│  回答：Version N 怎么安全地送到用户手上               │
│  视角：单次交付循环                                   │
│  终点：灰度完成、指标正常、版本稳定                    │
├────────────────────────────────────────────────────┤
│  长线运营工程（Live Ops Engineering）                 │
│  回答：Version 1 → Version 500 的过程中技术怎么不塌   │
│  视角：跨越产品生命周期的时间维度                      │
│  起点：交付工程的终点——版本已稳定上线                  │
└────────────────────────────────────────────────────┘
```

交付工程是**管道设计**——管道建好了，每次版本沿着管道走。
长线运营工程是**管道在跑了 200 次之后还能不能用**——以及管道本身需要怎样演进。

### 边界划分

| 维度 | 交付工程管 | 长线运营工程管 |
|------|-----------|--------------|
| 时间跨度 | 一个版本周期（周级） | 产品生命周期（年级） |
| 核心问题 | 这次发版怎么不出事 | 第 200 次发版时系统还能不能扛 |
| 线上监控 | 看什么指标、怎么告警 | 指标基线怎么随版本漂移、怎么治理趋势 |
| 热修复 | 怎么做一次热修 | 一年做了 40 次热修说明什么、怎么改 |
| 技术债 | 不涉及 | 核心主题 |
| 引擎/SDK 升级 | 不涉及 | 核心主题 |
| 资源管理 | 包体策略、CDN 部署 | 运营 2 年后资源总量翻 5 倍怎么办 |
| 性能 | 优化技术、性能预算 | 性能基线随版本漂移的自动化检测与治理 |
| 版本节奏 | 分支策略、内容冻结 | 多版本并行、内容/技术解耦、节奏长期演进 |
| 团队 | 角色分工、发布审批 | 人员轮换下的知识传承、on-call 体系 |

### 前置依赖

读者需要先理解"一次交付是怎么回事"，才能讨论"怎么持续交付数百次"。

具体依赖交付工程的：
- **V01 交付总论** — 理解交付飞轮模型
- **V16 CI/CD 管线** — 本专栏多处以成熟 CI/CD 为前提
- **V17 灰度上线与线上运营** — 本专栏从 V17 的终点开始

但**写作顺序不需要等交付工程全部完成**。Part I（思维转换）和 Part IV（技术可持续性）可以独立启动，因为依赖的是实际项目经验而非交付工程的具体章节。

|--------|---------|---------------|

## 二、设计原则

### 与交付工程共享的原则

沿用交付工程的三层内容模型（原理层 / 实践层 / 平台层）和长青原则。

### 本专栏特有原则

```
原则一：时间维度优先——每个主题必须回答"第 1 天和第 365 天有什么不同"
原则二：趋势比快照重要——运营期的问题不是某个时刻的状态，而是变化的方向
原则三：决策框架优先于方案——运营期没有标准答案，给框架让读者自己判断
原则四：反模式比正确做法更有教学价值——运营期的坑比开发期更隐蔽
```

### 写作原则

- 必须区分"开发期的做法"和"运营期的做法"，说清楚为什么不同
- 数字和趋势优先于定性描述——"Crash 率从 0.1% 涨到 0.3%"比"Crash 率上升"有价值
- 必须覆盖"做了正确的事但时机不对"的情况——运营期时机和做法同样重要
- 不写"应该做长期规划"这种空话，要给出具体的规划周期、检查节点、调整机制

### AI 集成策略

AI 不作为独立专栏，而是作为**横切关注点**嵌入各卷的具体场景中。原因：

1. AI 工具层半年过期，独立专栏保鲜期太短，与站调性冲突
2. AI 的价值在具体场景下才能体现，脱离场景的"AI 教程"缺乏工程深度
3. 读者搜索"运营期知识传承"时，希望在那个主题下直接看到 AI 方案

**三层结构**：

```
┌──────────────────────────────────────────────┐
│  判断框架层（长青，独立文章）                    │
│  1 篇 essays：游戏工程团队引入 AI 的判断框架    │
│  怎么判断该不该用、四种集成模式、ROI 评估        │
│  → 放在 essays/ 板块                          │
├──────────────────────────────────────────────┤
│  场景应用层（嵌入各卷章节）                      │
│  在 V03/V04/V05/V10/V12/V14 各增加 AI 小节    │
│  讲"在这个具体场景下 AI 怎么用"                 │
├──────────────────────────────────────────────┤
│  工具实践层（最易过期，最小化）                   │
│  只在实践层提及当前可用工具，不绑定特定产品        │
│  工具变了更新这一层，上两层不动                   │
└──────────────────────────────────────────────┘
```

## 三、核心问题集

以下 12 个问题是本专栏要回答的核心问题（交付工程 15 个核心问题均不覆盖这些）：

1. **从开发到运营，技术决策逻辑怎么变？** 开发期功能优先可以重来，运营期稳定优先每次变更都是风险。
2. **每两周发一个版本，怎么保证第 100 个版本的质量不低于第 1 个？** 质量不会自动维持，需要主动治理。
3. **线上问题怎么分级？什么紧急热修、什么等下个版本、什么可以不修？** 运营期的问题远多于能修的，分级是核心能力。
4. **技术债什么时候还、还多少、怎么跟版本节奏协调？** 不还会塌，全还会停，需要节奏。
5. **运营中的游戏怎么升引擎版本？** 最危险的操作之一，但迟早要面对。
6. **性能基线会随版本漂移，怎么发现、怎么治理？** 每个版本慢一点，半年后就是事故。
7. **资源总量每个版本都在涨，什么时候会成为问题？怎么控？** 内容是产品的命脉，不能不加，但加了要管。
8. **功能开关和 A/B 测试的工程基建怎么搭？** 运营期需要能安全地试错。
9. **多版本并行开发时，怎么管分支、管配置、管验证？** 运营期永远在同时做当前版本修复 + 下个版本开发 + 下下个版本预研。
10. **运营数据怎么反向驱动技术决策？** 用数据而不是直觉说服团队做技术改造。
11. **团队轮换了，技术知识怎么不丢？** 一个运营 3 年的项目，初始团队可能只剩 30%。
12. **什么时候该把项目经验抽象成团队基建？** 从"这个项目的做法"到"工作室的标准做法"。

## 八、写作模板

沿用交付工程的统一写作模板，增加以下运营期特有小节：

```markdown
# 标题

## 这篇解决什么运营问题
一句话说清。注意是"运营问题"，不是"技术问题"。

## 开发期 vs 运营期
明确说清楚这个主题在两个阶段的差异。如果没有差异，这篇不该存在于本专栏。

## 本质是什么（原理层）
引擎无关、项目无关的原理性解释。

## 反模式
运营期特有的错误做法。重点写"看起来合理但运营期会出问题的做法"。

## 工程化做法（实践层）
用当前技术栈展示落地方案。

## 时间维度
这个做法在第 1 天、第 100 天、第 365 天分别是什么样？
如果答案是"一样"，这一节可以省略。

## 度量与可见性
怎么知道做得好不好？看什么数字？

## 小结与检查清单
3-5 条要点 + Checklist。
```
