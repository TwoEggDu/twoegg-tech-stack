# HybridCLR AOT 泛型修法专题计划

## 定位

这不是新的 HybridCLR 主线系列，也不是对 `HCLR-02`、`HCLR-03` 的重写。

它的目标是：

`把现有系列里“已经讲过的原理、工具链、案例、决策图”进一步收束成一组可执行的修法专题，让读者知道共享类型怎么判断、DisStripCode 怎么写、AOTGenericReferences / metadata / 裁剪怎么分工，以及这些坑怎么在工程流程里前移。`

## 为什么值得单独补

当前 HybridCLR 系列已经有：

- `HCLR-02`：AOT 泛型与补充元数据的原理边界
- `HCLR-03`：`LinkXml`、`AOTDlls`、`MethodBridge`、`AOTGenericReference` 的工具链地图
- `HCLR-17`：async 崩溃案例
- `HCLR-18`：`Dictionary<ValueTuple, 热更类型>` 案例
- `HCLR-19`：`DisStripCode`、`link.xml`、补 metadata 的决策图

但中间还缺一条明显的带状区域：

- 共享类型和具体闭包类型怎么区分
- DisStripCode 到底该怎么写
- `AOTGenericReferences`、DisStripCode、补 metadata 之间怎么协作
- 高频坑型如何归类
- 怎么把这些坑前移到流程里

这正是读者最容易卡住、但目前还没有被系统化讲透的部分。

## 与现有文章的分工

- 不重讲 `LoadMetadataForAOTAssembly` 的底层运行链，见 `HCLR-02`
- 不重讲工具链按钮和生成物的全景地图，见 `HCLR-03`
- 不把专题写成单一事故复盘，案例边界留在 `HCLR-17`、`HCLR-18`
- 不重复 `HCLR-19` 的“先走哪条路”入口判断，而是承接它，往下讲“决定走这条路之后具体怎么做”

## HCLR-20 详细提纲

### 标题

HybridCLR 共享类型判断｜为什么报错是 `ValueTuple<int,string>`，你写的却可能是 `ValueTuple<int,object>`

### 定位

这篇不是 AOT 泛型总论，也不是 DisStripCode 模板篇。

它只做一件事：

`把“日志里看到的具体闭包类型”和“实际该写进 DisStripCode 的共享类型”拉开。`

### 建议字数

- 最低可用线：`2500` 字
- 标准完成线：`3500-4500` 字
- 尽量不要超过：`5500` 字

### 开头策略

- 从读者最直觉的误判切入
- 第一段就抛出一个反直觉例子
- 不先讲术语定义，而是先讲“为什么你照着日志写，会写错”

### 核心判断

如果只允许用一句话概括全文，我建议写成：

`HybridCLR AOT 泛型修法里，日志报出来的是“当前缺口长什么样”，而 DisStripCode 真正需要你写进去的，常常是“这组泛型在共享规则下该落到什么类型”。`

### 推荐结构

#### 1. 读者为什么会被日志里的类型名带偏

- 这一节只立问题
- 用 `ValueTuple<int,string>` 做最小示例
- 先告诉读者：`看到什么` 不等于 `该写什么`

#### 2. 什么是“具体闭包类型”，什么是“共享类型”

- 只讲和修法直接相关的最小定义
- 不展开 ECMA 或运行时实现细节
- 把“日志层”和“保留层”的视角分开

#### 3. 值类型、引用类型、嵌套泛型在共享判断上怎么不同

- 值类型为什么不能直接被 `object` 整体覆盖
- 引用类型为什么经常能落到 `object`
- 嵌套泛型为什么最容易误判

#### 4. 三组最小例子：`ValueTuple`、`List<HotfixType>`、`Dictionary<K,V>`

- 每组只讲一个判断点
- 不扩成模板大全
- 每个例子都要落到“日志类型 -> 共享类型 -> 为什么”

#### 5. 最后收成一张共享类型判断表

- 把本篇的规则压成可以查的形式
- 这一张表要能直接成为 `HCLR-21` 的前置输入

### 写作风险

- 最容易写成“泛型共享原理总论”，导致和 `HCLR-02` 重复
- 最容易提前滑进 DisStripCode 代码模板，导致和 `HCLR-21` 重复
- 最容易把“共享类型”和“metadata / native 路径选择”混在一起，导致和 `HCLR-19` 重复

### 文末导读

- **下一步应读：** `HCLR-21` — 因为判断完共享类型，下一步就是把它写进 DisStripCode
- **扩展阅读：** `HCLR-18` — 因为它提供了这组专题最具体的起点案例
