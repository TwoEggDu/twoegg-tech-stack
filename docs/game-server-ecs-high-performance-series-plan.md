# 高性能游戏服务端 ECS 系列规划

> 本系列是 `dots-mass-hardware-deep-series-plan.md` 向服务端的延伸，
> 也是 `系列十三（游戏后端）` 的深度补充层（十三·G）。
> 前者解决客户端 DOD；这里解决同一问题在服务端的不同形态。

---

## 系列定位

游戏服务端和客户端面对的是同一个计算问题——**大量同构对象的高频状态更新**——但约束完全不同：

| 维度 | 客户端 | 服务端 |
|------|--------|--------|
| 主要瓶颈 | CPU 模拟 + GPU 渲染 | CPU 模拟 + 网络 I/O |
| 并发模型 | 单 World，多线程 | 多 World（多房间），各自单/多线程 |
| 持久化 | 无 / 本地存档 | 必须写数据库 |
| 横向扩展 | 不需要 | 必须支持动态扩缩容 |
| 语言选择 | 基本锁死在引擎语言 | C++ / Rust / C# / Go 都有可能 |
| 渲染层 | 存在 | 不存在，完全砍掉 |

这些差异让服务端 ECS 的架构取舍和客户端完全不同。光把客户端 DOTS 搬过来是不够的。

**一句话定位**：讲清楚高性能游戏服务端怎样用 ECS 组织仿真逻辑，以及在语言、框架、I/O模型、持久化、横向扩展这几个约束下，各条路的取舍在哪里。

## 系列结构（13 篇）

### Part 1：问题定位（2 篇）

| 编号 | 标题 | 核心问题 |
|------|------|---------|
| SV-ECS-01 | 游戏服务端为什么需要 ECS：和客户端的问题空间有什么本质区别 | 不是"客户端 ECS 很好用所以服务端也用"，而是服务端本身有大量同构对象高频更新的问题——这正是 ECS 擅长的 |
| SV-ECS-02 | 服务端 ECS 的五个核心约束：无渲染、有 I/O、有持久化、有多 World、有横向扩展 | 这五个约束决定了服务端 ECS 的设计必须和客户端不同，也是选框架的判断依据 |

### Part 2：框架选型（4 篇）

| 编号 | 标题 | 核心问题 |
|------|------|---------|
| SV-ECS-03 | Flecs 深度：架构、Observer、REST API、Pipeline、World Streaming，以及和 DOTS 的设计哲学对比 | Flecs 是目前功能最完整的独立 ECS，内置 Web Explorer 和 REST，适合需要工具链的项目 |
| SV-ECS-04 | EnTT 深度：header-only C++、Signal、View、Group 的性能模型，Minecraft Bedrock 的选型理由 | EnTT 比 Flecs 更轻，没有内置调度器——极简但高性能，控制权在开发者手里 |
| SV-ECS-05 | Bevy ECS on Server：Rust 内存安全 + ECS 性能，Schedule、System、Resource 在无渲染环境下的用法 | Bevy 的 ECS 内核可以独立使用，Rust 的所有权模型天然契合多 World 隔离 |
| SV-ECS-06 | C# 服务端 ECS：Arch / Leopotam 的架构定位，与 Unity DOTS Headless Server 的对比和选型路径 | 对 Unity 技术栈团队来说，选 DOTS Headless 还是独立 C# ECS，取决于是否需要和客户端代码共享 |

### Part 3：核心工程问题（5 篇）

| 编号 | 标题 | 核心问题 |
|------|------|---------|
| SV-ECS-07 | Multi-World 隔离：每个房间一个 World，World 生命周期、资源共享与跨 World 查询的边界 | 服务端最典型的 ECS 形态——每个游戏房间是独立的 World，互不干扰，可以并行跑在不同线程 |
| SV-ECS-08 | ECS + 异步 I/O：游戏逻辑的同步/并行世界与网络 I/O 的异步世界怎样接起来 | ECS 系统是同步的，网络 I/O 是异步的，中间需要一层消息队列或通道作为缓冲——怎样设计这个边界 |
| SV-ECS-09 | 状态广播：哪些 Component 变化需要同步给哪些客户端，Delta 压缩与兴趣管理（AOI）的 ECS 实现 | Flecs 的 Observer 和 EnTT 的 Signal 如何用来驱动状态广播，与传统 Dirty Flag 方案的对比 |
| SV-ECS-10 | 持久化边界：ECS World 快照如何映射到数据库，冷热分离、存档时机与宕机恢复策略 | ECS 的 Chunk 布局和关系型数据库的行式存储是两种完全不同的模型，中间层怎样设计 |
| SV-ECS-11 | Tick 预算管理：固定时间步、服务器权威、Tick 超载的检测与降级策略 | 服务端 Tick 超时比客户端更危险（影响所有玩家），怎样做预算监控和自动降频 |

### Part 4：实战与选型（2 篇）

| 编号 | 标题 | 核心问题 |
|------|------|---------|
| SV-ECS-12 | 案例拆解：大规模多人游戏服务端的 ECS 落地（SpatialOS 架构回顾 + 自研方案参考） | SpatialOS 是第一个把 ECS 概念系统化用在大规模游戏服务端的商业方案，它的架构取舍和失败教训都值得看 |
| SV-ECS-13 | 服务端 ECS 选型决策地图：语言 × 规模 × 团队栈 × 客户端代码共享程度的四维判断 | 没有最好的框架，只有最合适的选择——这篇把前 12 篇的判断依据收回成一张可操作的决策图 |

## 写作约束

- **每篇必须给出语言/框架的具体 API 示例**，不停留在概念层
- **性能对比必须有数字**：Flecs/EnTT 的 benchmark 数字（官方 repo 有），不靠"据说更快"
- **服务端特有的问题优先**：不重复 `十八·B DOTS工程` 已经讲清楚的东西，只讲服务端额外面对的
- **每篇结尾必须回答"什么情况下不该用这个方案"**
