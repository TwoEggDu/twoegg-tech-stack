# 质量护栏子系列计划

## 这份计划解决什么

前面的 `code-quality` 栏目已经把三件事立住了：

- 代码质量不是洁癖，而是交付能力
- code review 应该优先拦结构风险
- 该自动化的规则不要继续靠人盯

但“自动化测试怎样保证工程质量”“AI 怎样参与代码评审”这两件事，还没有被收成一条连续主线。

这份计划专门回答四件事：

1. 这组文章到底在解决什么问题
2. 自动化测试、AI review、人类 review、CI 各自负责什么
3. 这组文章该按什么顺序写
4. 第一批该先推进哪几篇

## 子系列名称

建议固定为：

`质量护栏`

这个名字的好处是：

- 比“自动化测试”更宽，能自然包含 AI review 和 CI
- 比“QA”更贴近工程判断，不会滑成测试管理术语
- 能和现有 `code-quality` 栏目自然衔接

## 一句话定位

`人负责判断，机器负责重复规则，AI 负责第一轮风险扫描，CI 负责把已确定的质量门槛硬拦下来。`

## 目标读者

- 客户端 / 工具链 / 技术负责人
- 已经有多人协作、版本压力和发布节奏的团队
- 不满足于“大家认真一点”，希望把质量变成工程机制的人

## 这组文章不解决什么

- 不写成某个测试框架的 API 教程
- 不写成某个 AI 工具的产品说明书
- 不写成抽象 checklist 堆砌
- 不把质量问题全压成“上工具就行”

这组文章真正要讲的是：哪些该交给人，哪些该交给自动化，哪些适合让 AI 先扫一遍，以及它们怎样接成闭环。

## 核心判断框架

整组文章都围绕这 4 条判断展开：

1. 可枚举、可重复、规则明确的问题，优先自动化
2. 依赖上下文、阶段目标和设计取舍的问题，交给人判断
3. AI 最适合做第一轮风险扫描，不适合做最终裁决
4. 质量机制只有接进 CI / 发布流程，才算真正落地

## 文章目录

### QG-01 自动化测试不是写几个单测，而是质量门禁

要回答的问题：

`为什么“会写测试”不等于“团队有质量护栏”，以及真正有用的验证链应该怎样分层。`

必须覆盖：

- 编译 / lint / 静态检查 / 契约校验 / smoke / regression / baseline 的分层
- 测试不是孤立技能，而是门禁链的一部分
- 为什么很多团队“有测试”但还是不稳

状态：`已起草`

### QG-02 哪些问题该靠自动化测试拦，哪些问题不该

要回答的问题：

`测试边界在哪里，哪些问题必须变成自动化验证，哪些问题更适合 review 或人工判断。`

必须覆盖：

- 纯逻辑
- 配置契约
- 资源链路
- 构建产物
- 性能 / 包体 / 加载时间基线
- 哪些问题不值得写测试

状态：`已起草`

### QG-03 游戏 / 客户端项目最值得先自动化的 5 类验证

要回答的问题：

`如果团队资源有限，最值得先自动化什么，先后顺序怎么排。`

必须覆盖：

- 数值 / 规则回归
- 配置表契约
- 关键资源 smoke
- 构建产物校验
- 基线阈值

状态：`已起草`

### QG-04 AI code review 应该抓什么，不应该抓什么

要回答的问题：

`AI 在代码评审里最适合承担哪一层，哪些问题值得它先扫，哪些问题不能让它代替 reviewer 决策。`

必须覆盖：

- 边界被打穿
- 规则重复散落
- 临时补丁混进长期层
- 接口语义含糊
- 异常 / 回滚路径不清
- 缺少必要测试
- 为什么 AI 不适合做最终架构裁决

状态：`已起草`

### QG-05 AI review、人类 review、CI 怎样分工

要回答的问题：

`当 AI、reviewer、自动化测试和 CI 同时存在时，最稳的职责切分是什么。`

必须覆盖：

- AI 提疑点
- 人做裁决
- CI 做硬拦
- 什么规则适合前移到本地提交
- 什么规则适合放到 PR / nightly / release gate

状态：`第二批推进`

### QG-06 Quality Gate：怎样把自动化测试和 AI 评审接进发布流程

要回答的问题：

`如何把这整套质量护栏真正接进团队流程，而不是停留在文章和口头共识。`

必须覆盖：

- 提交前
- PR 阶段
- 合并后
- 夜间回归
- 发版前 gate
- 上线后监控回流

状态：`第二批推进`

## 推荐写作顺序

1. 先写 `QG-01`
原因：先把“测试不是单点技术，而是门禁链”立住。

2. 再写 `QG-02`
原因：不先讲边界，后面很容易变成“什么都写测试”。

3. 再写 `QG-03`
原因：给团队一个低成本可执行起点。

4. 再写 `QG-04`
原因：把 AI 放回正确位置，避免一上来就写工具测评。

5. 再写 `QG-05`
原因：收束 AI、人类和 CI 的职责边界。

6. 最后写 `QG-06`
原因：把前面几篇收成完整发布流程。

## 第一批执行方案

第一批先并行推进 4 篇：

- `QG-01 自动化测试不是写几个单测，而是质量门禁`
- `QG-02 哪些问题该靠自动化测试拦，哪些问题不该`
- `QG-03 游戏 / 客户端项目最值得先自动化的 5 类验证`
- `QG-04 AI code review 应该抓什么，不应该抓什么`

这 4 篇的关系是：

- `QG-01` 立框架
- `QG-02` 切边界
- `QG-03` 给落地顺序
- `QG-04` 把 AI 放回合理位置

等第一批稳定后，再用 `QG-05` 和 `QG-06` 做流程收束。

## 这组文章的写法约束

- 每篇只抓一个核心判断，不堆工具名
- 必须讲清“谁负责什么”，不混淆人、AI、自动化、CI
- 要给最小可执行路径，而不是理想状态
- 尽量复用你已有的资源、构建、回归、变体治理案例
- 避免写成“会用某某工具 = 工程质量好”
