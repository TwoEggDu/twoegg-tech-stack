# 质量护栏子系列计划

## 这份计划解决什么

这组文章现在不再只是回答“要不要写测试”或“AI review 有没有用”。
它真正要解决的是：

1. 哪些质量问题该交给机器
2. 哪些质量问题必须继续交给人
3. AI、人类 review、自动化测试、CI 各自站在哪一层
4. 这些能力怎样接成真正会影响合并和发版的 gate

## 子系列名称

建议固定为：

`质量护栏`

这个名字现在仍然成立，因为它天然能同时覆盖：

- 自动化测试
- AI review
- 人类 review
- CI / Quality Gate
- Baseline 和发布门槛

## 一句话定位

`人负责判断，AI 负责第一轮风险扫描，自动化负责证明行为和契约，CI 负责把已确定的门槛硬拦下来。`

## 目标读者

- 客户端 / 工具链 / 技术负责人
- 已经有多人协作、版本压力和发布节奏的团队
- 不满足于“大家认真一点”，希望把质量变成工程机制的人

## 这组文章不解决什么

- 不写成某个测试框架的 API 教程
- 不写成某个 AI 工具的产品说明书
- 不写成抽象 checklist 堆砌
- 不把质量问题全压成“上工具就行”

这组文章真正讲的是：谁负责什么，什么该自动化，什么该保留给人判断，以及这些判断怎样接成稳定流程。

## 核心判断框架

整组文章都围绕这 4 条判断展开：

1. 可枚举、可重复、规则明确的问题，优先自动化
2. 依赖上下文、阶段目标和设计取舍的问题，交给人判断
3. AI 最适合做第一轮风险扫描，不适合做最终裁决
4. 质量机制只有接进 CI / 发布流程，才算真正落地

## 文章目录

### QG-01 自动化测试不是写几个单测，而是质量门禁

职责：先把验证链的层次立住，解释为什么“会写测试”不等于“团队有质量护栏”。

状态：`已写`

### QG-02 哪些问题该靠自动化测试拦，哪些问题不该

职责：切清测试边界，说明哪些问题该前移成自动化验证，哪些更适合 review 或人工判断。

状态：`已写`

### QG-03 游戏 / 客户端项目最值得先自动化的 5 类验证

职责：给团队一个最小可执行起点，按优先级而不是按理想覆盖率排顺序。

状态：`已写`

### QG-04 AI code review 应该抓什么，不应该抓什么

职责：把 AI 放回第一轮风险扫描的位置，优先抓结构风险、异常路径和测试缺口，不让它越权做架构终判。

状态：`已写`

### QG-05 AI review、人类 review、CI 怎样分工

职责：把 AI、reviewer 和 CI 的职责边界拆开，说明谁提疑点、谁做裁决、谁做硬拦。

状态：`已写`

### QG-06 Quality Gate：怎样把自动化测试和 AI 评审接进发布流程

职责：把前面 5 篇收成一条完整流程，说明提交前、PR、合并后、nightly、发版前和上线后各自放什么 gate。

状态：`已写`

## 推荐阅读顺序

1. 先读 `QG-01`
原因：先把“测试不是单点技巧，而是门禁链”立住。

2. 再读 `QG-02`
原因：不先切边界，后面很容易退化成“什么都写测试”。

3. 再读 `QG-03`
原因：给团队一个低成本可执行起点。

4. 再读 `QG-04`
原因：先把 AI 放回正确位置，避免直接滑成工具测评。

5. 再读 `QG-05`
原因：把 AI、人类 review 和 CI 的职责切开。

6. 最后读 `QG-06`
原因：把前面几篇收成真正的发布流程。

## 这 6 篇现在形成的主线

- `QG-01` 立验证链
- `QG-02` 切测试边界
- `QG-03` 给落地优先级
- `QG-04` 放正 AI 位置
- `QG-05` 切清职责分工
- `QG-06` 接进流程门禁

也就是说，这条线现在已经不只是“怎么测试”，而是在回答“团队怎样把质量判断稳定地放进协作流程”。

## 写法约束

- 每篇只抓一个核心判断，不堆工具名
- 必须讲清“谁负责什么”，不混淆人、AI、自动化、CI
- 要给最小可执行路径，而不是理想状态
- 尽量复用你已有的资源、构建、回归、变体治理案例
- 避免写成“会用某某工具 = 工程质量好”
