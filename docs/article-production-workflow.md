# TwoEgg 单篇生产流程

> 这份文件回答的是：`一篇文章怎样从想法走到发布。`

## 1. 适用范围

这份流程适用于：

- 新文章起草
- 旧文章重写
- 系列中的单篇推进

它不替代：

- 系列 plan
- 写作方法规范
- 证据卡模板

也就是说：

- 系列 plan 决定写什么
- 写法规范决定怎么写
- 这份流程决定怎么推进到成稿

## 2. 默认流程总览

默认按下面这条链推进：

```text
选题
-> 判断文章类型
-> 确认系列与边界
-> 起 outline
-> 补前置资料 / 证据
-> 写初稿
-> 自查与修订
-> 构建验证
-> 发布 / 回写状态
```

这条链的核心目标不是增加流程感，而是避免两种常见坏法：

- 没有 outline 就直接写，最后结构散
- 文章写完了，但系列状态和索引没有回写

## 3. 第一步：选题

先回答：

- 这篇想解决什么问题
- 它属于哪个系列
- 它是否真的值得单独成篇

如果一个话题只是某篇正文里的一段，或者没有稳定问题空间，就不要硬拆成单篇。

## 4. 第二步：判断文章类型

开写前先判断这是哪一类文章。

默认从下面四种里选：

- 原理篇
- 映射篇
- 案例篇
- 索引篇

不要写到一半才决定它到底是什么。

文章类型一旦判断清楚，后面的 outline 才会稳定。

## 5. 第三步：确认系列与边界

在 outline 前先回答：

- 这篇在本系列里站在哪
- 它的前置文章是什么
- 它不该吞掉哪些别的系统

例如：

- 技能系统里的 AI 边界篇，不该写成完整 AI 系列
- 技能系统里的多人同步篇，不该写成完整网络同步百科

如果这一步不做，文章很容易越写越大，边界失控。

## 6. 第四步：起 outline

默认先读：

- `docs/article-writing-method.md`
- `docs/article-outline-template.md`

然后根据文章类型起 outline。

最少也要先有：

- 这篇在解决什么问题
- 抽象模型是什么
- 最后落到什么具体实现

没有 outline 就直接写，大概率会导致：

- 第一屏先讲 API
- 中间没有抽象模型
- 结尾回不到工程实现

## 7. 第五步：补前置资料 / 证据

不是每篇都要重证据驱动，但默认要先判断：

- 这篇是否需要查官方资料
- 这篇是否需要引用已有 plan / outline / evidence card
- 这篇是否需要交叉引用仓库内已有文章

建议分三种情况。

### 7.1 纯原理与方法文

如果主要在讲问题空间、抽象模型、工程边界，且内容稳定，可以少依赖外部资料。

### 7.2 引擎实现文

如果明显涉及：

- Unity / Unreal 具体机制
- 版本差异
- 官方 API / 包行为

就应该查证，不要只凭记忆写。

### 7.3 案例与诊断文

要优先整理：

- 现象
- 排查过程
- 根因证据
- 修法依据

不要只留最终结论。

## 8. 第六步：写初稿

写初稿时，默认遵守下面几条：

- 第一屏先立问题，不先讲 API
- 中间必须有抽象模型层
- 最后必须回到具体实现
- 要主动拆边界，而不是把交叉系统写糊
- 末尾用一句最短结论收口

写初稿时可以不追求“完全正确的字句”，但结构必须先站住。

## 9. 第七步：自查与修订

初稿出来之后，至少自查下面这些问题：

### 9.1 结构检查

- 问题空间是否清楚
- 抽象模型是否存在
- 是否落回具体实现
- 系统边界是否清楚

### 9.2 文章类型检查

- 原理篇有没有退化成工具说明书
- 映射篇有没有退化成 API 对照表
- 案例篇有没有只写现象，不写根因
- 索引篇能不能真的回答“先读哪篇”

### 9.3 交叉引用检查

- `series` 是否正确
- `slug` 是否稳定
- 文内链接 / `relref` 是否正确
- 是否需要在索引页或系列 plan 里回写

## 10. 第八步：构建验证

文章不是写完 Markdown 就结束了。

至少要做一次站点构建验证，确认：

- front matter 合法
- 页面可被 Hugo 正常生成
- `relref` 没断
- 没有明显渲染结构问题

如果文章涉及系列索引页，还要确认它的阅读路径和链接能成立。

## 11. 第九步：发布 / 回写状态

一篇文章发布之后，至少要回写这些地方中的相关项：

- 本系列专属 plan
- 总 `doc-plan`
- 系列索引页

如果只把文章发出去，不更新系列状态，后面的 plan 很快就会过期。

## 12. 什么时候需要单独 evidence card

不是每篇都要 evidence card。

更适合单独补 evidence card 的情况通常是：

- 引擎实现细节重
- 明显依赖官方资料
- 计划做系统性系列
- 同一主题会反复扩写

如果只是高层原理或方法文，不必机械地先建 evidence card。

## 13. 什么时候适合直接写，不必先补完整计划

下面这些情况通常可以直接写：

- 这个主题边界已经很清楚
- 文章类型已经明确
- 系列主线已经存在
- 这篇不会改变整个系列结构

换句话说，不是每篇都要先做大 planning。

但即使直接写，也应该至少先有 outline。

## 14. 发布前最小检查清单

发布前至少确认：

- 标题、slug、series 是否一致
- 文章类型是否明确
- 第一屏是否先立问题
- 是否有抽象模型
- 是否落回具体实现
- 是否讲清系统边界
- 是否有一句最短结论
- 本地构建是否通过
- 是否需要更新系列索引或 plan

## 15. 最短结论

`一篇文章的生产，不是“想到就写完发出去”，而是先定类型和边界，再起 outline，再补必要资料，再写稿、验证、回写状态。结构和维护都应该从一开始就被设计进去。`
