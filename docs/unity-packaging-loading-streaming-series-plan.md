# Unity 打包、加载与流式系统专栏规划

## 专栏定位

这组文章不写成“怎么点 Build 按钮”的操作教程，也不写成“Addressables / YooAsset 配置项速查”。

它真正要解决的问题是：

`把 Unity 资源交付链拆成一张稳定地图，讲清楚内容为什么要这样切包、运行时为什么会这样加载、项目为什么会在包体、热更、卡顿和内存上反复翻车。`

一句话说，这个专栏的重心不是 API 记忆，而是：

`资源交付链的工程判断 + 关键源码链路解读。`

---

## 为什么不能只写 3 篇

如果只写：

- 首包边界
- 压缩权衡
- 异步加载

这三篇只能解决“现象判断”，还不够解决“为什么它在 Unity 里会这样长”。

而打包链路的问题，恰恰有很大一部分不是配置问题，而是源于下面这些底层边界：

- `BuildPipeline / SBP / Addressables Build Script` 各自站在哪一层
- 资源在构建期是怎么被收集依赖、切分、序列化和压缩的
- `AssetBundle` 文件内部到底装了什么
- 运行时所谓“加载 bundle”到底经过了哪几段恢复链

所以这条专栏不能只有“工程判断文”，还必须配一条正式的：

`源码解读支线`

而且这条支线不是番外，而是主线证据层。

---

## 这条专栏在总规划里的位置

它最合适挂在现有的：

`系列八：打包、加载与流式系统`

而不是另起一个平行系列。原因很直接：

- 上游已经有 [`Unity 资产系统与序列化原理`](/E:/NHT/workspace/TechStackShow/docs/unity-asset-system-and-serialization-series-plan.md)，负责解释“资源是什么、引用怎么成立、序列化怎么恢复”。
- 下游已经有 `YooAsset`、`HybridCLR`、`CI/CD`、`质量保障` 等专题，负责解释框架接入、发布治理和验证闭环。
- “打包”本身不是孤立话题，它天然连着 `切包策略 -> 压缩与交付 -> 运行时加载 -> 内存与卡顿 -> 流式治理` 这一整条链。

所以更稳的做法不是重新造一个新栏目，而是把“打包”明确收束成系列八的主线入口，再把源码解读作为这一系列内部的证据支线接进来。

## 这组文章要持续回答的四个问题

1. 资源为什么要这样切包，而不是按目录随便分。
2. 运行时说“异步加载”时，到底异步了哪几段，卡顿又卡在了哪几段。
3. 大体量项目为什么最后一定会走到版本、缓存、回滚、按需下载和流式治理。
4. 这些工程判断在 Unity 的构建产物和源码链路里，证据到底落在哪。

## 专栏边界

这组文章明确讲：

- AssetBundle 的切包、压缩、交付和运行时加载链
- Addressables / YooAsset 站在资源交付链里的位置
- 首包、分包、按需下载、差量更新、缓存、回滚这些工程问题

- Android AAB / PAD、iOS ODR 的平台接入教程细节
- 贴图压缩格式、视频编码、音频编码的独立专题细节
- 把整套文章写成“Unity 引擎文件夹导览”

这些内容可以引用，但不喧宾夺主。

## 专栏结构

### 主线 A：打包策略层

职责：
回答“资源为什么这样切、这样压、这样发”。

- `打包-01` Asset Bundle 打包粒度：粗粒度 vs 细粒度，依赖关系与冗余控制
- `打包-02` 首包体积优化：分包策略、按需下载、差量更新（热更 patch）原理
- `打包-03` 资源压缩方案：LZ4 vs LZMA，解压速度与包体大小的权衡
- `打包-04` Addressables / YooAsset 打包配置：Group 划分、Bundle 布局最佳实践

### 主线 B：加载优化层

职责：
回答“资源到了客户端以后，为什么会这样加载、这样卡、这样爆内存”。

- `加载-01` 异步加载管线：AsyncOperation / UniTask / 协程在加载中的正确用法
- `加载-02` 加载优先级调度：关键路径资源优先、后台预加载、加载进度反馈
- `加载-03` 加载时的内存控制：缓存池、引用计数、卸载时机，避免内存尖峰
- `加载-04` 场景加载优化：Loading Screen 设计、帧时间分摊、分帧实例化

### 支线 C：源码解读层

职责：
把主线里的关键判断，落回到 `构建分层 -> 构建产物 -> 运行时恢复链` 的证据上。

- `源码-01` Unity 的资源构建管线到底分几层：BuildPipeline、SBP、Addressables Build Script 各自站在哪
- `源码-02` Unity 怎么把资源编成 AssetBundle：依赖、序列化、Manifest、压缩到底发生了什么
- `源码-03` AssetBundle 文件内部结构：Header、Block、Directory、SerializedFile 是怎么组织的
- `源码-04` AssetBundle 运行时加载链：下载、缓存、依赖、反序列化、Instantiate、Unload 怎么接起来
- `源码-05` 怎么看 Unity 资源构建产物：Manifest、BuildLayout、Catalog 和缓存目录到底在告诉你什么

### 主线 D：流式系统层

职责：
回答“为什么项目一旦进入大世界或长线运营，加载问题一定升级成流式系统问题”。

- `流式-01` Level Streaming 基础：Scene 异步加载/卸载、触发区域设计
- `流式-02` Unreal World Partition：新一代大世界管理，HLOD 自动生成

## 和相邻系列的边界

### 相对《Unity 资产系统与序列化原理》

那个系列负责解释：

- 资源对象是什么
- GUID / fileID / PPtr 怎么成立
- Importer、序列化、恢复链怎么工作

这个系列负责继续往后走，解释：

- 这些资源对象怎么被切进交付产物
- 交付产物怎么被下载、缓存、解压、反序列化和实例化
- 项目如何围绕这些链路做工程治理
- 这些结论在源码和构建产物里怎么落证据

### 相对《插件与框架》

`YooAsset`、`Addressables` 相关内容只在本系列里承担“管理层/调度层”的角色。

不要把它们写成底层格式本身，也不要把框架选型讨论写成二进制格式原理的替代。

### 相对《从引擎源码阅读到自研引擎》

那个系列更上位，负责建立“如何读引擎源码”的长期方法论。

这个系列里的源码解读更窄，只服务于：

`资源交付链`

也就是说，这里不做泛化的“Unity 源码导览”，只追打包、构建、交付和加载相关的关键链路。

### 相对《CI/CD 与工程质量》

本系列讲“资源交付链怎么设计”。
CI/CD 系列讲“这套链怎么进流水线、怎么验证、怎么阻断坏版本”。

## 源码支线的写法约束

源码支线不能写成“类名/函数名导览”，必须遵守这套顺序：

1. 先回答这篇到底要解释什么工程问题。
2. 先给问题地图，再进实现细节。
3. 只追关键主链，不平铺目录树。
4. 明确区分“源码事实”和“工程判断”。
5. 最后收回到项目侧：这个实现意味着你该怎么做切包、压缩、加载和验证。

推荐骨架：

1. 这篇要回答什么问题
2. 先给地图：这一层站在哪
3. 关键链路/关键数据结构是什么
4. 从源码或构建产物里直接能看到什么
5. 为什么会这样设计
6. 这对项目判断意味着什么
7. 常见误判
8. 最后压成一句话

## 证据边界

当前仓库里的 [`engine-source-roots`](/E:/NHT/workspace/TechStackShow/docs/engine-source-roots.md) 还没有把 Unity 本地源码路径标成 `READY`。

所以这条支线在真正落稿时，证据优先级应该是：

1. `官方可公开 package/source`，例如 `SBP`、`Addressables` 相关包代码
2. `构建产物证据`，例如 `Manifest`、`BuildLayout`、`Catalog`、缓存目录
3. `官方文档`
4. 本地完整引擎源码路径配置完成后，再补更底层引擎实现证据

也就是说，源码解读现在就能做，但要诚实地区分：

- 哪些结论来自公开 package/source 和构建产物
- 哪些结论需要等本地 Unity 引擎源码路径配置完成后再继续向下挖

## 每篇文章的统一写法约束

- 每篇只回答一个真实工程问题，不写成概念杂烩。
- 顺序固定为：`原理/地图 -> 实现/机制 -> 项目对应 -> 常见误判 -> 落地判断`。
- 如果提到 Addressables / YooAsset，先说它站在哪一层，再说它解决什么，不要直接进入配置项。
- 如果提到 AssetBundle，优先从“交付容器”角度解释，不先从 API 角度解释。
- 源码解读文不堆目录树，不按类名流水账写。
- 代码和配置示例默认以 `Unity 2022.3 LTS` 为主，`Unity 6` 的差异只在必要处注记。
- 文末必须落一张判断表、检查表或选型表，避免文章变成纯道理文。
