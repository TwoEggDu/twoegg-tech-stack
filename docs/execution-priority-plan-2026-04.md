# 执行优先级计划

> 日期：2026-04-13
> 原则：先修路，再立招牌，最后深挖矿

---

## 为什么是这个顺序

当前站点有 763 篇文章，技术深度已经是核心竞争力。
但三个结构性问题正在浪费已有资产：

1. **30 篇文章 series 字段为空** — Hugo 无法聚合，读者找不到
2. **5 个系列没有索引页** — 包括 59 篇的后端系列和 33 篇的质量板块
3. **只有 3 篇项目案例** — 700+ 篇知识没有"做过"的证据支撑

在写任何新系列之前，先把这三个问题修掉，投入产出比最高。

---

## 阶段一：结构修复（预计 1-2 天）

### 1A. 修复 series 字段（~30 篇）

| 受影响系列 | 预估篇数 | 修复内容 |
|-----------|---------|---------|
| Shader Variant 治理 | 15+ | 补 `series: "Unity Shader Variant 治理"` |
| HybridCLR | 3 | case study 文章补 `series: "HybridCLR"` |
| 资产系统 | 5+ | 补 `series: "Unity 资产系统与序列化"` |
| 机型分档 | 5+ | 统一 `series` 字段，去掉 `primary_series` |
| 其余零散 | 2-3 | 逐篇确认归属 |

**验收标准**：`grep -r '^series:$' content/` 返回 0 结果。

### 1B. 补索引页（5 个系列）

| 系列 | 篇数 | 优先级 |
|------|------|--------|
| 游戏后端基础 | 59 | P0 — 全站第二大系列 |
| Code Quality | 33 | P0 — 质量板块入口 |
| 渲染系统分档设计 | 6 | P1 |
| 数据导向行业横向对比 | 4 | P2 |
| 数据导向实战案例 | 3 | P2 |

每个索引页遵循已有模式：
- 一句总判断
- "这篇要回答什么"
- 推荐阅读顺序
- 按主题分组
- 问题跳转入口

**验收标准**：每个系列的索引页能在 Hugo 中正确聚合该系列所有文章。

## 阶段二：项目案例（6 篇）

**为什么优先于新系列**：对于目标岗位（1M+），决策者看的是"做过什么"而非"知道什么"。
当前 3 篇案例无法支撑 700+ 篇知识的可信度。

| 编号 | 标题 | 证明什么 |
|------|------|---------|
| 01 | 复杂 Unity 项目接手方法论 — 前两周怎么诊断 | 诊断能力 |
| 02 | 资源/构建/热更新的一次生产事故复盘 | 事故处理能力 |
| 03 | 性能优化项目 — 从低帧率到达标的分层治理过程 | 性能治理能力 |
| 04 | 多团队工程治理 — 规则、流程、CI、回滚怎么落地 | 治理落地能力 |
| 05 | 从项目内脚本到跨项目工具平台的边界判断 | 平台化判断力 |
| 06 | 我的工程边界：能解决什么、需要协作什么、不做什么 | 自我认知与定位 |

**写作标准**：
- 必须有具体场景（不是抽象方法论）
- 必须有决策过程（为什么选 A 不选 B）
- 必须有结果（量化或可验证）
- 必须有反思（哪里做对了、哪里可以更好）

## 阶段三：游戏质量保障体系（8 篇，新系列）

**为什么排在技能系统前面**：质量是你说的核心视角，它能串联客户端、服务端、工具链、团队协作。
写完这 8 篇，后续所有系列都有一条主线可以挂靠。

| 编号 | 标题 | 覆盖角色 |
|------|------|---------|
| 01 | 游戏质量的问题空间：为什么"测了没 bug"不等于"质量好" | 全员 |
| 02 | 质量门禁体系：从提交到发布，每一关卡什么、怎么卡 | 程序 / QA |
| 03 | 配表质量：策划数据的校验、回归、版本控制与防崩链 | 策划 / 程序 |
| 04 | 资源质量：美术资产的自动化检查、预算验证与 CI 集成 | 美术 / TA |
| 05 | 逻辑质量：游戏逻辑的可测试性设计与回归策略 | 客户端 / 服务端 |
| 06 | 兼容性质量：多机型、多系统版本、多分辨率的验证矩阵 | 工具链 / QA |
| 07 | 线上质量：崩溃率、ANR、卡顿、内存泄漏的监控与告警 | 运维 / 运营 |
| 08 | 质量文化：从"测试找 bug"到"全员质量意识"的团队转变 | 团队管理 |

这 8 篇天然覆盖了策划、美术、程序、QA、运维、管理六个角色。
