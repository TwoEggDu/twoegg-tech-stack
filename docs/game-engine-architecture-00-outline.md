## 详细结构

### 1. 这篇要回答什么

- 开篇切口：
  - 先写读者常见记法：`渲染 / 物理 / 音频 / UI`
  - 再指出这套记法会直接漏掉：`编辑器`、`对象组织`、`运行时底座`、`构建发布`、`数据导向扩展`
- 要抛出的核心问题：
  - `如果按架构理解，而不是按功能背词，现代游戏引擎最稳定的分层方式是什么。`
- 这一节要完成的动作：
  - 给出全文唯一主问题
  - 解释为什么这个问题必须先于 02 / 03 / 07 / 05 / 06
  - 说明本文只搭地图，不深入某一层内部机制
- 可直接引用的证据锚点：
  - 证据卡 `1` 到 `6` 的文档入口都显示，Unity / Unreal 的官方资料并不是按单一运行时或功能列表组织
- 本节事实与判断分界：
  - `事实`：两边官方资料同时覆盖 editor/workflow、对象模型、运行时机制、渲染系统、构建发布、数据导向框架
  - `判断`：因此“先搭一张总地图”比“先列功能模块”更稳

### 2. 这一层负责什么

- 本节在总论里改写为：`这张总地图里，各层分别负责什么`
- 先给一段总述：
  - 总地图不是把所有概念并排摆放
  - 它要回答的是：`哪些问题属于同一层，哪些问题根本不是同一类问题`
- 六层的段落顺序建议如下：

#### 2.1 内容生产层

- 负责的问题：
  - 编辑器如何承载内容制作
  - 包、插件、模板、工作流怎样组织团队生产
  - 工具链如何反过来塑造引擎本体
- 本段必须压出的判断：
  - `现代引擎不是纯运行时库，它首先是内容生产系统。`
- 证据锚点：
  - 证据卡 `1`

#### 2.2 世界模型层

- 负责的问题：
  - 世界里有哪些对象
  - 这些对象怎样组合、更新、协作
  - 游戏框架怎样把“能存在于世界里的东西”组织起来
- 本段必须压出的判断：
  - `对象存在方式和协作方式本身就是一层，不能和渲染/物理平铺。`
- 证据锚点：
  - 证据卡 `2`

#### 2.3 运行时底座层

- 负责的问题：
  - 脚本后端
  - 对象基类、反射、GC
  - 执行模型与运行时支撑机制
- 本段必须压出的判断：
  - `GC、脚本 VM、反射、任务系统属于执行地基，而不是又一组业务模块。`
- 证据锚点：
  - 证据卡 `3`

#### 2.4 专业子系统层

- 负责的问题：
  - 渲染、物理、动画、音频、UI 这类重型系统如何在引擎里各自成立
  - 它们为什么像“引擎里的国中国”
- 本段必须压出的判断：
  - `这些系统不是功能按钮集合，而是带自己数据结构、调度规则和工具链的专业子系统。`
- 证据锚点：
  - 证据卡 `4`

#### 2.5 资产与发布层

- 负责的问题：
  - 资源导入、序列化、构建、打包、发布
  - 内容如何从编辑状态重新组织成可交付产品
- 本段必须压出的判断：
  - `Build / Cook / Package 不是最后一步外壳，它们属于引擎交付能力。`
- 证据锚点：
  - 证据卡 `5`

#### 2.6 数据导向扩展层

- 负责的问题：
  - 当对象规模、更新方式、表示层桥接出现新成本时，引擎如何长出数据导向区域
  - DOTS / Mass 为什么值得单独占位
- 本段必须压出的判断：
  - `DOTS / Mass 不是普通功能模块，而是对既有层级的一次结构性扩展。`
- 证据锚点：
  - 证据卡 `6`

#### 2.7 本节收口

- 用一小段把六层串成一句话：
  - `先生产内容，再组织世界，再提供运行时底座，再挂载专业子系统，再完成资产与发布，必要时再长出数据导向扩展。`
- 点明后续文章关系：
  - `02` 负责把世界模型层讲透
  - `03` 负责把运行时底座层讲透
  - `07` 负责把数据导向扩展层讲透
  - `05` / `06` 负责把交付链和平台边界讲透

### 4. Unity 怎么做

- 本节不是做 Unity 功能导览，而是按六层地图快速落地：

#### 4.1 内容生产层

- 可用材料：
  - Package Manager
  - Packages 文档
- 可落下的事实：
  - Unity 官方把 package 视为 editor tools、runtime tools、assets、templates 的正式交付单元
- 可落下的暂定判断：
  - `Unity 的内容生产层天然带有包化和工具化气质`

#### 4.2 世界模型层

- 可用材料：
  - GameObjects 文档
- 可落下的事实：
  - GameObject 是核心概念，行为依赖 Components
- 可落下的暂定判断：
  - `Unity 默认世界组织是 Scene / GameObject / Component 这套组件化对象模型`

#### 4.3 运行时底座层

- 可用材料：
  - IL2CPP Overview
- 可落下的事实：
  - IL2CPP 是 scripting backend，会把 IL 转成 C++ 再生成平台原生二进制
- 可落下的暂定判断：
  - `脚本后端与执行模型在 Unity 里属于底座，不是外接插件`

#### 4.4 专业子系统层

- 可用材料：
  - URP 文档
- 可落下的事实：
  - URP 是正式 render pipeline 包
- 可落下的暂定判断：
  - `渲染能力在 Unity 里是专业子系统，不代表整台引擎的全部`

#### 4.5 资产与发布层

- 可用材料：
  - Build Settings
  - IL2CPP Overview
- 可落下的事实：
  - Build settings、target platform、native binary 都在正式手册内
- 可落下的暂定判断：
  - `交付链在 Unity 里属于引擎本体的一部分`

#### 4.6 数据导向扩展层

- 可用材料：
  - Entities 文档
- 可落下的事实：
  - Entities 被官方定义为 ECS implementation
- 可落下的暂定判断：
  - `DOTS 在 Unity 里是单独成体系的扩展方向`

#### 4.7 本节收口

- 收成一句话：
  - `从官方文档结构看，Unity 不是单一运行时，而是一台把生产工具、对象模型、脚本底座、专业子系统、交付链和数据导向框架并列组织起来的引擎。`

### 5. Unreal 怎么做

- 本节同样按六层地图落地，不做 Unreal 功能导览：

#### 5.1 内容生产层

- 可用材料：
  - Unreal Engine Interface and Navigation
  - Unreal Engine for Unity Developers
- 可落下的事实：
  - Unreal 官方单独提供 Editor 界面与导航文档，并把 workflows 作为正式入口
- 可落下的暂定判断：
  - `Unreal 的内容生产层以编辑器工作流和系统导航为明确组成部分`

#### 5.2 世界模型层

- 可用材料：
  - Unreal Engine Terminology
  - GameFramework API
- 可落下的事实：
  - UObject、Actor、Pawn、Character、Game State 等术语和类型被作为基础世界组织能力给出
- 可落下的暂定判断：
  - `Unreal 默认把世界对象组织和 gameplay framework 绑定得更紧`

#### 5.3 运行时底座层

- 可用材料：
  - Reflection System
  - Terminology
- 可落下的事实：
  - UObject 关联 GC、metadata、serialization；反射系统同时影响 editor/runtime
- 可落下的暂定判断：
  - `反射、对象基类、GC 在 Unreal 里构成运行时底座的一部分`

#### 5.4 专业子系统层

- 可用材料：
  - Unreal Engine for Unity Developers 中的 rendering 入口
- 可落下的事实：
  - rendering 被单列成正式入口，并列于 game objects、workflows、writing code
- 可落下的暂定判断：
  - `渲染是重型子系统，但不是对 Unreal 的全部描述`

#### 5.5 资产与发布层

- 可用材料：
  - Packaging Your Project
  - Building Multi-Platform Projects in Unreal Engine
- 可落下的事实：
  - Build、Cook、Stage、Package 是正式 build operations
- 可落下的暂定判断：
  - `Unreal 的交付链是体系内能力，不是项目外脚本`

#### 5.6 数据导向扩展层

- 可用材料：
  - MassGameplay Overview
  - MassEntity API
- 可落下的事实：
  - MassEntity 被官方定义为 data-oriented calculations framework，MassGameplay 在其上继续扩展
- 可落下的暂定判断：
  - `Mass 在 Unreal 里也是单独成立的结构性扩展方向`

#### 5.7 本节收口

- 收成一句话：
  - `从官方文档结构看，Unreal 也不是“Actor + Renderer”的二元体，而是一台把编辑器、世界框架、运行时机制、专业子系统、交付链和数据导向框架并列组织起来的引擎。`

### 6. 这些差异为什么不是表面 API 差异

- 本节要完成的动作：
  - 把前两节的 Unity / Unreal 例子收回到“工程组织方式不同”
- 必须压出来的 3 个判断：
  - `差异一`：Unity 与 Unreal 的对象世界组织方式不同，因此不能只用同名功能去对照
  - `差异二`：两边都把 editor/runtime/build/data-oriented 放进正式能力面，说明引擎差异不是只看运行时 API
  - `差异三`：如果按层看，后续讨论 GameObject / Actor、GC、DOTS / Mass、Build / Cook / Package 才不会互相串台
- 本段结尾建议：
  - 用一句话把“分层视角”与“功能列表视角”做正面对比

### 7. 常见误解

- 误解 `1`：
  - `游戏引擎就是渲染 / 物理 / 音频 / UI 的功能打包`
  - 纠正方式：指出这会漏掉编辑器、对象模型、运行时底座、构建发布
- 误解 `2`：
  - `编辑器和构建链只是外围工具`
  - 纠正方式：回到官方文档入口，说明它们就是正式能力面
- 误解 `3`：
  - `GC、DOTS、Mass 都只是优化选项`
  - 纠正方式：强调它们分属不同层级，不能并列压平
- 误解 `4`：
  - `Unity 和 Unreal 的区别主要是 API 写法不同`
  - 纠正方式：指出世界模型、运行时机制、交付链组织方式都不同
- 误解 `5`：
  - `总论应该一次性把所有专题讲透`
  - 纠正方式：明确总论的任务是搭地图，不是替代后续文章
