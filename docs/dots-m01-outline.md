## 一句话中心判断

- `DOTS 不是“对象多就该上”的性能开关；它只适合那些高密度仿真、数据布局和调度方式决定了项目上限的问题空间，而一旦团队、边界和验证链不成立，引入它只会把复杂度前移。`

## 详细结构

### 1. 为什么“对象多”不是充分条件

- 开头先摆一个最典型的错误说法：
  - `项目里对象很多，所以该上 DOTS`
- 然后拆它为什么粗：
  - 对象数量只是表象
  - 真正决定是否值得上的是：
    - 是否是高密度同构仿真
    - 是否真的被数据布局和调度方式卡住
    - 是否会长期吃到 Burst / Jobs / ECS 的收益
- 这一节的任务：
  - 先打断“性能焦虑 -> 技术选型”的自动跳跃
  - 把文章重心从“多不多”转到“问题空间对不对”

### 2. DOTS 真正擅长的问题空间

- 这里要给正面判断标准，不要只讲口号
- 建议拆成 4 类问题：
  - 大量同构对象的数值更新
  - 需要明确数据布局和调度收益的仿真层
  - 逻辑与表示层可以切开的系统
  - 需要可重复性能验证的核心热路径
- 每类问题都要配“为什么适合”：
  - SoA / Chunk / Burst / Fixed Step 在这里解决什么
- 最后收成一句话：
  - DOTS 擅长的是“仿真层重、表示层可分离、热路径稳定”的问题空间

### 3. 团队、工具链和维护成本怎么算

- 这是本篇必须和纯技术教程区分开的地方
- 至少要写 3 个成本面：
  - 团队认知成本：谁来维护 ECS / Burst / NativeCollection 这一套
  - 工具链成本：调试、Profiler、构建、CI 是否跟得上
  - 维护期成本：包升级、代码边界、长期可读性
- 这一节要明确指出：
  - 即使技术上“能上”，工程上也可能“不该上”
- 可以落一个简单表格：
  - `收益面` vs `成本面`

### 4. 哪些项目根本不该上 DOTS

- 这一节是全文的价值锚点
- 至少给 4 类反例：
  - UI / 内容驱动型项目
  - 热路径不稳定、热点并不集中在仿真层的项目
  - 团队没有维护 ECS / Burst / Native 内存经验的项目
  - 没有验证链、准备靠“感觉更快”来推动选型的项目
- 每类反例都要说明：
  - 不是“永远不能上”
  - 而是当前条件下不该上
- 这里一定要写清 `Exit Condition`：
  - 如果不满足哪些条件，应该停止迁移，而不是继续硬推

### 5. 一张可执行的判断矩阵

- 把全文压成一张可操作的矩阵或决策树
- 建议维度：
  - 问题空间
  - 团队栈
  - Hybrid 边界清晰度
  - 验证链准备度
  - 维护期风险
- 输出目标：
  - 让读者读完后能立刻判断“现在先别上 / 可以试点 / 可以进入迁移设计”
