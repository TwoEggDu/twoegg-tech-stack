# ET-Pre-01｜详细提纲：线程、任务、协程、纤程到底不是一回事

## 文章主问题与边界

- 这篇只回答：`为什么“线程 / 任务 / 协程 / 纤程”不是同一层概念，以及这个区分为什么是理解 ET Fiber 的前提。`

- `ET 里的 Fiber 不是“更轻一点的线程”，也不是“语法糖协程”，而是一种由框架定义调度边界和串行语义的运行时执行单元。`

## 详细结构

### 1. 这篇要回答什么

- 开篇先给读者熟悉的混用现场：
  - “开个线程去跑异步”
  - “协程就是单线程任务”
  - “Fiber 就是线程池任务”
- 直接指出这些说法的问题：
  - 它们都在谈“执行”，但谈的是不同层级的执行对象
  - 一旦混层，后面看 ET 的 Fiber、NetClient Fiber、Actor 串行处理都会走偏
- 本节收束成一个明确问题：
  - `如果不先拆清这四个词，就不可能稳定理解 ET 的 Fiber 模型。`

### 2. 为什么这些词总被混掉

- 先给出一个总判断：
  - 大家会混，不是因为概念故意复杂，而是因为这四个概念都在影响“代码什么时候继续跑”
- 要拆出的三层原因：
  - 层 1：都和“并发/切换/等待”有关
  - 层 2：业务代码经常只看见“用了某个 API 之后程序又继续跑了”
  - 层 3：框架往往故意把底层调度细节包起来，进一步强化错觉
- 这一段的目标不是定义，而是先解释“为什么读者会误解”

### 3. 先把四个词放回各自层级

#### 3.1 线程是什么

- 操作系统级执行单位
- 有自己的调度、切换、阻塞、栈
- 代价大，能力强，可以并行跑在不同核上
- 不要在这里展开锁和内存序，只要说明线程是最重的一层

#### 3.2 任务是什么

- 运行时/框架层的工作描述或待完成单元
- 任务不等于线程，它通常要被调度到线程上
- `Task` 解决的是“完成与 continuation 表达”，不是天然解决“在哪个线程跑”

#### 3.3 协程是什么

- 更像一种“挂起和恢复控制流”的语义能力
- 协程不是线程，也未必并行
- 在 Unity 里 `IEnumerator` 协程更接近“挂到 PlayerLoop 某个阶段继续执行”

#### 3.4 纤程是什么

- 历史上 Fiber 可以指用户态调度的轻量执行单元
- 但框架里的 Fiber 往往是一种“框架定义的执行边界”，不必和操作系统或语言标准术语完全重合
- ET 的 Fiber 更重要的是：`一个有明确归属、明确调度方式、明确消息和串行语义的运行时单元`

#### 3.5 小结：四者不是并列替代关系

- 线程是承载层
- 任务是等待/完成表达层
- 协程是控制流挂起恢复层
- 纤程是框架给出的执行与隔离边界
- 它们会组合出现，而不是彼此替代

### 4. 一个更稳的最小模型

- 用一个最小示意图来写：
  - 业务希望“这个对象的逻辑顺序执行”
  - 框架希望“多个这样的逻辑单元被安排到不同调度器上”
  - 底层再把这些调度器映射到线程或线程池上
- 最小模型应该至少包含：
  - 业务对象
  - 执行边界
  - 调度器
  - 底层线程
- 本节结论：
  - `Fiber 不是为了发明新名词，而是为了把业务串行语义和底层线程能力隔开。`

### 5. 放回 Unity / 游戏框架现场

- 先写 Unity 项目的常见执行层：
  - 主线程
  - ThreadPool 任务
  - `IEnumerator` 协程
  - Job Worker
- 然后说明 ET 为什么还要加 Fiber 这一层：
  - 因为它要给客户端和服务端都提供“可组织的执行单元”
  - 它不想让业务逻辑直接围着线程写
  - 它希望把消息、定时器、场景角色和执行边界绑起来
- 收到一句：
  - `ET 的 Fiber 是框架架构层抽象，不是语言层或操作系统层原生概念。`

### 6. 一个坏理解通常怎么坏

- 坏法 1：把 Fiber 当成“线程别名”
- 坏法 2：把 `async` 当成“开线程”
- 坏法 3：把协程、任务、纤程都当成“异步”一个词糊过去
- 坏法 4：以为切到 Fiber 就自动获得并行性能

### 7. 工程边界与取舍

- 为什么框架愿意多造一层 Fiber 概念：
  - 降低业务层直接面对线程的复杂度
  - 把串行消息语义收拢进统一边界
  - 更方便绑定场景、对象、角色、消息和调度器
- 这层抽象的代价：
  - 学习成本变高
  - 名词和已有语言术语部分重叠
  - 不先建图就会误会

### 8. 这篇真正想留下来的结论

- 最短结论：
  - `先把线程、任务、协程、纤程拆开，后面读 ET 时才能知道 Fiber 解决的是“框架执行边界”，不是“替代线程的语法糖”。`
- 明确导向后续：
  - 继续读 `ET-Pre-02`
  - 正文对应 `ET-13`、`ET-19`

# ET-Pre-02｜详细提纲：async/await 到底在调度什么

## 文章主问题与边界

- 这篇只回答：`async/await 到底改变了什么，以及为什么理解 continuation 是看懂 ETTask 的前提。`
- 这篇不展开：`Fiber 与线程的概念区分`，这部分由 `ET-Pre-01` 提前完成

## 一句话中心判断

- `async/await 不是多线程机制，而是一套把“暂停点之后怎样继续跑”交给状态机和 continuation 协议处理的语言级约定。`

## 详细结构

### 1. 这篇要回答什么

- 开篇直接点常见误解：
  - “加了 async 就异步了”
  - “await 会切线程”
  - “只要不阻塞主线程就是 async 的功劳”
- 然后指出这些误解会如何影响 ET：
  - 看不懂 ETTask 为何要有自己的 builder/上下文
  - 看不懂消息处理链为什么还能保留串行语义
- 收成文章主问题：
  - `await 一个东西之后，程序到底是怎样恢复执行的？`

### 2. 为什么 async 总被误解成并行

- 先解释误解来源：
  - 代码写成顺序样子
  - UI 没卡住
  - 等待完成后自动继续执行
- 然后指出真正发生的不是“自动并行”，而是：
  - 当前方法在 await 点被拆开
  - 后半段被注册成 continuation
  - 何时恢复、在哪恢复，要看 awaiter 和上下文

### 3. 先把最小运行链讲清

#### 3.1 状态机在做什么

- `async` 方法会被编译器改写成状态机
- 状态机负责记录“当前走到哪一步”
- 这一步的重点不是 IL 细节，而是语义：`方法被拆成了多段`

#### 3.2 await 在做什么

- await 首先问：当前等待对象是否已完成
- 如果完成，直接继续
- 如果未完成，注册 continuation，然后返回给上层

#### 3.3 continuation 是什么

- continuation 不是线程
- continuation 是“之后要继续执行的那段逻辑”
- 它能被不同调度器、不同上下文恢复

#### 3.4 上下文恢复在做什么

- 为什么有些代码 await 之后还在主线程
- 为什么有些代码继续在线程池
- `SynchronizationContext` / 自定义 awaiter / 框架调度器在这里各自站位

#### 3.5 小结

- `async/await` 真正改变的是控制流表达方式和 continuation 协议
- 它不承诺自动开线程
- 它也不保证业务层天然安全

### 4. 一个更稳的最小实现骨架

- 建议用一个最小伪流程来写：
  - 调用 async 方法
  - 进入状态机 MoveNext
  - 碰到 await
  - 注册 continuation
  - 返回
  - 等待对象完成
  - continuation 触发，再次 MoveNext
- 这里要把“上下文恢复点”明确画出来
- 本节要形成一个读者脑中可复用的最小模型

### 5. 放回 Unity 和 ET 现场

- 先回到 Unity：
  - Unity 主线程 API 访问边界为什么让“默认 Task 语义”经常错位
  - 为什么 Unity 项目经常需要自定义调度和切回主线程机制
- 再桥接到 ET：
  - ETTask 不只是想“少一点 GC”
  - ET9 README 明说 ETTask 实现了传递上下文能力
  - 对 ET 来说，重要的不只是 await，而是 await 之后上下文还能保住什么
- 收束：
  - `ETTask` 的关键价值在于：它为框架自己的执行边界提供了 continuation 承接能力

### 6. 一个坏理解通常怎么坏

- 坏法 1：把 await 当成“异步线程切换”
- 坏法 2：以为 async 方法天然更高性能
- 坏法 3：把 continuation 当成“回调换个写法”就结束了
- 坏法 4：忽略上下文恢复，导致误判框架设计意图

### 7. 工程边界与取舍

- 为什么框架不总是直接用原生 Task：
  - 默认调度语义不一定适合框架
  - GC、上下文、时序、生命周期可能都不合适
  - 游戏框架往往更在意“恢复到谁的执行边界”
- 代价：
  - 自定义 task 语义会提升理解门槛
  - 一旦读者没有 continuation 概念，就很容易看不懂

### 8. 这篇真正想留下来的结论

- 最短结论：
  - `async/await 的本质不是多线程，而是 continuation 协议；看懂这一点，后面才能理解 ETTask 为什么要接管上下文与恢复边界。`
- 明确导向后续：
  - 继续读 `ET-Pre-03`
  - 正文对应 `ET-12`、`ET-13`、`ET-15`

## 详细结构

### 1. 这篇要回答什么

- 开篇先写读者最常见的朴素理解：
  - DLL 只是编译出来的东西
  - 反射只是偶尔拿来取类型名
  - 动态加载只是热更方案的一部分
- 然后指出这套理解在游戏框架里为什么太浅：
  - 很多框架把代码本身也作为一种“被组织、被扫描、被切分、被装配”的资产
- 收成主问题：
  - `为什么 ET 这样的框架，会把代码装配本身做成架构层能力？`

### 2. 为什么游戏框架总离不开程序集边界

- 要先讲清几个现实需求：
  - 模块边界
  - 类型扫描
  - 按层装配
  - 动态替换
  - 工具链前移约束
- 然后指出：
  - 一旦框架需要这些能力，代码就不再只是“源码文件集合”
  - 它会变成有边界、有依赖、有加载时机的模块单元

### 3. 先把几个关键概念放回原位

#### 3.1 程序集是什么

- 程序集是代码、元数据、依赖边界的打包单元
- 重点不是文件后缀，而是边界和身份

#### 3.2 反射在框架里解决什么

- 类型扫描
- 自动注册
- 根据特性做分发
- 框架不是“偷懒”，而是在把规则前移到元数据层

#### 3.3 动态加载在框架里解决什么

- 不同阶段装不同代码
- 允许替换一部分逻辑而不推翻全部运行时
- 给热更新、可插拔模块、运行时重装配提供可能

#### 3.4 代码生成为什么总和装配边界一起出现

- 一旦框架需要稳定规则和约束，手写样板会失控
- 代码生成和分析器本质上是在“装配之前”就帮你限制结构

### 4. 一个更稳的最小装配模型

- 建议画一个最小模型：
  - 运行时底座程序集
  - 模型层程序集
  - 表现层程序集
  - 可替换逻辑程序集
- 说明这几层可能分别承担：
  - 稳定底座
  - 共享数据结构
  - 客户端表现逻辑
  - 高频改动逻辑
- 本节结论：
  - `代码装配边界是框架的组织手段，不是单纯构建产物。`

### 5. 放回 ET 现场：为什么会有 Model / ModelView / Hotfix / HotfixView

- 这里不逐行拆源码，而是建立最小直觉：
  - ET 公开 `CodeLoader` 会加载 `ET.Model.dll`、`ET.ModelView.dll`、`ET.Hotfix.dll`、`ET.HotfixView.dll`
  - 这说明 ET 对“稳定层”和“可变层”、“数据层”和“表现层”做了显式切分
- 应该讲清的判断：
  - ET 不是把热更当附属功能，而是把代码装配路径纳入主流程
  - 这也解释了为什么它会高度依赖反射、CodeTypes、特性扫描和 SourceGenerator
- 本节最后要把读者导向：
  - `ET-08` 看 CodeLoader
  - `ET-28` 看分析器与生成器
  - `ET-30` 看 Reload 与热更

### 6. 一个坏理解通常怎么坏

- 坏法 1：把 DLL 只当“编译副产物”
- 坏法 2：把反射理解成“临时工具”，看不到它在框架层的组织作用
- 坏法 3：把热更理解成“晚点再讲的附属功能”
- 坏法 4：看见多段程序集就以为只是项目拆层，没有看到运行时装配意义

### 7. 工程边界与取舍

- 这套设计的收益：
  - 分层清晰
  - 可扫描、可替换、可约束
  - 更适合长线项目和框架化组织
- 这套设计的代价：
  - 学习门槛明显升高
  - 编译、装配、调试链条更长
  - 读者如果只熟悉单体 Unity 项目，会觉得 ET 很“绕”

### 8. 这篇真正想留下来的结论

- 最短结论：
  - `ET 把代码分段装配，是因为它把“代码边界”本身也当成了框架能力；不理解这一点，就很难真正看懂 CodeLoader、热更和分析器。`
