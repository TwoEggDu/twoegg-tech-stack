# 网站 SEO / AI 引用 / 个人 IP 执行计划

## 一、项目目标

这个网站的目标，不是单纯“被收录”，而是让搜索引擎和 AI 能稳定理解、引用并传播我的内容，从而帮助更多人认识我、知道我是谁、知道我擅长什么，逐步建立我的个人技术 IP。

更具体地说，这个网站需要同时完成三件事：

1. 让搜索引擎能够发现并收录我的页面。
2. 让 AI 能在相关问题下引用我的页面，并把我的内容当作可参考来源。
3. 让第一次进入网站的人，能快速理解“我是谁、我做什么、我为什么值得关注”。

这个网站最终应该稳定回答下面四个问题：

1. 你是谁？
2. 你擅长什么？
3. 你有哪些代表内容？
4. 为什么应该相信你的判断？

如果这四个问题表达清楚，那么搜索引擎、AI 和首次访问者，都会更容易把我的名字与我的技术方向建立长期关联。

## 二、网站定位

这个网站不应该只是“文章存档”或“作品列表”，而应该是：

- 一个公开的个人技术品牌入口
- 一个能被搜索引擎和 AI 理解的专业身份页面集合
- 一个围绕特定主题持续积累信任的内容资产

当前站点地址：

- 站点入口：`https://twoeggdu.github.io/twoegg-tech-stack/`
- Sitemap：`https://twoeggdu.github.io/twoegg-tech-stack/sitemap.xml`

## 三、核心原则

### 1. 优先争取“被引用”，而不是先追求“被训练”

对个人 IP 来说，更有价值的是：

- 在 Google / Bing 搜索结果中出现
- 在 ChatGPT / Claude / 其他 AI 搜索回答中被引用
- 让用户能从结果页回到我的站点

相比之下，“是否进入训练语料”更黑箱，也更难验证，不应作为当前阶段的主要目标。

### 2. 优先让网站“可理解”

不是只追求页面数量，而是要让站点结构、首页文案、About 页面、文章摘要、内部链接都足够清晰，让机器和人都能快速形成正确认知。

### 3. 优先建设“主题权威感”

与其分散输出，不如围绕我的核心能力方向形成稳定主题簇，例如：

- 游戏客户端工程
- Unity / 引擎链路
- 资源 / 构建 / 热更新
- Shader Variant
- HybridCLR
- 工程质量与质量护栏

## 四、整体执行路径

整个项目分为两个优先级最高的阶段：

1. 第一阶段：站内改造
2. 第二阶段：平台提交

阶段顺序不能反过来。先把站内表达整理清楚，再去提交给 Google / Bing，效果更稳。

## 五、第一阶段：站内改造

### 阶段目标

让网站达到三个标准：

- 可被发现
- 可被理解
- 可被引用

### 重点任务

#### A. 补齐基础 SEO 元信息

需要补齐或检查：

- `canonical`
- `meta description`
- `robots`
- `Open Graph`
- `Twitter Card`
- 页面标题生成规则

目标是让首页、栏目页、文章页都有稳定、清晰、可复用的 head 信息。

#### B. 强化首页与 About 页的“身份表达”

首页和 About 页不能只承担导航功能，还必须承担身份说明功能。

需要做到：

- 首页第一屏就能回答“我是谁、我做什么、我擅长什么”
- About 页明确说明个人背景、长期关注方向、代表经验、推荐阅读入口
- 把“你为什么值得被关注”说清楚

#### C. 补齐结构化信息

建议逐步补齐：

- 首页：`WebSite`
- About 页：`ProfilePage` + `Person`
- 文章页：`Article`
- 导航层级：`BreadcrumbList`

目标是让搜索引擎和 AI 能更稳定识别这个站是“一个明确的个人专业站点”，而不是普通博客集合。

#### D. 强化内部链接结构

文章不能是孤页。

需要补：

- 面包屑
- 返回栏目入口
- 同主题继续阅读
- 系列索引页与单篇文章双向链接
- 首页固定代表作入口

目标是让机器和人都能看出：这些文章是围绕若干主线主题组织起来的。

#### E. 补齐文章摘要

所有重点文章都应该有明确 `description`。

摘要要求：

- 直接说明这篇文章解决什么问题
- 尽量包含主题关键词
- 不写空话，不写泛泛介绍
- 让搜索结果和 AI 摘要可以直接抽走关键结论

#### F. 明确 AI crawler 策略

如果目标是“被 AI 搜索引用”，那么需要优先保证搜索型 crawler 可以访问站点内容。

后续需要确认的方向：

- 是否允许 OpenAI 的搜索型抓取
- 是否允许 Anthropic 的搜索型抓取
- 是否区分“搜索引用”和“训练抓取”

现阶段建议优先支持“可引用”，训练抓取策略后续单独决定。

### 第一阶段完成标准

满足以下条件时，可视为阶段完成：

1. 首页、About、文章页都有完整 title / description / canonical。
2. 首页和 About 页能明确说明我的身份和方向。
3. 文章页已经具备面包屑和相关文章入口。
4. 至少 5 篇代表内容被固定为站内主入口。
5. 文章摘要覆盖核心内容。

## 六、第二阶段：平台提交

### 阶段目标

让 Google 和 Bing 尽快开始抓取、识别、建立索引，并逐步积累搜索可见性。

### 重点任务

#### A. Google Search Console

需要执行：

1. 新建 `URL-prefix` 属性  
   使用地址：`https://twoeggdu.github.io/twoegg-tech-stack/`

2. 完成站点验证  
   建议优先使用 HTML meta tag 验证方式

3. 提交 sitemap  
   地址：`https://twoeggdu.github.io/twoegg-tech-stack/sitemap.xml`

4. 手动请求关键页面建立索引  
   重点先提交：
   - 首页
   - About 页
   - 1 篇项目代表作
   - 1 篇引擎代表作
   - 1 篇代码质量代表作

5. 关注后续报告  
   - Pages / Indexing
   - Sitemaps
   - Performance

#### B. Bing Webmaster Tools

需要执行：

1. 创建或导入站点  
   优先考虑从 Google Search Console 导入

2. 验证站点归属

3. 提交 sitemap  
   地址同上

4. 请求关键页面抓取 / 提交

5. 查看索引覆盖与 sitemap 状态

#### C. 关键页面优先策略

不要一开始就试图推进全站。

优先推动这 5 个页面：

1. 首页
2. About 页
3. 1 篇最强项目案例
4. 1 篇最强引擎链路文章
5. 1 篇最强代码质量文章

这 5 个页面代表的是“你是谁”和“你最强的能力表达”，优先级高于一般内容页。

### 第二阶段完成标准

满足以下条件时，可视为阶段完成：

1. Google Search Console 已验证成功。
2. Bing Webmaster Tools 已验证成功。
3. Sitemap 已提交且无明显错误。
4. 关键页面已发起索引请求。
5. 后台已开始出现抓取或索引数据。

## 九、风险与注意事项

### 1. GitHub Pages 子路径站点的限制

当前站点运行在 GitHub Pages 的子路径下，而不是独立域名。

这意味着：

- 某些站点级信号会弱于独立域名
- `robots.txt` 的控制能力需要结合根主机实际情况理解
- 长期来看，若个人 IP 是核心目标，后续可评估迁移到独立域名

### 2. 收录不是即时结果

即使 sitemap 提交成功，也不代表立刻收录。

平台从发现、抓取、理解到建立索引，需要一定时间，尤其是新站、新路径或主题密度较弱的站点。

### 3. “可被抓取”不等于“会被引用”

真正决定 AI 和搜索引擎是否愿意引用的，除了可抓取性，还包括：

- 页面主题是否清晰
- 内容是否有明确结论
- 身份是否明确
- 内部结构是否清晰
- 是否具备可信度信号
