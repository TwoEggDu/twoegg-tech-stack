# 游戏引擎架构地图自动化约束

## 这份文件的作用

这份文件是给自动化任务读取的硬约束。

它解决的不是“这组文章写什么”，而是：

- 自动化每次运行时允许做什么
- 不允许做什么
- 什么情况下必须停下并报告阻塞
- 什么样的内容算“没有跑偏”

如果这份文件和其他规划文件冲突，以这份文件为准。

## 自动化的唯一目标

自动化任务只服务一个系列：

`游戏引擎架构地图`

它的工作不是泛化地写游戏引擎文章，也不是泛化地写 Unity / Unreal 内容。

它每次只允许推进这个系列中的一个明确里程碑。

## 每次运行前必须读取的文件

自动化每次运行时，必须按下面顺序读取：

1. `docs/game-engine-architecture-automation-constraints.md`
2. `docs/game-engine-architecture-series-execution-plan.md`
3. `docs/game-engine-architecture-series-plan.md`
4. `docs/engine-source-roots.md`，如果存在
5. `docs/game-engine-architecture-evidence-map.md`，如果存在

如果前 3 个文件任意一个不存在，不允许起草正文，只能报告阻塞。

## 每次运行只允许做一个“小而完整的增量”

每次运行只能完成下面 4 类动作中的一种：

1. 为下一篇文章建立详细提纲
2. 为下一篇文章补源码证据卡或证据映射
3. 为下一篇文章新建正文草稿
4. 在已有草稿上补一个完整章节，并同步补短观点

每次运行不要同时推进两篇文章。
每次运行不要同时做“证据整理 + 全文写完 + 全系列索引”。

## 文章顺序约束

默认只允许按下面顺序推进：

1. `00 总论`
2. `02 世界模型层`
3. `03 运行时底座层`
4. `07 数据导向扩展层`
5. `01 内容生产层`
6. `04 专业子系统层`
7. `05 资产与发布层`
8. `06 平台抽象层`
9. `08 总收束`

如果已有文件表明某篇已经完成，才允许进入下一篇。

## 推进阶段约束

自动化默认分成两个阶段工作：

### 第一阶段：先完成整套首稿

目标是：

- 先把计划中的每篇文章都推进到 `可读首稿`
- 不追求第一次就写成最终发布版
- 优先保证整套系列完整成形

在这一阶段，自动化应优先做：

1. 锁定问题边界
2. 建立证据卡
3. 建提纲
4. 补出首稿正文

### 第二阶段：再做逐篇细化

只有当第一阶段的目标已经完成后，才进入这一阶段。

这一阶段可以做：

1. 补源码级证据
2. 收紧结构
3. 改写表达
4. 增加对照表、短观点和系列内链
5. 提升从“可读首稿”到“可发布稿”的质量

## 不设总运行次数上限

自动化不需要预设总运行次数上限。

它可以持续运行，直到：

1. 系列所有计划文章都已有首稿
2. 或者进入明确阻塞状态

但是“不设上限”不等于“允许无目标重复”。

如果没有新的明确增量可做，自动化必须报告：

- 当前已完成到哪一步
- 剩余工作是什么
- 为什么这次没有继续改动

## 进度账本约束

为了支持长时间持续运行，自动化应维护一个进度账本文件：

- `docs/game-engine-architecture-progress.md`

如果这个文件不存在，自动化可以创建它。

这个进度账本至少要记录：

- 每篇文章的状态：`未开始 / 提纲中 / 首稿中 / 首稿完成 / 细化中 / 完成`
- 当前推进顺序
- 最近一次实际改动
- 下一次最该做什么

自动化每次运行前，应先读取这个文件；每次运行后，应更新这个文件。

如果进度账本显示当前文章已经 `完成`，才允许进入下一篇。

## 每篇文章只能回答一个主问题

这是最重要的约束。

每篇文章必须先写出一句话：

`这篇只回答 X，不展开 Y 和 Z。`

如果自动化无法把当前文章压成一个明确问题，就不允许写正文。

禁止在同一篇里同时大篇幅展开：

- 宏观分层
- 运行时 GC
- DOTS / Mass
- 渲染细节
- 平台适配
- 资源构建链

这些问题可以互相引用，但不能混写成一篇。

## 禁止跑偏的硬规则

自动化不得把文章写成下面任何一种形态：

1. `教程`
2. `功能百科`
3. `产品优劣比较`
4. `API 罗列`
5. `没有证据支撑的观点散文`

更具体地说，禁止出现下面这些倾向：

- 教用户怎么点按钮
- 教用户怎么创建项目、Actor、GameObject、Package
- 罗列 Unity 有什么功能、Unreal 有什么功能
- 讨论谁更强、谁更先进、谁更适合独立开发
- 用模糊印象替代源码或官方资料
- 把不同层级的东西并列

## 证据优先级约束

自动化必须优先使用本地源码证据，再使用官方文档证据。

证据优先级如下：

1. 本地 `Unity` / `Unreal` 源码
2. 官方文档
3. 已有系列文档
4. 工程推断

如果一个重要结论已经可以从本地源码直接支撑，不要只用概念描述代替。

## 事实与判断必须分开写

正文、提纲、证据卡里都必须明确区分：

- `源码里能直接看到什么`
- `官方资料明确写了什么`
- `基于这些事实，我的工程判断是什么`

禁止把“事实”和“判断”混写成同一句模糊表达。

例如：

- 可以写：`从源码结构可以直接看到 X，因此我判断它更接近 Y。`
- 不要写：`它显然就是 Y。`

## 没有证据卡，不允许输出强结论

如果当前文章涉及下面这些容易失真的主题：

- Unity / Unreal 世界模型差异
- GC 所在层级
- DOTS / Mass 所在层级
- Build / Cook / Package 的架构位置
- 平台抽象边界

那么在没有证据卡之前，不允许写出强结论式正文。

所谓证据卡，最少要包含：

- 主题名
- Unity 对应源码位置或官方文档入口
- Unreal 对应源码位置或官方文档入口
- 可直接证明的事实
- 基于事实得到的暂定判断

## 源码路径缺失时的行为

如果 `docs/engine-source-roots.md` 不存在，或者存在但没有可用路径：

- 不要假装自己已经基于源码验证过
- 不要输出“源码显示”这类表达
- 只能先整理官方证据
- 同时在 `docs/` 下留下一个阻塞说明

## 每篇文章的固定结构

自动化起草正文时，必须尽量使用下面结构：

1. 这篇要回答什么
2. 这一层负责什么
3. 这一层不负责什么
4. Unity 怎么做
5. Unreal 怎么做
6. 这些差异为什么不是表面 API 差异
7. 常见误解
8. 我的结论

如果当前运行只完成提纲，也要按这个结构搭骨架。

## 文件输出约束

自动化只能写到下面两个区域：

- `docs/`：提纲、证据卡、阻塞说明、短观点草稿
- `content/engine-notes/`：可发布正文草稿

不要把规划、证据和正文混在同一个文件里。

## 命名约束

与这个系列直接相关的新文件，命名应尽量包含：

- `game-engine-architecture`

短观点文件建议放到 `docs/`，并明确绑定某一篇文章。

## 每次运行结束时必须输出的内容

自动化每次运行后，必须在总结里明确写出：

1. 这次推进的是哪一篇
2. 这次做的是哪一种增量
3. 新增或更新了哪些文件
4. 哪些结论是事实，哪些还是推断
5. 下一次最应该做什么

## 什么情况必须停止推进

遇到下面任何一种情况，自动化必须停止正文写作并报告：

1. 当前文章的问题边界压不清
2. 本地源码路径缺失，但文章又需要源码级判断
3. 找不到足够证据支撑关键论点
4. 当前草稿开始滑向教程、百科或产品比较
5. 当前运行已经跨到第二篇文章

## 判断“没有跑偏”的标准

只有同时满足下面 5 条，才算没有跑偏：

1. 只推进了系列中的一篇文章
2. 这篇文章只回答了一个主问题
3. 关键结论有证据支撑
4. 没有写成教程、百科或产品比较
5. 输出能自然挂回整个系列地图

## 最后压成一句话

这套自动化最重要的约束只有一句：

`每次只基于证据推进一篇文章的一个明确问题，不做教程、不做百科、不做比较、不写没有证据的强结论。`
