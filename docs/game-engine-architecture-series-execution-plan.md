# 游戏引擎架构地图系列执行计划

## 这份计划解决什么

上一版规划已经回答了“这组文章写什么”。

这份执行计划专门回答 4 件更具体的事：

1. 先发哪几篇，后发哪几篇
2. 每篇文章的核心论点是什么
3. 每篇需要什么例子和证据
4. 6 周内怎么把这个系列稳定跑起来

## 系列名称

建议主系列名固定为：

`游戏引擎架构地图`

这个名字的好处是：

- 比“游戏引擎基础”更高阶
- 比“Unity vs Unreal”更稳，不会变成产品比较
- 能自然挂接你现有的 `游戏图形系统` 和 `数据导向运行时`

## 系列目标

这组文章最终要让读者拿到 3 个结果：

1. 知道现代游戏引擎可以按哪些大层来理解
2. 知道 Unity 和 Unreal 在这些层上的差异，不只是 API 风格差异
3. 知道 GC、DOTS、Mass、Build、Cook、RHI 这类词分别站在哪

## 内容形态

建议采用和你现有内容一致的组合：

- `8 篇中长文`

## 8 篇中长文的具体计划

### 00｜总论：现代游戏引擎到底该怎么分层

建议题目：

`游戏引擎架构地图 00｜总论：现代游戏引擎到底该怎么分层`

要回答的问题：

`如果不按零散功能记忆，而按架构理解，现代游戏引擎最稳定的分层方式是什么。`

核心论点：

`现代游戏引擎至少可以先分成内容生产层、世界模型层、运行时底座层、专业子系统层、资产与发布层、数据导向扩展层。`

必须讲的段落：

- 为什么“渲染 / 物理 / 音频 / UI”这种教材式分类不够
- 为什么编辑器和构建系统也属于引擎本体
- 为什么 GC 和 DOTS 不在一个层级
- 为什么 Unity 和 Unreal 的差异应该按层来看

必须放的例子：

- Unity 的 Package / Editor / GameObject / IL2CPP / URP / Build Pipeline
- Unreal 的 Editor / Actor / UObject / RHI / Cook / Package

建议字数：

`3000 - 4000`

可衍生短观点：

- 很多人把游戏引擎看成功能集合，但它本质上是分层系统
- 编辑器和构建链不是外围工具，而是引擎本体的一部分

### 01｜内容生产层：为什么游戏引擎首先是一套生产工具

建议题目：

`游戏引擎架构地图 01｜为什么游戏引擎首先是一套生产工具`

要回答的问题：

`为什么引擎不是单纯运行时库，而是一套服务内容生产的编辑器与工作流系统。`

核心论点：

`没有内容生产层，就没有现代游戏引擎；只有运行时库，不足以支撑大型内容团队。`

必须讲的段落：

- 编辑器工作流为什么会反过来塑造底层设计
- Prefab / Blueprint 为什么不是“方便功能”，而是生产组织方式
- Package / Plugin 为什么影响引擎扩展边界

Unity 例子：

- Inspector
- Scene View
- Prefab
- Package Manager

Unreal 例子：

- Content Browser
- Blueprint Editor
- Plugin
- Details 面板

建议字数：

`2500 - 3500`

可衍生短观点：

- 大型引擎最值钱的部分，往往不是运行时而是生产体系
- Prefab 和 Blueprint 不只是工具特性，而是团队协作边界

### 02｜世界模型层：Unity 的 GameObject 和 Unreal 的 Actor 到底差在哪

建议题目：

`游戏引擎架构地图 02｜Unity 的 GameObject 和 Unreal 的 Actor，到底差在哪`

要回答的问题：

`两台引擎默认如何组织“世界里的对象”，以及这种差异为什么会影响后续架构。`

核心论点：

`Unity 默认在强调组件化对象组织，Unreal 默认在强调 World/Actor/Gameplay Framework 的世界框架。`

必须讲的段落：

- Scene / World 的区别
- Component 在两边分别扮演什么角色
- MonoBehaviour 和 Actor 生命周期为什么不只是接口差异
- Unreal 为什么要多出 Pawn / Controller / GameMode / GameState

必须放的例子：

- Unity：Scene、GameObject、Transform、MonoBehaviour
- Unreal：World、Level、Actor、ActorComponent、Pawn、Controller

建议字数：

`3500 - 4500`

可衍生短观点：

- Unity 的对象模型和 Unreal 的对象模型，看起来像同类，实际上不是同一种世界观
- Actor 不只是 GameObject 的换皮，它带的是一整套 gameplay framework

### 03｜运行时底座层：脚本、反射、GC、任务系统到底站在哪

建议题目：

`游戏引擎架构地图 03｜脚本、反射、GC、任务系统，到底站在引擎的哪一层`

要回答的问题：

`什么是引擎运行时底座，为什么 GC、脚本 VM、任务系统不应该被当成功能模块平铺。`

核心论点：

`GC、脚本 VM、反射、任务系统都属于运行时底座，因为它们在支撑整台引擎的执行模型。`

必须讲的段落：

- Unity 的 C#、Mono、IL2CPP、PlayerLoop、GC、Job System、Burst
- Unreal 的 C++ Runtime、UObject、Reflection、GC、Blueprint VM、Task Graph
- 为什么 GC 是运行时机制，不是独立业务模块
- 为什么 Jobs / Task Graph 是执行地基

建议字数：

`3500 - 4500`

可衍生短观点：

- GC 和 DOTS 不在一个层级，前者是运行时机制，后者是架构路线
- 很多人把任务系统看成优化工具，但它其实是执行底座

### 04｜专业子系统层：渲染、物理、动画、音频、UI 为什么都像半台小引擎

建议题目：

`游戏引擎架构地图 04｜渲染、物理、动画、音频、UI，为什么都像半台小引擎`

要回答的问题：

`为什么这些看似并列的功能模块，实际上各自都像“引擎里的国中国”。`

核心论点：

`专业子系统之所以重，是因为它们不只是功能点，而是带自己数据结构、调度规则和工具链的子引擎。`

必须讲的段落：

- 为什么 Renderer 不等于“画图接口”
- 为什么 Animation 不只是播一段动作
- 为什么 UI 通常自带事件系统、布局系统和资源体系

建议字数：

`3000 - 4000`

可衍生短观点：

- 引擎里的专业模块，真正贵的从来不是 API 数量，而是独立系统边界
- 物理、动画、UI 往往不是功能点，而是半台小引擎

### 05｜资产与发布层：资源导入、Cook、Build、Package 为什么也是引擎本体

建议题目：

`游戏引擎架构地图 05｜资源导入、Cook、Build、Package，为什么也是引擎本体`

要回答的问题：

`为什么资源导入和发布流程不是外围工具，而是引擎架构的一部分。`

核心论点：

`现代引擎真正把内容变成可运行产品，必须经过资产与发布层的再组织。`

必须讲的段落：

- Unity 的 Asset Import、Serialization、Addressables、Build Pipeline
- Unreal 的 Asset Registry、Asset Manager、Cook、Package、UBT
- 为什么运行时能力最终都要重新经过构建链
- 为什么“资源系统”本质上是交付系统的一部分

建议字数：

`3000 - 4000`

可衍生短观点：

- 资源导入规则写在群公告里，不叫工程化
- Build / Cook / Package 不只是发布步骤，而是引擎对内容的二次组织

### 06｜平台抽象层：跨平台引擎到底在抽象什么

建议题目：

`游戏引擎架构地图 06｜跨平台引擎到底在抽象什么`

要回答的问题：

`Unity 和 Unreal 所谓跨平台，抽象掉了什么，又保留了什么。`

核心论点：

`跨平台不是消灭差异，而是在统一抽象上管理不可避免的平台差异。`

必须讲的段落：

- Unity 的 graphics backend、Build Target、platform split
- Unreal 的 RHI、Target Platform、Platform Abstraction
- 为什么同一套引擎不可能让所有平台表现完全一致
- 为什么跨平台最终会回到 feature detection、fallback、quality tier

建议字数：

`3000 - 4000`

可衍生短观点：

- 跨平台引擎抽象的不是差异消失，而是差异管理
- 真正成熟的跨平台，不是“一次开发，到处完全一样”

### 07｜数据导向扩展层：DOTS 和 Mass 应该放在整张地图的哪里

建议题目：

`游戏引擎架构地图 07｜为什么 DOTS 和 Mass 不能只算“一个模块”`

要回答的问题：

`DOTS 和 Mass 在整张引擎架构图上到底站哪，为什么它们不是普通功能模块。`

核心论点：

`DOTS 和 Mass 本质上是在重写或扩展世界模型层与运行时底座层，而不是增加一项业务功能。`

必须讲的段落：

- 为什么 DOTS 不只是 package 列表
- 为什么 Mass 不是简单插件
- 为什么两者都在回答“高规模对象和执行模型怎么重组”
- 它们与原有 GameObject / Actor 世界是什么关系

建议字数：

`3500 - 4500`

可衍生短观点：

- DOTS 不是传统模块，Mass 也不是普通插件
- 数据导向不是多一套 API，而是换一套成本模型

### 08｜总收束：Unity 和 Unreal 到底是两种什么气质的引擎

建议题目：

`游戏引擎架构地图 08｜Unity 和 Unreal，到底是两种什么气质的引擎`

要回答的问题：

`把所有分层收回之后，Unity 和 Unreal 各自最稳定的架构气质是什么。`

核心论点：

- `Unity 更像组件化、包化、通用工作流导向的引擎`
- `Unreal 更像 World/Actor 框架、重编辑器、模块化系统导向的引擎`

必须讲的段落：

- 不是做优劣比较
- 只讲架构气质和历史路径
- 解释为什么相似功能会落在不同系统边界里

建议字数：

`2500 - 3500`

可衍生短观点：

- 两台引擎的差异，不只是“谁功能更强”，而是谁把复杂度放在了哪里
- 同样是现代引擎，Unity 和 Unreal 的默认工程组织方式并不一样

## 不要写成什么样

这组文章最容易滑向 3 种错误方向：

### 错误一：写成功能百科

表现：

- 罗列 Unity 有什么
- 罗列 Unreal 有什么
- 但没有讲每层的边界和组织方式

### 错误二：写成产品优劣对比

表现：

- 谁更强
- 谁更先进
- 谁更适合独立开发

这不是这组文章的任务。

### 错误三：写成教程

表现：

- 教你怎么点按钮
- 教你怎么创建 Actor
- 教你怎么装 package

这也不是这组文章的任务。
