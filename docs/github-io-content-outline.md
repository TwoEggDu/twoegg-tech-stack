# GitHub.io 内容规划草稿

## 目的

先把站点想写的东西落下来，不追求一次定稿。
这一版先回答三个问题：

1. 这个站主要写什么
2. 这些内容怎么分组才不散
3. 后面优先写哪些系列

## 当前想写的内容

### 1. 游戏引擎架构

- Unity 架构剖析
- Unreal 架构剖析

### 3. 工程实践

- HybridCLR 原理解析
- 质量管理体系搭建

- 做过什么事情，解决过什么问题
- 一些有代表性的复盘

## 推荐的站点主线

目前最自然的整理方式是三条主线：

### A. Engine Architecture

核心问题：

- 一个游戏引擎由哪些层组成
- 核心系统之间怎么协作
- 同一个架构问题在 Unity / Unreal / Piccolo 中怎么落地

适合写成系列：

- 引擎总览系列
- 核心子系统系列
- 跨引擎对比系列

### B. Computer Science for Engine Developers

核心问题：

- 底层基础知识怎样映射到引擎设计
- 为什么理解操作系统、编译原理、数据结构，会影响工程判断

适合写成系列：

- 操作系统到引擎运行时
- 编译原理到脚本系统 / 反射 / 热更新
- 数据结构与算法到资源管理 / 调度 / 渲染数据组织
- 设计模式到引擎模块组织

### C. Engineering Practice

核心问题：

- 真正做项目时，架构分析如何落到工程实现
- 自己做过的东西到底体现在哪些能力上
- 怎么把质量要求沉淀成可执行的流程、规范和检查机制

适合写成系列：

- HybridCLR 原理解析系列
- 质量管理体系搭建系列
- 项目复盘系列
- 我做过什么系列

## 推荐的导航结构

顶层导航先尽量少，不要一开始分太细。

- Home
- Series
- Projects
- About

说明：

- Home：一句话定位 + 代表文章 + 最近更新
- Series：按专题组织文章
- Projects：放项目经历、工程实践、代表性成果

## 首页应该传达的感觉

不是“我会很多技术”，而是：

- 我长期关注游戏引擎架构
- 我会把底层基础知识和真实工程问题连接起来
- 我做过实际项目，也能把项目经验抽象成方法
- 我也关心如何把质量管理做成体系，而不是靠个人经验硬撑

可以先按这个方向去写首页定位：

“这里主要记录游戏引擎架构、底层系统和工程实践。内容会围绕 Unity、Unreal、Piccolo、HybridCLR，以及操作系统、编译原理、算法等基础知识在真实工程中的落地展开。”

其中也会包含质量管理体系、流程治理和交付稳定性相关的总结。

## 可直接展开的文章方向

### 引擎方向

- Unity 引擎架构可以从哪几个层面理解
- Unreal 和 Unity 的架构差异，背后反映了什么设计取舍
- Piccolo 适合拿来讲哪些“引擎基础件”
- 一个组件系统在不同引擎里会怎么长出来
- 渲染管线为什么是理解引擎边界的关键入口

### 基础方向

- 为什么做引擎的人需要操作系统视角
- 编译原理知识会怎样影响对脚本系统和热更新的理解
- 数据结构并不是刷题知识，而是工程结构选择
- 设计模式真正有用的时候，往往是在大型系统边界开始混乱之后

### 实践方向

- HybridCLR 解决了什么问题，它的代价是什么
- 质量管理为什么不能只靠测试同学兜底
- 一个项目里的质量体系，应该如何拆成规范、流程、工具和门禁
- 怎样把“线上质量要求”前移到研发和提测阶段
- 我做过的东西里，哪些最能代表我的技术判断
- 一个技术方案从“能跑”到“能长期维护”，中间到底差了什么

## 关于“我做过什么”

这个部分不要只写成经历罗列。
更好的写法是：

- 做过什么系统
- 搭过什么质量体系
- 解决过什么复杂问题
- 当时的约束是什么
- 为什么这么做
- 最后沉淀出了什么判断

这样它既是经历，也是能力证明。
