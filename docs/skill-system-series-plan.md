# 技能系统深度系列规划

## 系列定位

这组文章不写成 `做一个火球术教程`，也不写成 `Unity 技能框架 API 速查`。

它真正要解决的问题是：

`把技能系统拆成一条稳定的执行链，讲清楚输入意图怎样变成施法请求，怎样经过校验、目标选择、命中、效果执行、Buff 规则，再延伸到表现、AI、多人同步、工具链和测试。`

一句话说，这个系列的重点不是“某个技能怎么写”，而是：

`技能系统作为一条完整执行链，到底该怎样分层、怎样定边界、怎样长期维护。`

## 为什么值得单独做成系列

如果只把技能系统当成零散文章去写，后面很快会遇到三个问题：

- `Skill / Effect / Buff / Attribute / Tag / Cooldown / Resource` 会不断混写，越写越像一个大对象或一组大 if
- 输入、表现、AI、网络会被默认吞进“技能逻辑”，最后失去系统边界
- 工程后期真正会爆炸的问题，往往不在“能不能放出一个技能”，而在于同步、工具、回归和跨框架映射

所以更稳的做法不是继续补零散专题，而是先固定一条上位主线：

`问题空间 -> 概念边界 -> 数据骨架 -> 生命周期 -> 执行主链 -> 世界变化 -> 跨系统边界 -> 工程化落地 -> 框架映射`

## 目标读者

- 已经做过 Unity 项目，但脑子里还没有“技能系统完整执行链”稳定地图的客户端程序
- 技能逻辑已经写到一定规模，开始被边界、复用、多人、工具和回归问题反复卡住的人
- 想把“会堆功能”升级成“能设计一套长期可维护技能系统”的工程实践者

## 系列边界

### 属于这个系列的内容

- 技能系统到底在解决什么问题
- 技能、效果、Buff、属性、标签、冷却、资源等概念边界
- 定义层、运行时层、上下文层和生命周期骨架
- 输入请求、校验约束、目标选择、命中结算这条执行主链
- Effect 与 Buff 的系统拆分
- 与表现系统、AI、多人同步的交点边界
- 技能编辑器、配置校验、调试视图、回归测试等工程化问题
- Unity 自研技能系统与 Unreal GAS 的思想映射

### 不属于这个系列的内容

- 完整 AI 系列
- 完整网络同步系列
- 完整动画系统系列
- 完整 UI 输入系统系列
- 具体引擎插件、包管理器或商业框架的使用说明书

这组文章只覆盖这些主题和技能系统的交点，不吞掉它们各自的完整体系。

## 上游前置与配套文章

这组文章虽然已经能独立阅读，但更适合挂在下面这些公共底座之上：

- `Command` 模式：理解“施法请求”为什么应该对象化
- `State / FSM / 行为树`：理解生命周期与 AI 调度为什么不能混成一层
- `OCP`：理解为什么技能扩展点不该继续堆回主系统
- `拓扑排序 / 依赖图`：理解技能依赖、配置校验和工具图为什么是同类问题
- `从玩家输入到屏幕画面`：理解输入、运行时和表现链如何接起来

这些前置不属于技能系统系列编号本身，但会决定这组文章的理解速度。

## 系列核心主线

这组文章的主线不按“技能类型”拆，也不按“Unity API”拆。

更稳的主线是：

1. 先回答技能系统在解决什么问题
2. 再切清概念边界
3. 再立数据骨架和生命周期
4. 再把输入、校验、目标、命中接成执行主链
5. 再把世界变化收回 Effect / Buff 两层
6. 再处理表现、AI、网络这些高复杂度交叉边界
7. 最后收束到工具、测试和框架映射
